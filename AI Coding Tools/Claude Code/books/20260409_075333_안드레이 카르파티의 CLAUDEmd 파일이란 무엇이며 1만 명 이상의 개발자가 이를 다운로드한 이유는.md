안드레이 카르파티의 CLAUDE.md 파일이란 무엇이며, 1만 명 이상의 개발자가 이를 다운로드한 이유는?

2025년 11월, 안드레이 카르파티는 여전히 대부분의 코드를 직접 손으로 작성했습니다. 하지만 12월이 되자 더 이상 그러지 않았습니다. 2026년 1월에 널리 공유된 게시물에서 그는 단 몇 주 만에 80% 수동 코딩에서 클로드 코드(Claude Code)를 활용한 80% 에이전트 기반 코딩으로 전환된 과정을 설명했습니다. 그는 이를 약 20년간의 프로그래밍 경력에서 가장 큰 작업 흐름(workflow) 변화라고 불렀습니다. 점진적인 변화가 아닌, 상전이(phase shift)였습니다.

이 게시물은 많은 논의를 불러일으켰습니다. 그러나 그 안에는 AI와 소프트웨어 엔지니어링의 미래에 대한 거창한 주장보다 더 작고 실용적인 내용이 숨겨져 있었습니다. 그는 하나의 파일을 언급했습니다. CLAUDE.md라는 일반 마크다운(markdown) 파일이었습니다. 그가 이 파일에 대해 이야기하는 방식, 즉 이 파일이 해결하고자 했던 문제와 여전히 해결할 수 없었던 점들이 대부분의 사람들이 인식했던 것보다 더 중요하게 다가왔습니다.

### CLAUDE.md의 실제 용도

CLAUDE.md는 Anthropic의 터미널(terminal) 기반 코딩 에이전트인 클로드 코드(Claude Code)를 위한 구성 파일(configuration file)입니다. 이 파일을 프로젝트의 루트(root)에 두면 클로드 코드가 모든 세션을 시작할 때 자동으로 파일을 읽습니다. 이 파일은 대화가 시작되기 전에 모델의 작업 컨텍스트(working context)의 일부가 됩니다.

이를 상시 지시 문서로 생각할 수 있습니다. 매번 처음부터 설명해야 할 사항들, 예를 들어 테스트 실행 방법, 폴더 구조의 의미, 코드베이스(codebase)의 명명 규칙(naming conventions), 모델이 절대 건드려서는 안 되는 부분 등을 기록합니다. 클로드가 이를 읽으면 반복적으로 설명할 필요가 없습니다.

이것은 마크다운 파일로, 일반 텍스트이며 특별한 문법은 없습니다. 프로젝트 루트에 전체 팀을 위해 두거나, 모든 프로젝트에 적용되는 개인 구성(config) 폴더에 둘 수 있습니다. 다른 하위 디렉토리(subdirectories)에 여러 CLAUDE.md 파일을 가질 수 있으며, 클로드는 작업 중인 코드에 가장 가까운 파일을 적용합니다.

이것이 기본적인 메커니즘입니다. 하지만 카르파티가 언급했고 1만 명 이상의 개발자가 찾아 나선 이 파일은 단순히 폴더 구조나 테스트 명령어에 관한 것이 아닙니다.

### 해결하고자 했던 문제

카르파티의 2026년 1월 게시물은 단순히 생산성 향상에 관한 것이 아니었습니다. 그 중요 부분은 LLM(대규모 언어 모델)이 코드를 작성할 때 실패하는 구체적인 방식들의 목록이었으며, 모호하지 않으므로 주의 깊게 읽어볼 가치가 있습니다. 그가 지적한 문제점들은 다음과 같습니다:

*   모델은 잘못된 가정을 하고 확인 없이 계속 진행한다.
*   혼란스러울 때 플래그를 지정하지 않고, 절충안(tradeoffs)을 제시하지 않으며, 이의를 제기하지 않는다.
*   기본적으로 너무 복잡하게 만들어 100줄짜리 솔루션을 1,000줄로 부풀린다.
*   완전히 이해하지 못하는, 작업과 인접한 코드를 건드린다.
*   작업 후 사용하지 않는 코드(dead code)를 정리하지 않는다.

이것들은 드문 예외적인 경우가 아닙니다. 클로드 코드(Claude Code)나 커서(Cursor)를 몇 번 이상 사용해 본 사람이라면 누구나 이 모든 문제들을 인식할 것입니다. 작은 변경을 요청했는데, 6개의 파일을 건드리는 변경 사항(diff)을 받게 됩니다. 그리고 요청하지 않은 코드를 감사(auditing)하는 데 다음 20분을 소비하게 됩니다. 답답한 점은 이러한 실수들이 예측 가능하다는 것입니다. 동일한 방식으로 계속해서 발생합니다. 그리고 그러한 예측 가능성(predictability)이 바로 구성 파일(configuration file)이 다룰 수 있는 부분입니다.

### 확산된 카르파티에서 영감을 받은 파일

포레스트 창(Forrest Chang)이라는 개발자는 카르파티의 관찰 내용을 바탕으로, 깃허브(GitHub)의 'andrej-karpathy-skills' 저장소에 단일 다운로드 가능한 CLAUDE.md 파일을 게시했습니다. 이 파일은 모든 것을 시도하지 않습니다. 오직 네 가지 원칙만을 포함합니다:

*   **가정하지 말라. 혼란을 숨기지 말라. 절충안(tradeoffs)을 명확히 하라.**
    모델이 불확실하다면 질문해야 합니다. 여러 해석이 존재한다면, 조용히 선택하는 대신 이를 명시해야 합니다. 더 간단한 접근 방식이 있다면, 언급해야 합니다. 이 원칙은 카르파티가 설명한 암묵적인 가정(silent assumption) 문제에 대한 직접적인 대응입니다.
*   **문제를 해결하는 최소한의 코드. 추측성 작업은 금지.**
    요청된 것 이상의 기능은 만들지 않습니다. 한 번만 사용될 코드에 대해 추상화(abstractions)를 구축하지 않습니다. 요청되지 않은 유연성(flexibility)은 제공하지 않습니다. 내부 테스트는 다음과 같습니다: 시니어 엔지니어(senior engineer)라면 이것을 과도하게 복잡하다고 부를 것인가? 그렇다면 단순화합니다.
*   **필요한 것만 건드려라. 자신이 만든 것만 정리하라.**
    인접한 코드를 개선하지 말고, 작업의 일부가 아니었던 것을 다시 포맷(reformat)하지 말며, 모델이 이해하지 못하는 주석을 삭제하지 않습니다. 변경된 모든 줄은 실제로 요청된 사항에 직접적으로 연결되어야 합니다.
*   **성공 기준을 정의하라. 검증될 때까지 반복하라.**
    모델에 단계별로 무엇을 할지 지시하는 대신, 완료된 모습이 어떠해야 하는지 설명합니다. LLM은 엄격한 절차를 따르는 것보다 목표를 향해 반복(iterating)하는 데 더 능숙합니다. 통과할 테스트를 주고 반복하도록 합니다.

이것이 파일의 전부입니다. 대략 60줄 정도 됩니다.

이 저장소는 3,500개 이상의 깃허브(GitHub) 스타를 얻었습니다. 이를 기반으로 VS 코드(VS Code) 및 커서(Cursor) 확장 프로그램이 만들어졌고, 소스포지(SourceForge) 미러(mirror)도 등장했습니다. 미디엄(Medium) 게시물들이 이를 설명했으며, 개발자들은 단 하나의 curl 명령어로 자신의 프로젝트에 이 파일을 추가하기 시작했습니다.

### 공감을 얻은 이유

이 파일이 확산된 이유 중 하나는 카르파티의 본래 관찰이 구체적이고 솔직했기 때문입니다. 그는 아무것도 판매하지 않았습니다. 그는 자신이 실제로 겪은 불편한 부분들을 포함하여 관찰한 내용을 설명했습니다. 예를 들어, 수동 코딩 능력이 퇴화하는 것을 느끼기 시작했다는 점, AI에게 무엇을 작성하라고 지시하는 것이 이상하게 느껴졌다는 점, 그리고 CLAUDE.md를 사용했음에도 문제들이 완전히 사라지지 않았다는 점 등입니다.

마지막 부분은 주목할 가치가 있습니다. 카르파티조차도 나중 게시물에서 CLAUDE.md를 최신 상태로 유지하는 좋은 방법을 찾지 못했다고 인정했습니다. 이 파일은 도움이 되지만, 모든 것을 해결하지는 않습니다. 개발자들에게는 이러한 솔직함이 생산성 과장 기사에서는 느낄 수 없는 신뢰감을 주었습니다. 그는 완벽한 시스템을 묘사한 것이 아니었습니다. 그는 실제 마찰(friction)이 있는 실제 작업 흐름(workflow)을 설명했고, 이 파일은 특정 유형의 마찰에 대한 하나의 실용적인 대응책이었습니다.

네 가지 원칙이 공감을 얻은 또 다른 이유는 개발자들이 이미 스스로 관찰하고 있던 내용과 일치했기 때문입니다. 카르파티가 언급한 문제들은 이론적인 것이 아닙니다. AI 생성 코드 후의 정리 작업(cleanup pass), 슬그머니 들어오는 불필요한 추상화(abstractions), 모델이 의미를 확신하지 못해 사라지는 주석 등 이러한 일들은 실제로 발생합니다. 파일에 규칙을 추가하는 것이 이러한 문제들을 완전히 없애지는 못하지만, 줄여주며, 이 정도의 감소만으로도 일상적인 작업 흐름(workflow)에 충분한 변화를 가져올 수 있습니다.

### 실제 내용 및 사용 방법

카르파티에서 파생된 파일을 사용하는 가장 간단한 방법은 터미널(terminal)에서 다음 단일 명령어를 실행하는 것입니다:

```bash
curl -o CLAUDE.md https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md
```

이 명령어는 현재 프로젝트 디렉토리(directory)에 파일을 배치합니다. 클로드 코드(Claude Code)는 다음 세션(session) 시작 시 자동으로 이를 인식합니다. 만약 이미 CLAUDE.md 파일이 있다면, 교체하는 대신 추가할 수 있습니다:

```bash
echo "" >> CLAUDE.md
curl https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md >> CLAUDE.md
```

이 파일은 최종 버전이 아닌 시작점으로 설계되었습니다. 이 파일을 사용해 본 여러 개발자들의 권장 사항은 다음과 같습니다. 카르파티 원칙 아래에 프로젝트별 컨벤션(conventions)을 추가하고, 실제 작업 방식에 적용되지 않는 모든 것을 제거하세요. 이론적으로 모든 것을 다루는 200줄짜리 CLAUDE.md는 60줄이면서 실제로 정확한 파일보다 덜 유용합니다. 알아둘 가치가 있는 한 가지 실질적인 제약 사항은 CLAUDE.md의 모든 줄이 모델의 컨텍스트 창(context window) 공간을 실제 작업과 경쟁한다는 것입니다. 최신 모델(frontier models)은 약 150~200개의 지시를 합리적인 일관성으로 따를 수 있지만, 클로드 코드(Claude Code) 자체의 시스템 프롬프트(system prompt)는 파일이 로드되기도 전에 이미 그 중 약 50개를 사용합니다. 짧을수록 좋습니다.

### 해결하지 못하는 점

카르파티 파일은 과도한 설계(overengineering)를 줄이고 암묵적인 가정(silent assumptions)을 줄여줄 것입니다. 하지만 모델을 완벽하게 만들지는 못하며, 프로젝트가 발전함에 따라 자동으로 업데이트되지도 않습니다. 카르파티도 그렇게 말했습니다. 보다 솔직한 관점에서 보면: CLAUDE.md는 어차피 직접 할 수정 사항들을 미리 인코딩(encoding)하는 방법입니다. 매 세션마다 동일한 유형의 실수를 찾아내고 매번 수정 사항을 설명하는 대신, 한 번 작성하면 자동으로 적용됩니다. 정리 작업(cleanup pass)은 여전히 가끔 발생하지만, 그 횟수가 줄어들 뿐입니다. 이것이 가치 있는지는 클로드 코드(Claude Code)를 얼마나 자주 사용하는지, 그리고 반복되는 마찰(friction)로 인해 얼마나 많은 비용이 발생하는지에 달려 있습니다. 이미 하루 대부분을 에이전트 작업 흐름(agent workflow)에서 보내는 개발자들에게는 일반적으로 '그렇다'입니다.

이 파일은 다루는 문제가 실제적이고, 해결책이 실제로 사용하기에 충분히 작으며, 과장할 이유가 없는 출처에서 비롯되었기 때문에 확산되었습니다.
