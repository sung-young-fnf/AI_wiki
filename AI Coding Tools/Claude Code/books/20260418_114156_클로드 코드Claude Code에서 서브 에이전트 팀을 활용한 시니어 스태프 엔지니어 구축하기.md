# 클로드 코드(Claude Code)에서 서브 에이전트 팀을 활용한 시니어 스태프 엔지니어 구축하기
## TDD, 서브 에이전트 위임, 사양 검토, 포렌식 디버깅 등

주니어 엔지니어와 시니어 엔지니어의 차이는 구문 지식이 아니라 예측 가능성, 위험 관리, 그리고 압박감 속에서의 규율입니다. AI 에이전트들은 바로 이 지점에서 어려움을 겪습니다. 이들은 검토를 건너뛰고, 지름길을 합리화하며, 스스로는 자랑스러워하지만 아무도 실제로 검증하지 않은 결과물을 내놓습니다. 앤스로픽(Anthropic)은 에이전트들이 잘 협력할 수 있음을 보여주지만, 구조가 없으면 더 많은 에이전트가 더 많은 혼돈과 컴퓨팅 자원 낭비를 초래할 뿐입니다. 필요한 것은 실제 기업들이 가진 것과 같습니다. 즉, 설계부터 배포까지 규율 잡힌 파이프라인을 통해 전문화된 서브 에이전트(sub-agents)들을 조율하는 스태프 엔지니어(Staff Engineer)입니다. 그리고 이것이 바로 우리가 구축하려는 것입니다.

이 시스템은 책임이 명확하게 분리된 실제 엔지니어링 조직처럼 작동합니다:

*   **발견 및 설계(Discovery & Design):** 아키텍트(Architect)는 한 번에 하나의 질문을 하고, 장단점(trade-offs)을 가진 2~3가지 접근 방식을 탐색하며, Git에 커밋(commit)된 설계 사양(design spec)을 만듭니다. 인간의 승인 없이는 코드를 작성하지 않습니다.
*   **계획 및 검토(Planning & Review):** 테크 리드(Tech Lead)는 사양(spec)을 정확한 파일 경로와 완전한 코드를 포함하는 2~5분 분량의 작업으로 나눕니다. 새로운 서브 에이전트가 사양에 따라 계획을 감사하여 누락된 부분과 범위 확장(scope creep)을 잡아냅니다.
*   **실행 엔진(Execution Engine):** 매니저(Manager)는 각 작업에 대해 격리된 컨텍스트(context)를 가진 새로운 서브 에이전트를 파견합니다. 진행이 막힌 에이전트는 문제를 에스컬레이션(escalate)합니다. 불량 작업은 종료되고 재파견됩니다.
*   **품질 관문(Quality Gates):** 두 단계의 순차적 검토가 이루어집니다. 사양 준수(올바른 것인가?) 후 코드 품질(잘 만들어졌는가?)을 검토합니다. 검증 관문(verification gate)은 완료를 주장하기 전에 최신 터미널 출력(terminal output)을 요구합니다.
*   **엔지니어링 규율(Engineering Discipline):** TDD(Test-Driven Development)가 철칙으로 적용됩니다. 디버깅(Debugging)은 4단계의 포렌식(forensic) 프로세스를 따릅니다. 근본 원인 추적(Root cause tracing)은 5단계 깊이까지 진행됩니다. 기술(skill) 자체도 배포 전에 실제 에이전트를 대상으로 압력 테스트를 거칩니다.

이 블로그에서는 이 워크플로의 각 단계를 시각적으로 살펴보고, 훅(hook)부터 스킬(skill), 에이전트(agent)에 이르기까지 모든 구성 요소를 단계별로 구축하며, 클로드 코드(Claude Code)를 사용하여 이들이 규율 잡힌 엔지니어링 팀으로서 어떻게 함께 작동하는지 알아볼 것입니다.

## 팀 코드베이스 생성

모든 엔지니어링 조직에서 첫 번째 제품 코드가 작성되기 전에 누군가는 인프라를 설정해야 합니다. 스포티파이(Spotify), 쇼피파이(Shopify), 스트라이프(Stripe)와 같은 회사에서는 새로운 이니셔티브(initiative)에서 스태프 엔지니어의 첫 번째 임무가 바로 이것입니다.

이들은 기능 코딩부터 시작하지 않습니다. 프로젝트 구조, 개발 워크플로, CI/CD(Continuous Integration/Continuous Delivery) 파이프라인(pipeline), 그리고 팀 규범을 확립합니다. 이들은 다른 모든 사람의 작업을 가능하게 하는 조직적 기반(scaffolding)을 구축합니다.

클로드 코드(Claude Code)를 일반적으로 사용할 때는 몇 가지 스킬, MCP 서버, 도구 스크립트를 사용하거나, 아예 처음부터 시작하여 클로드에게 맡길 수도 있습니다.

하지만 우리는 단독 개발자를 만드는 것이 아닙니다. 우리는 협업하고, 프로세스를 따르며, 서로에게 책임을 묻는 에이전트 팀을 구축하고 있습니다. 이는 구조를 필요로 합니다. 10명으로 구성된 엔지니어링 팀이 프리랜서 한 명에게는 필요 없는 구조를 필요로 하는 것과 같습니다.

따라서 어떤 스킬을 작성하거나 에이전트를 정의하기 전에, 우리는 조직적 기반을 만들어야 합니다:

```
senior_staff_engineer/
├── agents/                             # 에이전트 및 서브 에이전트 정의
├── commands/                           # 커스텀 CLI 스타일 에이전트 명령
├── hooks/                              # 생명주기 훅(이벤트, 워크플로 제어)
├── skills/                             # 핵심 역량 모듈
│   ├── brainstorming/                  # 아이디어 생성 및 설계
│   │   └── scripts/                    # 시각/인터랙티브 브레인스토밍
│   ├── dispatching-parallel-agents/    # 여러 에이전트 조정
│   ├── executing-plans/                # 단계별 계획 실행
│   ├── finishing-a-development-branch/ # 병합 전 작업 마무리
│   ├── receiving-code-review/          # 피드백 처리 및 반복
│   ├── requesting-code-review/         # PR 준비 및 검토 요청
│   ├── subagent-driven-development/    # 서브 에이전트에게 위임
│   ├── systematic-debugging/           # 근본 원인 분석
│   ├── test-driven-development/        # 테스트 작성 → 구현 → 검증
│   ├── using-git-worktrees/            # 워크트리를 통한 작업 격리
│   ├── using-senior-staff-engineer/    # 이 시스템 구성 및 최적화
│   ├── verification-before-completion/ # 완료 전 최종 검증
│   ├── writing-plans/                  # 계획 생성
│   └── writing-skills/                 # 새로운 스킬 작성
```

처음에는 복잡해 보일 수 있지만, 각 디렉터리는 실제 엔지니어링 조직의 역할이나 기능에 매핑됩니다.

`agents/` 디렉터리는 누가 무엇을 하는지 정의하는 팀 명단입니다.
`skills/` 디렉터리는 모든 팀원이 따르는 프로세스와 표준을 담고 있는 직원 핸드북입니다.
`hooks/` 디렉터리는 정보의 흐름을 제어하고 모든 사람이 정렬된 상태로 시작하도록 보장하는 관리 계층입니다.

우리는 이들을 하나씩 구축하여, 계획부터 배포까지 복잡한 개발 작업을 처리하는 응집력 있는 팀으로 작동하게 만들 것입니다.

## 훅(Hooks) (관리 계층)

모든 회사에는 보이지 않게 작동하는 관리 인프라 계층이 있습니다. 하루의 첫 회의가 시작되기 전에 이미 사무실은 열려 있고, 커피 머신은 작동하며, 공유 캘린더는 동기화되어 있고, 팀의 슬랙(Slack) 채널에는 밤새 업데이트된 내용이 채워져 있습니다. 누구도 이 인프라가 고장 나기 전까지는 생각하지 않습니다.

우리 시스템에서 `hooks/`는 이러한 관리 계층입니다.

클로드 코드(Claude Code)에서 훅(hook)은 세션이 시작되거나, 컨텍스트가 지워지거나, 에이전트가 메모리를 압축하는 등 특정 생명주기(lifecycle) 이벤트에서 발생하는 작업을 제어할 수 있게 해줍니다. 단일 개발자에게 훅은 선택 사항이지만, 공유된 컨텍스트와 일관된 규칙으로 모든 세션을 시작해야 하는 에이전트 팀에게는 필수적입니다.

우리는 세 개의 파일로 시작합니다:

```
hooks/
├── hooks.json    # 주요 설정 파일
├── run-hook.cmd  # 크로스 플랫폼 스크립트 실행기
├── session-start # 세션 초기화 로직
```

`hooks.json`은 간단합니다. 이 파일은 세션이 시작될 때마다 초기화 스크립트를 실행하는 한 가지 규칙을 정의합니다.

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup|clear|compact",
        "hooks": [
          {
            "type": "command",
            "command": "\"${PROJECT_ROOT_DIR}/hooks/run-hook.cmd\" session-start",
            "async": false
          }
        ]
      }
    ]
  }
}
```

매처(matcher)는 세 가지 시나리오에서 트리거됩니다: `startup`(새 세션), `clear`(컨텍스트 재설정), `compact`(메모리 압축).
조직적인 관점에서 보면, 이것은 첫날뿐만 아니라 누군가가 이 프로젝트로 컨텍스트를 전환할 때마다 실행되는 온보딩(onboarding) 체크리스트를 갖는 것과 같습니다.

AI 에이전트 관점에서는 모든 세션이 '첫째 날'입니다. 에이전트는 어제 세션에 대한 기억이 없습니다. 매번 다시 온보딩되어야 합니다.

`async: false` 플래그는 에이전트가 응답하기 시작하기 전에 훅(hook)이 완료되도록 보장합니다. 이는 '아무도 노트북을 열기 전에 아침 스탠드업 회의가 끝나야 한다'고 말하는 것과 같습니다.

실제 회사에서는 모든 사람이 동일한 OS를 사용하지 않습니다. 엔지니어링 팀에는 Mac 사용자, Linux 서버, 그리고 가끔 Windows 머신이 있을 수 있습니다. 우리는 Windows와 Unix 모두에서 작동하는 다국어(polyglot) 스크립트인 `run-hook.cmd` 스크립트를 만들어 이 문제를 처리해야 합니다:

```bash
if "%~1"=="" (
    echo run-hook.cmd: missing script name >&2
    exit /b 1
)

set "HOOK_DIR=%~dp0"

REM Try Git for Windows bash in standard locations

if exist "C:\Program Files\Git\bin\bash.exe" (
    "C:\Program Files\Git\bin\bash.exe" "%HOOK_DIR%%~1" %2 %3 %4 %5 %6 %7 %8 %9
    exit /b %ERRORLEVEL%
)
```

Windows에서는 일반적인 Git 설치 경로에서 Bash를 검색합니다. Bash가 어디에도 없으면 충돌하는 대신 조용히 종료됩니다. Unix에서는 스크립트를 직접 실행합니다:

```bash
# Unix: run the named script directly
SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
SCRIPT_NAME="$1"

shift

exec bash "${SCRIPT_DIR}/${SCRIPT_NAME}" "$@"
```

에이전트는 자신이 어떤 OS에서 실행되고 있는지 알 필요가 없습니다. 게이트웨이 스크립트가 이를 파악하여 적절히 라우팅(route)합니다. 이는 도커(Docker)와 쿠버네티스(Kubernetes)의 기본 원리와 동일합니다: 애플리케이션 로직이 환경에 신경 쓸 필요 없도록 환경을 추상화하는 것입니다.

`session-start` 스크립트는 실제 온보딩이 이루어지는 곳입니다. 새 세션이 시작되면 이 스크립트는 스쿼드(squad)의 핵심 스킬 파일을 로드하고 에이전트의 컨텍스트에 주입합니다.

이는 스태프 엔지니어가 작업 시작 전에 모든 팀원에게 팀의 운영 원칙을 상기시키는 아침 스탠드업 회의와 같다고 생각할 수 있습니다.

스크립트는 경로를 해결하고 레거시 구성(legacy configurations)을 확인하는 것으로 시작합니다:

```bash
#!/usr/bin/env bash
set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
PLUGIN_ROOT="$(cd "${SCRIPT_DIR}/.." && pwd)"

# Legacy config check
warning_message=""

legacy_skills_dir="${HOME}/.config/staff_engineer/skills"

if [ -d "$legacy_skills_dir" ]; then
    warning_message="\n\n<important-reminder>⚠️ **WARNING:** 
    staff_engineer now uses Claude Code's skills system. 
    Move custom skills to ~/.claude/skills instead.</important-reminder>"
fi
```

그런 다음 모든 에이전트가 읽어야 하는 직원 핸드북인 핵심 스킬 파일을 로드합니다:

```bash
using_staff_engineer_content=$(
    cat "${PLUGIN_ROOT}/skills/using-staff_engineer/SKILL.md" 2>&1 \
    || echo "Error reading using-staff_engineer skill"
)
```

내용은 JSON을 위해 이스케이프(escape)되고 컨텍스트 페이로드(payload)로 래핑됩니다:

```bash
session_context="<EXTREMELY_IMPORTANT>\nYou have staff_engineer.

\n\n**Below is the full content of your 'staff_engineer:using-

staff_engineer' skill - your introduction to using skills. For 

all other skills, use the 'Skill' tool:**\n\n

${using_staff_engineer_escaped}\n\n${warning_escaped}\n

</EXTREMELY_IMPORTANT>"
```

마지막으로, 스크립트는 페이로드를 실행되는 플랫폼에 특화된 형식으로 출력합니다:

```bash
if [ -n "${CURSOR_PLUGIN_ROOT:-}" ]; then
    printf '{\n  "additional_context": "%s"\n}\n' "$session_context"

elif [ -n "${CLAUDE_PLUGIN_ROOT:-}" ] && [ -z "${COPILOT_CLI:-}" ]; then
    printf '{\n  "hookSpecificOutput": {\n    "hookEventName": 
    "SessionStart",\n    "additionalContext": "%s"\n  }\n}\n' \
    "$session_context"
else
    printf '{\n  "additionalContext": "%s"\n}\n' "$session_context"

fi
```

클로드 코드(Claude Code), 커서(Cursor), 코파일럿 CLI(Copilot CLI)는 각각 약간 다른 JSON 형식으로 컨텍스트를 기대합니다. 우리 스크립트는 이 세 가지 모두를 처리하며, 이는 동일한 스쿼드(squad)가 수정 없이 여러 AI 코딩 플랫폼에서 작동할 수 있음을 의미합니다.

이것은 스태프 엔지니어들이 인프라 작업에 가져오는 크로스 플랫폼(cross-platform) 사고방식으로, 자신의 머신에서만 작동하는 것이 아니라 모든 곳에서 작동하는 것을 구축하는 것입니다.

## 직원 핸드북 (1% 규칙)

모든 조직에는 문화를 확립하는 창립 문서가 있습니다. 아마존(Amazon)에는 리더십 원칙(Leadership Principles)이 있고, 넷플릭스(Netflix)에는 문화 데크(Culture Deck)가 있으며, 브리지워터(Bridgewater)에는 원칙(Principles)이 있습니다. 이 문서들은 회사가 무엇을 하는지뿐만 아니라, 회사 내 모든 사람이 어떻게 생각해야 하는지를 설명합니다.

`skills/using-senior-staff-engineer/SKILL.md`는 우리의 문화 데크입니다. 이것은 세션 시작 시 모든 에이전트의 컨텍스트에 로드되는 첫 번째 항목이며, 전체 시스템에서 가장 중요한 단일 규칙을 확립합니다:

```
<EXTREMELY-IMPORTANT>
수행 중인 작업에 스킬(skill)이 적용될 가능성이 1%라도 있다고 생각되면, 반드시 해당 스킬을 호출해야 합니다.

작업에 스킬이 적용된다면 선택의 여지가 없습니다. 반드시 사용해야 합니다.
이것은 협상할 수 없으며, 선택 사항이 아닙니다. 이것을 합리화하여 회피할 수 없습니다.
</EXTREMELY-IMPORTANT>
```

이것이 바로 1% 규칙(1% Rule)이며, 다른 모든 것의 기반이 됩니다. 이 규칙이 없으면 에이전트는 스킬을 제안으로만 취급합니다. 이들은 작업이 "간단해 보여서" 브레인스토밍 단계를 건너뛰고, "단순한 설정 변경이므로" TDD를 건너뛰며, "두 줄짜리 코드일 뿐이므로" 코드 검토를 건너뛸 것입니다.

모든 시니어 엔지니어는 팀이 이러한 합리화를 통해 서서히 프로세스를 포기하는 것을 목격해 왔으며, 그 결과는 항상 동일합니다: 품질 저하, 버그 배포, 그리고 모두가 무슨 일이 일어났는지 의아해합니다.

이 스킬은 또한 서브 에이전트를 위한 비상 탈출구(escape valve)를 포함해야 합니다. 메인 에이전트가 특정 작업을 구현하기 위해 집중된 서브 에이전트를 파견할 때, 해당 서브 에이전트는 전체 스킬 발견 흐름을 거칠 필요가 없습니다:

```
<SUBAGENT-STOP>
특정 작업을 실행하기 위한 서브 에이전트(subagent)로 파견된 경우, 이 스킬은 건너뛰세요.
</SUBAGENT-STOP>
```

이것은 조직적인 관점에서 계약자에게 '전체 회의에 참석할 필요 없이, 고용된 작업을 수행하세요'라고 말하는 것과 같습니다. 메인 에이전트는 전체 프로세스를 따르고, 서브 에이전트는 작업에 집중합니다.

## 지시 우선순위 (명령 체계)

모든 조직에는 명령 체계(chain of command)가 있으며, 계층 간의 충돌은 명확한 해결 규칙을 가져야 합니다. 이 스킬은 세 가지 수준을 정의합니다:

```
## 지시 우선순위

1. **사용자의 명시적 지시** (CLAUDE.md, 직접 요청) 
   - 가장 높은 우선순위

2. **senior_tech_engineers 스킬** - 충돌 시 기본 시스템 동작을 재정의

3. **기본 시스템 프롬프트** - 가장 낮은 우선순위

만약 CLAUDE.md가 "TDD를 사용하지 마세요"라고 말하고 스킬이 "항상 TDD를 사용하세요"라고 말한다면, 사용자의 지시를 따르세요. 사용자가 통제권을 가집니다.
```

인간(VP, 창업자, 제품 책임자)이 항상 최종 권한을 가집니다. 실제 회사에서 이것은 프로세스 중심 문화와 관료주의의 차이입니다.

프로세스 중심 문화는 더 나은 결과를 낳기 때문에 프로세스를 따르지만, 상황이 요구할 때는 프로세스를 무시합니다.
관료주의는 나쁜 결과를 낳을 때조차도 프로세스를 맹목적으로 따릅니다.

## 스킬 발견 흐름

핸드북에는 에이전트가 어떤 메시지에 응답하기 전에 스킬을 어떻게 확인하고 호출해야 하는지 정확히 정의하는 흐름도가 포함되어 있습니다. 흐름은 다음과 같습니다: 메시지를 받고, 어떤 스킬이 적용될 수 있는지 확인(1%의 가능성이라도), 적용되면 스킬을 호출하고, 어떤 스킬이 사용되는지 알리고, 스킬의 체크리스트나 지시사항을 따른 다음, 그 후에야 응답합니다.

이것을 효과적으로 만드는 것은 '적색 신호(red flags)' 표인데, 이는 에이전트가 프로세스를 따르지 않기 위해 합리화하고 있음을 나타내는 생각들의 목록입니다:

| 생각                     | 현실                                  |
|------------------------|---------------------------------------|
| "이건 단순한 질문일 뿐이야"   | 질문은 작업입니다. 스킬을 확인하세요. |
| "먼저 더 많은 컨텍스트가 필요해" | 스킬 확인이 질문보다 먼저 옵니다.     |
| "먼저 코드베이스를 살펴보자" | 스킬이 탐색 방법을 알려줍니다. 먼저 확인하세요. |
| "이건 정식 스킬이 필요 없어" | 스킬이 존재하면 사용하세요.           |
| "이 스킬을 기억하고 있어"     | 스킬은 진화합니다. 현재 버전을 읽으세요. |
| "스킬이 너무 과해"         | 단순한 것도 복잡해집니다. 사용하세요. |
| "이건 생산적인 것 같아"      | 규율 없는 행동은 시간을 낭비합니다.     |

이 표의 모든 항목은 에이전트가 스킬을 건너뛰려고 시도했던 실제 테스트에서 비롯되었습니다. "이건 단순한 질문일 뿐이야"가 가장 흔한 경우입니다. 에이전트는 작업을 사소하다고 판단하고 스킬을 건너뛰어 중요한 것을 놓칩니다.

실제 팀에서는, 이것은 "단 한 줄 변경일 뿐이야"라고 말하고 테스트 없이 배포했다가, 그 한 줄이 아무도 예상치 못한 부작용을 일으켜 프로덕션(production)을 다운시키는 개발자와 같습니다.

핸드북은 또한 여러 스킬이 적용될 수 있을 때 스킬 우선순위를 정의합니다:

```
## 스킬 우선순위

1. **프로세스 스킬 먼저** (브레인스토밍, 디버깅) - 이들은 작업을 어떻게 접근할지 결정합니다.

2. **구현 스킬 다음** (프런트엔드 설계, MCP 빌더) - 이들은 실행을 안내합니다.

"X를 만들자" → 먼저 브레인스토밍, 다음 구현 스킬.

"이 버그를 고쳐라" → 먼저 디버깅, 다음 도메인 특정 스킬.
```

그리고 엄격한(rigid) 스킬과 유연한(flexible) 스킬의 구별:

```
**엄격한** (TDD, 디버깅): 정확히 따르세요. 규율을 벗어나 재량껏 행동하지 마세요.

**유연한** (패턴): 원칙을 컨텍스트에 맞게 적용하세요.

스킬 자체가 어떤 범주에 속하는지 명시합니다.
```

당면한 문제에 따라 브레인스토밍 기법을 조정합니다. 핵심 통찰은 스킬 자체가 자신이 속한 범주를 선언하여, 에이전트가 지름길을 택하고 싶을 때 엄격한 스킬을 유연한 것으로 재분류할 수 있는 능력을 제거한다는 것입니다.

마지막으로, 핸드북은 에이전트(및 인간)가 끊임없이 사용하는 미묘한 합리화에 대해 다룹니다:

```
## 사용자 지시
지시는 무엇(WHAT)을 할지 말하며, 어떻게(HOW) 할지는 말하지 않습니다. "X를 추가하라" 또는 "Y를 고쳐라"는 워크플로를 건너뛰라는 의미가 아닙니다.
```

사용자가 "로그인 페이지를 추가하라"고 말하면, 에이전트는 "무엇을 만들지 알려줬으니, 그냥 만들면 되겠네"라고 생각할 수 있습니다.
하지만 "로그인 페이지 추가"는 '무엇(WHAT)'입니다. '어떻게(HOW)'는 여전히 브레인스토밍, 설계, 계획, TDD로 구현, 검토, 그리고 배포입니다. 사용자의 지시가 프로세스를 무시하는 것이 아니라, 프로세스에 입력으로 제공될 뿐입니다.

이제 관리 인프라, 온보딩 흐름, 그리고 문화 핸드북을 설정했으니, 시니어 스태프 엔지니어 에이전트가 작업을 효과적으로 수행할 수 있도록 실제 스킬을 구축하기 시작할 수 있습니다.

## 브레인스토밍 및 설계 (계획 단계)

어떤 코드도 존재하기 전에 가장 중요한 작업이 이루어지는 순간이 있습니다. 아마존(Amazon)에서는 제품을 만들기 전에 보도 자료를 작성합니다. 베이스캠프(Basecamp)에서는 피치 문서(pitch document)를 작성합니다. 구글(Google)에서는 단일 함수가 정의되기 전에 세 단계의 엔지니어에게 검토받는 설계 문서(design doc)를 작성합니다.

공통점은 '구축 전에 생각이 선행된다'는 것입니다.

그 이유는 간단하며 배우는 데 비용이 많이 듭니다. 모든 시니어 스태프 엔지니어는 설계 단계를 건너뛴 프로젝트에 대한 이야기를 가지고 있습니다. 누군가 "이건 간단하니까 바로 코딩을 시작하자"라고 말했습니다. 3주 후, 팀은 이해관계자들이 원했던 것과 맞지 않는 구현을 절반쯤 진행했고, 아키텍처는 아무도 묻지 않았던 요구사항을 지원할 수 없었으며, 재작성 비용은 전체 프로젝트의 원래 예상보다 더 많이 들었습니다.

스트라이프(Stripe)와 같은 회사에서는 스태프 엔지니어에게 명시적으로 "타이핑을 멈추고 생각하기 시작하라"고 말하는 역할을 기대합니다. 이는 생각이 즐거워서가 아니라(엔지니어는 코딩을 하고 싶어 합니다), 한 시간의 설계가 일주일의 재작업을 방지하기 때문입니다. 브레인스토밍(brainstorming) 스킬은 이러한 규율을 담고 있습니다.

`skills/brainstorming/SKILL.md`는 TDD 철칙에 해당하는 설계 규칙으로 시작합니다:

```
<HARD-GATE>
설계를 제시하고 사용자의 승인을 받을 때까지는 어떤 구현 스킬도 호출하거나, 코드를 작성하거나, 프로젝트를 스캐폴딩(scaffold)하거나, 구현 작업을 수행하지 마십시오. 이는 인지된 단순성에 관계없이 모든 프로젝트에 적용됩니다.
</HARD-GATE>
```

이것은 제안이 아니라 '하드 게이트(hard gate)'입니다. 설계가 존재하고 인간이 승인할 때까지 에이전트는 물리적으로 코딩으로 진행할 수 없습니다. 조직적인 관점에서, 이것은 개발자와 코드베이스 사이에 서서 "설계 문서를 먼저 보여주세요"라고 말하는 스태프 엔지니어와 같습니다.

### "너무 간단해"라는 함정

소프트웨어 엔지니어링에서 가장 위험한 합리화를 다루는 스킬이 필요합니다:

```
## 안티 패턴: "이것은 설계가 필요 없을 정도로 너무 간단하다"

모든 프로젝트는 이 과정을 거칩니다. 할 일 목록, 단일 기능 유틸리티, 설정 변경 — 이 모든 것들입니다. "간단한" 프로젝트는 검토되지 않은 가정으로 인해 가장 많은 작업 낭비를 초래하는 곳입니다. 설계는 짧을 수 있지만(정말 간단한 프로젝트의 경우 몇 문장), 반드시 제시하고 승인을 받아야 합니다.
```

모든 시니어 엔지니어는 이 교훈을 어렵게 배웠습니다. "간단한" 설정 변경이 세 가지 서비스에 영향을 미치는 것으로 밝히지기도 합니다.
아무도 언급하지 않았던 7가지 엣지 케이스(edge case)를 처리해야 했던 "빠른" 유틸리티 함수.
실제로 오프라인 동기화, 충돌 해결, 다중 사용자 지원이 필요했던 "사소한" 할 일 앱.

간단한 프로젝트는 아무도 검토를 멈추지 않기 때문에 검토되지 않은 가정이 숨어 있는 곳입니다.
설계는 복잡성에 비례할 수 있습니다. 설정 변경은 두 문장의 설계로 충분할 수 있습니다. 결제 시스템은 여러 페이지의 사양(spec)을 가집니다. 하지만 멈추고, 생각하고, 승인을 받는 과정은 항상 발생합니다.

### 설계 체크리스트

이 스킬은 실제 제품 팀이 작동하는 방식을 반영하는 9단계 체크리스트를 제공합니다. 회사에서는 이것이 킥오프 회의부터 설계 검토, 스프린트 계획까지의 흐름이 될 것입니다:

```
## 체크리스트

1. **프로젝트 컨텍스트 탐색** - 파일, 문서, 최근 커밋(commit) 확인
2. **시각적 동반자 제공** (시각적 질문이 필요한 주제인 경우)
3. **명확화 질문** - 한 번에 하나씩, 목적/제약조건/성공 기준 이해
4. **2~3가지 접근 방식 제안** - 장단점(trade-offs)과 추천 사항 포함
5. **설계 제시** - 복잡성에 맞춰 섹션을 나누어 제시, 각 섹션 후 사용자 승인 받기
6. **설계 문서 작성** - `docs/senior-staff-engineer/specs/`에 저장하고 커밋
7. **사양 자체 검토** - 플레이스홀더, 모순, 모호성 확인
8. **사용자 작성된 사양 검토** - 진행하기 전에 사용자에게 검토 요청
9. **구현으로 전환** - `writing-plans` 스킬 호출
```

1단계는 스태프 엔지니어가 어떤 회의 전에 하는 일입니다. 이들은 설계 논의에 사전 준비 없이 들어가지 않습니다. 기존 코드를 읽고, 최근 커밋을 확인하며, 문서를 살펴봅니다. 질문이 정보에 기반하고 중복되지 않도록 컨텍스트를 구축합니다.

3단계와 4단계는 실제 제품 설계 회의를 반영합니다. 아키텍트는 한 번에 하나의 질문을 하고(이해관계자를 압도하는 15개의 질문을 쏟아내지 않고), 장단점이 있는 여러 접근 방식을 제안하며, 권장 사항을 제시합니다. 이것이 시니어 엔지니어가 회의를 운영하는 방식입니다.

이들은 "무엇을 원하십니까?"라고 묻지 않습니다. 대신 "여기에 세 가지 옵션이 있습니다. X와 Y 때문에 옵션 B를 추천하는데, 어떻게 생각하십니까?"라고 말합니다.

이 스킬은 질문 설계에 대해 구체적입니다:

*   아이디어를 다듬기 위해 한 번에 하나의 질문을 하세요.
*   가능하면 객관식 질문을 선호하세요.
*   메시지당 하나의 질문만 하세요.
*   목적, 제약 조건, 성공 기준 이해에 집중하세요.

메시지당 하나의 질문만 합니다. 다섯 개가 아닙니다. "몇 가지 궁금한 점이 있습니다"도 아닙니다. 단 하나입니다. 이는 제품 관리자가 이해관계자 인터뷰를 진행하는 방식을 반영합니다. 각 질문은 다음 주제로 넘어가기 전에 완전한 주의, 완전한 컨텍스트, 그리고 완전한 답변을 받습니다.

### 범위 감지

상세 질문으로 들어가기 전에, 이 스킬은 스태프 레벨의 사고방식과 주니어 레벨의 사고방식을 구분하는 범위 평가(scope assessment)를 요구합니다:

```
상세 질문을 하기 전에, 범위를 평가하십시오: 요청이 여러 독립적인 서브 시스템을 설명하는 경우(예: "채팅, 파일 저장, 결제, 분석 기능을 포함하는 플랫폼을 구축하세요"), 즉시 이 점을 지적하십시오.

먼저 분해(decompose)해야 하는 프로젝트의 세부 사항을 다듬는 데 질문을 낭비하지 마십시오.
```

주니어 엔지니어는 "채팅, 파일 저장, 결제, 분석 기능을 갖춘 플랫폼을 구축하라"는 말을 듣고 채팅 메시지 형식에 대해 묻기 시작합니다. 스태프 엔지니어는 같은 요청을 듣고 "그것은 네 가지 별개의 시스템입니다. 이들 중 어떤 것을 먼저 구축할지, 그리고 어떻게 관련되는지 알아낸 다음에 설계합시다"라고 말합니다.

이것은 분해(decomposition)입니다. 프로젝트가 실제로는 여러 프로젝트가 하나인 것처럼 위장하고 있을 때 이를 인식하는 능력입니다.

### 격리를 위한 설계

이 스킬은 시니어 엔지니어가 시스템 설계에 대해 생각하는 방식을 반영하는 아키텍처 지침을 포함합니다:

*   시스템을 각각 하나의 명확한 목적을 가지고, 잘 정의된 인터페이스를 통해 통신하며, 독립적으로 이해하고 테스트할 수 있는 더 작은 단위로 분할하세요.
*   각 단위에 대해 다음 질문에 답할 수 있어야 합니다: 무엇을 하는가, 어떻게 사용하는가, 무엇에 의존하는가?
*   누군가 내부를 읽지 않고도 단위가 무엇을 하는지 이해할 수 있습니까? 내부를 변경해도 소비자에게 영향을 주지 않습니까? 그렇지 않다면 경계를 재작업해야 합니다.

이것은 스태프 엔지니어가 모든 아키텍처 검토 중에 묻는 질문입니다.
"소스 코드를 읽지 않고 이 컴포넌트를 이해할 수 있는가?"는 좋은 추상화(abstraction)를 위한 테스트입니다.
"내부 구현을 변경해도 소비자를 망가뜨리지 않는가?"는 좋은 캡슐화(encapsulation)를 위한 테스트입니다.
두 질문 중 하나라도 '아니오'라면, 구현을 시작하기 전에 설계 작업이 필요합니다.

또한, 경험 많은 엔지니어와 이상주의적인 엔지니어를 구분하는 기존 코드베이스 작업에 대한 실용적인 언급도 있습니다:

```
기존 코드에 작업에 영향을 미치는 문제(예: 너무 커진 파일, 불분명한 경계)가 있는 경우, 좋은 개발자가 작업 중인 코드를 개선하는 방식처럼 설계의 일부로 목표 개선 사항을 포함하십시오.

관련 없는 리팩토링(refactoring)을 제안하지 마십시오. 현재 목표에 도움이 되는 것에 집중하십시오.
```

주니어 엔지니어는 지저분한 코드를 보고 모든 것을 다시 작성하고 싶어 합니다. 스태프 엔지니어는 자신이 다루는 부분을 개선하고 나머지는 그대로 둡니다. 이는 십자군처럼 무차별적인 리팩토링이 아니라, 목표 지향적인 개선입니다.

### 설계 후

설계가 구두로 승인되면, 문서화되고 검토되며 커밋(commit)됩니다. 이는 성숙한 엔지니어링 조직이 설계 문서를 다루는 방식을 반영합니다. 설계는 회의실에서 오간 대화로 모두의 불완전한 기억 속에만 존재하는 것이 아닙니다. 그것은 버전 관리 시스템에 존재하는 문서입니다:

*   검증된 설계(spec)를 `docs/senior-staff-engineer/specs/YYYY-MM-DD-<topic>-design.md`에 작성합니다.
*   설계 문서를 Git에 커밋합니다.

그 후 사양(spec)은 가장 일반적인 문서화 실패를 잡아내는 자체 검토를 거칩니다:

```
**사양 자체 검토(Spec Self-Review):**

1. **플레이스홀더 검사:** "TBD", "TODO", 불완전한 섹션이 있습니까?

2. **내부 일관성:** 어떤 섹션이 서로 모순됩니까?

3. **범위 확인:** 단일 구현 계획에 충분히 집중되어 있습니까?

4. **모호성 확인:** 어떤 요구사항이 두 가지 방식으로 해석될 수 있습니까? 그렇다면 하나를 선택하고 명시하십시오.
```

그런 다음 명시적인 인간의 승인 없이는 시스템이 진행되지 않도록 하는 사용자 검토 게이트가 있습니다:

```
"사양이 작성되어 `<path>`에 커밋되었습니다. 구현 계획을 작성하기 전에 검토하고 변경 사항이 있는지 알려주십시오."

사용자의 응답을 기다리십시오. 사용자가 승인해야만 진행합니다.
```

이것은 설계 검토 승인(sign-off)입니다. 마이크로소프트(Microsoft)와 구글(Google) 같은 회사에서는 구현이 시작되기 전에 설계 문서에 검토자들의 명시적인 승인이 필요합니다. 암묵적인 승인("아무도 반대하지 않았으니 괜찮겠지")이 아닙니다. 명시적인 승인입니다. "이것을 검토했고 준비가 되었습니다." 우리 시스템도 동일한 표준을 따릅니다.

브레인스토밍 스킬의 최종 상태는 항상 동일합니다: `writing-plans`를 호출하여 구현 계획을 생성하는 것입니다. 코딩으로 바로 뛰어들거나, 바로 구축을 시작하는 것이 아닙니다. 계획을 생성합니다. 설계는 계획으로 이어지고, 계획은 구현으로 이어집니다. 각 단계는 다음 단계를 공급합니다. 지름길은 없습니다.

```
**구현:**

- `writing-plans` 스킬을 호출하여 상세한 구현 계획을 생성합니다.

- 다른 스킬은 호출하지 마세요. `writing-plans`가 다음 단계입니다.
```

### 시각적 옵션

어떤 설계 대화는 말만으로는 부족합니다. UI 레이아웃, 아키텍처 다이어그램 또는 시각적 비교가 논의에 포함될 때, 에이전트는 목업(mockup)과 다이어그램을 보여주기 위한 브라우저 기반의 동반자(companion)를 제공할 수 있습니다:

```
"우리가 작업하는 일부 내용은 웹 브라우저에서 보여드리면 더 쉽게 설명될 수 있습니다. 진행하면서 목업, 다이어그램, 비교 자료 및 기타 시각 자료를 함께 만들 수 있습니다. 시도해 보시겠습니까?"
```

이 스킬은 이 제안이 어떻게 이루어지는지에 대해 신중합니다:

```
**이 제안은 반드시 별도의 메시지여야 합니다.** 명확화 질문, 컨텍스트 요약 또는 다른 어떤 내용과도 결합하지 마십시오. 사용자의 응답을 기다린 후 계속 진행하십시오. 사용자가 거부하면 텍스트 전용 브레인스토밍으로 진행하십시오.
```

그리고 사용자가 수락한 후에도 에이전트는 질문별로 시각 자료를 사용할지 텍스트를 사용할지 결정합니다:

```
UI 주제에 대한 질문이 자동으로 시각적 질문이 되는 것은 아닙니다. "이 컨텍스트에서 '개성'은 무엇을 의미합니까?"는 개념적 질문입니다. 터미널을 사용하십시오.

"어떤 마법사 레이아웃이 더 효과적입니까?"는 시각적 질문입니다. 브라우저를 사용하십시오.
```

이것은 시니어 엔지니어의 커뮤니케이션 매체에 대한 직관입니다. 어떤 것은 말로 설명하는 것이 더 좋고, 어떤 것은 화이트보드가 필요합니다. 이 스킬은 에이전트에게 동일한 직관을 가르칩니다.

## 논리적 허점 찾기 (사양 검토)

어떤 엔지니어링 조직에서든 아키텍트는 시스템을 설계하고 그냥 떠나지 않습니다. 항상 설계 검토 회의에 앉아서 "데이터베이스가 다운되고 사용자가 세 번 재시도하면 어떻게 됩니까?"라고 묻는 시니어 엔지니어가 있습니다. 한 줄의 코드가 작성되기 전에 허점을 찾는 것이 이 사람의 임무입니다.

우리는 `skills/brainstorming/spec-document-reviewer-prompt.md`를 사용하여 이 역할을 재현합니다. 이것은 메인 에이전트가 직접 사용하는 스킬이 아니라, 별도의 서브 에이전트에게 파견되는 프롬프트 템플릿(prompt template)입니다. 아키텍트의 청사진을 검토하기 위해 프리랜서 감사관을 고용하는 것과 같다고 생각할 수 있습니다.

핵심 통찰은 관심사 분리(separation of concerns)입니다.

설계를 작성한 에이전트는 편향되어 있으며, 자신의 작업에 자부심을 가집니다. 그래서 우리는 완전히 새로운 클로드(Claude)를 하나의 임무, 즉 사양(spec)을 철저히 분석하는 임무를 위해 가동합니다.

```
Task tool (general-purpose):
  description: "사양 문서 검토"
  prompt: |
    당신은 사양 문서 검토자입니다. 이 사양이 완전하고 계획 준비가 되었는지 확인하십시오.
    **검토할 사양:** [SPEC_FILE_PATH]
```

검토자는 집중된 범주 세트를 기준으로 확인합니다:

| 범주             | 무엇을 찾아야 하는가                                     |
|------------------|----------------------------------------------------------|
| **완전성(Completeness)** | TODO, 플레이스홀더, "TBD", 불완전한 섹션                     |
| **일관성(Consistency)**  | 내부 모순, 충돌하는 요구사항                                 |
| **명확성(Clarity)**      | 잘못된 것을 만들게 할 정도로 모호한 요구사항                 |
| **범위(Scope)**          | 단일 계획에 충분히 집중되어 있는가 — 여러 독립적인 서브시스템을 다루지 않는가 |
| **YAGNI(You Ain’t Gonna Need It)** | 요청되지 않은 기능, 과도한 엔지니어링                    |

각 범주가 실제 설계 검토 문제와 어떻게 연결되는지 주목하십시오. 완전성은 "설계를 마쳤는가?"이고, 일관성은 "3페이지가 7페이지와 모순되는가?"이며, 명확성은 "두 명의 개발자가 이것을 읽고 두 가지 다른 것을 만들 수 있는가?"입니다. 그리고 YAGNI (You Ain't Gonna Need It, 필요 없을 것이다)는 시니어 엔지니어가 "클라이언트가 하나뿐인데 왜 플러그인 시스템을 설계하고 있는가?"라고 말하는 것입니다.

이 검토자가 까다롭기보다는 규율 있게 만드는 것은 '교정(calibration)' 섹션입니다:

```
구현 계획 단계에서 실제 문제를 일으킬 수 있는 문제점만 지적하십시오.

누락된 섹션, 모순, 또는 두 가지 방식으로 해석될 수 있을 정도로 모호한 요구사항 — 이들은 문제입니다.

사소한 문구 개선, 스타일 선호도, 그리고 "다른 섹션보다 덜 상세한 섹션"은 문제가 아닙니다.

심각한 누락으로 인해 결함 있는 계획으로 이어질 가능성이 없는 한 승인하십시오.
```

이것은 중요합니다. 실제 조직에서 최악의 검토자는 쉼표 위치 때문에 설계를 막는 사람들입니다. 우리 검토자는 실제로 하류에서 문제를 일으킬 수 있는 것들만 지적하도록 교정되어 있습니다. 예를 들어, 누락된 오류 처리 전략, 두 요구사항 간의 모순, 단일 스프린트(sprint) 내에서 계획할 수 없을 정도로 광범위한 범위 등입니다.

이것은 실제 설계 검토 흐름을 반영합니다: 감사관이 검토하고 보고서를 제출하면, 아키텍트는 계획으로 넘어가기 전에 어떤 조치를 취할지 결정합니다.

## 시각적 동반자 (화이트보드 세션)

모든 시니어 엔지니어는 어떤 문제는 말만으로는 해결될 수 없다는 것을 알고 있습니다. 사이드바 레이아웃을 사용할지 상단 내비게이션(top-nav)을 사용할지 논의할 때, 화이트보드에 그림을 그리는 것보다 더 좋은 방법은 없습니다. 우리 시스템에서는 `skills/brainstorming/visual-companion.md`와 보조 `scripts/` 디렉터리를 통해 이를 에이전트에 적용합니다.

시각적 동반자는 항상 켜져 있는 것이 아니라, 브레인스토밍 중에 질문별로 제공됩니다. 결정 규칙은 간단합니다:

```
세션당이 아니라 질문당 결정합니다. 판단 기준: **사용자가 읽는 것보다 보는 것이 더 잘 이해할 수 있는가?**
```

이 파일은 브라우저에 속하는 것과 터미널에 속하는 것 사이의 명확한 경계를 그립니다:

**브라우저를 사용하세요** (내용 자체가 시각적인 경우):
*   UI 목업(mockups) (와이어프레임, 레이아웃, 내비게이션 구조)
*   아키텍처 다이어그램 (시스템 컴포넌트, 데이터 흐름)
*   시각적 비교 (두 가지 레이아웃, 두 가지 색 구성표)
*   공간 관계 (상태 머신, 플로우차트, 엔티티 관계)

**터미널을 사용하세요** (내용이 텍스트 또는 표 형식인 경우):
*   요구사항 및 범위 질문
*   말로 설명된 개념적 A/B/C 선택
*   장단점 목록 (장점/단점, 비교표)
*   기술적 결정 (API 설계, 데이터 모델링)

여기에는 간단하지만 중요한 구분이 있습니다. UI 주제에 대한 질문이 자동으로 시각적 질문이 되는 것은 아닙니다. "어떤 종류의 마법사를 원하십니까?"는 개념적인 질문이므로 터미널을 사용합니다. "이 마법사 레이아웃 중 어떤 것이 적절하다고 생각하십니까?"는 시각적 질문이므로 브라우저를 사용합니다.

실제 팀에서 시니어 엔지니어는 언제 계속 말하고 언제 일어나서 화이트보드로 가야 할지 압니다. 이 스킬은 에이전트에게 동일한 직관을 가르칩니다.

### 작동 방식

이 기술은 HTML 파일을 감시하고 가장 최신 파일을 브라우저에 제공하는 경량 로컬 서버입니다.

```bash
# Start server with persistence
scripts/start-server.sh --project-dir /path/to/project

# Returns: {"type":"server-started","port":52341,
#   "url":"http://localhost:52341",
#   "screen_dir":"...content", "state_dir":"...state"}
```

에이전트는 HTML 콘텐츠 조각을 `screen_dir`에 작성합니다. 사용자는 이를 브라우저에서 보고 옵션을 클릭하여 선택할 수 있으며, 이 선택 사항은 `state_dir/events`에 JSON 라인으로 기록되어 에이전트가 다음 턴에 읽게 됩니다.

핵심 설계 결정은 콘텐츠 조각(content fragments) 대 전체 문서(full documents)입니다. 에이전트는 전체 HTML 페이지를 작성할 필요 없이 콘텐츠만 작성하고, 서버는 이를 테마, CSS 및 인터랙티브 인프라를 갖춘 프레임 템플릿(frame template)으로 자동으로 감쌉니다:

```html
<h2>Which layout works better?</h2>
<p class="subtitle">Consider readability and visual hierarchy</p>

<div class="options">
  <div class="option" data-choice="a" onclick="toggleSelect(this)">
    <div class="letter">A</div>
    <div class="content">
      <h3>Single Column</h3>
      <p>Clean, focused reading experience</p>
    </div>
  </div>

  <div class="option" data-choice="b" onclick="toggleSelect(this)">
    <div class="letter">B</div>
    <div class="content">
      <h3>Two Column</h3>
      <p>Sidebar navigation with main content</p>
    </div>
  </div>
</div>
```

`<html>`, CSS, `<script>` 태그는 없습니다. 서버가 이 모든 것을 제공합니다. 이는 에이전트가 상용구(boilerplate)가 아닌 무엇을 보여줄지에 집중하도록 하기 때문에 중요합니다.

### 피드백 루프

상호작용은 실제 화이트보드 세션을 반영하는 루프(loop)를 따릅니다: 그리고, 논의하고, 수정합니다:

```
HTML을 `layout.html`과 같은 의미 있는 이름으로 `screen_dir`의 새 파일에 작성합니다.
무엇이 화면에 표시되는지 사용자에게 알리고 피드백을 요청합니다.
다음 턴에 `state_dir/events`를 읽어 무엇을 클릭했는지 확인합니다.
피드백이 현재 화면을 변경하면 반복하거나 진행하고, `layout-v2.html`을 작성합니다. 이 단계가 검증될 때까지 다음 질문으로 넘어가지 마십시오.
```

사용자가 브라우저에서 옵션을 클릭하면, 전체 탐색 경로가 캡처됩니다:

```json
{"type":"click","choice":"a","text":"Option A","timestamp":1706000101}
{"type":"click","choice":"c","text":"Option C","timestamp":1706000108}
{"type":"click","choice":"b","text":"Option B","timestamp":1706000115}
```

클릭 패턴은 망설임을 보여줍니다. A를 시도했다가 C로 넘어갔고, 결국 B로 결정했습니다. 좋은 시니어 엔지니어라면 이를 알아차리고 "처음에는 A에 끌렸던 것 같은데, 왜 바꾸셨습니까?"라고 물을 것입니다. 에이전트도 같은 일을 할 수 있습니다.

실제 엔지니어링 규율을 보여주는 또 다른 세부 사항이 있습니다. 비시각적 질문을 위해 대화가 터미널로 돌아갈 때, 에이전트는 대기 화면을 표시합니다:

```html
<div style="display:flex;align-items:center;justify-content:center;min-height:60vh">
  <p class="subtitle">Continuing in terminal...</p>
</div>
```

이것은 대화가 다음으로 넘어갔을 때 사용자가 이미 해결된 선택을 계속 쳐다보는 것을 방지합니다. 작은 부분이지만, 시니어 엔지니어의 도구와 주니어의 임시방편을 구분하는 세련됨입니다.

## 계획 작성 (백로그 그루밍)

애자일(Agile) 팀에는 "백로그 그루밍(backlog grooming)"이라는 용어가 있습니다. 이는 테크 리드(tech lead)가 승인된 아키텍처를 가져와 어떤 개발자라도 질문 없이 바로 작업을 시작할 수 있을 만큼 명확한 티켓(ticket)으로 분해하는 과정입니다. `skills/writing-plans/SKILL.md`가 바로 이 역할을 합니다.

시작 문구는 철학을 제시합니다:

```
엔지니어가 우리 코드베이스에 대한 컨텍스트가 전혀 없고 의심스러운 취향을 가졌다고 가정하고 포괄적인 구현 계획을 작성하십시오. 각 작업에 대해 만져야 할 파일, 코드, 테스트, 확인해야 할 문서, 테스트 방법 등 그들이 알아야 할 모든 것을 문서화하십시오.
```

"컨텍스트가 전혀 없고 의심스러운 취향" — 이것이 시니어 테크 리드가 티켓 작성에 대해 생각하는 방식입니다. 개발자가 지난 3주 동안 스탠드업 회의에 참석했다고 가정하지 않습니다. 이들이 오류 처리(error handling)에 대해 스스로 좋은 결정을 내릴 것이라고 가정하지 않습니다.

오늘 막 팀에 합류한 숙련된 개발자라도 완벽하게 실행할 수 있도록 티켓을 작성합니다.
어떤 작업을 정의하기 전에, 이 스킬은 어떤 파일을 생성하거나 수정할지, 그리고 각 파일이 무엇을 책임질지 결정하는 파일 구조 매핑(file structure mapping)을 요구합니다:

```
## 파일 구조

- 명확한 경계와 잘 정의된 인터페이스를 가진 설계 단위를 만드십시오. 각 파일은 하나의 명확한 책임을 가져야 합니다.

- 함께 변경되는 파일은 함께 두십시오. 기술적 계층이 아닌 책임에 따라 분할하십시오.

- 기존 코드베이스에서는 확립된 패턴을 따르십시오.
```

이것은 테크 리드가 "티켓을 작성하기 전에, 무엇이 어디에 속할지 합의하자"고 말하는 것입니다. 어떤 개발자도 유효성 검사 로직이 컨트롤러에 들어갈지 서비스 계층에 들어갈지 추측해서는 안 됩니다.

### 2–5분 규칙

이곳에서 세분화(granularity) 요구사항이 규율을 갖게 됩니다:

```
## 한입 크기 작업 세분화(Bite-Sized Task Granularity)

**각 단계는 하나의 동작입니다 (2-5분):**
- "실패하는 테스트를 작성하라" - 단계
- "실패하는지 확인하기 위해 실행하라" - 단계
- "테스트를 통과시키기 위한 최소한의 코드를 구현하라" - 단계
- "테스트를 실행하고 통과하는지 확인하라" - 단계
- "커밋하라" - 단계
작업이 서브 에이전트에게 5분 이상 걸린다면, 이는 작업이 너무 광범위하여 에이전트가 가정을 하거나 집중력을 잃기 시작할 수 있다는 의미입니다.
```

이는 컨텍스트 전환 없이 한 번에 완료할 수 있을 만큼 티켓을 작게 유지하는 실제 관행을 반영합니다.

모든 작업은 정확한 파일 경로, 완전한 코드 블록, 예상 출력을 포함하는 정확한 명령, 그리고 커밋 단계를 포함해야 합니다. 애매모호한 설명은 허용되지 않습니다:

```
### 작업 N: [컴포넌트 이름]

**파일:**

- 생성: `exact/path/to/file.py`

- 수정: `exact/path/to/existing.py:123-145`

- 테스트: `tests/exact/path/to/test.py`

- [ ] **단계 1: 실패하는 테스트를 작성하십시오.**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **단계 2: 테스트가 실패하는지 확인하기 위해 실행하십시오.**

실행: `pytest tests/path/test.py::test_name -v`
예상: FAIL with "function not defined"
```

이 스킬은 엄격한 TBD 금지 정책을 시행하며, 무엇이 플레이스홀더로 간주되는지 구체적으로 명시합니다:

```
이것들은 **계획 실패**입니다 — 절대로 작성하지 마세요:

- "TBD", "TODO", "나중에 구현", "세부 사항 채워넣기"

- "적절한 오류 처리 추가" / "유효성 검사 추가"

- "위에 대한 테스트 작성" (실제 테스트 코드 없이)

- "작업 N과 유사" (코드를 반복 — 엔지니어가 작업을 순서 없이 읽을 수 있음)

- 어떤 작업에서도 정의되지 않은 타입, 함수, 또는 메서드 참조
```

마지막 항목은 미묘합니다. 만약 Task 7이 `clearFullLayers()`를 호출하지만 Task 3이 이를 `clearLayers()`로 정의했다면, 이는 계획의 버그이며 자체 검토 단계에서 이를 잡아냅니다:

```
**3. 타입 일관성:** 나중 작업에서 사용한 타입, 메서드 시그니처, 그리고 속성 이름이 이전 작업에서 정의한 것과 일치합니까?
```

계획은 `docs/senior-staff-engineer/plans/`에 저장되며, 완료되면 에이전트는 서브 에이전트 주도(권장) 또는 인라인 실행의 두 가지 실행 경로를 제공합니다.

이것은 테크 리드가 "각 티켓에 전문가를 둘 것인가, 아니면 한 개발자가 마라톤을 할 것인가?"라고 묻는 것과 같다고 생각하십시오.

## 계획 검토 (그루밍 게이트키퍼)

설계에 대한 사양 검토자가 있었던 것처럼, 우리는 구현 계획에 대한 계획 검토자를 가집니다. `skills/writing-plans/plan-document-reviewer-prompt.md`는 새로운 서브 에이전트를 파견하여 한 가지 질문에 답하게 합니다: 엔지니어가 막힘없이 이 계획을 실제로 따를 수 있을까?

```
당신은 계획 문서 검토자입니다. 이 계획이 완전하고 구현 준비가 되었는지 확인하십시오.

**검토할 계획:** [PLAN_FILE_PATH]

**참고할 사양:** [SPEC_FILE_PATH]
```

계획과 사양(spec) 모두를 입력으로 받는다는 점에 주목하십시오. 검토자는 네 가지를 확인합니다:

| 범주             | 무엇을 찾아야 하는가                         |
|------------------|----------------------------------------------|
| **완전성(Completeness)** | TODO, 플레이스홀더, 불완전한 작업            |
| **사양 정렬(Spec Alignment)** | 계획이 사양 요구사항을 다루는지, 범위 확장(scope creep)이 없는지 |
| **작업 분해(Task Decomposition)** | 작업 경계가 명확하고, 단계가 실행 가능한지 |\n| **구축 가능성(Buildability)** | 엔지니어가 막힘없이 이를 따를 수 있는지   |

"구축 가능성(Buildability)"이 가장 중요합니다.

계획은 완전하고, 사양과 일치하며, 잘 분해될 수 있지만, 만약 4단계가 3단계에서 생성되지 않은 것을 가정한다면, 개발자는 막히게 됩니다.
이것은 실제 스프린트 계획 회의에서 프로젝트 관리자가 요구사항 문서와 티켓 보드를 나란히 열고 "7번 요구사항을 다루는 티켓이 무엇인지 보여주세요"라고 말하는 순간과 같습니다.

사양 검토자와 동일한 교정 철학입니다. 즉, 실제 문제를 일으킬 만한 것들만 지적합니다:

```
심각한 누락 — 사양에서 빠진 요구사항, 모순되는 단계, 플레이스홀더 내용, 또는 너무 모호하여 실행할 수 없는 작업이 없는 한 승인하십시오.
```

검토자는 깨끗한 '승인됨' 또는 '문제 발견됨' 상태를 반환하며, 문제는 특정 작업 및 단계에 연결됩니다. 계획에 누락된 부분이 있다면, 테크 리드는 어떤 개발자도 작업을 시작하기 전에 돌아가서 이를 채웁니다.

## 서브 에이전트 주도 개발 (위임)

엔지니어링 조직에서 관리자는 모든 코드를 직접 작성하지 않습니다. 이들은 전문가를 고용하고, 명확한 지시를 내리고, 작업을 확인하며, 인수인계를 관리합니다. `skills/subagent-driven-development/SKILL.md`는 정확히 이러한 워크플로를 시스템으로 인코딩한 것입니다.

시작 원칙:

```
**서브 에이전트(subagents)를 사용하는 이유:** 당신은 격리된 컨텍스트를 가진 전문 에이전트에게 작업을 위임합니다. 이들의 지시와 컨텍스트를 정밀하게 조작함으로써, 이들이 작업에 집중하고 성공하도록 보장합니다. 이들은 당신의 세션 컨텍스트나 이력을 상속받아서는 안 됩니다 — 필요한 것을 정확하게 구성해야 합니다.
```

이것은 컨텍스트 격리(context isolation)입니다. 인간 개발자가 티켓을 집어들 때, 회사 전체의 이력이 필요한 것이 아니라 티켓, 관련 코드, 그리고 코딩 표준이 필요합니다. 여기도 마찬가지입니다. 새로운 에이전트, 깨끗한 마음, 집중된 작업.

이는 한 시간 동안 작업한 에이전트가 다른 기능의 요구사항을 혼동하기 시작하는 "혼란스러운 시니어(confused senior)" 문제를 방지합니다.

이 프로세스는 작업별 루프를 따릅니다: 구현자 파견 → 사양 검토 → 코드 품질 검토 → 완료 표시. 이 스킬은 이를 방향성 그래프(digraph)로 제시하지만, 인간적인 관점에서는 잘 운영되는 팀의 일상적인 리듬과 같습니다: 작업을 할당하고, 사양에 맞게 구축되었는지 확인하고, 잘 구축되었는지 확인한 다음 다음으로 넘어갑니다.

### 모델 선택 (적절한 수준 고용)

여기서 관리자 은유가 실용적이 됩니다. 변수 이름을 바꾸기 위해 수석 엔지니어(principal engineer)를 보내지는 않습니다:

```
## 모델 선택

**기계적 구현 작업** (격리된 함수, 명확한 사양, 1-2개 파일): 빠르고 저렴한 모델을 사용합니다.

**통합 및 판단 작업** (다중 파일 조정, 패턴 매칭, 디버깅): 표준 모델을 사용합니다.

**아키텍처, 설계 및 검토 작업**: 사용 가능한 가장 유능한 모델을 사용합니다.
```

이것은 티켓을 주니어, 미드 레벨 또는 시니어에게 할당하는 차이입니다. 관리자는 작업 복잡도를 능력 및 비용과 일치시킵니다. 잘 명시된 계획은 대부분의 작업이 "기계적"이라는 것을 의미하며, 이는 저렴하고 빠르다는 것을 의미합니다.

### 실패 처리

모든 관리 시스템에서 가장 드러나는 부분은 실패를 처리하는 방식입니다. 이 스킬은 구현자가 반환할 수 있는 네 가지 상태를 정의합니다:

```
**DONE (완료):** 사양 준수 검토로 진행합니다.

**DONE_WITH_CONCERNS (우려 사항과 함께 완료):** 구현자가 작업을 완료했지만 의구심을 표명했습니다. 진행하기 전에 우려 사항을 읽으십시오.

**NEEDS_CONTEXT (컨텍스트 필요):** 구현자가 제공되지 않은 정보가 필요합니다. 누락된 컨텍스트를 제공하고 재파견합니다.

**BLOCKED (차단됨):** 구현자가 작업을 완료할 수 없습니다.
```

그리고 중요한 규칙:

```
**절대** 에스컬레이션(escalation)을 무시하거나 동일한 모델에게 변경 없이 재시도하도록 강요하지 마십시오. 구현자가 막혔다고 말하면, 무언가가 변경되어야 합니다.
```

최악의 관리자는 "막혔다"는 말에 "더 열심히 해봐"라고 응답하는 사람들입니다. 우리 시스템은 이를 명시적으로 금지합니다. 에이전트가 막혔다면, 관리자는 세 가지 옵션을 가집니다: 더 많은 컨텍스트를 제공하거나, 모델을 업그레이드하거나, 작업을 더 세분화하는 것입니다. 만약 계획 자체가 잘못되었다면, 인간에게 에스컬레이션합니다. 이것이 성숙한 관리입니다.

## 구현자 프롬프트 (개발자 직무 기술서)

회사가 개발자를 고용할 때, 온보딩 문서, 코딩 표준, 그리고 첫 번째 작업이 있습니다. `skills/subagent-driven-development/implementer-prompt.md`는 바로 이것입니다. 즉, 모든 서브 에이전트가 받는 직무 기술서입니다.

이 프롬프트(prompt)는 대부분의 AI 시스템이 완전히 건너뛰는 것으로 시작합니다:

```
## 시작하기 전에

다음 사항에 대해 질문이 있다면:
- 요구사항 또는 인수 기준
- 접근 방식 또는 구현 전략
- 의존성 또는 가정
- 작업 설명에서 불분명한 모든 것

**지금 물어보십시오.** 작업을 시작하기 전에 우려 사항을 제기하십시오.
```

그리고 구현 중에 강화됩니다:

```
**작업 중:** 예상치 못한 일이 발생하거나 불분명한 점이 있다면, **질문하십시오.** 언제든지 멈추고 명확히 하는 것은 괜찮습니다. 추측하거나 가정을 하지 마십시오.
```

"추측하지 마라"는 규칙은 아마도 전체 코드베이스에서 가장 중요한 구절일 것입니다.
조직에서 가장 값비싼 실수는 무엇이 필요한지 안다고 가정한 개발자로부터 나옵니다.
멈춰서 "널(null) 처리 방식이 불분명합니다"라고 말하는 서브 에이전트는 5분의 비용이 듭니다. 잘못 추측하는 에이전트는 한 시간의 재작업 비용이 듭니다.

또한 실패에 대한 명시적인 허용도 있습니다:

```
## 역량 밖의 일을 만났을 때

언제든지 "이것은 나에게 너무 어렵다"고 말하고 멈추는 것은 괜찮습니다. 나쁜 작업은 작업 없음보다 나쁩니다. 에스컬레이션(escalate)한다고 해서 불이익을 받지 않을 것입니다.
```

이것은 좋은 엔지니어링 문화가 구축하는 심리적 안전(psychological safety)을 반영합니다.
둘째 날에 "이것을 이해하지 못하겠습니다"라고 말하는 주니어는 일주일 동안 조용히 잘못된 것을 만드는 주니어보다 훨씬 더 가치가 있습니다.

보고하기 전에, 구현자는 완전성, 품질, 규율, 테스트의 네 가지 차원에서 자체 검토를 수행합니다. 그런 다음 구조화된 상태로 보고합니다. 무엇이 완료되었는지, 무엇이 테스트되었는지, 또는 어떤 우려 사항이 남아 있는지에 대한 모호함이 없습니다.

## 사양 및 코드 품질 검토자 (준법 감시관)

구현자가 작업을 마치면, 작업은 두 개의 별도 검토 서브 에이전트를 거칩니다. 순서가 중요합니다: 사양 준수(spec compliance) 먼저, 코드 품질(code quality) 나중입니다. 잘못된 것을 만드는 코드를 다듬을 필요는 없습니다.

### 사양 준수 검토자

`spec-reviewer-prompt.md`는 이 시스템이 어떻게 생각하는지에 대한 모든 것을 알려주는 구절로 시작합니다:

```
## 중요: 보고서를 신뢰하지 마십시오.

구현자가 의심스러울 정도로 빠르게 작업을 마쳤습니다. 그들의 보고서는 불완전하거나, 부정확하거나, 낙관적일 수 있습니다. 당신은 모든 것을 독립적으로 확인해야 합니다.

**다음은 하지 마십시오:**
- 그들이 구현했다고 주장하는 것을 그대로 믿지 마십시오.
- 완전성에 대한 그들의 주장을 신뢰하지 마십시오.
- 요구사항에 대한 그들의 해석을 받아들이지 마십시오.

**다음은 하십시오:**
- 그들이 작성한 실제 코드를 읽으십시오.
- 실제 구현을 요구사항과 한 줄씩 비교하십시오.
```

이것은 자기 보고된 감사 결과를 받아들이지 않는 준법 감시관(compliance officer)과 같습니다. 이들은 세 가지를 확인합니다: 누락된 요구사항(무엇을 건너뛰었는가?), 추가 작업(과도하게 설계했는가?), 오해(잘못된 문제를 해결했는가?). 모든 발견 사항은 파일:줄 참조와 함께 제공되며, 모호한 피드백은 없습니다.

### 코드 품질 검토자

`code-quality-reviewer-prompt.md`는 사양 준수(spec compliance)를 통과한 후에만 실행됩니다. 표준 코드 검토 템플릿(template)을 사용하지만 구현별 검사를 추가합니다:

```
- 각 파일이 잘 정의된 인터페이스를 가지고 하나의 명확한 책임을 가집니까?

- 단위들이 독립적으로 이해하고 테스트할 수 있도록 분해되었습니까?

- 구현이 계획의 파일 구조를 따르고 있습니까?

- 이 구현으로 인해 이미 큰 새로운 파일이 생성되었습니까?
```

이러한 2단계 검토는 실제 조직이 기능 검토("올바른 일을 하는가?")와 기술 검토("올바른 방식으로 구축되었는가?")를 분리하는 방식을 반영합니다.

기능은 기능적으로는 올바를 수 있지만 아키텍처적으로는 끔찍할 수 있습니다. 두 가지 관점 모두 필요합니다.

두 검토자 중 한 명이라도 문제를 발견하면, 구현자는 이를 수정하고 검토자는 다시 확인합니다. 승인될 때까지 이 루프가 반복됩니다. "이 정도면 됐어"는 없습니다. 재검토를 건너뛰는 것도 없습니다. 이것이 규율 잡힌 팀이 배포하는 방식입니다.

## 테스트 주도 개발 (철칙)

구글(Google), 마이크로소프트(Microsoft), 스트라이프(Stripe), 쇼피파이(Shopify)와 같은 모든 진지한 엔지니어링 조직에는 전문가와 아마추어를 구분하는 한 가지 규칙이 있습니다.

이것은 어떤 프레임워크를 사용하거나 알고리즘이 얼마나 영리한지에 대한 것이 아닙니다. 압박감 속에서의 규율에 대한 것입니다.

구글에서는 이를 "테스팅 문화"라고 부릅니다. 스트라이프에서는 "프로덕션 준비성"이라고 부릅니다. 우리 시스템에서는 이를 '철칙(Iron Law)'이라고 부릅니다:

```
실패하는 테스트 없이는 프로덕션 코드도 없다.
```

`skills/test-driven-development/SKILL.md`는 전체 코드베이스에서 가장 타협하지 않는 파일입니다. 이것은 새벽 3시에 프로덕션 중단으로 고통받았고, 다시는 테스트되지 않은 코드를 배포하지 못하게 하는 퉁명스러운 시니어 엔지니어와 같습니다.

핵심 원칙은 한 줄입니다:

```
**핵심 원칙:** 테스트가 실패하는 것을 보지 못했다면, 그 테스트가 올바른 것을 테스트하는지 알 수 없습니다.
```

생각해보십시오. 코드를 작성한 후에 테스트를 작성했고, 그것이 즉시 통과한다면, 당신은 무엇을 증명한 것입니까? 아무것도 증명하지 못한 것입니다. 그 테스트는 완전히 엉뚱한 것을 테스트하고 있을지도 모릅니다.

테스트가 아무것도 잡아내지 못했으니 알 수 없습니다. 이것은 화재 경보기를 설치하고 배터리가 작동하는지 한 번도 확인하지 않는 것과 같습니다.

### 사이클

이 스킬은 고전적인 레드-그린-리팩터(Red-Green-Refactor) 사이클을 강제하지만, 대부분의 팀이 건너뛰는 검증 게이트(verification gate)를 포함합니다:

RED: 최소한의 실패하는 테스트 하나를 작성하십시오.

```javascript
test('retries failed operations 3 times', async () => {
  let attempts = 0;
  const operation = () => {
    attempts++;
    if (attempts < 3) throw new Error('fail');
    return 'success';
  };

const result = await retryOperation(operation);
  expect(result).toBe('success');
  expect(attempts).toBe(3);
});
```

그런 다음 테스트가 실패하는지 확인합니다. 단순히 실패하는 것이 아니라, 올바른 이유로 실패하는지 확인합니다. 오타 때문에 에러가 나는 테스트는 실패하는 테스트가 아닙니다. 기능이 아직 존재하지 않아서 실패하는 테스트, 그것이 우리가 원하는 것입니다.

GREEN: 통과하는 가장 간단한 코드를 작성하십시오.

```javascript
async function retryOperation<T>(fn: () => Promise<T>): Promise<T> {
  for (let i = 0; i < 3; i++) {
    try {
      return await fn();
    } catch (e) {
      if (i === 2) throw e;
    }
  }
  throw new Error('unreachable');
}
```

이 스킬은 "최소한"이 무엇을 의미하는지 명시합니다. 테스트가 "3번 재시도"만을 요구했을 때, 구성 가능한 백오프(backoff) 전략과 재시도 콜백(callback)으로 과도하게 설계하는 안티 패턴(anti-pattern)을 보여줌으로써 말입니다. 이것은 코드 레벨에서 YAGNI(You Ain't Gonna Need It)가 강제되는 것입니다.

### 삭제 규칙

여기서 이 스킬이 진정으로 '하드코어'해집니다. 대부분의 TDD 가이드는 "테스트를 먼저 작성하라"고 말합니다. 이 스킬은 이렇게 말합니다:

```
테스트 전에 코드를 작성했습니까? 삭제하십시오. 다시 시작하십시오.

**예외 없음:**
- "참조용"으로 보관하지 마십시오.
- 테스트를 작성하면서 "수정"하지 마십시오.
- 보지 마십시오.
- 삭제는 삭제를 의미합니다.
```

실제 회사에서 이것은 주니어 엔지니어들이 반발하는 순간입니다. "하지만 세 시간을 썼단 말이에요!" 이 스킬은 이를 직접적으로 다룹니다:

| 변명                 | 현실                                   |
|----------------------|----------------------------------------|
| "X 시간을 삭제하는 것은 낭비야" | 매몰 비용 오류(Sunk cost fallacy). 검증되지 않은 코드를 유지하는 것은 기술 부채(technical debt)입니다. |
| "참조용으로 보관하고 테스트를 먼저 작성하자" | 결국 수정하게 될 것입니다. 그것은 테스트 후입니다. 삭제는 삭제를 의미합니다. |
| "TDD는 나를 느리게 만들 거야" | TDD는 디버깅보다 빠릅니다. 실용적인 것은 테스트 우선입니다. |

아마존(Amazon)에는 이런 격언이 있습니다: "가장 비싼 버그는 배포한 버그이다." TDD를 건너뛰어 "절약한" 시간은 프로덕션 디버깅, 고객 불만, 새벽 3시 호출로 10배를 갚아야 합니다. 대규모 중단을 겪어본 모든 시니어 스태프 엔지니어는 이를 뼈저리게 알고 있습니다.

### 테스트 우선이 아닌 테스트 후가 같은 이유

이 스킬이 맞서 싸우는 가장 위험한 합리화는 "나중에 테스트를 작성할 거야 — 결과는 같겠지?"입니다. 틀렸습니다:

```
테스트 후에는 "이것이 무엇을 하는가?"에 답합니다.

테스트 우선은 "이것이 무엇을 해야 하는가?"에 답합니다.

테스트 후에는 구현에 의해 편향됩니다. 당신은 요구되는 것이 아니라 구축한 것을 테스트합니다. 기억된 엣지 케이스를 확인하고, 발견되지 않은 엣지 케이스는 확인하지 않습니다.
```

이것은 인지 편향(cognitive bias)에 대한 심오한 통찰입니다. 코드를 먼저 작성하면 테스트는 당신이 구축한 것에 의해 형성됩니다. 당신이 구현한 '해피 경로(happy path)'를 테스트하고, 잊어버린 엣지 케이스는 테스트하지 않습니다.

테스트 우선은 구현 전에 동작에 대해 생각하도록 강요합니다. 예를 들어 "이메일이 비어 있으면 어떻게 되어야 하는가?"와 같은 질문은 그렇지 않았다면 놓쳤을 엣지 케이스를 드러냅니다.

## 테스트 안티 패턴 (게으른 QA 훈련)

만약 TDD가 철칙이라면, `skills/test-driven-development/testing-anti-patterns.md`는 모든 신입 사원이 첫날에 시청하는 교육 비디오입니다. 이 파일은 엔지니어들이 보기에는 좋지만 아무것도 증명하지 못하는 테스트를 작성하는 특정 방식들을 문서화합니다. 시니어 스태프 엔지니어가 전체 조직의 테스트 스위트(test suite)에 대한 신뢰를 파괴하는 것을 목격한 패턴들 말입니다.

이 파일은 세 가지 자체적인 철칙으로 시작합니다:

1.  절대로 목(mock) 동작을 테스트하지 마십시오.
2.  절대로 프로덕션 클래스에 테스트 전용 메서드를 추가하지 마십시오.
3.  의존성(dependencies)을 이해하지 않고 목(mock)하지 마십시오.

### 목(Mock)을 테스트하고 코드를 테스트하지 않는 경우

첫 번째 안티 패턴은 가장 흔하고 가장 위험합니다:

```javascript
// ❌ 나쁨: 목(mock)이 존재하는지 테스트
test('renders sidebar', () => {
  render(<Page />);
  expect(screen.getByTestId('sidebar-mock')).toBeInTheDocument();
});
```
