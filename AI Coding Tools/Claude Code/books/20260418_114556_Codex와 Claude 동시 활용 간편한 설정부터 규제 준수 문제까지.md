# Codex와 Claude 동시 활용: 간편한 설정부터 규제 준수 문제까지

지난주 저는 Claude Code에 OpenAI의 Codex 플러그인을 설치했습니다. 네 개의 명령어와 5분 만에, 하나의 터미널 세션에서 두 개의 경쟁하는 AI 시스템(하나는 초안을 작성하고 다른 하나는 비평하는)을 동시에 실행하게 되었습니다.

이는 마치 마술처럼 느껴졌습니다. 그러던 중 저는 마이크로소프트가 같은 날 출시한 내용을 접하게 되었습니다.

이제 기업용 Microsoft 365 고객에게 제공되는 Copilot Cowork는 완전히 다른 규모로 동일한 기능을 수행합니다. GPT는 연구 보고서 초안을 작성하고, Claude는 이를 감사하며, 세 번째 모델은 둘을 종합합니다.

이는 가장 가까운 경쟁사의 자체 테스트로 측정되고 해당 공급업체의 모델로 평가되었을 때, 13.8%의 벤치마크 개선을 보였습니다. 그리고 규제 대상 기관에서 일하는 저희에게는, 아직 어떤 기술 언론도 다루지 않은 규제 준수(compliance) 격차가 존재합니다.

이 글은 두 가지 측면을 다룹니다.

*   **실용적인 측면**: 오늘날 여러분의 워크플로에서 Claude Code와 Codex를 어떻게 결합하는지, 그리고 두 번째 의견이 정확히 어디에서 중요한지 보여주는 전후(before/after) 사례를 제시합니다.
*   **아키텍처적인 측면**: 마이크로소프트가 실제로 무엇을 구축했는지, 벤치마크 이야기가 겉보기보다 왜 더 복잡한지, 그리고 이 제품을 배포하려는 모든 은행이 출시 전에 물어봐야 할 특정 질문이 무엇인지 다룹니다.

## 아무도 이야기하지 않는 공식 플러그인

3월 30일, OpenAI는 Claude Code용 공식 플러그인을 출시했습니다. 커뮤니티에서 분리되거나(fork) 해킹된 것이 아니라, OpenAI 자체 GitHub 계정으로 게시된 공식 릴리스이며, 그들의 직접적인 경쟁사 도구 내에서 Codex를 실행하도록 설계되었습니다.

이 점은 주목할 만합니다. 두 회사는 동일한 시장을 놓고 경쟁하며, 그들의 모델은 같은 벤치마크에서 경쟁합니다. 그런데 OpenAI는 Claude Code의 기능을 확장하는 플러그인을 막 출시한 것입니다.

돌이켜보면 사업적 논리는 명확합니다. Claude Code에 익숙한 개발자들은 떠나지 않을 것이고, Codex는 개발자들이 있는 곳에 있어야 합니다. 하지만 그 전략 이면과 관계없이 그 결과는 유용합니다.

**플러그인의 실제 기능**: 플러그인은 Claude 세션 내에 Codex를 백그라운드 하위 프로세스(subprocess)로 추가합니다. Claude는 파일 읽기, 편집, 컨텍스트 기반 추론과 같은 활성 작업을 처리합니다. Codex는 여러분이 부여한 작업을 비동기적으로(asynchronously) 실행하여, 다른 작업이 막히지 않도록 합니다. 두 개의 모델, 두 개의 실행 환경, 하나의 터미널입니다.

Claude Code + Codex: 두 개의 에이전트, 하나의 터미널

설정은 5분밖에 걸리지 않습니다. Node.js 18.18+ 이상, ChatGPT 계정(무료 티어 사용 가능) 또는 OpenAI API 키, 그리고 이미 실행 중인 Claude Code가 필요합니다.

**Codex CLI 설치:**

```
npm install -g @openai/codex
codex login
```

**Claude Code 내 플러그인 추가:**

```
/plugin marketplace add openai/codex-plugin-cc
/plugin install codex@openai-codex
/reload-plugins
/codex:setup
```

마지막 명령어는 모든 것이 연결되었는지 확인합니다. 버전 번호가 출력되면 준비된 것입니다.

**비용**: 자체 OpenAI API 키를 사용하는 경우, 각 `/codex:rescue` 호출은 GPT-5.4 요금(백만 토큰당 $2.50/$15)으로 약 2,000 입력 토큰과 1,000 출력 토큰이 소모되며, 호출당 약 $0.02입니다. 하루 50회 호출 시 약 $1/일, 즉 월 $30입니다. 백그라운드 연구 루프를 실행하는 헤비 유저는 토큰 사용량을 주시해야 합니다. 무료 ChatGPT 계정은 속도 제한(rate-limits)이 심하므로, 일반적인 사용에는 API 키가 더 안정적입니다.

## 적대적 검토(Adversarial Review)가 실제로 찾아내는 것

작성자 다이어그램: 적대적 검토(Adversarial Review)가 실제로 무엇을 찾아내는가

코드 작업의 경우 `/codex:adversarial-review`를 사용하세요. 이는 단순히 버그를 확인하는 것을 넘어, 설계 결정에 이의를 제기하고, 가정에 질문하며, 고려하지 못했던 엣지 케이스(edge cases)를 드러냅니다.

실제 사례는 다음과 같습니다. 저는 Claude에게 API 응답 배치(batch)를 처리하는 파이썬 함수를 작성해 달라고 요청했습니다.

**Claude의 출력 (요약):**

```python
def process_batch(responses):
    results = []
    for r in responses:
        if r.status_code == 200:
            results.append(r.json()["data"])
    return results
```

**Claude의 자체 코드 검토**: "올바르게 보입니다. 성공 사례를 처리하고 200이 아닌 응답을 필터링합니다."

`/codex:adversarial-review`를 실행한 후, Codex는 세 가지 문제를 지적했습니다.

*   API가 200 응답을 반환했지만 "data" 키가 없는 경우 `r.json()["data"]`에서 `KeyError`가 발생합니다. 이는 부분 응답(partial responses)의 실제 사례입니다.
*   `r.json()` 자체 실패에 대한 처리 부재 (200 응답에서 잘못된 형식의 JSON은 생각보다 흔합니다).
*   200이 아닌 응답을 묵묵히 버리면(silent discard) 오류가 사라져, 하류(downstream) 모니터링 시스템이 실패를 놓치게 됩니다.

Claude는 동일한 코드를 읽고 문제가 없다고 말했습니다. 두 개의 모델, 다른 답변. 둘 다 유용했지만, 그중 하나가 실제로 더 유용했습니다.

이것이 작동하는 이유는 마법이 아닙니다. Claude와 Codex는 다른 훈련 분포(training distributions), 다른 미세 조정 이력(fine-tuning histories), 다른 실패 모드(failure modes)를 가지고 있습니다. 그들이 의견이 다를 때, 그 불일치는 대개 실제 문제를 가리키고 있습니다.

코딩이 아닌 작업의 경우, `/codex:rescue`는 연구 방식을 변화시킵니다. 이 페어링(pairing)을 통해 가치를 얻기 위해 코드를 작성할 필요는 없습니다.

문서를 작성하는 도중에 특정 주제에 대한 배경 지식이 필요하지만, 현재 Claude의 컨텍스트를 잃고 싶지 않다고 가정해 봅시다. 이 작업을 위임하세요.

```
/codex:rescue research the EU AI Act Article 12 logging requirements
and summarize what a bank needs to document for high-risk AI systems
```

Codex는 이 작업을 백그라운드에서 실행합니다. 여러분은 계속 글을 쓸 수 있습니다. 완료되면:

```
/codex:result
```

요약 내용이 여러분의 세션으로 돌아옵니다. 탭 전환도, 컨텍스트 손실도, 기다림도 없습니다. 저는 경쟁사 조사, 초안 작성 중 사실 확인, 특정 기술적 접근 방식에 알려진 문제가 있는지 확인하는 데 이를 사용합니다.

**Codex 플러그인을 사용하지 말아야 할 때**: 이는 오버헤드(overhead)를 추가합니다. 한 시간 안에 버릴 작은 스크립트의 경우, 검토 주기가 도움이 되는 것보다 작업을 더 지연시킵니다. 문제를 아직 파악 중인 탐색적 작업에서는 초기 초안에 대한 적대적 검토가 주로 노이즈를 생성합니다. 그리고 Claude의 추론 깊이가 이미 작업의 복잡성을 초과하는 경우에는 두 번째 의견이 크게 도움이 되지 않습니다. 프로덕션(production)으로 넘어갈 PR(Pull Request), 의사 결정에 의존할 연구, 본인이 아닌 다른 사람이 읽을 문서 등 중요한 작업에 사용하세요.

이제 시야를 넓혀 봅시다. 여러분이 노트북에서 방금 설정한 방식, 즉 하나의 모델이 초안을 작성하고 다른 모델이 검토하는 방식은 마이크로소프트가 Outlook, Teams, Excel 내에서 기업 규모로 동일하게 제공하고 있습니다. 그리고 그 규모에서는 아직 어떤 기술 언론도 질문하지 않은 아키텍처적 문제가 발생합니다.

## 하나의 아키텍처가 아닌 두 개의 아키텍처

Copilot Cowork에 대한 대부분의 보도는 두 가지 별개의 기능을 혼동합니다. 이들은 서로 다른 아키텍처, 다른 장단점(tradeoffs), 그리고 책임(accountability)에 대한 다른 함의를 가집니다.

작성자 다이어그램: 비평(Critique): 순차적 검토 파이프라인

첫 번째는 **비평(Critique)**입니다.

비평은 순차적으로(sequential) 진행됩니다. GPT는 연구 작업을 계획하고, 검색을 반복하며, 인용(citations)이 포함된 초기 초안을 생성합니다. 그런 다음 Claude가 검토자로 나서서 소스 신뢰성, 보고서 완전성, 증거 기반(evidence grounding)의 세 가지 차원에서 초안을 감사합니다. 중간 초안은 사용자에게 보이지 않으며, 사용자가 받는 것은 단일 'Researcher' 답변으로 제공되는 최종 검토된 결과물입니다.

이 디자인은 의도적인 것입니다. 목표는 이미 검토된 하나의 답변을 제공하여 사용자 경험을 더 깔끔하게 만드는 것입니다. 검토는 백그라운드에서 이루어지며, 변경 사항이 있는 경우에만 표시됩니다.

이 패턴은 오늘날에도 복제할 수 있습니다. Codex 플러그인이 설치된 상태에서 `/codex:adversarial-review`는 Claude가 방금 생성한 코드, 문서, 연구 요약 등 모든 것에 대해 동일한 순차적 로직을 실행합니다. 검토자는 다른 훈련 분포와 다른 실패 모드를 가지고 있습니다. 불일치가 바로 가치가 있는 부분입니다.

작성자 다이어그램: 모델 협의회(Model Council): 병렬 아키텍처

두 번째는 **모델 협의회(Model Council)**이며, 작동 방식이 다릅니다.

모델 협의회는 병렬로(parallel) 실행됩니다. GPT와 Claude는 동시에 동일한 질문에 대해 완전하고 독립적인 보고서를 각각 생성합니다. 그런 다음 세 번째 "심판 모델(judge model)"이 두 가지 출력물을 평가하여, 모델들이 동의하는 부분, 갈라지는 부분, 그리고 각 모델이 다른 모델이 놓친 것을 무엇을 발견했는지 보여주는 종합적인 결과(synthesis)를 생성합니다. 불일치가 가시적으로 드러나며, 여러분은 그것을 볼 수 있습니다.

이것은 더 정직한 아키텍처입니다. 이는 두 개의 최첨단 모델이 동일한 소스 자료에 대해 다른 결론에 도달할 수 있음을 인정하고, 그 긴장감을 숨기지 않고 드러냅니다.

이 또한 복제할 수 있습니다. Claude가 활성 세션에서 동일한 질문에 답하는 동안 `/codex:rescue [your question]`을 실행하세요. `/codex:result`가 반환되면 두 결과를 비교합니다. 일치하는 부분에서는 신뢰도가 높아지고, 갈라지는 부분에서는 검토할 가치가 있는 것이 있습니다. 저는 제가 수치를 매기려는 모든 연구 주장에 이 방법을 사용합니다.

작성자 다이어그램: 비평 vs. 모델 협의회 vs. DIY 플러그인

장단점(tradeoff)은 투명성 대 편의성입니다. 비평은 깔끔한 답변을 제공합니다. 모델 협의회는 불일치의 지도를 제공합니다. 어떤 것을 원하는지는 출력물을 어떻게 사용할 것인지, 그리고 우리가 보게 되겠지만, 얼마나 책임감(accountable)을 져야 하는지에 따라 달라집니다.

두 기능 모두 3월 30일부로 Copilot Cowork의 Frontier 티어에서 사용할 수 있습니다. 둘 다 관리자 활성화가 필요합니다. 즉, IT 부서가 M365 테넌트(tenant)에 대한 Claude 액세스를 승인해야 활성화됩니다.

작성자 다이어그램: DRACO 벤치마크: 다중 모델 검토의 승리

간략한 상업적 맥락: 마이크로소프트의 주가는 2026년 1분기에 23% 하락하여 2008년 이래 최악의 분기를 기록했습니다. 4억 5천만 명의 M365 구독자 중 3.3%만이 Copilot에 비용을 지불하고 있습니다. ChatGPT와 직접 경쟁할 때, 사용자들은 18%의 경우에 Copilot을 선택합니다. 5월 1일부터 $99/시트(seat)로 제공되는 E7 번들(bundle)은 M365 E5 + Copilot + Agent 365 + Entra Suite를 묶는 수익화 수단입니다. Cowork의 다중 모델(multi-model) 기능은 가격 인상을 정당화합니다. 그리고 그들이 선택한 벤치마크인 DRACO는 그들의 직접적인 경쟁사인 Perplexity에 의해 만들어졌습니다. 마이크로소프트는 자체적으로 테스트를 실행하고 GPT-5.2를 심판(judge)으로 사용했으며(초안 작성 모델과 동일한 공급업체), 결과는 독립적으로 복제되지 않았습니다. 13.8%의 개선은 실제일 수 있지만, 결과에 이해관계가 없는 누구에 의해서도 검증되지 않았습니다.

한 가지 더 특이한 점은 Anthropic과의 파트너십입니다. 마이크로소프트는 Anthropic의 Claude를 사용하여 Anthropic 자체의 독립형 Claude Cowork 제품과 직접 경쟁하는 제품을 구동하고 있습니다.

동일한 기반 모델, 그러나 매우 다른 거버넌스(governance) 계층입니다.

## 귀속(Attribution) 격차

은행에서 일하는 사람으로서, 비평(Critique)에 대해 읽었을 때 제가 처음 던진 질문은 "작동하는가?"가 아니었습니다. 그것은 "어떤 모델이 어떤 말을 했고, 이를 감독관(examiner)에게 증명할 수 있는가?"였습니다.

오늘날 기준으로, 답은 "아니요"입니다.

구체적인 문제는 다음과 같습니다. 비평은 중간 초안을 숨기도록 설계되었습니다. 이는 두 개의 경쟁적인 출력물을 조정할 필요 없이 깔끔한 최종 답변을 제공하는 기능입니다. 하지만 이는 어떤 모델 버전이 어떤 주장을 생성했는지에 대한 정체성이 마이크로소프트의 오케스트레이션(orchestration) 계층 뒤에 추상화되어 있다는 것을 의미합니다. 여러분은 하나의 'Researcher' 결과물을 받게 되며, 그 결과물에는 모델 기여(model attribution)가 포함되어 있지 않습니다.

대부분의 사용 사례에서는 괜찮지만, 규제된 사용 사례에서는 그렇지 않습니다.

연방준비제도(Federal Reserve)의 모델 위험 관리 지침(SR 11-7)은 은행이 중대한 결정에 사용하는 모든 모델에 적용됩니다. 이는 은행이 각 모델을 식별하고, 독립적으로 검증하며, 그 한계를 문서화하고, 지속적인 성능을 모니터링할 것을 요구합니다. 중요한 점은, 벤더(vendor)에게 아웃소싱(outsourcing)한다고 해서 은행의 규제 책임(regulatory accountability)이 이전되지 않는다는 것입니다. OCC, 연준, FDIC는 모두 생성형 AI 배포에 SR 11-7 원칙을 적극적으로 적용하고 있습니다.

비평은 두 개의 모델 파이프라인(pipeline)을 생성합니다. SR 11-7은 두 모델 모두에 대한 문서화를 요구합니다. 출력물은 어떤 GPT 버전이 초안을 작성했고 어떤 Claude 버전이 비평했는지 알려주지 않습니다. 마이크로소프트가 어느 모델이든 조용히 업데이트할 경우(그럴 수 있습니다), 여러분은 알지 못할 수도 있습니다.

EU AI 법(EU AI Act)은 두 번째 계층을 추가합니다. 2026년 8월 2일부터 발효되는 제12조는 고위험 AI 시스템(high-risk AI systems)이 추적 가능성(traceability)을 갖춘 자동 이벤트 수준 로그(event-level logs)를 유지해야 한다고 요구합니다. 신용 평가, AML/사기 탐지, 대출 승인, KYC(고객 알기 제도) 시스템은 모두 부록 III에 따라 고위험으로 분류됩니다. 만약 은행이 이러한 기능 중 하나를 지원하는 연구에 비평을 배포한다면, 로깅(logging) 요구사항은 출력물뿐만 아니라 파이프라인에도 적용됩니다.

마이크로소프트가 현재 제공하는 것은 누가 언제 어떤 쿼리(query)를 실행했는지에 대한 M365 수준의 감사 로깅(audit logging)입니다. 이는 추론 호출(inference call)당 모델별 기여(per-model attribution)와는 다릅니다. E7 고객이 모델 수준 로그를 얻을 수 있는지 여부는 마이크로소프트가 공개적으로 답변하지 않은 질문입니다.

작성자 다이어그램: 귀속(Attribution) 격차: 필요한 것 vs. 비평이 제공하는 것

제가 우려하는 시나리오는 다음과 같습니다. 준법감시인(compliance officer)이 Copilot Researcher와 비평을 사용하여 특정 파생 상품 구조가 CFTC 지침에 따라 허용되는지 평가합니다. 파이프라인은 자신감 있고 잘 인용된 답변을 반환합니다. 하지만 한 인용문이 잘못 해석되었고, 한 규제 해석이 약간 틀렸습니다. 은행은 이에 의존합니다. 6개월 후 감독관이 묻습니다: 어떤 모델 버전이 그 해석을 생성했는가? SR 11-7에 따라 검증되었는가? 쿼리 시점에 모델 버전은 무엇이었는가?

현재 아키텍처에서는 은행이 이러한 질문에 답을 제시할 수 없습니다.

제가 모르는 부분에 대해 솔직히 말씀드리겠습니다. 마이크로소프트가 E7 고객에게 모델별 감사 로그를 제공하지만, 단순히 공개적으로 문서화하지 않았을 가능성도 있습니다. 다중 모델 AI 파이프라인에 대한 향후 FCA 지침이 비평과 같은 시스템에 대해 "적절한 로깅"이 무엇을 의미하는지 명확히 할 가능성도 있습니다. 이들은 미해결 질문입니다. 하지만 "가능성"은 은행 감독관에게 만족스러운 답변이 아닙니다. 그리고 현재로서는 공개 문서가 이 격차를 해소하지 못하고 있습니다.

## 무엇을 해야 할까요?

### 어떤 다중 모델 패턴을 사용해야 할까요?

작성자 다이어그램: 어떤 다중 모델 패턴을 사용해야 할까요?

**개발자 또는 지식 노동자라면**: 플러그인은 github.com/openai/codex-plugin-cc에서 찾을 수 있습니다. 5분이면 설정이 완료됩니다. 다음으로 중요한 작업에 `/codex:adversarial-review`를 사용해 보세요. 직접 처리할 경우 컨텍스트를 방해할 수 있는 작업에는 `/codex:rescue`를 사용하세요. 한 가지 주의할 점은 Claude와 Codex 간의 자동화된 루프(loops)가 API 사용량을 빠르게 소모할 수 있다는 것입니다. 자신의 패턴을 알게 될 때까지는 항상 사람을 개입시키세요(keep a human in the loop).

**규제 대상 기관에서 Copilot Cowork를 평가하고 있다면**, 출시 전 다음 벤더(vendor) 콜에서 세 가지 질문을 하십시오.

*   현재 비평(Critique) 파이프라인에는 어떤 모델 버전이 포함되어 있으며, 모델이 업데이트될 때 통보해 줄 것입니까?
*   E7 고객은 감사 로그에서 모델별 기여(per-model attribution)를 받습니까, 아니면 M365 쿼리(query) 로그만 통합하여 받습니까?
*   귀사의 문서는 특히 비평 기능에 대한 SR 11-7 모델 인벤토리 및 EU AI 법 제12조 추적 가능성(traceability) 의무를 어떻게 지원합니까?

답변이 모호하다면, 그것이 바로 답변입니다. 이는 배포하지 말라는 의미가 아니라, 문서화가 완료될 때까지 고위험 AI 사용 사례와 관련된 작업에 비평을 배포하지 말라는 의미입니다.

**모델 위험을 관리한다면**: 비평(Critique)을 출시 전에 SR 11-7 검토 대상에 올리세요. 모델 협의회(Model Council)가 더 방어적입니다. 두 출력물 모두 볼 수 있고, 둘 다 문서화할 수 있기 때문입니다. 감독관이 호출할 때, 그 투명성 장단점(transparency tradeoff)은 겉보기보다 훨씬 중요합니다.

**주목할 점**: 다중 모델 감사 추적(audit trails)에 대한 FCA 지침은 2026년 하반기에 발표될 예정입니다. 독립적인 DRACO 복제본(replications)은 몇 달 내에 나올 것입니다. EU AI 법은 8월 2일부터 은행에 적용됩니다.

## 제 워크플로에서 실제로 바뀐 점

적대적 검토(adversarial review) 패턴은 제 작업 방식에 미묘한 변화를 가져왔습니다. 이전에는 확신이 없는 내용에 대해 Claude에게 두 번째 검토를 요청했지만, 이제는 Codex를 통해 처리합니다. 이는 Codex가 더 똑똑해서가 아니라, 다른 맹점(blind spots)을 가진 검토자가 동일한 모델이 자신의 출력물을 읽는 것보다 더 유용하기 때문입니다.

또한, 저는 그들이 불일치하는 경우를 노이즈로 취급하는 것을 멈췄습니다. 그 불일치는 대개 중요한 발견입니다. 두 개의 최첨단 모델이 동일한 소스 자료에 대해 다른 결론에 도달하는 것은 거의 항상 검토할 가치가 있는 가정을 하고 있다는 것을 의미합니다.

제가 아직 직장에서 Copilot Cowork의 비평(Critique) 기능을 사용하지 않은 이유는 이 기능이 인상적이지 않아서가 아닙니다. 하지만 마이크로소프트가 귀속(attribution) 질문에 답하기 전까지는, 감독관 앞에 놓일 수 있는 어떤 작업에든 이를 배포하는 것은 제가 책임져야 할 문서화 격차처럼 느껴집니다. 모델 협의회(Model Council)가 더 정직합니다. 저는 그곳에서 시작할 것입니다.

다중 모델 품질 루프(multi-model quality loop)가 도래했습니다. 기술 아키텍처는 영리합니다. 그러나 거버넌스 아키텍처는 아직 따라잡고 있으며, 그 격차는 마이크로소프트가 아닌 여러분이 관리해야 할 부분입니다.
