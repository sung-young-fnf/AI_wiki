# `CLAUDE.md` 파일을 프롬프트 파일처럼 다루지 마세요

저는 한동안 사람들이 Claude Code를 사용하는 것을 지켜봐 왔는데, 계속해서 똑같은 실수를 발견합니다. 사람들은 모든 것을 `CLAUDE.md` 파일에 쏟아붓습니다. 모든 지시사항, 모든 예외 상황(edge case), 코드베이스에 대한 모든 세부사항까지요. 그러고는 왜 Claude가 자꾸 놓치거나 이상한 행동을 하는지 의아해합니다.

아무도 알려주지 않는 사실이 있습니다. Claude Code는 더 많은 정보를 필요로 하는 것이 아닙니다. 올바른 정보가 올바른 위치에 있는 것을 필요로 합니다.

이렇게 생각해 보세요. 만약 여러분이 시니어 엔지니어를 고용했다면, 첫날부터 50페이지짜리 문서를 건네주고 모든 것을 기억하라고 기대하지 않을 것입니다. 코드베이스의 지도를 주고, 어디에 무엇이 있는지 알려준 다음, 작업하면서 세부사항을 알아가도록 할 것입니다.

Claude Code도 마찬가지입니다. 여러분의 프로젝트는 정보의 양(volume)이 아니라 구조(structure)를 필요로 합니다.

## Claude가 항상 필요로 하는 네 가지

실제 파일과 폴더에 대해 알아보기 전에, Claude가 제대로 작동하기 위해 항상 필요한 네 가지를 설명해 드리겠습니다.

*   **목적(The why):** 이 시스템이 무엇을 하려고 하는가?
*   **지도(The map):** 각 요소는 어디에 있는가?
*   **규칙(The rules):** 허용되는 것과 그렇지 않은 것은 무엇인가?
*   **워크플로우(The workflows):** 여기서 작업은 실제로 어떻게 수행되는가?

이게 전부입니다. 그 외의 모든 것은 구현 세부사항(implementation detail)입니다.

이 네 가지를 중심으로 저장소(repo)를 정리하면, Claude는 세 메시지 전에 말한 내용을 잊어버리는 챗봇처럼 행동하는 것을 멈춥니다. 대신 코드베이스를 실제로 아는 엔지니어처럼 느껴지기 시작할 것입니다.

어떻게 하는지 보여드리겠습니다.

## 1. `CLAUDE.md` = 저장소의 기억 (짧게 유지하세요)

이 파일은 여러분의 북극성(north star) 파일입니다. 저장소(repo)의 루트(root)에 위치합니다.

하지만 함정이 있습니다. 너무 길게 만들면 모델이 중요한 부분을 건너뛰기 시작합니다. 저는 이런 일이 반복적으로 발생하는 것을 보았습니다. 누군가 800줄짜리 아름답고 포괄적인 `CLAUDE.md` 파일을 작성했는데, Claude는 그 중 절반을 무시하더군요.

세 가지에 집중하세요:

*   **목적(Purpose, WHY)** — 이 시스템은 무엇을 하는가? 최대 한두 단락(paragraph).
*   **저장소 지도(Repo map, WHAT)** — 중요한 디렉토리(directory)는 어디에 있는가? 무엇이 어디에 있는가?
*   **규칙 및 명령어(Rules + commands, HOW)** — 핵심 제약사항(constraint)과 일반적인 작업(operation).

좋은 예시는 다음과 같습니다:

```markdown
# Project Overview

This is an e-commerce API built with Node.js and PostgreSQL. 
Handles orders, payments, and inventory for retail clients.
## Structure
- `/src/api` - REST endpoints
- `/src/services` - Business logic
- `/src/models` - Database models
- `/src/jobs` - Background workers
- `/tests` - Integration and unit tests
## Rules
- Never modify `/src/auth` without explicit approval
- All DB changes require a migration file
- Run `npm test` before any PR
## Common Commands
- `npm run dev` - Start local server
- `npm run migrate` - Run pending migrations
- `npm run test:watch` - Run tests in watch mode
```

이것은 20줄에 불과합니다. 필요한 모든 것을 담고, 필요 없는 것은 없습니다.

핵심은 모든 뉘앙스(nuance)를 설명하려는 충동을 억제하는 것입니다. Claude가 필요할 때 더 많은 맥락(context)을 요청할 것이라고 믿으세요.

## 2. `.claude/skills/` = 같은 지시를 반복해서 작성하지 마세요

대부분의 사람들이 기회를 놓치는 지점입니다.

여러분은 반복적으로 수행하는 작업들을 알고 있을 것입니다. 코드 검토(code review), 오래된 모듈(module) 리팩토링(refactoring), 릴리스(release) 준비, 그리고 프로덕션(production) 문제 디버깅(debugging) 같은 것들이죠?

아마도 여러분은 매번 Claude에게 유사한 지시를 내릴 것입니다. "예외 상황(edge case)을 확인해라." "테스트가 통과하는지 확인해라." "우리의 코드 스타일을 따라라." 같은 것들이요.

그만하세요. 대신 그것들을 스킬(skill)로 만드세요.

`.claude/skills/` 디렉토리(directory)를 생성하고 재사용 가능한 워크플로우(workflow)를 거기에 넣으세요:

```
.claude/
  skills/
    code-review.md
    refactor.md
    release-prep.md
    debug-prod.md
```

스킬(skill) 파일의 예시는 다음과 같습니다:

```markdown
# Code Review Checklist

When reviewing code, always check:
1. **Tests** - Does the change have tests? Do they cover edge cases?
2. **Error handling** - What happens if this fails?
3. **Performance** - Any N+1 queries? Unnecessary loops?
4. **Security** - Does this touch user data? Auth? Payments?
5. **Documentation** - Do new functions have docstrings?
If you find issues, list them clearly. Don't just say "looks good."
```

이제 "코드 검토 스킬(code review skill)을 사용하여 이 PR(Pull Request)을 검토해 줘"라고 말하면, Claude는 매번 동일한 체크리스트(checklist)를 따릅니다. 변동도 없고, 단계를 잊는 일도 없습니다.

## 3. `.claude/hooks/` = 실제로 작동하는 안전장치(Guardrail)

모델은 정보를 잊어버립니다. 그게 현실입니다.

Claude에게 "항상 포매터(formatter)를 실행하라"고 수백 번 말해도, 가끔은 건너뛸 것입니다. 모델이 나빠서가 아니라, 언어 모델이 완벽한 기억력(recall)을 가지고 있지 않기 때문입니다.

훅(hook)이 이 문제를 해결합니다.

훅은 워크플로우(workflow)의 특정 지점에서 자동으로 실행되는 결정론적(deterministic) 검사입니다. 기억력이 필요 없습니다.

```
.claude/
  hooks/
    post-edit.sh
    pre-commit.sh
    on-test-change.sh
```

간단한 편집 후 훅(post-edit hook)의 예시는 다음과 같습니다:

```bash
#!/bin/bash
# .claude/hooks/post-edit.sh

# Run formatter on edited files
prettier --write "$EDITED_FILES"
# Check if we touched core auth files
if echo "$EDITED_FILES" | grep -q "src/auth/"; then
  echo "⚠️  Auth files modified - manual review required"
  exit 1
fi
# Run relevant tests
npm test -- --findRelatedTests "$EDITED_FILES"
```

이제 Claude에게 코드를 포맷(format)하라고 상기시킬 필요가 없습니다. 그저 매번 자동으로 일어날 뿐입니다.

다음과 같이 일관성을 유지해야 하는 작업에 훅(hook)을 사용하세요:

*   편집 후 포매터(formatter) 실행
*   핵심 파일 변경 시 테스트 실행
*   위험한 디렉토리(directory) (인증, 결제, 데이터베이스 마이그레이션) 편집 차단
*   파일 명명 규칙(naming convention) 유효성 검사

훅(hook)을 `pre-commit hook`에 넣을 법한 것들이라고 생각하되, Claude가 방금 무엇을 변경했는지 알기 때문에 더 스마트하다고 보면 됩니다.

## 4. `docs/` = Claude가 필요할 때 진실(정보)을 찾게 하세요

`CLAUDE.md` 파일에 전체 아키텍처(architecture) 문서를 넣지 마세요.

대신, Claude에게 진실(정보)이 어디에 있는지 알려주세요. 그러면 필요할 때 찾아볼 수 있습니다.

```
docs/
  architecture.md
  adr/
    001-chose-postgres.md
    002-event-sourcing.md
    003-auth-strategy.md
  runbooks/
    deploy.md
    rollback.md
    database-backup.md
```

여러분의 `CLAUDE.md`는 이것을 참조하기만 하면 됩니다:

```markdown
## Documentation

- Architecture overview: `/docs/architecture.md`
- Key decisions: `/docs/adr/`
- Operations: `/docs/runbooks/`
When making architectural changes, check relevant ADRs first.
```

이제 Claude는 여러분이 전체 기술 스택(tech stack)을 미리 설명할 필요 없이 어디를 봐야 할지 압니다.

이것은 점진적인 맥락(progressive context)입니다. Claude는 필요한 시점에 정확히 세부 정보를 얻고, 모든 것을 한 번에 얻지 않습니다.

## 5. 핵심 모듈(Sharp Edges)을 위한 로컬 `CLAUDE.md` 파일

제가 좋아하는 패턴이 있습니다. 위험한 코드 옆에 작은 `CLAUDE.md` 파일들을 두는 것입니다.

```
src/
  auth/
    CLAUDE.md
    session.js
    tokens.js
  persistence/
    CLAUDE.md
    migrations/
    models/
  payments/
    CLAUDE.md
    stripe.js
    refunds.js
```

각 파일은 해당 모듈(module)에 특화된 맥락(context)을 포함합니다:

```markdown
# Auth Module

⚠️ HIGH RISK - Changes here affect all users
## Critical Rules
- Never modify session validation logic without security review
- Token expiry is 24h - hardcoded for compliance
- Password hashing uses bcrypt with cost factor 12
## Testing
Changes to this module require:
- Unit tests for logic
- Integration tests for auth flow
- Manual QA in staging
## Common Gotchas
- Sessions are stored in Redis, not DB
- Refresh tokens live 30 days (separate from access tokens)
- SSO users don't have passwords - check `user.auth_type`
```

이제 Claude가 해당 디렉토리(directory)에서 작업할 때, 경고와 제약사항(constraint)을 즉시 확인합니다. 바로 중요할 때 말이죠.

이것은 루트(root) `CLAUDE.md` 파일에서 모든 모듈(module)의 모든 특별한 경우를 기억하려고 애쓰는 것보다 훨씬 좋습니다.

## 구조가 만드는 차이점

이렇게 정리했을 때 어떤 일이 일어나는지 보여드리겠습니다:

**구조가 없을 때:**

당신: "인증 모듈(auth module)을 리팩토링(refactor)해 줘."

Claude: 변경 시작

당신: "잠깐, 테스트(test)는 실행했니?"

Claude: "아, 제가 그렇게 하겠습니다."

당신: "그리고 토큰 만료(token expiry)를 변경할 수 없어. 준수(compliance) 사항이야."

Claude: "맞아요, 죄송합니다. 되돌리겠습니다."

**구조가 있을 때:**

당신: "인증 모듈(auth module)을 리팩토링(refactor)해 줘."

Claude: `src/auth/CLAUDE.md`를 읽고, 고위험(high risk)임을 인지

Claude: 제약사항(constraint)을 확인하고, 토큰 만료(token expiry)가 하드코딩(hardcoded)되어 있음을 인지

Claude: 변경사항을 적용하고, 편집 후 훅(post-edit hook)이 테스트(test)를 자동으로 실행

Claude: "세션(session) 처리를 리팩토링(refactor)했습니다. 테스트(test)는 통과했습니다. 토큰 만료(token expiry)는 (준수 사항이므로) 건드리지 않았습니다. 검토 준비 완료."

차이가 보이시나요? 여러분은 더 이상 잔소리꾼이 아닙니다. 협력자(collaborator)입니다.

## 작게 시작하여 필요할 때 추가하세요

첫날부터 이 모든 것이 필요하지는 않습니다.

기본적인 `CLAUDE.md` 파일부터 시작하세요. 필수적인 내용만 담으세요. 지시사항을 반복하고 있다는 것을 알게 되면, 그것들을 스킬(skill)로 만드세요. Claude가 중요한 것을 잊어버리는 것을 발견하면, 훅(hook)을 추가하세요. 모듈(module)이 혼란을 야기할 때, 해당 모듈에 로컬 `CLAUDE.md` 파일을 제공하세요.

여러분의 저장소(repo) 구조는 어떤 완벽한 이론적 프레임워크(framework)가 아니라 실제 필요에 따라 유기적으로 성장할 것입니다.

목표는 모든 것을 문서화하는 것이 아닙니다. Claude가 이미 여러분의 코드베이스(codebase)를 알고 있는 것처럼 느끼게 하는 것입니다.

## 이것이 실제로 중요한 이유

저는 팀들이 단순히 구조를 추가하는 것만으로 "Claude는 단지 멋진 자동 완성(autocomplete) 도구일 뿐"이라고 생각하던 것에서 "Claude는 기본적으로 우리의 첫 번째 코드 검토자(code reviewer)"라고 여기게 되는 것을 보았습니다.

Claude가 더 똑똑해졌기 때문이 아닙니다. 프로젝트를 탐색(navigate)하기가 더 쉬워졌기 때문입니다.

프롬프팅(prompting)은 일시적입니다. 한 번 작성하면 해당 세션(session)에 도움이 되고, 그 다음에는 사라집니다.

구조는 영구적입니다. 모든 세션에 도움이 됩니다. 모든 팀원에게 이득이 됩니다. 지식은 축적됩니다.

여러분의 저장소(repo)가 이런 식으로 설정되면, Claude는 더 이상 여러분이 관리해야 하는 도구처럼 느껴지지 않습니다. 실제로 무슨 일이 일어나고 있는지 아는 엔지니어처럼 느껴지기 시작합니다.

그리고 그때 비로소 유용해지는 것입니다.
