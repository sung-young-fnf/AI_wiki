전체 크기로 이미지를 보려면 Enter 키를 누르거나 클릭하세요.

저는 선임 머신러닝 엔지니어(Senior Machine Learning Engineer)로서 매일 Claude Code를 사용합니다. 대부분의 사람들은 AI에 대한 멘탈 모델(mental model)이 여전히 ChatGPT 시대에 머물러 있습니다. 즉, 질문을 입력하면 답변을 받는 식이죠. Claude Code는 다릅니다. 이는 실제 파일을 읽고, 쓰고, 편집하며, 터미널 명령(terminal command)을 실행하고, 코드베이스(codebase)에서 직접 작동하는 에이전트(agent)입니다. 이러한 특성 덕분에 강력하지만, 구조 없는 힘은 새로운 문제를 야기합니다.

이제 코드를 작성하는 것은 쉬운 부분입니다. 실제로 좋은 코드를 얻으려면 단순히 프롬프트(prompt)를 입력하는 것 이상의 노력이 필요합니다. 올바른 습관이 없으면 결국 나중에 다시 작성해야 할 "AI 슬롭(AI slop)"을 생성하게 됩니다. 저는 이를 어려운 방식으로 배웠습니다. 시간이 지나면서, Claude Code를 효과적으로 사용하는 간단한 프레임워크로 성공적이었던 것(그리고 그렇지 않았던 것)을 정리했습니다.

이 프레임워크는 두 가지 범주로 나뉩니다. 한 번 설정하면 모든 향후 세션을 더 스마트하게 만드는 '복합 습관(compounding habits)'과, 현재 진행 중인 작업을 올바르게 유지하는 '세션 습관(session habits)'입니다. 이 두 가지를 합하면 Claude Code에서 더 일관되고 신뢰할 수 있는 결과물을 얻을 수 있는 반복 가능한 시스템을 제공합니다.

### 파워 유저 습관

Claude Code를 최대한 활용하기 위해 복잡한 설정이 필요하지 않습니다. Claude Code의 개발자인 보리스 체르니(Boris Cherny)는 자신의 설정이 "놀랍도록 평범하다(surprisingly vanilla)"고 말했습니다. 그는 도구 자체가 이미 강력하기 때문에 크게 커스터마이징(customize)하지 않습니다. 우위는 복잡한 설정(config)이 아니라 습관에서 나옵니다.

#### 복합 습관 (Compounding Habits)

1.  **`CLAUDE.md`를 설정하고 최신 상태로 유지하세요.**
    프로젝트 루트(root)에 `CLAUDE.md` 파일을 생성하려면 `/init`을 입력하세요. Claude는 모든 세션 시작 시 이 파일을 읽으므로, 파일에 있는 내용은 Claude가 매번 그냥 알고 있습니다. 기술 스택(tech stack), 빌드(build) 및 테스트(test) 명령어, 그리고 적용하고 싶은 모든 규칙으로 채우세요. 200줄 미만으로 유지하세요. `CLAUDE.md`가 너무 길어지면, 규칙을 `.claude/rules/` 폴더 내의 별도 파일로 옮기세요. Claude는 이 파일들도 같은 방식으로 로드하며, 특정 파일이 편집될 때만 규칙이 적용되도록 범위를 지정할 수 있습니다. 규칙 문서(rules docs)를 참조하세요.

    핵심 습관: Claude가 잘못된 작업을 할 때마다, 다시는 그런 일이 발생하지 않도록 `CLAUDE.md`에 메모를 추가하세요. Anthropic의 Claude Code 팀은 내부적으로 이 작업을 수행하며 이를 "복합 엔지니어링(compounding engineering)"이라고 부릅니다. 각 수정 사항은 영구적인 컨텍스트(context)가 되므로, Claude는 해당 종류의 오류를 다시는 저지르지 않습니다. 문제가 발생했을 때만 `CLAUDE.md`를 업데이트하지 마세요. 기술 스택이 변경되거나, 컨벤션(convention)이 진화하거나, 코드베이스에 대해 새로운 것을 배울 때도 업데이트하세요. 팀에서는 이 습관이 훨씬 더 빠르게 복합적으로 작용합니다. 한 사람이 추가하는 모든 규칙이 자동으로 다른 모든 사람의 세션에 이점을 주기 때문입니다.

2.  **Claude에게 피드백 루프(feedback loop)를 제공하세요.**
    모든 변경 후 테스트, 린터(linter), 또는 빌드 명령을 실행하도록 Claude에게 지시하세요. 피드백 루프가 없으면 Claude는 변경 사항을 적용하고, 작동한다고 가정하며 다음으로 넘어갑니다. 피드백 루프가 있으면 Claude는 스스로 실수를 포착합니다. 체르니는 이 한 가지만으로도 결과물의 품질을 2~3배 향상시킬 수 있다고 말했습니다. 또 다른 복합 습관: `CLAUDE.md`에 피드백 루프를 연결하면, 모든 향후 세션에서 무료로 이를 활용할 수 있습니다. "Python 파일을 변경한 후에는 항상 `pytest`를 실행하라"와 같은 한 줄이면 충분합니다.

#### 세션 습관 (Session Habits)

1.  **복잡한 작업은 플랜(Plan) 모드에서 시작하세요.**
    작업이 여러 파일에 걸치거나 연구가 필요한 경우, 먼저 플랜 모드로 전환하세요. 오타 수정과 같은 한 줄짜리 변경에는 건너뛰세요. 체르니는 대부분 자신의 세션을 플랜 모드에서 시작합니다. 뭔가 이상하면 되돌리고, 만족스러우면 노멀(Normal) 또는 자동 승인(Auto-Accept) 모드로 전환하여 구현하도록 지시하세요. 이것이 핵심 세션 습관입니다. 10초의 계획이 10분의 되돌리기를 절약합니다.

2.  **구체적으로 명시하고 파일을 직접 참조하세요.**
    "인증 버그를 수정해줘"라고 말하는 대신, "@utils.py 파일의 42번째 줄 근처에 있는 마지막 항목을 건너뛰는 오프-바이-원 오류(off-by-one error)를 수정해줘"와 같이 시도해보세요. 처음부터 상세한 프롬프트는 여러 번의 수정 왕복(round trip)을 줄여줍니다.

3.  **컨텍스트 윈도우(context window)를 보호하세요.**
    모든 파일 읽기, 명령어 출력, 그리고 주고받는 메시지는 컨텍스트를 소모합니다. Claude는 이제 이 중 많은 부분을 관리하며, 컨텍스트가 무거워지면 압축(compact)하거나 지우도록(clear) 프롬프트를 띄웁니다. 큰 작업의 경우 더 신중하게 접근하세요. 요약하고 공간을 확보하려면 `/compact`를 사용하거나, 관련 없는 작업 사이에서 새롭게 시작하려면 `/clear`를 사용하세요. 연구 중심 작업의 경우, 서브 에이전트(subagent)를 사용하여 주요 컨텍스트에서 부담을 덜 수 있습니다 (자세한 내용은 아래 기능 섹션 참조).

4.  **작업을 작게 유지하세요.**
    "인증, 결제, 체크아웃 기능을 갖춘 전체 전자상거래 사이트를 만들어줘"는 사실 열 개의 작업을 하나로 묶은 것입니다. 작업이 클수록 Claude가 요구 사항을 놓치거나 연쇄적인 오류를 일으킬 가능성이 높아집니다. 작업을 나누세요. "인증, 결제, 체크아웃 기능을 갖춘 전체 전자상거래 사이트 계획을 세우고, 이를 여러 단계(waves)로 나누어줘"로 시작하세요. 각 단계를 하나씩 처리하고, 그 사이에는 컨텍스트 윈도우를 지우세요. 작은 작업은 또한 더 깨끗한 체크포인트(checkpoint)를 생성하므로, 문제가 발생했을 때 되돌려야 할 작업이 줄어듭니다.

이 모든 습관에 대해 더 자세히 알아보려면 모범 사례 문서(best practices docs)를 참조하세요.

## 흔한 함정

*   **Claude가 프로젝트 컨벤션(convention)을 무시합니다.**
    Claude는 작동하는 코드를 작성하지만, 코드베이스가 카멜 케이스(camelCase)를 사용하는데 스네이크 케이스(snake_case)를 사용하는 것과 같이 사용자의 패턴을 따르지 않을 수 있습니다. 이는 거의 항상 `CLAUDE.md` 규칙이 누락되었기 때문입니다. "변수에는 항상 `camelCase`를 사용하고 클래스에는 `PascalCase`를 사용하라"와 같은 내용을 추가하면 Claude는 그 시점부터 해당 컨벤션을 따릅니다. 이것이 바로 실제 복합 엔지니어링(compounding engineering)입니다. 여러분이 인코딩(encode)하는 모든 수정 사항은 모든 향후 세션을 더 좋게 만듭니다.

*   **복잡한 작업이 통제 불능 상태로 진행됩니다 (spirals).**
    Claude가 예상치 못한 파일을 편집하기 시작하거나 변경 사항이 눈덩이처럼 불어납니다. 작업을 중지하고(Ctrl+C), 필요하다면 Esc+Esc로 되감고(rewind), 플랜(Plan) 모드로 전환한 다음 작업을 더 작은 조각으로 나누세요. 그런 다음 각 조각을 한 번에 하나씩 실행하세요.

## 알아둘 만한 기능

Claude Code에는 더 고급 워크플로우를 가능하게 하는 몇 가지 기능이 있습니다.

*   **파이프 모드(pipe mode)를 위한 `claude -p`**
    `claude -p`는 Claude를 비대화형(non-interactive)으로 실행하므로, 스크립트(script) 및 자동화(automation)에 사용할 수 있습니다. 예를 들어, `git diff --cached | claude -p 'review these staged changes for bugs'`와 같이 Git 프리-커밋 훅(git pre-commit hook)에 연결하여 모든 커밋(commit) 전에 코드를 자동으로 검토할 수 있습니다. 전체 플래그(flag) 목록은 CLI 참조 문서(CLI reference)를 참조하세요.

*   **병렬 세션(parallel session)을 위한 `--worktree`**
    `claude --worktree feature-auth`는 격리된 Git 워크트리(git worktree)를 생성합니다. 충돌 없이 여러 브랜치(branch)에서 여러 Claude 세션을 실행할 수 있습니다. 워크트리 문서(worktree docs)를 참조하세요.

*   **컨텍스트 격리(context isolation)를 위한 내장 서브 에이전트(subagent)**
    Claude Code에는 자체 별도 컨텍스트 윈도우에서 실행되는 내장 서브 에이전트(탐색, 계획, 일반 목적)가 있습니다. Claude는 작업이 필요할 때 자체적으로 이러한 서브 에이전트를 생성합니다. 모든 자세한 출력(verbose output)은 서브 에이전트의 컨텍스트에 남아 있으며, 요약만 반환됩니다. 직접 지시할 수도 있습니다. 예를 들어, "인증 모듈(authentication module), 데이터베이스 계층(database layer), API 경로(API route)를 병렬로 조사하기 위해 세 개의 서브 에이전트를 가동하고, 찾은 내용을 요약해줘"와 같이 말이죠. 경험 법칙: 작업이 약 5개 이상의 파일과 관련되어 있다면, 서브 에이전트에 격리하는 것을 고려하세요. 서브 에이전트 문서(subagents docs)를 참조하세요.

*   **내장 스킬(skill)**
    Claude Code는 `/`로 호출할 수 있는 내장 스킬을 제공합니다. 예를 들어, `/simplify`는 코드의 재사용성(reuse), 품질(quality), 효율성(efficiency)을 검토한 후 수정 사항을 적용합니다. 새로운 스킬은 정기적으로 추가되므로, 세션에서 `/`를 입력하여 현재 사용 가능한 스킬을 확인하세요. 스킬 문서(skills docs)를 참조하세요.

*   **사용자 지정 서브 에이전트 및 스킬 (Custom subagents and skills)**
    직접 만들 수 있습니다. 가장 쉬운 방법은 Claude에게 "보안 문제를 검토하는 서브 에이전트를 만들어줘" 또는 "스테이징(staging)에 배포하는 스킬을 만들어줘"와 같이 요청하면 Claude가 파일을 생성해줍니다. Claude Code의 확장 시스템(스킬, 서브 에이전트, 플러그인)에 대해 더 자세히 알아보려면, 제가 별도로 작성한 글 "Claude Code를 위한 멘탈 모델: 스킬, 서브 에이전트, 플러그인 (A Mental Model for Claude Code: Skills, Subagents, and Plugins)"을 참조하세요. 직접 빌드하고 싶다면 서브 에이전트 문서(subagents docs) 및 스킬 문서(skills docs)를 참조하세요.

## 결론

Claude Code를 가장 잘 활용하는 사람들은 마법 같은 프롬프트(magic prompt)를 작성하지 않습니다. 그들은 두 가지 범주에 투자하고 있습니다. 이전 세션보다 모든 세션을 더 스마트하게 만드는 복합 습관(`CLAUDE.md`, 피드백 루프)과, 각 작업을 올바르게 유지하는 세션 습관(계획 수립, 구체성, 컨텍스트 관리, 작은 범위)입니다. 이것이 전체 프레임워크입니다.

그리고 도구가 개선됨에 따라 이러한 습관도 함께 복합적으로 작용합니다. 컨텍스트 윈도우는 커지고, 모델은 자기 수정(self-correcting) 능력이 향상되며, 모든 개선은 잘 관리된 `CLAUDE.md`와 규율 있는 세션의 가치를 증대시킵니다. 프레임워크는 동일하게 유지됩니다. 결과는 계속해서 더 좋아질 것입니다.
