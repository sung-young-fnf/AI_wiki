# 오픈소스 레포지토리 하나로 Claude Code를 n8n 아키텍트로 변신시키다: 더욱 강력해진 n8n 활용법

지난 화요일 밤 11시, 저는 **Claude Code**가 운영 중인 이커머스 스토어에서 140개의 상품 변형(product variants) 데이터를 삭제하는 광경을 목격했습니다. 악의적인 의도는 없었습니다. 기술적으로 틀린 작업을 수행한 것도 아니었습니다.

제가 수정을 요청했던 워크플로우(workflow)가 프레스타샵(PrestaShop)의 XML API에 단 4개의 필드만 포함된 PUT 요청을 보냈던 것이 화근이었습니다. 프레스타샵은 이를 "나머지 모든 필드를 빈 값으로 설정하라"는 명령으로 해석했습니다. 다른 터미널 탭에서 실행 로그를 비교하는 사이, 참조 코드와 바코드 등 140개 조합의 데이터가 순식간에 사라졌습니다.

**요약하자면:** n8n은 건재합니다. 다만 수동으로 워크플로우를 구축하는 시대가 저물고 있을 뿐입니다. `czlonkowski/n8n-mcp`라는 오픈소스 레포지토리(repository) 덕분에 Claude Code는 n8n 아키텍트로 진화했습니다. 저는 이를 55개 노드(node)로 구성된 실제 운영 파이프라인에 적용해 보았고, 반대로 토큰만 낭비하던 저의 자체 AI 에이전트는 비활성화했습니다. 아래에 무엇이 실제로 작동하고, 무엇이 문제이며, 어떤 도구를 언제 사용해야 하는지에 대한 프레임워크를 정리했습니다.

### n8n의 역할 정의를 바꾼 MCP 레포지토리

3개월 전, Claude Code에게 n8n 워크플로우 구축을 요청했을 때 그것은 `scheduleTrigger`라는 노드를 만들어냈습니다. 하지만 그런 노드는 존재하지 않습니다. 실제 명칭은 `schedule`입니다. 저는 유효해 보이지만 속성 이름이 미세하게 틀린 JSON 파일의 임포트 오류를 디버깅하는 데 20분을 허비했습니다. 완벽한 환각(hallucination) 증상이었습니다.

하지만 개발자 czlonkowski가 공개한 오픈소스 **MCP(Model Context Protocol)** 서버를 사용한 뒤로 상황이 달라졌습니다. 이제 Claude Code는 1,084개의 n8n 노드, 템플릿에서 추출된 2,600개 이상의 실제 설정값, 배포 전 실수를 잡아내는 유효성 검사 규칙을 완벽하게 파악하고 있습니다. 환각도, 추측도 사라졌습니다.

많은 이들이 Claude Code가 n8n을 대체할 것이라고 말하지만, 실제 변화는 더 구체적입니다. n8n의 시각적 빌더(visual builder)는 사라지는 것이 아니라 역할이 바뀌는 것입니다. 예전에는 '건설 도구'였다면, 이제는 '모니터링 대시보드'가 된 것입니다. **Claude Code가 구축하고, n8n이 실행합니다.**

워크플로우를 수정하는 주체가 변경 사항을 문서화하고, 실제로 무슨 일이 일어났는지 설명하는 메시지와 함께 깃(Git)에 커밋(commit)까지 수행합니다. n8n UI에서 수동으로 JSON을 내보내고 가져오는 클릭 작업 없이도 완벽한 롤백(rollback) 기능을 갖추게 된 것입니다.

### 실제 운영 환경에서의 디버깅 사례

유럽의 한 아웃도어 쇼핑몰 파이프라인(55개 노드 규모)을 운영하며 겪은 세 가지 결정적인 순간을 소개합니다.

#### 1. 보이지 않는 삭제(Silent Wipe)
앞서 언급한 프레스타샵 API의 부분 PUT 요청 문제였습니다. Claude Code는 MCP를 통해 실행 데이터를 가져와 손상 정도를 확인한 후, '수정 전 조회(GET-before-PUT)' 패턴을 구현하는 4개의 새로운 코드 노드를 구축했습니다. UI에서 직접 작업했다면 노드 배치와 연결, 설정에 20분은 걸렸을 작업을 API 호출 한 번으로 해결하고 3분 만에 검증까지 마쳤습니다.

#### 2. 존재하지 않는 타입 문제
구글 시트(Google Sheets)는 셀이 비어 있으면 빈 문자열(`""`)을 반환합니다. n8n의 `IF` 노드에서 `exists` 연산자는 빈 문자열도 '존재하는 것'으로 간주하여 통과시킵니다. MCP 실행 검사기(execution inspector) 덕분에 각 노드에서 나오는 데이터 구조를 정확히 확인하여 이 혼합 타입 문제를 즉시 파악할 수 있었습니다.

#### 3. 완벽하지 않은 테스트
복잡한 로직을 수정할 때 Claude Code는 정규식과 XML 로직을 로컬에서 검증했지만, n8n 표현식 참조나 노드 체이닝까지 완벽히 검증하지는 못했습니다. 아무리 똑똑한 도구라도 실제 환경의 규칙을 배우기 위해서는 때때로 실행 실패 과정이 필요합니다.

### 왜 자체 AI 에이전트를 포기했는가?

저는 $5짜리 VPS에서 작동하는 멀티 모델 AI 에이전트인 `OpenClaw`를 구축해 운영해 왔습니다. 하지만 결국 전원을 껐습니다. 투입된 토큰(token) 비용 대비 생산된 가치가 마이너스였기 때문입니다. 에이전트는 실제 업무를 수행하는 대신 자기 자신을 업데이트하고 도구를 수정하는 데 너무 많은 리소스를 소모했습니다.

반면, 저의 n8n 워크플로우는 수개월 동안 안정적으로 돌아가고 있습니다. 웹훅(webhook)이 실행되고, 크론(CRON) 작업이 수행되며, 상품 데이터가 동기화됩니다. 초기 구축 비용 외에는 추가 토큰 비용이 전혀 들지 않습니다.

비즈니스 자동화의 80%는 지루한 작업들입니다. 추론(reasoning)하는 에이전트가 필요한 것이 아니라, 저렴한 비용으로 안정적으로 실행되는 파이프라인이 필요할 뿐입니다.

### 프레임워크: 무엇을 언제 사용할 것인가?

실제 운영 경험을 바탕으로 내린 결정 기준입니다.

1.  **n8n 단독 사용:** 안정적이고 반복적인 프로세스. 웹훅, 예약된 크론 작업, API 간 데이터 동기화 등 가동 시간이 유연성보다 중요할 때 사용합니다.
2.  **Claude Code 단독 사용:** 반복되지 않는 작업. 데이터셋 변환을 위한 일회성 스크립트나 프로토타입 제작 등 배포보다 탐색이 우선일 때 사용합니다.
3.  **Claude Code + n8n (MCP 연동):** 복잡한 파이프라인의 디버깅, 기존 워크플로우의 마이그레이션 또는 리팩토링(refactoring), 수십 개의 워크플로우 자동 문서화가 필요할 때 최적입니다.
4.  **자율형 AI 에이전트:** 입력값이 예측 불가능하여 의사결정 트리를 미리 작성할 수 없는 경우, 즉 '진정한 추론'이 필요할 때만 사용합니다. 저의 경험상 이는 자동화 작업의 10% 미만입니다.
