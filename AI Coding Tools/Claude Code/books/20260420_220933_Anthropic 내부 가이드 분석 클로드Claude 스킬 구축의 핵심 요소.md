# Anthropic 내부 가이드 분석: 클로드(Claude) 스킬 구축의 핵심 요소

저는 클로드(Claude) 도구에 대해 오랫동안 글을 써왔습니다. 클로드 코드(Claude Code), 프롬프트 명령, 옵시디언(Obsidian) 통합 등이 있죠. 하지만 스킬(skills)은 항상 미뤄왔던 주제였는데, 설명이 부족하고 문서가 이미 사용법을 알고 있다고 가정하는 듯한 느낌을 받았기 때문입니다.

그러던 중 Anthropic이 30페이지에 달하는 완벽한 내부 가이드를 공개했습니다. 모든 패턴, 주의사항, 체크리스트가 담겨 있었죠. 저는 이 가이드를 전부 읽어보았습니다.

다음은 그중에서 정말 중요한 내용들입니다.

### 스킬(Skill)이란 무엇인가

스킬은 그저 폴더입니다. 그게 전부입니다.

이 폴더 안에는 핵심 지시 파일인 `SKILL.md` 파일이 있으며, 선택적으로 일부 스크립트, 참고 문서 또는 애셋(asset) 템플릿이 포함될 수 있습니다. 폴더를 압축한 다음 `Claude.ai`의 `Settings > Capabilities > Skills`를 통해 업로드하면, 그때부터 클로드는 사용자가 설명한 모든 워크플로(workflow)를 처리하는 방법을 알게 됩니다.

아이디어는 간단합니다. 매 대화마다 프로세스를 다시 설명하는 대신, 클로드에게 한 번만 가르치는 것입니다. 스프린트 계획, 문서 생성, 온보딩(onboarding) 워크플로, API 통합 등 반복 가능한 모든 것이 스킬이 됩니다.

폴더 구조는 다음과 같습니다:

```
your-skill-name/
├── SKILL.md          ← 필수
├── scripts/          ← 선택 사항
├── references/       ← 선택 사항
└── assets/           ← 선택 사항
```

스킬은 `Claude.ai`, 클로드 코드(Claude Code), 그리고 API에서 동일하게 작동합니다. 한 번 만들면 어디에서나 사용할 수 있습니다.

### 3단계 로딩 시스템

이 부분은 대부분의 사람들이 놓치는 부분이며, 일부 스킬이 작동하고 일부 스킬이 작동하지 않는 이유를 잘 설명해줍니다.

스킬은 세 가지 수준으로 로드(load)됩니다:

*   **레벨 1 — YAML 프론트매터(frontmatter):** 모든 대화에서 클로드의 시스템 프롬프트(system prompt)에 항상 존재합니다. 클로드가 스킬의 관련성을 판단하는 곳입니다. 몇 줄에 불과한데, 이것이 핵심입니다. 토큰 사용량(token usage)을 낮게 유지합니다.
*   **레벨 2 — SKILL.md 본문:** 클로드가 스킬이 적용된다고 판단할 때 로드됩니다. 실제 지시 사항이 있는 곳입니다.
*   **레벨 3 — `references/` 내 링크된 파일:** 클로드에게 필요할 때만 로드됩니다. 심층 문서, API 패턴, 예외 처리 가이드 등이 여기에 해당합니다.

이 시스템의 결과는 다음과 같습니다. 프론트매터(frontmatter) 설명(description)이 전체 스킬에서 가장 중요한 요소입니다. 설명이 좋지 않으면 클로드는 스킬을 로드하지 않습니다. 아무도 지시 사항을 읽지 않는다면 지시 사항은 무의미합니다.

### `SKILL.md` 파일 작성

이 파일은 YAML 헤더(프론트매터)와 본문(실제 지시 사항)의 두 부분으로 구성됩니다.

#### 프론트매터(Frontmatter)

최소한의 유효한 형식은 다음과 같습니다:

```yaml
---
name: your-skill-name
description: 스킬의 기능과 사용 시점.
---
```

`name` 필드는 반드시 케밥 케이스(kebab-case)여야 합니다. 공백, 대문자, 밑줄은 허용되지 않습니다. `my-cool-skill`은 작동하지만, `My Cool Skill`은 작동하지 않습니다.

`description` 필드는 대부분의 사람들이 실수를 저지르는 부분입니다. 스킬이 무엇을 하는지, 그리고 언제 사용해야 하는지 두 가지를 모두 포함해야 합니다. 둘 다 필요하며, 하나만으로는 안 됩니다.

**좋은 설명의 예시:**

*   Figma 디자인 파일을 분석하고 개발자 핸드오프 문서를 생성합니다. 사용자가 `.fig` 파일을 업로드하거나 "디자인 사양", "컴포넌트 문서", "디자인-코드 핸드오프"를 요청할 때 사용합니다.
*   스프린트 계획, 작업 생성, 상태 추적을 포함하는 Linear 프로젝트 워크플로를 관리합니다. 사용자가 "스프린트", "Linear 작업", "프로젝트 계획"을 언급하거나 "티켓 생성"을 요청할 때 사용합니다.

**작동하지 않는 설명의 예시:**

*   프로젝트에 도움을 줍니다. (너무 모호함)
*   정교한 다중 페이지 문서 시스템을 생성합니다. (트리거(trigger) 문구가 없어 클로드가 언제 로드해야 할지 알 수 없음)

선택적 필드도 추가할 수 있습니다:

```yaml
---
name: my-skill
description: ...
license: MIT
metadata:
  author: Your Name
  version: 1.0.0
  mcp-server: your-server-name
---
```

한 가지 엄격한 규칙은 프론트매터(frontmatter) 어디에도 XML 꺾쇠괄호(`< >`)를 사용해서는 안 된다는 것입니다. 이는 프론트매터가 클로드의 시스템 프롬프트(system prompt)에 직접 들어가기 때문에 보안 제한 사항입니다.

#### 지시 사항 본문

프론트매터 다음에는 일반 마크다운(Markdown)으로 지시 사항을 작성합니다. 가이드에서는 다음과 같은 구조를 권장합니다:

```markdown
# 스킬 이름

## 1단계: 첫 번째 작업
명확한 설명. 무엇을 해야 하는지, 성공적인 결과는 무엇인지.

## 2단계: 다음 작업
...
## 예시
사용자: "새 프로젝트를 설정해줘"
액션:
1. MCP를 통해 기존 프로젝트 가져오기
2. 제공된 매개변수로 새 프로젝트 생성
결과: 확인 링크와 함께 프로젝트 생성됨
## 문제 해결
오류: 연결 거부됨
원인: MCP 서버가 실행 중이 아님
해결책: Settings > Extensions > [Service] > Reconnect
```

구체적으로 작성해야 합니다. "진행하기 전에 데이터를 검증하세요"는 클로드에게 아무것도 알려주지 않습니다. "파이썬 스크립트(`python scripts/validate.py --input {filename}`)를 실행하고, 실패하면 필수 필드 누락 또는 날짜 형식을 확인하세요"와 같이 클로드가 실제로 조치할 수 있는 내용을 작성해야 합니다.

`SKILL.md`는 핵심에 집중하고, 심층 문서는 `references/`로 이동시켜 지시 사항에서 링크하도록 합니다. 이것이 점진적 공개(progressive disclosure) 시스템이 작동하는 방식입니다.

### 세 가지 스킬(Skill) 범주

Anthropic은 초기 개발자들로부터 세 가지 주요 사용 사례를 확인했습니다:

*   **범주 1: 문서 및 애셋(asset) 생성.** 문서, 프레젠테이션, 코드, 디자인 등 일관된 출력을 생성하는 스킬입니다. 프론트엔드 디자인(frontend-design) 스킬이 여기에 해당합니다. 핵심 패턴: 스타일 가이드 내장, 템플릿 사용, 완료 전 품질 검사 실행. 외부 도구는 필요하지 않습니다.
*   **범주 2: 워크플로 자동화.** 일관된 방법론을 가진 다단계 프로세스입니다. 스킬 생성자(skill-creator) 스킬 자체가 여기에 해당합니다. 핵심 패턴: 검증 게이트가 있는 단계별 진행, 반복적인 개선 루프(loop), 내장된 개선 제안.
*   **범주 3: MCP 향상.** 기존 MCP 서버 연결 위에 워크플로 지침을 추가하는 것입니다. Sentry의 코드 리뷰(code-review) 스킬이 그 예시인데, Sentry의 MCP를 사용하여 GitHub PR(Pull Request)의 버그를 분석합니다. 핵심 패턴: 여러 MCP 호출을 순서대로 실행, 도메인 전문 지식 내장, 일반적인 오류 처리.

### 스킬(Skills)과 MCP

이미 클로드에 연결된 MCP 서버가 있다면, 스킬은 부족했던 부분입니다.

가이드에서는 부엌 비유를 사용합니다. MCP는 도구, 재료, 모든 것에 대한 접근 권한을 제공하는 '부엌'과 같습니다. 스킬은 실제로 무엇을 만들어야 하는지에 대한 단계별 지시 사항인 '레시피'를 제공합니다.

스킬이 없으면 사용자는 MCP 서버를 연결한 후 무엇을 해야 할지 모릅니다. 일관성 없이 프롬프트(prompt)를 입력하고, 일관성 없는 결과를 얻으며, 결국 연결 도구를 탓하게 됩니다.

스킬이 있으면 워크플로가 자동으로 활성화됩니다. 클로드는 순서를 알고 있으며, 도메인 전문 지식(domain expertise)이 내장되어 있습니다.

Anthropic은 MCP 향상 스킬에 잘 맞는 다섯 가지 패턴을 확인했습니다:

*   **패턴 1: 순차적 워크플로 오케스트레이션(orchestration).** 단계가 순서대로 진행되어야 할 때. 계정 생성 → 결제 설정 → 구독 생성 → 환영 이메일 전송. 각 단계는 다음으로 이동하기 전에 유효성을 검사하며, 실패 시 롤백(rollback) 지시 사항이 포함됩니다.
*   **패턴 2: 다중 MCP 조정.** 워크플로가 여러 서비스에 걸쳐 있을 때. Figma에서 내보내기 → Drive에 업로드 → Linear에서 작업 생성 → Slack에 알림. 명확한 단계 분리, 서비스 간 데이터 전달, 중앙 집중식 오류 처리.
*   **패턴 3: 반복적 개선.** 반복을 통해 품질이 향상될 때. 초안 생성 → 검증 → 문제 식별 → 수정 → 재검증 → 임계값 충족까지 반복. 보고서, 문서 등 정의된 품질 기준이 있는 모든 것에 효과적입니다.
*   **패턴 4: 상황 인식 도구 선택.** 동일한 결과에 대해 처리되는 내용에 따라 다른 도구가 필요할 때. 10MB 초과 파일? 클라우드 저장소 사용. 협업 문서? Notion 사용. 코드 파일? GitHub 사용. 스킬은 결정을 내리고 사용자에게 그 이유를 설명합니다.
*   **패턴 5: 도메인 특화 지능.** 스킬이 단순히 도구를 실행하는 것을 넘어 전문 지식을 추가할 때. 거래 전에 규정 준수 검사를 실행하는 결제 처리 스킬. 관할권 규칙을 적용하는 법률 스킬. 전문 지식이 로직(logic)에 내장되어 사용자에게 맡겨지지 않습니다.

### 스킬(Skill) 테스트

가이드는 테스트를 세 가지 영역으로 나눕니다.

*   **트리거(Trigger) 테스트:** 스킬이 올바른 시점에 로드되는가? 스킬을 트리거해야 하는 쿼리(query) 10~20개를 실행하고 실제로 얼마나 많이 트리거되는지 추적합니다. 또한 관련 없는 쿼리에서는 트리거되지 않는지도 테스트합니다. 관련 쿼리에서는 90%의 정확도를 목표로 하고, 명확히 관련 없는 쿼리에서는 0%를 목표로 합니다.
*   **기능 테스트:** 스킬이 올바른 출력을 생성하는가? 동일한 요청을 3~5회 실행하고 결과를 비교합니다. 도구 호출 횟수를 세고, API 오류를 모니터링합니다.
*   **성능 비교:** 스킬을 사용하는 것이 실제로 사용하지 않는 것보다 나은가? 스킬 활성화 여부에 따른 토큰 사용량(token usage), 주고받는 메시지 수, 실패한 API 호출을 비교합니다.

가이드에는 실용적인 디버깅(debugging) 팁도 있습니다. 클로드에게 직접 "언제 [스킬 이름] 스킬을 사용하겠니?"라고 물어보십시오. 클로드는 스킬의 설명을 인용하여 답변할 것입니다. 이 답변에서 부족한 점이 드러나면, 무엇을 고쳐야 할지 정확히 알 수 있습니다.

반복 작업에 효과적인 한 가지 패턴은 다음과 같습니다. 먼저 클로드가 하나의 어려운 작업을 성공적으로 수행하도록 한 다음, 그 접근 방식을 스킬로 추출하는 것입니다. 처음부터 광범위한 기능을 구축하려고 하지 마십시오. 하나의 작업을 안정적으로 작동시킨 다음 확장하십시오.

### 일반적인 실패 및 해결책

*   **스킬이 업로드되지 않음:** 주로 이름 지정 문제입니다. 파일은 정확히 `SKILL.md`여야 하며, 대소문자를 구분하고 변형이 없어야 합니다. 폴더는 케밥 케이스(kebab-case)여야 합니다. "Invalid frontmatter" 오류가 표시되면 YAML 구분자(`---` 줄)를 확인하고 닫히지 않은 따옴표가 없는지 확인하십시오.
*   **스킬이 트리거(trigger)되지 않음:** 설명이 너무 모호하거나 트리거 문구가 누락되었습니다. 사용자가 실제로 말할 법한 구체적인 언어를 추가하십시오. 클로드에게 스킬을 무엇이 트리거하는지 물어보면, 그 답변이 무엇이 부족한지 보여줄 것입니다.
*   **스킬이 너무 자주 트리거됨:** 명시적으로 부정적인 트리거를 추가하십시오. "X에는 사용하지 마세요 — 대신 Y 스킬을 사용하세요." 범위에 대해 더 구체적으로 작성하십시오.
*   **MCP 호출은 실패하지만 스킬은 로드됨:** 먼저 MCP 연결을 독립적으로 테스트하십시오. 스킬 없이 클로드에게 MCP 도구를 직접 호출하도록 요청하십시오. 그것이 실패하면 문제는 인증 또는 MCP 서버에 있으며, 스킬의 문제가 아닙니다.
*   **클로드가 스킬을 로드하지만 지시 사항을 무시함:** 몇 가지 원인이 있습니다. 지시 사항이 너무 길거나(세부 사항을 `references/`로 이동), 중요한 단계가 너무 아래에 있거나(맨 위에 배치), 언어가 너무 모호한 경우("검증"이 무엇을 의미하는지 명확히 설명)입니다. 정말 중요한 검사에는 검증 스크립트(validation script)를 고려하십시오. 코드는 결정적이지만, 자연어는 그렇지 않습니다.
*   **컨텍스트(context)가 느리거나 저하된 느낌:** `SKILL.md`를 5,000단어 미만으로 유지하십시오. 동시에 20~50개 이상의 스킬이 활성화되어 있다면, 트리밍(trimming)을 고려하십시오. 상세 문서를 `references/`로 이동시키고 링크하십시오.

### 스킬(Skill) 배포

현재(2026년 초 기준) 배포 흐름은 다음과 같습니다:

*   스킬을 GitHub에 호스팅(public repo, 사람 방문자를 위한 명확한 `README`, 스크린샷이 포함된 사용 예시)
*   MCP 문서가 있다면 스킬을 MCP 문서에 추가
*   사용자는 폴더를 다운로드하고 압축하여 `Settings > Capabilities > Skills`를 통해 업로드

조직 관리자는 2025년 12월에 출시된 기능으로, 자동 업데이트(automatic updates) 및 중앙 집중식 관리(centralized management)를 통해 워크스페이스(workspace) 전반에 걸쳐 스킬을 배포할 수도 있습니다.

API를 기반으로 구축하는 팀의 경우, 스킬은 `/v1/skills` 엔드포인트(endpoint)를 통해 접근할 수 있으며, `container.skills` 매개변수(parameter)를 통해 Messages API 요청에 첨부할 수 있습니다. 이 기능은 코드 실행 도구 베타(Code Execution Tool beta)를 필요로 합니다.

`README` 또는 다른 곳에서 스킬에 대해 작성할 때는 메커니즘이 아닌 '결과'를 설명하십시오. "수동 설정에 30분을 소비하는 대신 팀이 몇 초 만에 완전한 프로젝트 워크스페이스를 설정할 수 있게 합니다"가 "YAML 프론트매터(frontmatter)와 마크다운(Markdown) 지시 사항을 포함하는 폴더"보다 훨씬 설득력 있습니다.

### 체크리스트 (요약)

*   **시작하기 전에:** 2~3가지 구체적인 사용 사례를 식별하십시오. 필요한 도구를 파악하십시오.
*   **개발 중:** 폴더 이름은 케밥 케이스(kebab-case). `SKILL.md` 철자는 정확하게. `---` 구분자를 가진 YAML. 설명은 기능과 사용 시기를 모두 포함. XML 태그는 어디에도 없음. 오류 처리 및 예시를 포함한 명확하고 구체적인 지시 사항.
*   **업로드 전:** 명확하고 의역된 쿼리에 대한 트리거(trigger)를 테스트하십시오. 관련 없는 주제에는 트리거되지 않는지 확인하십시오. 실제 워크플로를 테스트하십시오.
*   **업로드 후:** 과도하거나 부족한 트리거링(triggering)을 모니터링하십시오. 피드백을 수집하십시오. 설명과 지시 사항을 반복적으로 개선하십시오.

이 가이드 전체를 읽으면서 가장 눈에 띄는 점은 '설명(description)'이라는 단일 필드에 얼마나 많은 비중이 실리는지였습니다. 다른 모든 것은 개선할 수 있습니다. 하지만 설명이 클로드에게 스킬을 언제 로드해야 하는지 알려주지 않으면, 다른 어떤 것도 중요하지 않습니다.

그것부터 제대로 하십시오. 다른 모든 것은 거기에서부터 시작됩니다.
