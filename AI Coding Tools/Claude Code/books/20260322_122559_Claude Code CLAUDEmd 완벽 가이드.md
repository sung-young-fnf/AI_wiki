AI 코딩 도구의 발전 타임라인을 그려본다면, Claude Code(클로드 코드)는 분명히 중요한 이정표 중 하나일 것입니다. Anthropic(앤트로픽)은 2025년 2월, 초기 공개 프리뷰(public preview) 형태로 개발자들에게 Claude Code를 처음 선보였고, 2025년 5월에는 훨씬 더 광범위한 개발자층에 도달하기 시작했습니다. 특이했던 점은 단순히 채팅을 위한 또 다른 모델 인터페이스(model interface)를 추가한 것이 아니라, AI 에이전트(AI agent)가 터미널(terminal) 환경으로 진정으로 진입하여, 저장소(repository)를 읽고, 파일을 편집하며, 명령어를 실행하고, 문제를 해결하며, 작업을 추진하는 첫 사례였다는 것입니다. 그 순간부터 소프트웨어 개발의 중심은 조용히 변화하기 시작했습니다. 오랫동안 개발자들의 주된 업무는 수동 구현이었고, IDE(통합 개발 환경)는 주로 코드 자동 완성, 검색 및 분석을 위한 보조 도구 역할을 했습니다. Claude Code는 그 경계를 훨씬 더 확장시켰습니다. 점점 더 우리는 다음 코드 한 줄이 무엇이어야 할지 고민하는 대신, 목표가 무엇인지, 제약 조건은 무엇인지, 어떤 파일을 변경할 수 있는지, 그리고 어떤 결과가 '완료'로 간주되는지에 대해 생각하게 됩니다. 개발 프로세스는 수동으로 코드를 한 줄씩 작성하는 방식에서 명확한 의도와 AI 협업을 통해 구현을 이끌어가는 방식으로 전환되고 있습니다. 이것은 단순한 기능 업그레이드가 아니라, 업무 수행 방식의 진정한 재편이며, 많은 개발자들이 AI가 프로그래밍 작업을 자율적으로 완료할 수 있다는 영향을 실제로 느낀 첫 번째 경험이었습니다.

바로 그렇기 때문에 저는 Claude Code를 효과적으로 사용하는 방법에 대한 일련의 글을 쓰고 싶습니다. 이미 온라인에는 훌륭한 글들이 많고, 공식 문서도 반드시 읽어볼 가치가 있지만, 저는 여전히 이 시리즈를 쓰고 싶습니다. 한 가지 이유는 Claude Code가 너무 빠르게 발전하여 오래된 경험이 빠르게 효력을 잃기 때문입니다. 또 다른 이유는 제가 실제 사용을 통해 배운 것들, 즉 겪었던 함정들과 사고방식의 변화를 정리하고 싶기 때문입니다. 글을 쓰는 것은 저 자신의 학습 경로를 재구성하고 Claude Code에 대한 이해를 더욱 견고히 하는 좋은 방법이기도 합니다. 이 시리즈의 첫 번째 글로, 저는 `CLAUDE.md`로 시작하고자 합니다. 왜냐하면 이 파일이 Claude Code가 프로젝트를 이해하는 진입점이기 때문입니다. `CLAUDE.md`는 AI에 필요한 배경 정보와 규칙을 제공하며, 모델이 코드베이스(codebase) 내에서 얼마나 정확하고 신뢰성 있게 작동하는지에 직접적인 영향을 미치는 경우가 많습니다.

## CLAUDE.md란 무엇인가

`CLAUDE.md`는 Claude Code가 각 세션(session) 시작 시 자동으로 읽는 마크다운(Markdown) 파일입니다. 이 파일에 지침, 규칙, 선호 사항을 작성하면, Claude Code는 이후의 상호작용 전반에 걸쳐 이를 따릅니다.

어떤 사람이 온라인에서 2024년 파리 올림픽 사격 경기의 비교 사진을 바탕으로 재미있는 밈(meme)을 공유한 적이 있습니다. 이 밈은 Claude Code 사용자를 두 그룹으로 나누어 농담했습니다. 왼쪽에는 각종 플러그인(plugin)을 서둘러 설치하며 완벽하게 장비를 갖춘 선수, 오른쪽에는 한 손을 주머니에 넣고 오직 하나의 `CLAUDE.md`에만 의존하여 침착하게 서 있는 터키 저격수였습니다.

(이미지)

농담일 뿐이지만, 이는 중요한 점을 시사합니다. 화려한 플러그인 설정에 시간을 낭비하는 대신, 좋은 `CLAUDE.md`를 작성하는 데 시간을 투자하는 것이 종종 더 나은 결과를 가져옵니다. `CLAUDE.md`는 AI가 프로젝트와 상호작용하는 방식을 근본적으로 바꾸기 때문에, 이에 진정으로 노력을 기울이는 사용자들은 예상보다 훨씬 더 좋은 경험을 얻는 경우가 많습니다.

유용한 비유를 들어보겠습니다. Claude Code를 팀에 새로 합류하는 개발자로 생각한다면, `CLAUDE.md`는 프로젝트의 온보딩(onboarding) 안내서입니다. 이 안내서가 없으면 새로운 팀원은 프로젝트를 어떻게 빌드하는지, 어떤 테스트 프레임워크(test framework)를 사용하는지, 어떤 코드 스타일(code style)을 따르는지 등 기본적인 질문을 계속해서 물어봐야 합니다. 안내서가 있으면, 첫날부터 팀이 기대하는 방식으로 효율적으로 작업할 수 있습니다.

`CLAUDE.md`의 핵심 가치: Claude Code를 범용 AI 어시스턴트(general-purpose AI assistant)에서 프로젝트에 맞춰진 개발 도구로 전환시킨다는 점입니다.

## CLAUDE.md는 어떻게 로드되는가

많은 사람들이 `CLAUDE.md`를 처음 접할 때, 단순히 프로젝트 루트(root)에 있는 마크다운 파일이라고 생각합니다. 하지만 실제로는 그렇지 않습니다. Claude Code는 계층형 로딩 모델(layered loading model)을 통해 메모리(memory)와 지침을 읽습니다. 서로 다른 위치에 있는 `CLAUDE.md` 파일들은 다른 스코프(scope, 범위)에 적용됩니다. 일부는 모든 프로젝트에 영향을 미치고, 일부는 현재 저장소(repository)에만 영향을 미치며, 일부는 특정 하위 디렉토리(subdirectory) 내에서만 유효합니다.

이것이 사람들이 실제 사용에서 종종 혼란을 겪는 이유이기도 합니다. 왜 동일한 규칙이 프로젝트 A에서는 작동하지만 프로젝트 B에서는 무시되는가? 왜 명확하게 규칙을 작성했는데, Claude Code가 특정 디렉토리에서는 여전히 이를 무시했는가? 결국 이러한 질문들은 모두 `CLAUDE.md`가 로드되는 방식으로 귀결됩니다.

### 파일 계층 구조

핵심 파일 유형부터 살펴보겠습니다.

(이미지)

일상적인 사용에서 가장 일반적인 것은 보통 프로젝트 수준의 `./CLAUDE.md`입니다. 이 파일은 버전 관리(version control)를 통해 공유되며, 해당 저장소 내에서 Claude Code의 기본 동작을 정의합니다. `~/.claude/CLAUDE.md`는 사용자의 전역 개인 환경설정 파일(global personal preference file)과 같으며, `CLAUDE.local.md`는 커밋(commit)하고 싶지 않지만 현재 프로젝트에 적용하고 싶은 노트(note)에 유용합니다.

일부 팀은 규칙을 `.claude/rules/` 디렉토리에 분할하여 보다 모듈식(modular)으로 구성하기도 하지만, 이는 이미 좀 더 고급 설정입니다. 이 글에서는 `CLAUDE.md` 자체에 집중하겠습니다.

(이미지)

### 로딩 규칙

Claude Code가 이러한 파일을 읽을 때, 대략 다음과 같은 규칙을 따릅니다.

*   Claude Code가 시작될 때, 현재 작업 디렉토리(working directory)와 관련된 `CLAUDE.md` 파일을 먼저 읽습니다.
*   사용자 수준의 `~/.claude/CLAUDE.md`도 더 높은 수준의 기본 설정 레이어(default preference layer)로 포함됩니다.
*   하위 디렉토리 내의 `CLAUDE.md` 파일은 미리 모두 로드되지 않고, Claude Code가 해당 디렉토리의 콘텐츠를 실제로 읽을 때만 로드됩니다.
*   여러 `CLAUDE.md` 파일이 동시에 활성화될 때, 가장 가까운 스코프 규칙(nearest-scope rule)이 일반적으로 적용됩니다. 즉, 현재 작업에 더 가깝고 스코프가 좁은 지침이 우선합니다.
*   동일한 레이어 내에서는 모호하고 일반적인 진술보다 더 명확하고 구체적인 규칙이 일관되게 따를 가능성이 높습니다.

### 충돌 해결

서로 다른 `CLAUDE.md` 파일이 충돌할 때, 가장 이해하기 쉬운 규칙은 '가장 가까운 스코프 원칙'입니다. 규칙이 현재 작업에 더 가깝고, 적용되는 스코프가 더 좁을수록 일반적으로 우선순위가 높습니다. 예를 들어, 사용자 수준 `CLAUDE.md`가 4칸 들여쓰기를 사용하라고 명시했지만, 프로젝트 루트 `CLAUDE.md`가 명확히 2칸 들여쓰기를 사용하라고 명시하면, Claude Code는 해당 프로젝트 내에서 2칸 들여쓰기를 따를 것입니다.

이는 여러 `CLAUDE.md` 파일이 존재할 수 있다는 것을 아는 것만으로는 충분하지 않기 때문에 중요합니다. 각 파일이 어떤 스코프에 적용되는지, 그리고 충돌이 발생했을 때 어떤 파일이 다른 파일을 무시하는지(override) 이해해야 합니다.

## CLAUDE.md 작성 방법

### `/init`으로 빠르게 시작하기

가장 쉽게 시작하는 방법은 프로젝트 루트에서 Claude Code의 `/init` 명령어를 실행하여 초안(first draft)을 생성하도록 하는 것입니다.

```
$ claude
> /init
```

`/init`은 기술 스택(tech stack), 디렉토리 구조(directory structure), 일반적인 명령어들을 분석하여 `CLAUDE.md` 파일을 생성하고 기본적인 시작 골격(skeleton)을 제공합니다. 그러나 이것은 빈 파일에서 시작하는 문제를 해결할 뿐입니다. 프로젝트에 진정으로 적합한 `CLAUDE.md`를 이미 가지고 있다는 의미는 아닙니다.

초안을 얻은 후에는 일반적으로 최소한 한 번의 편집 과정을 거쳐야 합니다. 이유는 간단합니다. `/init`은 광범위하고 포괄적인 내용을 생성하는 경향이 있습니다. 감지할 수 있는 모든 것을 포함하려고 시도하지만, 그 정보 중 일부는 프로젝트에 실제로 중요하지 않으며, 일부는 정확하더라도 Claude Code를 효과적으로 안내하기에 충분히 유용하지 않을 수 있습니다. `CLAUDE.md`는 단순히 길다고 해서 더 좋은 것이 아닙니다. 한편으로는 컨텍스트 윈도우(context window) 공간을 차지하고, 다른 한편으로는 정보가 너무 분산되고 일반화되면 Claude Code가 정말 중요한 제약 조건을 식별하기 어려워집니다. 더 나은 접근 방식은 먼저 가치가 낮거나, 일반적이거나, 이미 다른 곳에 문서화된 중복 정보를 삭제한 다음, 팀만 아는 프로젝트 지식, 특히 Claude Code가 실수를 저지를 가능성이 가장 높은 세부 정보를 추가하는 것입니다.

### 처음 작성할 때 가장 유용한 정보부터 시작하기

`CLAUDE.md`를 처음 작성할 때, 처음부터 완전함을 목표로 할 필요는 없습니다. 더 중요한 것은 Claude Code의 성능에 가장 직접적으로 영향을 미치는 정보부터 시작하는 것입니다. 다시 말해, 파일이 완전해 보이게 하는 것보다, 빠뜨리면 틀리기 쉬운 것들을 우선적으로 추가해야 합니다.

가장 가치 있는 정보는 보통 몇 가지 범주로 나뉩니다.

*   **일반적인 명령어**: 빌드(build), 테스트(test), 린트(lint), 로컬 개발 명령어 등. Claude Code가 매번 추측할 필요가 없도록 합니다.
*   **프로젝트별 제약 조건 및 함정**: 편집해서는 안 되는 디렉토리, 소프트 삭제(soft delete)를 사용하는 테이블, 특정 미들웨어(middleware)를 거쳐야 하는 인터페이스 등.
*   **기본 워크플로(workflow)**: 브랜치(branch) 이름 지정, 프리-커밋 체크(pre-commit checks), 기본 PR(Pull Request) 요구사항 등.
*   **필수 아키텍처(architecture) 컨텍스트**: 모노레포(monorepo) 내 각 디렉토리의 역할, 특정 계층 경계(layering boundary)를 넘어서는 안 되는 모듈 등.

어떤 정보가 Claude Code가 프로젝트를 더 빨리 이해하도록 돕지 않고 일반적인 실수를 줄이지 않는다면, 그 정보를 파일에 서둘러 넣을 이유가 없습니다. `CLAUDE.md`의 가치는 모든 것을 다루는 데 있는 것이 아닙니다. 그 가치는 가장 중요한 프로젝트 사실과 협업 제약 조건을 명확하게 전달하는 데 있습니다.

### 파일이 커지면 `@imports`로 분할하기

프로젝트가 더 복잡해지고 규칙이 계속 늘어날 때, 모든 것을 단일 파일에 계속 채워 넣지 마십시오. 이때는 `@imports`를 사용하여 더 자세한 가이드를 별도의 파일로 옮기고 메인 `CLAUDE.md`에서 참조할 수 있습니다.

```markdown
# Project Instructions

See @README.md for project overview.
See @package.json for available commands and scripts.
See @docs/testing.md for testing conventions.
See @docs/api-guidelines.md for API design rules.
```

이점은 메인 파일이 가장 자주 필요하고 먼저 보여야 할 정보를 계속 담을 수 있고, 더 자세한 표준, 워크플로, 규칙들은 별도로 관리할 수 있다는 점입니다. 얼마나 모듈화(modularize)해야 하는지, 어떤 콘텐츠를 분할할 가치가 있는지는 모범 사례(best practices) 섹션에서 다루겠습니다.

## CLAUDE.md는 어떻게 발전하는가

`CLAUDE.md`는 한 번 작성하고 다시는 건드리지 않는 종류의 파일이 아닙니다. 오히려 프로젝트의 진화하는 기억(evolving memory)과 같습니다. 진정으로 가치 있는 부분은 한 번에 발명되는 것이 아니라, 반복적인 협업, 수정, 회고(retrospective)를 통해 점진적으로 다듬어지는 것들입니다.

(이미지)

### 협업 문제를 통해 지속적으로 개선하기

물론 `/init`이 생성한 초안에서 시작할 수 있지만, 나중에 `CLAUDE.md`에 추가해야 할 내용은 상상에서 나오는 것이 아닙니다. 일상 업무에서 구체적인 협업 문제를 반복적으로 발견하는 데서 나옵니다. 한 사람이 가끔 임의의 노트를 추가하는 것에 의존해서는 안 됩니다. 팀 전체가 프로젝트 경험을 지속적으로 기록하고 보존하는 장소가 되어야 합니다.

이러한 문제들은 종종 매우 구체적입니다.

*   Claude Code가 `pnpm` 대신 계속 `npm`을 사용하나요? 규칙을 추가하세요.
*   Claude Code가 생성된 테스트 파일을 잘못된 디렉토리에 계속 넣나요? 올바른 테스트 파일 배치 규칙을 문서화하세요.
*   Claude Code가 DB 스키마(schema) 파일을 편집했지만 기본 모듈(module)을 재빌드하는 것을 잊었나요? 종속성 관계와 필요한 빌드 단계를 명확히 작성하세요.

Claude Code를 수동으로 수정해야 할 때마다, 그것은 실제로 프로젝트 지식 중 일부가 아직 `CLAUDE.md`에 작성되지 않았다는 강력한 신호입니다. 동일한 수정이 팀 전체에서 이미 두세 번 발생했다면, 그 규칙은 공유되는 장기 기억으로 전환되어야 한다고 말하기에 충분합니다.

### `/reflection`을 사용하여 정기적인 회고 진행하기

이전 접근 방식은 여전히 사람들이 사용 중에 문제를 알아차리고 규칙을 업데이트하는 것을 기억하는 것에 달려 있습니다. `/reflection`의 가치는 이를 반복 가능한 마무리 단계로 전환한다는 점입니다. 각 세션이 끝날 때, Claude Code에게 해당 협업 라운드에서 `CLAUDE.md`에 추가할 가치가 있는 내용을 요약해달라고 요청한 다음, 그 내용을 더 안정적인 프로젝트 규칙으로 전환할 수 있습니다.

엄밀히 말하면 `/reflection`은 어떤 신비로운 내장 기능이 아닙니다. 본질적으로는 반복적으로 호출할 수 있는 명령어로 패키징된 프롬프트(prompt)일 뿐입니다. 원래 내용을 보려면 이 [reflection gist](https://gist.github.com/tonychan-anthropic/39df7811f0a2027b165147854eb16e78)를 직접 확인할 수 있습니다.

해당 프롬프트의 핵심 아이디어는 Claude Code가 방금 끝난 세션을 되돌아보고, 채팅별 컨텍스트(chat-specific context)에서 `CLAUDE.md` 내의 지속적인 규칙으로 전환될 만큼 안정화된 교훈이 무엇인지 판단하게 하는 것입니다. 사용자가 이를 확인하면, Claude Code는 `CLAUDE.md`를 업데이트하고 다른 파일은 수정하지 않습니다.

설정도 간단합니다. Anthropic의 공식 문서에 따르면 `~/.claude/commands/` 아래에 배치된 마크다운 파일은 자동으로 사용자 수준 명령어가 되며, 파일 이름이 명령어 이름이 됩니다. 따라서 해당 프롬프트를 다음과 같이 저장할 수 있습니다.

`~/.claude/commands/reflection.md`

그런 다음 Claude Code 내에서 직접 실행합니다.

`/reflection`

Claude Code는 해당 프롬프트에 따라 세션을 검토하고 확인된 규칙으로 `CLAUDE.md`를 업데이트합니다. 이는 `CLAUDE.md`의 진화를 사람들이 가끔 기억해서 하는 일에서, 매 세션 후에 발생할 수 있는 안정적인 단계로 전환시킵니다.

### 인사이트 리포트(Insights Reports)를 사용하여 최적화하기

이전 두 접근 방식이 개별 협업 세션에 더 밀접하게 연결되어 있는 반면, 인사이트(Insights)를 사용하면 더 긴 사용 기간에 걸쳐 확대하여 볼 수 있습니다. 이를 통해 어떤 문제가 계속 재발하는지, 어떤 습관이 안정화되었는지, 그리고 어떤 정보가 `CLAUDE.md`에 공식적으로 추가할 가치가 있는지 더 쉽게 파악할 수 있습니다.

인사이트는 Claude Code v2.1.x에서 도입된 분석 명령어입니다. 주로 Claude Code 사용 기록을 분석하고 보고서를 생성하는 데 사용됩니다. 해당 보고서의 일부는 `CLAUDE.md`를 개선하기 위한 직접적인 제안을 포함하며, 이는 지속적인 진화를 위한 특히 중요한 경로입니다.

사용법은 간단합니다. Claude Code에 들어간 후 `/insights` 명령어를 실행하십시오. 완료되면 생성된 HTML 보고서를 `~/.claude/usage-data` 디렉토리에서 찾을 수 있으며, 여기에서 시간 경과에 따른 사용 패턴과 관련 권장 사항을 검토할 수 있습니다.

예를 들어, 특정 테스트 명령어 세트를 사용하도록 Claude Code에게 자주 상기시키거나, 일부 생성된 디렉토리를 건드리지 말라고 자주 지시하거나, 동일한 유형의 작업에 대해 동일한 배경 정보를 자주 반복한다는 것을 알 수 있습니다. 이러한 것들이 여러 세션에 걸쳐 일관되게 나타났다면, 채팅 기록에 묻혀 있을 것이 아니라 `CLAUDE.md`로 승격되어야 합니다.

이러한 관점에서 인사이트 리포트는 `CLAUDE.md`가 계속 진화하도록 돕는 도구로 이해하는 것이 가장 좋습니다. 제공하는 수정 제안은 어떤 콘텐츠가 공식적으로 기록될 만큼 충분히 성숙했는지 결정하는 데 특히 유용합니다.

(이미지)

### 더 효과적인 진화 주기

위의 모든 방법을 연결하면, 자연스러운 진화 주기는 보통 다음과 같습니다.

*   `/init`으로 시작하여 빠르게 사용할 수 있는 초안을 얻습니다.
*   일상적인 협업에서 문제를 기록하고 반복적인 수정을 `CLAUDE.md`에 다시 작성합니다.
*   각 세션이 끝날 때 `/reflection`을 실행하여 세션별 교훈을 프로젝트 규칙으로 전환합니다.
*   정기적으로 인사이트 리포트를 검토하여 더 긴 시간 범위에 걸쳐 반복되는 패턴을 식별합니다.
*   오래되거나, 중복되거나, 가치가 낮은 규칙을 제거하여 파일을 계속 정리하고 다듬습니다.

이러한 방식으로 진화하는 `CLAUDE.md`는 처음부터 모든 것을 포괄하려고 시도한 버전보다 훨씬 더 유용합니다. 추상적으로 설계된 것이 아니라, 실제 협업을 통해 단계별로 다듬어지기 때문입니다.

## CLAUDE.md를 위한 모범 사례

많은 사람들은 `CLAUDE.md`가 로드되지 않았기 때문에 효과가 없다고 생각하지만, 실제 사용에서는 Claude Code가 파일을 보았음에도 불구하고 그 내용 중 많은 부분이 현재 작업과 별로 관련이 없다고 판단하여 의미 있는 방식으로 의존하지 않는 경우가 더 흔합니다. 다시 말해, 진짜 문제는 로딩 메커니즘뿐만 아니라, 작성한 내용이 장기적으로 유지할 만큼 충분히 관련성이 있고, 구체적이며, 가치가 있는가입니다. 그래서 이 섹션에서는 더 많은 규칙을 추가하는 방법에 초점을 맞추기보다, 무엇을 유지할 가치가 있는지 판단하는 방법에 더 집중하고자 합니다.

### 간결하게 유지하고 고가치 정보를 우선시하라

`CLAUDE.md`는 본질적으로 Claude Code를 위한 추가 컨텍스트(context)입니다. 즉, 파일이 길어질수록 더 많은 컨텍스트를 소비하고 실제 작업을 위한 공간이 줄어듭니다. 더 중요한 것은, 파일이 너무 복잡해지면 모델이 이를 관련성이 낮은 배경 자료로 취급할 가능성이 높아진다는 것입니다.

그렇기 때문에 메인 파일에 유지할 가치가 가장 높은 내용은 일반적으로 짧은 목록입니다. 고빈도 명령어, 핵심 프로젝트 제약 조건, 실수를 유발하기 쉬운 과거의 교훈, 그리고 누군가 저장소에 진입할 때마다 중요하게 여겨질 배경 정보입니다. 드문 상황에서만 적용되는 내용이나, 듣기에는 좋지만 명확한 실행 경로가 없는 진술은, 파일이 포괄적으로 보이도록 억지로 채워 넣는 것보다 제외하는 것이 좋습니다.

### 규칙이 실제로 실행 가능하도록 구체적으로 작성하라

많은 `CLAUDE.md` 파일은 겉으로는 완전해 보이지만, 실제 작업에 구체적인 지침을 제공하지 않고 단순히 올바른 방향만 제시하기 때문에 실제로는 별 도움이 되지 않습니다. "코드 품질에 주의하라", "프로젝트 규칙을 따르라", "변경 전에 컨텍스트를 이해하라"와 같은 문구는 괜찮게 들리지만, Claude Code가 특정 상황에서 더 나은 결정을 내리는 데 거의 도움이 되지 않습니다.

실제로 도움이 되는 것은 직접 따를 수 있는 수준으로 작성하는 것입니다. 예를 들어, 테스트를 위해 어떤 명령어를 사용해야 하는지, 어떤 디렉토리를 편집해서는 안 되는지, 커밋(commit)하기 전에 린트(lint)를 실행해야 하는지, API 레이어(API layer)가 어느 디렉토리에 속하는지, 어떤 상황에서 테스트를 추가해야 하는지 등입니다. 더 구체적일수록 Claude Code가 일상 업무 중에 여러분이 진정으로 원하는 것을 더 쉽게 이해할 수 있습니다.

### 단순히 규칙 목록이 아닌 의도를 전달하라

좋은 `CLAUDE.md`는 단순히 규칙을 나열하는 것을 넘어섭니다. 더 중요한 것은 Claude Code가 규칙 뒤에 있는 의도(intent)를 이해하도록 돕는 것입니다. 많은 실제 작업은 규칙 체크리스트(rule checklist)의 경계에 깔끔하게 들어맞지 않기 때문입니다. 팀이 특정 방식을 설계한 이유와 제약 조건이 존재하는 이유를 이해해야만, 새로운 상황에서 여러분이 기대하는 것에 더 가까운 결정을 내릴 수 있습니다.

예를 들어, `src/generated/` 아래의 파일을 수정하지 말라고만 작성하면 여전히 규칙입니다. 그러나 한 발 더 나아가 해당 파일이 OpenAPI 스키마(schema)에서 자동으로 생성되며, 실제 진실의 원천(source of truth)은 생성된 결과물이 아니라 스키마와 생성 워크플로에 있다고 설명한다면, Claude Code는 그 제한이 왜 존재하는지 이해하고 다음에 관련 작업이 발생할 때 올바른 것을 수정할 가능성이 훨씬 높아집니다.

### 점진적 노출(Progressive Disclosure)을 사용하고 메인 파일을 체계적으로 관리하라

모든 정보가 루트 `CLAUDE.md`에 속하는 것은 아닙니다. 더 나은 접근 방식은 일반적으로 가장 일반적이고 안정적이며 작업 전반에 걸친 정보를 메인 파일에 먼저 유지하고, 내용이 많아지면 `@imports`를 사용하거나 다른 디렉토리 수준에서 여러 `CLAUDE.md` 파일로 분할하는 것입니다. 전자는 구조와 유지보수성(maintainability)을 향상시키고, 후자는 특정 로컬 규칙이 Claude Code가 관련 디렉토리에 실제로 진입할 때만 적용되도록 합니다.

이점은 단순히 더 깔끔하게 보이는 파일이 아닙니다. 더 중요한 것은 각 세션 시작 시 컨텍스트에 덤프되는 관련 없는 정보의 양을 줄인다는 것입니다. 메인 파일을 체계적으로 관리하고, 필요할 때만 세부 정보가 확장되도록 하십시오. 이것이 `CLAUDE.md`가 끊임없이 늘어나는 프로젝트 잡다한 정보 더미가 아니라, 사려 깊은 협업 가이드처럼 느껴지기 시작하는 방법입니다.

강조할 한 가지 더 있습니다. `CLAUDE.md`에는 절대로 API 키(API keys), 비밀번호(passwords), 토큰(tokens) 또는 기타 보안 정보(secrets)를 넣지 마십시오. 이 파일은 종종 버전 관리에 포함되기 때문입니다. 거기에 자격 증명을 작성하는 순간, 저장소에 접근할 수 있는 모든 사람에게 노출하는 것과 같습니다.

## CLAUDE.md와 유사한 파일들

Claude Code와 유사한 AI 코딩 도구들 대부분은 이제 자체 지침 파일(instruction file)을 가지고 있습니다. 하지만 이 시점에서 정말 비교할 가치가 있는 것은 파일 이름만이 아닙니다. 더 중요한 질문은 각 도구가 해당 파일을 어떻게 사용하며, 어떤 기능을 파일 자체 외부에 두는지입니다.

### 이름뿐 아니라 메커니즘도 다르다

지침 시스템 설계 관점에서 Claude Code와 Gemini CLI(제미니 CLI)는 사실 서로 더 가깝습니다. 둘 다 이 파일을 모델에 주입되는 영구 메모리(persistent memory) 또는 컨텍스트(context)로 취급합니다. 차이점은 Claude Code가 `CLAUDE.md`에 대해 더 명확한 계층 구조(hierarchy)를 가지고 있어, 프로젝트 전체에 걸쳐 여러 컨텍스트 파일을 결합하기 더 쉽다는 것입니다. Gemini CLI도 프로젝트에서 여러 `GEMINI.md` 파일을 읽을 수 있지만, 더 간단한 방식으로 연결하는 경우가 많으며, 기본 파일 이름을 `AGENTS.md`로 변경할 수도 있어 생태계 호환성(ecosystem compatibility) 측면에서 더 개방적입니다.

Codex(코덱스), OpenCode(오픈코드), Droid(드로이드)는 동일한 진화의 다른 갈래처럼 보입니다. 세 가지 모두 `AGENTS.md`로 수렴하고 있지만, 공유하는 것은 공통 파일 이름만이 아닙니다. 또한 이를 통합된 진입점(unified entry point)으로 취급하는 경향이 있으며, 더 고급 기능을 파일 자체 외부로 옮깁니다. Codex의 경우, `AGENTS.md`를 재사용 가능한 교차 도구 지침 형식(reusable cross-tool instruction format)으로 전환하는 데 중점을 둡니다. OpenCode와 Droid의 경우, `AGENTS.md`를 중심으로 더 풍부한 협업 기능을 확장하는 데 중점을 둡니다. 다시 말해, 이 경로에서는 `AGENTS.md`가 시스템 전체라기보다 시작점에 가깝습니다.

### 형성되기 시작하는 트렌드

정말 주목할 가치가 있는 것은 더 이상 파일 자체가 아니라, 이러한 도구 뒤에 있는 지침 시스템들이 점차 수렴하고 있다는 점입니다. `CLAUDE.md`는 특히 계층형 로딩(layered loading)과 프로젝트 컨텍스트 관리(project context management)에서 여전히 분명한 강점을 가진 매우 인지 가능한 메모리 디자인(memory design)을 나타냅니다. 동시에 `AGENTS.md`는 점점 더 강력한 교차 도구 형식으로 자리 잡고 있습니다. 2025년 말까지 OpenAI(오픈AI)는 이미 `AGENTS.md`를 Agentic AI Foundation(에이전틱 AI 재단)에 기여했으며, 이를 간단하고 개방적이며 상호 운용 가능한 표준으로 명시적으로 설명했습니다. Gemini CLI, OpenCode, Droid 등도 다양한 정도로 해당 규칙을 따르고 있습니다.

이는 미래에 다른 AI 코딩 에이전트(AI coding agent) 간 전환 비용이 계속 저렴해질 것이라는 의미입니다. 그러나 현재 단계에서는 파일 이름은 수렴하고 있을지라도, 기능 모델(capability model)은 아직 통일되지 않았습니다. 일부 도구는 계층형 메모리를 강조하고, 일부는 에이전트 오케스트레이션(agent orchestration)을 강조하며, 일부는 스킬(skills), 서브 에이전트(subagents), 플러그인(plugins), 조직 수준 구성(organization-level configuration)을 지침 파일 외부로 옮깁니다. 따라서 `CLAUDE.md`의 가치는 파일 자체뿐만 아니라, 배경에서 프로젝트 컨텍스트를 조직하는 전체 방식에 있습니다.

## 요약

이 글은 `CLAUDE.md`가 무엇인지부터 시작하여, 로드 방식, 작성 방법, 진화 방식, 모범 사례, 그리고 유사 도구의 지침 시스템과의 관계를 살펴보았습니다. 정말 중요한 것은 그 뒤에 숨겨진 더 넓은 패턴을 이해하는 것입니다. 즉, 프로젝트 컨텍스트를 조직하고, 협업 제약 조건을 보존하며, AI 에이전트가 개발에 더 신뢰성 있게 참여하도록 하는 방법입니다.

실제 사용에서 `CLAUDE.md`는 처음부터 모든 것을 포괄하는 문서를 작성하려고 시도하는 대신, 사용할 수 있는 버전에서 시작하여 실제 협업을 통해 계속 추가하고, 정리하고, 다듬을 때 가장 잘 작동합니다. 미래에 파일 이름과 구현 방식이 도구들 간에 어떻게 진화하든, 기본적인 아이디어는 여전히 유효할 것입니다. 프로젝트 경험을 장기적인 규칙으로 전환하여 AI가 컨텍스트를 더 일관되게 이해하도록 하는 것입니다. 이것이 `CLAUDE.md`를 진지하게 이해할 가치가 있는 이유입니다.
