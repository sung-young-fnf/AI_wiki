# 번역 기사

- **원본 URL:** https://medium.com/@me_82386/i-never-used-claude-code-agents-now-i-cant-work-without-them-9cbb14bb3219
- **번역 일시:** 2026-04-09 16:32:19

---

**몇 달간 무시했던 Claude Code의 기능이 가장 중요했습니다**

저는 수개월 동안 Claude Code를 사용해 왔습니다. 나름 잘 활용하고 있다고 생각했죠. 작업을 붙여넣으면 파일을 편집하고, 테스트를 실행하고, 결과물을 내놓는 것을 보면서 생산적이라고 느꼈습니다.

하지만 저는 잘못 사용하고 있었습니다.

Claude를 마치 한 명이 다섯 가지 일을 하는 것처럼 대하고 있었던 겁니다. 계획 수립, 코딩, 검토, 테스트, QA(품질 보증) 이 모든 것을 한 세션 내에서 처리하도록 말이죠. 실제 사람과 마찬가지로 하나의 두뇌가 너무 많은 업무를 담당하면 실수가 생기기 마련입니다. 컨텍스트(context)가 뒤섞입니다. 검토 중에 확인하라고 요청했던 내용은 구현에 몰두하느라 잊어버리고, 테스트 커버리지(coverage)는 실제로 중요한 것이 아닌 그저 "완료된 것 같다"는 수준으로 작성되는 식이죠.

그래서 저는 각각의 전문 에이전트(agent)가 순차적으로 실행되며, 각자의 터미널 세션(terminal session)에서 다음 에이전트로 컨텍스트를 자동으로 전달하는 도구를 만들었습니다. 이제는 예전으로 돌아갈 수 없습니다.

### 모든 것을 하나의 에이전트가 처리할 때의 문제점

제가 진행하는 모든 프로젝트는 대체로 동일한 루프(loop)를 따릅니다:

*   무엇을 변경해야 할지 파악
*   코드 작성
*   코드 품질 확인
*   테스트 작성
*   올바르게 보이는지 확인

Claude가 이 다섯 단계를 한 번에 처리할 때는 속도에 최적화됩니다. 이미 무엇을 만들어야 할지 "알고 있다"고 생각해서 계획을 건너뛰고, 코드를 직접 작성했으니 당연히 괜찮을 것이라며 검토를 건너뜁니다. 테스트는 실행하지만, 테스트에 누락된 부분이 무엇인지 깊이 생각하지 않고 존재하는 것만 확인합니다.

저는 각 단계가 해당 단계에만 집중하는 전문가에 의해 처리되기를 원했습니다. 그리고 각 전문가가 이전 단계의 모든 컨텍스트를 받아 자동으로 전달되기를 바랐습니다. 그래서 DevPit을 만들었습니다.

### 한 번의 명령으로 전체 파이프라인 실행

`dp pipeline "Add a health check endpoint"`

이게 전부입니다. DevPit이 전체 워크플로(workflow)를 넘겨받아 실행합니다:

*   **Architect (아키텍트)**는 구현을 계획합니다. 읽기 전용 도구를 사용하며, 아무것도 편집할 수 없습니다. 이 에이전트의 유일한 임무는 무엇을 변경해야 할지, 어떤 파일이 영향을 받는지, 무엇이 잘못될 수 있는지, 어떤 테스트를 업데이트해야 할지 고민하는 것입니다.
*   **Coder (코더)**는 사용자의 실제 컨벤션(convention)에 따라 구현합니다. 아키텍트의 전체 계획을 컨텍스트로 받습니다. 주변 파일을 읽어 스타일을 맞추고, 린터(linter)를 실행합니다.
*   **Tester (테스터)**는 사용자의 실제 테스트 프레임워크(framework)를 사용하여 테스트를 작성하고 실행합니다. 코더의 결과물을 컨텍스트로 받아 무엇이 변경되었는지 정확히 파악합니다. 테스트가 실패하면, DevPit은 코더에게 자동으로 최대 3번까지 다시 루프를 돌려 통과할 때까지 반복합니다.
*   **Reviewer (리뷰어)**는 사용자가 구성한 우선순위(보안, 성능, 정확성 등)를 염두에 두고 최종 차이(diff)를 검토합니다. 모든 피드백에는 구체적인 수정 사항이 포함됩니다.
*   **Design QA (디자인 품질 보증)**는 시각적인 품질 검증 단계입니다. 영향을 받는 페이지를 스크린샷하고, 레이아웃, 반응형 동작, 접근성을 감사합니다. 뭔가 잘못된 부분이 있으면, 코더에게 돌아가 수정을 요청합니다.

각 에이전트는 독립적인 `tmux` 세션에서 실행됩니다. 각 에이전트는 깨끗한 컨텍스트와 필요한 정보만 가지고 새롭게 시작합니다. 작업이 완료되면 DevPit은 해당 에이전트의 결과물을 캡처하고 세션을 종료한 후, 그 결과물을 파이프라인의 다음 에이전트로 전달합니다.

### 모든 에이전트가 tmux에서 실행됩니다. 실시간으로 지켜볼 수 있습니다.

이 부분이 저를 사로잡은 핵심입니다. DevPit은 에이전트를 숨겨진 서브프로세스(subprocess)에서 실행하지 않습니다. 모든 에이전트는 자신만의 `tmux` 세션을 가지며, 실시간으로 어떤 세션에도 연결하여 지켜볼 수 있습니다.

*   `dp pipeline peek coder`       # 코더가 지금 무엇을 하는지 확인
*   `dp pipeline peek tester -n 200` # 테스터의 마지막 200줄 출력 확인
*   `dp pipeline status`            # 어떤 에이전트가 작업 중/대기 중인지 확인

또는 직접 연결할 수도 있습니다:

*   `tmux attach -t pipeline-coder`

실제 터미널에서 Claude Code(또는 Gemini, Codex)가 작업하는 것을 볼 수 있습니다. 에이전트가 읽는 모든 파일, 만드는 모든 편집, 실행하는 모든 명령을 볼 수 있습니다. 뭔가 잘못된 부분이 있다면, 로그로 나중에 확인하는 것이 아니라 실시간으로 발생하고 있는 것을 직접 볼 수 있습니다.

TUI (텍스트 사용자 인터페이스) 대시보드는 이를 더욱 쉽게 만듭니다. `dp` 명령을 인자 없이 실행하면 모든 파이프라인 실행에 대한 대화형 보기를 얻을 수 있습니다. 실행 중, 통과, 실패 상태를 확인하고, 각 실행을 드릴 다운하여 단계별 출력을 볼 수 있습니다.

### 워크플로는 YAML 파일입니다

기본 파이프라인(아키텍트, 코더, 테스터, 리뷰어, 디자인 QA)은 기본적으로 훌륭하게 작동합니다. 하지만 모든 프로젝트는 다릅니다. DevPit은 YAML 형식으로 사용자 정의 워크플로를 정의할 수 있게 해줍니다:

```yaml
name: optimize
description: Iterative benchmark-and-improve loop
steps:
  - name: baseline
    agent: benchmarker
  - name: analyst
    context: [baseline]
  - name: improver
    agent: coder
    context: [analyst]
    directive: "Implement the improvements proposed by the analyst"
  - name: verifier
    agent: benchmarker
    context: [improver]
    loop:
      goto: analyst
      max: 3
      pass: "PIPELINE_RESULT:PASS"
      fail: "PIPELINE_RESULT:FAIL"
```

`dp pipeline "Optimize the search endpoint" --workflow optimize`

이 워크플로는 코드를 벤치마킹하고, 결과를 분석하고, 개선 사항을 적용하고, 다시 벤치마킹하여 성능이 통과하거나 3번의 시도 횟수에 도달할 때까지 반복합니다. 각 단계는 자신이 의존하는 단계에서만 컨텍스트를 얻습니다.

보안 감사, 마이그레이션 파이프라인, 문서 생성기, 리팩토링 루프 등 무엇이든 위한 워크플로를 구축할 수 있습니다. 각 단계는 그저 컨텍스트가 흐르는 에이전트에 불과합니다.

워크플로를 대화식으로 생성할 수도 있습니다. `dp create` 명령은 Claude를 실행하여 프로젝트에 대해 인터뷰하고 스택(stack)에 맞춤화된 에이전트 및 워크플로 YAML을 생성합니다. 또는 대시보드에서 `e` 키를 눌러 TUI에서 편집할 수 있습니다. 터미널 내 편집기에서 단계를 재정렬하고, 필드를 편집하고, 루프를 추가하고, 저장할 수 있습니다.

### 컨텍스트 전달이 핵심 통찰력입니다

하나의 Claude 세션으로 모든 것을 처리할 때는 컨텍스트가 암시적(implicit)입니다. "계획"은 Claude가 코딩을 시작할 때 생각했던 모든 것이고, "검토"는 코드를 작성한 동일한 두뇌가 스스로를 평가하는 것입니다.

DevPit은 컨텍스트를 명시적(explicit)으로 만듭니다. 각 단계는 필요한 이전 단계를 선언합니다:

```yaml
- name: tester
  context: [coder]
```

테스터는 코더의 모든 출력, 즉 코더가 수행한 모든 작업, 변경한 모든 파일, 그 추론을 받습니다. 하지만 아키텍트의 계획이나 이전 실행에서 리뷰어의 피드백은 받지 않습니다. 테스터는 자신의 작업을 수행하는 데 필요한 것만 정확히 얻고, 그 이상은 받지 않습니다.

이것이 바로 격리(isolation)가 실제로 의미하는 바입니다. 단순히 별도의 세션이 아니라, 범위가 지정된 컨텍스트(scoped context)입니다. 아키텍트는 구현 세부 사항에 영향을 받지 않고 생각하고, 코더는 검토 문제에 방해받지 않고 구현합니다. 테스터는 코더가 "아마 괜찮을 것"이라고 생각했던 것을 알지 못한 채 테스트합니다.

Claude가 이 모든 것을 하나의 세션에서 처리하면 이러한 경계가 사라집니다. "계획"은 암시적입니다. "검토"는 코드를 작성한 동일한 두뇌가 스스로를 칭찬하는 것이 됩니다. 테스트는 코드가 이미 올바르다고 결정한 동일한 컨텍스트에 의해 작성됩니다.

격리는 책임감(accountability)을 만듭니다.

### 에이전트가 자신이 만드는 것을 볼 수 있습니다

Claude Code는 시각적 인터페이스를 편집하는 텍스트 기반 도구입니다. 코딩이 "눈먼" 상태에서 이루어집니다. CSS를 작성하고 커밋(commit)한 다음, 최선을 바라죠.

DevPit은 전용 `design-qa` 단계로 이를 해결합니다. 코더가 완료되면 `design-qa`가 작업을 이어받습니다. Playwright를 사용하여 여러 뷰포트(viewport)에서 영향을 받는 페이지를 스크린샷하고, 스크린샷을 읽어(Claude는 멀티모달(multimodal)이므로 이미지를 볼 수 있음) 레이아웃, 반응형 동작, 타이포그래피, 접근성 등 모든 것을 감사합니다. 뭔가 잘못된 부분이 있으면 정확히 무엇이 문제인지 보고하고, DevPit은 코더에게 돌아가 수정을 요청합니다.

모바일 프로젝트의 경우, Maestro와 시뮬레이터(simulator) 스크린샷으로 동일한 방식으로 작동합니다. iOS 시뮬레이터 또는 안드로이드 에뮬레이터에서 말이죠.

이 기능은 결과물에 대한 저의 신뢰도를 크게 높였습니다. 이전에는 변경할 때마다 브라우저를 열어 직접 눈으로 확인해야 했습니다. 이제는 만들어진 것을 보고 시각적인 문제를 찾는 것이 유일한 임무인 전담 에이전트가 있습니다.

### Claude만이 아닙니다

DevPit은 여러 AI 런타임(runtime)과 함께 작동합니다. `--agent` 플래그로 기본 CLI(명령줄 인터페이스)를 바꿀 수 있습니다:

*   `dp pipeline "task" --agent claude`    # Claude Code (기본)
*   `dp pipeline "task" --agent gemini`    # Gemini CLI
*   `dp pipeline "task" --agent codex`     # OpenAI Codex
*   `dp pipeline "task" --agent copilot`   # GitHub Copilot

각 런타임은 자체적인 준비 감지, 프롬프트 전달, 시작 대화 처리 기능을 가지고 있습니다. DevPit은 어떤 AI가 작업을 수행하든 상관없이 `tmux` 세션 관리, 유휴 감지, 컨텍스트 전달을 담당합니다.

이는 에이전트를 혼합하여 사용할 수 있음을 의미합니다. 추론이 가장 중요한 아키텍트 및 리뷰어에는 Claude를 사용하고, 코더에는 Gemini를 사용하는 등 워크플로와 예산에 맞는 모든 조합이 가능합니다.

### 컨텍스트를 유지하는 후속 작업

파이프라인이 완료되면 모든 에이전트 세션이 캡처됩니다. DevPit은 이러한 세션을 재개하여 후속 작업을 처리할 수 있습니다:

*   `dp pipeline follow "Make the button blue instead of green"`

후속 작업은 동일한 파이프라인을 통해 실행되지만, 각 에이전트의 대화를 재개하므로 첫 번째 실행의 모든 내용을 기억합니다. 다시 설명할 필요도, 컨텍스트를 잃을 필요도 없습니다. 코더는 자신이 무엇을 만들었는지 알고, 테스터는 무엇을 테스트했는지 알며, 후속 작업은 그 위에 구축됩니다.

후속 작업은 자동으로 대기열에 추가됩니다. 파이프라인이 아직 실행 중이면 작업은 대기했다가 유휴 상태일 때 실행됩니다.

### 새롭게 나타난 워크플로

몇 주 동안 이것을 사용한 후, 자연스러운 흐름이 생겼습니다:

*   저는 무엇을 만들고 싶은지 설명합니다.
*   DevPit의 아키텍트가 계획을 수립합니다: 영향을 받는 파일, 접근 방식, 위험 요소, 테스트 요구 사항.
*   코더는 계획에 따라 구현합니다.
*   테스터는 테스트를 작성하고 실행하며, 뭔가 실패하면 코더와 함께 루프를 돕니다.
*   리뷰어는 최종 차이(diff)를 확인합니다.
*   `design-qa`는 (UI 작업의 경우) 스크린샷을 찍고 감사하며, 뭔가 문제가 있으면 코더와 함께 루프를 돕니다.
*   저는 결과물을 검토하고, `dp pipeline follow` 명령으로 조정을 요청합니다.

이 시스템에서 제가 가장 좋아하는 점은 모든 단계가 가시적(visible)이라는 것입니다. 어떤 에이전트든 들여다볼 수 있고, 무엇을 하는지 정확히 볼 수 있으며, 왜 그런 선택을 했는지 이해할 수 있습니다. 뭔가 잘못되면 어떤 에이전트가 실패했고 어떤 컨텍스트를 가지고 있었는지 알 수 있습니다. 대시보드는 모든 실행의 전체 기록을 보여줍니다.

이것은 수백 개의 메시지를 거슬러 올라가 언제 문제가 발생했는지 파악해야 하는 단일 Claude 세션과는 근본적으로 다릅니다.
