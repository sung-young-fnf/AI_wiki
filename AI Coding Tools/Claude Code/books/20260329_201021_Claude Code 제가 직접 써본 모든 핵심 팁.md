저는 Claude Code를 오랫동안 사용해오면서 강한 의견을 갖게 되었습니다. 몇몇은 제가 직접 어렵게 터득했고, 몇몇은 다른 사람들에게서 배웠습니다. 이 모든 것이 여기에 담겨 있습니다.

## 1. 아이디어가 떠오르는 순간 — `/ce:plan`

제 업무 방식에서 가장 큰 변화는 아이디어를 "곰곰이 생각하는" 대신, 즉시 `/ce:plan`을 실행하기 시작했다는 것입니다.

새로운 제품 아이디어? `/ce:plan`. GitHub에 올라온 버그를 발견했다면? URL을 복사하고 붙여넣은 다음 `/ce:plan`. 터미널(terminal)에 오류가 발생했다면? Cmd+Shift+4로 스크린샷(screenshot)을 찍고, Ctrl+V로 Claude Code에 붙여넣은 다음 다음과 같이 입력합니다.

`/ce:plan fix this`

Claude Code는 이미지를 처리합니다. 스크린샷, 오류 메시지, 디자인 목업(design mockups), Slack 대화 등 모든 것을 읽고 계획을 수립합니다.

실제로 이렇게 하면 어떤 일이 일어나는지 설명하겠습니다. `/ce:plan`은 여러 연구 에이전트(research agents)를 병렬로 실행합니다. 하나는 여러분의 코드베이스(codebase) — 파일, 규칙(conventions), 패턴(patterns) — 를 읽습니다. 다른 하나는 이전에 해결했던 버그를 찾기 위해 `docs/solutions/` 폴더를 뒤집니다. 주제에 따라 더 많은 에이전트가 외부 문서와 모범 사례(best practices)를 찾아봅니다. 이 모든 과정이 동시에 진행됩니다.

그 결과 `plan.md` 파일이 나옵니다. 이 파일에는 문제점, 해결책, 수정할 파일, 체크박스가 있는 승인 기준(acceptance criteria), 그리고 일반적인 상용구(boilerplate)가 아닌 여러분의 코드에서 추출된 패턴이 포함됩니다.

그런 다음 `/ce:work`는 그 계획을 바탕으로 작업을 수행합니다. 한 번에 하나의 작업을 처리하고, 테스트를 실행하며, 체크박스를 확인합니다. 세션(session) 도중 맥락을 잃더라도, 새로 시작하여 계획을 지정하고 작업을 계속할 수 있습니다. 계획은 모든 상황에서 살아남습니다.

예전 방식: 코딩 80%, 계획 20%. 이 방식: 정반대입니다. 모든 사고 과정이 계획에 집중되고, 실행은 거의 기계적으로 이루어집니다.

이를 가능하게 하는 플러그인(plugin)은 Compound Engineering입니다:

```
/plugin marketplace add EveryInc/compound-engineering-plugin
```

저는 기여자(#3 on GitHub, 21 커밋(commits))가 되었습니다. 지난 30일 동안 70개의 계획 파일과 263개의 커밋을 기록했습니다. 저의 현재 규칙은 다음과 같습니다. 문자 그대로 한 줄 변경이 아닌 이상, 항상 `plan.md` 파일이 먼저 있어야 합니다. 항상요.

## 2. Claude Code와 대화하기 시작하세요

저는 음성 입력을 정말 싫어했습니다. Apple의 받아쓰기 기능은 제 노트북을 던져버리고 싶게 만들었죠. 하지만 음성을 LLM(거대 언어 모델)으로 변환하는 것은 완전히 다릅니다.

필사(transcription)가 완벽할 필요는 없습니다. Claude Code는 마이크가 잘못 인식한 부분을 채워줍니다. 중얼거려도 되고, 문장을 다시 시작해도 되며, 말을 흐려도 괜찮습니다. Claude Code가 알아서 처리합니다.

Monologue(@usemonologue)는 현재 활성화된 앱으로 여러분의 음성을 직접 전달합니다. 여러분이 말하면, 그것이 Claude Code에 입력됩니다. WhisperFlow도 훌륭한 대안입니다. 둘 중 하나를 선택하세요. 저는 제 책상에 구즈넥 마이크(gooseneck mic)를 구입했습니다.

저는 지금 아이들을 학교에 내려주는 차 안에서 이 단락을 받아쓰고 있습니다. 타이핑이 아니라 말로요.

## 3. 동시에 4~6개의 세션 실행하기

이것이 저의 실제 하루 모습입니다.

4개에서 6개의 Ghostty 창을 엽니다. 각 창은 독립적인 Claude Code 세션(session)입니다. 하나는 계획을 작성하고 있고, 다른 하나는 또 다른 계획을 바탕으로 작업을 빌드(build)하고 있습니다. 또 다른 하나는 연구를 진행 중이며, 다른 무언가를 테스트하다가 발견한 버그를 수정하는 세션도 있습니다.

첫 번째 창에서 `/ce:plan`이 연구 에이전트를 실행하는 동안, 저는 두 번째 창으로 전환하여 이미 완료된 계획에 대해 `/ce:work`를 실행합니다. 그것이 빌드되는 동안, 세 번째 창에는 새로운 버그가 붙여넣어집니다. 제가 다시 첫 번째 창으로 돌아올 때쯤이면, 계획이 준비되어 있습니다.

이것이 다음 섹션이 필수적인 이유이기도 합니다. 만약 모든 세션이 30초마다 "이것을 허용하시겠습니까?"라고 묻는다면, 여러분은 컨텍스트 전환(context-switch)을 할 수 없습니다. 각 세션이 독립적으로 실행되어야 합니다.

여담: 6개의 Claude 세션을 병렬로 실행하면 맥북 배터리가 약 한 시간 만에 소모됩니다. 그래서 저는 방금 새로운 MacBook Pro를 주문했습니다.

## 4. 실제로 중요한 세 가지 설정 변경

Claude Code의 기본 설정은 거의 모든 것에 대해 허가를 요청합니다. 이 세 가지 변경 사항으로 이를 해결할 수 있습니다.

권한 프롬프트(permission prompts)를 끄세요. `~/.claude/settings.json` 파일을 편집합니다.

```json
{
  "permissions": {
    "allow": [
      "WebSearch", "WebFetch", "Bash", "Read", "Write",
      "Edit", "Glob", "Grep", "Task", "TodoWrite"
    ],
    "deny": [],
    "defaultMode": "bypassPermissions"
  },
  "skipDangerousModePermissionPrompt": true
}
```

`skipDangerousModePermissionPrompt: true`가 핵심 줄입니다. 이 설정이 없으면 Claude는 모든 세션에서 이를 확인하라고 요청합니다. Shift+Tab을 눌러 전환할 수도 있습니다. 제가 친구를 위해 이 설정을 할 때, AI는 말 그대로 친구를 설득하여 포기하게 만들려고 했습니다. 그냥 단호하게 진행하세요. 여러분의 컴퓨터입니다.

Claude가 작업을 마쳤을 때 소리를 추가합니다. 동일한 파일에 추가합니다.

```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "afplay /System/Library/Sounds/Blow.aiff"
          }
        ]
      }
    ]
  }
}
```

자리를 비웠다가 소리가 들리면 돌아오세요. 4개에서 6개의 세션이 실행 중일 때, 어떤 세션이 방금 완료되었는지 알아야 합니다.

Zed에서 자동 저장(autosave)을 설정합니다. Cmd+,를 눌러 설정을 엽니다.

```json
{
  "autosave": {
    "after_delay": {
      "milliseconds": 500
    }
  }
}
```

Zed는 500밀리초(milliseconds)마다 저장합니다. Claude Code는 파일 시스템(filesystem)을 감시합니다. Claude가 파일을 편집하면, Zed에 즉시 나타납니다. 여러분이 Zed에 타이핑하면, Claude는 1초 안에 그것을 확인합니다. 한쪽에는 Ghostty, 다른 한쪽에는 Zed를 띄워놓고 동일한 파일을 열어두면, 공동 저자가 AI인 구글 문서(Google Doc)에서 작업하는 것과 같은 느낌을 받을 수 있습니다.

## 5. 계획하기 전에 연구하기

저는 `/ce:plan`을 실행하기 전에, 보통 해당 주제에 대해 `/last30days`를 먼저 실행합니다.

저는 Vercel의 에이전트 브라우저(agent-browser)와 Playwright 중에서 선택하는 중이었습니다. 두 시간 동안 문서를 읽는 대신, 다음과 같이 실행했습니다.

```
/last30days Vercel agent browser vs Playwright
```

몇 분 후: 78개의 Reddit 스레드, 76개의 X(트위터) 게시물, 22개의 YouTube 동영상, 15개의 Hacker News 기사가 나왔습니다. 에이전트 브라우저는 82~93% 적은 컨텍스트 토큰(context tokens)을 사용합니다. Playwright는 도구 정의(tool definitions)에만 13,700 토큰을 덤프(dump)합니다. 이는 상당한 차이입니다.

저는 전체 출력 내용을 `/ce:plan integrate agent-browser`에 입력했습니다. 그 계획은 6개월 전 Claude의 훈련 데이터(training data)에 있던 것이 아니라, *바로 지금* 커뮤니티가 실제로 알고 있는 것에 기반을 두어 나왔습니다.

`/last30days`는 오픈 소스(open source, 4.5K 별, `github.com/mvanhorn/last30days-skill`)입니다. Reddit, X, YouTube, TikTok, Instagram, HN, Polymarket 및 웹을 모두 병렬로 검색합니다. 연구, 계획, 구축. 이것이 진정한 루프(loop)입니다.

## 6. 모든 회의를 `Plan.md`로 전환하기

저는 잠재적 채용 후보자와 점심 식사를 했습니다. 우리는 제품 아이디어에 대해 이야기했고, 음식, 아이들, 식당 등 평범한 90분간의 점심 식사를 했습니다.

저는 **Granola**를 점심 내내 실행했습니다. 식사 후, 전체 대화록(transcript)을 Claude Code에 붙여넣었습니다.

```
/ce:plan turn this into a product proposal
```

특별했던 점은 이겁니다. Claude Code는 이미 저희 GitHub 코드베이스(codebase)와 저의 전체 전략 폴더 — 제가 작성했던 모든 이전 `plan.md` 파일 — 에 접근할 수 있었다는 것입니다.

그래서 점심 대화에서 아이디어를 단순히 가져온 것이 아닙니다. 저희의 실제 제품과 이전에 내렸던 모든 전략적 결정들과 교차 참조(cross-referencing)했습니다.

Claude Code는 단번에 완벽한 제안서(proposal)를 만들어냈습니다. 목표, 사용자 스토리(user stories), 기술적 접근 방식(technical approach), 이정표(milestones) 등이 담겨 있었죠. 스시 대화는 완전히 무시했습니다.

저는 그날 저녁 후보자에게 제안서를 보냈습니다. 그는 이제 그 제품을 위해 저희와 전일제로 일하고 있습니다.

Granola는 이제 MCP 지원을 하므로, Claude Code에 직접 연결됩니다. 더 이상 복사-붙여넣기 할 필요가 없습니다. 모든 회의 내용이 바로 Claude Code로 흘러들어 갑니다.

## 7. `plan` 파일을 모든 것에 활용하기 — 코드뿐만 아니라

저는 회사 전략 문서(strategy doc)를 작성 중이었습니다. Claude Code와 마크다운(Markdown) 파일을 나란히 열어두고 Monologue에 말했습니다.

*"시장 진출(go-to-market)을 위한 세 가지 접근 방식을 제시해 주세요. 각각의 장단점을 설명해 주세요."*

Zed에 세 가지 옵션이 나타났습니다.

*"두 번째 옵션이 가장 가깝지만, 첫 번째 옵션의 표현이 더 마음에 듭니다. 둘을 결합해 주세요."*

즉시 업데이트되었습니다.

*"이제 가장 큰 위험 요소를 다뤄주세요."*

추가되었습니다.

*"두 번째 단락이 너무 깁니다."*

짧아졌습니다.

Claude Code는 저희 GitHub와 저의 모든 이전 전략 계획들을 가지고 있기 때문에 제품을 잘 알고 있습니다. 제가 작성하는 모든 새로운 문서에는 이전에 내렸던 모든 결정의 완전한 맥락(context)이 담겨 있습니다. 이러한 축적 덕분에 각 계획은 이전 계획보다 더 나아집니다.

전략 문서, 제품 사양(product specs), 경쟁 분석(competitive analysis), 그리고 이 글까지. 매번 동일한 워크플로우(workflow)를 사용합니다.

## 8. 원격 Claude Code를 위해 Mac Mini 실행하기

24시간 내내 실행되는 Mac Mini는 여러분의 책상에서 벗어나서도 가능한 일들을 변화시킵니다.

**Telegram 통합.** Claude Code는 Telegram과 연결됩니다. 저는 제 휴대폰으로 Mac Mini에 메시지를 보냅니다. 저녁 식사 중에 버그가 생각나면 Telegram에 다음과 같이 입력합니다.

```
/ce:plan fix the timeout issue
```

제가 책상으로 돌아오면 Zed에 계획이 기다리고 있습니다.

**비행 중 `tmux`.** 이것은 Nathan Smith에게서 배운 것입니다. Claude Code는 비행기 와이파이(WiFi)를 매우 불안정하게 처리합니다. 연결이 끊기고, 세션(session)이 죽어버려도 알려주지도 않습니다. 하지만 먼저 Mac Mini에 `tmux`로 접속하면 세션은 그 기기에서 실행됩니다. 여러분의 노트북은 단지 창일 뿐이죠. 대서양 상공에서 와이파이가 20분 동안 끊겼다고요? 다시 연결하면, 세션은 정확히 그 자리에 있습니다.

저는 유럽에서 돌아오는 비행 내내 기능(features)을 배포했습니다.

### 토큰 소모량이 많아질 때

하루 종일 4개에서 6개의 병렬 Opus 세션을 실행하면 월 200달러짜리 Claude Max 요금제를 다 써버리게 됩니다. 이런 일이 발생합니다.

해결책: 월 200달러짜리 Codex 요금제도 함께 구입하세요. Codex CLI(명령줄 인터페이스)를 설치하면, Claude 크레딧(credits)이 부족할 때 Compound Engineering이 빌드(builds) 작업을 Codex로 넘겨줄 수 있습니다. 저는 방금 Compound Engineering에 이 기능을 정확히 수행하는 `/ce:work --codex`를 배포했습니다. 오늘 병합되었습니다.

어떤 사람들은 Claude Code 작업에 대한 코드 리뷰(code reviews)를 위해 Codex를 사용하고, 그 반대로도 사용합니다. 또 다른 사람들은 Codex의 코드 출력(code output)을 선호하지만, 오케스트레이션(orchestration)을 위해 Claude Code를 사용합니다. 두 가지 요금제는 서로를 잘 보완합니다.

### 실제 사례: 디즈니 월드 6단계 계획

이것은 코드와 전혀 관련 없는 일에 대한 전체 루프(loop)를 보여줍니다. 저는 아이들의 축구 경기에 있었습니다. 다른 부모님이 다가올 디즈니 월드(Disney World) 여행에 대해 언급했고, 저는 노트북을 꺼냈습니다.

**1단계:** `/last30days Disney World`. 2분 후: 66개의 Reddit 스레드(11,804 좋아요), 34개의 X(트위터) 게시물, 8개의 YouTube 동영상이 나왔습니다. 가격 충격이 주된 대화 내용이었는데, r/DisneyPlanning에 올라온 8,500달러짜리 여행 보고서는 183개의 댓글을 받았습니다. 3월에 6개의 놀이기구가 폐쇄되었고, 버즈 라이트이어(Buzz Lightyear)는 4월 8일에 재개장하며, 디노랜드(DinoLand)는 철거되었습니다.

**2단계:** 특정 날짜에 어떤 시설이 열려 있는지 물었습니다. Claude는 보수(refurbishment) 일정을 확인하고, 최신 데이터와 교차 참조하여 전체 개방/폐쇄 목록을 제공했습니다.

**3단계:** `/ce:plan 저는 디즈니 월드에 하루 동안 갈 겁니다. 최소 3개, 아마 4개, 아마도 4개 공원을 방문하고 싶어요. 왜냐하면 제가 좀 미쳤으니까요…` (오타가 있어도 상관없습니다.) Claude의 에이전트들은 `/last30days` 데이터를 교차 참조하여 `plan.md`를 작성했습니다. 공원 순서(AK → HS → Epcot → MK), 정확한 라이트닝 레인(Lightning Lane) 전략, 세 가지 알람(alarm) 알림, 싱글 패스(Single Pass)와 멀티 패스(Multi Pass)가 필요한 놀이기구, 아이들의 키 제한(height requirements) 등이 포함되었습니다.

**4단계:** Zed에서 계획을 열었습니다. 다른 부모님을 위한 두 번째 계획을 만들었습니다. 3일 일정, 8세와 5세 아이들과 함께요. 305줄 분량이었습니다. 일일 일정(day-by-day itineraries), 라이더 스위치(Rider Switch) 프로토콜(protocols), "이번 주에 5살 아이의 신발을 신긴 채 키를 재세요"라는 경고문까지 포함되었습니다.

**5단계:** "이 마지막 계획을 Vercel 사이트에 라이트 모드(light mode)로 게시해 줄 수 있을까요?" (오타가 있었지만, 중요하지 않았습니다.) Claude는 깔끔한 HTML 페이지를 만들고 배포(deploy)했습니다.

**6단계:** `OpenClaw`에 `Telegram`을 통해 `.md` 파일을 넣었습니다. 그리고 이중 백업(double backups) — 캘린더 이벤트(calendar events)와 크론 잡(cron job) 핑(pings) — 으로 알림(reminders)을 추가해 달라고 요청했습니다. 모든 예약 기간에 대한 이중 확인이었습니다. 4월 13일 오전 3시 50분: "지금 멀티 패스 구매!" 4월 16일 오전 3시 50분: "지금 싱글 패스 구매!" 모두 동부 표준시 오전 7시 예약 창이 열리기 10분 전이었습니다. 실행 후에는 자동 삭제됩니다.
