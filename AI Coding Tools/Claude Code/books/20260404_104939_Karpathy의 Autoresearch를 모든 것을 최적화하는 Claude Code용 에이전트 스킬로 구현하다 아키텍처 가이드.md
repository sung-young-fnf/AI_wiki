# Karpathy의 Autoresearch를 모든 것을 최적화하는 Claude Code용 에이전트 스킬로 구현하다: 아키텍처 가이드

Andrej Karpathy가 공개한 'Autoresearch'의 핵심은 단순한 학습 코드가 아니라 '제약 조건(Constraint)'에 있었습니다. 저는 이 루프를 API 속도, 광고 클릭률(CTR), 프롬프트 품질 등 모든 영역에 적용할 수 있는 에이전트 스킬로 구축했습니다.

### Karpathy가 실제로 구축한 것 (그리고 우리가 놓친 것)

Autoresearch의 핵심은 630줄의 코드 자체가 아니라 그 이면의 통찰, 즉 **'하나의 파일, 하나의 지표, 하나의 루프'**라는 제약 조건입니다. 수정하고, 평가하고, 유지하거나 버리는 과정을 반복하는 이 패턴은 머신러닝뿐만 아니라 모든 최적화의 원형입니다.

이 루프가 작동하는 핵심 제약 조건은 다음과 같습니다.

*   **고정된 시간 예산:** 모든 실험은 정확히 5분 동안 실행됩니다. 이를 통해 모델 크기나 배치(Batch) 크기에 상관없이 실험 결과를 직접 비교할 수 있습니다.
*   **단일 파일 범위:** 에이전트는 오직 `train.py` 하나만 수정합니다. 이를 통해 변경 사항을 명확히 추적하고 범위를 제한할 수 있습니다.
*   **메모리로서의 Git:** 성공한 실험은 커밋(Commit)하고 실패는 되돌립니다(Revert). Git 로그가 곧 실험의 역사가 됩니다.
*   **단일 지표:** 오직 하나의 지표(예: `val_bpb`)만 사용합니다. 모호함이나 다중 목표 간의 절충안(Trade-off)은 허용하지 않습니다.

### 일반화: 숫자가 있는 모든 것으로의 확장

저는 이 패턴을 API 응답 시간, 번들 크기, 테스트 통과율, 빌드 속도, 프롬프트 품질 등 수치화할 수 있는 모든 영역으로 일반화했습니다. 이를 위해 Claude Code, Cursor 등 다양한 도구에서 작동하는 **AgentSkills** 표준 기반의 `autoresearch-agent` 스킬을 구축했습니다.

이 아키텍처는 최적화 루프와 도메인을 분리합니다.

*   **Engineering:** 코드 속도, 메모리, 번들 크기 최적화
*   **Marketing:** 헤드라인, 광고 문구의 클릭률(CTR) 최적화
*   **Prompts:** 시스템 프롬프트 및 지시문 품질 최적화

설정 과정에서는 Karpathy의 구조를 따르되 모든 도메인에 대응할 수 있도록 매개변수화했습니다. 실험 폴더에는 대상 파일, 평가 명령, 지표 이름, 목표 방향(상승/하락) 등이 포함된 설정 파일과 에이전트의 전략을 담은 `program.md`가 생성됩니다.

### 평가의 난제: 숫자가 아닌 것들을 측정하기

머신러닝 지표와 달리 헤드라인의 품질이나 프롬프트의 효과는 단순한 숫자로 산출되지 않습니다. 이를 해결하기 위해 두 가지 유형의 평가자(Evaluator)를 도입했습니다.

1.  **무료 평가자(Free Evaluators):** 벤치마크 명령을 실행하여 응답 시간이나 파일 크기 같은 수치를 직접 파싱합니다. API 비용이 들지 않고 명확합니다.
2.  **LLM 평가자(LLM Judge Evaluators):** 헤드라인이나 시스템 프롬프트의 품질을 0~100점 척도로 점수화합니다. 사용자가 이미 구독 중인 도구(Claude Code, ChatGPT Pro 등)의 CLI를 활용해 추가 비용 없이 실행할 수 있습니다.

**중요한 제약 조건:** 에이전트는 평가 로직이 담긴 `evaluate.py`를 수정할 수 없습니다. 최적화 도구가 지표의 정의까지 제어하게 되면, 점수를 높이기 위해 기준 자체를 조작하는 '정렬 문제(Alignment Problem)'가 발생하기 때문입니다.

### 전략 고도화 (Strategy Escalation)

에이전트는 실험 횟수가 늘어남에 따라 사고 패턴을 발전시킵니다.

*   **1~5회:** 명확하고 쉬운 개선사항 적용
*   **6~15회:** 체계적인 파라미터 탐색
*   **16~30회:** 알고리즘 교체 등 구조적 변경
*   **30회 이후:** 근본적으로 다른 접근 방식 시도

또한, 매 10회의 실험마다 에이전트는 결과 로그를 검토하여 "캐싱 변경이 성능을 5~10% 일관되게 향상시킴"과 같은 패턴을 파악하고 스스로 전략(`program.md`)을 업데이트합니다.

### 현재의 한계와 향후 과제

이 시스템은 아직 몇 가지 한계를 지니고 있습니다.

*   **단일 파일 최적화:** 실제 성능은 여러 파일에 걸친 변경에 의존하는 경우가 많으나, 현재는 리뷰 가능성을 위해 단일 파일로 제한하고 있습니다.
*   **단일 지표:** 속도와 메모리 사이의 절충안 같은 다중 목표 최적화(Pareto optimization)는 아직 지원하지 않습니다.
*   **LLM 평가자의 한계:** LLM 점수는 상대적인 개선을 찾는 데 유용하지만, 실제 세상의 절대적인 결과(예: 실제 사용자 클릭률)를 보장하지는 않습니다.

### 결론: 왜 이 패턴이 중요한가

자율적이고 측정 가능하며 누적되는 반복(Iteration)은 향후 2년 내에 표준적인 엔지니어링 관행이 될 것입니다. 머신러닝 학습뿐만 아니라 마케팅 랜딩 페이지 최적화, 코드 최적화, 프롬프트 엔지니어링 모두 평가자만 다를 뿐 동일한 루프를 공유합니다.

지금 수동으로 반복하고 있는 작업(수정 -> 테스트 -> 확인 -> 반복)이 있다면, 그것이 바로 Autoresearch의 대상입니다. 이 루프를 통해 개인의 전문성을 시스템적인 인프라로 전환할 수 있습니다.
