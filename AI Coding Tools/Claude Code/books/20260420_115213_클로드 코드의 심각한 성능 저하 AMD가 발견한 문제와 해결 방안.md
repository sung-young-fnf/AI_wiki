# 클로드 코드의 심각한 성능 저하: AMD가 발견한 문제와 해결 방안

6,852회의 세션에서 클로드 코드(Claude Code)의 사고력 저하가 입증되었습니다. 사고력이 67% 감소한 것입니다.

"클로드는 복잡한 엔지니어링 작업을 신뢰할 수 없을 정도로 퇴보했습니다." — AMD 수석 AI 디렉터

클로드 코드는 읽지도 않은 파일을 편집해 왔습니다. 이 문제는 가끔이 아니라 몇 주 동안 지속적으로 발생했습니다.

AMD의 AI 디렉터인 스텔라 로렌조(Stella Laurenzo)는 이 문제를 보고했습니다. 그녀는 `~/.claude/projects/` 경로에 있는 6,852개 세션의 증거를 가지고 있습니다.

매일 클로드 코드를 사용하는 대부분의 개발자들은 자신들의 결과물 품질이 훨씬 나빠졌다는 사실을 전혀 인지하지 못하고 있습니다.

## 실제로 무엇이 바뀌었나

2월에 앤트로픽(Anthropic)은 "적응형 사고(adaptive thinking)" 기능을 출시했습니다. 멋지게 들리지만 실제로는 그렇지 않았습니다.

이전에는 클로드 코드가 기본적으로 높은 수준의 노력(effort)으로 추론했습니다. 이제 모델은 필요하다고 판단하는 만큼만 사고(think)하도록 결정합니다.

문제는 모델의 사고(추론)가 현저히 줄어들었다는 점입니다.

간단히 말해, 클로드 코드는 코드를 작성하기 전에 엔지니어와 유사한 문제 해결 과정을 거칩니다. 파일을 읽고, 비즈니스 로직을 이해하려 노력하며, 솔루션 내 잠재적인 문제를 파악하는 과정입니다. 이 과정을 추론(reasoning)이라고 합니다.

이 추론 과정은 클로드 코드 성공의 큰 부분이었습니다.

그러나 적응형 사고는 이 과정을 매우 극적으로 단축시켰고, 이전에 건너뛰지 않던 단계를 건너뛰기 시작했습니다.

이후 앤트로픽은 `redact-thinking-2026–02–12`를 출시하여 모델의 추론 단계(실질적으로는 대규모 성능 저하)를 숨겼습니다.

이것이 중요한 이유는 이제 모델이 덜 추론하는데도 사용자들은 이를 볼 수 없게 되었기 때문입니다. 작업은 자신감 있게 완료되지만, 실제로는 우리 모르게 엄청난 성능 저하(nerf)가 발생한 것입니다.

많은 사용자들이 모델 성능이 나빠졌다는 느낌을 받았지만(저를 포함해서), 이는 단순한 기분처럼 여겨졌습니다.

스텔라는 그러한 막연한 느낌을 측정 가능한 대규모 문제로 바꿔주는 분석을 수행했습니다.

## 수치로 입증된 증거

AMD AI 그룹의 수석 디렉터인 스텔라 로렌조는 50개 이상의 세션을 운영하며 `~/.claude/projects/`에 많은 로그(log)를 축적했다고 말합니다.

요컨대, 2월 이전의 클로드 코드는 오늘날보다 두 배 더 많은 사고(thinking)와 작업을 수행했습니다.

가장 충격적인 점은(개인적인 의견입니다만), 클로드 코드가 편집하는 파일당 6.6개의 파일을 읽던 것에서 2개의 파일만 읽게 되었다는 것입니다.

잠시 이것에 대해 생각해 보십시오. 만약 제가 여러분에게 어떤 작업을 하라고 지시하고 단 두 개의 파일만 읽도록 허용한다면, 여러분이 작성할 코드가 어떨지 상상해 보십시오.

"사고가 얕아지면 모델은 가장 저렴한(cheapest) 행동을 기본으로 삼습니다. 즉, 읽지 않고 편집하고, 끝내지 않고 중단하며, 실패에 대한 책임을 회피하고, 올바른 해결책 대신 가장 단순한 해결책을 택하는 것입니다. 이러한 증상들이 정확히 관찰되었습니다."

## 실제 작업에서 나타나는 현상

클로드 코드에 디버깅(debug) 및 문제 해결을 요청하면, 모델은 빠르고 자신감 있게 해결책을 제시합니다. 그러면 여러분은 그 해결책을 적용합니다.

여러분은 숨겨진 진실에 완전히 눈이 가려진 상태입니다.

클로드 코드는 복잡한 버그(bug)를 수정하기 위해 단 2개의 파일만 읽었을 뿐입니다.

또 다른 문제로는 클로드 코드가 작업 완료를 조작하는 현상이 보고되었습니다.

가장 나쁜 점은 이 모든 것이 겉보기에는 정상적이고 올바르게 보인다는 것입니다.

## 왜 눈치채지 못했을까

솔직히 말하면, 저도 뭔가 이상하다는 느낌은 받았지만 정확히 짚어낼 수는 없었습니다.

세션이 미묘하게 어긋나는 느낌을 받기 시작했습니다. 작은 수정 작업에도 더 많은 주의와 검토가 필요했습니다. 문제는 이러한 도구를 사용할 때, 사용자들이 자신의 프롬프트(prompt)나 기술, 또는 코드에 문제가 있다고 생각하게 된다는 점입니다. 이 조용한 문제가 제 머리 속에는 떠오르지 않았습니다.

하지만 이제 여러분은 이 사실을 알게 되었으니 쉽게 알아차릴 수 있을 것입니다. 클로드 코드는 이전만큼 추론/사고하지 않고 있습니다. 파일 호출 빈도가 줄어들고 있습니다. 출력 결과는 더 빨라졌지만, 그것들은 틀린 내용입니다.

## 해결책

가장 중요하고 첫 번째 해결책은 다음과 같습니다.

```
alias claude="claude --effort max"
alias claude-max="claude --model claude-opus-4-6 --effort max"
```

이 설정은 다른 곳에서 재정의(override)될 수 있으므로 중요합니다.

낮아진 추론 능력을 되돌리는 데 도움이 되는 또 다른 지시 사항은 다음 내용을 `CLAUDE.md` 파일에 추가하는 것입니다.

```
For every claim or finding, quote the exact code that proves it.
If you cannot quote the actual code, say "I need to read this 
file first" instead of making the claim.
```

하지만 참고로, 저는 클로드가 미래에 이러한 유형의 재정의를 막을 수도 있다고 생각합니다.

복잡한 작업에는 `ultrathink`를 사용하십시오.

## 더 큰 경고

AMD는 50개의 클로드 코드 에이전트(agent)를 사용하여 복잡한 시스템을 운영하고 있었습니다(이들은 AMD입니다, 말할 필요도 없이). 한 번의 업데이트가 이들에게 치명적인 문제를 일으켰습니다.

스텔라의 팀은 이미 다른 제공업체로 전환했으며, NDA(비밀유지협약) 때문에 이름을 밝히기를 거부했습니다. 하지만 그녀는 우리에게 다음과 같은 경고를 남겼습니다.

"6개월 전만 해도 클로드는 추론 품질과 실행 능력 면에서 독보적이었습니다. 하지만 이제 앤트로픽은 이전에 Opus가 차지했던 능력 수준에서 결코 혼자가 아닙니다."

이는 모든 대안을 고려해야 할 신호입니다.

추론 노력을 줄인 다음, 그 감소를 숨기는 기능을 출시한 것은 우리에게 나쁜 신호입니다. 이는 비용을 절감하면서도 우리의 결과물 품질을 저하시키려는 시도로 보입니다.

## 다음 조치 사항

이러한 노력(effort) 감소는 우리 모두에게 중요합니다. 조용히 발생했으며 우리를 방심하게 만들었습니다.

*   5분 여유가 있다면: 지금 바로 `alias`를 `.zshrc` 파일에 추가하십시오.
    ```
    alias claude="claude --effort max"
    alias claude-max="claude --model claude-opus-4-6 --effort max"
    ```
*   15분 여유가 있다면: `CLAUDE.md` 증거 규칙을 추가하고 테스트하십시오.
*   30분 여유가 있다면: 댄 번스(Dan Burns)의 훌륭한 해결책 팩을 읽어보십시오.
