# Hermes 에이전트: 어제 학습한 내용을 실제로 기억하는 오픈 소스 AI 에이전트

여러분들이 사용해 본 모든 AI 에이전트에는 동일한 문제가 있습니다. 월요일에 무언가를 가르치면 화요일에는 모든 것을 잊어버립니다. 그러면 다시 처음으로 돌아가 프로젝트 구조, 선호 사항, 전체 작업 흐름을 다시 설명해야 합니다. 마치 매일 아침 새로운 인턴을 교육하는 것과 같습니다.

Nous Research는 이러한 반복을 끝내기 위해 Hermes 에이전트를 개발했습니다.

Hermes는 자체 서버에서 실행되고, 수행하는 모든 작업에서 학습하며, 사용할수록 눈에 띄게 개선되는 오픈 소스(open-source), 자체 호스팅(self-hosted) AI 에이전트입니다. 2026년 2월 출시 이후 64,000개 이상의 GitHub 스타를 얻었으며, 개발자들이 이전까지 지배적이던 오픈 소스 에이전트 프레임워크인 OpenClaw로부터의 '마이그레이션 물결'이라고 부르는 현상을 촉발했습니다.

흥미로운 부분은 채택률이 아니라, 그 기반이 되는 아키텍처(architecture)와 이것이 매 세션마다 초기화되는 대신 시간이 지남에 따라 가치가 축적되는 AI를 원하는 모든 사람에게 어떤 의미를 가지는지입니다.

## 핵심 아이디어: 스스로 운영 지침서를 작성하는 에이전트

Hermes는 작업을 성공적으로 완료하면 단순히 결과를 기록하고 넘어가지 않습니다. 대신 실행 후 평가(post-execution evaluation)를 수행하여 특정 결과를 도출한 정확한 단계 순서, 도구 호출(tool calls) 및 추론을 식별한 다음, 해당 순서를 재사용 가능한 '기술(Skill)' 문서로 규격화합니다. 이 마크다운(markdown) 파일은 에이전트가 다음에 유사한 작업을 만났을 때 참고합니다.

이것이 Nous Research가 '폐쇄형 학습 루프(closed learning loop): 실행, 평가, 추출, 개선, 검색'이라고 부르는 방식입니다. 그리고 이는 점진적으로 축적됩니다. Nous Research가 발표한 벤치마크(benchmark)에 따르면, 자체 생성 기술을 실행하는 에이전트는 프롬프트 튜닝(prompt tuning)이 전혀 없는 새 인스턴스보다 연구 작업을 40% 더 빠르게 완료했습니다.

3개월 동안 5달러짜리 VPS(Virtual Private Server)에서 실행되는 Hermes 인스턴스는 여러분의 코드베이스, 배포 특이사항, 선호하는 커밋 메시지 형식, 그리고 특이한 레거시 통합(legacy integration)에 맞는 정확한 API 호출 시퀀스를 알고 있습니다. 새로 설치된 인스턴스는 이 중 아무것도 알지 못합니다.

## 기술(Skills) vs. 도구(Tools): 중요한 구분

Hermes는 기술(Skills)과 도구(Tools) 사이에 명확한 아키텍처적 구분을 두는데, 이는 생각보다 훨씬 중요합니다.

도구(Tools)는 JSON 스키마(schema)를 통해 AI 모델에 노출되는 파이썬(Python) 함수입니다. 이들은 결정론적(deterministically)으로 실행됩니다. 웹 브라우저 자동화, 파일 조작, 스트리밍 프로세스 등을 생각해보세요. 이를 변경하려면 핵심 파이썬 파일을 편집해야 합니다.

기술(Skills)은 마크다운 문서입니다. 에이전트는 이를 지침처럼 읽고 문서화된 절차를 자율적으로 따릅니다. 여기서 중요한 점은 에이전트가 스스로 기술을 작성할 수 있다는 것입니다. 코드 변경이 필요 없고, 사람이 설정 파일을 편집할 필요도 없습니다. 에이전트는 스스로 성공하는 과정을 관찰하고, 그 방법을 기록하며, 다음에 사용할 수 있도록 해당 지식을 저장합니다.

에이전트는 자체 소스 코드(source code)를 전혀 건드리지 않고도 더 똑똑해집니다. 결정론적 도구 계층(deterministic tool layer)은 안정적이고 안전하게 유지되며, 지식 계층(knowledge layer)은 지속적으로 성장합니다.

## 4단계 메모리 시스템

컨텍스트(context)가 너무 적으면 에이전트는 잘못된 결정을 내립니다. 너무 많으면 토큰(token)을 소모하고 지연 시간(latency)이 급증하며, 모델은 방대한 프롬프트(prompt)에 묻힌 지침을 잊기 시작합니다. Hermes는 에이전트가 정보를 얼마나 긴급하게 필요로 하는지에 따라 계층화된 4단계 시스템으로 이 문제를 해결합니다.

*   **1단계: 에이전트의 개인 노트 (MEMORY.md)**
    환경 사실, 프로젝트 규칙 및 운영 교훈을 저장합니다. 약 800토큰으로 제한되며 세션 시작 시 시스템 프롬프트에 직접 주입됩니다. 이를 에이전트의 '치트 시트(cheat sheet)'라고 생각하면 됩니다.
*   **2단계: 사용자 프로필 (USER.md)**
    여러분이 누구인지, 기술 숙련도, 시간대, 의사소통 스타일 등의 모델을 유지합니다. 약 500토큰이며, 역시 시작 시 주입됩니다.
*   **3단계: 세션 검색(Session Search)**
    모든 과거 세션의 내용을 SQLite 데이터베이스에 보관합니다. 에이전트는 LLM 요약과 결합된 전체 텍스트 검색(full-text search)을 사용하여 필요에 따라 이를 쿼리(query)합니다. 깊이 있는 과거 컨텍스트가 여기에 저장됩니다.
*   **4단계: 외부 메모리 플러그인(External Memory Plugins)**
    LightRAG, Supermemory 또는 사용자 정의 벡터 스토어(vector stores)와 같은 그래프 기반 검색 시스템에 연결됩니다. 이는 복잡한 관계 맵(relationship maps)을 가로지르는 다중 홉 쿼리(multi-hop queries)를 가능하게 하여 엔터프라이즈(enterprise) 사용 사례에 특히 중요합니다.

처음 두 계층은 '고정 스냅샷(Frozen Snapshot)' 패턴을 사용합니다. 변경 사항은 즉시 디스크에 기록되지만, 다음 세션이 시작될 때까지 활성 시스템 프롬프트(system prompt)를 수정하지 않습니다. 이는 언어 모델의 접두사 캐시(prefix cache)를 보존하여 긴 세션 동안 지연 시간을 낮게 유지합니다. 대화 도중 프롬프트를 변경하면 해당 캐시가 무효화되고 추론 속도(inference speed)가 저하됩니다.

또한 이해할 가치가 있는 '주기적 알림(periodic nudge)' 메커니즘도 있습니다. 에이전트가 중요한 내용을 저장하라는 지시를 수동적으로 기다리는 대신, Hermes 런타임(runtime)은 유휴 시간에 에이전트에게 최근 상호작용을 평가하고 컨텍스트 윈도우(context window)가 가득 차서 오래된 대화가 압축되어 사라지기 전에 중요한 사실을 추출하도록 적극적으로 안내합니다. 이 명시적인 플러시(flush) 과정에서 에이전트가 표시하지 않은 사실은 단순히 보존되지 않습니다. 이는 메모리 관리에 대한 '사용하지 않으면 잃는(use-it-or-lose-it)' 접근 방식입니다.

## 심층 연구: 단순한 고급 웹 검색이 아니다

Hermes에 복잡한 연구 작업을 할당하면, 에이전트는 먼저 'think_tool'을 호출하는 것으로 시작합니다. 이는 모델이 실제로 조사를 시작하기 전에 실제 조사 계획을 명확히 설명하도록 강제하는 전략적 일시 정지입니다. 데이터 벡터를 매핑하고, 범위를 정의하며, 가설을 수립합니다. 이 계획 단계가 끝난 후에야 에이전트가 실행을 시작할 수 있습니다.

그다음에는 병렬화합니다.

'ConductResearch' 도구는 단일 쿼리(query)만 실행하지 않습니다. 이는 특정 연구 주제를 독립적인 하위 에이전트(sub-agents)들에게 위임하며, 각 하위 에이전트는 자체적인 깨끗한 컨텍스트 윈도우, 격리된 터미널 세션, 제한된 도구 세트를 가지고 실행됩니다. 한 하위 에이전트는 헤드리스 브라우저(headless browser)를 통해 규제 서류를 스크랩(scrape)할 수 있고, 다른 에이전트는 데이터베이스를 쿼리하며, 세 번째 에이전트는 파이썬 스크립트를 실행하여 통계 분석을 생성할 수 있습니다. 이들은 동시에 실행되며 구조화된 JSON을 주 오케스트레이터(orchestrator)에게 반환합니다.

병렬 작업이 완료되면 에이전트는 'think_tool'을 다시 호출합니다. 이 두 번째 단계에서는 종합(synthesis)을 강제합니다. 즉, 돌아온 결과를 평가하고, 부족한 부분을 확인하며, 목표가 달성되었는지 판단합니다. 그렇지 않으면 새로운 연구 벡터(research vectors)로 루프를 반복합니다. 목표가 달성되면 'ResearchComplete'를 호출하고 보고서 생성으로 넘어갑니다. 미리 정의된 깊이 경계(depth boundaries)는 무한 재귀(infinite recursion)를 방지합니다.

최종 보고서를 위해 모듈형 미니 보고서들은 대규모 컨텍스트 모델(최소 64K 토큰 윈도우)에 의해 병합되어 응집력 있고 인용 가능한 문서로 만들어집니다.

## 보안 이야기: 기능보다 중요한 이유

여기서 OpenClaw 사용자들에게 OpenClaw와의 비교는 불편해질 수 있습니다.

2026년 3월 18일에서 21일 사이에 OpenClaw에 대해 4일 만에 9개의 CVE(Common Vulnerabilities and Exposures)가 발표되었으며, 여기에는 인증된 사용자가 관리자 권한으로 상승할 수 있게 하는 CVSS(Common Vulnerability Scoring System) 9.9점의 심각한 취약점도 포함되었습니다. 4월 초까지 보안 연구원들은 63일 동안 발견된 138개의 CVE를 추적하고 있었습니다. 이는 하루에 약 2.2개의 새로운 취약점이 발견된 셈입니다.

공급망(supply chain) 수치는 더욱 나쁩니다. 보안 회사 Antiy CERT는 ClawHub 마켓플레이스에서 1,184개의 악성 기술(malicious skills)을 확인했으며, 이는 한때 패키지 5개 중 약 1개꼴이었습니다. SecurityScorecard는 공용 인터넷에 노출된 135,000개 이상의 OpenClaw 인스턴스가 불안전한 기본값(insecure defaults)을 가지고 있음을 발견했습니다. 'ClawHavoc' 캠페인은 취약점을 찾을 필요도 없었습니다. 공격자들은 그저 설득력 있어 보이는 기술을 업로드하고 사람들이 설치하기를 기다렸습니다.

Hermes는 근본적으로 더 보수적인 접근 방식을 취합니다. 에이전트가 특정 작업 흐름을 기반으로 자체적으로 기술을 내부적으로 생성하기 때문에, 외부 공급망 공격 벡터(attack vector)를 완전히 우회합니다. 익명의 기여자가 작성한 실행 가능한 코드(executable code)를 공개 레지스트리에서 가져오는 것이 아닙니다. 에이전트가 자신의 환경에서 관찰된 성공을 바탕으로 자체적인 운영 지식을 작성하는 것입니다.

인프라(infrastructure) 측면에서 Hermes는 터미널(terminal) 실행을 위한 필수 컨테이너 격리(container isolation), 하위 에이전트 프로세스를 위한 암호화 네임스페이스 분리(cryptographic namespace segregation), 동적 자격 증명 순환(dynamic credential rotation)을 강제합니다. 또한 외부 데이터를 시스템 프롬프트에 주입하기 전에 프롬프트 인젝션(prompt injection)을 스캔합니다. OpenClaw의 패치(patch) 응답 시간은 빠르고 종종 당일에 이루어졌으며, 프로젝트가 근본적으로 고장난 것은 아닙니다. 그러나 아키텍처적 공격 표면(architectural attack surface)의 차이는 분명합니다.

## Hermes 실습: 주말 설치 가이드

Git만 설치되어 있으면 됩니다. 설치 프로그램이 Python 3.11+, Node.js v22, ripgrep, ffmpeg를 처리합니다. Linux, macOS, WSL2, 그리고 Termux를 통한 Android에서 실행됩니다. 네이티브 윈도우(Windows)는 지원하지 않습니다.

### 1단계: 설치

다음 명령어를 실행하여 설치합니다.

```bash
curl -fsSL https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.sh | bash
```

설치 프로그램은 운영 체제(OS)를 감지하고 종속성(dependencies)을 자동으로 프로비저닝(provisioning)합니다. 파이썬(Python)은 Rust 기반 패키지 관리자인 uv를 통해 실행되며, sudo 권한 없이 격리된 가상 환경을 생성합니다.

### 2단계: 설치 실행

```bash
hermes setup
```

대화형 마법사(interactive wizard)가 모델 선택 및 초기 구성을 안내합니다. OpenClaw 사용자라면, 마법사가 `~/.openclaw`를 자동으로 감지하고 설정, 메모리, 기술 및 API 키(API keys)를 마이그레이션(migrate)할 것을 제안합니다.

### 3단계: 모델 선택

```bash
hermes model
```

Hermes는 모델에 구애받지 않습니다(model-agnostic). Anthropic, OpenAI, DeepSeek 또는 OpenRouter(단일 엔드포인트(endpoint)를 통해 400개 이상의 모델에 접근 가능)에 연결할 수 있습니다. 심층 연구 작업에는 최소 64K 컨텍스트 토큰(context tokens)을 가진 모델이 필요합니다.

API 비용 없이 완전한 데이터 프라이버시(data privacy)를 원한다면, Ollama를 통해 로컬 모델을 가져올 수 있습니다.

```bash
ollama pull qwen2.5-coder:32b
```

그런 다음 Hermes를 로컬 엔드포인트인 `http://localhost:11434/v1`로 지정하세요. 완전한 자율성, 외부 API 호출 없음, 데이터가 머신을 벗어나지 않습니다.

### 4단계: 게이트웨이 설정 (선택 사항이지만 강력함)

```bash
hermes gateway setup
```

Hermes를 텔레그램(Telegram), 디스코드(Discord), 슬랙(Slack), 왓츠앱(WhatsApp) 또는 시그널(Signal)에 연결하세요. 게이트웨이(gateway)는 영구적인 백그라운드 서비스로 실행되므로, 서버에서 무거운 작업을 실행하는 동안 휴대폰으로 에이전트에 메시지를 보낼 수 있습니다. 텔레그램의 경우, BotFather를 통해 봇을 등록하고 `@userinfobot`을 통해 사용자 ID를 얻은 다음, 둘 다 구성(config)에 입력하세요.

한 가지 주의할 점: 메시징 인터페이스는 모니터링(monitoring) 및 비동기 업데이트(async updates)에는 훌륭하지만, 본질적으로 선형적입니다. 복잡한 코드베이스 변경과 같은 심층적인 협업 작업의 경우 메시징 스레드가 빠르게 병목 현상(bottleneck)이 됩니다. 그러한 종류의 상호작용에는 CLI(Command Line Interface) 또는 전용 작업 공간을 사용하세요.

### 5단계: 실행 컨테이너화 (필수 권장)

```bash
hermes config set terminal.backend docker
```

이 설정은 에이전트가 생성한 모든 코드가 격리된 컨테이너(container) 내부에서 실행되도록 보장합니다. 문제가 발생하더라도 호스트 시스템은 영향을 받지 않습니다.

### 6단계: 학습시키기

주간 보고서, PR 검토(PR reviews), API 데이터 처리, 연구 자료 정리와 같이 간단하고 반복 가능한 작업부터 시작하세요. Skills 디렉토리(directory)를 주시하세요. 성공적으로 작업을 완료한 후에는 에이전트가 스스로 작성한 운영 지침서인 새로운 `.md` 파일이 나타나는 것을 볼 수 있을 것입니다. 몇 주 후에는 몇 분 걸리던 작업이 몇 초 만에 완료되기 시작할 것입니다.

## Hermes의 한계 (솔직한 평가)

*   **로컬 추론(local inference)의 하드웨어 현실.** Ollama를 통해 qwen2.5-coder:32b를 가져와 무료 로컬 추론을 하라는 본문의 제안은 현실적인 점검이 필요합니다. 32B 모델로 주 오케스트레이터를 실행하는 것은 괜찮습니다. 그러나 이전에 설명된 심층 연구 병렬화는 여러 하위 에이전트가 각각 자체적인 활성 추론 스레드(inference thread)와 깨끗한 컨텍스트 윈도우를 필요로 하므로, 상당한 GPU 자원을 요구합니다. 보다 현실적인 로컬 설정은 하위 에이전트 라우팅(routing) 및 데이터 추출에는 더 작은 양자화된(quantized) 모델을 사용하고, 32B 모델은 주 오케스트레이터와 최종 종합에 할당하는 방식입니다. 단일 소비자 GPU를 사용하는 경우, 병렬 연구 작업 흐름에서 심각한 병목 현상을 예상해야 합니다.
*   **자체 작성 기술은 취약합니다(brittle).** 내부 기술 시스템은 OpenClaw의 공급망 위험을 우회하지만, 마크다운 운영 지침서는 기반 환경이 변경되면 작동하지 않게 됩니다. 레거시 API가 인증 흐름(auth flow)을 업데이트하거나 웹사이트가 DOM(Document Object Model)을 재설계하면 에이전트의 기억된 절차는 실패합니다. 시각적 오류 처리 경로가 있는 결정론적 워크플로우 도구와 달리, Hermes는 실패한 후 재평가하고 처음부터 기술을 다시 작성해야 합니다. 학습 루프가 이를 처리하지만, 즉각적으로 이루어지지는 않습니다.
*   **실제로 필요하지 않다면 4단계 메모리는 건너뛰세요.** 본문에서는 LightRAG 및 Supermemory 통합을 언급하지만, 대부분의 단일 서버 설정에서는 SQLite 전체 텍스트 검색(3단계)이 속도와 검색 성능 면에서 최상의 균형을 제공합니다. 무거운 외부 벡터 데이터베이스와 평면 MEMORY.md 파일을 동기화하는 것은 엔터프라이즈 규모의 다중 에이전트 워크플로우를 실행하지 않는 한 오버헤드(overhead)에 비해 가치가 없는 지연 시간 및 상태 충돌 문제를 야기합니다.
*   **설정 복잡성은 실제입니다.** OpenClaw는 설치부터 에이전트 작동까지 더 빠르게 진행됩니다. 50개 이상의 플랫폼 통합과 소비자 수준의 단순성이 필요하다면 OpenClaw가 더 접근하기 쉬운 선택으로 남아있습니다.
*   **프레임워크는 두 달밖에 되지 않았습니다.** 2026년 4월 8일에 버전 0.8.0이 출시되었으며, 개발 속도는 빠릅니다. 그러나 프로젝트는 여전히 성숙 단계에 있습니다. 미흡한 점을 예상해야 합니다.
*   **자기 개선은 점진적이지, 마법과 같지 않습니다.** 40% 벤치마크 개선은 짧은 데모가 아닌, 몇 주에 걸친 지속적인 사용을 통해 얻어진 것입니다. 하룻밤 사이의 변화를 기대하지 마세요.

## 더 큰 그림

AI 에이전트 시장은 두 가지 진영으로 나뉘고 있습니다. 연결성(Connectivity, OpenClaw의 전략: 모든 곳에, 모든 것에 연결) 대 인지(Cognition, Hermes의 전략: 시간이 지남에 따라 더 똑똑해짐)입니다. 이 두 가지 모두 다른 사용 사례에 유효합니다. 일부 팀은 OpenClaw를 데이터 수집 및 라우팅(routing)에 사용하고, Hermes를 심층 분석 작업에 사용하는 등 두 가지를 모두 운영하고 있습니다.

동일한 복잡한 워크플로우를 반복적으로 실행하는 모든 사람에게는 실제로 학습하는 에이전트의 축적되는 이점을 무시하기 어렵습니다. 설치는 5분이면 됩니다. 나머지는 학습 루프가 알아서 처리합니다.
