# 30개 이상의 OpenClaw 대안을 무시하다가 OpenFang을 만났습니다

OpenClaw에는 이미 30개 이상의 대안이 존재하며, 솔직히 대부분은 쉽게 무시할 수 있었습니다.

OpenClaw와 그 대안들은 동일한 것을 약속합니다. 즉, 정해진 스케줄에 따라 실행되고, 로컬 파일에 접근하며, 오전 6시에 텔레그램 메시지를 보내고, 당신이 채팅창에 계속 신경 쓰지 않아도 작동하는 에이전트(agent) 말이죠.

하지만 실제로 사용해 보면 대부분은 무겁고 과도하게 구축된 런타임(runtime)이며, 단순한 대화형 데모를 위해 설계되었을 뿐, 진지하고 상시 작동해야 하는 대규모 작업(workload swarm)에 신뢰할 수 있는 시스템은 아니라는 것을 알게 될 것입니다.

그래서 OpenFang이야말로 자세히 살펴볼 가치가 있다고 느낀 첫 번째 사례였습니다.

OpenFang은 완전히 오픈 소스(open-source) 에이전트 운영 체제(Agent Operating System)로, 전적으로 Rust로 작성되었으며, 단일 32MB 바이너리(binary) 형태로 제공되고 180ms의 콜드 스타트(cold start) 시간을 자랑합니다.

이는 벤치마크(benchmark)를 위한 과장된 수치처럼 들릴 수 있지만, 중요한 것은 언어 선택이나 시작 속도가 아닙니다.

런타임(runtime)이 매우 경량화되어 있어 OpenFang은 단일 가상 사설 서버(VPS)에서 여러 자율 에이전트(autonomous agent)를 동시에 24시간 내내 부하 없이 실행할 수 있습니다.

설정은 거의 의심스러울 정도로 간단합니다.

```bash
curl -fsSL https://openfang.sh/install | sh
```

`openfang init`을 실행하고
모델 제공자(model provider)를 선택한 다음
브라우저를 열면
바로 접속됩니다.

이 글에서는 OpenFang이 실제로 무엇을 하는지 자세히 설명하고 빠른 시작 가이드(quick start guide)를 제공할 것입니다.

OpenClaw 및 점점 늘어나는 대안들과 OpenFang이 어떻게 다른지, 그리고 OpenFang이 또 하나의 화려한 에이전트 프레임워크(agent framework)에 불과한지 아니면 당신의 작업 부하(workload)에 진정으로 중요한 몇 안 되는 프레임워크 중 하나가 될 수 있는지 살펴보겠습니다.

## 에이전트 프레임워크가 약속하는 것 vs. 실제로 제공하는 것

OpenFang 개발팀은 다음과 같이 명확히 말합니다.

"OpenClaw는 좋은 챗봇(chatbot)이지만, 실제 작업을 수행하려고 하면... 그저 할 수 없습니다. 그것이 OpenClaw의 설계 방식입니다."

그리고 이것은 OpenClaw만의 문제가 아닙니다.

저 역시 대안을 찾기 전에 정확히 이러한 좌절감을 겪었습니다.

LangGraph는 콜드 스타트(cold start)에 2.5초가 필요하며 유휴 메모리(idle memory)는 180MB입니다. CrewAI는 200MB에 약 3초가 걸립니다. AutoGen은 이보다 훨씬 무겁습니다.

이러한 모든 프레임워크는 인간이 앉아서 기다리며 프롬프트(prompt)를 입력한다는 가정을 기반으로 구축되었습니다.

이러한 타이밍은 중요합니다. 왜냐하면 산업이 자율 에이전트 작업 부하(autonomous agent workloads)를 향해 빠르게 나아가고 있기 때문입니다. 이는 크론 스케줄(cron schedule)에 따라 실행되는 에이전트, 깨어나서 조사를 수행하는 에이전트, 소셜 미디어 계정을 관리하는 에이전트, 잠자는 동안 리드(lead)를 생성하는 에이전트 등을 의미합니다.

그리고 기존 툴체인(toolchain)은 이러한 목적으로 전혀 구축되지 않았습니다. 이것이 바로 이 방향성이 중요한 이유입니다.

이 글을 읽어주셔서 감사합니다. 저는 에이전틱 서비스형 소프트웨어(Agentic SaaS), 즉 2026년 가장 혁신적인 스타트업(startup)들을 조용히 뒷받침하고 있는 새로운 설계 패턴에 대한 심층 전자책을 집필 중입니다.

이 책은 'Agentic SaaS Patterns Winning in 2026'이라는 제목으로, 다른 곳에서는 찾아볼 수 없는 실제 사례, 아키텍처, 워크플로(workflow)가 가득합니다. 여기서 읽어보실 수 있습니다.

## OpenFang은 정확히 무엇인가요?

OpenFang은 Rust로 처음부터 구축된 오픈 소스(open-source) 에이전트 운영 체제(Agent Operating System)입니다. MIT 라이선스(MIT-licensed)를 따르며, 현재 v0.3.34 버전입니다.

여기서 중요한 점은 이것이 대규모 언어 모델(LLM)을 감싼 파이썬 래퍼(Python wrapper)나 "멀티 에이전트 오케스트레이터(multi-agent orchestrator)"가 아니라는 것입니다.

이는 단일 바이너리(single binary)로 컴파일되는 완전한 운영 체제입니다.

실제로 다음과 같습니다.

```bash
curl -fsSL https://openfang.sh/install | sh
openfang init
openfang start
# 대시보드(Dashboard)는 http://localhost:4200 에서 활성화됩니다.
```

네 가지 명령이면 충분합니다.

당신의 에이전트 인프라(agent infrastructure)가 실행 중입니다.

전체 설치 용량은 약 32MB입니다.

OpenClaw가 약 500MB인 것과 비교해 보세요.

## 핵심 개념: "핸드(Hands)"

OpenFang은 7개의 사전 구축된 자율 에이전트인 "핸드(Hands)"와 함께 제공됩니다. 각 핸드는 독립적으로 실행되는 완전한 기능 패키지입니다.

각 핸드는 `HAND.toml` 매니페스트(manifest), 500단어 이상의 다단계 시스템 프롬프트(system prompt), `SKILL.md` 도메인 전문 지식 참조, 그리고 민감한 작업을 위한 가드레일(guardrails)을 포함하고 있습니다.

```bash
# Researcher Hand 활성화 — 즉시 작동 시작
openfang hand activate researcher

# 언제든지 진행 상황 확인
openfang hand status researcher

# 매일 일정으로 리드 생성 활성화
openfang hand activate lead

# 상태를 잃지 않고 일시 중지
openfang hand pause lead
```

제가 사용해 본 다른 모든 프레임워크들이 에이전트를 "나에게 응답하는 존재"로 취급하는 반면, OpenFang은 에이전트를 "나를 위해 일하는 존재"로 취급한다는 점에서 이 개념이 매우 훌륭하다고 생각합니다.

## 벤치마크는 하나의 이야기만 말할 뿐

수치는 OpenFang 리포지토리(repo)에서 가져왔으며, 공식 문서와 공개 리포지토리에서 출처를 밝혔습니다.

OpenFang과 OpenClaw 간의 콜드 스타트(cold start) 차이는 상당합니다.

메모리(memory) 격차는 거의 10배에 달합니다.

많은 사람들이 가상 사설 서버(VPS)에서 에이전트를 실행하는 경우, 이는 저렴한 서버에서 10개의 에이전트를 실행할 수 있느냐 아니면 인프라(infra)를 업그레이드해야 하느냐의 차이를 만듭니다.

참고로, ZeroClaw(또 다른 Rust 기반 시스템)는 순수 시작 시간(10ms)과 메모리(5MB) 면에서는 OpenFang을 능가하지만, OpenFang의 53가지 도구와 40가지 어댑터(adapter)에 비해 12가지 내장 도구와 15가지 채널 어댑터만 제공합니다.

이러한 트레이드오프(tradeoff)가 당신에게 적합한지는 즉시 사용할 수 있는 기능(out-of-the-box functionality)이 얼마나 필요한지 대 순수 성능에 따라 전적으로 달라집니다.

저는 이전에 다른 OpenClaw 변형들에 대해서도 다루었습니다.

[10달러 하드웨어와 10MB RAM에서 OpenClaw 변형](https://agentnativedev.medium.com/openclaw-variants-on-10-hardware-and-10mb-ram-01e4a3c1a329)

대부분의 제품 버그는 단순한 기능이 특정 서버에 배포될 때 나타납니다.

## 기능 매트릭스(Feature Matrix)

다음은 어떤 기능이 프로덕션 준비(production-ready)되었는지 알려주는 실제 기능별 비교입니다.

이 글을 읽는 엔지니어들에게 가장 중요한 것은 보안(security) 문제입니다.

버전 1.0 미만 프로젝트에 16개의 개별 보안 계층(discrete security layers)이 있다는 것은 놀라운 일입니다.

WASM 이중 계량 샌드박싱(WASM dual-metered sandboxing), 머클 해시 체인 감사 추적(Merkle hash-chain audit trails), Ed25519 서명 에이전트 매니페스트(Ed25519 signed agent manifests), 오염 추적(taint tracking), SSRF 보호(SSRF protection), 비밀 영점화(secret zeroization), 프롬프트 인젝션 스캐닝(prompt injection scanning) 등은 오픈 소스 프로젝트가 아닌 기업 보안 팀에서 기대할 만한 내용입니다.

## 이것이 실제로 어떻게 도움이 될까요?

이처럼 인프라(infrastructure)가 경량화되어 있을 때 자율 에이전트 패턴을 훨씬 더 기꺼이 실험하게 된다는 것을 알게 되었습니다.

실제 적용 사례는 다음과 같습니다.

### 시나리오 1: "설정하고 잊어버리는" 연구원

경쟁사 신제품 출시와 같이 모니터링해야 할 주제가 있습니다.

Researcher Hand를 활성화하고 대상을 지정하면 정해진 스케줄에 따라 실행됩니다.

Researcher Hand는 깨어나서 조사하고, 출처를 상호 참조하고, 신뢰성을 평가하며, 인용된 보고서를 전달합니다.

원할 때 대시보드(dashboard)를 확인하기만 하면 됩니다.

### 시나리오 2: 실제로 작동하는 리드(Lead) 생성

Lead Hand는 매일 실행됩니다.

이상적인 고객 프로필(ideal customer profile)에 맞는 잠재 고객을 발견하고, 웹 데이터로 정보를 보강하며, 점수를 매기고, 기존 연락처와 중복 제거를 한 후 깔끔한 CSV 파일로 전달합니다.

이 예시를 계속 언급하는 이유는, 에이전트 프레임워크가 약속하지만 실제로 제공하지 못하는 종류의 작업이기 때문입니다. 이러한 작업은 보통 프로세스를 계속 주시해야 하기 때문입니다.

### 시나리오 3: 다중 채널 에이전트 배포

이것은 에이전트가 여러 플랫폼에 걸쳐 존재해야 하는 프로젝트를 작업할 때 특히 유용합니다.

OpenFang은 40개의 채널 어댑터(channel adapter)를 제공합니다.

**코어(Core):** 텔레그램(Telegram), 디스코드(Discord), 슬랙(Slack), 왓츠앱(WhatsApp), 시그널(Signal), 매트릭스(Matrix), 이메일(IMAP/SMTP)
**엔터프라이즈(Enterprise):** 마이크로소프트 팀즈(Microsoft Teams), 매터모스트(Mattermost), 구글 챗(Google Chat), 웹엑스(Webex), 페이슈/라크(Feishu/Lark), 줄립(Zulip)
**소셜(Social):** 라인(LINE), 바이버(Viber), 페이스북 메신저(Facebook Messenger), 마스토돈(Mastodon), 블루스카이(Bluesky), 레딧(Reddit), 링크드인(LinkedIn), 트위치(Twitch)
**커뮤니티(Community):** IRC, XMPP, 길디드(Guilded), 리볼트(Revolt), 키베이스(Keybase), 디스코드(Discourse), 기터(Gitter)
**프라이버시(Privacy):** 쓰리마(Threema), 노스트르(Nostr), 머블(Mumble), 넥스트클라우드 톡(Nextcloud Talk), 로켓챗(Rocket.Chat), 엔티파이(Ntfy), 고티파이(Gotify)
**업무 공간(Workplace):** 펌블(Pumble), 플록(Flock), 트위스트(Twist), 딩톡(DingTalk), 잘로(Zalo), 웹훅(Webhooks)

각 어댑터는 채널별 모델 오버라이드(model override), DM/그룹 정책, 속도 제한(rate limiting), 출력 형식 지정(output formatting)을 지원합니다.

에이전트를 구축한 다음 5가지 다른 메시징 플랫폼과 수동으로 통합해야 한다는 것을 깨달았다면, 당신이 잘못한 것이 아닙니다. 다른 프레임워크들이 그 문제를 해결하지 못했을 뿐입니다.

### 시나리오 4: 개인 AI 스택(Stack)

커뮤니티에서 나타나고 있는 한 가지 패턴은 OpenFang을 Paperclip, Obsidian과 같은 도구와 페어링하여 슈퍼 스택(super-stack)으로 사용하는 것입니다. 이는 OpenFang의 자율 핸드(autonomous Hands)를 사용하여 작업을 실행하고, Obsidian이 지식 기반을 관리하며, Paperclip이 이들을 연결하는 방식입니다.

이는 당신의 개인 에이전트 군대가 일상적인 작업을 수행하여 중요한 일에 시간과 돈을 벌어줄 것이라는 약속을 담고 있습니다.

## 기존 프레임워크에서 마이그레이션(Migration)

이미 OpenClaw 설정이 있다면, 단일 명령으로 마이그레이션할 수 있습니다.

```bash
# 모든 것 마이그레이션 — 에이전트, 메모리, 스킬, 설정
openfang migrate --from openclaw

# 먼저 드라이 런(dry run)하여 변경될 내용을 확인
openfang migrate --from openclaw --dry-run
```

마이그레이션 엔진(migration engine)은 에이전트, 대화 기록, 스킬(skills), 설정을 가져옵니다.

OpenFang은 `SKILL.md`를 기본적으로 읽으며 ClawHub 마켓플레이스(marketplace)와 호환됩니다.

## OpenFang 시작하기

직접 시도해보고 싶다면, 가장 쉬운 방법은 다음과 같습니다.

### 설치(Install)

**macOS / Linux**
```bash
curl -sSf https://openfang.sh | sh
```

**Windows (PowerShell)**
```powershell
irm https://openfang.sh/install.ps1 | iex
```

**또는 Cargo를 통해 (Rust 1.75+ 필요)**
```bash
cargo install --git https://github.com/RightNow-AI/openfang openfang-cli
```

**또는 Docker**
```bash
docker pull ghcr.io/RightNow-AI/openfang:latest
```

### 초기화 및 구성(Initialize and Configure)
```bash
openfang init
```

이 명령은 `~/.openfang/`에 기본 설정 파일을 생성합니다. API 키를 설정하세요.

```bash
# 하나 이상 선택
export ANTHROPIC_API_KEY=sk-ant-...
export OPENAI_API_KEY=sk-...
export GROQ_API_KEY=gsk_...  # 무료 티어(Free tier) 사용 가능
```

`~/.openfang/config.toml`을 편집하여 기본 모델을 구성하세요.

```toml
[default_model]
provider = "groq"
model = "llama-3.3-70b-versatile"
api_key_env = "GROQ_API_KEY"

[memory]
decay_rate = 0.05

[network]
listen_addr = "127.0.0.1:4200"
```

### 확인 및 실행(Verify and Launch)
```bash
openfang doctor    # 설정, API 키, 툴체인(toolchain) 확인
openfang start     # 대시보드(Dashboard)는 http://127.0.0.1:4200 에서 활성화됩니다.
```

### 첫 번째 에이전트 생성(Spawn Your First Agent)


### 빠른 명령 참조(Quick Command Reference)

*   `openfang hand activate <name>`: 자율 핸드(autonomous Hand) 활성화
*   `openfang hand status <name>`: 핸드 진행 상황 확인
*   `openfang hand pause <name>`: 상태를 잃지 않고 일시 중지
*   `openfang agent spawn <manifest.toml>`: 사용자 정의 에이전트 생성
*   `openfang agent list`: 모든 에이전트 목록 표시
*   `openfang agent chat <id>`: 에이전트와 대화
*   `openfang migrate --from openclaw`: OpenClaw에서 마이그레이션
*   `openfang skill list`: 번들된 60가지 스킬(skills) 목록 표시
*   `openfang channel setup <channel>`: 메시징 채널 설정

## 결론

에이전트 우선, 채팅 다음이라는 아키텍처(architectural) 결정은 산업이 나아가는 방향에 정확히 부합하며, 180ms의 콜드 스타트(cold start)와 40MB의 유휴 메모리(idle memory)는 적당한 하드웨어(hardware)에서도 많은 에이전트를 실행할 수 있음을 의미합니다.

또한, 가드레일(guardrails)이 포함된 번들형 자율 기능 패키지인 핸드(Hands) 개념이 올바른 추상화(abstraction)라고 생각합니다.

하지만 개발자 경험(developer experience)과 생태계 완성도(ecosystem polish)는 OpenClaw에 비해 현저히 뒤처져 있습니다. 예를 들어, 최신 세대 모델(gpt-5.3-codex)은 아직 모두 지원되지 않습니다.

커뮤니티와 서드파티(third-party) 생태계도 초기 단계이며, 일부 핸드(Hands)는 다른 것들보다 더 성숙하다고 느꼈습니다 (Browser와 Researcher가 가장 많이 테스트되었습니다).

만약 당신이 OpenFang Hands를 실제 프로덕션 워크플로(lead generation, content ops, monitoring)에 배포했다면 댓글로 알려주세요.

이러한 프레임워크 중 하나라도 사용해 본 경험이 있고 의견이 있다면, 특히 데모(demo)가 아닌 실제 운영 시간을 경험했다면 진심으로 당신의 발견을 듣고 싶습니다!
