# article_wiki 병렬 세션 작업 가이드

## 목적

`ai_scrap`의 681개 AI/개발 스크랩을 카테고리별 wiki(`article_wiki`)로 재구성한다. 이 파일은 **Phase 2 병렬 세션(총 10개)이 반드시 먼저 읽어야 할 작업 규칙**이다.

전체 플랜은 `D:/project/ai_wiki/plan.md` 참조.

## 세션 역할

각 세션은 **자기 담당 `ai_scrapN/` 한 폴더만** 순차 처리한다. (N = 1~10)

### 입력

| 파일/폴더 | 역할 |
|---|---|
| `D:/project/ai_wiki/ai_scrapN/` | 자기 담당 원본 스크랩 (폴더명의 숫자 N 기준) |
| `D:/project/ai_wiki/article_wiki/_category_spec.md` | **Phase 1 산출물**. 카테고리 v0 + 초안 키워드. 제일 먼저 읽을 것 |
| `D:/project/ai_wiki/plan.md` | 전체 맥락 (선택적 참고) |

### 출력

1. 원본 `.md` → `article_wiki/{카테고리}/books/{파일명}.md` **이동**
2. `article_wiki/tables/summary_table_N.md` 에 한 줄씩 append

## 금지 사항 (병렬 충돌 방지 — 엄수)

- `wiki_index.md` 직접 수정 금지
- 각 카테고리의 `{카테고리} Index.md` 직접 수정 금지
- 통합 `summary_table.md` 수정 금지
- 다른 세션의 `summary_table_M.md` (M ≠ N) 수정 금지
- 새 카테고리 폴더 생성 금지 — `?` 카테고리 + `NEW` 플래그만 달고 계속 진행
- 원본 `ai_scrapN/` 외의 `ai_scrapM/` 건드리지 말 것

위 중 하나라도 어기면 10세션 병합이 깨진다.

## 처리 절차 (파일 1개당)

### Step 1. 본문 읽기
- `Read` 툴로 `ai_scrapN/{파일명}.md` 전체 읽기 (평균 ~224줄, 한 번에 들어옴)

### Step 2. 카테고리 판정
- `_category_spec.md` 의 v0 정의와 각 카테고리 키워드 세트 기준으로 **정확히 1개** 카테고리 선택
- 2개 이상 걸치면 **주 카테고리 하나만** 고르고, 다른 주제는 **키워드 필드에 반영**
- 어느 것에도 안 맞거나 모호 시:
  - 카테고리 = `?`
  - 플래그 = `NEW`
  - 추천 카테고리명을 요약 뒤에 `(NEW 제안: XXX)` 형태로 메모

### Step 3. 요약 + 키워드 추출
- **요약**: **2~3줄**. 제목 반복 금지. 다음 세 가지를 포함하도록:
  1. 문제/맥락 (어떤 상황에서 다뤄지는 주제인지)
  2. 핵심 주장·방법·결론
  3. 구체적 수치·도구·고유명사가 있으면 1개 이상 반영
- **키워드**: **5~10개, 쉼표 구분. 검색용 키워드만 — 일반어 금지**
  - 키워드 품질 기준은 아래 "키워드 품질 기준" 섹션 **엄수**

### Step 4. 파일 이동

| 상황 | 목적지 |
|---|---|
| 카테고리 확정 | `article_wiki/{카테고리}/books/{파일명}.md` |
| `?` / `NEW` | `article_wiki/_pending/books/{파일명}.md` (폴더 없으면 `mkdir -p` 로 생성 OK — 이 폴더만 예외) |

**이동은 Bash `mv` 사용. 복사 아닌 이동.**

파일명에 **공백·한국어·특수문자**가 있을 수 있으니 반드시 **쌍따옴표로 감싸기**:
```bash
mv "ai_scrapN/20260228_LLM에 지식 그래프 구축을 맡기지 않게 된 이유.md" \
   "article_wiki/Knowledge_Data_Systems/books/"
```

### Step 5. summary_table_N.md 에 한 줄 추가

**파일이 없으면** `Write`로 신규 생성 (헤더 포함):
```markdown
| 파일명 | 요약 | 키워드 | 카테고리 | 플래그 |
|---|---|---|---|---|
```

**파일이 있으면** `Edit`으로 **맨 끝에 한 줄 append** (Write 금지 — 기존 내용 날아감).

한 줄 예시:
```markdown
| 20260228_LLM에 지식그래프 구축을 맡기지 않게 된 이유.md | LLM이 엔터티를 노이즈하게 추출해 실패한 Graph RAG를 도메인 전문가가 정의한 고정 엔터티 아키텍처(FEA) 3계층으로 재설계. PaymentService가 5개 노드로 분열되던 문제를 온톨로지 선고정으로 해결. Accenture의 Irina Adamchic 시리즈(2024~2025) 기반. 마이크로서비스 코드 마이그레이션 플랫폼에 적용. | Fixed Entity Architecture, FEA, Graph RAG, 온톨로지, 마이크로서비스 마이그레이션, Irina Adamchic, Accenture | Knowledge_Data_Systems | OK |
```

요약은 셀 안에서 개행 없이 **문장으로 연결** (표 셀이 깨지지 않도록). 2~3개 문장이면 됨.

## 표 스키마

| 컬럼 | 규칙 |
|---|---|
| 파일명 | `.md` 포함 원본 파일명 (한국어/공백 포함 그대로) |
| 요약 | **2~3문장**. 제목 중복 금지. 표 셀 안에서 **개행 없이** 이어붙이기 |
| 키워드 | 5~10개 쉼표 구분. **검색용만** (아래 품질 기준 참조) |
| 카테고리 | `_category_spec.md` 대분류 폴더명 중 하나 or `?` |
| 플래그 | `OK` / `?` / `NEW` 중 하나 |

## 키워드 품질 기준 (엄수)

이 wiki는 **681개가 넘는 검색 대상**을 가진다. 범용 키워드는 검색 시 노이즈만 늘릴 뿐 쓸모없다. **"이 키워드로 검색했을 때 10개 이하 문서가 걸리면 OK, 100개 이상 걸리면 실패"** 를 기준으로 판단하라.

### ❌ 금지 — 일반어/범용어

다음 부류는 **키워드로 쓰지 말 것** (너무 많은 문서에 해당되어 검색 필터로 쓸모없음):

| 금지 키워드 예시 | 이유 |
|---|---|
| `AI`, `인공지능`, `LLM` (단독) | 거의 모든 문서에 해당 |
| `개발`, `코딩`, `프로그래밍`, `소프트웨어` | 너무 범용 |
| `기술`, `도구`, `시스템`, `아키텍처` (단독) | 맥락 없는 추상어 |
| `방법`, `가이드`, `사용법`, `팁` | 문서 형식 묘사일 뿐 |
| `최신`, `최고`, `혁신`, `강력한` | 마케팅어, 검색 쓸모 0 |

### ✅ 좋은 키워드 — 구체·고유·검색 가능

| 유형 | 예시 |
|---|---|
| **제품·도구 고유명** | `Claude Code`, `Cursor`, `Ollama`, `LM Studio`, `vLLM`, `n8n`, `OpenCode`, `Obsidian` |
| **모델명·버전** | `Gemini 3`, `Claude Opus 4.7`, `Gemma 4`, `DeepSeek R1`, `Kimi K2`, `GPT-5.4` |
| **기법·개념 고유명사** | `프롬프트 캐싱`, `prompt caching`, `ON SCHEMA`, `DDL_DML_ROLE`, `NOINHERIT`, `SET ROLE`, `Fixed Entity Architecture`, `PageIndex`, `HiChunk` |
| **프로토콜·스펙** | `MCP`, `gRPC`, `RSA`, `SSO`, `PAT` |
| **수치·스펙 포함** | `96GB 블랙웰`, `12B 매개변수`, `1200달러 훈련` (검색으로 기억을 유발하는 단위) |
| **인물·조직** | `Karpathy`, `Anthropic`, `Accenture`, `Irina Adamchic` |
| **동의어 쌍** | `패스워드, password, 비밀번호` 처럼 한/영 묶음 (검색 매칭 ↑) |

### 판별 테스트

키워드 하나 뽑을 때마다 스스로 질문:
- "이 단어로 검색했을 때 이 문서가 **왜 떠야 하는지** 한 문장으로 설명 가능한가?"
  - YES → 키워드로 채택
  - NO → 버림

키워드 5~10개 중 **최소 3개 이상**은 **이 문서가 아니면 안 나올 법한 고유 용어**여야 한다.

## 작업 종료 조건 (셀프 체크)

- [ ] `ai_scrapN/` 파일 수 == 0 (전부 이동됨)
- [ ] `summary_table_N.md` 데이터 줄 수 == 원래 `ai_scrapN/` 파일 수
- [ ] `_pending/books/` 에 들어간 파일은 `summary_table_N.md` 에서 `?` + `NEW` 플래그
- [ ] 공유 파일(`wiki_index.md`, `{카테고리} Index.md`, `summary_table.md`) 수정 흔적 없음

위 4개 모두 OK일 때 세션 종료 보고.

## 불확실할 때

- 카테고리 선택이 애매 → **`?` + `NEW` 플래그**로 일단 넘기기. Phase 3에서 처리됨
- 파일이 손상/비어있음 → 요약에 `[손상/빈 파일]`, 키워드 `불량`, 플래그 `?`, 카테고리 `?`
- 카테고리 spec 해석이 다를 경우 → **`_category_spec.md` 가 최종 기준**. 이 파일(CLAUDE.md)의 예시는 참고만.
- 요약에 쓸 핵심이 정말 없음(순수 잡담·일기성 글) → 요약 2줄 최소만, 키워드는 **있는 그대로의 고유명사**만 뽑기. 강제로 키워드 10개 채우지 말 것.
- 영어 원문 그대로인 파일 → 요약은 한국어로, 키워드는 **원문 고유명사(영어)** + **주제 한국어 번역** 둘 다 포함.
