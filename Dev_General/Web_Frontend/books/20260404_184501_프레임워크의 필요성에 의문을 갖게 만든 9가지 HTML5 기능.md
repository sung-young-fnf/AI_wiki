# 프레임워크의 필요성에 의문을 갖게 만든 9가지 HTML5 기능

3년 전, 한 고객을 위해 간단한 내부용 도구를 제작한 적이 있습니다. 직원이 파일을 업로드하고, 데이터를 미리 보며, 양식을 제출하는 평범한 대시보드였습니다.

당시 저는 대부분의 개발자가 그렇듯 당연하게 자바스크립트 프레임워크를 선택했습니다. 상태 관리(State management), 컴포넌트, 번들러, 그리고 마치 NASA의 관제 시스템처럼 복잡한 빌드 파이프라인(Build pipeline)을 구축했습니다. 2주 후 도구는 완성되었습니다.

그런데 그때 한 주니어 개발자가 제게 던진 질문이 저를 멈추게 했습니다.
"왜 이런 도구를 만드는 데 프레임워크를 사용하나요?"

처음에는 웃어넘겼지만, 이후 최신 HTML5 기능만을 사용하여 동일한 도구를 다시 만들어 보았습니다. 결과는 놀라웠습니다. 전체 인터페이스를 구현하는 데 자바스크립트 코드 양이 80%나 줄어든 것입니다. 가상 DOM(Virtual DOM)도, 복잡한 아키텍처도, 의존성(Dependency)의 혼란도 없었습니다. 그저 대부분의 개발자가 무시해 온 브라우저 자체의 기능만을 활용했을 뿐입니다.

이 실험은 제가 웹 애플리케이션을 구축하는 방식을 완전히 바꾸어 놓았습니다. 현대의 브라우저는 놀라울 정도로 강력하지만, 많은 개발자는 여전히 브라우저를 2012년 수준으로 취급하고 있습니다. 프론트엔드 아키텍처를 다시 고민하게 만든 9가지 HTML5 기능을 소개합니다.

### 1. 네이티브 파일 업로드 처리 (Native File Upload Handling)

과거에 파일을 업로드하려면 복잡한 UI 로직을 직접 구현해야 했습니다. 하지만 HTML은 이미 이 문제를 해결했습니다.

```html
<form>
  <label>데이터셋 선택</label>
  <input type="file" accept=".csv,.json">
  <button type="submit">업로드</button>
</form>
```

브라우저는 파일 선택, 유효성 검사, 드래그 앤 드롭 지원, 보안 제한 사항 등을 자동으로 처리합니다. 또한 자바스크립트를 통해 파일에 직접 접근할 수도 있습니다. 별도의 라이브러리 없이도 데이터 도구에 필요한 UI 패키지 전체를 대체할 수 있습니다.

### 2. 내장 폼 제약 조건 (Built-In Form Constraints)

이메일 형식, 입력 길이, 패턴 검사 등의 유효성 검사 로직을 구현하기 위해 수백 줄의 코드를 작성하곤 했습니다. 하지만 이제 HTML이 이 기능을 기본으로 제공합니다.

```html
<form>
  <input type="text" minlength="4" maxlength="12" required>
  <input type="email" required>
  <input type="number" min="1" max="50">
  <button>등록</button>
</form>
```

브라우저는 유효성 검사를 자동으로 강제하며 네이티브 에러 메시지까지 표시합니다. 네이티브 검사는 커스텀 자바스크립트 로직보다 빠르게 실행되므로 웹 접근성과 성능이 즉각적으로 향상됩니다.

### 3. 네이티브 오디오 및 비디오 API

예전에는 비디오를 삽입하려면 플래시(Flash)나 실버라이트(Silverlight) 같은 플러그인이 필요했습니다. HTML5는 이를 완전히 대체했습니다. 브라우저는 이제 재생 제어, 버퍼링, 스트리밍 지원, 접근성 기능을 직접 제공합니다. 미디어 앱을 만들 때 불필요한 의존성을 제거할 수 있습니다.

### 4. 즉각적인 편집을 위한 Contenteditable

어떤 요소든 편집 가능한 필드로 바꿀 수 있는, 마치 반칙처럼 강력한 기능입니다.

```html
<div contenteditable="true">
  이 텍스트를 브라우저에서 직접 수정해 보세요.
</div>
```

최소한의 자바스크립트만으로 편집 내용을 저장할 수 있습니다. 복잡한 위지윅(WYSIWYG) 라이브러리 없이도 간단한 문서 편집기를 구축할 수 있습니다.

### 5. 스마트한 폼을 위한 네이티브 자동 완성 (Native Autocomplete)

자동 완성 기능은 사용자 경험을 크게 향상합니다. `autocomplete` 속성을 사용하면 브라우저에 저장된 사용자 데이터를 활용해 양식을 즉시 채울 수 있습니다. 이는 입력 오류를 줄이고 전환율을 높이며, 추가적인 엔지니어링 없이도 애플리케이션의 완성도를 높여줍니다.

### 6. 실시간 그래픽을 위한 캔버스 (Canvas)

HTML5 캔버스를 사용하면 브라우저 내부에서 직접 픽셀 단위의 조작이 가능합니다. 많은 개발자가 이를 활용해 게임, 데이터 시각화 도구, 이미지 편집기 등을 제작합니다. 시중에 나와 있는 수많은 그래픽 라이브러리는 사실 캔버스를 추상화(Abstraction)한 것에 불과합니다.

### 7. 로컬 스토리지 (Local Storage)

규모가 작은 앱은 백엔드 데이터베이스가 굳이 필요하지 않을 때가 많습니다. HTML5의 로컬 스토리지는 사용자 설정 저장, API 결과 캐싱, 오프라인 설정 등을 위해 도메인당 5~10MB의 저장 공간을 제공합니다. 이는 가벼운 도구를 만들기에 충분한 용량입니다.

### 8. 서비스 워커를 통한 오프라인 지원 (Service Workers)

웹 앱을 인터넷 연결 없이도 작동하게 만드는 마법 같은 기능입니다. 서비스 워커(Service Worker) 내에서 자산을 캐싱하면 웹사이트를 프로그레시브 웹 애플리케이션(PWA)으로 변환할 수 있으며, 이는 마치 네이티브 앱처럼 작동하게 합니다.

### 9. 실시간 통신을 위한 웹소켓 (WebSockets)

웹소켓(WebSockets)은 브라우저와 서버 간의 양방향 실시간 통신을 가능하게 합니다. 이를 통해 실시간 대시보드, 멀티플레이어 게임, 협업 편집 도구와 같은 고수준의 애플리케이션을 브라우저 안에서 구현할 수 있습니다.

### 사고의 전환: 플랫폼 우선주의

프레임워크는 분명 유용하지만, 우리가 이미 가진 능력을 가리기도 합니다. 현대의 브라우저는 저장 시스템, 네트워크 레이어, 렌더링 엔진, GPU 접근, 미디어 파이프라인을 모두 갖춘 하나의 완전한 운영 환경입니다. 이러한 기능을 무시하는 것은 슈퍼컴퓨터를 사놓고 계산기로만 쓰는 것과 같습니다.

새로운 프론트엔드 라이브러리를 설치하기 전에 스스로에게 한 번 물어보세요.
**"브라우저가 이미 이 기능을 제공하고 있지는 않은가?"**

만약 그렇다면 라이브러리를 생략하십시오. 앱은 더 빨라지고, 번들 크기는 줄어들며, 디버깅은 훨씬 쉬워질 것입니다. 플랫폼에서 시작하고, 프레임워크는 플랫폼만으로 해결할 수 없는 진정한 문제가 발생했을 때만 추가하십시오. 대부분의 경우, 프레임워크는 필요하지 않습니다.
