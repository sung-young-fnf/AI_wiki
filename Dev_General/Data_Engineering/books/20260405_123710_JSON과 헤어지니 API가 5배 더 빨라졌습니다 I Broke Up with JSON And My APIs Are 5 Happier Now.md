# JSON과 헤어지니 API가 5배 더 빨라졌습니다 (I Broke Up with JSON (And My APIs Are 5× Happier Now))

### 1. JSON과의 애증 관계

솔직히 말해봅시다. 모든 개발자의 첫 API 응답은 아마 이런 모습일 것입니다.
`{ "message": "hello world" }`

그리고 3년이 지난 후에도 여전히 그 형태를 유지하고 있을 겁니다. 우리는 모두 JSON이 기본값이라는 사실에 암묵적으로 동의했기 때문입니다. JSON은 읽기 쉽고 생산성을 높여주지만, API가 수 메가바이트(MB)에 달하는 중첩 배열을 보내기 시작하고 클라이언트가 비명을 지르기 전까지만 그렇습니다.

실제로 API 응답 시간을 측정해 본 결과, JSON은 마치 함께 놀기에는 즐겁지만 메모리(RAM)를 전부 잡아먹는 친구와 같다는 것을 깨달았습니다. 그래서 저는 아무도 이야기하지 않는 숨겨진 보석 같은 다른 데이터 포맷들을 실험해 보기로 했습니다. 그 결과, API 속도는 5배 빨라졌고 서버의 과부하는 사라졌습니다.

### 2. JSON이 항상 정답은 아닌 이유

JSON은 인간에게는 훌륭하지만, 안타깝게도 API는 당신의 감정을 고려하지 않습니다.

*   **텍스트 중심:** 페이로드(Payload) 용량이 비대해집니다.
*   **데이터 타입의 부재:** 명확히 증명되기 전까지 모든 것은 문자열과 다름없습니다.
*   **느린 파싱 속도:** 대규모 애플리케이션에서는 파싱(Parsing) 속도가 병목 현상을 일으킵니다.

Node.js에서 간단한 테스트를 진행해 보았습니다.
```javascript
console.time("JSON test");
for (let i = 0; i < 100000; i++) {
  JSON.parse(JSON.stringify({ user: "dev", id: i }));
}
console.timeEnd("JSON test");
```
결과는 인생의 선택을 후회하게 만들 정도로 긴 밀리초(ms)를 기록했습니다. 요컨대, JSON은 패스트푸드처럼 편리하지만 그것만 먹고 살 수는 없다는 뜻입니다.

### 3. 메시지팩(MessagePack): JSON의 더 세련된 이진법 사촌

메시지팩(MessagePack)은 운동으로 다져진 JSON이라고 생각하면 됩니다. 이진(Binary) 포맷이며 효율적이고 압축률이 높습니다. JSON이 하는 일을 똑같이 수행하지만, 훨씬 더 빠르고 가볍습니다.

```javascript
const msgpack = require('msgpack5')();
const encoded = msgpack.encode({ hello: "world" });
console.log(encoded); // <Buffer 81 a5 68 65 6c 6c 6f a5 77 6f 72 6c 64>
const decoded = msgpack.decode(encoded);
console.log(decoded); // { hello: 'world' }
```
버퍼(Buffer) 형태가 낯설어 보일 수 있지만, 전송 속도가 훨씬 빠르며 클라이언트 역시 이를 반길 것입니다.

*   **속도 향상:** 직렬화(Serialization) 시 JSON보다 최대 2~4배 빠름.
*   **적합한 분야:** REST API, 모바일 앱, IoT 기기.
*   **단점:** 사람이 직접 읽을 수 없음.

### 4. 프로토콜 버퍼(Protocol Buffers): 기업 수준의 속도가 필요할 때

JSON이 친절한 인턴이라면, 프로토콜 버퍼(Protobuf)는 모든 것에 문서화를 요구하는 수석 엔지니어와 같습니다. 구글이 개발한 이 포맷은 스키마(Schema)를 사용하여 구조화된 데이터를 정의하고 이를 컴팩트한 이진 포맷으로 컴파일합니다.

```protobuf
message User {
  string name = 1;
  int32 id = 2;
}
```
이를 컴파일하여 Node.js, Go, Python 등 다양한 환경에서 사용할 수 있습니다.

*   **장점:** 매우 작고 타입이 명확하며, 언어 간 호환성이 뛰어나고 극도로 빠름.
*   **단점:** 스키마를 컴파일해야 하는 번거로움이 있음.

엔터프라이즈급 API, 마이크로서비스(Microservices) 또는 gRPC를 운영 중이라면 이것이 최종 목적지가 될 것입니다.

### 5. 아브로(Avro): 빅데이터의 비밀 병기

아파치 아브로(Apache Avro)는 빅데이터 엔지니어들이 "JSON을 대체할 더 멋진 것을 만들자"며 탄생시킨 포맷입니다. 스키마 진화(Schema Evolution) 기능을 갖춘 이진 포맷으로, 전체 시스템을 망가뜨리지 않고도 데이터 모델을 업데이트할 수 있습니다.

*   **장점:** 효율적이며 자가 설명적(Self-describing)이고, 카프카(Kafka)나 스파크(Spark) 같은 스트리밍 시스템에 최적화됨.
*   **단점:** 초보자가 접근하기에 진입장벽이 있음.

수백만 건의 이벤트를 처리하거나 기술 회의에서 전문성을 뽐내고 싶을 때 완벽한 선택입니다.

### 6. BSON: MongoDB가 가장 좋아하는 간식

마지막으로 MongoDB에서 사용하는 JSON의 이진 버전인 BSON이 있습니다. 날짜(Date)나 바이너리 같은 추가 데이터 타입을 지원하며 JSON보다 압축률이 좋습니다.

*   **장점:** 빠른 파싱, 데이터베이스 작업에 최적화, Node.js와 훌륭한 궁합.
*   **단점:** 메시지팩(MessagePack)보다는 크기가 약간 더 큼.

BSON은 사실상 MongoDB 스티커를 붙이고 파티에 나타나 우월감을 뽐내는 JSON의 독특한 사촌과 같습니다.

### 7. 요약 및 결론

JSON에서 벗어난 후, 제 API는 마치 구형 피처폰에서 양자 프로세서로 업그레이드한 것 같은 성능을 보여주었습니다.

| 포맷 | 특징 | 최적의 용도 |
| :--- | :--- | :--- |
| **MessagePack** | 작고 빠름, JSON과 유사함 | 일반적인 API, 모바일, IoT |
| **Protobuf** | 강력한 타입, 스키마 기반 | 마이크로서비스, gRPC |
| **Avro** | 스키마 진화에 강함 | 빅데이터, 스트리밍(Kafka) |
| **BSON** | 풍부한 데이터 타입 | MongoDB 기반 프로젝트 |

JSON이 안정적이지만 느린 '혼다 시빅'이라면, 위에 소개한 포맷들은 테슬라나 튜닝된 머스탱과 같습니다. API 성능이 정체되어 있다면 JSON에 대한 맹신을 버리고 이러한 대안들을 시도해 보십시오. 서버가 마침내 원활하게 작동하는 소리를 듣게 될 것입니다.

**"빠른 API는 우연히 만들어지지 않습니다. JSON을 종교처럼 받드는 것을 멈출 때 비로소 가능해집니다."**
