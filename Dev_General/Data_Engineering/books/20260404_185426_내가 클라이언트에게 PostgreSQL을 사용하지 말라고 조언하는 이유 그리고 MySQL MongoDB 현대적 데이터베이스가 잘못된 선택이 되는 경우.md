# 내가 클라이언트에게 PostgreSQL을 사용하지 말라고 조언하는 이유 (그리고 MySQL, MongoDB, '현대적' 데이터베이스가 잘못된 선택이 되는 경우)

데이터베이스 선택에 대해 아무도 말하지 않는 불편한 진실을 공유하고자 합니다. 이는 47건의 실패한 마이그레이션(Migration), 2,300만 달러(약 300억 원)의 비용 낭비, 그리고 10년간 타인의 데이터베이스 재앙을 해결하며 얻은 교훈을 바탕으로 합니다.

많은 이들이 따르는 "베스트 프랙티스"는 종종 틀렸으며, 모두가 추천하는 데이터베이스가 당신의 유스케이스에는 최악일 수 있습니다. 제가 주장하는 논쟁적인 입장은 다음과 같습니다.

*   PostgreSQL이 항상 MySQL보다 나은 것은 아닙니다.
*   NoSQL이 대규모 확장의 유일한 해답은 아닙니다.
*   데이터베이스가 분리된 마이크로서비스는 성능을 파괴할 수 있습니다.
*   "클라우드 네이티브" 데이터베이스는 때로 10배의 비용을 쓰면서도 더 나쁜 결과를 냅니다.
*   최신 유행하는 데이터베이스 기술은 대개 시한폭탄과 같습니다.

저는 지난 10년 동안 수많은 데이터베이스 문제를 해결해 왔습니다. 기업들이 소위 "올바른" 데이터베이스로 수백만 달러를 들여 이전했다가, 이전보다 더 느리고 비싸며 취약한 시스템을 마주하게 된 사례를 47번이나 목격했습니다. 이 글은 그 실패의 기록이며, 당신의 회사를 구할 수 있는 데이터베이스 선택 지침입니다.

### 제1부: 870만 달러를 낭비하고 실패한 PostgreSQL 마이그레이션

한 SaaS 스타트업은 "PostgreSQL이 더 좋다"는 말만 듣고 MySQL에서 이전을 결정했습니다. 18개월 후, 그들은 막대한 손실을 입고 다시 MySQL로 돌아가야 했습니다.

#### 왜 실패했는가?

**1. 과도한 업데이트 워크로드에서의 MVCC 블로트(Bloat)**
해당 시스템은 CRM으로, 레코드 업데이트가 빈번했습니다. PostgreSQL은 다중 버전 동시성 제어(MVCC)를 사용합니다. 모든 업데이트는 새로운 행 버전을 생성하며, 오래된 버전은 진공(VACUUM) 작업이 청소할 때까지 쌓입니다. 분당 5만 건의 업데이트가 발생하자 진공 작업이 따라가지 못했고, 테이블 크기는 80GB에서 280GB로 부풀어 올랐으며 쿼리 성능은 급격히 저하되었습니다. 반면 MySQL(InnoDB)은 가능할 경우 제자리 업데이트(In-place update)를 수행하므로 이런 문제가 발생하지 않았습니다.

**2. 커넥션 오버헤드(Connection Overhead)**
MySQL은 커넥션 생성 비용이 낮고 메모리 점유가 적지만(커넥션당 ~256KB), PostgreSQL은 커넥션당 약 10MB의 메모리를 사용하며 생성 시간도 6배 더 느립니다. 커넥션 풀링(Connection Pooling) 도구인 PgBouncer 없이 마이그레이션한 결과, CPU 시간의 40%가 커넥션 관리에 낭비되었습니다.

**3. 복제 지연(Replication Lag)**
MySQL의 바이너리 로그 복제는 성숙하고 최적화되어 있습니다. 반면 PostgreSQL의 WAL 기반 복제는 해당 기업의 높은 업데이트 속도를 견디지 못해 리플리카(Replica) 서버의 데이터가 수십 초씩 지연되는 현상이 발생했습니다.

### 제2부: 데이터 무결성을 파괴한 MongoDB 마이그레이션

한 이커머스 기업은 "확장성"을 위해 MongoDB를 선택했습니다. 하지만 그 결과는 데이터 부패, 트랜잭션 손실, 그리고 고객 소송이었습니다.

#### 주요 재앙
*   **주문 유실:** 애플리케이션과 DB 사이의 네트워크 문제로 주문은 생성되었으나 주문 항목(Items)이 저장되지 않는 사례가 빈번했습니다.
*   **재고 데이터 오염:** 데이터베이스 수준의 트랜잭션 보장이 없어 경합 상태(Race condition)가 발생했고, 실제 판매량과 재고 수치가 일치하지 않게 되었습니다.
*   **고객 잔액 오류:** 환불 과정에서 주문 상태 업데이트와 잔액 충전이 원자적(Atomic)으로 이루어지지 않아 수천 명의 고객 계정 잔액이 틀어졌습니다.

**교훈:** MongoDB는 유연한 스키마가 필요한 콘텐츠 관리 시스템(CMS)이나 이벤트 로그에는 훌륭하지만, 데이터 무결성이 중요한 금융 트랜잭션이나 재고 관리에는 부적합합니다. 관계형 DB(RDBMS)가 필요한 곳에는 관계형 DB를 사용하십시오.

### 제3부: 마이크로서비스 데이터베이스의 악몽

"서비스당 하나의 데이터베이스"라는 원칙을 맹목적으로 따르다가 분산 데이터 문제로 파산 위기까지 몰린 핀테크 기업의 사례입니다.

#### 발생한 문제들
1.  **JOIN 쿼리 불가:** 과거 45ms면 충분했던 복합 데이터 조회가 여러 서비스를 호출하고 애플리케이션에서 조합하는 방식으로 바뀌면서 187ms로 느려졌습니다.
2.  **데이터 일관성 유지의 어려움:** 단일 DB에서는 ACID 트랜잭션 한 번으로 끝날 일이 여러 서비스에 걸친 분산 트랜잭션 문제가 되었습니다. 결제는 성공했는데 재고 차감이 실패하는 등의 시나리오를 해결하기 위한 보상 트랜잭션(Saga 패턴 등) 구현은 지옥에 가깝습니다.
3.  **쿼리 성능 저하:** 리포트 작성을 위해 수만 건의 데이터를 서비스 간 통신으로 가져오면서 네트워크 호출이 폭증했고, 실행 시간이 수백 배 증가했습니다.

**해결책:** 마이크로서비스 아키텍처를 유지하되, 트랜잭션 공유가 필수적인 핵심 서비스들은 데이터베이스를 공유하는 것이 현명합니다.

### 제4부: 10배 더 비싼 "클라우드 네이티브" 데이터베이스

비용 절감을 위해 도입한 Aurora Serverless가 오히려 비용을 10배 폭증시킨 사례입니다.

*   **ACU 과금의 함정:** Aurora Serverless는 트래픽 증가 시 빠르게 확장(Scale-up)되지만, 축소(Scale-down)는 매우 보수적입니다. 단 15분간의 스파이크 때문에 이후 몇 시간 동안 높은 비용을 지불하게 됩니다.
*   **커넥션 변동:** 커넥션 풀링이 없는 애플리케이션은 잦은 커넥션 생성을 부하 급증으로 오판하게 만들어 불필요한 확장을 유도합니다.

**교훈:** 트래픽이 어느 정도 예측 가능하다면, 고정된 크기의 RDS 인스턴스를 사용하는 것이 비용과 성능 면에서 훨씬 유리합니다.

### 제5부: 카산드라(Cassandra) 재앙

"웹 스케일(Web Scale)"이라는 마케팅 용어에 속아 8TB의 데이터를 위해 Cassandra를 도입한 애드테크 기업은 운영 난이도와 비용 문제로 고통받았습니다.

*   **일관성 문제:** 읽기/쓰기 시 복제본 간의 데이터 불일치로 인해 고객 리포트 숫자가 조회할 때마다 달라지는 문제가 발생했습니다.
*   **쿼리 제약:** JOIN이 불가능하고 파티션 키가 없는 컬럼으로 필터링이 안 되어, 동일한 데이터를 23개의 다른 테이블 디자인으로 중복 저장해야 했습니다.
*   **운영 부담:** 노드 장애 시 복구 과정이 매우 복잡하며 전문 인력이 필수적입니다.

결국 이 기업은 PostgreSQL의 시간 기반 파티셔닝(Time-based Partitioning)으로 전환했고, 운영비용을 60% 절감하면서도 더 나은 성능을 얻었습니다.

### 제6부: 데이터베이스 선택을 위한 조언

2,300만 달러의 낭비를 지켜본 끝에 내린 결론은 다음과 같습니다.

1.  **PostgreSQL이나 MySQL로 시작하십시오.** 40년 이상 검증된 기술이며, 거의 모든 유스케이스를 처리할 수 있습니다. 구체적이고 측정 가능한 이유가 없다면 SQL 데이터베이스를 사용하십시오.
2.  **한계에 부딪히기 전에는 마이그레이션하지 마십시오.** "남들이 좋다고 해서"나 "언젠가 확장해야 하니까"는 이유가 될 수 없습니다. 현재 DB의 인덱스 최적화와 쿼리 튜닝이 먼저입니다.
3.  **지루한(Boring) 기술이 좋은 기술입니다.** 검증되고 생태계가 풍부한 기술을 선택하십시오. 페이스북, 인스타그램, 깃허브 같은 거대 서비스들도 여전히 MySQL과 PostgreSQL을 사용합니다.
4.  **최적화가 확장보다 저렴합니다.** 마이그레이션에 12개월을 쓰는 것보다 느린 쿼리를 찾아 인덱스를 추가하는 1주일이 훨씬 가치 있습니다.
5.  **"웹 스케일"은 마케팅 용어일 뿐입니다.** 당신의 트래픽이 인스타그램보다 적다면, 일반적인 SQL 데이터베이스로도 충분히 감당할 수 있습니다.

### 데이터베이스 선택 프레임워크

*   **1단계:** PostgreSQL이나 MySQL을 기본값으로 정합니다. (복잡한 쿼리/JSON은 Postgres, 단순하고 빠른 쓰기는 MySQL)
*   **2단계:** 특수한 이유가 있을 때만 대안을 고려합니다. (완전 유연한 스키마는 MongoDB, 전 세계적 가용성은 Cassandra 등)
*   **3단계:** 마이그레이션 전 실제 데이터로 성능, 비용, 복잡성을 측정하십시오.
*   **4단계:** 반드시 롤백(Rollback) 계획을 세우십시오.

트렌드를 쫓지 말고 데이터를 따르십시오. 가장 현대적인 데이터베이스가 반드시 당신의 문제를 해결해 주지는 않습니다.
