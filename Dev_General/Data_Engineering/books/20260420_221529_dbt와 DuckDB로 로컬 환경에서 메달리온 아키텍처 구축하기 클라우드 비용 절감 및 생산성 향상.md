# dbt와 DuckDB로 로컬 환경에서 메달리온 아키텍처 구축하기: 클라우드 비용 절감 및 생산성 향상

치솟는 클라우드 비용 시대에 dbt와 DuckDB를 활용하여 로컬 환경에서 메달리온 아키텍처(Medallion Architecture)를 구축하면, 프로덕션 수준의 고성능 데이터 스택을 비용 없이 개발할 수 있습니다.

본 문서에서는 로컬 머신에 견고한 데이터 파이프라인을 설정하는 방법에 대해 자세히 다룰 것입니다. 다룰 내용은 다음과 같습니다.

*   메달리온 아키텍처(Medallion Architecture)와 DuckDB의 근본적인 '이유'
*   소매 판매 데이터를 다루는 실제 비즈니스 시나리오
*   로컬 ELT(Extract, Load, Transform) 파이프라인 설계
*   DuckDB로 데이터를 추출하고 로드하기 위한 Python 스크립트 작성
*   dbt를 사용하여 브론즈(Bronze), 실버(Silver), 골드(Gold) 레이어 구축

바로 시작해 봅시다!

## 왜 메달리온 아키텍처인가?

메달리온 아키텍처(Medallion Architecture)는 레이크하우스(Lakehouse) 내에서 데이터를 논리적으로 구성하는 데 사용되는 데이터 설계 패턴입니다. 목표는 간단합니다. 시스템을 통해 흐르는 데이터의 품질을 개선하는 것입니다.

이것은 수질 정화 시스템과 같다고 생각할 수 있습니다. 강에서 '원수(raw water)'를 가져와(브론즈), 침전물과 화학 물질을 걸러내고(실버), 마지막으로 최종 소비자를 위해 미네랄을 추가하거나 탄산화합니다(골드).

이러한 단계를 분리함으로써, 최종 단계에서 계산이 잘못되더라도 더 많은 물을 얻기 위해 강으로 돌아갈 필요가 없습니다. 이미 필터링된 '실버(Silver)' 물을 다시 처리하기만 하면 됩니다.

## 왜 클라우드 대신 DuckDB인가?

DuckDB는 인-프로세스(in-process) SQL OLAP 데이터베이스 관리 시스템입니다. 쉽게 말해, SQLite와 유사하지만 분석 워크로드(analytical workloads)에 특화되어 구축되었습니다.

DuckDB는 별도의 구성이 필요 없는 Python 라이브러리 또는 바이너리(binary)이므로 설정이 간단하며, Parquet, CSV, JSON과 같은 최신 형식을 기본적으로 지원합니다.

가장 중요한 것은 로컬에서 실행되므로 지연 시간이나 비용에 대해 걱정할 필요가 없으며, Wi-Fi 없이도 완벽하게 작동합니다.

도구들을 이해했으니, 이제 우리가 다룰 비즈니스 시나리오를 살펴보겠습니다.

## 비즈니스 사례: 커피 로스터리 고객

고객은 데이터 저장의 어려움을 설명했습니다. 그들은 POS(Point of Sale) 시스템에서 CSV 파일로 판매 데이터를 가지고 있고, 창고에서 JSON 기반의 재고 목록을 가지고 있습니다.

그들은 일일 매출과 재고가 부족한 원두를 알고 싶어 하지만, 아직 대규모 빅쿼리(BigQuery)나 스노우플레이크(Snowflake)를 구축할 예산은 없습니다.

따라서, 당신은 그들을 위해 로컬 메달리온 파이프라인을 구축할 것을 제안했습니다.

고객을 돕기 위해, 먼저 CSV 및 JSON 파일을 추출하여 로컬 DuckDB 데이터베이스에 로드하는 것으로 시작할 것입니다. 그 후 dbt를 사용하여 변환을 처리합니다. 목표는 고객이 사용할 판매 총액과 현재 재고 수준을 구체적으로 추적하는 구조화된 '골드(Gold)' 레이어를 생성하는 것입니다.

## 아키텍처: 로컬 ELT 파이프라인

우리 파이프라인은 최신 ELT(Extract, Load, Transform) 패턴을 따릅니다. 먼저 원본 데이터를 DuckDB에 로드한 다음, dbt가 모든 변환 작업을 처리하도록 할 것입니다.

*   **추출(Extraction):** 소스에서 데이터를 가져오기 위한 Python 스크립트 작성.
*   **로드(Loading):** 데이터는 `.duckdb` 파일에 지속(persisted)됩니다.
*   **변환(Transformation):** dbt가 메달리온 레이어를 통해 데이터를 이동시키는 작업을 수행합니다.

## 1단계: 추출 및 로드 (E와 L)

먼저 데이터를 우리 환경으로 가져와야 합니다. 이 예시에서는 일일 판매 정보가 POS 시스템에서 CSV 파일로 저장되고, 재고 데이터는 별도의 소스에서 JSON 형식으로 제공된다고 가정합니다.

### 추출 코드

Python 스크립트가 데이터 수집을 처리합니다. DuckDB는 소스 파일에서 직접 테이블로 데이터를 로드할 수 있도록 해주므로 이 과정을 단순화합니다.

```python
import duckdb
import pandas as pd

# Connect to (or create) a local DuckDB file
con = duckdb.connect('coffee_shop.duckdb')

# Load Sales Data (CSV)
con.execute("""
    CREATE TABLE IF NOT EXISTS raw_sales AS 
    SELECT * FROM read_csv_auto('data/daily_sales.csv')
""")

# Load Inventory Data (JSON)
con.execute("""
    CREATE TABLE IF NOT EXISTS raw_inventory AS 
    SELECT * FROM read_json_auto('data/inventory.json')
""")

print("Successfully loaded raw data into DuckDB!")
con.close()
```

DuckDB를 사용하면 CSV, JSON, Parquet과 같은 파일을 미리 스키마(schema)를 정의하지 않고도 직접 읽을 수 있다는 장점이 있습니다. 반면 PostgreSQL과 같은 시스템은 데이터를 로드하기 전에 스키마를 설정해야 합니다.

우리는 DuckDB의 `read_csv_auto` 및 `read_json_auto` 함수를 사용합니다. 이들은 "스마트" 함수로, 데이터를 검사하여 열 유형을 추측(스키마 추론, schema inference)하고, 각 열을 수동으로 정의할 필요 없이 데이터 수집을 처리합니다.

## 2단계: dbt 설정

변환 작업을 시작하기 전에 먼저 dbt를 설정해야 합니다.

### 1. dbt 프로젝트 초기화

```bash
dbt init my_coffee_shop_project
```

터미널에서 위 명령어를 실행하면 메인 폴더 내에 dbt 프로젝트가 생성됩니다. 터미널에서 데이터베이스 유형을 묻는 메시지가 나타나면 `duckdb`를 선택하세요. 이렇게 하면 `models`, `seeds`, `tests`를 포함한 표준 폴더 구조가 생성됩니다.

### 2. `profiles.yml` 구성

이는 가장 중요한 단계입니다. dbt는 데이터베이스에 연결하는 방법을 이해하기 위해 `profiles.yml`이라는 파일(일반적으로 `~/.dbt/` 폴더에 숨겨져 있음)을 찾습니다. DuckDB의 경우 구성은 매우 간단합니다.

`profiles.yml`을 열고 다음과 같이 설정합니다.

```yaml
my_coffee_shop_project:
  outputs:
    dev:
      type: duckdb
      path: 'path/to/your/coffee_shop.duckdb'  # Point this to your local file
      threads: 4
  target: dev
```

루트 디렉토리에 있는 `coffee_shop.duckdb` 파일 경로를 제공해야 합니다. 이 파일은 Python 추출 스크립트에 의해 생성된 데이터베이스 파일입니다. dbt가 이 경로를 가리키도록 설정하면 Python 스크립트와 dbt 모두 동일한 로컬 웨어하우스(warehouse)를 공유할 수 있습니다.

### 3. 소스 정의

1단계 실행 후 `models` 폴더가 생성된 것을 볼 수 있습니다. 해당 폴더 안에 `sources.yml` 파일을 생성하세요. 이 파일은 dbt의 맵(map) 역할을 합니다.

```yaml
version: 2

sources:
  - name: raw_data
    schema: main # DuckDB defaults to the 'main' schema
    tables:
      - name: raw_sales
      - name: raw_inventory
```

이러한 소스를 정의함으로써, 나중에 SQL 모델 변환에서 `{{ source('raw_data', 'raw_sales') }}` 매크로(macro)를 사용할 수 있으며, 이는 원본 데이터와 변환 로직(transformation logic) 간의 공식적인 연결을 생성할 수 있습니다.

이제 '배관'이 설치되었고 dbt가 DuckDB와 통신하므로, 즐거운 부분인 변환 레이어 구축으로 넘어갈 수 있습니다.

## 3단계: dbt를 이용한 변환 (T)

모든 변환 파일은 `models` 폴더 아래에 생성됩니다. 구조는 다음과 같습니다.

```
models/
├── bronze/
├── silver/
└── gold/
```

### 브론즈(Bronze) 레이어

이 레이어에서는 데이터가 소스와 정확히 일치하도록 합니다. 오타를 수정하거나 데이터 유형을 변경하는 것은 피합니다. 유일한 수정 사항은 데이터 수집 프로세스를 추적하는 데 도움이 되는 '로드 시간(`_ingested_at`)'과 같은 메타데이터(metadata) 열을 추가하는 것입니다.

`models` 폴더 안에 `models/bronze/brz_sales.sql` 파일을 생성하세요.

```sql
{{ config(materialized='table') }}

SELECT 
    *,
    current_timestamp as _ingested_at
FROM {{ source('raw_data', 'raw_sales') }}
```

### 실버(Silver) 레이어

이제 데이터를 정리할 차례입니다. 여기서 핵심 작업이 이루어집니다. 문자열을 날짜로 캐스팅(cast)하고, Null 값을 처리하며, 테이블을 조인(join)하여 엔티티(entity)를 생성합니다.

모델: `models/silver/slv_sales_cleaned.sql`

```sql
{{ config(materialized='view') }}

WITH base_sales AS (
    SELECT 
        CAST(transaction_id AS INT) as transaction_id,
        CAST(sale_date AS DATE) as sale_date,
        product_id,
        CAST(quantity AS INT) as quantity,
        CAST(price AS DECIMAL(10,2)) as price
    FROM {{ ref('brz_sales') }}
    WHERE transaction_id IS NOT NULL
)

SELECT 
    s.*,
    (s.quantity * s.price) as total_line_amount
FROM base_sales s
```

### 골드(Gold) 레이어

이제 비즈니스에 집중할 시간입니다. 골드(Gold) 레이어에서 필요한 것은 보고용 테이블입니다. 이 테이블들은 고도로 집계(aggregated)되어 있으며 BI 도구(BI tool)나 대시보드(dashboard)에 바로 사용할 수 있습니다. 생성하는 테이블은 특정 비즈니스 질문에 답하는 데 도움이 될 수 있습니다.

예를 들어, 일일 총수익은 얼마인가요?

모델: `models/gold/gld_daily_revenue.sql`

```sql
{{ config(materialized='table') }}

SELECT 
    sale_date,
    SUM(total_line_amount) as daily_revenue,
    count(distinct transaction_id) as total_transactions
FROM {{ ref('slv_sales_cleaned') }}
GROUP BY 1
ORDER BY 1 DESC
```

골드 레이어는 데이터 분석가들의 작업을 단순화하도록 설계되었습니다. 최종 테이블이 보고용으로 준비될 수 있도록 데이터 조인 및 집계와 같은 복잡한 작업을 여기서 처리합니다.

이러한 작업을 미리 수행함으로써, 분석가들은 분석을 실행할 때마다 복잡한 조인이나 집계를 반복할 필요가 없습니다.

## 4단계: 파이프라인 실행

변환 준비가 끝났습니다. 이제 작동하는지 테스트할 시간입니다. dbt가 DuckDB와 통신하려면 `dbt-duckdb` 어댑터(adapter)가 필요합니다. `pip install dbt-duckdb` 명령어를 통해 pip으로 설치할 수 있습니다.

파이프라인 실행은 단일 명령어로 이루어지며, `profiles.yml`이 `coffee_shop.duckdb` 파일을 가리키고 있는지 확인해야 합니다.

```bash
dbt run
```

이를 실행하면 dbt는 의존성(Bronze -> Silver -> Gold)을 확인하고, 로컬 DuckDB 파일에 필요한 뷰(view)와 테이블을 생성하며, 실행 요약을 제공합니다.

그 후 DuckDB CLI 또는 DBeaver와 같은 도구를 사용하여 최종 골드 테이블이 성공적으로 생성되었는지 확인할 수 있습니다.

## 보너스

브론즈(Bronze) 및 골드(Gold) 레이어에는 테이블을 사용하고 실버(Silver) 레이어에는 뷰(view)를 사용하는 것을 눈치챘을 수도 있습니다. 그 차이점은 다음과 같습니다.

테이블은 물리적으로 데이터를 디스크에 저장하는 구체화(materialized)된 형태입니다. 공간을 차지하지만 쿼리(query) 속도가 매우 빠릅니다.

반대로 뷰는 저장된 쿼리일 뿐입니다. 호출할 때만 DuckDB가 코드를 실행하므로 공간을 차지하지 않습니다.

브론즈에서는 원본 데이터의 물리적 백업을 확보하기 위해 테이블을 사용합니다. 골드에서도 최종 결과가 미리 계산되어 분석가들이 빠르게 사용할 수 있도록 테이블을 사용합니다.

반면 실버는 정리를 위한 전환 레이어 역할을 하므로, 변환을 처리하는 동안 공간을 절약하기 위해 뷰를 사용할 수 있습니다.

## 결론

이 글이 dbt와 DuckDB를 사용하여 자신의 머신에 메달리온 아키텍처(Medallion Architecture)를 설정하는 데 자신감을 주었기를 바랍니다. 강력한 데이터 웨어하우스(data warehouse)를 운영하기 위해 복잡한 클라우드(cloud) 설정이 필요하지 않으며, 여러분의 노트북만으로도 충분히 시작할 수 있습니다.
