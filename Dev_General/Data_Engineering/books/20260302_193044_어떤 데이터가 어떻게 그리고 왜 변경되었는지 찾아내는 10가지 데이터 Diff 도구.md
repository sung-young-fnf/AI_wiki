엔지니어는 대시보드(dashboard)를 열어봅니다. 어제: 250만 달러. 오늘: 150만 달러. 하지만 파이프라인(pipeline)은 모두 초록색 체크 표시를 보여줍니다. 오류도, 알림도 없으며, 아무것도 실패하지 않았습니다.

하지만 분명 뭔가 바뀌었습니다. 파이프라인 어딘가에서 — 필터(filter)가 더 엄격해졌거나, `JOIN` 조건이 변경되었거나, 원본 데이터(source data)가 이동했습니다. 하지만 어디에서 말이죠?

데이터 차이 분석(data diffing) 없이는, 근본 원인을 찾는다는 것은 수백만 개의 행(row)을 수동으로 비교하고, 모든 변환(transformation)을 확인하며, 건초 더미에서 바늘을 찾기를 바라는 것과 같습니다.

이것이 바로 데이터 차이 분석(data diff) 문제입니다. 코드 차이 분석(code diffs)은 어떤 내용이 줄 단위(line-by-line)로 변경되었는지 정확히 보여줍니다. 데이터 차이 분석(data diffs)도 마찬가지로 — 데이터셋(datasets)에 대해 동일한 작업을 수행해야 합니다. 어떤 행이 추가, 삭제 또는 수정되었는지 보여줍니다. 열 값(column value)의 변경 사항을 강조 표시합니다. 스키마 드리프트(schema drifts)를 식별합니다. 그리고 이상적으로는 그 이유를 설명해야 합니다.

데이터 차이 분석 도구(data diff tools)는 이를 가능하게 합니다. 이 도구들은 대규모로 데이터셋을 비교하고, 의미 있는 변경 사항을 표면화하며, 노이즈(noise)를 필터링하고, 문제의 근원을 추적하는 데 도움을 줍니다. 최고의 도구들은 CI/CD(지속적 통합/지속적 배포)와 통합되어 프로덕션(production) 단계 이전에 문제를 감지합니다.

하지만 모든 차이 분석 도구가 동일하게 만들어진 것은 아닙니다. 일부는 작은 데이터셋만 처리합니다. 다른 일부는 스키마(schema) 차이로 어려움을 겪습니다. 가장 우수한 도구들은 수십억 개의 행으로 확장(scale)하면서도 사람이 읽을 수 있는 출력(human-readable output)을 제공합니다.

## 데이터 차이 분석(Data Diffing)이 어려운 이유

도구들을 살펴보기 전에, 데이터셋을 비교하는 것이 겉보기보다 어려운 이유를 설명합니다.

*   **규모(Scale)** — 100개의 행을 비교하는 것은 사소합니다. 1억 개의 행을 비교하는 것은 정교한 알고리즘(algorithms)과 상당한 컴퓨팅 자원(compute)을 필요로 합니다.
*   **스키마 진화(Schema evolution)** — 프로덕션 데이터는 스키마를 변경합니다. 좋은 차이 분석 도구는 열 추가, 이름 변경 및 유형 변경을 유연하게 처리합니다.
*   **부동 소수점 정밀도(Floating point precision)** — 0.1 + 0.2는 정확히 0.3과 같지 않습니다. 차이 분석 도구는 숫자 비교를 위한 허용 오차 임계값(tolerance thresholds)이 필요합니다.
*   **정렬 순서(Sort order)** — 두 데이터셋은 동일하지만 순서가 다를 수 있습니다. 도구는 행을 올바르게 매칭하기 위해 기본 키(primary keys)가 필요합니다.
*   **NULL 처리(NULL handling)** — `NULL`이 `NULL`과 같을까요? 시스템마다 다른 의미 체계(semantics)를 가집니다.
*   **성능(Performance)** — 단순 비교는 O(n²)입니다. 대규모 데이터셋에는 해시 기반(hash-based) 또는 병합 조인(merge-join) 접근 방식이 필요합니다.
*   **출력 가독성(Output readability)** — "100만 개의 행이 다릅니다"라고만 보여주는 것은 도움이 되지 않습니다. 좋은 차이 분석은 샘플 차이(sample differences)와 요약 통계(summary statistics)를 보여줍니다.

좋은 데이터 차이 분석 도구는 다음과 같은 방법으로 이러한 어려움을 처리합니다.

*   행 매칭을 위해 기본 키(primary keys) 사용
*   스키마(schema) 차이 지원
*   구성 가능한 허용 오차(tolerances) 제공
*   대규모 차이 집합을 지능적으로 샘플링(sampling)
*   실행 가능한 보고서(reports) 생성
*   버전 관리(version control) 및 CI/CD(지속적 통합/지속적 배포)와 통합

### 1. datacompy

PyPI: [datacompy](https://pypi.org/project/datacompy/)
**기능:** 상세 보고서와 함께 `pandas` `DataFrame`과 `Spark` `DataFrame`을 비교합니다.

`datacompy`는 사람이 읽을 수 있는 출력과 함께 직관적인 `DataFrame` 비교를 제공합니다.

**기본 비교:**

```python
import pandas as pd
import datacompy
# Production data (yesterday)
df1 = pd.DataFrame({
    'order_id': [1, 2, 3, 4],
    'user_id': [100, 101, 102, 103],
    'amount': [150.0, 200.0, 175.5, 225.0],
    'status': ['completed', 'completed', 'pending', 'completed']
})
# Production data (today)
df2 = pd.DataFrame({
    'order_id': [1, 2, 3, 5],  # Order 4 deleted, Order 5 added
    'user_id': [100, 101, 102, 104],
    'amount': [150.0, 205.0, 175.5, 300.0],  # Order 2 amount changed
    'status': ['completed', 'completed', 'completed', 'completed']  # Order 3 status changed
})
# Compare
compare = datacompy.Compare(
    df1,
    df2,
    join_columns='order_id',  # Primary key
    abs_tol=0.01,  # Tolerance for floats
    rel_tol=0.0,
    df1_name='Yesterday',
    df2_name='Today'
)
# Print report
print(compare.report())
```

**Spark 비교 (대규모 데이터셋):**

```python
from pyspark.sql import SparkSession
import datacompy
spark = SparkSession.builder.getOrCreate()
# Large Spark DataFrames
df1_spark = spark.read.parquet('s3://bucket/orders_yesterday.parquet')
df2_spark = spark.read.parquet('s3://bucket/orders_today.parquet')
# Compare (distributed computation)
compare = datacompy.SparkCompare(
    spark,
    df1_spark,
    df2_spark,
    join_columns=['order_id'],
    abs_tol=0.01
)
# Report (computed in parallel)
print(compare.report())
# Get mismatched rows
mismatches = compare.rows_both_mismatch
mismatches.show()
```

**CI/CD 통합:**

```python
# tests/test_data_diff.py
import datacompy
def test_prod_vs_dev_data():
    """개발(dev) 변환(transformation) 결과를 프로덕션(prod)과 비교"""
    
    prod_df = load_prod_data()
    dev_df = run_transformation_in_dev()
    
    compare = datacompy.Compare(
        prod_df,
        dev_df,
        join_columns='order_id',
        abs_tol=0.01
    )
    
    # 너무 많은 차이가 있으면 실패
    match_rate = compare.count_matching_rows() / len(prod_df)
    assert match_rate > 0.99, f"Only {match_rate:.1%} of rows match"
    
    # 스키마(schema)가 변경되면 실패
    assert compare.columns_match(), "Schema mismatch detected"
```

### 2. data-diff

GitHub: [datafold/data-diff](https://github.com/datafold/data-diff)
**기능:** 체크섬(checksum)을 사용하여 대규모 SQL 데이터베이스에 대한 빠른 차이 분석.

`data-diff`는 해싱(hashing)을 사용하여 데이터베이스 간 테이블을 효율적으로 비교합니다.

**설치:**

```bash
pip install data-diff
```

**기본 사용법:**

```bash
data-diff \
  postgresql://user:pass@host1/db1 orders \
  postgresql://user:pass@host2/db2 orders \
  --key-column order_id \
  --columns amount,status
```

**출력:**

```
차이 요약:
  총 행 수: 1,234,567
  DB1에만 있는 행: 123 (0.01%)
  DB2에만 있는 행: 89 (0.007%)
  차이가 있는 행: 456 (0.037%)
샘플 차이 (상위 10개 표시):
  order_id | amount (DB1) | amount (DB2) | status (DB1) | status (DB2)
  ---------|--------------|--------------|--------------|-------------
  12345    | 150.00       | 155.00       | completed    | completed
  12346    | 200.00       | 200.00       | pending      | completed
  ...
전체 차이 분석 출력을 보려면 --debug와 함께 실행하십시오.
```

**고급 기능:**

```python
# Python API
from data_diff import connect_to_table, diff_tables
# Connect to tables
table1 = connect_to_table(
    "postgresql://host1/db1",
    "orders",
    key_columns=("order_id",)
)
table2 = connect_to_table(
    "postgresql://host2/db2", 
    "orders",
    key_columns=("order_id",)
)
# Diff with options
diff = diff_tables(
    table1,
    table2,
    threaded=True,  # Parallel processing
    max_diff_rows=10000,  # Limit output size
    bisection_threshold=1000000,  # Use bisection for large diffs
    bisection_factor=16  # Parallelize into 16 segments
)
# Get results
for row in diff:
    print(f"Difference: {row}")
```

### 3. Apache Griffin

웹사이트: [griffin.apache.org](https://griffin.apache.org/)
**기능:** 데이터 품질(data quality) 및 조정 프레임워크(reconciliation framework).

`Griffin`은 행 수준 차이 분석(row-level diffing)을 포함한 데이터 품질 유효성 검사를 제공합니다.

**데이터 소스 정의:**

```json
{
  "name": "orders_source",
  "connectors": [
    {
      "type": "HIVE",
      "version": "2.3",
      "config": {
        "database": "production",
        "table": "orders"
      }
    }
  ]
}
```

**측정 지표 정의 (정확도 확인):**

```json
{
  "name": "orders_accuracy",
  "measure.type": "accuracy",
  "process.type": "batch",
  "data.sources": [
    {
      "name": "source",
      "connectors": [{"type": "HIVE", "database": "staging", "table": "orders"}]
    },
    {
      "name": "target",
      "connectors": [{"type": "HIVE", "database": "production", "table": "orders"}]
    }
  ],
  "evaluate.rule": {
    "rules": [
      {
        "dsl.type": "griffin-dsl",
        "dq.type": "accuracy",
        "rule": "source.order_id = target.order_id AND source.amount = target.amount"
      }
    ]
  }
}
```

**비교 실행:**

```scala
// Griffin 측정 지표를 실행하는 Spark 작업
val spark = SparkSession.builder().getOrCreate()
val sourceDF = spark.table("staging.orders")
val targetDF = spark.table("production.orders")
// 불일치 찾기
val mismatches = sourceDF
  .join(targetDF, Seq("order_id"), "full_outer")
  .where(
    sourceDF("amount") =!= targetDF("amount") ||
    sourceDF("amount").isNull ||
    targetDF("amount").isNull
  )
mismatches.show()
```

**주요 장점:**

*   아파치 프로젝트 (특정 벤더에 종속되지 않음)
*   Spark 기반 (확장성 우수)
*   데이터 품질(data quality)에 중점
*   여러 데이터 소스(data sources) 지원

**과제:**

*   복잡한 설정 (하둡(Hadoop) 생태계 필요)
*   UI는 배포 필요
*   간단한 차이 분석에는 무거움

### `dbt` 및 분석 차이 분석

### 4. Datafold

웹사이트: [datafold.com](https://datafold.com/)
**기능:** 풀 리퀘스트(pull request) 통합을 통해 `dbt`를 위한 CI/CD 데이터 차이 분석.

`Datafold`는 풀 리퀘스트(pull request)에서 `dbt` 모델 변경 사항을 자동으로 분석합니다.

**설정 (.github/workflows/datafold.yml):**

```yaml
name: Datafold 데이터 차이 분석
on:
  pull_request:
    paths:
      - 'models/**'
jobs:
  datafold:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Datafold 차이 분석 실행
        uses: datafold/datafold-action@v1
        with:
          datafold_api_key: ${{ secrets.DATAFOLD_API_KEY }}
          dbt_project_dir: ./
```

**자동 PR 댓글:**

## 일반적인 차이 분석 패턴

### 패턴 1: 스키마 인식 차이 분석

```python
# 스키마 진화(schema evolution)를 유연하게 처리
def smart_diff(df1, df2, key_col):
    """스키마 호환성을 고려한 차이 분석"""
    
    # 공통 열 찾기
    common_cols = set(df1.columns) & set(df2.columns)
    
    # 추가/제거된 열
    added = set(df2.columns) - set(df1.columns)
    removed = set(df1.columns) - set(df2.columns)
    
    if added:
        print(f"새 열: {added}")
    if removed:
        print(f"제거된 열: {removed}")
    
    # 공통 열만 비교
    compare = datacompy.Compare(
        df1[list(common_cols)],
        df2[list(common_cols)],
        join_columns=key_col
    )
    
    return compare.report()
```

### 패턴 2: 대규모 차이 분석 샘플링

```python
def sampled_diff(df1, df2, key_col, sample_size=1000):
    """샘플링을 통해 대규모 데이터셋 차이 분석"""
    
    # 모든 다른 행 찾기
    merged = df1.merge(df2, on=key_col, how='outer', indicator=True, suffixes=('_old', '_new'))
    
    # 차이 유형별로 분리
    only_old = merged[merged['_merge'] == 'left_only'].sample(min(sample_size, len(merged)))
    only_new = merged[merged['_merge'] == 'right_only'].sample(min(sample_size, len(merged)))
    both = merged[merged['_merge'] == 'both']
    
    # 공통 행에서 값 차이 찾기
    changed = []
    for col in [c for c in both.columns if c.endswith('_old')]:
        base_col = col[:-4]
        new_col = base_col + '_new'
        if new_col in both.columns:
            diff_mask = both[col] != both[new_col]
            if diff_mask.any():
                changed.append(both[diff_mask].sample(min(sample_size, diff_mask.sum())))
    
    return {
        'deleted_sample': only_old,
        'added_sample': only_new,
        'changed_sample': pd.concat(changed) if changed else pd.DataFrame()
    }
```

### 패턴 3: 임계값 기반 알림

```python
def diff_with_thresholds(df1, df2, key_col, thresholds):
    """구성 가능한 알림 임계값을 사용한 차이 분석"""
    
    compare = datacompy.Compare(df1, df2, join_columns=key_col)
    
    issues = []
    
    # 행 수 변경 확인
    row_change_pct = abs(len(df1) - len(df2)) / len(df1) * 100
    if row_change_pct > thresholds['row_count_pct']:
        issues.append(f"행 수가 {row_change_pct:.1f}% 변경되었습니다")
    
    # 일치율 확인
    match_rate = compare.count_matching_rows() / len(df1) * 100
    if match_rate < thresholds['min_match_rate']:
        issues.append(f"행의 {match_rate:.1f}%만 일치합니다")
    
    # 열 변경 확인
    for col in df1.columns:
        if col in df2.columns:
            diff_count = (df1[col] != df2[col]).sum()
            diff_pct = diff_count / len(df1) * 100
            
            if diff_pct > thresholds.get(f'{col}_change_pct', thresholds['default_col_change_pct']):
                issues.append(f"{col}: 값의 {diff_pct:.1f}%가 변경되었습니다")
    
    return issues
```

## 핵심 요점

데이터는 변경됩니다. 문제는 사용자가 불평하기 전에 이를 감지하느냐, 아니면 그 후에 감지하느냐입니다.

**`dbt` 프로젝트의 경우:**

*   자동화된 PR 차이 분석을 위해 `Datafold` 사용
*   리팩토링(refactoring) 유효성 검사를 위해 `dbt-audit-helper` 추가
*   중요 테이블에 대해 CI/CD에서 `data-diff` 실행

**파이썬(`Python`)/`pandas` 워크플로우(workflow)의 경우:**

*   상세 보고서를 위해 `datacompy` 사용
*   중첩 구조를 위해 `DeepDiff` 추가
*   빠른 확인을 위해 `pandas.compare()` 사용

**대규모 데이터의 경우:**

*   교차 데이터베이스 비교를 위해 `data-diff` 사용
*   데이터 품질(data quality)을 위해 `Apache Griffin` 추가
*   방대한 데이터셋을 위해 `Spark` 기반 비교 사용

**데이터 레이크(data lakes)의 경우:**

*   `Delta Lake`/`Iceberg` 타임 트래블(time travel) 사용
*   Git 유사 브랜칭(branching)을 위해 `Nessie` 추가
*   `DVC`로 버전 추적

가장 중요한 데이터셋에 하나의 도구부터 시작하십시오. CI/CD에 차이 분석 기능을 추가하고, 그 다음 다른 파이프라인(pipeline)으로 확장하십시오.

무엇이 변경되었는지 파악하는 것이 문제를 해결하는 것보다 어려워서는 안 되기 때문입니다.

데이터 차이 분석 기능을 추가하기 가장 좋은 시점은 지난번 프로덕션 사고(production incident)가 발생하기 전이었습니다. 두 번째로 좋은 시점은 바로 지금입니다.
