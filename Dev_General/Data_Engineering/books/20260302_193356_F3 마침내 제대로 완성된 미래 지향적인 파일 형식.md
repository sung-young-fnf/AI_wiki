F3: 마침내 제대로 완성된 미래 지향적인 파일 형식
오픈 소스 데이터 커뮤니티가 CMU의 새로운 컬럼형 형식에 열광하는 이유와 Parquet의 10년 통치가 막을 내릴 수도 있는 이유

"또 다른 파일 형식"에 대한 이야기를 처음 들었을 때, 너무나 눈을 굴려 🙄 목이 삘 뻔했습니다. 우리는 이미 이런 상황을 여러 번 겪었습니다. 몇 년마다 누군가는 Parquet을 대체하고 우리 모두를 구할 혁명적인 형식을 발표합니다. 저는 Lance, Nimble 등 수많은 형식이 엄청난 약속을 하며 등장했다 사라지는 것을 지켜봤습니다.

하지만 F3는 다릅니다. 이 형식이 CMU(카네기 멜런 대학교)에서 나왔기 때문만은 아닙니다(물론 그들이 쌓은 신뢰는 무시할 수 없지만요). 제가 지난 한 달 동안 직접 테스트해 본 결과, F3는 제가 해결할 수 있을 것이라고 생각조차 못 했던 문제들을 해결해 주었기 때문입니다.

⚠️ F3 프로젝트는 연구 프로토타입(research prototype)입니다. 프로덕션(production) 환경에서 사용해서는 안 됩니다.

## Parquet 현실 점검 (The Parquet Reality Check)

화려한 "Parquet이 놀라운 이유" 블로그 게시물에서 아무도 말해주지 않는 현실은 다음과 같습니다.

더 자세한 내용은 다음 링크를 참조하십시오: [https://parquet.apache.org/docs/file-format/](https://parquet.apache.org/docs/file-format/)

메타데이터(metadata) 파싱(parsing)이 시스템을 좀먹고 있습니다. 머신러닝(ML) 피처(features)를 위한 10,000개 컬럼(column)을 가진 테이블이 있습니까? 축하합니다. 푸터(footer)를 읽는 데만 100밀리초(ms) 이상을 소비하고 있습니다. 데이터가 아니라 목차인 푸터만 읽는 데 말입니다.

로우 그룹(row group)은 아름다운 거짓말입니다. 개념은 우아합니다. 데이터를 수평적으로 분할하고 각 파티션(partition) 내에 컬럼형으로 저장하는 것입니다. 실제로는 어떻습니까? 전체 로우 그룹이 준비될 때까지 플러시(flush)할 수 없기 때문에 라이터(writer)가 1GB 이상의 데이터를 RAM(Random Access Memory)에 버퍼링(buffering)하고 있습니다. 기본 설정으로 와이드 테이블(wide table)을 작성하려다가 프로덕션(production) 시스템에서 OOM(Out Of Memory) 오류가 발생하는 것을 본 적이 있습니다.

딕셔너리 인코딩(Dictionary encoding)은 전부 아니면 전무(all-or-nothing) 방식입니다. 모든 컬럼 청크(column chunk)는 정확히 하나의 딕셔너리(dictionary)를 가집니다. 예외는 없습니다. 첫 100만 개의 로우(row)에 10개의 고유(distinct) 값이 있고 마지막 100만 개의 로우에 10,000개의 고유 값이 있든 상관없습니다. 동일한 딕셔너리 범위(scope)가 적용되며, 이로 인해 최적화되지 않은 압축(suboptimal compression)이 발생합니다.

버전 지옥(Version hell)은 실재합니다. 2018년에 Parquet 사양(spec)에 나노초(nanosecond) 타임스탬프(timestamp)를 추가했습니까? 좋습니다. 2025년에도 Spark나 Iceberg에서는 여전히 지원되지 않습니다. 왜 그럴까요? 모든 언어에 걸쳐 모든 리더 구현(reader implementation)을 업그레이드하는 것이 사실상 불가능하기 때문입니다.

결정적인 문제점은 무엇일까요? 50,000개 이상의 컬럼을 가진 피처 스토어(feature store)에서 작업한다고 상상해 보십시오. "이 10개 컬럼을 가져와라"는 간단한 프로젝션 쿼리(projection query)조차 데이터를 읽는 시간보다 메타데이터를 파싱하는 데 더 많은 시간을 소비했습니다.

## F3가 실제로 무엇을 고치는가 (What F3 Actually Fixes)

F3 팀은 단순히 Parquet을 개선하는 데 그치지 않았습니다. 그들은 2025년의 컬럼형 형식(columnar format)이 어떠해야 하는지에 대해 근본적으로 재고했습니다. 그리고 그 결과는 놀랍습니다.

### 해결책 #1: 당신을 괴롭히지 않는 메타데이터 (Metadata That Doesn’t Hate You)

F3는 모든 메타데이터(metadata)에 FlatBuffers를 사용합니다. Thrift도, Protocol Buffers도 아닙니다. FlatBuffers입니다.

이것이 왜 중요할까요? 제로 카피 역직렬화(Zero-copy deserialization) 때문입니다.

```python
# Parquet이 당신에게 요구하는 것:
# 1. 전체 푸터(footer) 읽기 (50,000개 컬럼 전체)
# 2. 전체 Thrift 구조체(struct) 역직렬화(deserialize)
# 3. 마침내 10개 컬럼에 접근

# F3가 당신에게 허용하는 것:
# 1. 푸터 읽기 (매우 작음)
# 2. 필요한 10개 컬럼의 메타데이터만 읽기
# 3. 제로 역직렬화(deserialization) 오버헤드(overhead)
```

그 결과는 어떨까요? 앞서 언급한 10,000개 컬럼 테이블의 경우:
*   Parquet: 100ms+ 메타데이터 오버헤드
*   F3: 10ms

필요 없는 것을 읽지 않는 것만으로도 10배의 개선을 이룬 것입니다.

### 해결책 #2: IOUnit (일명 "쓰기 작업이 작동해야 하는 방식")

이것은 미묘하지만 탁월합니다. Parquet의 로우 그룹(row group)은 과부하 상태입니다. 이는 논리적 파티션(logical partition)이자 I/O 단위(I/O unit)이며, 동시에 메모리 플러시 경계(memory flush boundary)입니다. F3는 이들을 분리합니다.

```
// F3의 계층 구조(hierarchy):
로우 그룹(Row Group) -> 논리적 파티션 (100만 로우)
  └─ IOUnit -> 물리적 I/O (각 8MB)
      └─ EncUnit -> 인코딩 경계(Encoding boundaries)


// Parquet의 계층 구조:
로우 그룹(Row Group) -> 모든 것 (논리적 + 물리적 + 플러시 경계)
```

실질적인 영향은 무엇일까요? 10GB 로우 그룹을 메모리에 10GB를 유지하지 않고도 쓸 수 있습니다. F3는 컬럼이 8MB IOUnit을 채우자마자 플러시(flush)합니다. 메모리 사용량은 로우 그룹 크기에 관계없이 일정합니다.

### 해결책 #3: 합리적인 딕셔너리 (Dictionaries That Make Sense)

F3는 컬럼(column)별로 딕셔너리 범위(dictionary scope)를 선택할 수 있게 합니다. 전체 파일에 걸쳐 전역 딕셔너리(global dictionary)를 원하십니까? 가능합니다. 64K 로우당 로컬 딕셔너리(local dictionary)를 원하십니까? 그것도 가능합니다. 그 중간을 원하십니까? 그것도 작동합니다.

연구 결과는 놀랍습니다. 실제 데이터셋에서 2,080개의 컬럼을 테스트한 결과:
*   13%는 로컬 딕셔너리로 가장 잘 압축됩니다.
*   35%는 전역 딕셔너리로 가장 잘 압축됩니다.
*   52%는 그 중간 방식으로 가장 잘 압축됩니다.

하지만 여기서 핵심은, "전역 또는 로컬 중 최적의 방식"을 선택하기만 하면 88%의 컬럼에 대해 거의 최적의 압축을 얻을 수 있다는 것입니다. 그리고 F3는 쓰기 시점에 이를 자동으로 선택할 수 있습니다.

```python
# 예시: 카디널리티(cardinality)가 변하는 컬럼
# 처음 100만 로우: 10개의 고유(distinct) 값
# 다음 100만 로우: 10,000개의 고유(distinct) 값


# Parquet: 200만 로우 전체에 하나의 딕셔너리
# 결과: 불필요하게 커진 딕셔너리, 낮은 압축률

# F3: 범위(scope)별 개별 딕셔너리
# 결과: 첫 번째 청크(chunk)에는 작은 딕셔너리, 두 번째에는 더 큰 딕셔너리
# 전체 파일 크기: 15-20% 감소
```

### 해결책 #4: WebAssembly 트릭 (The WebAssembly Trick)

여기서부터 F3는 정말 흥미로워집니다(또는 누구에게 묻느냐에 따라 논쟁의 여지가 있을 수 있습니다).

핵심 문제: 모든 리더(reader)를 손상시키지 않고 새로운 인코딩 스킴(encoding scheme)을 어떻게 추가할 수 있을까요?

Parquet의 답변: 사양(spec)에 추가하고, 모든 언어(Java, C++, Rust, Go, Python 등)로 구현하며, 채택되기까지 3~7년을 기다리고, 절반의 리더는 절대 업데이트되지 않는 것을 지켜봅니다.

F3의 답변: 디코더(decoder)를 파일 자체에 WebAssembly (Wasm)로 임베드(embed)합니다.

잠깐, 뭐라고요?

```
┌─────────────────────────────────────┐
│         F3 파일                      │
├─────────────────────────────────────┤
│  데이터 (CustomCodec으로 압축됨)      │
│  메타데이터                            │
│  Wasm 바이너리 (CustomCodec 디코더)    │ <- 이것이 새로운 점
└─────────────────────────────────────┘
```

모든 F3 파일에는 다음이 포함됩니다:
*   실제 데이터 (인코딩됨)
*   데이터를 설명하는 메타데이터
*   데이터를 디코딩(decode)하는 Wasm 코드

이것은 다음을 의미합니다:
*   ✅ 구현체(implementation) 간 조율 없이 새로운 인코딩(encoding)을 추가할 수 있습니다.
*   ✅ 오래된 리더(reader)도 새로운 파일(임베드된 Wasm을 통해)을 디코딩할 수 있습니다.
*   ✅ 새로운 리더는 사용 가능한 경우 네이티브 코드(native code)를 사용하여 더 빠르게 작동합니다.
*   ✅ 파일은 자체 설명적(self-describing)이며 미래에도 사용 가능(future-proof)합니다.

오버헤드(overhead)는 얼마나 될까요? 인코딩(encoding)당 약 150KB입니다. 일반적인 파일의 경우, 이는 전체 크기의 0.001%에 불과합니다.

성능 저하는 어떻습니까? Wasm 경로의 경우 네이티브 코드(native code)보다 15~35% 느립니다. 하지만 중요한 점은, Wasm은 폴백(fallback)으로만 사용된다는 것입니다. 리더가 네이티브(native) 지원을 하는 경우, 해당 코드를 사용합니다.

## 실제로 작동하는 것과 그렇지 않은 것 (What Actually Works (And What Doesn’t))

저는 지난 3주 동안 피처 스토어(feature store) 파이프라인(pipeline)에서 F3를 운영해 보았습니다. 다음은 솔직한 평가입니다.

*   ✅ **메타데이터(metadata) 성능은 혁신적입니다.** 와이드 테이블(wide table) 쿼리(query)는 5~10배 더 빠릅니다. 이것만으로도 F3를 시도해 볼 가치가 있습니다.
*   ✅ **쓰기 메모리(write memory)는 예측 가능합니다.** 와이드 테이블에서 더 이상 OOM(Out Of Memory) 충돌이 발생하지 않습니다. 저희 라이터(writer)는 "제발 성공하길"에서 "그냥 작동한다"는 확신으로 바뀌었습니다.
*   ✅ **압축률(compression ratio)은 경쟁력이 있습니다.** 데이터에 따라 Parquet과 5% 이내의 차이를 보이거나 때로는 더 좋습니다.
*   ✅ **Wasm 기능은 실제로 작동합니다.** 시계열 데이터(time-series data)를 위한 사용자 지정 인코딩(custom encoding)으로 테스트했습니다. Wasm으로 임베드(embed)하자 리더(reader)가 그냥 작동했습니다. 업그레이드는 필요 없었습니다.
*   ❌ **생태계(ecosystem) 지원이 아직 초기 단계입니다.** 아직 Spark나 Snowflake에 F3를 바로 적용할 수 없습니다. 이는 많은 팀에게 걸림돌이 됩니다.
*   ❌ **쓰기 성능(write performance)은 약간 느립니다.** 제 테스트에서는 Parquet에 비해 약 10~15% 정도였습니다. 유연성에는 대가가 따릅니다.
*   ❌ **문서(documentation)는 학술적입니다.** SIGMOD 논문은 박사 학위 소지자에게는 훌륭하지만, 단순히 레이아웃(layout)을 이해하려는 사람에게는 적합하지 않습니다.
*   ❌ **도구(tooling)가 최소한입니다.** Parquet처럼 멋진 인스펙터 UI(inspector UI)를 원하십니까? 아직 없습니다. 파일 손상 디버깅(file corruption debugging)은? 헥스 덤프(hex dump)를 좋아하시길 바랍니다.

## 결론 (The Verdict)

F3는 제가 본 "차세대" 컬럼형 형식(columnar format) 중 실제로 과대광고가 아닌 유일한 것입니다. 메타데이터(metadata) 설계는 확실히 더 낫습니다. IOUnit 분리(decoupling)는 실제 문제를 해결합니다. 그리고 Wasm 기능은 — 이상하게 들릴지 모르지만 — 사실 탁월합니다.

하지만 아직 초기 단계입니다. 정말 초기 단계입니다. 내일 당장 Spark 통합이 필요하다면 Parquet을 고수하십시오. 새롭게 무언가를 구축하고 있고 초기 단계의 미숙함을 감수할 수 있다면 F3는 진지하게 고려할 가치가 있습니다.

이 형식의 세 가지 핵심 원칙 — 상호 운용성(interoperability), 확장성(extensibility), 효율성(efficiency) — 은 단순히 마케팅 문구가 아닙니다. 이는 모든 설계 결정에 깊이 반영되어 있습니다. 그리고 이것은 드문 일입니다.
