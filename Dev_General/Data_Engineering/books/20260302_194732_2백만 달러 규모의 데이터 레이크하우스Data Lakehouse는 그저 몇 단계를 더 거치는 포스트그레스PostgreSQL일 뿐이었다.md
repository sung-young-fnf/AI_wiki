**2백만 달러 규모의 '데이터 레이크하우스(Data Lakehouse)'는 그저 몇 단계를 더 거치는 포스트그레스(PostgreSQL)일 뿐이었다**

마케팅이 '데이터 아키텍처의 미래'라고 부르는 것을 구축하는 데 2년을 보냈습니다. 결국 더 많은 벤더를 거느린 데이터베이스였습니다.

작년에 우리 회사의 데이터 아키텍처는 다음과 같았습니다.

*   데이터 웨어하우스(Data Warehouse): 스노우플레이크(Snowflake) (연간 20만 달러)
*   데이터 레이크(Data Lake): S3 버킷(S3 buckets) (연간 5만 달러)
*   문제점: 데이터가 두 곳에 분산되어 있었음

그래서 우리는 당연히 합리적인 회사라면 할 법한 일을 했습니다. 두 시스템에 데이터가 분산된 문제를 해결하기 위해 2백만 달러를 들여 "레이크하우스(lakehouse)"를 구축했습니다. 그 해결책이란… 이전에 존재하던 두 시스템인 척하는 세 번째 시스템을 만드는 것이었습니다.

*   투자: 18개월간 2백만 달러
*   성능 개선: -30% (네, 마이너스입니다)
*   복잡성 증가: 10배
*   해결된 문제: 0개
*   새로 발생한 문제: 47개
*   실상은 이랬습니다: 포스트그레스(PostgreSQL)만으로도 충분했을 것입니다.

데이터 레이크하우스는 데이터 역사상 가장 위대한 마케팅 성과입니다. 기업들에게 기존의 혼란을 해결하는 대신, 이전의 두 가지 혼란을 모두 결합한 '새로운' 혼란이 필요하다고 설득했으니 말입니다. 정말 기발합니다.

## 아무것도 아닌 곳으로의 2백만 달러 여정

### 1년 차: 약속의 시간 (80만 달러)

*   벤더의 제안: "통합 아키텍처! 두 세계의 장점만! 단일 진실 공급원!"
*   현실: 하나의 '통합' 시스템을 만들기 위해 6가지 다른 기술이 필요했습니다.

**구매 내역:**

*   데이터브릭스(Databricks) 라이선스: 연간 30만 달러
*   델타 레이크(Delta Lake) 구현: 컨설팅 비용 20만 달러
*   아파치 아이스버그(Apache Iceberg) "만약을 대비해서": 컨설팅 비용 10만 달러
*   AWS 인프라(AWS infrastructure): 연간 15만 달러
*   모니터링 도구: 연간 5만 달러

### 2년 차: 현실의 시간 (120만 달러)

*   실제로 일어난 일: 모든 것이 더 나빠졌습니다.

**추가 비용:**

*   성능 컨설턴트: 30만 달러 (우리가 망가뜨린 것을 고치기 위해)
*   데이터 엔지니어: 3명 x 20만 달러 = 60만 달러 (이 괴물을 유지하기 위해)
*   마이그레이션(Migration) 도구: 10만 달러
*   팀원들의 심리 치료: 값을 매길 수 없음
*   이 프로젝트를 주도한 임원: 승진 (상향식 실패)

## 아무도 이해하지 못하는 아키텍처

다음은 우리의 "간단하고 통합된" 레이크하우스 아키텍처입니다.

원시 데이터(Raw Data) → S3 버킷(S3 Buckets) → 델타 레이크(Delta Lake) → 스파크 처리(Spark Processing) → 메타데이터 레이어(Metadata Layer) → 카탈로그 서비스(Catalog Service) → 쿼리 엔진(Query Engine) → 또 다른 처리 레이어(Processing Layer) → 캐시 레이어(Cache Layer) → 마침내 쿼리 결과 (운이 좋다면 말이죠)

*   연관된 구성 요소: 14개
*   장애 지점: 14개
*   전체를 이해하는 사람: 0명
*   데이터 쿼리 시간: 이전보다 3배 더 오래 걸림

실제로 우리에게 필요했던 것과 비교:

데이터 → 포스트그레스(PostgreSQL) → 쿼리

*   구성 요소: 1개
*   장애 지점: 1개
*   이해하는 사람: 모두
*   쿼리 시간: 밀리초(Milliseconds) 단위

## 레이크하우스 아키텍처의 세 가지 거짓말

### 첫 번째 거짓말: "두 세계의 장점을 결합한다"

*   현실: 두 세계의 '복잡성'을 결합합니다.

**데이터 레이크(Data Lake)에서 얻은 것:**

*   비정형 데이터의 혼란 ✓
*   스키마 온 리드(Schema-on-read) 방식의 혼란 ✓
*   "데이터 늪(Data swamp)"이 될 가능성 ✓

**데이터 웨어하우스(Data Warehouse)에서 얻은 것:**

*   높은 비용 ✓
*   엄격한 구조 요구사항 ✓
*   충족할 수 없는 성능 기대치 ✓

**얻지 못한 것:**

*   어느 쪽에서도 얻지 못한 단순성
*   어느 쪽에서도 얻지 못한 성능
*   어느 쪽에서도 얻지 못한 비용 절감

### 두 번째 거짓말: "단일 시스템이다"

*   현실: 15개의 시스템이 하나인 척합니다.

**우리의 "단일" 레이크하우스가 사용하는 것들:**

*   객체 스토리지(Object storage) (S3)
*   테이블 형식(Table format) (델타 레이크(Delta Lake))
*   카탈로그(Catalog) (AWS 글루(AWS Glue))
*   처리 엔진(Processing engine) (스파크(Spark))
*   쿼리 엔진(Query engine) (프레스토(Presto))
*   메타데이터 저장소(Metadata store) (하이브 메타스토어(Hive Metastore))
*   오케스트레이션(Orchestration) (에어플로우(Airflow))
*   모니터링(Monitoring) (데이터독(Datadog))
*   보안 레이어(Security layer) (레인저(Ranger))
*   캐싱 레이어(Caching layer) (알룩시오(Alluxio))
*   피처 스토어(Feature store) (피스트(Feast))
*   머신러닝 플랫폼(ML platform) (ML플로우(MLflow))
*   노트북 환경(Notebook environment) (데이터브릭스(Databricks))
*   버전 관리(Version control) (깃(Git))
*   소규모 데이터용 또 다른 데이터베이스 (포스트그레스(Postgres))

"단일 시스템"이라니, 말도 안 되는 소리입니다.

### 세 번째 거짓말: "데이터 이동을 없앤다"

*   현실: 이전보다 훨씬 더 많은 데이터를 이동시킵니다.

**레이크하우스 도입 전:**

*   소스에서 웨어하우스로 ETL
*   끝.

**레이크하우스 도입 후:**

*   브론즈(Bronze) 레이어로 유입
*   실버(Silver) 레이어로 처리
*   골드(Gold) 레이어로 변환
*   자주 접근하는 데이터 캐싱
*   성능 향상을 위한 구체화(Materialize)
*   피처 스토어(Feature store)로 복사
*   다양한 엔진으로 동기화
*   모든 것을 백업

데이터 이동 횟수가 2번에서 8번으로 늘어났습니다. 대단한 발전이군요!

## 아무도 인정하지 않는 성능 참사

우리 레이크하우스의 실제 벤치마크 결과:

**간단한 쿼리:** “SELECT COUNT(*) FROM customers”

*   포스트그레스(PostgreSQL): 15밀리초
*   우리의 기존 스노우플레이크(Snowflake): 200밀리초
*   우리의 새로운 레이크하우스(Lakehouse): 3.2초

**왜 이렇게 느릴까요?**

*   카탈로그(catalog)에서 메타데이터 읽기 (500밀리초)
*   쿼리 최적화기(Query optimizer)가 고민 (800밀리초)
*   스파크 익스큐터(Spark executors) 실행 (1초)
*   S3에서 읽기 (500밀리초)
*   3개 레이어(layer)를 통해 처리 (400밀리초)
*   결과 반환 (마침내)

**복잡한 분석 쿼리:**

*   기존 스노우플레이크(Snowflake): 30초
*   새로운 레이크하우스(Lakehouse): 4분
*   적절한 인덱스(index)가 적용된 포스트그레스(PostgreSQL): 8초

하지만 적어도 "통합"은 되었잖아요!

## 수백만 달러를 낭비하는 형식 전쟁

테이블 형식을 선택하지 않고는 레이크하우스를 가질 수 없습니다! 선택지는 다음과 같습니다.

*   **델타 레이크(Delta Lake) (데이터브릭스(Databricks))**
    *   장점: 데이터브릭스와 잘 작동합니다.
    *   단점: 벤더 종속(Vendor lock-in), 그 외 모든 것
    *   비용: 당신의 영혼

*   **아파치 아이스버그(Apache Iceberg) (넷플릭스(Netflix))**
    *   장점: "개방형" 표준
    *   단점: 47가지의 다른 구현체, 호환되는 것은 전무
    *   비용: 끝없는 컨설팅

*   **아파치 후디(Apache Hudi) (우버(Uber))**
    *   장점: 아무도 사용하지 않으므로 독특해질 수 있습니다.
    *   단점: 아무도 사용하지 않습니다.
    *   비용: 당신의 온전한 정신

우리는 형식을 평가하는 데 3개월을 보냈습니다. 그리고는 우리 컨설턴트가 데이터브릭스와 관계가 있었기 때문에 델타(Delta)를 선택했습니다. 동전 던지기로 결정해도 되었을 것입니다.

## 마이그레이션 지옥

레이크하우스로의 이전은 쉬울 것이라고 예상했습니다. 현실은 이랬습니다.

*   **1단계: "간단한 마이그레이션(Simple Migration)" (6개월)**
    *   웨어하우스에서 내보내기
    *   레이크로 가져오기
    *   아무것도 작동하지 않음을 발견
    *   패닉

*   **2단계: "문제 해결" (6개월)**
    *   모든 곳에서 스키마(Schema) 충돌
    *   데이터 타입(Data type) 불일치
    *   성능 참사
    *   더 큰 패닉

*   **3단계: "최적화" (무한 개월)**
    *   여전히 진행 중
    *   절대 끝나지 않을 것입니다.
    *   컨설턴트 자녀들의 대학 학자금은 확보
    *   운명에 대한 체념

*   성공적으로 마이그레이션된 데이터: 60%
*   새 시스템에서 접근 가능한 데이터: 40%
*   실제로 사용되는 데이터: 5%
*   투자수익률(ROI): 마이너스 무한대

## 우리가 했어야 할 일

충격적인 비밀을 말씀드리자면, 대부분의 회사에는 포스트그레스(PostgreSQL)만 있으면 됩니다.

**우리의 실제 데이터:**

*   크기: 2테라바이트(TB) (페타바이트(PB)가 아님)
*   사용자: 50명 (수천 명이 아님)
*   쿼리: 하루 1,000건 (수백만 건이 아님)
*   성장률: 월 10기가바이트(GB) (하루 테라바이트가 아님)

**포스트그레스(PostgreSQL)는 다음을 처리할 수 있었습니다.**

*   비용: 강력한 서버에 월 5천 달러
*   성능: 1초 미만의 쿼리
*   복잡성: 없음
*   필요한 기술: 기본 SQL
*   구현 시간: 1주

**하지만 우리는 레이크하우스를 선택했습니다. 그 이유는:**

*   "모두가 그렇게 하니까"
*   "그것이 미래니까"
*   "우리는 확장성(scale)이 필요할지도 모르니까"
*   "AI 준비" (우리는 AI를 하지 않습니다)

## 당신의 데이터 요구 사항에 대한 진실

99%의 회사들은:

*   페타바이트(PB) 단위의 데이터를 가지고 있지 않습니다.
*   실시간 스트리밍(real-time streaming)이 필요하지 않습니다.
*   복잡한 머신러닝(ML)을 하지 않습니다.
*   대기 중인 데이터 과학자(data scientists)가 없습니다.
*   15개의 처리 레이어(processing layers)가 필요하지 않습니다.

그들이 실제로 필요한 것:

*   데이터베이스
*   잘 작동하는
*   백업이 있는
*   적절한 성능을 가진
*   사람들이 이해할 수 있는

이 모든 것을 제공하는 것이 무엇인지 아십니까? 포스트그레스(PostgreSQL)입니다.

## 해방: 지루함으로의 회귀

우리는 레이크하우스를 해체하고 있습니다.

**1단계: 현실을 받아들이기**

*   우리는 구글(Google)이 아닙니다.
*   우리는 구글의 문제를 가지고 있지 않습니다.
*   우리는 구글의 해결책이 필요하지 않습니다.

**2단계: 포스트그레스(PostgreSQL)로 마이그레이션(Migration)**

*   읽기 복제본(read replicas) 설정
*   적절한 인덱스(indexes) 추가
*   대형 테이블에 파티셔닝(partitioning) 사용
*   끝.

**3단계: 모든 것 취소**

*   데이터브릭스(Databricks) ✓
*   스파크 클러스터(Spark clusters) ✓
*   컨설턴트 ✓
*   교육 프로그램 ✓
*   콘퍼런스 참석 ✓

**현재까지의 결과:**

*   비용 90% 절감

## 정신 차리라는 요구

레이크하우스를 구축하기 전에 다음을 물어보십시오.

*   10테라바이트(TB) 이상의 활성 데이터(아카이브(archived) 제외)를 가지고 있습니까?
*   하루 1백만 건 이상의 쿼리를 처리합니까?
*   100명 이상의 동시 사용자(concurrent users)가 있습니까?
*   데이터가 매달 두 배씩 증가하고 있습니까?
*   실제로 프로덕션(production) 환경에서 머신러닝(ML)을 수행하고 있습니까?

이 중 하나라도 "아니오"라고 답했다면, 레이크하우스는 필요 없습니다.

당신에게 필요한 것:

*   포스트그레스(PostgreSQL) (또는 MySQL, 심지어 SQLite)
*   좋은 인덱스(indexes)
*   적절한 하드웨어
*   기본적인 백업
*   SQL을 아는 사람

*   총 비용: 월 1만 달러 미만
*   총 복잡성: 최소한
*   테이블 형식에 대해 논쟁하는 총 시간: 없음

## 최종 판결

데이터 레이크하우스는 문제에 대한 해결책을 찾아 헤매는 격입니다. 이는 벤더들이 당신의 단순한 요구 사항에 복잡한 솔루션이 필요하다고 설득하는 것이며, 컨설턴트들이 아키텍처 우주비행학(architecture astronautics)을 판매하는 것입니다. 최악의 경우, 이는 이력서 위주의 개발(resume-driven development)입니다.

*   데이터 레이크는 늪이 되어 실패했습니다.
*   데이터 웨어하우스는 벤더들이 터무니없는 가격을 매겨서 "실패"했습니다.
*   데이터 레이크하우스는 이 두 가지 실패가 결합된 것이기 때문에 실패합니다.

진정한 승자는? 그저 작동할 뿐인 지루한 데이터베이스입니다.

현재 2백만 달러짜리 레이크하우스를 포스트그레스(PostgreSQL)로 되돌리는 마이그레이션을 진행 중입니다. 2년간의 복잡성을 되돌리는 데 3개월이 걸릴 것입니다. 이 데이터베이스는 월 5천 달러의 비용으로 우리의 모든 요구 사항을 처리할 것이며, 모두가 이해할 수 있을 것입니다. 레이크하우스 벤더들은 필사적으로 전화하고 있습니다. 우리는 받지 않을 것입니다.

**추신** — "하지만 확장성(scale)이 필요할 때는 어떻게 하죠?" 그런 일은 없을 것입니다. 99%의 회사들은 포스트그레스(PostgreSQL)가 실패할 정도의 확장성에 도달하지 못합니다. 우리가 구글(Google)이 된다면 그때 걱정하겠습니다. 스포일러: 우리는 구글이 되지 않을 것입니다.

**추신 2** — 레이크하우스를 추진했던 임원은요? 그는 지금 다른 회사에서 또 다른 레이크하우스를 구축하고 있습니다. 복잡성의 순환은 계속됩니다. 그의 새로운 직함은요? "최고 데이터 레이크하우스 책임자(Chief Data Lakehouse Officer)"입니다. 이건 농담이 아닙니다.
