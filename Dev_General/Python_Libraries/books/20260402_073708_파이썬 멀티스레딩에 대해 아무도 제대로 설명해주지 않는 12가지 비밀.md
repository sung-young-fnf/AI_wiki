# 파이썬 멀티스레딩에 대해 아무도 제대로 설명해주지 않는 12가지 비밀

파이썬 프로그래머를 가장 빠르게 혼란에 빠뜨리는 방법은 멀티스레딩(multithreading) 주제를 꺼내는 것입니다. 흔히 "파이썬에는 GIL이 있다", "스레드는 병렬로 실행되지 않는다", "그냥 멀티프로세싱(multiprocessing)을 써라" 같은 말들을 하지만, 실제로 스레드 코드를 작성하기 시작하면 상황은 훨씬 복잡해집니다.

어떤 스레드는 예상보다 빠르고, 어떤 스레드는 이유 없이 멈추며, 어떤 스레드는 IO 작업의 속도를 마법처럼 높여주기도 합니다. 반면 앱을 더 느리게 만드는 스레드도 있습니다. 파이썬의 멀티스레딩 규칙은 대개 결과 위주로 교육될 뿐, 그 이면의 메커니즘은 간과되곤 합니다.

이 글에서는 중급 파이썬 개발자라면 반드시 알아야 할, 하지만 명쾌하게 설명해 주는 사람이 드문 12가지 비밀을 파헤칩니다.

### 1. GIL은 모든 스레드를 막지 않는다 — 파이썬 바이트코드만 막을 뿐이다

글로벌 인터프리터 락(GIL, Global Interpreter Lock)이 파이썬 바이트코드(bytecode)의 실행을 직렬화하는 것은 맞습니다. 하지만 중요한 점은 **스레드가 C 수준의 작업(C-level operations)을 기다릴 때는 병렬로 실행된다**는 점입니다.

여기에는 다음과 같은 작업이 포함됩니다.
* 네트워크 IO
* 파일 IO
* Sleep 연산
* 특정 C 확장 모듈 (NumPy, lxml, Pillow 등)
* 블로킹 시스템 콜 (Blocking system calls)

이것이 `requests` 라이브러리를 사용한 다운로드 작업에서 스레드가 효율적으로 작동하는 이유입니다. 네트워크 IO 동안 GIL이 해제되므로 스레드들이 실제로 겹쳐서 실행됩니다.

### 2. "CPU-Bound"와 "IO-Bound"의 구분은 실제 문제의 핵심이 아니다

보통 "스레드는 IO 작업에, 프로세스는 CPU 작업에 써라"라고 가르치지만, 이는 지나치게 단순화된 규칙입니다. 진짜 규칙은 **"코드가 계산이 아니라 대기(waiting)에 상당한 시간을 할애할 때 스레드를 사용하라"**는 것입니다.

심지어 CPU 집약적인 작업이라도 다음과 같은 경우라면 스레드의 이점을 얻을 수 있습니다.
* GIL을 해제하는 C 확장 모듈 내에서 무거운 작업이 일어날 때
* 작업이 IO 연산 사이에 조각나 있을 때
* 동일한 워크플로우 내에서 CPU와 IO 작업이 결합되어 있을 때

### 3. 컨텍스트 스위칭은 숨겨진 성능 킬러다

스레드가 GIL에 의해 차단되지 않더라도 다음과 같은 비용이 발생합니다.
* 커널 컨텍스트 스위칭 (Kernel context switching)
* 파이썬 수준의 작업 전환
* 락(Lock) 획득 오버헤드
* 부하 상황에서의 스케줄링 불공정성

사소한 작업을 위해 200개의 스레드를 띄우면 오히려 시스템 전체가 느려집니다. 대부분의 IO 작업에 최적화된 스레드 개수는 보통 10개에서 50개 사이입니다.

### 4. GIL이 성능에 도움이 될 때도 있다

GIL은 한 번에 하나의 스레드만 파이썬 바이트코드를 실행하도록 보장함으로써 다음과 같은 이점을 제공합니다.
* 값비싼 락 없이도 객체를 보호함
* 원자적 연산(atomic operations)의 필요성을 줄임
* 메모리 관리를 더 단순하고 빠르게 유지함
* 멀티코어 동기화 오버헤드를 감소시킴

GIL을 제거하려는 시도가 10년 넘게 걸린 이유는 전체 파이썬 생태계가 이러한 성능상의 장단점에 의존하고 있기 때문입니다.

### 5. 파이썬 스레드는 죽을 때 소리 없이 죽는다

스레드가 대상 함수를 마치면 조용히 종료됩니다. 하지만 **예외(exception)가 발생했을 때도 조용히 죽습니다.**

많은 초보자가 이 부분을 놓칩니다. 메인 스레드는 자식 스레드에서 에러가 났는지 알 수 없습니다. 따라서 예외 처리를 수동으로 하거나, 예외를 제대로 캡처하는 `concurrent.futures`를 사용해야 합니다.

### 6. daemon=True는 "배경 스레드"가 아니라 "킬 스위치"다

많은 개발자가 `daemon=True`를 단순히 "백그라운드에서 실행하라"는 의미로 오해합니다. 하지만 실제 의미는 **"프로그램 종료 시 정리 작업 없이 즉시 강제 종료하라"**는 뜻입니다. 이 경우 열려 있는 파일이 그대로 남거나, 데이터 쓰기가 중단되어 상태가 오염될 수 있습니다.

### 7. GIL은 생각보다 훨씬 자주 해제된다

파이썬은 다음과 같은 상황에서 GIL을 해제합니다.
* 약 5ms 동안 바이트코드를 실행했을 때
* 수많은 블로킹 연산 중에
* 일부 순수 C 연산 중에
* 대부분의 IO 작업 및 최적화된 라이브러리 실행 중에

GIL은 병목(bottleneck)일 수 있지만, 결코 감옥(prison)은 아닙니다.

### 8. ThreadPoolExecutor는 2025년에 가장 권장되는 API다

과거의 `threading.Thread` API도 여전히 작동하지만, 다음과 같은 단점이 있습니다.
* 예외 캡처가 어려움
* 스레드 생명주기를 수동으로 관리해야 함
* 결과 값을 반환받기 어려움

반면 `ThreadPoolExecutor`는 깔끔한 구문, 자동 결과 처리, 스레드 재사용, 타임아웃 및 취소 기능을 제공하여 90% 이상의 사용 사례에 적합한 현대적인 접근 방식입니다.

### 9. 생각보다 락(Lock)이 필요하지 않은 경우가 많다

많은 개발자가 락을 과용합니다. 파이썬은 더 안전한 대안들을 제공합니다.
* **Queue**: 스레드 세이프한 생산자-소비자 파이프라인 구현
* **deque**: 원자적인 append 및 pop 지원
* **Event**: 신호 전달을 위한 단순한 도구
* **Semaphore**: 속도 제한(rate limiting)에 이상적

락은 꼭 필요한 경우에만 사용하세요. 그렇지 않으면 코드가 취약해지고 데드락(deadlock)에 빠지기 쉽습니다.

### 10. 스레드 안전성은 디자인의 문제다

멀티스레딩 문제는 스레드 자체보다 설계(architecture)에서 발생합니다. 스레드가 가변 데이터 구조, 글로벌 변수, 공유 카운터나 캐시를 공유한다면 문제가 발생할 수밖에 없습니다. 최선의 방법은 **"데이터를 공유하지 말고, 메시지를 전달하라(Don't share data. Pass messages.)"**는 원칙을 지키는 것입니다.

### 11. print()는 스레드 세이프하지 않다

출력 결과가 뒤섞여 나오는 이유는 `print()` 함수가 스레드 간에 원자적이지 않기 때문입니다. 이를 해결하려면 출력 시 락을 사용하거나, 스레드 세이프한 `logging` 모듈을 사용하거나, 큐(queue)에 출력을 수집하는 방식을 써야 합니다.

### 12. 멀티스레딩은 마법처럼 빨라지는 게 아니라 응답성을 높이는 것이다

스레드는 단순히 속도를 높이는 도구가 아닙니다.
* 앱의 응답성을 유지하고 UI 프리징을 방지함
* 동시 다운로드를 가능하게 함
* 대기 작업을 병렬화함
* 실시간 모니터링을 가능하게 함

속도보다 중요한 것은 **응답성(Responsiveness)**입니다. 이 사고방식의 전환이 있어야 스레드를 의도에 맞게 제대로 사용할 수 있습니다.

### 결론

파이썬 멀티스레딩은 예측 불가능해 보일 수 있지만, GIL, 컨텍스트 스위칭, 공유 상태 설계, IO 모델을 이해하면 모든 것이 명확해집니다. 이 12가지 원칙을 마스터하면 더 빠르고 깨끗하며 안전한 코드를 작성할 수 있을 것입니다.
