새벽 1시 42분, 휴대폰이 울렸습니다. 슬랙(Slack)도, 왓츠앱(WhatsApp)도 아니었습니다. 몇 달 전에 제가 작성한 파이썬(Python) 스크립트에서 보낸 일반 텍스트 알림이었습니다. "새로운 중요 문서 감지됨. 분류됨. 보관됨. 요약 전송됨." 저는 미소 지으며 몸을 뒤척인 후 다시 잠들었습니다.

그 순간은 과도하게 설계된 인공지능(AI) 플랫폼이나 12개 서비스로 이루어진 마이크로서비스(microservice) 악몽에서 온 것이 아니었습니다. 그저 작은 파이썬 라이브러리들이 조립되어 제대로 작동하는 실제 자동화 시스템을 통해 이루어졌습니다.

이 글은 초보자를 위한 목록이 아닙니다. 이 글은 겉보기에는 단순해 보이지만, 조합했을 때 그 잠재력을 알게 되는 라이브러리들을 활용하여 솔직하게 풀어낸 실제 자동화 프로젝트 이야기입니다.

아래에 소개된 모든 내용은 제가 직접 배포하고, 유지보수하며, 신뢰하는 것들입니다.

### 해결한 자동화 문제

저는 수많은 인입 파일들을 다룹니다:

*   이메일로 받은 PDF
*   CSV 내보내기 파일
*   로그(Log)
*   보고서
*   폴더에 무작위로 저장된 다운로드 파일

이러한 파일들을 수동으로 정리하고, 검증하고, 요약하고, 보관하는 것은 엄청난 시간 낭비였습니다.

그래서 저는 자동화된 파일 인텔리전스 파이프라인(Automated File Intelligence Pipeline)을 구축했습니다:

*   새로운 파일 감지
*   파일 검증 및 정규화
*   메타데이터(metadata) 추출
*   결과 캐시(cache)
*   모든 것을 깔끔하게 로깅(logging)
*   필요할 때만 사람에게 알림

이제 자세히 살펴보겠습니다.

### 1. `pathlib` — 파일 시스템(Filesystem)의 중추

여전히 `os.path`를 사용하고 있다면 위험하게 작업하고 있는 것입니다.

```python
from pathlib import Path

INBOX = Path("incoming")
ARCHIVE = Path("archive")

for file in INBOX.iterdir():
    if file.is_file():
        print(file.name)
```

자동화에서 이 라이브러리가 중요한 이유:

*   크로스 플랫폼(Cross-platform) 지원
*   조합 가능(Composable)
*   문자열(string) 관련 혼란 없음

이 라이브러리 위에 다른 모든 것이 구축됩니다.

### 2. `watchfiles` — 실시간 파일 감지

폴더를 계속 폴링(polling)하는 것은 비효율적인 방식입니다.

```python
from watchfiles import watch

for changes in watch("incoming"):
    print("Change detected:", changes)
```

이 라이브러리는 이벤트 기반(event-driven) 자동화를 가능하게 했습니다. 크론(cron) 작업이나 무한 반복(sleep loops) 같은 방식이 필요 없습니다.

새로운 파일이 나타나면 → 파이프라인(pipeline)이 시작됩니다.

### 3. `python-magic` — 추측이 필요 없는 파일 유형 감지

파일 확장자는 항상 거짓말을 합니다.

```python
import magic

file_type = magic.from_file("incoming/report", mime=True)
print(file_type)
```

이 라이브러리가 중요한 이유:

*   보안(Security)
*   올바른 파싱(parsing)
*   잘못된 가정 방지

이 하나의 라이브러리 덕분에 손상된 업로드 파일을 처리하는 실수를 수없이 막을 수 있었습니다.

### 4. `cerberus` — 확장 가능한 스키마(Schema) 검증

나쁜 데이터가 유입되면 자동화는 무력해집니다.

```python
from cerberus import Validator

schema = {
    "name": {"type": "string"},
    "size": {"type": "integer", "min": 1}
}

v = Validator(schema)
v.validate({"name": "file.pdf", "size": 2048})
```

`cerberus` 덕분에 데이터가 저장소나 다운스트림(downstream) 시스템에 도달하기 전에 메타데이터를 검증할 수 있었습니다.

항상 빠르게 실패를 감지해야 합니다.

### 5. `diskcache` — 인프라(Infrastructure) 없이 영구 캐싱(Caching)

동일한 파일을 두 번 처리하는 것은 시간 낭비입니다.

```python
from diskcache import Cache

cache = Cache("cache_dir")

if not cache.get(file.name):
    cache[file.name] = "processed"
```

레디스(Redis)도, 별도의 서버도 필요 없습니다.
이 라이브러리 하나만으로도 파이프라인 실행 시간을 60% 단축할 수 있었습니다.

### 6. `tenacity` — 실패를 이해하는 자동화

API는 실패할 수 있고, 디스크(Disk)는 오작동할 수 있으며, 네트워크(Network)는 불안정할 수 있습니다.

```python
from tenacity import retry, wait_fixed, stop_after_attempt

@retry(wait=wait_fixed(2), stop=stop_after_attempt(3))
def archive_file(path):
    path.rename(ARCHIVE / path.name)
```

재시도(retry)는 임시방편이 아닙니다. 그것은 정책입니다.

`tenacity`는 이러한 정책을 명시적으로 구현할 수 있게 해줍니다.

### 7. `humanize` — 사람이 실제로 읽을 수 있는 로그(Log)

새벽 3시에 원시 타임스탬프(timestamp)는 아무런 의미가 없습니다.

```python
import humanize
import time

print(humanize.naturaltime(time.time() - 300))
```

이제 로그는 다음과 같이 표시됩니다:
"5 minutes ago" (5분 전)
"2 hours ago" (2시간 전)

작은 변화지만, 사용자 경험(UX) 측면에서는 큰 성과입니다.

### 8. `structlog` — 깔끔한 로깅

`print` 문은 확장성이 없습니다. JSON 로그는 확장성이 있습니다.

```python
import structlog

log = structlog.get_logger()

log.info(
    "file_processed",
    filename="report.pdf",
    status="archived"
)
```

이 라이브러리 덕분에 자동화 디버깅(debugging)이 지루해졌습니다. 이는 칭찬입니다.

### 9. `schedule` — 경량 태스크 오케스트레이션(Task Orchestration)

일부 태스크(task)는 이벤트 기반이어서는 안 됩니다.

```python
import schedule
import time

def cleanup():
    print("Running cleanup")

schedule.every().day.at("02:00").do(cleanup)

while True:
    schedule.run_pending()
    time.sleep(1)
```

에어플로우(Airflow)도, DAG 지옥도 없습니다. 오직 논리(logic)만 있습니다.

### 10. `python-box` — 골치 아픈 설정(Config) 관리 해결

중첩된 설정은 고통스러워서는 안 됩니다.

```python
from box import Box

config = Box({
    "paths": {
        "inbox": "incoming",
        "archive": "archive"
    }
})

print(config.paths.archive)
```

깔끔한 설정은 프로덕션(production) 환경에서 실수를 줄여줍니다.

### 11. `invoke` — 안정적으로 Bash 스크립트(Bash Scripts) 대체

자동화 배포(deployment)도... 자동화되어야 합니다.

```python
from invoke import task

@task
def run(c):
    c.run("python pipeline.py")
```

Bash 스크립트는 낡아가지만, 파이썬 태스크는 발전합니다.

이것이 제 제어 플레인(control plane)이 되었습니다.

### 내가 배운 어려운 교훈

대부분의 자동화 실패는 기술적인 문제가 아닙니다.

그것은 아키텍처(architectural)적 나태함에서 비롯됩니다:

*   과도한 의존성(dependencies)
*   지나친 추상화(abstraction)
*   지루해 보이는 도구에 대한 존중 부족

위에 소개된 라이브러리들은 제가 더 빠르게 구축하는 데 도움이 되었을 뿐만 아니라, 코드를 삭제하는 데도 기여했습니다.

그것이 진정한 힘입니다.
