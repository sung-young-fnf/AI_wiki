# 파이썬 변수가 스프레드시트처럼 스스로 업데이트된다면 어떨까요?

엑셀 사용자들이 파이썬을 접하는 순간, 그들의 첫 질문은 항상 똑같습니다. "왜 내 변수들은 스스로 업데이트되지 않는 거죠?"

*(비회원 독자께서는 여기에서 전문을 읽으실 수 있습니다)*

파이썬 코드를 10년간 작성하면서, 저는 수동적인 상태 관리(manual state management)에 익숙해졌다고 생각했습니다. 그런데 어느 날 한 재무 분석가가 세율 변경으로 인한 하위 셀(downstream cells) 재계산을 잊어버려 가격 책정 모델을 디버깅하는 데 4시간을 소비하는 것을 보았습니다. 그들은 저를 보며 물었습니다. "파이썬이 뭐가 어디에 종속되는지 그냥 알 수는 없을까요?"

이제는 가능해졌습니다.

## 명시적인 방식의 문제점

기존 파이썬의 상태 관리(state management)는 콜백(callback), 옵저버(observer), 상투적인 코드(boilerplate)의 난무였습니다. 데이터가 변경될 때 대시보드를 업데이트하고 싶으신가요? 콜백을 작성해야 합니다. 가격이 변동될 때 총계를 재계산해야 하나요? 또 다른 콜백이 필요합니다. 반응형 UI(reactive UI)를 구축하시겠다고요? 이벤트 핸들러(event handler)를 좋아하시길 바랍니다.

```python
# 기존 방식: 수동 종속성 지옥
class Order:
    def __init__(self):
        self._price = 0
        self._tax_rate = 0.08
        self._callbacks = []
    
    @property
    def price(self):
        return self._price
    
    @price.setter
    def price(self, value):
        self._price = value
        self._notify()
    
    @property
    def total(self):
        return self._price * (1 + self._tax_rate)
    
    def _notify(self):
        for callback in self._callbacks:
            callback(self.total)
```

모든 종속성은 수동 연결(manual wiring)이라는 지뢰밭과 같습니다. 콜백 하나라도 놓치면, 시스템은 조용히 동기화가 어긋나게 됩니다.

## Reaktiv의 등장: 파이썬의 스프레드시트

Reaktiv는 엑셀의 자동 종속성 추적(automatic dependency tracking) 기능을 파이썬으로 가져온 라이브러리입니다. 추적된 변수(tracked variables)로 반응형 상태(reactive state)를 정의한 다음, 종속성이 변경될 때 자동으로 재계산되는 계산된 값(computed values)을 생성합니다.

```python
from reaktiv import tracked, compute

# 진실의 근원(source of truth)을 정의합니다
price = tracked(100)
tax_rate = tracked(0.08)

# 계산된 값은 그저 '알아서' 작동합니다
total = compute(lambda: price.value * (1 + tax_rate.value))

print(total.value)  # 108.0

# 근원을 변경하고 마법을 지켜보세요
price.value = 200
print(total.value)  # 216.0 (자동으로 재계산됨!)
```

콜백도, 수동 알림(manual notifications)도, 상투적인 코드(boilerplate)도 없습니다. 그저 흐르는 상태(state)만이 있을 뿐입니다.

## 작동 방식: 숨겨진 비밀

Reaktiv는 반응형 그래프 추적(reactive graph tracking)이라는 기술을 사용합니다. `compute` 함수가 실행될 때, 속성 접근(property accesses)을 가로채서 동적으로 종속성 그래프(dependency graph)를 구축합니다. 각 추적된 변수는 누가 자신을 지켜보고 있는지 알고 있으며, 변경될 경우 직접적인 관찰자(direct observers)만을 무효화합니다.

다음은 우리의 세금 계산기에 대한 종속성 그래프입니다.

```
┌─────────────┐      ┌─────────────┐
│ price (100) │      │ tax (0.08)  │
└──────┬──────┘      └──────┬──────┘
       │                    │
       └────────┬───────────┘
                │
         ┌──────▼──────┐
         │ total (108) │
         └─────────────┘
```

`price`를 변경하면 Reaktiv는 그래프를 따라 `total`을 업데이트합니다. `tax_rate`를 변경해도 결과는 같습니다. 그래프는 자동으로 구축되므로, 종속성을 명시적으로 선언할 필요가 없습니다.

## 성능: 놀랍도록 빠르다

저는 10,000개의 파생 셀(derived cells)을 가진 재무 모델에서 Reaktiv를 기존 옵저버 패턴(observer patterns)과 비교하여 벤치마킹했습니다. 그 결과는 저를 놀라게 했습니다.

| Method           | Setup Time | Update Time | Lines of Code |
| :--------------- | :--------- | :---------- | :------------ |
| Manual Callbacks | 12.3ms     | 0.8ms       | 45            |
| reaktiv          | 0.1ms      | 1.2ms       | 8             |

Reaktiv는 설정(setup) 시간이 123배 빠르고, 업데이트(update)는 50% 느릴 뿐입니다. 이는 유지보수 비용(maintenance costs)을 고려할 때 충분히 감수할 만한 절충점(tradeoff)입니다. 절약된 37줄의 코드는 발생할 수 있는 버그 37개를 줄여주는 것과 같습니다.

## 실제 사용 사례

### 1. 재무 모델(Financial Models)

이 모든 것이 시작된 곳입니다. 이제 포트폴리오 관리자(portfolio manager)는 Reaktiv를 사용하여 실시간으로 업데이트되는 위험 모델(risk models)을 구축합니다. 시장 데이터가 스트리밍(streaming)되면, 모든 파생 지표(derived metric) (VaR, 샤프 비율(Sharpe ratio), 베타(beta))가 자동으로 재계산됩니다.

```python
market_price = tracked(150.0)
position_size = tracked(1000)
volatility = tracked(0.25)

portfolio_value = compute(lambda: market_price.value * position_size.value)
risk_exposure = compute(lambda: portfolio_value.value * volatility.value)
```

### 2. 반응형 대시보드(Reactive Dashboards)

한 데이터 과학자(data scientist)는 필터가 자동으로 연쇄적으로 적용되는 스트림릿(Streamlit) 대시보드를 만들었습니다. 날짜 범위를 변경하면, 단 하나의 콜백(callback) 없이 모든 차트, 표, 통계가 업데이트됩니다.

### 3. 게임 상태 관리(Game State Management)

한 인디 게임 개발자(indie game developer)는 플레이어의 체력, 위치, 인벤토리(inventory)를 반응적으로 추적합니다. 체력이 20% 미만으로 떨어지면, UI가 자동으로 저체력 경고를 표시하며, 게임 루프(game loop)에서 수동으로 상태를 확인할 필요가 없습니다.

```python
health = tracked(100)
is_low_health = compute(lambda: health.value < 20)

# UI 컴포넌트는 is_low_health만 지켜봅니다
# 게임 로직은 UI 코드를 건드리지 않습니다
```

### 4. 과학 시뮬레이션(Scientific Simulations)

한 기후 연구원(climate researcher)은 대기층(atmospheric layers)에 걸친 온도 종속성을 모델링합니다. 지표면 온도가 변하면, 모든 층의 열 플럭스(heat flux)가 자동으로 재계산되어 물리적 일관성(physical consistency)을 유지합니다.

## 엑셀과의 유사점: 왜 중요한가

엑셀의 진가는 그리드(grid)가 아니라 반응형 계산 모델(reactive computation model)에 있습니다. 엑셀 사용자 중 90%는 "반응형 프로그래밍(reactive programming)"이 무엇을 의미하는지 모르지만, A1 셀을 변경하면 B2 셀이 자동으로 업데이트된다는 것을 직관적으로 이해합니다.

Reaktiv는 파이썬에도 동일한 직관을 부여합니다. 제 분석가 친구는 옵저버 패턴(observer patterns)을 이해할 필요 없이, `price.value`가 변경되면 이를 사용하는 모든 것이 함께 변경된다는 사실만 알면 되었습니다.

## 어두운 면: 언제 사용하지 않아야 할까

Reaktiv는 마법이 아닙니다. 리스트(lists)나 딕셔너리(dicts) 내부의 변경(mutations)을 추적할 수는 없습니다 (이를 위해서는 `tracked_list`를 사용하세요). 또한 속성 접근(property access)에 약간의 오버헤드(overhead)가 추가됩니다. 그리고 간단한 스크립트(scripts)에는 과도한 기능(overkill)일 수 있습니다.

```python
# 이것은 작동하지 않습니다 (추적된 객체 내부의 변경)
items = tracked([1, 2, 3])
items.value.append(4)  # 자동 업데이트 없음!

# 대신 tracked_list를 사용하세요
from reaktiv import tracked_list
items = tracked_list([1, 2, 3])
items.append(4)  # 작동합니다!
```

## 미래: 반응형 파이썬?

Reaktiv는 더 큰 흐름의 일부입니다. ReactPy(반응형 UI) 및 River(반응형 머신러닝)와 같은 라이브러리들은 파이썬 프로그램이 스프레드시트처럼 흐르는 생태계(ecosystem)를 구축하고 있습니다.

우리는 언어가 "이것을 하고, 저것을 해라"는 명령형(imperative) 방식에서 "이것은 저것이다"는 선언형(declarative) 방식으로 진화하는 것을 보고 있습니다. 이는 미묘한 변화이지만, 상태(state)에 대한 우리의 사고방식을 바꿉니다.
