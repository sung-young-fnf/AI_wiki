# 공개 IP, 라우터 설정 없이 집 컴퓨터를 어디서든 연결하는 방법 — "편집증적인" 에디션

집 컴퓨터를 어디서든 사용할 수 있다고 상상해 보세요.

카페에서 노트북을 켜고 주소를 입력하면 집 컴퓨터로 바로 연결되어 파일, 도구, 서버 등 모든 것을 이용할 수 있습니다. "그 폴더 동기화했나?" 또는 "집에 가서 해야겠다"는 걱정 없이 작업을 계속할 수 있습니다.

이것이 모두의 꿈이지만, 막상 구현하려고 하면 오래된 규칙에 부딪히게 됩니다.

외부에서 컴퓨터에 연결하려면, 그 컴퓨터가 외부에서 접근 가능해야 합니다.

전통적으로 이는 공개 IP(Public IP)와 포트 포워딩(Port Forwarding)을 의미합니다.

그래서 인터넷 서비스 제공업체(ISP)에 전화해서 "공개 IPv4 / CGNAT 제거"나 "고정 IP(Static IP) 추가"를 요청합니다. 그들이 서비스를 제공하더라도 종종 유료이거나 속도가 느리며, 가끔은 CGNAT이라는 재미있는 약어를 힘든 방식으로 배우게 됩니다. 그런 다음 라우터 설정을 파고들어 포트를 열고 방화벽을 해제하다 보면, "빠른 원격 접속" 계획이 어느새 복잡한 네트워크 사이드 퀘스트처럼 느껴집니다.

1부에서는 탈출구를 제시했습니다: Tailscale + NoMachine.

*   Tailscale은 기기들에 어디서든 작동하는 사설 네트워크를 제공합니다.
*   NoMachine은 타이핑이 실제로 자연스럽게 느껴지는 진정한 원격 데스크톱을 제공합니다.

라우터나 공개 IP는 필요 없었습니다. 몇 분 만에 설정이 가능했죠.

그리고 잘 작동했습니다. 많은 사람이 시도했죠.

"Tailscale은 클로즈드 소스(closed-source) 서버에 의존합니다."
"제어 플레인(control plane)이 모든 것을 조정하는데 어떻게 안전하다는 거죠?"
"그렇게 많은 접근 권한을 누구에게도 맡기고 싶지 않습니다."

솔직히 말해서, 이는 타당한 본능입니다.

그래서 2부에서는 동일한 편리함을 유지하되, 가장 큰 심리적 장애물인 신뢰 문제를 해결하고자 합니다!

저희는 Tailscale 클라이언트(매우 훌륭하므로)와 NoMachine(실용적이므로)을 그대로 사용하고, 호스팅된 조정 서버(coordination server)만 자체 호스팅(self-hosted) 서버로 교체할 것입니다:

Headscale. (Tailscale과 함께 사용합니다.)

## 왜 여전히 Tailscale이 필요할까요?

Headscale은 이 솔루션에서 오직 제어 서버(control server), 즉 "조정 두뇌" 역할만 합니다. VPN 터널을 자체적으로 생성하지는 않습니다.

각 기기에는 여전히 다음 기능을 수행하는 클라이언트가 필요합니다.

*   WireGuard 인터페이스 생성
*   트래픽 암호화
*   NAT 순회/중계(NAT traversal/relaying)
*   패킷 라우팅

이 클라이언트가 바로 Tailscale 클라이언트입니다 (또는 일부 설정에서는 headscale-client 대안과 같은 다른 호환 클라이언트도 있지만, 표준적이고 신뢰할 수 있는 것은 Tailscale입니다).

따라서 둘의 관계는 다음과 같습니다.

*   **Headscale** = 자체 호스팅 제어 플레인 (네트워크 내 기기 관리, 키, 조정)
*   **Tailscale 클라이언트** = 기기의 데이터 플레인 에이전트 (실제 암호화된 네트워크 통신)

이것이 Headscale과 Tailscale을 모두 사용하는 이유입니다. Headscale은 Tailscale의 호스팅 서버를 대체하는 것이지, 클라이언트 소프트웨어를 대체하는 것이 아닙니다.

## 10초 요약 진실

*   트래픽은 양쪽 모두 종단 간 암호화됩니다(end-to-end encrypted).
*   조정 서버는 제어 플레인(기기 검색/인증/키 배포)이며, 트래픽 경로가 아닙니다.
*   Tailscale의 호스팅 조정 서버는 클로즈드 소스입니다.
*   Headscale을 사용하면 자체 제어 서버를 실행할 수 있습니다.

Headscale은 이러한 제어 서버의 오픈 소스(open-source), 자체 호스팅 구현입니다.

이제 구축해봅시다.

## 필요한 것

*   작은 VPS (Hetzner가 완벽하며, 저렴한 VPS면 무엇이든 좋습니다, 예: ovh)
*   제어할 수 있는 도메인(Domain) + 서브도메인(Subdomain) (예: `hs.mydomain.cc` 사용)
*   집 컴퓨터 (Windows/Linux/macOS)
*   여행용 컴퓨터 (MacBook 등)
*   양쪽 끝에 NoMachine (편집, 타이핑, 클립보드 기능 때문)

> 다음 과정에서 잘못되면 모든 것이 조용히 망가질 수 있으니 주의하세요.

### 0.1 Hetzner DNS 콘솔에서 DNS 존(Zone) 생성

Hetzner DNS 콘솔에서:

*   존 추가: `mydomain.cc`
*   Hetzner는 세 개의 권한 있는 네임서버(authoritative nameservers)를 보여줄 것입니다.

### 0.2 등록기관에 Hetzner 네임서버 사용 지시

등록기관(예: Spaceship)에서 사용자 지정 네임서버를 세 개의 Hetzner 네임서버로 설정합니다.

> 이 단계를 건너뛰면 Hetzner 기록은 영원히 "비활성" 상태로 유지됩니다.

### 0.3 Headscale용 A 레코드(Record) 생성

Hetzner 존 `mydomain.cc`에서 다음을 추가합니다.

*   유형(Type): A
*   호스트(Host): `hs`
*   값(Value): `<VPS IPv4 주소>` (예: `2x.xx.xx.xxx`)

그런 다음 어디에서든 다음 명령으로 확인합니다.

```bash
dig A hs.mydomain.cc +short
```

이 명령이 VPS IP를 반환하지 않으면, 여기서 멈추고 먼저 DNS를 수정해야 합니다.

```bash
dig @1.1.1.1 A hs.mydomain.cc +short 
dig @8.8.8.8 A hs.mydomain.cc +short
```

이들은 도메인을 Hetzner의 IPv4 주소로 올바르게 확인해야 합니다!

## 1단계 — VPS에 Headscale 설치 (Ubuntu 22.04+ / Debian 12+)

Headscale 자체 문서에서는 서비스/사용자 설정을 위해 Debian/Ubuntu에 공식 `.deb` 패키지 설치를 권장합니다.

브라우저에서 다음 링크를 엽니다 (복사/붙여넣기):
[https://headscale.net/stable/setup/install/official/](https://headscale.net/stable/setup/install/official/)

다운로드한 `.deb`를 VPS에 설치합니다.

```bash
sudo apt update
sudo apt install -y ./headscale_*_linux_amd64.deb
sudo systemctl enable --now headscale
sudo systemctl status headscale
```

또는 다음과 같이 수동으로 설치할 수 있습니다.

Headscale 웹사이트로 이동하여 최신 Headscale 패키지 링크를 클릭합니다. (예: 현재 최신 버전은 v0.28.0입니다.)

```bash
HEADSCALE_VERSION="" # 위 URL에서 최신 버전 확인, 예: "X.Y.Z" (참고: "v" 접두사 붙이지 마세요!)
HEADSCALE_ARCH="" # 시스템 아키텍처, 예: "amd64"
wget --output-document=headscale.deb \
 "https://github.com/juanfont/headscale/releases/download/v${HEADSCALE_VERSION}/headscale_${HEADSCALE_VERSION}_linux_${HEADSCALE_ARCH}.deb"
```

다음과 같이 내용을 채웁니다.

```bash
HEADSCALE_VERSION="0.28.0" #(참고: "v" 접두사 붙이지 마세요!)
HEADSCALE_ARCH="arm64" # 시스템 아키텍처, 예: "amd64"
wget --output-document=headscale.deb \
 "https://github.com/juanfont/headscale/releases/download/v${HEADSCALE_VERSION}/headscale_${HEADSCALE_VERSION}_linux_${HEADSCALE_ARCH}.deb"
```

`headscale.deb`가 다운로드됩니다.

```bash
sudo apt install ./headscale.deb

# 설정 (아무것도 하지 않음)
sudo nano /etc/headscale/config.yaml

# 프로세스 시작
sudo systemctl enable --now headscale

# 모든 것이 작동하는지 확인
sudo systemctl status headscale
```

## 2단계 — HTTPS 설정 (Caddy, 최소 구성)

Tailscale 설치 및 설정(또한 필요함)에 대해서는 이전 글("공개 IP나 라우터 설정 없이 집 컴퓨터를 어디서든 연결하는 방법")의 1단계 및 후속 단계를 참조하세요.

Headscale은 TLS 없이 실행될 수 있으며, 리버스 프록시(reverse proxy)가 HTTPS를 처리하도록 할 수 있습니다.

```bash
sudo apt install -y caddy
```

`/etc/caddy/Caddyfile`을 생성합니다.

```bash
sudo nano /etc/caddy/Caddyfile
```

내용은 다음과 같습니다.

```
hs.mydomain.cc {
 reverse_proxy 127.0.0.1:8080 
}
```

이제 Headscale에 공개 URL을 알리고 로컬에 바인딩(bind)하도록 설정합니다. 다음 파일을 편집합니다.

```bash
sudo nano /etc/headscale/config.yaml
```

다음과 같이 설정합니다 (Headscale 문서의 리버스 프록시 패턴과 일치시킵니다).

```yaml
server_url: https://hs.yourdomain.com
listen_addr: 127.0.0.1:8080
tls_cert_path: ""
tls_key_path: ""
# 파일의 나머지 부분은 그대로 둡니다 - 변경 사항을 저장하세요!
```

`tls_cert_path`와 `tls_key_path`는 비워 둡니다. `Caddy`가 자동으로 설정할 것입니다!

재시작합니다.

```bash
sudo systemctl restart headscale 
sudo systemctl reload caddy
```

(또한: VPS 방화벽이 TCP 80/443을 허용해야 합니다.)

VPS에서 빠른 건전성 확인:

```bash
sudo ss -ltnp | egrep ':80|:443|:8080'
```

다음과 같이 보여야 합니다.

*   Caddy가 :80 및 :443에서 수신 대기
*   Headscale이 127.0.0.1:8080에서 수신 대기

### 2.5단계 — Hetzner 방화벽: 포트 열기 (필수)

UFW가 "비활성" 상태이더라도 Hetzner Cloud 방화벽은 여전히 트래픽을 차단할 수 있습니다.

Hetzner 콘솔 → 방화벽 규칙에서 설정합니다.

그런 다음 노트북에서 확인합니다.

```bash
nc -vz hs.mydomain.cc 443 
curl -I https://hs.mydomain.cc
```

443 포트가 응답하지 않으면, Tailscale이 제어 서버와 통신할 수 없습니다.

## 3단계 — 사용자 생성 + 접속 키(Join Key) 생성

사용자를 생성합니다.

```bash
sudo headscale users create home 
sudo headscale users list
```

중요: 여러분의 버전에서는 사전 인증 키(preauth keys)에 숫자 사용자 ID가 필요할 수 있습니다.

따라서 `users list`에서 다음과 같이 표시된다면:

```
ID Name 
1 home
```

재사용 가능한 키(기기 온보딩 시 권장)를 생성하고 충분한 유효 기간을 부여합니다.

```bash
sudo headscale preauthkeys create --user 1 --reusable --expiration 720h
```

키를 복사합니다.

**주의사항: "authkey expired"**

너무 오래 기다려서 키가 만료되면, 클라이언트가 "authkey expired" 메시지와 함께 로그아웃됩니다.

새로운 키를 생성하고 `tailscale up` 명령을 다시 실행하세요.

## 4단계 — 기기를 제어 플레인에 연결

이 단계에서 사람들이 실수로 이전의 Tailnet에 머무는 경우가 많습니다.

규칙:

이전에 호스팅된 Tailscale을 사용했다면, 서버를 가리키도록 깨끗하게 "down"하고 "up"해야 합니다.

### Linux (Acer)
Linux에서:

```bash
sudo tailscale down
sudo tailscale up --login-server https://hs.mydomain.cc --authkey <PASTE_KEY>
```

### macOS에서:

```bash
sudo tailscale down 
sudo tailscale up --login-server https://hs.mydomain.cc --authkey <PASTE_KEY>
```

### Windows (Scoop 이용)

```powershell
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
iwr -useb get.scoop.sh | iex
scoop bucket add extras
scoop install tailscale
tailscale up --login-server https://hs.mydomain.cc --authkey <PASTE_KEY>
```

(`--login-server` 플로우는 Tailscale 클라이언트가 사용자 지정 제어 서버/Headscale을 위해 구성되는 방식과 정확히 일치합니다.)

확인합니다.

```bash
tailscale status 
tailscale ip -4
```

익숙한 100.x.y.z 주소를 볼 수 있을 것입니다.

만약 다음 메시지가 보인다면:

`failed to resolve hs.mydomain.cc`
또는 `Temporary failure in name resolution but dig @1.1.1.1 hs.mydomain.cc works…`

…이는 로컬 리졸버(resolver) 문제입니다.

### Linux 해결법 (Acer)

이는 Wi-Fi 인터페이스에서 깨끗한 DNS를 강제합니다.

```bash
sudo resolvectl dns wlp1s0 1.1.1.1 8.8.8.8
sudo resolvectl domain wlp1s0 "~."
sudo resolvectl flush-caches
sudo systemctl restart systemd-resolved
sudo systemctl restart NetworkManager
```

### macOS 해결법

활성 네트워크 서비스(일반적으로 "Wi-Fi")에서 DNS 서버를 설정합니다.

```bash
sudo networksetup -setdnsservers "Wi-Fi" 1.1.1.1 8.8.8.8
sudo dscacheutil -flushcache
sudo killall -HUP mDNSResponder
```

이후:

```bash
dig A hs.mydomain.cc +short
```

명령이 VPS IP를 반환해야 합니다.

## 5단계 — NoMachine 설치 (실제로 체감하는 부분입니다)

NoMachine 다운로드 페이지:
[https://www.nomachine.com/everybody/nomachine-for-your-os](https://www.nomachine.com/everybody/nomachine-for-your-os)

NoMachine은 기본적으로 4000 포트에서 NX를 사용합니다.

노트북에서 집 컴퓨터로 빠른 테스트:

```bash
nc -vz <HOME_TAILSCALE_IP> 4000
```

그런 다음 NoMachine에서:

*   호스트(Host): `<HOME_TAILSCALE_IP>`
*   포트(Port): `4000`
*   연결(Connect)

이제 편집, 클립보드 사용, OAuth 버튼 동작이 모두 정상적입니다. X11 포워딩이나 Electron 재그리기 지터(jitter)와 씨름할 필요가 없습니다.

NoMachine 설정 및 사용에 대한 자세한 지침은 이전 글의 1부를 참조하세요.

## "이것이 실제로 신뢰 문제를 해결하는가?"

사람들이 주로 논쟁했던 주요 문제는 해결합니다.

*   Tailscale 호스팅 버전: 클로즈드 소스 조정 서버를 신뢰합니다.
*   Headscale 사용: 조정 서버를 직접 실행합니다.

제거되지 않는 것은: 엔드포인트 보안(endpoint security)은 여전히 엔드포인트 보안입니다. 원격 접근은 가장 약한 장치와 암호만큼만 안전합니다.

따라서 여기에 짧고 실용적인 강화 단계가 있습니다.

## 2분 만에 강화 (이것을 하고 편안히 잠드세요)
### 1) NoMachine을 VPN으로만 잠급니다.

Linux 홈 컴퓨터에서 (UFW 예시):

```bash
sudo ufw allow in on tailscale0 to any port 4000 proto tcp 
sudo ufw deny 4000/tcp
```

Windows에서: Tailscale 범위 100.64.0.0/10에서만 4000 포트를 허용합니다 (고급 방화벽 GUI도 작동합니다).

### 2) ACLs: 기본적으로 거부

노트북만 데스크톱(4000) 및 SSH(22)에 접근할 수 있도록 시작합니다. 풀 메시(full mesh)는 아닙니다.

### 3) 계정 보안

가능한 한 강력한 암호 + 2단계 인증(2FA). 대부분의 침해는 "계정이 털린 것"이지 "WireGuard가 고장 난 것"이 아닙니다.

이제 1부와 동일한 "어디서든 집 컴퓨터" 설정을 갖추게 되었지만, 조정 서버는 여러분의 통제하에 있습니다.

조만간 이 솔루션의 Docker 버전을 만들 수도 있습니다. 하지만 Hetzner 서버에서 Docker를 실행하고 싶지 않다면, 이 글의 설정이 도움이 될 것입니다!

우리 중 "편집증적인(paranoid)" 분들을 포함한 모두에게 행복한 원격 컴퓨팅이 되기를 바랍니다!

## 90초 만에 문제 해결 (실패의 95%를 잡는 5가지 확인 사항)
### 1) 클라이언트에서 DNS가 확인됩니까?

Linux/macOS 클라이언트에서:

```bash
dig A hs.mydomain.cc +short 
dig @1.1.1.1 A hs.mydomain.cc +short
```

### 2) 클라이언트에서 HTTPS에 연결할 수 있습니까?

```bash
nc -vz hs.mydomain.cc 443 
curl -I https://hs.mydomain.cc | head
```

이것이 실패하면, 거의 항상 Hetzner Cloud 방화벽(80/443이 열려 있지 않음) 또는 잘못된 IP를 가리키는 DNS 문제입니다.

### 3) Caddy가 실제로 VPS에서 수신 대기하고 있습니까?

VPS에서:

```bash
sudo ss -ltnp | egrep ':80|:443|:8080'
```

다음과 같이 보여야 합니다.

*   Caddy on :80 and :443
*   Headscale on 127.0.0.1:8080

### 4) 실제로 Headscale 네트워크에 연결되어 있습니까 (이전 Tailscale 네트워크가 아님)?

각 장치에서:

```bash
tailscale status 
tailscale ip -4
```

장치들이 서로를 보지 못한다면, 깨끗하게 다시 연결합니다.

```bash
sudo tailscale down 
sudo tailscale up --login-server https://hs.mydomain.cc --authkey <NEW_KEY>
```

### 5) "authkey expired" (흔한 문제)

VPS에서 새로운 키를 생성합니다 (재사용 가능 + 온보딩에 충분한 긴 유효 기간).

```bash
sudo headscale users list 
sudo headscale preauthkeys create --user <ID> --reusable --expiration 720h
```

## DNS 해결법 (두 가지 "동일한 버그, 다른 OS" 솔루션)
### Linux (systemd-resolved) — SERVFAIL / 빈 dig 문제 해결

```bash
sudo resolvectl dns wlp1s0 1.1.1.1 8.8.8.8
sudo resolvectl domain wlp1s0 "~."
sudo resolvectl flush-caches
sudo systemctl restart systemd-resolved
sudo systemctl restart NetworkManager
```

### macOS — Tailscale이 DNS를 변경한 후 "can’t resolve" 문제 해결

```bash
sudo networksetup -setdnsservers "Wi-Fi" 1.1.1.1 8.8.8.8
sudo dscacheutil -flushcache
sudo killall -HUP mDNSResponder
```

(선택 사항, Tailscale이 macOS에서 DNS를 건드리지 않게 하려면):

```bash
sudo tailscale set --accept-dns=false
```

## Hetzner 방화벽 알림 (조용한 차단자)

UFW가 비활성 상태이더라도 Hetzner Cloud 방화벽 규칙은 트래픽이 VPS에 도달하기 전에 차단할 수 있습니다.

Hetzner 콘솔의 최소 인바운드 규칙:

*   TCP 80
*   TCP 443
