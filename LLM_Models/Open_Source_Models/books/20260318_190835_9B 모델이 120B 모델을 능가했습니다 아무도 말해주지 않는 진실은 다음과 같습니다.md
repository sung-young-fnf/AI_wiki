하지만 "소형 모델이 거대 모델을 파괴한다"는 이야기에 휩쓸리기 전에, 이 비교가 겉보기보다 더 미묘한 이유를 설명해 드리겠습니다.

## 맥락이 필요한 비교 (The Comparison That Needs Context)

GPT-oss-120B는 총 1,170억 개의 파라미터(parameters)를 가지고 있지만, 포워드 패스(forward pass)당 51억 개의 파라미터만 활성화하는 전문가 혼합(Mixture-of-Experts, MoE) 모델입니다 [1][5]. Qwen3.5–9B는 모든 토큰(token)에 대해 90억 개의 파라미터 전부를 활성화하는 Dense 모델(dense model)입니다.

따라서 사람들이 "9B 모델이 120B 모델을 이겼다"고 말할 때, 총 파라미터 수(total parameter counts)에 대해서는 기술적으로 정확합니다. 그러나 토큰(token)당 실제 연산량(compute) 측면에서는 Qwen3.5–9B가 GPT-oss-120B보다 약 1.8배 더 많은 파라미터를 활성화합니다. 이 이야기는 사실 작은 모델이 큰 모델을 이겼다는 것이 아닙니다. 51억 개의 활성 파라미터(active parameters)를 가지고 있으면서 우연히 메모리(memory)에 1,170억 개의 총 가중치(total weights)를 가지고 있는 모델을 9B Dense 모델이 능가했다는 이야기입니다.

여전히 인상적입니다. 하지만 "13배 더 많은 파라미터, 더 낮은 점수"라는 것과는 다른 종류의 인상적인 결과입니다.

VentureBeat는 80.1점을 보고했습니다. 81.7점과 71.5점의 격차는 엄청납니다. 81.7점과 80.9점의 격차는 미미하며 실행 간 편차(run-to-run variance) 범위 내에 있습니다. 어떤 비교를 신뢰하느냐에 따라 전체 이야기가 바뀝니다.

## Qwen3.5 Small의 실제 모습 (What Qwen3.5 Small Actually Is)

Qwen3.5 Small 시리즈는 Qwen의 주력 397B MoE 모델 [9]과 동일한 기본 아키텍처(foundational architecture)를 기반으로 2026년 3월 2일에 출시된 네 가지 소형 Dense 모델(compact dense models) 제품군입니다 [2][6]. 이 모델들은 더 큰 모델에서 기능을 제거하거나 단순화한 버전이 아닙니다. Gated Delta Networks와 주력 모델의 설계 원칙을 결합한 하이브리드 아키텍처(hybrid architecture)를 사용하여 처음부터 작고 효율적으로 설계되었습니다.

네 가지 모델은 배포 계층(deployment tiers)별로 신중하게 배치되었습니다:

*   **Qwen3.5–0.8B 및 2B**: 엣지 디바이스(edge devices)용으로 설계되었습니다. 스마트폰 칩, IoT 하드웨어(hardware), 오프라인 추론(offline inference) 등을 생각해 보세요. r/LocalLLaMA 커뮤니티 벤치마크(community benchmarks)에 따르면 2B 모델은 MLX 최적화(optimization)를 통해 아이폰(iPhone)에서 초당 30~50개 토큰(tokens)을 원활하게 생성하며 실행됩니다.
*   **Qwen3.5–4B**: 경량 에이전트(lightweight agents)를 위한 다중 모달(multimodal) 기반으로 포지셔닝되었습니다. 텍스트와 이미지가 처음부터 동일한 잠재 공간(latent space)을 공유하는 초기 융합 훈련(early-fusion training)을 통해 네이티브 다중 모달(native multimodal)을 지원합니다. 이는 언어 모델(language model)에 비전 인코더(vision encoder)를 추가하는 구식 어댑터 기반(adapter-based) 접근 방식과는 다릅니다. 4B 모델은 거의 8배 작음에도 불구하고 ScreenSpot Pro와 같은 에이전트 평가(agent evaluations)에서 이전 세대 Qwen3-VL-30B-A3B의 성능과 일치한다고 보고되었습니다.
*   **Qwen3.5–9B**: 소형 시리즈의 주력 모델입니다. 이 모델이 사람들의 관심을 사로잡았습니다.

## 핵심 역할을 하는 아키텍처 (The Architecture Doing the Heavy Lifting)

네 가지 모델 모두 선형 어텐션 블록(linear attention blocks)과 완전 소프트맥스 어텐션 블록(full softmax attention blocks)의 비율이 3:1인 Gated DeltaNet 하이브리드 아키텍처(hybrid architecture)를 사용합니다 [2][8]. 선형 어텐션 레이어(linear attention layers)는 일정한 메모리 복잡도(constant memory complexity)를 제공하는데, 이를 통해 Qwen은 2B 모델에 262,144개 토큰(token)의 네이티브 컨텍스트 창(native context window)을 제공하면서도 실행 불가능하게 만들지 않았습니다 [7]. 완전 어텐션 블록(full attention blocks)은 정확한 토큰(token) 관계가 중요한 정밀 작업을 처리합니다.

훈련(training)은 대규모 시뮬레이션(simulated) 환경에서 스케일드 강화 학습(Scaled Reinforcement Learning, SRL)을 사용했습니다. 이는 벤치마크(benchmark) 데이터셋(datasets)에 대한 단순한 미세 조정(fine-tuning)과는 다릅니다. Qwen 문서에 따르면, RL 훈련은 도구 사용 및 구조화된 다단계 워크플로(multi-step workflows)를 포함한 실제 환경 적응성(adaptability)에 최적화되었습니다.

또한 훈련(training) 중 다중 토큰 예측(multi-token prediction)을 사용하여 모델이 한 단계에서 여러 개의 후속 토큰(tokens)을 예측할 수 있도록 함으로써 추론(inference) 속도를 높였습니다 [8]. 그리고 어휘(vocabulary)는 201개 언어에 걸쳐 248,000개 토큰을 포함합니다 [2][7].

한 가지 중요한 점: 소형 모델(0.8B부터 9B까지)은 Dense 모델(dense models)입니다. 주력 397B 및 Qwen3.5 제품군의 일부 중형 모델은 희소 전문가 혼합(sparse Mixture-of-Experts)을 사용하지만, 소형 시리즈는 그렇지 않습니다.

## "내가 실행할 수 있는가"가 중요한 이유 (Why I Care About “Can I Run It”)

저는 박사 과정 동안 꽤 큰 모델들을 다룰 수 있는 행운을 누렸습니다. 그리고 그 경험이 저에게 가르쳐 준 한 가지는 "인상적인 벤치마크(benchmark)"와 "실제로 무언가를 만드는 데 유용한" 것 사이의 간극이 엄청나다는 것입니다.

Qwen3.5 시리즈는 그 간극을 좁힙니다. 1년 전만 해도 로컬(local)에서 다중 모달(multimodal) 모델을 실행하려면 13B+ 파라미터(parameter) 모델과 상당한 VRAM을 가진 전용 GPU가 필요했습니다. Qwen3.5–4B는 262K 컨텍스트 창(context window)으로 텍스트, 이미지, 비디오를 처리합니다.

BF16 정밀도(precision)의 9B 모델은 약 18GB의 VRAM이 필요하며, 이는 양자화(quantization) 없이도 RTX 3090 또는 RTX 4090과 같은 24GB 소비자 GPU에 적합하다는 의미입니다. 4비트 양자화(4-bit quantization)를 사용하면 약 5GB로 줄어들어 Apple Silicon Mac을 포함한 더 넓은 범위의 하드웨어(hardware)에서 실행 가능합니다. 커뮤니티 보고서에 따르면 소비자 하드웨어(consumer hardware)에서 초당 30~50개의 토큰(tokens)을 생성하지만, 처리량(throughput)은 설정에 따라 크게 달라집니다.

의료 AI 도구(healthcare AI tools), 에이전트 파이프라인(agent pipelines) 또는 민감한 데이터를 처리하는 로컬 애플리케이션(local applications)을 구축하는 사람들에게 이것은 의미 있는 변화입니다. API 호출(API calls)이 필요 없고, 데이터가 장치를 떠나지 않으며, 클라우드 엔드포인트(cloud endpoint)로 왕복하는 지연(latency)이 없습니다.

이것은 결코 작은 일이 아닙니다.

## 좋습니다, 그런데 어떻게 실행하나요? (Okay, But How Do You Actually Run It?)

이것은 더 많은 모델 출시 게시물에 포함되었으면 하는 부분입니다. 벤치마크(benchmarks)는 좋지만, 내 기계에서 작동하는 모델이 더 좋습니다.

가장 쉬운 방법은 Ollama입니다. ollama.com에서 설치한 다음 (Mac, Linux, Windows에서 단일 앱 다운로드), 한 가지 명령어를 실행하세요:

```bash
ollama run qwen3.5
```

그러면 기본적으로 9B 모델을 다운로드하여 채팅 세션(chat session)으로 들어갑니다. 특정 크기를 원한다면:

```bash
ollama run qwen3.5:4b    # 더 가볍고, 여전히 다중 모달(multimodal)
ollama run qwen3.5:2b    # 구형 하드웨어(hardware)용
ollama run qwen3.5:0.8b  # 기본적으로 모든 것, 심지어 CPU(CPU)에서도 실행 가능
```

Ollama는 NVIDIA와 Apple Silicon 모두에서 GPU 가속(GPU acceleration)을 자동으로 처리합니다. 설정이 필요 없습니다.

보유한 하드웨어(hardware)에 따라 크기를 선택하세요. 0.8B 모델은 약 1GB가 필요하며, 필요시 CPU(CPU)에서도 실행됩니다. 2B 모델은 약 3GB VRAM이 필요합니다. 4B 모델은 약 4~5GB. 9B 모델은 4비트 양자화(4-bit quantization) 시 약 5GB에 들어갑니다. BF16 전체 정밀도(full precision)에서는 24GB GPU가 필요합니다.

코드에서 사용하고 싶다면, Ollama는 localhost:11434에서 OpenAI 호환 API(OpenAI-compatible API)를 제공합니다:

```python
from ollama import chat
response = chat(
    model='qwen3.5:9b',
    messages=[{'role': 'user', 'content': 'Hello!'}],
)
print(response.message.content)
```

이는 OpenAI API(API)와 이미 연동되는 모든 도구를 단 한 번의 설정 변경만으로 로컬 Qwen3.5에 연결할 수 있음을 의미합니다. API 비용, 속도 제한, 데이터 유출이 없습니다.

프로덕션 워크로드(production workloads) 또는 고처리량 시나리오(high-throughput scenarios)의 경우, 공식 문서에서는 Ollama보다 SGLang, KTransformers 또는 vLLM과 같은 전용 서빙 엔진(serving engines)을 권장합니다.

## 놓치고 있는 점 (What’s Missing)

저는 솔직하게 말하고 싶습니다. 이 모델들에 대한 과대광고(hype)가 이미 증거보다 앞서가고 있기 때문입니다.

첫째, 헤드라인을 장식한 파라미터(parameter) 비교는 오해의 소지가 있습니다. GPT-oss-120B가 토큰(token)당 51억 개의 파라미터만 활성화한다는 점을 언급하지 않고 "9B가 120B를 이겼다"고 말하는 것은 가장 중요한 맥락을 생략하는 것입니다. Qwen3.5–9B는 GPT-oss-120B보다 토큰당 더 많은 활성 연산량(active compute)을 사용합니다. 이는 Qwen이 달성한 것을 폄하하는 것이 아니라, 그 의미를 재정의하는 것입니다.

둘째, GPT-oss-120B의 GPQA Diamond 점수는 추론 수준과 도구 사용 구성에 따라 출처별로 71.5점에서 80.9점까지 크게 다릅니다. 어떤 수치를 사용하느냐에 따라 비교 결과가 매우 다르게 보입니다. 벤치마크(benchmark) 비교는 이러한 조건을 명시해야 합니다.

셋째, 벤치마크(benchmarks)는 강력합니다. 하지만 벤치마크는 통제된 환경입니다. 에이전트 다단계 워크플로(Agentic multi-step workflows)는 그렇지 않습니다. 이러한 모델을 구조화된 작업에 능하게 만드는 동일한 RL 훈련(training)은 체인(chain) 초기의 작은 오류가 5단계쯤에서 무의미한 결과로 이어질 수 있음을 의미하기도 합니다. 프로덕션(production) 사용을 위해서는 직접 테스트해야 합니다.

넷째, 개인 실험용으로 노트북(laptop)에서 9B 모델을 실행하는 것은 괜찮습니다. 실제 제품을 위한 추론(inference) 규모로 실행하는 것은 다른 이야기입니다. 전체 정밀도(full precision)에서는 24GB GPU가 필요합니다. 양자화(quantization)를 사용할 경우, 특정 사용 사례에 대한 품질 절충(quality trade-offs) 평가가 필요합니다.

다섯째, 네 가지 모델 모두 알리바바(Alibaba)에서 나왔습니다. 가중치(weights)의 출처가 중요한 무언가를 구축하고 있다면, 이는 신중하게 고려해야 할 사항입니다.

## 더 큰 그림 (The Bigger Picture)

Qwen3.5 Small 출시는 Qwen3.5 제품군을 완성합니다: 397B 주력 모델(2월 16일 출시), 중형 시리즈(2월 24일), 그리고 이제 소형 모델들입니다. 이 모든 모델은 동일한 아키텍처(architectural) DNA를 공유하며, 모두 네이티브 다중 모달(natively multimodal)을 지원하고, Apache 2.0 라이선스(licensed)로 상업적 사용이 가능합니다.

이것이 시사하는 바는 "소형 및 로컬(small and local)"과 "대형 및 클라우드(large and cloud)" 모델 간의 성능 격차가 대부분의 사람들이 예상했던 것보다 빠르게 무너지고 있다는 것입니다. 소형 모델이 모든 면에서 수조 개의 파라미터(parameter)를 가진 시스템을 마법처럼 따라잡고 있기 때문이 아니라, 많은 실용적인 작업에서 올바른 훈련(training)을 거친 잘 설계된 9B 모델이 이제 API 호출(API call)을 대체할 만큼 충분히 좋아졌기 때문입니다.

저는 이 점에 주목할 가치가 있다고 생각합니다. 벤치마크(benchmarks)가 핵심이어서가 아니라, 우리가 실제로 현실 세계에서 사용할 수 있는 도구들이 중요한 방식으로 발전하고 있음을 의미하기 때문입니다.

이 모델들은 Apache 2.0 라이선스(licensed)로 HuggingFace와 ModelScope에서 이용 가능합니다. 연구 및 미세 조정(fine-tuning)을 위한 기본 모델(Base models)도 포함되어 있습니다. 로컬 추론(local inference)을 실험한다면 4B 모델이 다중 모달(multimodal) 작업에 가장 흥미로운 시작점일 것입니다. 서빙(serving)의 경우, 공식 문서에서는 Ollama와 같은 간단한 도구보다 SGLang, vLLM 또는 KTransformers를 권장합니다.

저는 AI, 헬스케어 ML, 그리고 Towards Deep Learning에서 주목할 만한 가치가 있다고 생각하는 것들에 대해 글을 씁니다. 이 글이 유용했다면 팔로우해주세요.
