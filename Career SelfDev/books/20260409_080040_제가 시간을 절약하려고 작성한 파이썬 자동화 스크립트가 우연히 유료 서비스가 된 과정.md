제가 시간을 절약하려고 작성한 파이썬 자동화 스크립트가 우연히 유료 서비스가 된 과정
개인적인 스크립트가 클라이언트들이 매달 비용을 지불하는 시스템으로 발전하기까지

### 사소한 업무에 허덕이며 파이썬이 구원해 줄 거라 착각했습니다

모든 프리랜서는 똑같이 말합니다. "나중에 자동화할 거야."

그 '나중'은 결코 오지 않습니다. 대신, 파일 이름을 수동으로 변경하고, CSV를 정리하고, 같은 이메일을 보내고, 같은 스프레드시트를 업데이트하고, 같은 보고서를 내보내는 일을 반복하게 되죠. 어느 것 하나 어려운 일은 없지만, 모든 것이 영혼을 갉아먹는 고통입니다.

그게 바로 저였습니다. 저는 "파이썬(Python)을 사용하고 있었지만" 여전히 모든 것을 수동으로 처리했습니다. 여기저기 흩어진 스크립트, 노트북 파일들, 그 어떤 것도 연결되어 있지 않았죠. 한계에 다다른 지점은 복잡성이 아니라 반복이었습니다. 일주일에 몇 시간씩 컴퓨터가 훨씬 더 잘할 수 있는 일에 시간을 낭비하고 있다는 사실을 깨달았을 때, 무언가가 변했습니다.

그래서 저는 단발성 스크립트를 작성하는 것을 멈추고 파이프라인(pipeline)을 구축하기로 결심했습니다. 그 결정이 모든 것을 바꾸어 놓았습니다.

### 중요한 전환점: 스크립트 사고방식에서 시스템 사고방식으로

대부분의 파이썬 자동화는 단 하나의 질문에 답하려 하기 때문에 실패합니다.

"이 작업을 어떻게 더 빠르게 수행할 수 있을까?"

더 나은 질문은 이것입니다.

"이 작업을 어떻게 영구적으로 없앨 수 있을까?"

이는 다음을 의미합니다.

*   입력이 예측 가능하다.
*   출력이 유용한 곳으로 전달된다.
*   오류가 새벽 2시에 당신을 깨우지 않는다.
*   스크립트가 당신의 개입 없이 실행된다.

저는 매우 흔한 프리랜서 문제를 택했습니다: 바로 클라이언트 보고서 작성입니다. 원본 데이터가 들어오고, 이를 정리하고, 요약하고, 이메일로 보내는 일. 매주, 영원히 반복되는 작업이죠.

자동화를 위한 완벽한 미끼였습니다.

### 1단계: 지저분한 실제 데이터를 수집하기 (데이터는 항상 지저분하니까요)

클라이언트들은 깨끗한 데이터를 보내지 않습니다. 손상된 헤더, 누락된 값, 무작위 인코딩, 키보드를 마구 친 것처럼 보이는 컬럼명(열 이름)을 가진 CSV 파일을 보냅니다.

파일을 일일이 관리하는 수고를 덜어준 수집 계층은 다음과 같습니다.

```python
import pandas as pd
from pathlib import Path

def load_client_data(folder="data"):
    frames = []
    for file in Path(folder).glob("*.csv"):
        try:
            df = pd.read_csv(file)
            df["source_file"] = file.name
            frames.append(df)
        except Exception as e:
            print(f"Failed to load {file}: {e}")
    return pd.concat(frames, ignore_index=True)
```

사용자 인터페이스(UI)도, 프롬프트도 없습니다. 파일을 폴더에 넣고 그냥 두면 됩니다. 이것만으로도 어떤 기발한 머신러닝(ML) 기술보다 더 많은 시간을 절약할 수 있었습니다.

### 데이터가 손상될 것이라고 예상하고 정리하기 (실제로 그러하니까요)

초기 실수: 데이터가 "대부분 잘 작동할 것"이라고 가정했던 것.
올바른 사고방식: 아무것도 가정하지 마라.

```python
def clean_data(df):
    df.columns = (
        df.columns
        .str.strip()
        .str.lower()
        .str.replace(" ", "_")
    )

    df = df.dropna(how="all")
    df = df.fillna(0)

    numeric_cols = df.select_dtypes(include="object").columns
    for col in numeric_cols:
        try:
            df[col] = pd.to_numeric(df[col])
        except ValueError:
            pass

    return df
```

화려하지는 않지만, 여기서 신뢰성이 탄생합니다. 클라이언트들은 기발함을 위해 돈을 지불하지 않습니다. 그들은 고장 나지 않는 지루한 것을 위해 돈을 지불합니다.

### 원시 데이터를 단순한 숫자가 아닌 실제 통찰력으로 바꾸기

처음에는 정리된 CSV 파일만 보냈습니다. 아무도 신경 쓰지 않았죠.

클라이언트가 정말로 신경 쓰는 것은 '답변'입니다. 그래서 저는 통찰력을 스크립트에 직접 넣어 주었습니다.

```python
def generate_summary(df):
    summary = {
        "rows": len(df),
        "columns": len(df.columns),
        "totals": {},
    }

    for col in df.select_dtypes(include="number").columns:
        summary["totals"][col] = {
            "sum": df[col].sum(),
            "average": df[col].mean(),
            "max": df[col].max(),
        }

    return summary
```

이때부터 파이썬은 조용히 사고하는 작업을 시작합니다. 이 단계에 이르면 클라이언트들은 더 이상 "데이터"를 요구하는 대신 "보고서"를 요청하기 시작합니다.

### 클라이언트가 무시할 수 없는 보고서 자동 생성하기

원시 통계치를 보내는 것은 게으른 일입니다. 그것을 잘 포장하는 것이 바로 영향력(leverage)입니다.

```python
from jinja2 import Template

REPORT_TEMPLATE = """
Weekly Report

Total Rows Processed: {{ rows }}

{% for col, stats in totals.items() %}
{{ col.upper() }}
- Sum: {{ stats.sum }}
- Average: {{ stats.average }}
- Max: {{ stats.max }}

{% endfor %}
"""

def render_report(summary):
    template = Template(REPORT_TEMPLATE)
    return template.render(**summary)
```

이제 파이썬은 단순히 숫자를 처리하는 것을 넘어, 클라이언트에게 제공할 수 있는 결과물(deliverables)을 생산합니다. 이것이 스크립트와 서비스의 차이점입니다.

### 자동화가 가치를 발하는 지점: 전달

당신의 컴퓨터에만 머무는 보고서는 가치가 없습니다. 자동으로 도착하는 보고서는 마법과 같습니다.

```python
import smtplib
from email.mime.text import MIMEText

def send_report(body, to_email):
    msg = MIMEText(body)
    msg["Subject"] = "Your Weekly Automated Report"
    msg["From"] = "reports@yourdomain.com"
    msg["To"] = to_email

    with smtplib.SMTP("smtp.gmail.com", 587) as server:
        server.starttls()
        server.login("your_email", "app_password")
        server.send_message(msg)
```

이 시점에서 전체 파이프라인(pipeline)이 완성됩니다.
파일 → 정리된 데이터 → 통찰력 → 보고서 → 받은 편지함

대시보드도, 로그인도 필요 없습니다. 오직 결과만 있을 뿐입니다.

### 스케줄링: 더 이상 아무것도 "실행"하지 않게 된 순간

수동 실행은 세금과 같습니다. 스케줄링(scheduling)은 자유입니다.

```bash
# crontab -e
0 9 * * 1 /usr/bin/python3 /path/to/report_pipeline.py
```

매주 월요일 아침, 제가 신경 쓰지 않아도 보고서가 도착합니다. 이때 저는 이것이 단순한 생산성 해킹이 아니라, 사용자 인터페이스(UI) 없는 제품이라는 것을 깨달았습니다.

### 예상치 못한 수익 부분 (늘 이런 식으로 흘러가니까요)

저는 이 시스템을 저를 위해 만들었습니다. 그러다 한 클라이언트가 "자기에게도 해줄 수 있느냐"고 물었고, 또 다른 클라이언트도 마찬가지였습니다. 그러더니 누군가는 여러 데이터 세트에 대해 매주 실행될 수 있는지 문의했습니다.

그때부터 가격 책정 방식이 시간당에서 월간으로 바뀌었습니다.

이 동일한 파이썬 시스템은 다음과 같이 판매될 수 있습니다.

*   보고서 자동화 서비스
*   틈새 서비스형 소프트웨어(SaaS)를 위한 백엔드(backend)
*   에이전시를 위한 화이트 라벨(white-labeled) 도구
*   영원히 조용히 실행되는 프리랜서 업셀링(upselling) 도구

코드는 거의 변하지 않았습니다. 관점(framing)이 달라졌을 뿐입니다.
