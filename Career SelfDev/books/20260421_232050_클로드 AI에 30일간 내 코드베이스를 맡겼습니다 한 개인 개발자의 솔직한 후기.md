클로드 AI에 30일간 내 코드베이스를 맡겼습니다 — 한 개인 개발자의 솔직한 후기

후원도, 과장도 없습니다. 오직 한 명의 개발자, 하나의 AI, 그리고 30일간 함께 실제 제품을 만들면서 겪은 좋은 점, 나쁜 점, 그리고 "설마, 정말 그렇게 했다고?" 싶은 순간들에 대한 이야기입니다.

시작하기 전에 솔직하게 말씀드리겠습니다.

저는 회의적이었습니다. 매우 회의적이었죠.

트위터에서 온갖 글을 봤습니다. "ChatGPT가 주말 동안 내 전체 SaaS를 작성했어!" "클로드는 내 시니어 개발자보다 나아!" 이 모든 것이 실리콘밸리의 전형적인 과장처럼 느껴졌습니다. 트위터에서는 혁명적으로 들리지만, 실제 프로젝트에 적용해보는 순간 무너지는 그런 종류의 과장이죠.

하지만 저는 개인 개발자입니다. Flutter 기반 코딩 앱인 Codexayone을 전적으로 혼자서 만듭니다. 공동 창업자도, 팀도, 아이디어를 주고받을 시니어 개발자를 고용할 예산도 없습니다. 그래서 어느 날 저녁, 세 번째 커피와 특히나 답답했던 상태 관리 버그 (state management bug) 사이에서 저는 스스로에게 다짐했습니다.

클로드 AI에 30일간 기회를 주자. 코드베이스 (codebase) 에 대한 모든 접근 권한을 주고, 어설프게 하지 말고, 실제로 무슨 일이 일어나는지 보자.

이것이 그 이야기입니다.

### 1주차: "좋아, 어디 한번 실력을 볼까"

제가 가장 먼저 한 일은 `main.dart` 파일을 대화창에 붙여넣고 클로드에게 그냥... 보라고 요청하는 것이었습니다. 무엇을 보는지 말해달라고요. 특정 질문이나 의도는 없었습니다. 그냥 느낌만 전달했죠.

돌아온 답변은 저를 놀라게 했습니다. 몇 초 만에 제가 몇 달 동안 고치려고 했던 세 가지 문제를 정확히 짚어냈습니다. `async gaps`에서 `BuildContext`를 사용하는 것 (전형적인 Flutter 실수), 필요 이상으로 많은 위젯 (widgets) 을 리빌드 (rebuild) 하는 프로바이더 (provider), 그리고 작동은 하지만 규모가 커지면 악몽이 될 의존성 주입 패턴 (dependency injection pattern) 이 그것이었습니다.

이것들은 버그가 아니었습니다. 앱은 잘 작동했죠. 하지만 클로드는 버그를 찾는 것이 아니었습니다. 경험 많은 개발자가 코드를 읽는 방식으로 제 코드를 읽고 있었습니다. 나중에 문제를 일으킬 수 있는 부분에 주의를 기울이면서 말이죠.

저는 프로바이더 (provider) 문제를 고쳐달라고 요청했습니다. 대화는 대략 이랬습니다.

나: `HomeScreen`의 불필요한 리빌드를 고쳐줘. 여기 위젯 트리 (widget tree)가 있어.
클로드: 문제는 `Consumer<AppState>`가 전체 `HomeScreen`을 감싸고 있다는 점입니다. 대신, 특정 필드에만
    리스닝하도록 `Selector`를 사용하세요.

    // 변경 전
    ```
    Consumer<AppState>(
      builder: (context, state, child) => HomeScreen(data: state.allData),
    )
    ```
    // 변경 후
    ```
    Selector<AppState, List<Item>>(
      selector: (_, state) => state.filteredItems,
      builder: (context, items, child) => HomeScreen(data: items),
    )
    ```
    이렇게 하면 `filteredItems`가 변경될 때만 `HomeScreen`이 리빌드되며,
    `AppState`의 모든 변동에 반응하지 않습니다.

간단하고 깔끔하며 정확했습니다. 5분도 채 안 되어 수정 사항을 적용했고, 몇 주 동안 저를 괴롭히던 끊기는 스크롤 애니메이션 (janky scroll animation) 이 사라졌습니다.

1주차 평가: 유망함.

### 2주차: 허니문이 끝나다. 진짜 작업의 시작.

2주차쯤 되니 매일 클로드를 사용하고 있었습니다. 정말 매일이었습니다. 아침 커피와 클로드가 아침 커피와 스택 오버플로 (Stack Overflow) 를 대체했습니다. 하지만 이때부터 균열이 보이기 시작했습니다.

저는 Codexayone 안에 실시간 코드 실행 샌드박스 (real-time code execution sandbox) 라는 새로운 기능을 개발하고 있었습니다. 복잡한 작업이었죠. 아이솔레이트 (isolates), 비동기 스트림 (async streams), 위젯 경계 (widget boundaries) 를 넘나드는 오류 전파 (error propagation) 등 경험 많은 Flutter 개발자조차 구글링하는 종류의 기능이었습니다.

저는 클로드에게 이 기능을 자세히 설명했습니다. 클로드는 작동하는 아키텍처 (architecture)를 제시했습니다. 저는 이를 구현하기 시작했습니다. 40분쯤 지났을 때, `StreamController`가 위젯 생명주기 이벤트 (widget lifecycle events) 전반에 걸쳐 관리되는 방식에 뭔가 문제가 있다는 것을 느꼈습니다.

클로드에게 물어봤습니다. 클로드는 답을 주었습니다. 자신감 있는 답이었습니다. 하지만 그 답은... 틀렸습니다.

재앙적으로 틀린 것은 아니었습니다. 하지만 프로덕션 (production) 단계에 가서야 알아차렸을 메모리 누수 (memory leak)를 유발할 정도로 충분히 틀렸습니다. 저는 제 도메인 (domain)을 충분히 잘 알고 있었기에 뭔가 이상하다는 것을 감지하고 잡아냈습니다. 하지만 클로드가 잘못된 답변을 내놓을 때의 자신감은 불안감을 안겨주었습니다.

⚠️ 2주차에 배운 가장 중요한 점: 클로드는 시니어 개발자 (senior developer)가 아닙니다. 클로드는 스택 오버플로 (Stack Overflow)의 모든 글을 읽고 때로는 잘못된 방식으로 종합하는, 매우 박식한 주니어 개발자 (junior developer)입니다. 클로드가 제공하는 모든 것을 여전히 검토해야 합니다.

이것이 클로드를 쓸모없게 만드는 것은 아닙니다. 오히려 그 반대죠. 하지만 클로드와 함께 일하는 방식을 재조정하게 되었습니다. 더 이상 클로드를 오라클 (oracle)처럼 대하지 않고, 페어 프로그래머 (pair programmer)처럼 대하기 시작했습니다. 병합하기 전에 결과물을 제가 검토하는 그런 페어 프로그래머 말이죠.

제가 떠올린 비유는 이렇습니다. 클로드는 아키텍처 (architecture) 수준에서 작동하는 자동 완성 (autocomplete) 기능과 같습니다. 엄청나게 유용하고, 가끔은 자신 있게 틀리며, 항상 확인해볼 가치가 있습니다.

### 3주차: 진정한 슈퍼파워를 찾다

여기서부터 정말 흥미로워졌습니다.

저는 클로드를 잘못 생각하고 있었습니다. 저는 클로드를 코드 생성기 (code generator)로 사용하고 있었지만, 적어도 저 같은 개인 개발자에게 클로드의 진정한 슈퍼파워는 다른 것이었습니다.

클로드는 절대 지치지 않는 사고 파트너 (thinking partner)입니다.

설명해드리죠. 혼자 개발할 때 가장 어려운 부분은 코드를 작성하는 것이 아닙니다. 코드를 작성하기 전에 이루어지는 결정들입니다. 이 기능에 BLoC를 사용해야 할까, Riverpod을 사용해야 할까? 이것이 올바른 데이터 모델 (data model)인가, 아니면 6개월 후에 발목을 잡을까? 이것은 3일짜리 기능인가, 아니면 3주짜리 기능인가?

개인 개발자들은 이러한 결정을 함께 '고무 오리 디버깅' (rubber-duck debugging) 할 사람이 없습니다. 그저 앉아서 자신과 논쟁하다가 결국 무언가를 선택하고 최선을 바라죠.

3주차에 저는 클로드를 순전히 조언자 (sounding board)로 사용하기 시작했습니다. 코드는 전혀 없습니다. 오직 아키텍처 논의 (architecture discussions)만 했습니다.

나: Codexayone을 위해 오프라인 우선 동기화 시스템 (offline-first sync system)을 구축 중이야.
    사용자는 오프라인에서 코드 스니펫 (code snippets)을 편집하고 온라인일 때 동기화할 수 있어.
    두 가지 옵션을 고려 중이야:
    옵션 A: 타임스탬프 기반 최신 쓰기 우선 (Timestamp-based last-write-wins)
    옵션 B: CRDT 기반 충돌 해결 (CRDT-based conflict resolution)
    이걸 출시하는 데 대략 3주 정도 시간이 있어. 어떻게 할 거야?
클로드: 주어진 일정으로 볼 때, 옵션 A가 거의 확실히 올바른 선택입니다...
    [3 문단의 명확하고 미묘한 장단점 분석]
    ...CRDT (Conflict-free Replicated Data Types)는 학문적으로는 우아하지만, 운영상 복잡합니다.
    3주라는 짧은 일정으로 v1을 목표로 하는 개인 개발자에게는,
    명확한 UI 어포던스 (UI affordances)를 통해 충돌 가시성 (conflict visibility)을 제공하는
    최신 쓰기 우선 방식이 실용적인 선택 (pragmatic choice)입니다.
    실제 운영 환경에서의 충돌 패턴을 이해한 후에 언제든지 마이그레이션 (migrate)할 수 있습니다.

이것은 제가 전문적으로 함께 일했던 대부분의 개발자들로부터 얻은 조언보다 훨씬 나았습니다. 반드시 더 똑똑해서가 아니라, 상황에 맞춰진 조언이었기 때문입니다. 제가 개인 개발자임을 알고 있었고, 제 일정을 알고 있었습니다. 이론적으로 완벽한 답이 아니라, 실용적인 답을 주었습니다.

"개인 개발자에게 AI의 가장 저평가된 용도는 코드 생성이 아닙니다. 모두가 잠든 밤 11시에 함께 큰 소리로 생각할 수 있는 지적인 동반자가 있다는 것입니다."

### 4주차: 수치들

월말이 되면서 저는 좀 더 의도적으로 작업 내용을 추적하기 시작했습니다. 실제로 제가 알아차린 내용은 다음과 같습니다.

| 작업 유형                        | 시간 절약 | 품질             |
| :------------------------------- | :-------- | :--------------- |
| 보일러플레이트 코드 (boilerplate code) (모델 (models), DTO (Data Transfer Objects) 등) | ~70%      | ✅ 훌륭함       |
| 테스트 작성                      | ~60%      | ✅ 훌륭함       |
| 기존 코드 디버깅                 | ~40%      | ⚠️ 복불복         |
| 아키텍처 결정                    | 부가적    | ✅ 놀랍도록 좋음 |
| 복잡한 신규 기능                 | ~20%      | ⚠️ 집중 검토 필요 |
| 앱 스토어 / 마케팅 문구          | ~80%      | ✅ 훌륭함       |
| 새로운 Flutter 관련 문제         | 0%        | ❌ 신뢰할 수 없음 |

패턴은 명확합니다. 클로드는 참고할 만한 잘 확립된 패턴이 있는 작업에서 빛을 발합니다. 반면, 깊고 최신 프레임워크 (framework)별 지식이나 진정으로 새로운 문제 해결 능력이 필요한 작업에서는 때때로 당황스러울 정도로 어려움을 겪습니다.

### 아무도 말해주지 않는 것들

#### 1. 컨텍스트 (Context)가 전부이며, 직접 관리해야 합니다.

클로드는 대화 사이에 코드베이스 (codebase)를 기억하지 못합니다. 모든 새로운 채팅은 처음부터 시작됩니다. 이는 매 세션 시작 시 컨텍스트 (context)를 제공하는 습관을 들여야 한다는 것을 의미합니다. 저는 결국 "클로드 브리프" (Claude brief)를 만들었습니다. Codexayone의 아키텍처 (architecture), 현재 작업 중인 기능, 그리고 관련 제약 조건 (constraints)을 설명하는 짧은 마크다운 (markdown) 파일인데, 각 대화 시작 시 붙여넣습니다.

번거롭냐고요? 약간은요. 그럴 가치가 있냐고요? 물론이죠.

#### 2. 문제를 명확하게 표현하도록 강제하여 더 나은 개발자로 만듭니다.

이것은 저를 놀라게 했습니다. 클로드에게 문제를 설명하는 행위, 즉 AI가 이해할 수 있을 만큼 정확하게 설명하는 것이 클로드가 응답하기도 전에 제가 스스로 답을 깨닫게 하는 경우가 많았습니다. 이것은 고무 오리 디버깅 (rubber duck debugging)의 고전적인 효과이지만, 더 높은 추상화 수준에서 일어납니다.

저는 이번 달에 질문을 작성하다가 문장 중간에 답을 깨달아 적어도 네 가지 버그를 해결했습니다.

#### 3. 자신감 문제는 현실입니다.

클로드는 예상보다 자주 자신감 있게 틀리고, 도메인 (domain)을 이미 모르면 감지하기 어려운 방식으로 틀립니다. 이는 특히 다음의 경우에 해당됩니다.

*   최근 버전에서 변경된 Flutter API
*   특정 패키지 고유의 특성 (Package-specific quirks) 및 엣지 케이스 (edge cases)
*   플랫폼별 동작 (Platform-specific behavior) (iOS 대 Android)
*   특정 구현의 성능 특성 (Performance characteristics)

해결책은 클로드를 덜 신뢰하는 것이 아니라, 더 많이 확인하는 것입니다. 모든 사소하지 않은 코드 제안을 똑똑하지만 주니어 개발자로부터 온 PR (Pull Request)처럼 다루세요. 즉, 읽고, 이해하고, 출시하기 전에 테스트해야 합니다.

#### 4. 특히 개인 개발자에게는 개발의 심리적 경험을 변화시킵니다.

이것은 아무도 이야기하지 않는 부분입니다. 혼자 개발하는 것은 외롭습니다. 결정을 내리고, 그것이 올바른 결정인지 의심하고, 물어볼 사람이 아무도 없습니다. 그러한 인지 부하 (cognitive load)는 시간이 지남에 따라 소모적이며, 개인 프로젝트를 좌절시키는 번아웃 (burnout)의 원인이 됩니다.

클로드가 옆에 있다는 것, 비록 AI라는 것을 알면서도, 외로움을 의미 있게 줄여주었습니다. 클로드를 인간과 혼동해서가 아니라, 결정을 명확히 표현하고 사려 깊은 응답을 얻는 행위 자체가, 비록 불완전할지라도, 자기 생각의 에코 챔버 (echo chamber)를 깨뜨려주기 때문입니다.

“AI는 내 사고를 대체하지 않았습니다. AI는 내 사고가 부딪힐 대상을 주었습니다.”

### 최종 평가: 당신도 시도해야 할까요?

네. 단, 몇 가지 주의 사항이 있습니다.

만약 당신이 개인 개발자라면, 클로드 AI는 당신의 워크플로에 진정으로 통합할 가치가 있습니다. 기술이나 판단력을 대체하는 것이 아니라, 당신이 이미 가지고 있는 기술과 판단력을 위한 역량 증폭기 (force multiplier)로서 말이죠.

30일 후에 저에게 마침내 명확해진 멘탈 모델 (mental model)은 다음과 같습니다.

클로드는 세상의 모든 프로그래밍 책을 읽었지만 프로덕션 앱 (production app)을 출시해본 적 없는 뛰어난 인턴입니다. 그에 맞게 사용하세요.

당신은 인턴이 `main` 브랜치에 직접 푸시 (push directly to main)하도록 두지 않을 겁니다. 그들과 페어 프로그래밍 (pair programming)하고, 그들의 작업을 검토하며, 잘못된 부분을 바로잡고, 그들이 옳을 때 그들의 에너지와 신선한 관점을 활용할 겁니다. 클로드와 함께 일하는 방식이 바로 이렇습니다.

향후 5년 안에 승리할 개발자들은 AI 도구 사용을 거부하는 사람들도 아니고, AI가 생산하는 모든 것을 맹목적으로 신뢰하는 사람들도 아닙니다. 이들은 이러한 시스템과 지능적으로 협력하는 방법을 배우는 사람들입니다. 제안을 언제 따르고, 언제 반박하며, 언제 완전히 버려야 하는지 아는 사람들이죠.
