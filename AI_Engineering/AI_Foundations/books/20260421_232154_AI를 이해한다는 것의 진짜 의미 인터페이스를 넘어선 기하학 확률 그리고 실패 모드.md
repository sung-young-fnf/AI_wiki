# AI를 '이해한다'는 것의 진짜 의미: 인터페이스를 넘어선 기하학, 확률, 그리고 실패 모드

모두가 AI를 배우고 있다고 말하지만, 대다수는 그저 구글 검색을 더 잘하는 방법을 배우는 것에 불과합니다.

만약 여러분의 'AI 교육'이 프롬프트 템플릿을 암기하거나 클로드(Claude)에게 코드를 작성하도록 만드는 완벽한 문구를 찾는 것에 그친다면, 여러분은 AI를 배우는 것이 아닙니다. 그저 사용자 인터페이스(UI)를 배우고 있는 것입니다. 자판기를 조작하는 방법을 배우는 것이지, 기계 내부에서 결제를 처리하고 제품을 배출하는 메커니즘을 배우는 것이 아닙니다.

프롬프트 작성은 AI 리터러시(AI literacy)가 아닙니다. 허깅페이스(HuggingFace)에서 찾은 JSONL 파일로 모델을 미세 조정하는 것 또한 마찬가지입니다.

진정한 AI 리터러시는 소프트웨어를 바라보는 방식의 불편한 전환을 요구합니다. 컴퓨터가 여전히 수학적 연산을 수행하지만, 그 수학이 기하학(geometry)을 정의하고 있다는 것을 깨달아야 합니다.

## 'UI 기술' 문제

현재 업계는 인텔리전스 자체가 아닌 인터페이스만 숙달한 'AI 전문가'들로 넘쳐납니다. 이러한 현상은 곳곳에서 발견됩니다:

*   '4시간 만에 ChatGPT 마스터하기'와 같은 강좌들.
*   소스 코드를 읽지 않고 랭체인(LangChain)의 상용구(boilerplate) 코드를 복사 붙여넣기하는 행위.
*   API를 호출하여 결과를 얻고, 평가 도구(eval suite) 없이 바로 제품에 배포하는 경우.

이러한 것들은 피상적이고 취약한 기술들입니다. 모델 아키텍처가 변경되거나 (반드시 변경될 것입니다) 추상화 계층이 달라지면, '프롬프트 엔지니어'들은 쓸모없는 지식만을 가진 채 남겨지게 될 것입니다.

## 거의 아무도 이해하지 못하는 한 가지

여기 도구 사용자(tool user)와 엔지니어(engineer)를 구분 짓는 핵심 진실이 있습니다:

AI 모델은 사실(fact)을 '알고' 있지 않습니다. 대신, 고차원 표현 공간(high-dimensional representation space)에서 확률을 탐색합니다.

대규모 언어 모델(LLM)에 '애플의 CEO는 누구인가요?'라고 물었을 때, 모델은 데이터베이스(database)를 검색하거나 지식 그래프(knowledge graph)를 조회하지 않습니다. 인간이 가진 방식의 '기억'도 없습니다.

이는 궤적(trajectory)을 계산하는 것입니다.

## 지식은 실제로 어디에 있는가 (매개변수 안에 있지 않다)

지식은 매개변수(parameters) 자체에 개별적인 사실로 저장되는 것이 아니라, 통계적 구조(statistical structure)로 인코딩됩니다.

이를 이해하려면 벡터 공간(Vector Space)에 대해 이야기해야 합니다.

전통적인 소프트웨어에서 데이터는 이산적(discrete)입니다. 문자열은 문자열이고, A는 A입니다. 하지만 AI에서 데이터는 연속적(continuous)입니다. 모든 것, 즉 단어, 이미지, 개념은 벡터(vector) 또는 임베딩(embedding)이라는 숫자 목록으로 변환됩니다.

지도를 상상해 보세요. 하지만 이 지도에는 동서남북 대신 1,536개(또는 모델에 따라 더 많을 수 있음)의 차원(dimensions)이 있습니다.

이 초복잡한 지도에서는 '의미'가 위치로 정의됩니다.

*   '개'라는 개념은 특정 좌표에 위치합니다.
*   '강아지'라는 개념은 '개'와 매우 가깝습니다.
*   '기화기(Carburettor)'라는 개념은 매우 멀리 떨어져 있습니다.

지식은 곧 기하학입니다.

모델이 올바른 답을 제시할 때, 그것은 답을 '기억한' 것이 아닙니다. 데이터 분포의 곡선 형태인 매니폴드(manifold)를 성공적으로 탐색하여 답이 확률적으로 속하는 좌표에 도달한 것입니다.

이것이 바로 '환각(Hallucination)'이라는 용어가 부적절한 이유입니다. 모델은 거짓말을 하거나 혼란스러워하는 것이 아닙니다. 프롬프트의 궤적(trajectory)을 기반으로 수학적으로 옳다고 느껴지는 좌표로 이동했을 뿐인데, 그 좌표가 현실 세계에서는 사실상 비어있는 경우입니다.

진실의 도심(city centre)이 아니라 교외(suburbs)에 착륙한 셈이죠.

## 이것이 모든 것을 바꾸는 이유

AI가 기하학적 공간에서 작동하는 확률 엔진(probability engine)이라는 것을 이해하면, 일반적인 오류들이 비로소 설명됩니다.

### '그냥 데이터를 더 많이 추가하면 된다'는 생각이 실패하는 이유:

추론 능력을 개선하기 위해 단순히 더 많은 텍스트를 컨텍스트 창(context window)에 밀어 넣을 수는 없습니다. 모델의 내부 지도(즉, 사전 학습된 가중치(pre-trained weights))가 A가 B에 가깝다고 암시한다면, 아무리 많은 컨텍스트도 추론 과정에서 그 기하학을 근본적으로 재구성하지 못합니다. 이는 모델 학습의 중력(gravity)과 싸우는 것과 같습니다.

### 모델이 확신에 차 있지만 틀리는 이유:

LLM에서 '확신'이란 단순히 확률 곡선(probability curve)이 얼마나 가파른지를 나타내는 척도입니다. 모델은 수학적으로 99.9% 올바른 경로에 있다고 확신할 수 있지만, 사실적으로는 100% 틀릴 수 있습니다. 이는 GPS 신호가 튕겨 잘못된 방향으로 시속 160km(100mph)로 달리는 것과 같습니다.

## 차이점: 도구 사용자 대 AI 사상가

향후 2년간의 엔지니어링 환경에서 살아남으려면, 사용자(user)에서 사상가(thinker)로 전환해야 합니다.

## 기술 구현: '의미' 측정하기

실질적인 예를 들어보겠습니다. '도구 사용자'는 AI 답변이 올바른지 읽어보고 확인하는 반면, 'AI 사상가(AI Thinker)'는 의미론적 거리(semantic distance)를 측정합니다.

검색 증강 생성(RAG) 시스템을 구축하고 있다면, 모델이 실제로 제공된 컨텍스트를 사용하는지 아니면 단순히 지어내는지 알아야 합니다.

여기에 벡터 유사도(vector similarity) 개념을 사용하여 답변을 검증하는 예시가 있습니다. 우리는 단순히 키워드를 확인하는 것이 아니라, 진실의 원천(source of truth)과 생성된 답변 사이의 기하학적 각도(geometric angle)를 측정하는 것입니다.

참고: 이 예시는 넘파이(numpy)와 OpenAI의 임베딩(embedding) 형식을 사용하며, 이는 일반적인 워크플로우에서 표준적으로 활용됩니다.

```python
# 간단한 코드 예시
import numpy as np
from typing import List, Dict

# 임베딩 모델(예: text-embedding-3-large)에서 벡터를 가져오는 시뮬레이션
def get_embedding(text: str) -> np.array:
    # 실제 프로덕션 환경에서는 API 또는 로컬 ONNX 모델을 호출합니다.
    # 시연을 위해 무작위 벡터를 반환합니다.
    return np.random.rand(1536)

def cosine_similarity(v1: np.array, v2: np.array) -> float:
    """
    두 벡터 사이의 각도 코사인 값을 계산합니다.
    1.0 = 같은 방향 (의미 동일)
    0.0 = 직교 (무관)
    -1.0 = 반대 방향
    """
    dot_product = np.dot(v1, v2)
    norm_v1 = np.linalg.norm(v1)
    norm_v2 = np.linalg.norm(v2)
    return dot_product / (norm_v1 * norm_v2)

def evaluate_hallucination_risk(
    context_text: str,
    generated_answer: str,
    threshold: float = 0.75
) -> Dict:
    """
    답변이 컨텍스트에 의미론적으로 기반하고 있는지 판단합니다.
    """
    context_vector = get_embedding(context_text)
    answer_vector = get_embedding(generated_answer)

    score = cosine_similarity(context_vector, answer_vector)

    # 각도가 너무 넓으면 모델이 컨텍스트에서 벗어난 것입니다.
    is_risky = score < threshold

    return {
        "similarity_score": round(score, 4),
        "risk_detected": is_risky,
        "reasoning": "답변 벡터가 컨텍스트 벡터 공간에서 너무 멀리 벗어났습니다." if is_risky else "답변이 컨텍스트에 기반하고 있습니다."
    }

# 사용 예시
context = "The API rate limit is 500 requests per minute."
response = "You can make 500 calls every 60 seconds." # 의미론적으로 동일
# 단어는 다르지만, 그들의 벡터는 같은 방향을 가리킵니다.
eval_result = evaluate_hallucination_risk(context, response)
print(eval_result)
```

이것이 바로 AI를 '배우는' 방법입니다. '시니어 개발자처럼 행동해 주세요'라고 입력하는 방법을 배우는 것이 아니라, 출력 벡터가 여러분의 진실 벡터(ground truth vector)와 수학적으로 일치하는지 검증하는 방법을 배우는 것입니다.

## 'AI 학습'은 실제로 어떤 모습이어야 하는가

2026년에 앞서나가고 싶다면, 로드맵(roadmap) 영상을 보는 것을 멈추고 다음을 시작하십시오.

*   **실패 모드(Failure Modes)를 연구하십시오**: 모델이 올바르게 수행하는 것만 보지 마십시오. 의도적으로 모델을 망가뜨리려 노력하고, '매니폴드의 가장자리(edge of the manifold)' 즉, 모델이 반복이나 무의미한 결과를 내도록 만드는 입력값을 찾아내십시오.
*   **평가(Evals)를 배우십시오**: 이것은 지루하고 매력적이지 않지만 가장 큰 성과를 가져다주는 작업입니다. 1,000개의 모델 출력을 자동으로 채점하는 스크립트를 작성할 수 있습니까? 그렇지 못하다면, AI 제품을 확장(scale)할 수 없습니다.
*   **시스템적 사고(Systems Thinking)를 하십시오**: 모델은 단지 하나의 구성 요소일 뿐입니다. 진정한 마법은 라우터(router), 메모리 계층(memory layer), 그리고 검증기(verifier)에 있습니다.

## 불편한 진실

다음 세대의 AI 버그는 구문 오류(syntax error)가 아닐 것입니다. 여러분의 서버를 다운시키지도 않을 것입니다.

그것들은 미묘하고 조용한 실패일 것입니다. 예를 들어, '피고인(defendant)'과 '원고인(plaintiff)'의 벡터 공간이 엄청나게 가깝기 때문에 법률 요약기가 단어를 바꿔 버리거나, 통계적으로는 확률이 높지만 생물학적으로는 치명적인 복용량을 추천하는 의료 봇과 같은 경우입니다.

대부분의 사람들이 뒤쳐지는 이유는 늦게 시작했기 때문이 아닙니다. 잘못된 추상화(abstraction)를 배우고 있기 때문입니다.

AI 리터러시는 입력 상자에 무엇을 타이핑해야 하는지를 아는 것이 아닙니다. 출력되는 내용을 왜 정확히 신뢰해서는 안 되는지를 아는 것입니다.
