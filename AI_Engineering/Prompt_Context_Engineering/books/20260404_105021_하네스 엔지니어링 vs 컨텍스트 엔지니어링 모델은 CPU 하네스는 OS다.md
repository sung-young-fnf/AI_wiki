# 하네스 엔지니어링 vs 컨텍스트 엔지니어링: 모델은 CPU, 하네스는 OS다

운영 환경(Production)에서 AI 에이전트가 여전히 20~40%의 실패율을 보이고 있다면, 접근 방식을 재고해야 할 때입니다. 단순히 데모를 만드는 수준을 넘어 실제 업무를 수행할 수 있는 신뢰성 높은 시스템을 구축하려면 **컨텍스트 엔지니어링(Context Engineering)**과 **하네스 엔지니어링(Harness Engineering)**의 결정적인 차이를 이해해야 합니다.

핵심 요약하자면, 컨텍스트 엔지니어링은 AI 모델이 추론 시점에 무엇을 보는지를 최적화하는 것이고, 하네스 엔지니어링은 여러 번의 추론 과정 전반에서 신뢰성을 보장하는 전체 시스템 설계를 의미합니다. 컨텍스트 엔지니어링은 하네스 엔지니어링의 하위 집합입니다.

### 1. 컨텍스트 엔지니어링(Context Engineering)이란?

컨텍스트 엔지니어링은 LLM이 추론 시점에 접하는 모든 정보의 설계 및 관리를 의미합니다. 모델의 컨텍스트 윈도우(Context Window)에 입력되는 모든 토큰—시스템 프롬프트, 도구 정의, RAG(검색 증강 생성) 결과, 메시지 이력, 출력 스키마, 이전 세션의 메모리 등—이 이에 해당합니다.

안드레이 카파시(Andrej Karpathy)는 이 용어를 '프롬프트 엔지니어링'의 후계자로 대중화했습니다. 프롬프트 엔지니어링이 단순히 영리한 지침을 만드는 것이라면, 컨텍스트 엔지니어링은 에이전트가 과제를 해결할 수 있도록 **전체 정보 환경을 설계**하는 예술입니다.

**컨텍스트 엔지니어링의 한계:**
아무리 완벽한 컨텍스트 엔지니어링이라도 단일 추론만 최적화할 뿐입니다. 모델이 실수했을 때 어떻게 대처할지, 혹은 다음 주에 동일한 실패가 반복되는 것을 어떻게 방지할지에 대해서는 답을 주지 못합니다. 컨텍스트 엔지니어링에는 자체적인 실패에 대한 '기억'이 없기 때문입니다.

### 2. 하네스 엔지니어링(Harness Engineering)이란?

하네스 엔지니어링은 에이전트 환경 전체에 대한 시스템 설계입니다. 모델 외부에서 작동하는 모든 것, 즉 행동 제약(Behavioral Constraints), 피드백 루프(Feedback Loops), 품질 게이트(Quality Gates), 그리고 수천 번의 추론 과정에서 신뢰성을 보장하는 개선 사이클을 포함합니다.

마틴 파울러(Martin Fowler)의 사이트에서 비르기타 뵈켈러(Birgitta Boeckeler)가 정의한 하네스 엔지니어링의 세 가지 구성 요소는 다음과 같습니다.

1.  **컨텍스트 엔지니어링:** 코드베이스 내의 지속적으로 향상되는 지식 베이스와 관측 데이터로부터 얻은 동적 컨텍스트.
2.  **아키텍처 제약(Architectural Constraints):** LLM 에이전트뿐만 아니라 결정론적(Deterministic) 커스텀 린터(Linter) 및 구조적 테스트에 의해 모니터링되는 제약 사항.
3.  **가비지 컬렉션(Garbage Collection):** 문서의 불일치나 아키텍처 제약 위반을 찾아내기 위해 주기적으로 실행되는 에이전트.

하네스 엔지니어링의 핵심 철학은 **"에이전트가 실수를 할 때마다, 다시는 같은 실수를 반복하지 않도록 솔루션을 엔지니어링하는 것"**입니다.

### 3. 주요 차이점 비교

| 구분 | 컨텍스트 엔지니어링 (Context Engineering) | 하네스 엔지니어링 (Harness Engineering) |
| :--- | :--- | :--- |
| **핵심 질문** | 에이전트에게 무엇을 보여줄 것인가? | 무엇을 방지하고, 측정하고, 제어하고, 고칠 것인가? |
| **초점** | 개별 추론의 최적화 | 시스템의 신뢰성과 지속적인 개선 |
| **성격** | 정보 환경 설계 | 운영체제(OS) 수준의 시스템 설계 |
| **비유** | 신입 사원을 위한 완벽한 온보딩 문서 | 린터, CI/CD, 코드 리뷰가 갖춰진 업무 환경 |
| **목표** | 더 나은 개별 출력값 생성 | 자율적으로 업무를 수행할 수 있는 신뢰 구축 |

### 4. 하네스가 모델의 성능을 압도한다는 증거

모델의 성능보다 하네스 설계가 더 중요하다는 사실은 여러 연구를 통해 입증되었습니다.

*   **SWE-agent 연구 (프린스턴 대학):** 동일한 모델(GPT-4 Turbo)을 사용했을 때, 커스텀 에이전트-컴퓨터 인터페이스(ACI)를 갖춘 하네스 설정이 기본 설정보다 문제 해결률을 **64% 향상**시켰습니다.
*   **SWE-Bench Mobile 평가:** 동일한 모델(Claude Opus 4.5)이 하네스 설계의 차이에 따라 **성공률이 2%에서 12%로 6배 차이**가 났습니다. 성능 차이는 전적으로 도구 사용 관리, 편집 인터페이스 구조화, 실패 복구 방식에서 발생했습니다.
*   **Stripe 사례:** 주당 1,300개의 AI 작성 풀 리퀘스트(PR)를 생성하는 시스템은 샌드박스 실행 환경, 병렬 에이전트 실행, 인간의 리뷰 게이트 등 철저한 하네스 엔지니어링을 통해 운영됩니다.

### 5. 실전 구현 가이드

#### 1단계: 컨텍스트 엔지니어링 기초
*   **지침 파일(Instruction Files):** `CLAUDE.md`, `AGENTS.md` 등을 통해 프로젝트 고유의 규칙과 코딩 스타일을 모델에 직접 주입합니다.
*   **RAG 및 검색 파이프라인:** 코드베이스나 문서에서 관련 컨텍스트를 검색하여 제공합니다.
*   **메모리 시스템:** 세션 간에 이전 결정과 사용자 선호도를 기억하도록 합니다.

#### 2단계: 하네스 엔지니어링 구축
*   **반복적 오류 로그:** 에이전트가 실패할 때마다 `AGENTS.md`에 새로운 규칙을 추가하여 동일한 실수를 방지합니다.
*   **생명주기 훅(Lifecycle Hooks):** 위험한 명령어가 실행되기 전에 이를 차단하거나 검증하는 결정론적 제어 장치를 마련합니다.
*   **린터 및 구조적 테스트:** 에이전트의 모든 행동 직후에 테스트를 실행하여 아키텍처 제약을 강제합니다.
*   **가비지 컬렉션 에이전트:** 주기적으로 오래된 문서나 일치하지 않는 코드를 정리하여 시스템의 엔트로피(Entropy)를 관리합니다.

#### 3단계: 피드백 루프 (Feedback Loop)
에이전트가 실패하면 사람이 분석하고, 그 결과에 따라 지침을 업데이트하거나(컨텍스트 개선), 새로운 린터를 추가하거나(아키텍처 제약), 가비지 컬렉션을 강화하는 선순환 구조를 만듭니다.

### 결론

컨텍스트 엔지니어링은 에이전트에게 무엇을 할지 알려주지만, 하네스 엔지니어링은 에이전트가 실제로 올바르게 수행하는지 보장하고 프로젝트가 진화하더라도 궤도를 유지하게 해줍니다.

만약 에이전트가 100번 실행했을 때 예측 불가능한 결과를 내놓는다면, 그것은 모델의 문제가 아니라 **하네스의 문제**일 가능성이 큽니다. 신뢰할 수 있는 AI 시스템을 구축하고 싶다면, 모델(CPU)에만 집중하지 말고 에이전트의 운영체제인 하네스(OS)를 구축하는 데 투자하십시오.
