# AI 시대, JSON은 저물고 있는가 (JSON Is Dying in the AI Era)

우리는 매일 API, 마이크로서비스, 설정 파일 등에서 JSON을 사용합니다. JSON은 개발자들에게 기본 중의 기본입니다. 하지만 대규모 언어 모델(LLM)과 본격적으로 작업하기 시작하면, JSON은 다소 어색하고 비효율적으로 느껴지기 시작합니다.

JSON은 분명 작동하지만, 토큰(Token) 사용량 측면에서 볼 때 부피가 크고, 느리며, 비용이 많이 듭니다. AI 세계에서 토큰은 곧 돈입니다. 여기서 **TOON(Token Oriented Object Notation)**이라는 대안이 등장합니다. 모든 곳에서 JSON을 대체하려는 것이 아니라, LLM과 통신할 때 활용할 수 있는 매우 실질적인 대안입니다.

### JSON과 LLM의 불편한 진실

LLM에 데이터를 보낼 때, 모델은 '객체'나 '구조'를 받는 것이 아니라 '토큰'을 받습니다. 중괄호, 따옴표, 쉼표, 공백, 줄 바꿈 등 모든 문자가 토큰이 됩니다. LLM은 인간에게 보기 좋은 중괄호(`{}`) 구조에 관심이 없습니다. 그저 처리해야 할 토큰이 늘어날 뿐입니다.

토큰이 많아진다는 것은 다음과 같은 의미입니다:
- 비용 상승
- 추론(Inference) 속도 저하
- 모델이 혼란을 느낄 여지 증가

JSON은 오직 인간의 가독성을 위해 존재하는 요소들로 가득 차 있습니다. 이름, 이메일, 국가 등의 간단한 데이터만 보더라도 따옴표와 쉼표, 들여쓰기 등이 의미 없는 토큰을 생성합니다.

### TOON (Token Oriented Object Notation)의 등장

TOON의 핵심 아이디어는 간단합니다. **"구조는 유지하되, 노이즈는 제거하자"**는 것입니다. TOON은 JSON의 손실 없는 표현 방식이면서도 LLM 입력에 최적화되어 있습니다. 따옴표, 쉼표, 대괄호를 없애고 공백을 최소화하여 가독성은 유지하면서도 가볍게 만듭니다.

#### JSON vs TOON 비교 (토큰 수)
작은 프로필 데이터를 변환했을 때의 결과입니다:
- **JSON:** 68 토큰
- **TOON:** 48 토큰
- **YAML:** 51 토큰
- **XML:** 77 토큰

특별한 기술 없이 불필요한 문자만 제거했을 뿐인데 벌써 약 30%의 토큰이 줄어들었습니다.

### 배열과 계층 구조에서의 강점

TOON의 진가는 배열 데이터에서 드러납니다. 50명의 사용자 리스트를 전달할 때의 토큰 차이는 다음과 같습니다:
- **JSON:** 1,481 토큰
- **TOON:** 993 토큰

이것은 단순한 최적화가 아니라 데이터 전달 방식의 변화입니다. TOON은 균일한 배열 데이터를 CSV와 같은 테이블 형태로 평탄화(Flatten)합니다. 헤더 행을 두고 각 객체를 한 줄씩 표현함으로써 노이즈를 완전히 제거하며, LLM은 이러한 패턴을 매우 잘 이해합니다.

### LLM이 TOON을 선호하는 이유

TOON은 두 가지 개념을 결합합니다:
1. YAML과 같은 **들여쓰기 기반 구조**
2. CSV와 같은 **테이블 스타일 레이아웃**

이를 통해 모델은 불필요한 기호 없이 신호(Signal)에 더 집중할 수 있습니다. 벤치마크 결과에 따르면 토큰 사용량은 30~50% 감소하며, JSON 대비 소폭 향상된 정확도를 보이기도 합니다.

### 현실적인 관점: TOON은 "JSON 킬러"가 아니다

분명히 말하자면, JSON은 사라지지 않습니다. REST API나 프런트엔드 계약, 설정 파일 등을 당장 TOON으로 바꿀 필요는 없습니다. TOON은 인간을 위한 것도, 저장이나 상호운용성을 위한 것도 아닙니다. 오직 **"LLM과 효율적으로 통신하기 위한"** 용도입니다.

실제 시스템에서의 흐름은 다음과 같습니다:
1. 앱은 평소처럼 JSON으로 작동합니다.
2. LLM을 호출하기 직전에 JSON을 TOON으로 변환합니다.
3. 변환된 TOON을 프롬프트로 보냅니다.
4. 필요하다면 응답을 다시 변환합니다.

이렇게 하면 시스템 전체를 재작성하지 않고도 토큰 낭비를 막을 수 있습니다.

### 고려해야 할 트레이드오프(Trade-offs)

물론 단점도 존재합니다:
- 도구 생태계가 아직 미성숙함
- 대부분의 프레임워크에서 네이티브 지원이 없음
- 사람이 수동으로 디버깅하기에 불편함

TOON은 필수 사항이 아닌 '최적화'의 영역입니다. 모델은 여전히 JSON을 잘 처리합니다.

### 결론

JSON이 죽은 것은 아닙니다. 하지만 JSON이 LLM 시대를 위해 설계되지 않았음은 분명합니다. 우리는 캐릭터 단위로 비용을 지불하는 기계와 대화하면서, 인간 중심적인 포맷을 사용하고 있습니다.

JSON은 인간을 위한 것이고, 토큰 효율적인 포맷은 모델을 위한 것입니다. LLM에 대규모 구조화된 데이터를 보내면서 토큰 최적화를 고민하지 않고 있다면, 이제는 그 비용과 효율성을 다시 생각해야 할 때입니다.
