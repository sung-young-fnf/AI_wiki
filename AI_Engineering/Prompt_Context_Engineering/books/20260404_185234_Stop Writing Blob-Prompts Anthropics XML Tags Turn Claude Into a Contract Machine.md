# Stop Writing Blob-Prompts: Anthropic’s XML Tags Turn Claude Into a Contract Machine

여러분의 프롬프트는 "나쁜" 것이 아닙니다. 단지 확률론적 머신(probabilistic machine)에 전달되는 '형식이 지정되지 않은 입력값'일 뿐입니다. 그런데도 모델이 예시를 지시 사항으로 오해할 때마다 매번 놀라곤 합니다.

이제 '덩어리형 프롬프트(blob-prompts)'를 배포하는 것을 멈춰야 합니다.

문맥을 위해 "도움이 될 만한" 단락 하나를 추가할 때마다, 모델이 작업을 수행하는 대신 예시를 복제하기 시작하는 임계점이 옵니다. 이는 모델이 멍청해서가 아니라, 경계가 없는 문서를 주었기 때문입니다.

XML 구조화 프롬프팅은 앤스로픽(Anthropic)이 제시하는 가장 실용적인 신뢰성 해킹 기술입니다. 이는 프롬프트를 문맥(context), 지시 사항(instructions), 예시(examples), 형식(format) 등으로 라벨링된 구획으로 나누어 하나의 '계약서'처럼 기능하게 합니다.

### 가장 중요한 단 하나의 지표

이 글에서 가장 중요한 지표는 **'10회 실행 시 형식 오류율(Format-break rate over 10 runs)'**입니다.

"더 똑똑해진 것 같다"거나 "더 자신감 있게 들린다"는 식의 느낌(vibes)이 아닙니다. 실제로 10번을 실행해 보고, 여러분이 요청한 스키마(schema)를 출력이 얼마나 자주 위반하는지 확인해 보십시오. 앤스로픽은 지시 사항이 예시나 문맥과 섞이는 것을 방지하기 위해 XML 태그 사용을 권장하며, 이는 형식 준수 실패를 직접적으로 해결합니다.

### 작동 원리 (핵심 요약)

XML 프롬프팅은 단순히 XML 형식을 사용하는 것이 아니라 **'경계(boundaries)'**를 설정하는 것입니다.

1.  **라벨이 붙은 패킷:** 프롬프트가 라벨링된 필드를 가진 데이터 패킷이 됩니다.
2.  **모호함 제거:** 모델은 어떤 단락이 "법(지시)"이고 어떤 단락이 "배경(문맥)"인지 추측할 필요가 없습니다.
3.  **유연성:** 프롬프트 전체를 다시 쓰지 않고도 특정 섹션을 추가, 제거 또는 교체할 수 있습니다.
4.  **파싱 가능성:** 클로드(Claude)에게 태그가 지정된 섹션을 출력하도록 요청하여, 애플리케이션에서 해당 데이터를 안정적으로 추출할 수 있습니다.

### 활용 방법: 3가지 마이크로 테스트

#### 테스트 1: 배신하는 덩어리형 프롬프트 (Blob-prompt)
긴 이메일이나 정책 문서 등을 준비하고 아래와 같이 구조화되지 않은 프롬프트를 사용해 보십시오.
> "아래 텍스트를 읽고 1) 3문장 요약, 2) 5가지 실행 항목, 3) 위험 요소를 반환해 줘. [텍스트 붙여넣기]"

10번 실행 시, 요청한 형식이 깔끔하게 지켜지지 않는 횟수를 기록해 보십시오.

#### 테스트 2: 구획화된 프롬프트
동일한 작업에 구획을 나눕니다.
```xml
<instructions>
<context>의 텍스트를 요약하고 <formatting>에 정의된 대로 출력을 생성하세요. 추가 섹션은 포함하지 마세요.
</instructions>

<context>
[텍스트 붙여넣기]
</context>

<formatting>
반드시 다음 형식을 따르세요:
요약: (3문장)
실행 항목: (5개)
위험 요소: (없으면 "없음"으로 표기)
</formatting>
```
다시 10번 실행하여 형식 위반 횟수를 확인하십시오. 형식 오류율이 떨어진다면, 여러분은 문장 부호만으로 신뢰성을 확보한 것입니다.

#### 테스트 3: 예시 추가 (규칙 오염 방지)
개발자가 아닌 사용자들이 가장 자주 실수하는 부분입니다. 평문으로 예시를 추가하면 모델은 이것이 시연(demonstration)인지 명령(command)인지 구분하지 못합니다. 멀티샷(multishot) 프롬프팅 시에는 반드시 태그를 사용해야 합니다.

```xml
<instructions>
사용자 메시지를 <formatting>의 스키마로 변환하세요. 누락된 필드는 "unknown"으로 작성하세요.
</instructions>

<examples>
  <example>
    <input>사용자: "이번 달에 이중 결제됐어요. 송장 번호 1829."</input>
    <output>카테고리: 결제 / 긴급도: 상 / 참조: 1829</output>
  </example>
</examples>

<input>[사용자 메시지]</input>
<formatting>[출력 스키마]</formatting>
```

### 보너스: 복잡한 작업을 위한 생각의 분리

작업이 복잡할 때는 모델이 추론할 공간이 필요합니다. 앤스로픽의 생각의 사슬(Chain-of-thought) 가이드는 `<thinking>`과 `<answer>` 태그를 사용하여 추론 과정과 최종 답변을 분리할 것을 권장합니다.

```xml
<instructions>
문제를 해결하세요. 추론 과정은 <thinking>에, 최종 답변만 <answer>에 작성하세요.
</instructions>

<problem>
[복잡한 문제 입력]
</problem>

<thinking>
</thinking>

<answer>
</answer>
```
답변에 추론 과정이 섞여 나온다면, " `<thinking>`의 내용을 `<answer>`에 포함하지 마세요"라는 지시를 추가하여 경계를 강화할 수 있습니다.

### 냉정한 진실

XML 태그가 모호한 작업 자체를 구원해주지는 않습니다. 단지 여러분의 의도를 명확하게 전달하여 오해를 줄여줄 뿐입니다.

또한, 모든 작업에 '생각(thinking)' 단계가 필요한 것은 아닙니다. 앤스로픽은 생각의 사슬 방식이 출력 길이를 늘리고 지연 시간(latency)에 영향을 줄 수 있다고 경고합니다. 수학, 다단계 분석, 복잡한 문서 처리 등 인간도 생각을 해야 하는 작업에만 선별적으로 사용하십시오.

XML 프롬프팅은 취미로 프롬프트를 만드는 사람에게는 지루해 보일 수 있지만, 실제 기능을 배포해야 하는 팀에게는 생명줄과 같습니다. 형식 오류율을 측정하고 있지 않다면, 그것은 프롬프트 엔지니어링이 아니라 일기를 쓰고 있는 것입니다. 지금 바로 15분만 투자하여 10번의 테스트를 수행해 보십시오.
