OpenClaw 메모리 시스템 연구 보고서: 주요 발견 사항

지난 한 해 동안 거의 모든 AI 제품은 한 가지 단어, 즉 '메모리(memory)'에 대해 이야기해왔습니다.

ChatGPT는 "장기 기억(long-term memory)"을 추가하고 있고, Claude는 "맥락을 더 잘 이해하기 시작했다(understand context better)"고 말하며, 다양한 에이전트 프레임워크들은 "지속적인 대화(continuous dialogue)"를 강조하고 있습니다. 그러나 여전히 의문은 남습니다. 이 기억들은 누구에게 속하는가?

최근 입소문을 탄 오픈소스 프로젝트인 OpenClaw는 완전히 다른, 심지어 다소 파격적인(dangerous) 해답을 제시합니다.

OpenClaw는 기억을 클라우드(cloud)나 회사에 맡기거나 블랙박스(black box) 안에 숨기지 않습니다. 모든 기억을 온전히 사용자 자신의 하드 드라이브(hard drive)에 저장합니다.

MIT 라이선스(MIT license)를 받은 이 오픈소스 개인 AI 비서(personal AI assistant)는 Peter Steinberger가 개발했습니다. 폭발적인 성장을 거듭한 지 일주일 만에 Clawdbot은 GitHub에서 10만 개 이상의 스타(stars)를 빠르게 확보했습니다.

이 프로젝트의 기능은 특별할 것이 없습니다. 대규모 언어 모델(large models)이 잘하는 모든 것, 즉 이메일 관리, 여행 일정 계획, 항공편 체크인, 예약된 백그라운드 작업(background tasks) 실행 등입니다.

그러나 가장 중요한 차이점은 Discord, WhatsApp, Telegram, Slack과 같은 인기 채팅 도구(chat tools)와 통합되었을 때 놀라운 화학 반응(chemical reaction)이 일어난다는 것입니다. 바로 '잊지 않는다'는 점입니다.

3일 또는 3개월 동안 사용하지 않더라도, 컴퓨터를 다시 시작하거나(restart), 모델을 변경하거나, 맥락(context)을 지우더라도, OpenClaw는 사용자가 내린 결정, 선호도, 그리고 기록을 여전히 기억할 것입니다.

이는 업계 개발자들의 즉각적인 주목을 받았습니다. Clawdbot은 대체 어떻게 이런 일을 해낼 수 있었을까요?

얼마 전, Manthan Gupta라는 AI 연구 엔지니어(AI research engineer)가 이 문제에 대한 특별 연구를 수행했습니다. 그는 OpenClaw의 메모리 시스템을 분석하여 ChatGPT 및 Claude와는 매우 다르다는 것을 발견했습니다.

하지만 Manthan Gupta의 눈길을 사로잡은 것은 바로 OpenClaw의 '영구적인 메모리 시스템(persistent memory system)'이었습니다. 이 시스템은 24시간 내내 맥락적 기억(contextual memory)을 유지하고, 이전 대화를 기억하며, 과거 상호작용(interactions)을 기반으로 무한히 발전할 수 있었습니다.

ChatGPT 및 Claude와 비교했을 때, Clawdbot은 완전히 다른 길을 택했습니다. 클라우드 기반의(cloud-based), 기업이 제어하는(company-controlled) 메모리에 의존하지 않고, 모든 것을 로컬(local)에 유지하여 사용자에게 자신의 맥락(context)과 기술(skills)에 대한 완전한 통제권을 부여합니다.

### 맥락(Context) ≠ 기억(Memory)

이 문제를 명확히 하기 위해 Manthan Gupta는 업계에서 쉽게 혼동되는 개념인 '맥락 ≠ 기억'을 이해하는 것이 필수적이라고 언급했습니다.

"Clawdbot을 이해하려면, 많은 제품이 의도적으로 혼동하는 두 가지를 먼저 분리해야 합니다."

**맥락(Context)이란 무엇인가?** 맥락은 본질적으로 모델이 이번 요청(round of requests)에서 볼 수 있는 모든 정보입니다. 여기에는 시스템 프롬프트(System Prompt), 대화 기록(conversation history), 도구 반환 결과(tool return results), 현재 메시지(current message) 등이 포함됩니다.

대규모 언어 모델(large models)이나 에이전트(agents)를 개발하는 사람들이 맥락과 관련하여 골머리를 앓는 이유는 맥락이 세 가지 본질적인 단점을 가지고 있기 때문입니다. 즉, 짧고(short), 비싸며(expensive), 제한적입니다(limited).

*   **짧음(Short):** 현재 라운드(this round)까지만 남습니다.
*   **비쌈(Expensive):** 토큰(token)이 추가될수록 비용과 지연 시간(latency)이 증가합니다.
*   **제한적(Limited):** 맥락 창(context window) 제한(200,000, 1,000,000 토큰)의 제약을 받습니다.

그렇다면 **기억(Memory)이란 정확히 무엇인가?** Clawdbot에서 기억은 이미 모든 사람이 익숙한 또 다른 개념입니다. 바로 **기억 = 디스크에 저장된 파일(Files stored on the disk)** 입니다.

결국 프롬프트(prompt)와 도구의 결과는 AI를 위한 것입니다. 하지만 디스크에 있는 마크다운(Markdown) 파일은 사람이 읽을 수 있고(human-readable), 편집 가능하며(editable), 검색 가능합니다(searchable).

이 마크다운 파일은 사람들이 "기억"이라는 현상을 이해하는 방식과 더 잘 부합합니다.

*   **영속적(Durable):** 재시작 후에도, 다음 날에도, 다음 달에도 남아 있습니다.
*   **무한정(Unbounded):** 이론적으로 무한히 확장될 수 있습니다.
*   **저렴함(Cheap):** API 비용이 필요하지 않습니다.
*   **검색 가능(Searchable):** 의미론적 색인(semantic index)이 구축되어 있습니다.

Manthan은 이 개념을 바탕으로 Clawdbot이 AI를 "대화 도구(conversational tool)"에서 "장기적인 협력자(long-term collaborator)"로 변화시켰다고 말했습니다.

### 업계의 통념을 거스르는 설계: 마크다운 파일로 충분, "맥락 창(context windows)"은 필요 없다.

Clawdbot의 핵심 메모리 설계는 한 문장으로 요약할 수 있습니다. **기억은 그저 마크다운(Markdown)이다.**

이 시스템은 독점적인 데이터베이스(proprietary database)도, 사적인 클라우드 형식(private cloud format)도, 이해할 수 없는 내부 상태(internal state)도 가지고 있지 않습니다. 기본 디렉터리 구조는 다음과 같습니다.

```
~/clawd/
├── MEMORY.md              - Layer 2: Long-term curated knowledge
└── memory/
    ├── 2026-01-26.md      - Layer 1: Today's notes
    ├── 2026-01-25.md      - Yesterday's notes
    ├── 2026-01-24.md      - ...and so on
    └── ...
```

"AI 특권"은 존재하지 않습니다.

### 두 가지 계층의 메모리 설계

주목할 만한 핵심 설계 특징 중 하나는 Clawdbot이 기억을 두 가지 계층으로 나눈다는 점입니다.

**첫 번째 계층: 일일 기록(Daily log, 단기적, 원본).** 이는 사람의 업무 수첩과 유사합니다.

오늘 나눈 대화, 내린 결정, 그리고 우연히 언급한 선호도 등 모든 것을 기록합니다.

```
## 10:30 AM - API Discussion
Decided to use REST over GraphQL.
## 4:00 PM - User Preference
User prefers TypeScript over JavaScript
```

**두 번째 계층: 장기 기억(long-term memory, 조직화 및 침전).** 이는 인간 두뇌의 "지식 기반(knowledge base)"으로 이해할 수 있습니다.

특정 정보가 반복적으로 언급되거나, 확인되거나, 인용되면 에이전트(agent)는 이를 `MEMORY.md` 파일로 정리합니다.

이 단계는 메모장을 상식으로 '업그레이드'하는 것과 같습니다.

```
# Long-term Memory

## User Preferences
- Prefers TypeScript over JavaScript
- Likes concise explanations
- Working on project "Acme Dashboard"

## Important Decisions
- 2026-01-15: Chose PostgreSQL for database
- 2026-01-20: Adopted REST over GraphQL
- 2026-01-26: Using Tailwind CSS for styling

## Key Contacts
- Alice (alice@acme.com) - Design lead
- Bob (bob@acme.com) - Backend engineer
```

세션 런타임(session runtime) 시 자동으로 로드되는 `AGENT.md` 파일은 이러한 기억을 읽는 몇 가지 원칙을 명시합니다.

`SOUL.md` (참고: 여기서 "영혼(soul)"은 개발자 Peter가 아직 완전히 오픈소스화되지 않았다고 언급한 유일한 '비밀'입니다.)는 에이전트가 누구인지 알려줍니다.

`USER.md`는 에이전트가 누구를 위해 봉사하는지 알려줍니다.

`memory/YYYY-MM-DD.md` 파일은 현재 맥락(current context)을 저장합니다.

세션 중이라면 `MEMORY.md` 파일도 읽어야 합니다.

```
## Every Session

Before doing anything else:

1. Read SOUL.md — this is who you are

2. Read USER.md — this is who you are helping

3. Read memory/YYYY-MM-DD.md (today and yesterday) for recent context

4. If in MAIN SESSION (direct chat with your human), also read MEMORY.md
```

**허락을 구하지 말고, 그냥 실행하라.**

흥미롭게도 Peter는 특별한 규칙을 설정했습니다. "허락을 구하지 말고, 그냥 실행하라(Don’t ask for permissions, just do it!)"

### 기억 파일은 어떻게 색인화되는가?

기억 파일이 저장되면, 백그라운드 프로세스(background process)는 이 기억들을 조각내어 벡터 의미론적 색인(vector semantic index)을 생성합니다.

```
┌─────────────────────────────────────────────────────────────┐
│  1. File Saved                                              │
│     ~/clawd/memory/2026-01-26.md                            │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  2. File Watcher Detects Change                             │
│     Chokidar monitors MEMORY.md + memory/**/*.md            │
│     Debounced 1.5 seconds to batch rapid writes             │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  3. Chunking                                                │
│     Split into ~400 token chunks with 80 token overlap      │
│                                                             │
│     ┌────────────────┐                                      │
│     │ Chunk 1        │                                      │
│     │ Lines 1-15     │──────┐                               │
│     └────────────────┘      │                               │
│     ┌────────────────┐      │ (80 token overlap)            │
│     │ Chunk 2        │◄─────┘                               │
│     │ Lines 12-28    │──────┐                               │
│     └────────────────┘      │                               │
│     ┌────────────────┐      │                               │
│     │ Chunk 3        │◄─────┘                               │
│     │ Lines 25-40    │                                      │
│     └────────────────┘                                      │
│                                                             │
│     Why 400/80? Balances semantic coherence vs granularity. │
│     Overlap ensures facts spanning chunk boundaries are     │
│     captured in both. Both values are configurable.         │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  4. Embedding                                               │
│     Each chunk -> embedding provider -> vector              │
│                                                             │
│     "Discussed REST vs GraphQL" ->                          │
│         OpenAI/Gemini/Local ->                              │
│         [0.12, -0.34, 0.56, ...]  (1536 dimensions)         │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  5. Storage                                                 │
│     ~/.clawdbot/memory/<agentId>.sqlite                     │
│                                                             │
│     Tables:                                                 │
│     - chunks (id, path, start_line, end_line, text, hash)   │
│     - chunks_vec (id, embedding)      -> sqlite-vec         │
│     - chunks_fts (text)               -> FTS5 full-text     │
│     - embedding_cache (hash, vector)  -> avoid re-embedding │
└─────────────────────────────────────────────────────────────┘
```

다음 사항에 주목하십시오.

*   `sqlite-vec`은 외부 벡터 데이터베이스(vector database) 없이 SQLite 내에서 직접 벡터 유사성 검색(vector similarity searches)을 수행할 수 있게 해주는 SQLite 확장(extension)입니다.
*   `FTS5`는 SQLite에 내장된 전체 텍스트 검색 엔진(full-text search engine)으로, BM25 키워드 매칭(keyword matching)을 지원합니다.
*   이 둘은 OpenCLAW가 단일의 경량 데이터베이스(lightweight database) 파일에서 하이브리드 검색(hybrid searches, 의미론적 검색 + 키워드 검색)을 실행할 수 있도록 합니다.

### 비결: OpenClaw는 이 기억들을 어떻게 "기억"하는가?

이제 중요한 부분입니다.

OpenClaw는 모든 기억을 한꺼번에 맥락(context)에 쏟아붓지 않습니다. 그 접근 방식은 간단합니다. **먼저 검색하고(search first), 그 다음 주입(inject)합니다.**

**어떻게 검색을 수행하는가?**

"과거 사건"과 관련된 경우, 에이전트(agent)는 먼저 기억 검색 과정을 거쳐야 합니다.

*   벡터 의미론적 검색(Vector semantic search, 사용자가 말하는 것을 이해)
*   BM25 키워드 검색(keyword BM25 search, 고유 명사가 누락되지 않도록 확인)

두 가지의 가중치 융합(weighted fusion):

`finalScore = 0.7 * semantic + 0.3 * keyword`

결과가 충분히 관련성이 없으면 버려집니다.

이것이 무엇을 의미할까요? 이는 맥락이 "잊어버릴까 봐 모든 것을 다 집어넣는" 대신, 현재 진정으로 필요한 기억만을 포함해야 한다는 것을 의미합니다.

**한 가지 중요한 점: 외부 데이터베이스를 사용하지 않았습니다.**

많은 사람들이 이 점을 간과합니다.

위에서 언급했듯이, 모든 색인(indexes)은 단일 로컬(local) `.sqlite` 파일에 있습니다.

Clawdbot의 벡터 검색(vector search)은 외부 벡터 데이터베이스를 사용하지 않고, SQLite, `sqlite-vec`, `FTS5`를 활용합니다. SaaS 종속성(SaaS dependencies)이 없습니다.

이는 또한 Peter의 "개인 에이전트(personal agents)"에 대한 설계 철학을 보여줍니다. 즉, 개인 AI 비서는 단일 파일(single file)이어야 하며, 이식 가능하고(portable), 백업 가능해야(backable) 합니다.

### 긴 대화는 어떻게 처리하는가? 맥락 창(Context window).

현실은 가혹합니다. 아무리 큰 맥락 창(context window)이라도 결국 공간이 부족해질 것입니다.

OpenClaw의 해결책은 간단합니다. **압축(compression)**입니다. (편집자 주: 이는 지난 주말 OpenAI가 공개한 Codex의 접근 방식과 유사하며, 이 역시 압축을 포함합니다.)

*   초기 대화를 구조화된 요약(structured summary)으로 요약합니다.
*   가장 최근의 원본 메시지(original message)를 유지합니다.
*   요약을 프롬프트(prompt)에 저장하는 대신 디스크에 기록합니다.

여기에 특별한 단계가 있습니다. 압축 전에 기억이 강제로 새로 고쳐집니다(forcibly refreshed). 즉, 모델이 "잊기" 전에 중요한 정보를 마크다운(Markdown) 파일에 기록하는 것입니다.

이 단계는 매우 영리합니다. 맥락을 압축하면서도 "요약할 때 중요한 결정을 부주의하게 지우는" 고전적인 실수를 방지하기 때문입니다.

```
┌─────────────────────────────────────────────────────────────┐
│  Context Approaching Limit                                  │
│                                                             │
│  ████████████████████████████░░░░░░░░  75% of context       │
│                              ↑                              │
│                    Soft threshold crossed                   │
│                    (contextWindow - reserve - softThreshold)│
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Silent Memory Flush Turn                                   │
│                                                             │
│  System: "Pre-compaction memory flush. Store durable        │
│           memories now (use memory/YYYY-MM-DD.md).          │
│           If nothing to store, reply with NO_REPLY."        │
│                                                             │
│  Agent: reviews conversation for important info           │
│         writes key decisions/facts to memory files        │
│         -> NO_REPLY (user sees nothing)                     │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Compaction Proceeds Safely                                 │
│                                                             │
│  Important information is now on disk                       │
│  Compaction can proceed without losing knowledge            │
└─────────────────────────────────────────────────────────────┘
```

### 다중 에이전트, 다중 페르소나, 그러나 그들의 기억은 서로 격리되어 있다.

OpenCLAW는 여러 에이전트(agents)를 지원합니다: 개인(Personal), 업무(work), 실험(experimental), 자동화 스크립트(automation scripts).

각 에이전트(Agent)는 독립적인 작업 공간(workspace), 기억(memory), 그리고 색인(index)을 가지며, 기본적으로 서로의 데이터를 읽지 않습니다.

WhatsApp의 "생활 비서"가 Slack에서 하는 업무에 대해 완전히 알지 못하게 할 수 있습니다.

이는 오늘날의 AI 제품에서는 거의 '사치스러운' 기능(luxury feature)입니다.

```
~/.clawdbot/memory/              # State directory (indexes)
├── main.sqlite                  # Vector index for "main" agent
└── work.sqlite                  # Vector index for "work" agent

~/clawd/                         # "main" agent workspace (source files)
├── MEMORY.md
└── memory/
    └── 2026-01-26.md

~/clawd-work/                    # "work" agent workspace (source files)
├── MEMORY.md
└── memory/
    └── 2026-01-26.md
```

### 요약: Clawdbot 메모리 시스템의 네 가지 원칙

Clawdbot의 메모리 시스템은 다음과 같은 몇 가지 핵심 원칙을 따랐기 때문에 성공했습니다.

1.  **블랙박스(black box)보다 투명성이 우수하다.**
    기억은 일반 마크다운(Markdown) 형식을 사용합니다. 사용자는 이를 읽고, 편집하며, 버전 관리(version control)할 수 있습니다. 모호한 데이터베이스나 독점적인 형식(proprietary formats)을 사용하지 않습니다.
2.  **주입(injection)보다 검색(search)이다.**
    모든 정보를 맥락에 쑤셔 넣는 대신, 에이전트(agent)는 관련 정보를 검색합니다. 이는 비용을 줄이면서 맥락의 집중도를 유지합니다.
3.  **세션 기반(session-based)보다 영속성(persistence)이 우수하다.**
    중요한 정보는 채팅 기록(chat history)뿐만 아니라 디스크의 파일에도 저장됩니다. 압축(compression)은 이미 저장된 내용을 삭제할 수 없습니다.
4.  **단일 검색(single search)보다 하이브리드 검색(hybrid search)이 우수하다.**
    벡터 검색(vector search)만으로는 정확한 일치(exact match)를 제공할 수 없으며, 키워드 검색(keyword search)만으로는 의미론적 정보(semantic information)를 포착할 수 없습니다. 그러나 하이브리드 검색은 이 둘 모두를 달성할 수 있습니다.

### 정말로 중요한 것: 사용자가 자신의 데이터를 제어한다

돌이켜보면, 구현 방식에만 집중한다면 OpenClaw는 신비롭지 않습니다. 그러나 Peter가 이 프로젝트를 처음 만들었을 때 의도했던 바는 다음과 같습니다.

"올해는 개인 에이전트(personal agents)의 해가 될 것이며, OpenAI나 Anthropic과 같은 대기업들이 이 분야를 지배할 가능성이 높습니다. 하지만 저는 다른 선택을 하고 싶습니다. 거대 기업들에게 더 많은 데이터를 넘겨주는 대신, 당신이 당신 자신의 데이터를 제어할 수 있도록 하는 것입니다."

따라서 OpenClaw는 하나의 신호를 보냅니다. 즉, AI의 기억이 사용자에게로 돌아오기 시작했다는 것입니다.

기억은 로컬 디스크(local disk)에 저장될 수 있으며, 사용자는 자신의 채팅 기록(chat history)처럼 읽고(readable), 제어하고(controllable), 삭제하고(deletable), 전송할(transferable) 수 있습니다.

분명히, 이 방식은 사용자들에게 더 큰 인기를 얻고 있습니다. 2026년 초에 이 프로젝트가 소셜 미디어(social media)를 휩쓸었던 이유도 바로 이 때문입니다.

결국, 에이전트(agents)가 사용자들과 장기간 동반하기 시작하고, AI가 단순히 질문에 답하는 것을 넘어 의사 결정(decision-making), 실행(execution), 협업(collaboration)에 참여하게 되면, 기억은 "기능(function)"에서 "권력(power)"으로 변모합니다.

이 기억을 소유하는 자가 AI가 궁극적으로 어느 편에 설지를 결정하게 될 것입니다.
