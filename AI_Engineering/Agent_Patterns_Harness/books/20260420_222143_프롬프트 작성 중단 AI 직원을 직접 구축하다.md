# 프롬프트 작성 중단: AI 직원을 직접 구축하다

하루 종일 AI를 '사용'했지만, 실제로 완료된 일은 없었습니다. 수많은 도구를 연결하고 제 받은 편지함이 스스로 답장을 보내기 시작한 날의 이야기입니다.

저의 일상은 매우 생산적으로 보였습니다. 이메일을 붙여넣고 AI에 질문한 다음, 답장을 복사하고 수정하여 보내는 과정을 40번 반복했습니다. 기술적으로는 AI의 도움을 받고 있었지만, 업무량은 줄어들지 않았습니다. 저는 여전히 애플리케이션(앱) 사이를 이어주는 운영 체제(operating system) 역할에 머물러 있었습니다.

그때 깨달았습니다.
AI는 저를 '보조'하는 것이 아니라, '워크플로우(workflow) 자체를 대체'해야 한다는 것을요.

목표는 더 나은 프롬프트를 만드는 것이 아니었습니다.
목표는 '입력(input)부터 결과(outcome)까지 수동 단계를 0으로 만드는 것'이었습니다.

그래서 저는 위험한 것을 만들었습니다. 수신 메시지를 읽고, 의도를 이해하고, 조치를 취하며, 자동으로 답장하는 AI를 말입니다.

## 모든 것을 바꾼 시스템 설계

저는 챗봇(chatbot) 대신 파이프라인(pipeline)을 설계했습니다.

*   수신 메시지 읽기
*   의도 이해
*   조치 결정
*   도구 사용 (검색, 데이터베이스, 캘린더, 이메일)
*   응답 생성
*   제가 아무것도 건드리지 않고 전송

AI는 더 이상 '기능(feature)'이 아니었습니다.
AI는 '운영자(operator)'가 된 것입니다.

## 1단계 — 두뇌: 자유 형식 채팅 대신 구조화된 추론

자동화에서 가장 큰 실수는 모델이 자유롭게 떠들도록 내버려 두는 것입니다. 우리는 '결정'을 원하지, '에세이'를 원하는 것이 아닙니다.

우리는 모델이 JSON 형식의 지시를 출력하도록 강제했습니다.

```python

당신은 운영 보조원입니다.

{

 "action": "...",
 "reply": "...",

}
"""
```

이것이 전환점이었습니다.
우리는 AI에게 '답장'을 요청하지 않았습니다.
우리는 AI에게 '결정'을 요청했습니다.

프롬프트도, 감독도 필요 없습니다.

## 예상치 못한 수익화 기회

저는 이 시스템을 원래 제 개인 받은 편지함을 위해 만들었습니다. 그러다 한 프리랜서 친구가 고객 FAQ에 답할 수 있는지 물어봤습니다. 그다음에는 한 에이전시가 리드 자격(lead qualification)을 위해 원했고, SaaS 창업가는 온보딩(onboarding) 자동화를 원했습니다.

핵심 엔진은 동일하고, 겉껍질(wrapper)만 다를 뿐입니다.

이는 다음과 같은 서비스로 확장됩니다.

*   유료 받은 편지함 자동화
*   고객 지원 AI
*   리드 자격 봇
*   예약 서비스

코드는 거의 바뀌지 않습니다. 단지 프롬프트와 도구만 변경될 뿐입니다.

반복 결제(recurring billing)가 갑자기 합리적으로 들리기 시작했습니다. 왜냐하면 당신은 AI 응답을 파는 것이 아니기 때문입니다.
당신은 '작업이 자동으로 완료되는 것'을 팔고 있는 것입니다.
