AI 에이전트를 위한 CLI 대 MCP: CLI 툴 호출 에이전트 구축 방법!

AI 에이전트 생태계에서 CLI(Command Line Interface) 툴과 MCP(Model Context Protocol) 간의 논쟁이 가열되고 있습니다. 이 글은 실제 CLI 툴 호출 에이전트를 구축하는 방법을 보여주고, LLM(거대 언어 모델)이 사용자 의도를 추론하고 셸 명령어를 스킬(skill)로 실행하는 방법을 설명합니다.

AI 에이전트는 인보이스 생성부터 CRM(고객 관계 관리) 데이터 관리까지, 비즈니스 워크플로우를 자동화하는 데 점점 더 많이 사용되고 있습니다. 전통적으로 이러한 에이전트를 외부 툴과 통합하려면 표준화된 접근을 위해 MCP와 같은 프로토콜에 의존해야 했습니다. 하지만 더 간단하고 매우 효과적인 대안이 부상하고 있습니다. 바로 CLI(명령줄 인터페이스) 툴을 에이전트 스킬로 활용하는 것입니다. 셸 스크립트와 명령어를 "스킬"로 래핑(wrapping)함으로써, 에이전트는 특수 프로토콜의 오버헤드 없이 추론하고, 작업을 실행하며, 결과를 반환할 수 있습니다. 이 데모에서는 Tools4AI가 이러한 아키텍처를 어떻게 구현하는지 정확히 보여주며, 가볍고 강력하며 실용적인 AI 자동화 에이전트를 쉽게 구축할 수 있도록 합니다.

이 프로젝트의 코드는 다음에서 확인할 수 있습니다: [https://github.com/vishalmysore/cli-tool-calling](https://github.com/vishalmysore/cli-tool-calling)

⚖️ **아키텍처적 트레이드오프(Architectural Trade-offs): CLI 대 MCP**

MCP(Model Context Protocol)와 CLI 기반의 툴 호출은 서로 다른 추상화 방식을 통해 유사한 목표를 달성합니다. 각각을 언제 사용해야 하는지 이해하는 것이 시스템 아키텍처에 매우 중요합니다.

**CLI 기반 실행 (직접 경로 - Direct Path)**

*   **비결합 다중 언어 지원 (Decoupled Polyglotism)**: 스크립트는 서버 래퍼(wrapper)가 필요 없이 모든 언어(Python, Bash, Go 등)로 작성될 수 있습니다.
*   **낮은 지연 시간 (Low Latency)**: 네트워크나 IPC(Inter-Process Communication) 오버헤드 없이, 스크립트가 에이전트 프로세스의 직접적인 하위 프로세스로 호출됩니다.
*   **레거시 통합 (Legacy Integration)**: '서비스'로 설계되지 않은 기존 엔터프라이즈 스크립트를 래핑하는 데 이상적입니다.
*   **복잡성 (Complexity)**: 메타데이터는 레지스트리(여기서 사용된 YAML과 같은)에 수동으로 유지 관리되어야 합니다.

**MCP (프로토콜화된 경로 - Protocolized Path)**

*   **표준화된 검색 (Standardized Discovery)**: 독립적인 서비스 간의 동적 툴 협상이 가능합니다.
*   **분산 아키텍처 (Distributed Architecture)**: 툴은 별도의 서버나 컨테이너에 위치할 수 있습니다.
*   **리소스 관리 (Resource Management)**: 내부 상태를 유지하는 장기 실행 툴 서비스에 더 적합합니다.
*   **오버헤드 (Overhead)**: 특정 전송 계층(STDIO/HTTP)을 구현하고 프로토콜 수준의 핸드셰이크를 처리해야 합니다.

**일부 팀이 CLI를 선호하는 이유**

**토큰 효율성 (Token Efficiency)**

LLM(거대 언어 모델)은 툴 정의를 컨텍스트 윈도우(context window)에 로드해야 합니다.

MCP 툴은 종종 매개변수와 기능을 설명하는 큰 JSON 스키마를 포함합니다.

하지만 CLI 명령어는 매우 작습니다.

이는 다음을 의미합니다:

*   추론을 위한 더 많은 컨텍스트 사용 가능
*   더 낮은 토큰 비용
*   더 빠른 응답

많은 경우, 에이전트는 전체 툴 사양을 로드하는 대신 단일 명령만 실행하면 됩니다.

**더 간단한 인프라 (Simpler Infrastructure)**

MCP는 여러 구성 요소를 필요로 합니다:
에이전트 → MCP 클라이언트 → MCP 서버 → 툴

CLI 워크플로우는 더 간단합니다:
에이전트 → 명령 → 결과

계층이 적다는 것은 실패 지점이 적고 디버깅이 더 쉽다는 것을 의미합니다.

**LLM은 이미 CLI를 이해합니다 (LLMs Already Understand CLIs)**

LLM은 Git 명령어, Linux 유틸리티, 클라우드 CLI, 데이터베이스 툴 등 방대한 양의 개발자 데이터로 훈련되었습니다.

이는 에이전트가 종종 `git`, `docker`, `psql`, `kubectl`, `aws`와 같은 툴을 사용하는 방법을 이미 알고 있음을 의미합니다.

추가적인 스키마나 프로토콜이 필요하지 않습니다.

**MCP가 여전히 중요한 이유 (Why MCP Still Matters)**

CLI 툴의 장점에도 불구하고, MCP는 몇 가지 중요한 문제를 해결합니다.

**1. 보안 및 거버넌스 (Security and Governance)**

엔터프라이즈 환경에서 셸 접근은 위험할 수 있습니다.

MCP는 다음을 가능하게 합니다:
*   권한 제어
*   감사(Auditing)
*   범위가 지정된 접근
*   안전한 실행 환경

**2. 다중 에이전트 협업 (Multi-Agent Collaboration)**

여러 에이전트가 상호 작용할 때, 공유된 컨텍스트와 표준화된 툴 접근이 필요합니다.

MCP와 같은 프로토콜은 조직 및 서비스 간의 이러한 조정을 가능하게 합니다.

**3. 엔터프라이즈 데이터 접근 (Enterprise Data Access)**

대규모 조직은 내부 API, 데이터 레이크(data lakes), SaaS 시스템, 고객 데이터와의 구조화된 통합을 필요로 합니다.

MCP 서버는 더 안전한 추상화 계층을 제공합니다.

**결론: 하이브리드 에이전트 아키텍처를 향하여**

이 데모는 CLI 기반 툴 호출이 로컬 자동화, 신속한 프로토타이핑(prototyping), 기존 스크립트 오케스트레이션(orchestration)을 위한 효율적이고 실용적인 접근 방식임을 보여줍니다. 동시에 MCP와 같은 프로토콜은 분산 또는 장기 실행 서비스에 대한 표준화, 동적 검색(discovery), 거버넌스를 제공합니다. CLI와 MCP를 상호 배타적인 것으로 여기기보다는, 가장 견고한 에이전트 아키텍처는 종종 이 둘을 결합합니다. 즉, 빠르고 로컬 실행을 위해 CLI 툴을 사용하고, 원격, 상태 저장(stateful) 또는 엔터프라이즈급 서비스를 위해 MCP를 사용합니다. 하이브리드 접근 방식을 수용함으로써 개발자는 작업에 따라 각 방법의 강점을 활용하여 가볍고 확장 가능한 AI 에이전트를 구축할 수 있습니다.

에이전트 자동화의 미래는 어느 하나를 선택하는 것이 아니라, 효율성, 유연성, 신뢰성을 극대화하기 위해 CLI와 MCP 툴을 지능적으로 오케스트레이션하는 데 있습니다.
