# 안정적인 AI 에이전트 시스템 구축을 위한 7가지 핵심 하네스 구성 요소

운영 환경에서 에이전트가 없어서는 안 될 인프라 계층이지만, 대부분의 엔지니어는 이를 구축하지 않습니다.

---

에이전트가 단일 질의에 대해 사용자에게 38달러를 청구했습니다.

복잡한 작업을 수행해서가 아니었습니다. 동일한 문서를 47번 연속으로 요약했기 때문입니다. 이미 작업을 수행했음을 감지했음에도 불구하고 다시 작업을 반복했습니다. 충돌도, 경고도 없었습니다. 그저 무한 루프(spinning loop)에 빠져 청구서만 계속 늘어났을 뿐입니다.

모델 로그를 확인했습니다. 모델은 훈련된 대로 정확히 작동하고 있었습니다.

문제는 모델을 둘러싼 모든 것이 이전에 수행한 작업을 전혀 기억하지 못하고, 상태 파일도 없고, 중지 조건도 없었다는 점입니다. 시스템은 다음과 같이 말할 방법이 없었습니다.

"우리는 여기 와 본 적이 있어."

이것이 바로 데모와 실제 운영 에이전트 간의 차이입니다.

이 글에서는 안정적인 AI 에이전트를 구축하는 데 사용할 수 있는 7가지 하네스(Harness)에 대해 설명합니다.

## 아무도 경고하지 않는 격차

한 번 작동하는 에이전트를 구축하는 것은 정말 쉽습니다. LLM을 호출하고, 도구를 제공하고, 루프를 돌게 합니다. 20줄의 파이썬 코드면 충분합니다. 데모를 기록하면 깔끔해 보입니다.

그런 다음 배포합니다. 실제 사용자들은 예상치 못한 입력을 보냅니다.

도구 호출이 빈 값을 반환합니다. 컨텍스트(Context)는 40분 후에 가득 찹니다. 두 개의 서브 에이전트(subagent)가 서로 모순됩니다. 모델은 무기한으로 어떤 것을 재시도하기로 결정합니다.

데모에서는 보이지 않던 모든 것이 운영 환경에서는 실패가 됩니다.

이 격차는 모델의 품질 문제가 아닙니다. 하네스(Harness)의 품질 문제입니다.

> "하네스 없는 모델은 신경계 없는 두뇌와 같습니다. 생각은 하지만, 그 외에는 아무것도 하지 못합니다."

**에이전트 = 모델 + 하네스**

이러한 프레임워크는 구축 방식을 변화시킵니다.

**에이전트 = 모델 + 하네스**

*   **모델** → 추론, 언어, 결정
*   **하네스** → 모델이 안정적으로 작동하는 데 필요한 모든 것

모델이 아니라면, 당신은 하네스입니다.

하네스는 모델을 감싸고 텍스트 생성기를 실제로 작동하는 무언가로 바꾸는 모든 코드, 모든 설정(config), 모든 실행 후크(execution hook)를 의미합니다.

모델은 무엇을 할지 결정합니다. 하네스는 모델이 안전하고 반복적으로, 그리고 대규모로 그 작업을 수행할 수 있도록 보장합니다.

대부분의 엔지니어는 모델에 90%의 시간을 할애합니다. 더 나은 프롬프트, 최신 모델, 더 많은 예시 등을 연구합니다. 하지만 운영 환경에서의 실패는 거의 항상 건너뛴 10%에서 발생합니다.

## 실제로 중요한 7가지 구성 요소

### 1. 제어 루프 (Control Loop)

루프는 에이전트의 심장 박동과 같습니다. 루프가 없으면 모델 호출 한 번과 응답 한 번만 발생합니다. 이는 에이전트가 아니라 챗봇입니다.

루프는 모델을 실행하고, 모델이 반환한 내용을 읽고, 모든 도구 호출을 실행하고, 결과를 다시 피드하고, 모델이 더 이상 도구를 호출하지 않거나 단계 제한(step limit)에 도달할 때까지 반복됩니다.

```python
while agent_is_running:
    response = call_model(context)

    if response.has_tool_calls:
        results = execute_tools(response.tool_calls)
        append_to_context(results)
        continue

    if response.is_final_answer:
        return response.content

    if step_count > MAX_STEPS:
        return "Task incomplete. Max steps reached."
```

`MAX_STEPS` 라인은 선택 사항이 아닙니다. 이는 잘 작동하는 에이전트와 38달러 청구 사건의 차이입니다. 단일 도구를 작성하기 전에 이를 구축해야 합니다.

나쁜 루프는 루프가 없는 것보다 더 나쁩니다. 중지 조건, 상태 추적, 반복적인 도구 호출 감지 기능이 없으면, 모델은 이미 완료한 작업을 무기한으로 계속 수행할 수 있습니다.

### 2. 상태 관리 (State Management)

모델은 기본적으로 무상태(stateless)입니다. 모든 API 호출은 새로 시작됩니다. 하네스가 발생한 일을 명시적으로 추적하지 않으면, 에이전트는 이미 무엇을 했는지, 무엇이 성공했는지, 어디까지 진행되었는지 전혀 기억하지 못합니다.

두 가지 유형의 상태가 필요합니다.

*   **세션 상태 (Session state)**: 현재 대화에서 발생한 일을 다룹니다. 대화 기록, 도구 결과, 현재 단계 번호 등이 포함됩니다.
*   **영구 상태 (Persistent state)**: 세션이 종료된 후에도 유지되는 상태입니다. 장기 작업의 진행 상황, 완료된 하위 작업(subtask), 이미 처리된 파일 등이 해당합니다.

가장 간단한 운영 환경용 상태 저장소는 JSON 파일입니다. 작업 진행 상황, 이미 처리된 항목, 현재 상태를 추적합니다. 이는 읽기 쉽고, 디버깅 가능하며, 프로세스 재시작에도 살아남고, 별도의 인프라를 필요로 하지 않습니다.

```json
{
  "task_id": "refactor-auth-module",
  "completed_files": ["auth.py", "middleware.py"],
  "pending_files": ["routes.py", "tests/test_auth.py"],
  "current_step": 3
}
```

대규모 코드베이스에서 작업하는 코딩 에이전트의 경우, 이 파일은 진행 상황을 만드는 에이전트와 매 루프마다 동일한 파일을 다시 편집하는 에이전트를 구분하는 역할을 합니다. Git은 여기에 버전 관리(versioning)를 추가하여 에이전트가 작업을 추적하고, 실수를 되돌리고, 실험을 분기할 수 있게 합니다.

### 3. 메모리 (Memory)

상태(State)는 에이전트가 이 세션에서 무엇을 했는지 추적합니다. 메모리(Memory)는 세션 전반에 걸쳐 에이전트가 무엇을 알고 있는지를 의미합니다.

*   **단기 메모리 (Short-term memory)**는 대화 기록입니다. 모든 메시지, 도구 호출 및 결과가 모델에 전달되는 목록에 추가됩니다.
    이는 구현하기는 쉽지만, 관리하지 않으면 비용이 많이 듭니다. 목록이 길어질수록 토큰(token) 비용은 증가하고, 하드 한계에 도달하기 전에 성능이 저하됩니다.
*   **장기 메모리 (Long-term memory)**는 더 어렵습니다. 코드 작성을 돕는 에이전트는 사용자가 예외 처리(exceptions)보다 명시적 오류 처리(explicit error handling)를 선호한다는 것을 기억해야 합니다. 고객 지원을 처리하는 에이전트는 특정 고객이 지난주에 결제 문제가 있었다는 것을 알아야 합니다. 이는 일반적으로 의미론적 검색(semantic retrieval)을 위해 벡터 데이터베이스(vector database)에 저장되거나, 사실이 구체적일 때는 구조화된 파일에 저장됩니다.

좋은 운영 패턴:

**세션 시작:**
1.  AGENTS.md 또는 프로젝트 메모리 파일 로드 → 시스템 프롬프트(system prompt)에 주입
2.  현재 작업에 따라 관련 메모리 검색 → 컨텍스트에 추가

**세션 중:**
3.  롤링 대화 기록 유지

**세션 종료:**
4.  주요 학습 요약 → 메모리 저장소에 기록

하네스는 1, 2, 4단계를 처리합니다. 모델은 자체 메모리를 관리하지 않습니다. 모델은 그럴 수 없습니다.

장기 메모리가 없는 에이전트는 매 실행마다 컨텍스트를 다시 학습합니다. 사용자들은 이를 알아챕니다. 모델은 완벽하게 기능할 수 있음에도 불구하고 에이전트가 자신을 잊어버린다고 느끼기 시작합니다. 이러한 신뢰 상실은 모델 문제가 아니라 하네스 문제입니다.

### 4. 도구 (Tools)와 Bash 이스케이프 해치 (Bash Escape Hatch)

도구는 언어를 행동으로 전환시키는 역할을 합니다. 도구가 없으면 모델은 무언가를 하는 것에 대한 텍스트를 생성할 뿐입니다. 도구가 있으면 실제로 그 일을 수행합니다.

도구 설계는 도구 개수보다 중요합니다. 도구를 하나 추가할 때마다 컨텍스트 비용(도구 설명이 프롬프트에 포함됨)이 발생하고, 모델이 잘못된 도구를 선택할 가능성이 높아집니다. 훌륭한 설명을 가진 3개의 도구가 모호한 설명을 가진 15개의 도구보다 성능이 뛰어날 것입니다.

좋은 도구 설명은 세 가지 질문에 답합니다.

*   이 도구는 실제로 무엇을 하는가?
*   언제 사용해야 하는가 (사용할 수 있을 때뿐만이 아니라)?
*   출력은 어떤 모습이며, 이를 통해 작동했는지 어떻게 알 수 있는가?

**Bash 이스케이프 해치 (bash escape hatch)**는 에이전트가 수행할 수 있는 작업을 변화시키는 아키텍처적 조치입니다. 가능한 모든 도구를 미리 설계하는 대신, 에이전트에게 Bash 접근 권한을 부여하여 필요에 따라 자체 도구를 즉석에서 작성할 수 있게 합니다. 이것이 Claude Code가 개방형 작업을 처리하는 방식입니다. 모델은 고정된 도구 세트에 얽매이지 않고, 필요한 것을 설계합니다.

그에 대한 대가(tradeoff)는 보안입니다. Bash가 사용되는 순간 샌드박스 격리(sandbox isolation)는 필수 불가결한 요소가 됩니다.

에이전트가 생성한 코드를 로컬에서 실행하는 것은 위험하며, 단일 환경은 동시 워크로드에 확장되지 않습니다. 샌드박스는 에이전트에게 안전하고 격리된 실행 환경을 제공합니다. 코드를 실행하고, 패키지를 설치하고, 파일을 검사하는 모든 작업을 호스트 시스템에 영향을 주지 않고 수행할 수 있습니다. 샌드박스는 필요에 따라 스핀업되고, 병렬 작업을 위해 확장되며, 작업이 완료되면 해체됩니다.

잘 구성된 샌드박스는 또한 적절한 기본값(언어 런타임, Git CLI, 테스트 러너, 브라우저)을 함께 제공합니다. 이를 통해 에이전트가 자체 검증을 할 수 있습니다. 코드를 작성하고, 테스트를 실행하고, 로그를 검사하고, 실패를 수정합니다. 하네스가 환경을 구축하고, 모델은 이를 사용합니다.

### 5. 컨텍스트 관리 (Context Management)

**컨텍스트 고갈 (Context rot)**은 가장 교묘한 운영 환경 실패 중 하나입니다.

에이전트는 40분 동안 잘 작동했습니다. 이제는 자체 시스템 프롬프트(system prompt)를 무시하고 있습니다. 아무것도 충돌하지 않았고, 오류도 발생하지 않았습니다. 컨텍스트 창이 가득 차고 중요한 지시사항이 중간에 묻혀버려서, 모델은 점차적으로 그것들을 고려하지 않게 되었습니다.

하네스가 모델이 보는 것을 제어합니다. 모델은 그렇지 않습니다.

운영 환경에서 실제로 작동하는 세 가지 패턴:

*   **압축 (Compaction)**은 컨텍스트 창이 채워질 때 오래된 대화 기록을 단순히 삭제하는 대신 요약하여 처리합니다. 핵심 제약 사항은 원래의 작업 정의(task definition)나 시스템 프롬프트는 절대 압축하지 않는 것입니다. 다른 모든 것은 협상 가능합니다.
*   **도구 출력 잘라내기 (Tool output truncation)**는 대규모 도구 결과가 컨텍스트를 범람시키는 것을 방지합니다. fetch 호출에서 50페이지짜리 문서가 그대로 반환되면 전체 예산을 소모하고 유용한 모든 것을 가려버릴 것입니다. 하네스는 처음과 마지막 N개의 토큰을 유지하고, 전체 출력은 파일 시스템에 저장하며, 모델이 더 많은 정보가 필요할 경우 포인터를 제공합니다.
*   **점진적 공개를 통한 스킬 로딩 (Skills via progressive disclosure)**은 시작(startup) 문제를 해결합니다. 세션 시작 시 모든 도구 설명을 로드하는 것은 에이전트가 아무것도 하기 전에 컨텍스트를 비대하게 만듭니다. 스킬은 모델이 해당 기능을 필요로 한다고 판단할 때 온디맨드(on demand)로 핵심 정보(front-matter)를 로드합니다. 50개의 스킬을 지연 로딩(lazily loaded)하는 에이전트는 10개의 도구를 미리 로드하는 에이전트보다 성능이 더 좋은 경우가 많습니다. 실제 작업이 시작될 때 컨텍스트 부담이 적기 때문입니다.

운영 규칙: 시스템 프롬프트와 작업 정의는 항상 눈에 띄게 유지되어야 합니다. 이것들을 건드리기 전에 기록을 압축해야 합니다.

### 6. 계획 (Planning)

계획이 없는 모델은 목표로 향하는 일관된 경로의 일부인지 여부와 관계없이 가장 명확한 다음 단계만 수행합니다.

간단한 작업에는 문제가 없지만, 복잡한 다단계 작업에서는 비일관성을 초래합니다. 단계가 순서 없이 진행되거나, 반복되거나, 즉시 명확하지 않아 건너뛰게 됩니다. 에이전트는 매우 유능할 수 있지만, 실행할 구조가 주어지지 않았기 때문에 작업을 실패할 수 있습니다.

**계획 파일 패턴 (plan file pattern)**은 운영 환경에서 실제로 작동하는 가장 간단한 해결책입니다.

```
task: Migrate database schema from v1 to v2
steps:
  - Backup current schema         [ ]
  - Generate migration script     [ ]
  - Run migration on staging      [x]
  - Verify data integrity         [ ]
  - Run migration on production   [ ]
  - Update documentation          [ ]
current_step: 4
```

하네스는 매 루프 시작 시 이를 컨텍스트에 주입합니다. 에이전트는 단계를 완료할 때마다 체크합니다. 세션이 종료되어도 계획은 유지됩니다. 에이전트가 재개되면 정확히 어디에 있었는지 알 수 있습니다.

**자체 검증 (Self-verification)**은 루프를 닫습니다. 각 단계를 완료한 후 에이전트는 다음 단계로 넘어가기 전에 결과를 검증합니다. 하네스는 테스트 스위트(test suite)를 실행하고 실패를 피드백하여 이를 강제할 수 있습니다. 마이그레이션 스크립트를 작성하고 즉시 스테이징 환경(staging environment)에서 검증하는 에이전트는 작성만 하고 가정한 에이전트보다 훨씬 더 안정적입니다.

**랄프 루프 (Ralph Loop)**는 기억할 가치가 있는 이름입니다. 에이전트가 긴 작업을 수행하면서 컨텍스트 창을 모두 사용했지만 목표를 완료하지 못한 경우, 랄프 루프는 후크(hook)를 통해 이 종료를 가로채고, 새로운 컨텍스트 창에 원래 목표를 주입하여 계속 진행하도록 강제합니다. 파일 시스템이 이를 가능하게 합니다. 각 새로운 컨텍스트는 이전 반복의 상태를 읽습니다. 이것이 여러 컨텍스트 창에 걸쳐 진정한 장기 자율성(long-horizon autonomy)이 작동하는 방식입니다.

### 7. 오류 처리 (Error Handling)

실제 세상은 협조적이지 않습니다. 도구가 실패하고, API는 처리율 제한(rate-limit)에 걸리며, 파일이 누락되기도 합니다. 모델은 때때로 파싱할 수 없는 출력을 반환합니다.

명시적인 오류 처리(error handling)가 없으면, 이러한 상황 중 하나라도 발생했을 때 에이전트는 두 가지 좋지 않은 선택을 하게 됩니다. 충돌하거나, 오류가 발생하지 않은 것처럼 조용히 환각(hallucinate)을 일으키는 것입니다. 둘 다 운영 환경에서의 실패입니다.

*   **도구 실패:**
    *   재시도 가능? (타임아웃, 처리율 제한) → 지수 백오프(exponential backoff)
    *   데이터 오류? → 대체 접근 방식 시도
    *   권한 오류? → 사람에게 에스컬레이션(escalate)
*   **모델 출력 형식 불량:**
    *   명시적인 형식 상기 메시지와 함께 재시도
    *   세 번 실패 → 구조화된 출력 강제(structured output enforcement)로 폴백(fall back)
*   **에이전트 루프:**
    *   단계 카운터 작동 → 강제 중지
    *   반복되는 동일한 도구 호출 감지 → 중단 및 재지정
*   **신뢰도 낮음:**
    *   비동기적(async) 인간 검토를 위해 플래그 지정
    *   사용자가 기다리는 동안 작업을 차단하지 않음

**에스컬레이션 경로 (escalation path)**는 대부분의 하네스가 갖지 못한 가장 중요한 기능입니다. 언제 멈추고 도움을 요청해야 하는지 아는 에이전트는 항상 작업을 완료하려고만 하는 에이전트보다 운영 환경에서 더 유용합니다. 배포하기 전에 명시적인 신뢰도 임계값(confidence thresholds)을 구축해야 합니다.

모든 도구 호출에는 정의된 실패 동작이 있어야 합니다. 단순히 "오류를 우아하게 처리"하는 것이 아니라, 구체적으로: 이것이 빈 값을 반환하면 X를 수행합니다. 오류가 발생하면 Y를 수행합니다. 타임아웃되면 Z를 수행합니다.

## 실제 사례: 운영 에이전트 내부에서 일어나는 일

사용자 입력: "지난달 EU AI 규제에 대한 뉴스 보도의 주요 주장들을 요약해 주세요."

**1단계: 계획 생성**
*   EU 규제 뉴스 검색 (지난 30일)
*   상위 5개 결과 읽기
*   주장 클러스터(argument clusters) 추출
*   구조화된 요약으로 통합

**2단계: 상태 확인**
기존 진행 상황 없음. 단계 카운터 0으로 초기화.

**3단계: 검색 도구 호출**
8개 기사 반환. 하네스가 각 기사를 500 토큰으로 잘라 컨텍스트에 추가.

**4단계: 상위 5개 결과에 대해 `fetch_url()` 호출**
전체 텍스트는 파일 시스템에 저장. 에이전트는 요약 및 파일 포인터를 얻음.

**5단계: 컨텍스트 확인**
60% 용량. 아직 압축 필요 없음.

**6단계: 통합 (Synthesis)**
에이전트가 3개의 주요 주장 클러스터를 찾아 컨텍스트에 작성.

**7단계: 검증 (Verification)**
에이전트가 기사 날짜 확인. 두 기사가 45일 전 것임을 감지. 플래그 지정.
더 엄격한 날짜 필터로 다시 검색. 새로운 기사 2개 추가.

**8단계: 최종 출력**
인용이 포함된 구조화된 요약. 단계 카운터: 9. `MAX_STEPS`: 20.

**9단계: 상태 업데이트**
계획 파일: 모든 단계 완료.
향후 세션을 위해 주요 발견 사항을 메모리 저장소에 기록.

모델은 요약을 작성했습니다. 하네스는 상태를 추적하고, 컨텍스트를 관리하며, 검증을 강제하고, 단계 제한을 적용하고, 메모리에 기록했습니다. 9단계, 인간 개입 없음, 정확한 출력.

이것이 바로 하네스가 실제로 하는 일입니다. 화려하지는 않습니다. 하지만 한 번만 작동하는 도구와 안정적으로 작동하는 도구의 차이입니다.

## 당신을 곤경에 빠뜨릴 수 있는 엣지 케이스 (Edge Cases)

*   **도구가 있음에도 불구하고 환각 (Hallucination despite tools).** 에이전트가 검색 도구를 가지고 있음에도 불구하고 훈련 데이터를 사용합니다. 이는 도구 설명이 언제 도구 호출이 필요한지(사용 가능한 때뿐만 아니라) 명확하게 밝히지 않을 때 발생합니다. 해결책: 답변하기 전에 어떤 질문에 도구 호출이 필요한지 명시적으로 지시합니다.
*   **무한 루프 (Infinite loops).** 모델이 빈 결과를 얻은 후 사소한 변형으로 동일한 도구를 재시도합니다. 빈 결과를 "다시 시도"로 해석하고 "이 접근 방식은 틀렸다"고 해석하지 않습니다. 해결책: 반복되는 동일한 도구 호출을 감지하고, 재지정 프롬프트(redirect prompt)로 중단합니다.
*   **컨텍스트 오버플로 (Context overflow).** 에이전트가 처음 30분 동안은 훌륭했습니다. 이제 자체 지시사항을 무시합니다. 컨텍스트가 서서히 가득 차서 시스템 프롬프트가 묻히고, 성능이 눈에 띄지 않게 저하되었습니다. 해결책: 압축 전략과 작업 정의가 항상 컨텍스트 시작과 끝에 나타나야 한다는 엄격한 규칙을 만듭니다.
*   **도구 오용 (Tool misuse).** 읽기 전에 쓰기, 아카이브(archive) 대신 삭제. 이런 일은 도구 설명이 전제 조건에 대해 모호할 때 발생합니다. 해결책: 모든 설명은 도구를 언제 사용해야 하는지뿐만 아니라 언제 사용하지 말아야 하는지도 명시해야 합니다.
*   **지연 시간 폭증 (Latency explosion).** 일련의 합리적인 도구 호출이 45초의 응답 시간을 발생시킵니다. 해결책: 독립적인 도구 호출은 순차적으로가 아니라 동시에 실행됩니다. 모델을 건드리기 전에 어떤 하네스 선택이 지연 시간을 추가하는지 측정합니다.

## 대부분의 엔지니어들이 모델-하네스 결합에 대해 깨닫지 못하는 것

Claude Code와 같은 최신 코딩 에이전트는 모델과 하네스가 함께 실행되는 방식으로 사후 훈련(post-trained)됩니다. 모델은 파일 시스템 작업, Bash 실행 및 계획 수립을 학습하는데, 이는 이러한 행동을 보상하는 하네스 내부에서 실행되는 동안 훈련되었기 때문입니다.

이로 인해 흥미로운 부작용이 발생합니다.

도구 로직을 변경하면 새로운 로직이 동일하더라도 모델 성능이 저하되는 경우가 많습니다. 특정 패치(patching) 형식으로 훈련된 모델은 형식을 변경하면 성능이 저하됩니다. 두 형식이 논리적으로 동일하더라도 마찬가지입니다. 루프 내에서 하네스와 함께 훈련하는 것은 해당 하네스 설계에 대한 일종의 과적합(overfitting)을 생성합니다.

실질적인 결과: 즉시 사용 가능한(out-of-the-box) 하네스가 항상 작업에 최적인 것은 아닙니다. Terminal Bench 2.0 리더보드에서 Claude Code 내부의 Opus 4.6은 커스텀 튜닝된 하네스 내부의 Opus 4.6보다 훨씬 낮은 점수를 기록했습니다. 동일한 모델, 다른 하네스, 측정 가능한 다른 순위입니다.

대부분의 팀은 하네스 최적화에 전혀 손대지 않았습니다. 바로 그곳에 실제 성능 개선의 기회가 숨어 있습니다.

## 에이전트를 사용하지 말아야 할 때

에이전트는 사람들이 인정하고 싶어 하는 것보다 더 자주 잘못된 도구입니다.

*   동일한 입력이 항상 동일한 단계를 통해 동일한 출력을 생성하는 경우 **결정론적 파이프라인 (deterministic pipeline)**을 사용하십시오. 이를 하드 코딩하십시오. 에이전트가 하는 어떤 작업보다 빠르고, 저렴하며, 더 안정적입니다.
*   실수가 프로덕션 데이터 삭제 또는 잘못된 사람에게 이메일 발송을 의미하는 경우 **명시적인 인간 개입 지점 (explicit human gates)**을 사용하십시오. 에이전트의 추천과 실행을 분리하십시오. 에이전트는 실수를 합니다. 그 실수가 되돌릴 수 없는 것이 아니도록 확인하십시오.
*   입력이 구조화되어 있고 처리가 규칙 기반인 경우 에이전트를 완전히 건너뛰십시오. 양식 제출 워크플로에 복잡성을 추가하는 에이전트는 개선이 아니라 과잉 엔지니어링(overengineering)입니다.

> "에이전트는 워크플로의 업그레이드가 아닙니다. 다른 종류의 문제에 대한 다른 도구입니다. 당신이 어떤 문제를 가지고 있는지 아십시오."

과잉 엔지니어링을 하고 있다는 가장 분명한 신호는 워크플로의 모든 단계에 하나의 정확한 작업이 있고, 경로가 완전히 정의되어 있으며, 에이전트를 원하는 주된 이유가 에이전트가 인상적으로 보이기 때문이라는 것입니다. 물론 인상적입니다. 하지만 결정론적 파이프라인에는 과합니다.

## 처음부터 구축하는 경우 어디서부터 시작해야 하는가

이 순서대로 추가하십시오. 각 계층은 해당 단계에서 가장 흔한 실패를 해결합니다.

1.  **단계 제한(step limit)이 있는 제어 루프.** 어떤 도구보다 먼저. `MAX_STEPS = 10`은 하룻밤 사이에 발생하는 청구 사고를 방지합니다.
2.  **상태 파일 (State file).** 무엇이 일어났고 다음은 무엇인지 추적하는 JSON 파일. 모든 루프 시작 시 이를 읽으십시오.
3.  **도구 세트 (Tool set).** 세 개에서 다섯 개 정도, 잘 설명된 것. 특정 공백을 발견할 때만 더 추가하십시오.
4.  **오류 처리 (Error handling).** 배포하기 전에 모든 도구에 대한 실패 동작을 정의하십시오.
5.  **컨텍스트 압축 (Context compaction).** 장기 세션에서 성능 저하가 보일 때 추가하십시오. 그전에는 필요 없습니다.
6.  **메모리 (Memory).** 사용자가 에이전트가 알아야 할 것을 잊어버린다고 느낄 때 추가하십시오.
7.  **계획 (Planning).** 작업이 여러 세션에 걸쳐 있거나 단일 컨텍스트 창을 초과할 때 추가하십시오.

이 순서는 임의적이지 않습니다. 2단계 없이 6단계로 건너뛰면 잘못된 것을 디버깅하게 될 것입니다.

## 마지막 생각

잘 구축된 에이전트의 가장 좋은 점은 모든 것이 제대로 작동할 때 하는 일이 아닙니다.

무언가 잘못되었을 때 하는 일입니다.

모델이 개선됨에 따라 오늘날 하네스에 포함된 일부 기능은 모델 내부에 흡수될 것입니다. 모델은 더 많은 프롬프트 지원 없이도 계획 및 자체 검증을 더 잘 수행할 것입니다. 일부 하네스 복잡성은 실제로 불필요해질 것입니다.

그러나 모델 지능을 둘러싼 엔지니어링, 올바른 도구, 견고한 상태, 컨텍스트 관리, 검증 루프는 모델의 능력과 관계없이 어떤 모델이든 더 효과적으로 만듭니다. 이것은 결함을 패치하는 것이 아닙니다. 이것은 시스템 설계입니다.

모델은 몇 달마다 개선됩니다. 하네스는 당신이 구축해야 할 몫입니다.
