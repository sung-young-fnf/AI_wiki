# AI 코딩 효율성을 극대화하는 4가지 핵심 기법

이 글에서는 AI 코딩 도구를 효과적으로 활용하는 방법에 대해 논의합니다.

이전 글에서는 AI 에이전트로 효과적인 코딩을 위해 활용하는 세 가지 중요한 기법을 다뤘습니다. 이번 글에서는 제가 매일 사용하는 네 가지 추가 기법에 대해 이어서 설명합니다.

오늘날 효율적인 프로그래머가 되려면 AI 도구를 적극적으로 활용해야 한다고 생각합니다. AI 에이전트를 사용하지 않고 코딩한다면 뒤처지게 될 것입니다. 또한, 에이전트는 코딩 외에도 훨씬 더 많은 용도로 사용될 수 있습니다.

*   Linear 이슈를 읽고 생성합니다.
*   관심 있는 주제에 대해 심층적인 조사를 수행합니다.
*   운영 코드의 로그 메시지를 검토합니다.

이 모든 것은 프로그래머가 정기적으로 수행해야 하는 중요한 작업입니다.

저는 가능한 한 효율적으로 작업하기 위해 AI 에이전트의 적극적인 사용을 지지합니다. 이 글에서는 저를 더욱 효율적인 프로그래머로 만들어준다고 믿는 네 가지 기법을 개괄적으로 다룰 것입니다.

*   빠른 에이전트 프롬프트 작성을 위한 MacWhisper
*   Claude를 활용한 코드 리뷰
*   병렬 에이전트 활용
*   에이전트를 이용한 GitHub 상호작용

여러분의 프로그래밍 워크플로에서 중요한 기법이 있다면 언제든지 알려주세요. 기꺼이 듣고 싶습니다.

## AI 에이전트로 코딩해야 하는 이유

저는 이전에 AI 에이전트를 사용한 코딩이 프로그래머로서 저를 훨씬 더 효과적으로 만든다고 설명했습니다. AI 사용을 통해 프로그래밍 생산성을 여러 배로 늘렸으며, 이전보다 훨씬 더 많은 작업을 수행할 수 있게 되었습니다.

AI 에이전트에 대한 일반적인 반론은 코드를 운영 환경에 배포하기 전에 코드를 이해해야 한다는 것입니다. 종단 간(end-to-end) 테스트를 수행하기 어려운 중요 시스템을 다루는 경우, 어느 정도 이 의견에 동의합니다.

그러나 대부분의 웹사이트와 애플리케이션은 그렇지 않습니다. 첫째, 그들은 그렇게 중요하지 않으며, 둘째, 프로그래머로서 수행하는 대부분의 작업은 검증 가능합니다. 이는 기능을 직접 사용해봄으로써 작동 여부를 간단히 테스트할 수 있다는 의미입니다.

따라서 저는 AI 에이전트의 더 많은 사용을 주장하며, 모든 프로그래밍 관련 작업에 이를 활용할 것을 권장합니다. 예를 들어:

*   Linear 이슈 생성
*   Linear 이슈 링크를 통해 버그 수정
*   새로운 기능 기획 및 개발

## 코딩 효율성을 위한 4가지 기법

이 섹션에서는 제가 AI 기반 프로그래밍 워크플로에서 사용하는 네 가지 기법을 다룰 것입니다. 이들은 제가 매일 코딩할 때 실제로 사용하는 구체적인 기법들입니다.

### MacWhisper

MacWhisper는 Mac에서 사용할 수 있는 훌륭한 받아쓰기(transcription) 도구입니다. 간단히 말해, MacWhisper를 사용하면 버튼을 누르고 컴퓨터에 대고 말하면, 텍스트가 자동으로 받아쓰기되어 마우스 커서가 있는 곳에 붙여넣어집니다.

이는 제 프로그래밍 워크플로의 상당 부분이 순수한 코드에서 자연어로 전환되었기 때문에 유용합니다. 코딩에는 콜론, 괄호, 탭과 같은 많은 특수 문자가 필요하며 이는 키보드로 입력하는 것이 더 빠르기 때문에, 코딩에 받아쓰기 도구를 사용하는 것은 당연히 어려울 것입니다.

AI 에이전트를 사용하면 코딩 언어 대신 자연어로 점점 더 많은 작업이 수행됩니다.

따라서 Cursor 에이전트에 프롬프트를 입력할 때마다, 저는 보통 버튼을 누르고 에이전트에 말하고 싶은 내용을 소리 내어 말합니다. 예를 들어, 다음과 같이 물을 수 있습니다:

> 이 문서 ID에 대한 로그를 확인해줘, 올바르게 처리되었니 <문서 ID>

이 예시에서는 문장을 소리 내어 말한 후에 문서 ID를 붙여넣습니다.

제가 MacWhisper를 사용하는 이유는 단순히 제가 타이핑하는 것보다 말하는 속도가 더 빠르기 때문입니다. 평균 말하기 속도는 분당 약 150단어인 반면, 대부분의 사람은 최대 속도로 분당 100단어를 타이핑하지 못합니다. 또한, 생각까지 해야 할 때는 최대 속도로 타이핑하기 어렵습니다.

### Claude 코드 리뷰

이 단계는 두 부분으로 나뉩니다.

*   기능을 구현한 후, Cursor에게 코드가 운영 환경에 배포할 준비가 되었는지 묻고, Cursor가 만족할 때만 푸시(push)합니다.
*   PR(Pull Request)을 생성할 때마다, PR 설명과 병합할 브랜치(branch)에 대한 Git diff 파일 외에 다른 맥락 없이 Claude가 코드 리뷰를 수행하도록 합니다.

이 방식은 매우 효과적입니다. Cursor에게 코드가 운영 준비가 되었는지 물으면 Cursor는 변경 사항을 검토하고 의도대로 작동하지 않을 수 있는 작은 문제들을 수정합니다.

또한, 구현 방식에 대한 맥락 없이 완전히 별개의 LLM(대규모 언어 모델)이 코드를 검토하게 하는 것은 매우 유용합니다. 이는 PR에서 코드를 구현할 때 저(또는 Cursor)가 생각하지 못했던 다른 오류들을 종종 발견하게 합니다. 이는 또한 운영 환경에서 발생하는 버그의 양을 크게 줄여주며, CICD(지속적 통합/지속적 배포) 파이프라인에 추가할 수 있는 비교적 저렴한 방법입니다.

### 병렬 에이전트 (Fire and Forget)

또 다른 중요한 기법은 병렬 에이전트를 사용하는 것입니다. 어떤 에이전트가 작업을 수행하느라 막힐 때마다, 저는 항상 새로운 에이전트를 시작합니다. 이는 다른 기능을 구현하는 코딩 에이전트일 수도 있고, 관심 있는 주제를 심층적으로 조사하는 Gemini 딥 리서치일 수도 있습니다. 핵심은 에이전트가 작업을 마칠 때까지 아무것도 하지 않고 단순히 기다리지 않는다는 것입니다.

병렬 에이전트를 실행할 때, 컨텍스트 스위칭(context switching)으로 어려움을 겪을 수 있습니다. 컨텍스트를 자주 전환하는 것은 뇌에 많은 부담을 주므로, 최소화하고 싶은 부분입니다.

따라서 저는 완전히 막힐 때까지 한 가지 작업을 수행하도록 항상 노력합니다. 컨텍스트를 전환하는 횟수를 최소화하고, 코딩 에이전트가 구현을 완료하기까지 시간이 걸릴 것이라고 확신할 때만 병렬 작업을 시작합니다.

여기서 또 다른 중요한 점은 코딩 에이전트가 장시간 실행될 수 있도록 충분한 권한을 부여해야 한다는 것입니다. Cursor가 권한을 요청하며 계속 방해하면 병렬 워크플로는 제대로 작동하지 않습니다.

### 에이전트를 이용한 커밋 및 PR

마지막으로, 제가 직접 명령어를 작성하는 대신 코딩 에이전트를 사용하여 항상 GitHub와 상호작용하는 방법을 강조하고 싶습니다. 이렇게 하는 이유는 단순히 더 빠르고, 에이전트가 사전 커밋 훅(precommit hooks) 실행, 커밋(commits), 푸시(pushes), 풀 리퀘스트(pull requests)를 수행하는 동안 저는 다른 작업을 할 수 있기 때문입니다.

커밋 메시지, 풀 리퀘스트 제목 및 설명을 작성하는 데는 예상외로 많은 시간이 소요됩니다. 특히 번역 추가나 UI 버튼 이동과 같은 빠른 작업을 수행할 때 더욱 그렇습니다. 따라서 저는 항상 Claude를 활용하여 커밋 메시지, PR 제목 및 설명을 작성합니다.

이는 시간을 절약해줄 뿐만 아니라, Claude가 저를 위해 이러한 메시지를 더 잘 작성한다고 생각합니다. 예를 들어 풀 리퀘스트의 경우, 사람이 모든 변경 사항을 기억하고 깔끔하게 요약하는 것이 어려운 경우가 많습니다. Claude가 Git diff를 보고 모든 변경 사항을 요약하여 제공하는 것이 훨씬 쉽습니다.

따라서 저는 Cursor에게 저를 대신하여 GitHub와 상호작용할 권한을 부여했습니다. 풀(pull), 리베이스(rebase), 커밋(commit), 수정(amend), 푸시(push), PR 생성과 같은 모든 GitHub 작업을 직접 수행하는 대신, Cursor에게 작업을 지시하기만 합니다. 따라서 저는 "지시하고 잊어버리기(fire and forget)"가 가능합니다. 새로운 기능을 구현한 후의 제 워크플로는 Cursor에게 다음 프롬프트를 제공하는 것입니다.

```
모든 사전 커밋 검사(black, mypy, pytest)를 실행하고, 커밋 및 푸시해줘.
그리고 이 브랜치(<linear branch name>)에 PR을 생성하고, PR 링크를 알려줘.
```

이는 직접 GitHub 명령어를 작성하는 것보다 훨씬 빠릅니다. GitHub UI에서 풀 리퀘스트를 생성할 때 매우 시간이 많이 걸렸던 점을 고려하면, 직접 풀 리퀘스트를 작성하지 않아도 되는 것이 아마도 가장 중요한 부분일 것입니다. 이제 에이전트가 제공하는 링크를 클릭하기만 하면 PR이 준비됩니다. 그런 다음 Claude가 제공한 코드 리뷰를 확인하고 잠재적인 문제들을 수정합니다.

## 결론

이 글에서는 제가 매일 코딩할 때 사용하는 네 가지 구체적인 기법에 대해 논의했습니다. 받아쓰기(transcription)를 위한 MacWhisper, Claude 코드 리뷰, 병렬 에이전트, 그리고 에이전트를 이용한 GitHub 상호작용에 대해 다뤘습니다. 이 기법들을 종합하면 매일 최소 1시간을 절약할 수 있다고 추정하며, 이는 상당한 시간입니다. 이 시간을 확보함으로써 프로젝트 진행 과정에서 훨씬 더 많은 작업을 완료할 수 있습니다. AI 에이전트를 효과적으로 사용하는 것은 매우 중요한 기술이며, 시간을 들여 능숙해져야 할 주제라고 생각합니다.
