Do AGENTS.md/CLAUDE.md 파일이 코딩 에이전트에 도움이 되는가? 새로운 논문이 이 가정에 도전하다

지난 1년간, 코딩 에이전트(coding agent)를 사용할 때 AGENTS.md 또는 CLAUDE.md 파일을 코딩 저장소(repository)에 추가하는 것이 조용히 표준 관행이 되었습니다. 에이전트 공급업체들이 이를 권장하며, Anthropic, OpenAI, Qwen 모두 이를 장려합니다. 현재 작성 시점 기준으로 6만 개 이상의 공개 GitHub 저장소에 이미 해당 파일이 포함되어 있습니다.

그 전제는 직관적입니다. 코딩 에이전트에게 도구(tooling), 구조(structure), 스타일 가이드(style guides) 등 저장소의 지도를 제공하면, 에이전트가 더 효율적으로 작동할 것이라는 생각입니다. 일리가 있어 보이죠?

ETH 취리히(ETH Zurich)와 LogicStar.ai의 새로운 논문인 "Evaluating AGENTS.md: Are Repository-Level Context Files Helpful for Coding Agents?"은 이러한 가정을 엄격하게 테스트한 첫 번째 연구입니다. 그리고 그 결과는 대부분의 사람이 예상했던 것과 달랐습니다.

목차:
1.  실험: 두 가지 벤치마크, 세 가지 조건
2.  결과: 더 많은 컨텍스트, 더 나쁜 결과
3.  에이전트는 지시를 따릅니다. 그것이 문제입니다.
4.  "아하" 실험: 모든 문서를 제거하면 어떻게 될까?
5.  더 강력한 모델이 도움이 될까?
6.  개발자에게 이것은 무엇을 의미하며, AGENTS.md를 계속 작성해야 하는가?

---

### 1. 실험: 두 가지 벤치마크, 세 가지 조건

저자들은 두 가지 벤치마크(benchmark)와 세 가지 조건에서 네 가지 주요 코딩 에이전트(Claude Code (Sonnet-4.5), Codex (GPT-5.2 및 GPT-5.1 Mini), Qwen Code (Qwen3–30B))를 평가했습니다.

*   **컨텍스트 파일 없음(No context file):** 에이전트가 저장소(repo)와 작업 설명만 가지고 작동합니다.
*   **LLM 생성 컨텍스트 파일(LLM-generated context file):** 에이전트 공급업체에서 권장하는 `/init` 명령을 사용하여 자동 생성됩니다.
*   **사람이 작성한 컨텍스트 파일(Human-written context file):** 저장소 개발자가 직접 작성합니다.

이를 적절하게 수행하기 위해, 연구팀은 AGENTBENCH라는 새로운 벤치마크를 도입했습니다. AGENTBENCH는 개발자가 커밋(commit)한 컨텍스트 파일이 모두 포함된 12개 저장소의 실제 GitHub 이슈(issue) 138개로 구성됩니다. 이는 SWE-Bench와 같은 기존 벤치마크가 컨텍스트 파일 시대 이전에 만들어진 인기 저장소를 기반으로 구축되었다는 점에서 중요합니다.

### 2. 결과: 더 많은 컨텍스트, 더 나쁜 결과

에이전트와 설정별 문제 해결률 (왼쪽은 SWE-Bench Lite, 오른쪽은 AGENTBENCH).

두 벤치마크 모두에서, LLM이 생성한 컨텍스트 파일(빨간색/분홍색)은 컨텍스트 파일이 없는 경우(회색)보다 지속적으로 성능이 낮거나 같았습니다. 사람이 작성한 파일(짙은 녹색)은 약간 더 나은 성능을 보였지만, 그 차이는 미미했습니다.

수치는 명확한 결과를 보여줍니다.

*   LLM이 생성한 컨텍스트 파일은 평균적으로 작업 성공률을 약 2~3% 감소시켰습니다.
*   사람이 작성한 컨텍스트 파일은 작업 성공률을 약 4% 향상시켰습니다. 이는 미미한(marginal) 개선입니다.
*   두 경우 모두 추론 비용(inference cost)을 20% 이상 증가시켰고, 에이전트가 작업을 완료하는 데 더 많은 단계를 거치게 했습니다.

결론적으로, 에이전트는 사람이 작성한 컨텍스트 파일의 경우에도 약간의 개선만 있을 뿐, 더 많은 작업을 하고, 더 많은 비용이 들며, 더 적은 작업을 해결하고 있었습니다. 이는 문제입니다.

### 3. 에이전트는 지시를 따릅니다. 그것이 문제입니다.

여기서 흥미로운 점이 드러납니다. 문제는 에이전트가 컨텍스트 파일을 무시한다는 것이 아닙니다. 오히려 그 반대입니다.

LLM이 생성한 (밝은 녹색) 또는 개발자가 제공한 (짙은 녹색) 컨텍스트 파일을 포함할 때의 평균 도구 사용량 증가 (컨텍스트 파일이 없는 경우의 평균 도구 사용량과 비교). 도구 이름의 경우, CODEX 및 QWEN CODE 도구를 CLAUDE CODE의 해당 도구에 매핑했습니다.

모든 에이전트에서 컨텍스트 파일의 존재는 테스트 실행(test runs), 파일 읽기(file reads), `grep` 검색 및 저장소(repo) 특정 도구 사용을 증가시켰습니다. 에이전트는 거의 흠잡을 데 없이 지시를 따릅니다.

컨텍스트 파일에 `UV`가 언급되면 에이전트는 거의 전적으로 `UV`를 사용합니다. 저장소 특정 도구가 언급되면 해당 도구는 2.5배 더 자주 호출됩니다. 지시는 따르지만, 이러한 지시를 따르는 것이 결과의 신뢰할 만한 개선 없이 더 광범위한 탐색, 더 심층적인 테스트 및 더 많은 추론(reasoning)을 유발합니다.

컨텍스트 파일 유무에 따른 추론 토큰(reasoning tokens) 사용량.

LLM이 생성한 컨텍스트 파일은 GPT-5.2의 추론 토큰을 최대 22%까지 증가시켰습니다. 에이전트가 더 열심히 일하지만, 더 스마트하게 일하지는 않는 것입니다.

AI 엔지니어로서 이 부분은 크게 와닿았습니다. 우리는 종종 더 많은 컨텍스트가 더 나은 성능으로 이어진다고 가정합니다. 그러나 자율 에이전트(autonomous agents)의 경우, 불필요한 요구 사항은 인지 부하(cognitive load)를 증가시킵니다. 에이전트 또한 사람처럼 과도한 지시를 받을 수 있습니다.

### 4. "아하" 실험: 모든 문서를 제거하면 어떻게 될까?

이 논문에서 가장 흥미로운 부분은 간단한 제거(ablation) 실험입니다. 저장소에서 README, 문서 폴더(docs folders), 마크다운 파일(markdown files) 등 모든 문서를 제거하고 컨텍스트 파일만 남겨두면 어떻게 될까요?

다른 모든 문서가 제거되었을 때의 성능.

다른 문서가 전혀 없을 경우, LLM이 생성한 컨텍스트 파일은 도움이 될 뿐만 아니라, 사람이 작성한 파일보다 더 나은 성능을 보였습니다.

이것은 모든 것을 다시 생각하게 만듭니다. 컨텍스트 파일 자체가 나쁜 것이 아닙니다. 잘 문서화된 저장소에서는 중복될(redundant) 뿐입니다. 그리고 중복성은 에이전트에게 비용이 많이 드는 것으로 밝혀졌습니다. 컨텍스트 파일이 도움이 되었다는 일화적인 보고들은 어디에서 온 것일까요? 아마도 에이전트가 가질 수 있는 유일한 정보원이 컨텍스트 파일이었던, 문서화가 덜 되어 있거나 틈새(niche) 저장소에서 비롯된 것일 가능성이 높습니다.

### 5. 더 강력한 모델이 도움이 될까?

이 논문에서는 두 가지 자연스러운 후속 질문에 답합니다.

*   **더 강력한 모델이 더 나은 컨텍스트 파일을 생성할 수 있는가?** 아닙니다. GPT-5.2가 생성한 컨텍스트 파일은 SWE-Bench에서는 성능을 향상시켰지만, AGENTBENCH에서는 저하시켰습니다. 더 나은 LLM이 더 나은 컨텍스트 파일을 의미하지는 않습니다.
*   **파일을 생성하는 데 사용된 프롬프트(prompt)가 중요한가?** 일관적이지 않습니다. Claude Code와 Codex의 생성 프롬프트 사이를 전환해도 에이전트 전반에 걸쳐 신뢰할 수 있는 개선은 나타나지 않았습니다.

### 6. 개발자에게 이것은 무엇을 의미하며, AGENTS.md를 계속 작성해야 하는가?

답변은 일률적으로 "컨텍스트 파일을 사용하지 마라"가 아닙니다. 이는 저장소 문서화(documentation) 상태에 따라 달라집니다.

*   **잘 문서화된 저장소(well-documented repo)인가?** LLM이 생성한 컨텍스트 파일은 완전히 건너뛰세요. 대부분 README와 문서에 이미 있는 내용을 반복할 것이며, 이러한 중복성(redundancy)은 결과 개선 없이 비용만 증가시킵니다. 컨텍스트 파일이 꼭 필요하다면, `pip` 대신 `uv`를 사용하거나 `pytest -x`로 테스트를 실행하는 등 에이전트가 스스로 추론할 수 없는 내용에만 초점을 맞춰 짧게 수동으로 작성하세요.
*   **문서가 전혀 없는가(zero documentation)?** 그렇다면 생성하세요. 이 논문은 LLM이 생성한 컨텍스트 파일이 실제로 제 역할을 하고 성능이 향상되며, 심지어 사람이 작성한 파일보다도 뛰어난 성능을 보이는 시나리오가 바로 이 경우임을 보여줍니다. 에이전트가 활용할 다른 정보가 없기 때문에 컨텍스트 파일이 중복이 아닌 진정으로 유용해집니다.
*   **부분적으로 문서화된 저장소(partially documented repo) (가장 흔한 실제 사례)인가?** 이 논문은 명확한 답을 제시하지 않지만, 논리는 분명합니다. 파일을 생성한 다음, 다른 곳에 이미 설명된 내용은 수동으로 제거하세요. 남은, 명확하지 않은 내용이 실제로 도움이 되는 부분입니다.

모든 경우에 해당되는 한 가지 사실은 AGENT.md 또는 CLAUDE.md에 디렉토리 개요(directory overviews)를 피해야 한다는 것입니다. 이 논문은 이를 직접 테스트했습니다. 상세한 디렉토리 맵을 가진 에이전트가 관련 파일을 더 빨리 찾지 못했습니다. 그들은 단지 더 많은 작업을 수행했을 뿐입니다. 더 많은 읽기, 더 많은 검색, 더 많은 추론을 했지만, 정확도에서는 의미 있는 이득이 없었습니다. 개요는 작성하는 데 도움이 될 것 같지만, 더 나은 에이전트 행동으로 이어지지는 않습니다.

결론은 다음과 같습니다. 컨텍스트 파일은 문서화가 부족한 부분을 보완하는 메커니즘이지, 잘 된 문서화 위에 성능을 향상시키는 부스터가 아닙니다. 이에 맞춰 파일을 다루세요.

이 논문은 AI 엔지니어링에서 우리의 직관이 종종 증거보다 앞선다는 것을 상기시켜주는 좋은 사례입니다. 우리는 컨텍스트 파일이 이론적으로 합리적이고, 모두가 사용하며, 공급업체가 권장했기 때문에 이를 채택했습니다. 그러나 "합리적이다"가 "작동한다"와 같은 의미는 아닙니다.

진정한 과제는 에이전트에게 더 많은 컨텍스트를 제공하는 것이 아닙니다. 적절한 시기에 최소한의, 구체적이며, 중복되지 않는 올바른 컨텍스트를 제공하는 것입니다.

그것은 훨씬 더 어려운 문제이자, 훨씬 더 흥미로운 문제입니다.
