웹 전체를 파일 시스템으로 전환하기

코드 환각(hallucinations)은 모델의 문제가 아니라 데이터의 문제입니다.

매일 API는 호환성을 깨는 변경 사항(breaking changes)을 적용하고, 엔드포인트(endpoints)를 사용 중단하며, 파라미터(parameters) 이름을 변경합니다. 모델은 훈련 데이터가 몇 달, 때로는 몇 년씩 뒤처지기 때문에 이러한 변화를 따라잡을 수 없습니다. 그래서 에이전트(agent)가 완벽해 보이고 깔끔하게 컴파일되는 코드를 작성하더라도, 라이브러리가 6개월 전에 업데이트되었다면 런타임(runtime)에 실패하게 됩니다.

표준적인 해결책은 RAG(Retrieval Augmented Generation)입니다: 문서를 청크(chunk)로 나누고, 임베딩(embed)한 다음, 일치하는 상위 K개의 조각(fragments)을 검색하는 방식입니다. 이 방식은 80% 정도의 정확도를 제공하지만, 답변이 세 페이지에 걸쳐 있거나 에이전트가 청킹(chunking) 과정에서 손실된 정확한 함수 시그니처(function signatures)를 필요로 할 때 한계에 부딪힙니다. 검색은 조각을 제공하지만, 에이전트는 전체적인 그림을 필요로 합니다.

우리는 다른 것을 원했습니다. 만약 에이전트가 문서를 *읽을* 수 있다면 어떨까요? 청크를 검색하는 것이 아니라, 실제 문서를 탐색(browse)하는 방식입니다. 마치 코드베이스를 `grep`으로 찾거나 `README` 파일을 `cat`으로 읽는 것과 동일하게 말이죠.

### 파일 시스템 아이디어

유닉스(Unix)는 50년 전에 이 문제를 해결했습니다. 장치, 프로세스, 소켓 — 모든 것이 파일입니다. 하나의 인터페이스로 `open`, `read`, `write`, `close`를 사용합니다. 각 리소스마다 새로운 API를 배울 필요 없이, 파일에 대한 지식 하나로 모든 곳에 적용할 수 있었습니다.

에이전트는 유닉스로 사전 훈련되었습니다. 파일 시스템 상호 작용의 수십억 개 토큰(tokens)이 가중치(weights)에 녹아 있습니다. `tree`, `grep`, `find`. 이것들은 에이전트가 *배워야 할* 도구가 아닙니다. 이미 *알고 있는* 도구입니다. 모든 코딩 모델은 `cat README.md`, `grep -r "auth" .`, `find . -name "*.md"`와 같은 수백만 가지 예시를 접했습니다.

이를 MCP(Multi-tool Coordination Protocol)와 비교해 보십시오. 모든 도구는 JSON 스키마(schema), 자연어 설명, 그리고 신중한 인자(argument) 구성을 필요로 합니다. 각각은 컨텍스트 윈도우(context window) 공간을 차지하고 오용될 여지를 만듭니다. 파일 시스템은 그 중 아무것도 필요하지 않습니다.

@jerryjliu0가 잘 설명했듯이, 파일 시스템 도구와 코드 인터프리터(code interpreter)를 가진 에이전트는 100개 이상의 MCP 도구를 가진 에이전트보다 더 보편적(general)이거나, 적어도 그만큼 보편적입니다.

그래서 우리는 물었습니다: 웹상의 모든 문서 사이트가 `cd` 명령어로 들어갈 수 있는 디렉터리(directory)라면 어떨까요?

### 작동 방식

이것이 전체 인터페이스(interface)입니다. API 키(API keys)도, 설치도, 설정(config)도 없습니다. 단 하나의 명령어로 모든 문서가 파일 시스템으로 마운트(mount)된 Bash 셸(shell)을 제공합니다. `tree`로 구조를 보고, `grep`으로 검색하고, `cat`으로 개별 페이지를 읽은 다음, 에이전트에게 파이프(pipe)로 전달할 수 있습니다.

내부적으로는 세 가지 과정이 발생합니다.

**1. 색인 (Index)**
처음으로 URL에 접근하면, 백엔드(backend)가 해당 사이트를 크롤링(crawl)합니다. `llms.txt`를 존중하고, OpenAPI 스펙(specs)을 감지하며, 리다이렉트(redirects)를 처리합니다. 각 페이지는 하나의 파일이 됩니다. 예를 들어 `https://docs.stripe.com/api/charges/create`라는 URL은 `/api/charges/create.md`가 됩니다.

까다로운 부분은 경로 정규화(path normalization)입니다. 문서 사이트들은 URL 구조가 매우 불규칙합니다. 어떤 곳은 모든 것을 `/docs/` 아래에, 어떤 곳은 `/api/reference/` 아래에 두거나, 단순히 루트(root)를 사용하기도 합니다. 우리는 색인된 모든 URL에서 공통 경로 접두사(common path prefix)를 자동으로 감지하고 제거하여, 실제 콘텐츠에 도달하기 전에 파일 시스템이 세 단계 깊이로 들어가지 않도록 합니다.

결과적으로 `https://better-auth.com/docs/installation`은 `/docs/installation.md`가 아니라 `/installation.md`가 됩니다. 파일 시스템은 URL이 어떻게 구성되어 있는지보다 사용자가 문서를 어떻게 생각하는지에 맞춰 반영됩니다.

**2. 서비스 (Serve)**
백엔드는 파일 시스템 작업(operations)을 API 엔드포인트(endpoints)로 노출합니다. 모든 데이터는 `gzip`으로 압축됩니다. `/load` 엔드포인트는 단일 요청으로 상태(status)와 모든 파일을 반환합니다. 셸 세션(shell session)을 시작하는 데 한 번의 왕복(round trip)이면 충분합니다. 응답은 캐시(cache)되며(`Cache-Control: public, max-age=300`), CLI(Command Line Interface)는 네임스페이스(namespace)와 `indexed_at` 타임스탬프(timestamp)를 키(key)로 하는 디스크 캐시(disk cache)를 `~/.cache/nia-docs/`에 유지합니다. 같은 문서에 두 번째 방문 시 즉시 로드됩니다.

네임스페이스는 공유됩니다. 누군가 `docs.stripe.com`을 색인하면, `shelldocs-{canonical_id}` 아래에 저장되어 모든 사람이 사용할 수 있습니다. 전체 시스템은 의도적으로 인증(authentication)이 필요 없습니다. 이것들은 공개 문서이기 때문입니다.

**3. 셸 (Shell)**
셸 자체는 클라이언트(client)에서 실행됩니다. 컨테이너(container)도, 샌드박스(sandbox)도, 가상 머신(VM)도 없습니다. 우리는 `just-bash`를 사용하는데, 이는 `grep`, `cat`, `ls`, `find`, `cd`, `tree`, 파이프(pipes), 그리고 별칭(aliases)을 지원하는 Bash의 TypeScript 재구현입니다. 전체 파일 시스템은 인메모리(in-memory) 자바스크립트(JavaScript) 객체(object)입니다.

모든 파일은 사전에 로드되어 가상 파일 시스템(virtual filesystem)에 마운트됩니다. 명령은 인-프로세스(in-process)로 실행됩니다. 500페이지 이상의 문서에서 `grep -r "webhook" .` 명령은 단순히 메모리 내 문자열 일치(string matching)이기 때문에 밀리초(milliseconds) 단위로 완료됩니다. 대규모 사이트(1,000페이지 이상)의 경우, 파일 트리(file tree)를 사전에 로드하고 `cat` 명령 시 콘텐츠를 지연 로드(lazily fetch)하며, 이는 세션(session) 동안 캐시됩니다.

### 왜 실제 샌드박스를 사용하지 않을까요?

*   **부팅 시간(Boot time):** 약 100ms(캐시된 경우), 약 2초(이미 색인된 경우), 약 30~120초(새 사이트의 콜드 색인)
*   **세션당 컴퓨팅(Per-session compute):** 서버에서는 제로. 파일은 Postgres에서 오고, 명령은 클라이언트에서 실행
*   **오케스트레이션(orchestration)이나 웜 풀(warm pools) 불필요**

명백한 대안은 에이전트에게 실제 파일 시스템을 제공하는 것입니다. 컨테이너(container)를 띄우고, 문서를 복제(clone)한 다음 에이전트를 작동시키는 방식입니다. 일부 팀은 이렇게 합니다. 이는 올바르고 작동하지만, 본질적으로 정적 텍스트(static text)에 대한 읽기 전용(read-only) 작업 부하(workload)에는 느리고 비용이 많이 듭니다.

문서 셸은 프로세스 격리(process isolation), 쓰기 가능한 저장소(writable storage) 또는 커널(kernel)이 필요하지 않습니다. 알려진 파일 집합에 대한 문자열 일치만 있으면 됩니다. 이를 마이크로-VM(micro-VM) 내에서 실행하는 것은 책꽂이를 보관하기 위해 창고를 빌리는 것과 같습니다.

우리는 다른 방향을 택했습니다. 셸은 인-프로세스(in-process) TypeScript Bash 인터프리터(interpreter)로서 전적으로 클라이언트에서 실행됩니다. 백엔드는 단순한 파일 서버(file server)입니다. 부팅할 컨테이너도, 관리할 세션도 없습니다.

### 에이전트는 이미 이것을 알고 있습니다.

이는 에이전트의 지시 파일(instruction file)에 몇 줄을 추가합니다. 에이전트는 이를 감지합니다. 무언가가 어떻게 작동하는지 확인할 필요가 있을 때, 명령을 실행하여 실제 최신 문서를 가져오고, 이를 기반으로 코드를 작성합니다. 훈련 데이터에 있던 내용이 아니라요.

`-c` 플래그(flag)는 단일 명령을 실행하고 종료합니다. 에이전트는 대화형 세션(interactive session)에 들어가지 않습니다. `grep`을 실행하고, 결과를 얻고, 관련 파일을 읽고, 다음으로 넘어갑니다. 이것은 표준적인 도구 사용 루프(tool-use loop)입니다. 파일 시스템은 단지 인터페이스일 뿐입니다.

### 문서로 시작하는 이유

우리는 문서로 시작합니다. 이는 환각(hallucinations)이 가장 큰 피해를 입히고 데이터 문제가 가장 다루기 쉬운(tractable) 영역이기 때문입니다. 문서 사이트는 공개되어 있고, 구조화되어 있으며, 비교적 작습니다(대부분 1,000페이지 미만). 훈련 데이터가 항상 오래될 만큼 자주 변경되지만, 5일간의 캐시 TTL(Time-To-Live)로 최신 상태를 유지하기에 충분할 만큼 느리게 변경됩니다.

하지만 이 아이디어는 문서 그 이상입니다. API 레퍼런스(reference)는 디렉터리이고, 변경 로그(changelog)는 파일입니다. OpenAPI 스펙(spec)은 이미 `cat`으로 읽을 수 있는 JSON입니다. 우리는 문서와 함께 OpenAPI 스펙을 자동으로 감지하고 색인합니다. 이들은 파일 시스템에서 `/api-spec/` 아래에 나타납니다.

웹 전체가 코드베이스(codebase)처럼 탐색 가능해야 합니다. 우리는 현재 에이전트에게 가장 중요한 부분부터 시작하여 이를 현실로 만들 인프라(infrastructure)를 구축하고 있습니다.

### 측정 내용

에이전트가 셸 세션에서 실행하는 모든 명령은 기록됩니다(NIA_DOCS_TELEMETRY=off를 통해 옵트아웃 가능): 명령 이름, 성공 여부, 소요 시간, `grep`과 일치하는 파일. 내용은 기록되지 않고, 패턴만 기록됩니다. 우리는 에이전트가 실제로 문서를 어떻게 탐색하는지 이해하여 파일 시스템을 더 좋게 만들고자 합니다.

초기 데이터는 에이전트가 일관된 워크플로우(workflow)로 수렴한다는 것을 보여줍니다: `tree`로 방향을 잡고, `grep -rl`로 관련 파일을 찾고, `cat`으로 읽는 방식입니다. 이는 인간 개발자가 할 법한 행동과 정확히 일치합니다. 파일 시스템 추상화(abstraction)는 에이전트가 *배워야 할* 것이 아니라, 선택권이 주어졌을 때 *기본적으로 사용하는* 것입니다.

웹은 거대합니다. 하지만 모든 문서 사이트, 모든 API 레퍼런스, 모든 변경 로그는 `cd` 명령어로 들어갈 수 있는 디렉터리가 됩니다.
