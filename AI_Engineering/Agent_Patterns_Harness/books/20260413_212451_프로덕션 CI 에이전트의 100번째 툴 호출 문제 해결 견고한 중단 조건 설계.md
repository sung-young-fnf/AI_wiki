프로덕션 CI 에이전트의 '100번째 툴 호출 문제' 해결: 견고한 중단 조건 설계

CI 파이프라인이 빨간색으로 변하고 배포가 중단됩니다. 새로운 AI 에이전트에게 근본 원인을 찾아 해결책을 제시해 달라고 요청합니다. 에이전트가 작업을 시작하면 터미널로 전환하여 로그를 확인합니다.

에이전트는 워크플로(workflow) 실행 메타데이터(metadata)를 가져오고, 실패한 작업을 식별하며, 빌드 로그(build log)의 마지막 부분을 가져옵니다. 여기까지는 순조로워 보입니다.

하지만 곧 로그의 방대한 텍스트 더미에 혼란스러워합니다. 특정 오류 문자열을 찾기 위해 리포지토리(repository)를 검색하기로 결정하지만 아무것도 찾지 못합니다. 약간 다른 문자열로 다시 검색해도 마찬가지입니다. 결국 로그 툴(tool)로 돌아가 동일한 로그의 마지막 부분을 다시 가져옵니다.

당신은 터미널 출력이 스크롤되는 것을 지켜봅니다. 툴 호출(Tool Call) 10회. 툴 호출 50회. 툴 호출 100회. 에이전트는 여전히 맴돌고 있습니다. 이미 시간당 API 예산을 모두 소진했고, 컨텍스트 윈도우(context window)는 거대하고 읽을 수 없는 텍스트 덩어리로 불어나 있습니다. 하지만 여전히 당신에게 답을 주지 못합니다.

이것이 바로 '100번째 툴 호출 문제'입니다.

에이전트를 충분히 오래 개발하다 보면 이 문제에 부딪히게 됩니다. 시스템 프롬프트(system prompt)를 조정하여 해결하려 합니다. 모델에게 다섯 단계 후 중단하라고 대문자로 지시하는 줄을 추가해도 모델은 이를 무시합니다. API 비용을 고려하라고 말해도 마찬가지입니다.

중단 조건은 프롬프트 엔지니어링(prompt engineering) 문제가 아닙니다. 파이썬(Python) 코드에 의해 강제되는 런타임 계약(runtime contract)입니다.

오늘 우리는 깃허브 액션(GitHub Actions)을 위한 CI 실패 분류 에이전트(CI Failure Triage Agent)를 구축할 것입니다. 이를 랭그래프(LangGraph)와 연결할 것입니다. 하지만 더 중요한 것은 엄격한 중단 컨트롤러(StopController)를 구축하는 것입니다. 예산을 추적하고, 정체된 루프(stagnant loops)를 감지하며, 회로 차단기(circuit breaker)를 구현하여 에이전트가 더 이상 의미 있는 작업을 수행하지 못할 때 강제로 중단시킬 것입니다.

### 에이전트가 제어 불능 상태에 빠지는 이유

수정 사항을 작성하기 전에 실패의 원리를 이해해야 합니다. ReAct 에이전트(agent)는 추론과 행동을 번갈아 수행합니다. 대화 기록을 보고 어떤 툴을 호출할지 결정하고, 그 결과를 관찰한 다음 이 과정을 반복합니다. 이 루프는 강력하지만, 자연적인 제동 장치가 없습니다.

나선형으로 제어 불능 상태에 빠지는 첫 번째 원인은 모호한 목표입니다. 에이전트에게 'CI를 수정하라'고 말하는 것은 바람일 뿐 측정 가능한 목표가 아닙니다. 에이전트는 언제 성공했는지 알지 못합니다. '완료'에 대한 명확한 정의가 없기 때문에 계속해서 더 많은 정보를 찾으려 합니다.

두 번째 원인은 툴 쓰레싱(tool thrash)입니다. 에이전트가 로그 파일을 읽고 핵심 오류를 놓칩니다. 더 많은 컨텍스트가 필요하다고 생각하여 동일한 툴을 동일한 인수로 호출합니다. 그러면 정확히 동일한 응답을 받습니다. 이제 에이전트는 결정론적 루프(deterministic loop)에 갇히게 됩니다.

세 번째 원인은 나쁜 툴 사용성(tool ergonomics)입니다. 툴이 10,000줄의 원시 출력(raw output)을 담은 거대한 JSON 블롭(JSON blob)을 반환하면, 언어 모델(language model)은 주의력을 잃습니다. 원래 찾던 것을 잊어버리고, 다시 균형을 잡기 위해 다른 툴을 호출하려 합니다.

에이전트가 툴을 호출할 때마다 관찰 결과가 메시지 기록에 추가됩니다. 컨텍스트 윈도우는 커지고, 단계당 비용은 제곱으로 증가합니다.

### 중단 조건 계약

우리는 중단 조건을 에이전트 런타임(runtime)의 API 계약(contract)처럼 다루어야 합니다. 모델에게 중단을 요청하는 것이 아니라 강제로 중단시켜야 합니다. 이를 위해 네 가지 방어 계층이 필요합니다.

첫째, 엄격한 예산(hard budgets)이 필요합니다. 그래프 전환의 총 수를 제한하기 위해 `max_steps`를 추적합니다. CI 파이프라인이 중단 해제되기를 사람들이 기다리고 있으므로 `max_wall_clock_seconds`를 추적합니다. 에이전트가 깃허브(GitHub) API를 과도하게 호출하여 IP 주소가 차단되는 것을 방지하기 위해 `max_tool_calls_total`과 `max_tool_calls_per_tool`을 추적합니다.

둘째, 진행 없음 규칙(no-progress rules)이 필요합니다. 에이전트가 여러 번 툴을 호출했지만 새로운 증거를 전혀 발견하지 못하면 중단시킵니다. 동일한 툴과 인수를 여러 번 연속해서 반복하면 에이전트를 중단시킵니다.

셋째, 회로 차단기(circuit breakers)가 필요합니다. 깃허브 API가 연속으로 두 번 500 오류를 반환하면 중단합니다. 세 번째 시도할 이유가 없습니다. 액세스 거부(access denied) 오류가 발생하면 즉시 중단합니다.

넷째, 에스컬레이션 규칙(escalation rules)이 필요합니다. 에이전트를 강제로 중단시킬 때 단순히 파이썬 예외(Python exception)를 발생시키지 않습니다. 대신 최종화 노드(finalization node)로 전환합니다. `NEEDS_HUMAN` 또는 `BUDGET_EXCEEDED`와 같은 종료 이유(exit reason)를 전달합니다. 그러면 에이전트는 찾아낸 모든 것을 포맷하고 사람에게 도움을 요청해야 합니다.

### CI 분류 에이전트 설계

이 계약을 깃허브 액션 분류 에이전트에 적용해 봅시다. 입력은 간단합니다. 리포지토리 이름, 실행 ID, 그리고 질문을 제공합니다.

출력은 구조화되어야 합니다. 실패에 대한 가장 높은 순위의 가설, 최소한의 수정 권장 사항, 특정 로그 라인과 파일 참조를 포함하는 증거 목록을 원합니다. 또한 에이전트가 왜 중단되었는지 알 수 있도록 정확한 종료 이유도 필요합니다.

우리는 '완료'에 대한 엄격한 정의를 가지고 있습니다. 에이전트는 실패한 작업을 명명하고, 실패 유형을 분류하며, 수정 사항을 제안하고, 증거를 인용할 수 있을 때 완료된 것입니다. 예산이 소진되기 전에 이를 수행할 수 없으면 종료하고 구체적인 질문을 해야 합니다.

### 효율적인 툴 구축

에이전트용 툴은 경계가 명확해야 합니다(bounded). 원시 API 응답을 컨텍스트 윈도우(context window)에 무작정 덤프(dump)할 수는 없습니다.

이 에이전트를 위해 다섯 가지 툴을 만들 것입니다. `get_workflow_run`은 상위 수준의 메타데이터(metadata)를 가져오고, `get_failed_jobs`는 충돌한 특정 작업을 나열하며, `get_job_log_tail`은 로그의 마지막 몇 줄을 가져오고, `search_repo`는 코드 스니펫(code snippet)을 찾고, `read_file`은 특정 파일 내용을 가져옵니다.

### 상태 스키마(State Schema)

중단 조건에는 상태(state)가 필요합니다. 대부분의 기본적인 랭그래프(LangGraph) 데모는 메시지 목록만 추적합니다. 이는 프로덕션(production) 환경에는 충분하지 않습니다. 에이전트의 진행 상황을 측정하기 위해 에이전트가 수행하는 모든 것을 추적해야 합니다.

우리는 `TypedDict`를 사용하여 `CIAgentState`를 정의합니다.

```python
import operator
from typing import Annotated, Any, Literal, TypedDict
from langchain_core.messages import BaseMessage
from langgraph.graph.message import add_messages

ExitReason = Literal[
    "SOLVED", "BUDGET_EXCEEDED", "NO_PROGRESS",
    "NEEDS_HUMAN", "ACCESS_DENIED", "UNKNOWN",
]

class CIAgentState(TypedDict):
    repo: str
    run_id: int
    question: str
    messages: Annotated[list[BaseMessage], add_messages]

    # Budgets
    step_count: int
    max_steps: int
    started_at: float
    max_wall_clock_seconds: int
    tool_calls_total: int
    max_tool_calls_total: int
    max_tool_calls_per_tool: int
    tool_calls_by_tool: dict[str, int]
    
    # History and Streaks
    tool_call_history: Annotated[list[dict[str, str]], operator.add]
    last_tool_args_key: str
    repeated_tool_args_streak: int
    consecutive_tool_failures: int

    # Evidence and Progress
    evidence: Annotated[list[dict[str, str]], operator.add]
    stagnant_turns: int

    # Output
    exit_reason: ExitReason
    final_answer: str
```

이 스키마(schema)는 중단 컨트롤러의 '두뇌'입니다.

`tool_call_history`와 `evidence`에 `operator.add`를 사용합니다. 이는 랭그래프에게 각 노드(node) 전환 시 전체 목록을 덮어쓰는 대신 새 항목을 목록에 추가하도록 지시합니다.

우리는 `tool_calls_total`을 `step_count`와 별도로 추적합니다. 단일 추론 단계에서 여러 병렬 툴 호출이 생성될 수 있습니다. 그래프 루프(graph loop)뿐만 아니라 실제 API 요청에 대한 예산도 책정해야 합니다.

또한 `repeated_tool_args_streak`도 추적합니다. 이는 '100번째 툴 호출 문제'에 대한 우리의 주요 방어선입니다. 에이전트가 정확히 동일한 툴을 정확히 동일한 인수로 세 번 연속 호출하면, 에이전트는 고장난 것입니다. 실행을 중단시켜야 합니다.

### 증거 추출 및 상태 업데이트

루프는 'reason' 노드에서 시작됩니다. 언어 모델은 메시지를 보고 툴을 호출하기로 결정합니다. 그래프는 툴 노드로 라우팅됩니다. 툴은 실행되고 관찰 결과를 메시지 목록에 추가합니다.

대부분의 아키텍처는 여기서 멈춥니다. 우리는 툴 실행이 끝난 직후 새로운 노드를 삽입할 것입니다. 이를 `extract_evidence_node`라고 부릅니다.

이 노드는 두 가지 역할을 합니다. 원시 툴 출력을 파싱하여 구체적인 증거를 찾고, 모든 카운터와 연속 추적기를 업데이트합니다.

첫째, 툴 인수를 해싱(hashing)하는 방법이 필요합니다. `{"query": "error"}`와 `{"query": "error"}`가 동일한지 알고 싶기 때문입니다. 딕셔너리(dictionary)를 직렬화(serialize)하고 키를 정렬한 다음 해싱합니다.

```python
import hashlib

def _stable_args_hash(args: dict[str, Any]) -> str:
    payload = json.dumps(args, sort_keys=True, separators=(",", ":"))
    return hashlib.sha1(payload.encode("utf-8")).hexdigest()[:12]
```

또한 로그에서 오류를 빠르게 찾는 방법도 필요합니다. 언어 모델은 쉽게 주의가 산만해집니다. 로그의 마지막 부분에서 일반적인 오류 시그니처(signature)를 스캔하고 추출하기 위해 간단한 파이썬 함수를 사용할 것입니다.

```python
_ERROR_MARKERS = ("ModuleNotFoundError", "ImportError", "AssertionError",
                  "ERROR", "FAIL", "Traceback")

def _extract_sigs(text: str) -> list[str]:
    seen: set[str] = set()
    out: list[str] = []
    for line in text.splitlines():
        s = line.strip()
        if s and any(m in s for m in _ERROR_MARKERS) and s not in seen:
            seen.add(s)
            out.append(s[:200])
        if len(out) >= 5:
            break
    return out
```

이 노드는 진행 상황 추적의 핵심입니다. 최근 툴 메시지를 반복 처리하고 JSON 페이로드(payload)를 파싱합니다. 로그의 마지막 부분을 찾으면 오류 시그니처를 추출하고, 파일 경로를 찾으면 파일 참조로 저장합니다.

그 다음에는 집합 차이(set difference)를 계산합니다. 새로 발견된 증거 키와 이전 증거 키를 비교합니다. 만약 집합 차이가 비어 있다면, 에이전트가 툴을 호출했지만 새로운 것을 전혀 학습하지 못했음을 의미합니다. 이때 `stagnant_turns` 카운터를 증가시킵니다.

또한 툴 페이로드의 오류도 확인합니다. 툴이 파싱에 실패하거나 명시적인 오류 문자열을 반환하면 `consecutive_tool_failures` 카운터를 증가시킵니다.

### 중단 컨트롤러 로직

이제 상태가 업데이트되었으므로 `stop_controller_node`로 라우팅합니다. 이 함수는 순수한 파이썬 함수이며, 언어 모델을 호출하지 않습니다. 단지 우리의 엄격한 제약 조건에 대해 상태를 평가할 뿐입니다.

```python
import time

def stop_controller_node(state: CIAgentState) -> dict[str, Any]:
    elapsed = time.time() - state["started_at"]

    if state["step_count"] >= state["max_steps"]:
        return _stop("BUDGET_EXCEEDED", "max_steps reached")
        
    if state["tool_calls_total"] >= state["max_tool_calls_total"]:
        return _stop("BUDGET_EXCEEDED", "max_tool_calls_total reached")
        
    if any(v >= state["max_tool_calls_per_tool"] for v in state["tool_calls_by_tool"].values()):
        return _stop("BUDGET_EXCEEDED", "max_tool_calls_per_tool reached")
        
    if elapsed >= state["max_wall_clock_seconds"]:
        return _stop("BUDGET_EXCEEDED", "wall clock exceeded")
        
    if state["stagnant_turns"] >= 3:
        return _stop("NO_PROGRESS", "no new evidence for 3 turns")
        
    if state["repeated_tool_args_streak"] >= 3:
        return _stop("NO_PROGRESS", "same tool+args repeated")
        
    if state["consecutive_tool_failures"] >= 2:
        return _stop("NEEDS_HUMAN", "circuit breaker: repeated tool failures")

    for e in state["evidence"]:
        if e["kind"] == "log_signature" and ("403" in e["value"] or "401" in e["value"]):
            return _stop("ACCESS_DENIED", "auth error in logs")

    return {"exit_reason": "UNKNOWN"}

def _stop(reason: ExitReason, note: str) -> dict[str, Any]:
    return {"exit_reason": reason, "messages": [AIMessage(content=f"Stop: {note}.")]}
```

이 코드가 바로 API 예산을 절약해 주는 부분입니다. 우리는 실제 경과 시간(wall clock time)을 확인합니다. 에이전트가 20초 동안 실행되었다면 중단시킵니다. CI 분류는 빨라야 합니다. 툴별 예산을 확인합니다. 에이전트가 `search_repo`를 8번 호출했다면 중단시킵니다. 9번째 시도에서 답을 찾을 리 없습니다. 정체된 턴(stagnant turns)을 확인합니다. 새로운 증거를 찾지 못한 채 3턴이 지났다면 중단시킵니다.

이러한 조건 중 하나라도 충족되면, `exit_reason`을 업데이트하고 히스토리(history)에 합성된 `AIMessage`를 추가하는 상태 패치(state patch)를 반환합니다. 이 메시지는 모델에게 왜 중단되었는지 명시적으로 알려줍니다.

그 다음에는 조건부 라우팅(conditional routing) 함수를 사용하여 그래프가 다음에 어디로 갈지 결정합니다.

```python
def route_after_stop(state: CIAgentState) -> str:
    if state["exit_reason"] in ("BUDGET_EXCEEDED", "NO_PROGRESS", "ACCESS_DENIED", "NEEDS_HUMAN"):
        return "finalize"
    return "reason"
```

에이전트가 명확한 상태라면, 다음 움직임을 계획하기 위해 'reason' 노드로 돌아갑니다. 중단 조건에 도달했다면 'finalize' 노드로 강제로 이동합니다.

### 최종화 노드(The Finalize Node)

에이전트가 문제를 해결했든, 한계에 도달했든, 어느 쪽이든 사용자에게 구조화된 응답을 제공해야 합니다.

모델이 수집한 모든 증거를 종합하도록 강제하는 프롬프트(prompt)를 구축합니다. 모델이 실패했는지 알 수 있도록 `exit_reason`을 프롬프트에 명시적으로 전달합니다.

```python
from langchain_core.messages import HumanMessage

async def finalize_node(state: CIAgentState, llm: Any) -> dict[str, Any]:
    ev_block = "\n".join(
        f"- {e['kind']}: {e['value']}" for e in state["evidence"][-12:]
    ) or "- (no evidence captured)"

    exit_reason = state["exit_reason"] if state["exit_reason"] != "UNKNOWN" else "NEEDS_HUMAN"

    prompt = (
        f"Repo: {state['repo']}  Run: {state['run_id']}\n"
        f"Question: {state['question']}\n"
        f"Exit reason: {exit_reason}\n\n"
        "Write:\n"
        "1) Ranked hypotheses (top 3) with confidence 1-10\n"
        "2) Smallest fix OR smallest experiment to confirm\n"
        "3) Evidence (bulleted)\n"
        "4) If NEEDS_HUMAN: ask 1-2 targeted questions\n\n"
        f"Evidence:\n{ev_block}\n"
    )
    resp = await llm.ainvoke([HumanMessage(content=prompt)])
    return {"final_answer": str(resp.content), "exit_reason": exit_reason}
```

에이전트가 `NO_PROGRESS` 중단 조건에 도달했다면, 프롬프트에서 이를 확인하게 됩니다. 수집한 부분적인 증거를 살펴보고 가설을 세운 다음, 누락된 부분을 사람에게 질문할 것입니다. 이는 영원히 맴돌기만 하고 아무것도 반환하지 않는 에이전트보다 훨씬 낫습니다.

### 전체 구성

랭그래프(LangGraph)의 `StateGraph`를 사용하여 그래프를 컴파일(compile)합니다. 툴을 언어 모델에 바인딩하고 노드를 연결합니다.

```python
from functools import partial
from langgraph.graph import END, START, StateGraph
from langgraph.prebuilt import ToolNode

SYSTEM_PROMPT = (
    "You are a CI failure triage agent for GitHub Actions.\n"
    "Goal: identify the most likely root cause and smallest fix.\n"
    "Use tools to fetch only bounded evidence.\n"
    "Do not loop: if you can't make progress, prepare to finalize with NEEDS_HUMAN.\n"
    "Return tool calls when you need evidence. Otherwise, draft the final response."
)

async def reason_node(state: CIAgentState, llm_with_tools: Any) -> dict[str, Any]:
    sys = SystemMessage(content=SYSTEM_PROMPT)
    msgs = [sys] + (state["messages"] or [HumanMessage(content=state["question"])])
    resp = await llm_with_tools.ainvoke(msgs)
    return {"messages": [resp], "step_count": state["step_count"] + 1}

def route_after_reason(state: CIAgentState) -> str:
    last = state["messages"][-1]
    if isinstance(last, AIMessage) and last.tool_calls:
        return "tools"
    return "finalize"

def build_app(tools: list, llm: Any):
    llm_with_tools = llm.bind_tools(tools)
    tool_node = ToolNode(tools)

    g = StateGraph(CIAgentState)

    g.add_node("reason", partial(reason_node, llm_with_tools=llm_with_tools))
    g.add_node("tools", tool_node)
    g.add_node("extract_evidence", extract_evidence_node)
    g.add_node("stop_controller", stop_controller_node)
    g.add_node("finalize", partial(finalize_node, llm=llm))

    g.add_edge(START, "reason")
    g.add_conditional_edges("reason", route_after_reason, {"tools": "tools", "finalize": "finalize"})
    g.add_edge("tools", "extract_evidence")
    g.add_edge("extract_evidence", "stop_controller")
    g.add_conditional_edges("stop_controller", route_after_stop, {"reason": "reason", "finalize": "finalize"})
    g.add_edge("finalize", END)

    return g.compile()



def parse_args() -> argparse.Namespace:
    p = argparse.ArgumentParser(description="CI triage agent with stop conditions.")
    p.add_argument("--repo", default="acme/widgets")
    p.add_argument("--run-id", type=int, default=999111)
    p.add_argument("--question", default="Why did CI fail and what's the smallest fix?")
    p.add_argument("--max-steps", type=int, default=10)
    p.add_argument("--max-seconds", type=int, default=20)
    p.add_argument("--max-tool-calls-total", type=int, default=24)
    p.add_argument("--max-tool-calls-per-tool", type=int, default=8)
    p.add_argument("--live", action="store_true")
    return p.parse_args()


async def main() -> None:
    args = parse_args()
    model = os.getenv("GEMINI_MODEL", "gemini-2.5-flash")
    llm = ChatGoogleGenerativeAI(model=model, temperature=0.1)

    tools = [get_workflow_run, get_failed_jobs, get_job_log_tail, search_repo, read_file]
    app = build_app(tools, llm)

    result = await app.ainvoke({
        "repo": args.repo,
        "run_id": args.run_id,
        "question": args.question,
        "messages": [HumanMessage(content=args.question)],
        "step_count": 0,
        "max_steps": args.max_steps,
        "started_at": time.time(),
        "max_wall_clock_seconds": args.max_seconds,
        "tool_calls_total": 0,
        "max_tool_calls_total": args.max_tool_calls_total,
        "max_tool_calls_per_tool": args.max_tool_calls_per_tool,
        "tool_calls_by_tool": {},
        "tool_call_history": [],
        "last_tool_args_key": "",
        "repeated_tool_args_streak": 0,
        "consecutive_tool_failures": 0,
        "evidence": [],
        "stagnant_turns": 0,
        "exit_reason": "UNKNOWN",
        "final_answer": "",
    })

    print(f"\n{'=' * 72}")
    print(f"CI Triage | {args.repo} | Run {args.run_id} | {model}")
    print(f"Exit: {result.get('exit_reason')}")
    print(f"{'=' * 72}\n")
    print(result.get("final_answer", ""))


if __name__ == "__main__":
    asyncio.run(main())
```

흐름에 주목하십시오. 'reason'에서 시작하여 'tools'로 라우팅됩니다. 툴이 실행되면 즉시 'extract_evidence'로 흐릅니다. 증거가 파싱되고 상태가 업데이트됩니다. 그 다음 'stop_controller'로 흐릅니다. 컨트롤러는 상태를 평가하고 'reason'으로 다시 루프할지, 아니면 'finalize'로 탈출할지 결정합니다.

이러한 그래프 구조는 중단 로직(stop logic)이 모든 단일 관찰 후에 실행됨을 보장합니다. 에이전트는 이를 우회할 수 없습니다.

이제 경계가 명확하고(bounded), 감사 가능하며(auditable), 종료가 보장되는 CI 분류 에이전트를 갖게 되었습니다. 이 사실 하나만으로도 대부분의 '프로덕션 에이전트' 관련 게시물보다 앞서 나갈 수 있습니다.
