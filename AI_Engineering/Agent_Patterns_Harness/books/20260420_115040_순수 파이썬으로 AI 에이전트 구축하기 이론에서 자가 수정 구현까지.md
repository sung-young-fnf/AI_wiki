# 순수 파이썬으로 AI 에이전트 구축하기: 이론에서 자가 수정 구현까지

이전 글인 "챗봇을 넘어(Beyond the Chatbot)"에서 우리는 에이전트(Agent)에 대한 간단한 정의를 내렸습니다: 에이전트는 도구(Tools)와 루프(Loop)를 갖춘 LLM입니다.

LangChain이나 CrewAI와 같은 프레임워크가 강력하지만, 종종 로직을 너무 많은 계층으로 감싸서 "기계적(mechanical)" 실체를 파악하기 어렵게 만듭니다. AI를 진정으로 마스터하려면 순수 파이썬(raw Python)과 직접적인 API 호출을 사용하여 처음부터 에이전트를 구축해야 합니다.

이 가이드에서는 구조화된 계획-실행(Structured Plan-and-Execute) 에이전트를 구축할 것입니다. 이 아키텍처는 종종 "ReAct"의 범주에 속하지만, 복잡한 워크플로에 더 견고합니다. 이는 실행 전에 쿼리를 JSON 기반의 작업 목록으로 분해하기 때문입니다.

ReAct를 찾으셨나요? 걱정 마세요. 다음 글에서는 완전한 ReAct 에이전트를 처음부터 다루고 구현할 것입니다. 하지만 에이전트 분야가 처음이고 이 분야에 진지하다면, 이 패턴(계획-실행)부터 시작하는 것이 좋습니다. 계획-실행(Plan-and-Execute) 방식은 고정밀의 실제 시스템에 필요한 구조적 규율과 예측 가능성을 제공합니다. 여기서부터 시작해야 단순히 "프롬프트만 다루는 사람"이 아닌 진정한 전문가가 될 수 있습니다.

## 우리가 구축할 것:

*   **손(The Hands)**: LLM 호환성을 위한 파이썬 함수와 JSON 스키마 정의.
*   **플래너(The Planner)**: 추론 중심의 LLM 호출을 사용하여 복잡한 쿼리를 작업 목록으로 분해.
*   **실행자(The Executor)**: 작업을 반복하고 특정 도구에 매핑.
*   **안전망(The Safety Net)**: 비결정론적 JSON의 "혼돈"을 처리하기 위한 3단계 재시도 루프(retry loop) 구현.

## 아키텍처: 계획 후 실행 (Plan then Execute)

고전적인 ReAct 패턴(한 번에 한 단계씩 생각하고 행동)과 달리, 우리의 에이전트는 더 체계적인 3단계 파이프라인을 따릅니다:

*   **플래너(The Planner)**: LLM이 사용자 질문을 받아 JSON 형식의 작업 목록을 반환합니다.
*   **실행자(The Executor)**: 각 작업에 대해 LLM이 적절한 도구를 선택하고, 파이썬 함수를 실행한 후, 그 결과를 추가합니다.
*   **종합자(The Synthesizer)**: 마지막으로 LLM이 전체 실행 이력을 바탕으로 최종 답변을 제공합니다.

파이프라인을 이해하는 것이 우리의 지도이지만, 적절한 장비 없이는 지도가 무용지물입니다. 이 청사진을 작동하는 기계로 만들기 위해, 우리는 AI가 현실 세계와 접촉할 수 있도록 하는 인터페이스를 먼저 구축해야 합니다. 에이전트 세계에서는 이를 "손(Hands)"이라고 부릅니다.

## 1단계: "손" 구현 (API 계약)

에이전트가 계획을 실행하기 전에, 에이전트가 실제로 무엇을 할 수 있는지 알아야 합니다. 1단계는 "손(Hands)", 즉 도구(Tools)를 정의하는 것입니다.

이를 파이썬 환경과 LLM 간의 API 계약(API Contract)을 생성하는 것으로 생각해보세요. LLM은 파이썬 코드를 볼 수 없으며, 여러분이 제공하는 JSON 스키마만 볼 수 있습니다. 따라서 모든 도구는 두 부분으로 구성됩니다:

*   **로직(The Logic)**: 실제 작업을 수행하는 파이썬 함수.
*   **스키마(The Schema)**: LLM에게 언제, 어떻게 해당 함수를 호출해야 하는지 알려주는 설명.

에이전트의 손을 만들어 봅시다: 먼저 파이썬 함수와 그 JSON 스키마를 정의해야 합니다.

### 두뇌(LLM)와 손 연결하기

순수 파이썬 코드를 살펴보기 전에, AI 엔지니어링에서 가장 큰 문제인 '언어'를 해결해야 합니다. 파이썬 환경은 "코드"(함수 및 변수)를 말하지만, LLM은 "텍스트"를 말합니다. 이 간극을 메우기 위해 JSON 스키마를 사용합니다. 이를 마치 "번역 매뉴얼"처럼 생각하여, 두뇌가 손을 움직이는 방법을 이해할 수 있게 합니다.

우리 아키텍처에서 1단계는 이 매뉴얼을 만드는 데 중점을 둡니다. 단순히 함수를 작성하는 것이 아니라, LLM에게 다음을 알려주는 함수 선언(Function Declaration)을 작성합니다:

*   도구가 무엇을 하는지 (설명).
*   어떤 특정 데이터가 필요한지 (매개변수).

이것이 "API 계약(API Contract)"이며, 플래너가 "화성의 질량을 가져와"라고 말할 때, 실행자가 정확히 어떤 함수를 트리거해야 하는지 알도록 보장합니다. 다음은 함수 선언의 예시입니다:

```python
# 1. 스키마 (LLM이 보는 것)
tools_schema = [
    {
        "type": "function",
        "name": "get_planet_mass",
        "description": "주어진 행성의 질량을 가져옵니다.",
        "parameters": {
            "type": "object",
            "properties": {
                "planet": {"type": "string", "description": "행성 이름 (예: Earth)"},
            },
            "required": ["planet"],
        },
    },
    # ... 'calculate'와 같은 추가 도구
]

# 2. 로직 (파이썬이 실행하는 것)
def get_planet_mass(planet_dict):
    planet = planet_dict["planet"].lower().strip()
    masses = {"earth": "5.972e24 kg", "mars": "6.39e23 kg", "jupiter": "1.898e27 kg"}
    return masses.get(planet, "알 수 없는 행성입니다.")
```

API 계약이 확립되고 에이전트의 손이 준비되었으니, 이제 언제 어떻게 손을 움직일지 정확히 결정하는 추론 엔진, 즉 플래너(The Planner)를 구현해야 합니다.

## 2단계: 플래너 (내부 청사진)

이제 에이전트가 손을 갖추었으니, 플래너를 구현해야 합니다.

플래너의 역할은 작업을 수행하는 것이 아니라, 인간의 쿼리(query)라는 혼돈을 구조화되고 실행 가능한 JSON 목록으로 분해하는 것입니다. 우리는 LLM에게 규칙서 역할을 하는 시스템 프롬프트(System Prompt)를 제공하여, 대화형 에세이가 아닌 유효한 JSON을 출력하도록 강제합니다.

여기서 JSON을 사용하는 이유: 플래너가 구조화된 작업 목록을 출력하도록 강제함으로써, 깔끔한 인계(hand-off)를 생성합니다. 실행자(Executor)는 다음에 무엇을 할지 "추측"할 필요 없이, 단순히 목록을 반복합니다.

```python
def __plan_tasks(self, user_prompt: str)-> json:
        '''LLM이 수행할 작업을 계획합니다.'''
        planner_system_prompt = (
            "당신은 정교한 플래너 에이전트입니다. "
            "당신의 역할은 복잡한 사용자 질문을 순차적이고 간단한 작업으로 분해하는 것입니다. "
            "단일 키를 가진 JSON 객체를 반환하세요 "
            " - 'tasks': list[string] # 문자열 목록을 포함합니다."
        )

        action_plan = self.call_llm(planner_system_prompt, user_prompt)
        return json.loads(action_plan.output_text)
```

하지만 계획은 단지 의도의 목록일 뿐입니다. 이러한 의도를 현실로 만들기 위해서는 각 작업을 1단계에서 정의한 특정 도구에 매핑하는 메커니즘이 필요합니다. 이는 실행자(Executor)에서 구현됩니다.

## 3단계: 실행자 (매핑 및 수행)

실행자(Executor)는 에이전트의 핵심 동력원(Workhorse)입니다. 플래너의 목록을 반복하며 특정 작업에 맞는 도구를 식별하고 실제 파이썬 실행을 처리합니다. 또한 이곳에서 재시도 로직(Retry Logic)을 구현하여, LLM이 형식 오류를 발생시키더라도 에이전트가 중단되지 않고 스스로 수정하도록 합니다.

간단히 말해: 계획의 각 작업에 대해 우리는 LLM에게 "이 특정 작업에 어떤 도구가 적합한가?"라고 묻습니다. 그런 다음 해당 도구를 실행하고 결과를 `execution_plan`에 저장합니다.

단순히 도구를 실행하는 것이 아니라, 결과를 다시 `execution_plan`에 저장하는 방식에 주목하세요. 이를 통해 에이전트는 방금 수행한 작업을 "기억"하여, 이후의 작업이 이전 결과에 의존할 수 있도록 합니다.

```python
def __plan_tools(self, action_plan, tools):
    execution_plan = []
    for task in action_plan["tasks"]:
        # 1. 두뇌에 묻습니다: "이 작업에 어떤 도구가 적합한가?"
        response = self.call_llm(execution_system_prompt, user_prompt)
        # 2. LLM의 문자열 선택을 실제 파이썬 함수에 매핑합니다.
        function_to_call = self.available_tools_dict[response["function"]]
        response["result"] = function_to_call(*kwargs)
    return execution_plan.append(response)
```

기계가 기계와 대화하게 하려면, 매우 구체적인 프롬프트(prompt)가 필요합니다. 우리는 LLM이 특정 키(key)를 가진 JSON 객체를 반환하도록 강제합니다. 이 구조 덕분에 우리의 파이썬 루프는 사람의 개입 없이 실행될 수 있습니다.

전문가 팁: 아래 `execution_system_prompt`를 주목하세요. 우리는 LLM에게 명시적으로 "만약 적합한 도구가 없으면, None이라고 작성해라"라고 지시합니다. 이는 에이전트가 적절한 장비가 없을 때 "가짜" 행동을 하는 것을 방지합니다.

```python
execution_system_prompt  = (
            "당신은 정교한 플래너 에이전트입니다. "
            "당신의 역할은 주어진 작업을 해결하기 위해 시스템에 제공된 올바른 도구를 찾는 것입니다."
            "사용자로부터 사용 가능한 도구와 작업을 받게 됩니다."
            "작업에 맞는 도구가 없으면 None이라고 작성하세요."
            "단일 키를 가진 JSON 객체를 반환하세요."
            " - 'id: int # 작업을 식별하는 고유 ID "
            " - 'task': {task}"
            " - 'function': string # 도구의 이름"
            " - 'properties: list # 함수를 실행할 속성"
            " - 'dependencies': list # 다른 작업의 필요한 결과 ID"
        )

user_prompt = (f"다음과 같은 작업이 있습니다: {task}"
               f"작업 해결을 위해 다음 도구들을 사용할 수 있습니다: {tools}"
               "주어진 작업에 사용할 올바른 도구를 알려주세요."
               f"다음은 전체 작업 목록입니다: {action_plan}"
               F"다음은 이미 완료된 실행입니다: {execution_plan} 종속성이 있는 작업의 결과를 가져오세요."
            )
```

이제 우리는 손(도구), 계획(청사진), 그리고 실행 추적(원시 데이터)을 갖추었습니다. 하지만 코드를 실행하고 종료하는 에이전트는 그저 스크립트에 불과합니다. 이를 파트너로 바꾸려면 이러한 기술적 결과를 인간의 통찰력으로 다시 번역할 "목소리", 즉 종합자(The Synthesizer)가 필요합니다.

## 4단계: 종합자 (목소리)

종합자(The Synthesizer)는 에이전트의 최종 품질 관리 역할을 합니다. 그 역할은 다음 세 가지를 검토하는 것입니다:

*   **원래 질문(The Original Question)**: 사용자가 실제로 무엇을 원했는가?
*   **계획(The Plan)**: 어떤 단계를 밟기로 결정했는가?
*   **실행 추적(The Execution Trace)**: 우리의 도구에서 나온 실제 출력값은 무엇인가(행성 질량, 계산 결과 등)?

5.972e24 kg와 같은 원시 데이터를 단순히 쏟아내는 대신, 종합자는 이를 자연스럽고 권위 있는 응답으로 엮어냅니다. 이는 데이터가 유용하게 사용될 수 있도록 "그래서 무엇인가?"에 대한 답을 제공합니다.

```python
    def __synthesize_answer(self, user_prompt, execution_results):
        synthesis_prompt = (
            "당신은 유용한 비서입니다. 당신은 사용자 질문과 다양한 도구에서 나온 실행 결과를 받았습니다. "
            "당신의 목표는 이러한 결과를 바탕으로 최종적이고 간결한 답변을 제공하는 것입니다."
        )

        # 우리는 이력을 LLM을 위한 단일 'context' 문자열로 결합합니다.
        context = f"사용자 질문: {user_prompt}\n결과: {execution_results}"
        return self.call_llm(synthesis_prompt, context)
```

## 루프 닫기: 이것이 왜 중요한가

플래너, 실행자, 종합자를 분리함으로써, 여러분은 기본적인 챗봇보다 훨씬 더 회복력(resilient) 있는 에이전트를 구축한 것입니다:

*   **감사 가능성(Auditability)**: 답변이 잘못된 경우, 플래너를 확인하여 로직에 결함이 있었는지, 또는 실행자를 확인하여 도구가 실패했는지 파악할 수 있습니다.
*   **토큰 효율성(Token Efficiency)**: LLM에게 매 초마다 전체 대화를 "다시 읽도록" 강요하는 것이 아니라, 특정 작업에 대한 특정 컨텍스트(context)를 제공합니다.
*   **사용자 신뢰(User Trust)**: 최종 종합은 사용자가 깔끔하게 다듬어진 답변을 받을 수 있도록 보장하며, 복잡한 JSON 위주의 중간 과정을 숨깁니다.

## 전문가 섹션: 엔터프라이즈 준비

AI 엔지니어링에서 가장 어려운 교훈 중 하나는 LLM이 예측 불가능하다는 것입니다. GPT-4조차도 때때로 잘못된 형식의 JSON을 반환하거나 존재하지 않는 매개변수를 "환각(hallucinate)"합니다. 실험실에서는 사소한 버그이지만, 엔터프라이즈(Enterprise) 환경에서는 시스템 장애로 이어집니다.

프로토타입에서 프로덕션 수준의 에이전트로 나아가려면, 여러 겹의 방어 계층이 필요합니다. 여기서는 시작을 위한 세 가지를 소개합니다:

### 1. 엄격한 스키마 유효성 검사 (Pydantic 도입)

LLM이 JSON을 올바르게 반환할 것이라고 전적으로 의존하지 마세요. 대신, Pydantic과 같은 라이브러리를 사용하여 코드 수준에서 "계약(contract)"을 강제하세요. 도구를 Pydantic 모델로 정의함으로써, LLM의 출력이 여러분의 함수에 도달하기 전에 자동으로 유효성을 검사할 수 있습니다.

```python
from pydantic import BaseModel, ValidationError

class PlanetQuery(BaseModel):
    planet: str
    include_moons: bool = False  # 기본값은 견고성을 더합니다.

# 만약 LLM이 "world" 대신 "planet"을 보낸다면, Pydantic이 즉시 이를 감지합니다.
```

더 자세한 정보는 OpenAI 문서: [Structured model outputs | OpenAI API](https://platform.openai.com/docs/guides/function-calling/structured-outputs)에서 찾을 수 있습니다. 제 코드 예제에서는 단순화를 위해 이를 구현하지 않았지만, 엔터프라이즈 환경에서는 강력히 권장됩니다.

### 2. "자가 수정" 재시도 루프 (Self-Correction Retry Loop)

저희 구현에서는 단순히 "잘 되기를 바라는" 것이 아니라, 실패에 대비해 구축합니다. 특정 오류 처리가 포함된 재시도 루프(Retry Loop)를 사용합니다. JSON 파싱이 실패하면 에이전트는 중단되지 않고 다시 시도합니다.

유효성 검사가 실패하면 단순히 중단하지 마세요. 오류 메시지 자체를 피드백으로 활용하세요. 오류를 LLM에게 다시 보내 "형식이 잘못되었습니다. 오류: [ValidationError]입니다. 다시 시도해 주세요."라고 말하세요.

```python
max_retries = 3
attempts = 0
success = False # 루프 조건에 필요한 success 플래그 초기화
while attempts < max_retries and not success:
    try:
        response = self.call_llm(execution_system_prompt, user_prompt)
        response_json = json.loads(response.output_text)
        success = True
    except (json.JSONDecodeError, KeyError):
        attempts += 1
```

이것이 중요한 이유: 이 3회 시도 패턴은 구조화된 워크플로의 신뢰성을 크게 높여, 에이전트를 단순한 실험실 프로젝트가 아닌 프로덕션(production) 준비 상태로 만듭니다.

### 3. 관찰 가능성 및 추적 (Observability & Tracing)

엔터프라이즈 환경에서 "왜 그랬는지 모르겠습니다"는 용납될 수 없는 답변입니다. 모든 생각(Thought), 행동(Action), 관찰(Observation)을 기록해야 합니다. 이 "실행 추적(Execution Trace)"은 에이전트의 의사결정 과정을 감사할 수 있게 해주며, 1만 달러의 트랜잭션이 트리거되었을 때 그 이면에 있는 추론에 대한 기록(paper trail)을 확보할 수 있도록 보장합니다.

## 실제 시나리오: 행성에서 수익까지

우리의 예시에서는 행성 질량과 간단한 덧셈을 사용했지만, 복잡한 엔터프라이즈 시스템에서도 아키텍처는 동일하게 유지됩니다. 프로덕션 환경에서는 에이전트의 로직을 변경하는 것이 아니라, 에이전트가 사용하는 도구만 교체하는 것입니다.

이 "계획-실행(Plan-and-Execute)" 패턴이 전문적인 워크플로를 어떻게 변화시키는지 생각해 보세요:

*   **데이터베이스 전문가**: `get_planet_mass()` 대신, 도구는 `query_sql_database()`가 됩니다. 플래너는 어떤 컬럼을 가져올지 결정하고, 실행자는 연결 문자열과 원시 데이터 검색을 처리합니다.
*   **금융 분석가**: `calculate()` 대신, 에이전트는 `analyze_market_trends(ticker)`를 사용합니다. API에서 실시간 데이터를 가져오고, 최근 뉴스에 대한 감성 분석을 수행하며, 위험 점수를 반환합니다.
*   **운영 관리자**: `update_inventory_levels()`와 같은 도구는 에이전트가 재고에 대해 "대화"하는 것을 넘어, 구매가 확정된 후 실제로 창고 데이터를 조정할 수 있게 합니다.

참고: 처음부터 구축하는 것의 강력한 장점은 보안 계층(Security Layer)을 직접 제어할 수 있다는 것입니다. 엔터프라이즈 환경에서 "실행자(Executor)"는 권한 확인을 포함할 수 있으며, 이를 통해 LLM이 권한 없는 도구를 절대 트리거하지 않도록 보장합니다.
