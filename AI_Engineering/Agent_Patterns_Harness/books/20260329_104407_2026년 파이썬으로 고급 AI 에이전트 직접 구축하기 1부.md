2026년에 AI 에이전트(AI Agent)를 배우는 가장 좋은 방법이 무엇이냐고 묻는다면, 저는 직접 처음부터 구축해보는 것이라고 단언할 것입니다. 이는 학습 목적뿐만 아니라, 고도로 효율적이고 개인화되며 견고한 프로덕션(production) 수준의 AI 에이전트를 구축하려는 경우에도 가장 좋은 선택입니다. 예를 들어, 현재 사용 가능한 코딩 에이전트(예: 클로드 코드(Claude Code), 코덱스(Codex), 커서(Cursor) 등)는 모두 제품에 특화된 맞춤형 아키텍처(architecture)로 구축되었습니다.

오해하지 마세요. 랭체인(LangChain), 랭그래프(LangGraph), 라마인덱스(LlamaIndex)와 같은 프레임워크(framework)는 RAG(Retrieval Augmented Generation)나 자동화된 워크플로우(workflow)와 같은 표준 작업에도 유용할 수 있습니다. 핵심은 실제 복잡한 작업에 라이브러리(library)를 사용하기로 결정하기 전에 해당 라이브러리의 기능과 한계를 명확히 인지해야 한다는 것입니다.

저는 여전히 랭그래프를 많이 사용하지만, 주로 프로토타이핑(prototyping) 단계에서 활용합니다. 데모(demo)를 만들거나 디자인 패턴(design pattern) 및 에이전트형 아키텍처(agentic architecture)를 가르치는 데 매우 효과적입니다.

이번 글과 다음 글에서는 필수 기능과 몇 가지 고급 기능을 갖춘 AI 에이전트를 단계별로 구축하는 방법을 보여드릴 것입니다. 또한 여러 상황에서 유용하게 활용될 수 있는 몇 가지 디자인 패턴을 구현하는 방법도 알려드릴 것입니다.

## AI 에이전트란 무엇인가요?

요즘에는 수많은 종류의 AI 에이전트가 있으며, 어디에서나 찾아볼 수 있습니다. 챗GPT(ChatGPT)와 같은 단순한 챗봇(chatbot)이었던 것들이 이제는 웹 검색, 추론 능력, 이미지 생성 등 다양한 도구를 활용할 수 있는 AI 에이전트가 되었습니다. 에이전트의 복잡성은 달성해야 하는 목표에 따라 달라집니다.

예를 들어, 웹사이트 방문자를 지원하는 고객 지원 에이전트는 정확하고 최신 정보에 기반한 응답을 위한 RAG(Retrieval Augmented Generation) 도구를 갖춘 챗봇일 수 있으며, 신뢰할 수 있는 답변을 찾을 수 없거나 문의가 사람의 개입을 필요로 할 때 자동으로 인간 지원팀에 이메일을 작성하는 또 다른 도구를 가질 수 있습니다.

본질적으로 AI 에이전트는 다음과 같은 기능을 수행할 수 있는 시스템입니다:

*   환경을 인지하고(사용자 입력 이해)
*   취할 행동을 추론하며
*   도구를 사용하거나 응답을 제공하여 행동하고
*   결과로부터 학습합니다(이 부분은 2부에서 다룰 예정입니다).

오늘 우리는 리액트(ReAct: Reasoning + Acting) 패턴을 사용하여 처음 세 가지 기능을 구현하는 기본적인 에이전트를 구축할 것입니다.

## 아키텍처 개요

코딩 부분으로 들어가기 전에, 에이전트가 어떻게 구성되는지 이해해봅시다. 우리는 세 가지 주요 구성 요소를 구축할 것입니다:

*   **도구 시스템(Tool System):** 사용 가능한 모든 도구를 관리하는 유연한 레지스트리(registry)
*   **LLM 래퍼(LLM Wrapper):** 언어 모델(language model)과 상호 작용하기 위한 추상화 계층
*   **에이전트 오케스트레이터(Agent Orchestrator):** 모든 것을 조율하는 두뇌

이러한 분리가 중요한 이유는 다음과 같습니다:

**도구 추상화(Tool Abstraction):** 도구 레지스트리를 생성함으로써 핵심 로직을 수정하지 않고도 에이전트에 새로운 기능을 쉽게 추가할 수 있습니다. 데이터베이스(database) 쿼리(query) 함수가 필요하신가요? 새로운 도구를 등록하기만 하면 됩니다. 이는 확장성(extensibility) 원칙이 실제로 적용되는 것이며, 생각할 수 있는 모든 에이전트에서 사용됩니다.

**LLM/에이전트 분리(LLM/Agent Separation):** 이는 프로덕션 시스템에 매우 중요합니다. 에이전트는 오케스트레이터로서 대화 흐름을 관리하고, 도구를 호출할 시기를 결정하며, 전체 워크플로우를 처리합니다. LLM은 추론을 제공하는 구성 요소일 뿐이며, 에이전트의 두뇌와 같지만, 이 경우에는 필요할 때마다 두뇌를 쉽게 교체할 수 있어야 합니다.

따라서 이들을 분리함으로써:

*   에이전트 로직을 다시 작성하지 않고도 다양한 LLM 제공업체(제미니(Gemini), 오픈AI(OpenAI), 클로드(Claude)) 간에 전환할 수 있습니다.
*   한 제공업체가 실패할 경우 대체 전략(fallback strategies)을 구현할 수 있습니다.
*   다양한 작업에 다른 모델을 사용하여 비용을 최적화할 수 있습니다.
*   LLM을 독립적으로 모의(mock)할 수 있으므로 테스트가 더 쉬워집니다.

## 1단계: 도구 시스템 구축

기반이 되는 도구 시스템부터 시작하겠습니다. 도구는 에이전트가 외부 세계와 상호 작용할 수 있도록 하는 손과 발입니다.

### 도구 클래스(Tool Class)

먼저, 개별 도구를 나타내는 방법이 필요합니다:

```python
from typing import Dict, List, Callable, Any

class Tool:
    def __init__(
        self,
        name: str,
        description: str,
        input_schema: Dict[str, Any],
        output_schema: Dict[str, Any],
        func: Callable[..., Any],
    ):
        self.name = name
        self.description = description
        self.input_schema = input_schema
        self.output_schema = output_schema
        self.func = func
    def __call__(self, **kwargs):
        return self.func(**kwargs)
```

각 도구는 다섯 가지 핵심 구성 요소를 가집니다:

*   `name`: 고유 식별자
*   `description`: 도구가 수행하는 작업(LLM이 언제 사용할지 이해하는 데 중요함)
*   `input_schema`: 도구가 예상하는 매개변수 정의
*   `output_schema`: 도구가 반환하는 값
*   `func`: 실제 작업을 수행하는 함수

`__call__` 메서드는 `Tool` 인스턴스를 호출 가능하게 만들어 일반 함수처럼 사용할 수 있게 합니다: `tool(a=5, b=3)`.

### 도구 레지스트리(Tool Registry)

이제 모든 도구를 관리할 중앙 장소가 필요합니다.

```python
from typing import Union, Literal
from pydantic import BaseModel

class ToolRegistry:
    def __init__(self):
        self.tools: Dict[str, Tool] = {}
    def register(self, tool: Tool):
        self.tools[tool.name] = tool
    def get(self, name: str) -> Tool:
        if name not in self.tools.keys():
            raise ValueError(f"Tool '{name}' not found")
        return self.tools[name]
    def list_tools(self) -> List[Dict[str, Any]]:
        return [
            {
                "name": tool.name,
                "description": tool.description,
                "input_schema": tool.input_schema.model_json_schema(),
            }
            for tool in self.tools.values()
        ]
    def get_tool_call_args_type(self) -> Union[BaseModel]:
        input_args_models = [tool.input_schema for tool in self.tools.values()]
        tool_call_args = Union[tuple(input_args_models)]
        return tool_call_args
    def get_tool_names(self) -> Literal[None]:
        return Literal[*self.tools.keys()]
```

레지스트리(registry)는 사용 가능한 모든 기능의 중앙 카탈로그(catalog) 역할을 합니다. 주로 도구를 등록하고 검색하는 데 사용됩니다.

#### `list_tools()` 메서드: LLM에게 무엇을 할 수 있는지 알려주기

이 메서드는 사용 가능한 모든 도구의 기계 판독 가능한(machine-readable) 설명을 생성하므로 특히 중요합니다. 이 설명을 시스템 프롬프트(system prompt)에서 LLM에 전달하면, LLM은 어떤 기능에 접근할 수 있는지 학습합니다. 이 메서드는 다음과 같은 것을 반환합니다:

```json
[
  {
    "name": "add",
    "description": "Add two numbers",
    "input_schema": {
      "type": "object",
      "properties": {
        "a": {"type": "integer"},
        "b": {"type": "integer"}
      },
      "required": ["a", "b"]
    }
  },
  {
    "name": "multiply",
    "description": "Multiply two numbers",
    "input_schema": {...}
  }
]
```

이 JSON 스키마(schema)는 LLM에게 각 도구를 호출하는 방법을 정확히 알려줍니다. 이것이 없으면 LLM은 존재하지 않는 도구 이름을 환각(hallucinate)하거나 잘못된 형식으로 인자(argument)를 제공할 수 있습니다.

#### `get_tool_call_args_type()` 메서드: 런타임 유효성 검사

이 메서드는 가능한 모든 도구 인자(argument) 스키마의 유니온(Union) 타입을 생성합니다. 파이썬(Python) 타입 힌팅(typing)에서 `Union`은 "이 타입들 중 하나"를 의미합니다. 따라서 두 개의 도구가 있다면 `Union[ToolAddArgs, ToolMultiplyArgs]`와 같이 생성됩니다.

왜 이것이 중요할까요? LLM이 도구 호출로 응답할 때, 파이단틱(Pydantic)은 인자가 이러한 스키마 중 하나와 일치하는지 검증합니다. LLM이 `{"a": "five", "b": 3}` (정수 대신 문자열)을 전달하려고 하면, 도구가 실행되기 전에 파이단틱이 이를 잡아냅니다. 이는 런타임(runtime) 오류를 방지하고 명확한 피드백을 제공합니다.

**참고:** 우리는 파이단틱을 LLM API(Application Programming Interface)의 도구 호출과 구조화된 출력(structured output)을 처리하는 표준으로 사용하고 있습니다. 하지만 간단한 JSON(JavaScript Object Notation) 형식을 사용하려면 파이단틱 모델을 JSON으로 쉽게 변환할 수 있습니다.

#### `get_tool_names()` 메서드: 환각 방지

이 메서드는 유효한 도구 이름만 포함하는 리터럴(Literal) 타입을 생성합니다: `Literal["add", "multiply"]`. 이는 강력한 제약 조건으로, LLM은 레지스트리에 실제로 존재하는 도구 이름만 반환할 수 있습니다.

이것이 없으면 LLM은 존재하지 않는 'divide'라는 도구를 자신 있게 호출할 수 있습니다. 구조화된 출력과 리터럴 타입을 사용하면 LLM은 허용된 세트(set) 내에서 선택하도록 강제됩니다. 유효하지 않은 이름을 사용하려고 하면 API가 응답을 거부하고 모델이 유효한 도구 이름으로 다시 시도하도록 강제합니다.

이 세 가지 메서드는 함께 LLM의 확률론적 세계와 파이썬 코드의 결정론적 세계 사이의 간극을 메우는 견고한 타입 안전(type-safety) 시스템을 만듭니다. 이는 모호한 요청을 검증되고 실행 가능한 함수 호출로 전환합니다.

이제 도구 추상화 클래스와 레지스트리를 사용하여 새로운 도구를 생성하고 등록하는 방법을 살펴보겠습니다.

### 첫 번째 도구 등록하기

시스템을 시연하기 위해 두 가지 간단한 도구를 만들어봅시다:

```python
def add(a: int, b: int) -> int:
    return a + b

def multiply(a: int, b: int) -> int:
    return a * b
```

이들은 실행 중에 호출될 도구 함수입니다.

이제 파이단틱이 등장합니다. 각 도구의 입력에 대한 스키마를 정의합니다:

```python
class ToolAddArgs(BaseModel):
    a: int
    b: int

class ToolMultiplyArgs(BaseModel):
    a: int
    b: int
```

그런 다음 도구 레지스트리를 인스턴스화하고 새로 생성한 도구를 추가합니다.

```python
registry = ToolRegistry()

registry.register(
    Tool(
        name="add",
        description="Add two numbers",
        input_schema=ToolAddArgs,
        output_schema={"result": "int"},
        func=add,
    )
)
registry.register(
    Tool(
        name="multiply",
        description="Multiply two numbers",
        input_schema=ToolMultiplyArgs,
        output_schema={"result": "int"},
        func=multiply,
    )
)
```

## 2단계: 파이단틱을 사용한 타입 안전성

궁금할 수 있습니다: "왜 일반 딕셔너리(dictionary)나 JSON 대신 파이단틱을 사용하나요?" 좋은 질문입니다. 이는 구조화된 출력(structured output)과 타입 안전성(type safety)에 관한 문제입니다.

LLM과 작업할 때 가장 큰 과제 중 하나는 코드가 안정적으로 처리할 수 있는 형식으로 데이터를 반환하도록 보장하는 것입니다. 오늘날 2026년에는 도구 사용에 대해 고도로 훈련된 신뢰할 수 있는 LLM이 있지만, 환각(hallucination)은 여전히 해결되지 않은 문제입니다. 이것이 우리가 타입 및 구조 유효성 검사(validation)가 필요한 이유입니다.

파이단틱 모델은 계약 역할을 합니다. 다음과 같은 기능을 제공합니다:

*   들어오는 데이터를 자동으로 검증합니다.
*   데이터가 유효하지 않을 때 명확한 오류 메시지를 제공합니다.
*   더 나은 개발자 경험을 위해 IDE 자동 완성(autocomplete)을 가능하게 합니다.
*   최신 LLM이 구조화된 출력에 사용할 수 있는 JSON 스키마를 생성합니다.

에이전트가 취할 수 있는 가능한 행동을 정의해봅시다:

```python
# Get type-safe tool names and arguments
ToolNameLiteral = registry.get_tool_names()
ToolArgsUnion = registry.get_tool_call_args_type()

class ToolCall(BaseModel):
    action: Literal["tool"]
    thought: str
    tool_name: ToolNameLiteral
    args: ToolArgsUnion
class FinalAnswer(BaseModel):
    action: Literal["final"]
    answer: str
LLMResponse = Union[ToolCall, FinalAnswer]
```

이 구조는 리액트(ReAct) 패턴을 강제합니다. LLM은 다음을 수행해야 합니다:

*   행동 유형("tool" 또는 "final")을 선택합니다.
*   도구를 호출하는 경우: 사고 과정(thought process), 도구 이름, 유효한 인자(argument)를 제공합니다.
*   최종 답변을 제공하는 경우: 답변 텍스트를 제공합니다.

`ToolNameLiteral`은 LLM이 실제로 존재하는 도구만 호출하도록 보장합니다. `ToolArgsUnion`은 어떤 도구가 호출되든 인자가 예상 스키마와 일치하도록 보장합니다.

## 3단계: LLM 래퍼(LLM Wrapper)

이제 구글의 제미니(Gemini) API와 통합합니다. 튜토리얼에서는 항상 무료 티어(tier) API를 제공하는 제미니를 사용하는 것을 선호합니다. 하지만 다른 API 서비스 제공업체를 사용할 수도 있으며, 필요한 것은 API 문서에 따라 이 클래스를 수정하는 것뿐입니다.

```python
import json
from google import genai
from google.genai import types

class GeminiLLM:
    def __init__(self, client, tool_registry, model="gemini-2.5-flash"):
        self.client = client
        self.model = model
        self.tool_registry = tool_registry
        self.system_instruction = self._create_system_instruction()
```

### 시스템 프롬프트(System Prompt)

시스템 프롬프트는 에이전트에게 어떻게 행동해야 하는지 가르치는 부분입니다. 여기서는 간단한 시스템 프롬프트를 사용했지만, 실제 제품에서는 훨씬 더 상세해질 수 있습니다. 이 부분이 가장 중요한 부분 중 하나입니다:

```python
def _create_system_instruction(self) -> str:
    tools_description = json.dumps(
        self.tool_registry.list_tools(),
        indent=2
    )

    system_prompt = """
You are a conversational AI agent that can interact with external tools.
CRITICAL RULES (MUST FOLLOW):
- You are NOT allowed to perform operations internally that could be performed by an available tool.
- If a tool exists that can perform any part of the task, you MUST use that tool.
- You MUST NOT skip tools, even for simple or obvious steps.
- You MUST NOT combine multiple operations into a single step unless a tool explicitly supports it.
- You may ONLY produce a final answer when no available tool can further advance the task.
TOOL USAGE RULES:
- Each tool call must perform exactly ONE meaningful operation.
- If the task requires multiple operations, you MUST call tools sequentially.
- If multiple tools could apply, choose the most specific one.
RESPONSE FORMAT (STRICT):
- You MUST respond ONLY in valid JSON.
- Never include explanations outside JSON.
- You must choose exactly one action per response.
Tool call format:
{
  "action": "tool",
  "thought": "...",
  "tool_name": "...",
  "inputs": { ... }
}
Final answer format:
{
  "action": "final",
  "answer": "..."
}""" + "\\n\\nAvailable tools with description:\\n" + tools_description
    return system_prompt
```

이 규칙들이 왜 그렇게 엄격할까요?

LLM은 도움이 되도록 훈련되었으며, 종종 내부적으로 수학 계산이나 추론을 수행하여 "도움을 주려고" 시도합니다. 그러나 우리는 에이전트가 관찰 가능하고(observable) 신뢰할 수 있기를 원합니다. 모든 작업에 도구를 사용하도록 강제함으로써:

*   각 단계를 기록하고 디버깅할 수 있습니다.
*   에이전트를 변경하지 않고 도구 구현을 교체할 수 있습니다.
*   도구를 독립적으로 테스트할 수 있습니다.
*   명확한 감사 추적(audit trail)을 유지할 수 있습니다.

이것이 바로 리액트(ReAct) 패턴의 핵심입니다: 명시적인 추론("thought") 뒤에 명시적인 행동("tool_name" + "args")이 따릅니다.

### 제미니를 위한 채팅 히스토리(Chat History) 형식 지정

다양한 LLM 제공업체는 서로 다른 메시지 형식을 예상합니다. 여기서는 일반적인 히스토리(history)를 제미니 형식으로 변환하는 방법을 보여줍니다:

```python
def _format_gemini_chat_history(self, history: list[dict]) -> list:
    formatted_history = []
    for message in history:
        if message["role"] == "user":
            formatted_history.append(types.Content(
                    role="user",
                    parts=[
                        types.Part.from_text(text=message["content"])
                    ]
                )
            )
        if message["role"] == "assistant":
            formatted_history.append(types.Content(
                    role="model",
                    parts=[
                        types.Part.from_text(text=message["content"])
                    ]
                )
            )
        if message["role"] == "tool":
            formatted_history.append(types.Content(
                    role="tool",
                    parts=[
                        types.Part.from_function_response(
                            name=message["tool_name"],
                            response={'result': message["tool_response"]},
                        )
                    ]
                )
            )
    return formatted_history
```

이 추상화(abstraction)가 핵심입니다. 우리 에이전트는 간단하고 제공업체에 구애받지 않는 메시지 형식으로 작동합니다. 각 LLM 래퍼(wrapper)는 자체 형식화 특성을 처리합니다.

### 구조화된 출력으로 응답 생성

마지막으로, 구조화된 출력(structured output)이 활성화된 LLM을 호출합니다:

```python
def generate(self, history: list[dict]) -> str:
    gemini_history_format = self._format_gemini_chat_history(history)
    response = self.client.models.generate_content(
        model=self.model,
        contents=gemini_history_format,
        config=types.GenerateContentConfig(
            temperature=0,
            response_mime_type="application/json",
            response_schema=LLMResponse,
            system_instruction=self.system_instruction,
            automatic_function_calling=types.AutomaticFunctionCallingConfig(disable=True)
        ),
    )
    return response.text
```

주요 매개변수:

*   `temperature=0`: 결정론적이고 일관된 동작을 원합니다.
*   `response_mime_type="application/json"`: JSON 출력을 강제합니다.
*   `response_schema=LLMResponse`: 유효성 검사를 위해 파이단틱 모델을 사용합니다.
*   `automatic_function_calling` 비활성화: 도구 실행에 대한 수동 제어를 원합니다.

## 4단계: 에이전트 오케스트레이터(Agent Orchestrator)

이제 모든 것을 통합합니다. 에이전트는 대화 루프(loop)를 관리하는 오케스트레이터입니다:

```python
class Agent:
    def __init__(self, llm, tool_registry, max_steps=5):
        self.llm = llm
        self.tool_registry = tool_registry
        self.history = []
        self.max_steps = max_steps
```

`max_steps` 매개변수는 에이전트가 멈췄을 때 무한 루프를 방지하는 안전 메커니즘입니다.

### 리액트(ReAct) 루프

여기서 마법이 일어납니다:

```python
def run(self, user_input: str):
    self.history.append({"role": "user", "content": user_input})
    for step in range(self.max_steps):
        # Get LLM decision
        llm_output = self.llm.generate(self.history)
        action = json.loads(llm_output)
        if action["action"] == "tool":
            # Record the thought process
            self.history.append(
                {"role": "assistant", "content": llm_output}
            )
            # Execute the tool
            tool = self.tool_registry.get(action["tool_name"])
            result = tool(**action["args"])
            # Record the result
            observation = f"Tool {tool.name} returned: {result}"
            self.history.append(
                {"role": "tool", "tool_name": tool.name, "tool_response": result}
            )
            continue
        if action["action"] == "final":
            self.history.append(
                {"role": "assistant", "content": llm_output}
            )
            return action["answer"]
    raise RuntimeError("Agent did not terminate within max_steps")
```

이 루프를 분석해봅시다:

*   **사용자 입력이 히스토리에 추가됩니다:** 에이전트는 컨텍스트(context)가 필요합니다.
*   **LLM이 결정을 생성합니다:** 전체 대화를 기반으로 합니다.
*   **도구 호출인 경우:**
    *   결정("thought")을 기록합니다.
    *   도구를 실행합니다.
    *   결과("observation")를 기록합니다.
    *   다음 반복으로 진행합니다.
*   **최종 답변인 경우:** 작업이 완료됩니다!
*   **안전 점검:** `max_steps`에 도달하면 오류를 발생시킵니다.

## 5단계: 모든 것을 통합하기

모든 것을 초기화하고 채팅 인터페이스를 만들어봅시다:

```python
from google import genai

# Initialize the client (you'll need your API key)
client = genai.Client(api_key=GEMINI_API_KEY)
# Create LLM and Agent
llm = GeminiLLM(client, registry)
agent = Agent(llm, registry)
def chat_with_agent(agent: Agent):
    print("Welcome! Type 'exit' to quit.\n")
    while True:
        user_input = input("You: ")
        if user_input.lower() in ["exit", "quit", "q"]:
            print("Goodbye!")
            break
        try:
            response = agent.run(user_input)
            print(f"Agent: {response}")
        except RuntimeError as e:
            print(f"Agent error: {e}")
        except Exception as e:
            print(f"Unexpected error: {e}")
# Start chatting
chat_with_agent(agent)
```

## 예시: 에이전트 동작 살펴보기

다음 질문을 했을 때 어떤 일이 일어나는지 살펴봅시다: "5 더하기 3은 얼마인가요? 그리고 그 결과에 2를 곱하세요."

**1단계:** LLM이 질문을 받고 다음과 같이 응답합니다:

```json
{
  "action": "tool",
  "thought": "I need to first add 5 and 3",
  "tool_name": "add",
  "args": {"a": 5, "b": 3}
}
```

**2단계:** 에이전트가 `add(5, 3)`을 실행 → 8을 반환합니다.

**3단계:** LLM이 결과를 보고 다음과 같이 응답합니다:

```json
{
  "action": "tool",
  "thought": "Now I need to multiply 8 by 2",
  "tool_name": "multiply",
  "args": {"a": 8, "b": 2}
}
```

**4단계:** 에이전트가 `multiply(8, 2)`를 실행 → 16을 반환합니다.

**5단계:** LLM이 다음과 같이 응답합니다:

```json
{
  "action": "final",
  "answer": "The result is 16"
}
```

에이전트가 작업을 개별적인 단계로 분해하고, 각 작업에 도구를 사용하며, 명확한 설명을 제공하는 방식에 주목하세요. 이러한 투명성은 AI 에이전트를 디버깅 가능하고 신뢰할 수 있게 만듭니다.

## 이 아키텍처가 중요한 이유

이것이 "간단한 계산기에 너무 많은 상투적인 코드(boilerplate)처럼 보인다"고 생각할 수 있습니다. 그리고 맞는 말입니다! 하지만 이 기반이 강력한 이유는 다음과 같습니다:

### 1. 확장성(Extensibility)

날씨 API(API)를 추가하고 싶으신가요? 함수를 만들고 등록하기만 하면 됩니다:

```python
class WeatherArgs(BaseModel):
    city: str

def get_weather(city: str) -> str:
    # API call here
    return f"Weather in {city}: Sunny, 72°F"
registry.register(Tool(
    name="get_weather",
    description="Get current weather for a city",
    input_schema=WeatherArgs,
    output_schema={"weather": "str"},
    func=get_weather
))
```

에이전트 로직을 변경할 필요가 없습니다. LLM은 시스템 프롬프트로부터 새로운 도구에 대해 자동으로 학습합니다.

### 2. 제공업체 유연성

제미니에서 오픈AI(OpenAI)로 전환해야 하나요? 동일한 인터페이스를 구현하는 `OpenAILLM` 클래스를 만드세요:

```python
class OpenAILLM:
    def __init__(self, client, tool_registry, model="gpt-4"):
        # Similar structure, different API calls
        pass
    def generate(self, history: list[dict]) -> str:
            # OpenAI-specific implementation
            pass
# Swap it in
llm = OpenAILLM(openai_client, registry)
agent = Agent(llm, registry)  # 나머지 모든 것은 동일하게 유지됩니다!
```

### 3. 테스트 용이성

각 구성 요소를 독립적으로 테스트할 수 있습니다:

*   도구를 개별적으로 테스트합니다.
*   에이전트 로직 테스트를 위해 LLM을 모의(mock)합니다.
*   전체 흐름을 종단 간(end-to-end)으로 검증합니다.

### 4. 관찰 가능성(Observability)

모든 단계는 히스토리에 기록됩니다. 다음을 수행할 수 있습니다:

*   디버깅을 위해 모든 도구 호출을 기록합니다.
*   어떤 도구가 가장 많이 사용되는지 분석합니다.
*   에이전트가 어려움을 겪는 부분을 식별합니다.
*   분석 대시보드를 구축합니다.

## 구축한 것

이 글에서는 다음과 같은 특징을 가진 기본적인 AI 에이전트를 만들었습니다:

*   모듈식 도구 시스템(Modular tool system).
*   타입 안전성을 갖춘 구조화된 출력(Type-safe structured outputs).
*   제공업체에 구애받지 않는 LLM 통합(Provider-agnostic LLM integration).
*   리액트(ReAct) 추론 패턴.
*   명확한 관심사 분리(Clear separation of concerns).

하지만 이것은 시작에 불과합니다. 저희 에이전트는 상태가 없고(대화 간에 메모리가 없음), 인간의 감독이 없으며, 제한적인 관찰 가능성만 제공합니다.
