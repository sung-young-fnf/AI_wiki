저는 꽤 오랫동안 Claude 스택 (stack) 에서 작업해왔습니다. Airtable 및 Notion에 연결된 MCP (Model Context Protocol) 서버 (server), 사용자 지정 스킬 (skill) 및 `CLAUDE.md` 계층 구조 (hierarchy) 로 팀 프로젝트 전반에 구성된 Claude Code, 콘텐츠 추출 및 연구를 위한 다중 에이전트 파이프라인 (pipeline) 등을 다루었습니다. 오케스트레이션 (orchestration) 실패를 디버깅 (debug) 했고, 컨텍스트 (context) 격리 문제에 부딪혔으며, 제가 게으른 설명 (description) 을 작성하여 도구 (tool) 가 잘못 라우팅 (misroute) 되는 것을 목격했습니다.

Anthropic이 Claude Certified Architect 시험 가이드를 출시했을 때, 저는 그 안에 있는 모든 개념 (concept) 을 알아보았습니다. 대부분은 이미 제가 잘못 이해하여 문제를 일으켰기 때문입니다.

이 시험은 Claude 파트너 (partner) 에게만 응시 자격이 주어집니다. 파트너가 아니면 응시할 수 없습니다. 하지만 이 가이드는 공개되어 있으며, 프로덕션 (production) Claude 개발에 필요한 것과 직접적으로 연관되는 다섯 가지 도메인 (domain) 을 다룹니다.

저는 전체 가이드를 검토하고, 알아둘 가치가 있는 모든 개념을 뽑아내고, 가장 중요한 패턴 (pattern) 에 대해 실행 가능한 코드 (code) 를 작성했으며, 각 도메인에 대해 Claude를 해당 주제의 튜터 (tutor) 로 바꿔주는 복사-붙여넣기 프롬프트 (prompt) 를 만들었습니다.

이것은 제가 Claude 스택에서 에이전트를 구축하기 시작했을 때 있었으면 하고 바랐던 가이드입니다.

저는 매주 The AI Engineer에서 이러한 개념들을 분석합니다. 이곳에서 소프트웨어 (software) 엔지니어는 매우 뛰어난 AI 엔지니어가 됩니다. 무료로 구독하세요.

## 다섯 가지 도메인 (The Five Domains)
전체 크기 이미지 보려면 Enter 키를 누르거나 클릭하세요.

시험 형식은 시나리오 (scenario) 기반의 객관식입니다. 정답은 하나이고, 그럴듯한 오답이 세 개 있습니다. 합격 점수는 720/1000점입니다.

모든 도메인을 살펴보면서 한 가지 주목한 점은 다음과 같습니다. 시험은 위험이 높을 때 확률적 해결책보다 결정론적 해결책을 일관되게 보상합니다. 증상을 패치 (patch) 하는 것보다 근본 원인 (root cause) 을 해결하는 것을 보상합니다. 그리고 가장 정교한 방법보다는 실제로 문제를 해결하는 가장 간단한 개입 (intervention) 을 보상합니다.

이러한 철학은 실제로도 유효합니다.

### 도메인 1: 에이전트 아키텍처 및 오케스트레이션 (Agentic Architecture & Orchestration) (27%)

이것은 가장 비중이 큰 도메인이며, 그럴 만한 이유가 있습니다. 저는 사람들이 오케스트레이션 (orchestration) 을 '그냥 에이전트들을 연결하면 돼'라고 대충 넘기는 것을 싫어합니다. 오케스트레이션이 망가지면 다운스트림 (downstream) 의 모든 것이 함께 망가집니다.

#### 1) 에이전트 루프 (The Agentic Loop)

모든 Claude 에이전트는 동일한 루프 (loop) 를 따릅니다. 요청을 보내고, `stop_reason` 필드를 확인한 다음, 요청된 도구를 실행하거나 중지합니다.

```python
import anthropic
client = anthropic.Anthropic()
messages = [{"role": "user", "content": "Your task here"}]
while True:
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=4096,
        tools=tools,
        messages=messages
    )
    # CORRECT: check stop_reason, not content type
    if response.stop_reason == "end_turn":
        break
    if response.stop_reason == "tool_use":
        tool_results = execute_tools(response.content)
        messages.append({"role": "assistant", "content": response.content})
        messages.append({"role": "user", "content": tool_results})
    # WRONG: this kills your agent mid-task
    # if response.content[0].type == "text":
    #     break  # Claude returns text AND tool_use in the same response
```

시험에서는 세 가지 안티패턴 (anti-pattern) 을 테스트하며, 이 세 가지 모두 실제 코드베이스 (codebase) 에서 나타납니다.

1.  **완료를 감지하기 위해 자연어 (natural language) 를 파싱 (parsing) 하는 것.** 모델이 "다 됐어" 또는 "여기 답이 있어"라고 말했는지 확인하는 것입니다. `stop_reason` 필드는 바로 이 목적을 위해 존재합니다. 이를 사용하세요.
2.  **임의의 반복 상한 (iteration caps).** `max_loops = 10`으로 설정하면 12단계 작업을 일찍 중단시키거나, 6단계 작업을 4번 불필요하게 반복할 수 있습니다. 모델은 작업이 완료되었을 때 신호를 보냅니다.
3.  **콘텐츠 유형 (content type) 을 완료 지표로 확인하는 것.** 이것은 미묘한 문제입니다. `if response.content[0].type == "text"`는 합리적으로 보이지만, Claude는 동일한 응답에서 텍스트와 `tool_use` 블록 (block) 을 함께 반환할 수 있습니다. 에이전트에게 고객을 찾아 기록을 업데이트하라고 요청합니다. Claude는 텍스트 블록("고객을 찾아보겠습니다")과 `tool_use` 블록(실제 검색 호출)으로 응답합니다. 코드는 텍스트를 먼저 보고 에이전트가 완료되었다고 판단하여 검색을 실행하지 않습니다. 에이전트는 요청을 인지한 것처럼 보이지만 아무것도 하지 않습니다.

#### 2) 다중 에이전트 오케스트레이션 (Multi-Agent Orchestration)

아키텍처 (architecture) 는 허브 앤 스포크 (hub-and-spoke) 방식입니다. 코디네이터 (coordinator) 에이전트가 중심에 있고, 하위 에이전트 (subagent) 가 전문화된 작업을 처리합니다. 모든 통신은 코디네이터를 통해 이루어집니다. 하위 에이전트는 서로 직접 대화하지 않습니다.

다중 에이전트 시스템에서 가장 흔한 실수는 하위 에이전트가 코디네이터와 메모리 (memory) 를 공유하지 않는다는 것입니다. 대화 기록을 상속받지 않으며, 호출 간에 아무것도 유지하지 않습니다. 합성 에이전트가 웹 검색 결과를 필요로 한다면, 해당 결과를 합성 에이전트의 프롬프트 (prompt) 에 명시적으로 전달해야 합니다. 암묵적 컨텍스트 공유 (implicit context sharing) 는 없습니다. 저는 합성 에이전트가 출처 표기 (source attribution) 가 전혀 없는 보고서를 계속 생성했을 때 이 점을 깨달았습니다. "이것은 어디서 온 거야?"라고 물으면 어디에서도 답을 찾을 수 없을 때까지는 모든 것이 올바르게 보였습니다.

코디네이터가 광범위한 쿼리 (query) 를 너무 좁게 분해 (decompose) 하면, 전체 파이프라인은 불완전하지만 완전해 보이는 출력을 생성합니다. 예를 들어, 다중 에이전트 시스템이 "재생 에너지 (renewable energy)"에 대한 연구 보고서를 생성했는데, 태양광 및 풍력만을 다룹니다. 근본 원인은 검색 에이전트(지시받은 것을 찾음)나 합성 에이전트(받은 것을 합성함)에 있지 않습니다. 근본 원인은 코디네이터의 분해에 있습니다. 코디네이터가 범위를 태양광 및 풍력으로만 결정한 것입니다. 다운스트림의 모든 것이 잘못된 계획에 따라 올바르게 실행된 것입니다.

#### 3) 훅 (Hook) vs 프롬프트 (Prompt)

이것은 프로토타입 (prototype) 에이전트와 프로덕션 (production) 에이전트를 구분하는 개념입니다.

**프롬프트 기반 가이드 (Prompt-based guidance)**는 시스템 프롬프트 (system prompt) 에 지침을 넣는 것을 의미합니다. "환불을 처리하기 전에 항상 고객의 신원을 확인하십시오." 이것은 대부분의 경우에 작동합니다. 하지만 항상 작동하는 것은 아닙니다.

**프로그래밍 방식의 강제 (Programmatic enforcement)**는 선행 조건이 완료될 때까지 다운스트림 도구를 물리적으로 차단하는 훅 (hook) 을 사용하는 것을 의미합니다. 이것은 항상 작동합니다.

**결정 규칙:** 단 한 번의 실패가 비용을 발생시키거나, 보안 사고를 유발하거나, 규정 준수 요구 사항을 위반하는 경우 프로그래밍 방식의 강제를 사용합니다. 그 외의 모든 경우(형식 선호도, 스타일 가이드라인)에는 프롬프트 기반 가이드도 괜찮습니다.

Agent SDK의 `PreToolUse` 훅이 어떻게 보이는지 살펴보세요.

```python
from claude_agent_sdk import ClaudeAgentOptions, ClaudeSDKClient, HookMatcher
async def block_large_refunds(input_data, tool_use_id, context):
    tool_input = input_data.get("tool_input", {})
    if tool_input.get("amount", 0) > 500:
        return {
            "hookSpecificOutput": {
                "hookEventName": "PreToolUse",
                "permissionDecision": "deny",
                "permissionDecisionReason": "Refunds over $500 require manager approval",
            }
        }
    return {}
options = ClaudeAgentOptions(
    hooks={
        "PreToolUse": [
            HookMatcher(matcher="process_refund", hooks=[block_large_refunds]),
        ],
    }
)
```

`PostToolUse` 훅은 도구 실행 후 모델이 결과를 보기 전에 이를 가로챕니다. 데이터 형식 (data format) 을 정규화하거나, 파일 변경 사항을 로깅 (log) 하거나, 모델이 필요 없는 필드 (field) 를 제거할 수 있습니다.

```python
async def log_file_changes(input_data, tool_use_id, context):
    file_path = input_data.get("tool_input", {}).get("file_path", "unknown")
    print(f"Modified: {file_path}")
    return {}
options = ClaudeAgentOptions(
    hooks={
        "PostToolUse": [
            HookMatcher(matcher="Edit|Write", hooks=[log_file_changes]),
        ],
    }
)
```

#### 4) 작업 분해 (Task Decomposition)

알아둘 가치가 있는 두 가지 패턴 (pattern) 입니다.

**고정 순차 파이프라인 (Fixed sequential pipeline)**은 예측 가능한 작업에 잘 작동합니다. 각 파일을 개별적으로 분석한 다음, 파일 간 통합 패스 (pass) 를 실행합니다. 단계를 미리 알고 있는 경우입니다.

**동적 적응형 분해 (Dynamic adaptive decomposition)**는 문제의 형태를 예측할 수 없는 개방형 조사에 작동합니다. 에이전트가 각 단계에서 발견한 내용을 기반으로 하위 작업 (subtask) 을 생성합니다.

**주의 분산 문제 (attention dilution problem)**는 실재합니다. 14개 이상의 파일을 한 번에 처리하면 일부 파일에 대한 상세한 피드백 (feedback) 이 생성되지만, 다른 파일에서는 명백한 버그 (bug) 를 놓칩니다(모델이 한 파일에서 특정 패턴을 문제점으로 지적하고도 세 파일 뒤에 동일한 코드를 승인하는 경우가 발생합니다). 해결책은 **다중 패스 아키텍처 (multi-pass architecture)**입니다. 일관된 깊이를 위해 파일별 분석을 수행한 다음, 별도의 파일 간 통합 패스를 수행합니다.

#### 5) 세션 관리 (Session Management)

세 가지 옵션이 있습니다. `--resume`은 이전 컨텍스트 (context) 가 여전히 유효할 때 명명된 세션 (session) 을 계속합니다. `fork_session`은 발산하는 접근 방식을 탐색하고 싶을 때 공유된 기준선 (baseline) 에서 독립적인 브랜치 (branch) 를 생성합니다. 도구 결과가 오래되었거나 긴 세션으로 인해 컨텍스트가 저하되었을 때 요약 주입을 통한 새로운 시작 (fresh start with summary injection) 이 작동합니다.

**한 가지 함정:** 3개의 파일을 수정한 후 세션을 재개합니다. 에이전트가 실제 코드를 더 이상 반영하지 않는 캐시 (cache) 된 도구 결과로 추론하기 때문에 모순된 조언을 제공합니다. 이 경우, 에이전트에게 특정 파일 변경 사항을 알려서 표적 재분석을 수행하거나, 새로 시작하여 이전에 발견한 내용에 대한 요약을 주입해야 합니다.

#### 🤖 도메인 1 튜터 프롬프트 (Tutor Prompt)

이것을 Claude에 붙여넣으세요. 전체 도메인에 대한 당신의 튜터가 됩니다.

당신은 Claude Certified Architect 시험의 도메인 1(에이전트 아키텍처 및 오케스트레이션)을 가르치는 전문 강사입니다. 이 도메인은 총 시험 점수의 27%를 차지합니다.

백보드 (whiteboard) 에 있는 수석 아키텍트처럼 가르치십시오: 직접적이고, 구체적이며, 프로덕션 시나리오 (production scenario) 에 기반을 두세요. 모호함 없이. 불필요한 내용 없이.

에이전트 시스템에 대한 나의 친숙도(없음 / 간단한 에이전트 구축 / 다중 에이전트 시스템 구축)를 먼저 물어보고, 그에 따라 깊이를 조정하세요.

7가지 작업 설명 (task statement) 을 순서대로 진행하세요. 각 설명에 대해: 구체적인 프로덕션 예시로 설명하고, 시험 함정을 강조하고, 다음으로 넘어가기 전에 1-2개의 확인 질문을 하고, 다음 설명과 연결하세요.

**작업 설명:**

1.  **에이전트 루프 (AGENTIC LOOPS):** 완전한 라이프사이클 (lifecycle) (메시지 API, `stop_reason` 처리, 도구 결과 추가). 세 가지 안티패턴: 자연어 종료, 임의 반복 상한, 텍스트 콘텐츠를 완료로 인식. 모델 기반 의사 결정 vs 사전 구성된 의사 결정 트리 (decision tree).
2.  **다중 에이전트 오케스트레이션 (MULTI-AGENT ORCHESTRATION):** 허브 앤 스포크 아키텍처 (hub-and-spoke architecture). 코디네이터의 책임 (작업 분해, 하위 에이전트 선택, 컨텍스트 전달, 결과 집계). 중요한 격리 원칙: 하위 에이전트는 코디네이터 또는 서로 메모리를 공유하지 않습니다. 좁은 분해 실패 및 근본 원인 추적.
3.  **하위 에이전트 호출 및 컨텍스트 전달 (SUBAGENT INVOCATION AND CONTEXT PASSING):** 하위 에이전트를 생성하는 `Task` 도구. 출처 표기를 보존하기 위한 구조화된 메타데이터 (source URL, 문서 이름, 페이지 번호) 를 통한 컨텍스트 전달. 지연 시간 (latency) 을 위한 병렬 생성. 발산 탐색을 위한 `fork_session`.
4.  **워크플로 강제 및 핸드오프 (WORKFLOW ENFORCEMENT AND HANDOFF):** 프롬프트 기반 가이드 vs 프로그래밍 방식 훅. 결정 규칙: 재정/보안/규정 준수 = 프로그래밍 방식 강제. 다중 관심사 요청 분해. 인간 개입 (human escalation) 을 위한 구조화된 핸드오프 프로토콜 (customer ID, 대화 요약, 근본 원인, 권장 조치). 인간 에이전트는 대화 기록에 접근할 수 없습니다.
5.  **에이전트 SDK 훅 (AGENT SDK HOOKS):** 데이터 정규화를 위한 `PostToolUse` 훅. 정책 강제를 위한 도구 호출 가로채기. 결정 프레임워크 (framework): 훅 = 결정론적 보장, 프롬프트 = 확률적 가이드.
6.  **작업 분해 (TASK DECOMPOSITION):** 고정 파이프라인 vs 동적 적응형 분해. 주의 분산 문제. 다중 패스 아키텍처 해결책.
7.  **세션 상태 (SESSION STATE):** 재개 (resume) vs 분기 (fork) vs 요약 주입을 통한 새로운 시작. 오래된 컨텍스트 문제.

7가지 모두 완료 후, 10문제의 연습 시험을 실행하세요. 점수를 매겨주세요. 8/10 미만이면 약한 부분을 다시 검토하세요.

### 도메인 2: 도구 설계 및 MCP 통합 (Tool Design & MCP Integration) (18%)

이 도메인은 과소 평가되었습니다. 저는 프로덕션 (production) 에서 여러 MCP 서버 (Airtable, Notion, Google Drive) 를 운영하며, 도구 설명 품질에 디버깅 (debugging) 시간을 가장 많이 할애했습니다. 시험에서는 18%를 할당했지만, 저는 25%를 할당할 것입니다.

#### 1) 도구 설명은 선택 메커니즘 (Selection Mechanism) 이다

이것은 Claude 스택에서 다른 어떤 것보다 간과되는 부분입니다. 도구 설명은 인간을 위한 문서 (documentation) 가 아닙니다. 그것은 Claude가 어떤 도구를 호출할지 결정하는 **주요 메커니즘 (mechanism)**입니다.

시험에서는 `get_customer`와 `lookup_order`가 거의 동일한 설명을 가지고 있어서 에이전트가 계속해서 요청을 잘못 라우팅 (misroute) 하는 시나리오를 제시합니다. 네 가지 해결책이 제공되는데, 세 가지는 틀렸습니다: 퓨샷 예시 (few-shot example) (잘못된 근본 원인에 대한 토큰 오버헤드), 라우팅 분류기 (routing classifier) (과도하게 설계된 첫 단계), 도구 통합 (tool consolidation) (완전히 다른 문제). **올바른 해결책:** 각 도구가 언제 적용되는지 명확하게 구분하도록 설명을 다시 작성하는 것입니다.

좋은 설명에는 도구가 하는 일, 예상되는 입력 (input), 처리하는 쿼리 (query) 예시, 엣지 케이스 (edge case), 그리고 유사한 도구와의 명시적인 경계가 포함됩니다. 저는 설명 문제로 인해 잘못 라우팅된 도구 호출 때문에 오후 전체를 날린 적이 여러 번 있습니다. 라우팅 계층 (routing layer) 을 구축하는 데 3일을 보내기 전에, 더 나은 설명을 작성하는 데 30분을 투자하세요.

#### 2) 4~5개 도구 제한 (The 4–5 Tool Limit)

에이전트에 18개의 도구를 제공하면 선택 신뢰성 (reliability) 이 저하됩니다. 각 하위 에이전트의 역할을 고려하여 4~5개의 관련 도구로 범위를 제한하세요.

고빈도의 간단한 작업의 경우, 코디네이터 (coordinator) 를 통해 라우팅 (routing) 하는 대신 에이전트에 직접 제한된 도구를 제공하세요. 85%의 사실 확인이 간단한 조회인 경우, 합성 에이전트에 범위가 지정된 `verify_fact` 도구를 제공하세요. 복잡한 15%는 코디네이터를 통해 라우팅합니다. 이렇게 하면 작업당 2~3번의 왕복 호출을 줄일 수 있습니다.

`tool_choice`

세 가지 모드 (mode):

```python
# Default: model decides whether to use a tool
response = client.messages.create(
    tool_choice={"type": "auto"},  # May return text OR tool call
    ...
)

# Force a tool call (model picks which one)
response = client.messages.create(
    tool_choice={"type": "any"},  # MUST call a tool. Guaranteed structured output.
    ...
)
# Force a specific tool
response = client.messages.create(
    tool_choice={"type": "tool", "name": "extract_metadata"},  # MUST call this exact tool.
    ...
)
```

#### 3) 구조화된 오류 응답 (Structured Error Responses)

네 가지 범주가 있으며, 각각 다른 처리가 필요합니다.

**일시적 오류 (Transient errors)** (타임아웃 (timeout), 서비스 (service) 사용 불가)는 재시도 가능합니다. **유효성 검사 오류 (Validation errors)** (잘못된 입력 (input), 잘못된 형식 (format))는 재시도 전에 입력 수정이 필요합니다. **비즈니스 오류 (Business errors)** (환불이 정책 한도를 초과함)는 재시도할 수 없으며 다른 워크플로 (workflow) 가 필요합니다. **권한 오류 (Permission errors)** (액세스 거부)는 개입 (escalation) 이 필요합니다.

대부분의 에이전트 오류 처리를 어렵게 만드는 구분은 **액세스 실패 vs 유효한 빈 결과**입니다. 도구가 빈 배열 (array) 을 반환합니다. 데이터베이스 (database) 에 연결할 수 없어서일까요, 아니면 고객이 존재하지 않아서일까요? 이 둘을 동일하게 처리하면 에이전트가 조회를 3번 재시도하고 실패하며, 시스템에 없는 고객에 대해 인간에게 개입을 요청합니다. 해결책은 `isRetryable` 불리언 (boolean) 및 `errorCategory` 필드를 포함한 구조화된 오류 메타데이터 (metadata) 입니다.

#### 4) MCP 서버 구성 (MCP Server Configuration)

```json
// .mcp.json (project-level, version-controlled, shared with team)
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    }
  }
}
```

프로젝트 레벨 (project-level) (`.mcp.json`)은 버전 관리 (version control) 되고 팀과 공유됩니다. 사용자 레벨 (user-level) (`~/.claude.json`)은 개인적이며 공유되지 않습니다. 시험에서 가장 좋아하는 함정은 새 팀원이 사용자 레벨 대신 프로젝트 레벨에 구성이 있어서 도구가 누락되는 시나리오입니다.

표준 통합 (integration) (Jira, GitHub, Slack)을 위해서는 커뮤니티 MCP 서버를 사용하세요. 커뮤니티 서버가 처리할 수 없는 워크플로에 대해서만 사용자 지정 서버를 구축하세요.

#### 5) Grep vs Glob

Grep은 파일 내용 (file content) 을 검색합니다 (함수 호출자, 오류 메시지, import 문 찾기). Glob은 파일 경로 (file path) 를 일치시킵니다 (`**/*.test.tsx` 찾기, 구성 파일 위치 찾기). 잘못된 것을 사용하면 시간을 낭비합니다.

코드베이스 (codebase) 탐색의 경우: Grep으로 진입점 (entry point) 을 찾고, Read를 사용하여 import 문을 따르고 흐름을 추적합니다. 모든 파일을 미리 읽지 마세요. 그렇게 하면 컨텍스트 예산 (context budget) 이 소모됩니다.

🔗 MCP는 이 모든 도구 통합 (tool integration) 을 작동시키는 프로토콜 (protocol) 입니다. 저는 The AI Engineer의 "MCP란 무엇인가?"에서 전체 사양 (specification), USB-C 비유 (analogy), 그리고 주요 AI 플랫폼 (platform) 들이 왜 몇 주 내에 이를 채택했는지 다루었습니다.

### 도메인 3: Claude Code 구성 및 워크플로 (Claude Code Configuration & Workflows) (20%)

이 도메인은 순수한 구성 지식입니다. 파일이 어디에 있고 플래그 (flag) 가 무엇을 하는지 알거나 모르거나 둘 중 하나입니다. 저는 사용자 지정 스킬 (skill), 경로별 규칙 (path-specific rule) 및 MCP 서버 (server) 를 사용하여 콘텐츠 운영을 위한 Claude Code를 구성합니다. 이 도메인의 시험 함정 대부분은 실제 프로젝트 (project) 에서 잘못되었던 것들을 제가 직접 목격한 것들입니다.

#### 1) `CLAUDE.md` 계층 구조 (The CLAUDE.md Hierarchy)

세 가지 레벨이 있습니다.

*   **사용자 레벨 (`~/.claude/CLAUDE.md`):** 자신에게만 적용됩니다. 버전 관리 (version control) 되지 않습니다. Git을 통해 공유되지 않습니다. 팀의 다른 누구도 이 지침을 받지 못합니다.
*   **프로젝트 레벨 (`.claude/CLAUDE.md`):** 모든 사람에게 적용됩니다. 버전 관리되고 공유됩니다.
*   **디렉터리 레벨 (subdirectory `CLAUDE.md` files):** 특정 디렉터리 (directory) 에서 작업할 때 적용됩니다.

시험은 대부분의 팀이 겪었던 시나리오를 테스트합니다. 개발자 A의 Claude Code는 명명 규칙 (naming convention) 을 완벽하게 따릅니다. 지난주에 합류한 개발자 B는 일관성 없는 출력을 얻습니다. 동일한 리포지토리 (repository), 동일한 도구. **근본 원인:** 지침이 개발자 A의 머신에 있는 `~/.claude/CLAUDE.md`에 있습니다. 개발자 B는 그것을 받지 못했습니다. 해결책은 30초밖에 걸리지 않습니다(`.claude/CLAUDE.md`로 이동). 근본 원인을 찾는 데는 훨씬 더 오래 걸립니다.

#### 2) 경로별 규칙 (Path-Specific Rules)

```markdown
# .claude/rules/testing.md
---
paths: ["**/*.test.tsx"]
---
모든 테스트 파일은 특정 어설션 메시지 (assertion message) 를 사용하여 `describe/it` 블록을 사용해야 합니다.
외부 종속성 (external dependencies) 을 모의 (mock) 하세요. 내부 모듈 (internal modules) 은 절대 모의하지 마세요.
```

Glob 패턴 (pattern) 은 전체 코드베이스 (codebase) 에서 파일을 일치시킵니다. 디렉터리 레벨 `CLAUDE.md`는 단일 디렉터리만 다룹니다. 50개 이상의 디렉터리에 분산된 테스트 파일에 테스트 규칙을 적용해야 하는 경우, 경로별 규칙이 유일한 깔끔한 해결책입니다. 또한 일치하는 파일을 편집할 때만 로드되므로, 관련 없는 컨텍스트 (context) 가 토큰 예산 (token budget) 에서 제외됩니다.

#### 3) 계획 모드 (Plan Mode) vs 직접 실행 (Direct Execution)

*   **계획 모드 (Plan Mode)**는 여러 가지 유효한 접근 방식, 아키텍처 (architecture) 결정 또는 커밋 (commit) 하기 전에 탐색해야 하는 다중 파일 수정과 같은 복잡한 작업에 사용됩니다.
*   **직접 실행 (Direct Execution)**은 범위가 명확한 작업에 사용됩니다. 명확한 스택 트레이스 (stack trace) 가 있는 단일 파일 버그 수정. 유효성 검사 조건문 하나 추가. 올바른 접근 방식이 이미 알려진 경우입니다.

일반적인 하이브리드 (hybrid) 방식: 조사를 위해 계획 모드를 사용하고, 계획을 구현하기 위해 직접 실행을 사용합니다.

탐색 (Explore) 하위 에이전트 (subagent) 는 자세한 발견 출력을 주 대화에서 격리하고 요약을 반환합니다. 이것이 없으면 다단계 작업은 실제 작업에 도달하기 전에 발견 노이즈 (noise) 로 컨텍스트 창 (context window) 을 소모시킵니다.

#### 4) 스킬 (Skills) 및 명령 (Commands)

*   `.claude/commands/`는 프로젝트 범위이며 버전 제어를 통해 공유됩니다.
*   `~/.claude/commands/`는 개인용입니다.
*   `CLAUDE.md` 파일이 있는 `.claude/skills/`는 구성 (configuration) 과 함께 온디맨드 (on-demand) 로 작동합니다.

**스킬 프론트매터 (Skill frontmatter) 가 중요합니다.** `context: fork`는 격리된 하위 에이전트에서 스킬을 실행합니다 (자세한 출력은 격리되고, 주 대화는 깨끗하게 유지됩니다). `allowed-tools`는 스킬이 사용할 수 있는 도구를 제한합니다.

**핵심 구분:** 스킬은 온디맨드 방식의 작업별 워크플로 (workflow) 입니다. `CLAUDE.md`는 항상 로드되는 보편적인 표준입니다. 작업별 절차를 `CLAUDE.md`에 넣지 마세요. 보편적인 표준을 스킬에 넣지 마세요.

#### 5) CI/CD

`-p` 플래그 (flag) 는 Claude Code를 비대화형 모드 (non-interactive mode) 로 실행합니다. 이 플래그가 없으면 CI 작업이 입력 대기 상태로 중단됩니다. 시험은 이를 직접적으로 테스트합니다.

`--output-format json`과 `--json-schema`는 PR (Pull Request) 주석으로 게시할 수 있는 기계가 파싱 (parse) 할 수 있는 결과물을 생성합니다.

기억할 가치가 있는 한 가지: 코드를 생성한 동일한 Claude 세션 (session) 은 자체 변경 사항을 검토하는 데 덜 효과적입니다. 자체 결정을 의심할 가능성을 낮추는 추론 컨텍스트 (reasoning context) 를 유지합니다. CI 코드 검토 (code review) 에는 독립적인 검토 인스턴스 (instance) 를 사용하세요.

### 도메인 4: 프롬프트 엔지니어링 및 구조화된 출력 (Prompt Engineering & Structured Output) (20%)

이 도메인 전체의 핵심 원칙: 명시적으로 표현해야 합니다.

#### 1) 모호한 지침보다 명시적 기준 (Explicit Criteria Over Vague Instructions)

"보수적으로 행동하라"는 정확도 (precision) 를 향상시키지 않습니다. "높은 신뢰도 (confidence) 의 결과만 보고하라"는 오탐 (false positive) 을 줄이지 않습니다.

**작동하는 것:** "주장된 행동이 실제 코드 행동과 모순될 때만 주석을 플래그 (flag) 합니다. 버그 (bug) 및 보안 취약점 (security vulnerability) 을 보고합니다. 사소한 스타일 선호도 (style preference) 와 로컬 패턴 (local pattern) 은 건너뜝니다."

심각도 보정 (severity calibration) 에는 각 레벨에 대한 구체적인 코드 예시가 필요합니다. "치명적"이 무엇을 의미하는지에 대한 산문 설명이 아니라, 치명적인 버그와 사소한 스타일 문제 사이의 실제 코드를 보여주는 예시입니다.

이해할 가치가 있는 한 가지 동적 요소: 한 범주에서 높은 오탐 (false positive) 율은 모든 범주에 대한 신뢰를 떨어뜨립니다. 개발자들은 출력을 완전히 읽는 것을 중단합니다. 저는 이것이 두 개의 다른 팀에서 AI 코드 검토 도구의 채택을 막는 것을 보았습니다. 해결책은 직관적이지 않습니다. 해당 프롬프트 (prompt) 를 개선하는 동안 높은 오탐 범주를 일시적으로 비활성화합니다. 먼저 신뢰를 회복한 다음 반복 (iterate) 합니다.

#### 2) 퓨샷 예시 (Few-Shot Examples)

일관된 출력 (output) 을 얻기 위한 가장 효과적인 기술입니다. 더 많은 지침이 아닙니다. 신뢰도 임계값도 아닙니다. **예시입니다.**

모호한 시나리오에 대해 2~4개의 특정 예시를 만드세요. 각 예시는 가능한 대안 중에서 특정 조치를 선택한 이유에 대한 추론 (reasoning) 을 보여주어야 합니다. 이것은 제공된 예시와의 엄격한 일치가 아니라, 새로운 패턴 (pattern) 에 대한 일반화 (generalization) 를 가르칩니다.

#### 3) `tool_use` 및 JSON 스키마 (JSON Schemas)

`tool_use`와 JSON 스키마 (JSON Schema) 는 구문 오류 (syntax error) 를 제거합니다. 그것이 바로 `tool_use`가 하는 일입니다. `tool_use`가 **방지하지 않는 것**: 의미 오류 (semantic error) (명시된 총합과 일치하지 않는 항목), 필드 배치 (values in wrong fields) (값이 잘못된 필드에 배치됨), 정보 조작 (fabrication) (원본에 정보가 없을 때 모델이 필수 필드에 대한 값을 만들어냄).

스키마 (schema) 설계는 사람들이 깨닫는 것보다 더 중요합니다.

```json
tools = [{
    "name": "extract_invoice",
    "input_schema": {
        "type": "object",
        "properties": {
            "vendor_name": {"type": "string"},
            "total_amount": {"type": "number"},
            "currency": {"type": "string"},
            "payment_terms": {
                "type": "string",
                "nullable": True  // Source may not contain this
            },
            "confidence": {
                "type": "string",
                "enum": ["high", "medium", "low", "unclear"]
            },
            "category": {
                "type": "object",
                "properties": {
                    "primary": {"type": "string", "enum": [
                        "services", "goods", "subscription", "other"
                    ]},
                    "detail": {"type": "string"}  // Freeform when "other"
                }
            }
        },
        "required": ["vendor_name", "total_amount", "currency", "confidence"]
    }
}]
```

원본에 데이터가 없을 수 있는 경우 널러블 필드 (nullable field) 를 사용하세요. 모호한 경우 "unclear" 열거형 (enum) 을 사용하세요. 확장 가능한 분류를 위해 "other"와 자유 형식의 상세 문자열 (detail string) 을 사용하세요. 모든 필드를 필수로 만들면, 모델은 데이터가 없더라도 모든 필드를 채워 넣습니다.

#### 4) 유효성 검사-재시도 루프 (Validation-Retry Loops)

원본 문서, 실패한 추출 결과 및 특정 유효성 검사 오류 (validation error) 를 다시 보냅니다. 모델은 오류를 사용하여 자체 수정합니다.

기억해야 할 경계: 이것은 형식 불일치, 구조적 오류, 잘못 배치된 값에는 효과적입니다. 하지만 원본 문서에 존재하지 않는 정보에는 효과적이지 않습니다. 없는 데이터에 재시도 사이클을 낭비하지 마세요.

#### 5) 배치 처리 (Batch Processing)

메시지 배치 API (Message Batches API) 는 24시간 처리 창과 함께 50%의 비용 절감 효과를 제공합니다. 지연 시간 (latency) 보장은 없습니다. 단일 요청 내에서 다중 턴 (multi-turn) 도구 호출을 지원하지 않습니다.

블로킹 (blocking) 워크플로 (pre-merge checks, 개발자가 기다리는 모든 것)에는 동기 API (synchronous API) 를 사용하세요. 지연 시간 허용 워크플로 (overnight reports, weekly audits, nightly test generation)에는 배치 API (Batch API) 를 사용하세요.

나의 프롬프트 엔지니어링 (prompt engineering) 경험(기본 프롬프트 / 퓨샷 사용 / 추출 파이프라인 구축)에 대해 물어보고 깊이를 조정하세요.

### 도메인 5: 컨텍스트 관리 및 신뢰성 (Context Management & Reliability) (15%)

가중치가 가장 작은 도메인이지만, 컨텍스트 관리 (context management) 실수는 다른 모든 도메인으로 파급됩니다.

#### 1) 점진적 요약 함정 (The Progressive Summarization Trap)

대화 기록을 응축하면 트랜잭션 데이터 (transactional data) 가 압축됩니다. "3월 3일에 주문된 #8891 주문에 대한 $247.83 환불"이 "고객이 최근 주문에 대한 환불을 원함"이 됩니다. 금액, 주문 번호, 날짜가 모두 사라집니다.

**해결책:** 트랜잭션 사실을 영구적인 "사례 사실 (case facts)" 블록 (block) 으로 추출합니다. 모든 프롬프트 (prompt) 에 포함합니다. 절대 요약하지 마세요.

```json
case_facts = {
    "customer_id": "cust_8891",
    "order_id": "#8891",
    "order_date": "2025-03-03",
    "refund_amount": 247.83,
    "currency": "USD",
    "issue": "Defective product received"
}
system_context = f"""
CASE FACTS (do not summarize or paraphrase):
{json.dumps(case_facts, indent=2)}
"""
```

**"중간 누락 효과 (lost in the middle effect)":** 모델은 긴 입력의 시작과 끝을 안정적으로 처리하지만, 중간에 묻힌 발견은 놓칩니다. 주요 요약을 시작 부분에 배치하세요.

자세한 도구 결과는 컨텍스트 (context) 에 추가하기 전에 잘라내세요. 주문 조회는 40개 이상의 필드 (field) 를 반환합니다. 5개만 필요합니다. 나머지는 컨텍스트 창 (context window) 을 채우기 전에 제거하세요.

#### 2) 에스컬레이션 트리거 (Escalation Triggers)

인간에게 개입 (escalation) 할 유효한 세 가지 트리거 (trigger) 입니다.

1.  **고객이 명시적으로 인간을 요청합니다.** 즉시 존중하세요. 먼저 해결하려고 시도하지 마세요, "확실합니까?"라고 묻지 마세요, 진단 (diagnostic) 을 한 번 더 실행하지 마세요. 그들은 사람을 요청했습니다.
2.  **정책 예외 또는 공백.** 요청이 문서화된 정책 (policy) 범위를 벗어납니다.
3.  **여러 번의 시도 후 의미 있는 진전을 이루지 못함.**

시험에서 오답으로 사용되는 두 가지 신뢰할 수 없는 트리거: 감정 기반 에스컬레이션 (좌절한 고객이라도 간단한 배송 질문이라면 인간이 필요하지 않고, 추적 번호가 필요합니다)과 자체 보고된 신뢰도 점수 (confidence score) (모델은 어려운 경우에는 종종 잘못된 자신감을 보이고, 쉬운 경우에는 잘못된 불확실성을 보입니다).

#### 3) 오류 전파 (Error Propagation)

다중 에이전트 시스템의 신뢰성 (reliability) 을 파괴하는 두 가지 안티패턴 (anti-pattern) 입니다.

*   **자동 오류 억제 (Silent suppression):** 하위 에이전트 (subagent) 가 오류를 만나지만 성공으로 표시된 빈 결과를 반환합니다. 코디네이터 (coordinator) 는 모든 것이 작동한 것처럼 계속합니다. 아무도 실패를 알지 못하므로 복구할 수 없습니다.
*   **워크플로 (workflow) 종료:** 단일 하위 에이전트 실패가 전체 파이프라인 (pipeline) 을 중단시킵니다. 다른 하위 에이전트가 달성한 모든 것이 버려집니다.

**올바른 패턴:** 구조화된 오류 컨텍스트. 하위 에이전트가 실패하면, 실패 유형, 시도한 내용, 수집한 부분적 결과 및 가능한 대안을 보고합니다. 코디네이터는 재시도할지, 재라우팅 (reroute) 할지, 또는 부분적 결과로 진행하고 공백을 주석 (annotate) 처리할지 결정합니다.

#### 4) 정보 출처 (Information Provenance)

다중 에이전트 합성의 각 발견은 **주장-출처 매핑 (claim-source mapping)**이 필요합니다: 주장 (claim), 출처 URL, 문서 이름, 관련 발췌문 (excerpt), 발행일. 이것이 없으면 합성 에이전트가 여러 출처를 요약하는 순간 출처 표기 (attribution) 가 사라집니다.

두 개의 신뢰할 수 있는 출처가 다른 통계를 보고할 때, 하나를 선택하지 마세요. 두 값 모두 출처를 표시하세요. 발행일을 포함하세요. 2023년 연구와 2025년 연구가 다른 숫자를 보고하는 것은 서로 모순되는 것이 아닙니다. 그것들은 다른 시점을 측정하는 것입니다.

#### 🤖 도메인 5 튜터 프롬프트

당신은 Claude Certified Architect 시험의 도메인 5(컨텍스트 관리 및 신뢰성)를 가르치는 전문 강사입니다. 이 도메인은 15%를 차지합니다.

가중치가 가장 작지만, 이 개념들은 도메인 1, 2, 4로 파급됩니다.

긴 컨텍스트 (long-context) 애플리케이션 (application) 및 다중 에이전트 시스템 (multi-agent system) 경험에 대해 물어보고 깊이를 조정하세요.

6가지 작업 설명을 진행하세요:

1.  **컨텍스트 보존 (CONTEXT PRESERVATION):** 점진적 요약 함정 (progressive summarization trap). 영구적인 사례 사실 (case facts) 블록 (block). 중간 누락 효과 (lost-in-the-middle effect). 도구 결과 트리밍 (trimming).
2.  **개입 및 모호성 (ESCALATION AND AMBIGUITY):** 세 가지 유효한 트리거 (trigger) vs 두 가지 신뢰할 수 없는 트리거. 좌절감 뉘앙스. 모호한 고객 매칭.
3.  **오류 전파 (ERROR PROPAGATION):** 구조화된 오류 컨텍스트 (structured error context). 자동 오류 억제 (silent suppression) vs 워크플로 (workflow) 종료. 액세스 실패 vs 유효한 빈 결과.
4.  **코드베이스 탐색 (CODEBASE EXPLORATION):** 컨텍스트 저하. 스크래치패드 (scratchpad) 파일. 하위 에이전트 (subagent) 위임. 요약 주입 (summary injection). `/compact`.
5.  **인간 검토 및 신뢰도 (HUMAN REVIEW AND CONFIDENCE):** 유형별 오류를 가리는 총체적 정확도 (aggregate accuracy). 계층적 샘플링 (stratified sampling). 필드 레벨 신뢰도 보정 (field-level confidence calibration).
6.  **정보 출처 (INFORMATION PROVENANCE):** 합성을 통한 주장-출처 매핑 (claim-source mapping). 충돌 처리. 시간적 인식 (temporal awareness).

6가지 모두 완료 후, 6문제의 연습 시험을 실행하세요. 점수를 매겨주세요. 5/6 미만이면 다시 검토하세요.

구축 연습: "두 개의 하위 에이전트를 포함하는 코디네이터 (coordinator) 를 구축합니다. 영구적인 사례 사실 (case facts) 을 구현합니다. 구조화된 오류 전파 (error propagation) 와 함께 타임아웃 (timeout) 을 시뮬레이션 (simulate) 합니다. 충돌하는 출처로 테스트합니다."

## 시험이 제대로 다루는 것과 놓치는 것

가중치는 대부분 적절합니다. 에이전트 아키텍처 (architecture) 의 27%는 프로덕션 (production) 현실을 반영합니다. 오케스트레이션 (orchestration) 결정은 다른 어떤 것보다 더 큰 영향을 미칩니다.

도구 설계 및 MCP는 18%보다 높아야 합니다. 도구 설명 품질과 오류 처리 (error handling) 는 가중치가 암시하는 것보다 더 많은 프로덕션 문제를 야기합니다. 에이전트당 4~5개 도구 제한은 자의적으로 들릴 수 있지만, 이를 위반하고 로그 (log) 를 통해 잘못 라우팅된 도구 호출을 추적하는 데 일주일을 보내고 나면 달라집니다.

**완전히 누락된 것:** 비용 최적화 (cost optimization). 시험은 기본적인 요약 이상의 토큰 예산 책정 (token budgeting), 컨텍스트 창 관리 (context window management), 또는 에이전트 역할 전반에 걸친 모델 선택의 장단점 (tradeoff) 을 다루지 않습니다. 하위 에이전트에 Haiku로 충분할 때와 Sonnet이 필요할 때는 언제일까요? 프로덕션에서 모든 하위 에이전트가 Opus로 실행되는 다중 에이전트 시스템은 훌륭한 결과를 생성하겠지만, 필요한 비용의 40배가 들 것입니다.

**또한 누락된 것:** 관측 가능성 (observability). 시험은 오류 전파 (error propagation) 를 다루지만, 프로덕션에서 다중 에이전트 시스템을 실제로 모니터링 (monitor) 하는 방법은 다루지 않습니다. 어떤 하위 에이전트가 느린가요? 어떤 도구가 자동으로 오류를 발생시키나요? 코디네이터의 분해 단계에서 p95 지연 시간 (latency) 은 얼마인가요? 측정할 수 없는 것은 개선할 수 없습니다.
