# MCP 앱: AI, 드디어 프런트엔드를 만나다 — 그리고 우리의 개발 이야기

과거 인포그램(Infogram)과 프레지(Prezi)에서 일할 때, 인포그램의 공동 창립자 중 한 명인 앨리스 디리카(Alise Dīrika)는 회사의 창립 스토리를 간단한 이미지로 설명하곤 했습니다. 왼쪽에는 텍스트 벽, 오른쪽에는 차트. 같은 데이터였지만, 하나는 스쳐 지나가고 다른 하나는 즉시 이해할 수 있었죠.

우리 모두 고개를 끄덕입니다. 하지만 앨리스가 정말로 보여주고 싶었던 것은 더 깊은 통찰이었습니다. 바로 올바른 표현 방식이 사람들이 보고 있는 것을 실제로 이해하는지 여부를 바꾼다는 것이죠.

'이미지는 천 마디 말의 가치가 있다.'

### UI는 만 마디 말의 가치가 있다

저는 5,500개 이상의 깃허브(GitHub) 스타와 389,000회 이상의 설치(클로드(Claude) 기준)를 기록한 MCP 서버인 데스크톱 커맨더(Desktop Commander)의 CEO 에두아르드(Eduard)입니다. 이 서버는 AI가 로컬 파일 시스템에 직접 접근할 수 있도록 합니다. 지난 해 데스크톱 커맨더 MCP를 출시하고 사용자 활성화를 조사할 때, 앨리스의 '차트 vs. 텍스트' 이미지를 많이 떠올렸습니다.

MCP는 백엔드(backend)입니다. 사용자 인터페이스(UI)가 없습니다. 호스트 챗이 제공하는 사용자 경험(UX)에 좌우됩니다. 온보딩(onboarding)도, 브랜딩(branding)도, 첫인상에 대한 통제도 없습니다. 우리는 신규 사용자에게 이러한 무력감을 가장 절실히 느꼈습니다. 20~80%의 사용자가 도구를 설치하고도 사용 횟수가 0회 또는 1회에 불과했으며, 우리가 할 수 있는 일은 거의 없었습니다. 우리는 그들을 안내하기 위해 프롬프트 인젝션(prompt injection) 실험을 진행했습니다. 환영 화면을 A/B 테스트했습니다. 하지만 근본적으로 우리는 다른 사람의 텍스트 상자 안에서 작동하는 강력한 도구를 만들고 있었습니다.

클로드가 파일을 읽으면? 원시 텍스트(raw text)를 챗에 덤프합니다. 사용자는 그냥 스크롤하여 지나칩니다. 클로드가 이미지를 읽으면? 베이스64(base64) 형태의 암호 같은 문자열을 반환합니다. 클로드가 디렉터리 목록을 보여주면? 일반 텍스트 경로가 나타납니다. 우리는 프런트엔드(frontend) 없는 백엔드를 만들고 있었고, 다른 모든 MCP 서버들도 마찬가지였습니다.

켄트 C. 도즈(Kent C. Dodds)는 신택스 팟캐스트(Syntax Podcast) (에피소드 973)에서 이를 정확히 지적했습니다:

"지난 몇 년간 모두가 자신의 애플리케이션에 챗봇을 넣으려고 했습니다. 이제 우리는 그것을 뒤집었습니다. 애플리케이션을 챗봇 안에 넣은 것이죠."

이러한 변화에는 UI가 필요합니다. 챗봇이 텍스트로만 응답할 수 있다면 애플리케이션을 챗봇 안에 넣을 수 없습니다.

MCP는 AI에 초능력을 부여했지만, 모든 결과는 텍스트로 돌아왔습니다. 이미지가 천 마디 말의 가치가 있다면, 좋은 MCP 앱은 만 마디 말의 가치가 있습니다.

2026년 1월 26일에 상황이 바뀌었습니다. MCP 앱(Apps)이 출시되면서 AI 도구는 단순히 정보를 '말해주는' 것을 넘어 '보여줄' 수 있게 되었습니다. 이제 상호작용 가능한 위젯(widget), 렌더링된 미리 보기, 클릭 가능한 UI가 AI 대화 안에 모두 담기게 됩니다.

이것이 어떻게 일어났고, 왜 중요하며, 우리가 이것으로 무엇을 구축하고 있는지 설명하겠습니다.

### MCP 앱은 실제로 어떻게 탄생했는가

이것은 기업 주도의 이니셔티브가 아니었습니다. 풀뿌리(grassroots) 운동이었습니다.

2025년 5월 — 이도 살로몬(Ido Salomon, 팔로알토 네트웍스(Palo Alto Networks))과 리아드 요세프(Liad Yosef, 쇼피파이(Shopify))는 MCP-UI를 사이드 프로젝트로 시작했습니다. MCP 서버가 텍스트 대신 상호작용 가능한 HTML을 반환할 수 있도록 하는 오픈소스 SDK였습니다. 그들은 AI 도구가 호스트의 도움 없이는 시각적인 것을 보여줄 수 없다는 동일한 문제에 좌절하여 이것을 만들었습니다.

쇼피파이가 이를 가장 먼저 도입했습니다. 제품 카드, 변형 선택기, 이미지 갤러리, 결제 흐름 등 모든 것이 AI 대화 내에서 렌더링(rendering)되었습니다. 그들의 엔지니어링 블로그에서는 "커머스(commerce)에서 시각적 맥락은 단순히 유용한 것을 넘어 필수적입니다"라고 명확히 밝혔습니다.

그 후 블록(Block)의 오픈소스 에이전트(agent)인 구스(Goose)가 이를 채택했습니다. 좌석 배치도, 식당 카드, 날씨 위젯 등 모든 것이 에이전트 내에서 렌더링되었습니다.

2025년 8월 — 샌프란시스코에서 MCP Night 2.0이 열렸습니다. 700명의 엔지니어들이 모였고, 이도와 리아드는 MCP-UI를 라이브로 시연했습니다. 앤스로픽(Anthropic), 오픈AI(OpenAI), 노션(Notion), 블록(Block) 모두 참석했습니다. 커뮤니티는 열광했습니다. (두 창립자는 이후 먼데이닷컴(Monday.com)에 각각 수백만 달러 규모의 인수합병(acqui-hire)으로 영입되었다고 전해집니다.)

2025년 9월 — 그리고 이때부터 흥미로워집니다. 오픈AI는 독립적으로 자체적인 '챗 내 UI' 시스템인 ChatGPT 앱 SDK를 구축했습니다. 이는 MCP를 기반으로 했지만, 그 위에 자체 UI 프레임워크를 사용했습니다. 그들은 주간 활성 사용자(weekly active users)가 9억 명에 달했습니다. 독점적으로 운영하여 모든 개발자가 ChatGPT 전용으로만 앱을 만들게 할 수도 있었습니다.

하지만 그들은 개방을 선택했습니다.

2025년 11월 — 앤스로픽, 오픈AI, 그리고 MCP-UI 팀은 SEP-1865: MCP 앱을 제안했습니다. 이는 최초의 공식 MCP 확장 기능이었습니다. 단일 표준. 모든 호스트 지원. 공급업체 종속(lock-in) 없음.

2026년 1월 26일 — MCP 앱이 클로드, ChatGPT, VS Code, 구스에 적용되어 출시되었습니다. 피그마(Figma), 아사나(Asana), 슬랙(Slack), 칸바(Canva), 박스(Box)가 48시간 이내에 통합했습니다. 2월에는 에어리아(Airia)와 같은 기업 플랫폼이 지원을 시작했고, 조달팀은 MCP 앱을 AI 플랫폼 요구사항에 추가하고 있습니다.

사이드 프로젝트에서 업계 표준이 되기까지 단 9개월이 걸렸습니다.

### 데스크톱 커맨더의 첫 MCP 앱 기능

우리는 데스크톱 커맨더 MCP v0.2.36에서 첫 UI 위젯을 출시했습니다. 바로 파일 미리 보기 위젯입니다. 간단하게 들리지만, 그것이 핵심입니다.

이전에는 클로드에게 파일을 읽어달라고 요청하면, 클로드는 원시 콘텐츠(raw content)를 컨텍스트(context)에 덤프했습니다. 클로드는 기본적으로 숨겨진 원시 구문(raw syntax)으로 렌더링했습니다. 사용자가 클릭하고, 읽으려 애쓰고, 눈을 가늘게 뜨는 방식이었습니다.

이제는 깨끗한 타이포그래피(typography)로 렌더링된 마크다운(markdown), 구문 강조(syntax-highlighted)된 코드, PNG, JPEG, GIF, WebP 등 이미지가 인라인(inline)으로 실제 표시됩니다. 현재 위치를 보여주는 브레드크럼 경로(breadcrumb path)도 있습니다. 파인더(Finder)나 탐색기(Explorer)로 바로 이동하는 "폴더에서 열기(Open in Folder)" 버튼과, 파일을 다시 읽지 않고도 미리 보기를 확장하는 "더 많은 줄 로드(Load more lines)" 컨트롤도 있습니다.

추가 종속성(dependencies)은 전혀 없습니다. macOS와 Windows에서 모두 작동합니다.

이것이 제가 "누락된 프런트엔드(missing frontend)"라고 말하는 의미입니다. 기능은 항상 존재했습니다. 데스크톱 커맨더는 항상 파일을 읽을 수 있었습니다. 하지만 경험은 대규모 언어 모델(LLM)을 통해 해석된 텍스트였습니다. 이제 경험은 시각적입니다. 그리고 시각적인 것은 인간이 실제로 처리하는 방식입니다.

### MCP 앱의 작동 방식 (내부 구조)

아키텍처(architecture)는 우아합니다. 도구는 UI 메타데이터(metadata)를 선언합니다. 기본적으로 호스트에게 "나는 시각적 구성 요소를 가지고 있다"고 알리는 것이죠. 서버는 `ui://` 스키마(scheme)를 통해 HTML/JS 번들(bundle)을 제공합니다. 이는 정적이며, 미리 선언되고, 렌더링 전에 감사(auditable)할 수 있습니다.

모든 것은 샌드박스(sandboxed) 처리된 아이프레임(iframes) 내에서 실행됩니다. 부모 도메인 객체 모델(parent DOM) 접근은 없으며, 임의의 네트워크 요청도 없습니다. 기본적으로 보안이 강화되어 있습니다. UI와 호스트는 `postMessage`를 통한 양방향 JSON-RPC(JSON-RPC)로 통신합니다. 양쪽 모두 동일한 MCP SDK를 사용합니다.

하지만 이것이 단순히 "아이프레임" 이상인 이유는 다음과 같습니다. 사양(spec)은 UI가 사용할 수 있는 풍부한 기능 집합을 정의합니다:

*   `tools/call` — UI는 MCP 서버의 모든 도구를 호출할 수 있습니다. 파일 미리 보기 위젯이 파일 이름 변경을 트리거(trigger)할 수 있고, 대시보드(dashboard)가 자체 데이터를 새로 고칠 수 있습니다.
*   `ui/message` — UI는 대화로 메시지를 다시 보낼 수 있습니다. 위젯의 버튼 클릭이 "사용자가 옵션 B를 선택했습니다"라고 말하면 LLM이 거기서부터 대화를 이어갑니다.
*   `ui/update-model-context` — UI는 LLM의 컨텍스트(context)를 조용히 업데이트할 수 있습니다. 사용자가 필드를 채우는 양식을 생각해보세요. 모델은 사용자가 직접 입력할 필요 없이 해당 선택 사항을 학습합니다.
*   `resources/read` — UI는 MCP 리소스(resource)를 직접 읽어, 도구 호출(tool call)을 거치지 않고 서버에서 데이터를 로드(load)할 수 있습니다.
*   `Host context at init` — `ui/initialize` 시, 호스트는 테마 정보(theme info) (밝은 모드/어두운 모드, 색상, 타이포그래피, 테두리를 위한 CSS 사용자 정의 속성(CSS custom properties)), 컨테이너 크기, 기능 플래그(capability flags)를 전달합니다. 위젯은 처음부터 호스트의 모양과 느낌에 맞출 수 있습니다.
*   `Display modes` — 인라인(inline) (챗 흐름에 포함), 전체 화면(fullscreen) (창을 모두 차지), PIP(picture-in-picture) (타이머나 음악 플레이어와 같은 지속적인 위젯을 위한 오버레이(overlay)).

이는 AI 도구에 적용된 웹 플랫폼(web platform)입니다. 특별한 것이 없습니다. 그래서 채택이 빨랐습니다. 모든 개발자는 이미 이것을 구축하는 방법을 알고 있습니다.

### 사용자 경험은 아직 완벽하지 않지만, 괜찮습니다.

우리의 현주소에 대해 솔직하게 말씀드리고 싶습니다. 표준은 확고합니다. 하지만 클라이언트(client)가 여전히 경험을 제어하고, 각 호스트는 기능을 다르게 구현합니다.

`ui/message`를 예로 들어보겠습니다. 위젯이 대화에 메시지를 보낼 수 있는 기능입니다. 이론적으로 이는 위젯이 대화를 진행시키는 방식입니다: 사용자가 버튼을 클릭 → UI가 메시지를 보냄 → LLM이 응답.

실제로는? 클로드에서는 위젯이 메시지를 처음 보낼 때, 텍스트가 입력 상자에 들어가고 사용자가 직접 보내기 버튼을 눌러야 합니다. 두 번째부터는 입력 상자에 있는 내용을 바꿀 것인지 묻는 토스트(toast) 알림이 뜹니다.

이것이 데스크톱 커맨더가 아직 '메시지 보내기' 기능을 사용하지 않는 이유입니다. 기존의 '선택/복사/제출' 패턴보다 나은 점이 없습니다.

그들은 조심스럽게 접근하고 있으며, 이는 보안을 위한 올바른 본능입니다. 하지만 이것이 "원활한(seamless)" 흐름이 아직 원활하지 않다는 것을 의미합니다.

또는 `ui/update-model-context`를 예로 들어보겠습니다. 조용한 컨텍스트 업데이트 기능입니다. 사양에는 UI가 모델에 컨텍스트를 푸시(push)할 수 있다고 명시되어 있습니다. 하지만 클로드에서는 사용자가 명시적으로 컨텍스트에 대해 질문해야만 LLM이 실제로 그 컨텍스트를 인식합니다. 컨텍스트는 저장되지만, 선제적으로 드러나지 않습니다. 따라서 위젯이 양식 상태를 업데이트하고 모델이 반응하기를 기대한다면, 사용자가 확인하도록 프롬프트(prompt)하지 않는 한 모델은 반응하지 않을 것입니다.

이것들은 버그(bug)가 아닙니다. 신뢰와 사용자 제어에 대한 호스트 수준의 의도적인 선택입니다. 모든 클라이언트는 여기에서 다른 절충안을 만들 것입니다. 그리고 이것이 MCP 앱의 근본적인 긴장 관계입니다. 표준은 가능한 것을 정의하지만, 각 호스트는 허용되는 것을 결정합니다.

켄트 C. 도즈는 훌륭한 예를 들었습니다. 요리 중 AI에게 타이머를 설정해달라고 요청할 때, "30분 타이머를 설정했습니다. 언제든지 물어보세요"라고 말하는 것을 원치 않습니다. 일시정지 버튼이 있는 카운트다운을 원합니다. MCP 앱은 정확히 이를 위해 PIP 모드(pip mode)를 지원합니다. 하지만 대부분의 도구는 아직 이를 구축하지 않았습니다.

### 우리가 다음에 구축할 것

데스크톱 커맨더 MCP를 위해 적극적으로 탐색하고 있는 몇 가지 아이디어가 있습니다:

*   **대화형 터미널 위젯(Interactive Terminal Widget)** — 터미널 출력(terminal output)을 실시간으로 확인합니다. SSH 세션, 데이터베이스 프롬프트, 개발 서버 등과 챗에서 직접 상호작용합니다. 대화를 떠나지 않고도 대화형 프로세스(interactive processes)에 로그인합니다.
*   **편집 내용 되돌리기 버튼(Revert Button for Edits)** — AI가 파일을 편집할 때, 시각적 차이(visual diff)를 확인하고 문제가 있으면 '되돌리기(Revert)'를 클릭합니다. 깃(git)을 뒤적거리지 않고 실행 취소(undo)할 수 있습니다.
*   **챗 내 협업 편집(Collaborative Editing in Chat)** — AI와 나란히 대화 안에서 마크다운, HTML 또는 코드를 편집합니다. 때로는 사용자가 빠르게 수정하는 것이 더 빠르고, 때로는 AI에게 요청하는 것이 더 빠릅니다. 라이브 편집기 위젯을 사용하면 선택할 필요가 없습니다. 두 사용자 모두 같은 장소에서 같은 파일을 작업할 수 있습니다. 앱을 전환할 필요가 없습니다.

그리고 우리는 빠르게 개발하고 있습니다. 현재 주 2회 릴리스(release)를 진행하며, 매 릴리스마다 더 많은 MCP 앱 기능을 추가하고 있습니다.

### 데스크톱 커맨더 앱 — 이제 공개 베타 중

위의 모든 내용은 우리의 오픈소스 MCP 서버에 적용됩니다. 하지만 우리는 더 큰 것을 구축하고 있습니다. 바로 데스크톱 커맨더 앱(Desktop Commander App)입니다. 이것은 사용자 컴퓨터에서 직접 실행되는 완전한 데스크톱 AI 에이전트입니다.

자신의 ChatGPT 구독을 사용하거나, 어떤 모델이든 선택할 수 있습니다. 추론(reasoning)을 위한 오푸스(Opus), 정확한 실행을 위한 GPT, 비전(vision)을 위한 제미니(Gemini), 또는 개인 정보 보호를 위해 로컬 모델을 실행할 수 있습니다. 자동화를 위한 칸바(Canva)를 생각해보세요. 개발자가 아니어도 컴퓨터 작업을 자동화하거나, 전담 IT 인력을 고용할 수 없는 팀을 위한 저렴한 기술 지원으로 사용할 수 있습니다.

그리고 우리를 차별화하는 점은 다음과 같습니다. 우리는 MCP와 이를 사용하는 앱을 모두 구축합니다. 우리는 프로토콜(protocol)의 양쪽 측면에서 독특한 통찰력을 가지고 있습니다. 이는 마찰이 어디에 있는지 정확히 이해하고 양쪽 끝에서 이를 해결할 수 있기 때문에 더 나은 사용자 경험(UX)으로 이어집니다.

Mac, Windows, Linux에서 사용할 수 있습니다. 한번 사용해보시고, 문제가 발생하는 부분을 알려주세요. 그렇게 해야 더 나아질 수 있습니다.
