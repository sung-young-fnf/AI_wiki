이런 상황에 익숙하실 겁니다. 복잡한 백엔드 문제를 해결하며 몰입(flow state)하고 있는데, 프로덕트 매니저로부터 슬랙(Slack) 알림이 도착합니다:

“안녕하세요, 지난주 독일에 가입한 모든 사용자 목록을 빠르게 뽑아줄 수 있을까요? 10분 뒤 회의에 필요합니다.”

당신은 SQL을 잘 알고 있고, 눈을 감고도 쿼리를 작성할 수 있습니다. 하지만 굳이 그럴 필요는 없습니다.

임시(ad-hoc) 보고 쿼리를 작성하는 데 드는 매 순간은 실제 제품 기능을 개발하는 데서 빼앗긴 시간입니다.

해결책은 또 다른 복잡한 관리 대시보드를 만드는 것이 아닙니다. 팀원들이 사용하는 언어로 소통할 수 있는 도구를 제공하는 것입니다.

이 튜토리얼에서는 '데이터베이스와 대화하는(Chat with your Database)' 애플리케이션을 구축할 것입니다. 이 앱은 비기술 사용자(non-technical users)가 일상적인 영어로 질문하면, AI가 안전한 읽기 전용(read-only) SQL을 작성하고 백그라운드에서 실행하도록 합니다.

가장 좋은 점은? 100% 로컬(locally) 환경에서 실행된다는 것입니다.

*   **API 키 불필요**: 비용은 $0.00입니다.
*   **데이터 프라이버시(Data Privacy) 위험 없음**: 프로덕션 스키마(production schema)가 OpenAI로 전송될 일이 없습니다.
*   **‘글루 코드(Glue Code)’ 악몽 없음**: 새로운 모델 컨텍스트 프로토콜(Model Context Protocol, MCP)을 사용하여 아키텍처(architecture)를 깔끔하게 유지할 것입니다.

이 가이드가 끝날 때쯤이면, “이 데이터 좀 뽑아줄 수 있나요?”라는 요청을 “네, 봇에게 물어보세요.”로 바꿔주는 배포 가능한 도커(Docker) 컨테이너(container)를 갖게 될 것입니다.

이제 당신의 시간을 되찾으세요.

## 아키텍처: “AI를 위한 USB-C”

코드를 한 줄도 작성하기 전에, 모델 컨텍스트 프로토콜(Model Context Protocol, MCP)을 왜 사용하는지 이해해야 합니다.

대규모 언어 모델(LLM) 초기에는 모델을 데이터베이스에 연결하려면 지저분한 ‘글루 코드(glue code)’를 작성해야 했습니다. 프롬프트(prompt)에 테이블 정의를 수동으로 주입하고, SQL을 찾기 위해 복잡한 정규 표현식(regex)을 파싱(parse)하며, 모든 오류 예외(edge-case)를 직접 처리해야 했습니다. 이는 취약하고 유지보수가 어려웠습니다.

MCP는 연결을 표준화하여 이 문제를 해결합니다. AI를 위한 USB-C 포트(port)라고 생각해보세요:

*   **리소스(Resources, Context)**: AI가 “어떤 데이터가 있나요?”라고 물으면, 서버가 데이터베이스 스키마(database schema)로 응답합니다.
*   **도구(Tools, Action)**: AI가 “쿼리를 실행해야 해요.”라고 결정하면, 서버가 해당 함수를 안전하게 실행합니다.

우리는 두뇌(Brain, LLM)와 손(Hands, 데이터베이스 실행)을 분리하고 있습니다.

우리가 구축할 스택(stack)은 다음과 같습니다:

*   **두뇌**: Ollama에서 실행되는 Mistral 7B. (Mistral은 더 큰 모델들보다 엄격한 JSON 명령어(instructions)를 더 잘 따른다는 이유로 선택했습니다).
*   **인터페이스(Interface)**: 사용자가 채팅하는 스트림릿(Streamlit) 앱(app).
*   **서버**: SQLite 연결을 유지하는 FastMCP를 사용하는 파이썬 스크립트(Python script).

![Diagram of the architecture showing Brain (Mistral 7B on Ollama), Interface (Streamlit), and Server (Python script with FastMCP and SQLite)](https://miro.medium.com/v2/resize:fit:700/1*n8H38vTq-8pSj5_Yx7LwGw.png)

## 1단계: 환경 설정 (도커는 당신의 친구)

‘내 컴퓨터에서는 잘 되는데(it works on my machine)’ 증후군을 피하기 위해 도커(Docker)를 사용할 것입니다. 우리의 UI와 데이터베이스를 담고 있으면서도, 호스트 머신(host machine)에서 실행 중인 Ollama 인스턴스(instance)와 통신할 수 있는 컨테이너가 필요합니다.

`docker-compose.yml` 파일을 생성하세요. 이것이 바로 ‘원클릭(one-click)’ 설정입니다:

```yaml
services:
  mcp-reporter:
    build: .
    container_name: pocket_analyst
    ports:
      - "8501:8501"
    volumes:
      - ./data:/app/data
    environment:
      # 중요: 컨테이너가 노트북의 Ollama와 통신할 수 있도록 합니다.
      - OLLAMA_HOST=http://host.docker.internal:11434
    extra_hosts:
      - "host.docker.internal:host-gateway"
```

## 3단계: 두뇌 (Mistral이 우세한 이유)

이제 대화를 관리하는 ‘호스트(Host)’인 스트림릿(Streamlit) 앱이 필요합니다.

모델 실행을 위해 Ollama를 사용하고 있습니다. 하지만 제 테스트에서 얻은 중요한 통찰은 다음과 같습니다: Llama 3 대신 Mistral 7B를 사용하세요.

Llama 3가 강력하긴 하지만, 작은 모델들은 종종 엄격한 JSON 형식 지정(formatting)에 어려움을 겪습니다. `{ “sql”: “…” }`를 요청하면 Llama는 “네! 여기 JSON입니다: `{ … }`”와 같이 응답할 수 있습니다. 이러한 불필요한 대화형 텍스트는 파서(parser)를 손상시킵니다.

Mistral 7B는 놀랍도록 규율이 잘 잡혀 있습니다. 군말 없이 도구를 트리거(trigger)하는 데 필요한 JSON을 출력합니다.

다음은 `src/app.py`의 로직입니다:

```python
# ... 임포트 및 스트림릿 페이지 설정 생략 ...

# 시스템 프롬프트(System Prompt)는 동작을 '프로그래밍'하는 곳입니다.
# 스키마와 출력 형식에 대해 모델에게 명시적으로 지시합니다.
SYSTEM_PROMPT = """
당신은 데이터 분석가(Data Analyst)입니다. 
1. 도구 호출(tool call)이 포함된 JSON 객체만 출력하십시오.
2. 형식은: { "tool": "query_database", "sql": "SELECT..." } 이어야 합니다.
3. 최종 답변에서는 항상 센트(cents)를 주요 통화 단위로 변환하십시오.
"""

# ... 주요 채팅 루프(chat loop) 내부 ...

if prompt := st.chat_input("질문하세요..."):
    # 1. 사용자의 요청을 Ollama로 보냅니다.
    # 참고: 구조화된 출력을 강제하기 위해 format='json'을 사용합니다.
    response = client.chat(
        model="mistral",
        messages=[
            {"role": "system", "content": SYSTEM_PROMPT},
            {"role": "user", "content": prompt}
        ],
        format="json" 
    )

    # 2. 도구 호출을 파싱합니다.
    # (전체 오류 처리 및 JSON 파싱 로직은 레포지토리에 있습니다.)
    tool_call = json.loads(response['message']['content'])

    if tool_call['tool'] == 'query_database':
        # 3. MCP 서버를 통해 SQL을 실행합니다.
        result = call_mcp_tool(tool_call['sql'])
        
        # 4. 데이터를 Mistral로 다시 전달하여 요약합니다.
        final_answer = client.chat(
            model="mistral",
            messages=[..., {"role": "user", "content": f"Data: {result}"}]
        )
        st.write(final_answer['message']['content'])
```

## 4단계: 데이터와 대화하기

`docker-compose up`을 실행하면 컨테이너가 시작됩니다. 이 컨테이너는 자동으로 `seed_data.py` 스크립트(레포지토리에 포함됨)를 실행하여 SQLite 데이터베이스(database)를 2,000개의 실제와 같은 ‘모의 스트라이프(Mock Stripe)’ 거래(transactions)로 채웁니다.

이제 이것이 정적 대시보드(dashboard)보다 왜 더 나은지 살펴보겠습니다.

### 테스트 1: “복잡한 필터”

기존 대시보드에서는 ‘독일에서 실패한 결제’를 보려면 상태(Status) 필터와 국가(Country) 필터가 필요했을 것입니다.

**프롬프트:**

어제 독일에서 실패한 신용카드 결제가 몇 건 있었나요?

**로직(Logic):**
Mistral은 스키마를 분석합니다. `country_code`, `status`, `payment_method`, `created_at`를 확인하고 다음을 생성합니다:

```sql
SELECT count(*) FROM transactions 
WHERE country_code = 'DE' 
AND status = 'failed' 
AND payment_method = 'card'
AND created_at > date('now', '-1 day');
```

### 테스트 2: “임시 변환”

이것이 바로 핵심 기능(killer feature)입니다. 프로덕트 매니저는 종종 자신들의 보고서를 위해 특정 방식으로 데이터가 포맷(format)되기를 원합니다.

**프롬프트:**

지난달 국가별 총 EUR 결제액을 계산하고 CSV 형식으로 포맷하세요.

**결과:**
MCP 도구는 원시 JSON(raw JSON)을 반환합니다. LLM은 이 JSON을 가져와 다음과 같이 다시 작성합니다:

```csv
"Country Code","Total EUR Payments"
"US",2873932
"DE",1093878
"GB",533471
"BR",492770
"FR",454523
"JP",348649
```

기존 대시보드의 치명적인 결함은 사용자가 물어볼 것이라고 예측한 질문에만 답한다는 것입니다. 이 아키텍처는 예상치 못했던 문제를 해결하며, 제한된 수의 하드코딩된 버튼을 언어의 무한한 유연성으로 대체합니다.
