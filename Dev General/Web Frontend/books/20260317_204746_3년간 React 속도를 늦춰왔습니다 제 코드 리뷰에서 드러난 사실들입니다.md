## 메모이제이션(Memoization)의 비용

`useMemo`의 기본 아이디어는 간단합니다: 비용이 많이 드는 계산의 결과를 캐시(cache)하여 React가 매 렌더링(render)마다 해당 계산을 다시 수행할 필요가 없도록 하는 것입니다.

```javascript
const sortedList = useMemo(() => {
  return items.sort((a, b) => a.name.localeCompare(b.name));
}, [items]);
```

매 렌더링마다 정렬하는 대신, React는 결과를 저장하고 `items`가 변경될 때만 다시 계산합니다. 당연한 것처럼 들리죠?

제가 간과하고 있던 부분은 바로 이것입니다: `useMemo` 자체에도 비용이 든다는 것입니다.

React는 캐시된 값을 저장하고, 매 렌더링마다 의존성(dependencies)을 비교하며, 캐시 생명주기(lifecycle)를 관리합니다.

간단한 계산에는 과잉입니다.

```javascript
// 이것은 거의 확실히 메모이제이션할 가치가 없습니다.
const fullName = useMemo(() => {
  return `${user.firstName} ${user.lastName}`;
}, [user.firstName, user.lastName]);

// 그냥 이렇게 하세요.
const fullName = `${user.firstName} ${user.lastName}`;
```

문자열 연결은 마이크로초 단위로 처리됩니다. 메모이제이션 래퍼(wrapper)의 비용이 계산 비용보다 더 듭니다. 제 코드베이스에는 이런 경우가 수십 개나 있었습니다.

저는 이런 코드를 수십 개나 가지고 있었고, 최악의 부분은 제가 그것들을 자랑스럽게 생각했다는 것입니다. 저는 제가 철저하게 코드를 작성하고 있다고 생각했습니다.

## `useMemo`를 사용할 명확한 이유를 가지세요

`useMemo`를 사용할 때는 명확한 이유가 있어야 합니다. 그 이유는 **측정 가능한 성능 문제**이거나 **참조 안정성(referential stability) 요구사항**이어야 합니다.

**사례 1: 비용이 많이 드는 계산**

계산이 정말로 비용이 많이 드는 경우(예: 대규모 데이터셋 필터링, 복잡한 알고리즘 실행, 많은 데이터 처리)에는 메모이제이션이 합리적입니다. 핵심 단어는 **측정 가능한**입니다. 프로파일링을 하지 않았다면, 그것이 비용이 많이 드는지 알 수 없습니다.

```javascript
const filteredProducts = useMemo(() => {
  return products
    .filter(p => p.category === selectedCategory)
    .filter(p => p.price >= minPrice && p.price <= maxPrice)
    .sort((a, b) => b.rating - a.rating);
}, [products, selectedCategory, minPrice, maxPrice]);
```

**사례 2: 참조 안정성**

이것은 대부분의 개발자가 놓치는 부분이며, `useMemo`가 실제로 가장 가치 있는 부분입니다.

이것은 제가 정말로 몰랐던 부분이었습니다. 저는 잘못된 이유(계산 절약)로 `useMemo`를 사용하고 있었고, 때로는 우연히 올바른 결과를 얻기도 했습니다. 참조 안정성을 이해하게 되자, 과거의 혼란스러운 버그(bug)들이 갑자기 이해되기 시작했습니다.

자바스크립트(JavaScript)에서 `[] !== []` 이고 `{}` `!== {}` 입니다. 내용이 동일한 두 배열 또는 객체는 동일한 참조(reference)가 아닙니다. 이것은 React의 의존성 비교가 참조 동등성(reference equality)에 기반하기 때문에 중요합니다.

```javascript
// useMemo 없이 — 매 렌더링마다 새로운 객체 참조가 생성됩니다.
// 이것은 아무것도 변경되지 않았더라도 자식 컴포넌트와 useEffect가 실행되도록 만듭니다.
const config = { threshold: 0.5, maxItems: 100 };

// useMemo 사용 — 안정적인 참조, 값이 실제로 변경될 때만 자식이 다시 렌더링됩니다.
const config = useMemo(() => ({
  threshold: 0.5,
  maxItems: 100
}), []);
```

메모이제이션된 자식 컴포넌트(component)에 객체나 배열을 `prop`으로 전달하거나, `useEffect`의 의존성으로 포함할 경우, 계산 시간을 절약하기 위해서가 아니라 참조 안정성을 유지하기 위해 `useMemo`가 거의 확실히 필요합니다.

## `React.memo`와 혼동하지 마세요

`React.memo`는 값을 감싸는 것이 아니라 컴포넌트를 감쌉니다. 이것은 React에게 "이 컴포넌트의 `prop`이 변경된 경우에만 다시 렌더링하라"고 지시합니다.

```javascript
const UserCard = React.memo(({ user, onSelect }) => {
  return (
    <div onClick={() => onSelect(user.id)}>
      {user.name}
    </div>
  );
});
```

`React.memo` 없이는 `UserCard`는 `user`와 `onSelect`가 변경되지 않았더라도 부모 컴포넌트가 다시 렌더링될 때마다 다시 렌더링됩니다. `React.memo`를 사용하면 React는 다시 렌더링할지 결정하기 전에 `prop`을 비교합니다.

하지만 여기에 함정이 있습니다: `React.memo`는 얕은 비교(shallow comparison)를 사용합니다. 기본적으로 매 렌더링마다 새로운 객체나 함수 참조를 전달하면, 메모이제이션은 무용지물이 됩니다.

```javascript
// 이것은 React.memo를 망가뜨립니다 — 부모가 다시 렌더링될 때마다 새로운 함수 참조가 생성됩니다.
<UserCard
  user={user}
  onSelect={(id) => handleSelect(id)}
/>

// 이것은 작동합니다 — useCallback을 통한 안정적인 참조
const handleSelectUser = useCallback((id) => handleSelect(id), []);
<UserCard
  user={user}
  onSelect={handleSelectUser}
/>
```

이것이 `useMemo`, `React.memo`, `useCallback`이 함께 나타나는 경향이 있는 이유입니다. 이들은 모두 참조 불안정성(reference instability)으로 인한 불필요한 작업을 방지하는 동일한 근본적인 문제를 해결합니다.

## 함수를 위한 `useMemo` — `useCallback`

`useCallback`은 본질적으로 함수를 위한 `useMemo`입니다. 이것은 함수 참조를 메모이제이션하여 렌더링 전반에 걸쳐 안정적으로 유지합니다.

```javascript
// useCallback 없이 — 매 렌더링마다 새로운 함수가 생성됩니다.
const handleSubmit = (values) => {
  submitForm(values);
};

// useCallback 사용 — 의존성이 변경되지 않는 한 동일한 참조를 유지합니다.
const handleSubmit = useCallback((values) => {
  submitForm(values);
}, [submitForm]);
```

주요 사용 사례는 메모이제이션된 자식 컴포넌트에 콜백(callback)을 전달하거나 `useEffect` 의존성에 포함시키는 것입니다. `useCallback`이 없으면 `React.memo` 최적화를 깨뜨리거나 무한 `useEffect` 루프(loop)를 유발할 수 있습니다.

```javascript
// 고전적인 무한 루프 — handleFetch는 매 렌더링마다 새로운 참조입니다.
useEffect(() => {
  handleFetch();
}, [handleFetch]);

// useCallback으로 수정
const handleFetch = useCallback(() => {
  fetchData(userId);
}, [userId]);

useEffect(() => {
  handleFetch();
}, [handleFetch]);
```

한 가지 경험칙(rule of thumb): `prop`이나 의존성으로 어디에도 전달되지 않는 함수에 `useCallback`을 작성하고 있다면, 아마도 필요 없을 것입니다.

## 다시 렌더링 프로파일링

이러한 도구들을 사용하기 전에, 문제가 있는지 알아야 합니다. 추측만으로 모든 곳에 메모이제이션을 적용하여 실제로는 나아지지 않은 성능을 얻게 된 것이 저의 경우입니다.

`React DevTools Profiler`는 올바른 시작점입니다. 사용자 상호작용(user interaction)을 기록하면, 어떤 컴포넌트가 왜 다시 렌더링되었고 얼마나 오래 걸렸는지 정확히 보여줍니다. 눈에 띄는 이유 없이 자주 렌더링되는 컴포넌트를 찾아보세요.

실제 문제를 나타내는 몇 가지 패턴:

*   부모 입력창(input)에서 키 입력(keystroke)이 있을 때마다 컴포넌트가 다시 렌더링되는데, 실제로는 해당 입력값을 사용하지 않는 경우
*   목록에서 항목 하나만 변경되었는데도 모든 항목이 다시 렌더링되는 경우
*   `useEffect`가 예상보다 더 자주 실행되는 경우

이러한 패턴이 보이지 않거나 렌더링 시간이 몇 밀리초(milliseconds) 이하라면, 메모이제이션은 아마 도움이 되지 않을 것입니다. 추상화(abstraction)의 비용이 이점보다 더 큽니다.

## 메모이제이션을 조용히 망가뜨리는 흔한 실수들

이러한 도구들을 올바르게 사용하려 해도, 쉽게 잘못 설정하여 아무런 이점도 얻지 못할 수 있습니다.

**JSX 내 인라인 객체**

```javascript
// 메모이제이션을 망가뜨립니다 — 매 렌더링마다 새로운 객체가 생성됩니다.
<Chart options={{ color: 'blue', width: 300 }} />
// 수정됨
const chartOptions = useMemo(() => ({ color: 'blue', width: 300 }), []);
<Chart options={chartOptions} />
```

**누락되거나 잘못된 의존성**

```javascript
// 오래된 클로저(stale closure) 버그 — 오래된 userId를 읽습니다.
const fetchUser = useCallback(() => {
  api.get(`/users/${userId}`);
}, []); // userId가 의존성에서 누락됨
// 올바름
const fetchUser = useCallback(() => {
  api.get(`/users/${userId}`);
}, [userId]);
```

**너무 상위 레벨에서 메모이제이션**

전체 페이지 컴포넌트를 `React.memo`로 감싸는 것은 거의 도움이 되지 않습니다. 해당 `prop`은 거의 항상 변경되기 때문입니다.

메모이제이션은 안정적인 `prop`을 받고 자주 렌더링되는 리프 컴포넌트(leaf component)에서 가장 효과적입니다.

## 의사 결정 프레임워크

*   **`useMemo`의 경우** — 이 계산이 비용이 많이 드나요, 아니면 참조 안정성이 중요한가요? 둘 다 아니라면 건너뛰세요.
*   **`React.memo`의 경우** — 이 컴포넌트가 자주 렌더링되나요? `prop`이 실제로 안정적인가요? `prop`이 어쨌든 자주 변경된다면, 이점 없이 비용만 추가됩니다.
*   **`useCallback`의 경우** — 이 함수가 메모이제이션된 컴포넌트의 `prop`으로 전달되나요? `useEffect`의 의존성인가요? 둘 다 아니라면 건너뛰세요.

## 결론

모든 것을 `useMemo`로 감싸고 싶었던 본능은 좋은 의도에서 비롯되었습니다: 저는 고성능 코드를 작성하고 싶었습니다. 하지만 React에서 성급한 최적화(premature optimization)는 시간 낭비일 뿐만 아니라, 적극적으로 복잡성을 추가합니다:

*   추적해야 할 더 많은 의존성.
*   오래된 클로저(stale closure) 버그가 숨어들 더 많은 공간.
*   나중에 코드를 읽는 사람에게 더 많은 인지 부하(cognitive overhead).

그 리뷰 이후, 저는 코드베이스를 살펴보며 제가 작성했던 `useMemo` 호출의 약 80%를 제거했습니다. 앱은 느려지지 않았습니다. 코드는 더 읽기 쉬워졌습니다. 그 사실이 조금 쓰라렸습니다.
