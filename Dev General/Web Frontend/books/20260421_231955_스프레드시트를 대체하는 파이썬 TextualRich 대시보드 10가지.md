# 스프레드시트를 대체하는 파이썬 Textual/Rich 대시보드 10가지

운영, 분석 및 일상적인 의사 결정을 위해 복잡한 스프레드시트보다 뛰어난, 빠르고 가독성 높은 파이썬 터미널 대시보드를 구축하세요.

파이썬으로 만들 수 있는 Textual 및 Rich 기반의 실용적인 대시보드 10가지는 프로덕션 환경에 적합한 사용자 경험(UX), 실시간 데이터, 그리고 복사-붙여넣기 가능한 코드를 제공하여 스프레드시트를 대체할 수 있습니다.

스프레드시트는 놀랍도록 유용하지만, 그렇지 않은 순간이 옵니다. 실시간 데이터, 안전 장치(guardrails), 또는 새벽 2시에도 문제없는 사용자 경험(UX)이 필요할 때, 스프레드시트의 격자(grid)는 굳어버린 시멘트처럼 다루기 어려워집니다. Textual과 Rich로 구축된 터미널 대시보드는 스크립트의 속도와 제품 사용자 인터페이스(UI)의 명확성을 제공합니다. 그리고 네, 이들은 전체 스프레드시트를 대체할 수 있습니다.

아래는 제가 직접 구축했거나 팀이 구축하는 데 도움을 주었던 10가지 대시보드입니다. 이들은 스프레드시트 작업을 터미널로 옮겨 사람들이 실제로 즐겁게 사용할 수 있도록 했습니다.

## 1) KPI 그리드: "지금 어떤 지표에 문제가 있나요?"

**사용 사례:** 경영진 스탠드업 회의, 일상적인 운영.
**스프레드시트보다 나은 이유:** 실시간 지표, 키보드 우선 내비게이션, 셀 깨짐 걱정 없음.

```python
# pip install textual rich
from textual.app import App, ComposeResult
from textual.widgets import DataTable
from rich import box

class KPIs(App):
    CSS = "DataTable {height: 100%;}"

    def compose(self) -> ComposeResult:
        t = DataTable(zebra_stripes=True)
        t.add_columns("Metric", "Today", "Target", "Δ")
        yield t

    def on_mount(self) -> None:
        t = self.query_one(DataTable)
        rows = [("Signups", 412, 390, "+22"),
                ("Failed payments", 9, 5, "−4"),
                ("Latency p95 (ms)", 188, 200, "+12")]
        for r in rows: t.add_row(*map(str, r))
        t.styles.border = box.SIMPLE

if __name__ == "__main__":
    KPIs().run()
```

**전문가 팁:** 조건부 스타일과 단축키(j/k)를 사용하여 Δ 셀에 색상을 적용하고, 문제가 있는(red) 항목으로 바로 이동할 수 있습니다.

## 2) 실시간 CSV 모니터: "스프레드시트, 그러나 실시간으로"

**사용 사례:** CSV로 지표를 덤프하는 팀(크론, DB 익스포트).
**스프레드시트보다 나은 이유:** 자동 새로 고침, 스키마 변경 감지, 마우스 사용 불필요.

```
┌──────────────────────────┐
 │  /var/exports/metrics.csv│  ← tail + schema watch
 ├───────────┬──────────────┤
 │ metric    │ value        │
 ├───────────┼──────────────┤
 │ orders    │ 124          │
 │ refunds   │ 3            │
 │ aov       │ 57.90        │
 └───────────┴──────────────┘
```

Textual의 `watch_files()` 기능을 사용하여 `DataTable`을 새로 고치고, 스키마 변경 시 깜빡이게 할 수 있습니다.

## 3) 인시던트 지휘 보드: "PagerDuty의 최소 버전"

**사용 사례:** 운영 인시던트, 온콜(on-call) 로테이션.
**스프레드시트보다 나은 이유:** 역할 명확성, 타이머, 액션 로그.

**특징:** 타임라인 + 시계가 있는 왼쪽 패널, 런북 단계, 담당자, 노트 피드로 분할된 오른쪽 패널로 구성됩니다. `a` 키를 눌러 메모를 추가하고, `o` 키로 재할당하며, `c` 키로 미리 만들어진 상태 업데이트를 복사할 수 있습니다(더 이상 셀을 찾아다닐 필요 없음).

## 4) 영업 파이프라인 칸반

**사용 사례:** 공유 스프레드시트로 거래를 추적하는 중소기업(SMB) 팀.
**스프레드시트보다 나은 이유:** 드래그 없이 키보드로 이동, 단계별 작업 진행 중(WIP) 한도, 자동 합계.

```python
from textual.app import App
from textual.widgets import Static
from textual.containers import Grid

class Deal(Static): pass

class Pipeline(App):
    CSS = """
    Grid { grid-size: 4 1; grid-gutter: 1; }
    .col { border: tall $accent; padding: 1; }
    """

    def compose(self):
        yield Grid(
            Static("New\n— ACME\n— Initech", classes="col"),
            Static("Qualified\n— BigCo", classes="col"),
            Static("Negotiation", classes="col"),
            Static("Won / Lost", classes="col"),
        )

Pipeline().run()
```

단축키(>/<)를 추가하여 `Deal` 위젯을 열 간에 이동시키고, 각 단계별 예상 가치 합계를 자동으로 다시 계산하는 푸터(footer)를 만들 수 있습니다. 더 이상 수식이 깨질 걱정이 없습니다.

## 5) 용량 히트맵: "시스템이 과부하에 걸릴까요?"

**사용 사례:** 사이트 신뢰성 엔지니어링(SRE) 예측.
**스프레드시트보다 나은 이유:** 터미널에서 트루 컬러(true color) 히트맵, 키보드 줌, 경고 기능.

Rich의 `Console`을 활용하여 활용률(예: 0-100%)에 따라 블록에 색상을 입히세요. Textual의 `Slider`와 결합하여 시간 범위를 스크럽(scrub)할 수 있습니다. 셀이 임계값을 넘어서면 토스트(toast) 알림을 띄우고 해당 위치로 포커스를 이동시킵니다.

## 6) 손익계산서(P&L) 탐색기: "피벗 테이블 불안감 없는 재무 관리"

**사용 사례:** 창업자 및 재무 담당자.
**스프레드시트보다 나은 이유:** 계층 구조, 빠른 필터, 저장된 뷰.

계정을 Rich의 `Tree` 위젯으로 트리(tree) 구조화하고 접을 수 있는 노드를 만듭니다: 매출(Revenue) → 구독/서비스(Subscriptions / Services), 매출원가(COGS), 운영비(OpEx). 월별 필터링을 위한 Textual `Input`과 시나리오(실제 / 예산 / 예측) 선택을 위한 `Select`를 추가하세요. 노드를 접을 때마다 합계가 다시 계산되어, 행을 놓치는 `SUM` 범위 문제 없이 정확한 결과를 제공합니다.

## 7) A/B 테스트 컨트롤 타워

**사용 사례:** 성장 및 제품 분석.
**스프레드시트보다 나은 이유:** 즉각적인 통계, 안전 장치(guardrails), 수동 수식 불필요.

**패널:**

*   실험 목록(상태, 시작 날짜, 실험군).
*   베이즈(Bayesian) / 홀드아웃(HT) 빠른 조회 기능을 갖춘 지표(클릭률(CTR), 전환율, 사용자당 평균 매출(ARPU)).
*   의사 결정 힌트(계속 실행, 승리 시 중단, 무의미할 경우 중단).

DuckDB 또는 Polars에서 집계(aggregates)를 미리 계산한 다음 Rich `Table`로 가져올 수 있습니다. `?` 키를 할당하여 통계적 가정을 표시하면 사람들이 p-값을 오해하는 것을 방지할 수 있습니다.

## 8) 백로그 번다운 & 플로우

**사용 사례:** 배포 리드(delivery leads).
**스프레드시트보다 나은 이유:** 번다운(burndown) + 처리량(throughput) + 진행 중 작업(WIP)을 한 화면에 결합.

```
[Sprint 14]  Tickets: 68 → 12     Velocity: 22/wk
 ┌─────── Burndown ───────┐ ┌── Flow (last 7d) ┐
 │#######.....            │ │ToDo: ▓▓▓         │
 │#####........           │ │Doing: ▓▓▓▓▓      │
 │###............         │ │Review: ▓▓        │
 │#..............         │ │Done: ▓▓▓▓▓▓▓     │
 └────────────────────────┘ └──────────────────┘
```

번다운은 단지 문자로 표현됩니다. 중요한 것은 추세이며, 픽셀 아트가 아닙니다. 값은 트래커 API에서 가져오므로, 오후 6시에 누군가 셀 업데이트를 "잊어버리는" 일은 없습니다.

## 9) ETL 작업 실행기 및 큐

**사용 사례:** "시트 매크로 좀 실행해 줄 수 있나요?"라는 요청을 대체하려는 데이터 팀.
**스프레드시트보다 나은 이유:** 동시성 제어, 재시도, 로그, 소유권.

**레이아웃:**

*   왼쪽: 작업 카탈로그(이름, SLA, 담당자).
*   중앙: 상태 및 기간을 포함한 활성 실행 작업.
*   오른쪽: 큐 깊이, 재시도율, 마지막 오류.

액션을 키에 연결하세요: `r`은 실행, `R`은 실패한 작업을 동일한 매개변수로 재실행, `l`은 최신 로그 보기(파일 패널 테일), `o`는 설정 열기. 한 번 구축하면 CSV 매크로를 영원히 관리할 필요가 없습니다.

## 10) 개인 재무 CLI: "실제로 사용하게 될 예산 관리"

**사용 사례:** 개인 창업가, 학생, 스프레드시트에 거부감을 느끼는 모든 사람.
**스프레드시트보다 나은 이유:** 은행 동기화 + 규칙 + 집중된 워크플로우.

계정 / 예산 / 인사이트를 위한 Textual 사이드바, 거래 내역을 위한 메인 패널, 그리고 규칙("식료품 → 카테고리 식비") 생성을 위한 오버레이 대화상자. 색상을 아껴서 사용하세요. 너무 많은 색상으로 채워진 스프레드시트보다 뇌가 패턴을 더 빨리 학습할 것입니다.

## Textual/Rich가 스프레드시트를 이기는 방법 (실제로)

*   **안전 장치(Guardrails) > 그리드:** 자유로운 형식의 셀 대신 흐름과 제약 조건을 인코딩합니다. 수식이 깨질 염려가 없습니다.
*   **실시간 상태:** 스프레드시트는 수동 새로 고침이나 취약한 가져오기를 요구하는 반면, Textual 앱은 기본적으로 반응형(reactive)입니다.
*   **가치 창출 속도:** 파이썬(Python)은 API, CSV, DuckDB, Postgres 등 팀이 이미 보유한 모든 소스에서 데이터를 가져올 수 있게 해줍니다.
*   **인체 공학(Ergonomics):** 키보드 단축키와 집중된 패널은 셀을 찾아다니는 것보다 빠릅니다. 사용자 인터페이스(UI)는 작업에 맞춰지며, 그 반대가 아닙니다.

## 아키텍처 스케치 (아주 간단합니다)

```
데이터 소스(들) ──► 어댑터 (CSV/SQL/API) ──► 모델 (pydantic/dataclasses)
                                      │
                                      ├─► 캐시 (DuckDB/SQLite/Redis)
                                      │
  키보드 이벤트 ──► Textual 위젯 ┴─► 뷰 (테이블, 트리, 히트맵)
                                     └─► 액션 (작업 실행, 로그 열기)
```

이것은 의견이 반영된 MVC(Model-View-Controller) 패턴입니다: 어댑터와 모델을 단순하게 유지하고, UI를 반응형으로 유지하며, I/O 중에는 이벤트 루프를 절대 차단하지 마세요.

## 적용할 수 있는 작은 "스타터"

```python
from textual.app import App, ComposeResult
from textual.widgets import Header, Footer, DataTable, Input, Select
from textual import events
import csv, pathlib

class SheetReplacer(App):
    BINDINGS = [("q", "quit", "Quit"), ("/", "focus_filter", "Filter")]

    def __init__(self, csv_path="data.csv"):
        super().__init__()
        self.csv_path = pathlib.Path(csv_path)

    def compose(self) -> ComposeResult:
        yield Header(show_clock=True)
        yield Input(placeholder="Filter…")
        yield Select(options=[("All","all"),("Errors","err")])
        yield DataTable(zebra_stripes=True)
        yield Footer()

    def on_mount(self):
        self.load_table()

    def load_table(self):
        t = self.query_one(DataTable)
        t.clear(columns=True)
        with self.csv_path.open() as f:
            rows = list(csv.DictReader(f))
        if rows:
            t.add_columns(*rows[0].keys())
            for r in rows: t.add_row(*r.values())

    def action_focus_filter(self):
        self.query_one(Input).focus()

    async def on_input_changed(self, event: Input.Changed):
        q = event.value.lower()
        table = self.query_one(DataTable)
        table.show_row_labels = False
        for i, row in enumerate(table.rows):
            visible = q in " ".join(map(str, row.get_values())).lower()
            table.set_row_visibility(i, visible)

if __name__ == "__main__":
    SheetReplacer().run()
```

이것은 필터링 가능하고 키보드 친화적이며, 합계, 색상 규칙 또는 쓰기(write-back) 기능으로 확장할 준비가 된 단일 파일 "안전 장치(guardrails)를 갖춘 시트 뷰어"입니다.

## 솔직히 말해서: 스프레드시트를 대체하지 말아야 할 때

*   매일 변경되고 수작업이 필요한 임시 모델링.
*   댓글과 변경 추적(track-changes) 기능이 중요한 대규모 다중 사용자 협업.
*   일회성 작업. (2분짜리 스프레드시트로 충분할 때 앱을 배포하지 마세요.)

하지만 명확한 흐름을 가진 반복적인 의사 결정에는 어떨까요? Textual/Rich 대시보드가 더 안정적이고 빠르며, 쉽게 망가지지 않습니다.

## 결론

스프레드시트는 연산을 대중화했습니다. Textual과 Rich는 인터페이스를 대중화합니다. 이는 깨지기 쉬운 그리드와 사람들이 사랑하는 도구의 차이입니다. 위의 대시보드 중 하나(KPI 그리드, 인시던트 보드, 또는 A/B 컨트롤 타워)부터 시작하여 점차 확장해 보세요. 미래의 당신(과 당신의 팀)이 감사할 것입니다.
