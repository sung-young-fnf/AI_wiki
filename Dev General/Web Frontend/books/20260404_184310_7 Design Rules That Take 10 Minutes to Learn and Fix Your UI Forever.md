7 Design Rules That Take 10 Minutes to Learn and Fix Your UI Forever

코드는 깔끔하고 로직은 탄탄하며 앱은 완벽하게 작동할 수 있습니다. 하지만 사용자가 앱을 여는 순간, 무언가 잘못되었다는 느낌을 받을 때가 있습니다. 사용자도, 개발자도 정확히 무엇이 문제인지 짚어내지는 못하지만, 그저 '보기에 좋지 않다'라고 느끼는 것이죠.

이러한 느낌에는 반드시 원인이 있습니다. 대개 한 가지 이상의 원인이 복합적으로 작용합니다. 다음은 프로그래밍 튜토리얼에서는 알려주지 않지만, '작동만 하는 UI'와 '전문적인 UI'를 가르는 디자인 규칙들입니다.

### #7 — 순수한 검정색(#000000)을 사용하지 마세요

`#000000`은 논리적으로 보입니다. 대비가 극대화되어 가독성이 가장 좋을 것 같기 때문입니다. 하지만 거의 모든 전문 디자인 도구와 실제 서비스 중인 UI, 주요 디자인 시스템(Design System)에서는 이를 피합니다.

흰색 배경에 순수한 검정색 텍스트를 사용하면 시간이 지날수록 눈의 피로(Eye strain)를 유발합니다. 흰색은 100%의 밝기를 가지고 검정색은 0%입니다. 이 큰 격차로 인해 망막은 계속해서 적응해야 하며, 이는 읽는 시간이 길어질수록 불편함을 줍니다.

**잘못된 예시:**
```css
body {
  background-color: #ffffff;
  color: #000000;
}
```

**개선된 예시:**
```css
body {
  background-color: #f5f5f5;
  color: #111111;
}
```
에디터에서는 미묘한 차이처럼 보일지 몰라도, 사용자가 체감하는 가독성의 차이는 매우 큽니다.

### #6 — 무작위적인 간격을 피하세요

스타일시트의 여백(Margin)과 패딩(Padding) 값을 확인해 보세요. `13px`, `22px`, `7px` 같은 숫자들이 여기저기 흩어져 있다면 그것이 문제입니다. 개별 요소로는 괜찮아 보일지 몰라도, 전체적으로는 일관성이 없어 보이며 사용자는 이유를 모르더라도 본능적으로 불협화음을 느낍니다.

해결책은 **8px 그리드 시스템(8px Grid System)**입니다. 애플의 휴먼 인터페이스 가이드라인(Human Interface Guidelines)과 구글의 머티리얼 디자인(Material Design) 모두 이를 권장합니다. 모든 수치가 수학적으로 연관되어 있기 때문에 시각적으로 정렬된 느낌을 줍니다.

임의의 값을 사용하는 대신, 간격 척도(Spacing scale)를 한 번 정의하고 사용하세요.

**개선된 예시:**
```css
:root {
  --space-1: 8px;
  --space-2: 16px;
  --space-3: 24px;
  --space-4: 32px;
}

.card {
  padding: var(--space-2);
  margin-bottom: var(--space-3);
  gap: var(--space-1);
}
```
무작위적인 간격은 개발자가 만든 UI가 '왠지 모르게 어설퍼 보이는' 가장 흔한 이유입니다.

### #5 — 너무 많은 폰트를 사용하지 마세요

많은 개발자가 제목용 폰트, 본문용 폰트, 그리고 흥미로워 보이는 세 번째 폰트까지 섞어 쓰곤 합니다. 그러면 페이지가 마치 짜깁기한 문서처럼 보일 수 있습니다.

규칙은 간단합니다. **최대 두 개의 폰트 패밀리(Font families)**만 사용하세요. 계층 구조(Hierarchy)를 만들고 싶다면 같은 폰트 내에서 서로 다른 굵기(Weight)를 활용하는 것이 훨씬 효과적입니다.

**개선된 예시:**
```css
:root {
  --font-base: 'Inter', sans-serif;
}

h1 {
  font-family: var(--font-base);
  font-weight: 700;
  font-size: 2.5rem;
}

p {
  font-family: var(--font-base);
  font-weight: 400;
  font-size: 1rem;
}
```
Inter, Plus Jakarta Sans, DM Sans 같은 폰트는 범용성이 높고 수많은 앱에서 검증된 폰트입니다. 하나의 폰트와 다양한 굵기만으로도 훨씬 의도적이고 세련된 UI를 만들 수 있습니다.

### #4 — 컬러 시스템을 구축하세요

버튼에 파란색을 쓰고, 링크에는 또 다른 파란색을 쓰고, 배지(Badge)에는 어울리지 않는 청록색을 쓰다 보면 서로 연관성 없는 수많은 색상이 남발됩니다.

여기에 적용할 수 있는 **60-30-10 법칙**이 있습니다.
*   **60%:** 지배적인 무채색 (보통 배경색)
*   **30%:** 보조 색상 (표면, 카드 등)
*   **10%:** 강조색(Accent color) (주요 버튼 및 핵심 하이라이트)

**개선된 예시:**
```css
:root {
  /* 60% — 배경 */
  --color-bg: #f8fafc;
  /* 30% — 표면, 카드, 사이드바 */
  --color-surface: #ffffff;
  --color-border: #e2e8f0;
  /* 10% — 강조색, 제한적으로 사용 */
  --color-accent: #3b82f6;
  
  --color-text-primary: #111827;
}
```
많은 UI가 실패하는 이유는 강조색을 남발하여 정작 중요한 요소가 눈에 띄지 않기 때문입니다.

### #3 — 텍스트 계층 구조를 명확히 하세요

이 부분은 미묘하지만 치명적입니다. 만약 H1 크기가 28px이고 H2가 24px라면, 뇌는 이를 거의 같은 크기로 인식합니다. 페이지가 평면적으로 느껴지고 사용자의 시선을 끄는 지점이 사라집니다.

제목 수준 간의 **최소 비율은 1.25배**여야 하며, 명확성을 위해서는 **1.5배**가 더 좋습니다.

**개선된 예시 (1.5배 비율):**
```css
:root {
  --text-base: 1rem;    /* 16px — 본문 */
  --text-xl: 1.5rem;    /* 24px — h3 */
  --text-2xl: 2rem;     /* 32px — h2 */
  --text-4xl: 3rem;     /* 48px — h1 */
}
```
H1은 페이지 내의 다른 어떤 요소와도 혼동되지 않을 만큼 확실히 커야 합니다.

### #2 — 행 너비(Line Length)를 조절하세요

내용을 전체 너비 컨테이너에 담으면 텍스트가 화면 끝에서 끝까지 늘어납니다. 대형 모니터에서는 한 줄에 120자에서 160자까지 길어지기도 합니다.

이는 가독성을 해칩니다. 눈이 다음 줄의 시작점을 찾기 위해 너무 멀리 이동해야 하기 때문입니다. 사람들은 읽는 위치를 놓치고, 읽는 속도가 느려지며, 결국 페이지를 떠나게 됩니다.

본문 텍스트의 **이상적인 한 줄 길이는 50~75자** 사이입니다.

**개선된 예시:**
```css
.content {
  max-width: 65ch; /* 65자 내외의 너비로 제한 */
  margin-inline: auto;
}
```
콘텐츠 컨테이너에 `max-width: 65ch`를 설정하는 것만으로도 가독성이 즉각적으로 향상됩니다.

### #1 — 시각적 계층 구조(Visual Hierarchy)를 완성하세요

앞서 언급한 모든 규칙을 하나로 묶어주는 핵심입니다. 모든 버튼이 똑같이 중요해 보인다면 사용자는 어디를 클릭해야 할지 알 수 없습니다.

모든 요소에 동일한 무게감, 색상 강도, 크기를 부여하지 마세요.

**개선된 예시:**
*   **주요(Primary) 버튼:** 페이지당 하나, 가장 눈에 띄게 (강조색 사용)
*   **보조(Secondary) 버튼:** 눈에 띄지만 한 단계 뒤로 (테두리만 사용 등)
*   **고스트(Ghost) 버튼:** 가장 낮은 우선순위 (배경과 테두리 없음)

강력한 주요 버튼 하나, 지배적인 제목 하나, 명확한 초점 하나를 만드세요. 나머지 요소들은 시각적으로 한 걸음 물러나 있어야 합니다.

### 맺음말

웹 개발 과정에서 이런 디자인적 사고를 가르쳐주는 곳은 드뭅니다. 보통 HTML, CSS, JavaScript, 프레임워크 등을 배우며 '작동하는 것'을 만드는 법에 집중하기 때문입니다.

하지만 1인 개발자나 사이드 프로젝트를 진행하는 이들에게 디자인은 결국 본인의 몫입니다. 이러한 규칙들은 디자인 전공 지식을 요구하지 않습니다. 단지 이런 규칙이 **존재한다는 사실을 아는 것**만으로도 충분합니다. 이제 여러분은 그 방법을 알게 되었습니다.
