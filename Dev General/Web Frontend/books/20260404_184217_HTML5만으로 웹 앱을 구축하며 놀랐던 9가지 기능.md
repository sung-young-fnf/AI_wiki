# HTML5만으로 웹 앱을 구축하며 놀랐던 9가지 기능

3개월 전, 저는 프런트엔드 코드를 완전히 삭제했습니다. 리팩터링(Refactoring)이나 최적화가 아니라, 말 그대로 전부 지워버렸습니다.

해당 앱은 4만 줄의 자바스크립트(JavaScript) 코드로 이루어진 비대한 대시보드였습니다. 상태 관리 버그, 유효성 검사 문제, 복잡한 모달 로직 등 작은 기능을 하나 추가하는 것조차 수술처럼 느껴지는 프로젝트였습니다.

좌절감 섞인 도전 정신으로 저는 스스로에게 질문을 던졌습니다. "이 내부 도구 전체를 오직 HTML5만으로 만든다면 어떨까?"

React도, UI 라이브러리도, 유효성 검사 프레임워크도 없이 오직 구조와 네이티브(Native) 동작, 그리고 절제만으로 구축해 보았습니다. 처음에는 제약이 많을 것이라 예상했지만, 결과는 오히려 강력했습니다.

자바스크립트로 작성하던 업무의 절반을 자동으로 해결해 준, 저를 놀라게 한 9가지 HTML5 기능을 소개합니다.

### 1. 모달 시스템을 대체한 `<dialog>` 태그

과거에는 모달 컴포넌트를 만들기 위해 다음과 같은 기능들을 직접 구현해야 했습니다.
*   백드롭(Backdrop) 처리
*   포커스 트래핑(Focus trapping, 포커스가 모달 밖으로 나가지 않게 함)
*   ESC 키 입력 로직
*   접근성(Accessibility) 패치

이제는 `<dialog>` 요소 하나로 해결됩니다. 이 태그는 내장된 백드롭을 제공하며, ESC 키를 누르면 자동으로 닫히고, 포커스를 내부에 유지하며, 스크린 리더가 즉시 인식합니다. 2026년에도 여전히 모달 라이브러리를 임포트하고 있다면 `<dialog>`를 다시 살펴봐야 합니다.

### 2. 자바스크립트 계층을 없앤 폼 유효성 검사(Form Validation)

모든 항목에 대해 유효성 검사 함수를 작성하는 대신, 브라우저의 기본 제약 조건을 활용했습니다. `min`, `max`, `required`, `minlength` 등의 속성을 사용하면 잘못된 입력을 즉시 차단하고 백엔드 에러 노이즈를 줄이며 모바일 사용자 경험(UX)을 자동으로 개선할 수 있습니다. 브라우저는 시중의 어떤 유효성 검사 라이브러리보다 오랫동안 검증되어 왔습니다.

### 3. 토글 UI를 위한 `<details>`와 `<summary>`

이 조합은 드롭다운 로직의 절반을 대체했습니다. 별도의 상태 관리나 CSS 트릭 없이도 클릭 시 토글되는 기능을 구현할 수 있으며, 키보드 접근성도 기본으로 보장됩니다. 자바스크립트 없이 상호작용을 구현하는 것, 이것이 바로 자동화입니다.

### 4. 네이티브 날짜 및 시간 입력

한때 300KB에 달하는 날짜 선택기(Date picker) 라이브러리를 사용했던 적이 있습니다. 하지만 `<input type="datetime-local">`을 사용하면 모바일에 최적화된 선택기와 네이티브 유효성 검사를 제공받을 수 있으며, 시간대 파싱으로 인한 골칫거리도 사라집니다. 외부 의존성을 제거하는 것만으로도 번들 크기가 눈에 띄게 줄어듭니다.

### 5. AI처럼 느껴지는 자동 완성(Autocomplete)

브라우저가 사용자 데이터를 기억하게 하는 `autocomplete` 속성은 사소해 보이지만 강력합니다. 사용자가 폼을 일일이 채울 필요가 없게 만들어 주며, 이는 백엔드 로직이 아닌 브라우저의 기억 능력을 활용한 자동화의 일종입니다.

### 6. 프레임워크 없는 동적 콘텐츠를 위한 `<template>`

React 없이 반복 가능한 UI 블록이 필요할 때 HTML5의 `<template>` 태그가 답이 될 수 있습니다. 이 태그 내부의 콘텐츠는 페이지 로드 시 렌더링되지 않으며, 자바스크립트의 `cloneNode`를 통해 필요할 때만 복제하여 사용할 수 있습니다. 프레임워크 오버헤드 없이 깨끗하고 재사용 가능한 구조를 유지할 수 있습니다.

### 7. 라이브러리 없는 드래그 앤 드롭(Drag and Drop)

작업 관리 패널을 만들 때 드래그 앤 드롭 라이브러리를 쓰지 않았습니다. 브라우저 네이티브 이벤트(`draggable="true"`, `ondragstart`, `ondrop` 등)만으로도 충분히 커스터마이징 가능한 로직을 구현할 수 있습니다. 무조건 외부 솔루션을 찾기 전에 브라우저가 이미 할 수 있는 일인지 먼저 물어야 합니다.

### 8. 백그라운드 작업을 위한 `<progress>`

비동기 작업이나 리포트 생성 등의 진행 상황을 알릴 때 `<progress>` 태그를 사용했습니다. 복잡한 CSS나 캔버스 애니메이션 없이도 사용자에게 즉각적인 시각적 신뢰를 줄 수 있습니다. 사용자는 당신의 기술 스택이 아니라 적절한 피드백에 관심을 가집니다.

### 9. 추가 작업이 필요 없는 네이티브 접근성(Accessibility)

시맨틱 HTML(Semantic HTML)을 올바르게 사용하면 스크린 리더가 정상적으로 작동하고, 키보드 내비게이션과 포커스 순서가 자동으로 맞춰집니다. ARIA 속성을 많이 작성하고 있다면, 오히려 기본 HTML 구조가 잘못된 것은 아닌지 의심해 봐야 합니다.

---

### 깨달음: HTML5는 자동화의 기반입니다

가장 놀라웠던 점은 HTML5가 원시적인 도구가 아니라, 이미 '자동화된' 도구라는 사실입니다. 위에서 언급한 기능들은 자바스크립트 코드와 버그를 줄이고, 의존성 업데이트와 정신적 오버헤드를 감소시켰습니다.

프런트엔드의 복잡성은 프레임워크 때문이 아니라, 브라우저의 내장 기능을 무시할 때 발생합니다. HTML과 싸우고, 네이티브 동작을 덮어쓰고, 이미 존재하는 기능을 다시 만들고 있다면 미래의 자신에게 넘겨줄 일감을 쌓고 있는 것과 같습니다.

웹 앱을 설계할 때 "브라우저가 이미 할 수 있는 것은 무엇인가?"를 먼저 자문해 보십시오. 자바스크립트는 그다음에 꼭 필요한 곳에만 추가하면 됩니다. 이 단순한 사고의 전환만으로도 개발 시간을 몇 주나 단축할 수 있습니다. HTML5는 기초적인 수준에 머물러 있는 도구가 아니라, 여전히 많은 개발자가 과소평가하고 있는 강력한 근간입니다.
