# 429: 당신의 API에 속도 제한이 없다면, 그것은 문제입니다.

늦은 밤, PagerDuty 알림을 받습니다. 프로덕션 API(Production API)의 속도가 느려지고 있으며, 거대한 트래픽 급증(Spike)이 관찰됩니다. 코드 버그 때문이 아니라, 누군가 초당 수천 개의 요청으로 엔드포인트(Endpoint)를 맹렬히 공격하는 공격적인 봇(Bot)을 실행했기 때문입니다. 데이터베이스 연결이 고갈되고 서버의 CPU 사용률이 100%에 달합니다. 그리고 정상적인 사용자들은 서비스에 접근할 수 없습니다.

이것이 바로 속도 제한(Rate Limiting)이 해결하는 문제입니다.

하지만 속도 제한은 양날의 검입니다. 너무 낮게 설정하면 정상적인 트래픽을 거부하게 되고, 너무 높게 설정하면 정작 필요할 때 보호 역할을 하지 못합니다.

이 블로그 게시물에서는 속도 제한의 동기, 핵심 알고리즘(토큰 버킷(Token Bucket), 리키 버킷(Leaky Bucket), 고정 윈도우(Fixed Window) 대 슬라이딩 윈도우(Sliding Window)), 그리고 Resilience4j, 커스텀 필터(Custom Filter), Redis를 사용한 분산 속도 제한(Distributed Rate Limiting)의 실제 Java 구현 방법을 다룹니다.

## 무제한 접근의 대가

최신 API(Application Programming Interface)는 웹 브라우저, 모바일 앱, 파트너 통합 시스템, 그리고 때로는 악성 봇(Bot) 등 다양한 클라이언트(Client)로부터 호출됩니다. 제한이 없다면, 단 하나의 오작동하는 클라이언트가 수천 명의 사용자를 위한 리소스(Resource)를 모두 소모할 수 있습니다.

### API 과부하의 해부

**정상적인 상황:**
- 500명의 사용자에게서 초당 1,000건의 요청
- 평균 응답 시간(Response Time): 50ms
- 데이터베이스 풀(Database Pool): 100개 중 20개의 연결 사용

**스크래퍼(Scraper) 공격 시작:**
- 단일 클라이언트가 초당 5,000건의 요청 생성 시작
- 총 부하(Total Load): 초당 6,000건의 요청
- 데이터베이스 풀: 100개 중 95개의 연결 사용
- 모든 사용자의 응답 시간: 800ms

**5분 후:**
- 데이터베이스 풀 고갈
- 모든 요청 타임아웃(Time Out) 발생
- 헬스 체크(Health Check)마저 실패
- 애플리케이션(Application)이 완전히 중단된 것처럼 보임

이는 이론적인 이야기가 아닙니다. 매주 누군가에게 실제로 일어나는 일입니다. 악의적인 의도 없이도 버그가 있는 재시도 루프(Retry Loop)나 개발자가 실수로 API를 무한 루프(Tight Loop)로 호출하는 경우에도 동일한 효과를 낼 수 있습니다.

## "그냥 확장(Scale Up)하면 되지"가 답이 아닌 이유

서버를 더 추가하여 문제를 해결할 수도 있습니다. 하지만:

*   **스케일링(Scaling)에는 시간이 걸립니다.** 자동 스케일링(Autoscaling) 정책이 새로운 인스턴스(Instance)를 프로비저닝(Provisioning)하는 데 몇 분이 걸리며, 그 사이에 이미 피해는 발생합니다.
*   **비용 증가** — 정상적인 트래픽이 아닌, 공격을 처리하기 위한 용량에 비용을 지불하게 됩니다.
*   **근본적인 문제를 해결하지 못합니다.** 무제한 접근은 본질적으로 안전하지 않습니다.

속도 제한은 1차 방어선입니다. 실행 중인 서버 수와 관계없이 단일 클라이언트가 야기할 수 있는 피해의 양을 제한합니다.

## 속도 제한 알고리즘: 핵심 메커니즘

코드를 살펴보기 전에, 기본적인 전략들을 이해해 봅시다. 각 전략은 버스트(Burst) 처리, 공정성, 구현 복잡성 측면에서 서로 다른 장단점을 가집니다.

### 1. 고정 윈도우 (Fixed Window)

가장 간단한 접근 방식: 고정된 시간 윈도우(예: 분당) 내의 요청 수를 계산합니다. 윈도우가 재설정되면 카운터는 0으로 돌아갑니다.

```
윈도우: 09:00:00 - 09:00:59
제한: 100개 요청

09:00:15 → 요청 #1  → 허용됨 (1/100)
09:00:30 → 요청 #50 → 허용됨 (50/100)
09:00:58 → 요청 #100 → 허용됨 (100/100)
09:00:59 → 요청 #101 → 거부됨
09:01:00 → 윈도우 재설정 → 카운터 0으로 복귀
```

**장점:** 구현하기 간단합니다. 예측 가능한 동작을 합니다.

**단점:** 경계 악용(Boundary Gaming) 가능성이 있습니다. 클라이언트는 09:00:59에 100개의 요청을 하고, 09:01:00에 다시 100개의 요청을 하여 2초 만에 200개의 요청을 할 수 있으며, 이는 속도 제한의 목적에 어긋납니다.

### 2. 슬라이딩 윈도우 (Sliding Window)

고정된 경계 대신 이동하는 시간 윈도우를 통해 요청을 추적합니다. 1분 윈도우의 경우, 현재 시점으로부터 지난 60초를 고려하며, 분의 시작 시점부터가 아닙니다.

```
현재 시간: 09:00:30
윈도우: 지난 60초 (08:59:30 - 09:00:30)

해당 윈도우 내의 요청 수: 95
새로운 요청 도착 → 총 요청 수 96
제한: 100
결과: 허용됨

현재 시간: 09:00:31
윈도우: 지난 60초 (08:59:31 - 09:00:31)
(일부 이전 요청은 이제 윈도우에서 만료됨)
```

**장점:** 더 부드러운 분배가 가능합니다. 경계 악용이 없습니다.

**단점:** 계산 비용이 더 많이 듭니다. 개별 요청 타임스탬프(Timestamp)를 추적하고 오래된 요청을 정리해야 합니다.

### 3. 토큰 버킷 (Token Bucket)

토큰을 담는 버킷(Bucket)을 상상해 보세요. 각 요청은 하나의 토큰을 소모합니다. 토큰은 고정된 속도(예: 초당 10개의 토큰)로 버킷에 추가됩니다. 버킷이 가득 차면 새로운 토큰은 버려집니다.

```
버킷 용량: 100개 토큰
재충전 속도: 초당 10개 토큰

09:00:00 → 버킷에 100개 토큰이 있음
09:00:01 → 클라이언트가 50개 요청 → 50개 토큰 소모 → 50개 남음
09:00:02 → 10개 토큰 추가 (재충전) → 60개 토큰 사용 가능
09:00:03 → 클라이언트가 70개 요청 → 10개 거부됨 (60개 토큰만 사용 가능)
```

**장점:** 버킷 용량까지 버스트(Burst)를 허용합니다. 사용자가 유휴(Idle) 상태였다면 토큰이 쌓여 합법적인 버스트 요청을 스로틀링(Throttling)되지 않고 수행할 수 있습니다.

**단점:** 더 복잡한 상태 관리(State Management)가 필요합니다. 클라이언트가 토큰을 "저장"할 수 있으며, 이는 일부 API에 바람직하지 않을 수 있습니다.

### 4. 리키 버킷 (Leaky Bucket)

요청이 큐(Queue) (버킷)에 들어가 고정된 속도("누출")로 처리됩니다. 버킷이 넘치면 요청이 삭제됩니다.

```
버킷 용량: 50개 큐에 대기 중인 요청
누출 속도: 초당 10개 요청 (일정한 처리 속도)

09:00:00 → 30개 요청 도착 → 큐: 30
09:00:01 → 10개 처리 (누출) → 큐: 20
09:00:01 → 40개 더 도착 → 큐: 60 → 10개 삭제됨 (오버플로우)
09:00:02 → 10개 처리 → 큐: 50
```

**장점:** 버스트 트래픽(Bursty Traffic)을 부드럽게 합니다. 출력 속도가 일정하여 다운스트림(Downstream) 서비스를 보호합니다.

**단점:** 지연 시간(Latency)을 추가합니다 (큐잉 지연(Queueing Delay)). 즉시 거부하지 않고 요청이 큐(Queue)에 남아 있어 사용자 경험(User Experience) 문제를 야기할 수 있습니다.

대부분의 공용 REST API(Representational State Transfer API)의 경우, 토큰 버킷이 가장 좋은 균형점을 제공합니다.

## 실습: Resilience4j를 이용한 속도 제한

Resilience4j는 프로덕션 환경(Production Environment)에 적합하고 어노테이션(Annotation) 기반의 속도 제한기(Rate Limiter)를 제공하며, Spring Boot와 완벽하게 통합됩니다.

### Maven 의존성 (Dependencies)

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
<dependency>
    <groupId>io.github.resilience4j</groupId>
    <artifactId>resilience4j-spring-boot3</artifactId>
    <version>2.2.0</version>
</dependency>
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-aop</artifactId>
</dependency>
```

### 애플리케이션 구성 (Application Configuration)

```yaml
# application.yml
resilience4j:
  ratelimiter:
    instances:
      # Public API - strict limits
      publicApi:
        limit-for-period: 10          # 10 requests
        limit-refresh-period: 1s      # per second
        timeout-duration: 0           # don't wait, reject immediately
        allow-health-indicator-to-fail: true
# Partner API - more generous
      partnerApi:
        limit-for-period: 100
        limit-refresh-period: 1s
        timeout-duration: 500ms       # wait up to 500ms for a permit
      # Internal service calls - high limits
      internalService:
        limit-for-period: 1000
        limit-refresh-period: 1s
        timeout-duration: 0
management:
  endpoints:
    web:
      exposure:
        include: health,metrics,prometheus
  metrics:
    export:
      prometheus:
        enabled: true
  health:
    ratelimiters:
      enabled: true
```

### 속도 제한이 적용된 서비스 계층 (Service Layer with Rate Limiting)

```java
package com.example.ratelimit;

import io.github.resilience4j.ratelimiter.annotation.RateLimiter;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.stereotype.Service;
import java.math.BigDecimal;
import java.util.List;
import java.util.stream.IntStream;

@Service
public class ProductService {
    private static final Logger log = LoggerFactory.getLogger(ProductService.class);

    @RateLimiter(name = "publicApi", fallbackMethod = "searchFallback")
    public ProductSearchResponse searchProducts(ProductSearchRequest request) {
        log.info("Searching for products: query={}, page={}", request.query(), request.page());
        // 데이터베이스 쿼리(Database Query) 시뮬레이션
        var products = IntStream.range(0, request.pageSize())
                .mapToObj(i -> new Product(
                        "prod-" + i,
                        request.query() + " Item " + i,
                        "Description for " + request.query(),
                        BigDecimal.valueOf(9.99 + i)
                ))
                .toList();
        return new ProductSearchResponse(products, 1000, request.page());
    }

    private ProductSearchResponse searchFallback(ProductSearchRequest request, Exception ex) {
        log.warn("Rate limit exceeded for search query: {}", request.query());
        // 빈 결과 또는 캐시(Cache)된 데이터 반환
        return new ProductSearchResponse(List.of(), 0, request.page());
    }
}

@Service
public class PartnerIntegrationService {
    private static final Logger log = LoggerFactory.getLogger(PartnerIntegrationService.class);

    @RateLimiter(name = "partnerApi", fallbackMethod = "syncFallback")
    public SyncResult syncInventory(String partnerId, List<InventoryUpdate> updates) {
        log.info("Syncing inventory for partner {}: {} updates", partnerId, updates.size());
        // 외부 API 호출 시뮬레이션
        return new SyncResult(partnerId, updates.size(), "SUCCESS");
    }

    private SyncResult syncFallback(String partnerId, List<InventoryUpdate> updates, Exception ex) {
        log.warn("Rate limit exceeded for partner {}", partnerId);
        // 재시도(Retry)를 위한 큐(Queue)에 추가
        return new SyncResult(partnerId, 0, "RATE_LIMITED");
    }
}
record SyncResult(String partnerId, int processedCount, String status) {}
record InventoryUpdate(String productId, int quantity) {}
```

### 속도 제한 오류 처리 기능이 있는 REST 컨트롤러 (REST Controller with Rate Limit Error Handling)

```java
package com.example.ratelimit;

import io.github.resilience4j.ratelimiter.RequestNotPermitted;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;
import java.time.Instant;

@RestController
@RequestMapping("/api/products")
public class ProductController {
    private final ProductService productService;

    public ProductController(ProductService productService) {
        this.productService = productService;
    }

    @GetMapping("/search")
    public ResponseEntity<ProductSearchResponse> search(
            @RequestParam String query,
            @RequestParam(defaultValue = "1") int page,
            @RequestParam(defaultValue = "20") int pageSize
    ) {
        var request = new ProductSearchRequest(query, page, pageSize);
        var response = productService.searchProducts(request);
        return ResponseEntity.ok(response);
    }

    @ExceptionHandler(RequestNotPermitted.class)
    public ResponseEntity<ApiError> handleRateLimitExceeded(RequestNotPermitted ex) {
        var error = new ApiError(
                "RATE_LIMIT_EXCEEDED",
                "Too many requests. Please try again later.",
                Instant.now()
        );
        return ResponseEntity.status(HttpStatus.TOO_MANY_REQUESTS)
                .header("X-RateLimit-Retry-After", "1")  // 1초 후 재시도
                .body(error);
    }
}
```

**주요 내용:**

*   `@RateLimiter(name = "publicApi")` 어노테이션(Annotation)은 `YAML` 파일에 구성된 속도 제한기를 적용합니다.
*   제한을 초과하면 `RequestNotPermitted` 예외(Exception)가 발생합니다.
*   `429 Too Many Requests` 상태 코드와 `Retry-After` 헤더(Header)를 반환합니다.
*   폴백(Fallback) 메서드(Method)는 우아한 성능 저하(Graceful Degradation)를 제공합니다.

## 커스텀 속도 제한: 서블릿 필터(Servlet Filter) 방식

사용자 ID(User ID), API 키(API Key) 또는 IP 주소(IP Address) 기반 속도 제한과 같이 더 많은 제어가 필요한 사용 사례의 경우, 커스텀 서블릿 필터(Custom Servlet Filter)를 구현할 수 있습니다.

### Guava를 이용한 인메모리(In-Memory) 속도 제한기

```java
package com.example.ratelimit;

import com.google.common.cache.CacheBuilder;
import com.google.common.cache.CacheLoader;
import com.google.common.cache.LoadingCache;
import com.google.common.util.concurrent.RateLimiter;
import jakarta.servlet.*;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import org.springframework.stereotype.Component;
import java.io.IOException;
import java.util.concurrent.TimeUnit;

@Component
public class IpBasedRateLimitFilter implements Filter {
    // IP 주소별 속도 제한기 캐시(Cache)
    private final LoadingCache<String, RateLimiter> limiters = CacheBuilder.newBuilder()
            .maximumSize(10_000)
            .expireAfterAccess(10, TimeUnit.MINUTES)
            .build(new CacheLoader<>() {
                @Override
                public RateLimiter load(String key) {
                    // IP당 초당 10개 요청
                    return RateLimiter.create(10.0);
                }
            });

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {
        var httpRequest = (HttpServletRequest) request;
        var httpResponse = (HttpServletResponse) response;

        String clientIp = getClientIp(httpRequest);
        RateLimiter limiter = limiters.getUnchecked(clientIp);

        // 기다리지 않고 허용량(Permit)을 획득 시도
        if (!limiter.tryAcquire()) {
            httpResponse.setStatus(429);
            httpResponse.setHeader("X-RateLimit-Retry-After", "1");
            httpResponse.getWriter().write("{\"error\":\"Too many requests from " + clientIp + "\"}");
            return;
        }

        chain.doFilter(request, response);
    }

    private String getClientIp(HttpServletRequest request) {
        String ip = request.getHeader("X-Forwarded-For");
        if (ip == null || ip.isEmpty()) {
            ip = request.getRemoteAddr();
        }
        return ip;
    }
}
```

**장점:** 키 추출(IP, 사용자 ID, API 키), 속도 제한 로직(Logic), 응답 형식(Response Formatting)에 대한 완전한 제어가 가능합니다.

**단점:** 인메모리(In-Memory) 상태 — 분산 조정(Distributed Coordination) 없이는 다중 인스턴스(Multi-Instance) 배포 환경에서 작동하지 않습니다.

## Redis를 이용한 분산 속도 제한 (Distributed Rate Limiting)

여러 서비스 인스턴스(Service Instance)를 사용하는 프로덕션 배포(Production Deployment)의 경우, 공유 상태(Shared State)가 필요합니다. Redis는 가장 적합한(Go-to) 솔루션입니다.

### Redis와 함께 Bucket4j 사용하기

Bucket4j는 토큰 버킷(Token Bucket) 알고리즘을 구현하고 Redis를 백엔드(Backend)로 지원하는 Java 속도 제한 라이브러리(Library)입니다.

### 구성 (Configuration)

```java
package com.example.ratelimit;

import com.bucket4j.distributed.proxy.ProxyManager;
import com.bucket4j.redis.lettuce.cas.LettuceBasedProxyManager;
import io.lettuce.core.RedisClient;
import io.lettuce.core.api.StatefulRedisConnection;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class RateLimitConfiguration {
    @Bean
    public RedisClient redisClient() {
        return RedisClient.create("redis://localhost:6379");
    }

    @Bean
    public StatefulRedisConnection<String, byte[]> redisConnection(RedisClient redisClient) {
        return redisClient.connect(io.lettuce.core.codec.RedisCodec.of(
                io.lettuce.core.codec.StringCodec.UTF8,
                io.lettuce.core.codec.ByteArrayCodec.INSTANCE
        ));
    }

    @Bean
    public ProxyManager<String> proxyManager(StatefulRedisConnection<String, byte[]> connection) {
        return LettuceBasedProxyManager.builderFor(connection)
                .build();
    }
}
```

### 분산 속도 제한 필터 (Distributed Rate Limit Filter)

```java
package com.example.ratelimit;

import com.bucket4j.*;
import com.bucket4j.distributed.proxy.ProxyManager;
import jakarta.servlet.*;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import org.springframework.stereotype.Component;
import java.io.IOException;
import java.time.Duration;
import java.util.function.Supplier;

@Component
public class DistributedRateLimitFilter implements Filter {
    private final ProxyManager<String> proxyManager;

    public DistributedRateLimitFilter(ProxyManager<String> proxyManager) {
        this.proxyManager = proxyManager;
    }

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {
        var httpRequest = (HttpServletRequest) request;
        var httpResponse = (HttpServletResponse) response;

        String apiKey = httpRequest.getHeader("X-API-Key");
        if (apiKey == null || apiKey.isEmpty()) {
            httpResponse.setStatus(401);
            httpResponse.getWriter().write("{\"error\":\"Missing API key\"}");
            return;
        }

        // 이 API 키(API Key)에 대한 버킷(Bucket)을 가져오거나 생성합니다.
        Bucket bucket = proxyManager.builder()
                .build(apiKey, bucketConfiguration());

        ConsumptionProbe probe = bucket.tryConsumeAndReturnRemaining(1);
        if (probe.isConsumed()) {
            // 요청 허용됨
            httpResponse.setHeader("X-RateLimit-Remaining", String.valueOf(probe.getRemainingTokens()));
            chain.doFilter(request, response);
        } else {
            // 속도 제한 초과
            long waitForRefill = probe.getNanosToWaitForRefill() / 1_000_000_000;
            httpResponse.setStatus(429);
            httpResponse.setHeader("X-RateLimit-Retry-After", String.valueOf(waitForRefill));
            httpResponse.setHeader("X-RateLimit-Remaining", "0");
            httpResponse.getWriter().write("{\"error\":\"Rate limit exceeded\"}");
        }
    }

    private Supplier<BucketConfiguration> bucketConfiguration() {
        return () -> BucketConfiguration.builder()
                .addLimit(Bandwidth.builder()
                        .capacity(100)                        // 버킷은 100개의 토큰을 보유
                        .refillIntervally(100, Duration.ofMinutes(1))  // 1분마다 100개의 토큰 재충전
                        .build())
                .build();
    }
}
```

**진행 중인 작업:**

*   각 API 키는 Redis에 저장된 자체 버킷을 가집니다.
*   버킷의 용량은 100개의 토큰입니다.
*   토큰은 1분당 100개(초당 약 1.67개)의 속도로 재충전됩니다.
*   요청이 허용되면, 남은 토큰 수를 헤더(Header)에 반환합니다.
*   요청이 거부되면, 클라이언트(Client)에게 재시도(Retry)하기까지 기다려야 하는 시간을 알려줍니다.

## 모니터링: 트래픽을 거부할 때를 알아야 합니다.

모니터링(Monitoring) 없는 속도 제한은 눈 가리고 비행하는 것과 같습니다. 다음 항목에 대한 가시성(Visibility)이 필요합니다.

*   **거부율(Rejection Rate)** — 얼마나 자주 제한에 도달하는가?
*   **가장 많이 위반하는 클라이언트(Top Violators)** — 어떤 클라이언트가 가장 많이 스로틀링(Throttling)되는가?
*   **남은 용량(Remaining Capacity)** — 제한에 얼마나 근접했는가?

### Micrometer를 이용한 메트릭(Metrics)

### 분산 속도 제한을 위한 커스텀 메트릭(Custom Metrics)

```java
package com.example.ratelimit;
import io.micrometer.core.instrument.Counter;
import io.micrometer.core.instrument.MeterRegistry;
import org.springframework.stereotype.Component;

@Component
public class RateLimitMetrics {
    private final Counter allowedRequests;
    private final Counter rejectedRequests;

    public RateLimitMetrics(MeterRegistry registry) {
        this.allowedRequests = Counter.builder("rate_limit.requests")
                .tag("result", "allowed")
                .register(registry);
        this.rejectedRequests = Counter.builder("rate_limit.requests")
                .tag("result", "rejected")
                .register(registry);
    }

    public void recordAllowed() {
        allowedRequests.increment();
    }

    public void recordRejected() {
        rejectedRequests.increment();
    }
}
```

## 다차원 속도 제한 (Rate Limiting by Multiple Dimensions)

실제 API는 종종 다차원(Multi-Dimensional) 제한이 필요합니다.

*   API 키당 초당 10회 요청
*   사용자당 일일 1,000회 요청
*   전역적으로 시간당 100,000회 요청

다음은 여러 제한기를 조합하는 패턴입니다.

```java
package com.example.ratelimit;
import com.bucket4j.Bucket;
import com.bucket4j.BucketConfiguration;
import com.bucket4j.distributed.proxy.ProxyManager;
import org.springframework.stereotype.Component;
import java.time.Duration;
import java.util.function.Supplier;

@Component
public class MultiDimensionalRateLimiter {
    private final ProxyManager<String> proxyManager;

    public MultiDimensionalRateLimiter(ProxyManager<String> proxyManager) {
        this.proxyManager = proxyManager;
    }

    public boolean allowRequest(String apiKey, String userId) {
        // API 키당 제한 확인 (초당 10회)
        Bucket apiKeyBucket = getBucket("api_key:" + apiKey, perSecondConfig(10));
        if (!apiKeyBucket.tryConsume(1)) {
            return false;
        }

        // 사용자당 일일 제한 확인 (일일 1000회)
        Bucket userDailyBucket = getBucket("user_daily:" + userId, perDayConfig(1000));
        if (!userDailyBucket.tryConsume(1)) {
            return false;
        }

        // 전역 시간당 제한 확인 (시간당 10만 회)
        Bucket globalBucket = getBucket("global_hourly", perHourConfig(100_000));
        return globalBucket.tryConsume(1);
    }

    private Bucket getBucket(String key, Supplier<BucketConfiguration> config) {
        return proxyManager.builder().build(key, config);
    }

    private Supplier<BucketConfiguration> perSecondConfig(long limit) {
        return () -> BucketConfiguration.builder()
                .addLimit(Bandwidth.simple(limit, Duration.ofSeconds(1)))
                .build();
    }

    private Supplier<BucketConfiguration> perDayConfig(long limit) {
        return () -> BucketConfiguration.builder()
                .addLimit(Bandwidth.simple(limit, Duration.ofDays(1)))
                .build();
    }

    private Supplier<BucketConfiguration> perHourConfig(long limit) {
        return () -> BucketConfiguration.builder()
                .addLimit(Bandwidth.simple(limit, Duration.ofHours(1)))
                .build();
    }
}
```

## 속도 제한이 해결하는 문제보다 더 많은 문제를 야기할 때

속도 제한에는 비용이 듭니다. 지연 시간(Redis 확인, 토큰 계산), 운영 복잡성(제한 설정, 거부 모니터링), 사용자 불만(정상적인 사용자가 스로틀링될 때)을 야기합니다.

**속도 제한을 건너뛰어야 할 때:**

*   소규모 내부 API로, 소수의 알려진 클라이언트(Client)가 사용하는 경우
*   인증(Authentication) + 할당량(Quota)을 사용하는 경우 — 모든 클라이언트가 인증된 경우, 요청당이 아닌 구독/결제 수준(Subscription/Billing Level)에서 제한을 적용할 수 있습니다.
*   시스템이 효과적으로 자동 스케일링(Auto-scaling)되는 경우 — 공격자가 트래픽을 늘리는 것보다 더 빠르게 용량을 프로비저닝(Provisioning)할 수 있다면, 속도 제한은 거의 가치가 없습니다.
*   팀이 합리적인 제한을 정의할 수 없는 경우 — 데이터에 근거하지 않은 "분당 100회 요청"과 같은 임의의 숫자를 설정하면 종종 더 큰 해를 끼칠 수 있습니다.

**속도 제한을 사용해야 할 때:**

*   알려지지 않은 클라이언트에 노출되는 공용 API를 가지고 있는 경우
*   다운스트림(Downstream) 서비스가 비싼 경우 (예: 데이터베이스 쓰기, 외부 API 호출, 결제 처리)
*   이전에 공격을 받은 경험이 있는 경우
*   성숙한 관측 가능성(Observability)을 갖추고 있는 경우 (거부율을 측정하고 실제 트래픽 패턴에 따라 제한을 조정할 수 있는 경우)
*   공정한 사용이 중요한 경우 (예: SaaS API, 프리미엄(Freemium) 계층, 파트너 통합)
