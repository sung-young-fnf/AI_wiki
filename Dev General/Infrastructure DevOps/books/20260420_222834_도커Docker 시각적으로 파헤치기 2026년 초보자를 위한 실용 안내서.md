# 도커(Docker) 시각적으로 파헤치기: 2026년 초보자를 위한 실용 안내서

아마도 이런 답답함을 들어본 적이 있거나 직접 경험했을 것입니다.

당신의 코드는 노트북에서는 완벽하게 작동합니다. 하지만 동료에게 보내거나, 서버에 배포하거나, 프로덕션 환경에 올리면 갑자기 문제가 발생합니다. "하지만 제 컴퓨터에서는 잘 작동해요"는 해결책이 아닙니다. 이것이야말로 도커(Docker)가 영구적으로 제거하고자 만들어진 문제입니다.

이 글은 도커를 완벽하게 이해하게 해 줄 단 하나의 글입니다. 명령어를 암기하게 하는 대신, 내부에서 어떤 시각적 논리가 작동하는지 보여줄 것입니다.

흐름을 이해하고 나면, 명령어들이 더 이상 임의적으로 느껴지지 않고 명확하게 느껴지기 시작할 것입니다.

개발자, 학생, 커리어 전환을 준비하는 사람, 또는 오랫동안 컨테이너(container)를 피해왔던 엔지니어 누구에게나 이 글은 도움이 될 것입니다.

## 도커의 문제 해결 방식 (쉬운 설명)

소프트웨어는 특정 환경 내에서 실행됩니다. 이 환경에는 운영 체제 버전, 설치된 라이브러리, 설정, 그리고 수많은 보이지 않는 의존성(dependencies)이 포함됩니다.

당신이 자신의 장비에서 코드를 작성할 때는 당신의 환경이 정확히 맞아떨어집니다. 하지만 다른 사람이 그 코드를 실행할 때는 그들의 환경이 다릅니다.

이것은 작은 문제가 아닙니다. 셀 수 없이 많은 프로덕션 장애, 배포 지연, 그리고 고통스러운 디버깅(debugging) 세션을 야기했습니다.

도커는 당신의 코드와 그 전체 환경을 단일하고 이식 가능한(portable) 단위로 패키징하여 이 문제를 해결합니다. 이 단위는 당신의 노트북, 동료의 장비, 싱가포르의 클라우드 서버 등 어디에서나 동일하게 실행됩니다.

핵심 개념: 도커를 배송 컨테이너처럼 생각해보세요. 표준화된 배송 컨테이너가 있기 전에는 선박, 트럭, 항구 간 화물 운송이 물류 악몽이었습니다. 서로 다른 크기, 다른 취급 요구 사항, 호환되지 않는 시스템 등이 문제였죠. 표준화된 컨테이너가 이 모든 것을 없애주었습니다. 화물을 컨테이너에 넣으면, 컨테이너는 어디든 갈 수 있습니다. 도커는 소프트웨어에 대해 정확히 이 역할을 합니다.

## 네 가지 핵심 개념 (각 개념은 다음 개념의 기초가 됩니다)

무엇보다 먼저 네 단어를 이해해야 합니다. 도커의 모든 것은 이 네 단어가 함께 작동하는 방식일 뿐입니다.

### 1. 도커파일(Dockerfile): 레시피

도커파일은 애플리케이션의 환경을 구축하기 위한 지침이 담긴 일반 텍스트 파일입니다. 레시피라고 생각해보세요. "이 기본 운영 체제에서 시작하여 이 도구들을 설치하고, 이 코드를 복사하고, 시작 시 이 명령어를 실행하라"고 지시합니다.

```dockerfile
# 공식 Python 베이스 이미지로 시작
FROM python:3.12-slim

# 컨테이너 내부의 작업 디렉토리 설정
WORKDIR /app

# requirements 파일을 복사하고 의존성 설치
COPY requirements.txt .
RUN pip install -r requirements.txt

# 나머지 애플리케이션 코드 복사
COPY . .

# 컨테이너 시작 시 실행할 명령어를 도커에 알림
CMD ["python", "app.py"]
```

도커파일은 절대 당신의 앱을 실행하지 않습니다. 단지 앱을 위한 환경을 어떻게 구축할지 설명할 뿐입니다. 당신은 한 번 작성하고, 도커는 매번 그것을 사용합니다.

### 2. 이미지(Image): 청사진

도커가 당신의 도커파일을 읽고 모든 지침을 실행하면 이미지가 생성됩니다.

이미지는 운영 체제, 설치된 도구, 당신의 코드 등 전체 환경의 읽기 전용 스냅샷(read-only snapshot)입니다.

이미지는 레이어(layer)로 구축됩니다. 도커파일의 각 지침은 하나의 레이어를 생성합니다. 이는 빌드 속도에 직접적인 영향을 미치기 때문에 이해하는 것이 매우 중요합니다.

레이어는 다음과 같이 작동합니다. 만약 당신이 애플리케이션 코드(마지막 `COPY . .` 줄)만 변경한다면, 도커는 이전 레이어들을 캐시(cache)에서 재사용하고 변경된 지점부터만 다시 빌드합니다. 만약 첫 번째 줄(`FROM python:3.12-slim`)을 변경하면, 그 아래 모든 것이 다시 빌드되어야 합니다.

이것이 숙련된 엔지니어들이 거의 변경되지 않는 것들(의존성 설치)을 도커파일 초반에, 자주 변경되는 것들(소스 코드 복사)을 후반에 배치하는 이유입니다.

### 3. 컨테이너(Container): 실행 인스턴스

컨테이너는 이미지의 실행 중인 인스턴스(instance)입니다. 이미지와 컨테이너의 관계는 프로그래밍에서 클래스(class)와 객체(object)의 관계, 또는 레시피와 실제 요리의 관계와 같습니다.

하나의 이미지는 여러 컨테이너를 생성할 수 있습니다. 각 컨테이너는 격리되어 있습니다. 즉, 자체 파일 시스템, 자체 네트워크 인터페이스, 자체 프로세스 공간을 가집니다. 컨테이너 내부의 변경 사항은 그것이 생성된 이미지에 영향을 미치지 않으며, 다른 컨테이너에도 영향을 주지 않습니다.

격리에 대한 핵심 통찰은 다음과 같습니다. 컨테이너는 호스트 머신(host machine)의 운영 체제 커널(kernel)을 공유하지만, 기본적으로 서로를 보거나 간섭할 수 없습니다. 이는 인스턴스당 전체 OS 복사본이 필요한 가상 머신(Virtual Machine, VM)과는 근본적으로 다릅니다.

### 4. 레지스트리(Registry): 배포 계층

레지스트리는 도커 이미지를 위한 저장 및 배포 시스템입니다. 도커 허브(Docker Hub)는 기본 공개 레지스트리입니다. 당신의 팀이나 회사는 프라이빗 레지스트리를 운영할 수 있습니다.

도커파일에서 `python:3.12-slim`과 같은 기본 이미지를 사용할 때, 도커는 이를 도커 허브에서 풀(pull)합니다. 당신이 자신만의 이미지를 빌드하고 이를 서버에 배포하고 싶을 때, 레지스트리에 푸시(push)합니다. 서버는 거기서부터 이미지를 풀합니다.

이것이 도커가 일관된 배포를 가능하게 하는 방법입니다. 로컬에서 테스트한 것과 정확히 동일한 이미지 비트(bits)가 프로덕션에 배포됩니다.

## 도커가 실제로 이미지를 빌드하는 방법 (단계별)

대부분의 튜토리얼에서 건너뛰는 시각적인 흐름입니다. 이 순서를 이해하면 다른 모든 것이 이해될 것입니다.

`docker build`를 실행하면 어떤 일이 일어나는가:

1.  도커는 도커파일을 위에서 아래로 읽습니다.
2.  각 지침은 이전 레이어 위에 새로운 레이어를 생성합니다.
3.  도커는 각 레이어에 대해 먼저 빌드 캐시를 확인합니다.
    *   해당 지침에 변경 사항이 없으면: 캐시됨 (즉시)
    *   변경 사항이 있으면: 해당 지점부터 다시 빌드
4.  최종 결과는 고유 ID를 가진 새 이미지입니다.

레이어 캐시의 작동 방식:

*   **단계 1:** `FROM python:3.12-slim` ← 한 번 풀링되면 영원히 캐시됨
*   **단계 2:** `WORKDIR /app` ← 캐시됨
*   **단계 3:** `COPY requirements.txt .` ← 파일이 변경되지 않았다면 캐시됨
*   **단계 4:** `RUN pip install -r req...` ← `requirements.txt`가 변경되지 않았다면 캐시됨
*   **단계 5:** `COPY . .` ← 코드가 변경될 때마다 다시 빌드됨
*   **단계 6:** `CMD ["python", "app.py"]` ← 다시 빌드됨

만약 `app.py`에서 한 줄만 변경했다면, 1단계부터 4단계까지는 시간이 전혀 걸리지 않습니다. 오직 5단계와 6단계만 다시 빌드됩니다. 이것이 레이어 순서가 개발 속도에 엄청나게 중요한 이유입니다.

## 도커 네트워킹(Networking): 컨테이너들이 서로 통신하는 방법

초보자들이 가장 혼란스러워하는 것 중 하나는 컨테이너들이 어떻게 통신하는지입니다. 다음은 개념 모델입니다.

기본적으로 각 컨테이너는 격리되어 있습니다. 명시적으로 연결하지 않는 한 다른 컨테이너나 서비스에 접근할 수 없습니다.

도커는 당신이 실제로 알아야 할 세 가지 네트워킹 모드(networking mode)를 제공합니다.

*   **브리지 네트워크(Bridge network, 기본값):** 도커는 사설(private) 내부 네트워크를 생성합니다. 동일한 브리지 네트워크 상의 컨테이너는 컨테이너 이름으로 서로에게 접근할 수 있습니다. 외부 세계로의 트래픽은 도커의 네트워크 계층을 통해 전달됩니다.
*   **호스트 네트워크(Host network):** 컨테이너가 호스트 머신의 네트워크를 직접 공유합니다. 포트 매핑(port mapping)이 필요 없습니다. 고성능 시나리오에 사용됩니다.
*   **커스텀 브리지 네트워크(Custom bridge network, 올바른 방식):** 당신이 자신만의 네트워크를 생성할 때, 도커는 컨테이너 간에 자동 DNS(Domain Name System) 해석을 추가합니다. 컨테이너 A는 서비스 이름을 호스트 이름(hostname)으로 사용하여 컨테이너 B에 간단히 접근할 수 있습니다. 이것이 도커 컴포즈(Docker Compose)가 사용하는 방식입니다.

**포트 매핑(Port mapping) 설명:** 컨테이너는 자체 내부 포트를 가집니다. 컨테이너 내부에서 8000번 포트에서 수신 대기하는 웹 서버는 당신이 이를 매핑(mapping)하지 않는 한 노트북에서 접근할 수 없습니다. `-p 3000:8000` 플래그는 "당신 장비의 3000번 포트에서 오는 요청을 컨테이너 내부의 8000번 포트로 전달하라"는 의미입니다.

`당신 노트북:3000` → `[도커 네트워크 계층]` → `컨테이너:8000`

## 볼륨(Volumes): 컨테이너 데이터가 사라지지 않는 이유

이것은 미묘하지만 중요한 개념입니다.

컨테이너는 설계상 일시적(ephemeral)입니다. 컨테이너를 삭제하면 그 안에 있는 모든 것이 사라집니다. 이는 의도적인 것으로, 컨테이너를 깔끔하고 재현 가능하게 유지합니다. 하지만 데이터베이스, 사용자 업로드 파일, 애플리케이션 로그의 경우 지속되어야 할 데이터가 필요합니다.

볼륨이 그 해답입니다. 볼륨은 호스트 머신에 존재하지만, 컨테이너 내부에 지정된 경로로 마운트(mount)되는 디렉터리입니다. 컨테이너가 해당 경로에 쓰면, 실제로는 호스트 머신에 쓰는 것입니다. 컨테이너가 삭제되어도 볼륨은 살아남습니다.

```yaml
# docker-compose.yaml 파일 내
volumes:
  - postgres_data:/var/lib/postgresql/data
```

이것은 도커에게 "컨테이너가 `/var/lib/postgresql/data`에 쓸 때마다, 그 데이터를 호스트의 `postgres_data`라는 볼륨에 저장하라"고 지시합니다. 데이터베이스는 컨테이너 재시작, 재구축, 업데이트 후에도 유지됩니다.

개념 모델: 볼륨은 호스트 머신에 항상 연결되어 있는 USB 드라이브와 같습니다. 어떤 컨테이너에도 마운트하고, 꺼내고, 새로운 컨테이너에 다시 마운트할 수 있으며, 모든 데이터가 그대로 유지됩니다.

## 도커 컴포즈(Docker Compose): 여러 컨테이너를 시스템으로 실행하기

실제 애플리케이션은 단일 컨테이너가 아닙니다. 웹 서버, 데이터베이스, 캐시, 백그라운드 워커와 같은 시스템입니다. 이들을 수동으로 조정하는 것은 복잡합니다. 도커 컴포즈가 그 해결책입니다.

도커 컴포즈는 선언적 모델(declarative model)로 작동합니다. 단일 YAML 파일에 전체 애플리케이션 스택을 정의하면, 컴포즈가 모든 컨테이너의 생성, 네트워킹 및 생명 주기 관리를 처리합니다.

다음은 데이터베이스와 캐시를 사용하는 웹 애플리케이션을 위한 실제 컴포즈(Compose) 파일입니다.

```yaml
# compose.yaml (2026년 현재 표준 파일명)
services:
  web:
    build: .                    # 로컬 Dockerfile에서 빌드
    ports:
      - "8000:8000"             # 머신의 8000번 포트를 외부에 노출
    environment:
      DATABASE_URL: postgres://user:pass@db:5432/myapp
    depends_on:
      db:
        condition: service_healthy  # db가 단순히 시작된 것이 아니라 준비될 때까지 기다림
    networks:
      - app-network
  db:
      image: postgres:16-alpine   # 공식 이미지 사용, Dockerfile 필요 없음
      environment:
        POSTGRES_USER: user
        POSTGRES_PASSWORD: pass
        POSTGRES_DB: myapp
      volumes:
        - postgres_data:/var/lib/postgresql/data  # 데이터 지속
      healthcheck:
        test: ["CMD-SHELL", "pg_isready -U user"]
        interval: 5s
        retries: 5
      networks:
        - app-network
  cache:
      image: redis:7-alpine
      networks:
        - app-network
volumes:
  postgres_data:               # 도커가 관리하는 이름 지정 볼륨
networks:
  app-network:                 # 이 스택을 위한 격리된 사설 네트워크
      driver: bridge
```

여기서 각 줄별로 어떤 일이 일어나는지 설명합니다.

*   `services`는 시스템의 각 컨테이너를 정의합니다. 각 서비스는 네트워크상에서 호스트 이름이 되는 이름을 가집니다.
*   `depends_on`과 `condition: service_healthy`는 데이터베이스가 단순히 시작된 것이 아니라 연결을 받아들일 준비가 될 때까지 기다립니다. 이는 웹 앱이 데이터베이스가 준비되기 전에 시작되는 경쟁 조건(race condition)을 방지합니다.
*   `db` 서비스에는 `pg_isready`를 5초마다 실행하여 연결을 수락하는지 확인하는 `healthcheck`가 있습니다. `service_healthy`가 기다리는 것이 바로 이것입니다.
*   세 서비스 모두 `app-network`를 공유합니다. 이는 `web`이 `db`를 호스트 이름으로 사용하여 데이터베이스에 연결할 수 있음을 의미합니다. 도커가 DNS 조회를 자동으로 처리합니다.
*   하단의 `postgres_data` 볼륨은 도커가 그 생명 주기를 관리하도록 명시적으로 선언됩니다.

실행은 한 가지 명령으로 가능합니다.

```bash
docker compose up -d    # -d는 detached 모드로 백그라운드에서 실행
docker compose logs -f  # 모든 서비스의 로그를 실시간으로 확인
docker compose down     # 컨테이너 중지 및 제거 (볼륨은 유지)
docker compose down -v  # 볼륨도 함께 삭제 (주의: 데이터를 삭제함)
```

## 2025년과 2026년에 변경된 사항

두세 년 전에 도커를 배웠다면, 몇 가지 알아야 할 변경 사항이 있습니다.

*   **컴포즈 V2가 이제 유일한 버전입니다.** 도커 컴포즈 V1(하이픈이 있는 파이썬 기반 `docker-compose`)은 2023년 7월에 지원이 종료되었습니다. 도커 컴포즈 V1은 독립형 파이썬 바이너리(binary)였습니다. V2는 도커 CLI(Command Line Interface)에 직접 통합된 고(Go) 플러그인이며, `docker compose` (공백 포함)으로 호출됩니다. V2는 훨씬 빠르고, 컴포즈 스펙(Compose Specification) 표준을 지원하며, 서비스 프로파일 및 GPU(Graphics Processing Unit) 접근과 같은 기능을 추가했습니다. 만약 하이픈이 있는 `docker-compose`를 사용하는 튜토리얼을 본다면, 그것은 구식입니다.
*   **컴포즈 스펙 v5가 현재 버전입니다.** 2025년 12월 "몽블랑(Mont Blanc)"이라는 코드명으로 출시된 컴포즈 스펙 v5.0.0은 여러 아키텍처 변경 사항을 도입했습니다. 내부 빌더는 `Docker Bake`로 위임하는 방식으로 제거되었으며, 이는 빌드 프로세스를 표준 `docker build` 동작과 일치시킵니다. 이는 더 빠른 빌드, 더 나은 캐싱, 그리고 고급 빌드킷(BuildKit) 기능 지원을 의미합니다.
*   **빌드킷이 이제 기본 빌더입니다.** 빌드킷은 도커 데스크톱(Docker Desktop) 및 도커 엔진(Docker Engine) v23.0 이상 사용자에게 기본 빌더입니다. 빌드킷이 레거시 빌더보다 개선된 주요 영역은 성능, 스토리지 관리 및 확장성입니다. 따로 활성화할 필요가 없습니다. 이미 활성화되어 있습니다. 이것이 실질적으로 의미하는 바는: 빌드가 더 빠르고, 가능한 경우 병렬로 진행되며, 캐싱에 대해 더 스마트하다는 것입니다.
*   **`docker-compose.yml`보다 `compose.yaml` 파일명이 선호됩니다.** 둘 다 작동하지만, `compose.yaml`이 현재 표준이며 현대적인 컴포즈 스펙을 사용하고 있음을 나타냅니다.
*   **도커 데스크톱에 공식 Go SDK가 포함된 컴포즈 v5가 포함됩니다.** 도커 데스크톱에는 이제 새로운 공식 Go SDK를 도입한 도커 컴포즈 v5가 포함됩니다. 이 SDK는 포괄적인 API(Application Programming Interface)를 제공하여 컴포즈 CLI에 의존하지 않고도 컴포즈 기능을 애플리케이션에 직접 통합할 수 있게 함으로써 멀티 컨테이너 환경을 로드, 검증 및 관리할 수 있습니다.

## 멀티스테이지 빌드(Multi-Stage Builds): 이미지 크기를 작고 안전하게 유지하기

이것은 프로덕션 사용에 있어 가장 영향력 있는 개념 중 하나이며, 초보자 튜토리얼에서는 가장 많이 건너뛰는 부분 중 하나입니다.

문제: Node.js 또는 파이썬 애플리케이션을 빌드할 때, 빌드 프로세스 중(컴파일러, 테스트 프레임워크, 개발 의존성) 필요한 빌드 도구들이 최종 프로덕션 이미지에는 포함되기를 원치 않습니다. 이는 이미지 크기를 늘리고 공격 표면(attack surface)을 확대합니다.

멀티스테이지 빌드는 빌드용 이미지와 실행용으로 다른 최소한의 이미지를 사용할 수 있게 해줍니다.

```dockerfile
# 스테이지 1: 빌드 스테이지
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json .
RUN npm ci                    # 개발 의존성을 포함한 모든 의존성 설치
COPY . .
RUN npm run build             # TypeScript 컴파일, 자산 번들링 등

# 스테이지 2: 프로덕션 스테이지
# 이 스테이지의 내용만 최종 이미지에 포함됨
FROM node:20-alpine AS production
WORKDIR /app
COPY package*.json .
RUN npm ci --only=production  # 프로덕션 의존성만 설치
COPY --from=builder /app/dist ./dist  # 컴파일된 결과물만 복사
CMD ["node", "dist/index.js"]
```

최종 이미지는 프로덕션 런타임(runtime)과 컴파일된 결과물만 포함합니다. TypeScript 컴파일러도, 테스트 라이브러리도, 이미 컴파일된 소스 파일도 없습니다. 예전에는 800MB였던 이미지가 120MB가 됩니다. 더 작은 이미지는 더 빠르게 풀링되고, 더 빠르게 배포되며, 패치할 취약점이 더 적습니다.

## 흔히 발생하는 실수와 그 의미

*   **"Port already in use" (포트가 이미 사용 중입니다):** 당신의 장비에서 다른 프로세스가 이미 해당 포트를 사용하고 있습니다. 다른 프로세스를 중지시키거나 `-p` 매핑에서 호스트 측 포트를 변경하세요.
*   **"Cannot connect to the Docker daemon" (도커 데몬에 연결할 수 없습니다):** 도커가 실행되고 있지 않습니다. 도커 데스크톱 또는 도커 서비스를 시작하세요.
*   **컨테이너가 즉시 종료됩니다:** 컨테이너의 메인 프로세스가 완료되었거나 충돌했습니다. `docker compose logs [서비스명]`으로 로그를 확인하세요.
*   **컨테이너 내부에 변경 사항이 반영되지 않습니다:** 당신의 코드 변경 사항이 다시 빌드되지 않고 있습니다. `docker compose build`를 사용한 다음 `docker compose up`을 사용하거나, 개발 중 라이브 리로드(live reload)를 위해 `docker compose watch`를 사용하세요.
*   **시작 시 데이터베이스 연결이 거부됩니다:** 당신의 앱이 데이터베이스가 준비되기 전에 시작되었습니다. `depends_on`에 `condition: service_healthy`를 추가하고 데이터베이스 서비스에 `healthcheck`를 추가하세요. 위의 코드가 정확한 방법을 보여줍니다.

## 다음 단계

이제 도커가 실제로 어떻게 작동하는지 이해하게 되었습니다. 단순히 명령어가 아니라 그 뒤에 숨겨진 논리를 말입니다. 여기서 구축한 개념 모델은 엔지니어들이 프로덕션 배포를 디버깅할 때 사용하는 것과 동일합니다.

가고자 하는 방향에 따라 몇 가지 구체적인 다음 단계를 제안합니다.

*   **웹 애플리케이션을 구축한다면:** 다음 프로젝트에 도커 컴포즈를 사용하여 로컬 환경을 설정하세요. 위의 템플릿을 시작점으로 활용하세요.
*   **데이터 또는 AI 작업을 한다면:** 컨테이너에서 주피터 노트북(Jupyter notebooks)과 모델 서빙(model serving)을 실행하는 방법을 배우세요. 격리(isolation)와 재현성(reproducibility)은 ML 파이프라인에 엄청나게 중요합니다.
*   **데브옵스(DevOps) 또는 플랫폼 엔지니어링을 목표로 한다면:** 도커 네트워킹, 빌드킷을 사용한 멀티플랫폼 빌드, 컨테이너 보안 모범 사례에 대해 더 깊이 파고드세요.

가장 중요한 규칙은 이것입니다. 각 개념을 사용하기 전에 그 개념이 어떤 문제를 해결하는지 이해하세요. 볼륨은 컨테이너가 일시적이기 때문에 존재합니다. 멀티스테이지 빌드는 프로덕션 이미지가 간결해야 하기 때문에 존재합니다. 네트워크는 컨테이너가 기본적으로 격리되어야 하기 때문에 존재합니다.

도커는 복잡하지 않습니다. 일관성이 있을 뿐입니다. 그리고 이제 당신은 그 이유를 이해합니다.
