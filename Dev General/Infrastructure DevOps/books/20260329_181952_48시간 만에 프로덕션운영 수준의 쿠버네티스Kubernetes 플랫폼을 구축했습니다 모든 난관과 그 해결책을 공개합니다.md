지난주, 저는 완전한 데브섹옵스(DevSecOps) 인프라를 처음부터 구축하는 것에 도전했습니다. 단일 Nginx 파드(pod)를 사용하는 장난감 프로젝트가 아니라, 깃옵스(GitOps), 보안(secrets management), 관측 가능성(observability), 자동화된 배포 기능을 갖춘 실질적인 다중 노드 쿠버네티스(Kubernetes) 클러스터였습니다.

실제로 프로덕션(운영) 환경에서 볼 수 있는 종류의 설정 말입니다.

저는 스스로에게 48시간을 주었고, 상상할 수 있는 모든 난관에 부딪혔습니다. 그리고 이틀 동안 수개월간의 튜토리얼(tutorial)을 통해서보다 더 많은 것을 배웠습니다.

이것이 바로 그 이야기입니다.

### 왜 이 작업을 했는가

데브옵스(DevOps) 튜토리얼에서 가르치는 내용과 기업이 실제로 기대하는 것 사이의 격차에 지쳐 있었습니다. 모든 채용 공고에는 "프로덕션(운영) 쿠버네티스(Kubernetes) 경험"을 요구하지만, 이미 그 직무를 가지고 있지 않고서야 어떻게 그러한 경험을 얻을 수 있겠습니까?

그래서 저는 저만의 프로덕션(운영) 환경을 구축하기로 결정했습니다. 제가 직접 가리키며 "이것을 설계했고, 디버깅했으며, 이 모든 조각들이 어떻게 연결되는지 이해합니다"라고 말할 수 있는 무언가를 말이죠.

목표는 간단했습니다. 실제 데브섹옵스(DevSecOps) 관행을 보여주는 완벽한 플랫폼을 배포하는 것이었습니다. 코드형 인프라(Infrastructure as Code), 구성 관리(Configuration management), 깃옵스(GitOps) 워크플로우(workflow), 중앙 집중식 보안(secrets), 완벽한 관측 가능성(observability) 등이요.

지름길 없이. 무거운 작업을 처리하는 관리형 서비스(managed services)도 사용하지 않았습니다.

### 아키텍처 (The Architecture)

제가 구축하고자 했던 아키텍처(Architecture)는 다음과 같습니다.

**인프라(Infrastructure) 계층**
*   AWS에 EC2 인스턴스 3개 (마스터 노드 1개, 워커 노드 2개)
*   배스천(bastion) 접근이 가능한 프라이빗 서브넷(private subnet)
*   모든 프로비저닝(provisioning)을 위한 테라폼(Terraform)

**클러스터(Cluster) 계층**
*   kubeadm을 통한 쿠버네티스(Kubernetes) 설치
*   파드(pod) 네트워킹(networking)을 위한 칼리코(Calico)
*   베어메탈(bare-metal) 스타일 환경에서 로드 밸런싱(load balancing)을 위한 메탈LB(MetalLB)

**플랫폼(Platform) 계층**
*   인그레스 컨트롤러(ingress controller)로 트래픽(Traefik)
*   TLS 자동화를 위한 cert-manager
*   보안(secrets) 관리를 위한 해시코프 볼트(HashiCorp Vault)
*   볼트(Vault) 보안을 쿠버네티스(Kubernetes)로 동기화(sync)하기 위한 External Secrets Operator

**배포(Delivery) 계층**
*   깃옵스(GitOps) 배포를 위한 아르고CD(ArgoCD)
*   커스텀(custom) 3계층 애플리케이션(application) (프론트엔드, 백엔드, 데이터베이스)

**관측 가능성(Observability) 계층**
*   메트릭(metrics) 수집을 위한 프로메테우스(Prometheus)
*   시각화를 위한 그라파나(Grafana)
*   트래픽(Traefik) 메트릭(metrics) 통합

### 1-4시간: 기반이 무너지다

테라폼(Terraform)은 완벽하게 작동했습니다. 20분도 채 되지 않아 프라이빗 서브넷(private subnet)에 EC2 인스턴스 3개가 실행되고, 보안 그룹(security groups)이 구성되었으며, 마스터 노드(master node)를 배스천(bastion)으로 사용하여 SSH 접속도 가능했습니다.

그리고 kubeadm 차례였습니다.

첫 클러스터(cluster) 초기화는 조용히 실패했습니다. 오류 메시지(error messages)는 없었고, 컨트롤 플레인(control plane)은 준비 상태가 되지 않았습니다. kubectl 로그(logs)를 30분 동안 들여다본 후, 범인을 찾아냈습니다. t3.small 인스턴스(instance)는 쿠버네티스(Kubernetes) 컨트롤 플레인(control plane)을 실행할 만큼 충분한 메모리(memory)를 가지고 있지 않았던 것입니다.

**배운 점:** 쿠버네티스(Kubernetes)는 실제 리소스(resource) 요구사항이 있습니다. 마스터 노드(master node)는 최소 2GB의 RAM이 필요합니다. t3.medium 인스턴스(instance)로 업그레이드(upgrade)하자 클러스터(cluster)가 제대로 작동하기 시작했습니다.

### 5-12시간: 네트워킹(Networking) 악몽

작동하는 클러스터(cluster)를 확보한 후, 플랫폼 서비스(platform services)로 넘어갔습니다. 메탈LB(MetalLB)는 문제없이 설치되었습니다. 트래픽(Traefik)도 성공적으로 배포되었습니다. 첫 인그레스(Ingress) 리소스(resource)를 생성하고, nip.io 호스트명(hostname)을 구성한 뒤 서비스(service)에 접근을 시도했습니다.

아무것도 없었습니다. `curl` 요청(request)은 그저 멈춰 있었고, 타임아웃(timeout)도, 오류(error)도, 응답(response)도 없었습니다.

이것이 6시간 동안의 디버깅(debugging) 세션(session)의 시작이었으며, 어떤 강의보다도 쿠버네티스(Kubernetes) 네트워킹(networking)에 대해 더 많은 것을 가르쳐주었습니다.

문제는 여러 겹이었습니다. 첫째, 트래픽(Traefik)이 웹시큐어(websecure) 진입점(entrypoint)에서만 수신하도록 구성되어 있었는데, 저는 HTTP 요청(request)을 보내고 있었습니다. 그것을 수정한 후에는 인그레스(Ingress)가 존재하지 않는 TLS 보안(secret)을 참조하고 있어, 트래픽(Traefik)이 조용히 라우트(route) 구성을 실패한다는 것을 발견했습니다.

일단 이해하고 나니 해결책은 놀랍도록 간단했습니다.

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-service
  annotations:
    traefik.ingress.kubernetes.io/router.entrypoints: web # 웹 진입점 명시
spec:
  ingressClassName: traefik
  rules:
  - host: myapp.10.0.1.200.nip.io
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: my-service
            port:
              number: 80
```

TLS 블록(block) 없음. 명시적인 HTTP 진입점(entrypoint). 간단한 라우팅(routing).

### 13-18시간: 동기화(Sync)를 거부하는 보안(Secrets)

해시코프 볼트(HashiCorp Vault)는 개발 모드(dev mode)로 쉽게 설치되었습니다. 몇몇 보안(secrets)을 저장하고, 존재하는지 확인한 다음, External Secrets Operator를 설정하여 쿠버네티스(Kubernetes)로 가져오도록 했습니다.

ExternalSecret 리소스(resource)는 "SecretSynced: True"를 표시했지만, 실제 쿠버네티스(Kubernetes) 보안(Secret)은 비어 있었습니다.

또 다른 디버깅(debugging) 난관이었습니다.

문제는 볼트(Vault) 경로였습니다. 볼트(Vault)의 KV v2 보안 엔진(secrets engine)을 사용할 때, 실제 데이터(data)는 `secret/mypath`가 아닌 `secret/data/mypath`에 있습니다. API 경로(path)에는 UI가 숨기는 'data' 세그먼트(segment)가 추가되어 있습니다.

```yaml
apiVersion: external-secrets.io/v1
kind: ExternalSecret
metadata:
  name: taskapp-secrets
  namespace: taskapp
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-backend
    kind: ClusterSecretStore
  target:
    name: taskapp-secrets
  data:
    - secretKey: DB_PASSWORD
      remoteRef:
        key: secret/data/taskapp/config  # /data/ 세그먼트에 유의
        property: DB_PASSWORD
```

경로를 수정한 후, 보안(secrets)은 볼트(Vault)에서 애플리케이션 파드(pod)로 자동으로 전달되었습니다.

### 19-30시간: 아르고CD(ArgoCD)와 깃옵스(GitOps)의 발견

아르고CD(ArgoCD) 설치는 쉬운 부분이었습니다. 하지만 브라우저(browser)에서 접근 가능하게 만드는 것은 그렇지 않았습니다.

제 설정은 클러스터(cluster)가 프라이빗 서브넷(private subnet)에 있었기 때문에 접속을 위해 SSH 터널(tunnels)을 사용했습니다. 9090번 포트(port)를 메탈LB(MetalLB) IP로 터널링(tunneling)하고 아르고CD(ArgoCD)에 접속을 시도했을 때, 404 오류(error)가 발생했습니다.

문제는 미묘했습니다. 트래픽(Traefik)은 호스트 헤더(Host header)를 기반으로 라우팅(routing)합니다. `argocd.10.0.1.200.nip.io:9090`으로 접속하면, 브라우저(browser)는 포트 번호(port number)를 포함하는 호스트 헤더(Host header)를 보냅니다. 하지만 인그레스(Ingress)는 포트 없이 호스트명(hostname)만 일치하도록 구성되어 있었습니다.

해결책은 80번 포트(port)로 직접 터널링(tunneling)하는 것이었습니다.

```bash
sudo ssh -i k8s-key.pem ubuntu@master-ip -L 80:10.0.1.200:80 -N
```

그런 다음 로컬 호스트(hosts) 파일에 127.0.0.1을 가리키는 항목을 추가했습니다. 이제 호스트 헤더(Host header)가 정확히 일치했고, 아르고CD(ArgoCD)가 나타났습니다.

아르고CD(ArgoCD)가 작동하면서, 저는 마침내 깃옵스(GitOps)의 마법을 경험했습니다. 컨테이너 이미지(container image) 태그(tag)를 v1에서 v2로 업데이트(update)하는 변경 사항을 깃허브(GitHub) 저장소(repository)에 푸시(push)했습니다. 몇 초 만에 아르고CD(ArgoCD)는 변경을 감지하고 새 버전(version)을 자동으로 롤아웃(roll out)했습니다.

kubectl 명령(commands)도, 수동 개입(manual intervention)도 없었습니다. 그저 `git push`만 하고 지켜보면 되는 것이었습니다.

**배운 점:** 포트(port) 문제를 처리하지 않으면 호스트 기반 라우팅(Host-based routing)과 SSH 터널(tunnels)은 잘 어울리지 않습니다. 데모(demos)를 할 때는 항상 표준 포트(standard ports)로 터널링(tunneling)하세요.

### 31-40시간: 관측 가능성(Observability)의 공백

프로메테우스(Prometheus)와 그라파나(Grafana)는 `kube-prometheus-stack` 헬름 차트(Helm chart)를 통해 배포되었습니다. 하지만 그라파나(Grafana)의 트래픽(Traefik) 대시보드(dashboard)는 "데이터 없음(No data)" 패널(panels)만 보여주었습니다.

문제는 프로메테우스(Prometheus)가 트래픽(Traefik)을 스크래핑(scraping)하고 있지 않았다는 것입니다. ServiceMonitor를 생성해야 했지만, ServiceMonitor는 프로메테우스 오퍼레이터(Prometheus Operator)가 실행 중일 때만 작동합니다. 그리고 이 오퍼레이터(operator)는 프로젝트 초기에 메모리(memory) 부족으로 인해 레플리카(replicas)가 0으로 축소(scaled to zero)되어 있었습니다.

다시 스케일 업(scale up)하자 또 다른 문제가 드러났습니다. 오퍼레이터(operator)가 자체 ServiceMonitor 중 하나에 대한 TLS 인증서(certificate)가 없어서 크래시 루프(crash-loop)에 빠지고 있었던 것입니다. 이것은 헬름 차트(Helm chart) 설치에 이미 존재하던 문제였고, 저는 이를 해결해야 했습니다.

결국, 오퍼레이터(operator)를 트래픽(Traefik) ServiceMonitor를 처리할 수 있을 만큼 안정화시켰습니다.

```yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: traefik
  namespace: traefik
  labels:
    release: prometheus  # 이 레이블이 중요함
spec:
  selector:
    matchLabels:
      app.kubernetes.io/name: traefik
  endpoints:
    - port: metrics
      interval: 15s
```

`release: prometheus` 레이블(label)이 핵심이었습니다. 이것이 없으면 프로메테우스 오퍼레이터(Prometheus Operator)는 ServiceMonitor를 완전히 무시합니다.

**배운 점:** 프로메테우스 오퍼레이터(Prometheus Operator)는 레이블 셀렉터(label selectors)를 사용하여 ServiceMonitor를 발견합니다. 프로메테우스(Prometheus) 인스턴스(instance)가 어떤 레이블(labels)을 기대하는지 항상 확인하세요.

### 41-48시간: 모든 것을 통합하다

모든 조각들이 작동하자, 깃옵스(GitOps) 파이프라인(pipeline)을 통해 실제 애플리케이션(application)을 배포했습니다.

*   쿠버네티스 매니페스트(manifests)를 깃허브(GitHub)에 푸시(push)했습니다.
*   저장소(repository)를 가리키는 아르고CD(ArgoCD) 애플리케이션(Application)을 생성했습니다.
*   아르고CD(ArgoCD)가 리소스(resources)를 자동으로 동기화(sync)하는 것을 지켜봤습니다.
*   트래픽(Traefik)을 통해 애플리케이션(application)에 접근 가능한지 확인했습니다.
*   볼트(Vault)에서 보안(secrets)이 주입되었는지 확인했습니다.
*   그라파나(Grafana)에서 메트릭(metrics)이 정상적으로 흐르고 있는지 확인했습니다.

모든 것이 함께 작동했습니다. 깃(Git) 저장소(repository)에 대한 변경은 배포를 트리거(trigger)했습니다. 보안(secrets)은 깃(Git) 외부에 안전하게 보관되어 볼트(Vault)에서 흘러나왔습니다. 메트릭(metrics)은 실시간(real-time)으로 요청률(request rates)과 응답 시간(response times)을 보여주었습니다.

### 다음에 다르게 할 일

*   **더 큰 인스턴스(instance)로 시작했을 것입니다.** 메모리(memory) 관련 문제를 디버깅(debugging)하는 데 보낸 시간은 가치 있는 학습이 아니라 그저 좌절감만 안겨주었습니다.
*   **관측 가능성(observability)을 먼저 설정했을 것입니다.** 처음부터 프로메테우스(Prometheus)와 그라파나(Grafana)를 실행했다면 디버깅(debugging)이 훨씬 쉬웠을 것입니다. 보이지 않는 것은 고칠 수 없습니다.
*   **컨트롤러 로그(controller logs)를 즉시 확인했을 것입니다.** 모든 쿠버네티스(Kubernetes) 컨트롤러(controller)는 오류를 기록합니다. 트래픽(Traefik), cert-manager, 아르고CD(ArgoCD), External Secrets Operator 등 무엇이든 작동하지 않을 때, 해답은 거의 항상 로그(logs)에 있습니다.
*   **각 구성 요소(component)를 개별적으로 테스트했을 것입니다.** 저는 모든 것을 한 번에 설치하고 상호 작용을 디버깅(debugging)하려고 시도했습니다. 각 부분이 독립적으로 작동하는지 먼저 확인하는 것이 더 빨랐을 것입니다.

### 그만한 가치가 있었는가?

물론입니다.

이제 저는 실제 프로덕션(운영) 관행을 보여주는 깃허브(GitHub) 저장소(repository)를 가지고 있습니다. 강의에서 얻은 이론적 지식이 아니라, 제가 직접 구축하고 디버깅(debugging)한 작동하는 시스템입니다.

더 중요하게는, 저는 이 도구들이 왜 존재하는지 이해하게 되었습니다. 볼트(Vault) 없이 보안(secrets)을 관리하려고 시도했기 때문에 볼트(Vault)가 어떤 문제를 해결하는지 알고 있습니다. 깃옵스(GitOps)의 대안을 경험했기 때문에 깃옵스(GitOps)가 왜 중요한지 이해합니다. 관측 가능성(observability) 없이 몇 시간을 디버깅(debugging)했기 때문에 관측 가능성(observability)의 가치를 진정으로 깨닫습니다.

데브옵스(DevOps) 분야로 진입하거나 기술을 한 단계 발전시키고 싶다면, 이 접근 방식을 강력히 추천합니다. 야심 찬 프로젝트를 선택하고, 마감일을 정하고, 직접 구축해 보세요.

난관에 부딪힐 것입니다. 좌절할 것입니다. 하지만 배울 것입니다.

직접 시도해보거나 참고 자료로 사용하고 싶다면 제 깃허브(GitHub)에서 코드를 확인할 수 있습니다: [github.com/NanaGyamfiPrempeh30/k8s-devsecops](https://github.com/NanaGyamfiPrempeh30/k8s-devsecops)
