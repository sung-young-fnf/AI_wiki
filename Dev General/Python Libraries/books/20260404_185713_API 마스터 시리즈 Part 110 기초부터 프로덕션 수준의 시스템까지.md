# API 마스터 시리즈 (Part 1–10): 기초부터 프로덕션 수준의 시스템까지

API는 현대 소프트웨어를 하나로 묶어주는 보이지 않는 접착제와 같습니다. 하지만 많은 엔지니어는 업무 과정에서 스택 오버플로우(Stack Overflow) 검색에 의존하며 API를 단편적으로만 익히곤 합니다. 본 시리즈는 이러한 문제를 해결하기 위해 기획되었습니다. 1부에서 10부까지는 API의 개념, 통신 방식, 보안 방법 등 핵심적인 기초 레이어를 다룹니다.

불필요한 수식어는 빼고, 백엔드 엔지니어라면 반드시 알아야 할 핵심 개념과 코드, 구조를 정리했습니다.

### 1. 엔드포인트 (Endpoint)

엔드포인트는 서버의 특정 리소스를 나타내는 URL입니다.

`GET https://api.example.com/users/42`

엔드포인트를 주소라고 생각하면 쉽습니다. 서버는 고정된 위치에 있고, 엔드포인트는 클라이언트가 어디를 두드려야 할지 알려줍니다.

```text
클라이언트                 서버
  |                        |
  |-- GET /users/42 -----> |
  |<-- 200 { user data } --|
```

각 엔드포인트는 하나의 리소스와 하나의 작업에 매핑되어야 합니다. 여러 관심사를 섞는 것(예: `/getUserAndUpdateProfile`)은 좋지 않은 설계 방식입니다.

### 2. HTTP 메서드 (HTTP Methods)

HTTP 메서드는 리소스에 대해 수행할 작업을 정의합니다.

| 메서드 | 목적 |
| :--- | :--- |
| **GET** | 조회 |
| **POST** | 생성 |
| **PUT** | 교체(전체 수정) |
| **PATCH** | 부분 수정 |
| **DELETE** | 삭제 |

```python
# FastAPI 예시
@app.get("/products/{id}")
def get_product(id: int):
    return db.query(Product).get(id)

@app.post("/products")
def create_product(product: ProductSchema):
    return db.add(product)
```

올바른 작업에 올바른 메서드를 사용하십시오. 데이터를 수정하는 기능에 `GET`을 사용하는 것은 위험합니다. 이는 캐싱, 멱등성(Idempotency) 및 중간의 모든 HTTP 프록시 서버의 메커니즘을 깨뜨릴 수 있습니다.

### 3. 요청-응답 주기 (Request–Response Cycle)

모든 API 호출은 동일한 루프를 따릅니다. 클라이언트가 요청을 보내면 서버가 응답을 반환합니다.

*   **요청(Request):**
    *   메서드 (GET, POST...)
    *   헤더 (Content-Type, Authorization...)
    *   쿼리 파라미터 (?limit=10&page=2)
    *   본문 (POST/PUT을 위한 JSON 페이로드)
*   **응답(Response):**
    *   상태 코드 (200, 404, 500...)
    *   헤더
    *   본문 (JSON, XML, 이진 데이터 등)

이 전체 주기를 이해해야만 맹목적인 디버깅이 아닌 빠른 문제 해결이 가능한 엔지니어가 될 수 있습니다.

### 4. 상태 코드 (Status Codes)

상태 코드는 클라이언트가 응답 본문을 파싱하여 추측하지 않고도 어떤 일이 발생했는지 즉시 알 수 있게 해줍니다.

*   **2xx → 성공:** `200 OK`, `201 Created`, `204 No Content`
*   **4xx → 클라이언트 오류:** `400 Bad Request`, `401 Unauthorized`, `403 Forbidden`, `404 Not Found`, `429 Too Many Requests`
*   **5xx → 서버 오류:** `500 Internal Server Error`, `503 Service Unavailable`

흔히 하는 실수 중 하나는 본문에 `{ "error": "not found" }`를 포함하면서 상태 코드는 `200 OK`로 반환하는 것입니다. 이는 오해의 소지가 있으며 상태 코드에 의존하는 모니터링 도구들을 무용지물로 만듭니다.

### 5. 인증 (Authentication)

인증은 **"당신은 누구인가?"**라는 질문에 답하는 과정입니다.

```text
클라이언트                     인증 서버
  |                                |
  |--- POST /login (자격 증명) --->|
  |<-- JWT 토큰 -------------------|
  |                                |
  |--- GET /profile (+ 토큰) ----> API
  |<-- 200 사용자 데이터 ----------|
```

비밀번호를 원문(Raw) 그대로 저장하지 마십시오. URL에 토큰을 포함해 반환하지 마십시오. 항상 HTTPS를 사용하십시오.

### 6. 인가 (Authorization)

인가는 **"당신에게 허용된 작업은 무엇인가?"**라는 질문에 답하는 과정입니다.

인증과 인가는 서로 다릅니다. 유효한 토큰은 신원을 증명(인증)하지만, 액세스 가능 여부는 권한(인가)이 결정합니다.

```python
def require_role(required_role: str):
    def decorator(func):
        def wrapper(user=Depends(get_current_user), *args, **kwargs):
            if user.role != required_role:
                raise HTTPException(status_code=403, detail="Access denied")
            return func(*args, **kwargs)
        return wrapper
    return decorator

@app.delete("/users/{id}")
@require_role("admin")
def delete_user(id: int):
    ...
```

### 7. 액세스 토큰 (Access Tokens)

액세스 토큰은 매 요청마다 비밀번호를 전송하지 않고도 권한을 증명할 수 있는 단기 자격 증명입니다.

*   **액세스 토큰 (Access token):** 수명이 짧으며(분~시간 단위), API 호출에 사용됩니다.
*   **리프레시 토큰 (Refresh token):** 수명이 길며, 액세스 토큰이 만료되었을 때 새 토큰을 발급받는 데 사용됩니다.

### 8. OAuth 2.0

OAuth 2.0은 위임된 액세스(Delegated access)를 위한 업계 표준입니다. 제3자 앱에 비밀번호를 제공하는 대신, 토큰을 통해 제한된 권한을 부여합니다.

가장 많이 사용되는 방식은 **인가 코드 흐름(Authorization Code Flow)**입니다.

1.  사용자를 인가 서버로 리다이렉트합니다.
2.  사용자가 로그인하고 액세스 권한을 승인합니다.
3.  인가 서버가 앱으로 인가 코드(Auth Code)를 보냅니다.
4.  앱은 이 코드를 액세스 토큰으로 교환합니다.
5.  이후 API 호출 시 액세스 토큰을 사용합니다.

**주의:** 클라이언트 비밀키(`client_secret`)를 프론트엔드 코드에 포함하지 마십시오. 이는 서버 측에서만 관리해야 합니다.

### 9. 속도 제한 (Rate Limiting)

속도 제한은 클라이언트가 일정 시간 동안 보낼 수 있는 요청 수를 제한합니다. 이는 의도적이든 실수이든 인프라가 남용되는 것을 방지합니다.

*   **고정 윈도우(Fixed window):** 분당 X회 요청, 정각에 초기화.
*   **슬라이딩 윈도우(Sliding window):** 임의의 60초 간격 내에 X회 요청.
*   **토큰 버킷(Token bucket):** 토큰이 일정 비율로 채워지며, 일시적인 폭주(Burst) 허용.

제한을 초과할 경우 반드시 `429 Too Many Requests` 상태 코드와 함께 `Retry-After` 헤더를 반환하여 클라이언트가 적절히 대응할 수 있게 해야 합니다.

### 10. 스로틀링 (Throttling)

스로틀링은 흔히 속도 제한과 혼동되지만 차이점이 있습니다.

*   **속도 제한(Rate limiting):** 엄격한 차단. 한도 초과 시 즉시 차단.
*   **스로틀링(Throttling):** 유연한 제한. 한도 초과 시 차단하지 않고 응답 속도를 늦춤.

스로틀링은 서비스 중단보다 서비스 품질 저하가 더 나은 상황(예: 공개 API의 트래픽 급증)에서 유용합니다.

### 요약: 단일 요청의 흐름

위의 10가지 개념은 하나의 요청 흐름 속에서 다음과 같이 작동합니다.

1.  **엔드포인트** 라우팅 확인 (`/users/42`)
2.  **HTTP 메서드** 확인 (`GET`인가? `POST`인가?)
3.  **속도 제한 및 스로틀링** 체크
4.  **인증** (토큰이 유효한가?)
5.  **인가** (이 사용자가 이 작업을 수행할 수 있는가?)
6.  **액세스 토큰** 스코프(Scope) 확인
7.  **핸들러 로직** 실행
8.  **응답 생성** (헤더 + 본문)
9.  올바른 **상태 코드** 반환
10. 클라이언트 응답 수신

프로덕션 환경의 모든 API는 어떤 방식으로든 이 체인을 통과합니다. 각 레이어를 개별적으로 이해하면 API를 올바르게 설계하고 디버깅하는 것이 훨씬 수월해집니다.
