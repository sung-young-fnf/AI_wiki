### 1. 결정론적 작업 스케줄러(Deterministic Task Scheduler) 만들기 (cron 없이)

시간을 이해하는 것은 개발자에게 강력한 무기입니다.

대부분의 개발자는 스케줄링을 '크론(cron)이 알아서 하겠지'라며 넘깁니다. 하지만 새벽 3시에 크론이 아무 소리 없이 실패하기 전까지만 유효한 생각입니다. 그래서 저는 실행 순서, 재시도, 충돌 후 상태 복구를 보장하는 결정론적 스케줄러를 직접 구축했습니다.

**이 프로젝트가 흥미로운 이유:**
*   시간 드리프트(time drift)에 대해 고민하게 됩니다.
*   "5분마다 실행"이라는 말이 왜 모호할 수 있는지 배우게 됩니다.
*   메모리상의 환상이 아닌 영속성(persistence)을 다루게 됩니다.

**핵심 아이디어:**
실행 간격이 아니라, '다음 실행 타임스탬프'를 저장하는 것입니다.

```python
import time
import json
from datetime import datetime, timedelta

TASK_FILE = "tasks.json"

def load_tasks():
    try:
        with open(TASK_FILE) as f:
            return json.load(f)
    except FileNotFoundError:
        return {}

def save_tasks(tasks):
    with open(TASK_FILE, "w") as f:
        json.dump(tasks, f, indent=2)

def should_run(task):
    return datetime.utcnow() >= datetime.fromisoformat(task["next_run"])

def run_task(task):
    print(f"Running: {task['name']} at {datetime.utcnow().isoformat()}")
    task["next_run"] = (datetime.utcnow() + timedelta(seconds=task["interval"])).isoformat()

def scheduler_loop():
    tasks = load_tasks()
    while True:
        for task in tasks.values():
            if should_run(task):
                run_task(task)
        save_tasks(tasks)
        time.sleep(1)

scheduler_loop()
```

**배운 점:** 크론은 시간을 예약하지만, 프로덕션 시스템은 '보장'을 예약해야 합니다.

### 2. grep 재구현하기 — 제대로 동작하게

정규표현식(Regex)은 쉽지만, 정확한 텍스트 처리는 쉽지 않습니다.

단순히 정규표현식만 쓰다 보면 수년간 발견되지 않는 버그가 생기곤 합니다. 저는 다음 기능을 갖춘 간소화된 grep을 다시 만들었습니다.

*   파일을 한 줄씩 스트리밍(streaming) 처리
*   다양한 인코딩을 유연하게 처리
*   수 기가바이트(GB)의 로그에서도 메모리 폭발이 일어나지 않음

```python
import re

def grep(pattern, filepath, ignore_case=False):
    flags = re.IGNORECASE if ignore_case else 0
    regex = re.compile(pattern, flags)

    with open(filepath, "rb") as f:
        for line_no, raw in enumerate(f, 1):
            try:
                line = raw.decode("utf-8")
            except UnicodeDecodeError:
                continue
            if regex.search(line):
                yield line_no, line.rstrip()
```

**배운 점:** 텍스트 처리는 단순히 문자열의 문제가 아니라 시스템의 문제입니다.

### 3. 직접 메모리 누수 탐지기(Memory Leak Detector) 작성하기

메모리를 이해하지 못하면, 파이썬은 여러분에게 정중하게 거짓말을 할 것입니다.

파이썬은 메모리 관리를 매우 잘 숨기기 때문에, 서버가 스스로 재시작되기 전까지 개발자는 누수를 알아채지 못하는 경우가 많습니다. 저는 `tracemalloc`을 사용하여 간단한 할당 변화량(allocation delta) 추적기를 만들었습니다.

```python
import tracemalloc
import time

tracemalloc.start()

def snapshot(label):
    current, peak = tracemalloc.get_traced_memory()
    print(f"{label} | Current: {current / 1024:.2f}KB | Peak: {peak / 1024:.2f}KB")

snapshot("시작")

leaky = []
for _ in range(10_000):
    leaky.append("x" * 10_000)

snapshot("할당 후")
```

**배운 점:** 대부분의 "메모리 누수"는 진짜 누수가 아니라 잊혀진 참조(reference) 때문입니다. 파이썬은 잊지 않고 기다릴 뿐입니다.

### 4. 이진 로그(Binary Log) 포맷 구현 (JSON이 느리기 때문)

가독성이 좋다고 해서 효율적인 것은 아닙니다. JSON은 인간에게는 훌륭하지만, 기계에게는 더 나은 방식이 필요합니다. 저는 고빈도 이벤트를 위해 바이너리 기반의 추가 전용(append-only) 로그를 설계했습니다.

```python
import struct
import time

LOG_FORMAT = ">dI"  # 타임스탬프(double), 페이로드 크기(uint)

def write_event(file, payload: bytes):
    header = struct.pack(LOG_FORMAT, time.time(), len(payload))
    file.write(header)
    file.write(payload)

def read_events(file):
    while True:
        header = file.read(struct.calcsize(LOG_FORMAT))
        if not header:
            break
        ts, size = struct.unpack(LOG_FORMAT, header)
        payload = file.read(size)
        yield ts, payload
```

**배운 점:** 엔디언(endianness)과 프레이밍(framing)에 대해 배우게 되며, 원래 용도에 맞지 않는 곳에 JSON을 남용하는 것을 멈추게 됩니다. 실제로 구조화된 데이터에서 바이너리 포맷은 JSON보다 5~10배 더 작습니다.

### 5. 데드락 시뮬레이터(Deadlock Simulator) 구축

동시성(concurrency)은 시끄럽게 망가지지 않습니다. 정중하게 멈출 뿐입니다.

대부분의 사람들은 데드락이 의도적으로 발생하는 것을 보지 못한 채 '스레딩'을 배웁니다. 그래서 저는 데드락을 보장하는 시뮬레이터를 작성했습니다.

```python
import threading
import time

lock_a = threading.Lock()
lock_b = threading.Lock()

def task_one():
    with lock_a:
        time.sleep(0.1)
        with lock_b:
            print("Task one 완료")

def task_two():
    with lock_b:
        time.sleep(0.1)
        with lock_a:
            print("Task two 완료")

threading.Thread(target=task_one).start()
threading.Thread(target=task_two).start()
```

이 코드를 작성하고 수정하는 과정에서 그 어떤 튜토리얼보다 동시성에 대해 많은 것을 배웠습니다.

**배운 점:** 동시성 버그는 테스트를 실패하게 만드는 것이 아니라, 신뢰를 무너뜨립니다.

### 6. 코드 고고학(Code Archaeology) 도구 만들기

성숙한 코드베이스는 마치 사건 현장과 같습니다. 저는 Git과 파이썬만을 사용하여 다음과 같은 질문에 답하는 도구를 만들었습니다.

*   어떤 파일이 가장 자주 변경되는가?
*   어떤 함수가 버그를 끌어들이는가?
*   기술 부채(technical debt)가 실제로 어디에 쌓여 있는가?

```python
import subprocess
from collections import Counter

def churn():
    output = subprocess.check_output(
        ["git", "log", "--name-only", "--pretty=format:"],
        text=True
    )

    files = [line for line in output.splitlines() if line]
    return Counter(files).most_common(10)

for file, count in churn():
    print(f"{file}: {count}번의 변경")
```

**배운 점:** 어디를 리팩토링할지 추측하는 것을 멈추고 데이터에 기반한 직관을 갖게 됩니다. 개인의 의견이 아닌 역사를 읽는 법을 배우게 됩니다.
