# 자동화를 극도로 쉽게 만들어주는 6가지 파이썬 라이브러리

단순히 작동하는 자동화를 원합니다. 다음 엣지 케이스(edge case)에서 터져버리는 임시방편 스크립트가 아니라요. 아래는 제가 취약한 접착 코드(glue code)와 씨름하지 않고도 신뢰할 수 있는 자동화를 원할 때 사용하는, 널리 쓰이지 않지만 실전에서 검증된(battle-ready) 파이썬 라이브러리 6가지입니다. 이 라이브러리들은 흔히 블로그에서 볼 수 있는 일반적인 것들이 아닙니다. 크고 복잡한 문제를 사소하게 만드는 작고 정밀한 도구(surgical tools)들이죠.

각 항목에는 "왜 도움이 되는지"에 대한 한 단락 설명, 복사하여 바로 사용할 수 있는 작은 예제, 그리고 즉시 적용할 수 있는 프로 팁(Pro tip)이 포함되어 있습니다. 불필요한 내용은 없습니다. 바로 시작하겠습니다.

## 1) plumbum — 셸 우선 자동화 (로컬 + 원격)

`plumbum`을 "셸(shell)을 유창하게 구사하는 파이썬"이라고 생각해보세요. 깔끔한 파이썬 객체를 통해 하위 프로세스(subprocess), 파이프(pipe), 심지어 SSH를 통한 원격 명령어까지 제공하여, `subprocess` 모듈의 복잡함보다 훨씬 좋습니다.

```python
from plumbum import local, SshMachine

# 로컬 명령어 실행
git = local["git"]
print(git("status"))

# 셸처럼 명령어를 파이프(|)로 연결
out = (local["git"]["log"]["-n", "1"] | local["sed"]["-n", "1,3p"])()
print(out)

# SSH를 통해 원격 명령어 실행 (암호 없는 SSH 권장)
with SshMachine("host.example.com", user="deploy") as m:
    print(m["uname"]("-a"))
```

**왜 도움이 되는가:** 취약한 셸 문자열을 작성하는 대신, 조합 가능한 명령어 객체(composable command objects)를 얻게 됩니다. 이는 셸과 파이썬 작업을 결합하는 배포 단계(deployment steps)나 데이터 파이프라인을 자동화할 때 매우 유용합니다.

**프로 팁:** 키 기반 인증(key-based authentication)과 함께 `SshMachine`을 사용하고, 원격 작업을 작은 함수로 묶으세요. `SshMachine`을 `local`로 바꾸면 로컬에서 테스트하기가 더 쉬워집니다.

## 2) pexpect — 사람처럼 (하지만 더 빠르게) 대화형 CLI 자동화

어떤 도구가 대화형 프롬프트(`passwd`, 설치 프로그램, `telnet`, 사용자 정의 CLI)를 고집할 때, `pexpect`는 스크립트가 이를 안정적으로 제어할 수 있게 해줍니다.

```python
import pexpect

# 예제: 비밀번호를 묻는 일회성 ssh 자동화
child = pexpect.spawn("ssh user@host.example.com 'echo hello && exit'")
i = child.expect(["password:", pexpect.EOF, pexpect.TIMEOUT], timeout=10)
if i == 0:
    child.sendline("your_password_here")
child.expect(pexpect.EOF)
print(child.before.decode())
```

**왜 도움이 되는가:** 자동화를 위해 설계되지 않은 시스템도 스크립트화할 수 있습니다. `pexpect`는 전용 API가 없는 레거시 설치 프로그램이나 네트워크 장비를 자동화하는 방법입니다.

**프로 팁:** 가능한 경우 키 기반 인증을 선호하세요. 피할 수 없는 대화형 시스템에 `pexpect`를 사용하고, 그 상호작용을 작고 잘 문서화된 헬퍼(helper) 함수로 묶으세요.

## 3) watchfiles — 실제로 프로덕션에서 사용할 작고 번개처럼 빠른 파일 감시자

파일 변경 트리거(file-change triggers)는 기본적인 자동화의 연결 고리(automation glue)입니다. `watchfiles` (기존 감시자보다 현대적이고 빠름)는 플랫폼 전반에 걸쳐 파일 시스템(FS) 변경에 반응하는 간단한 이터레이터 API(iterator API)를 제공합니다.

```python
from watchfiles import watch
import subprocess

# 디렉토리를 감시하고 변경 시 빌드 단계 실행
for changes in watch('src'):
    print("Detected:", changes:", changes)
    subprocess.run(["./build.sh"])
```

**왜 도움이 되는가:** 가벼운 CI (Continuous Integration), 로컬 파일 기반 워크플로우 또는 무거운 도구를 원하지 않는 자동 재로드 작업에 완벽합니다.

**프로 팁:** 대량 쓰기(bulk writes) 중에 동일한 작업을 여러 번 실행하는 것을 피하기 위해 `watchfiles`를 디바운싱(debouncing, 200~500ms 동안 불필요한 이벤트 무시)과 결합하세요.

## 4) fs (PyFilesystem2) — 모든 저장소를 하나의 API처럼 다루기 (로컬, zip, sftp, 메모리...)

자동화가 여러 저장소 유형(SFTP, ZIP, AWS S3 어댑터, 인메모리)을 다룰 때, `fs`는 단일 파일 시스템 추상화(filesystem abstraction)를 제공하여 코드가 특별한 케이스로 인해 비대해지는 것을 방지합니다.

```python
from fs import open_fs

# 일반 FS처럼 zip 아카이브에서 파일 읽기
with open_fs('zip://archive.zip') as z:
    print(z.readtext('docs/readme.md'))

# SFTP 서버에 연결 (paramiko 필요)
with open_fs('sftp://user:pass@host/path') as remote:
    for f in remote.listdir('/'):
        print(f)
```

**왜 도움이 되는가:** 일단 I/O를 `fs` 뒤에 추상화하면, 로컬 파일을 원격 저장소로 바꾸는 것은 한 줄의 설정 변경으로 가능하며, 테스트와 배포에 강력합니다.

**프로 팁:** 단위 테스트에서 `fs.memoryfs.MemoryFS()`를 사용하여 파일 처리 코드가 즉시, 그리고 결정적으로 실행되도록 하세요.

## 5) tenacity — 백오프(backoff) 및 지터(jitter)를 통한 완벽한 재시도

자동화는 원격 호출이 불안정할 때 종종 실패합니다. `tenacity`는 견고한 재시도 전략(robust retry strategies)을 쉽게 구현할 수 있게 해줍니다. 데코레이터(decorator), 백오프, 조건부 재시도 및 취소 기능을 제공합니다.

```python
from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type
import requests

@retry(stop=stop_after_attempt(5), wait=wait_exponential(multiplier=1, min=1, max=10),
       retry=retry_if_exception_type(requests.RequestException))
def fetch(url):
    r = requests.get(url, timeout=5)
    r.raise_for_status()
    return r.text

print(fetch("https://example.com/api"))
```

**왜 도움이 되는가:** 모든 네트워크 호출을 맞춤형 `try/except/backoff`로 감싸는 대신, `tenacity`는 프로젝트 전체에서 재사용할 수 있는 안정적인 재시도 의미론(retry semantics)을 중앙 집중화합니다.

**프로 팁:** `tenacity`를 서킷 브레이커(circuit-breaker) 또는 카운터(counter)와 결합하여 지속적인 실패가 조용히 영원히 재시도되는 대신 모니터링 시스템에 보고되도록 하세요.

## 6) doit — 의존성(deps) 및 캐싱과 잘 작동하는 선언적 작업 자동화

`doit`은 의존성, 점진적 빌드(incremental builds), 병렬 실행을 이해하는 태스크 러너(task runner, 강화된 Makefile이라고 생각하세요)입니다. 데이터 작업, ETL (Extract, Transform, Load), 복잡한 자동화 흐름에 탁월합니다.

`dodo.py` 파일을 생성하세요:

```python
# dodo.py
def task_download():
    return {
        'actions': ['curl -sS -o data.csv https://example.com/data.csv'],
        'targets': ['data.csv'],
    }

def task_process():
    return {
        'actions': ['python process.py data.csv processed.csv'],
        'file_dep': ['data.csv'],
        'targets': ['processed.csv'],
    }
```

다음과 같이 실행하세요:

```bash
$ doit        # 작업을 실행하고, 의존성을 파악하며, 최신 상태의 작업은 건너뜁니다.
$ doit -n     # 드라이 런 (dry-run)
```

**왜 도움이 되는가:** 최소한의 형식으로 재현 가능한 파이프라인을 얻을 수 있으며, 내장된 최신 상태 확인(up-to-date checks) 기능 덕분에 긴 작업이 불필요하게 다시 실행되지 않습니다.

**프로 팁:** 상세도(verbosity)와 리포터(reporter)를 사용하여 `doit` 출력을 CI (Continuous Integration) 로그에 통합하세요. 가장 작고 안정적인 작업을 먼저 배치하여 의존성 그래프가 무거운 작업을 조기에 가지치기(prunes heavy work early)하도록 만드세요.
