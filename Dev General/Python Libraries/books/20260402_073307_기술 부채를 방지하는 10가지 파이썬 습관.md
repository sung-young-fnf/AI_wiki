# 기술 부채를 방지하는 10가지 파이썬 습관

기술 부채(Technical debt)는 미리 경고하고 찾아오지 않습니다. 보통 새벽 2시경, "작은 변경 사항" 하나가 서로 관련 없는 대여섯 개의 기능을 망가뜨릴 때 그 모습을 드러내곤 합니다.

4년 이상 프로덕션 환경에서 파이썬(Python) 코드를 작성하며 깨달은 불편한 진실은 다음과 같습니다. 대부분의 기술 부채는 나쁜 코드 때문에 발생하는 것이 아니라, 규율 없이 작성된 좋은 코드 때문에 발생합니다.

당장은 번거로워 보일 수 있는 아래의 습관들이 여러분의 시간을 수개월씩 아껴줄 것입니다.

### 1. 임포트(Import)를 의존성 계약으로 취급하기

모듈이 프로젝트의 절반을 임포트하고 있다면, 그 설계는 이미 잘못된 것입니다. 임포트는 단순히 코드에 접근하는 수단이 아니라, 모듈 간의 결합도(Coupling)를 정의합니다. 변동성이 큰 의존성에는 **지연 임포트(Lazy imports)**를 활용하는 습관을 들이세요.

```python
def load_user(user_id):
    from services.user_store import fetch_user  # 필요한 시점에 의존성 주입
    return fetch_user(user_id)
```

**이 방식이 중요한 이유:**
* 순환 임포트(Circular imports)를 방지합니다.
* 리팩토링(Refactoring) 시 영향 범위를 줄여줍니다.
* 콜드 스타트(Cold start) 속도를 높입니다.
* 모듈 간의 경계를 명확히 고민하게 만듭니다.

### 2. 리팩토링 전 골든 마스터 테스트(Golden Master Tests)로 동작 고정하기

레거시 코드를 수정하기 전에는, 설령 그 코드가 엉망일지라도 현재의 동작 방식을 그대로 고정해야 합니다. 골든 마스터 테스트란 "이 코드가 정확히 무엇을 하는지는 몰라도, 결과값은 반드시 이전과 같아야 한다"는 원칙입니다.

```python
def snapshot(func, *args):
    return repr(func(*args))

def test_legacy_behavior():
    assert snapshot(legacy_calc, 10, 5) == "42"
```

논리를 리팩토링하는 것이 아니라, 결과에 대한 확신을 리팩토링하는 것입니다. 이 습관은 시스템 전체를 맹목적으로 재작성하는 위험으로부터 당신을 구해줄 것입니다.

### 3. `__all__`을 공개 API 방화벽으로 사용하기

모든 것을 임포트할 수 있게 열어두면, 모든 것이 다른 코드의 의존성이 됩니다. 내부 헬퍼 함수가 영구적인 유지보수 대상이 되는 이유가 바로 이것입니다.

```python
__all__ = ["calculate_invoice", "InvoiceError"]

def _round_tax(value):
    ...

def calculate_invoice(data):
    ...
```

`__all__`에 포함되지 않은 것은 의도적으로 비공개(Private)임을 명시하세요. 이는 라이브러리가 버전 업데이트 시에도 살아남는 비결입니다.

### 4. 사후 처리가 아닌 `tracemalloc`으로 조기에 메모리 측정하기

성능 버그는 눈에 띄지만, 메모리 누수(Memory leak)는 소리 없는 암살자와 같습니다. 사용자가 불평하기 전에 메모리 사용량을 측정하세요.

```python
import tracemalloc

tracemalloc.start()
process_data()
current, peak = tracemalloc.get_traced_memory()

print(f"현재: {current / 1024:.1f}KB, 최대: {peak / 1024:.1f}KB")
tracemalloc.stop()
```

측정하지 않으면 추측하게 되고, 추측은 결국 시스템 재작성으로 이어집니다.

### 5. 함수를 지루할 정도로 예측 가능하게 만들기

함수가 파일 읽기, 전역 변수 수정, API 호출, 로깅, 데이터 반환을 동시에 수행한다면 그것은 함수가 아니라 잠재적인 부채입니다. **결정(Decision)과 실행(Execution)을 분리**하세요.

```python
# 순수 함수 (Pure function): 결정 로직
def compute_discount(price, tier):
    return price * (0.1 if tier == "gold" else 0.05)

# 실행 함수: 부수 효과(Side-effect) 관리
def apply_discount(user, price):
    discount = compute_discount(price, user.tier)
    save_transaction(user.id, discount)
```

순수 함수는 시간이 지날수록 가치가 높아지지만, 부수 효과가 섞인 함수는 금방 부패합니다.

### 6. 조용히 넘기지 말고 명시적으로 실패하기 (Fail Loudly)

조용한 실패는 나중에 더 큰 버그로 돌아옵니다. 예외 처리 시 맥락을 유지하세요.

*   **나쁜 예:** 예외를 잡아서 무시함 (`pass`)
*   **좋은 예:** **예외 체이닝(Chained exceptions)** 활용

```python
except Exception as e:
    raise RuntimeError("처리 실패") from e
```

연결된 예외는 맥락을 보존하며, 이는 나중에 코드의 역사를 추적할 때 고고학자가 될 필요가 없게 만들어 줍니다.

### 7. `inspect` 모듈로 죽은 코드(Dead Code) 제거하기

사용하지 않는 코드는 무해하지 않습니다. 미래의 개발자를 기만할 뿐입니다. 주기적으로 사용되지 않는 매개변수와 시그니처를 스캔하세요.

```python
import inspect

def audit(func):
    print(inspect.signature(func))

audit(legacy_handler)
```

전혀 사용되지 않는 매개변수나 아무 역할도 하지 않는 설정들을 제거하는 것만으로도, 어떤 리팩토링보다 인지적 부하(Mental load)를 크게 줄일 수 있습니다.

### 8. 마법 같은 설정이 아닌 명시적인 설정 지향하기

설정값이 환경 변수, 기본값, 숨겨진 임포트 등에 흩어져 있다면 그것은 시스템이 아니라 수수께끼입니다.

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class Config:
    retries: int
    timeout: float
```

설정을 옆으로 전달(숨은 의존성)하지 말고, 아래로 명시적으로 전달하세요. 이 습관은 몇 년 후에도 시스템을 테스트 가능하게 유지해 줍니다.

### 9. 코드뿐만 아니라 데이터 구조도 버전 관리하기

코드는 빠르게 변하지만, 데이터는 영원히 남습니다. 데이터 구조에 버전이 없다면 미래의 당신이 그 대가를 치르게 됩니다.

```python
def migrate_v1_to_v2(record):
    record["full_name"] = f"{record.pop('first')} {record.pop('last')}"
    record["_version"] = 2
    return record
```

데이터 버전 관리가 없는 기술 부채는 복리로 늘어납니다.

### 10. 자신의 코드를 낯선 사람처럼 읽기

코드를 배포하기 전 스스로에게 질문해 보세요.
"만약 내가 모르는 코드베이스에서 이 코드를 봤다면, 이 코드를 신뢰할 수 있을까?"

만약 대답이 "글쎄..."라면, 그 "글쎄"는 곧 부채가 됩니다. 초기에 스스로에게 엄격해지는 것이 나중에 코드를 재작성하는 수고를 덜어주는 가장 확실한 방법입니다.
