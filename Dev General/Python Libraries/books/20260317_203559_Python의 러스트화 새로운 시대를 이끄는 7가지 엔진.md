# Python의 러스트화: 새로운 시대를 이끄는 7가지 엔진

러스트(Rust) 기반 도구들이 파이썬(Python)의 인프라(infrastructure) 계층을 재편하고 있습니다.

## 서론

지난 몇 년간 파이썬(Python) 생태계에서 명확한 패턴이 나타나고 있습니다.

여전히 대다수의 개발자는 파이썬으로 코드를 작성하지만, 파이썬 주변의 많은 도구와 라이브러리가 점점 더 러스트(Rust)로 작성되고 있습니다.

이는 파이썬에 대한 반란이나 파이썬 언어를 대체하려는 움직임이 아닙니다. 대신, 언어의 단순성과 도구 레이어(tooling layer)의 유용성을 유지하면서 애플리케이션(apps), 라이브러리(libraries), 워크플로(workflows)의 핵심 측면에서 효율성을 높이는 데 목적이 있습니다.

아래에서는 이러한 변화의 가장 강력한 사례와 파이썬 기반의 기존 도구들과 어떻게 비교되는지 살펴보겠습니다.

## 1. uv vs Poetry

포에트리(Poetry)는 파이썬 패키징(packaging)을 현대화했습니다. `pyproject.toml`을 중심으로 의존성 관리(dependency management), 잠금(locking), 가상 환경(virtual environments)을 통합하여 파편화된 생태계에 구조를 제공했습니다.

`uv`는 시스템적 관점(systems perspective)에서 동일한 문제에 접근합니다.

### `uv`의 장점:
*   러스트(Rust)로 작성되었으며 단일 바이너리(single binary)로 배포됨
*   매우 빠른 의존성 해결(dependency resolution)
*   적극적인 전역 캐싱(global caching)
*   내장된 파이썬 버전 관리(version management)
*   네이티브 워크스페이스(workspace) 지원
*   `uv pip` 호환성 레이어(compatibility layer)

### 기본 워크플로(workflow) 비교:

**Poetry:**
```bash
poetry add fastapi
poetry install
poetry run python main.py
```

**uv:**
```bash
uv add fastapi
uv sync
uv run python main.py
```

철학은 비슷하지만, 성능과 통합성에서 차이가 있습니다. `uv`는 포에트리의 역할 외에도 `pip`, `virtualenv`, `pipx`, 그리고 `pyenv`의 일부 기능을 대체합니다.

## 2. Ruff vs Flake8 (및 유사 도구)

파이썬 린팅(linting)은 역사적으로 여러 도구를 포함했습니다:
*   `Flake8`
*   `isort`
*   `pylint`
*   `Black` (포매팅(formatting)용)

`Ruff`는 이들 중 다수를 단일 러스트 바이너리로 통합합니다.

### `Ruff`의 장점:
*   매우 빠름 (대규모 저장소(repo)에서 종종 밀리초 단위)
*   여러 도구의 규칙을 구현
*   플러그인 로딩(plugin-loading) 오버헤드(overhead) 없음
*   린터(linter) 및 포매터(formatter) 역할 모두 수행 가능

### 예시:

**Flake8 + isort:**
```bash
flake8 .
isort .
```

**Ruff:**
```bash
ruff check .
ruff format .
```

얻는 것은 단순히 속도만이 아닙니다. 바로 단순화입니다. 더 적은 의존성(dependencies), 더 적은 설정 레이어(config layers), 더 빠른 CI(Continuous Integration) 피드백 루프(feedback loops)를 제공합니다.

## 3. Polars vs pandas

`pandas`는 파이썬에서 데이터프레임(dataframe) 워크플로를 정의했으며, 해당 API는 10년 이상 데이터 과학(data science)의 방향을 제시했습니다.

러스트(Rust)로 작성된 `Polars`는 실행 방식에 대한 재고를 제시합니다.

### `Polars`의 장점:
*   지연 실행 엔진(lazy execution engine)
*   자동 쿼리 최적화(query optimization)
*   기본적으로 병렬 처리(parallelism)
*   대규모 데이터셋(datasets)에서 더 나은 성능

**pandas:**
```python
df = df[df["value"] > 10]
result = df.groupby("category").sum()
```

**Polars (지연 모드):**
```python
result = (
    df.lazy()
      .filter(pl.col("value") > 10)
      .groupby("category")
      .sum()
      .collect()
)
```

구문은 유사하지만, 실행 모델(execution model)이 다릅니다. `Polars`는 실행 전에 쿼리 계획(query plan)을 구축하고 최적화하는 반면, `pandas`는 즉시(eagerly) 실행합니다.

이러한 변화는 스크립팅(scripting) 방식의 데이터 조작(data manipulation)보다는 데이터베이스(database) 방식의 사고방식과 더 유사합니다.

## 4. Pydantic (v2) vs v1 구현체

`Pydantic` 자체는 파이썬 라이브러리이지만, 버전 2에서는 주요 내부 재작성(internal rewrite)이 이루어졌습니다.

내부적으로는 러스트로 작성된 `pydantic-core`에 의존합니다.

### `Pydantic v2`의 장점:
*   유효성 검사(validation)에서 상당한 성능 향상
*   더 낮은 메모리 오버헤드(memory overhead)
*   더 예측 가능한 파싱(parsing) 동작

사용자 대면 코드(user-facing code)는 거의 변하지 않았습니다:
```python
from pydantic import BaseModel

class User(BaseModel):
    id: int
    name: str
```

러스트 레이어(Rust layer)는 보이지 않지만, 성능 향상은 측정 가능합니다. 이는 반복되는 주제입니다: 러스트는 엔진, 파이썬은 인터페이스(interface)입니다.

## 5. cryptography (러스트 컴포넌트) vs 기존 C 기반 확장

`cryptography` 라이브러리는 역사적으로 `OpenSSL` 바인딩(bindings)과 C 확장(C extensions)에 크게 의존했습니다.

최근 몇 년 동안 그 일부가 러스트로 재작성되었습니다.

### 러스트 컴포넌트(components)의 장점:
*   향상된 메모리 안전성(memory safety)
*   안전하지 않은 C 패턴(patterns)에 대한 의존성 감소
*   더 나은 장기 유지보수성(maintainability)

보안에 민감한 코드(security-sensitive code)는 러스트의 보장으로부터 직접적인 이점을 얻습니다. 이 경우 러스트는 속도보다는 안전성에 더 중점을 둡니다.

## 6. PyO3 vs 전통적인 CPython C-API 확장

러스트가 등장하기 전에는 성능에 중요한 파이썬 확장(extensions)이 일반적으로 CPython C-API를 사용하여 C로 작성되었습니다.

그 방식은 작동하지만, 복잡하고 안전하지 않습니다.

### `PyO3`는 다음을 제공합니다:
*   파이썬에 대한 더 안전한 러스트 바인딩
*   더 깔끔한 오류 처리(error handling)
*   더 쉬운 패키징(packaging) (특히 `maturin`과 같은 도구 사용 시)
*   메모리 안전성에 대한 더 강력한 보장

이는 네이티브 확장(native extensions) 작성의 장벽을 낮추며, 러스트 기반의 파이썬 라이브러리가 더 많이 등장하는 이유를 부분적으로 설명합니다.

## 7. ty vs mypy

파이썬의 정적 타입 검사(static type checking)는 오랫동안 `mypy`와 연관되어 있었습니다.

러스트로 개발된 `ty`는 다음에 중점을 둔 새로운 접근 방식을 나타냅니다:
*   성능
*   증분 검사(incremental checking)
*   대규모 저장소에서 더 낮은 지연 시간(latency)

아직 발전 중이지만, 이는 동일한 경향을 반영합니다: 정적 분석 도구(static analysis tools)가 규모를 처리하기 위해 컴파일된 구현(compiled implementations)으로 이동하고 있다는 것입니다.

## 일관된 패턴

이러한 사례들 전반에 걸쳐 구조는 유사합니다:
*   파이썬은 사용자 대면 API(user-facing API)로 남습니다.
*   러스트는 성능에 중요한 내부 요소(internals)를 처리합니다.
*   도구들은 기능을 통합합니다.
*   단일 바이너리 배포(single-binary distribution)가 보편화됩니다.
*   CI 피드백 루프가 더 빨라집니다.

이전의 도구들(antagonists)이 "나쁜" 선택이었다는 것은 아닙니다. `Poetry`, `Flake8`, `pandas`, `mypy` 모두 생태계를 형성하는 데 기여했습니다.

그러나 저장소가 커지고 CI 파이프라인(CI pipelines)이 개발의 중심이 되면서, 성능과 시작 시간(startup time)은 "있으면 좋은 것"에서 "기본적인 기대치"로 변화하고 있습니다.

러스트는 이러한 제약 조건(constraints)에 잘 부합합니다.

## 결론

파이썬의 러스트화는 파이썬을 대체하는 것이 아닙니다. 많은 최신 도구와 라이브러리에서 파이썬은 인터페이스이고, 러스트는 엔진입니다.

위에 언급된 모든 사례들은 고립된 혁신이 아닙니다. 이들은 생태계가 기반을 구축하는 방식에 있어서 구조적인 변화를 나타냅니다.

이는 다음 사이의 경계를 재정의하는 것입니다:
*   높은 수준의 개발자 생산성(developer productivity).
*   낮은 수준의 실행 성능(execution performance)

다시 말해:
*   파이썬은 계속해서 핵심적인 위치를 차지합니다.
*   러스트는 파이썬을 둘러싼 기계 장치(machinery)에 점점 더 많은 동력을 제공합니다.

그리고 이러한 계층화된 모델(layered model)이 파이썬 진화의 다음 단계를 정의할 수 있습니다.
