# 5 “Hidden” Python Libraries That Solved Problems I Didn’t Know I Was Fighting

1년 전, 제 코드는 작동하고 있었지만 제 삶은 그렇지 못했습니다. 스크립트는 지연 실행되었고, 크론 작업(Cron jobs)은 아무런 예고 없이 실패했으며, 로그는 저를 속였습니다. 자동화는 마치 테이프로 겨우 붙여놓은 유리처럼 위태로웠습니다.

저는 초보자가 아니었습니다. 수년간 파이썬을 다뤄왔지만, 시스템적인 문제 대신 표면적인 문제만 해결하고 있었던 것입니다. 어느 날 새벽, '단순한' 자동화 스크립트를 디버깅하다가 불편한 진실을 깨달았습니다. 제가 사용하는 도구들은 대중적이었을 뿐, 실용적이지 않았다는 사실입니다.

그때부터 저는 깃허브의 별점이나 유행이 아닌, 실제 자동화의 고통을 해결해 줄 도구들을 찾기 시작했습니다. 그리고 제가 찾아낸 라이브러리들은 제가 코드를 작성하는 방식을 완전히 바꾸어 놓았습니다.

### 1. tenacity — 두려움 없는 자동화
**문제: 불안정한 시스템**

API는 타임아웃이 발생하고, 네트워크는 일시적으로 끊기며, 파일은 무작위로 잠깁니다. 대부분의 자동화 스크립트가 실패하는 이유는 로직 때문이 아니라, 외부 세계가 불안정하기 때문입니다.

`tenacity`를 사용하기 전, 저의 재시도(retry) 로직은 조잡한 `try-except`와 `time.sleep`의 반복이었습니다. 하지만 이는 회복 탄력성이 아니라 단순한 '희망 사항'에 불과했습니다.

`tenacity`는 다음과 같은 기능을 제공합니다:
- **지수 백오프(Exponential backoff):** 실패할수록 재시도 간격을 늘려 시스템 부담을 줄임
- **깔끔한 데코레이터:** 코드 가독성 향상
- **상용구(Boilerplate) 제거:** 핵심 로직에만 집중 가능

**자동화의 교훈:** 스크립트가 우아하게 실패하지 못한다면, 그것은 자동화가 아니라 '예약된 불안'일 뿐입니다.

### 2. watchfiles — 살아있는 실시간 자동화
**문제: 폴링(Polling)의 한계**

과거에는 특정 폴더를 확인하고, 잠시 멈췄다가, 다시 확인하는 방식으로 자동화를 구현했습니다. 이러한 폴링 방식은 비효율적이며 이벤트를 놓치기 쉽습니다.

**해결책: 이벤트 기반 스크립트**
`watchfiles`를 사용하면 루프가 실제로 변화가 일어날 때만 실행됩니다. 크론이나 `sleep` 루프 없이도 CPU 낭비를 막을 수 있습니다. 저는 이 라이브러리를 다음과 같은 작업에 사용합니다:
- 송장 자동 처리
- 머신러닝 파이프라인 트리거
- 데이터 드롭 모니터링
- 로컬 ETL 파이프라인

**자동화의 교훈:** 주기적으로 확인하는 것보다 사건에 반응하는 것이 매번 더 효율적입니다.

### 3. python-dotenv — 정신 건강을 지켜주는 비밀 관리
**문제: 설정 관리의 부주의**

설정값을 코드에 하드코딩하거나, 비밀번호를 커밋하고, "작은 변경" 하나로 운영 환경을 망가뜨리는 실수는 자동화의 효율보다 고통을 더 크게 만듭니다.

`python-dotenv`를 사용하면 모든 자동화 스크립트에서 코드를 수정하지 않고도 환경 변수를 관리할 수 있습니다.
- 코드 내에 비밀 정보가 남지 않음
- 로컬과 운영 환경에서 동일하게 작동
- AWS, Docker, CI 환경에서 깔끔한 배포 가능

**자동화의 교훈:** 설정 과정이 지루하게 느껴질 만큼 단순하지 않다면, 그것은 위험한 신호입니다.

### 4. schedule — 크론(Cron)이 과할 때의 대안
**문제: 크론의 가독성과 이식성 문제**

크론은 강력하지만 파이썬 코드 외부에서 관리해야 하며, 디버깅이 어렵고 이식성이 떨어집니다. 저는 애플리케이션 수준의 자동화를 위해 크론 대신 `schedule`을 사용하기 시작했습니다.

이 라이브러리는 파이썬 내부에서 사람이 읽기 쉬운 언어로 스케줄링을 구현하게 해줍니다.
- `schedule.every().day.at("09:00").do(job)`와 같이 직관적인 작성 가능
- 테스트 및 배포가 용이함
- Docker 컨테이너나 EC2 백그라운드 서비스 내에서 실행 가능

**자동화의 교훈:** 파이썬 내에서 디버깅할 수 없다면, 파이썬 밖에서 실행하지 마십시오.

### 5. invoke — Bash를 대체하는 자동화
**문제: 확장되지 않는 쉘 스크립트**

수많은 쉘 스크립트와 관리되지 않는 리드미(README) 파일은 자동화의 신뢰도를 떨어뜨립니다. 저는 이 모든 것을 `invoke`를 활용한 파이썬 작업(task)으로 교체했습니다.

`invoke`를 사용하면 빌드 파이프라인, 배포 자동화, 재현 가능한 워크플로우 등을 파이썬 언어로 관리할 수 있습니다. 쉘 스크립트와 동일한 힘을 가지면서도 파이썬의 강력한 테스트 및 버전 관리 기능을 그대로 누릴 수 있습니다.

**자동화의 교훈:** 중요한 작업이라면 Bash가 아닌 파이썬으로 관리할 가치가 있습니다.

### 자동화에 대한 불편한 진실

대부분의 자동화가 실패하는 이유는 그것을 '시스템'이 아닌 '스크립트'로 작성하기 때문입니다. 위에서 언급한 라이브러리들은 새로운 기능을 추가하는 것이 아니라, **취약성을 제거**하는 데 목적이 있습니다. 코드를 기발하게 만드는 것이 아니라, **지루할 정도로 안정적**으로 만들어 줍니다. 그리고 지루할 정도의 안정성, 그것이 바로 자동화가 꿈꾸는 최종 단계입니다.

오늘도 여전히 수동으로 재시도하고, 폴더를 계속 확인하며, 한밤중에 크론을 디버깅하고 있다면 당신이 파이썬에 서툰 것이 아닙니다. 단지 **적절한 도구**를 놓치고 있을 뿐입니다. 최고의 자동화 코드는 당신이 낭비하고 있던 수많은 시간을 구원해 줄 때까지 눈에 띄지 않는 법입니다.
