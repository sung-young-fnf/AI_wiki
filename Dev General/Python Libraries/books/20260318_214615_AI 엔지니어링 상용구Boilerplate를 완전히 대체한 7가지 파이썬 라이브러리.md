"스트리밍을 원하세요? SSE를 직접 파싱하겠습니다. 얼마나 어렵겠어요?"

어려웠습니다. 터무니없이 어려웠죠. OpenAI의 형식을 처리하는 스트리밍 파서(parser)를 작성하는 데 주말을 보냈지만, Anthropic이 형식을 변경하자 제 "범용" 파서는 단 하나의 공급업체에만 적용되는 것이 되어버렸습니다.

바로 그 순간, 저는 사용자 지정 API 래퍼(wrapper) 작성을 중단했습니다. 6개월이 지나고 수많은 코드를 삭제한 후, 여기 저의 상용구(boilerplate) 코드를 대체한 7가지 라이브러리가 있습니다. 각각은 동일한 패턴을 따릅니다: 제가 수동으로 계속 해결하던 문제, 몇 줄로 그 문제를 해결해 준 라이브러리, 그리고 README에서는 아무도 언급하지 않는 솔직한 한계점입니다.

### 1. LiteLLM: 모든 LLM 공급업체를 위한 하나의 인터페이스

**문제:** GPT-4와 Claude, Llama를 비교 테스트하고 싶습니다. 그래서 세 가지 별도의 클라이언트 구성, 세 가지 응답 파서, 세 가지 오류 핸들러를 작성해야 합니다.

```python
from litellm import completion
# 동일한 인터페이스. 모든 공급업체에 적용.
response = completion(
    model="gpt-4",
    messages=[{"role": "user", "content": "Explain RAG in one paragraph"}]
)

# Claude로 전환하려면? 문자열 하나만 변경하세요.
response = completion(
    model="claude-3-5-sonnet-20241022",
    messages=[{"role": "user", "content": "Explain RAG in one paragraph"}]
)
```

지난 분기에 GPT-4, Claude Sonnet, Llama 3 70B 간에 동일한 워크로드(workload)로 비용 비교를 실행했습니다. 한 달에 847달러의 차이가 있었습니다. LiteLLM이 없었다면, 이 테스트를 수행하는 데 일주일간의 리팩토링(refactoring)이 필요했을 것입니다.

**대체한 것:** 3개의 개별 SDK(Software Development Kit) 통합 및 비용 추적 스프레드시트.

**솔직한 한계점:** 공급업체가 API를 변경하면 LiteLLM이 업데이트될 때까지 기다려야 합니다. 미션 크리티컬(mission-critical)한 프로덕션 환경에서는 일부 팀이 여전히 직접 SDK를 선호합니다.

### 2. Instructor: 실제로 작동하는 구조화된 출력

**문제:** LLM에게 JSON을 요청했는데, 마크다운(markdown) 백틱으로 감싸진 JSON을 받거나, 추가 필드가 있는 JSON, 혹은 JSON처럼 보이지만 후행 쉼표가 있는 문자열을 받습니다.

```python
import instructor
from pydantic import BaseModel, Field
from openai import OpenAI
client = instructor.from_openai(OpenAI())

class ToolRecommendation(BaseModel):
    name: str
    category: str = Field(description="e.g. 'orchestration', 'observability'")
    maturity: str = Field(description="'production-ready' or 'experimental'")
    best_for: list[str]
    avoid_when: list[str]

tools = client.chat.completions.create(
    model="gpt-4",
    messages=[{
        "role": "user",
        "content": "Recommend 3 Python tools for building RAG pipelines"
    }],
    response_model=list[ToolRecommendation]
)
# 유효성 검사를 거친 구조화된 객체 목록. 파싱(parsing) 불필요. 엣지 케이스 없음.
```

Instructor를 사용하기 전에는 12가지 엣지 케이스를 처리하는 `parse_llm_json()` 함수가 있었지만, 여전히 프로덕션 데이터에서는 일주일에 한 번 정도 실패했습니다. 위와 같은 중첩된 추출(복잡한 객체의 유효성 검사된 목록)은 Instructor가 진정으로 탁월한 부분입니다. 정규 표현식(regex)과 `json.loads()`로 이것을 시도해 보세요.

**대체한 것:** 수동 유효성 검사가 포함된 150줄의 JSON 파싱 함수.

**솔직한 한계점:** 지연 시간(latency)을 추가합니다. 모델의 첫 시도가 유효성 검사를 통과하지 못할 경우 Instructor는 재시도 주기(retry cycle)가 필요할 때가 있습니다. 높은 처리량(high-throughput) 파이프라인의 경우 오버헤드(overhead)를 측정해야 합니다.

### 3. MarkItDown: 모든 문서를 LLM 친화적인 텍스트로 변환

**문제:** RAG(Retrieval Augmented Generation) 파이프라인이 PDF, Word 문서, PowerPoint, Excel 파일을 수집해야 합니다. 그래서 PDF용으로는 pdfminer를, Word용으로는 python-docx를, 슬라이드용으로는 python-pptx를, 스프레드시트용으로는 pandas를 설치합니다. 네 가지 라이브러리. 네 가지 파싱 파이프라인. 네 가지 엣지 케이스 세트.

```python
from markitdown import MarkItDown

md = MarkItDown()

# PDF? 완료.
result = md.convert("quarterly_report.pdf")

# PowerPoint? 이것도 완료.
result = md.convert("client_deck.pptx")

# Excel? 어떤 식인지 아실 겁니다.
result = md.convert("budget_2025.xlsx")
print(result.text_content) # 매번 깔끔한 마크다운
```

마이크로소프트가 이 라이브러리를 만들었습니다(GitHub 스타 86K개). 이것은 제목, 표, 목록을 마크다운으로 보존합니다. 내부 지식 기반을 위한 문서 Q&A 시스템을 구축할 때, 데이터 수집 파이프라인이 전체 코드베이스의 40%를 차지했습니다. MarkItDown은 그것을 몇 줄로 줄일 수 있었을 것입니다.

**대체한 것:** 4개의 개별 파싱 라이브러리를 사용한 사용자 지정 추출 파이프라인.

**솔직한 한계점:** PDF 추출은 텍스트 레이어(text-layer)만 지원합니다. OCR(광학 문자 인식)이 없는 스캔된 이미지는 아무것도 반환하지 않습니다. PDF에서 표를 추출하는 것도 일관성이 없으므로, 금융 관련 또는 표가 많은 내용은 반드시 확인해야 합니다.

### 4. FastMCP: MCP 서버를 며칠이 아닌 몇 분 만에 구축

**문제:** Model Context Protocol (MCP)은 LLM이 외부 도구와 데이터에 접근할 수 있도록 하는 표준이 되고 있습니다. 그러나 처음부터 규격에 맞는 MCP 서버를 구축하는 것은 수백 줄의 전송 처리, 스키마(schema) 정의, 프로토콜 상용구(protocol boilerplate)를 의미합니다. 저는 MCP의 보안 모델에 대한 면밀한 조사가 필요한 이유에 대해 썼지만, 보안 문제를 차치하고서라도 구현 부담은 분명합니다.

```python
from fastmcp import FastMCP

mcp = FastMCP("My Data Tools")

@mcp.tool()
def search_database(query: str, limit: int = 10) -> list[dict]:
    """제품 데이터베이스를 검색합니다."""
    return db.search(query, limit=limit)

@mcp.tool()
def get_user_profile(user_id: str) -> dict:
    """ID로 사용자 프로필을 가져옵니다."""
    return db.get_user(user_id)

mcp.run()
```

이것은 완전하고 작동하는 MCP 서버입니다. Claude Desktop 또는 모든 MCP 클라이언트에 사용할 수 있는 두 가지 도구가 준비되어 있습니다. 데코레이터(decorator) 패턴은 FastAPI를 사용해 본 경험이 있다면 익숙하게 느껴질 것입니다(우연이 아닙니다). 저는 내부 검색 인덱스(internal search index)를 위한 도구 서버를 약 20분 만에 프로토타입으로 만들었습니다. 제가 시작했던 순수 MCP 버전은 200줄이었고 여전히 전송을 처리하지 못했습니다.

**대체한 것:** 200줄 이상의 프로토콜 상용구.

**솔직한 한계점:** FastMCP는 아직 초기 단계입니다(22K 스타를 기록하며 성장 중). 스택이 MCP를 지원하지 않는다면 도움이 되지 않습니다. 그리고 무엇을 노출할지 신중하게 생각해야 합니다.

### 5. PydanticAI: 프레임워크 부담 없는 타입 안전(Type-Safe) 에이전트

**문제:** LLM과 구조화되고 타입 안전한(type-safe) 상호작용을 원합니다. 하지만 전체 에이전트 프레임워크(agent framework)는 의존성(dependency) 과부하, 불투명한 추상화, 그리고 프레임워크 내부 깊숙한 세 개의 계층에서 무언가를 디버깅해야 할 때의 절망감을 수반합니다. (이러한 고통을 느껴본 적이 있다면, 당신은 혼자가 아닙니다.)

```python
from pydantic_ai import Agent
from pydantic import BaseModel

class CodeReview(BaseModel):
    issues: list[str]
    severity: str
    suggested_fix: str
    confidence: float
reviewer = Agent(
    "openai:gpt-4",
    result_type=CodeReview,
    system_prompt="You review Python code for bugs and anti-patterns."
)
result = reviewer.run_sync("def connect(url): return requests.get(url, verify=False)")
print(result.data.issues)
# ['SSL verification disabled', 'No timeout specified', 'No error handling']
```

PydanticAI는 순수 API 호출과 무거운 프레임워크 사이의 적절한 위치에 있습니다. 타입이 지정된 입력, 타입이 지정된 출력, 도구 정의, 의존성 주입(dependency injection)을 제공합니다. Pydantic 팀(OpenAI SDK, LangChain 등 거의 모든 것을 구동하는 유효성 검사 라이브러리를 만든 바로 그 사람들)이 만들었습니다. 저는 단일 Instructor 호출로는 너무 복잡하지만 완전한 LangGraph 워크플로가 필요하지 않은 모든 구조화된 작업에 PydanticAI를 사용하기 시작했습니다.

**대체한 것:** 타입이 지정되지 않은 프롬프트 체인(prompt chain)과 수동 응답 유효성 검사.

**솔직한 한계점:** PydanticAI의 "에이전트(agent)"는 "자율적인 다단계 추론(autonomous multi-step reasoning)"이라기보다는 "구조화된 LLM 상호작용"에 가깝습니다. 지속적인 상태를 가진 복잡한 다중 에이전트 워크플로(multi-agent workflow)의 경우 LangGraph나 CrewAI가 여전히 그 영역을 지배합니다.

### 6. tiktoken: 전송하기 전에 비용을 파악하세요

**문제:** 컨텍스트(context)를 연결하여 프롬프트를 만들고 있는데, 컨텍스트 창(context window)을 초과하거나 0.02달러로 충분할 API 호출에 2달러를 지출하게 될지 전혀 알 수 없습니다.

```python
import tiktoken

enc = tiktoken.encoding_for_model("gpt-4")
prompt = "Your very long prompt with lots of context..."
tokens = enc.encode(prompt)
print(f"Token count: {len(tokens)}")
# 호출 전에 비용 추정
input_cost = len(tokens) * 0.00003  # GPT-4 입력 가격
print(f"Estimated input cost: ${input_cost:.4f}")
```

재귀적인 컨텍스트 구성 함수가 45,000 토큰(token)의 프롬프트를 생성한 후 tiktoken을 사용하기 시작했습니다. 그 API 호출은 1.35달러였습니다. 단일 요청에 말이죠. 시간당 200번 쿼리되었습니다. 이제 동적 컨텍스트를 포함하는 모든 LLM 호출 전에 토큰 검사를 추가합니다.

**대체한 것:** 맹목적인 API 호출과 예상치 못한 청구서.

**솔직한 한계점:** tiktoken은 OpenAI 전용입니다. Anthropic의 경우 그들의 토큰 카운팅 API가 필요합니다. LiteLLM의 `token_counter()`는 여러 토크나이저(tokenizer)를 감싸지만 정확도는 다릅니다. 그리고 제가 위에서 한 것처럼 요금을 하드코딩하는 것은 언젠가 잘못된 정보를 제공할 지름길입니다.

### 7. Pydantic Evals: 프롬프트 "느낌적인" 테스트를 멈추세요

**문제:** 시스템 프롬프트를 변경했습니다. 더 나아졌을까요? 세 가지 예시를 수동으로 실행하고 괜찮아 보여서 배포합니다. 이틀 후, 사용자들이 테스트하지 않은 회귀(regression)에 대해 불평합니다.

```python
from pydantic_evals import Case, Dataset

dataset = Dataset(
    cases=[
        Case(
            name="basic_extraction",
            inputs="Extract the company name: Apple announced new MacBooks",
            expected_output="Apple",
        ),
        Case(
            name="ambiguous_input",
            inputs="The orange company released a new phone",
            expected_output="Orange",
        ),
    ]
)

report = dataset.evaluate(your_extraction_function)
report.print()
```

저는 몇 달 동안 평가 프레임워크(eval framework)에 저항했습니다. "제 프롬프트는 잘 작동하고, 수동으로 테스트합니다."라고 말했죠. 그러다가 한 가지 엣지 케이스를 개선하기 위해 시스템 프롬프트를 변경했는데, 제가 확인하지 않던 다른 세 가지 문제를 조용히 망가뜨렸습니다. Pydantic Evals는 제가 찾은 가장 간단한 평가 프레임워크입니다. 케이스를 정의하고, 실행하고, 통과/실패 보고서를 받으세요. 실제로 사용할 만큼 충분히 간단합니다 (이것이 핵심입니다).

**대체한 것:** "나한테는 괜찮아 보여" 식의 테스트와 배포 후의 놀라움.

**솔직한 한계점:** 새로 나왔고 API는 여전히 진화 중입니다. 대규모 프로덕션 평가 파이프라인의 경우 Braintrust 또는 Langfuse가 더 심층적인 통합을 제공합니다. 하지만 "이 변경 후에도 내 프롬프트가 여전히 작동하는가?"라는 질문에는 정확히 맞습니다.
