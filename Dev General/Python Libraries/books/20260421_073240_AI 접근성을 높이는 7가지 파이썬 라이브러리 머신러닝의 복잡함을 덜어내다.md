# AI 접근성을 높이는 7가지 파이썬 라이브러리: 머신러닝의 복잡함을 덜어내다

솔직히 말해두자면, AI가 어려운 것은 수학이 복잡해서가 아닙니다. AI가 어려운 것은 도구가 다루기 어렵기 때문입니다.

너무 많은 라이브러리가 사용자들을 박사 학위 연구자이거나 새벽 2시에 CUDA 오류를 디버깅하는 것을 즐기는 사람이라고 가정합니다. 대부분의 개발자는 그 중간 어딘가에 속합니다: 똑똑하고, 실용적이며, 참을성이 부족하죠.

그래서 흔히 듣는 "텐서플로(TensorFlow)나 파이토치(PyTorch)를 쓰고 행운을 빌어라"는 조언 대신, 저는 실제 AI 작업의 마찰을 조용히 제거해 주는 덜 알려진 7가지 파이썬 라이브러리를 소개하고자 합니다.

이것들은 과장된 도구들이 아닙니다. 어려운 방식으로 몇 주를 허비한 후에야 비로소 발견하게 될 그런 종류의 도구들이죠.

자, 시작해 봅시다.

## 1. River — 실제로 작동하는 온라인 머신러닝

대부분의 ML 라이브러리는 데이터가 정적이라고 가정합니다.

현실을 직시합시다: 데이터는 움직입니다. 로그는 계속 유입됩니다. 사용자 행동은 매일 달라집니다. 모델은 노후화됩니다.

River는 증분 학습(incremental learning), 즉 한 번에 하나의 데이터 포인트로 모델을 학습하도록 설계되었습니다.

재학습이 필요 없습니다. 메모리에 대규모 데이터셋을 올릴 필요도 없습니다.

```python
from river import linear_model, preprocessing, metrics

model = preprocessing.StandardScaler() | linear_model.LogisticRegression()
metric = metrics.Accuracy()

for x, y in data_stream:   # data arrives continuously
    y_pred = model.predict_one(x)
    metric.update(y, y_pred)
    model.learn_one(x, y)

print(metric)
```

**이 라이브러리가 중요한 이유:**

*   사기 탐지(fraud detection), 모니터링, IoT(사물 인터넷), 로그 처리에 완벽합니다.
*   메모리 효율적입니다 (운영 환경에 필수).
*   라즈베리 파이(Raspberry Pi)에 울지 않고 배포할 수 있습니다.

**사실:** 대부분의 운영 ML 시스템은 모델이 나빠서 실패하는 것이 아니라, 재학습 파이프라인이 고장 나서 실패합니다. River는 이를 완전히 방지합니다.

## 2. Skorch — 상투적인 코드(boilerplate) 없는 파이토치(PyTorch)

파이토치(PyTorch)는 강력합니다. 하지만 간단한 실험조차 박사 논문처럼 느껴질 정도로 코드가 장황합니다.

Skorch는 파이토치 모델을 사이킷런(scikit-learn) 인터페이스로 감싸줍니다.

동일한 성능, 절반의 코드, 불필요한 연결 논리(glue logic) 없음.

```python
from skorch import NeuralNetClassifier
import torch.nn as nn
class Net(nn.Module):
    def __init__(self):
        super().__init__()
        self.fc = nn.Linear(20, 2)

    def forward(self, X):
        return self.fc(X)

net = NeuralNetClassifier(
    Net,
    max_epochs=10,
    lr=0.01,
)

net.fit(X_train, y_train)
```

**이 라이브러리가 중요한 이유:**

*   GridSearchCV가 바로 작동합니다.
*   깔끔한 실험 파이프라인을 구축합니다.
*   빠른 반복 작업에 이상적입니다.

"파이토치는 좋지만, 학습 루프(training loops)를 일일이 관리하고 싶지 않아"라고 생각해 본 적이 있다면, 이 라이브러리가 바로 당신을 위한 것입니다.

## 3. Cleanlab — 모델이 아닌 데이터를 수정하세요

논쟁적 의견: 모델의 성능이 낮은 것이 아닙니다. 레이블(label)이 잘못된 것입니다.

Cleanlab은 통계적 방법을 사용하여 잘못 레이블링된 데이터(mislabeled data)를 자동으로 찾아냅니다.

휴리스틱(heuristics)이나 추측이 필요 없습니다.

```python
from cleanlab.filter import find_label_issues
import numpy as np

label_issues = find_label_issues(
    labels=y_train,
    pred_probs=predicted_probabilities
)

print("Suspicious samples:", label_issues[:10])
```

**이 라이브러리가 중요한 이유:**

*   실제 데이터셋은 지저분합니다.
*   레이블 노이즈(label noise)는 조용히 정확도를 저하시킵니다.
*   데이터를 정제하는 것이 레이어를 추가하는 것보다 효과적일 때가 많습니다.

**사실:** 5~10%의 레이블 오류율은 "잘못된" 알고리즘을 사용하는 것보다 모델 성능을 더 많이 저하시킬 수 있습니다. 대부분의 사람들은 이 부분을 간과하지만, 전문가들은 주의 깊게 살펴봅니다.

## 4. Hummingbird — ML 모델을 번개처럼 빠른 코드로 전환

모델을 학습시켰지만, 운영 환경에서는 느립니다.

Hummingbird는 기존의 ML 모델(사이킷런(sklearn), XGBoost)을 최적화된 텐서 기반 표현(tensor-based representations)으로 변환합니다.

생각해보세요: 모든 것을 다시 작성할 필요 없이 모델 가속화가 가능합니다.

```python
from hummingbird.ml import convert
import xgboost as xgb

model = xgb.XGBClassifier().fit(X, y)
torch_model = convert(model, 'torch')

torch_model.predict(X)
```

**이 라이브러리가 중요한 이유:**

*   더 빠른 추론(inference)
*   GPU / 엣지(edge) 환경에 더 쉬운 배포
*   재학습 불필요

이것은 평소에는 필요 없다가도, 갑자기 지연 시간(latency)이 중요해지는 순간 절실해지는 그런 도구입니다.

## 5. Evidently — 추측 없는 ML 모니터링

모델은 시끄럽게 실패하지 않습니다. 조용히 실패합니다.

Evidently는 다음을 탐지하는 데 도움을 줍니다:

*   데이터 드리프트(Data Drift)
*   콘셉트 드리프트(Concept Drift)
*   성능 저하

추측이 아닌 실제 통계를 기반으로 합니다.

```python
from evidently.report import Report
from evidently.metrics import DataDriftPreset

report = Report(metrics=[DataDriftPreset()])
report.run(reference_data=train_df, current_data=prod_df)
report.show()
```

**이 라이브러리가 중요한 이유:**

*   운영 ML은 캐글(Kaggle)이 아닙니다.
*   모니터링은 필수 불가결합니다.
*   이해관계자들은 시각적인 보고서를 좋아합니다.

**사실:** 구글(Google)에 따르면, 운영 환경에서 발생하는 대부분의 ML 사고는 코드 버그가 아닌 분포 변화(distribution shifts)로 인해 발생합니다. Evidently는 이러한 조기 경고를 제공합니다.

## 6. Optuna — 불법적인 느낌마저 드는 하이퍼파라미터 튜닝

그리드 서치(Grid Search)는 느리고, 랜덤 서치(Random Search)는 게으릅니다.

Optuna는 지능적인 샘플링을 사용하여 더 나은 하이퍼파라미터(hyperparameter)를 더 빠르게 찾아냅니다.

```python
import optuna

def objective(trial):
    lr = trial.suggest_float("lr", 1e-4, 1e-1, log=True)
    depth = trial.suggest_int("depth", 3, 10)
    return train_model(lr, depth)

study = optuna.create_study(direction="maximize")
study.optimize(objective, n_trials=50)

print(study.best_params)
```

**이 라이브러리가 중요한 이유:**

*   적은 실험으로 더 나은 결과
*   내장된 조기 종료(pruning) 기능 (성능이 좋지 않은 시도를 일찍 중단)
*   노트북부터 클러스터까지 확장 가능

Optuna를 한 번 사용하고 나면, 그리드 서치는 팩스 기기를 사용하는 것처럼 느껴질 것입니다.

## 7. Haystack — 구글을 재창조하지 않고도 AI 검색 시스템 구축

다음을 구축해 본 적이 있다면:

*   시맨틱 검색(Semantic Search)
*   질의응답(Question Answering)
*   검색 증강 생성(Retrieval-Augmented Generation, RAG)

...얼마나 빠르게 상황이 복잡해지는지 알 것입니다.

Haystack은 완전한 파이프라인을 제공합니다: 문서 → 임베딩(embeddings) → 검색(retrieval) → 추론(inference).

```python
from haystack.document_stores import InMemoryDocumentStore
from haystack.nodes import BM25Retriever

store = InMemoryDocumentStore()
store.write_documents([
    {"content": "Python makes AI accessible."}
])

retriever = BM25Retriever(document_store=store)
docs = retriever.retrieve("What makes AI easy?")

print(docs[0].content)
```

**이 라이브러리가 중요한 이유:**

*   내부 도구에 완벽합니다.
*   LLM(대규모 언어 모델)과 잘 통합됩니다.
*   운영 환경 지향적인 설계

이것이 실제 AI 시스템이 구축되는 방식이며, 장난감 같은 노트북(notebooks)으로 만들어지는 것이 아닙니다.
