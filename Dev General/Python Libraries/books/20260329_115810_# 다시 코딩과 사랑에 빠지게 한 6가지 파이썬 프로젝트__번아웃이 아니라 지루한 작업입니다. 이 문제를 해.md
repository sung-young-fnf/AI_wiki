## 1. 확정적 작업 스케줄러(Deterministic Task Scheduler) 구축 (Cron 없이)

시간을 이해하는 것은 엄청난 능력(superpower)이기 때문입니다.

대부분의 개발자는 스케줄링(scheduling)을 "Cron(크론)이 처리해 줄 것"이라고 생각합니다. 새벽 3시에 Cron이 조용히 실패할 때까지는 말이죠.

그래서 저는 실행 순서, 재시도, 그리고 충돌 후 상태 복구를 보장하는 저만의 확정적 스케줄러를 만들었습니다.

**흥미로운 이유**
*   시간 편차(time drift)에 대해 생각하게 됩니다.
*   "5분마다 실행"이 왜 모호한지 배우게 됩니다.
*   메모리 착각(memory illusions)이 아닌 영속성(persistence)을 다루게 됩니다.

**핵심 아이디어**
간격(intervals) 대신 다음 실행 타임스탬프(timestamps)를 저장합니다.

```python
import time
import json
from datetime import datetime, timedelta

TASK_FILE = "tasks.json"

def load_tasks():
    try:
        with open(TASK_FILE) as f:
            return json.load(f)
    except FileNotFoundError:
        return {}

def save_tasks(tasks):
    with open(TASK_FILE, "w") as f:
        json.dump(tasks, f, indent=2)

def should_run(task):
    return datetime.utcnow() >= datetime.fromisoformat(task["next_run"])

def run_task(task):
    print(f"Running: {task['name']} at {datetime.utcnow().isoformat()}")
    task["next_run"] = (datetime.utcnow() + timedelta(seconds=task["interval"])).isoformat()

def scheduler_loop():
    tasks = load_tasks()
    while True:
        for task in tasks.values():
            if should_run(task):
                run_task(task)
        save_tasks(tasks)
        time.sleep(1)

scheduler_loop()
```

**이 프로젝트를 통해 배운 것:**
Cron은 시간을 스케줄링합니다.
프로덕션 시스템(Production systems)은 보장(guarantees)을 스케줄링합니다.

## 2. Grep 재구축 — 하지만 올바르게 만들기

정규 표현식(Regex)은 쉽습니다. 올바른 텍스트 처리는 그렇지 않습니다.

모두가 "그냥 정규 표현식을 사용해"라고 말합니다. 그렇게 하면 몇 년 동안 살아남는 버그가 생기게 됩니다.

저는 다음과 같은 기능을 하는 단순화된 Grep(그렙)을 재구축했습니다:

*   파일을 한 줄씩 스트리밍(streams)합니다.
*   인코딩(encodings)을 유연하게 처리합니다.
*   여러 기가바이트(GB) 로그에서도 메모리가 폭주하지 않습니다.

```python
import re

def grep(pattern, filepath, ignore_case=False):
    flags = re.IGNORECASE if ignore_case else 0
    regex = re.compile(pattern, flags)

    with open(filepath, "rb") as f:
        for line_no, raw in enumerate(f, 1):
            try:
                line = raw.decode("utf-8")
            except UnicodeDecodeError:
                continue
            if regex.search(line):
                yield line_no, line.rstrip()
```

이렇게 사용하세요:

```python
for ln, text in grep("ERROR", "server.log"):
    print(f"{ln}: {text}")
```

**이 프로젝트를 통해 배운 것:**
텍스트 처리는 문자열(string) 문제가 아닌 시스템 문제입니다.

## 3. 자체 코드용 메모리 누수 탐지기 작성

메모리를 이해하지 못하면, 파이썬은 정중하게 거짓말을 할 것입니다.

파이썬은 메모리를 너무 잘 숨겨서, 대부분의 개발자는 서버가 재시작될 때까지 누수(leaks)를 전혀 알아채지 못합니다.

저는 `tracemalloc`을 사용하여 간단한 할당 변화 추적기(allocation delta tracker)를 작성했습니다.

```python
import tracemalloc
import time

tracemalloc.start()

def snapshot(label):
    current, peak = tracemalloc.get_traced_memory()
    print(f"{label} | Current: {current / 1024:.2f}KB | Peak: {peak / 1024:.2f}KB")

snapshot("Start")

leaky = []
for _ in range(10_000):
    leaky.append("x" * 10_000)

snapshot("After allocations")
```

그다음 이 코드를 실제 서비스에 적용했습니다.

**저를 놀라게 한 점:**
대부분의 "메모리 누수(memory leaks)"는 실제 누수가 아니라 잊힌 참조(forgotten references)입니다.

파이썬은 잊지 않습니다.
그저 기다릴 뿐입니다.

## 4. 이진 로그 형식(Binary Log Format) 구현 (JSON은 느리기 때문)

가독성(Readable) ≠ 효율성(efficient).

JSON(제이슨)은 사람에게는 훌륭합니다. 하지만 기계는 더 나은 것을 받을 자격이 있습니다.

그래서 저는 고빈도 이벤트를 위한 이진(binary) 형식의 추가 전용 로그(append-only log)를 설계했습니다.

```python
import struct
import time

LOG_FORMAT = ">dI"  # timestamp (double), payload size (uint)

def write_event(file, payload: bytes):
    header = struct.pack(LOG_FORMAT, time.time(), len(payload))
    file.write(header)
    file.write(payload)

def read_events(file):
    while True:
        header = file.read(struct.calcsize(LOG_FORMAT))
        if not header:
            break
        ts, size = struct.unpack(LOG_FORMAT, header)
        payload = file.read(size)
        yield ts, payload
```

**이것이 중요한 이유**
*   엔디언(endianness)을 배우게 됩니다.
*   프레이밍(framing)을 배우게 됩니다.
*   JSON을 본래 목적이 아닌 용도로 오용하는 것을 멈추게 됩니다.

**사실:**
이진 형식(Binary formats)은 구조화된 데이터(structured data)의 경우 JSON보다 5~10배 더 작습니다.

## 5. 교착 상태(Deadlock) 시뮬레이터 구축

동시성(Concurrency)은 요란하게 고장 나지 않습니다. 그저 정중하게 멈출 뿐입니다.

대부분의 사람은 의도적으로 교착 상태(deadlock)가 발생하는 것을 보지 않고도 "스레딩(threading)을 배웁니다."

그래서 저는 교착 상태를 보장하는 시뮬레이터를 작성했습니다.

```python
import threading
import time

lock_a = threading.Lock()
lock_b = threading.Lock()

def task_one():
    with lock_a:
        time.sleep(0.1)
        with lock_b:
            print("Task one done")

def task_two():
    with lock_b:
        time.sleep(0.1)
        with lock_a:
            print("Task two done")

threading.Thread(target=task_one).start()
threading.Thread(target=task_two).start()
```

이제 이 문제를 해결해 보세요.

그 단 하나의 연습만으로 저는 어떤 튜토리얼보다 더 많은 것을 동시성에 대해 배우게 되었습니다.

**이 프로젝트를 통해 배운 것:**
동시성 버그(Concurrency bugs)는 테스트를 통과하지 못하는 것이 아닙니다.
신뢰를 저버리는 것입니다.

## 6. 코드 고고학 도구(Code Archaeology Tool) 구축

모든 성숙한 코드베이스(codebase)는 범죄 현장입니다.

저는 다음과 같은 질문에 답하는 도구를 만들었습니다:

*   어떤 파일이 가장 많이 변경될까요?
*   어떤 함수가 버그를 유발할까요?
*   기술 부채(technical debt)는 실제로 어디에 있을까요?

오직 Git(깃)과 파이썬만을 사용했습니다.

```python
import subprocess
from collections import Counter

def churn():
    output = subprocess.check_output(
        ["git", "log", "--name-only", "--pretty=format:"],
        text=True
    )

    files = [line for line in output.splitlines() if line]
    return Counter(files).most_common(10)

for file, count in churn():
    print(f"{file}: {count} changes")
```

**이것이 강력한 이유**
*   리팩토링(refactors)이 중요한 곳을 추측하는 것을 멈추게 됩니다.
*   데이터에 기반한 직관을 얻게 됩니다.
*   의견이 아닌 역사를 읽는 법을 배우게 됩니다.
