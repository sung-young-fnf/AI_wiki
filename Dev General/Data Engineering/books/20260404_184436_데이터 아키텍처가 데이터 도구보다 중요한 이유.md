# 데이터 아키텍처가 데이터 도구보다 중요한 이유

데이터 엔지니어링 분야에서 6개월이라는 시간을 낭비하는 가장 쉬운 방법은 아키텍처의 문제를 도구 선택으로 해결하려 하는 것입니다.

현업에서는 팀이 인정하는 것보다 훨씬 자주 이런 일이 발생합니다. 파이프라인(Pipeline) 관리가 어려워지면 새로운 오케스트레이터(Orchestrator)를 찾고, 쿼리 속도가 느려지면 웨어하우스(Warehouse)를 이전하며, 대시보드의 수치가 일치하지 않으면 또 다른 변환 계층(Transformation layer)을 추가하곤 합니다.

때로는 이러한 결정이 도움이 될 수도 있습니다. 하지만 근본적인 문제는 여전히 해결되지 않은 채로 남습니다. 시스템이 명확한 모델, 소유권, 경계, 또는 데이터 진화 방식에 대한 규칙 없이 설계되었기 때문입니다. 도구 구성(Stack)을 바꾸더라도 혼란은 계속됩니다.

이것이 바로 데이터 아키텍처가 데이터 도구보다 중요한 이유입니다. 도구는 속도와 개발자 경험을 개선할 수 있지만, 아키텍처는 실제 복잡성이 닥쳤을 때 시스템이 이해 가능하고 신뢰할 수 있으며 확장 가능한 상태를 유지할지를 결정합니다. 다시 말해, 도구는 더 빨리 구축하도록 돕지만, 아키텍처는 구축한 결과물이 무너지지 않고 버틸 수 있는지를 결정합니다.

### 데이터 도구로 해결할 수 없는 5가지 아키텍처 문제

많은 데이터 팀이 도구를 업그레이드하고 파이프라인을 현대화하며 새로운 플랫폼을 도입함에도 불구하고 결국 같은 패턴의 문제에 직면합니다. 문제는 처음부터 아키텍처에 있었기 때문입니다. 데이터 시스템을 서서히 약화시키는 5가지 흔한 실수는 다음과 같습니다.

#### 1. 시스템 설계가 아닌 파이프라인 구축에만 집중하는 것
대부분의 데이터 플랫폼은 작게 시작합니다. 매출 데이터를 로드하는 파이프라인, 고객 기록을 변환하는 파이프라인 등이 하나씩 늘어납니다. 처음에는 관리 가능해 보이지만, 시간이 흐르면서 파이프라인은 기하급수적으로 늘어납니다. 여러 팀이 각자의 변환 로직을 만들면서 정의는 어긋나고 로직은 수십 개의 워크플로우(Workflow)로 흩어집니다.

결국 조직 내에 '매출', '활성 사용자', '고객 생애 가치(LTV)'에 대한 여러 버전이 존재하게 됩니다. 이는 파이프라인 코드의 문제가 아니라 시스템 수준 설계의 부재 때문입니다. 건강한 데이터 플랫폼은 파이프라인을 구축하기 전에 공유 모델과 명확한 계층을 정의해야 합니다. 로우 데이터(Raw data), 정제된 데이터셋, 비즈니스용 모델은 각각 고유한 목적을 가져야 하며, 파이프라인은 아키텍처를 구현하는 수단이어야지 아키텍처를 임의로 만들어내서는 안 됩니다.

#### 2. 도구가 데이터 모델을 결정하게 두는 것
사용 중인 도구에 맞춰 데이터 모델을 설계하는 것도 흔한 실수입니다. 특정 웨어하우스나 레이크하우스(Lakehouse)가 권장하는 패턴에 맞추다 보면, 스키마(Schema)가 비즈니스 로직이 아닌 기술적 특성에 종속됩니다.

이는 초기에는 효율적으로 보일 수 있지만, 플랫폼을 변경해야 할 때 큰 고통으로 다가옵니다. 강력한 아키텍처는 비즈니스를 대변해야 하며, 고객, 제품, 주문 등의 데이터 모델은 스노우플레이크(Snowflake), 빅쿼리(BigQuery), 델타 레이크(Delta Lake) 등 어떤 환경에서도 독립적으로 존재할 수 있어야 합니다. 도구는 모델을 구현할 뿐, 모델을 정의해서는 안 됩니다.

#### 3. 데이터셋을 내부 파일처럼 취급하는 것
많은 파이프라인이 단순히 단계별로 데이터를 전달하는 내부 스크립트처럼 동작합니다. 한 작업이 테이블을 쓰고, 다른 작업이 이를 읽고, 또 다른 작업이 수정하지만 명확한 소유자는 없습니다.

이 방식은 여러 팀이 동일한 데이터에 의존하기 시작하면 무너집니다. 상류(Upstream)의 작은 변화가 하류(Downstream)의 대시보드나 머신러닝 기능을 망가뜨리기 때문입니다. 현대적인 데이터 아키텍처는 데이터셋을 하나의 '제품(Product)'으로 취급합니다. 각 데이터셋에는 소유자가 있고, 스키마가 문서화되며, 변경 사항이 공유됩니다. 이를 **데이터 계약(Data Contracts)**이라고 합니다. 생산자와 소비자가 데이터의 형태에 합의할 때 시스템은 비로소 안정화됩니다.

#### 4. 모든 데이터 의사결정을 중앙 집중화하는 것
성장하는 회사는 모든 요청이 단일 데이터 팀이라는 병목 구간을 통과하게 됩니다. 새로운 데이터셋은 백로그(Backlog)에서 대기하고, 스키마 변경에는 몇 주가 소요됩니다.

확장 가능한 접근 방식은 소유권을 분산하는 것입니다. 도메인 팀이 자신의 비즈니스 프로세스와 가장 가까운 데이터셋을 소유하고, 플랫폼 팀은 공유 인프라와 표준을 제공해야 합니다. 아키텍처는 일관성을 유지하면서도 각 팀의 자율성을 보장해야 합니다.

#### 5. 데이터 품질을 단순히 모니터링의 문제로 보는 것
데이터 품질 문제가 발생하면 흔히 알림과 모니터링을 추가합니다. 하지만 모니터링은 원인이 아닌 증상을 치료할 뿐입니다. 많은 데이터 품질 문제는 불분명한 데이터 정의, 파편화된 변환 로직, 정의되지 않은 소유권 등 아키텍처적 요인에서 비롯됩니다.

아키텍처가 탄탄하면 품질 유지는 쉬워집니다. 수집 단계에서 검증이 이루어지고, 변환은 예측 가능한 패턴을 따르며, 데이터 계보(Lineage) 추적이 용이해집니다. 데이터 품질은 단순히 도구의 기능이 아니라 잘 설계된 아키텍처의 결과물입니다.

### 도구는 중요하지만 기초는 아닙니다

훌륭한 도구는 개발자의 생산성을 높이고 운영을 단순화하며 시스템 구축의 마찰을 줄여줍니다. 하지만 도구는 기초가 아닙니다. 도구를 건설 장비라고 한다면 아키텍처는 설계도(Blueprint)와 같습니다. 성능 좋은 기중기가 설계가 잘못된 건물을 바로잡을 수는 없지만, 훌륭한 설계도는 장비가 바뀌어도 살아남습니다.

데이터 엔지니어링의 미래는 아키텍처에 달려 있습니다. 도구에만 집중하는 엔지니어는 끊임없이 다음 플랫폼을 배우는 데 급급하겠지만, 아키텍처를 이해하는 엔지니어는 전체 데이터 시스템의 작동 방식을 설계하게 될 것입니다. 장기적으로 기업이 진정으로 필요로 하는 사람은 바로 후자입니다.
