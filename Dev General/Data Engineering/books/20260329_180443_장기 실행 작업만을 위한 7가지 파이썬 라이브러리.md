# 장기 실행 작업만을 위한 7가지 파이썬 라이브러리
며칠, 몇 주, 그리고 끝없이 이어지는 작업을 위한 도구들

파이썬(Python) 스크립트가 5초 이내에 끝난다면, 이 글은 당신을 위한 것이 아닙니다.

이 글은 작업을 너무 오래 실행하여 왜 시작했는지조차 잊어버릴 정도로 긴 작업들을 위한 것입니다. 노트북 절전 모드, 커피 리필, 사무실 Wi-Fi 고장, 그리고 가끔 찾아오는 존재론적 위기 속에서도 살아남는 작업들 말입니다.

제가 말하는 장기 실행 작업(long-running jobs)이란 크롤러(crawlers), 데이터 파이프라인(data pipelines), 백필(backfills), 모델 학습(model training), ETL 괴물(ETL monsters), 그리고 "그냥 밤새 돌려놓을게"라는 거짓말 같은 작업들을 의미합니다.

4년 이상 프로덕션(production) 환경에서 파이썬을 작성한 후 깨달은, 대부분의 튜토리얼이 간과하는 진실이 있습니다.

장기 실행 작업은 다르게 실패합니다.
그리고 이러한 작업에는 다른 도구가 필요합니다.

아래에는 멈추기를 거부하는 작업들을 위해 특별히 제작된 7가지 파이썬 라이브러리가 있습니다. 일부는 잘 알려지지 않았고, 일부는 터무니없이 저평가되어 있습니다. 이 모든 라이브러리는 새벽 3시에 문제가 발생한 후에야 비로소 깨닫게 되는 문제들을 해결해 줍니다.

그럼 시작해 볼까요.

### 1. tenacity — 재시도는 `while` 루프가 아닌 과학입니다.

대부분의 사람들은 이렇게 재시도합니다.

```python
while True:
    try:
        do_the_thing()
        break
    except:
        time.sleep(5)
```

이것은 작동합니다… 작동하지 않을 때까지는요.

`tenacity`는 재시도에 다음과 같은 규칙이 필요하기 때문에 존재합니다.

*   지수적 백오프(exponential backoff)
*   지터(jitter)
*   최대 시도 횟수(max attempts)
*   특정 예외(exception) 발생 시에만 재시도

**장기 실행 작업에 완벽한 이유**

작업이 며칠 동안 실행되면, 일시적인 오류(transient failures)는 필연적으로 발생합니다. API는 타임아웃되고, 네트워크는 깜빡이며, 데이터베이스는 재채기를 합니다(즉, 예상치 못한 작은 문제가 발생합니다).

`tenacity`는 재시도를 예측 가능하고 측정 가능하게 만듭니다.

**예시: 지수적 백오프(exponential backoff)와 최대 시간(max time)을 사용한 재시도**
```python
from tenacity import retry, stop_after_delay, wait_exponential, retry_if_exception_type
import requests

@retry(
    wait=wait_exponential(multiplier=1, min=2, max=60),
    stop=stop_after_delay(3600),
    retry=retry_if_exception_type(requests.exceptions.RequestException),
)
def fetch_data(url):
    response = requests.get(url, timeout=10)
    response.raise_for_status()
    return response.json()

data = fetch_data("https://api.example.com/huge-endpoint")
```

이는 최대 한 시간 동안 지능적으로(intelligently) 백오프(backoff)하며 재시도할 것입니다.

이것은 단순한 루프(loop)가 아닙니다.
이것은 운영 규율(operational discipline)입니다.

### 2. apscheduler — 당신을 싫어하지 않는 크론 작업(Cron Jobs)

시스템 크론(system cron)은 훌륭합니다… 다음의 경우가 아니라면요.

*   상태 관리(state)가 필요할 때
*   지속성(persistence)이 필요할 때
*   작업을 동적으로 일시 중지(pause), 재개(resume) 또는 재스케줄링(reschedule)해야 할 때

`apscheduler`는 장기 실행 프로세스(long-running process) 내에서 작동하는 파이썬 네이티브(Python-native) 스케줄링을 위한 것입니다.

**끝없이 이어지는 프로세스에 탁월한 이유**

*   영구적인 작업 저장소(persistent job stores) (SQLite, PostgreSQL, Redis)
*   재시작 후에도 유지
*   실행되어야 할 작업에 대한 미스파이어(misfire) 처리

**예시: 스케줄된 작업을 포함하는 장기 실행 서비스**
```python
from apscheduler.schedulers.blocking import BlockingScheduler
from datetime import datetime

scheduler = BlockingScheduler()

def nightly_job():
    print("Running heavy nightly task at", datetime.utcnow())

scheduler.add_job(
    nightly_job,
    trigger="cron",
    hour=2,
    misfire_grace_time=3600
)

scheduler.start()
```

서비스가 새벽 2시 5분에 재시작하더라도,
해당 작업은 여전히 실행됩니다.

크론은 그렇게 하지 않습니다.
`apscheduler`는 가능합니다.

### 3. tqdm (고급 사용법) — 며칠이 걸리는 작업을 위한 진행률 바(Progress Bars)

모두가 `tqdm`을 알고 있습니다. 하지만 장기 실행 작업을 위해 올바르게 사용하는 사람은 거의 없습니다.

비밀은 이렇습니다: `tqdm`은 단순히 진행률 바가 아니라 모니터링 도구(monitoring tool)입니다.

**장기 실행 작업에 중요한 이유**

작업이 48시간 동안 실행될 때, 가시성(visibility)은 생존 그 자체입니다.

`tqdm`이 제공하는 것:

*   처리량(throughput) (초당 항목 수)
*   실제로 수렴하는 예상 완료 시간(ETA)
*   로그로 남길 수 있는 실시간 지표(live metrics)

**예시: 재시작에도 유지되는 진행률**
```python
from tqdm import tqdm
import time

total_items = 10_000
with tqdm(total=total_items, unit="records") as pbar:
    for i in range(total_items):
        process_item(i)
        time.sleep(0.01)
        pbar.update(1)
```

이제 이것을 로깅(logging)과 함께 사용해 보세요.

```python
pbar.set_postfix(memory="512MB", errors=3)
```

이것은 거저 얻는 관측 가능성(observability)입니다.

### 4. persist-queue — 메모리가 거짓말을 할 때

장기 실행 작업은 중단됩니다. 메모리(memory)는 휘발성(volatile)입니다. 당신의 큐(queue)는 그래서는 안 됩니다.

`persist-queue`는 충돌, 재시작, 정전 후에도 살아남는 디스크 기반 큐(disk-backed queues)를 제공합니다.

**장기 실행 작업에 필수적인 이유**

작업이 수백만 개의 항목을 처리한다면:

*   충돌할 것입니다.
*   재시작할 것입니다.
*   처음부터 다시 시작해서는 안 됩니다.

**예시: 내구성 있는 작업 큐(durable task queue)**
```python
from persistqueue import Queue

q = Queue("task_queue")

# Producer (생산자)
for i in range(100_000):
    q.put(i)

# Consumer (소비자)
while not q.empty():
    item = q.get()
    process(item)
    q.task_done()
```

프로세스를 종료하고 다시 시작하세요.
큐는 여전히 남아 있습니다.

이 라이브러리 덕분에 몇 주간의 재계산 시간을 절약했습니다.

### 5. watchdog — 실행 중에도 세상에 반응하기

장기 실행 작업은 종종 외부 이벤트를 기다립니다.

*   새 파일
*   업데이트된 설정(configs)
*   수신된 시그널(signals)

폴링(polling)은 비효율적입니다. `watchdog`는 당신의 작업이 반응하도록 합니다.

**저평가되는 이유**

수동적인 스크립트(script)를 이벤트 기반 시스템(event-driven system)으로 전환합니다.

**예시: 파일이 도착하는 즉시 처리하기**
```python
from watchdog.observers import Observer
from watchdog.events import FileSystemEventHandler
import time

class Handler(FileSystemEventHandler):
    def on_created(self, event):
        if not event.is_directory:
            print("New file detected:", event.src_path)
            process_file(event.src_path)

observer = Observer()
observer.schedule(Handler(), path="./incoming", recursive=False)
observer.start()

try:
    while True:
        time.sleep(1)
except KeyboardInterrupt:
    observer.stop()
observer.join()
```

이것은 영원히 — 효율적으로 실행됩니다.

폴링 없음.
낭비되는 CPU 없음.
놓치는 파일 없음.

### 6. py-spy — 작업을 멈추지 않고 디버깅하기

이 라이브러리는 터무니없이 적게 사용됩니다.

`py-spy`는 코드를 수정하거나 재시작하지 않고도 실행 중인 파이썬 프로세스(Python process)를 검사할 수 있게 해줍니다.

**장기 실행 작업에 이것이 중요한 이유**

작업이 3일 동안 실행되고 있다면:

*   재시작할 수 없습니다.
*   지금 무엇을 하고 있는지 이해해야 합니다.

**예시: 라이브 프로세스에 연결하기**
```bash
py-spy top --pid 12345
```

또는 플레임 그래프(flame graph)를 생성합니다.

```bash
py-spy record -o profile.svg --pid 12345
```

당신은 프로덕션 환경을 건드리지 않고…
프로덕션 환경을 디버그(debug)했습니다.

그것이 바로 힘입니다.

### 7. ray — "장기 실행"이 "분산"과 만날 때

어떤 작업은 단순히 오래 실행되는 것을 넘어섭니다. 그것들은 거대하게 실행됩니다.

`ray`는 다음을 위해 만들어졌습니다.

*   분산 실행(distributed execution)
*   내결함성(fault tolerance)
*   재개 가능한 작업 부하(resumable workloads)

**여기에 속하는 이유**

`Ray`는 다음과 같은 작업 부하(workloads)를 위해 설계되었습니다.

*   몇 주 동안 실행됨
*   여러 머신(machines)에 걸쳐 실행됨
*   워커(worker) 실패 후에도 유지됨

**예시: 분산된 장기 실행 태스크(tasks)**
```python
import ray
import time

ray.init()

@ray.remote
def long_task(x):
    time.sleep(10)
    return x * x

results = ray.get([long_task.remote(i) for i in range(10)])
print(results)
```

워커가 중단되면 `Ray`는 재스케줄링합니다.
당신의 작업은 계속 진행됩니다.

그것은 편의성이 아닙니다.
그것은 엔지니어링(engineering)입니다.
