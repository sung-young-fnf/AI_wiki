## 이 비교가 더 중요한 이유

### Polars는 실제로 SQL과 비교할 만합니다

pandas 대(vs) SQL 비교가 공정하지 않았던 이유는 다음과 같습니다.

*   pandas는 단일 스레드(single-threaded) 방식입니다 (SQL은 대규모 병렬(massively parallel) 방식입니다).
*   pandas는 모든 데이터를 RAM에 로드합니다 (SQL은 웨어하우스(warehouse)에서 처리합니다).
*   pandas에는 쿼리 옵티마이저(query optimizer)가 없습니다 (SQL에는 정교한 최적화 기능이 있습니다).

Polars 대(vs) SQL 비교가 공정한 이유는 다음과 같습니다.

*   Polars는 멀티 스레드(multi-threaded) 방식입니다 (SQL의 병렬 처리와 유사합니다).
*   Polars는 RAM보다 큰 데이터셋(dataset)을 스트리밍(streaming) 방식으로 처리할 수 있습니다 (SQL과 유사합니다).
*   Polars에는 쿼리 옵티마이저(query optimizer)가 있습니다 (예: 조건자 푸시다운(predicate pushdown), 프로젝션 최적화(projection optimization)).
*   Polars는 컬럼형 아파치 애로우(columnar Apache Arrow) 형식을 사용합니다 (현대적인 웨어하우스(warehouse)와 동일합니다).

즉, 이제서야 공정한 비교가 가능해졌습니다.

## 테스트 #1: 단순 집계 (Simple Aggregation)

작업: 제품 카테고리별 총매출 계산

### SQL 구현 (동일)

```sql
SELECT
    product_category,
    COUNT(DISTINCT order_id) AS order_count,
    SUM(order_total) AS total_revenue,
    AVG(order_total) AS avg_order_value
FROM orders
GROUP BY product_category
ORDER BY total_revenue DESC;
```

### Polars 구현

```python
import polars as pl

# Lazy scan with streaming
result = (
    pl.scan_parquet('s3://bucket/orders.parquet')
    .group_by('product_category')
    .agg([
        pl.col('order_id').n_unique().alias('order_count'),
        pl.col('order_total').sum().alias('total_revenue'),
        pl.col('order_total').mean().alias('avg_order_value')
    ])
    .sort('total_revenue', descending=True)
    .collect(streaming=True)  # Execute with streaming
)
```

코드 줄 수: SQL: 10줄, Polars: 12줄

### 성능 결과

(성능 결과는 원문 이미지 참조)

메모리 사용량:

*   SQL: 일정함 (웨어하우스(warehouse)에서 처리)
*   Polars: 최대 ~2GB (스트리밍(streaming)으로 낮게 유지)

### 주요 통찰

Polars가 모든 데이터 크기에서 승리했습니다! 그 이유는 무엇일까요?

*   ✅ 멀티 스레드(multi-threaded) 집계 (8 코어 대(vs) Snowflake의 4 코어)
*   ✅ 스트리밍 실행 (청크(chunk) 단위 처리, 낮은 메모리 사용량)
*   ✅ 쿼리 최적화 (필요한 컬럼(column)만 읽음)
*   ✅ 네트워크 오버헤드(overhead) 없음 (데이터가 직접 흐름)

SQL이 여전히 유효한 경우:

*   데이터가 이미 웨어하우스(warehouse)에 있는 경우 (데이터 전송 불필요)
*   팀 전체에서 쿼리(query)를 공유하는 경우 (BI 도구, 대시보드(dashboard))
*   거버넌스(governance)가 필요한 경우 (감사 로그(audit log), 접근 제어)

```python
result = (
    pl.scan_parquet('s3://bucket/customers.parquet')
    .with_columns([

          .when(pl.col('email').str.contains('@yahoo.'))
          .then(pl.lit('yahoo'))
          .when(pl.col('email').str.contains('@outlook.|@hotmail.'))
          .then(pl.lit('microsoft'))
          .otherwise(pl.lit('other'))
          .alias('email_provider'),

        # Validate
        pl.col('email').str.contains(
            r'^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$'
        ).alias('is_valid_email')
    ])
    .collect(streaming=True)
)
```

## 테스트 #3: 윈도우 함수 및 랭킹 (Window Functions & Ranking)

작업: 누계(running totals), 순위(ranks), 지연/선행(lag/lead) 연산

### Polars 구현

```python
result = (
    pl.scan_parquet('s3://bucket/orders.parquet')
    .filter(pl.col('order_date') >= '2024-01-01')
    .sort(['customer_id', 'order_date'])
    .with_columns([
        # Running total
        pl.col('order_total')
          .cum_sum()
          .over('customer_id')
          .alias('running_total'),

        # Daily rank
        pl.col('order_total')
          .rank(method='min', descending=True)
          .over('order_date')
          .alias('daily_rank'),

        # Previous order
        pl.col('order_total')
          .shift(1)
          .over('customer_id')
          .alias('previous_order_total'),

        # Order number
        pl.col('order_id')
          .cum_count()
          .over('customer_id')
          .alias('order_number')
    ])
    .collect(streaming=True)
)
```

### 성능 결과

(성능 결과는 원문 이미지 참조)

### 주요 통찰

대규모 데이터에서는 성능이 수렴합니다:

*   Polars는 소형-중형 데이터셋(dataset)에서 우세합니다.
*   SQL은 1,000만 행 이상에서 따라잡습니다.

## 테스트 #4: 복잡한 조인 (Complex Joins)

작업: 필터링(filtering)을 포함한 4개 테이블 조인(join)

### Polars 구현

```python
# Lazy scan all tables
orders = pl.scan_parquet('s3://bucket/orders.parquet')
customers = pl.scan_parquet('s3://bucket/customers.parquet')
products = pl.scan_parquet('s3://bucket/products.parquet')
categories = pl.scan_parquet('s3://bucket/categories.parquet')

result = (
    orders
    .filter(pl.col('order_date') >= '2024-01-01')
    .join(
        customers.filter(pl.col('customer_segment').is_in(['Enterprise', 'SMB'])),
        on='customer_id'
    )
    .join(products, on='product_id')
    .join(
        categories.filter(pl.col('category_name') != 'Returns'),
        on='category_id'
    )
    .with_columns([
        (pl.col('quantity') * pl.col('unit_price')).alias('line_total')
    ])
    .select([
        'order_id', 'order_date', 'customer_name',
        'customer_segment', 'product_name', 'category_name', 'line_total'
    ])
    .collect(streaming=True)
)
```

## 테스트 #5: 실제 시나리오 (The Real-World Scenario)

작업: 복잡한 비즈니스 로직을 사용한 고객 LTV(생애 가치) 계산

여기부터 정말 흥미로워집니다.

### Polars 구현 (SQL보다 깔끔함!)

```python
import polars as pl
import polars.selectors as cs

# Calculate RFM scores and LTV
result = (
    pl.scan_parquet('s3://bucket/orders.parquet')
    .group_by('customer_id')
    .agg([
        pl.col('order_id').n_unique().alias('order_count'),
        pl.col('order_total').sum().alias('total_spent'),
        pl.col('order_date').max().alias('last_order_date')
    ])
    .with_columns([
        # Days since last order
        (pl.lit(datetime.now()) - pl.col('last_order_date'))
        .dt.total_days()
        .alias('days_since_last_order'),

        # Recency score (using when/then chains)
        pl.when(pl.col('days_since_last_order') <= 30).then(5)
          .when(pl.col('days_since_last_order') <= 90).then(4)
          .when(pl.col('days_since_last_order') <= 180).then(3)
          .when(pl.col('days_since_last_order') <= 365).then(2)
          .otherwise(1)
          .alias('recency_score'),

        # Frequency score (using cut for binning)
        pl.col('order_count')
          .cut([0, 5, 10, 20, 50], labels=['1', '2', '3', '4', '5'])
          .cast(pl.Int8)
          .alias('frequency_score'),

        # Monetary score
        pl.col('total_spent')
          .cut([0, 500, 1000, 5000, 10000], labels=['1', '2', '3', '4', '5'])
          .cast(pl.Int8)
          .alias('monetary_score')
    ])
    .with_columns([
        # Weighted score
        (
            pl.col('recency_score') * 0.3 +
            pl.col('frequency_score') * 0.3 +
            pl.col('monetary_score') * 0.4
        ).alias('weighted_score'),

        # LTV with decay
        (
            (pl.col('recency_score') * 0.3 +
             pl.col('frequency_score') * 0.3 +
             pl.col('monetary_score') * 0.4) * 100 *
            pl.col('days_since_last_order').truediv(-365.0).exp()
        ).alias('customer_ltv'),

        # Segment
        pl.when(pl.col('weighted_score') >= 4.0).then(pl.lit('Champion'))
          .when(pl.col('weighted_score') >= 3.0).then(pl.lit('Loyal'))
          .when(pl.col('weighted_score') >= 2.0).then(pl.lit('Potential'))
          .otherwise(pl.lit('At Risk'))
          .alias('customer_segment')
    ])
    .collect(streaming=True)
)
```

### 성능 결과

(성능 결과는 원문 이미지 참조)

### 주요 통찰

바로 이 부분이 Polars가 빛을 발하는 지점입니다:

*   ✅ 코드가 더 깔끔합니다 — 메서드 체이닝(method chaining) 대(vs) 중첩된 CTE(Common Table Expressions)
*   ✅ 테스트 가능 — 개별 변환(transformation)에 대한 단위 테스트(unit test) 가능
*   ✅ 재사용 가능 — 점수 계산 로직을 함수로 추출
*   ✅ 수학적 표현 가능 — 넘파이(numpy) 함수를 코드 내에서 직접 사용 가능

재사용성 예시:

```python

    """Reusable RFM scoring function"""
    return df.with_columns([
        # Recency
        pl.when(pl.col('days_since_last_order') <= 30).then(5)
          .when(pl.col('days_since_last_order') <= 90).then(4)
          # ... etc
          .alias('recency_score'),
        # Frequency
        pl.col('order_count').cut([...]).alias('frequency_score'),
        # Monetary
        pl.col('total_spent').cut([...]).alias('monetary_score')
    ])

# Use in multiple places
customers_rfm = calculate_rfm_score(customers_lazy)
segments_rfm = calculate_rfm_score(segments_lazy)
```

## SQL이 여전히 우세한 경우

✅ 다음의 경우 SQL을 사용하세요:

    ```sql
    -- 쿼리(query) 로그 추적
    -- GRANT를 통한 접근 제어
    -- 감사 추적(audit trail) 내장
    SELECT * FROM sensitive_pii_data
    WHERE query_user = CURRENT_USER()
    ```
3.  **팀이 SQL 우선(SQL-first)인 경우**
    *   분석가(analyst)들이 유지보수할 수 있음
    *   Python 지식 불필요
    *   모두에게 익숙함
4.  **매우 큰 규모의 데이터 (1억 행 이상을 정기적으로 처리)**
    *   노드(node) 간 웨어하우스(warehouse) 병렬 처리
    *   1TB 이상의 데이터셋(dataset)에 더 적합
    *   높은 동시성을 위한 비용 최적화
5.  **표준 분석**
    *   단순 집계
    *   BI 대시보드(dashboard) 쿼리(query)
    *   표준 보고서
    *   사용자 정의 로직(custom logic) 불필요

## 하이브리드 접근 방식 (2026년 모범 사례)

현대적인 데이터 팀은 둘 다 사용합니다:

```
┌─────────────────────────────────────┐
│         데이터 파이프라인 (Data Pipeline)             │
├─────────────────────────────────────┤
│                                     │
│ 1. 추출 (Extraction) → Polars       │
│    - API 호출 (API calls)             │
│    - 파일 처리 (File processing)      │
│    - 데이터 강화 (Data enrichment)    │
│                                     │
│ 2. 로드 (Load) → 웨어하우스 (Warehouse)       │
│    - Polars는 Snowflake에 작성           │
│                                     │
│ 3. 변환 (Transform) → SQL (dbt)     │
│    - 표준 분석 (Standard analytics)   │
│    - 공유 모델 (Shared models)        │
│                                     │
│ 4. 특수 케이스 (Special cases) → Polars │
│    - 복잡한 알고리즘 (Complex algorithms)│
│    - ML 특성 공학 (ML feature engineering) │
│    - 일회성 분석 (One-off analysis)     │
│                                     │
│ 5. 제공 (Serve) → SQL (BI 도구)      │
│    - 대시보드 (Dashboards)            │
│    - 보고서 (Reports)                 │
└─────────────────────────────────────┘
```

실제 파이프라인 예시:

```python
# Step 1: Polars extracts and enriches
import polars as pl
import requests

def fetch_and_enrich():
    # Call API (Polars better for this)
    response = requests.get('https://api.example.com/orders')
    df = pl.DataFrame(response.json())

    # Enrich with complex logic
    df = df.with_columns([
        pl.col('email').str.extract(r'@(.+)').alias('email_domain'),
        calculate_custom_score(pl.col('value')).alias('score')
    ])

    # Step 2: Write to warehouse
    df.write_database(
        table_name='staging_orders',
        connection=warehouse_conn,
        if_exists='replace'
    )

fetch_and_enrich()
```

## 코드 유지보수성: Polars 대(vs) SQL

저는 동일한 5명의 엔지니어에게 Polars 코드에 대한 평가를 요청했습니다.

(코드 유지보수성 평가 그래프는 원문 이미지 참조)

주요 결과:

*   Polars는 복잡한 로직(logic)에서 우세합니다 (메서드 체이닝(method chaining), 중간 단계 출력).
*   SQL은 단순 분석에서 우세합니다 (모두가 `SELECT`/`GROUP BY`를 알고 있습니다).
*   Polars는 디버깅(debugging)이 더 쉽습니다 (각 단계에서 검사 가능).
*   SQL은 공유하기 더 쉽습니다 (Python 환경 불필요).

## 비용 분석: Polars 대(vs) SQL

컴퓨팅(compute) 비용 (100만 행 집계):

**Snowflake의 SQL:**

*   X-Small 웨어하우스(warehouse): 시간당 2달러
*   실행 시간: 2.3초
*   실행당 비용: 0.0013달러

**EC2의 Polars:**

*   c6i.2xlarge: 시간당 0.34달러
*   실행 시간: 1.7초
*   실행당 비용: 0.00016달러

**하지만: Polars의 숨겨진 비용:**

*   웨어하우스(warehouse)에서 데이터 이그레스(egress): GB당 0.09달러
*   100만 행 (~85MB)의 경우: 전송 비용 0.0076달러
*   Polars 총비용: 0.00176달러

**결과:** 데이터가 이미 웨어하우스(warehouse)에 있는 경우 SQL이 약간 더 저렴합니다 (0.0013달러 대(vs) 0.0017달러)!

**하지만:** 데이터가 아직 웨어하우스(warehouse)에 없는 경우:

*   Polars는 로컬(local)에서 처리합니다 (전송 비용 0).
*   SQL은 먼저 로드(load)해야 합니다.
*   Polars가 10배 우세합니다.

## 평결

### 성능 승자: Polars (근소하게)

*   일반적인 워크로드(workload)에서 평균 1.2배 빠름
*   소형-중형 데이터(1,000만 행 미만)에서 확실하게 우세
*   대규모 데이터(5,000만 행 이상)에서는 SQL과 동등
*   스트리밍 엔진(streaming engine)은 판도를 바꿀 만함

### 개발자 경험(Developer Experience) 승자: Polars

*   복잡한 로직(logic)에 더 깔끔한 코드
*   더 나은 디버깅(debugging) (중간 결과 출력)
*   테스트 가능 (pytest를 사용한 단위 테스트(unit test))
*   Python 생태계와 통합

### 프로덕션(Production) 승자: 상황에 따라 다름

**다음의 경우 SQL을 사용하세요:**

*   데이터가 이미 웨어하우스(warehouse)에 있는 경우
*   팀이 SQL 우선(SQL-first)인 경우
*   거버넌스(governance) 및 감사 추적(audit trail)이 필요한 경우
*   조직 전체에 쿼리(query)를 공유하는 경우

**다음의 경우 Polars를 사용하세요:**

*   로컬 개발
*   복잡한 사용자 정의 알고리즘(custom algorithm)
*   ML 특성 공학(feature engineering)
*   여러 데이터 소스(data source)를 혼합하는 경우

## 저의 업데이트된 권장 사항

2026년 데이터 엔지니어링 스택(stack):

```
Python (Polars) → 웨어하우스 (Warehouse) → SQL (dbt) → BI 도구 (BI Tools)
       ↓                           ↓
   추출 (Extraction)             변환 (Transformation)
   강화 (Enrichment)             표준화 (Standardization)
   복잡한 로직 (Complex Logic)     공유 (Sharing)
```

*   **시작:** 추출(extraction) 및 복잡한 전처리(prep)를 위해 Polars
*   **로드(Load) 대상:** 웨어하우스(Warehouse) (Snowflake, BigQuery, Databricks)
*   **변환(Transform):** 공유 분석을 위해 SQL/dbt
*   **특수 케이스:** SQL이 복잡해질 때 Polars로 다시 가져옴

**결론:**

*   pandas는 이제 그만 (너무 느림, 대신 Polars 사용)
*   Polars 대(vs) SQL은 경쟁적 (Polars가 성능 면에서 약간 우세)
*   둘 다 전략적으로 사용 (Polars는 전처리, SQL은 공유 분석)
