# 데이터 엔지니어가 직접 경험하기 전엔 믿지 못했을 최고의 SQL 핵심 기술

10년 넘게 SQL을 사용하면서 이 언어를 속속들이 알고 있다고 생각했습니다. 하지만 이 기술들을 접하게 된 후 그 생각이 바뀌었습니다. 각 기술을 접할 때마다 잠시 멈춰 다시 읽어보고 세 번씩 테스트한 후, 왜 아무도 이전에 이것을 가르쳐주지 않았을까 하는 의문을 가졌습니다.

이것은 일반적인 '인덱스 사용' 조언이 아닙니다. 실제 현장에서 검증된 핵심 기술(hack)들로, 수많은 작업 시간과 수천 줄의 코드, 그리고 프로덕션 환경에서의 여러 골칫거리를 해결해 주었습니다.

이제 자세히 살펴보겠습니다.

## 1. 인라인 테이블로 `VALUES` 사용하기 (실제 테이블 불필요)

대부분의 사람들은 작은 데이터셋으로 작업할 때 물리적 테이블이나 `CTE`가 필요하다고 생각합니다. 하지만 그렇지 않습니다.

```sql
SELECT *
FROM (VALUES
    (1, 'Alice', 'Engineering'),
    (2, 'Bob',   'Marketing'),
    (3, 'Carol', 'Engineering')
) AS employees(id, name, department)
WHERE department = 'Engineering';
```

**이 기술이 놀라웠던 이유:** 쿼리 내에서 임시 테이블을 바로 만들 수 있다는 점입니다. 데이터베이스 스키마를 건드리지 않고도 개발 중 빠른 조회, 값 매핑, 데이터 목업(mocking)에 완벽합니다.

**실제 활용 사례:** 한 번은 15개의 상태 코드(status code)를 사람이 읽을 수 있는 레이블(label)로 매핑해야 했습니다. 매핑 테이블을 생성하고 DBA의 승인을 기다리는 대신, `VALUES`를 사용하여 매핑을 인라인으로 정의하고 메인 쿼리에 조인했습니다.

## 2. 집계 함수 내 `CASE WHEN`의 우아한 대안, `FILTER` 절

만약 다음과 같은 쿼리를 작성해 본 적이 있다면, 그 고통을 아실 겁니다.

```sql
-- 기존의 보기 흉한 방식
SELECT
    department,
    COUNT(CASE WHEN status = 'active' THEN 1 END) AS active_count,
    COUNT(CASE WHEN status = 'inactive' THEN 1 END) AS inactive_count,
    SUM(CASE WHEN status = 'active' THEN salary END) AS active_salary
FROM employees
GROUP BY department;
```

이제 다음 쿼리를 살펴보세요.

```sql
-- 깔끔한 방식 (PostgreSQL, SQLite 3.30+, DuckDB)
SELECT
    department,
    COUNT(*) FILTER (WHERE status = 'active') AS active_count,
    COUNT(*) FILTER (WHERE status = 'inactive') AS inactive_count,
    SUM(salary) FILTER (WHERE status = 'active') AS active_salary
FROM employees
GROUP BY department;
```

**이 기술이 놀라웠던 이유:** 영어처럼 읽힙니다. "모든 것을 세어라, 상태가 활성인 경우로 필터링하여." 더 깔끔하고 오류 발생 가능성이 적으며, 대시보드 쿼리에 10개 이상의 조건부 집계가 있을 때 가독성을 획기적으로 향상시킵니다.

## 3. 아무것도 없는 상태에서 전체 날짜 시리즈 생성하기

특정 범위 내의 모든 날짜에 대해 행(row)이 필요했지만, 해당 날짜에 데이터가 없는 경우에도 행이 필요했던 적이 있나요? 대부분의 사람들은 파이썬(Python)에서 루프(loop)를 작성합니다. 그럴 필요 없습니다.

```sql
-- PostgreSQL
SELECT generate_series(
    '2024-01-01'::date,
    '2024-12-31'::date,
    '1 day'::interval
)::date AS date;

-- MySQL 8.0+ (재귀 CTE 방식)
WITH RECURSIVE dates AS (
    SELECT '2024-01-01' AS date
    UNION ALL
    SELECT date + INTERVAL 1 DAY
    FROM dates
    WHERE date < '2024-12-31'
)
SELECT * FROM dates;
```

**이 기술이 놀라웠던 이유:** 이 날짜 시리즈를 실제 데이터에 `LEFT JOIN`하면, 시계열 보고서에서 '누락된 날짜' 공백이 다시는 발생하지 않을 것입니다. 여러분의 대시보드가 고마워할 겁니다.

```sql
SELECT
    d.date,
    COALESCE(o.order_count, 0) AS order_count
FROM generate_series('2024-01-01'::date, '2024-01-31'::date, '1 day') AS d(date)
LEFT JOIN (
    SELECT order_date, COUNT(*) AS order_count
    FROM orders
    GROUP BY order_date
) o ON o.order_date = d.date
ORDER BY d.date;
```

더 이상 "1월 17일이 차트에서 왜 빠졌죠?" 같은 대화는 없을 겁니다.

## 4. `GROUPING SETS` — 단일 패스로 여러 `GROUP BY` 수행

매니저가 "부서별 총계, 지역별 총계, 그리고 전체 총계가 필요합니다. 세 개의 쿼리를 작성해 주시겠어요?"라고 묻는다고 상상해 보세요.

아니요. 단 하나의 쿼리를 작성합니다.

```sql
SELECT
    department,
    region,
    SUM(revenue) AS total_revenue
FROM sales
GROUP BY GROUPING SETS (
    (department),
    (region),
    (department, region),
    ()  -- 전체 총계 (grand total)
);
```

**샘플 출력:**

| department | region | total_revenue |
| :--------- | :----- | :------------ |
| Engineering | NULL | 500000 |
| Marketing | NULL | 300000 |
| NULL | East | 450000 |
| NULL | West | 350000 |
| Engineering | East | 280000 |
| Engineering | West | 220000 |
| NULL | NULL | 800000 |

**이 기술이 놀라웠던 이유:** 단일 테이블 스캔, 단일 쿼리로 여러 수준의 집계가 가능합니다. 데이터베이스 옵티마이저(optimizer)는 세 개의 개별 쿼리를 실행하는 것보다 훨씬 효율적으로 처리합니다. `GROUPING()` 함수와 함께 사용하면 실제 `NULL` 값과 "이것은 소계(subtotal)입니다"라는 `NULL`을 구분할 수 있습니다.

## 5. `LATERAL JOIN` — SQL의 "각 항목에 대한" 루프

이 기술은 심각하게 과소평가되어 있습니다. `LATERAL JOIN`은 오른쪽의 서브쿼리가 왼쪽의 열(column)을 참조할 수 있게 하여, 본질적으로 각 행(row)에 대해 반복하는 것과 같습니다.

**문제:** 각 부서별로 급여가 가장 높은 상위 3명의 직원을 가져옵니다.

```sql
-- PostgreSQL / MySQL 8.0.14+
SELECT
    d.department_name,
    top_emp.name,
    top_emp.salary
FROM departments d
CROSS JOIN LATERAL (
    SELECT name, salary
    FROM employees e
    WHERE e.department_id = d.id
    ORDER BY salary DESC
    LIMIT 3
) AS top_emp;
```

**이 기술이 놀라웠던 이유:** `LATERAL JOIN` 이전에는 "그룹별 상위 N개(top-N per group)" 문제를 해결하기 위해 `CTE`와 함께 윈도우 함수(window function)를 사용하거나 상관 서브쿼리(correlated subquery)를 사용해야 했습니다. `LATERAL JOIN`은 이를 너무나 자연스럽게 표현하여 거의 반칙처럼 느껴집니다. `SQL Server` 사용자들은 이를 `CROSS APPLY` / `OUTER APPLY`로 알고 있습니다.

## 6. `::` 연산자와 문자열-배열 변환 기술

`PostgreSQL`에서는 순수 SQL만으로 문자열을 분할하고, 배열을 역정규화(unnest)하고, 다시 조립할 수 있습니다.

```sql
-- 콤마로 구분된 문자열을 행으로 분할
SELECT unnest(string_to_array('apple,banana,cherry', ',')) AS fruit;
```

| fruit |
| :---- |
| apple |
| banana |
| cherry |

이제 반대로 — 행을 단일 값으로 다시 합칩니다.

```sql
SELECT
    department,
    STRING_AGG(employee_name, ', ' ORDER BY employee_name) AS team_members
FROM employees
GROUP BY department;
```

| department | team_members |
| :--------- | :----------- |
| Engineering | Alice, Bob, Carol |
| Marketing | Dave, Eve |

**이 기술이 놀라웠던 이유:** 과거에는 문자열을 분할하고 재결합하기 위해 데이터를 파이썬(Python)으로 내보내곤 했습니다. 알고 보니 SQL은 수년 전부터 이 기능을 기본적으로 수행할 수 있었습니다. 이 함수들을 함께 사용하면 비정규화된 CSV 열을 정규화하고, 정규화된 행을 비정규화할 수 있습니다 — 쿼리를 벗어나지 않고 말이죠.

## 7. `RANGE`와 `ROWS`를 사용한 윈도우 함수 — 동일하지 않습니다

대부분의 튜토리얼에서는 `ROWS BETWEEN`을 가르칩니다. `RANGE BETWEEN`이 언제 위기를 구할 수 있는지 설명하는 경우는 거의 없습니다.

```sql
-- 일별 수익의 7일 이동 평균 계산
SELECT
    order_date,
    daily_revenue,
    AVG(daily_revenue) OVER (
        ORDER BY order_date
        RANGE BETWEEN INTERVAL '6 days' PRECEDING AND CURRENT ROW
    ) AS rolling_7day_avg
FROM daily_sales;
```

**결정적인 차이:**

*   `ROWS BETWEEN 6 PRECEDING AND CURRENT ROW`는 날짜 간격에 관계없이 이전 6개의 행을 살펴봅니다.
*   `RANGE BETWEEN INTERVAL '6 days' PRECEDING AND CURRENT ROW`는 이전 6일의 실제 달력 날짜를 살펴보고, 누락된 날짜를 올바르게 처리합니다.

데이터에 공백(주말, 공휴일)이 있는 경우, `ROWS`는 잘못된 평균을 제공합니다. `RANGE`는 올바른 평균을 제공합니다.

**이 기술이 놀라웠던 이유:** '약간 어긋난' 이동 평균을 이틀 동안 디버깅하다가 문제의 원인이 `ROWS`와 `RANGE`의 차이임을 깨달았습니다. 한 단어만 바꾸니 모든 것이 해결되었습니다.

## 8. `INSERT … ON CONFLICT` (업서트(Upsert)) — '삭제 후 삽입' 방식에 작별을 고하다

기존의 "기존 항목 삭제 후 새 항목 삽입" 패턴은 취약하고 원자적(atomic)이지 않습니다. 업서트(upsert)가 등장했습니다.

```sql
-- PostgreSQL
INSERT INTO user_settings (user_id, theme, language)
VALUES (42, 'dark', 'en')
ON CONFLICT (user_id)
DO UPDATE SET
    theme = EXCLUDED.theme,
    language = EXCLUDED.language;

-- MySQL 동등한 코드
INSERT INTO user_settings (user_id, theme, language)
VALUES (42, 'dark', 'en')
ON DUPLICATE KEY UPDATE
    theme = VALUES(theme),
    language = VALUES(language);
```

**이 기술이 놀라웠던 이유:** 이 방식은 원자적이며 단일 구문이고, 모든 `ETL` 파이프라인(pipeline)에서 나타나는 "새 항목이면 삽입, 기존 항목이면 업데이트" 패턴을 처리합니다. 더 이상 경쟁 조건(race condition)도, 애플리케이션 코드의 `try-catch` 블록도 필요 없습니다.

## 9. `DISTINCT ON` — PostgreSQL의 비장의 무기

이것은 단연코 제가 가장 좋아하는 `PostgreSQL` 전용 기능입니다.

**문제:** 각 고객의 가장 최근 주문을 가져옵니다.

```sql
-- 일반적인 (장황한) 방식
WITH ranked AS (
    SELECT *, ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date DESC) AS rn
    FROM orders
)
SELECT * FROM ranked WHERE rn = 1;

-- PostgreSQL의 단축 방식
SELECT DISTINCT ON (customer_id)
    customer_id, order_id, order_date, total
FROM orders
ORDER BY customer_id, order_date DESC;
```

동일한 결과에 코드는 절반이며, 옵티마이저가 인덱스 스캔(index scan)을 직접 사용할 수 있어 종종 더 빠릅니다.

**이 기술이 놀라웠던 이유:** 모든 "그룹별 최신/최초 항목 가져오기" 문제가 두 줄짜리 솔루션으로 바뀝니다. 거의 매일 사용하고 있습니다.

## 10. 쓰기 가능한 단계로 `CTE` 활용하기 (데이터 수정 `CTE`)

대부분의 사람들은 데이터를 읽기 위해 `CTE`를 사용하지만, 데이터를 쓰고 작업을 연쇄적으로 연결하는 데 사용할 수 있다는 것을 아는 사람은 적습니다.

```sql
-- 한 번의 원자적(atomic) 구문으로 오래된 주문을 아카이브하고 삭제
WITH archived AS (
    INSERT INTO orders_archive
    SELECT * FROM orders WHERE order_date < '2023-01-01'
    RETURNING *
),
deleted AS (
    DELETE FROM orders
    WHERE order_id IN (SELECT order_id FROM archived)
    RETURNING *
)
SELECT
    COUNT(*) AS rows_processed
FROM deleted;
```

**이 기술이 놀라웠던 이유:** 단일 구문입니다. 원자적입니다. 임시 테이블이 없습니다. 여러 단계의 스크립트도 필요 없습니다. `RETURNING` 절은 한 작업의 출력을 다음 작업으로 공급합니다. 이것이 새벽 3시에 고장 나지 않는 `ETL` 로직을 작성하는 방법입니다.

## 보너스: `EXPLAIN ANALYZE` 활용 마인드셋

이것은 '핵심 기술'이 아니라 습관입니다. 하지만 이것은 여러분을 10배 더 나은 SQL 개발자로 만들 것입니다.

```sql
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT ...
```

출력을 아래에서 위로 읽으세요. 다음 사항을 찾으세요.

*   대용량 테이블에 대한 `Seq Scan` (아마 인덱스가 필요할 것입니다)
*   높은 행 수를 가진 `Nested Loop` (구조 조정을 통해 해시 조인(hash join) 고려)
*   실제 행 수(`Actual rows`)와 예상 행 수(`Estimated rows`)가 크게 다른 경우 (오래된 통계(`stale statistics`) — `ANALYZE` 실행)

**사고방식의 전환:** 쿼리가 느린 이유를 추측하지 마세요. 데이터베이스에게 직접 물어보세요. 제가 아는 모든 시니어 데이터 엔지니어(data engineer)들은 거의 문신처럼 `EXPLAIN ANALYZE`를 기억하고 있습니다.
