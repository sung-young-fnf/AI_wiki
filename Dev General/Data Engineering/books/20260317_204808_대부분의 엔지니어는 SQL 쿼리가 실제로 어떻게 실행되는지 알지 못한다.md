# 대부분의 엔지니어는 SQL 쿼리가 실제로 어떻게 실행되는지 알지 못한다

SQL은 위에서 아래로 작성되지만, 데이터베이스는 그렇게 읽지 않습니다.

SQL을 작성할 때의 모습과 데이터베이스가 실제로 처리하는 방식 사이에는 간극이 존재합니다. 대부분의 엔지니어들은 이 간극을 인지하지 못한 채 수년간 쿼리를 작성합니다. 그러다 성능 문제(performance problem)나 혼란스러운 오류에 직면하면, 그동안 가지고 있던 사고 모델(mental model)이 조용히 무너지게 됩니다.

해결책은 간단합니다. 논리적 실행 순서(logical execution order)를 이해하는 것입니다. 이를 이해하면 쿼리 작성 방식, 디버깅 방식, 그리고 특정 최적화(optimization)가 작동하고 다른 최적화는 작동하지 않는 이유가 달라집니다.

## 구문(Syntax)에 숨겨진 진실

다음은 대부분의 사람들이 실행된다고 생각하는 방식대로 작성된 일반적인 쿼리입니다.

```sql
SELECT user_id, COUNT(order_id) AS total_orders
FROM orders
JOIN users ON orders.user_id = users.id
WHERE orders.status = 'completed'
GROUP BY user_id
HAVING COUNT(order_id) > 5
ORDER BY total_orders DESC
LIMIT 10;
```

순차적으로 보입니다. `SELECT`로 시작하여 테이블에서 데이터를 가져오고, 필터링하고, 그룹화하고, 정렬하고, 제한합니다. 이것이 읽는 순서입니다.

하지만 이것은 실행 순서가 아닙니다.

## 실제 실행 파이프라인(Pipeline)

### SQL의 논리적 실행 순서
────────────────────────────────────────────────────────────
**STEP 1 FROM**        → 기본 데이터 소스 식별
         │
**STEP 2 JOIN**        → 관련 테이블 가져오기
         │
**STEP 3 ON**          → 테이블 간 행 일치 방식 정의
         │
**STEP 4 WHERE**       → 그룹화 전에 행 필터링
         │
**STEP 5 GROUP BY**    → 행을 그룹으로 통합
         │
**STEP 6 HAVING**      → 그룹/집계(aggregated) 결과 필터링
         │
**STEP 7 SELECT**      → 출력 컬럼 계산
         │
**STEP 8 ORDER BY**    → 결과 집합(result set) 정렬
         │
**STEP 9 LIMIT**       → 반환되는 행 수 제한
────────────────────────────────────────────────────────────

`SELECT`는 거의 마지막에 위치합니다. 데이터베이스는 모든 데이터, 모든 조인(join), 모든 그룹화를 처리한 후에 어떤 컬럼을 사용자에게 반환할지 결정합니다.

이것은 사소한 정보가 아닙니다. 직접적인 실질적 결과(practical consequences)를 가져옵니다.

## `WHERE`와 `HAVING`이 같지 않은 이유

이 부분이 실행 순서가 가장 큰 혼란을 야기하는 지점입니다.

```sql
-- 잘못된 예: WHERE 절에서 별칭(alias)으로 필터링 시도
SELECT user_id, COUNT(order_id) AS total_orders
FROM orders
WHERE total_orders > 5   -- 실패: total_orders는 아직 존재하지 않음
GROUP BY user_id;

-- 올바른 예: 집계 후(post-aggregation) 필터링에는 HAVING 사용
SELECT user_id, COUNT(order_id) AS total_orders
FROM orders
GROUP BY user_id
HAVING COUNT(order_id) > 5;  -- ✓ GROUP BY 이후에 실행됨
```

첫 번째 쿼리의 오류는 실행 순서를 알기 전까지는 이해되지 않습니다. `WHERE`는 `SELECT`보다 먼저 실행됩니다. `WHERE`가 평가될 때 `total_orders`라는 별칭(alias)은 아직 존재하지 않습니다. 아직 계산되지 않았기 때문입니다.

`HAVING`은 `GROUP BY`와 집계(aggregation) 이후에 실행되므로 계산된 값을 참조할 수 있습니다.

## 이것이 성능에 중요한 이유

실행 순서는 또한 최적화(optimization)의 지도이기도 합니다.

### 각 단계별 데이터 크기 (일반적인 쿼리)
──────────────────────────────────────────────────────
FROM        ████████████████████████  전체 테이블 스캔(full table scan)
JOIN        ████████████████████████  잠재적으로 행을 두 배로 늘림
WHERE       ███████                   초기에 필터링하여 줄임 ← 핵심
GROUP BY    █████                     행을 그룹으로 통합
HAVING      ████                      두 번째 필터링 과정
SELECT      ████                      컬럼 프로젝션(column projection)
ORDER BY    ████                      결과 정렬
LIMIT       █                         최종적으로 잘라냄
──────────────────────────────────────────────────────

`WHERE`는 일찍 실행됩니다. 이는 잘 작성된 `WHERE` 절이 그룹화(grouping)나 정렬(sorting)과 같은 비용이 많이 드는 작업이 데이터를 건드리기 전에 행 개수(row count)를 극적으로 줄여준다는 것을 의미합니다.

가장 선별적인 필터(selective filters)는 `HAVING`이 아닌 `WHERE`에 배치하세요. `HAVING`은 집계(aggregation) 후에 필터링하므로, 그때는 이미 많은 작업이 완료된 상태입니다.

```sql
-- 비효율적: 모든 행을 그룹화한 후에 필터링 발생
SELECT department, AVG(salary)
FROM employees
GROUP BY department
HAVING department = 'Engineering';

-- 더 효율적: 그룹화 전에 필터링하여 상위 작업 감소
SELECT department, AVG(salary)
FROM employees
WHERE department = 'Engineering'
GROUP BY department;
```

두 쿼리 모두 동일한 결과를 반환합니다. 두 번째 쿼리는 `GROUP BY`가 실행되기 전에 'Engineering'이 아닌 행들을 제거했기 때문에 훨씬 적은 수의 행을 그룹화합니다.

## 조인 함정(The JOIN Trap)

`JOIN`은 `WHERE` 필터링보다 파이프라인(pipeline)의 초기에 실행됩니다. 이것이 바로 조건(conditions) 없이 큰 테이블을 조인(join)하는 것이 쿼리 성능을 조용히 저하시킬 수 있는 이유입니다.

### `WHERE` 전 `JOIN` 예시
─────────────────────────────────────────────────────────
orders 테이블:      10,000,000 행
users 테이블:          500,000 행

JOIN (2단계):     잠재적으로 1천만 개 이상의 행으로 구성된 중간 집합(intermediate set) 생성
WHERE (4단계):    그 거대한 중간 집합을 필터링
vs.
만약 WHERE가 먼저 실행될 수 있다면:   50,000 행 정도
50,000 행에 대한 JOIN:        빠름
─────────────────────────────────────────────────────────

이것이 `JOIN` 컬럼과 `WHERE` 컬럼에 인덱스(indexes)를 사용하는 것이 그토록 중요한 이유입니다. 데이터베이스 옵티마이저(database optimizer)는 가능한 경우 작업 순서를 재정렬(reorder)하려고 시도하지만, 올바른 데이터 구조(data structures)를 제공했을 때 가장 잘 작동합니다.

## 실용적인 사고 모델(Mental Model)

쿼리를 작성하기 전에 다음 파이프라인(pipeline)을 따라가 보세요.

1.  **FROM / JOIN** → 어떤 데이터를 결합할 것인가?
2.  **WHERE**       → 이 데이터 집합을 얼마나 일찍 줄일 수 있는가?
3.  **GROUP BY**    → 무엇을 기준으로 집계(aggregate)할 것인가?
4.  **HAVING**      → 집계 결과에 대해 필터링해야 하는가?
5.  **SELECT**      → 실제로 출력에 필요한 것은 무엇인가?
6.  **ORDER BY**    → 정말 정렬(sorting)이 필요한가? (비용이 많이 든다)
7.  **LIMIT**       → 출력을 더 일찍 줄일 수 있는가?

대부분의 쿼리 성능 문제(query performance problems)는 이 파이프라인(pipeline)을 위반하는 것에서 비롯됩니다. 잘못된 위치의 필터, 대규모 데이터셋(dataset)에 대한 불필요한 정렬, 충분히 일찍 제한되지 않은 조인 등이 그 예입니다.

## 한 가지 더: `SELECT *`는 늦고 비쌉니다.

`SELECT`가 거의 마지막에 실행되기 때문에, `SELECT *`는 데이터베이스가 필요 없는 컬럼(column)을 버리기 전에 조인, 필터, 집계(aggregation) 등 전체 파이프라인(pipeline)을 통해 모든 컬럼을 전달하도록 강제합니다.

```sql
-- 파이프라인의 모든 단계에서 모든 컬럼을 전달
SELECT * FROM orders JOIN users ON orders.user_id = users.id
WHERE orders.status = 'completed';

-- 각 단계에서 필요한 것만 전달
SELECT orders.id, orders.total, users.email
FROM orders JOIN users ON orders.user_id = users.id
WHERE orders.status = 'completed';
```

작은 테이블에서는 그 차이가 미미합니다. 하지만 20개의 컬럼과 수백만 개의 행을 가진 테이블에서는 컬럼 프로젝션(column projection)이 중요합니다.

## 핵심 요약(The Takeaway)

SQL은 실행 모델(execution model)을 드러내기보다는 읽기 쉽게 설계되었습니다. 이것은 합리적인 설계 선택이지만, 대부분의 엔지니어가 쿼리를 작성하는 것만으로 개발하는 사고 모델(mental model)이 불완전하다는 것을 의미합니다.

논리적 실행 순서(logical execution order)는 학술적인 내용에 그치지 않습니다. 이는 특정 오류가 발생하는 이유, 일부 최적화(optimization)가 작동하는 이유, 그리고 동일해 보이는 쿼리가 부하(load) 상황에서 완전히 다르게 성능을 발휘하는 이유를 설명해 줍니다.

읽기 순서: `SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY`
실제 순서: `FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT`

이 순서가 내재화되면 쿼리 설계(query design)가 달라집니다. 필터는 더 일찍 적용되고, 별칭(aliases)은 올바르게 사용되며, 집계(aggregations)는 실행 전에 제한됩니다. 성능 문제(performance problems)를 추론하기가 더 쉬워집니다.

이는 하위의 모든 것을 조용히 개선하는 기본적인 지식 중 하나입니다.
