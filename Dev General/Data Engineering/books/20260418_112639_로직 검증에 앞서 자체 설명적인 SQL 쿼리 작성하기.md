# 로직 검증에 앞서 자체 설명적인 SQL 쿼리 작성하기

추가 문서 없이 의도, 가정, 계산이 스스로 설명되는 쿼리를 구조화하는 방법

지난달, 저는 한 SQL 파일을 열었습니다. 그리고 속으로 외쳤죠, '이게 도대체 뭐야?!'

주석도, 구조도 없는 1400줄짜리 중첩된 혼돈이었습니다. 그야말로 엉망진창이었죠.

tbl_a, final_temp_v3 같은 별칭(alias)만 가득했습니다.

그 코드를 이해하고 푸는 데만 하루 반나절이 걸렸습니다.

가장 심각한 건, 그 쿼리가 작동했다는 겁니다.

하지만 재무팀에서 수익 수치가 왜 잘못되었는지 물었을 때, 완전히 침묵할 수밖에 없었습니다.

지저분한 SQL은 똑똑한 것이 아니라, 자폭 행위입니다. 한번은 새벽 2시에 동료가 쿼리 전체를 다시 작성하는 것을 본 적이 있습니다. 왜냐고요? WHERE 절이 여섯 겹의 서브쿼리(subquery) 깊숙이 파묻혀 있었기 때문이죠.

팀들은 매년 '똑똑하다'고 여겨지는 코드를 고치는 데 몇 주를 허비합니다. 만약 여러분의 SQL에 번역기가 필요하다면, 그것은 코딩하는 것이 아니라 미래의 악몽을 만드는 일입니다.

## 명확성의 아키텍처

여러분의 로직에 대한 검증을 막으려면, 가독성을 염두에 두고 코드를 작성해야 합니다. 쿼리가 비밀을 숨기는 대신 이야기를 전달하도록 제가 사용하는 프레임워크(framework)는 다음과 같습니다.

### 1. 서브쿼리 지옥 대신 CTE 활용하기

중첩된 서브쿼리(subquery)는 양파와 같아서 여러분을 울게 만듭니다.

공통 테이블 표현식(CTE: Common Table Expression)은 로직을 위에서 아래로 정의하여 데이터 흐름을 선형적으로 만듭니다.

### 2. "사람이 읽기 쉬운" 규칙

*   **별칭(alias) 이름 바꾸기**: t1이나 a 대신 active_customers 또는 monthly_recurring_revenue와 같이 의미를 명확히 하는 이름으로 바꿉니다.
*   **불리언 플래그(Boolean Flag)**: `status_id IN (1, 4, 7)`을 확인하는 대신, `is_eligible_for_discount`와 같은 이름의 컬럼(column)을 만듭니다.
*   **선형 로직(Linear Logic)**: SQL을 요리 레시피처럼 다루세요. 재료를 준비하고(CTE), 섞어서(JOIN), 요리를 접시에 담습니다(Final SELECT).

## "자체 설명적인" 변환

고객 유지율(customer retention)을 계산한다고 가정해봅시다. '이전' 버전에서는 괄호가 여러 겹으로 쌓여 있었을 것입니다. '이후' 버전에서는 로직(logic)이 명확해집니다.

**이전 (혼돈)**
```sql
SELECT t1.id, t1.rev, t2.last_login
FROM (SELECT * FROM sales WHERE amt > 100) t1
JOIN (SELECT user_id, MAX(dt) as last_login FROM logs GROUP BY 1) t2 
ON t1.id = t2.user_id
WHERE t2.last_login < '2025-01-01';
```

**이후 (자체 설명적)**
```sql
WITH high_value_sales AS (
    -- 100달러 초과 거래를 필터링하여 주요 계정만 격리합니다.
    SELECT 
        customer_id, 
        amount AS revenue 
    FROM sales 
    WHERE amount > 100
),

customer_activity_tracking AS (
    -- 사용자별 가장 최근 상호작용을 식별합니다.
    SELECT 
        user_id, 
        MAX(login_date) AS last_active_date 
    FROM user_logs 
    GROUP BY user_id
)
SELECT 
    hvs.customer_id, 
    hvs.revenue, 
    cat.last_active_date
FROM high_value_sales hvs
JOIN customer_activity_tracking cat ON hvs.customer_id = cat.user_id
WHERE cat.last_active_date < '2025-01-01';
```

**이 방식이 성공적인 이유:**

*   **로직 분리**: '수익' 로직이 변경되면, JOIN 로직을 건드리지 않고도 어떤 CTE를 수정해야 하는지 정확히 알 수 있습니다.
*   **감사 가능성**: 재무팀에서 해당 수익에 대해 물을 때, `high_value_sales` 블록을 가리킬 수 있습니다. 이름 자체에 모든 것이 담겨있죠.
*   **10초 테스트**: 주니어 개발자가 이것이 무엇을 하는지 10초 안에 이해할 수 있을까요? 그렇지 않다면, 리팩토링(refactor)하세요.

이 변화는 제 팀에 엄청난 차이를 가져왔습니다. 더 이상 대시보드가 어떻게 작동하는지 설명하기 위한 '비상 회의'를 가질 필요가 없어졌습니다.

전문가 팁: 1년 후 새벽 3시에 졸린 눈으로 봐도 이해할 수 있는 SQL을 작성하세요.

## SQL을 자체 설명적으로 만드는 4가지 프레임워크 변경 사항

젠장! 6개월 전에 직접 작성한 SQL을 고대 외계어처럼 쳐다본 적이 있으신가요? 😅

한번은 저의 '영리한' 쿼리를 해독하는 데 오후 내내 시간을 허비한 적이 있습니다. 😅

결국 '영리함'이란 '미래의 내가 고통받는다'는 의미였습니다. 큰 실수였죠. 로직에 대한 검증이 시작되기도 전에 이를 막고 싶다면, 스스로를 설명하는 쿼리를 작성해야 합니다.

참고: 각 항목에 대한 전체 워크플로우(workflow)는 마지막에 공유되어 있습니다. 여기서는 중요성과 접근 방식을 설명합니다.

### 1. CTE는 생명을 구합니다

WHERE EXISTS (SELECT… FROM (SELECT…)) 같은 말도 안 되는 코드요? 절대로 안 됩니다! 서브쿼리 중첩(subquery nesting)은 인간의 뇌에는 재귀적인 악몽입니다.

공통 테이블 표현식(CTE)을 사용하여 선형적인 이야기로 나누세요.

지난달에 저는 이 방법으로 200줄짜리 괴물 같은 쿼리를 수정했습니다. 동료의 반응은 어땠을까요? "잠깐… 이게 정말 말이 되잖아?!" 🎉

### 2. 서술적인 컬럼 이름 (줄임말은 피해주세요!)

컬럼 이름은 중요합니다. `revenue_after_refunds`는 더 길지만, 더 친절한 이름입니다.

a, b, t1과 같은 별칭(alias)을 사용하는 것은 주방의 모든 병에 "음식"이라고 라벨을 붙이는 것과 같습니다.

새벽 2시에 디버깅할 때 데이터가 무엇을 의미하는지 정확히 알게 되면, 미래의 여러분은 "고마워!"라고 소리칠 것입니다.

### 3. 들여쓰기는 산소와 같습니다 (쿼리가 숨 쉴 공간을 주세요)

들여쓰기(indentation)는 로직이 숨 쉴 수 있는 여백입니다.

한번은 JOIN 절이 해적 보물처럼 파묻혀 있는 버그를 발견했습니다. 5분이면 찾을 수 있었을 버그였는데, 두 시간이 걸렸습니다. 이제 저는 월급이 걸린 것처럼 포맷팅(formatting)합니다. JOIN, AND, CASE 문을 정렬하세요.

### 4. 가정을 명시하세요 (네, 제대로 들으셨습니다!)

코드가 무엇을 하는지(SQL이 이미 말해주고 있으니)만 작성하지 마세요. 왜 그렇게 하는지 작성하세요.

예: `-- # 테스트 계정 제외 (오늘 2초 소요)`

석 달 후? "젠장, 내가 왜 user_id > 1000을 제외했지?!" 여러분은 기억하지 못할 것입니다. 주석이 기억해 줄 겁니다.

## "검증 방지" SQL 변환

여기 전문적이고 진보된 변환의 예시가 있습니다. 내부 테스트 데이터를 필터링하고 복잡한 환불을 처리하면서 순수익(net revenue)을 계산하는 경우입니다.

**로직 악몽 (이전)**
```sql
SELECT t1.id, SUM(t1.amt - t2.ref_amt) as net
FROM orders t1
LEFT JOIN (SELECT ord_id, SUM(amount) as ref_amt FROM refunds GROUP BY 1) t2 ON t1.id = t2.ord_id
WHERE t1.status_id NOT IN (4, 9) AND t1.u_id > 1000
GROUP BY 1;
```

**자체 설명적인 걸작 (이후)**
```sql
WITH refund_aggregates AS (
    -- 일대다 JOIN 인플레이션을 피하기 위해 주문별 환불액을 합산합니다.
    SELECT 
        order_id, 
        SUM(refund_amount) AS total_refunded_amount
    FROM order_refunds
    GROUP BY order_id
),

filtered_active_orders AS (
    -- 취소(4) 및 사기(9) 주문을 제외합니다.
    -- 내부 테스트 계정(ID > 1000은 실제 사용자임)을 제외합니다.
    SELECT 
        order_id, 
        customer_id, 
        gross_transaction_amount
    FROM transaction_ledger
    WHERE status_id NOT IN (4, 9) 
      AND user_id > 1000
)
SELECT 
    orders.order_id,
    (orders.gross_transaction_amount - COALESCE(refunds.total_refunded_amount, 0)) AS net_realized_revenue
FROM filtered_active_orders AS orders
LEFT JOIN refund_aggregates AS refunds 
    ON orders.order_id = refunds.order_id;
```

뼈아픈 진실: 이것은 완벽함에 관한 것이 아닙니다. 나중에 자신을 미워하지 않는 것에 관한 것이며, 여러분의 직업을 사랑하는 것에 관한 것입니다.

지금 5분을 더 투자하면, 분노에 찬 디버깅(debugging) 5시간을 절약할 수 있습니다.

이해관계자(stakeholder)나 선임 설계자(senior architect)가 여러분의 로직을 검증할 때, 식은땀을 흘릴 필요가 없을 것입니다.

## 향상된 가독성을 위한 고급 패턴

또한, 저는 해결책이 단순히 더 나은 주석에만 있는 것이 아니라는 것을 빠르게 깨달았습니다. 구조적인 문제였습니다.

저는 CTE를 사용하여 텍스트 벽을 시간 순서대로 서술하는 이야기로 바꾸기 시작했습니다.

고고학자처럼 서브쿼리(subquery) 계층을 파고드는 대신, 책을 읽듯이 쿼리를 위에서 아래로 읽습니다.

### "중첩된 지옥" vs. "서술적인 CTE"

규칙: 로직을 파악하는 데 5초 이상 걸린다면, 그것은 잘못된 것입니다. 다시 작성하세요. 명확성은 사치가 아니라 산소와 같습니다.

**리팩토링(Refactored) 표준:**
```sql
/* 목적: 2025 회계연도 연간 이탈률(churn rate)을 계산합니다.
   근거: 2025년 11월 12일 리더십 지침에 따라 체험 계정은 제외합니다.
*/

WITH valid_subscriptions AS (
    -- 내부 테스트 계정 및 만료된 체험 계정을 필터링합니다.
    SELECT 
        user_id, 
        subscription_start_date,
        total_recurring_revenue
    FROM production_schema.subscriptions
    WHERE is_internal_account = FALSE 
      AND subscription_status != 'trial_expired'
),
annual_revenue_calculation AS (
    -- 다중 플랜 계정을 처리하기 위해 사용자별 수익을 집계합니다.
    SELECT 
        user_id, 
        SUM(total_recurring_revenue) AS total_user_revenue
    FROM valid_subscriptions
    WHERE subscription_start_date >= '2025-01-01'
    GROUP BY 1
)
SELECT 
    COUNT(user_id) AS total_active_customers,
    SUM(total_user_revenue) AS total_fiscal_revenue
FROM annual_revenue_calculation;
```

## 극도로 엄격한 설계자를 위한 세 가지 원칙

참고: 이 부분은 문서화에 있어 '매우 어려운 모드(Hard Mode)'에 해당합니다. (관심이 없다면 건너뛰어도 좋습니다.)

1.  **메타데이터 레이어(Meta-Data Layer)**: 저는 모든 테이블(table)과 컬럼(column)에 `COMMENT ON` 명령어를 사용하기 시작했습니다. 이제 설명은 데이터베이스 스키마(schema) 내에 직접 존재합니다. 동료들은 답변이 이미 메타데이터에 있기 때문에 기본적인 질문을 멈췄습니다. 개발자가 스키마를 탐색하면 문서화가 그들을 반길 것입니다.
2.  **린터(Linter) 의무화**: 저는 개발 환경에 SQL 린터(Linter)를 통합했습니다. `rev_tot` 또는 `t1`과 같은 불필요한 이름을 금지하도록 설정했습니다. 이는 깔끔한 들여쓰기와 명시적인 이름 지정을 강제합니다. 이제 푸시(pull request)를 고려하기도 전에 환경이 오류를 표시합니다.
3.  **맥락적 근거**: 저는 코드가 무엇을 하는지(코드는 이미 그것을 말하고 있으니)만 주석으로 달지 않습니다. 왜 그렇게 하는지 주석으로 답니다.
    예: `-- # 존재 이유: 레거시 데이터 노이즈를 피하기 위해 이해관계자가 2023년 기록만 요청함`
    더 이상 '팀 채팅' 패닉은 없습니다. 이제 제 쿼리는 스스로의 존재 이유를 설명합니다.

## 자체 설명적인 SQL 쿼리 작성을 위한 심층 단계

### STEP 1. 서브쿼리를 이름이 있는 CTE로 대체하기

**워크플로우(Workflow):**

1.  특정 비즈니스 규칙을 처리하는 중첩된 서브쿼리(subquery)를 식별합니다.
2.  이를 서술적이고 결과 기반의 이름을 가진 WITH 절로 추출합니다.
3.  기본 로직 블록에서 해당 이름의 CTE를 참조합니다.

**예시:**

```sql
-- 이전 (로직이 숨겨져 있어 정신적 매핑이 필요합니다.)
SELECT order_id
FROM orders
WHERE customer_id IN (SELECT id FROM customers WHERE status = 'active');

-- 이후 (로직이 선형적이고 자체 설명적입니다.)
WITH active_customers AS (
    SELECT id 
    FROM customers 
    WHERE status = 'active'
)
SELECT order_id
FROM orders
WHERE customer_id IN (SELECT id FROM active_customers);
```

**결과**: 쿼리는 일련의 지침처럼 읽힙니다. 미래의 여러분(또는 동료)은 괄호를 따라가지 않고도 종속성(dependencies)을 즉시 파악할 것입니다.

### STEP 2. 전체 스토리를 말해주는 컬럼 별칭 사용하기

**워크플로우:**

1.  SELECT 문이나 복잡한 계산에서 모호한 컬럼(column) 이름을 찾습니다.
2.  데이터의 상태를 명시하는 의도를 드러내는 별칭(alias)으로 바꿉니다.
3.  테이블(table) 간의 관계를 명확히 하기 위해 JOIN 조건에 적용합니다.

**예시:**

```sql
-- 이전 (모호한 계산)
SELECT
    (price * quantity) - discount AS final_amount
FROM orders;

-- 이후 (명시적인 비즈니스 로직)
SELECT
    (price * quantity) - discount AS total_revenue_net_of_discounts
FROM orders;
```

**결과**: 비즈니스 분석가(Business Analyst)는 데이터 사전(data dictionary)을 교차 참조할 필요 없이 쿼리 결과를 활용할 수 있습니다. '무엇'이 이름에 정의되어 있기 때문에 디버깅(debugging)이 즉시 가능해집니다.

### STEP 3. 전략적인 공백 및 들여쓰기 강제하기

**워크플로우:**

1.  주요 절(SELECT, FROM, WHERE, GROUP BY)을 새로운 줄로 나눕니다.
2.  계층 구조를 보여주기 위해 JOIN 조건과 하위 로직을 들여쓰기(indent)합니다.
3.  '데이터 준비' 단계(CTE)와 '최종 집계' 단계(Final Aggregation)를 구분하기 위해 빈 줄을 사용합니다.

**예시:**

```sql
-- 이전 (밀집되어 있고 파싱(parsing)이 어렵습니다.)
SELECT o.id, c.name, SUM(p.amount) AS revenue FROM orders o JOIN customers c ON o.customer_id = c.id JOIN payments p ON o.id = p.order_id WHERE o.status = 'completed' AND c.is_active = TRUE GROUP BY o.id, c.name;

-- 이후 (구조화되어 있고 스캔하기 매우 쉽습니다.)
SELECT
    o.id,
    c.name,
    SUM(p.amount) AS lifetime_settled_revenue
FROM orders o
JOIN customers c
    ON o.customer_id = c.id
JOIN payments p
    ON o.id = p.order_id
WHERE
    o.status = 'completed'
    AND c.is_active = TRUE
GROUP BY
    o.id,
    c.name;
```

**결과**: 쿼리가 전문적인 기술 사양처럼 보입니다. 이는 동료들에게 로직이 규율 있고 신뢰할 수 있다는 신호를 보냅니다.

### STEP 4. 가정을 쿼리에 직접 삽입하기

**워크플로우:**

1.  암묵적인 로직(예: '이것은 미국 사용자에게만 적용됨' 또는 '내부 테스트 계정 제외')을 식별합니다.
2.  `-- #`와 같이 눈에 띄는 접두사를 사용하여 비즈니스 규칙을 코드 설명과 구분하는 인라인 주석(inline comment)을 추가합니다.
3.  이러한 메모를 로직이 적용되는 정확한 위치에 배치하고, 헤더(header)에만 두지 않습니다.

**예시:**

```sql
WITH regional_customer_base AS (
    SELECT 
        user_id,
        signup_date
    FROM customers
    WHERE country_code = 'US'
    -- # 비즈니스 규칙: 2025년 세금 규정 업데이트에 따라 영토(PR, GU, VI)는 제외합니다.
    AND state_code NOT IN ('PR', 'GU', 'VI')
)
```

**결과**: 쿼리가 감사(audit)에 대비할 수 있게 됩니다. 이해관계자가 숫자가 왜 변경되었는지 물을 때, 그 이유는 이미 버전 관리(version control) 기록에 문서화되어 있습니다.
