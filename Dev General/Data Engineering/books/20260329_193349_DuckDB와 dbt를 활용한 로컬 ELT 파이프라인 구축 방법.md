# DuckDB와 dbt를 활용한 로컬 ELT 파이프라인 구축 방법

노트북에서 데이터 변환(Data Transformation)을 마스터하기 위한 단계별 가이드 — 클라우드 비용 불필요

---

미디엄(Medium) 회원이 아니더라도 여기에서 기사를 읽을 수 있습니다.

데이터 엔지니어링(Data Engineering)은 종종 신용카드와 클라우드 구독을 필요로 하지만, 최신 데이터 변환(Data Transformation) 기술은 실제로는 로컬 머신(local machine)에서 습득할 수 있습니다.

이 글에서는 분석 엔진(analytical engine)으로 DuckDB와 데이터 변환(data transformation) 도구로 dbt를 사용하여 기능적인 ELT(Extract, Load, Transform) 파이프라인을 구축할 것입니다.

다룰 내용은 다음과 같습니다:

*   **아키텍처(Architecture):** '로컬 스택(Local Stack)' 접근 방식 이해하기.
*   **데이터 수집(Ingestion):** Python을 사용하여 원시 비즈니스 데이터를 DuckDB로 로드하기.
*   **구성(Configuration):** dbt-duckdb를 설정하여 로컬 데이터베이스(local database)에 연결하기.
*   **데이터 변환(Transformation):** 영업 및 재무 팀을 위한 특정 데이터 마트(Data Marts) 구축하기.

자, 시작해봅시다!

## DuckDB와 dbt를 사용하는 이유

코딩을 시작하기 전에 먼저 우리가 사용할 도구들을 이해해봅시다. DuckDB는 데이터 크기가 100GB에서 몇 TB 정도일 경우 스노우플레이크(Snowflake)와 유사한 강력한 기능을 제공할 수 있습니다. 클라우드 구성(cloud configuration)을 거칠 필요 없이, 노트북에서 로컬(locally)로 데이터를 엄청나게 빠르게 처리할 수 있습니다.

dbt를 사용하면 버전 관리(version control), 의존성 관리(dependency management), 자동화된 테스트(automated testing)와 같은 소프트웨어 엔지니어링(software engineering) 모범 사례를 SQL 코드에 적용하여, 스크립트 폴더를 신뢰할 수 있고 문서화된 전문적인 데이터 변환(data transformation) 레이어(layer)로 전환할 수 있습니다.

다음으로, 이 두 도구를 활용하여 커피숍을 지원하는 방법을 알아보겠습니다.

## 우리의 ELT 데이터 파이프라인

당신은 성장하는 커피숍 체인의 데이터 엔지니어(Data Engineer)로 고용되었습니다. 현재 그들의 데이터 저장 방식은 체계적이지 않습니다.

*   판매 데이터(Sales Data)는 POS(Point-of-Sale) 시스템에서 CSV 파일로 내보내져 있습니다.
*   예산 데이터(Budget Data)는 재무 부서에서 관리하는 엑셀(Excel) 시트에 있습니다.

하지만 이들은 여전히 수동으로 행(rows)을 마스터 스프레드시트(master spreadsheet)에 복사하여 붙여넣고 있습니다. 이는 느리고, 오류 발생 가능성이 높으며, 확장 불가능합니다(unscalable).

따라서 당신은 여러 소스에서 데이터를 추출(extract)하고, 중앙 집중식 데이터베이스(centralized database, DuckDB)에 데이터를 로드(load)하며, DBT를 사용하여 원시 데이터(raw data)를 영업 및 재무 팀을 위한 깨끗한 '데이터 마트(Data Marts)'로 변환(transformation)하는 ELT 파이프라인을 구축하기로 결정합니다.

## 1단계: 추출(Extract) 및 로드(Load)

첫 번째로, 파일에 있는 데이터를 DuckDB로 가져와야 합니다.

**1-1: 프로젝트 폴더를 생성하고 필요한 라이브러리 설치**

```bash
pip install duckdb dbt-duckdb pandas
```

**1-2: load.py라는 파일을 생성합니다. 이 스크립트는 데이터베이스를 생성하고 원시 파일을 DuckDB로 로드할 것입니다.**

```python
import duckdb

def load_data():
    # Connect to a persistent DuckDB file
    # If the file doesn't exist, this creates it
    con = duckdb.connect('coffeeshop.duckdb')

    print("🚀 Starting ELT Process...")

    # 1. create a schema for raw data
    con.sql("CREATE SCHEMA IF NOT EXISTS raw;")

    # 2. Load Sales Data (CSV)
    # read_csv_auto is a DuckDB magic function that infers types
    print("... Loading Sales Data")
    con.sql("""
        CREATE OR REPLACE TABLE raw.source_sales AS 
        SELECT * FROM read_csv_auto('data/sales_data.csv');
    """)

    # 3. Load Budget Data (CSV/Excel converted to CSV for simplicity)
    print("... Loading Budget Data")
    con.sql("""
        CREATE OR REPLACE TABLE raw.source_budget AS 
        SELECT * FROM read_csv_auto('data/budget_data.csv');
    """)
    
    # Validation check - to see if data is loaded into database successfully
    count = con.sql("SELECT count(*) FROM raw.source_sales").fetchone()
    print(f"✅ Loaded {count[0]} sales records.")
    
    con.close()

if __name__ == "__main__":
    load_data()
```

모든 원시 데이터(source data)는 프로젝트 루트 폴더 내의 `data` 폴더에 저장됩니다.
`duckdb.connect('coffeeshop.duckdb')`: 이 코드를 실행하면 프로젝트 폴더에 `coffeeshop.duckdb`라는 파일이 생성되며, 이것이 바로 여러분의 서버리스 데이터베이스(serverless database)입니다. 정말 쉽죠?
`read_csv_auto`: 이것은 DuckDB의 강력한 기능입니다. CSV 파일을 스캔하여 컬럼(column)이 정수(integer), 날짜(date), 문자열(string)인지 자동으로 추론하고 테이블 정의를 생성합니다. DuckDB는 파일을 직접 읽어 데이터베이스로 변환할 수 있습니다.

## 2단계: dbt 설정 및 구성(Configuration)

이제 모든 데이터가 `coffeeshop.duckdb`에 저장되었습니다. 다음으로 dbt를 사용할 차례입니다. 먼저 dbt 프로젝트를 설정해봅시다.

**2-1: 터미널에서 dbt 프로젝트 초기화:**

```bash
dbt init coffee_analytics
```

데이터베이스 어댑터(database adapter)를 묻는 메시지가 나타나면 `duckdb`를 선택합니다.

**2-2: profiles.yml 설정**

이 파일은 일반적으로 홈 디렉토리(`~/.dbt/`)에 있지만, 프로젝트 내에서도 구성할 수 있습니다. `profiles.yml`은 dbt가 우리의 DuckDB 데이터베이스에 연결하도록 지시합니다.

```yaml
coffee_analytics:
  target: dev
  outputs:
    dev:
      type: duckdb
      # This path must point to the yourdatabase
      path: 'absolute/full/path/to/your/project/coffee.duckdb'
      threads: 1
```

**2-3: 소스(Sources) 정의**

dbt 프로젝트에서 다음과 같은 폴더 구조를 볼 수 있습니다.

```
coffee_analytics/
├── dbt_project.yml
├── profiles.yml
└── models/
    ├── marts/
    │   ├── finance_variance.sql   <-- 재무팀을 위한 최종 결과
    │   └── sales_performance.sql  <-- 영업팀을 위한 최종 결과
    ├── staging/
    │   ├── stg_budget.sql         <-- 정리된 원시 예산 데이터
    │   └── stg_sales.sql          <-- 정리된 원시 판매 데이터
    └── sources.yml                <-- 원시 테이블(source_sales, source_budget) 정의
```

다음으로, `models` 폴더로 이동하여 `sources.yml`이라는 파일을 생성합니다. 이 파일은 DBT에게 원시 데이터(raw data)가 어디에 있는지(스키마와 테이블) 알려줍니다.

```yaml
version: 2

sources:
  - name: bean_raw
    # Our schema is raw - can be found in step 1
    schema: raw # <-- 이것은 dbt에게 데이터베이스(DuckDB)의 실제 스키마 이름을 알려줍니다.
    tables:
       # We have two tables - source_sales and source_budget
      - name: source_sales
      - name: source_budget
```

`bean_raw`는 SQL에서 사용될 `source_name`을 정의합니다.

## 3단계: 변환(Transformation) (데이터 마트)

가장 흥미로운 부분입니다. 이제 DBT로 데이터 마트(Data Marts)를 생성할 것입니다.

일반적으로 두 가지 모델 계층(layer)을 생성합니다.

*   **스테이징(Staging):** 컬럼 이름(column names)을 정리하고 데이터 타입(data types)을 수정합니다.
*   **마트(Marts):** 비즈니스 질문에 답하기 위해 데이터를 조인(join)합니다.

### 3-1: 스테이징 레이어(Staging Layer)

`models` 폴더 안에 `staging`이라는 폴더를 생성하고, 이 새로 생성된 폴더 안에 두 개의 파일을 만들 것입니다.

**models/staging/stg_sales.sql**

```sql
select
    order_id,
    store_id,
    -- Convert date string to actual date object
    strptime(transaction_date, '%Y-%m-%d') as transaction_date,
    product_name as product,
    quantity,
    unit_price,
    (quantity * unit_price) as total_revenue
from {{ source('bean_raw', 'source_sales') }}
```

**models/staging/stg_budget.sql**

```sql
select
    store_id,
    date_part('month', date) as month,  
    date_part('year', date) as year,    
    target_revenue,
    regional_manager_name,
    is_promotional_month -- (T/F)
from {{ source('bean_raw', 'source_budget') }}
```

### 3-2: 데이터 마트(Data Marts) 레이어

이제 이해관계자(stakeholders)들의 질문에 답해봅시다. `models/marts` 폴더를 생성합니다.

**models/marts/sales_performance.sql**

첫 번째 데이터 마트는 영업팀(Sales team)을 위해 설계되었으며, 매장별 매출(revenue)을 추적할 수 있도록 합니다.

```sql
with sales as (
    select * from {{ ref('stg_sales') }}
)

select
    store_id,
    date_trunc('month', transaction_date) as sales_month,
    sum(quantity) as total_items_sold,
    sum(total_revenue) as gross_revenue
from sales
group by 1, 2
order by 1, 2
```

이 구문은 CTE(Common Table Expression)와 유사하며, 테이블에서 데이터를 선택하는 대신 이전에 생성된 `stg_sales`에서 선택합니다.

**models/marts/finance_variance.sql**

두 번째 데이터 마트는 재무팀(finance team)이 실제 지출을 예산 목표와 비교하는 데 도움을 주기 위함입니다.

```sql
with actuals as (
    select * from {{ ref('sales_performance') }}
),

budgets as (
    select * from {{ ref('stg_budget') }}
)

select
    a.store_id,
    a.sales_month,
    a.gross_revenue as actual_revenue,
    b.target_revenue,
    -- Calculate variance
    (a.gross_revenue - b.target_revenue) as revenue_variance,
    round((a.gross_revenue / b.target_revenue) * 100, 2) as achievement_pct
from actuals a
left join budgets b 
    on a.store_id = b.store_id 
    and month(a.sales_month) = b.month
```

여러분은 이렇게 물을 수 있습니다:

dbt를 사용하지 않고 원시 데이터(raw data)에서 직접 쿼리(query)할 수 있지 않을까요?

DBT를 사용하는 이유는 실제 애플리케이션(application)이 위 예시보다 훨씬 복잡하기 때문입니다. 보고서는 여러 테이블을 집계하고(aggregate different tables) 다른 인덱스(index)를 표시해야 할 수 있습니다.

또한, 재무팀은 매주 결과를 필요로 할 수 있습니다. 따라서 데이터 마트(Data Mart)를 생성하는 것은 비즈니스 요구사항을 명확하게 충족시킬 수 있습니다.

## 4단계: 파이프라인 실행

이제 모든 준비가 끝났으니, 전체 흐름을 실행할 시간입니다. 터미널(terminal)에서 프로젝트 루트 폴더(root folder)에 있는지 확인해 주세요.

1.  `python load.py`를 실행하여 모든 원시 데이터(source data)를 데이터베이스(database)에 로드합니다. 그러면 `loaded 5000 sales records`와 유사한 출력을 볼 수 있습니다.
2.  `dbt build`를 실행합니다. 이 명령어는 여러분이 구성한 모델(models)과 모든 테스트(tests)를 실행할 것입니다. DBT가 DuckDB 파일 내에 뷰(views)와 테이블(tables)을 직접 생성하는 것을 볼 수 있습니다.
3.  결과를 확인합니다. DuckDB CLI(Command Line Interface)를 사용하거나 Python 스크립트를 생성하여 최종 마트(final mart)를 쿼리함으로써 데이터를 확인할 수 있습니다:

    **quick_check.py**

    ```python
    import duckdb

    con = duckdb.connect('beanthere.duckdb')
    # Querying the dbt model (which is now a table/view in the DB)
    df = con.sql("SELECT * FROM finance_variance WHERE achievement_pct < 100").df()
    print(df)
    ```

결과를 얻을 수 있다면, DBT로 데이터를 성공적으로 변환(transform)한 것입니다.

## 결론

우리는 단 하나의 클라우드 리소스(cloud resource)도 사용하지 않고 ELT 파이프라인을 구축했습니다. Python으로 원시 데이터를 수집(ingested)하고, DuckDB에 저장했으며, dbt를 사용하여 깔끔하고 신뢰할 수 있는 데이터 마트(Data Marts)를 설계했습니다.

모든 회사가 클라우드 기반 솔루션(cloud-based solution)을 필요로 하는 데이터 볼륨(data volumes)을 다루는 것은 아닙니다. 이러한 경우 DuckDB는 프로덕션(production) 환경에서 효과적으로 활용될 수 있습니다.

DuckDB와 dbt와 같은 경량 도구(lightweight tools)를 선택함으로써, 코드만큼이나 민첩한(agile) 데이터 스택(data stack)을 유지할 수 있습니다. 이러한 민첩성(agility)은 데이터를 기반으로 한 의사 결정(data-driven decisions)을 효과적으로 내리는 데 매우 중요합니다.
