# JSON 대 TOON — 구조화된 입력의 새로운 시대가 열리는가?

## 구조화된 입력이 그 어느 때보다 중요한 이유

프롬프트(prompt)는 점점 길어지고 AI 모델은 더욱 강력해지는 요즘, 한 가지 의문이 끊임없이 제기됩니다. 바로 비용과 처리 시간을 어떻게 낮게 유지할 수 있을까 하는 것입니다.

LLM(대규모 언어 모델)과 프로그래밍 방식으로 작업할 때, 구조화된 출력은 이미 표준적인 접근 방식이 되었습니다. AI에게 특정 형식(예: JSON)으로 응답하도록 지시할 수 있습니다. 모델 스키마(model schema)를 정의하고 각 필드의 의미를 이상적으로 설명함으로써, AI는 문맥을 이해하고 '최선을 다해'(또는 LLM이 그렇게 할 수 있는 가장 가까운 방식으로) 출력을 채우려고 노력합니다.

이는 AI 결과물을 다루는 것을 그 어느 때보다 쉽게 만들었습니다.

하지만 입력은 어떻습니까?

출력을 깔끔하고 구조화할 수 있음에도 불구하고, 대다수는 여전히 방대한 JSON, YAML 또는 일반 텍스트 데이터셋을 프롬프트에 그대로 넣고 있습니다. 이는 느리고 비용이 많이 들 뿐만 아니라 토큰 효율성(token-efficiency) 측면에서도 매우 비효율적입니다.

따라서 이러한 문제를 해결할 새로운 형식이 등장하는 것은 시간 문제였습니다.

그리고 여기 TOON이 등장합니다! 🎉

## TOON 소개 — JSON의 토큰 친화적인 사촌

TOON은 JSON과 CSV 사이에 있는 새로운 파일 형식입니다. 여전히 사람이 읽을 수 있으면서도, LLM 및 토큰화 효율성(tokenization efficiency)에 최적화되어 있습니다. 개발자들에 따르면, 이는 토큰 수를 30~60% 줄일 수 있으며, 토큰 가격 책정 방식을 고려할 때 상당한 비용 절감으로 이어질 수 있습니다.

TOON을 흥미롭게 만드는 요소들은 다음과 같습니다:

💸 **토큰 효율성**: JSON보다 토큰 수가 일반적으로 30~60% 적습니다.
🤿 **LLM 친화적인 가드레일(guardrails)**: 명시적인 길이와 필드를 통해 유효성 검사(validation)가 가능합니다.
🍱 **최소한의 구문**: 불필요한 구두점(중괄호, 대괄호, 대부분의 따옴표)을 제거합니다.
📐 **들여쓰기 기반 구조**: YAML처럼 중괄호 대신 공백을 사용하여 구조를 나타냅니다.
🧺 **테이블형 배열**: 키(key)를 한 번 선언하고 데이터를 행(row)으로 스트리밍합니다.

JSON
```json
[
  {"id": 1, "name": "Alice", "department": "Engineering", "salary": 120000},
  {"id": 2, "name": "Bob", "department": "Marketing", "salary": 95000},
  {"id": 3, "name": "Charlie", "department": "Engineering", "salary": 110000}
]
```

TOON
```
[3]{Id,Name,Department,Salary}:
  1,Alice,Engineering,120000
  2,Bob,Marketing,95000
  3,Charlie,Engineering,110000
```

자세히 보면, YAML이 CSV와 만나 커피를 마시고 구조화된 아기를 함께 키우기로 결정한 것 같은 느낌입니다.

개발자들이 공개한 벤치마크(benchmark) 결과는 이미 인상적입니다:
👉 Toon 벤치마킹 및 주요 기능

## 그렇다면 왜 관심을 가져야 할까요?

챗봇, AI 기반 코드 생성, 다단계 워크플로(workflow) 등 LLM에 구조화된 데이터를 정기적으로 공급하는 시스템을 구축한다면, TOON은 프롬프트 크기를 크게 줄일 수 있습니다.

단순히 비용 문제만은 아닙니다(물론 토큰 사용량을 약 50% 절감하는 것은 결코 나쁘지 않습니다).

속도 또한 중요합니다. 토큰 수가 적으면 추론(inference) 속도가 빨라지고, 특히 실시간 시스템이나 스트리밍 API를 사용할 때 잠재적으로 지연 시간(latency)을 줄일 수 있습니다.

가장 좋은 점은 이미 여러 언어에서 사용할 수 있다는 것입니다(다음 포함):

🟦 .NET: ToonSharp
🐍 Python: python-toon / pytoon
🦫 Go: gotoon

## 실제 평가

그래서 저는 TOON이 JSON과 비교하여 실제로 어떻게 작동하는지 확인하기 위해 작은 벤치마크 도구를 만들었습니다.

직원들의 간단한 데이터셋을 사용하여 GPT에게 데이터를 분석하고 부서별 평균 급여를 계산하도록 요청했습니다. 이 도구는 프롬프트 크기, 완성 토큰(completion token) 수, 그리고 전체 응답 시간을 측정합니다.

다음은 제가 실행한 결과입니다:

```
+---------------------------------------------------------------------------+
| Type         | Prompt Tokens   |  Completions Tokens  |  Duration         |
|--------------+-----------------+----------------------+-------------------|
| JSON         | 1344            |  3475                |  00:00:28.3932721 |
| TOON         | 589             |  2928                |  00:00:23.4953152 |
+---------------------------------------------------------------------------+
```

이는 프롬프트 토큰에서 약 56% 감소와 함께, 모델의 출력 품질은 동일하게 유지하면서 5초의 눈에 띄는 속도 향상을 보여줍니다.

결론적으로, TOON은 이론적으로만 좋은 것이 아닙니다. 실제로는 더 빠르고, 더 저렴하며, 여전히 읽기 쉽습니다.

## 마무리 생각

우리가 어떻게 한 바퀴를 돌아 다시 제자리로 왔는지 보는 것은 정말 흥미롭습니다. 수년 동안 AI에게 구조화된 데이터를 출력하도록 가르쳤고, 이제는 AI의 언어를 더 잘 이해하도록 입력 데이터를 최적화하고 있습니다.

TOON이 새로운 표준이 될지, 아니면 단지 또 하나의 영리한 틈새 아이디어로 남을지는 알 수 없지만, 분명히 주목할 가치가 있습니다. 특히 성능, 비용, 효율성을 중요하게 생각한다면 말이죠(솔직히 누가 그렇지 않겠습니까?).
