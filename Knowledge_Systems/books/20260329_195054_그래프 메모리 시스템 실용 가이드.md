## 1. 그래프 메모리 시스템 소개

AI 에이전트(AI agents)는 조직의 업무 방식을 혁신하고 있지만, 근본적인 문제에 직면해 있습니다. 바로 '메모리'입니다. 에이전트에게 지난주 대화 내용을 물어보면 종종 멍한 반응을 보입니다. 여러 에이전트를 조직 전체에 배포해도 학습한 내용을 공유할 수 없습니다. 문제가 발생했을 때, 에이전트가 특정 결정을 내린 이유를 설명하는 감사 추적(audit trail)이 없습니다.

미디엄 회원이 아니시라면, [미디엄 친구 링크](Medium Friend Link)를 확인하세요.

그래프 메모리 시스템(Graph memory systems)은 에이전트 컨텍스트(agent context)를 그래프 데이터베이스(graph databases)에 저장하여 이러한 문제를 해결합니다. 이를 통해 풍부한 관계 모델링(relationship modeling)과 효율적인 검색(retrieval)이 가능해집니다. 단순한 키-값 저장소(key-value stores)나 벡터 데이터베이스(vector databases)와 달리, 그래프 메모리는 실제 지식의 상호 연결된(interconnected) 특성을 포착합니다.

### 1.1 에이전트 메모리의 세 가지 유형

효과적인 에이전트 메모리는 통합된 그래프(unified graph) 내에서 함께 작동하는 세 가지 명확한 유형을 필요로 합니다.

### 1.2 에이전트 메모리에 그래프 데이터베이스를 사용하는 이유

업계는 AI 에이전트 메모리에 그래프 데이터베이스가 필수적이라는 점을 점점 더 인식하고 있습니다. Zep, Cognee, Mem0 등 여러 기업들은 Neo4j를 사용하여 대화 및 엔티티(entity) 지식을 저장하는 인상적인 시스템을 구축했습니다. 그래프 데이터베이스의 강점:

*   **관계 모델링(Relationship Modeling):** 엔티티와 그 연결을 자연스럽게 표현
*   **다중 홉 추론(Multi-hop Reasoning):** 관계를 탐색하여 연결된 정보 찾기
*   **출처 추적(Provenance Tracking):** 정보의 출처와 결정 과정 이해
*   **시계열 쿼리(Temporal Queries):** 시간 경과에 따른 사실 및 관계 변화 추적

## 2. Neo4j-Agent-Memory

Neo4j-agent-memory는 Neo4j Labs 프로젝트로, AI 에이전트를 위한 완전한 그래프 네이티브(graph-native) 메모리 시스템을 제공합니다. 이 시스템은 Neo4j의 강력한 그래프 데이터베이스를 기반으로 단기(short-term), 장기(long-term), 추론(reasoning) 메모리를 단일 연결 그래프(single connected graph)로 통합합니다.

**엔티티 추출 파이프라인(Entity Extraction Pipeline)**

*   **spaCy (~5ms, 무료):** 빠른 통계적 개체명 인식(NER, Named Entity Recognition)으로 일반적인 엔티티 파악
*   **GLiNER2 (~50ms, 무료):** 도메인별 스키마(schemas)를 사용하는 제로샷 개체명 인식(Zero-shot NER)
*   **LLM 폴백(fallback) (gpt-4o-mini) (~500ms, 유료):** 복잡한 경우에 대한 최고 정확도

이 파이프라인은 또한 중복 제거("Brian"이 "Brian Chesky"와 동일한가?), 관계 추출(relationship extraction) 및 선택적 위키백과 정보 보강(Wikipedia enrichment)을 처리합니다.

```python
from neo4j_agent_memory.extraction import ExtractorBuilder

extractor = (
    ExtractorBuilder()
    .with_spacy(model="en_core_web_sm")
    .with_gliner_schema("podcast", threshold=0.5)  # 8 domain schemas available
    .with_llm_fallback(model="gpt-4o-mini")
    .with_merge_strategy("confidence")
    .build()
)
result = await extractor.extract(
    "Brian Chesky founded Airbnb in San Francisco in 2008"
)

# 추출 결과: 인물(Brian Chesky), 조직(Airbnb),
#           장소(San Francisco), 사건(설립, 2008)
```

### 2.3 빠른 시작 예제
```python
import asyncio
from pydantic import SecretStr
from neo4j_agent_memory import MemoryClient, MemorySettings, Neo4jConfig

async def main():
    # 설정 구성
    settings = MemorySettings(
        neo4j=Neo4jConfig(
            uri='bolt://localhost:7687',
            username='neo4j',
            password=SecretStr('your-password'),
        )
    )
    
    # 메모리 클라이언트 사용
    async with MemoryClient(settings) as memory:
        # 대화 메시지 저장
        await memory.short_term.add_message(
            session_id='user-123',
            role='user',
            content='Hi, I am John and I love Italian food!'
        )
        
        # 선호도 추가
        await memory.long_term.add_preference(
            category='food',
            preference='Loves Italian cuisine',
            context='Dining preferences'
        )
        
        # 관련 메모리 검색
        preferences = await memory.long_term.search_preferences(
            'restaurant recommendation'
        )
        
        # LLM 프롬프트(prompt)를 위한 결합된 컨텍스트 가져오기
        context = await memory.get_context(
            'What restaurant should I recommend?',
            session_id='user-123'
        )
        print(context)

asyncio.run(main())
```

## 3. Zep과 Graphiti

Zep은 AI 에이전트를 위한 상용 메모리 레이어(memory layer) 서비스로, 메모리 검색 벤치마크(memory retrieval benchmarks)에서 현재 최첨단 시스템을 능가하는 성능을 보여줍니다. 그 핵심에는 Graphiti가 있습니다. 이는 오픈 소스(open-source)로, 시간 정보를 인식하는 지식 그래프 엔진(knowledge graph engine)이며, 구조화되지 않은 대화 데이터와 구조화된 비즈니스 데이터를 과거 관계를 유지하면서 동적으로 통합합니다.

Enter를 누르거나 클릭하여 전체 크기 이미지 보기
[https://www.getzep.com/](https://www.getzep.com/)

### 3.1 Graphiti 아키텍처(Architecture)

Graphiti는 시간에 따라 엔티티 간의 복잡하고 진화하는 관계를 나타내는 동적이고 시간 정보를 인식하는 지식 그래프(Knowledge Graphs)를 구축합니다. 배치 처리(batch processing)에 의존하는 기존 RAG(Retrieval-Augmented Generation) 방식과 달리, Graphiti는 다음을 제공합니다.

*   **실시간 증분 업데이트(Real-Time Incremental Updates):** 배치 재계산 없이 새로운 데이터를 즉시 통합
*   **양방향 시간 데이터 모델(Bi-Temporal Data Model):** 이벤트 발생 시간 및 데이터 수집 시간 명시적 추적
*   **효율적인 하이브리드 검색(Efficient Hybrid Retrieval):** 의미론적 임베딩(semantic embeddings), 키워드(BM25), 그래프 탐색(graph traversal) 결합
*   **사용자 정의 엔티티 정의(Custom Entity Definitions):** Pydantic 모델을 통한 유연한 온톨로지(ontology) 생성

### 3.3 Graphiti 빠른 시작
```python
from graphiti_core import Graphiti
from graphiti_core.nodes import EpisodeType

# Graphiti 초기화
graphiti = Graphiti('bolt://localhost:7687', 'neo4j', 'password')

# 인덱스(indices) 및 제약 조건(constraints)으로 그래프 데이터베이스 초기화
await graphiti.build_indices_and_constraints()

# 에피소드(텍스트 또는 구조화된 JSON) 추가
await graphiti.add_episode(
    name='User Conversation 1',
    episode_body='User: I love Italian food!',
    source=EpisodeType.message,
    source_description='Chat message',
    group_id='user-123'
)

# 하이브리드 검색(hybrid search)을 사용하여 관계 검색
results = await graphiti.search(
    'What food does the user like?',
    group_ids=['user-123']
)

# 그래프 거리(graph distance)를 사용하여 결과 재랭킹(Rerank)
reranked = await graphiti.rerank_results(
    'What food does the user like?',
    results
)
```

```
  점수: 0.923 - 사용자는 멕시코 음식을 좋아합니다.
  점수: 0.456 - 사용자는 이전에 이탈리아 음식을 좋아했습니다 (무효화됨).
  재랭킹 후 상위 결과: 사용자는 멕시코 음식을 좋아합니다.
```

## 4. Cognee

Cognee는 AI 에이전트에게 컨텍스트 인식(context-aware) 메모리를 제공하는 오픈 소스 메모리 시스템입니다. 이 시스템은 데이터의 임베딩(embeddings)으로 구동되는 쿼리 가능한 지식 그래프(queryable knowledge graphs)를 생성하여 검색 작업(retrieval tasks)에서 최대 92.5%의 정확도를 달성합니다. Cognee는 모든 것을 ECL(추출-인지-로드, Extract-Cognify-Load) 파이프라인(pipelines)으로 구성합니다.

## 5. LangMem과 Mem0
### 5.1 LangMem

LangMem은 LangGraph 애플리케이션(applications)을 위해 특별히 설계된 메모리 시스템입니다. 이 시스템은 여러 LLM 호출을 통해 쿼리당 토큰(tokens)을 최소화하고 가장 관련성 높은 메모리 스니펫(memory snippet)만 반환합니다. 이 접근 방식은 비용 최적화에 현명하지만, 토큰 효율성을 위해 여러 API 호출을 감수합니다.

Enter를 누르거나 클릭하여 전체 크기 이미지 보기
[메모리 유형](https://langchain-ai.github.io/langmem/concepts/conceptual_guide/#memory-types)

```python
# LangGraph를 사용한 LangMem 예제
from langmem import MemoryManager

# 메모리 관리자 초기화
memory = MemoryManager(
    model='gpt-4o-mini',
    extract_prompt='Extract user preferences'
)

# 메모리 저장
await memory.store(
    session_id='user-123',
    content='User prefers dark mode'
)

# 관련 메모리 검색
memories = await memory.retrieve(
    session_id='user-123',
    query='What are user preferences?'
)


print(f'Retrieved {len(memories)} memories')
for m in memories:
print(f' - {m.content}')
```
**출력:**
```
  메모리 저장됨
  1개의 메모리 검색됨
  - 사용자는 다크 모드(dark mode)와 큰 글꼴(large fonts)을 선호합니다.
  참고: LangMem은 3번의 LLM 호출을 했지만, 전체 대화 기록(~500 토큰) 대신
  약 50개의 토큰만 반환했습니다.
```

### 5.2 Mem0

Mem0는 복잡한 시간 기반 아키텍처(temporal architectures)를 필요로 하지 않고 그래프 강화(graph-enhanced) 메모리를 제공하는 선도적인 메모리 시스템입니다. 이 시스템은 벡터, 그래프, 키-값 저장소를 결합한 하이브리드 아키텍처(hybrid architecture)를 제공하며, 적응형 메모리 업데이트(adaptive memory updates)와 다단계 회수(multi-level recall) 기능을 갖추고 있습니다.

Enter를 누르거나 클릭하여 전체 크기 이미지 보기
[Mem0g의 그래프 기반 메모리 아키텍처](https://arxiv.org/abs/2504.19413)

#### 5.2.1 주요 기능

*   **다단계 메모리(Multi-Level Memory):** 사용자, 세션, 에이전트 수준의 메모리 계층(hierarchy)
*   **그래프 메모리 지원(Graph Memory Support):** 엔티티 관계를 위한 선택적 그래프 계층(graph layer)
*   **프레임워크 독립적(Framework Agnostic):** API를 통해 모든 에이전트 프레임워크(framework)와 연동 가능
*   **관리형 서비스(Managed Service):** 자체 호스팅(self-hosted) 또는 관리형 클라우드 서비스(managed cloud service)로 이용 가능

#### 5.2.2 Mem0 빠른 시작
```python
from mem0 import MemoryClient

# 클라이언트 초기화
client = MemoryClient(api_key='your-api-key')

# 메모리 저장
client.add(
    messages=[{
        'role': 'user',
        'content': 'I prefer vegetarian food'
    }],
    user_id='user-123'
)

# 메모리 검색
memories = client.search(
    query='What food preferences?',
    user_id='user-123'
)

print(memories)
```
**출력:**
```
  메모리 추가됨: mem-abc123-def456
  2개의 메모리 발견
  점수: 0.945 - 사용자는 채식주의(vegetarian) 음식을 선호합니다.
  점수: 0.823 - 사용자는 식단에서 글루텐(gluten)을 피합니다.
  참고: Mem0는 단일 메시지에서 두 개의 개별 사실을 자동으로 추출하여
  사용자 엔티티에 연결했습니다.
```

### 7.2 메모리를 사용한 환자 대화
```python
async def patient_interaction(memory, patient_id):
    # 단기 메모리(SHORT-TERM): 대화 저장
    await memory.short_term.add_message(
        session_id=patient_id,
        role='user',
        content='I have been feeling dizzy lately, especially in the morning'
    )
    
    # 장기 메모리(LONG-TERM): 엔티티 추출 및 저장
    patient = await memory.long_term.add_entity(
        name='Sarah Johnson',
        entity_type='PERSON',
        subtype='PATIENT',
        description='Patient reporting dizziness symptoms'
    )
    
    # 장기 메모리(LONG-TERM): 증상을 사실로 저장
    await memory.long_term.add_fact(subject='Sarah Johnson',
        predicate='has_symptom',
        obj='dizziness',
        valid_from=datetime.now()
    )
    
    # 추론 메모리(REASONING): 진단 과정 추적
    trace = await memory.reasoning.start_trace(
        session_id=patient_id,
        task='Diagnose dizziness symptoms'
    )
    
    await memory.reasoning.add_step(
        trace.id,
        thought='Dizziness in morning could be related to blood pressure',
        action='check_medication_history'
    )
    
    await memory.reasoning.record_tool_call(
        trace.id,
        tool_name='medication_lookup',
        arguments={'patient': 'Sarah Johnson'},
        result={'medications': ['Lisinopril']},
        status=ToolCallStatus.SUCCESS
    )
    
    await memory.reasoning.complete_trace(
        trace.id,
        outcome='Recommend blood pressure monitoring',
        success=True
    )
```
**출력:**
```
  메시지 저장됨: msg-abc123
  엔티티 생성됨: ent-def456
  사실 기록됨: fact-ghi789
  트레이스(trace) 시작됨: trace-jkl012
  트레이스 완료됨
  
  상호작용 후 그래프 상태:
    - 메시지 노드 1개
    - 인물 엔티티 1개 (사라 존슨)
    - 증상 사실 1개 (어지러움)
    - 2단계의 추론 트레이스 1개
    - 도구 호출 1개 기록됨
```

### 7.3 컨텍스트 검색
```python
async def get_patient_context(memory, patient_id, query):
    # 모든 메모리 유형에서 결합된 컨텍스트 가져오기
    context = await memory.get_context(
        query,
        session_id=patient_id
    )
    # 관련 증상 검색
    symptoms = await memory.long_term.search_entities(
        'dizziness symptoms'
    )
    
    # 대화 기록 가져오기
    conversation = await memory.short_term.get_conversation(patient_id)
    
    # 유사한 과거 사례 찾기
    similar_cases = await memory.reasoning.get_similar_traces(
        'dizziness diagnosis'
    )
    
    return {
        'context': context,
        'symptoms': symptoms,
        'conversation': conversation,
        'similar_cases': similar_cases
    }
```
**출력:**
```
  === 결합된 컨텍스트 ===
  환자 사라 존슨은 특히 아침에 어지럼증을 호소합니다.
  현재 복용 중인 약물은 리시노프릴(혈압약)입니다.
  유사한 과거 사례들은 혈압 모니터링을 권장합니다.
  
  === 관련 증상 (2개) ===
    - 어지럼증
    - 현기증
  
  === 대화 기록 (1개 메시지) ===
    [사용자]: 최근 어지러움을 느끼고 있습니다. 특히...
  
  === 유사 사례 (3개) ===
    - 어지럼증 증상 진단: 혈압 모니터링 권장
    - 아침 어지럼증 평가: 수분 섭취 및 혈압 확인 권장
```

## 8. 모범 사례
### 8.1 올바른 시스템 선택

*   **상용 엔터프라이즈(enterprise) 사용의 경우:** SLA(Service Level Agreement)가 포함된 관리형 서비스(managed service)에는 **Zep**을, 유연한 배포를 위해서는 **Mem0**을 선택하십시오.
*   **오픈 소스(Open Source) 유연성을 위해서는:** 시간 지식 그래프에는 **Graphiti**를, ECL 파이프라인에는 **Cognee**를 선택하십시오.
*   **Neo4j 통합의 경우:** 네이티브(native) 세 가지 메모리 유형 아키텍처에는 **neo4j-agent-memory**를 선택하십시오.
*   **LangGraph 프로젝트의 경우:** 긴밀한 프레임워크 통합을 위해서는 **LangMem**을 선택하십시오.

### 8.2 구현 팁

1.  **로컬에서 시작:** 그래프 데이터베이스를 위한 도커 컨테이너(Docker containers)를 사용하여 로컬 개발부터 시작하십시오.
2.  **엔티티 추출 파이프라인:** 프로덕션 환경에서는 다단계 추출(spaCy + GLiNER + LLM 폴백)을 사용하십시오.
3.  **적극적인 가지치기:** 그래프 성능 유지를 위해 오래된 메모리를 제거하십시오.
4.  **조기 시각화:** 그래프 시각화를 사용하여 메모리 구조를 디버그(debug)하고 이해하십시오.
5.  **도구 호출 모니터링:** 에이전트 의사 결정 과정을 이해하기 위해 추론 트레이스(trace)를 추적하십시오.

### 8.3 일반적인 문제점

*   **메모리 비대화(Memory Bloat):** 가지치기를 하지 않으면 그래프가 다루기 어려울 정도로 커질 수 있습니다. 보존 정책(retention policies)을 구현하십시오.
*   **엔티티 중복:** 중복 노드(nodes)를 피하기 위해 적절한 중복 제거 전략(deduplication strategies)을 구성하십시오.
*   **속도 제한(Rate Limiting):** LLM 기반 추출은 속도 제한에 도달할 수 있습니다. 로컬 모델을 사용하거나 백오프(backoff)를 구현하십시오.
*   **스키마 드리프트(Schema Drift):** 업데이트 전반에 걸쳐 일관된 엔티티 유형과 관계를 유지하십시오.

## 9. 결론

그래프 메모리 시스템은 AI 에이전트 기능에 있어 상당한 발전을 의미합니다. 대화, 엔티티, 추론 트레이스를 통합된 그래프 구조에 저장함으로써 이러한 시스템은 에이전트가 다음을 수행할 수 있도록 합니다.

*   **컨텍스트 유지(Maintain Context):** 세션(sessions) 전반에 걸쳐 이전 대화와 사용자 선호도 기억
*   **지식 구축(Build Knowledge):** 비정형 데이터(unstructured data)에서 엔티티를 자동으로 추출하고 연결
*   **결정 설명(Explain Decisions):** 의사 결정 과정을 추론 단계를 통해 추적하여 이유 이해
*   **학습 및 개선(Learn and Improve):** 과거 상호작용 분석을 통해 미래 성능 향상

포괄적인 세 가지 메모리 유형 아키텍처를 제공하는 neo4j-agent-memory를 선택하든, 시간 지식 그래프를 위한 Zep/Graphiti를 선택하든, ECL 파이프라인을 위한 Cognee를 선택하든, 또는 프로덕션(production) 준비가 된 관리형 서비스를 위한 Mem0를 선택하든, 그래프 메모리는 중요한 AI 에이전트 배포의 필수 구성 요소가 되고 있습니다.

AI 에이전트의 미래는 더 나은 모델뿐만 아니라 인간 지식과 경험의 풍부하고 상호 연결된 특성을 포착할 수 있는 더 나은 메모리 시스템에 달려 있습니다.
