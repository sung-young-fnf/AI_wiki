# 벡터 데이터베이스를 버리다: PageIndex로 RAG 성능을 비약적으로 개선한 방법

LLM(대규모 언어 모델)이 사람이 문서를 탐색하는 방식으로 작동하게 했을 때 어떤 일이 일어나는지, 그리고 금융 벤치마크에서 98.7%의 점수를 달성한 방법에 대해 설명합니다.

라이브 데모에서 AI가 200페이지짜리 연간 보고서에서 잘못된 섹션을 자신감 있게 가져오는 것을 지켜볼 때만 느낄 수 있는 특정 종류의 굴욕감이 있습니다.

저는 그것을 두 번 경험했습니다.

두 번째 실패 시점에는 첫 번째 실패의 문제를 이미 "해결"했다고 생각했습니다. 더 나은 청킹(chunking) 전략, 더 스마트한 오버랩, 더 신중하게 조정된 임베딩(embedding) 모델 등 모든 RAG(Retrieval-Augmented Generation) 엔지니어가 겪는 업그레이드 과정을 거치며 다음 조정이 실제로 작동할 것이라고 스스로에게 말했습니다.

하지만 그렇지 않았습니다.

투자자가 2023년 매출 예상에 대해 질문했을 때, 시스템은 회사의 2021년 자본 지출 계획에 대한 단락을 검색했습니다. 두 섹션 모두 유사한 금융 용어를 사용했습니다. 임베딩은 서로 관련이 있다고 생각했지만 실제로는 그렇지 않았습니다.

그때 저는 전체 접근 방식이 근본적으로 잘못되었는지에 대한 의문을 갖기 시작했습니다.

## 벡터 RAG의 핵심적인 거짓

모든 RAG 튜토리얼은 동일한 내용을 가르칩니다. 문서를 500단어 조각으로 청킹하고, 임베딩합니다. 누군가가 질문을 하면 가장 가까운 벡터를 찾습니다. 그 조각들을 LLM에 전달하고 최선을 바랍니다.

이러한 접근 방식은 우아하게 들립니다. 시맨틱 검색(semantic search)이 언어 생성과 만나는 것이죠.

하지만 아무도 언급하지 않는 것이 있습니다. 바로 유사성이 관련성은 아니라는 점입니다. 벡터 데이터베이스는 질문과 시맨틱적으로 가까운 것을 찾을 뿐, 실제로 질문에 답하는 것을 찾지는 않습니다. 약물 위험 요소에 대한 단락과 약물 효능에 대한 단락은 서로 가깝게 임베딩됩니다. 그러나 "회사의 재정적 위험은 무엇인가?"라는 질문에는 둘 중 하나만 답이 됩니다.

다음으로, 청킹 재앙이 있습니다. 200페이지짜리 법률 문서를 500토큰(token) 크기로 나누면 저자가 신중하게 구축한 모든 것이 파괴됩니다. 섹션, 상호 참조, 논리적 흐름이 사라집니다. 남는 것은 조각난 파편들의 묶음일 뿐, 그것들이 어디에 속하고 어떻게 연결되는지에 대한 감각이 전혀 없습니다.

더 스마트한 청킹은 약간 도움이 될 수 있지만, 버린 구조적 컨텍스트를 복구할 수 없다는 핵심 문제를 절대 해결하지 못합니다.

인간 분석가는 이런 식으로 작업하지 않습니다. 목차를 훑어보고, 관련 섹션으로 이동하며, 주변 장의 내용을 인지한 상태로 읽고, 각주를 상호 참조할 것입니다. 문서를 문서답게 탐색하는 것이죠.

우리가 같은 방식으로 작동하는 RAG 시스템을 구축한다면 어떨까요?

## PageIndex: 벡터 데이터베이스 없는 RAG

VectifyAI 팀이 PageIndex를 구축했을 때, 이 질문에 정확히 답했습니다.

핵심적인 통찰력은 듣고 나면 거의 불쾌할 정도로 분명합니다. 문서를 벡터로 변환하고 유사성으로 검색하는 대신, 문서의 계층적 트리 인덱스(본질적으로 기계가 읽을 수 있는 목차)를 구축한 다음, LLM 추론을 사용하여 그 트리를 탐색하여 관련 정보를 찾는 것입니다.

벡터 데이터베이스도, 청킹도, 시맨틱 유사성 점수도 없습니다. 오직 구조화된 인덱싱과 지능적인 탐색만 있을 뿐입니다.

실제 아키텍처는 다음과 같습니다.

## PageIndex 작동 방식: 2단계 프로세스

### 1단계: 인덱스 생성

문서를 PageIndex에 입력하면 문서를 조각내지 않습니다. 대신, 문서의 자연스러운 구조(섹션, 서브섹션, 제목, 논리적 그룹화)를 분석하고, 인간 전문가가 콘텐츠를 정리하는 방식을 반영하는 계층적 인덱스를 구축합니다.

이를 자동으로 생성되고 기계가 파싱할 수 있는 목차라고 생각해보세요. 하지만 훨씬 더 풍부합니다. 트리의 각 노드는 단순히 제목만 갖는 것이 아니라, 해당 브랜치(branch)가 다루는 내용, 어떤 질문에 답할 수 있는지, 그리고 형제 노드 및 부모 노드와 어떻게 관련되는지에 대한 요약을 포함합니다.

100페이지짜리 기술 사양서의 경우, 3~4단계 깊이의 트리를 얻을 수 있습니다. 최상위 레벨은 주요 기능 영역을 다룹니다. 어떤 노드로든 드릴다운하면 점점 더 구체적인 주제를 다루는 하위 노드를 찾을 수 있습니다. 문서의 논리적 계층 구조는 저자가 의도한 그대로 보존됩니다.

이 단계는 문서당 한 번, 초기에 발생하며 인덱스는 저장됩니다. 원본 문서 내용은 손상되지 않고 계속 액세스할 수 있습니다.

### 2단계: 추론 기반 트리 검색

쿼리가 들어오면 PageIndex는 쿼리를 임베딩하여 최근접 이웃 검색을 실행하지 않습니다. 대신, LLM을 사용하여 인덱스 트리를 탐색합니다.

LLM은 최상위 노드를 읽고 "이 질문을 고려할 때, 문서의 어느 브랜치가 가장 답변을 포함할 가능성이 높은가?"라고 추론합니다. 해당 브랜치를 따라 다음 레벨의 노드를 읽고 다시 추론합니다. 관련 있는 특정 문서 섹션을 식별할 때까지 계속 드릴다운합니다.

이것이 모든 것을 바꾸는 부분입니다. PageIndex는 원본 문서에서 해당 완전한 섹션을 조각화 없이 검색하고, 이를 질문에 답변하기 위한 컨텍스트(context)로 사용합니다.

검색은 설계상 설명 가능합니다. LLM이 트리의 어떤 노드를 탐색했는지, 원본 문서의 어떤 페이지에 도달했는지 정확히 확인할 수 있습니다. 블랙박스 같은 코사인 유사성(cosine similarity) 점수는 없습니다. 추론 체인이 투명합니다.

## 시작하기: PageIndex 로컬 실행

실제로 PageIndex를 실행해 봅시다. 설정은 최소한입니다.

### 설치

저장소(repository)를 클론(clone)하고 종속성(dependencies)을 설치합니다.

```bash
git clone https://github.com/VectifyAI/PageIndex.git
cd PageIndex
pip3 install --upgrade -r requirements.txt
```

### 설정

루트 디렉토리에 `.env` 파일을 생성합니다.

```ini
CHATGPT_API_KEY=your_openai_api_key_here
```

### PDF에서 인덱스 실행

이제 문서를 지정합니다. PageIndex가 탁월한 분야이므로 금융 보고서를 예시로 사용해 봅시다.

```bash
python3 run_pageindex.py --pdf_path /path/to/annual_report.pdf
```

마크다운(markdown) 문서의 경우:

```bash
python3 run_pageindex.py --md_path /path/to/technical_spec.md
```

이것이 전체 설정입니다. 구성할 벡터 데이터베이스도, 선택할 임베딩 모델도, 신경 쓸 청킹 매개변수도 없습니다.

## 인덱스는 실제로 어떻게 생겼는가

PageIndex가 문서를 처리하면 다음과 유사한 인덱스 구조를 생성합니다 (단순화된 형태):

```json
{
  "document": "Apple Inc. Annual Report 2023",
  "index": {
    "title": "Apple Inc. Annual Report 2023",
    "summary": "수익, 제품 부문, 위험 및 전략적 전망을 다루는 종합 재무 및 운영 보고서",
    "children": [
      {
        "title": "사업 개요",
        "summary": "회사 설명, 제품 라인 및 시장 포지션",
        "pages": [1, 8],
        "children": [...]
      },
      {
        "title": "재무 성과",
        "summary": "2023 회계연도 수익, 영업 이익, EPS 및 부문별 실적",
        "pages": [45, 72],
        "children": [
          {
            "title": "제품 카테고리별 수익",
            "summary": "iPhone, Mac, iPad, 웨어러블 및 서비스 수익 세부 정보",
            "pages": [46, 52]
          },
          {
            "title": "지역별 수익 분포",
            "summary": "미주, 유럽, 중화권, 일본, 기타 아시아 태평양",
            "pages": [53, 58]
          }
        ]
      },
      {
        "title": "위험 요소",
        "summary": "운영, 시장, 규제 및 경쟁 위험",
        "pages": [89, 110]
      }
    ]
  }
}
```

이제 분석가가 "2023 회계연도 iPhone 수익은 얼마였습니까?"라고 질문하면, LLM은 시맨틱적으로 유사한 텍스트 단편을 검색하지 않습니다. 인덱스를 읽고, "재무 성과 > 제품 카테고리별 수익"이 관련 브랜치라고 추론하여 그곳으로 이동한 다음, 46-52페이지를 통째로 검색합니다.

분석가는 정확한 페이지 인용과 함께 답변을 얻습니다. LLM은 '위험 요소' 섹션을 전혀 보지 않았습니다. 불필요한 정보도, 혼란도 없습니다.

## 인덱스 쿼리: 내부 작동 방식

코드에서 추론 기반 검색이 어떻게 보이는지 단순화된 버전은 다음과 같습니다.

```python
import json
from openai import OpenAI

client = OpenAI()
def navigate_index(query: str, index_node: dict, depth: int = 0) -> list[dict]:
    """
    LLM 추론을 사용하여 문서 인덱스를 재귀적으로 탐색합니다.
    페이지 참조가 있는 관련 리프 노드 목록을 반환합니다.
    """
    children = index_node.get("children", [])
    if not children:
        # 리프 노드: 이 섹션을 관련 섹션으로 반환
        return [index_node]
    # LLM에게 쿼리와 관련된 브랜치가 무엇인지 묻습니다.
    children_summary = "\n".join([
        f"[{i}] {child['title']}: {child['summary']}"
        for i, child in enumerate(children)
    ])
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {
                "role": "system",
                "content": (
                    "질문과 관련된 섹션을 찾기 위해 문서 인덱스를 탐색합니다. "
                    "답변을 포함할 가능성이 있는 섹션의 인덱스 번호를 선택하세요. "
                    "선택된 인덱스들의 JSON 배열을 반환하세요."
                )
            },
            {
                "role": "user",
                "content": (
                    f"질문: {query}\n\n"
                    f"사용 가능한 섹션:\n{children_summary}\n\n"
                    f"어떤 섹션을 살펴봐야 합니까? 인덱스만 JSON 배열로 반환하세요."
                )
            }
        ],
        temperature=0,
        response_format={"type": "json_object"}
    )
    selected = json.loads(response.choices[0].message.content).get("indices", [])
    relevant_nodes = []
    for idx in selected:
        if idx < len(children): # 유효한 인덱스인지 확인
            # 선택된 브랜치로 재귀 탐색
            relevant_nodes.extend(
                navigate_index(query, children[idx], depth + 1)
            )
    return relevant_nodes

def answer_with_pageindex(query: str, index: dict, document_pages: dict) -> str:
    """
    PageIndex 검색 및 답변 생성의 전체 프로세스.
    """
    # 관련 섹션을 찾기 위해 인덱스 탐색
    relevant_nodes = navigate_index(query, index)
    # 식별된 페이지에서 전체 텍스트 검색
    context_parts = []
    citations = []
    for node in relevant_nodes:
        pages = node.get("pages", [])
        if pages:
            page_start, page_end = pages[0], pages[1]
            for page_num in range(page_start, page_end + 1):
                if page_num in document_pages:
                    context_parts.append(document_pages[page_num])
                    citations.append(f"p.{page_num}")
    context = "\n\n".join(context_parts)
    # 조각나지 않은 전체 컨텍스트로 답변 생성
    answer_response = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {
                "role": "system",
                "content": (
                    "제공된 문서 섹션을 기반으로 질문에 답변하세요. "
                    "정확하게 답변하세요. 답변에 숫자나 날짜가 포함된 경우, 정확히 인용하세요."
                )
            },
            {
                "role": "user",
                "content": f"문서 섹션:\n{context}\n\n질문: {query}"
            }
        ],
        temperature=0
    )
    answer = answer_response.choices[0].message.content
    citation_str = ", ".join(set(citations))
    return f"{answer}\n\n**출처:** {citation_str}"
```

이 코드에서 빠진 것이 무엇인지 주목하세요: 임베딩, 코사인 유사성, 벡터 데이터베이스, 청킹 로직. 전체 검색 파이프라인이 문서 인덱스에 대한 구조화된 추론으로 대체되었습니다.

## 주목하게 만든 FinanceBench 결과

이 접근 방식이 진지하다는 것을 확신시킨 벤치마크 결과입니다.

FinanceBench는 SEC 보고서(SEC filings), 연간 보고서(annual reports), 실적 발표(earnings calls)와 같은 실제 금융 문서에 대한 질문으로 구성된 데이터셋입니다. 이 질문들은 정확한 수치 답변, 섹션 간의 정확한 상호 참조, 각주에 대한 신중한 읽기를 요구합니다. 금융 문서가 서로 유사하게 들리는 다른 주제에 대한 언어로 밀집되어 있기 때문에 벡터 기반 RAG 시스템에는 매우 어려운 것으로 악명이 높습니다.

PageIndex 기반으로 구축된 Mafin 2.5는 FinanceBench에서 98.7%의 정확도를 달성했습니다.

참고로, 유사한 금융 문서 벤치마크에서 일반적인 벡터 RAG 시스템은 구성에 따라 65~80% 정확도를 맴돕니다. 이 격차는 미미한 것이 아닙니다. 투자 분석가가 실제로 신뢰할 수 있는 도구와 한 줄 한 줄 확인해야 하는 그럴듯한 헛소리를 생산하는 도구 사이의 차이입니다.

그 이유는 구조적입니다. 금융 문서는 문서 계층 구조가 가장 중요한 콘텐츠 유형입니다. 연결 재무제표(consolidated balance sheet), 재무제표 주석(notes to financial statements), 경영진 분석 및 논의(MD&A) 섹션 등은 인간 분석가가 끊임없이 사용하는 명확하고 의도적인 구조를 가지고 있습니다. PageIndex는 그 구조를 보존하고 활용합니다. 벡터 청킹은 그것을 파괴합니다.

## PageIndex와 기존 RAG의 사용 시점

PageIndex가 벡터 RAG를 보편적으로 대체하는 것은 아닙니다. 올바른 선택은 구축하려는 대상에 따라 크게 달라집니다.

PageIndex는 다음과 같은 경우에 탁월합니다.

*   문서에 명확한 계층 구조가 있는 경우 (보고서, 매뉴얼, 계약서, 사양서)
*   정확성과 정확한 인용이 필수적인 경우
*   전문 분야인 경우 (금융, 법률, 의료, 규제)
*   질문이 단일 문서의 여러 섹션을 상호 참조해야 하는 경우
*   설명 가능성이 중요한 경우: 시스템이 반환한 내용을 보여줘야 할 때

기존 벡터 RAG는 다음과 같은 경우에 여전히 유효합니다.

*   수천 개의 짧고 독립적인 문서를 검색하는 경우 (예: 지원 티켓, 제품 리뷰)
*   문서에 일관된 구조가 없는 경우
*   광범위한 코퍼스(corpus)에 걸쳐 대략적인 시맨틱 검색이 필요한 경우
*   지연 시간(latency)이 주요 관심사이며 이미 청킹 전략을 최적화한 경우

솔직히 말해서, 대부분의 기업 문서 사용 사례, 즉 AI가 데모에서 인상적인 것을 넘어 실제로 유용해야 하는 경우들은 두 번째 목록보다는 첫 번째 목록과 훨씬 더 유사합니다.

## PageIndex 클라우드 API 사용

인프라 오버헤드 없이 프로덕션 배포를 위해 PageIndex는 REST API와 MCP 프로토콜 통합을 제공합니다. 개발자 설정은 다음과 같습니다.

```python
import requests

PAGEINDEX_API_KEY = "your_api_key"
BASE_URL = "https://api.pageindex.ai/v1"
def upload_document(file_path: str) -> str:
    """문서를 업로드하고 document_id를 반환합니다."""
    with open(file_path, "rb") as f:
        response = requests.post(
            f"{BASE_URL}/documents",
            headers={"Authorization": f"Bearer {PAGEINDEX_API_KEY}"},
            files={"file": f}
        )
    return response.json()["document_id"]

def query_document(document_id: str, question: str) -> dict:
    """인덱싱된 문서에 쿼리하고 인용된 답변을 얻습니다."""
    response = requests.post(
        f"{BASE_URL}/query",
        headers={
            "Authorization": f"Bearer {PAGEINDEX_API_KEY}",
            "Content-Type": "application/json"
        },
        json={
            "document_id": document_id,
            "question": question
        }
    )
    return response.json()

# 사용 예시
doc_id = upload_document("q3_earnings_report.pdf")
result = query_document(doc_id, "3분기 총 수익은 얼마였습니까?")
print(result["answer"])
print(f"출처: {result['citations']}")
```

API는 업로드 시 자동으로 인덱스 구성을 처리하고, 쿼리 시 추적 가능한 페이지 인용과 함께 구조화된 답변을 반환합니다.

## 이것이 나타내는 더 깊은 변화

PageIndex가 하는 일에는 벤치마크 점수를 넘어선 철학적으로 중요한 무언가가 있습니다.

기존 벡터 RAG는 문서를 시맨틱적으로 관련된 텍스트 단편들의 묶음으로 취급합니다. 문서 자체의 조직적 지능을 무시하는 것이죠. 잘 작성된 문서의 모든 제목, 섹션 구분, 상호 참조는 신호이며, 저자가 독자에게 콘텐츠를 탐색하는 방법을 알려주는 방식입니다. 벡터 청킹은 이 모든 것을 버리고 관련성을 처음부터 재구성하려고 합니다.

PageIndex는 문서를 실제 모습, 즉 지능적인 독자가 탐색하도록 설계된 구조화된 정보 아티팩트(artifact)로 취급합니다.

선임 분석가가 연간 보고서를 읽을 때, 페이지 1부터 선형적으로 읽지 않습니다. 목차를 확인하고, 관심 있는 부분으로 이동하며, 각주와 숫자를 대조하고, MD&A 섹션으로 돌아갑니다. 그들은 구조화된 아티팩트를 탐색하는 것입니다. PageIndex의 LLM도 같은 일을 합니다.

이것이 이 접근 방식이 전문 분야에 걸쳐 일반화되는 이유입니다. 제약 시험 보고서는 법률 계약이나 기술 사양서와 다른 계층 구조를 가집니다. 그러나 세 가지 경우 모두, 저자는 인간 전문가가 읽을 때 활용하는 내부 논리에 따라 문서를 구성했습니다. PageIndex는 그 논리를 포착하고, 검색 단계는 그것을 통해 추론합니다.

## 오늘날 문서 AI를 구축한다면 이것이 의미하는 것

PageIndex와 시간을 보내고 나서 제가 솔직하게 깨달은 점은: 저는 수년 동안 잘못된 문제를 해결하고 있었다는 것입니다.

문서 구조를 처음부터 파괴하고 있다는 것이 진짜 문제였는데, 저는 청킹 전략에만 집착했습니다. 시맨틱 유사성 자체가 잘못된 검색 신호였는데, 저는 임베딩 모델을 튜닝하고 있었습니다. 검색 접근 방식 자체가 병목 현상이었는데, 검색 품질을 개선하기 위해 리랭커(reranker)를 추가하고 있었습니다.

RAG 생태계는 청킹이라는 원죄를 보상하기 위해 수년간 정교한 장치들을 구축하는 데 시간을 보냈습니다. PageIndex는 다른 경로를 제시합니다: 청킹하지 말고, 임베딩하지 말고, 유사성으로 검색하지 마세요. 문서의 실제 구조를 반영하는 인덱스를 구축하고 추론으로 그것을 탐색하세요.

이것이 모든 문제에 대한 올바른 접근 방식은 아닙니다. 수백만 개의 짧은 비정형 문서에 대한 시맨틱 검색에는 여전히 벡터 검색이 우세합니다. 그러나 "길고 구조화된 문서에 대한 질문"이라는 문제의 큰 범주, 즉 대부분의 기업이 실제로 필요로 하는 문제에 대해서는 계층적 트리 접근 방식을 진지하게 고려할 가치가 있습니다.

저는 PageIndex를 사용하여 재무 문서 분석 파이프라인을 재구축했습니다. 저를 두 번이나 당황하게 했던 데모요? 다시 실행했습니다. 투자자가 2023년 매출 예상을 물었습니다. 시스템은 올바른 섹션으로 이동하여 정확한 페이지를 검색하고, 정확한 인용과 함께 답변을 제공했습니다.

더 이상 잘못된 연도, 잘못된 섹션, 그리고 자신감 넘치는 헛소리는 없습니다.
