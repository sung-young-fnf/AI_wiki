# 4천만 개의 문서. 하나의 CPU. GPU 없음.

이진 검색(binary search)과 int8 재점수화(rescoring)가 대규모 의미 검색(semantic search)의 규칙을 어떻게 조용히 다시 쓰는가.

임베딩(embeddings)을 오랫동안 다루다 보면 특정한 종류의 좌절감에 부딪히게 됩니다.

마침내 검색(retrieval)이 작동합니다. 정확합니다. 스마트하게 느껴집니다. 자부심을 느낍니다.
그러고 나서 청구서를 봅니다.

아니면 더 나쁘게는, RAM 사용량을 봅니다.

그보다 더 나쁜 것은 — "간단한" 의미 검색(semantic search)에 180GB의 메모리가 필요하다는 것을 깨닫는 순간입니다.

대개 이때 조용한 가정이 스며듭니다:
"음… 벡터 검색(vector search)을 제대로 하려면 이 정도 비용은 감수해야겠지."

하지만 어쩌면 그렇지 않을 수도 있습니다.

왜냐하면 제가 스크롤을 멈추게 한 사실은 바로 이것이기 때문입니다:

*   **4천만 개의 텍스트를 약 200ms 만에 검색할 수 있습니다.**
*   **CPU만 사용합니다.**
*   **8GB RAM으로.**
*   **45GB 디스크로.**

GPU는 없습니다.
이국적인 하드웨어(exotic hardware)도 없습니다.
마법도 없습니다.

단지 매우 의도적인 추론 전략(inference strategy)일 뿐입니다.

그리고 일단 이것을 보면, 이전으로는 돌아갈 수 없습니다.

## fp32 임베딩에 대한 불편한 진실

솔직히 말해봅시다.

대부분의 우리는 float32(fp32) 임베딩을 기본으로 사용합니다. 왜냐하면 모델이 그것을 제공하기 때문입니다. 우리는 그것들을 신성하게 여깁니다. 건드릴 수 없는 것으로. 고정밀도(high precision)가 아니면 아무것도 아니라고 생각합니다.

하지만 검색(retrieval) 과정에서 우리가 실제로 무엇을 하는지 생각해 보세요.

우리는 물리학 방정식을 푸는 것이 아닙니다.
우리는 문서를 **순위 매기고** 있습니다.

그리고 순위 매기기(ranking)는, 특히 퍼널(funnel)의 상단에서는, 놀라울 정도로 관대합니다.

이것이 전체 접근 방식이 파고드는 틈새입니다.

## 비법

신비로움을 걷어내면 설정은 거의 지루할 정도입니다:

**재현율(recall)을 위한 이진 검색(binary search). 정확도(precision)를 위한 int8 재점수화(rescoring).**

그것이 전부입니다.

품질을 버리는 것이 아닙니다.
**단계별로 적용하는 것입니다.**

먼저 저렴한 표현(cheap representations)이 무거운 작업을 처리하도록 한 다음, 필요한 곳에서만 정밀도(precision)를 다시 가져옵니다.

전체 흐름이 어떻게 작동하는지 설명합니다.

## 단계별 추론 전략(inference strategy)

마치 함께 디버깅하는 것처럼 천천히 살펴보겠습니다.

### 1단계: 쿼리(query) 임베딩

우리는 예상했던 대로 시작합니다.

밀집 임베딩 모델(dense embedding model).
표준 fp32 벡터.

```python
#1. 쿼리를 float32로 임베딩
start_time = time.time()
query_embedding = model.encode_query(query)
embed_time = time.time() - start_time
```

아직 특별한 것은 없습니다. 이것은 당신의 참조 신호입니다. 나중에 신뢰할 대상입니다.

### 2단계: 쿼리를 이진(binary)으로 양자화(quantize)

이제 사고방식의 첫 번째 전환이 옵니다.

fp32 쿼리 임베딩을 가져와서 이진으로 양자화합니다.
동일한 의미론적 방향을 가집니다.
표현 크기는 **32배 작아집니다.**

```python
#2. 쿼리를 ubinary로 양자화
start_time = time.time()
query_embedding_ubinary = quantize_embeddings(
    query_embedding.reshape(1, 1), "ubinary"
)
quantize_time = time.time() - start_time
```

이 단계는 빠릅니다. 거의 의심스러울 정도로 빠릅니다.

그리고 이것이 이어지는 모든 것을 가능하게 합니다.

### 3단계: 이진 인덱스(binary index) 검색

여기서부터 스케일(scale)이 갑자기 무섭지 않게 됩니다.

이진 인덱스(binary indices)는 매우 작습니다.
빠릅니다.
그리고 메모리(memory)에서 행복하게 살아갑니다.

여기서 정확(exact) 또는 근사(approximate) 검색을 선택할 수 있습니다.

```python
#3. 이진 인덱스 검색 (정확 또는 근사)
index = binary_ivf_index if use_approx else binary_index
start_time = time.time()
_scores, binary_ids = index.search(
    query_embedding_ubinary, top_k_rescore_multiplier
)
binary_ids = binary_ids[0]
search_time = time.time() - start_time
```

다음과 같이 묻는 대신,
"4천만 개의 문서 중 어느 것이 가장 좋을까?"

당신은 다음과 같이 묻습니다,
"약 40개 중 어떤 것이 유망해 보이는가?"

이러한 프레임 변경은 중요합니다.

### 4단계: 상위 후보(top candidates)에 대한 int8 임베딩 로드

이제 범위를 좁혔으니, 더 많은 신호(signal)를 다시 도입합니다.

상위 이진(binary) 일치 항목에 대해서만 int8 임베딩을 로드합니다.

```python
#4. 해당 int8 임베딩 로드
start_time = time.time()
int8_embeddings = np.array(
    title_text_int8_dataset[binary_ids]["embedding"], dtype=np.int8
)
load_int8_time = time.time() - start_time
```

여전히 작습니다.
여전히 저렴합니다.
하지만 이진(binary)보다 훨씬 더 표현력이 풍부합니다.

### 5단계: 원본 fp32 쿼리(query)를 사용하여 재점수화(rescore)

정확도(precision)가 돌아오는 순간입니다.

원본 fp32 쿼리 임베딩을 가져와서 int8 문서 임베딩과 비교하여 점수를 매깁니다.

```python
#5. float32 쿼리 임베딩과 int8 문서 임베딩을 사용하여 top_k_rescore_multiplier 재점수화
start_time = time.time()
scores = query_embedding @ int8_embeddings.T
rescore_time = time.time() - start_time
```

이 단계는 거의 비용이 들지 않습니다.
그리고 이전에 "잃었던" 품질의 대부분을 되찾게 해줍니다.

### 6단계: 정렬하고 최상의 결과 선택

```python
#6. 점수를 정렬하고 상위 k개 반환
start_time = time.time()
indices = scores.argsort()[::-1][:top_k]
top_k_indices = binary_ids[indices]
top_k_scores = scores[indices]
sort_time = time.time() - start_time
```

이 시점에서, 당신은 기본적으로 순위 매기기를 완료했습니다.

### 7단계: 실제 텍스트 로드

이제서야 제목, URL 및 콘텐츠를 건드립니다.

```python
#7. 상위 k개 결과에 대한 제목과 텍스트 로드
start_time = time.time()
top_k_titles = title_text_int8_dataset[top_k_indices]["title"]
top_k_urls = title_text_int8_dataset[top_k_indices]["url"]
top_k_texts = title_text_int8_dataset[top_k_indices]["text"]
top_k_titles = [f" [{title}] ({url})" for title, url in zip(top_k_titles, top_k_urls)]
load_text_time = time.time() - start_time
```

이것이 바로 그것들을 로드해야 할 때입니다.

그 이전이 아닙니다.

## 왜 이것이 작동하는가 (그리고 왜 처음에는 잘못된 느낌이 드는가)

이것이 잘못된 느낌이 드는 이유는 우리가 어떤 대가를 치르더라도 정밀도(precision)를 보호하도록 배웠기 때문입니다.

하지만 검색(retrieval)은 분류(classification)가 아닙니다.
회귀(regression)도 아닙니다.
심지어 생성(generation)도 아닙니다.

그것은 **분류(triage)**입니다.

이진 임베딩(binary embeddings)은 불필요한 것을 걸러내는 데 탁월합니다.
Int8 임베딩은 후보(contenders)의 순위를 매기는 데 좋습니다.
Fp32는 미묘한 차이(nuance)가 정말로 필요할 때 가장 좋습니다 — 하지만 문서가 40개 정도로 줄어들면 그런 경우는 드뭅니다.

따라서 모든 곳에서 fp32 비용을 지불하는 대신, 스포트라이트(spotlight)처럼 정밀도(precision)를 단계별로 적용합니다.

먼저 넓은 빔.
나중에 좁은 빔.

## 저장 공간 및 메모리

이것을 표준 fp32 설정과 비교해 봅시다.

**Float32 검색 (기준선)**
*   RAM: ~180GB
*   디스크: ~180GB
*   속도: 느림
*   비용: 고통스러움

**이진 + int8 검색**
*   이진 인덱스(Binary index): 32배 작음
*   Int8 임베딩: 4배 작음
*   RAM 사용량: ~6GB
*   디스크 사용량: ~45GB

그리고 여기 놀라운 점이 있습니다.

메모리에는 이진 인덱스만 유지합니다.

여기서 진정한 절약 효과가 복합적으로 발생합니다.

## 성능 트레이드오프(Performance trade-offs)

손실이 전혀 없다고 가장하지는 맙시다.

실험 결과는 다음과 같습니다:

| 표현 방식(Representation) | 인덱스 크기(Index Size) | 속도(Speed)       | 성능(Performance) |
| :----------------------- | :---------------------- | :---------------- | :---------------- |
| float32                  | 1배                     | 1배               | 100%              |
| int8 / uint8             | 4배 작음                | 최대 4배 빠름     | ~99.3%            |
| binary / ubinary         | 32배 작음               | 최대 45배 빠름    | ~96%              |

이제 중요한 부분입니다.

이진 검색(binary search)을 통해 더 많은 후보를 검색하고 int8로 재점수화(rescoring)함으로써, fp32 품질의 약 99%를 회복할 수 있습니다.

이 마지막 1%는 거의 아무런 비용도 들지 않습니다.

## 이것이 실질적으로 바꾸는 것

솔직히 말해서, 많은 것을 바꿉니다.

그것은 다음을 의미합니다:

*   대규모 의미 검색(semantic search)에 GPU가 필요하지 않습니다.
*   상용 하드웨어(commodity hardware)에서도 진지한 검색을 구현할 수 있습니다.
*   모든 곳에서 fp32에 대한 "고급세(luxury tax)"를 지불하지 않아도 됩니다.
*   모든 것을 다시 작성할 필요 없이 컬렉션(collections)을 확장할 수 있습니다.

그리고 아마도 더 큰 변화는 정신적인 것입니다.

우리는 임베딩을 깨지기 쉬운 유리처럼 취급해 왔습니다.

그렇지 않습니다.

그것들은 도구입니다.
그리고 도구는 모든 못에 하나의 망치만 사용하지 않을 때 가장 잘 작동합니다.
