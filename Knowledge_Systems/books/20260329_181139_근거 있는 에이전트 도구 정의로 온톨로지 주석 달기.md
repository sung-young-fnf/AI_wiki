지식 기반 시스템(knowledge-driven systems)에서 우리는 종종 온톨로지(ontologies)를 마치 이야기의 끝인 양 이야기합니다. 온톨로지는 도메인 개념(domain concepts)을 인코딩하는 형식 클래스(formal classes), 관계(relationships) 및 분류 체계(taxonomies)를 말하죠. 하지만 세상과 상호작용하고, 의사결정을 내리고, API(Application Programming Interface)를 호출하며, 작업을 실행하는 지능형 시스템(intelligent systems)을 구축하는 것이 목표라면, 온톨로지는 단순히 정적인 지식 이상이어야 합니다. 실행 가능한 동작(executable behavior)에 깔끔하게 매핑되는 시맨틱스(semantics)가 필요합니다.

미디엄 회원이 아니시라면, 이곳에서 미디엄 친구 링크를 통해 읽어보세요.

이 게시물에서는 온톨로지에 도구 정의(tool definitions)를 주석으로 추가하는 패턴을 설명할 것입니다. 이를 통해 AI 에이전트(AI agent)가 개념을 이해하는 것을 넘어, 그 개념으로 무엇을 할 수 있는지까지 알게 됩니다. 우리가 사용할 기술 스택(technical stack)은 다음과 같습니다:

*   **Neo4j** — 시맨틱 지식 그래프(semantic knowledge graph)를 위해
*   **LangChain** — 도구 정의 및 실행을 위해
*   **LangGraph** — 조율된 추론(orchestrated reasoning) 및 다단계 워크플로우(multi-step workflows)를 위해

최종 결과는 에이전트가 구조화된 지식을 바탕으로 추론하고, 근거 있고 검증 가능한 방식(grounded, verifiable ways)으로 행동하는 아키텍처입니다. 더 이상 환각적인 API(hallucinated APIs)는 없습니다.

목표는 영리한 프롬프팅(clever prompting)이 아닙니다.
목표는 근거 있는 시맨틱 실행(grounded, semantic execution)입니다.

## 온톨로지 주석이 중요한 이유

온톨로지는 일반적으로 클래스(classes)와 관계(relationships)로 형식화됩니다. 예를 들어, 헬스케어 도메인(healthcare domain)에서는 다음과 같습니다:

`환자 → 처방 보유 → 약물`

이는 구조, 즉 어떤 엔티티(entities)가 존재하고 어떻게 관련되어 있는지 알려줍니다. 하지만 어떤 조치를 취할 수 있는지에 대해서는 아무것도 알려주지 않습니다. LLM은 텍스트 문맥을 기반으로 동작을 추측할 수 있지만("환자 데이터를 가져오는 API가 있을지도 몰라"), 우리가 가지고 있지 않은 기능을 필연적으로 만들어낼 것입니다.

전통적인 소프트웨어 공학(traditional software engineering)에서는 기능(capabilities)이 타입 시스템(type system)의 일부입니다. `Patient` 타입에 `get_record()` 메서드(method)가 있다면 안전하게 호출할 수 있습니다. LLM에서는 이러한 바인딩(binding)이 부족합니다.

온톨로지에 도구 정의를 주석으로 추가하면 바로 그 바인딩이 생깁니다. 그래프의 개념을 실행 가능한 작업에 연결하여, 추론과 행동이 동일한 기본 구조를 공유하게 만듭니다.

온톨로지는 우선 시맨틱 모델(semantic model)이며, 그래프 데이터베이스 스키마(graph database schema)가 아닙니다.

따라서 깔끔하게 생각하는 방식은 다음과 같습니다:

1.  온톨로지를 공식적으로 정의합니다 (RDF/OWL / TTL).
2.  그다음 Neo4j에 이를 구체화합니다.
3.  그다음 실행 가능한 도구 메타데이터(executable tool metadata)를 연결합니다.
4.  그다음 LangGraph로 조율합니다.

먼저 Turtle(.ttl) 관점에서 이를 살펴보고, 나중에 Neo4j 및 도구로 번역해 봅시다.

## Turtle(TTL)로 온톨로지 정의하기

Neo4j나 LangGraph를 사용하기 전에, 우리는 RDF/OWL 시맨틱스(semantics)를 사용하여 도메인을 설명합니다.

이것이 '지식 공학(knowledge engineering)' 계층입니다.

Turtle은 단순히 RDF 트리플(triples)의 간결한 구문일 뿐입니다.

생각해 보세요:

`주어 서술어 객체`

따라서 우리가 이전에 작성했던:

`환자 → 처방 보유 → 약물`

Turtle에서는 명시적인 트리플(triples)이 됩니다.

### TTL로 작성된 최소한의 헬스케어 온톨로지

다음은 동일한 구조가 공식적으로 어떻게 보이는지입니다:

```turtle
@prefix ex:   <http://example.org/health#> .
@prefix rdf:  <http://www.w3.org/1999/02/22-rdf-syntax-ns#> .
@prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> .
@prefix owl:  <http://www.w3.org/2002/07/owl#> .

########################
# Classes
########################

ex:Patient        rdf:type owl:Class .
ex:Doctor         rdf:type owl:Class .
ex:Prescription   rdf:type owl:Class .
ex:Appointment    rdf:type owl:Class .

########################
# Relationships
########################

ex:hasPrescription rdf:type owl:ObjectProperty ;
    rdfs:domain ex:Patient ;
    rdfs:range  ex:Prescription .

ex:treatedBy rdf:type owl:ObjectProperty ;
    rdfs:domain ex:Patient ;
    rdfs:range  ex:Doctor .
```

이것은 순수한 구조입니다.

이는 우리에게 다음을 알려줍니다:

*   무엇이 존재하는지
*   사물들이 어떻게 연결되는지
*   도메인/범위 제약 조건(domain/range constraints)

하지만 여전히 행동에 대해서는 아무것도 알려주지 않습니다.

### 이것만으로는 충분하지 않은 이유

이런 온톨로지는 다음 질문에 답할 수 있습니다:

*   환자는 무엇인가요?

하지만 다음 질문에는 답할 수 없습니다:

*   환자로 무엇을 할 수 있나요?

LLM은 실행 가능한 기능(executable capabilities)을 안전하게 추론할 수 없습니다.

우리는 다음 사이의 다리가 필요합니다:

`시맨틱스 → 동작`

## 온톨로지 수준에서 도구 시맨틱스 추가하기

이제 온톨로지 자체를 확장합니다.

나중에 코드에서 도구를 발명하는 대신, 도구를 온톨로지 개념으로 모델링합니다.

이것이 핵심 설계 결정입니다.

### 도구 개념을 포함한 TTL

새로운 클래스와 속성을 소개합니다:

```turtle
########################
# Tool layer
########################

ex:Tool rdf:type owl:Class .
ex:hasTool rdf:type owl:ObjectProperty .

ex:inputSchema rdf:type owl:DatatypeProperty .
ex:description rdf:type owl:DatatypeProperty .
```

이제 도구는 온톨로지의 일등 시민(first-class citizens)입니다.

### 환자에 도구 주석 달기

```turtle
ex:getPatientRecord rdf:type ex:Tool ;
    ex:description "환자의 의료 기록을 가져옵니다" ;
    ex:inputSchema "{ patient_id: string }" .

ex:summarizePatientRecord rdf:type ex:Tool ;
    ex:description "환자 기록을 요약합니다" ;
    ex:inputSchema "{ patient_id: string }" .

ex:Patient ex:hasTool ex:getPatientRecord .
ex:Patient ex:hasTool ex:summarizePatientRecord .
```

이제 우리는 새로운 것을 얻었습니다.

시맨틱적으로, 우리는 다음과 같이 인코딩했습니다:

*   환자는 이러한 작업들을 수행할 수 있습니다.

## TTL을 Neo4j로 가져오기

이제 온톨로지를 Neo4j에 로드하여 시맨틱 모델링(semantic modeling)에서 구현(implementation)으로 넘어갑니다. RDF/TTL 데이터는 트리플을 속성 그래프 모델(property graph model)로 매핑하는 Neosemantics (n10s) 플러그인(plugin)을 사용하여 직접 가져올 수 있거나, RDF를 Cypher(Neo4j 쿼리 언어)로 변환하여 대량 로딩(bulk loading) 및 더 큰 제어를 가능하게 하는 사용자 정의 변환 스크립트(custom conversion scripts)를 통해 가져올 수 있습니다.

### 가져오기 예시

```cypher
CALL n10s.rdf.import.fetch(
  "file:///healthcare.ttl",
  "Turtle"
);
```

Neo4j는 다음과 같은 트리플을 구체화할 것입니다:

`(:Resource {uri:"Patient"})`
`(:Resource {uri:"getPatientRecord"})`
`(:Resource)-[:HAS_TOOL]->(:Resource)`

또는 매핑된 경우:

`(:Class {name:"Patient"})`
`(:Tool {name:"getPatientRecord"})`

이제 온톨로지는 Neo4j 내에서 쿼리할 수 있는 그래프로 존재합니다.

중요한 점에 주목하세요. 도구는 동일한 지식 그래프(knowledge graph) 내의 일반적인 노드일 뿐입니다. 특별하지 않으며, 동일한 시맨틱스에 참여합니다. 이것이 나중에 검색을 쉽게 만듭니다.

순수한 Cypher로:

```cypher
CREATE (:Class {name:"Patient"});
CREATE (:Class {name:"Appointment"});
CREATE (:Class {name:"Prescription"});
MATCH (p:Class {name:"Patient"}), (a:Class {name:"Appointment"})
CREATE (p)-[:HAS_APPOINTMENT]->(a);
```

이 단계에서는 구조는 있지만 실행 가능한 시맨틱스(actionable semantics)는 없습니다. 사람은 의미를 보지만, 기계는 작업이 필요합니다.

## 시스템의 계층별 보기

시스템을 계층별로 생각하는 것이 도움이 됩니다. 각 계층은 명확한 책임을 가집니다.

*   **Neo4j**는 의미(meaning)를 저장합니다.
*   **LangChain**은 동작(behavior)을 저장합니다.
*   **LangGraph**는 흐름(flow)을 조율합니다.

## 도구 정의로 개념 주석 달기 (Neo4j)

이것이 이 패턴의 핵심입니다. 에이전트에 도구 동작을 하드코딩(hardcoding)하는 대신, 온톨로지 자체에 이를 기술합니다.

각 도구는 이름, 설명 및 입력 스키마(input schema)를 가진 노드(node)로 모델링됩니다:

```cypher
CREATE (t:Tool {
  name: "get_patient_record",
  description: "주어진 환자의 상세 의료 기록을 반환합니다",
  input_schema: "{ patient_id: string }"
});
MATCH (c:Class {name:"Patient"}), (t:Tool {name:"get_patient_record"})
CREATE (c)-[:HAS_TOOL]->(t);
```

이제 우리는 선언적 매핑(declarative mapping)을 갖게 되었습니다:

*   환자를 다룬다면, `get_patient_record`라는 도구를 사용할 수 있습니다.

이것은 메타데이터(metadata)이며, 아직 실행 로직(execution logic)은 없다는 점에 주목하세요. 하지만 텍스트 프롬프트(text prompts)에 숨겨져 있지 않은 구조화된 방식으로 실행을 설명합니다.

## LangChain으로 도구 정의하기

이러한 그래프 주석을 실행 가능하게 만들려면, LangChain에서 해당 함수를 구현합니다. 각 함수는 에이전트가 호출할 수 있는 "도구"가 됩니다:

```python
from langchain.tools import tool

@tool
def get_patient_record(patient_id: str) -> dict:
    """데이터베이스에서 완전한 환자 기록을 가져옵니다"""
    return database.fetch_patient(patient_id)
```

함수 시그니처(function signature)는 Neo4j에 저장한 스키마와 일치합니다. 이러한 대칭성은 의도된 것이며, 자동 등록(automated registration) 및 유효성 검사(validation)를 간단하게 만듭니다.

수동으로 도구를 등록하는 대신, 온톨로지에서 사용 가능한 도구를 기반으로 동적으로 로드할 수 있습니다.

## 동적 도구 검색

질문이 들어오면 첫 번째 단계는 관련 엔티티(relevant entity)를 식별하는 것입니다. 식별되면 에이전트는 그래프에서 적용 가능한 도구(applicable tools)를 가져옵니다:

```python
def fetch_tools_for_entity(entity):
    query = """
    MATCH (:Class {name:$name})-[:HAS_TOOL]->(t:Tool)
    RETURN t.name AS name, t.description AS description
    """
    return run_neo4j_query(query, {"name": entity})
```

이는 취약한 프롬프트 엔지니어링(brittle prompt engineering)을 확정적 조회(deterministic lookup)로 대체합니다.

결과는 다음과 같을 수 있습니다:

```json
[
  {
    "name": "get_patient_record",
    "description": "주어진 환자의 상세 의료 기록을 반환합니다"
  }
]
```

그러면 에이전트는 그 개념으로 무엇을 할 수 있는지 정확히 알게 됩니다.

## LangGraph로 조율하기

단일 도구 호출만으로는 충분하지 않은 경우가 많습니다. 종종 개념 식별, 도구 선택, 실행, 결과 형식화와 같은 다단계 워크플로우(multi-step workflow)가 필요합니다.

이것이 바로 LangGraph가 도움이 되는 지점입니다. LangGraph를 사용하면 프롬프트에 암시적으로 포함하는 대신 명시적인 실행 흐름(explicit execution flows)을 정의할 수 있습니다.

다음은 간단한 그래프입니다:

```python
from langgraph.graph import StateGraph

def reason(state):
    question = state["question"]
    entity = extract_entity(question)  # 엔티티 추출 로직
    return {"entity": entity}

def lookup_tools(state):
    tools = fetch_tools_for_entity(state["entity"])
    return {"tools": tools}

def execute_tool(state):
    tool_info = state["tools"][0]
    result = invoke_tool(tool_info)  # LangChain 도구 호출
    return {"result": result}

builder = StateGraph(dict)
builder.add_node("reason", reason)
builder.add_node("lookup_tools", lookup_tools)
builder.add_node("execute_tool", execute_tool)

builder.add_edge("reason", "lookup_tools")
builder.add_edge("lookup_tools", "execute_tool")

agent_graph = builder.compile()
```

## 시스템의 일반적인 쿼리

사용자가 "환자 1234의 기록을 보여줘."라고 질문했다고 가정해 봅시다. 단계는 다음과 같습니다:

1.  오케스트레이터(orchestrator)가 엔티티(entity) — 환자를 추출합니다.
2.  Neo4j에서 환자에 주석이 달린 도구를 쿼리합니다.
3.  `get_patient_record` 도구를 로드합니다.
4.  구조화된 매개변수(parameter)로 LangChain 함수를 호출합니다.
5.  실행 결과로 JSON 반환 값을 수신합니다.
6.  응답 형식을 지정합니다.

에이전트는 도구를 추측하지 않습니다. 모든 작업은 온톨로지에 근거합니다.

## 이 패턴이 가치 있는 이유

실행 가능한 주석이 있는 온톨로지라는 이 방식은 여러 가지 이점을 제공합니다:

*   **확정적 동작(Deterministic behavior):** 도구는 프롬프트에 의해 추측되지 않습니다.
*   **설명 가능성(Explainability):** 왜 특정 도구가 선택되었는지 추적할 수 있습니다.
*   **확장성(Extensibility):** 그래프에 도구를 추가하면 프롬프트 변경 없이 에이전트가 기능을 얻게 됩니다.
*   **관심사의 분리(Separation of concerns):** 지식 → 시맨틱스; 로직 → 코드; 에이전트가 이들을 통합합니다.

이는 기능이 일등 시민(first-class citizens)인 전통적인 소프트웨어처럼 AI 시스템이 동작하게 만들면서도, 대규모 언어 추론의 유연성을 유지하기 위한 단계 중 하나입니다.

## 마무리 생각

온톨로지는 구조를 제공합니다. 도구는 행동을 부여합니다. LangGraph는 흐름을 제공합니다.

이러한 요소들이 신중하게 결합될 때, 에이전트는 단순히 텍스트 생성기(text generator)를 넘어 근거 있는 의사 결정(grounded decision-making)과 행동을 하는 시스템으로 격상됩니다. 헬스케어, 금융, 비즈니스 워크플로우(business workflows)와 같이 실제 행동이 수반되는 도메인에서는 이 패턴이 유용할 뿐만 아니라 필수적입니다.

이 패턴을 기반으로 구축한다면, 시맨틱 이해(semantic understanding)와 실행 가능한 기능(actionable capability) 사이의 격차가 훨씬 줄어들고, 에이전트가 훨씬 더 안정적으로 동작한다는 것을 알게 될 것입니다.
