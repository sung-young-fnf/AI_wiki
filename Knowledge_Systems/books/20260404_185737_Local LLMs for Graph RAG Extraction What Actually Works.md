# Local LLMs for Graph RAG Extraction: What Actually Works

그래프 RAG(Graph RAG)의 숨겨진 병목 현상은 검색(Retrieval)이 아니라 **추출(Extraction)**에 있습니다.

그래프 RAG는 문서를 엔티티(entity)와 관계(relationship)로 연결된 지식 그래프로 구조화하여, 단순한 단어의 공통 출현이 아닌 관계의 맥락을 이해하는 검색 시스템을 구축하는 유망한 방식입니다. 하지만 대중적으로 논의되지 않는 문제가 하나 있는데, 바로 추출 단계입니다.

지식 그래프를 쿼리하기 전에는 먼저 그래프를 구축해야 합니다. 즉, 가공되지 않은 텍스트에서 '(마리 퀴리, 출생지, 바르샤바)'와 같은 구조화된 엔티티-관계 트리플(entity-relation triples)을 추출해야 합니다. 많은 연구가 이 과정에 GPT-4를 사용한다고 가정하지만, 데이터 보안이 중요한 규제 환경에서 로컬 인프라를 사용해야 하는 경우 로컬 모델(Local Models)을 활용해야 합니다. 그러나 로컬 모델은 생각보다 구조화된 추출 작업에서 큰 어려움을 겪습니다.

저는 로컬 모델의 실제 성능을 파악하기 위해 구체적인 벤치마크를 수행했습니다.

### 그래프 RAG에 로컬 모델이 필요한 이유

엔터프라이즈 환경(의료, 법률, 금융 등)에서는 문서를 외부 API로 전송하는 것이 불가능한 경우가 많습니다. 규제 준수(Compliance), 지연 시간(Latency), 그리고 대규모 운영 비용 문제로 인해 추출 작업을 로컬에서 실행해야 합니다.

문제는 70억~90억 개(7B–9B)의 파라미터를 가진 소형 모델이 지문 내의 모든 엔티티를 식별하고, 유형을 분류하며, 관계를 결정하고, 이를 구조화된 트리플 형태의 유효한 JSON으로 출력하는 '인지적 부하'를 동시에 견뎌야 한다는 점입니다.

### 실험 설계

그래프 RAG 추출 품질을 측정하기 위해 특화된 벤치마크를 설계했습니다. 단순한 지문부터 복잡한 관계가 얽힌 지문까지 20개의 텍스트를 대상으로 4개의 주요 로컬 모델을 테스트했습니다.

*   **대상 모델:** Qwen 2.5 7B Instruct, Llama 3.1 8B Instruct, Mistral 7B Instruct v0.3, Gemma 2 9B IT
*   **프롬프트 전략:**
    *   **Naive:** 단순한 추출 지시
    *   **Schema-in-prompt:** 정확한 JSON 스키마를 포함한 지시
    *   **Few-shot:** 입출력 예시 2개를 포함한 지시
*   **평가 방식:** 유의어를 고려한 퍼지 매칭(fuzzy matching)을 적용하여, 'born_in'과 'was_born_in'을 동일하게 취급하는 등 실제 지식 그래프 구축 상황에 맞게 평가했습니다.

### 주요 결과

#### 1. 엔티티 인식은 해결되었으나, 관계 추출은 여전히 난제
모든 모델이 엔티티 인식(Entity Recognition)에서는 0.78~0.91의 높은 F1 점수(F1 score)를 기록했습니다. 하지만 트리플 추출(Triple F1) 점수는 가장 높은 설정(Llama 3.1 + Few-shot)에서도 0.732에 불과했으며, 대부분은 0.52~0.60 수준에 머물렀습니다. 엔티티는 잘 찾아내지만, 그들 사이의 관계를 원자적 트리플로 분해하고 일관된 형식으로 출력하는 것은 훨씬 더 어려운 작업임을 보여줍니다.

#### 2. 완벽한 모델은 없다: 품질과 신뢰성의 트레이드오프
*   **Llama 3.1 8B:** 추출 품질은 가장 뛰어나지만(Few-shot 시 0.732), 신뢰성이 매우 낮습니다. 퓨샷 프롬프트를 사용할 경우 출력물의 20%만이 유효한 스키마를 준수했습니다.
*   **Mistral 7B v0.3:** 품질(0.564)은 다소 낮지만, 스키마 준수율이 100%에 달하며 속도가 타 모델 대비 약 3배 빠릅니다. 예측 가능하고 안정적인 운영에 적합합니다.
*   **Gemma 2 9B:** 품질과 신뢰성 사이에서 가장 좋은 균형을 보여줍니다. 파라미터 수가 더 많은 만큼 복잡한 지문에서 강점을 보였습니다.

#### 3. 퓨샷(Few-shot)은 품질을 높이지만 신뢰성을 해친다
모든 모델에서 퓨샷 프롬프팅은 트리플 추출 품질을 향상시켰습니다. 그러나 프롬프트 길이가 3~5배 길어지면서 지연 시간이 크게 증가하고, 소형 모델이 형식을 무시하고 부연 설명을 출력하는 등 출력 오류가 발생할 확률도 높아졌습니다. (단, Mistral은 퓨샷 환경에서도 100% 신뢰성을 유지하는 유일한 예외였습니다.)

#### 4. 텍스트 복잡도가 미치는 영향
지문이 복잡해질수록(중첩된 관계, 시간적 한정어 등) 엔티티 인식 성능은 유지되지만, 트리플 추출 성능은 급격히 떨어집니다. 모델들은 밀도가 높은 텍스트를 개별 트리플로 분해하는 데 어려움을 겪으며, 여러 정보를 하나의 객체에 몰아넣는 경향을 보였습니다.

### 평가의 중요성

벤치마크 과정에서 깨달은 점은 평가 로직이 추출만큼이나 어렵다는 것입니다. 모델이 의미상으로는 정답을 맞혔음에도 불구하고 단어 선택이 달라 오답 처리되는 경우가 많았습니다. '동의어 그룹 매핑', '조동사 제거', '부분 문자열 매칭' 등 평가 로직을 정교화한 결과, 모델의 성능을 변경하지 않고도 측정된 F1 점수를 15~34% 높일 수 있었습니다.

### 실무적인 권장 사항

실제 그래프 RAG 시스템 구축 시 다음과 같은 전략을 추천합니다.

1.  **신뢰성이 최우선인 경우 (자동화 파이프라인):** **Mistral 7B v0.3 + Few-shot** 조합을 권장합니다. 속도가 가장 빠르고 스키마 오류가 거의 없습니다.
2.  **추출 품질이 최우선인 경우 (수동 검수 병행):** **Gemma 2 9B + Few-shot** 조합을 추천합니다. 오류가 발생할 수 있으나, 성공했을 때의 데이터 품질이 유의미하게 높습니다.
3.  **최고의 성능을 원하며 재시도가 가능한 경우:** **Llama 3.1 8B + Few-shot**을 먼저 시도하고, 스키마 검증 실패 시 **Schema-in-prompt** 방식으로 재시도(fallback)하는 전략이 가장 효과적입니다.

### 결론

엔티티 인식과 관계 추출 사이의 성능 격차는 로컬 그래프 RAG의 가장 큰 병목입니다. 이를 해결하기 위해서는 더 나은 프롬프팅 기술(Chain-of-Thought 등), 토큰 수준에서의 제약 디코딩(Constrained Decoding), 또는 추출 작업에 특화된 미세 조정(Fine-tuning)이 필요합니다.

다행히 현재 수준으로도 0.55~0.65의 트리플 F1 점수를 확보할 수 있으며, 이는 인간의 검수나 신뢰도 기반 필터링을 결합한다면 외부 API 없이도 충분히 실무 수준의 지식 그래프를 구축할 수 있음을 의미합니다.
