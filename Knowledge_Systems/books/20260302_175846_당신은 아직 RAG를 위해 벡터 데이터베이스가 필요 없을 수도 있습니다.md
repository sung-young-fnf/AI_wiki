당신은 (아직) RAG를 위해 벡터 데이터베이스가 필요 없을 수도 있습니다.
넘파이(Numpy) 및/또는 사이킷런(SciKit-Learn)으로도 모든 검색 요구사항을 충족할 수 있습니다.

현재, 검색 증강 생성(Retrieval Augmented Generation, RAG)의 부상과 함께, 벡터 데이터베이스(Vector Database)가 AI 업계에서 많은 주목을 받고 있습니다.

많은 사람들이 RAG 시스템을 구축하고 임베딩(Embedding)을 관리하려면 Pinecone, Weaviate, Milvus 또는 Qdrant와 같은 도구가 필요하다고 말합니다. 수억 개의 벡터를 다루는 엔터프라이즈(Enterprise) 애플리케이션을 개발하는 경우, 이러한 도구는 필수적입니다. 이 도구들은 CRUD 작업(Create, Read, Update, Delete), 메타데이터(Metadata) 기반 필터링, 그리고 컴퓨터 메모리를 넘어서는 디스크 기반 인덱싱(Disk-based Indexing)을 가능하게 합니다.

하지만 대부분의 내부 도구, 문서 봇(Documentation Bot), 또는 최소 기능 제품(Minimum Viable Product, MVP) 에이전트(Agent)의 경우, 전용 벡터 데이터베이스를 추가하는 것은 과도할 수 있습니다(overkill). 이는 복잡성을 증가시키고, 네트워크 지연(Network Delay)을 유발하며, 직렬화 비용(Serialisation Cost)을 추가하고, 관리 부담을 가중시킵니다.

사실 "벡터 검색(Vector Search)" (즉, RAG의 검색(Retrieval) 부분)은 단지 행렬 곱셈(Matrix Multiplication)에 불과합니다. 그리고 파이썬(Python)은 이미 이를 위한 세계 최고의 도구들을 보유하고 있습니다.

이 글에서는 넘파이(NumPy)와 사이킷런(SciKit-Learn)만을 사용하여 중소 규모 데이터 볼륨을 위한 RAG 파이프라인(Pipeline)의 운영 환경 준비 완료(Production-Ready) 검색 구성 요소(Retrieval Component)를 구축하는 방법을 보여드리겠습니다. 수백만 개의 텍스트 문자열을 밀리초(millisecond) 단위로, 모든 작업을 인메모리(In-Memory)로, 그리고 어떠한 외부 종속성(External Dependencies) 없이 검색할 수 있다는 것을 확인하실 수 있을 겁니다.

### RAG를 행렬 연산으로 이해하기

RAG는 주로 네 가지 단계를 포함합니다.

*   **임베드(Embed):** 소스 데이터의 텍스트를 벡터(부동 소수점 숫자의 목록)로 변환합니다.
*   **저장(Store):** 이 벡터들을 데이터베이스에 저장합니다.
*   **검색(Retrieve):** 쿼리 벡터(Query Vector)와 수학적으로 "가까운" 벡터를 찾습니다.
*   **생성(Generate):** 해당 텍스트를 대규모 언어 모델(Large Language Model, LLM)에 입력합니다.

1단계와 4단계는 대규모 언어 모델에 의존합니다. 2단계와 3단계는 벡터 데이터베이스(Vector DB)의 영역입니다. 우리는 1, 2, 3단계에 중점을 두고, 벡터 데이터베이스를 완전히 사용하지 않는 방법을 다룰 것입니다.

하지만 벡터 데이터베이스를 검색할 때, "근접성(closeness)"이란 정확히 무엇일까요? 일반적으로 이는 코사인 유사도(Cosine Similarity)입니다. 벡터가 단위 크기(Unit Magnitude)로 정규화(Normalised)되어 있다면, 코사인 유사도는 단순히 내적(Dot Product)과 같습니다.

크기 N의 1차원 쿼리 벡터(Q(1xN))와 크기 M x N의 문서 벡터(Document Vector) 데이터베이스(D(MxN))가 있을 때, 가장 일치하는 항목을 찾는 것은 데이터베이스 쿼리가 아닙니다. 이는 D와 Q의 전치(Transpose) 행렬 간의 행렬 곱셈(Matrix Multiplication) 연산, 즉 내적(Dot Product)입니다.

`Scores = D.Q^T`

넘파이(NumPy)는 벡터화(Vectorisation)와 같은 최신 CPU 기능(CPU Feature)을 활용하는 루틴(Routine)을 사용하여 이러한 종류의 연산을 효율적으로 수행하도록 설계되었습니다.

#### 인메모리 벡터 스토어(In-Memory Vector Store)

복잡한 서버는 필요 없습니다. 입력 파일에서 텍스트 데이터(Text Data)를 로드하여 바이트 크기의 조각(Byte-sized Pieces)으로 청크(Chunk)를 분할하는 함수와, 원시 텍스트 청크(Raw Text Chunk)용 목록 하나와 임베딩 행렬(Embedding Matrix)용 목록 하나를 가진 클래스만 있으면 됩니다. 다음은 코드입니다.

```python
import numpy as np
import os
from sentence_transformers import SentenceTransformer
from sklearn.metrics.pairwise import cosine_similarity
from typing import List, Dict, Any
from pathlib import Path


class SimpleVectorStore:
    def __init__(self, model_name: str = 'all-MiniLM-L6-v2'):
        print(f"Loading embedding model: {model_name}...")
        self.encoder = SentenceTransformer(model_name)
        self.documents = []  # Stores the raw text and metadata
        self.embeddings = None # Will become a numpy array 

    def add_documents(self, docs: List[Dict[str, Any]]):
        """
        Ingests documents.
        docs format: [{'text': '...', 'metadata': {...}}, ...]
        """
        texts = [d['text'] for d in docs]
        
        # 1. Generate Embeddings
        print(f"Embedding {len(texts)} documents...")
        new_embeddings = self.encoder.encode(texts)
        
        # 2. Normalize Embeddings 
        # (Critical optimization: allows dot product to approximate cosine similarity)
        norm = np.linalg.norm(new_embeddings, axis=1, keepdims=True)
        new_embeddings = new_embeddings / norm
        
        # 3. Update Storage
        if self.embeddings is None:
            self.embeddings = new_embeddings
        else:
            self.embeddings = np.vstack([self.embeddings, new_embeddings])
            
        self.documents.extend(docs)
        print(f"Store now contains {len(self.documents)} documents.")

    def search(self, query: str, k: int = 5):
        """
        Retrieves the top-k most similar documents.
        """
        if self.embeddings is None or len(self.documents) == 0:
            print("Warning: Vector store is empty. No documents to search.")
            return []

        # 1. Embed and Normalize Query
        query_vec = self.encoder.encode([query])
        norm = np.linalg.norm(query_vec, axis=1, keepdims=True)
        query_vec = query_vec / norm
        
        # 2. Vectorized Search (Matrix Multiplication)
        # Result shape: (1, N_docs)
        scores = np.dot(self.embeddings, query_vec.T).flatten()
        
        # 3. Get Top-K Indices
        # argsort sorts ascending, so we take the last k and reverse them
        # Ensure k doesn't exceed the number of documents
        k = min(k, len(self.documents))
        top_k_indices = np.argsort(scores)[-k:][::-1]
        
        results = []
        for idx in top_k_indices:
            results.append({
                "score": float(scores[idx]),
                "text": self.documents[idx]['text'],
                "metadata": self.documents[idx].get('metadata', {})
            })
            
        return results

def load_from_directory(directory_path: str, chunk_size: int = 1000, overlap: int = 200):
    """
    Reads .txt files and splits them into overlapping chunks.
    """
    docs = []
    # Use pathlib for robust path handling and resolution
    path = Path(directory_path).resolve()
    
    if not path.exists():
        print(f"Error: Directory '{path}' not found.")
        print(f"Current working directory: {os.getcwd()}")
        return docs
        
    print(f"Loading documents from: {path}")
    for file_path in path.glob("*.txt"):
        try:
            with open(file_path, "r", encoding="utf-8") as f:
                text = f.read()
                
            # Simple sliding window chunking
            # We iterate through the text with a step size smaller than the chunk size
            # to create overlap (preserving context between chunks).
            step = chunk_size - overlap
            for i in range(0, len(text), step):
                chunk = text[i : i + chunk_size]
                
                # Skip chunks that are too small (e.g., leftover whitespace)
                if len(chunk) < 50:
                    continue
                    
                docs.append({
                    "text": chunk,
                    "metadata": {
                        "source": file_path.name,
                        "chunk_index": i
                    }
                })
        except Exception as e:
            print(f"Warning: Could not read file {file_path.name}: {e}")
            
    print(f"Successfully loaded {len(docs)} chunks from {len(list(path.glob('*.txt')))} files.")
    return docs
```

### 사용된 임베딩 모델(Embedding Model)

우리는 `all-MiniLM-L6-v2` 모델을 선택했습니다. 그 이유는 다음과 같습니다.

*   빠르고 가볍습니다.
*   대형 모델보다 적은 메모리를 사용하는 384차원 벡터(Vector)를 생성합니다.
*   특별한 미세 조정(Fine-tuning) 없이도 다양한 영어 작업에서 우수한 성능을 보입니다.

이 모델은 단지 제안일 뿐입니다. 특별히 선호하는 모델이 있다면 어떤 임베딩 모델이든 사용할 수 있습니다.

### 왜 정규화(Normalise)해야 할까요?

코드에서 정규화(Normalisation) 단계를 보셨을 겁니다. 이전에 언급했지만, 명확히 하자면 두 벡터 X와 Y가 주어졌을 때 코사인 유사도(Cosine Similarity)는 다음과 같이 정의됩니다.

`Similarity = (X · Y) / (||X|| * ||Y||)`

여기서:

*   X · Y는 벡터 X와 Y의 내적(Dot Product)입니다.
*   ||X||는 벡터 X의 크기(Magnitude, 길이)입니다.
*   ||Y||는 벡터 Y의 크기입니다.

나눗셈은 추가적인 계산을 필요로 하므로, 모든 벡터가 단위 크기(Unit Magnitude)를 가진다면 분모가 1이 되어 공식이 X와 Y의 내적으로 단순화되며, 이는 검색 속도를 훨씬 빠르게 만듭니다.

### 사이킷런(SciKit-Learn): 업그레이드 경로

넘파이(NumPy)는 무차별 대입 검색(Brute-Force Search)에 잘 작동합니다. 하지만 수십 개의 문서가 있고 무차별 대입 방식이 너무 느리다면 어떨까요? 벡터 데이터베이스로 전환하기 전에 사이킷런(SciKit-Learn)의 `NearestNeighbors`를 시도해 볼 수 있습니다. 이는 KD-트리(KD-Tree) 및 볼-트리(Ball-Tree)와 같은 트리 기반 구조(Tree-based Structures)를 사용하여 검색 속도를 O(N)에서 O(log N)으로 향상시킵니다.

다음은 사이킷런(SciKit-Learn)을 사용한 코드입니다. 이번에는 다음과 같은 질문을 할 것입니다.

"크리스마스 연휴 후에 니콜라스는 누구를 사랑한다고 어머니에게 말했는가?"

혹시 모르셨다면, 이 질문은 소설 "전쟁과 평화"에서 나온 것입니다. 이 방대한 정보에 대해 우리의 새로운 검색이 어떻게 작동하는지 봅시다.

먼저, 사이킷런(SciKit-Learn)의 최근접 이웃(Nearest Neighbor) 알고리즘을 구현하는 새로운 클래스(Class)가 있습니다.

```python
from sklearn.neighbors import NearestNeighbors

class ScikitVectorStore(SimpleVectorStore):
    def __init__(self, model_name='all-MiniLM-L6-v2'):
        super().__init__(model_name)
        # Brute force is often faster than trees for high-dimensional data 
        # unless N is very large, but 'ball_tree' can help in specific cases.
        self.knn = NearestNeighbors(n_neighbors=5, metric='cosine', algorithm='brute')
        self.is_fit = False

    def build_index(self):
        print("Building Scikit-Learn Index...")
        self.knn.fit(self.embeddings)
        self.is_fit = True

    def search(self, query: str, k: int = 5):
        if not self.is_fit: self.build_index()
        
        query_vec = self.encoder.encode([query])
        # Note: Scikit-learn handles normalization internally for cosine metric 
        # if configured, but explicit is better.
        
        distances, indices = self.knn.kneighbors(query_vec, n_neighbors=k)
        
        results = []
        for i in range(k):
            idx = indices[0][i]
            # Convert distance back to similarity score (1 - dist)
            score = 1 - distances[0][i]
            results.append({
                "score": score,
                "text": self.documents[idx]['text']
            })
        return results
```

### 실제 벡터 데이터베이스(Vector Database)를 사용해야 할 때

저는 벡터 데이터베이스가 쓸모없다고 주장하는 것이 아닙니다. 이들은 넘파이(NumPy)와 사이킷런(SciKit-Learn)으로는 처리할 수 없는 특정 문제들을 해결합니다. 다음과 같은 조건 중 하나라도 해당할 경우, `SimpleVectorStore`나 `ScikitVectorStore`와 같은 방식에서 Weaviate/Pinecone/pgvector와 같은 도구로 마이그레이션(Migrate)해야 합니다.

*   **RAM이 병목 현상(Bottleneck)인 경우:** 임베딩 행렬(Embedding Matrix)이 서버의 메모리를 초과할 때. 참고: 768차원 [float32] 벡터 100만 개는 약 3GB의 RAM만 차지하므로, 상당한 양을 메모리에 담을 수 있습니다.
*   **CRUD 빈도가 높은 경우:** 읽는 동안 개별 벡터를 지속적으로 업데이트하거나 삭제해야 할 때. 예를 들어, 넘파이 배열은 불변(Immutable)하며, 요소를 추가하려면 전체 배열을 복사해야 하므로 느립니다.
*   **메타데이터(Metadata) 필터링이 필요한 경우:** "user_id=10이고 date > 2023인 X 근처의 벡터를 찾으세요"와 같은 복잡한 쿼리(Query)가 필요할 때. 넘파이에서 이를 수행하려면 복잡해질 수 있는 부울 마스크(Boolean Mask)가 필요합니다.
*   **영속성(Persistence)이 필요한 경우:** 서버 재시작 후에도 매번 소스 파일에서 인덱스(Index)를 재구축할 필요 없이 데이터가 유지되어야 할 때. `np.save`나 피클링(Pickling)은 간단한 영속성을 위해 작동하지만, 한계가 있습니다.

### 요약

엔지니어링은 항상 장단점(Trade-off)을 수반합니다. 벡터 데이터베이스(Vector Database)를 사용하면 지금 당장은 필요 없을 수도 있는 확장성(Scalability)을 얻는 대신 설정에 복잡성이 추가됩니다. 넘파이(NumPy) 및/또는 사이킷런(SciKit-Learn)을 검색 프로세스에 사용하여 더 간단한 RAG 설정을 시작하면 다음을 얻을 수 있습니다.

*   **더 낮은 지연 시간(Latency).** 네트워크 홉(Network Hop)이 없습니다.
*   **더 낮은 비용.** SaaS 구독이나 추가 인스턴스(Instance)가 필요 없습니다.
*   **단순성.** 그저 파이썬 스크립트(Python Script)일 뿐입니다.

식료품점에 가기 위해 스포츠카가 필요하지 않은 것처럼, 많은 경우 넘파이(NumPy)나 사이킷런(SciKit-Learn)만으로도 필요한 모든 검색을 수행할 수 있습니다.
