## 1. 그래프 메모리 시스템 소개

AI 에이전트(AI agents)는 조직의 업무 방식을 변화시키고 있지만, 근본적인 문제인 '메모리'에 직면해 있습니다. 에이전트에게 지난주 대화에 대해 물어보면 종종 멍한 반응을 얻게 됩니다. 조직 전체에 여러 에이전트를 배포하면 학습한 내용을 공유할 수 없습니다. 문제가 발생했을 때, 에이전트가 특정 결정을 내린 이유를 설명하는 감사 추적(audit trail)도 없습니다.

그래프 메모리 시스템(Graph memory systems)은 에이전트의 컨텍스트(context)를 그래프 데이터베이스(graph databases)에 저장하여 이러한 문제를 해결하며, 풍부한 관계 모델링(relationship modeling)과 효율적인 검색을 가능하게 합니다. 단순한 키-값 저장소(key-value stores)나 벡터 데이터베이스(vector databases)와 달리, 그래프 메모리는 실제 지식의 상호 연결된 특성을 포착합니다.

### 1.2 에이전트 메모리에 그래프 데이터베이스를 사용하는 이유

업계에서는 그래프 데이터베이스가 AI 에이전트 메모리에 필수적이라는 인식이 점점 커지고 있습니다. Zep, Cognee, Mem0 등과 같은 회사들은 Neo4j를 사용하여 대화와 개체 지식(entity knowledge)을 저장하는 인상적인 시스템을 구축했습니다. 그래프 데이터베이스는 다음 분야에서 뛰어난 성능을 발휘합니다:

*   **관계 모델링(Relationship Modeling)**: 개체(entities)와 그 연결을 자연스럽게 표현
*   **다중 홉 추론(Multi-hop Reasoning)**: 관계를 탐색하여 연결된 정보 검색
*   **출처 추적(Provenance Tracking)**: 정보의 출처와 결정이 이루어진 방식 파악
*   **시간 쿼리(Temporal Queries)**: 사실(facts)과 관계가 시간 경과에 따라 어떻게 변하는지 추적

## 2. Neo4j-Agent-Memory

Neo4j-agent-memory는 AI 에이전트를 위한 완전한 그래프 네이티브 메모리 시스템(graph-native memory system)을 제공하는 Neo4j Labs 프로젝트입니다. Neo4j의 강력한 그래프 데이터베이스를 기반으로 단기 메모리(short-term), 장기 메모리(long-term), 추론 메모리(reasoning memory)를 단일 연결 그래프로 통합합니다.

![Entity Extraction Pipeline | https://neo4j.com/blog/developer/meet-lennys-memory-building-context-graphs-for-ai-agents/](https://miro.medium.com/v2/resize:fit:1400/1*O-iY-UeL_x8WlYdFm-2cbg.png)
개체 추출 파이프라인 (Entity Extraction Pipeline) | https://neo4j.com/blog/developer/meet-lennys-memory-building-context-graphs-for-ai-agents/

### 2.1 주요 기능

![Table 2: Neo4j-Agent-Memory Core Features](https://miro.medium.com/v2/resize:fit:1400/1*bJ5-l5tK0r19Vd9B_G_5hA.png)
표 2: Neo4j-Agent-Memory 핵심 기능

**개체 추출 파이프라인(Entity Extraction Pipeline)**

*   **spaCy (~5ms, 무료)**: 일반적인 개체(entities)를 빠르게 포착하는 통계적 NER(Named Entity Recognition)
*   **GLiNER2 (~50ms, 무료)**: 도메인별 스키마(domain-specific schemas)를 사용하는 제로샷 NER(Zero-shot NER)
*   **LLM 폴백(fallback) (gpt-4o-mini) (~500ms, 유료)**: 복잡한 경우에 최고의 정확도 제공

이 파이프라인은 또한 중복 제거(deduplication, "Brian"과 "Brian Chesky"가 같은 사람인가?), 관계 추출(relationship extraction), 그리고 선택적 위키피디아 강화(Wikipedia enrichment)를 처리합니다.

```python
from neo4j_agent_memory.extraction import ExtractorBuilder

extractor = (
    ExtractorBuilder()
    .with_spacy(model="en_core_web_sm")
    .with_gliner_schema("podcast", threshold=0.5)  # 8 domain schemas available
    .with_llm_fallback(model="gpt-4o-mini")
    .with_merge_strategy("confidence")
    .build()
)
result = await extractor.extract(
    "Brian Chesky founded Airbnb in San Francisco in 2008"
)

# Extracts: Person(Brian Chesky), Organization(Airbnb), 
#           Location(San Francisco), Event(founding, 2008)
```

### 2.3 빠른 시작 예제

```python
import asyncio
from pydantic import SecretStr
from neo4j_agent_memory import MemoryClient, MemorySettings, Neo4jConfig

async def main():
    # 설정 구성
    settings = MemorySettings(
        neo4j=Neo4jConfig(
            uri='bolt://localhost:7687',
            username='neo4j',
            password=SecretStr('your-password'),
        )
    )
    
    # 메모리 클라이언트 사용
    async with MemoryClient(settings) as memory:
        # 대화 메시지 저장
        await memory.short_term.add_message(
            session_id='user-123',
            role='user',
            content='Hi, I am John and I love Italian food!'
        )
        
        # 선호도 추가
        await memory.long_term.add_preference(
            category='food',
            preference='Loves Italian cuisine',
            context='Dining preferences'
        )
        
        # 관련 메모리 검색
        preferences = await memory.long_term.search_preferences(
            'restaurant recommendation'
        )
        
        # LLM 프롬프트(prompt)를 위한 결합된 컨텍스트 가져오기
        context = await memory.get_context(
            'What restaurant should I recommend?',
            session_id='user-123'
        )
        print(context)

asyncio.run(main())
```

**출력:**

```

  [food] Loves Italian cuisine
  
  Context: User John loves Italian food. Previous preferences indicate
  
  a preference for Italian cuisine in dining contexts.
```

## 3. Zep과 Graphiti

Zep은 AI 에이전트를 위한 상용 메모리 계층 서비스(memory layer service)로, 메모리 검색 벤치마크에서 현재 최첨단 시스템을 능가합니다. 그 핵심에는 Graphiti가 있습니다. Graphiti는 오픈소스(open-source)의 시간 인지형 지식 그래프 엔진(temporally-aware knowledge graph engine)으로, 구조화되지 않은 대화 데이터와 구조화된 비즈니스 데이터를 동적으로 통합하면서 과거 관계(historical relationships)를 유지합니다.

![https://www.getzep.com/](https://miro.medium.com/v2/resize:fit:1400/1*6B-yvY5y_tL_2s_o66Kxsw.png)
[https://www.getzep.com/](https://www.getzep.com/)

### 3.1 Graphiti 아키텍처

Graphiti는 시간 경과에 따른 개체(entities) 간의 복잡하고 진화하는 관계를 나타내는 동적이고 시간 인지형 지식 그래프(temporally-aware Knowledge Graphs)를 구축합니다. 배치 처리(batch processing)에 의존하는 전통적인 RAG(Retrieval-Augmented Generation) 접근 방식과 달리 Graphiti는 다음을 제공합니다:

*   **실시간 증분 업데이트(Real-Time Incremental Updates)**: 배치 재계산(batch recomputation) 없이 새로운 데이터를 즉시 통합
*   **이중 시간 데이터 모델(Bi-Temporal Data Model)**: 이벤트 발생 및 수집 시간(ingestion times)을 명시적으로 추적
*   **효율적인 하이브리드 검색(Efficient Hybrid Retrieval)**: 의미 임베딩(semantic embeddings), 키워드(BM25), 그래프 탐색(graph traversal)을 결합
*   **사용자 정의 개체 정의(Custom Entity Definitions)**: Pydantic 모델을 통한 유연한 온톨로지(ontology) 생성

### 3.3 Graphiti 빠른 시작

```python
from graphiti_core import Graphiti
from graphiti_core.nodes import EpisodeType

# Graphiti 초기화
graphiti = Graphiti('bolt://localhost:7687', 'neo4j', 'password')

# 인덱스(indices) 및 제약 조건(constraints)으로 그래프 데이터베이스 초기화
await graphiti.build_indices_and_constraints()

# 에피소드 추가 (텍스트 또는 구조화된 JSON)
await graphiti.add_episode(
    name='User Conversation 1',
    episode_body='User: I love Italian food!',
    source=EpisodeType.message,
    source_description='Chat message',
    group_id='user-123'
)

# 하이브리드 검색을 사용하여 관계 검색
results = await graphiti.search(
    'What food does the user like?',
    group_ids=['user-123']
)

# 그래프 거리(graph distance)를 사용하여 결과 재순위 지정(rerank)
reranked = await graphiti.rerank_results(
    'What food does the user like?',
    results
)
```

```
  Score: 0.923 - User likes Mexican food  
  Score: 0.456 - User previously liked Italian food (invalidated)  
  Top result after reranking: User likes Mexican food
```

## 5. LangMem 및 Mem0

### 5.1 LangMem

LangMem은 LangGraph 애플리케이션을 위해 특별히 설계된 메모리 시스템입니다. 여러 LLM 호출을 수행하고 가장 관련성 높은 메모리 조각(snippet)만 반환하여 쿼리당 토큰(tokens per query)을 최소화합니다. 이 접근 방식은 비용 최적화에는 영리하지만, 토큰 효율성을 위해 여러 API 호출을 필요로 합니다.

![Types of Memory | https://langchain-ai.github.io/langmem/concepts/conceptual_guide/#memory-types](https://miro.medium.com/v2/resize:fit:1400/1*3yKx-jUjTj-q-K0Y-S_o8A.png)
메모리 유형 | https://langchain-ai.github.io/langmem/concepts/conceptual_guide/#memory-types

```python
# LangMem 예제 (LangGraph 포함)
from langmem import MemoryManager

# 메모리 관리자 초기화
memory = MemoryManager(
    model='gpt-4o-mini',
    extract_prompt='Extract user preferences'
)

# 메모리 저장
await memory.store(
    session_id='user-123',
    content='User prefers dark mode'
)

# 관련 메모리 검색
memories = await memory.retrieve(
    session_id='user-123',
    query='What are user preferences?'
)


print(f'Retrieved {len(memories)} memories')
for m in memories:
print(f' - {m.content}')
```

**출력:**

```
  Memory stored  
  Retrieved 1 memories  
  - User prefers dark mode and large fonts  
  Note: LangMem made 3 LLM calls but returned only ~50 tokens  
  instead of the full conversation history (~500 tokens).
```

### 5.2 Mem0

Mem0은 복잡한 시간 아키텍처(temporal architectures) 없이 그래프 강화 메모리(graph-enhanced memory)를 제공하는 선도적인 메모리 시스템입니다. 벡터(vector), 그래프(graph), 키-값 저장소(key-value stores)를 결합한 하이브리드 아키텍처(hybrid architecture)를 제공하며, 적응형 메모리 업데이트(adaptive memory updates) 및 다단계 회수(multi-level recall) 기능을 갖추고 있습니다.

![Graph-based memory architecture of Mem0g | https://arxiv.org/abs/2504.19413](https://miro.medium.com/v2/resize:fit:1400/1*Kx1yR9-dE-zE_qO8P6kH4Q.png)
Mem0g의 그래프 기반 메모리 아키텍처 | [https://arxiv.org/abs/2504.19413](https://arxiv.org/abs/2504.19413)

#### 5.2.1 주요 기능

*   **다단계 메모리(Multi-Level Memory)**: 사용자, 세션, 에이전트 수준의 메모리 계층
*   **그래프 메모리 지원(Graph Memory Support)**: 개체 관계(entity relationships)를 위한 선택적 그래프 계층
*   **프레임워크 비종속적(Framework Agnostic)**: API를 통해 모든 에이전트 프레임워크와 연동 가능
*   **관리형 서비스(Managed Service)**: 자체 호스팅 또는 관리형 클라우드 서비스로 제공

#### 5.2.2 Mem0 빠른 시작

```python
from mem0 import MemoryClient

# 클라이언트 초기화

    messages=[{
        'role': 'user',
        'content': 'I prefer vegetarian food'
    }],
    user_id='user-123'
)

# 메모리 검색
memories = client.search(
    query='What food preferences?',
    user_id='user-123'
```


### 7.2 메모리를 사용한 환자 대화

```python
async def patient_interaction(memory, patient_id):
    # 단기 메모리(SHORT-TERM): 대화 저장
    await memory.short_term.add_message(
        session_id=patient_id,
        role='user',
        content='I have been feeling dizzy lately, especially in the morning'
    )
    
    # 장기 메모리(LONG-TERM): 개체 추출 및 저장
    patient = await memory.long_term.add_entity(
        name='Sarah Johnson',
        entity_type='PERSON',
        subtype='PATIENT',
        description='Patient reporting dizziness symptoms'
    )
    
    # 장기 메모리(LONG-TERM): 증상(symptoms)을 사실(facts)로 저장
    await memory.long_term.add_fact(subject='Sarah Johnson',
        predicate='has_symptom',
        obj='dizziness',
        valid_from=datetime.now()
    )
    
    # 추론 메모리(REASONING): 진단 과정 추적
    trace = await memory.reasoning.start_trace(
        session_id=patient_id,
        task='Diagnose dizziness symptoms'
    )
    
    await memory.reasoning.add_step(
        trace.id,
        thought='Dizziness in morning could be related to blood pressure',
        action='check_medication_history'
    )
    
    await memory.reasoning.record_tool_call(
        trace.id,
        tool_name='medication_lookup',
        arguments={'patient': 'Sarah Johnson'},
        result={'medications': ['Lisinopril']},
        status=ToolCallStatus.SUCCESS
    )
    
    await memory.reasoning.complete_trace(
        trace.id,
        outcome='Recommend blood pressure monitoring',
        success=True
    )
```

**출력:**

```
  Message stored: msg-abc123
  Entity created: ent-def456
  Fact recorded: fact-ghi789
  Trace started: trace-jkl012
  Trace completed
  
  Graph state after interaction:
    - 1 Message node
    - 1 Person entity (Sarah Johnson)
    - 1 Symptom fact (dizziness)
    - 1 Reasoning trace with 2 steps
    - 1 Tool call recorded
```

### 7.3 컨텍스트 검색

```python
async def get_patient_context(memory, patient_id, query):
    # 모든 메모리 유형에서 결합된 컨텍스트 가져오기
    context = await memory.get_context(
        query,
        session_id=patient_id
    )
    # 관련 증상 검색
    symptoms = await memory.long_term.search_entities(
        'dizziness symptoms'
    )
    
    # 대화 기록 가져오기
    conversation = await memory.short_term.get_conversation(patient_id)
    
    # 유사한 과거 사례 찾기
    similar_cases = await memory.reasoning.get_similar_traces(
        'dizziness diagnosis'
    )
    
    return {
        'context': context,
        'symptoms': symptoms,
        'conversation': conversation,
        'similar_cases': similar_cases
    }
```

```
  Patient Sarah Johnson reports dizziness, especially in the morning.

  Similar past cases suggest monitoring blood pressure.
  
  === Related Symptoms (2) ===
    - dizziness
    - vertigo
  
  === Conversation History (1 messages) ===
    [user]: I have been feeling dizzy lately, especially in...

    - Vertigo assessment: Refer to ENT specialist
```

## 8. 모범 사례 (Best Practices)

### 8.1 올바른 시스템 선택

*   **프로덕션 엔터프라이즈 용도(Production Enterprise Use)**: SLA가 있는 관리형 서비스(managed service)를 원한다면 Zep을, 유연한 배포를 원한다면 Mem0을 선택하세요.
*   **오픈소스 유연성(Open Source Flexibility)**: 시간 인지형 지식 그래프(temporal knowledge graphs)를 원한다면 Graphiti를, ECL 파이프라인을 원한다면 Cognee를 선택하세요.
*   **Neo4j 통합(Neo4j Integration)**: 네이티브 세 가지 메모리 유형 아키텍처(native three-memory-type architecture)를 원한다면 neo4j-agent-memory를 선택하세요.
*   **LangGraph 프로젝트(LangGraph Projects)**: 긴밀한 프레임워크 통합을 원한다면 LangMem을 선택하세요.

### 8.2 구현 팁 (Implementation Tips)

1.  **로컬에서 시작(Start Local)**: 그래프 데이터베이스용 Docker 컨테이너를 사용하여 로컬 개발부터 시작하세요.
2.  **개체 추출 파이프라인(Entity Extraction Pipeline)**: 프로덕션(production) 환경에서는 다단계 추출(spaCy + GLiNER + LLM fallback)을 사용하세요.
3.  **적극적인 가지치기(Prune Aggressively)**: 그래프 성능을 유지하기 위해 오래된 메모리를 제거하세요.
4.  **조기 시각화(Visualize Early)**: 그래프 시각화를 사용하여 메모리 구조를 디버그하고 이해하세요.
5.  **도구 호출 모니터링(Monitor Tool Calls)**: 에이전트의 의사 결정 과정을 이해하기 위해 추론 추적(reasoning traces)을 추적하세요.

### 8.3 일반적인 함정 (Common Pitfalls)

*   **메모리 과부하(Memory Bloat)**: 가지치기(pruning)가 없으면 그래프가 다루기 힘들어질 수 있습니다. 보존 정책(retention policies)을 구현하세요.
*   **개체 중복(Entity Duplication)**: 중복 노드(duplicate nodes)를 피하기 위해 적절한 중복 제거 전략을 구성하세요.
*   **요율 제한(Rate Limiting)**: LLM 기반 추출은 요율 제한에 도달할 수 있습니다. 로컬 모델을 사용하거나 백오프(backoff)를 구현하세요.
*   **스키마 드리프트(Schema Drift)**: 업데이트 전반에 걸쳐 일관된 개체 유형(entity types)과 관계를 유지하세요.

## 9. 결론

그래프 메모리 시스템은 AI 에이전트 기능의 중요한 발전입니다. 대화, 개체 및 추론 추적을 통합된 그래프 구조에 저장함으로써 이러한 시스템은 에이전트가 다음을 수행할 수 있도록 합니다:

*   **컨텍스트 유지(Maintain Context)**: 이전 대화 및 사용자 선호도를 세션 전반에 걸쳐 기억
*   **지식 구축(Build Knowledge)**: 비정형 데이터에서 개체를 자동으로 추출 및 연결
*   **결정 설명(Explain Decisions)**: 결정이 이루어진 이유를 이해하기 위해 추론 단계(reasoning steps)를 역추적
*   **학습 및 개선(Learn and Improve)**: 과거 상호 작용을 분석하여 미래 성능 향상

종합적인 세 가지 메모리 유형 아키텍처를 위해 neo4j-agent-memory를 선택하든, 시간 인지형 지식 그래프를 위해 Zep/Graphiti를 선택하든, ECL 파이프라인을 위해 Cognee를 선택하든, 프로덕션 준비가 된 관리형 서비스를 위해 Mem0을 선택하든, 그래프 메모리는 중요한 AI 에이전트 배포의 필수 구성 요소가 되고 있습니다.

AI 에이전트의 미래는 더 나은 모델뿐만 아니라 인간 지식과 경험의 풍부하고 상호 연결된 특성을 포착할 수 있는 더 나은 메모리 시스템에 있습니다.
