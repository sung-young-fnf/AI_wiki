# PageIndex를 활용한 추론 기반 RAG로 검색 성능 개선하기

검색 증강 생성(RAG, Retrieval-augmented generation)은 대규모 문서군에 포함된 외부 지식을 거대언어모델(LLM)에 추가하는 기술입니다. 기존의 RAG는 최적화된 벡터 데이터베이스(Vector database)를 사용하여 임베딩 벡터(Embedding vector)를 효율적으로 저장하고, 주어진 쿼리에 적합한 일치 항목을 찾아냅니다.

OpenAI의 o1 모델 시리즈 출시 이후, 많은 LLM이 내부 사고 과정을 통한 추론(Reasoning) 능력을 갖추게 되었습니다. 이에 따라 인간이 정보를 검색하는 방식과 유사하게 추론을 활용하여 관련 정보를 찾는 차세대 RAG가 등장하고 있습니다. 본 기사에서는 문서의 목차(ToC)를 주로 고려하는 벡터리스(Vector-less) 추론 기반 RAG 알고리즘인 'PageIndex'에 대해 살펴봅니다.

### 전통적인 벡터 기반 RAG의 작동 방식

현재 기본적인 RAG 시스템은 문서 집합, 임베딩 모델, 검색을 위한 벡터 데이터베이스, 그리고 답변 생성을 위한 LLM으로 구성됩니다. 워크플로우는 크게 오프라인 단계와 온라인 단계로 나뉩니다.

*   **오프라인 단계 (문서 청킹, 임베딩 및 벡터 데이터베이스 저장):**
    1. 대규모 문서 집합을 관리하기 쉬운 작은 텍스트 단위인 '청크(Chunk)'로 나눕니다.
    2. 각 청크는 임베딩 모델을 통해 수치형 벡터로 변환됩니다.
    3. 이 임베딩은 고차원 벡터 검색에 최적화된 벡터 데이터베이스에 저장됩니다.

*   **온라인 단계 (사용자 쿼리부터 LLM 답변 생성까지):**
    1. 사용자가 질문이나 지시사항이 담긴 쿼리를 제출합니다.
    2. 임베딩 모델이 쿼리를 임베딩 벡터로 변환합니다.
    3. 시스템은 쿼리 벡터와 저장된 모든 문서 청크 벡터를 비교하여 상위 K개의 가장 유사한 청크를 찾습니다.
    4. 선택된 청크들은 추가 컨텍스트(Context)로서 사용자의 원래 프롬프트에 추가됩니다.
    5. 마지막으로 LLM은 지식 베이스에서 검색된 정보를 바탕으로 더 정확하고 근거 있는 답변을 생성합니다.

### 벡터 기반 RAG 시스템의 한계

RAG 시스템은 "가비지 인, 가비지 아웃(Garbage in, garbage out)"의 전형적인 사례입니다. LLM은 사용자의 쿼리에 답하기 위해 리트리버(Retriever, 검색기)가 제공하는 컨텍스트에 의존합니다. 하지만 리트리버가 올바른 정보를 찾지 못하거나, 너무 많은 검색 결과 속에 핵심 정보가 묻혀버리는 등 검색 단계에서 실패하는 경우가 많습니다.

주요 실패 원인은 다음과 같습니다.

1.  **청킹(Chunking):** 수백 페이지 분량의 문서를 작은 조각으로 나누는 과정에서 논리적 단위가 단절될 수 있습니다. 예를 들어, 법률 문서에서 특정 조항이 다른 조항을 참조할 때 해당 내용이 서로 다른 청크로 나뉘면 문맥 파악이 어려워집니다.
2.  **시맨틱 검색(Semantic Search):** 코사인 유사도(Cosine similarity) 등을 이용한 수학적 비교는 항상 완벽하지 않습니다. 모호한 사용자 쿼리나 긴 텍스트를 하나의 벡터로 압축하는 과정에서 발생하는 정보 손실은 검색 오류로 이어질 수 있습니다.

### PageIndex를 활용한 추론 기반 검색

2024년 말 추론형 LLM이 등장하면서 RAG와 추론을 결합하려는 연구가 시작되었습니다. PageIndex는 추론형 LLM을 사용하여 정보를 추출하는 '추론 증강 검색(Reasoning-augmented retrieval)' 방식입니다. 이 방식은 벡터 데이터베이스나 임베딩 기반의 시맨틱 검색을 사용하지 않습니다.

PageIndex 방식에서 추론형 LLM은 생성된 문서의 목차(ToC)를 검토하여 사용자 쿼리와 관련된 섹션을 결정합니다. 선택된 섹션을 읽은 후, LLM은 추가 정보가 더 필요한지 판단합니다.

이 접근 방식은 교과서, 법률 또는 금융 문서와 같이 구조가 잘 잡힌 문서에서 매우 효과적입니다. 추론 프로세스 루프를 통해 LLM은 연결된 정보들을 찾아낼 수 있습니다. 예를 들어, 처음 추출한 섹션에 다른 장에 대한 참조가 포함되어 있다면, LLM은 다시 돌아가 해당 참조 내용을 찾아낼 수 있습니다.

### PageIndex 트리 구조

PageIndex는 문서의 목차를 JSON 형식의 계층적 트리 구조로 생성합니다. 각 노드(Node)는 장, 절, 문단 또는 페이지와 같은 논리적 섹션을 나타내며 메타데이터, 요약, 하위 섹션 링크를 포함합니다.

LLM은 이 트리를 읽고 쿼리에 적합한 `node_id` 목록을 출력합니다. 각 `node_id`는 문서의 원본 텍스트 데이터와 매핑되어 최종적으로 LLM의 컨텍스트로 추가됩니다.

### PageIndex의 한계점

현재 PageIndex는 몇 가지 한계가 있습니다.

1.  **파일 형식 제한:** 현재는 PDF 파일만 처리할 수 있습니다. 지저분한 실제 문서를 처리할 때 목차 추출 과정에서 오류가 발생할 가능성이 있습니다.
2.  **높은 비용 및 시간:** 트리 구조와 요약을 생성하는 과정이 시간 소모적이며 연산 집약적입니다. 또한 온라인 쿼리마다 발생하는 추론용 LLM 호출은 실제 서비스 환경에서 토큰 비용 부담을 높일 수 있습니다.
3.  **확장성 문제:** 방대한 문서 컬렉션에서 적절한 문서를 먼저 찾는 기능이 부족합니다. 이를 해결하기 위해 메타데이터 필터링이나 기존의 시맨틱 검색을 병행하는 하이브리드 방식이 제안되고 있지만, 이는 효율성 측면에서 개선의 여지가 있습니다.

### 결론: 벡터 기반 vs 추론 기반 RAG

전통적인 벡터 기반 RAG는 대규모 문서 컬렉션에서 효율적이지만 결과가 항상 완벽하지는 않습니다. 반면 PageIndex와 같은 추론 기반 RAG는 인간이 책을 찾아보는 것과 유사한 방식으로 정밀하게 섹션을 찾아내지만, 단일 문서가 아닌 방대한 데이터로 확장하기에는 아직 비효율적입니다.

결론적으로, 검색 시스템에 추론형 LLM을 통합하는 것은 시스템을 더 지능적으로 만들고 모호한 쿼리를 처리하는 데 매우 유용한 전략입니다. 머지않은 미래에 벡터 기반의 효율성과 추론 기반의 지능을 결합한 하이브리드 검색 시스템이 주류가 될 것으로 보입니다.
