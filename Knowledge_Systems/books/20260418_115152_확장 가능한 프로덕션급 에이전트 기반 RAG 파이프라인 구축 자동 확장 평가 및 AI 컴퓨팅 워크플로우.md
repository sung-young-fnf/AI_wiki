# 확장 가능한 프로덕션급 에이전트 기반 RAG 파이프라인 구축: 자동 확장, 평가 및 AI 컴퓨팅 워크플로우

대규모 데이터셋과 연동하는 산업 표준 에이전트 기반 RAG(Retrieval Augmented Generation) 파이프라인은 명확하고 확장 가능한 계층으로 구축되어야 합니다. 에이전트가 병렬로 추론하고, 컨텍스트를 가져오며, 도구를 사용하고, 데이터베이스와 통신할 수 있도록 시스템을 구성합니다. 각 계층은 데이터 수집부터 모델 서빙 및 에이전트 조정까지 특정 책임을 처리합니다. 이러한 계층화된 접근 방식은 시스템이 안정적으로 확장되고 최종 사용자의 응답 시간을 낮게 유지하는 데 도움이 됩니다.

에이전트를 중심으로 구축된 확장 가능한 RAG 파이프라인은 일반적으로 6개의 핵심 계층을 가집니다.

*   **데이터 수집 계층 (Data Ingestion Layer):** 원시 데이터를 문서 로딩, 청킹(chunking) 및 인덱싱을 통해 구조화된 지식으로 변환합니다. S3, 관계형 데이터베이스(RDBMS), Ray를 통해 확장 가능합니다.
*   **AI 컴퓨팅 계층 (AI Compute Layer):** 대규모 저지연 추론을 위해 LLM(Large Language Model) 및 임베딩을 GPU/CPU에 효율적으로 매핑하여 실행합니다.
*   **에이전트 기반 AI 파이프라인 (Agentic AI Pipeline):** API, 캐싱 및 분산 실행을 통해 에이전트의 추론, 쿼리 강화 및 워크플로우 오케스트레이션을 지원합니다.
*   **도구 및 샌드박스 (Tools & Sandbox):** 프로덕션 워크로드에 영향을 주지 않으면서 계산, 검색 및 API 테스트를 위한 안전한 환경입니다.
*   **코드형 인프라 (Infrastructure as Code, IaC):** 재현 가능하고 확장 가능한 인프라를 위해 배포, 네트워킹, 클러스터 설정 및 자동 확장을 자동화합니다.
*   **배포 및 평가 (Deployment & Evaluation):** 대규모 환경에서 안정적인 운영을 보장하기 위해 비밀(secrets), 데이터베이스, 클러스터, 모니터링 및 로깅을 처리합니다.

이 블로그에서는 확장 가능한 시스템을 구축하는 방법을 보여주면서 에이전트가 포함된 RAG 파이프라인의 이 여섯 가지 계층을 모두 구축할 것입니다.

## 노이즈 데이터가 포함된 빅데이터

일반적으로 기업 코드베이스에는 프레젠테이션, 기술 문서, 보고서, 웹 페이지 등 다양한 유형의 문서가 포함되며, 이 중에는 RAG 파이프라인을 신뢰할 수 있고 정확하게 만들기 위해 사용되는 노이즈 데이터도 항상 존재합니다.

우리는 Kubernetes의 실제 문서를 '진실 데이터'로 사용하고, 실제 기업 코드베이스를 시뮬레이션하기 위해 다양한 주제의 무작위 PDF가 포함된 `tpn/pdfs` 저장소에서 95%의 노이즈 데이터를 추가할 것입니다. 이 방법을 통해 우리는 과도한 노이즈가 존재하는 환경에서 기업 RAG 파이프라인이 얼마나 잘 작동하는지 실제로 평가할 수 있습니다.

`tpn/pdfs` 저장소를 클론하여 노이즈 데이터를 가져올 것이며, 이 저장소에는 PDF, DOCX, TXT, HTML 파일 등 다양한 유형의 문서가 포함되어 있습니다. 이 저장소에서 950개의 문서를 무작위로 샘플링하여 `noisy_data` 디렉토리에 저장합니다. '진실 지식'을 위해 오픈소스 Kubernetes 공식 문서를 스크랩하여 `true_data` 디렉토리에 PDF, DOCX, TXT, HTML 파일 등 여러 형식으로 저장했습니다.

이제 노이즈 데이터와 진실 데이터를 하나의 `data` 디렉토리로 병합하여 총 1000개의 문서를 갖게 되며, 이 중 950개는 노이즈 문서이고 50개는 진실 문서입니다.

이제 엔터프라이즈급 RAG 파이프라인을 구축하는 다음 단계로 넘어갑시다.

## 엔터프라이즈 코드베이스 생성

일반적으로 에이전트 기반 RAG 파이프라인 코드베이스는 단일 벡터 데이터베이스, 몇 개의 AI 모델, 간단한 수집 파이프라인으로 구성됩니다. 하지만 이 파이프라인의 복잡성이 증가함에 따라 전체 아키텍처를 더 작고 관리하기 쉬운 구성 요소로 분해해야 합니다.

RAG 파이프라인을 위한 구조화된 디렉토리 레이아웃을 생성합니다.

*   `deploy/`: Ray, 인그레스(ingress) 컨트롤러, 비밀(secrets) 관리 등 다양한 구성 요소에 대한 배포 구성을 포함합니다.
*   `infra/`: Terraform 및 Karpenter를 사용하여 클라우드 리소스를 설정하기 위한 코드형 인프라(IaC) 스크립트입니다.
*   `pipelines/`: 문서 로더, 청킹, 임베딩 계산, 그래프 추출 및 인덱싱을 포함한 수집 및 작업 관리 파이프라인입니다.
*   `services/`: API 서버, 게이트웨이 구성, 신뢰할 수 없는 코드를 실행하기 위한 샌드박스 환경을 포함한 주요 애플리케이션 서비스입니다.

많은 구성 요소들이 (로더, 청킹과 같은) 자체 하위 폴더를 가지고 있어 책임 분리를 강화하고 코드베이스 유지보수를 용이하게 합니다.

### 개발 워크플로우 관리

에이전트 아키텍처를 코딩하기 전 가장 첫 번째 단계는 로컬 개발 환경을 설정하는 것입니다. 확장된 프로젝트는 새로운 개발자가 팀에 합류할 때마다 수동 설정을 피하기 위해 이를 자동화합니다.

이러한 개발 설정은 일반적으로 세 가지 방식으로 제공됩니다.

*   `.env.example`: 로컬 개발에 필요한 환경 변수를 공유하는 표준 방식입니다. 개발자는 이 파일을 `.env`로 복사하여 (개발, 스테이징, 프로덕션 단계에 따라) 자신의 값을 채워 넣을 수 있습니다. 여기에는 애플리케이션 설정, 데이터베이스(Aurora Postgres), 캐시(Redis), 벡터 데이터베이스(Qdrant), 그래프 데이터베이스(Neo4j) 및 AWS, Ray 클러스터, 가시성(Observability) 설정이 포함됩니다. 우리는 RAG 파이프라인에서 채팅 기록 및 메타데이터 저장을 위한 Aurora Postgres, 자주 액세스하는 데이터 캐싱을 위한 Redis, 임베딩 저장을 위한 벡터 데이터베이스로 Qdrant, 엔티티 간의 관계 저장을 위한 그래프 데이터베이스로 Neo4j를 사용합니다. 또한 효율적인 문서 저장 및 검색을 위해 AWS S3를 주요 문서 저장소로 사용하며, AI 모델(LLM, 임베딩, 재랭커)을 호스팅하기 위해 클러스터화된 Ray 서빙을 함께 사용합니다.
*   `Makefile`: 빌드, 테스트, 배포 등 일반적인 작업을 자동화하는 명령어를 포함합니다. 설치, 개발, 로컬 데이터베이스 시작/중지, AWS EKS 배포, Terraform 적용, 테스트와 같은 명령어를 정의합니다. 이는 많은 확장된 프로젝트에서 가장 일반적으로 사용되는 빌드 자동화 도구입니다.
*   `docker-compose.yml`: Docker 컨테이너를 사용하여 전체 RAG 파이프라인을 로컬에서 실행하는 데 필요한 모든 서비스를 정의합니다. Postgres, Redis, Qdrant, Neo4j, MinIO (S3 Mock) 서비스가 포함되어 있으며, 각 서비스는 충돌 방지를 위해 다른 포트를 사용하고 종속성을 격리합니다.

### 핵심 공유 유틸리티

이제 프로젝트 구조와 개발 워크플로우를 설정했으니, 가장 먼저 필요한 것은 고유 ID 생성 전략입니다. 사용자가 RAG 봇에 채팅을 보내면 많은 일이 동시에 발생하며, 이를 모두 함께 매핑하면 RAG 파이프라인의 다양한 구성 요소에서 해당 특정 채팅 세션과 관련된 문제를 추적하는 데 도움이 됩니다. 이는 프로덕션 시스템에서 사용자 작업(클릭 또는 요청 등)별로 모든 것을 연관시켜 나중에 모니터링, 디버깅 및 추적을 용이하게 하는 매우 일반적인 접근 방식입니다.

이를 위해 `libs/utils/ids.py` 파일을 생성하여 채팅 세션, 파일 업로드 및 OpenTelemetry 추적을 위한 고유 ID 생성을 처리합니다.
유사하게, 성능 모니터링 및 최적화를 위해 RAG 파이프라인의 다양한 함수의 실행 시간을 측정해야 합니다. `libs/utils/timing.py` 파일을 생성하여 동기 및 비동기 함수의 실행 시간 측정을 처리하는 데코레이터를 정의합니다.
마지막으로, 재시도 메커니즘이 필요하며, 프로덕션급 RAG 시스템에서는 일반적으로 지수 백오프(exponential backoff)를 사용하여 RAG 파이프라인의 오류를 처리합니다. `libs/retry/backoff.py` 파일을 생성하여 지터(jitter)가 포함된 지수 백오프를 구현합니다.

## 데이터 수집 계층 (Data Ingestion Layer)

RAG 파이프라인의 첫 번째 구성 요소는 크기에 관계없이 문서를 시스템에 수집하는 것입니다. 소규모 프로젝트에서는 파일을 순차적으로 읽는 간단한 스크립트로 충분합니다. 하지만 기업 RAG 파이프라인에서 수집은 API 서버를 충돌시키지 않으면서 수천 개의 파일을 동시에 처리해야 하는 고처리량(high-throughput), 비동기 작업입니다.

우리는 Ray Data를 사용하여 수집 로직을 분산 파이프라인으로 분리합니다. Ray Data를 사용하면 클러스터의 여러 노드에서 병렬로 실행할 수 있는 DAG(Directed Acyclic Graph) 작업을 생성할 수 있습니다. 이를 통해 구문 분석(CPU-bound) 및 임베딩(GPU-bound) 작업을 독립적으로 확장할 수 있습니다.

### 문서 로딩 및 구성

먼저, 수집 매개변수를 관리하기 위한 중앙 집중식 구성이 필요합니다. 프로덕션 환경에서 청크(chunk) 크기나 데이터베이스 컬렉션과 같은 값을 하드코딩하는 것은 재앙을 초래할 수 있습니다. `pipelines/ingestion/config.yaml` 파일을 생성하여 수집 파이프라인의 모든 구성을 포함합니다.

이제 로더가 필요합니다. 기업 시스템에서 PDF는 용량이 큽니다. 100MB PDF를 RAM에 로드하면 Kubernetes 워커에서 OOM(Out of Memory) 오류가 발생할 수 있습니다. 따라서 `pipelines/ingestion/loaders/pdf.py`와 같이 별도의 처리 파일을 사용하여 `unstructured`와 임시 파일을 활용하여 메모리를 효율적으로 처리합니다. 다른 형식의 경우 더 가벼운 파서를 사용합니다. Word 문서를 위한 `pipelines/ingestion/loaders/docx.py`와 웹 콘텐츠를 위한 `pipelines/ingestion/loaders/html.py`를 생성하여 CSS나 JavaScript 코드로 벡터를 오염시키지 않도록 스크립트와 스타일을 제거합니다.

### 청킹(Chunking) 및 지식 그래프

원시 텍스트가 추출되면, RAG에서는 이를 변환해야 합니다. `pipelines/ingestion/chunking/splitter.py`에 스플리터(splitter)를 정의하여 텍스트를 512토큰 청크로 분할하는데, 이는 많은 임베딩 모델의 표준 제한입니다. 또한 `pipelines/ingestion/chunking/metadata.py`에서 이러한 청크를 보강합니다. 분산 시스템에서는 중복 제거가 중요하므로, 콘텐츠 해시를 생성합니다.

이제 GPU 작업인 임베딩 생성을 수행합니다. 수집 스크립트 내부에 모델을 로드하는 대신(느린 시작 시간), Ray Serve 엔드포인트를 호출합니다. 이를 통해 수집 작업은 지속적으로 실행되는 모델 서비스에 단순히 HTTP 요청을 보낼 수 있습니다. 이를 별도로 관리하기 위해 `pipelines/ingestion/embedding/compute.py`를 생성해야 합니다.

동시에 지식 그래프를 추출합니다. 그래프를 깨끗하게 유지하려면 `pipelines/ingestion/graph/schema.py`에 엄격한 스키마를 정의해야 합니다. 이렇게 하지 않으면 LLM이 무작위 관계 유형을 환각(hallucinate)할 수 있습니다. 이 스키마는 `pipelines/ingestion/graph/extractor.py`에 적용됩니다. 이는 텍스트의 구조를 이해하기 위해 LLM을 사용하며, 단순히 의미론적 유사성만을 고려하지 않습니다.

### 고처리량(High-Throughput) 인덱싱

대규모 RAG에서는 레코드를 하나씩 삽입하지 않습니다. 배치(batch) 쓰기를 수행합니다. 배치 처리는 GPU 및 CPU 워크로드를 줄여 메모리 사용량을 절감하므로 나중에 다른 작업에 파티션을 사용할 수 있습니다.

벡터의 경우, `pipelines/ingestion/indexing/qdrant.py`를 사용합니다. 이는 Qdrant 클러스터에 대한 연결을 처리하고 원자적(atomic) upsert를 수행합니다.
그래프의 경우, `pipelines/ingestion/indexing/neo4j.py`를 생성합니다. Cypher MERGE 문을 사용하여 수집을 두 번 실행해도 중복 노드가 발생하지 않도록 멱등성(idempotency)을 보장하며, 이는 CPU 또는 GPU 노드에서 발생할 수 있는 메모리 문제를 방지합니다.

### 이벤트 기반 워크플로우 (Event-Driven Workflows) with Ray

마지막으로, 이 모든 구성 요소를 함께 결합해야 합니다. 단일 스크립트 대신, CPU 클러스터의 여러 노드에서 읽기, 청킹, 임베딩이 병렬로 발생하는 파이프라인을 원하므로, `pipelines/ingestion/main.py`를 우리의 지휘자로 생성해야 합니다. 우리는 Ray Data를 사용하여 지연 실행 DAG(Directed Acyclic Graph)를 생성할 것입니다.

이것을 Kubernetes 클러스터에서 실행하기 위해 `pipelines/jobs/ray_job.yaml`에 런타임 환경을 정의합니다. 이는 워커(worker)가 필요한 모든 Python 종속성을 설치했는지 확인합니다.

엔터프라이즈 아키텍처에서는 이를 수동으로 트리거하지 않습니다. 이벤트 기반 패턴을 사용합니다. 파일이 S3에 도착하면, `pipelines/jobs/s3_event_handler.py`에 정의된 람다(Lambda) 함수를 트리거하는 이벤트가 발생합니다.

그리고 마지막으로, 이 전체 프로세스를 테스트하기 위해 `scripts/bulk_upload_s3.py`를 사용하여 이전에 준비한 노이즈 데이터셋을 멀티 스레드 업로드 방식으로 S3 버킷에 대량 업로드합니다.

이제 데이터 수집 계층을 구축했습니다. 이제 Kubernetes 클러스터에 노드를 더 추가하는 것만으로 수백만 개의 문서로 확장할 수 있는 시스템을 갖추게 되었습니다.

다음 부분에서는 분산 패턴을 사용하여 모델을 포함하는 모델 서버 계층을 구축할 것입니다.

## AI 컴퓨팅 계층 (AI Compute Layer)

이제 데이터를 수집하는 파이프라인이 있으므로 해당 데이터를 처리할 프로세스가 필요합니다. 모놀리식(monolithic) 애플리케이션에서는 API 서버 내부에 LLM을 직접 로드할 수 있지만, 기업 RAG 플랫폼에서는 이는 치명적인 실수입니다. 70B 매개변수 모델을 웹 서버 내부에 로드하면 요청 처리량이 저하되고 확장이 불가능해집니다.

FastAPI를 AI 모델로부터 분리해야 합니다. Ray Serve를 사용하여 GPU 가용성과 수신 트래픽에 따라 자동 확장되는 독립적인 마이크로서비스로 모델을 호스팅할 것입니다.

### 모델 구성 및 하드웨어 매핑

프로덕션 환경에서는 모델 매개변수를 하드코딩하지 않습니다. 코드를 다시 작성하지 않고도 모델을 교체하고, 양자화 수준을 조정하며, 배치(batch) 크기를 튜닝할 수 있는 유연한 구성이 필요합니다.

`models/llm/llama-70b.yaml`에 우리의 주요 모델을 정의합니다. Llama-3-70B-Instruct를 사용하지만, 70B 매개변수가 FP16에서 약 140GB의 VRAM을 필요로 하므로, AWQ 양자화(quantization)를 사용하여 더 저렴한 GPU에 맞춥니다. 이는 많은 회사가 RAG 시스템의 주요 구성 요소로 최종 AI 모델보다는 데이터와 에이전트에 중점을 두는 방식과 일치합니다.
`tensor_parallel_size: 4`에 주목하십시오. 이는 엔터프라이즈급 구성입니다. 이는 서빙 엔진에게 이 단일 모델이 하나의 GPU에는 너무 크므로, 가중치 행렬을 4개의 GPU에 걸쳐 원활하게 분할해야 한다고 알려줍니다.
또한 쿼리 재작성 또는 요약과 같은 가벼운 작업을 위해 비용을 절감할 수 있는 더 작은 모델인 `models/llm/llama-7b.yaml`에 대한 구성을 유지합니다.

검색 측면에서는 `models/embeddings/bge-m3.yaml`에 임베딩 모델 구성을 정의합니다. BGE-M3는 조밀(dense), 희소(sparse), 다국어 검색을 모두 처리하므로, 영어가 아닌 언어로 질문할 수 있는 글로벌 엔터프라이즈 플랫폼에 중요합니다.

마지막으로, 정확도를 향상시키기 위해 `models/rerankers/bge-reranker.yaml`에 정의된 재랭커(reranker)를 사용할 것입니다. 이 모델은 LLM에 도달하기 전에 상위 검색된 문서를 재평가하여 오탐(false positives)을 필터링하므로, 환각(hallucination)을 크게 줄입니다.

### vLLM 및 Ray를 이용한 AI 모델 서빙

이제 실제로 이 모델들을 실행해야 합니다. 표준 HuggingFace 파이프라인은 고동시성(high-concurrency) 프로덕션 사용에는 너무 느립니다. 우리는 PagedAttention을 사용하여 메모리를 효율적으로 관리하는 고처리량 서빙 엔진인 vLLM을 사용할 것입니다.

vLLM을 `services/api/app/models/vllm_engine.py`의 Ray Serve 배포(deployment)로 래핑합니다. 이 스크립트는 엔진 초기화를 처리하고 생성을 위한 비동기 엔드포인트를 노출합니다. `@serve.deployment`를 사용함으로써 Ray에게 이 클래스의 수명 주기를 관리하도록 지시합니다. `autoscaling_config`는 여기서 핵심 엔터프라이즈 기능입니다. 트래픽이 급증하면 Ray는 부하를 감지하고 자동으로 더 많은 복제본을 생성하며, 유휴 상태일 때는 클라우드 비용을 절감하기 위해 스케일 다운합니다.

### 임베딩 및 재랭커 서빙

임베딩 모델은 vLLM의 무거운 백엔드를 필요로 하지 않지만, 효율적인 배치(batching) 처리가 필요합니다. 50명의 사용자가 동시에 검색하면, 50번의 순차적인 패스가 아닌 단일 GPU 패스에서 50개의 모든 쿼리를 인코딩하고 싶습니다.

이를 `services/api/app/models/embedding_engine.py`에서 구현합니다. `num_gpus: 0.5`를 할당하는 방식을 주목하십시오. 이를 통해 두 개의 임베딩 모델을 하나의 GPU에 패킹할 수 있어 엄청난 비용 절감 효과를 가져옵니다. 이 배포는 문서 임베딩(수집 중)과 쿼리 임베딩(채팅 중)을 동일한 공유 리소스 풀을 사용하여 처리합니다. `torch.compile`을 사용하여 모델 실행 그래프를 최적화하여 GPU의 모든 성능을 최대한 활용합니다.

### 비동기 내부 클라이언트

마지막으로, API 계층은 이러한 Ray 서비스와 통신할 방법이 필요합니다. 모델이 별도의 마이크로서비스로 실행되므로 HTTP를 통해 통신합니다. GPU를 기다리는 동안 FastAPI 서버가 응답성을 유지하도록 비블로킹(non-blocking) 클라이언트가 필요합니다. `services/api/app/clients/ray_llm.py`를 생성하여 연결 풀링(connection pooling) 및 재시도 기능을 갖춘 LLM 통신을 처리합니다. 여기서 `backoff`의 사용은 분산 시스템에서 중요합니다. 네트워크가 끊기거나 Ray가 바쁜 경우에도 시스템이 충돌하지 않고, 지수적으로 대기하며 재시도하여 고가용성을 보장합니다.

마찬가지로, 임베딩 서비스를 위한 `services/api/app/clients/ray_embed.py`를 생성합니다. 이 클라이언트는 "쿼리" 임베딩(검색용)과 "문서" 임베딩(수집용) 사이의 구분을 처리합니다.

이 클라이언트들이 이제 코딩됨으로써, 우리의 API 코드(다음에 구축할)는 70B 매개변수 모델을 단순한 비동기 함수 호출로 취급할 수 있게 되어, GPU 관리 및 분산 추론의 복잡성을 완전히 추상화합니다.

다음 부분에서는 에이전트 기반 RAG로 변환하기 위한 에이전트 기반 파이프라인(Agentic pipeline), 즉 사용자 질문에 답하기 위해 이러한 모델을 언제 사용할지 결정하는 로직을 구축할 것입니다.

## 에이전트 기반 AI 파이프라인 (Agentic AI Pipeline)

데이터 수집 파이프라인과 분산 클러스터에서 AI 모델 서빙을 갖췄습니다. 간단한 RAG 애플리케이션에서는 검색 호출을 LLM 생성 호출에 연결할 수도 있습니다. 그러나 기업 에이전트 기반 플랫폼의 경우 선형 체인은 취약합니다. 사용자가 주제를 변경하거나, 수학을 묻거나, 모호하게 말할 때 실패합니다.

FastAPI와 LangGraph를 사용하여 이벤트 기반 에이전트(Event-Driven Agent)를 구축할 것입니다. 이를 통해 시스템은 사용자의 의도를 '추론'하고, 자체적으로 루프를 돌며 수정하고, 도구를 동적으로 선택하면서 수천 개의 동시 웹소켓 연결을 비동기적으로 처리할 수 있습니다.

### API 기반 및 가시성 (API Foundation & Observability)

먼저 환경 및 보안 표준을 정의합니다. 기업 API는 블랙박스일 수 없으므로, 특정 쿼리가 500ms 대신 5초가 걸린 이유를 디버깅하기 위해 구조화된 로그와 추적(tracing)이 필요합니다.

`services/api/requirements.txt`에서 종속성을 확인합니다. 속도를 위한 `fastapi`, 오케스트레이션을 위한 `langgraph`, 가시성(observability)을 위한 `opentelemetry`를 포함합니다. 이 요구사항 파일은 고성능 시스템에 필요한 모든 비동기 드라이버(asyncpg, redis) 및 가시성 도구(opentelemetry)를 갖추고 있음을 보장합니다. `backoff`와 `tenacity`의 포함은 내부 호출 실패 시 자동으로 재시도하는 로직의 신뢰성을 강조합니다.

다음으로, Pydantic Settings를 사용하여 `services/api/app/config.py`를 생성합니다. 이는 모든 데이터베이스 URL과 API 키가 시작 시 존재함을 검증하여 나중에 런타임 충돌을 방지합니다. 이는 전체 API의 중앙 소스 역할을 합니다. Pydantic을 사용함으로써 `JWT_SECRET_KEY`와 같은 중요한 비밀(secret)이 누락된 경우, 파드가 즉시 시작에 실패하여("Fail Fast" 원칙), 운영 팀에 즉시 경고합니다.

Datadog 또는 Splunk와 같은 도구에서 기계가 읽을 수 있는 로그를 만들기 위해 `services/api/app/logging.py`에 구조화된 JSON 로깅을 구현할 것입니다. 표준 텍스트 로그는 50개 파드가 실행되는 대규모 환경에서는 쓸모가 없습니다. JSON 로그는 데이터베이스처럼 로그를 쿼리할 수 있게 하여, 오류 수준이나 특정 요청 ID로 필터링하여 분산 노드에서 버그를 추적할 수 있습니다.

또한 `services/api/app/observability.py`에서 분산 추적(distributed tracing)을 활성화합니다. 이는 API에서 Redis로, 다시 벡터 데이터베이스(Vector DB)로, 마지막으로 Ray 클러스터로 이어지는 요청 흐름을 추적합니다.

보안은 협상 불가능합니다. `services/api/app/auth/jwt.py`에 JWT(JSON Web Token) 유효성 검사를 구현합니다. 이 미들웨어는 승인된 사용자만 값비싼 GPU 리소스를 쿼리할 수 있도록 보장합니다.

마지막으로, `libs/schemas/chat.py`에 데이터 계약을 정의합니다. 이는 프론트엔드가 우리가 예상하는 것을 정확히 보내고 일관된 응답 구조를 받도록 보장합니다. 이러한 엄격한 타이핑(typing)은 스웨거(Swagger) 문서 자동 생성을 가능하게 하고, "쓰레기 데이터 유입(garbage-in)"으로 인해 파이프라인의 에이전트가 충돌하는 것을 방지합니다.

### 비동기 데이터 게이트웨이 (Asynchronous Data Gateways)

확장 가능한 API는 절대로 메인 스레드를 블로킹(blocking)해서는 안 됩니다. 데이터베이스가 데이터를 처리하는 동안 하나의 API 워커가 수백 개의 대기 중인 연결을 처리할 수 있도록 모든 데이터베이스에 대한 비동기 클라이언트가 필요합니다.

먼저 고속 캐시 및 속도 제한을 담당할 `services/api/app/clients/redis.py`를 구축해야 합니다. 이 싱글톤(singleton) 패턴은 연결을 효율적으로 재사용하여, 잘못 설계된 앱에서 자주 발생하는 "너무 많은 연결" 오류를 방지합니다.

다음으로 벡터 검색을 위한 `services/api/app/clients/qdrant.py`를 생성할 것입니다. 비블로킹(non-blocking) 유사성 검색을 수행하기 위해 Qdrant 클라이언트의 비동기 버전을 사용합니다.
그리고 그래프 검색을 위한 `services/api/app/clients/neo4j.py`를 생성합니다. 이 드라이버는 Cypher 쿼리를 효율적으로 실행하기 위한 연결 풀을 관리합니다.

### 문맥 메모리 및 의미론적 캐싱 (Contextual Memory & Semantic Caching)

에이전트는 그 메모리만큼만 똑똑합니다. LLM이 5턴 전의 대화를 기억할 수 있도록 Postgres를 사용하여 전체 대화 기록을 저장해야 합니다.

이를 위해 `services/api/app/memory/models.py`에 스키마를 정의해야 합니다. 그런 다음 `services/api/app/memory/postgres.py`에 CRUD 로직을 구현합니다. 가장 최근 컨텍스트를 LLM에 공급하기 위해 역시간 순서(reverse chronological order)로 기록을 가져옵니다.

비용을 절감하기 위해 `services/api/app/cache/semantic.py`에 의미론적 캐싱(Semantic Caching)을 구현합니다. 사용자가 "Kubernetes는 무엇인가요?"라고 묻고 다른 사용자가 "K8s를 설명해주세요"라고 물으면 임베딩 유사성이 높을 것입니다. 70B 모델에 GPU 크레딧을 소모하지 않고도 캐시된 답변을 즉시 제공할 수 있습니다. 이는 기업 환경에서 자주 묻는 질문(FAQ)의 지연 시간을 확실히 낮춥니다.

### LangGraph를 이용한 워크플로우 (Workflows with LangGraph)

이것이 에이전트의 핵심입니다. 우리는 대화를 상태 머신으로 취급합니다. 에이전트는 "고려 중(Thinking)", "검색 중(Retrieving)", "도구 사용 중(Using Tools)", "답변 중(Answering)" 사이를 전환할 수 있습니다.

먼저, `services/api/app/agents/state.py`에 `AgentState`를 정의합니다. 이 사전은 그래프 노드를 통해 흐르는 대화 컨텍스트를 전달합니다.
플래너 노드(`services/api/app/agents/nodes/planner.py`)는 사용자 쿼리를 보고 다음에 무엇을 할지 결정합니다. 이는 트래픽을 검색, 도구 또는 직접 답변으로 라우팅하는 라우터 역할을 합니다. 이 노드는 간단한 인사말이나 관련 없는 질문에 대해 시스템이 값비싼 검색을 수행하는 것을 방지하기 때문에 중요합니다.

또한 하이브리드 검색(Hybrid Search)을 실행하는 리트리버 노드(`services/api/app/agents/nodes/retriever.py`)가 필요합니다. 이는 `asyncio.gather`를 사용하여 Qdrant(벡터) 및 Neo4j(그래프)를 병렬로 호출하여 의미론적 검색과 구조적 검색 모두에서 최상의 결과를 제공합니다.
응답자 노드(`services/api/app/agents/nodes/responder.py`)는 검색된 문서를 가져와 Ray Serve를 통해 Llama-70B 모델을 사용하여 최종 답변을 합성합니다.

또한 LLM이 자체적으로 산술 연산을 시도하지 않도록(그들은 잘 못함) 외부 코드 또는 수학을 실행하기 위한 도구 노드(`services/api/app/agents/nodes/tool.py`)도 있습니다.
마지막으로, `services/api/app/agents/graph.py`에서 이 모든 것을 결합합니다. 이는 워크플로우를 정의합니다: 시작 -> 플래너 -> (리트리버 또는 도구) -> 응답자 -> 종료.

### 쿼리 강화 및 애플리케이션 진입점 (Query Enhancement & Application Entrypoint)

검색 정확도를 높이기 위해 고급 RAG 기술을 사용하며, 그중 하나는 `services/api/app/enhancers/hyde.py`에 구현할 HyDE(Hypothetical Document Embeddings)입니다. 이는 LLM이 가상의 답변을 환각하도록 요청하고, 이를 임베딩하여 유사한 의미 패턴을 가진 실제 문서를 찾는 데 사용합니다.

또한 `services/api/app/enhancers/query_rewriter.py`를 사용하여 상호 참조(예: "비용이 얼마나 드나요?")를 해결합니다. 이는 검색 엔진이 "Kubernetes 비용은 얼마인가요?"와 같은 완전한 쿼리를 얻도록 보장합니다.

최종 구성 요소는 `services/api/main.py`입니다. 이것이 애플리케이션 진입점입니다. 앱이 시작될 때 데이터베이스 연결을 초기화하고, 종료될 때 정상적으로 닫는 것과 같은 수명 주기 이벤트를 관리합니다.

이제 우리는 계획, 검색, 추론이 가능한 에이전트를 갖추었습니다. 그러나 에이전트 시스템은 외부 지식과 상호 작용해야 합니다. 다음 섹션에서는 에이전트가 안전하게 코드를 실행하고 웹을 검색할 수 있도록 도구 및 샌드박스 계층을 구축할 것입니다.

## 도구 및 샌드박스 (Tools & Sandbox)

기업 RAG 플랫폼에서 AI 모델에 코드 실행 또는 내부 데이터베이스 쿼리에 대한 직접 액세스 권한을 부여하는 것은 엄청난 보안 위험입니다. 이런 식으로 해커는 프롬프트를 통해 LLM을 환각시키고 중요한 데이터베이스 정보에 접근할 수 있습니다.

우리는 계층화된 전략이 필요합니다. 안전하고 결정론적인 도구 세트를 구축하고, 위험한 작업(Python 실행 등)을 위해 격리되고 강화된 샌드박스 환경을 만들 것입니다.

### 보안 코드 샌드박스 (Secure Code Sandbox)

LLM이 Python 코드를 생성하고 실행할 수 있게 하는 것은 강력합니다. 복잡한 수학 문제를 해결하고, 차트를 생성하거나, CSV를 분석할 수 있습니다. 하지만 이는 위험하기도 합니다. LLM이 `os.system("rm -rf /")`를 환각시키거나 데이터를 유출하려고 시도할 수 있습니다.

이 문제를 해결하기 위해 샌드박스 마이크로서비스(Sandbox Microservice)를 구축합니다. 이는 신뢰할 수 없는 코드를 실행하는 별도의 Docker 컨테이너입니다. 네트워크 액세스가 없고, CPU/RAM이 제한되며, 짧은 타임아웃(timeout)을 가집니다.

먼저, `services/sandbox/Dockerfile`을 코딩하여 컨테이너 내부의 권한을 제한하기 위해 비루트(non-root) 사용자를 생성합니다.
다음으로, `services/sandbox/runner.py` 파일입니다. 이는 코드를 받아 별도의 프로세스에서 실행하고(걸려도 강제 종료할 수 있도록), 표준 출력(stdout)을 캡처하는 간단한 Flask 서버입니다.
`services/sandbox/limits.yaml`에 리소스 제한을 정의합니다. 이는 사용자가 `while True` 루프나 메모리 폭탄을 실행하더라도 샌드박스 파드만 충돌하고 노드는 충돌하지 않도록 보장합니다.
또한 `services/sandbox/network-policy.yaml`에 Kubernetes 네트워크 정책(Network Policy)을 적용해야 합니다. 이는 모든 송신(egress) 트래픽을 차단하는 방화벽 규칙 역할을 합니다. 샌드박스는 인터넷, 데이터베이스 또는 다른 서비스에 연결할 수 없습니다.

마지막으로, API는 이 샌드박스와 통신하기 위한 클라이언트가 필요합니다. `services/api/app/tools/sandbox.py`는 내부 샌드박스 서비스에 대한 HTTP 요청을 처리합니다.

### 결정론적 및 검색 도구 (Deterministic & Search Tools)

코드 실행은 위험하므로, 가능한 한 안전하고 결정론적인 도구를 선호합니다. 수학의 경우, `services/api/app/tools/calculator.py`에서 `simpleeval`을 사용합니다. 이는 `eval()`을 사용하지 않고 표현식을 구문 분석하고 계산합니다.

지식 검색을 위해, 그래프 및 벡터 데이터베이스(Vector DB)를 도구로 노출합니다. `services/api/app/tools/graph_search.py`는 에이전트가 Neo4j를 쿼리할 수 있도록 합니다. 먼저 LLM을 사용하여 엔티티를 추출한 다음, 고정된 Cypher 쿼리를 실행합니다. *주목할 점은 주입 공격을 방지하기 위해 LLM이 직접 원시 Cypher 쿼리를 작성하도록 허용하지 않는다는 것입니다.*

마찬가지로, `services/api/app/tools/vector_search.py`를 코딩하여 Qdrant 포트를 노출해야 합니다. 이는 플래너가 사용자가 "질문에 답하기"보다 "문서를 찾는" 것을 특별히 원한다고 결정할 때 유용합니다.

외부 지식의 경우, `services/api/app/tools/web_search.py`를 추가합니다. 이는 Tavily(에이전트에 최적화됨)와 같은 API를 사용하여 실시간 정보를 얻습니다. 원하는 다른 공급자를 사용할 수 있지만, Tavily는 널리 신뢰할 수 있고 저렴합니다.

### API 경로 및 게이트웨이 로직 (API Routes & Gateway Logic)

도구가 준비되었으니 REST 엔드포인트를 통해 시스템을 노출해야 합니다. `services/api/app/routes/chat.py`는 주요 진입점입니다. 이는 스트리밍 응답을 처리하고, LangGraph 실행을 관리하며, 로깅을 위한 백그라운드 작업을 처리합니다.

파일 업로드의 경우 `services/api/app/routes/upload.py`의 로직이 필요합니다. 기업 환경에서는 API 서버를 통해 대용량 파일을 절대 업로드하지 않습니다. 프론트엔드가 S3에 직접 업로드할 수 있도록 미리 서명된 URL(Presigned URL)을 생성합니다.

또한 시간이 지남에 따라 시스템을 개선하기 위한 피드백(`services/api/app/routes/feedback.py`, RLHF)과 Kubernetes Liveness Probe를 위한 헬스 체크(`services/api/app/routes/health.py`)가 필요합니다.

마지막으로, API를 남용으로부터 보호하는 것은 중요합니다. `services/gateway/rate_limit.lua`를 사용하여 게이트웨이(Gateway) 수준에서 속도 제한을 구현합니다. 이는 Nginx 또는 Kong 내부에서 실행되며, 사용자가 요청을 스팸으로 보내는지 확인하기 위해 Redis를 확인합니다. 이 로직을 Lua/Nginx로 이동함으로써, 악성 트래픽이 Python API에 도달하기 전에 차단하여 귀중한 CPU 사이클을 절약합니다.

이제 수집, 모델, 에이전트, 도구 등 전체 애플리케이션 스택을 구축했습니다. 다음 섹션에서는 이 거대한 시스템을 AWS에서 실행하기 위한 인프라를 프로비저닝할 것입니다.

## 코드형 인프라 (Infrastructure as Code, IaC)

이제 소프트웨어 스택을 구축했으니, 이제 실행할 공간이 필요합니다. 기업 RAG 플랫폼에서는 AWS 콘솔에서 버튼만 클릭할 수 없습니다. 이는 "클릭옵스(ClickOps)" 이탈, 보안 취약점, 반복 불가능한 환경으로 이어집니다.

우리는 Terraform을 사용하여 전체 클라우드 발자국을 코드로 정의합니다. 이를 통해 몇 분 만에 동일한 개발, 스테이징, 프로덕션 환경을 구축할 수 있습니다. 또한 Karpenter를 사용하여 표준 AWS Auto Scaling 그룹보다 빠르고 저렴한 지능형 적시 노드 스케일링을 수행합니다.

### 기반 및 네트워킹 (Foundation & Networking)

모든 클라우드 아키텍처는 네트워크로 시작합니다. 민감한 데이터베이스를 공용 인터넷으로부터 격리하면서 API가 트래픽을 처리할 수 있도록 Virtual Private Cloud(VPC)가 필요합니다.

먼저 `infra/terraform/main.tf`를 구축하는 것부터 시작하겠습니다. 원격 상태 백엔드(remote state backend)를 구성할 것입니다. 이는 팀워크에 중요합니다. 없으면 두 명의 엔지니어가 동시에 `terraform apply`를 실행하여 인프라 상태를 손상시킬 수 있습니다. 상태 파일은 버전 관리되는 S3 버킷에 DynamoDB 잠금과 함께 저장합니다. 여기에서 AWS 및 Kubernetes 제공자를 초기화합니다. `backend "s3"` 블록은 인프라의 민감성이 개발자의 노트북이 아닌 클라우드에 암호화되고 잠긴 상태로 존재함을 보장합니다.

`infra/terraform/variables.tf`에 사용자 정의 가능한 매개변수를 정의합니다. 이는 재사용성을 촉진하며, 이러한 변수를 오버라이딩하는 것만으로 복잡한 로직 파일을 건드리지 않고 `us-west-2`에 배포하거나 IP 범위를 변경할 수 있습니다.

이제 `infra/terraform/vpc.tf`입니다. 3계층 네트워크 아키텍처를 구축해야 합니다. 이는 로드 밸런서(Public)를 애플리케이션 서버(Private) 및 데이터 저장소(Database)로부터 물리적으로 분리하여 공격 표면을 급격히 줄입니다. 여기서 `enable_nat_gateway = true`를 설정하면 사설 애플리케이션 파드가 수신 연결에 노출되지 않고 Docker 이미지와 Python 패키지를 인터넷에서 안전하게 다운로드할 수 있습니다. 태그는 필수적입니다. 태그 없이는 Kubernetes의 AWS Load Balancer Controller가 ALB(Application Load Balancer)를 어디에 배치해야 할지 알 수 없습니다.

### 컴퓨팅 클러스터 (EKS & IAM)

우리 플랫폼의 클라우드 통신 부분은 Amazon EKS(Elastic Kubernetes Service) 클러스터입니다. 모든 컨테이너를 오케스트레이션할 컨트롤 플레인을 프로비저닝하고 있습니다. 이 설정은 안전하고 세분화된 권한 관리가 가능해야 합니다.

`infra/terraform/eks.tf`에서 OIDC(OpenID Connect)를 활성화합니다. 이는 Kubernetes 서비스 계정(Service Account)이 AWS IAM 역할(IRSA)을 맡을 수 있도록 하는 연결입니다. 이는 엔터프라이즈 환경에서 주요 보안 위험인 장기 실행 AWS 액세스 키의 필요성을 없앱니다. 이 모듈은 EKS 컨트롤 플레인과 소규모 "시스템" 노드 그룹을 프로비저닝합니다. 이 노드 그룹은 CoreDNS 및 Karpenter와 같은 시스템에 필수적인 파드를 실행하는 것이 유일한 작업이므로 작게 유지합니다. 무거운 작업은 나중에 동적으로 생성되는 노드에 의해 수행될 것입니다.

다음으로, 인프라 `infra/terraform/iam.tf`를 코딩하여 수집 파이프라인에 최소 권한 원칙(Principle of Least Privilege)을 구현할 것입니다. 문서 버킷에만 액세스 권한을 부여하는 특정 IAM 정책을 생성하고, 이를 수집 작업이 사용하는 특정 Kubernetes 서비스 계정에 바인딩합니다. 이는 Ray 워커가 손상되더라도 공격자가 데이터베이스 백업이나 청구 정보와 같은 AWS 계정의 다른 부분에 액세스할 수 없도록 보장합니다.

### 관리형 데이터 저장소 (Managed Data Stores)

Kubernetes 내부에서 상태 저장 데이터베이스를 실행하는 것은 운영상 복잡하고 위험합니다. 프로덕션급 시스템의 경우, 백업, 패치 및 페일오버(failover)를 자동으로 처리하기 위해 AWS 관리형 서비스를 사용합니다.

`infra/terraform/rds.tf`를 통해 Aurora Serverless v2 (Postgres)를 배포합니다. 이는 RAG와의 채팅과 같은 가변적인 워크로드에 매우 중요합니다. 활성 연결에 따라 컴퓨팅 용량(ACU)을 자동으로 스케일 업 및 다운하여, 피크 시간에는 고성능을 보장하고 유휴 시간에는 낮은 비용을 유지합니다. `serverlessv2_scaling_configuration`은 데이터베이스가 2 ACU(저렴)에서 64 ACU(강력)로 연결을 끊지 않고 원활하게 확장될 수 있도록 합니다.

고속 캐싱을 위해 `infra/terraform/redis.tf`에 ElastiCache (Redis)를 코딩하고 사용해야 합니다. 우리는 특히 `cache.t4g.medium` 인스턴스를 선택합니다. 이 인스턴스는 AWS Graviton (ARM) 프로세서에서 실행되며, 표준 x86 인스턴스보다 최대 40% 더 나은 가격-성능을 제공합니다. 2개의 노드로 구성된 복제 그룹(Replication Group)을 구성하여 고가용성을 보장해야 합니다. 기본 노드가 실패하면 AWS가 자동으로 복제본을 승격하여 캐시를 온라인 상태로 유지합니다.

원시 파일을 덤프할 장소를 생성할 것입니다. `infra/terraform/s3.tf`는 Transfer Acceleration이 활성화된 S3 버킷을 설정합니다. 글로벌 기업은 전 세계에 사용자를 보유하고 있으며, Transfer Acceleration은 Amazon CloudFront의 전 세계적으로 분산된 엣지 로케이션을 사용하여 최적화된 AWS 백본 네트워크를 통해 버킷으로 업로드를 라우팅하여 대용량 파일 전송 속도를 크게 높입니다. `lifecycle_configuration`은 30일 후 오래된 파일을 더 저렴한 스토리지 클래스(Glacier 등)로 자동으로 이동시키며, 이는 장기 스토리지 비용을 제어하기 위한 표준 FinOps(금융 운영) 관행입니다.

Kubernetes에 Neo4j를 자체 호스팅하고 있으므로(AuraDB Enterprise 대비 라이선스 비용 절감), `infra/terraform/neo4j.tf`에 보안 그룹을 정의합니다. 이러한 파드에 대한 네트워크 액세스를 엄격하게 제어해야 합니다. 이 보안 그룹은 방화벽 역할을 하여 VPC 내에서 발생하는 트래픽만 Neo4j Bolt 포트에 도달할 수 있도록 합니다. 외부 인터넷 트래픽은 완전히 차단됩니다.

마지막으로, `infra/terraform/outputs.tf`는 방금 생성한 엔드포인트를 내보냅니다. Terraform 모듈은 함수와 같으며, 이러한 출력은 다음 단계에서 Kubernetes Secrets를 구성하기 위해 쉽게 쿼리할 수 있는 반환 값입니다.

### Karpenter를 이용한 자동 확장 (Autoscaling with Karpenter)

표준 클러스터 오토스케일러(Cluster Autoscaler)는 느리고 반응적입니다. 파드(Pod)가 스케줄링에 실패할 때까지 기다렸다가 노드를 추가합니다. Karpenter는 능동적이고 지능적입니다. 보류 중인 파드의 특정 리소스 요구 사항(GPU 유형, 메모리)을 분석하여 정확히 일치하는 EC2 인스턴스를 몇 초 안에 시작합니다.

`infra/karpenter/provisioner-cpu.yaml`에 CPU 프로비저너를 정의합니다. 우리의 스테이트리스(stateless) API 및 웹 서버의 경우, 중단을 허용할 수 있으므로 스팟 인스턴스(Spot Instances)를 사용합니다. 이는 온디맨드(On-Demand) 인스턴스에 비해 컴퓨팅 비용을 약 70% 절감합니다. `consolidation: true` 설정은 Karpenter에게 파드를 더 촘촘하게 패킹하고 활용도가 낮은 인스턴스를 종료하도록 적극적으로 이동하여 빈 공간에 대한 비용을 지불하지 않도록 지시합니다.

AI 워크로드의 경우 `infra/karpenter/provisioner-gpu.yaml`을 정의합니다. 여기서 비용 제어의 마법이 일어납니다. TTL(Time To Live) 설정을 포함합니다. `ttlSecondsAfterEmpty: 30`은 Scale-to-Zero 아키텍처를 가능하게 합니다. 아무도 Chat API를 사용하지 않으면 Ray는 모델 복제본을 0으로 스케일 다운합니다. Karpenter는 노드가 비어 있음을 감지하고 30초 이내에 값비싼 `g5.xlarge` 인스턴스를 종료합니다. 이는 실제로 추론을 수행할 때만 GPU 비용을 지불하도록 보장합니다.

이제 이 클러스터에 애플리케이션을 배포할 수 있으며, 기본 하드웨어가 우리가 가하는 모든 부하에 동적으로 적응할 것이라는 것을 알 수 있습니다.

## 배포 (Deployment)

프로덕션급 RAG 플랫폼에서 배포는 단순히 `kubectl apply -f file.yaml`이 아닙니다. 구성 불일치를 관리하고, 비밀(secrets)을 안전하게 처리하며, 데이터베이스가 장애에 강하도록 보장해야 합니다.

우리는 Helm을 사용하여 애플리케이션을 패키징합니다. Helm을 사용하면 배포를 템플릿으로 정의할 수 있어, 수천 줄의 YAML을 다시 작성하지 않고도 다양한 환경에 맞게 값(복제본 수 또는 저장소 크기 등)을 쉽게 변경할 수 있습니다.

### 클러스터 부트스트랩 및 비밀 (Cluster Bootstrapping & Secrets)

사용자 정의 코드를 배포하기 전에 클러스터를 설치해야 합니다. 여기에는 인그레스 컨트롤러(Ingress Controller, 트래픽 관리자), 외부 비밀 연산자(External Secrets Operator, 보안 관리자), KubeRay 연산자(AI 관리자)가 포함됩니다.

`scripts/bootstrap_cluster.sh`를 사용하여 이를 자동화합니다. 이 스크립트는 이러한 중요한 시스템 구성 요소에 대한 Helm 차트(chart)를 설치합니다. 이는 애플리케이션을 배포하기 전에 클러스터에 필요한 모든 컨트롤러가 실행 중인지 확인하는 "사전 비행 점검(pre-flight check)"과 같습니다.

다음으로, 비밀(secrets)을 처리하기 위해. 프로덕션 환경에서는 `.env` 파일을 절대 커밋하지 않습니다. AWS Secrets Manager를 사용하여 데이터베이스 암호와 API 키를 저장합니다. External Secrets Operator를 사용하여 이러한 비밀을 가져와 Kubernetes 파드에 안전하게 주입합니다. `deploy/secrets/external-secrets.yaml`에 이를 정의합니다. 이 구성은 Kubernetes에게 "AWS Secrets Manager로 이동하여 `prod/rag/db_creds`를 찾아 `neo4j_password`를 가져온 다음, 내 파드가 사용할 로컬 Kubernetes 비밀을 생성하라"고 지시합니다. 이는 민감한 데이터를 Git 기록에서 완전히 제외시킵니다.

### 데이터베이스 및 인그레스 배포 (Database & Ingress Deployment)

이제 스테이트풀(stateful) 워크로드를 배포합니다. Postgres에 RDS를 사용했음에도 불구하고, 성능 및 비용상의 이유로 Qdrant와 Neo4j는 자체 호스팅합니다.

표준 Helm 차트를 사용하지만, 기본값을 프로덕션 준비 상태로 재정의합니다. Qdrant의 경우 `deploy/helm/qdrant/values.yaml`을 수정하여 데이터가 영구적이고 고가용성을 갖도록 보장합니다. `replicaCount: 3`은 노드 하나가 실패해도 벡터 데이터베이스(Vector DB)가 온라인 상태를 유지하도록 합니다. `on_disk_payload`는 RAG에 대한 중요한 최적화로, 수백만 개의 문서를 인덱싱하여 RAM 한도를 초과하더라도 Qdrant가 충돌하는 것을 방지합니다.

Neo4j의 경우 `deploy/helm/neo4j/values.yaml`에 유사한 구조를 구성해야 합니다. 그래프 순회(Graph traversals)는 RAM 집약적이므로 메모리(8Gi)를 할당합니다. 또한 지식 그래프(Knowledge Graph)가 파드 재시작 후에도 유지되도록 100GB의 영구 볼륨을 연결합니다.

API를 인터넷에 노출하기 위해 `deploy/ingress/nginx.yaml`에 인그레스(Ingress) 리소스를 구성합니다. 이는 AWS 로드 밸런서(Load Balancer)에서 내부 서비스로 HTTP 트래픽을 라우팅하는 API 게이트웨이 역할을 합니다. 여기에서 주석(annotations)은 중요합니다. `proxy-read-timeout`은 1시간으로 설정되는데, 70B 모델로 긴 답변을 생성하는 데 시간이 걸릴 수 있으며, Nginx가 연결을 조기에 끊는 것을 원치 않기 때문입니다.

### Ray AI 클러스터 배포 (Ray AI Cluster Deployment)

마지막으로, 우리 운영의 핵심인 Ray 클러스터를 배포합니다. 이는 헤드 노드(관리자)와 여러 워커 노드(실행)가 독립적으로 확장되어야 하므로 복잡합니다.

먼저 `deploy/ray/autoscaling.yaml`에 스케일링 로직을 정의합니다. 이 구성 맵(config map)은 Ray 오토스케일러(Autoscaler)에게 부하가 증가할 때 추가할 워커(worker) 수를 알려줍니다. 이 구성은 Scale-to-Zero 패턴을 위한 것입니다. 5분 동안 AI 모델을 사용하지 않으면 Ray는 워커 파드를 종료합니다. 그러면 Karpenter는 빈 노드를 감지하고 EC2 인스턴스를 종료하여 막대한 비용을 절감합니다.

`deploy/ray/ray-cluster.yaml`에 클러스터 구조를 정의합니다. 이는 KubeRay 연산자가 관리하는 CRD(Custom Resource Definition)입니다. `tolerations` 섹션은 우리의 값비싼 AI 워크로드가 GPU가 있는 노드에만 스케줄링되도록 보장하며, 이전에 Karpenter 구성에서 설정한 Taint와 일치시킵니다.

이제 RayService를 사용하여 이 클러스터에 실제 모델을 배포합니다. 이 추상화는 모델에 대한 고가용성 및 무중단 업그레이드를 처리합니다.
임베딩 모델의 경우 `deploy/ray/ray-serve-embed.yaml`을 생성하여 클러스터에서 임베딩 모델을 서빙해야 합니다.
LLM의 경우 `deploy/ray/ray-serve-llm.yaml`을 생성해야 하며, 이는 효율적인 추론을 위해 VLLM을 사용하여 Meta Llama 3 모델을 서빙합니다.

이러한 객체를 RayService 객체로 정의함으로써 운영 능력을 확보합니다. `MODEL_ID`를 업데이트하면 연산자는 새 모델로 새 파드를 생성하고, 건강 상태가 될 때까지 기다린 다음, 트래픽을 원활하게 전환하여 사용자에게 무중단 서비스를 보장합니다.

마지막으로, 테스트가 끝나거나 개발 환경에서 비용을 절감해야 할 때, 모든 것을 해체하여 비용을 절감하기 위한 `scripts/cleanup.sh`도 구축해야 합니다. 이 스크립트는 안전 밸브 역할을 합니다. 주말 동안 20개의 GPU 노드 클러스터가 실수로 실행되어 엄청난 요금이 부과되는 것을 방지합니다.

우리는 Terraform, Helm, Ray를 결합하여 인프라를 완벽하게 제어하면서 분산 시스템의 복잡성을 자동화했습니다. 이제 프로덕션 규모로 이 아키텍처를 모니터링하고 평가하는 방법을 살펴보겠습니다.

## 평가 및 운영 (Evaluation & Ops)

기업 RAG 플랫폼에서 평가는 대부분 개발 단계에서 이루어지는 매우 중요한 부분입니다. 가시성(Observability: 메트릭/추적), 평가(정확도 확인), 운영(부하 테스트/유지보수)을 위한 전략이 필요합니다. 프로덕션 시스템에서 일반적으로 볼 수 있는 평가는 답변에 대한 사용자 만족도(좋아요/싫어요)입니다. 이는 유용한 피드백이지만, 너무 희박하고 지연되어 많은 사용자에게 영향을 미 미치기 전에 문제를 파악하기 어렵습니다.

### 가시성 및 추적 (Observability & Tracing)

측정할 수 없으면 개선할 수 없습니다. Ray, Kubernetes 및 여러 데이터베이스를 포함하는 분산 시스템에서 병목 현상을 찾는 것은 분산 추적(distributed tracing) 없이는 불가능합니다.

가시성 또한 평가의 한 유형입니다. 시스템 성능 및 사용자 행동을 이해하는 데 도움이 될 수 있기 때문에, 예산을 초과하는 토큰 사용량이나 지연 시간(latency) 급증은 예산 및 사용자 만족도에 문제를 일으킬 수 있습니다.

`libs/observability/metrics.py`에 사용자 정의 메트릭(metrics)을 정의합니다. Prometheus 카운터(Counter)를 사용하여 토큰 사용량(비용)을 추적하고, 히스토그램(Histogram)을 사용하여 지연 시간(성능)을 추적할 것입니다. 이를 통해 Llama-3이 소비하는 토큰 수와 특정 엔드포인트가 느려지는지 여부를 정확히 보여주는 Grafana 대시보드를 구축할 수 있습니다.

추적을 위해 `libs/observability/tracing.py`를 생성할 것입니다. OpenTelemetry를 사용하여 API, 벡터 검색(Vector Search), LLM 생성에 이르는 전체 요청 수명 주기를 시각화할 수 있습니다. 사용자가 "느린 응답"을 보고할 때, 추적 ID(Trace ID)를 확인하여 예를 들어 Neo4j 그래프 쿼리를 기다리는 데 90%의 시간이 소요되었음을 확인하고 정확히 어디를 최적화해야 하는지 찾아낼 수 있습니다.

### 연속 평가 파이프라인 (Continuous Evaluation Pipeline)

RAG 파이프라인이 정확한지 어떻게 알 수 있을까요? 모든 답변을 수동으로 확인할 수는 없습니다. "LLM-as-a-Judge"를 사용하여 품질 관리를 자동화합니다.

먼저, "골든 데이터셋(Golden Dataset)"을 생성해야 합니다. 질문과 정답(Ground Truth) 세트입니다. 프로젝트 초기에 수집한 Kubernetes 문서(예: pods_autoscale.html)를 기반으로 `eval/datasets/golden.json`을 생성합니다. 이 파일은 우리의 벤치마크 역할을 합니다. 프롬프트 또는 검색 로직을 업데이트할 때마다 이 데이터셋에 대해 시스템을 실행합니다.

일반적으로 골든 데이터셋은 더 큰 LLM을 통해 합성적으로 생성되거나 사람의 주석(annotation)을 통해 생성됩니다. 단순화를 위해 Gemini(Google의 최신 모델)를 통해 이를 구축했습니다. 비공개 데이터로 작업할 때는 하위 RAG 파이프라인을 생성하여 골든 데이터셋을 생성합니다. 이는 강력한 LLM에게 비공개 문서를 읽고 질문-답변 쌍을 생성하도록 프롬프트를 제공하여 수행됩니다. 이 방법을 통해 민감한 데이터가 타사 서비스에 노출되는 것을 방지합니다.

결과를 평가하기 위해 `eval/judges/llm_judge.py`에 평가 로직을 코딩할 것입니다. 이 스크립트는 강력한 모델(GPT-4 또는 Llama-3-70B 등)을 사용하여 시스템의 실제 답변을 정답과 비교할 것입니다.

Ragas는 RAG 시스템을 평가하기 위한 인기 있는 프레임워크입니다. `eval/ragas/run.py`에 주요 평가 스크립트를 작성하여 Ragas를 사용하여 충실도(Faithfulness, AI가 환각했는지 여부) 및 컨텍스트 리콜(Context Recall, 올바른 문서를 찾았는지 여부)과 같은 메트릭을 계산하고 보고서를 생성합니다. 이 스크립트는 `eval/reports/evaluation_results.csv` 보고서를 생성하며, 이는 파이프라인의 정확도에 대한 시각적 요약을 제공합니다. 우리는 이를 CI/CD 파이프라인에서 사용합니다. 만약 "충실도" 점수가 0.8 미만으로 떨어지면 배포를 차단합니다.

### 운영 우수성 및 유지보수 (Operational Excellence & Maintenance)

마지막으로, 프로덕션에서 시스템을 유지 관리하는 도구가 필요합니다. 코드 변경, 부하 증가 및 데이터 드리프트(data drift)는 피할 수 없습니다.

Karpenter 자동 스케일링 설정이 작동하는지 확인하기 위해 `scripts/load_test.py`의 Locust를 사용하여 과도한 트래픽을 시뮬레이션합니다. 이 스크립트를 실행하면 `ray/autoscaling.yaml` 구성을 튜닝하는 데 도움이 되며, 100명의 사용자가 동시에 접속할 때 새 GPU 노드가 충분히 빠르게 생성되는지 확인할 수 있습니다.

애플리케이션이 발전함에 따라 데이터베이스 스키마도 변경될 것입니다. `scripts/migrate_db.py`를 (래핑하여) 코딩하여 다운타임(downtime) 없이 스키마 마이그레이션을 안전하게 처리해야 합니다.

마지막으로, 새 버전을 배포할 때 캐시가 비어 있어 초기 사용자 몇 명에게 느린 응답을 초래합니다. `scripts/warmup_cache.py`를 사용하여 새 파드로 트래픽을 전환하기 전에 일반적인 쿼리로 Redis 및 Qdrant를 미리 채웁니다. 이러한 FAQ는 골든 데이터셋과 실제 사용자 쿼리에서 파생됩니다. 이를 캐시에 미리 로드함으로써, 가장 일반적인 질문에 즉시 답변하여 배포 직후 사용자 경험을 향상시킵니다.

## 엔드 투 엔드 실행 (End-to-End Execution)

코드를 구축하고, 인프라를 정의했으며, 평가 파이프라인을 설정했습니다. 이제 이 모든 것을 실제로 배포하여 RAG 플랫폼을 실행해야 합니다.

Terraform이 무거운 작업을 자동화하지만, "원격 상태(Remote State)" 백엔드를 부트스트랩하기 위해 AWS 콘솔에서 몇 가지 초기 수동 단계를 수행해야 합니다. 이는 일회성 설정입니다.
Terraform 상태 파일을 저장하기 위한 S3 버킷과 상태 잠금(동시 수정 방지)을 처리하기 위한 DynamoDB 테이블을 생성해야 합니다. 비용을 최적화하기 위해 버전 관리, 암호화 및 수명 주기 정책과 같은 여러 규칙을 버킷에 할당할 수 있습니다. EKS는 GPU 인스턴스 유형의 가용성이 가장 넓으므로 `us-east-1` (N. Virginia) 지역을 선택해야 합니다. 마찬가지로, 상태 잠금을 위한 DynamoDB 테이블을 생성해야 합니다. 이는 여러 사람이 동시에 변경 사항을 적용하는 것을 방지하는 데 중요합니다. 파티션 키를 유형이 `String`인 `LockID`로 설정해야 합니다. 나머지 설정은 기본값으로 유지할 수 있습니다. 테이블 이름을 `terraform-state-lock`과 같이 지정하고 최상의 성능을 위해 S3 버킷과 동일한 지역에 있는지 확인하십시오.

### EKS 클러스터 생성

이제 Terraform 백엔드가 준비되었습니다. 터미널로 전환해야 합니다. VPC, EKS 클러스터 컨트롤 플레인 및 데이터베이스를 생성할 것입니다.
참고: 이 단계는 EKS 컨트롤 플레인과 시스템 도구를 실행하기 위한 작은 "시스템 노드 그룹"(2개의 작은 CPU)을 생성합니다. GPU 노드는 아직 생성하지 않습니다. 비용을 절감하기 위해 소프트웨어가 실제로 요청할 때만 GPU 노드를 생성합니다.
인프라 `make` 명령을 실행하면 Terraform 코드를 초기화하고 적용하는 작업을 처리합니다.

```bash
# 프로젝트 루트에서
make infra
```

이 명령은 `terraform init` 다음에 `terraform apply`를 실행합니다.

출력에서 VPC, EKS 클러스터, RDS Postgres 및 ElastiCache Redis를 포함한 여러 리소스가 생성되었음을 확인할 수 있습니다. 이 프로세스는 일반적으로 약 15-20분 정도 소요됩니다. 완료되면 원시 Kubernetes 클러스터가 준비됩니다.

이제 `kubectl`이 새 클러스터와 통신하도록 구성하고 시스템 컨트롤러를 설치합니다.

```bash
# kubeconfig를 새 클러스터로 업데이트
aws eks update-kubeconfig --region us-east-1 --name rag-platform-cluster

# 부트스트랩 스크립트 실행
./scripts/bootstrap_cluster.sh
```

KubeRay Operator, External Secrets Operator, Nginx Ingress Controller가 성공적으로 설치되었음을 확인할 수 있습니다.
이 단계에서 클러스터는 설치되었지만 GPU 인스턴스는 아직 0개입니다.

다음으로, GPU 노드의 동적 프로비저닝을 관리하기 위해 Karpenter가 필요합니다. 이전에 정의한 Karpenter 구성을 적용합니다. Ray 구성은 `nvidia.com/gpu: 1`을 명시적으로 요청합니다.
Kubernetes는 파드를 스케줄링하려고 시도하지만, GPU 노드가 없으므로 실패합니다(Pending 상태). Karpenter는 이 Pending 상태의 파드를 감지합니다. Karpenter는 자동으로 AWS EC2 API를 호출하여 `g5.xlarge` 인스턴스를 구매합니다.

```bash
# Ray Cluster 및 Ray Service 정의 적용
kubectl apply -f deploy/ray/
```

이제 파드를 주의 깊게 지켜보십시오.

```bash
kubectl get pods -w
```

초기에는 `Pending` 상태의 파드가 GPU 노드를 기다리다가, 약 45초 지연 후 파드가 `Running` 상태로 전환됩니다. 정확히 이 순간, AWS EC2 콘솔을 확인하면 `karpenter-rag-platform-cluster-...`라는 이름의 새 인스턴스가 초기화되고 있는 것을 볼 수 있습니다. "인스턴스 생성"을 클릭하지 않았지만, 소프트웨어가 그렇게 했습니다.

Karpenter가 필요한 인프라를 적시에 성공적으로 생성/업데이트했습니다. 이제 워커 내부의 vLLM 엔진이 새로 생성된 인스턴스의 VRAM에 Llama-3-70B 가중치(약 40GB 양자화)를 로드하고 있습니다.

### 추론 및 지연 시간 테스트 (Inferencing and Latency Test)

첫 섹션에서 1000개의 문서(많은 노이즈 포함)가 준비되어 있었음을 기억할 것입니다. 이제 이를 S3에 업로드하고 Ray 작업을 트리거할 것입니다. 현재 클러스터가 너무 바쁘면 이 작업은 추가 CPU 노드를 생성할 것입니다. 이 모든 것이 Karpenter 접근 방식 덕분입니다.

```bash
# 데이터 업로드 (Terraform 출력에서 얻은 버킷 이름 사용)
python scripts/bulk_upload_s3.py ./data rag-platform-documents-prod

# 수집 작업 트리거
python -m pipelines.jobs.s3_event_handler
```

이것은 문서를 업로드하고 임베딩하며 청크에서 지식 그래프를 생성하는 수집 작업을 트리거할 것입니다.

출력에서 처리량에 주목하십시오. Ray가 PDF 구문 분석을 여러 CPU 코어에 걸쳐 병렬화하고 임베딩을 GPU로 일괄 처리했기 때문에, 1000개의 기업 문서를 2분 이내에 처리했습니다.

이제 지식 그래프와 벡터 데이터베이스(Vector DB)가 채워졌으므로, 에이전트가 "계획"해야 하는 복잡한 질문을 해봅시다.

```bash
# 로드 밸런서 URL 가져오기
kubectl get ingress
# 출력: api.your-rag-platform.com

# Curl 요청
curl -X POST https://api.your-rag-platform.com/api/v1/chat/stream \
  -H "Content-Type: application/json" \
  -d '{
    "message": "Compare the cost of the HPA vs VPA in Kubernetes based on the docs.",
    "session_id": "demo-session-1"
  }'
```

이것은 전체 RAG 파이프라인이 콘솔 출력에서 어떻게 작동하는지 보여줍니다. 에이전트가 먼저 의도를 "검색(Retrieve)"으로 식별하는 것을 볼 수 있습니다. 그런 다음 Qdrant와 Neo4j에 걸쳐 하이브리드 검색을 수행하고, 관련 문서를 찾아 인용과 함께 포괄적인 답변을 합성합니다.

답변의 품질도 중요하지만, 기업 환경에서는 지연 시간(Latency)이 곧 사용자 경험입니다. 사용자가 10초 동안 스피너를 응시해야 한다면, 그들은 떠날 것입니다.

방금 실행한 HPA vs VPA 쿼리에 대한 내부 지연 시간 분석을 살펴보겠습니다. 구조화된 로깅을 활성화했기 때문에 시간이 정확히 어디에 소요되었는지 알 수 있습니다.

```
{
  "request_id": "req-a1b2-c3d4",
  "total_latency_ms": 2850,
  "breakdown": {
    "planner_node_ms": 420,
    "retrieval_node_ms": 780,
    "generation_ttft_ms": 1650 
  }
}
```

일반적으로 2~4초 이내에 응답을 받습니다. 이 아키텍처가 이러한 속도를 가능하게 하는 이유는 다음과 같습니다.

*   **병렬 검색 (Parallel Retrieval):** `retrieval_node_ms`는 780ms입니다. 여기에는 벡터 검색(Qdrant)과 그래프 쿼리(Neo4j)가 모두 포함됩니다. API 코드에서 `asyncio.gather`를 사용했기 때문에 이 쿼리들은 다른 CPU 스레드에서 동시에 실행되었습니다. 순차적으로 실행했다면 이 단계만 1.5초가 걸렸을 것입니다.
*   **vLLM 가속 (vLLM Acceleration):** `generation_ttft_ms` (첫 토큰 생성 시간, Time To First Token)는 1.6초입니다. 이는 70B 모델이 컨텍스트를 읽고 말하기 시작하는 데 걸린 시간입니다. 표준 HuggingFace 파이프라인은 여기에서 4~5초가 걸릴 것입니다. vLLM의 PagedAttention은 메모리 액세스를 최적화하여 이를 낮게 유지합니다.

하지만 "코드 검토자" 또는 "웹 검색자"와 같은 더 많은 에이전트를 추가하면 이 지연 시간은 자연스럽게 늘어날 것입니다. 단순히 이들을 연결하면 응답 시간이 10초 이상 걸릴 수 있습니다. 이 문제를 해결하기 위해 네트워크를 더욱 분산할 것입니다. 하나의 CPU에서 하나의 에이전트 루프가 실행되는 대신, 여러 Ray 액터(Actor)를 생성하여 다른 에이전트를 병렬로 실행하여 복잡성이 증가하더라도 약 3초의 스위트 스팟을 유지할 것입니다.

### Redis 및 Grafana 대시보드 분석 (Redis and Grafana Dashboard Analysis)

마지막으로, 시스템이 압력 하에서 버티는지 확인할 수 있습니다. Locust 부하 테스트를 500명의 동시 사용자와 함께 실행하여 급격한 트래픽을 시뮬레이션합니다.

```bash
locust -f scripts/load_test.py --headless -u 500 -r 10 --host https://api.your-rag-platform.com
```

이것이 실행되는 동안, 대시보드를 모니터링하여 실시간으로 자동 스케일링을 확인합니다.

Ray 대시보드에서 1000개 문서 수집으로 GPU가 최대치에 도달했음을 나타내는 급증을 볼 수 있습니다. 오토스케일러(Autoscaler)는 증가된 부하를 감지하고 더 많은 GPU 노드(최대 3개)를 요청합니다. 이 테스트를 위해 최대 GPU 노드를 3개(총 16개의 GPU)로 설정했으며, CPU 코어는 192개로 설정했습니다. 이는 낮은 지연 시간으로 500명의 동시 사용자를 처리하기에 충분하지만, 예산과 예상 부하에 따라 조정할 수 있습니다.

Grafana 대시보드를 방문하면 요청 지연 시간이 처음에 5초로 급증하지만, 새 노드가 온라인 상태가 되면서 약 1.2초로 안정화되는 것을 볼 수 있습니다.

오토스케일러 로그는 이를 보여줍니다.

```
### Autoscaler 로그 ###

[Autoscaler Log]

스케일업(Scale-up) 트리거됨: 대기 중인 작업 > 100.

현재 노드: 1개 (g5.xlarge).

대상 노드: 3개 (g5.xlarge).

Karpenter를 통해 3개의 새 노드 시작 중...
```

GPU 사용량이 급증하는 것을 볼 수 있습니다. Ray 오토스케일러는 큐(queue) 깊이 증가를 즉시 감지하고 2개의 GPU 노드를 추가로 요청합니다. Karpenter는 이 요청을 즉시 이행합니다. 90초 이내에 클러스터 용량이 두 배로 늘어나고 요청 지연 시간(latency)이 안정화됩니다.

이는 노이즈 데이터를 처리하고, 효율적으로 처리하며, 사용자 요구에 따라 자동으로 확장되고, 사용자가 떠나면 다시 0으로 축소됩니다.
