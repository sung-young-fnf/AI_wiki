# MemPalace: API 호출 없이 96.6% 회상률 달성? 원본 텍스트 기반 AI 메모리 시스템의 혁신

**요약 (TLDR)**

MemPalace는 API 호출, 검색 루프 내 LLM (대규모 언어 모델) 사용, 구독료 없이 LongMemEval 벤치마크에서 96.6%의 R@5 (회상률@5)를 달성했으며, 이 결과는 제3자 하드웨어에서 독립적으로 재현되었습니다.

핵심 명제는 급진적이면서도 간단합니다. 대화 텍스트를 글자 그대로 (verbatim) 저장하고 최신 임베딩 (embeddings) 모델이 검색을 처리하게 하는 것입니다. 요약, 추출, 정제 과정 없이 신호가 온전히 유지됩니다.

5단계 하이브리드 검색 파이프라인은 의미론적 검색 (semantic search) 위에 키워드 중복, 시간적 부스팅 (temporal boosting), 선호도 추출을 계층화하여 쿼리당 0.001달러 미만의 비용으로 정확도를 99.4% R@5까지 끌어올립니다.

공간적 "궁전" (palace) 아키텍처는 수백만 개의 토큰을 탐색 가능한 '윙' (wings), '룸' (rooms), '홀' (halls), '터널' (tunnels)로 구성합니다. 이는 AI 대화 기록을 위한 파일 시스템으로 생각할 수 있습니다.

오픈 소스 커뮤니티는 모든 주장을 철저히 검증했고, 저자들은 필요한 부분을 수정하며 프로젝트를 더욱 강화했습니다. 이것이야말로 협업 지능 (collaborative intelligence)의 정수입니다.

여기 스크롤을 멈추게 할 만한 숫자가 있습니다.

96.6%.

**저자의 솔직한 의견.** 이 수치는 너무 좋아서 믿기 어려울 정도입니다... 하지만 이 이야기에는 좋은 점도 많으니 계속 살펴보겠습니다.

이것은 MemPalace가 LongMemEval 벤치마크에서 달성한 Recall@5 (회상률@5) 점수입니다. LongMemEval은 과거 세션의 방대한 양 속에서 메모리 시스템이 올바른 대화를 찾을 수 있는지 테스트하기 위해 고안된 500개의 질문으로 구성된 벤치마크입니다. 원문 텍스트 (raw verbatim text), ChromaDB 임베딩 (embeddings), 검색 루프에 언어 모델 없음, 클라우드 호출 없음, API 키 없음이라는 조건에서 얻은 결과입니다.

참고로, Mastra는 94.87%, Hindsight는 91.4%, 실제 사용되는 Supermemory는 약 85%를 기록합니다. 대부분의 사람들이 실제로 사용하는 Mem0 시스템은 75,000개 이상의 Q&A 쌍을 포함하는 ConvoMem에서 30~45%를 관리합니다.

이것은 작은 차이가 아닙니다. 검색에서 85%와 96.6%의 차이는 "가끔 유용함"과 "실제로 신뢰할 수 있음"의 차이입니다. 실질적으로, 85%는 대략 7번의 쿼리 중 1번이 실패한다는 의미입니다. 96.6%는 50개의 질문당 1~2번의 실패를 의미합니다.

하지만 중요한 것은, 이 숫자가 흥미로운 부분이 아닙니다. 이 숫자를 만들어내는 아키텍처가 진짜 흥미로운 부분입니다.

## 원본 텍스트 명제: 모든 것을 보존하는 이유

MemPalace의 핵심 아이디어는 거의 민망할 정도로 간단합니다. 요약하지 마세요. 압축하지 마세요. 정제하지 마세요. 그저 원본 대화 내용을 글자 그대로 저장하고 최신 임베딩 모델이 검색을 처리하도록 하세요.

이것은 일반적인 점진적 최적화가 아닙니다. 이는 전체 AI 메모리 산업이 지향해 온 방향에 정면으로 배치됩니다. Mem0, Zep, 그리고 모든 LLM 기반 메모리 시스템은 대화를 저장하기 전에 추출, 요약, 압축해야 한다고 가정합니다. 그 이유는 명백해 보입니다. 원본 대화는 지저분하고, 불필요한 내용으로 가득하며, 대규모로 검색하기에는 비용이 많이 든다는 것입니다.

MemPalace는 그 가정이 틀렸다고 말합니다.

그리고 추출 과정 없이 96.6%의 R@5는 그들이 옳을 수도 있음을 시사합니다.

통찰력은 미묘하지만 심오합니다. 대화를 요약할 때마다 정보를 잃게 됩니다. 사용자가 선호도를 표현한 미묘한 뉘앙스일 수도 있고, 스쳐 지나가듯 언급된 특정 도구 이름일 수도 있으며, 무언가가 논의된 시간적 맥락일 수도 있습니다. 요약은 자연어에 적용되는 손실 압축 (lossy compression)이며, 이러한 압축 아티팩트 (artifacts)는 하류에서 검색 실패로 나타납니다.

이렇게 생각해 보세요. 당신은 Claude Code, ChatGPT, Cursor를 사용하는 세 가지 프로젝트를 관리하는 테크 리더입니다. 6개월 동안 수천 개의 대화 세션이 축적되었습니다.

두 달째쯤, 당신은 AI 어시스턴트와 데이터베이스 스키마 결정에 대해 논의했습니다. 정확한 추론, 고려했던 절충안, 선택한 특정 마이그레이션 접근 방식 등 모든 것이 그 대화에 있습니다.

요약 기반 시스템은 이를 "사용자가 백엔드 데이터베이스로 PostgreSQL을 결정했습니다"로 요약했을 것입니다.

**기술적으로는 정확하지만 그 외에는 쓸모없습니다.**

왜 그렇게 결정했는지, 어떤 대안이 거부되었는지, 어떤 예외 상황을 걱정했는지 기억해야 할 때 완전히 쓸모없습니다.

MemPalace는 전체 대화를 원본 그대로, 글자 그대로, 모든 단어를 보존합니다. 그리고 6개월 후에 검색하면, 임베딩 모델이 전체 신호가 여전히 존재하기 때문에 그것을 찾아냅니다.

원본 그대로 저장하는 방식은 모든 것을 보존합니다. 임베딩 모델은 의미론적 매칭 (semantic matching)을 처리하고, ChromaDB는 벡터 검색 (vector search)을 처리합니다. 그리고 원본 신호를 버리지 않았기 때문에 검색할 때 더 많은 정보를 활용할 수 있습니다.

이것이 바로 명제입니다. 원본 텍스트와 좋은 임베딩은 정보 손실이 없기 때문에, 누구도 인식하지 못했던 것보다 훨씬 강력한 기준점입니다.

## 궁전 내부: AI 메모리를 위한 공간 아키텍처

은유는 매력적이지만, 중요한 것은 구현입니다. MemPalace는 고대 기억 궁전 (memory palace) 기법에서 영감을 받아 대화 기록을 공간적 계층 구조로 구성합니다.

*   **윙 (Wings)**
    최상위 카테고리입니다. 작업 중인 프로젝트, 협업하는 사람, 학습 중인 도메인 등입니다. 네임스페이스 (namespaces)라고 생각하세요.

*   **룸 (Rooms)**
    윙 안에 있습니다. 각 룸은 특정 주제를 나타냅니다. 예를 들어, "백엔드" 윙 안의 "파이썬 마이그레이션" 룸입니다. 룸 탐지기 (room_detector_local.py)는 패턴 매칭을 통해 수신되는 대화를 자동으로 분류합니다.

*   **드로어 (Drawers)**
    실제 콘텐츠, 즉 원본 대화 파일을 보관합니다. 모든 대화는 수정되지 않고 드로어에 들어갑니다. 검색은 이곳에서 이루어집니다.

*   **클로젯 (Closets)**
    드로어를 가리키는 요약본을 저장합니다. 이들은 빠른 인간의 스캔 및 탐색을 위해 존재하며, 검색 정확도를 위한 것이 아닙니다. 검색 엔진은 드로어를 직접 방문합니다.

*   **홀 (Halls)**
    동일한 윙 내의 관련 룸을 연결합니다. "파이썬 마이그레이션" 룸과 "데이터베이스 스키마" 룸이 컨텍스트를 공유하는 경우, 홀이 이들을 연결하여 palace_graph.py의 BFS (너비 우선 탐색)가 다중 홉 연결을 찾을 수 있도록 합니다. 터널 (Tunnels)은 윙을 가로질러 동일한 작업을 수행합니다. 예를 들어, 협업자의 윙에 있는 "API 디자인" 룸을 백엔드 윙의 "데이터베이스 스키마" 룸에 연결하는 식입니다.

이 계층 구조는 탐색을 위한 것이지 검색 정확도를 위한 것이 아닙니다. 저자들은 이 점에 대해 투명하게 밝히고 있습니다. 궁전 구조는 1,950만 토큰의 대화 기록을 탐색하는 데 도움이 됩니다. 검색 정확도는 드로어 내용에 대한 원본 임베딩에서 나옵니다.

## 지식 그래프 (Knowledge Graph) 레이어

궁전 아래에는 시간적 엔티티 관계 (temporal entity-relationships)를 포함하는 SQLite 기반 지식 그래프가 있습니다. Zep의 클라우드 기반 Neo4j 접근 방식과 달리, MemPalace는 모든 것을 로컬에서 제로 비용으로 유지합니다.

entity_detector.py 모듈은 대화에서 엔티티 (사람, 도구, 프로젝트)를 추출하고, knowledge_graph.py는 유효성 기간 (validity windows)을 통해 엔티티 간의 관계를 추적합니다. 1월에 사실이었던 엔티티 관계가 4월에는 사실이 아닐 수도 있습니다. 시간적 차원 (temporal dimension)이 중요합니다.

entity_registry.py 모듈은 엔티티를 짧은 코드 (아래 AAAK 섹션에서 자세히 설명)에 매핑하고 전체 대화 말뭉치에서 누가 무엇에 대해 무엇을 아는지에 대한 장부를 관리합니다.

## 선호도 윙 (Preference Wing)

가장 영리한 기능 중 하나는 MemPalace가 수집 시점에 채워지는 전용 선호도 윙을 유지한다는 것입니다. general_extractor.py의 16개 정규 표현식 (regex) 패턴이 수신 대화에서 선호도 표현을 스캔하고 합성 문서 (synthetic documents)를 생성합니다. "사용자는 TypeScript를 JavaScript보다 선호한다고 언급했습니다; 땅콩 알레르기가 있습니다; vim 바인딩을 사용합니다."

이러한 합성 선호도 문서는 ChromaDB에서 일급 검색 가능한 엔티티 (first-class searchable entities)가 됩니다. 미래의 쿼리가 선호도와 관련될 때, 시스템은 2,000단어 대화 속에 묻힌 올바른 문장을 임베딩이 찾기를 바라는 대신 이러한 집중된 신호와 일치시킬 수 있습니다.

## 5단계 하이브리드 검색 파이프라인

MemPalace가 정말 영리해지는 부분은 하이브리드 검색 모드입니다. 이 파이프라인은 5가지 문서화된 버전을 거쳐 진화했으며, 각 버전은 특정 유형의 검색 실패에 대한 표적 수정 사항을 추가했습니다. HYBRID_MODE.md에 세심하게 문서화되어 있으며, 실시간 엔지니어링 사후 분석처럼 읽힙니다.

**1단계: 깔때기 확장 (기준선)**
원본 모드는 ChromaDB의 상위 10개 결과를 검색합니다. 하이브리드 파이프라인은 이를 상위 50개 후보로 확장하는 것부터 시작합니다. 더 많은 후보는 정밀도 (precision)를 희생하여 더 높은 회상률 (recall)을 의미합니다. 이후 단계에서 정밀도 문제를 처리합니다.

이 기준선만으로도 96.6%의 R@5를 달성합니다. 실질적으로, 500개의 테스트 질문 중 483개가 상위 5개 결과 내에서 올바른 대화를 반환한다는 의미입니다. 17번의 실패입니다.

**2단계: 키워드 중복 재순위화 (v1, +1.2pp)**
의미론적 검색은 강력하지만 알려진 사각지대가 있습니다. 정확한 키워드 일치를 놓칠 수 있습니다. "GraphQL에 대해 뭐라고 말했지?"라고 물었는데 관련 대화에 "GraphQL"이라는 단어가 12번 사용되었다면, 순수한 임베딩 검색은 의미론적으로는 유사하지만 다른 대화의 순위를 더 높게 매길 수도 있습니다.

하이브리드 v1은 간단한 융합 거리 공식으로 이 문제를 해결합니다.

`fused_dist = dist × (1.0 − 0.30 × overlap)`

시스템은 쿼리에서 불용어 (stop words)를 제외한 단어 (예: "what", "when", "where", "how"와 같은 40개 이상의 불용어를 필터링)를 추출하고, 각 후보 문서와의 키워드 중복을 계산하여 거리 점수를 비례적으로 줄입니다. 강력한 키워드 일치에 대해 최대 30%의 거리 감소 효과가 있습니다.

결과: 97.8% R@5.

여섯 개의 질문이 더 정확하게 답변되었습니다. 잃어버릴 뻔했던 여섯 개의 대화가 찾아졌습니다.

**3단계: 시간적 날짜 부스트 (v2, +0.6pp)**
여기 실제 사용에서만 나타나는 실패 모드가 있습니다. 시간 기반 질문입니다. "며칠 전에 뭐 얘기했지?" "지난주에 그거 뭐였지?"

하이브리드 v2는 "N days ago", "a couple days ago", "last week", "N months ago", "recently"에 대한 정규 표현식 (regex patterns)을 사용하여 시간적 참조를 파싱합니다. 목표 날짜 범위 (target date window)를 계산하고 그 안에 속하는 문서의 점수를 높입니다. 시간적으로 관련 있는 결과에 대해 최대 40%의 거리 감소 효과가 있습니다.

이는 또한 AI 어시스턴트가 이전에 말한 내용에 대한 질문을 위해 2단계 검색 (two-pass retrieval)을 도입했습니다. 쿼리가 "you suggested", "you told me", "you recommended"와 같은 트리거를 포함할 때, 첫 번째 단계는 사용자 발화 전용 인덱스 (user-turn-only index)를 검색한 다음, 두 번째 단계는 전체 대화 텍스트 (사용자 + 어시스턴트 발화)를 사용하여 상위 5개 결과를 재인덱싱합니다.

결과: 98.4% R@5. 시간적 부스트만으로 임베딩이 본질적으로 "3일 전"을 인코딩하지 못하기 때문에 의미론적 검색이 찾을 수 없었던 대화를 복구했습니다.

**4단계: 선호도 추출 (v3, +0.6pp)**
여기서 선호도 윙 (preference wing)이 작동합니다. 수집 시점에 16개의 정규 표현식 패턴이 선호도 표현을 추출하고 합성 문서를 생성합니다. 쿼리 시점에는 재순위화 후보 풀이 상위 10개에서 상위 20개로 확장되어 선호도 일치 항목이 더 잘 나타날 수 있도록 합니다.

결과: LLM 재순위화 (reranking) 없이 98.4% R@5. 쿼리당 약 0.001달러의 Haiku 재순위화를 사용하면 99.4% R@5까지 올라갑니다. 500개의 질문 중 3개의 질문만 놓쳤습니다.

**5단계: LLM 재순위화 (선택 사항, +1.0pp)**
마지막 단계는 선택 사항이며 비용이 드는 유일한 단계입니다. Claude Haiku는 상위 20개 후보를 받아 의미론적 관련성 판단을 내립니다. 쿼리당 약 0.001달러의 비용으로 점수를 98.4%에서 99.4%로 끌어올립니다.

이 단계에서의 카테고리별 세부 분석은 흥미롭습니다. 지식 업데이트 질문은 100% (n=78), 다중 세션 질문은 100% (n=133), 단일 세션 사용자 질문은 100% (n=70)를 기록했습니다. 시간적 추론 (temporal reasoning)은 99.2% (n=133)에 도달했습니다. 가장 약한 카테고리는 단일 세션 선호도 (single-session-preference)로 96.7% (n=30)이며, 이는 정규 표현식 기반 선호도 추출이 모든 표현 방식을 포착할 수 없기 때문일 가능성이 높습니다.

## AAAK: 실패했지만 (아직은) 가치 있는 압축 실험

MemPalace는 AAAK (이름은 의도적으로 설명되지 않음; README에는 "묻지 마세요; 그 자체로 긴 이야기입니다"라고 명시되어 있음)라는 손실 압축 기호 요약 형식 (lossy symbolic summarization format)을 제공합니다.

AAAK가 실제로 무엇을 하는지 이해하는 것이 중요합니다. 왜냐하면 그 야망과 현재 결과 사이의 간격이 원본 텍스트 명제가 왜 중요한지를 정확히 보여주기 때문입니다.

AAAK는 텍스트를 다음 다섯 가지 구성 요소로 추출하여 압축합니다.

*   **엔티티 (Entities)**
    엔티티는 3글자의 대문자 코드로 매핑됩니다. "Alice"는 "ALC"가 되고 "Bob"은 "BOB"이 됩니다. 이 코드는 구성 가능한 엔티티 레지스트리 (entity registry)에서 가져옵니다.

*   **토픽 (Topics)**
    이는 150개 이상의 불용어 (stop words)를 필터링한 후 가장 빈번하게 나타나는 내용 단어이며, 고유 명사 (proper nouns)와 CamelCase 기술 용어에 대한 부스트가 적용됩니다.

*   **핵심 문장 (A key sentence)**
    결정 키워드와 길이 (시스템은 40-80자 조각을 선호)로 점수가 매겨진 다음, 최대 55자로 잘립니다.

*   **감정 (Emotions)**
    감정은 키워드 신호를 통해 감지되며, 30개 이상의 감정 라이브러리에서 3-4자 코드로 매핑됩니다. "vul"은 취약함 (vulnerable), "determ"은 단호함 (determined), "convict"은 확신 (conviction)을 의미합니다.

*   **플래그 (Flags)**
    플래그는 중요도를 표시하는 태그와 같습니다: ORIGIN, CORE, SENSITIVE, PIVOT, GENESIS, DECISION, TECHNICAL.

문장 출력은 다음과 같습니다: `ALC+BOB|GraphQL_REST|"chose GraphQL instead"|0.85|determ+convict|DECISION`

어떤 LLM도 이를 네이티브로 읽을 수 있습니다. 디코더는 필요 없습니다. 이론적으로, 대규모로 "ALC"와 같은 엔티티 코드가 말뭉치 전체에서 수백 번 "Alice"를 대체할 때 토큰 절약이 상당할 것이라는 것입니다.

하지만 솔직한 결과는 다음과 같습니다.

LongMemEval에서 AAAK 모드는 96.6%의 원본 모드 (raw mode)에 비해 84.2% R@5를 기록했습니다. 이는 12.4%포인트의 퇴보입니다. 요약 과정에서 손실된 정보가 토큰 절약의 이점보다 검색 정확도에 더 큰 손해를 입혔습니다. 그리고 작은 규모에서는 이 형식이 오히려 토큰을 추가합니다. README 자체 예시에서는 66개의 영어 토큰을 73개의 AAAK 토큰으로 압축합니다.

AAAK 실험은 실패했음에도 불구하고 가치가 있습니다.

이는 원본 텍스트 명제에 대한 실증적 증거를 제공합니다. 압축이 손실 압축 (lossy)일 때, 검색 정확도를 잃게 됩니다. AAAK의 엔티티 코드 접근 방식이 진정으로 대규모 (수백만 개의 반복되는 엔티티 참조)에서 효과를 볼 것인지에 대한 질문은 여전히 남아 있습니다. 이슈 #95는 법률 및 의료 RAG (검색 증강 생성)와 같은 고정밀 도메인에서의 해시 충돌 충실도 (hash collision fidelity)에 대한 사려 깊은 우려를 제기했습니다. 이는 연구 방향이지 실제 프로덕션 기능은 아닙니다.

## 경제성: 연간 0.70달러 대 507달러

여기 충분한 주목을 받지 못하는 숫자가 있습니다.

MemPalace는 1,950만 토큰의 대화 기록을 가정하고 6개월간 매일 AI 메모리를 실행하는 비용을 계산했습니다. LLM 기반 요약 (산업 표준 접근 방식)을 사용하면 연간 약 507달러가 소요됩니다. Zep은 구독료를 청구하고, Mem0에는 클라우드 티어가 있습니다.

MemPalace의 놀라운 비용은?

연간 0.70달러입니다.

월간이 아닙니다. 연간입니다.

매일 5번의 검색을 추가해도 연간 약 10달러입니다. 쿼리당 0.001달러의 선택적 Haiku 재순위화를 추가해도 여전히 15달러 미만입니다.

비용 차이는 미미한 수준이 아닙니다. 두 자릿수 차이입니다. 그리고 이는 검색 정확도를 이끄는 동일한 아키텍처 결정에서 비롯됩니다. 즉, LLM이 필요 없는 곳에서는 LLM을 사용하지 마십시오. 원본 텍스트를 저장하고, 검색에는 임베딩을 사용하십시오. 마지막 3%의 정확도를 위해서만 LLM을 호출하십시오.

개인 지식 관리 시스템 (personal knowledge management system)을 구축하는 사람이나 AI 지원 개발 (AI-assisted development)을 운영하는 소규모 팀에게는 이것이 "감당할 수 있는" 것과 "무료"의 차이입니다.

## 제품을 구축하고 있다면 이것이 의미하는 것

10명으로 구성된 엔지니어링 팀을 운영하고 있으며, 모두 매일 Claude Code 또는 Cursor를 사용한다고 가정해 봅시다. 6개월 후에는 수만 개의 대화 세션이 쌓일 것입니다. 기술적 결정, 디버깅 접근 방식, 아키텍처 논의 등 모든 것이 이 세션에 존재하지만, 컨텍스트 윈도우 (context window)가 지나가면 즉시 손실됩니다.

MemPalace는 이러한 세션을 검색 가능한 말뭉치로 만듭니다.

새로운 엔지니어가 합류하여 "데이터베이스 스키마는 어떻게 결정했나요?"라고 묻습니다. 답은 어딘가에 있는 드로어에 글자 그대로 저장되어 있으며, 96.6%의 정확도로 검색 가능합니다.

아무도 문서를 작성할 필요가 없었습니다. 아무도 위키를 유지할 필요가 없었습니다. 대화 자체가 문서입니다.

19가지 도구를 갖춘 MCP 서버는 이를 실용적으로 만듭니다. `claude mcp add mempalace -- python -m mempalace.mcp_server`라는 하나의 명령만으로 AI 어시스턴트가 전체 대화 기록에 액세스할 수 있습니다. 프로덕션 문제를 디버깅 중이고 3개월 전에 비슷한 문제를 해결했던 것이 막연히 기억난다면, 채팅 기록을 grep 검색하는 대신 AI가 MemPalace를 쿼리하여 관련 대화를 글자 그대로 검색하고 정확히 중단했던 지점에서 작업을 재개할 수 있습니다.

Mem0 또는 Zep의 클라우드 티어 구독료를 감당할 수 없는 사람들에게는, 이것이 로컬에서 무료로 실행되는 영구 AI 메모리의 전체 가치 제안입니다.

이것은 저의 관점입니다. 당신은 당신이 편안하게 느끼는 것을 해야 합니다. 또한, MemPalace에 대한 저의 첫 번째 시도입니다. 저는 용어에 다소 불쾌감을 느낍니다. 저는 구현이 매우 원시적이라고 생각합니다.

하지만 작동한다면 작동하는 것입니다. 저는 OpenPDFLoader를 핵심 파이프라인으로 사용하여 진지하게 시도해 볼 예정입니다 (코딩 관련 기사를 기대하고 있습니다).

## 커뮤니티가 발견한 것 (그리고 그것이 프로젝트를 더 좋게 만든 이유)

MemPalace의 GitHub 이슈 페이지는 협업적 동료 검토 (collaborative peer review)와 같습니다. 커뮤니티는 단순히 버그를 제기한 것이 아니라, 가정을 철저히 검증하고, 벤치마크를 독립적으로 재현하며, 문서화 및 코드를 개선하는 건설적인 분석을 제공했습니다.

주목할 만한 몇 가지 주요 내용은 다음과 같습니다.

*   **실제 토큰 수를 세지 않았습니다.**
    사용자 `panuhorsmalahti` (이슈 #43)는 AAAK 압축 예시를 OpenAI의 실제 토크나이저를 통해 실행했고, README의 토큰 추정치가 실제 토크나이저 대신 `len(text)//3` 휴리스틱 (heuristic)을 사용했음을 발견했습니다. 커뮤니티가 이를 지적했고, 저자들은 즉시 인정했으며, 수정 사항이 README 업데이트에 반영되었습니다.

*   **LoCoMo 정답 데이터 (Ground Truth)에 오류가 포함되어 있습니다.**
    사용자 `dial481` (이슈 #29)은 벤치마크 주장에 대한 상세한 방법론적 검토를 제출하며, LoCoMo 정답 데이터 (ground truth)에 오류가 있고, 일부 코드 패치가 특정 테스트 질문을 대상으로 했으며, ConvoMem 비교가 서로 다른 평가 메트릭스를 혼합했다고 지적했습니다. 이러한 종류의 방법론적 관찰은 발표된 숫자를 신뢰할 수 있게 만듭니다.

*   **원본 모드 (Raw Mode) 자체로 96% 정확도를 보였습니다.**
    이것은 저를 기쁘게 하면서도 동시에 슬프게 합니다. 원본 텍스트가 실제로 96% 더 잘 작동했다는 것을 믿기에는 너무 높습니다.

    사용자 `gizmax` (이슈 #39)는 M2 Ultra Mac Studio에서 전체 벤치마크 스위트를 독립적으로 실행했습니다. 원본 모드의 96.6% 결과가 정확히 재현되었습니다. 5분도 채 걸리지 않았습니다. 독립적인 하드웨어에서 말입니다. 그러나 `gizmax`는 또한 AAAK 압축률이 주장보다 낮고 룸 탐지 (room detection) 작업이 필요하다는 것을 확인했습니다. 핵심 결과를 검증하고 개선 영역을 식별하는 독립적인 재현입니다. 이것이 바로 과학이 올바르게 작동하는 방식입니다.

*   **다른 많은 커뮤니티 기여가 활발합니다.**
    다른 커뮤니티 기여에는 `.mempalace-ignore` 기능 (이슈 #102), Gemini CLI 통합 (이슈 #107), 다중 홉 탐색 경로 (multi-hop traversal paths) (이슈 #101), 단일 바이너리 배포 (single-binary deployment)를 위한 Zig 재작성 제안 (이슈 #30) 등이 있습니다. Sara Brain 프로젝트와의 아키텍처 유사성에 대한 사려 깊은 논의 (이슈 #104)도 있으며, 이는 AI 메모리 시스템의 수렴적 설계 (convergent design)라는 더 넓은 질문을 다룹니다.
