현재 실제 운영 중인 가장 성공적인 세 가지 AI 에이전트 플랫폼은 여러분이 장보기 목록을 보관하는 방식과 동일한 방식으로 메모리(기억)를 저장합니다. 왜 이것이 생각보다 터무니없는 일이 아닌지 그 이유를 알려드립니다.

저는 지난 2주 동안 Manus, OpenClaw, Claude Code의 아키텍처를 분석하는 데 시간을 보냈습니다. 이들은 주말 취미 프로젝트가 아닙니다. Manus는 출시 8개월 만에 연간 반복 매출(ARR) 1억 달러를 달성했고, Meta에 20억~30억 달러에 인수되었습니다. Claude Code의 연간 매출액(run-rate revenue)은 2026년 2월에 25억 달러를 넘어섰습니다. OpenClaw는 310,000개가 넘는 GitHub 별을 보유하고 있습니다.

이 세 플랫폼은 모두 일반 마크다운(Markdown) 파일을 주된 메모리 시스템으로 사용합니다. 관리형 벡터 데이터베이스(managed vector database)도, 5천만 달러짜리 인프라 제품도 아닙니다. 그냥 마크다운 파일입니다.

이 글에서 배울 내용:

*   **수렴 패턴 (The Convergence Pattern):** 세 개의 독립적인, 실제 운영 규모의 에이전트 시스템이 모두 파일 기반 메모리에 도달한 이유
*   **컨텍스트 엔지니어링 논리 (The Context Engineering Logic):** Manus가 "파일 시스템을 컨텍스트로 활용"하는 방식을 통해 실제 비용을 절감하는 단위 경제학(unit-economics) 전략으로 전환한 방법
*   **하이브리드 아키텍처 (The Hybrid Architecture):** OpenClaw가 외부 데이터베이스에 권한을 넘기지 않으면서도 마크다운 위에 시맨틱 검색(semantic retrieval) 레이어를 추가하는 방법
*   **균형점 (The Equilibrium Point):** 파일이 작동을 멈추는 시점과 실제로 데이터베이스가 필요한 시점 (생각보다 늦습니다)

## 세 팀, 하나의 기묘한 비결

이것이 흥미로운 이유는 다음과 같습니다. 이 세 프로젝트는 서로를 베낀 것이 아닙니다. 각기 다른 사용자를 위해 다른 문제를 해결하면서도 독립적으로 동일한 패턴에 도달했습니다.

Manus는 복잡한 다단계 작업을 처리하는 소비자용 AI 에이전트입니다. OpenClaw는 오픈 소스 개인 AI 비서입니다. Claude Code는 Anthropic의 공식 코딩 에이전트입니다. 서로 다른 팀. 서로 다른 코드베이스. 서로 다른 비즈니스 모델입니다.

## Manus: 파일이 비즈니스 전략임을 증명한 1억 달러의 사례

Manus는 그 팀이 데이터베이스 대신 파일을 선택한 이유를 실제로 공개했기 때문에 가장 흥미로운 사례입니다. 2025년 7월 엔지니어링 게시물에서 Manus의 공동 창립자 Yichao "Peak" Ji는 이 논리를 이례적으로 솔직하게 설명했습니다.

핵심 통찰력은 이렇습니다. Manus는 출력 토큰 1개당 평균 100개의 입력 토큰(input tokens)을 처리합니다. 이 비율 때문에 입력 비용이 지배적인 변수가 됩니다. 그리고 Claude Sonnet에서 캐시된(cached) 입력 토큰과 캐시되지 않은(uncached) 입력 토큰 간의 가격 차이는 대략 10배입니다 (100만 토큰당 약 0.30달러 대 3달러).

이 단 하나의 숫자가 Manus의 아키텍처 결정 거의 모든 것을 설명합니다.

입력-대-출력 비율이 100:1이고 캐시된 토큰이 10분의 1 비용인 경우, 안정적이고 예측 가능한 프롬프트 접두사(prompt prefixes)를 원하게 됩니다. 추가 전용(append-only) 컨텍스트(context)를 원하게 됩니다. 메모리 시스템이 키-값 캐시(KV-cache) 적중률과 잘 작동하기를 원하게 됩니다. 이것들은 스타일 선호도가 아닙니다. 바로 단위 경제학입니다.

## todo.md: 하나의 파일로 메모리와 주의 제어

전체 이미지 보기를 위해 Enter를 누르거나 클릭하세요.

Manus는 작업당 평균 50개의 도구 호출을 합니다. 이는 긴 일련의 동작이며, LLM은 긴 컨텍스트(context)의 중간에 있는 정보가 무시되는 "중간 정보 손실 효과 (lost-in-the-middle effects)"로 어려움을 겪는 것으로 유명합니다.

Manus의 해결책은 우아합니다. 복잡한 작업 중에 `todo.md` 체크리스트를 생성하고 반복적으로 업데이트합니다. 에이전트가 한 단계를 마칠 때마다 `todo` 파일을 다시 작성하며, 이 재작성은 현재 계획을 컨텍스트 창(context window)의 가장 최근 부분, 즉 모델이 가장 주의를 기울이는 곳에 배치합니다.

이것은 미묘하지만 중요한 지점입니다. 마크다운 파일은 단순히 정보를 저장하는 것이 아닙니다. 그것은 주의를 형성하고 있습니다. 데이터베이스 기반의 검색 증강 생성(RAG) 시스템은 사실을 검색할 수 있지만, 계획을 명시적인 계획 아티팩트(planning artifact)로 만들어 최근 컨텍스트에 다시 도입하지 않는 한 "계획을 최신 상태로 유지하는 (keep the plan hot)" 문제를 해결하지 못합니다. Manus는 바로 텍스트 파일로 그 작업을 수행하고 있습니다.

유출된 Manus 시스템 프롬프트(2,500개 이상의 GitHub 별을 받은 널리 알려진 GitHub 기스트(gist))는 독립적인 관점에서 이를 확인합니다. 프롬프트에는 에이전트가 `todo.md`를 체크리스트로 만들고 각 항목을 완료한 직후 업데이트하도록 요구하는 명시적인 규칙이 포함되어 있습니다. 또한 에이전트가 중간 결과를 저장하고 다른 참조 정보를 별도의 파일에 보관하도록 지시합니다.

마케팅 이야기에 회의적이라 할지라도, 실제 운영 프롬프트는 같은 이야기를 합니다. 파일 쓰기 규율은 시스템에 내장되어 있습니다.

## OpenClaw: 통제권을 넘기지 않고 검색 추가하기

Manus가 "순수 파일" 사례라면, OpenClaw는 다음 단계가 어떤 모습인지 보여줍니다. 마크다운을 정식 메모리 표면(canonical memory surface)으로 유지하면서 그 위에 시맨틱 검색(semantic retrieval) 레이어를 추가하며, 이 모든 것이 외부 관리형 벡터 데이터베이스를 요구하지 않습니다.

**메모리 레이아웃 (The Memory Layout)**

OpenClaw의 메모리 시스템은 두 가지 종류의 파일을 사용합니다.

*   `MEMORY.md`는 영구적이고 선별된 정보(에이전트가 항상 알아야 할 것)를 담습니다.
*   `memory/YYYY-MM-DD.md` 형식의 날짜 파일은 일상적인 메모리를 기록합니다.

이것은 "문서로서의 메모리 (memory as documentation)"입니다. 사람이 이 파일을 열고, 읽고, 편집하고, Git으로 버전 제어할 수 있습니다. 벡터 데이터베이스로는 그렇게 할 수 없습니다.

**압축 문제 (The Compaction Problem) (그리고 이것이 임베딩보다 더 중요한 이유)**

OpenClaw는 "그냥 데이터베이스를 사용하라"는 주장이 간과하는 실제 운영상의 문제를 드러냅니다. 컨텍스트 창이 가득 차면 어떻게 될까요?

OpenClaw는 자동 메모리 플러시(flush)를 구현합니다. 세션이 컨텍스트 한도(context limit)에 가까워지면, 시스템은 대화 기록이 압축(compaction)으로 잘려나가기 전에 모델에게 영구적인 메모리(durable memories)를 메모리 파일에 작성하도록 지시하는 자동 에이전트 턴(agent turn)을 트리거합니다.

이것은 핵심 문제가 "임베딩(embeddings)을 어디에 저장해야 하는가?"가 아니라는 실질적인 증거입니다. 문제는 "압축 경계를 넘어 상태를 어떻게 보존하는가?"입니다. 파일은 이를 직접적으로 해결합니다. 에이전트는 기억해야 할 것을 작성하고, 다음 세션은 이를 다시 읽습니다.

## 시맨틱 계층 (파일 위에 구축되었지, 파일을 대체하지 않음)

"벡터 검색은 어쩌지?" 하는 사람들에게 OpenClaw가 흥미로운 점은 바로 여기입니다.

OpenClaw는 마크다운 메모리 파일 위에 작은 벡터 인덱스(vector index)를 구축할 수 있습니다. 하지만 핵심은 "위에 (over)"입니다. 파일이 진실의 원천(source of truth)입니다. 벡터 인덱스는 키워드 검색(keyword search)이 부족할 때 메모리를 찾는 파생된 기능입니다.

원격 임베딩(remote embeddings)과 로컬 임베딩(local ones) 모두에서 작동하며, SQLite 내에서 벡터 검색을 실행하기 위해 `sqlite-vec`을 사용합니다. 관리형 인프라도, 외부 데이터베이스도 없습니다. 선택적 검색 계층이 추가된 로컬 파일일 뿐입니다.

OpenClaw는 명시적인 튜닝 노브(tuning knobs)를 사용한 하이브리드 검색(hybrid retrieval)도 포함합니다. 기본적으로 `vectorWeight`는 0.7, `textWeight`는 0.3이며, 중복 결과를 줄이기 위한 MMR 방식의 다양화(MMR-style diversification)와 오래된 메모리가 새로운 메모리보다 우선하지 않도록 하는 시간 경과 감쇠(temporal decay, 구성 가능한 반감기(half-life) 포함)를 제공합니다.

이는 성숙하고 운영을 고려한 설계입니다. 순수 키워드 검색이 재구성(paraphrases)과 동의어에서 실패한다는 것을 인정합니다. 그러나 이 문제를 기존 파일을 인덱싱하여 해결하며, 데이터베이스로 파일을 대체하지 않습니다.

## Claude Code: 벤더가 직접 패턴을 제공할 때

Claude Code가 중요한 이유는 마크다운 메모리 패턴을 커뮤니티 컨벤션에서 문서화된 제품 기능으로 전환했기 때문입니다.

**CLAUDE.md: 데이터베이스 없는 계층적 지침**

Claude Code의 메모리 시스템은 영구 지침(persistent instructions), 규칙, 워크플로우, 프로젝트 아키텍처 노트(project architecture notes)를 포함하는 `CLAUDE.md` 파일을 중심으로 합니다. 모델은 모든 세션 시작 시 이 파일을 읽습니다.

영리한 부분은 계층 구조에 있습니다.

*   조직 전체 규칙을 위한 관리형 정책 지침 (OS별 시스템 위치)
*   소스 제어(source control)를 통해 공유되는 프로젝트 지침 (`./CLAUDE.md`)
*   개인 기본 설정을 위한 사용자 지침 (`~/.claude/CLAUDE.md`)
*   Claude가 하위 디렉토리의 파일을 읽을 때 필요에 따라 로드되는 디렉토리 수준 파일

이것은 점진적 공개(progressive disclosure)입니다. 시작 시 모든 것을 컨텍스트에 덤핑하는 대신, 시스템은 범위에 따라 관련성 있는 것을 로드합니다. 이러한 종류의 "검색"에는 벡터 데이터베이스가 필요하지 않습니다. 파일 시스템의 디렉토리 구조(directory structure) 자체가 검색 메커니즘입니다.

## 한 가지 중요한 정정

Anthropic의 블로그 게시물은 `CLAUDE.md`가 "시스템 프롬프트(system prompt)의 일부가 된다"고 설명합니다. 하지만 실제 문서는 더 정확합니다. `CLAUDE.md` 내용은 시스템 프롬프트 *안쪽*이 아니라, 시스템 프롬프트 *이후* 사용자 메시지(user message)로 전달됩니다. 이는 강제 구성이 아닌 컨텍스트입니다.

이것은 "메모리"가 운영상 의미하는 바를 바꿉니다. 파일은 마법의 강제 경계가 아닙니다. 잘 작동하려면 간결하고 일관성 있게 유지해야 하는 표준화된 컨텍스트 채널입니다. Anthropic은 심지어 각 `CLAUDE.md` 파일을 약 200줄 미만으로 유지하고 주기적으로 모순을 제거할 것을 권장합니다.

**자동 메모리 (Auto-Memory): 제한된 파일 기반의 두 번째 두뇌**

Claude Code는 또한 프로젝트별로 `~/.claude/projects/<project>/memory/` 경로에 `MEMORY.md` 파일과 선택적 주제 파일(topic files)을 포함하는 자동 메모리 디렉토리(auto-memory directory)를 유지합니다. `MEMORY.md`의 처음 200줄만 세션 시작 시 로드됩니다. 주제 파일은 필요에 따라 로드됩니다.

이것은 "메모리 파일이 무한정 커지는" 비판에 대한 벤더(vendor)가 설계한 답변입니다. 무한 메모리가 무료인 척하는 대신, Claude Code는 항상 로드되는 인덱스에 하드 캡(hard cap)을 부과하고 주제 파일로 넘치게(spillover) 하는 것을 권장합니다. 이는 "그냥 마크다운을 사용하라"는 주장이 무시하는 확장성 문제를 피하는 간단하고 실용적인 제약입니다.

전체 이미지 보기를 위해 Enter를 누르거나 클릭하세요.

## 마크다운 파일이 실제로 실패하는 지점

파일이 모든 것을 해결한다고 주장한다면 저는 여러분에게 실례를 범하는 것입니다. 그렇지 않습니다. 이 주장의 솔직한 버전은 더 좁고 더 유용합니다. 단일 사용자(single-user) 또는 단일 스레드 에이전트 워크플로우(single-threaded agent workflows) (특히 로컬 코딩 에이전트)의 경우, 파일 기반 메모리는 LLM의 작동 방식과 실행 비용에 부합하기 때문에 불균형적으로 큰 가치를 제공합니다.

하지만 실제 실패 모드도 있습니다.

*   **컨텍스트 예산 압박 (Context budget pressure):** Claude Code는 `CLAUDE.md`가 매 세션마다 토큰을 소비하며, 너무 큰 파일은 준수도를 떨어뜨린다고 명시적으로 경고합니다. 파일은 내용이 너무 많아지고 내부적으로 모순될 때까지 작동합니다.
*   **동시성 (Concurrency):** 여러 에이전트나 사용자가 동일한 메모리에 접근해야 하는 순간, 동시 파일 쓰기는 데이터를 손상시킬 수 있습니다. 그 시점에서는 데이터베이스 보장(database guarantees) (원자성(atomicity), 격리(isolation), 조정(coordination))이 필요합니다. 이것은 마크다운 대 데이터베이스의 문제가 아닙니다. 파일 시스템 대 데이터베이스의 문제이며, 동시 접근 시 데이터베이스가 우위에 있습니다.
*   **대규모 시맨틱 검색 (Semantic retrieval at scale):** `grep` 및 키워드 검색은 재구성과 동의어에서 실패합니다. 지식 베이스가 커짐에 따라 벡터 검색은 선택 사항이 아니라 필수 사항이 됩니다. OpenClaw의 하이브리드 모드는 이를 실용적으로 인정하는 것입니다.
*   **캐시 무효화 비용 (Cache invalidation costs):** Manus는 도구를 중간에 추가하고 제거하는 것과 같은 일부 "영리한" 동적 동작이 키-값 캐시(KV caches)를 무효화하고 모델을 혼란스럽게 하여 실제로 성능을 저하시킨다는 것을 발견했습니다. 모든 복잡성이 기능을 제공하는 것은 아닙니다. 일부는 그저 더 높은 비용을 초래할 뿐입니다.

전체 이미지 보기를 위해 Enter를 누르거나 클릭하세요.

## 균형 아키텍처 (The Equilibrium Architecture)

이 세 시스템의 증거는 "파일 대 데이터베이스"보다 더 유용한 잠재적 균형점을 시사합니다.

전체 이미지 보기를 위해 Enter를 누르거나 클릭하세요.

*   **기본 인터페이스로서의 파일 (주로 마크다운):** 계획, 규칙, 학습 내용 및 지침은 사람이 읽을 수 있고 버전 제어가 가능한 텍스트 파일에 존재합니다. 이는 모델과 사람이 모두 직접 상호작용하는 계층입니다.
*   **디스크로의 적극적인 오프로딩:** 큰 도구 출력(tool outputs)은 복원 가능한 참조(restorable references, 경로, URL)와 함께 파일로 저장되어 정보 손실 없이 컨텍스트를 축소할 수 있습니다. Manus는 이를 "컨텍스트 창(Context Window) = RAM, 파일 시스템(Filesystem) = 디스크(disk)" 모델이라고 부릅니다.
*   **파생된 검색 계층:** 규모가 시맨틱 검색을 요구할 때, 파일 위에 인덱스(index)를 구축합니다. OpenClaw는 SQLite와 `sqlite-vec`을 사용하여 이를 수행합니다. 파일은 진실의 원천으로 남고, 인덱스는 검색 최적화(search optimization)입니다.
*   **명확한 확장 경로:** 동시 접근 하에서 메모리가 공유되고 정확해야 할 때, 기반 구조(substrate)를 데이터베이스로 옮깁니다. 하지만 그 경우에도 에이전트가 마주하는 인터페이스는 여전히 "폴더 안의 파일"처럼 보일 수 있습니다.

이것은 급진적인 주장이 아닙니다. 그저 가장 간단하게 작동하는 것부터 시작하고, 운영상의 요구가 발생할 때 인프라를 추가하라는 말입니다. 현재 방대한 종류의 에이전트 워크플로우에 있어, 가장 간단하게 작동하는 것은 마크다운 파일입니다.

전체 이미지 보기를 위해 Enter를 누르거나 클릭하세요.

## 이것이 벡터 데이터베이스 시장에 미치는 영향

벤처 캐피탈(VC-funded)이 투자한 메모리 인프라에 대한 불편한 질문은 현실입니다. 하지만 이 주장 하에서 자금 조달 그림이 붕괴되는 것은 아닙니다. 오히려 양분됩니다.

Mem0 (2,400만 달러 모금), Letta (7천만 달러 기업 가치로 1천만 달러 시드 투자), Zep (Graphiti 프로젝트가 GitHub 별 20,000개를 넘어섬)과 같은 메모리 계층 스타트업(Memory-layer startups)은 파일 우선 시스템이 어려움을 겪는 부분들, 즉 배포 간 내구성(durability), 사용자 수준 개인화(user-level personalization), 대규모 검색(retrieval at scale), 거버넌스(governance), 기업 제어(enterprise controls)를 특별히 해결하고 있습니다.

한편, 벡터 검색은 보편적인 기능(commodity feature)이 되어가고 있습니다. 설문 조사 데이터에 따르면 채택률이 꾸준히 상승하고 있으며, 벡터 기능은 Postgres 확장(Postgres extensions), 통합 데이터베이스 기능, 관리형 서비스(managed services) 등에서 나타나고 있습니다. 시장은 파일과 벡터 중 하나를 선택하는 것이 아니라, 둘 모두의 구성 가능한 조합(composable combinations)으로 나아가고 있습니다.

## 결론 (불필요한 미사여구 없이)

"5천만 달러짜리 벡터 데이터베이스를 이긴 마크다운 파일"이라는 주장은 데이터베이스의 역할이 없다는 예측이 아니라, 기본 시작 아키텍처에 대한 주장으로서 가장 잘 작동합니다.

더 정확하고 증거에 기반한 주장은 이렇습니다. 성공적인 에이전트 스택(stacks)은 파일을 상태(state)를 위한 일급 인터페이스(first-class interface)로 취급하고, 운영상의 요구가 발생할 때만 검색(retrieval) 및 더 강력한 기반 구조(substrates)를 추가합니다.

오늘날 에이전트를 구축 중이고 첫 번째 본능이 벡터 데이터베이스를 설정하는 것이라면, 한 발 물러서서 Manus, OpenClaw, Claude Code가 실제로 출시하는 것을 살펴보세요. 마크다운 파일로 시작하세요. 데이터베이스는 나중에 언제든지 추가할 수 있습니다.

그 반대의 경우는 아마도 그렇게 말할 수 없을 겁니다.

이 글이 에이전트 메모리에 대한 여러분의 생각을 바꾸었다면, 여러분의 아키텍처 선택에 대해 듣고 싶습니다. 현재 스택은 무엇인가요? 파일 우선인가요, 데이터베이스 우선인가요, 아니면 그 중간 어딘가인가요?
