# Vectorless RAG: RAG 파이프라인에 벡터 데이터베이스가 필요 없는 이유

AI 비서에게 200페이지 분량의 계약서에 대해 질문했을 때, 비서는 자신감 있게 대답하지만 그 답이 틀릴 수 있습니다. 모델은 정확한 주제에서 텍스트를 가져왔지만, 잘못된 조항을 참조했고 이를 인지하지 못했습니다.

이는 모델의 문제가 아니라 검색(retrieval) 문제입니다. 벡터 RAG의 핵심 전제인 '유사해 보이는 텍스트가 관련 있는 텍스트'라는 가정이 구조화된 문서에서는 유효하지 않기 때문에 이러한 문제가 발생합니다.

대안적인 접근 방식이 있습니다: Vectorless RAG. 이 방식은 임베딩(embeddings), 벡터 데이터베이스, 청킹(chunking) 파이프라인을 사용하지 않습니다. 대신 유사한 텍스트를 찾는 대신, 인간 전문가가 하듯이 문서 내에서 답변이 있을 만한 위치를 추론합니다.

이 글에서는 Vectorless RAG의 개념, 기존 RAG보다 뛰어난 부분, 그리고 약 50줄의 파이썬 코드로 이를 구현하는 오픈 소스 라이브러리인 PageIndex를 사용하여 구축하는 방법을 설명합니다.

## 전통적인 RAG란 무엇인가요?

RAG(Retrieval-Augmented Generation)는 AI 모델이 학습된 공용 데이터 외에 사용자의 특정 문서를 알지 못한다는 문제를 해결합니다. RAG는 쿼리 시점에 문서에서 관련 정보를 가져와 모델에 제공함으로써 이 문제를 해결합니다.

RAG의 두 가지 주요 단계는 다음과 같습니다:

**1단계: 색인화 (한 번 수행)**
*   문서가 300~500단어 분량의 청크(chunk)로 분할됩니다.
*   각 청크는 해당 의미의 수학적 지문인 임베딩(embedding)이라는 숫자 목록으로 변환됩니다.
*   모든 임베딩은 유사한 숫자를 빠르게 찾는 데 최적화된 벡터 데이터베이스(Pinecone, Weaviate, pgvector 등)에 저장됩니다.

**2단계: 쿼리 (질문마다 수행)**
*   질문이 동일한 숫자 형식으로 임베딩됩니다.
*   데이터베이스는 질문의 숫자와 가장 가까운 청크를 반환합니다.
*   이 청크와 질문은 LLM(거대 언어 모델)으로 전달되어 답변을 생성합니다.

이러한 두 단계 파이프라인은 일반 지식에 대한 광범위한 질문에는 잘 작동하지만, 구조화된 전문 문서에서는 한계를 드러냅니다.

### 실제 문제점

문제는 청크 크기나 임베딩에 있지 않습니다. 더 근본적인 문제입니다. 벡터 검색은 진실이 아닌 유사성에 최적화되어 있습니다. 따라서 "3분기 매출 성장을 견인한 요인은 무엇인가요?"와 같은 실제 질문을 하면, 매출을 언급하는 모든 청크를 가져오지, 실제로 매출을 설명하는 청크를 가져오지 않습니다.

청킹은 상황을 악화시킵니다. 의미가 분할되고, 정의는 한 청크에, 의존성은 다른 청크에 존재하며, 상호 참조는 끊어집니다. 모델은 컨텍스트가 불완전하기 때문에 추측하게 됩니다.

그리고 실패했을 때, 사용자에게는 아무런 단서가 없습니다. 유사성 점수만 받을 뿐, 왜 특정 정보가 검색되었는지에 대한 추론이 없습니다. 금융이나 법률과 같은 분야에서는 이는 심각한 결함입니다.

## Vectorless RAG란 무엇인가요?

Vectorless RAG는 전체 임베딩-검색-청크 파이프라인을 LLM(거대 언어 모델) 기반의 구조화된 문서 표현 추론으로 대체하는 검색 접근 방식입니다.

핵심 아이디어는 문서를 벡터로 변환하고 가장 유사한 것을 찾는 대신, 언어 모델이 문서의 구조화된 맵을 읽고 어떤 섹션을 열지 결정하도록 하는 것입니다. 모델은 복잡한 보고서를 탐색할 때 인간이 하는 방식과 동일하게 문서 계층 구조에 대해 추론합니다: 구조를 보고, 답변이 있을 가능성이 높은 위치를 식별한 다음 해당 섹션을 읽습니다.

Vectorless RAG가 제거하는 것:
*   선택하고 유지 관리할 임베딩 모델 불필요
*   호스팅하고 쿼리할 벡터 데이터베이스 불필요
*   조정할 청킹 전략 불필요
*   잘못된 청크가 높은 유사성 점수와 함께 반환되는 조용한 검색 실패 방지

Vectorless RAG가 추가하는 것:
*   **문서 트리:** 각 노드가 제목, 페이지 범위, LLM이 생성한 요약을 포함하는 계층적 JSON(JavaScript Object Notation) 구조
*   **추론 단계:** LLM이 트리를 읽고 질문에 따라 어떤 노드를 검색할지 결정
*   **추적 가능한 답변:** 어떤 섹션이 검색되었고 그 이유를 정확히 확인할 수 있음

Vectorless RAG가 모든 사용 사례에 적합한 것은 아닙니다. 이는 답변이 특정 식별 가능한 섹션에 있는 구조화된 문서, 즉 재무 보고서, 법률 계약서, 기술 매뉴얼, 학술 논문 등을 위해 설계되었습니다. 이러한 문서의 경우, 느슨하게 관련된 청크 조각들을 조합하는 대신 특정 페이지에 기반한 답변을 생성합니다.

## PageIndex의 역할

PageIndex(github.com/VectifyAI/PageIndex)는 Vectorless RAG를 구현하는 오픈 소스 라이브러리입니다. 이 라이브러리는 두 가지를 처리합니다:
*   **트리 생성:** PDF를 읽고 계층적 문서 트리를 구축합니다.
*   **트리 검색:** 해당 트리를 기반으로 추론 기반 탐색(navigation)을 위한 유틸리티를 제공합니다.

PageIndex는 개념 자체라기보다는 도구입니다. 이 글의 나머지 부분에서는 PageIndex를 사용하여 이 접근 방식을 구현하지만, 기본 패턴(트리를 구축하고, 그 위에서 추론하며, 올바른 섹션을 가져오는 것)은 모든 LLM과 모든 트리 생성 방법으로 작동합니다.

## PageIndex의 Vectorless RAG 구현 방식

200페이지 분량의 연간 보고서를 연구 분석가에게 주고 질문하는 상황을 상상해 보세요. 분석가는 모든 페이지를 읽지 않습니다. 목차를 보고 답변이 포함될 가능성이 가장 높은 섹션을 식별한 다음 해당 페이지를 찾아 읽고 답변합니다. PageIndex는 이와 똑같은 작업을 수행합니다. 다만, '분석가'는 LLM이고 '목차'는 문서에서 생성된 구조화된 JSON(JavaScript Object Notation) 트리입니다.

두 가지 단계는 다음과 같습니다:

**1단계: 트리 색인 구축**
PageIndex는 문서를 읽고 모든 노드가 섹션인 계층적 트리를 생성합니다. 각 노드에는 섹션 제목, 고유한 노드 ID, 페이지 범위(시작 및 끝 페이지), LLM이 생성한 해당 섹션 콘텐츠 요약, 그리고 하위 섹션에 대한 자식 노드가 저장됩니다. 문서는 자연스러운 구조를 유지합니다. 예를 들어, 섹션 4.2는 섹션 4에 속한다는 것을 알며, 서론은 결론과 연결됩니다. 어떠한 관계도 끊어지지 않습니다.

**2단계: 트리를 통한 추론**
질문이 들어오면 LLM은 트리(전체 텍스트가 아닌 제목과 요약만)를 읽고 다음을 생성합니다:
*   **사고 필드(thinking field):** LLM의 단계별 탐색 추론을 평이한 영어로 설명
*   **노드 목록(node_list):** 검색하기로 결정한 특정 노드 ID

그런 다음 LLM은 해당 노드의 전체 텍스트만 가져옵니다. 코사인 유사도(cosine similarity)는 사용되지 않습니다. 이는 사용자가 읽고 검증할 수 있는 감사 가능한(auditable) 명시적인 결정입니다.

DeepSeek-R1 연구 논문을 색인화하고 결론에 대한 질문에 답하는 예제를 통해 PageIndex를 사용하는 방법을 보여줍니다. 벡터 데이터베이스나 임베딩 모델은 필요하지 않습니다. PageIndex 라이브러리를 설치하고 OpenAI API 키를 설정한 후, PDF 문서를 다운로드하여 PageIndex에 제출하면 문서 ID가 반환됩니다. PageIndex가 문서 트리를 성공적으로 구축하면 각 노드의 제목과 요약을 포함하는 계층적 트리 구조를 검색할 수 있습니다.

질문이 주어지면 LLM은 전체 텍스트 없이 문서 트리의 제목과 요약만을 읽고, 질문에 답할 가능성이 가장 높은 노드를 식별합니다. 이 과정에서 LLM은 어떤 노드가 관련 있는지에 대한 단계별 추론 과정을 설명하는 '사고 필드'와 검색할 특정 노드 ID 목록인 '노드 목록'을 생성합니다.

예를 들어, "이 문서의 결론은 무엇인가요?"라는 질문에 대한 LLM의 추론은 다음과 같습니다.

**추론 과정:**
질문은 결론에 대한 것입니다. 일반적으로 결론은 명확하게 '결론'이라는 제목이 붙은 섹션에서 찾을 수 있습니다. 이 문서에서 노드 0019('5. 결론, 한계 및 향후 작업')가 가장 직접적으로 관련이 있습니다. 초록(노드 0001)에는 높은 수준의 요약이 포함될 수 있지만 전체 결론을 담을 가능성은 낮습니다. 논의(노드 0018)는 시사점을 다루지만 명시적으로 결론은 아닙니다. 따라서 기본 노드는 0019입니다.

**검색된 노드:**
노드 ID: 0019 페이지: 16 제목: 5. 결론, 한계 및 향후 작업

이 추론을 통해 LLM이 세 가지 후보를 평가하고, 그중 두 가지가 올바른 답변이 아닌 이유를 설명하며, 명확한 정당성을 가지고 노드 0019를 선택했음을 알 수 있습니다. 사용자는 이를 읽고 검증할 수 있으며, 오류가 발생했을 경우 특정 추론 단계로 거슬러 올라갈 수 있습니다. 이는 벡터 검색이 제공할 수 없는 부분입니다.

마지막으로, 검색된 노드의 전체 텍스트를 가져와서 LLM에 질문과 함께 제공하여 최종 답변을 생성합니다. 이 전체 파이프라인은 질문부터 답변까지 단 두 번의 LLM 호출(트리 탐색을 위한 한 번, 답변 생성을 위한 한 번)만 사용합니다.

생성된 답변:
- DeepSeek-R1-Zero는 콜드 스타트 데이터 없이 순수한 RL 접근 방식을 통해 다양한 작업에서 강력한 성능을 달성합니다.
- DeepSeek-R1은 콜드 스타트 데이터와 반복적인 RL 미세 조정을 결합하여 OpenAI-o1-1217과 비슷한 성능을 달성합니다.
- DeepSeek-R1의 추론을 더 작은 모델로 증류하는 것은 유망합니다. DeepSeek-R1-Distill-Qwen-1.5B는 수학 벤치마크에서 GPT-4o 및 Claude-3.5-Sonnet를 능가합니다.

이는 정확하며 16페이지에 근거합니다. 모든 주장은 노드 0019로 직접 추적됩니다. 만약 답변이 틀렸다면, 오류가 어디서 발생했는지 정확히 알 수 있습니다. 즉, 트리 검색이 노드 0019를 검색하기로 결정했고, 그 결정을 즉시 검토할 수 있습니다.

## 고려해야 할 한계점

Vectorless RAG를 도입하기 전에 다음과 같은 한계점을 알아두는 것이 중요합니다.

*   **지연 시간(Latency):** 모든 쿼리는 답변 생성 호출 외에 트리 탐색을 위해 최소 한 번의 추가 LLM 호출을 수행합니다. 이는 표준 RAG보다 상당한 지연 시간을 추가합니다. 사용자가 거의 즉각적인 응답을 기대하는 대화형 제품에서는 이러한 지연이 눈에 띌 수 있습니다.
*   **대규모 비용:** 벡터 유사성 검색은 색인 구축 후 쿼리당 한계 비용이 거의 없습니다. 반면 트리 검색은 모든 쿼리에서 LLM을 호출합니다. 정확한 비용은 모델 선택과 컨텍스트 길이에 따라 다르지만, 상당한 쿼리 볼륨에서는 순수 벡터 접근 방식에 비해 실제 비용 항목이 됩니다.
*   **문서 구조에 의존:** Vectorless RAG는 문서에 명확한 섹션 제목, 논리적 중첩, 목차와 같은 실제 계층 구조가 있을 때 가장 잘 작동합니다. 형식이 불량한 PDF, OCR(광학 문자 인식)이 없는 스캔 문서, PDF로 내보낸 프레젠테이션 자료는 약한 요약과 좋지 않은 검색 성능을 가진 평면적인 트리를 생성합니다.
*   **다중 문서 확장성(Multi-document scaling)은 여전히 개발 중:** 이 접근 방식은 단일 문서 질문 답변에는 강력합니다. 어떤 문서에 답변이 포함되어 있는지 알 수 없는 대규모 컬렉션의 경우 문서당 트리 오버헤드가 빠르게 증가합니다. PageIndex는 이를 알려진 한계로 문서화하고 있으며, 다중 문서 검색이 개발 중임을 명시하고 있습니다.
*   **모델 품질이 검색에 직접적인 영향:** 탐색 결정은 이를 수행하는 LLM의 품질에만 좌우됩니다. 더 강력한 추론 모델은 더 나은 트리 탐색을 생성합니다. 소형 모델을 사용하여 완전히 로컬에서 실행하려면, 프로덕션에서 결과를 신뢰하기 전에 신중하게 테스트해야 합니다.

## Vectorless RAG와 벡터 RAG 사용 시점

다음과 같은 경우 Vectorless RAG를 선택하세요:
*   문서에 명확한 구조(예: 10-K 보고서, 법률 계약서, 학술 논문, 기술 매뉴얼, 정책 문서)가 있는 경우
*   정확성이 최우선이며, 잘못된 답변이 실제 결과를 초래할 수 있는 경우
*   검색된 섹션, 출처 페이지, 선택에 대한 추론 등 추적 가능한 감사 기록이 필요한 경우
*   대규모 컬렉션 전체를 검색하는 것이 아니라 한 번에 특정 문서 하나로 작업하는 경우

다음과 같은 경우 벡터 RAG를 선택하세요:
*   수천 개의 문서로 이루어진 컬렉션이 있고 쿼리가 모든 문서를 포괄해야 하는 경우
*   "X에 대한 모든 것을 찾아라" 또는 "Y에 대해 무엇을 알고 있는가"와 같이 질문이 광범위하고 의미론적인 경우
*   쿼리 볼륨이 높고 응답 지연 시간이 엄격한 요구 사항인 경우
*   문서 형식이 불량하거나 명확한 구조가 없는 경우

실용적인 테스트: 실제 사용 사례에서 20~30개의 실제 질문을 가져와 두 가지 접근 방식을 모두 실행하고 정확도를 비교해 보세요. 하루 동안의 평가가 몇 달간의 아키텍처 논쟁보다 훨씬 효과적입니다.

## 결론

벡터 RAG는 질문과 유사한 텍스트를 검색하는 방식으로 작동합니다. 반면 Vectorless RAG는 문서 내에서 답변이 있을 만한 위치를 추론하는 방식으로 작동합니다.

대규모 문서 컬렉션에 걸친 대부분의 일반적인 사용 사례에서는 벡터 RAG가 여전히 올바른 시작점입니다.

그러나 정확성이 중요하고 잘못된 답변이 결과를 초래하는 구조화된 전문 문서의 경우, 문서를 트리로 구축하고 그 위에서 추론하는 것이 문서가 구성된 방식에 근본적으로 더 적합합니다.

PageIndex는 이러한 패턴의 한 가지 오픈 소스 구현입니다. 문서의 구조화된 맵을 구축하고 LLM이 읽을 섹션을 추론하도록 하며, 해당 섹션의 전체 텍스트를 가져오는 이 패턴 자체는 어떤 LLM로도 구축할 수 있습니다.
