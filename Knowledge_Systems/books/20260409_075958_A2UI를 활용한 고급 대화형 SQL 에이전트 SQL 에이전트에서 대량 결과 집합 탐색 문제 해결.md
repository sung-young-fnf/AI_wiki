A2UI를 활용한 고급 대화형 SQL 에이전트: SQL 에이전트에서 대량 결과 집합 탐색 문제 해결

SQL 에이전트는 질문을 SQL로 변환하는 데 능숙하지만, 챗봇 형태의 SQL 에이전트는 진정한 탐색 UI를 제공하지 않습니다. 결과 집합이 커지면 사용자 경험은 일반적으로 몇 가지 답답한 패턴으로 이어집니다. 예를 들어, 사용자는 다음과 같이 요청할 수 있습니다.

*   지난달의 모든 송장과 해당 품목을 나열해 주세요.
*   지난 7일간의 모든 고객과 그들의 구매 내역을 보여주세요.

운영 데이터베이스에서 이는 수천 개 이상의 행을 의미할 수 있습니다. 이런 경우 대부분의 SQL 에이전트 경험은 몇 가지 패턴으로 나타납니다.

*   **모든 곳에 `LIMIT 50`**: 에이전트가 수천 개의 행을 채팅에 쏟아내어 컨텍스트를 폭발시키지 않도록 작은 제한을 추가합니다.
*   **다음 50개를 보여줄 수 있나요? 반복**: 채팅에 실제 페이지 매김(paging) UI가 없기 때문에 사용자는 "다음 페이지 / 더 보기 / 계속"과 같은 내용을 입력하게 되고, 에이전트는 매번 쿼리를 다시 실행하거나 재구성해야 합니다 (종종 다른 LLM 호출과 함께).

이 시점에서 사용자는 더 많은 추론을 요구하는 것이 아니라 탐색을 요구하고 있습니다. 다음은 전통적인 SQL 에이전트의 일반적인 프롬프트 반복입니다.

1.  LLM이 SQL을 생성합니다.
2.  백엔드가 SQL을 실행합니다.
3.  에이전트가 채팅 응답 내에 결과를 반환하려고 시도합니다.
4.  컨텍스트 창이 폭발하므로 `LIMIT 50`을 추가합니다.
5.  사용자가 더 많은 것을 요청합니다. 반복이 계속됩니다.

개발자들이 페이지 매김을 추가하더라도, 많은 구현은 모든 상호작용에 대해 LLM을 다시 호출합니다.

`다음 클릭` → `메시지 전송` → `LLM이 다음 테이블 뷰를 생성` → `응답 반환`

이것은 데이터 탐색(data exploration)에 있어 두 가지 가장 큰 사용자 경험(UX) 문제를 야기합니다.

*   **지연 시간(Latency)**: 클릭당 2~5초
*   **비용(Cost)**: 클릭당 토큰 소모

따라서 진정한 문제는 SQL 생성(SQL generation)이 아니라 SQL 생성 후에 발생하는 일입니다. 챗봇은 대량의 테이블을 탐색하기에는 부적절한 공간입니다. 사용자가 실제로 원하는 것은 고정 열(stable columns), 페이지 제어(paging controls), 즉시 필터링되는 검색 상자(search box)와 같은 적절한 테이블 경험입니다. 바로 이 지점에서 A2UI가 도움이 됩니다. A2UI를 통해 "탐색"을 프롬프트 문제로 다루는 것을 멈추고 UI + 데이터 문제로 다루기 시작할 수 있기 때문입니다.

이 글은 A2UI를 활용한 실용적인 해결책을 제시합니다. 실제 테이블 UI를 한 번 구축한 다음, 데이터 전용 업데이트(data-only updates)를 통해 페이지 매김(pagination)과 검색(search)을 구동하여 탐색이 빠르고 일관되며 LLM을 다시 호출할 필요가 없도록 하는 것입니다.

### 아키텍처 개요

저는 Chinook SQLite 데이터베이스 위에 작은 텍스트-SQL 익스플로러(Text to SQL Explorer) 샘플을 구축했습니다. 목표는 완전한 BI 도구(BI tool)를 구축하는 것이 아니라, LLM이 첫 번째 단계(의도 → SQL)에만 사용되고 그 이후의 모든 탐색 작업은 일반적인 백엔드 로직으로 처리되는 깔끔한 아키텍처를 보여주는 것입니다.

시스템은 크게 백엔드와 프런트엔드로 구성됩니다.

#### 백엔드

백엔드는 요청이 LLM을 필요로 하는지 여부를 결정하는 한 가지 주요 작업을 수행합니다.

*   **텍스트 쿼리 → LLM**
    LLM은 사용자 질문을 SQL로 변환하고, 백엔드는 1페이지를 실행하며, 쿼리 세션(queryId)을 생성하고 다음을 반환합니다.
    *   데이터테이블 위젯(DataTable widget)을 위한 UI 설명 (프런트엔드가 이를 생성할 수 있도록)
    *   1페이지 데이터 (행 + 페이지 매김 메타데이터)
*   **UI 액션 → SQL 전용 경로 (LLM 없음)**
    페이지 변경 및 검색은 저장된 SQL 템플릿을 사용하여 세션 관리자(session manager)에 의해 직접 처리됩니다.

주요 백엔드 구성 요소:

*   **요청 라우터(Request router)**: 텍스트를 LLM으로 보내고, 사용자 액션(페이지/검색/초기화)을 세션 관리자로 보냅니다.
*   **LLM 에이전트(LLM agent)**: 기본 SQL과 초기 테이블 응답을 생성합니다.
*   **세션 관리자(Session manager)**: SQL 템플릿(queryId)을 저장하고 `LIMIT`/`OFFSET` 페이지와 검색 필터를 실행합니다.

#### 프런트엔드

사용자가 원시 결과(raw results)를 요청하면 에이전트는 데이터테이블(DataTable) UI를 반환하고, UI는 한 번만 생성됩니다. 이후의 상호작용은 경량 데이터 업데이트로 처리될 수 있습니다.

*   새로운 쿼리의 경우, 앱은 전체 응답(테이블 화면 포함)을 처리합니다.
*   페이지 매김/검색의 경우, 액션을 전송하고 데이터 전용 업데이트(data-only updates)를 적용합니다 (UI 재구축 없음).

주요 프런트엔드 구성 요소:

*   **사용자 정의 데이터테이블 컴포넌트(Custom DataTable component)**: 행 + 페이지 매김 + 검색을 렌더링하고 사용자 상호작용 시 `userAction` 이벤트를 발생시킵니다.
*   **A2UI 렌더러(A2UI renderer)**: 데이터 모델을 컴포넌트에 바인딩하여 업데이트가 자동으로 다시 렌더링되도록 합니다.

### 워크플로: 초기 쿼리 vs LLM 비활용 상호작용

이제 초기 쿼리(LLM 개입)부터 페이지 매김(pagination) 및 검색(search)을 위한 LLM 비활용 상호작용에 이르기까지 전체 워크플로를 살펴보겠습니다.

1.  **초기 쿼리 (LLM 개입)**

    예시: 사용자가 "모든 아티스트를 보여줘(Show all artists)"라고 입력합니다.

    *   앱은 텍스트 쿼리를 백엔드로 보냅니다.
    *   LLM은 의도를 해석하고 기본 SQL 쿼리(예: `SELECT * FROM Artist`)를 생성합니다.
    *   백엔드는 1페이지를 실행하고 SQL 템플릿을 저장하는 쿼리 세션(queryId)을 생성합니다.
    *   백엔드는 두 가지를 반환합니다.
        *   `surfaceUpdate`: 프런트엔드에 데이터테이블 위젯(UI 구조)을 생성하도록 지시합니다.
        *   `dataModelUpdate`: 1페이지 행과 페이지 매김 메타데이터를 제공합니다.
    *   프런트엔드는 `surfaceUpdate`를 처리하고, 사용자 정의 데이터테이블 컴포넌트를 인스턴스화하며, 데이터 모델에 바인딩하고 1페이지를 렌더링합니다.

2.  **페이지 매김 (LLM 없음)**

    사용자가 "다음(Next)"을 클릭합니다.

    *   UI는 `queryId`와 `currentPage`를 포함하는 `userAction` 이벤트를 발생시킵니다.
    *   백엔드는 `queryId`를 사용하여 저장된 SQL 템플릿을 찾습니다.
    *   `LIMIT`/`OFFSET`을 사용하여 다음 페이지를 실행합니다.
    *   응답은 `dataModelUpdate`만 포함합니다 (UI 재구축 없음).

3.  **검색 (LLM 없음)**

    사용자가 "이름(Name)" 열에서 "Accept"를 검색합니다.

    *   UI는 `queryId`, `searchTerm`, `searchColumn`을 포함하는 `userAction` 이벤트를 발생시킵니다.
    *   백엔드는 기본 SQL 위에 필터를 적용하고 필터링된 결과의 1페이지를 반환합니다.
    *   마찬가지로, `dataModelUpdate`만 반환합니다.

위 워크플로를 작동시키기 위해, 저는 프런트엔드와 백엔드 에이전트에 사용자 정의 A2UI 데이터테이블 컴포넌트(Custom A2UI DataTable component)를 구축했습니다.

*   **사용자 정의 데이터테이블 컴포넌트(클라이언트 측)**: 동적 열(dynamic columns), 페이지 제어(paging controls) 및 검색(search)을 렌더링하고, 사용자가 상호작용할 때 액션을 발생시킵니다.
*   **간단한 JSON 템플릿(에이전트 측)**: 에이전트 출력을 작고 일관되게 유지하여 UI를 한 번 생성하고 재사용할 수 있도록 합니다.

### 메시지 유형: A2UI 계약

에이전트 측의 간단한 JSON 템플릿을 통해 에이전트는 테이블 UI를 한 번 생성합니다. 그 후, 페이지 매김과 검색은 단지 액션이 들어오고 새로운 행이 나가는 것일 뿐이며, UI 재구축이나 반복적인 LLM 호출은 필요하지 않습니다.

**`surfaceUpdate` UI 구조 (한 번 전송):**

이는 사용자 정의 데이터테이블 위젯(Custom DataTable widget)을 생성하고 데이터 모델 경로에 바인딩합니다.

```json
{
  "surfaceUpdate": {
    "surfaceId": "sql-results",
    "components": [
      {
        "id": "root",
        "component": {
          "DataTable": {
            "title": {"path": "/title"},
            "rows": {"path": "/rows"},
            "currentPage": {"path": "/currentPage"},
            "totalPages": {"path": "/totalPages"},
            "queryId": {"path": "/queryId"}
          }
        }
      }
    ]
  }
}
```

**`dataModelUpdate` 데이터 전용 (페이지/검색당 전송):**

이는 UI를 변경하지 않고 행과 메타데이터(페이지 번호, 총계 등)를 업데이트합니다.

```json
{
  "dataModelUpdate": {
    "surfaceId": "sql-results",
    "contents": [
      {"key": "queryId", "valueString": "abc12345"},
      {"key": "currentPage", "valueNumber": 2},
      {"key": "totalPages", "valueNumber": 14},
      {"key": "totalCount", "valueNumber": 275},
      {"key": "rows", "valueArray": [
        {"ArtistId": 21, "Name": "Caetano Veloso"},
        {"ArtistId": 22, "Name": "Chico Buarque"}
      ]}
    ]
  }
}
```

**`userAction` 클라이언트 상호작용 이벤트:**

이는 사용자가 "다음(Next)"/"이전(Previous)"을 클릭하거나 검색을 실행할 때 데이터테이블(DataTable)이 다시 보내는 내용입니다.

```json
{
  "userAction": {
    "surfaceId": "sql-results",
    "name": "page_change",
    "context": {
      "direction": "next",
      "queryId": "abc12345",
      "currentPage": 1
    }
  }
}
```

스크린샷에서 볼 수 있듯이, 데이터테이블(DataTable)은 LLM을 다시 호출하지 않고도 동적 열(dynamic columns), 서버 측 검색(server-side search) 및 페이지 매김(pagination)을 지원합니다.

### 데모 살펴보기

다음으로, 전체 경험을 보여주는 간단한 데모를 살펴보겠습니다. "모든 아티스트를 보여줘(Show all artists)"라고 입력한 다음, 결과를 페이지별로 넘겨보고 서버 측 검색을 실행하는 과정입니다.

### 결론

SQL 에이전트는 텍스트 질문에서 쿼리를 생성하는 데 뛰어나지만, 결과 집합이 커지면 어려움을 겪습니다. 사용자 정의 A2UI 데이터테이블 컴포넌트(Custom A2UI DataTable component)를 추가함으로써, 우리는 이 경험을 실제 SQL 익스플로러(SQL explorer)에 더 가까운 형태로 만들 수 있습니다. 즉, 안정적인 테이블 렌더링, 빠른 페이지 매김(pagination), 그리고 서버 측 검색(server-side search)을 제공합니다.
