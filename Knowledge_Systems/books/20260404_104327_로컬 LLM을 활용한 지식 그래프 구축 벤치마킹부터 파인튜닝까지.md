# 로컬 LLM을 활용한 지식 그래프 구축: 벤치마킹부터 파인튜닝까지

실제 운영 환경의 그래프 RAG(GraphRAG) 파이프라인에서 추출(Extraction) 단계는 종종 소리 없이 실패하곤 합니다.

필자는 한동안 그래프 RAG 시스템을 운영해 왔습니다. 이 시스템의 핵심 아이디어는 매우 매력적입니다. 텍스트 청크(Chunk)를 벡터 저장소(Vector store)에 밀어 넣고 의미적 유사성만으로 적절한 맥락을 찾길 바라는 대신, 엔티티(Entity)와 그들 간의 관계를 구조화한 네트워크인 지식 그래프(Knowledge Graph)를 구축하고 쿼리 시점에 이를 검색하는 것입니다. 이 시스템은 단순히 단어들이 서로 가깝게 배치되어 있다는 것만 아는 것이 아니라, '마리 퀴리가 파리 대학교에서 근무했다', '그녀가 라듐을 발견했다', '라듐은 원소다'와 같은 구조화된 정보를 파악합니다. 이러한 구조적 이해는 여러 문서에 흩어진 정보를 연결해야 하는 복잡한 질문에 답할 때 큰 차이를 만듭니다.

하지만 지식 그래프를 쿼리하기 전에 먼저 구축해야 한다는 사실을 간과해서는 안 됩니다. 즉, 기사, 법률 문서, 의료 기록, 기술 문서와 같은 비정형 텍스트에서 'CRISPR-Cas9, 개발자, 제니퍼 다우드나'와 같은 구조화된 엔티티-관계 트리플(Triple)을 추출해야 합니다. 대부분의 그래프 RAG 가이드는 이를 GPT-4 호출 한 번으로 해결하지만, 외부로 유출할 수 없는 데이터를 다루는 환경에서는 로컬 모델(Local models)이 필요합니다. 그리고 로컬 모델은 구조화된 데이터 추출에 있어 예상보다 훨씬 더 큰 어려움을 겪습니다.

우리는 파이프라인에서 이 추출 작업이 얼마나 어려운지, 그리고 이를 개선할 수 있는지 파악해야 했습니다. 이를 위해 종합적인 내부 벤치마크를 구축하고, 주요 오픈 웨이트(Open-weight) 모델들을 테스트한 뒤 파인튜닝(Fine-tuning)을 진행했습니다. 다음은 그 과정에서 얻은 교훈입니다.

## Part 1: 벤치마크(Benchmark)

### 테스트 내용
우리는 실무적인 질문에 답하고자 했습니다. "70억~90억 개의 파라미터를 가진 모델을 GPU 인프라에서 실행할 때, 텍스트에서 지식 그래프를 얼마나 신뢰성 있게 추출할 수 있는가?"

우리는 법률 도메인(계약서, 판례, 규제 준수 등)에 집중하여 50개의 샘플로 구성된 벤치마크를 구축했습니다. 각 구절에는 단순형부터 에지 케이스(Edge cases)까지 네 가지 복잡도 수준의 골드 표준(Gold-standard) 라벨이 지정되었습니다.

vLLM을 통해 NVIDIA RTX 3090(24GB) 한 장에서 다음 네 가지 모델을 테스트했습니다:
*   **Qwen 2.5 7B Instruct**
*   **Llama 3.1 8B Instruct**
*   **Mistral 7B Instruct v0.3**
*   **Gemma 2 9B IT**

각 모델은 세 가지 프롬프트 전략(단순 지시, 스키마 포함 프롬프트, 퓨샷(Few-shot) 프롬프트)으로 테스트되었습니다.

### 품질 측정 방법
추출 품질을 평가하는 것은 추출 작업만큼이나 어렵습니다. 모델이 `born_in` 대신 `was_born_in`을 출력하더라도 이는 의미적으로 정답입니다. 이를 오답 처리하면 모델의 성능이 실제보다 낮게 평가됩니다.

우리는 세 단계의 퍼지 매칭(Fuzzy matching) 파이프라인을 구축했습니다:
1.  **엔티티 매칭**: 토큰 정렬 비율(Token-sort ratio)을 사용하여 부분 일치 허용.
2.  **술어(Predicate) 매칭**: 유의어 정규화 레이어를 통해 의미가 같은 관계명(예: `birthplace`, `place_of_birth`)을 통합.
3.  **부분 문자열 매칭**: 모델이 장황한 엔티티 이름을 생성하는 경우 처리.

이러한 개선을 통해 모델의 출력을 수정하지 않고도 측정된 트리플 F1(Triple F1) 점수를 15~34% 높일 수 있었습니다.

### 주요 결과
*   **엔티티 감지는 해결되었으나 관계 추출은 미흡**: 모든 모델의 엔티티 F1은 0.78~0.91로 우수했습니다. 그러나 트리플 F1은 최고치가 0.732(Llama 3.1 퓨샷)에 불과했으며, 대부분 0.52~0.60 사이에 머물렀습니다.
*   **신뢰성과 정확도의 트레이드오프**: 모든 조건을 만족하는 완벽한 모델은 없었습니다. **Llama 3.1 8B**는 퓨샷 사용 시 가장 높은 정확도를 보였으나, 유효한 JSON을 출력하는 비율(Schema Conformance)은 20%에 불과했습니다. 반면 **Mistral 7B**는 정확도는 낮았으나 100%의 신뢰성을 보여주었습니다.
*   **퓨샷(Few-shot)의 양날의 검**: 퓨샷 프롬프트는 추출 품질을 높이지만, 프롬프트 길이가 3~5배 길어져 지연 시간(Latency)을 증가시키고 소형 모델의 출력 안정성을 해칩니다.
*   **복잡도에 따른 성능 저하**: 텍스트가 복잡해질수록 엔티티 감지는 안정적이었으나, 트리플 추출 성능은 급격히 떨어졌습니다. 모델들은 밀집된 텍스트를 개별 트리플로 분해하는 데 어려움을 겪었습니다.

## Part 2: 파인튜닝을 통한 간극 극복

우리는 신뢰성(신뢰도 100%)과 품질(정확도) 사이의 간극을 좁히기 위해 지도 학습(Supervised fine-tuning)을 선택했습니다. 파인튜닝된 모델은 퓨샷 예시 없이도 짧은 시스템 프롬프트만으로 유효한 JSON을 출력해야 합니다.

### 학습 데이터 및 설정
*   **데이터셋**: 위키피디아 텍스트와 위키데이터(Wikidata) 트리플을 정렬한 **REBEL** 데이터셋을 사용했습니다. 이를 대상 JSON 형식으로 변환하여 3,000개의 학습 샘플을 구성했습니다.
*   **학습 환경**: RTX 3090 GPU 2장에서 **QLoRA**를 사용했습니다. 구조화된 추출 작업은 복잡하므로 LoRA의 랭크(Rank)를 일반적인 수준보다 높은 64로 설정했습니다.
*   **최적화**: 사용자 턴 마스킹(User-turn masking)을 통해 모델이 JSON 추출 결과물에만 집중하여 학습하도록 했습니다.

### 학습 결과
파인튜닝된 **Qwen3-8B** 모델은 모든 샘플에서 **100%의 JSON 유효성 및 스키마 준수율**을 기록했습니다. 파이프라인에서의 파싱 실패와 재시도 로직이 완전히 사라진 것입니다.

하지만 트리플 F1 점수는 0.211로 낮게 나타났는데, 이는 두 가지 흥미로운 이유 때문이었습니다:
1.  **도메인 전이(Domain Transfer) 문제**: 모델이 학습한 REBEL 데이터셋은 백과사전식 관계명(예: `born_in`)을 사용한 반면, 벤치마크는 법률 도메인의 정규화된 용어를 기대했기 때문입니다.
2.  **지식 베이스 완결 vs 텍스트 기반 추출**: REBEL은 문장에 직접 명시되지 않았더라도 지식 베이스에 존재하는 사실을 학습시키는 반면, 로컬 모델은 텍스트에 명시된 관계를 그대로 추출하려는 경향을 보였습니다. 그래프 RAG의 목적에는 후자의 동작이 오히려 더 유용할 수 있습니다.

## Part 3: 실무적인 시사점

1.  **스키마 포함 프롬프트(Schema-in-prompt)로 시작하기**: 대부분의 모델에서 가장 안전한 기본 전략입니다. 신뢰성 대비 품질 비율이 가장 좋습니다.
2.  **추출 실패에 대비한 예산 책정**: 가장 신뢰할 수 있는 모델도 품질을 일부 희생합니다. 재시도 로직이나 휴먼 인 더 루프(Human-in-the-loop) 검토 프로세스를 구축하십시오.
3.  **평가 시스템에 투자하기**: 엄격한 문자열 일치 방식은 모델의 성능을 과소평가하게 만듭니다. 유의어를 인식하는 술어 매칭과 퍼지 엔티티 비교 시스템을 구축하십시오.
4.  **파인튜닝은 신뢰성 문제를 해결함**: 파싱 오류나 잘못된 형식의 출력이 문제라면 수천 개의 예시로 파인튜닝하는 것이 가장 효과적입니다. QLoRA를 사용하면 저비용으로도 엔지니어링 리소스를 크게 절약할 수 있습니다.
5.  **도메인 간극 주의**: 일반적인 데이터셋(REBEL 등)으로 학습하면 구조적 패턴은 익히지만, 특정 도메인에서 필요한 추출 철학을 놓칠 수 있습니다. 50~100개의 도메인 전용 샘플을 섞어 학습시키는 것이 좋습니다.
6.  **로컬 모델의 생산성**: 오픈 웨이트 모델의 트리플 F1이 0.55~0.65 수준이라는 것은 생성된 정보의 절반 이상이 정확하다는 의미입니다. 이는 적절한 필터링을 거치면 외부 API 없이도 충분히 운영 가능한 수준입니다.

엔티티 인식과 관계 추출 사이의 간극은 로컬 그래프 RAG의 주요 병목 구간이지만, 이는 모델 크기의 한계라기보다는 엔지니어링으로 해결 가능한 문제입니다. 파인튜닝 결과는 구조적 신뢰성이 이미 확보되었음을 보여주며, 남은 과제는 모델에게 특정 도메인의 추출 철학을 가르치는 것입니다.
