## 고전하던 Graph RAG 시스템을 기업 코드 마이그레이션 플랫폼의 중추로 만든 고정 엔터티 아키텍처(Fixed Entity Architecture)의 힘

![이미지: 아키텍처 다이어그램](https://miro.medium.com/v2/resize:fit:700/1*y0xS_F9yG12h00b-jP-jUg.png)
엔터 키를 누르거나 클릭하여 전체 크기로 이미지를 확인하세요.

오늘날 Graph RAG 시스템을 구축하는 대부분의 팀은 동일한 접근 방식을 따릅니다. 즉, 문서들을 LLM에 입력하여 엔터티(entity)와 관계(relationship)를 추출하게 하고, 이를 그래프 데이터베이스(graph database)에 로드한 다음, 검색(retrieval) 성능이 충분하기를 바랍니다.

저도 그랬습니다. 하지만 몇 달 동안 결과를 지켜본 결과, 기대에 미치지 못했습니다.

저는 마이크로서비스(microservices) 코드베이스 전체를 프로그래밍 언어 간에 수집(ingest), 이해 및 번역하도록 설계된 기업 코드 마이그레이션 플랫폼을 구축하고 있었습니다. 이 시스템의 중심에 있는 지식 그래프는 서비스 간 호출 방식, 경계(boundary)를 넘나드는 타입(type) 흐름, 비즈니스 로직(business logic)이 연결되는 방식을 매핑하는 '두뇌' 역할을 해야 했습니다. 하지만 실상은 엉망이었습니다.

LLM이 추출한 엔터티는 노이즈(noisy)가 많았습니다. "PaymentService"가 다섯 개의 다른 노드(node)로 나타났고, 관계는 환각(hallucinated) 현상을 보였습니다. 그래프에 새 서비스를 추가할 때마다 일관성(consistency)을 유지하기 위해 모든 것을 재처리해야 했습니다. 수천 건에 달하는 LLM 호출 비용은 계속 증가했지만, 그래프 품질은 나아지지 않았습니다.

그러던 중 저의 전체 접근 방식을 바꾼 일련의 글들을 발견했습니다.

## 사고방식을 재구성한 글들

2024년 말, 이리나 아담칙(Irina Adamchic) 박사는 미디엄(Medium)에 고정 엔터티 아키텍처(Fixed Entity Architecture, FEA)라고 부르는 일련의 글을 발표했습니다. 이는 그래프 구축 과정에서 LLM 사용을 의도적으로 피하는, RAG용 지식 그래프 구축을 위한 3계층 접근 방식입니다.

핵심 아이디어는 놀랍도록 간단합니다. LLM이 처음부터 온톨로지(ontology)를 발견하게 하는 대신, 직접 정의하는 것입니다.

그녀의 세 가지 핵심 글은 다음과 같습니다.

*   **"Three-Layer Fixed Entity Architecture for Efficient RAG on Graphs" (2024년 9월)** — 3계층 개념을 소개하는 아키텍처 청사진(architectural blueprint)
*   **"RAG on Graph using Fixed Entity Architecture: make your retrieval work for you" (2024년 9월)** — 이들 그래프를 구축하고 쿼리(query)하는 실용적인 방법론(practical methodology)
*   **"Build your hybrid-Graph for RAG & GraphRAG applications using the power of NLP" (2025년 3월)** — 온톨로지 정의가 어려운 경우를 위해 온톨로지 계층을 제외한 진화된(evolution) 방법

세 편 모두 액센츄어(Accenture)의 첨단 AI 센터(Center of Advanced AI)에 소속된 아담칙 박사가 집필했습니다. 이 글들을 꼭 읽어보시길 강력히 추천합니다. API 예산(budget)을 소진하지 않고 Graph RAG를 프로덕션(production) 환경에서 작동시키는 방법에 대한 제가 본 가장 실용적인 사고방식 중 일부를 담고 있습니다.

중심적인 주장은 깊이 공감되었습니다. 이미 자신의 도메인(domain)을 이해하고 있다면, 왜 LLM에게 비용을 지불하여 그것을 제대로 파악하지 못하게 하는가?

## 세 가지 계층 (그리고 작동하는 이유)

FEA는 지식 그래프를 세 가지의 개별적인 계층으로 구성합니다.

### Layer 1: 고정 엔터티 온톨로지(Fixed Entity Ontology) (이른바 '물고기 뼈대')

이 계층은 다른 모든 것이 작동하게 만드는 핵심입니다. LLM이 아닌 도메인 전문가가 문제 영역의 핵심 어휘를 나타내는 개념 집합을 정의합니다. 각 개념은 풍부한 텍스트 설명과 임베딩 벡터(embedding vector)를 가지며, 안정적이고, 큐레이션(curated)되며, 사람이 검증합니다.

저의 경우, 마이크로서비스 코드베이스를 위해 15가지 도메인 개념을 정의했습니다. 예를 들어 '결제 및 지급 흐름(Checkout & Payment Flow)'과 '장바구니 관리(Shopping Cart Management)' 같은 비즈니스 로직 개념, 'gRPC 통신(gRPC Communication)' 및 '카프카 이벤트 스트리밍(Kafka Event Streaming)'과 같은 통신 패턴, 그리고 '캐싱(Caching)' 및 '데이터 영속성(Data Persistence)'과 같은 인프라(infrastructure) 관련 개념이 포함됩니다.

### Layer 2: 문서 계층(Document Layer) (코드 덩어리 및 심볼)

여기에 실제 콘텐츠가 존재합니다. 제 경우, 파싱(parsed)된 코드 함수, 메서드(method), 스트럭트(struct) 및 임베딩을 포함한 코드 덩어리들이 해당합니다. 이 계층은 제 시스템에 이미 존재했습니다. FEA의 핵심 통찰력은 이러한 문서들이 LLM 호출이 아닌 코사인 유사도(cosine similarity)를 통해 Layer 1에 연결된다는 점입니다. 순수한 수학입니다. 환각 현상도 없고, API 비용도 발생하지 않습니다.

### Layer 3: NLP 추출 엔터티(NLP-Extracted Entities)

경량 NLP 도구(spaCy, 정규 표현식 패턴(regex patterns))를 사용하여 콘텐츠에서 명명된 엔터티(named entities)를 추출합니다. 'Redis' 및 'Kafka'와 같은 기술 이름, 'Stripe'와 같은 조직 이름, 'CartService'와 같은 API 이름 등이 이에 해당합니다. 이러한 엔터티는 단 한 번의 LLM 호출 없이 그래프를 더욱 풍부하게 만듭니다.

세 계층을 연결하는 접착제는 무엇일까요? 내적(dot product)을 통해 계산되는 코사인 유사도입니다. 이는 밀리초(milliseconds) 단위로 소요되며, 비용이 들지 않고, 결정론적(deterministic)이며 재현 가능한(reproducible) 결과를 생성합니다.

## FEA가 저에게 해결해준 문제 (마이크로소프트 GraphRAG는 해결할 수 없었던 문제)

FEA를 채택하기 전에 마이크로소프트(Microsoft)의 GraphRAG 접근 방식을 평가했었습니다. 강력했지만, 제 사용 사례(use case)에는 적합하지 않은 특징들이 있었습니다.

|                 | **Microsoft GraphRAG**                       | **FEA**                                |
| :-------------- | :------------------------------------------- | :------------------------------------- |
| **비용**        | 그래프 구축에 수천 건의 LLM 호출             | 수학 + NLP만 사용 (LLM은 최종 답변에만) |
| **엔터티 중복** | "PaymentService" vs "Payment_Service" vs "payment service" | 하나의 큐레이션된 개념, 설계상 중복 없음 |
| **새 데이터 추가** | 재처리 또는 복잡한 증분 업데이트 필요          | 유사성을 통해 즉시 연결 가능 — 재구축 필요 없음 |
| **도메인 정확도** | 처음부터 발견하며, 종종 도메인 미묘한 차이를 놓침 | 전문가 지식 기반, 현실에 근거 |
| **그래프 품질** | 노이즈가 많고, 엔터티 해결(entity resolution) 후처리 필요 | 깔끔하고 통제되며, 사람이 검증함 |

기업 마이크로서비스와 같이 잘 정의된 도메인의 경우, FEA는 단순히 더 저렴할 뿐만 아니라 더 나은 방식이었습니다. 그래프는 더 깔끔했고, 검색은 더 정확했으며, 시스템 유지보수(maintenance)는 훨씬 더 단순해졌습니다.

## HyDE 혁신: 언어와 코드 사이의 의미론적 격차 해소

여기서 제 구현 방식이 아담칙 박사의 원래 작업과 달라진 부분이 있는데, 커뮤니티와 공유할 가치가 있다고 생각합니다.

FEA는 원래 PDF, 웹 페이지, 보고서와 같은 자연어(natural language) 문서를 위해 설계되었습니다. 하지만 제 Layer 2는 소스 코드(source code)였고, 이것이 근본적인 문제를 야기했습니다.

고정 온톨로지(Layer 1)를 코드 임베딩(Layer 2)에 처음 연결했을 때, 평균 코사인 유사도는 약 0.09였습니다. 이는 본질적으로 무작위(random)적인 수준이었고, 연결은 쓸모가 없었습니다.

그 이유는 개념 설명은 자연어("결제 처리를 포함한 엔드투엔드(end-to-end) 결제 오케스트레이션(orchestration)...")이지만, 코드 덩어리는 말 그대로 코드(func (s *CheckoutService) PlaceOrder(ctx context.Context, req *pb.PlaceOrderRequest))이기 때문입니다. 이들은 임베딩 공간(embedding space)에서 완전히 다른 영역을 차지합니다.

저는 다리(bridge)가 필요했고, HyDE(Hypothetical Document Embeddings, 가설적 문서 임베딩) (Gao et al., 2022)에서 그 답을 찾았습니다.

아이디어는 이렇습니다. 각 도메인 개념에 대해 LLM을 사용하여 해당 개념을 구현할 가상의 코드 스니펫(code snippet)을 생성합니다. 그런 다음, 자연어 설명이 아닌 생성된 코드를 임베딩하고, 이 임베딩을 유사도 매칭(similarity matching)에 사용합니다.

과정은 다음과 같습니다.

```
도메인 개념 (자연어 설명)
       |
       v
   LLM이 가상의 Go 함수 생성
       |
       v
   생성된 코드 임베딩 (실제 코드와 동일한 임베더(embedder) 사용)
       |
       v
   HyDE 임베딩 ('코드 공간'에 존재)
       |
       v
   실제 코드 덩어리와의 코사인 유사도: 평균 ~0.30 이상
```

결과적으로 평균 유사도는 0.09에서 0.30 이상으로 3배 이상 향상되었으며, 연결은 무의미한 노이즈에서 진정으로 유용한 의미론적 연결(semantic links)로 바뀌었습니다. 0.35~0.45의 임계값(threshold)으로 시스템은 이제 코드와 개념 사이에 정밀하고 고품질의 `RELATES_TO` 엣지(edge)를 생성합니다.

이 접근 방식의 장점은 HyDE가 일회성 비용이라는 것입니다. 개념당 한 번만 가상의 코드를 생성하고(15개 개념 온톨로지에 대해 15번의 LLM 호출), 그 이후에는 모든 서비스의 모든 코드 덩어리에 대한 모든 연결이 순수한 수학으로 이루어집니다. 문서-개념 쌍마다 LLM을 호출하는 것에 비해 경제성이 매우 뛰어납니다.

이 내용은 아담칙 박사의 원본 글에는 없으며, 코드 도메인 Graph RAG를 위해 제가 특별히 개발한 각색(adaptation)입니다. 비텍스트 콘텐츠(non-textual content) (코드, 구조화된 데이터(structured data), 로그(log))를 다루고 있다면, HyDE가 귀하의 도메인에서 FEA를 작동시키는 데 필요한 빠진 조각(missing piece)이 될 수 있습니다.

## 슈퍼 노드(Super-Node) 함정 (뼈아픈 교훈)

매우 중요하다고 입증된 하나의 설계 결정은 온톨로지에 무엇을 포함시키는지만큼 무엇을 제외시키는가도 중요하다는 것입니다.

저의 첫 온톨로지 초안에는 "오류 처리(Error Handling)", "관측 가능성(Observability)", "로깅(Logging)"과 같은 개념이 포함되어 있었습니다. 이들은 중요한 도메인 개념처럼 보였습니다. 실제로 중요했지만, 동시에 전체 코드의 90% 이상에 존재했습니다. 모든 함수가 오류를 처리하고, 모든 서비스가 트레이싱(tracing)을 가지며, 모든 모듈이 로그를 기록합니다.

이러한 개념들은 "슈퍼 노드(super-nodes)"가 되었습니다. 거의 모든 것에 연결되어 변별적 가치(discriminative value)를 전혀 제공하지 못했습니다. "결제 처리"에 대한 쿼리가 코드베이스의 90%에 연결된 개념을 통과할 때, 검색은 노이즈로 붕괴됩니다.

해결책은 '외과적 제거(surgical removal)'였습니다. 저는 온톨로지를 18개 개념에서 15개로 줄이고, 코드의 50% 이상과 일치하는 모든 것을 제거했습니다. 남은 15개 개념(비즈니스 로직 9개, 통신 3개, 인프라 3개)은 실제로 검색 품질을 향상시키는 날카롭고 변별적인 연결을 제공했습니다.

아담칙 박사는 자신의 글에서 고품질 개념 설명의 중요성을 언급합니다. 저는 여기에 다음의 귀결(corollary)을 추가하고 싶습니다. 온톨로지의 품질은 포함하는 것만큼이나 제외하는 것에 의해 정의됩니다.

## 실제 아키텍처

최종 시스템은 12단계의 수집 파이프라인(ingestion pipeline)으로 작동합니다. 1~9단계는 표준 코드 파싱(parsing), AST(Abstract Syntax Tree) 추출, 임베딩 생성 및 그래프 구축을 처리합니다. 10~12단계는 FEA 계층입니다.

*   **10단계:** 임베딩을 Neo4j 벡터 인덱스(vector indexes)에 동기화(sync)
*   **11단계:** 구조적 요소(structural elements)를 상위 서비스에 연결
*   **12단계:** FEA 통합 — 코드를 코사인 유사도를 통해 도메인 개념에 연결하고, NLP 엔터티를 추출하며, 의미론적 계층(semantic layer) 생성

모든 작업은 관측 가능성(observability)을 위해 Langfuse를 통해 추적됩니다. 모든 연결은 결정론적이며 재현 가능합니다. 그리고 전체 FEA 계층(15개 도메인 개념과 수천 개의 코드 심볼(symbol)에 걸쳐)은 수집 파이프라인에 약 2초를 추가할 뿐입니다. LLM 호출도 없고, API 비용도 없으며, 환각된 관계도 없습니다.

검색 계층은 상호 순위 융합(Reciprocal Rank Fusion, RRF)을 통해 세 가지 인덱스(index)를 결합합니다.

*   코드 임베딩에 대한 벡터 검색(vector search)
*   코드 콘텐츠 및 설명에 대한 전체 텍스트 검색(full-text search)
*   FEA 온톨로지 계층을 통한 개념 기반 검색(concept-guided search)

이 하이브리드(hybrid) 접근 방식은 "결제 처리는 어떻게 작동하는가"와 같은 쿼리(query)가 유사한 임베딩을 가진 코드만 찾는 것이 아니라, 어휘적 유사성(lexical similarity)과 관계없이 "결제 및 지급 흐름" 개념과 구조적으로 관련된 모든 코드를 찾기 위해 온톨로지를 탐색한다는 것을 의미합니다.

## 핵심 요약

이 시스템을 6개월 동안 구축하고, 반복하며, 때로는 재구축한 후에 프로덕션 환경에서 Graph RAG를 고려하는 모든 사람에게 다음과 같이 말하고 싶습니다.

1.  **도메인을 안다면, 온톨로지를 직접 정의하십시오.** LLM에 외주를 주지 마십시오. LLM은 텍스트 생성에는 탁월하지만, 일관되고 중복 없는 지식 구조(knowledge structures)를 구축하는 데는 신뢰할 수 없습니다. 도메인 전문 지식(domain expertise)이 있다면, 그것을 직접 인코딩(encode)하십시오.

2.  **수학은 과소평가되어 있습니다.** 코사인 유사도, 내적, 임계값 기반 엣지 생성은 지루할 수 있습니다. 하지만 빠르고, 결정론적이며, 재현 가능하고, 무료입니다. 제 시스템에서 가장 신뢰할 수 있는 부분은 AI를 전혀 사용하지 않는 부분입니다.

3.  **임베딩 공간의 격차는 실제하며 해결 가능합니다.** 콘텐츠가 자연어가 아닌 경우(코드, 구조화된 데이터, 로그, 구성 파일(configs)), 자연어 설명과 콘텐츠 임베딩 간의 직접적인 유사성 매칭은 실패할 것입니다. 온톨로지를 콘텐츠의 임베딩 공간으로 투영(project)하는 HyDE 또는 유사한 기술은 선택 사항이 아니라 필수적입니다.

4.  **적극적으로 큐레이션하십시오.** 날카롭고 변별력 있는 15개 개념 온톨로지는 슈퍼 노드로 가득 찬 50개 개념 온톨로지보다 더 나은 성능을 발휘할 것입니다. 적을수록 좋습니다. 모든 것에 일치하는 것은 무엇이든 제외하십시오.

5.  **하이브리드 검색은 선택 사항이 아닙니다.** 단일 검색 방법만으로는 충분하지 않습니다. 벡터 검색은 구조적 관계를 놓치고, 전체 텍스트 검색은 의미론적 유사성을 놓칩니다. 개념 기반 검색은 정확한 일치를 놓칩니다. 이들을 RRF와 결합하여 각 방법이 다른 방법의 사각지대(blind spots)를 보완하도록 하십시오.

## 참고 자료 및 추가 자료

이 작업은 이리나 아담칙 박사(액센츄어, 첨단 AI 센터)의 기초 연구 없이는 존재할 수 없었습니다. 그녀의 고정 엔터티 아키텍처(Fixed Entity Architecture)에 대한 글들은 Graph RAG 시스템을 구축하는 모든 사람에게 필수적인 읽을거리입니다.

*   "Three-Layer Fixed Entity Architecture for Efficient RAG on Graphs" — Medium, 2024년 9월
*   "RAG on Graph using Fixed Entity Architecture: make your retrieval work for you" — Medium, 2024년 9월
*   "Build your hybrid-Graph for RAG & GraphRAG applications using the power of NLP" — Medium, 2025년 3월

그녀는 또한 Neo4j Live (2025년 1월) 및 NLP 서밋(Summit)에서도 이 작업을 발표했는데, 여기서 FEA는 생식 의학 분야의 임상 문서에 적용되었습니다. 이는 아키텍처의 일반화 가능성(generalizability)을 입증하는 매우 다른 도메인입니다.

HyDE 기술은 다음 논문에서 가져왔습니다.

*   Gao et al., "Precise Zero-Shot Dense Retrieval without Relevance Labels" (arXiv:2212.10496, 2022)

저는 지식 그래프, Graph RAG, 그리고 AI 시스템 엔지니어링에 대한 현장 경험을 바탕으로 글을 씁니다. 유사한 시스템을 구축 중이거나 귀하의 도메인에 FEA를 적용해 본 경험이 있다면, 기꺼이 이야기를 나누고 싶습니다.
