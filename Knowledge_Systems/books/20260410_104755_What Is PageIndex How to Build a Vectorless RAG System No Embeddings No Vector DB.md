What Is PageIndex? How to Build a Vectorless RAG System (No Embeddings, No Vector DB)

PageIndex는 임베딩(embeddings), 청킹(chunking) 또는 벡터 데이터베이스(vector database)를 사용하지 않고 긴 문서에서 답변을 검색하는 벡터리스(vectorless), 추론 기반의 검색 증강 생성(Retrieval-Augmented Generation, RAG) 방식입니다.

PageIndex는 의미론적 유사성 검색(semantic similarity search)에 의존하는 대신, 문서에서 계층적 목차(Table of Contents, TOC) 트리를 구축하고 대규모 언어 모델(Large Language Model, LLM)을 사용하여 해당 구조를 추론합니다. 모델은 먼저 문서의 계층 구조를 사용하여 가장 관련성이 높은 섹션을 식별한 다음, 해당 섹션으로 이동하여 정확하고 출처가 명시된 답변을 생성합니다.

전통적인 RAG는 유사성을 통해 정보를 검색합니다.
PageIndex는 구조에 대한 추론을 통해 정보를 검색합니다.

이러한 특징은 재무 보고서, 법률 계약, 규제 서류, 정책 문서, 학술 논문과 같은 구조화된 장문 콘텐츠에 특히 효과적입니다.

대부분의 검색 증강 생성(RAG) 시스템은 임베딩과 벡터 데이터베이스에 의존합니다. 이들은 문서를 청크(chunks)로 분할하고, 이를 벡터로 변환하며, 코사인 유사성(cosine similarity)을 사용하여 답변을 검색합니다.

그러나 유사성은 추론이 아닙니다.

PageIndex는 의미 검색(semantic search)을 수행하는 대신 문서 구조에 대한 추론을 통해 정보를 검색하는 벡터리스(vectorless) RAG 아키텍처입니다. 문서를 평평한 텍스트 덩어리로 취급하기보다는, 목차가 있는 교과서처럼 구조화된 계층으로 취급합니다.

이 원리가 어떻게 직관적으로 작동하는지 이해하기 위해 고전 영화 <숄레이(Sholay)>의 예를 들어 살펴보겠습니다.

### 핵심 아이디어: 검색 전 구조화

<숄레이>의 스크립트나 상세한 줄거리를 PageIndex에 입력하면, 임의의 500단어 청크(chunk)로 문서를 분할하지 않습니다.

대신, 이야기의 구조적 트리를 구축합니다.

*   (PageIndex 방식) 문서 → 계층적 인덱스 → 추론 기반 검색 → 답변
*   (기존 방식) 문서 → 청크 → 임베딩 → 벡터 DB → 유사성 검색 → 답변

### 1단계: 트리 생성 (인덱싱 단계)

첫 번째 단계는 구조적 인덱싱(structural indexing)입니다.

**1. 구조 탐지**

LLM은 스크립트를 읽고 다음과 같은 자연스러운 경계를 탐지합니다.

*   장면 제목("SCENE 1 — THE TRAIN ROBBERY")
*   등장인물 소개
*   막(Act) 구분
*   주요 서사 전환

고정된 청크 크기에 의존하지 않고, 서사 구조에 의존합니다.

(이 부분은 시각적 설명이므로, 텍스트로 풀어 설명합니다.)

*   어두운 루트(root) 노드는 전체 문서를 나타냅니다.
*   파란색 노드는 주요 이야기 세그먼트를 나타냅니다.
*   빨간색 노드는 가바르(Gabbar) 관련 아크(arc)를 나타냅니다.
*   보라색 노드는 중요한 사건 노드를 나타냅니다.
*   황금색 노드는 특정 사실적 사건을 나타냅니다.

**2. 계층적 매핑(Hierarchical Mapping)**

PageIndex는 트리를 구축합니다.

만약 루트(root) 노드가 <숄레이>라면, 첫 번째 레벨의 브랜치는 다음과 같을 수 있습니다.

*   프롤로그
*   비루(Veeru)와 자이(Jai)의 영입
*   람가르(Ramgarh)에서의 삶
*   가바르의 공포 통치
*   최후의 대결

각 브랜치는 자식 노드(하위 섹션)를 포함할 수 있습니다.

예를 들어:

가바르의 은신처 (Gabbar’s Den)
요약: "이 섹션은 가바르 싱(Gabbar Singh)의 소개, 'Kitne aadmi the' 대사, 그리고 그의 부하들에 대한 처벌을 다룹니다."

각 노드는 다음을 포함합니다.

*   제목 (title)
*   노드 ID (nodeId)
*   요약 (summary)
*   자식 노드 (child nodes)

여기서 핵심은 요약입니다. LLM은 해당 섹션에서 어떤 내용이 발생하는지 간결하고 의미론적인 설명을 작성합니다. 이 요약은 나중에 검색 신호가 됩니다.

### 2단계: 질의 단계

이제 사용자가 다음과 같이 질문한다고 가정해 봅시다.

"타쿠르(Thakur)는 왜 팔을 잃었나요?"

다음과 같은 일은 발생하지 않습니다.

*   전체 스크립트가 즉시 전송되지 않습니다.
*   임베딩이 생성되지 않습니다.
*   벡터 유사성 검색이 수행되지 않습니다.

대신 LLM은 다음을 수신합니다.

*   사용자의 질문
*   계층적 맵(JSON 트리)
*   각 노드의 요약

전체 스크립트가 아닌, 오직 구조만 받습니다.

#### LLM이 답변을 찾는 방법 (수학이 아닌 추론)

**A단계: 구조적 검색**

LLM은 트리를 읽습니다.

LLM은 다음과 같은 노드를 확인합니다.

*   "타쿠르 가족의 학살 (The Massacre of Thakur’s Family)"
*   "가바르의 복수 (Gabbar’s Revenge)"
*   "람가르에서의 삶 (Life in Ramgarh)"

요약 내용을 바탕으로 LLM은 다음과 같이 추론합니다.

답변은 가바르와 타쿠르의 부상과 관련된 섹션에 존재할 가능성이 높다.

이것은 논리적 추론이지, 벡터 유사성이 아닙니다.

**B단계: 심층 탐색**

PageIndex는 해당 특정 노드에 해당하는 원본 텍스트(raw text)만 검색합니다.

50페이지를 스캔하는 대신, 2~3개의 집중된 섹션만 검색합니다.

**C단계: 최종 답변**

이제 LLM은 그 작고 매우 관련성 높은 스니펫(snippet)을 읽고 다음과 같이 답변합니다.

"타쿠르가 팔을 잃은 것은 수년 전 타쿠르가 자신을 체포한 것에 대한 복수로 가바르 싱이 그의 팔을 잘랐기 때문입니다."

또한 출처를 인용할 수 있습니다.

(노드 ID: massacre-thakur-family)

이 검색은 설명 가능합니다.

### 전통적인 RAG보다 PageIndex가 나은 이유 (구조화된 문서의 경우)

전통적인 벡터 RAG 시스템에서 "타쿠르의 팔"을 검색하면, 시스템은 다음을 검색할 수 있습니다.

*   자이와 비루가 팔을 사용하는 격투 장면
*   유사한 어휘를 포함하는 대화
*   "손" 또는 "부상"에 대한 관련 없는 언급

이는 벡터 검색이 서사적 관련성이 아닌 의미론적 근접성을 통해 검색하기 때문입니다. 이를 "분위기 매칭(vibe matching)"이라고 설명할 수 있습니다.

PageIndex는 학살 장면의 요약에서 "이 섹션은 가바르가 타쿠르의 가족을 공격하고 그의 팔을 절단하는 방법을 설명합니다."라고 명시적으로 언급하고 있기 때문에 이 문제를 피합니다.

LLM은 추측하지 않습니다. 탐색합니다.

### PageIndex가 작동하는 이유

PageIndex는 두 가지 인지 작업을 분리하기 때문에 작동합니다.

*   **탐색(Navigation)** — 답변이 어디에 존재해야 하는지 결정합니다.
*   **추출(Extraction)** — 해당 섹션만 읽고 답변을 생성합니다.

이는 인간이 독서하는 방식과 유사합니다.

소설에서 어떤 일이 왜 일어났는지 알고 싶을 때, 무작위로 모든 페이지를 훑어보지 않습니다.
관련 사건이 발생한 장으로 이동합니다.
PageIndex는 LLM이 같은 방식으로 행동하도록 만듭니다.

### 이 접근 방식이 빛을 발할 때

이 아키텍처는 다음 분야에 특히 효과적입니다.

*   재무 보고서
*   법률 문서
*   정책 보고서
*   규제 서류
*   학술 연구
*   긴 서사 콘텐츠

표면적 유사성보다 구조가 더 중요한 모든 곳에 적용됩니다.

### 더 큰 통찰

전통적인 RAG는 다음을 가정합니다.

*   관련성 = 의미론적 유사성 (semantic similarity)

PageIndex는 다음을 가정합니다.

*   관련성 = 구조적 추론 (structural reasoning)

이러한 차이는 작아 보일 수 있지만, 길고 계층적인 문서에서는 엄청난 영향을 미칩니다.

더 나은 검색 엔진을 구축하는 대신, PageIndex는 탐색 지도를 구축하고 LLM이 읽기 전에 생각하도록 만듭니다.
