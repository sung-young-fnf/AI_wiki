저는 그 좌절감을 생생하게 기억합니다. 강력한 70B(700억) 매개변수 모델을 다운로드하여 Q4_K_M으로 깔끔하게 양자화하고 추론 서버(inference server)를 가동했습니다. 첫 번째 응답이 나왔는데… 거의 기어가는 수준이었습니다. 초당 약 8토큰이었습니다. 텍스트가 생성되는 것을 보는 것은 페인트가 마르기를 기다리는 것만큼 지루했습니다. 제 RTX 3090이 더 나은 성능을 낼 수 있다는 것을 알고 있었지만, 두 번째 GPU에 수천 달러를 투자하여 채팅 응답 속도를 빠르게 만들 여유는 없었습니다.

그래서 저는 깊이 파고들었습니다. 양자화 조정, 배치 크기(batch size) 조절, 스레드 개수, 오프로딩(offloading) 전략 등 제가 찾을 수 있는 모든 최적화 기법을 몇 주 동안 테스트했습니다. 그리고 그 과정에서 로컬 LLM 경험을 완전히 바꿔놓은 두 가지 기법을 발견했습니다. 바로 투기적 디코딩(speculative decoding)과 멀티 토큰 예측(multi-token prediction)입니다.

오늘날, 동일한 RTX 3090에서 동일한 70B 모델이 초당 160토큰 이상으로 실행됩니다. 오타가 아닙니다. 새 하드웨어에 단 한 푼도 쓰지 않고 말 그대로 두 배 — 사실 두 배 이상 — 추론 속도(inference speed)를 향상시켰습니다.

제가 이를 단계별로 정확히 어떻게 달성했는지 설명해 드리겠습니다.

## 로컬 LLM이 느리게 실행되는 이유 이해하기

해결책을 자세히 알아보기 전에 문제점을 이해해야 합니다. 괜찮은 GPU를 가지고 있더라도 대규모 언어 모델이 일반 소비자용 하드웨어에서 왜 그렇게 느리게 실행될까요?

병목 현상은 계산 능력이 아니라 **메모리 대역폭(memory bandwidth)**입니다.

<img src="https://via.placeholder.com/800x400?title=Standard+Autoregressive+Generation" alt="Standard autoregressive generation processes one token at a time, requiring a full forward pass for each new token." title="표준 자기회귀 생성은 한 번에 하나의 토큰을 처리하며, 각 새 토큰마다 전체 순방향 패스를 필요로 합니다." width="800" height="400">

*표준적인 자기회귀 생성(autoregressive generation)은 한 번에 하나의 토큰을 처리하며, 각 새 토큰마다 전체 순방향 패스(forward pass)를 필요로 합니다.*

내부적으로는 다음과 같은 일이 발생합니다. LLM을 실행할 때 모델 가중치(수십억 개의 매개변수)는 GPU의 VRAM(Video RAM)에 저장됩니다. 모델이 생성하는 단일 토큰마다 해당 가중치는 VRAM에서 GPU의 연산 코어(compute core)로 읽혀야 합니다. 128GB/s의 메모리 대역폭을 가진 시스템에서 4비트 정밀도(약 35GB)의 70B 모델은 초당 약 3~4개의 토큰 읽기로 제한됩니다. 계산은 냉혹합니다.

자기회귀 생성(autoregressive generation)이라고 불리는 표준 접근 방식은 이 상황을 더욱 악화시킵니다. 모델은 한 번에 하나의 토큰을 순차적으로 생성합니다. 토큰 1에 대한 순방향 패스, 토큰 2에 대한 순방향 패스, 토큰 3에 대한 순방향 패스. 각 패스는 해당 가중치를 다시 읽어야 합니다.

이것이 우리가 모두 맞서 싸우는 근본적인 속도 제한입니다. 하지만 좋은 소식은, 이 제한을 우회할 방법이 있다는 것입니다.

## 기법 #1: 투기적 디코딩 (Speculative Decoding) - 판도를 바꾸는 기술

제가 구현한 첫 번째 기법은 투기적 디코딩(speculative decoding)이었습니다. 솔직히 처음 들으면 너무 좋아서 믿기지 않을 정도입니다.

### 투기적 디코딩 작동 방식

이를 경험 많은 관리자와 빠른 인턴 간의 협업으로 생각하면 됩니다.

*   **대규모 기본 모델(메인 LLM)**은 관리자입니다. 지식은 풍부하지만 느리고, 각 단어를 생각하는 데 시간이 걸립니다.
*   **작은 초안 모델(draft model)**은 인턴입니다. 훨씬 빠르지만 정확도가 떨어지고, 아이디어를 빠르게 작성할 수 있습니다.

저에게 모든 것을 바꿔놓은 워크플로우는 다음과 같습니다.

1.  관리자가 한 번에 한 단어를 작성하는 대신, 인턴이 다음 5~16개 단어의 짧은 시퀀스(sequence)를 빠르게 초안으로 작성합니다.
2.  그런 다음 관리자는 이 전체 초안을 단일 병렬 단계로 검토합니다.
3.  초안의 각 단어를 동시에 검증합니다.
4.  인턴의 예측이 관리자가 작성했을 내용과 일치하는 한, 해당 단어들은 즉시 수락됩니다.
5.  관리자가 다르게 선택했을 단어를 발견하는 순간, 해당 단어 하나만 수정하고 나머지는 폐기합니다.

<img src="https://via.placeholder.com/800x400?title=Speculative+Decoding+Workflow" alt="Speculative decoding uses a small draft model to generate token sequences in parallel, verified by the larger target model." title="투기적 디코딩은 작은 초안 모델을 사용하여 토큰 시퀀스를 병렬로 생성하고, 이를 더 큰 대상 모델이 검증합니다." width="800" height="400">

*투기적 디코딩은 작은 초안 모델을 사용하여 토큰 시퀀스를 병렬로 생성하고, 이를 더 큰 대상 모델이 검증합니다.*

이것이 더 빠른 이유는 5개의 순차적인 순방향 패스(토큰당 하나)를 수행하는 대신, 초안 작성에 하나, 검증에 하나, 총 두 개의 패스를 수행하기 때문입니다. 메모리 대역폭에 의해 제한되는 시스템에서는 엄청난 속도 향상을 가져올 수 있습니다.

### 실제 결과

Llama-3.3–70B 모델을 대상(target)으로, Llama-3.2–1B를 초안(draft)으로 사용하여 테스트한 결과, RTX 3090에서 초당 약 30토큰이던 처리량(throughput)이 160t/s 이상으로 급증했습니다. 다른 사용자들도 비슷한 성능 향상, 즉 4~5배의 속도 증가가 흔히 보고되었습니다.

가장 좋은 점은? 출력 결과가 대규모 모델이 단독으로 생성했을 결과와 동일하다는 것입니다. 대규모 모델이 수락된 모든 토큰을 검증하므로, 품질 손실은 전혀 없습니다.

### 하드웨어 요구 사항

주요 단점은 두 모델을 동시에 로드할 충분한 VRAM이 필요하다는 것입니다. 실제 사례는 다음과 같습니다.

<img src="https://via.placeholder.com/800x400?title=VRAM+Requirements+for+Speculative+Decoding" alt="VRAM Requirements for Speculative Decoding" title="투기적 디코딩을 위한 VRAM 요구 사항" width="800" height="400">

제 RTX 3090(24GB VRAM)의 경우, 70B + 3B 조합은 실제로 VRAM 용량을 초과했습니다. 대신 1B 초안 모델을 사용해야 했고, 이는 편안하게 맞았습니다.

### Llama.cpp에서 설정하는 방법

llama.cpp 서버로 투기적 디코딩을 실행하는 데 사용하는 정확한 명령어는 다음과 같습니다.

```bash
./llama-server \
 -m ./models/Llama-3.3-70B-Instruct-Q4_K_M.gguf \
 -md ./models/Llama-3.2-1B-Instruct-Q4_K_M.gguf \
 -ngl 99 -ngld 99 -fa --port 9999 -c 8192 \
 --draft-max 16 --draft-min 5
```

각 플래그(flag)가 무엇을 의미하는지 설명해 드리겠습니다.

*   `-m`: 메인 (대규모) 모델 경로
*   `-md`: 초안 (소규모) 모델 경로
*   `-ngl 99`: 메인 모델의 모든 레이어(layer)를 GPU로 오프로드(offload)
*   `-ngld 99`: 초안 모델의 모든 레이어를 GPU로 오프로드
*   `-fa`: 플래시 어텐션(flash attention) 활성화
*   `--draft-max 16`: 초안 모델이 생성하는 최대 토큰 수
*   `--draft-min 5`: 검증 전 최소 토큰 수

**전문가 팁: 다중 GPU 설정**

여러 개의 GPU가 있다면, 초안 모델을 별도의 느린 GPU에 배치하여 메인 모델을 위해 주 GPU의 대역폭을 확보하세요. `-ngld` 플래그를 사용하여 이를 제어할 수 있습니다.

## 기법 #2: 멀티 토큰 예측 (Multi-Token Prediction) - 다음 단계

투기적 디코딩을 작동시킨 후, 저는 훨씬 더 흥미로운 것, 즉 멀티 토큰 예측(multi-token prediction, MTP)을 우연히 발견했습니다. 이 기법은 동일한 핵심 아이디어를 기반으로 하지만, 별도의 초안 모델이 필요 없습니다.

### MTP가 다른 점

멀티 토큰 예측은 일종의 자체 투기적 디코딩(self-speculative decoding)입니다. 별도의 초안 모델을 사용하는 대신, 메인 모델 자체가 여러 미래 토큰을 한 번에 예측하도록 훈련됩니다.

<img src="https://via.placeholder.com/800x400?title=Multi-Token+Prediction+Workflow" alt="Multi-token prediction generates multiple tokens in parallel, dramatically reducing the number of forward passes required." title="멀티 토큰 예측은 여러 토큰을 병렬로 생성하여 필요한 순방향 패스 수를 획기적으로 줄입니다." width="800" height="400">

*멀티 토큰 예측은 여러 토큰을 병렬로 생성하여 필요한 순방향 패스 수를 획기적으로 줄입니다.*

이 과정은 다음과 같이 작동합니다.

1.  **추측 패스(Speculation Pass)**: 모델은 프롬프트(prompt)를 처리하고 동시에 추측 블록 — 예를 들어, [토큰 1, 토큰 2_추측, 토큰 3_추측, 토큰 4_추측, 토큰 5_추측]을 생성합니다.
2.  **검증 패스(Verification Pass)**: 각 추측을 하나씩 확인하는 대신, 모델은 모든 추측을 단일 병렬 패스로 검증합니다. [프롬프트 + 토큰 1 + 토큰 2_추측 + …] 시퀀스를 입력으로 받아 모든 위치에 대한 '올바른' 다음 토큰을 동시에 계산합니다.
3.  **수락**: 모든 추측이 정확하면, 단 두 번의 순방향 패스로 5개의 토큰을 생성한 것이며, 이는 2.5배의 속도 향상을 의미합니다.

### VRAM 이점

이것이 MTP가 하드웨어 제약이 있는 사용자들에게 진정으로 빛을 발하는 지점입니다. 표준 투기적 디코딩은 완전히 별도의 초안 모델을 로드해야 하며, 이는 자체 VRAM을 소모합니다. MTP를 사용하면 메인 모델이 자체 초안 모델이 되도록 훈련됩니다. 유일한 오버헤드(overhead)는 모델에 추가되는 작은 예측 '헤드(heads)'에서 발생하며, 이는 종종 추가 VRAM의 1% 미만을 소모합니다.

저에게는 이것이 엄청난 이점이었습니다. 더 큰 컨텍스트 창(context window)에 필요한 VRAM을 희생하지 않고도 투기적 디코딩 방식의 속도 향상을 얻을 수 있었습니다.

### 단점 (그리고 이는 중요한 단점입니다)

솔직히 말씀드리자면, 모델은 MTP를 위해 특별히 훈련되어야 합니다. 어떤 모델에서든 단순히 MTP를 활성화할 수는 없습니다. 이 기능을 지원하려면 모델이 MTP 헤드로 미세 조정(fine-tune)되어야 합니다.

이는 표준 Llama 3나 Mistral과 같은 인기 있는 GGUF 모델의 기존 라이브러리가 기본적으로 MTP와 작동하지 않는다는 의미입니다. 커뮤니티가 혜택을 받으려면 모델 개발자들이 MTP가 내장된 버전을 출시해야 합니다.

### 어떤 모델이 MTP를 지원하나요?

점점 더 많은 모델들이 MTP 기능을 탑재하여 출시되고 있습니다.

*   DeepSeek V3 / R1
*   Qwen3-Next
*   GLM-4.5

vLLM을 사용한다면, 간단한 명령줄 플래그(command-line flag)로 MTP를 활성화할 수 있습니다.

DeepSeek의 경우:

```bash
vllm serve deepseek-ai/DeepSeek-R1 --speculative-config='{"method": "deepseek_mtp", "num_speculative_tokens": 1}'
```

Qwen3-Next의 경우:

```bash
vllm serve Qwen/Qwen3-Next-80B-A3B-Instruct --speculative-config '{"method": "qwen3_next_mtp", "num_speculative_tokens": 2}'
```

참고: llama.cpp는 아직 MTP를 지원하지 않지만, 개발은 활발히 진행 중입니다. 제가 MTP 모델을 로드하려고 시도했을 때, 이 레이어들은 건너뛰어졌습니다. 특히 CPU 및 하이브리드 추론(hybrid inference)에서 잠재적인 성능 향상을 고려할 때, 이는 프로젝트에서 가장 기대되는 기능 중 하나입니다.

## 기법 #3: 하이퍼파라미터 튜닝 (The Cherry on Top)

투기적 디코딩을 작동시킨 후, 저는 마지막 한 토큰까지도 초당 더 많이 뽑아내고 싶었습니다. 그때 저는 자동화된 하이퍼파라미터(hyperparameter) 튜닝을 발견했습니다.

### 하이퍼파라미터가 중요한 이유

하이퍼파라미터는 AI 모델의 제어 장치입니다. 이를 적절히 조정하는 것은 느리고 반응 없는 모델과 빠르고 효율적인 모델 간의 차이를 만들 수 있습니다.

속도에 영향을 미치는 주요 매개변수는 다음과 같습니다.

*   **배치 크기(Batch size)**: 함께 처리되는 토큰 수
*   **스레드 개수(Thread count)**: 병렬 처리를 위한 CPU 스레드
*   **GPU 레이어 개수(GPU layer count)**: GPU로 오프로드되는 레이어 수
*   **컨텍스트 크기(Context size)**: 최대 시퀀스 길이

### llama-optimus를 사용한 자동 최적화

저는 튜닝 워크플로우를 완전히 바꾼 llama-optimus라는 도구를 찾았습니다. 이 도구는 베이시안 최적화(Bayesian optimization)를 사용하여 최대 처리량을 위한 llama.cpp 성능 플래그를 자동으로 최적화하는 파이썬 도구입니다.

사용 방법은 다음과 같습니다.

```bash
# 설치
pip install llama-optimus
# 최적화 실행
llama-optimus --llama-bin ~/llama.cpp/build/bin --model ~/models/my-model.gguf --trials 25 -r 3 --metric tg
```

이 도구는 다음과 같은 놀라운 일을 합니다.

*   오해의 소지가 있는 콜드 스타트(cold-start) 결과를 피하기 위해 시스템을 워밍업(warm-up)합니다.
*   베이시안 최적화(Optuna를 통해)를 사용하여 최적의 매개변수 조합을 찾습니다.
*   다양한 배치 크기, 스레드 개수, GPU 레이어 개수를 테스트합니다.
*   최적의 추론을 위한 복사 가능한 명령어를 출력합니다.

제 설정에서 이를 실행했을 때, 수동으로는 시도해보지 않았을 구성을 발견했습니다.

```
Best config: {'batch': 4096, 'flash': 1, 'u_batch': 1024, 'threads': 4, 'gpu_layers': 93}
Best tg tokens/sec: 73.5
```

이 도구가 제공한 최적화된 명령어:

```bash
llama-server --model my_model.gguf -t 4 --batch-size 4096 --ubatch-size 1024 -ngl 93 --flash-attn
```

### 워밍업 비법

제가 배운 한 가지는 콜드 스타트가 기만적이라는 것입니다. 모델을 처음 로드할 때 시스템 캐시(cache)가 워밍업되지 않아 인위적으로 높은 토큰 속도를 볼 수 있습니다. Llama-optimus는 실제 세계의 안정 상태 성능을 측정할 수 있도록 워밍업 단계를 포함합니다.

## 종합적인 최적화 워크플로우

새로운 모델에 대해 제가 현재 사용하는 단계별 프로세스는 다음과 같습니다.

### 1단계: 모델 선택

*   **투기적 디코딩의 경우**: 동일 계열(동일 토크나이저(tokenizer))의 메인 모델과 호환되는 초안 모델을 선택합니다.
*   **MTP의 경우**: MTP 지원을 위해 명시적으로 훈련된 모델(DeepSeek, Qwen3-Next)을 찾습니다.

### 2단계: 적절하게 양자화

저는 속도와 품질의 최상의 균형을 위해 Q4_K_M을 사용합니다. 초안 모델의 경우, 더 작은 양자화(Q4_0)도 잘 작동합니다.

### 3단계: 자동 튜닝 실행

```bash
llama-optimus --llama-bin ~/llama.cpp/build/bin --model ~/models/main-model.gguf --trials 30 --metric tg
```

### 4단계: 투기적 디코딩으로 실행

최적화된 매개변수를 가져와 투기적 디코딩 플래그를 추가합니다.

```bash
./llama-server \
 -m ./models/main-model.gguf \
 -md ./models/draft-model.gguf \
 -t [최적 스레드] \
 --batch-size [최적 배치] \
 --ubatch-size [최적 u_batch] \
 -ngl [최적 레이어] \
 -ngld 99 \
 --draft-max 16 \
 --draft-min 5 \
 --flash-attn
```

### 5단계: 벤치마킹 및 비교

항상 전후로 벤치마킹(benchmark)하세요. 저는 신뢰할 수 있는 숫자를 얻기 위해 llama-bench를 사용합니다.

```bash
llama-bench --model my-model.gguf -t 4 -ngl 99 -n 128 -p 128 -r 3
```

## 결과: 두 배 속도는 어떤 모습일까?

최적화 전에는 제 70B 모델이 초당 30토큰으로 기어가는 수준이었습니다. 사용할 수는 있었지만 채팅에는 유쾌하지 않았습니다. 1B 초안 모델을 사용한 투기적 디코딩을 구현하고 하이퍼파라미터를 최적화한 후, 저는 꾸준히 초당 160토큰 이상을 얻고 있습니다.

<img src="https://via.placeholder.com/800x400?title=Before+vs+After+Performance" alt="Before vs After Performance" title="최적화 전후 성능 비교" width="800" height="400">

*캡션: RTX 3090에서 투기적 디코딩을 통한 실제 성능 향상.*

이는 다음을 의미합니다.

*   500토큰 응답이 16초 대신 3초가 걸린다는 것
*   실시간 채팅이 즉각적으로 느껴진다는 것
*   배치 처리 작업(batch processing jobs)을 훨씬 짧은 시간에 실행할 수 있다는 것

이 모든 것이 제가 2년 동안 사용해온 바로 그 하드웨어에서 이루어졌습니다.

## 제가 저질렀던 일반적인 실수 (여러분은 하지 마세요)

### 실수 #1: 잘못된 모델 순서

한번은 메인 모델과 초안 모델을 바꿔서 사용했습니다. 속도 향상 대신 엄청난 속도 저하가 발생했습니다. 항상: 대규모 모델은 `-m`으로, 소규모 모델은 `-md`로 사용하세요.

### 실수 #2: 토크나이저 호환성 무시

초안 모델은 메인 모델과 동일한 토크나이저를 공유해야 합니다. 그래서 동일 계열의 모델(Llama 3.3 70B와 Llama 3.2 1B)을 사용하는 것이 매우 효과적입니다.

### 실수 #3: 워밍업 안 하기

콜드 스타트는 실제 성능을 반영하지 않는 인위적으로 높은 수치를 보여주었습니다. 항상 벤치마킹 전에 시스템을 워밍업하세요.

### 실수 #4: VRAM 과부하

제 70B 메인 모델과 3B 초안 모델을 함께 사용하려 했지만, 둘이 합쳐 제 24GB VRAM을 초과했습니다. 시스템은 시스템 RAM으로 스와핑(swapping)하기 시작했고, 성능은 급락했습니다. VRAM이 부족하다면 더 작은 초안 모델을 고수하세요.

## 저의 다음 최적화 목록

이 분야는 빠르게 발전하고 있습니다. 제가 주목하는 내용은 다음과 같습니다.

### QAT(Quantization-Aware Training)

구글은 최근 QAT(Quantization-Aware Training) 최적화가 적용된 Gemma 3 모델을 출시했습니다. 기존 양자화(훈련 후 변환)와 달리, QAT는 모델이 처음부터 양자화에 강하도록 훈련합니다. 초기 테스트 결과, QAT 모델은 4비트 정밀도에서도 품질을 유지하며, 퍼플렉시티(perplexity) 저하가 거의 없음을 보여줍니다.

### 더 나은 MoE 오프로딩

DeepSeek과 같은 MoE(Mixture-of-Experts) 모델의 경우, 새로운 기법은 더 스마트한 오프로딩을 가능하게 합니다. 즉, "항상 활성" 매개변수는 GPU에 유지하고 전문가(expert)들은 CPU로 라우팅하는 방식입니다. 성능 향상은 극적입니다.

### Glinthawk 아키텍처

마이크로소프트의 연구원들은 어텐션 메커니즘(attention mechanism)을 저사양 연산 장치로 오프로드하는 2계층 아키텍처(two-tiered architecture)를 시연하여 5.9배의 처리량 향상을 달성했습니다. 이것은 최첨단 기술이지만, 이 분야가 어디로 향하고 있는지를 보여줍니다.

## 결론

로컬 LLM에서 훨씬 더 나은 성능을 얻기 위해 새 하드웨어를 구매할 필요는 없습니다. 제가 공유한 기법들, 즉 투기적 디코딩, 멀티 토큰 예측, 그리고 자동화된 하이퍼파라미터 튜닝은 여러분이 이미 가지고 있는 것을 더 효율적으로 활용하는 순수한 소프트웨어 최적화입니다.

주요 핵심 사항:

*   투기적 디코딩은 호환 가능한 모델 쌍으로 4~5배의 속도 향상을 제공할 수 있습니다.
*   멀티 토큰 예측은 (지원되는 경우) VRAM 오버헤드 없이 유사한 성능 향상을 제공합니다.
*   자동화된 튜닝은 수동으로는 절대 발견할 수 없는 최적의 설정을 찾아줍니다.
*   항상 워밍업된 시스템으로 벤치마킹하여 실제 수치를 얻으세요.

오늘날 제 설정은 6개월 전의 제 설정보다 훨씬 뛰어납니다. 그리고 가장 좋은 점은? 제가 마침내 하드웨어를 업그레이드할 때, 이 동일한 기법들이 함께 확장될 것이라는 점입니다.

자신만의 속도를 두 배로 늘릴 준비가 되셨나요? 좋아하는 LLM을 위한 호환 가능한 초안 모델을 다운로드하고, 제가 공유한 플래그를 사용하여 llama.cpp를 실행한 다음, 토큰 카운터가 치솟는 것을 지켜보세요. 여러분의 지갑 — 그리고 인내심 — 이 감사할 것입니다.
