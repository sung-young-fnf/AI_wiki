# Qwen 3.5 35B-A3B: 800달러짜리 GPU가 최첨단 AI 워크스테이션이 된 이유

오랫동안 로컬 모델(local models)을 실행해왔기 때문에 소비자용 하드웨어(consumer hardware)의 한계에 대해 어느 정도 파악하고 있다고 생각했습니다.

하지만 350억(35B) 개의 매개변수(parameter)를 가진 모델이 토큰당 30억(3B) 개의 매개변수만 활성화하면서도 2,350억(235B) 개의 매개변수를 가진 이전 모델을 능가했을 때, 이는 프로덕션(production) 워크로드(workload)에 적합한 로컬 모델을 기다리던 개발자들에게 전환점이 되었습니다.

단 30억(3B) 개의 활성화로 Sonnet 4.5 수준의 성능을 얻는 것은 매우 인상적입니다. 이는 전체 가중치(weight)의 약 8.6%에 해당합니다.

가격 대비 성능(price performance ratio) 비율이 매우 매력적입니다.

## 왜 지금 이야기하는가?

같은 주에 세 가지 일이 일어났는데, 이들의 융합이 흥미로운 점이라고 생각합니다.

첫째, 알리바바(Alibaba)의 Qwen 팀은 Apache 2.0 라이선스(license) 하에 네 가지 모델을 포함하는 Qwen 3.5 미디엄(medium) 모델 시리즈를 출시했습니다:

*   35B-A3B
*   27B 덴스 모델(dense model)
*   122B-A10B
*   플래그십(flagship) 397B-A17B

Qwen 3.5 미디엄 모델들은 모든 이전 Qwen 모델들을 능가하며, 6배 더 큰 모델들보다 뛰어나고, Sonnet 4.5보다 더 스마트하며, 거의 모든 최신 컴퓨터에서 완전히 무료로 실행될 수 있습니다.

이 발표에서 더 흥미로운 부분은 30억(3B) 개의 활성 매개변수를 가진 모델이 이제 이전 세대의 220억(22B) 개의 활성 매개변수를 가진 모델을 능가한다는 것입니다.

저는 점진적인 개선(incremental improvements)을 여러 번 보아왔지만, 이것은 분명히 최첨단 모델(frontier models)에서 '작다'는 것의 의미를 변화시킵니다.

둘째, Unsloth 팀은 데이 제로(day-zero) 접근 권한을 포함한 최적화된 GGUF 양자화(quantization)를 출시했습니다. 여기에는 중요한 레이어(layer)를 8비트(8-bit) 또는 16비트(16-bit)로 업캐스트(upcast)하면서도 모델을 컴팩트(compact)하게 유지하는 Dynamic 2.0 양자화가 포함됩니다.

그들은 또한 모든 양자화 업로더(uploader)에서 함수 호출(function-calling) 워크플로우(workflow)를 조용히 깨뜨리던 도구 호출(tool-calling) 채팅 템플릿(template) 버그(bug)를 찾아 수정했습니다.

만약 에이전트(agent)의 도구 호출이 계속해서 불필요한 JSON(JavaScript Object Notation)을 반환하는 이유를 궁금해하셨다면, 아마 이것이 원인이었을 것입니다.

셋째, 로컬 추론(inference) 커뮤니티는 이에 대해 열광했습니다.

48시간 이내에, 누군가가 단일 프롬프트(prompt)로 전체 게임을 자율적으로 구축하고, 검열 없는 에이전트 파이프라인이 자동 조종으로 서비스에 가입하며, 세 개의 플래그로 생성 속도를 두 배로 늘리는 처리량 최적화(throughput optimizations)에 대한 보고가 있었습니다.

## Qwen 3.5 35B-A3B란 무엇인가?

35B는 총 매개변수(total parameters)를 의미하고, A3B는 토큰당 30억(3B) 개의 활성 매개변수를 의미하며, 이는 희소 전문가 혼합(sparse Mixture-of-Experts, MoE) 모델입니다.

여기서 중요한 점은 디스크(disk)와 VRAM(Video RAM)에 350억(35B) 개의 매개변수를 저장하지만, 추론(inference) 시에는 GPU(Graphics Processing Unit)가 30억(3B) 개의 계산만 처리한다는 것입니다.

이는 30억(3B) 개의 모델을 실행하는 것과 유사한 계산 비용으로 350억(35B) 개 모델의 메모리, 즉 세상 지식, 훈련된 표현, 매개변수적 폭넓음을 얻을 수 있다는 의미입니다.

## 작동을 가능하게 하는 아키텍처

긴 컨텍스트(long-context) 모델을 다루는 모든 사람들이 직면했던 문제는 다음과 같습니다: 표준 트랜스포머(transformer) 어텐션(attention)은 시퀀스 길이(sequence length)에 따라 2차 함수적으로(quadratically) 확장됩니다.

컨텍스트 창을 두 배로 늘리면, 컴퓨팅(compute) 요구량은 네 배가 됩니다.

우리는 10만(100K) 개 이상의 토큰 컨텍스트를 가진 RAG(Retrieval Augmented Generation)를 시도할 때 추론(inference) 속도가 매우 느려지는 고통을 모두 경험했습니다.

Qwen 3.5 하이브리드(hybrid)는 두 가지 어텐션 메커니즘(attention mechanisms)을 3:1 비율로 교차 적용하여 이 문제를 해결합니다:

4개 블록(block) 중 3개는 **Gated DeltaNet**을 사용합니다. 이는 Mamba2의 게이팅 감쇠(gated decay)와 숨겨진 상태(hidden states)를 업데이트(update)하는 델타 규칙(delta rule)을 결합한 선형 어텐션(linear attention) 변형입니다.

각 레이어(layer)는 입력 시퀀스(input sequence)를 고정된 크기의 상태로 압축하여, 시퀀스 길이에 거의 선형적으로 확장됩니다. 게이팅 메커니즘(gating mechanism, 어텐션 출력을 적응적으로 조절하는 시그모이드(sigmoid))은 어텐션 싱크(attention sink) 문제를 해결하고 대규모 훈련(training) 시 발생하는 대량 활성화(massive activations)를 방지합니다.

4개 블록 중 1개는 표준 **전체 어텐션(full attention, 또한 게이팅됨)**을 유지하여 세밀한 토큰 간 추론(token-to-token reasoning)을 보존합니다.

이러한 레이어들이 모델이 시퀀스 내의 어떤 위치에도 정확하게 주의를 기울일 수 있게 하며, 코드 생성(code generation) 작업을 해본 적이 있다면 임포트(imports), 변수 참조(variable references), 함수 시그니처(function signatures)를 올바르게 작성하는 데 이것이 얼마나 중요한지 알 것입니다.

이는 Kimi Linear의 작업에서 나타나던 것과 일치하지만, Kimi는 전체 어텐션 레이어에서 채널별 게이팅(channel-wise gating)과 다중 헤드 잠재 어텐션(multi-head latent attention)으로 이를 개선합니다.

두 개의 독립적인 팀이 동일한 3:1 비율에 수렴하는 것을 보면, 이 분야가 어디로 향하고 있는지 짐작할 수 있습니다.

## 배포 시 실제 의미

추론 실행(inference runs) 중에 `nvidia-smi`를 충분히 오랫동안 지켜보면서 3:1 비율이 실제로 무엇을 하는지 체감할 수 있었습니다.

세 가지 구체적인 결과:

(1) **KV-캐시(KV-cache) 메모리가 크게 감소합니다.** 선형 어텐션 레이어는 모든 토큰에 대한 키-값(key-value) 쌍을 저장하는 대신 컨텍스트를 고정 크기 상태로 압축합니다. 이것이 바로 35B-A3B가 26만 2천(262K) 컨텍스트로 소비자용 GPU에서 실행되는 진짜 이유이며, 유효 KV-캐시 공간(footprint)은 순수 어텐션 모델(pure-attention model)이 필요로 하는 것의 극히 일부에 불과합니다. 특히 전체 코드베이스(codebases)를 컨텍스트에 담아야 하는 에이전트 파이프라인(agentic pipelines) 작업을 할 때 매우 유용합니다.

(2) **컨텍스트 확장(context scaling)이 거의 평탄합니다.** RTX 5090의 커뮤니티 벤치마크(benchmark)에 따르면, Qwen 3.5 모델은 512에서 8,192 토큰의 컨텍스트에 이르기까지 -0.9%의 처리량 저하(degradation)만을 보이며 처리량을 유지했습니다. 이전 Qwen3 30B-A3B (순수 어텐션 + MoE, DeltaNet 없음)는 동일 범위에서 -21.5% 저하되었습니다. RAG 파이프라인이나 장문 대화 에이전트를 구축하는 경우 이는 중요합니다.

(3) **아키텍처로 인해 처리량에 대한 대가(tax)를 지불합니다.** 솔직히 말해서, 3.5 모델은 동일한 하드웨어에서 이전 30B-A3B 모델보다 순수 생성 속도(raw generation speed)가 약 32% 느립니다. DeltaNet 레이어와 더 큰 어휘(vocabulary, 24만 8천 토큰 대 15만 2천 토큰)는 오버헤드(overhead)를 추가하지만, 그 대가로 컨텍스트 안정성(context stability)을 얻습니다. 이러한 장단점(tradeoff)이 당신의 워크로드에 적합한지는 전적으로 당신의 선택에 달려 있으며, 어떤 것을 선택할지 신중하게 결정할 가치가 있다고 생각합니다.

## 위에 얹힌 MoE 레이어

하이브리드 어텐션(hybrid attention) 외에도, 각 피드포워드 블록(feedforward block)은 희소 전문가 혼합(sparse MoE) 레이어를 통해 토큰(token)을 라우팅(route)합니다. 이 모델은 MoE 레이어당 256개의 전문가 서브네트워크(expert sub-networks)를 가지며, 학습된 라우터(router)가 각 토큰에 대해 상위 9개를 선택합니다. 이는 토큰당 피드포워드 매개변수의 약 3.5%만 활성화된다는 것을 의미합니다.

선형 어텐션(메모리 감소)과 희소 MoE(계산 감소)의 조합이 이 모델이 자신의 체급을 뛰어넘는 성능을 발휘하게 하는 요인입니다.

## 로컬 성능 및 얻는 방법

Ollama, LM Studio, 심지어 단순한 llama.cpp 설정의 기본 구성은 24GB 이상의 VRAM 카드에서 초당 40~70 토큰을 생성합니다. 최적화된 설정은 동일한 하드웨어에서 초당 100~122 토큰을 생성합니다.

대부분의 도구가 f16 KV 캐시(KV cache)를 기본으로 사용하고, MoE 전문가 오프로딩(expert offloading)을 최적화하지 않으며, 플래시 어텐션(flash attention) 플래그를 건너뛰기 때문입니다.

RTX 5080 (16GB)에서의 커뮤니티 벤치마크는 명확한 그림을 보여주었습니다:

*   Q4_K_M (수동 MoE 오프로딩: `--n-cpu-moe 24`): ~70 tok/s
*   Q4_K_M (자동 최적화: `auto-fit`): ~65 tok/s
*   Q8_0 (전체 오프로딩: `full offload`): ~35 tok/s

f16에서 q8_0 KV 캐시로 전환하면 VRAM을 덜 사용하면서도 12~38%의 처리량 향상(throughput boost)을 얻을 수 있으니, 사용하지 않을 이유를 찾을 수 없습니다.

저는 이와 매우 유사하게 실행해 왔으며 작동함을 확인할 수 있습니다:

```bash
# llama.cpp를 소스에서 빌드 (필수 -- 미리 빌드된 바이너리에는 최적화가 누락될 수 있음)

git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp
cmake -B build -DGGML_CUDA=ON -DCMAKE_BUILD_TYPE=Release
cmake --build build --config Release -j
  
# 최적화된 플래그로 실행
./build/bin/llama-server \
-hf unsloth/Qwen3.5-35B-A3B-GGUF:UD-Q4_K_XL \
--cache-type-k q8_0 \
--cache-type-v q8_0 \
-np 1 \
-ngl 99 \
-fa \
--ctx-size 65536 \
--jinja \
--port 8080
```

각 플래그(flag)의 기능:

*   `--cache-type-k q8_0 --cache-type-v q8_0`: f16 기본값 대비 12-38% 향상.
*   `-np 1`: 병렬 요청 슬롯(slot)을 1로 설정합니다. 단일 사용자 로컬 추론(single-user local inference)의 경우 스케줄링 오버헤드(scheduling overhead)를 제거하므로 중요합니다.
*   `-ngl 99`: 모든 레이어를 GPU로 오프로드(offload)합니다. VRAM 한계에 도달하는 경우에만 줄이십시오.
*   `-fa`: 플래시 어텐션(flash attention)을 활성화합니다.

이 설정으로:

*   RTX 3090 (24GB): 100 tok/s (기본 플래그 사용 시 약 50 tok/s에서 향상)
*   RTX 4090 (24GB): 122 tok/s (기본 플래그 사용 시 약 70 tok/s에서 향상)
*   전체 26만 2천(262K) 컨텍스트 유지, 속도 손실 없음

유사한 하드웨어와 설정에서 이러한 수치를 재현할 수 있을 것입니다.

## Ollama가 아직 올바른 도구가 아닌 이유

Ollama는 MoE 전문가 오프로딩을 지원하지 않습니다.

MoE 모델이 VRAM 용량을 초과하면 Ollama는 레이어 수준에서 분할하여 전체 트랜스포머 블록(transformer blocks)을 CPU나 GPU 중 한 곳으로 보냅니다. 따라서 GPU는 CPU가 레이어 처리를 마칠 때까지 유휴 상태로 기다립니다. 이는 MoE 아키텍처에 있어서는 완전히 잘못된 전략입니다.

올바른 접근 방식은 전문가 전용 오프로딩(`-ot "exps=CPU"` 또는 `--n-cpu-moe`)입니다. 이는 어텐션(attention), 노름(norms) 및 공유 전문가(shared experts)를 GPU에 유지하고 전문가 FFN(FeedForward Network) 가중치(weights)만 PCIe를 통해 라우팅(routing)합니다.

Ollama는 또한 f16 KV 캐시를 기본으로 사용하고, 종종 플래시 어텐션이 꺼져 있으며, 배치 크기(batch size)를 제어할 수 없습니다.

더 자세한 내용은 [여기 커뮤니티 토론](https://github.com/ggerganov/llama.cpp/discussions/6321)에서 확인할 수 있습니다.

## Claude Code + 로컬 LLM 패턴

이것은 제 작업에 실질적으로 유용합니다.

Claude Code는 `claude-code-router`라는 도구를 통해 Anthropic의 API(Application Programming Interface) 대신 로컬 llama-서버(llama-server) 엔드포인트(endpoint)를 가리키도록 설정할 수 있습니다.

이를 통해 로컬 모델이 생성(generation)을 제공하면서도 Claude Code의 에이전트 스캐폴딩(agentic scaffolding), 즉 파일 관리, 오류 복구(error recovery), 다중 파일 편집 등 모든 이점을 얻을 수 있습니다.

```json
// 최소한의 claude-code-router 설정 (~/.claude-code-router/config.json)

{
  "providers": [
    {
      "name": "llamacpp",
      "api_base_url": "http://localhost:8080/v1/chat/completions",
      "api_key": "not-needed",
      "models": ["Qwen3.5"]
    }
  ]
}
```

고급 에이전트 코딩을 많이 하는 경우, 호스팅(hosted) 모델의 API 비용은 빠르게 누적될 수 있습니다. 이 설정에서는 그냥 실행하게 두면 됩니다. 토큰당 요금이 부과되지 않는다는 것을 알면, 더 야심찬 프롬프트(prompt)로 실험하는 데 훨씬 더 적극적이 될 것입니다.

## 한계점

하지만 이 모델이 만능 해결책은 아니며, 이 모델을 기반으로 구축하기 전에 알아둘 가치가 있는 실제적인 한계에 부딪혔습니다.

예를 들어, 35B-A3B에 다단계 논리(multi-step logic), 미묘한 코드 아키텍처 결정(nuanced code architecture decisions) 또는 심오한 수학적 증명(deep mathematical proofs)을 시도하면, 27B 덴스(dense) 모델과는 달리 힘들어하는 것을 느낄 수 있습니다.

생각해보면 이는 당연합니다. 30억(3B) 개의 활성 매개변수는 대략 100억(10B) 개에 해당하는 추론 깊이(reasoning depth)를 의미하며, 350억(35B) 개를 의미하지는 않습니다. 지식은 존재하지만, 토큰당 처리 깊이는 근본적으로 작업에 참여하는 매개변수의 수에 의해 제한됩니다.

따라서 복잡한 작업에는 온도(temperature) 1.0의 사고 모드(thinking mode)를 사용해 볼 수 있으며, 어려운 작업의 경우 여전히 덴스 27B 또는 더 큰 모델을 사용할 수 있습니다.

여러분들의 경험이 궁금합니다. MoE 모델을 로컬(local)에서 실행하고 계신가요? KV 캐시 q8_0 플래그를 사용해 보셨나요? 로컬 에이전트 설정으로 무엇을 구축하고 계신가요? 댓글을 남겨주시면 다른 사람들에게 무엇이 작동하고 무엇이 작동하지 않는지 진심으로 알고 싶습니다.
