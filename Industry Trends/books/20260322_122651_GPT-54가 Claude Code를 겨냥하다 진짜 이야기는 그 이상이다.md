## 요약 (TL;DR)

*   GPT-5.4는 코딩(coding), 추론(reasoning), 컴퓨터 사용(computer use)을 하나의 모델(model)로 통합했지만, 코딩 벤치마크(benchmark) 점수 상승은 단 2.1점에 불과합니다. 진정한 변화는 컴퓨터 사용(OSWorld에서 +27.7점)과 토큰(token) 효율성(도구 검색(Tool Search)으로 47% 절감)에 있습니다.
*   빠른 출시 주기(cadence)는 돌파구(breakthrough)를 의미하는 것이 아닙니다. 이는 사전 훈련(pretraining) 스케일링(scaling)이 정체되었고, 모든 연구소가 제품 레이어(product-layer) 혁신을 더 빠르게 출시하는 방향으로 전환했음을 나타냅니다.
*   모델(model)은 상품화되고 있습니다. 이제 동일한 MCP 프로토콜(protocol)과 SKILL.md 포맷(format)이 Claude Code, Codex CLI, Gemini CLI, Cursor에서 모두 작동합니다. 경쟁은 모델 레이어(model layer) 위로 이동했습니다.
*   Claude Code는 9개월 만에 매출 0달러에서 25억 달러(연간)를 달성했으며, 엔터프라이즈(enterprise) 코딩(coding) 시장 점유율 54%를 차지하고 있습니다. GPT-5.4는 이에 대한 직접적인 반격입니다.
*   OpenAI는 모델 레이어(model layer)에서의 광범위한 통합(하나의 모델이 모든 것을 처리)에 투자합니다. Anthropic은 런타임 레이어(runtime layer)에서의 깊이 있는 전문화(터미널 네이티브(terminal-native) 에이전트(agents), 서브 에이전트(sub-agents), 9,000개 이상의 플러그인(plugins))에 투자합니다. 서로 다른 레이어(layer), 서로 다른 문제입니다.
*   솔직한 빌더(builder)의 전략: 브랜드(brand) 충성도가 아닌 작업 복잡성에 따라 라우팅(route)하세요. 속도를 위해서는 GPT-5.4 모델을 사용하고, 깊이를 위해서는 Claude Code 런타임(runtime)을 사용하세요. 인프라(infrastructure) 결정을 내리기 전에 독립적인 벤치마크(benchmark)를 기다리세요.

## 왜 이렇게 빨리 출시되는가

11월에 GPT-5.1, 12월에 GPT-5.2, 2월에 GPT-5.3-Codex, 3월에 GPT-5.4. 넉 달 동안 네 번의 포인트 릴리스(point-releases)가 있었습니다. 한편, Anthropic은 Claude 4.5와 4.6을, Google은 Gemini 3와 3.1을 출시했습니다. 2월에만 Gemini 3.1 Pro, Claude Opus 4.6, GPT-5.3-Codex가 모두 출시되었고, 이제 3월 초에 GPT-5.4가 나왔습니다. 약 한 달 만에 네 개의 최첨단 모델(frontier models)이 등장한 것입니다.

이러한 주기는 정상적이지 않습니다. 그리고 연구소들이 연달아 네 가지 돌파구(breakthrough)를 발견했기 때문도 아닙니다.

*글쓴이 제공 이미지 — 세 가지 스케일링(scaling) 패러다임(paradigms): 사전 훈련(pretraining)은 한계 효용 체감(diminishing returns)에 도달했고, 산업은 사후 훈련(post-training)과 테스트 시 컴퓨트(test-time compute)로 전환했습니다.*

진정한 원인은 사전 훈련(pretraining) 스케일링(scaling)이 한계 효용 체감(diminishing returns)에 도달했기 때문입니다. OpenAI의 내부 차세대 모델(next-gen model)(Orion)은 GPT-3에서 GPT-4로의 도약보다 훨씬 작은 성능 향상을 보였습니다. Google의 Gemini 팀(team)은 다음 훈련 실행(training run)에서 “실망스러운 결과”를 보았다고 전해집니다. UC Berkeley의 연구(Snell et al.)는 연구소들이 내부적으로 확인하고 있던 사실을 확증했습니다: 쉽거나 중간 수준의 작업에서는 추론(inference) 시 더 많은 컴퓨트(compute)를 사용하는 것이 사전 훈련에 컴퓨트(compute)를 투자하는 것보다 종종 더 효과적이라는 것입니다.

OpenAI의 전 수석 과학자(chief scientist) Ilya Sutskever는 “2010년대는 스케일링(scaling)의 시대였습니다. 이제 우리는 경이와 발견의 시대로 돌아왔습니다”라고 단도직입적으로 말했습니다.

그래서 연구소들은 방향을 전환했습니다. 하나의 새로운 것이 아닌, 세 가지로:

*   **사후 훈련 강화 학습(Post-training RL)** — 사전 훈련(pretraining) 후 강화 학습(reinforcement learning)을 통해 특정 기능(추론(reasoning), 코딩(coding), 도구 사용(tool use))을 강화
*   **테스트 시 컴퓨트(Test-time compute)** — 더 큰 모델(model)을 훈련하는 대신 추론(inference) 시 더 많은 컴퓨트(compute)를 사용하여 “더 오래 생각”하게 함
*   **에이전트(Agent) 기능** — 컴퓨터 사용(computer use), 도구 검색(tool search), 다단계 워크플로(multi-step workflows) — 새로운 기본 모델(base model)을 필요로 하지 않는 제품 레이어(product-layer) 기능

*글쓴이 제공 이미지 — GPT-5.4의 모든 주요 기능(headline feature)은 사전 훈련(pretraining) 돌파구(breakthrough)가 아닌 추론 시간(inference-time) 혁신으로 연결됩니다.*

여기 반직관적인 진실이 있습니다: GPT-5.4의 모든 주요 기능(headline feature)은 추론 시간(inference-time) 혁신입니다. 도구 검색(Tool Search)은 추론(inference) 시 토큰(token)을 절약합니다. 컴퓨터 사용(computer use)은 추론(inference) 시 스크린샷(screenshot) → 액션(action) 루프(loops)를 실행합니다. 사고 모드(Thinking mode)는 추론(inference) 시 추론(reasoning)을 확장합니다. 100만 토큰(1M token) 컨텍스트 창(context window)은 추론(inference) 시 더 많은 데이터를 처리합니다. 이들 중 어느 것도 근본적으로 더 크거나 똑똑한 기본 모델(base model)을 필요로 하지 않았습니다 — 그들은 더 나은 제품 패키징(product wrapping)을 필요로 했습니다.

이것이 빠른 출시 주기(cadence)를 설명합니다. 주요 혁신 루프(innovation loop)가 “더 큰 모델(model) 훈련”(18개월 주기)에서 “더 나은 제품 레이어(product layer) 출시”(주에서 월 단위 주기)로 이동할 때, 훨씬 더 자주 출시할 수 있습니다. 모든 릴리스(release)는 제품 레이어(product-layer)의 움직임이지, 훈련 돌파구(training breakthrough)가 아닙니다.

재정적 압력은 속도를 증폭시킵니다. OpenAI는 2026년 2월 8,400억 달러(840B)의 기업 가치(valuation)로 1,100억 달러(110B)를 조달했지만, 올해 140억 달러(14B)의 손실을 예상하고 있습니다. Anthropic의 매출은 같은 달 연간 140억 달러(14B)에 도달했으며, 3년 연속 전년 대비 10배 성장했습니다. 시장 점유율(market share) 경쟁은 다음 사전 훈련(pretraining) 실행을 기다리지 않습니다.

OpenAI 엔지니어(engineers)는 8:1 비율로 Anthropic으로 이직합니다. OpenAI의 인력 유지율(retention rate)은 67%인 반면 Anthropic은 80%입니다. 인재가 이탈하고 투자자들이 추진력에 대한 증거를 요구할 때, 훈련 돌파구(training breakthrough)가 있든 없든 더 빠르게 출시하게 됩니다.

솔직한 결론: 각 릴리스(release)를 기능적인 단계적 변화로 여기지 마세요. 제품 반복(product iteration)으로 간주하세요. 모델(model)들은 훈련(training) 비용 대비 극적으로 더 똑똑해지고 있지 않습니다. 그들은 이미 알고 있는 것을 사용하는 데 극적으로 더 효율적이 되고 있습니다.

## 진정한 전쟁은 한 단계 위에서 벌어진다

모델(model)이 상품화(commoditizing)되고 벤치마크(benchmark)가 수렴(converging)한다면, 경쟁은 어디로 갔을까요?

돈의 흐름을 따르세요. Claude Code는 약 9개월 만에 매출 0달러에서 연간 25억 달러(2.5B)를 달성했습니다. Claude는 엔터프라이즈(enterprise) 코딩(coding) 시장의 54%를 차지하며, OpenAI의 21%보다 두 배 이상 높습니다. Anthropic은 전체 엔터프라이즈(enterprise) LLM 시장 점유율(market share)에서 OpenAI를 추월했습니다: 2년 전 OpenAI의 50%에서 32% 대 25%로 역전되었습니다.

GPT-5.4가 Claude Code를 겨냥하는 것은 모델(model) 품질 경쟁이 아닙니다. 이것은 시장 점유율(market share) 회복 전략입니다.

*이미지를 보려면 Enter를 누르거나 클릭하세요.*
*글쓴이 제공 이미지 — 새로운 경쟁 스택(stack): 모델(model)은 상품화(commoditizing)되고, 싸움은 에이전트 런타임(agent runtime) 및 생태계(ecosystem) 레이어(layers)로 이동했습니다.*

MCP(Model Context Protocol) 수렴(convergence)이 가장 명확한 증거입니다. Anthropic은 2024년 11월에 모델 컨텍스트 프로토콜(Model Context Protocol, MCP)을 만들었습니다. 2025년 3월까지 OpenAI는 이를 에이전트 SDK(Agents SDK), 응답 API(Responses API), ChatGPT 데스크톱(desktop)에 걸쳐 채택했습니다. 2025년 12월에는 두 회사가 리눅스 재단(Linux Foundation) 산하에 에이전틱 AI 재단(Agentic AI Foundation)을 공동 설립하여 MCP를 공유 표준으로 관리했습니다. Google, Microsoft, Amazon 및 수십 개의 엔터프라이즈(enterprise) 기업이 회원으로 참여했습니다.

경쟁사가 당신의 프로토콜(protocol)을 채택하고, 심지어 그것을 유지하기 위해 당신과 함께 재단을 공동 설립한다면 — 경쟁은 더 이상 프로토콜 레이어(protocol layer)에 있지 않습니다. 그 위로 이동한 것입니다.

*이미지를 보려면 Enter를 누르거나 클릭하세요.*
*글쓴이 제공 이미지 — 엔터프라이즈(enterprise) 시장 점유율 역전: Anthropic의 부상, OpenAI의 하락, 그리고 Claude Code의 25억 달러(2.5B) 궤적.*

스킬(Skills)도 같은 이야기를 들려줍니다. 2025년 12월, Anthropic은 에이전트 스킬(Agent Skills) 사양을 오픈 표준(open standard)으로 공개했습니다. OpenAI는 Codex CLI에 동일한 SKILL.md 포맷(format)을 채택했습니다. 이제 동일한 스킬(skill)이 Claude Code, Codex CLI, Gemini CLI, Cursor, Windsurf에서 실행됩니다. SkillsMP에만 35만 개 이상의 스킬(skill)이 색인되어 있습니다. Claude Code의 플러그인(plugin) 생태계(ecosystem)는 9,000개 이상의 패키지(packages)로 성장했습니다.

기반 모델(model)은 상호 교환 가능한 배관이 되고 있습니다. 이제 차별점은 다음과 같습니다:

*   누가 가장 풍부한 생태계(ecosystem)를 가지고 있는가 — Claude Code: 9,000개 이상의 플러그인(plugins), 520만 VS Code 설치 (Codex 490만 대비)
*   누가 최고의 에이전트 런타임(agent runtime)을 가지고 있는가 — 서브 에이전트(sub-agents), 훅(hooks), 에이전트 팀(Agent Teams), 권한 모델(permission models)
*   누가 개발자 워크플로(developer workflow)를 장악하는가 — 일단 당신의 팀(team)이 특정 도구의 MCP 서버(servers), 훅(hooks), CLAUDE.md 컨벤션(conventions)을 중심으로 구축되면, 전환 비용(switching costs)은 현실이 됩니다.

*이미지를 보려면 Enter를 누르거나 클릭하세요.*
*글쓴이 제공 이미지 — 스킬(Skills) 상호 운용성(interoperability): 동일한 SKILL.md가 5가지 다른 에이전트 런타임(agent runtimes)에서 실행됩니다.*

이것은 이론적인 관찰이 아닙니다. 이미 채택(adoption)에서 명확하게 나타납니다. Codex CLI보다 몇 달 늦게 출시되었음에도 불구하고, Claude Code는 VS Code 설치(520만 대 490만)와 평점(5점 만점에 4.0점 대 3.4점)에서 앞서고 있습니다. 이제 생태계(ecosystem) 해자(moat)가 형성되고 있습니다.

진정한 통찰력: 다음에 새로운 모델(model)이 출시되고 벤치마크(benchmark)가 인상적으로 보인다면, 그것이 에이전트 런타임(agent runtime), 생태계(ecosystem) 또는 워크플로(workflow) 장악에 변화를 주는지 스스로에게 물어보세요. 만약 모델 점수(model scores)만 변경한다면, 경쟁적 영향은 보도 자료(press release)가 시사하는 것보다 작습니다.

## 두 가지 전략, 두 가지 아키텍처(Architectures)

모델 레이어(model layer)가 상품화(commoditizing)되고 프로토콜 레이어(protocol layer)가 공유된다면, Claude Code와 Codex CLI는 왜 사용하기에 그렇게 다르게 느껴질까요? 그것은 근본적으로 다른 아키텍처(architectural) 철학을 기반으로 구축되었기 때문입니다.

**GPT-5.4에 대한 OpenAI의 전략:** 모든 것을 하나의 모델(model)로 통합하는 것입니다. GPT-5.3-Codex의 코딩(coding), O-시리즈의 추론(reasoning), Operator의 컴퓨터 사용(computer use), 효율성을 위한 도구 검색(Tool Search). 하나의 통합된 모델(unified model), 클라우드 우선(cloud-first), 토큰(token) 효율적입니다. 개발자(developers)에게 제시하는 제안: 어떤 모델을 사용할지 고민할 필요가 없습니다 — GPT-5.4가 모든 것을 처리합니다. 비즈니스 로직(business logic): 주간 사용자 9억 명, 2026년 4분기 잠재적 IPO(기업공개), 그리고 광범위한 확장을 통한 플랫폼 플레이(platform play)입니다.

**Claude Code에 대한 Anthropic의 전략:** 에이전트 레이어(agent layer)에 깊이 파고드는 것입니다. 모델(model)(Opus 4.6)은 훌륭하지만, 그것이 제품의 전부는 아닙니다. 제품은 터미널 네이티브(terminal-native) 런타임(runtime)입니다: 고유한 프롬프트(prompts)와 권한으로 작업을 위임하는 서브 에이전트(sub-agents), 세션(sessions) 간 병렬 조정을 위한 에이전트 팀(Agent Teams), 세션 중간에 Jira, Google Docs, Slack에서 컨텍스트(context)를 가져오는 MCP 서버(servers), 그리고 9,000개 이상의 패키지(packages)를 가진 플러그인(plugin) 생태계(ecosystem)입니다. 비즈니스 로직(business logic): 엔터프라이즈(enterprise) 매출 85%, 코딩(coding) 시장의 54% 점유, 2027년까지 긍정적인 현금 흐름.

*글쓴이 제공 이미지 — 두 가지 아키텍처(architectures) 나란히 비교: GPT-5.4의 통합 모델(unified model) 대 Claude Code의 심층 에이전트 런타임(deep agent runtime)*

이들은 같은 축에서 경쟁하지 않습니다. 하나는 광범위함(breadth)을 최적화합니다 — 하나의 모델(model)에서 모든 것을 적절하게 수행합니다. 다른 하나는 깊이(depth)를 최적화합니다 — 풍부한 에이전트 레이어(agent layer)를 통해 복잡한 소프트웨어 엔지니어링(software engineering)을 탁월하게 수행합니다.

그 차이는 일상적인 사용에서 드러납니다. Codex CLI는 오픈소스(open-source)(Rust 기반), 빠르며, 깔끔한 3단계 권한 모델(permission model)을 가집니다: 제안(Suggest), 자동 편집(Auto-Edit), 완전 자동(Full Auto). 작업을 지정하면 빠르게 실행됩니다. Claude Code는 더 많은 초기 투자(CLAUDE.md 파일, 훅(hook) 구성, 서브 에이전트(sub-agent) 설정)를 필요로 하지만, 일단 설정되면 프런트엔드(frontend), 백엔드(backend), 데이터베이스(database), 문서(docs)를 단일 워크플로(workflow)에서 다루는 다중 파일(multi-file) 기능을 처리합니다.

*이미지를 보려면 Enter를 누르거나 클릭하세요.*
*글쓴이 제공 이미지 — 어떤 도구가 어떤 작업에 적합할까요? 실용적인 라우팅(routing) 의사결정 트리(decision tree).*

그들이 최적화하는 엔지니어링(engineering) 문화도 다릅니다. Codex CLI는 속도, 낮은 마찰, 빠른 반복을 위한 “빠른 인턴(fast intern)”을 원하는 팀(teams)에 적합합니다. Claude Code는 아키텍처(architectural) 추론(reasoning), 복잡한 리팩토링(refactoring), 심층적인 코드베이스(codebase) 이해를 위한 “선임 개발자(senior developer)”가 필요한 팀에 적합합니다.

솔직한 표현: 이것은 “누가 더 나은가”의 질문이 아닙니다. “어떤 문제를 해결하고 있는가”의 질문입니다.

*이미지를 보려면 Enter를 누르거나 클릭하세요.*
*글쓴이 제공 이미지 — 철학적 분할: 플랫폼 플레이(platform play) 대 인프라 플레이(infrastructure play).*

주목할 한 가지 신호: 수렴(convergence)입니다. Codex CLI는 최근 MCP 지원과 스킬(skills) 시스템을 추가했습니다. Claude Code는 최근 /fast 모드(mode)와 속도 최적화(speed optimizations)를 추가했습니다. 둘 다 서로의 강점 쪽으로 나아가고 있습니다. 2026년의 질문은 아키텍처(architectural) 분할이 심화될 것인가 — 아니면 두 도구 모두 생태계(ecosystem) 종속성(lock-in)만으로 차별화되며 동일한 작업을 수행하게 될 것인가입니다.

두 철학 모두에 언급할 만한 맹점(blind spot)이 있습니다. OpenAI와 Anthropic 모두 세상을 “우리의 최첨단 모델(frontier model)이 필요하다”고 규정합니다. 그러나 분류(classification), 요약(summarization), 추출(extraction), 고객 대면 채팅(customer-facing chat), 구조화된 데이터 파이프라인(structured data pipelines)과 같은 대부분의 실제 응용 프로그램은 최첨단 모델을 전혀 필요로 하지 않습니다. 작고 전문화된 오픈 가중치(open-weight) 모델(models)은 종종 더 나은 지연 시간(latency), 더 낮은 비용, 더 엄격한 배포 제어(deployment control), 그리고 특정 작업에 대해 동등한 품질을 제공합니다. GPT-5.4와 Claude Code 간의 코딩 에이전트(coding agent) 싸움은 현실이지만, 그것은 훨씬 더 큰 그림의 한 조각에 불과하며, 여기서 올바른 답은 종종 작업에 맞게 미세 조정(fine-tuned)된 더 작은 모델(model)이지, 사용 가능한 가장 큰 모델이 아닙니다.

## 각 모델이 실제로 승리하는 지점 (그리고 "최고의 모델"이 틀린 질문인 이유)

OpenAI도 Anthropic의 마케팅(marketing)도 당신에게 말해주지 않을 불편한 진실이 있습니다: 단일 모델(model)이 모든 것을 이기는 경우는 없습니다. GPT-5.4도, Opus 4.6도, Gemini 3.1 Pro도 마찬가지입니다. 벤치마크(benchmark)는 이를 명확히 보여줍니다 — 각 모델(model)은 다른 카테고리(categories)에서 최고이며, 그 격차는 종종 오차 범위(margin of error) 내에 있습니다.

하지만 벤치마크(benchmark)가 전혀 포착하지 못하는 두 번째 레이어(layer)가 있습니다: 런타임(runtime) — 모델(model) 위에 있는 에이전트 레이어(agent layer)입니다. 그리고 여기서 논의가 혼란스러워집니다. 사람들은 “Claude Code가 다중 파일(multi-file) 리팩토링(refactoring)에서 승리한다”고 말합니다. 이는 부분적으로 Opus 4.6 모델(model) 덕분이지만, 대부분은 Claude Code 런타임(runtime)(서브 에이전트(sub-agents), 훅(hooks), MCP 컨텍스트(context)) 덕분입니다. 이 두 레이어(layer)를 분리하는 것이 중요합니다. 왜냐하면 모델은 상호 교환 가능해지고 있기 때문입니다. 런타임은 그렇지 않습니다.

*글쓴이 제공 이미지 — 솔직한 스코어카드(scorecard): 모델 수준의 승리 대 런타임 수준의 승리, 명확성을 위해 분리되었습니다.*

**모델 수준의 승리 — GPT-5.4:**

*   **토큰(token) 효율성.** 동일한 작업에서 약 4배 적은 토큰. 100만 토큰당 2.50달러/15달러 대 Opus 4.6의 5달러/25달러로, 고용량 팀(high-volume teams)에게는 실제적인 비용 격차가 있습니다.
*   **컴퓨터 사용(Computer use).** OSWorld에서 75% — 런타임(runtime) 기능이 아니라 GPT-5.4 자체에 내장된 모델(model) 기능입니다.
*   **간단한 작업에서의 속도.** 빠른 수정, 스크립트(scripts), 프로토타입(prototypes)에서 3~5배 빠른 추론(inference). 이는 모델 수준의 처리량(throughput)입니다.

**모델 수준의 승리 — Opus 4.6:**

*   **복잡한 코딩(coding).** SWE-bench Verified에서 80.8%. 모델(model) 자체가 다중 파일(multi-file) 코드베이스(codebases)에서 더 잘 추론(reasoning)합니다.
*   **시각적 추론(Visual reasoning).** MMMU Pro에서 85.1% — 세 가지 최첨단 모델(frontier models) 중 가장 높습니다.
*   **심층 아키텍처(architectural) 추론(reasoning).** 속도보다 철저함이 중요한 작업에서, Opus 4.6의 확장된 사고는 더 신뢰할 수 있는 결과물을 생성합니다.

**모델 수준의 승리 — Gemini 3.1 Pro:**

*   **과학적 추론(Scientific reasoning).** GPQA Diamond에서 94.3% — 세 가지 모델(model) 중 가장 높은 점수입니다.
*   **추상적 추론(Abstract reasoning).** ARC-AGI-2에서 77.1%.
*   **컨텍스트 창(Context window).** 100만 토큰(1M tokens) 네이티브(native), 처음 272K에 대해서는 이중 청구 없음.

**런타임 수준의 승리 — Codex CLI:**

*   **오픈소스(Open-source).** Rust 코드베이스(codebase), 감사 가능, 확장 가능.
*   **GitHub 네이티브(GitHub-native).** PR 자동 검토(auto-review), CI/CD 통합(integration), GitHub 워크플로(workflow)에 최적화.
*   **낮은 마찰.** 3단계 권한 모델(permission model)(제안(Suggest), 자동 편집(Auto-Edit), 완전 자동(Full Auto)). 최소한의 구성.

**런타임 수준의 승리 — Claude Code:**

*   **에이전트 오케스트레이션(Agent orchestration).** 다섯 가지 서브 에이전트(sub-agent) 유형, 병렬 세션(parallel sessions)을 위한 에이전트 팀(Agent Teams), 라이프사이클(lifecycle) 제어를 위한 훅(hooks). 다른 코딩(coding) 런타임(runtime)은 이 깊이에 미치지 못합니다.
*   **MCP(Model Context Protocol) 생태계(ecosystem).** 9,000개 이상의 플러그인(plugins), 35만 개 이상의 색인된 스킬(skills). Jira, Slack, Google Docs, Confluence — 모두 세션(session) 중간에 실시간 컨텍스트(live context)로 사용 가능.
*   **엔터프라이즈(enterprise) 워크플로(workflow) 장악.** CLAUDE.md 컨벤션(conventions), 도구별 권한, 심층 구성. 일단 설정되면, 전환 비용(switching costs)은 현실이 됩니다.

**동점 영역** — 그리고 이들은 계속 증가하고 있습니다: 단일 파일(single files)에서의 일반 코딩(coding), 표준 추론(reasoning) 작업, 작은 PR(Pull Request)에 대한 코드 리뷰(code review). 여기서는 모델(model)들이 충분히 비슷하여 벤치마크(benchmark)보다 개인적인 선호도와 생태계(ecosystem) 적합성이 더 중요합니다.

*2026년, 오차 범위(margin of error) 내에 있는 세 가지 최첨단 모델(frontier models)이 있는 상황에서, 현명한 투자는 이 모든 모델을 사용하는 오케스트레이션 레이어(orchestration layer)에 있습니다.*

더 넓게 보면, 스코어카드(scorecard) 자체는 구조적 변화를 보여줍니다: 모델(model)은 실시간으로 상품화(commoditizing)되고 있습니다.

18개월 전에는 하나의 모델(model)을 선택하고 그 주위에 구축했습니다. 오늘날, 제가 이야기하는 똑똑한 팀(teams)들은 다른 방식으로 일하고 있습니다 — 각 모델(model)이 가장 잘하는 것에 따라 작업을 다른 모델로 라우팅(routing)하고 있습니다. 빠른 스크립트(script)요? GPT-5.4. 복잡한 리팩토링(refactoring)요? Claude Code의 런타임(runtime)을 통한 Opus 4.6. 연구 종합이요? 100만 토큰(1M token) 컨텍스트 창(context window)을 가진 Gemini 3.1 Pro. 각 모델은 보편적인 솔루션이 아니라 전문화된 엔진(engine)입니다.

하지만 최첨단 모델(frontier model) 스코어카드(scorecard)가 보여주지 않는 것이 있습니다: 대부분의 프로덕션(production) 애플리케이션(applications)은 최첨단 모델을 전혀 필요로 하지 않습니다. 이 글이 초점을 맞추는 코딩 에이전트(coding agent) 사용 사례는 가장 크고 유능한 모델로부터 진정으로 이점을 얻습니다. 다중 파일(multi-file) 추론(reasoning), 아키텍처(architectural) 계획, 복잡한 리팩토링(refactoring)은 모델(model)이 할 수 있는 것의 한계를 밀어붙입니다. 하지만 코딩 에이전트(coding agent)의 영역을 벗어나면, 풍경은 완전히 바뀝니다.

분류(classification), 요약(summarization), 추출(extraction), 구조화된 데이터 파이프라인(structured data pipelines), 고객 대면 채팅(customer-facing chat), 콘텐츠 검토(content moderation), 번역(translation) — 이러한 작업은 프로덕션(production) 환경에서 대규모로 실행되며, 더 작고 전문화된 모델(models)에서 가장 잘 작동합니다. 오픈 가중치(open-weight) 모델, 미세 조정(fine-tuned) 모델. 벤치마크(benchmark) 점수보다는 지연 시간(latency), 비용, 배포 유연성(deployment flexibility)에 최적화된 모델들입니다. 특정 추출 작업을 10ms의 지연 시간과 100만 토큰당 0.10달러의 비용으로 수행하는 잘 조정된 7B 모델(model)은 해당 사용 사례에서 100만 토큰당 25달러의 최첨단 모델(frontier model)을 능가할 것입니다 — 벤치마크(benchmark)에서가 아니라, 프로덕션(production)에서 중요한 지표(metrics)들, 즉 쿼리(query)당 비용, p99 지연 시간, 배포 제어(deployment control)에서 말입니다.

상품화(commoditization) 이론은 빅 3를 넘어 확장됩니다. 단순히 “GPT-5.4 대 Opus 4.6 대 Gemini 3.1 Pro”가 아닙니다. 그것은 “최첨단 작업(frontier tasks)을 위한 최첨단 모델(frontier models), 그 외 모든 것을 위한 전문화된 모델(specialized models)”입니다. 가장 견고한 프로덕션(production) 시스템(systems)을 구축하는 팀(teams)은 OpenAI와 Anthropic 중에서 선택하는 것이 아닙니다 — 그들은 최첨단 모델(frontier model)이 어려운 추론(reasoning)을 처리하고 더 작은 모델(model)이 필요하지 않은 80%의 작업을 처리하는 모델 불가지론적(model-agnostic) 아키텍처(architectures)를 구축하고 있습니다.

이것은 3절의 상품화(commoditization) 추세의 논리적 결론입니다. 모든 최첨단 모델(frontier model)이 유사한 기능으로 수렴(converges)한다면 — 그리고 실제로 수렴하고 있습니다 — “최고의 모델(model)”이라는 질문은 더 이상 유용하지 않습니다. 답은 항상 “무엇을 위해?”입니다. 그리고 대부분의 프로덕션(production) 워크로드(workloads)에 대한 답은 가장 큰 모델(model)이 아닙니다.

*글쓴이 제공 이미지 — 작업 라우팅 매트릭스(Task routing matrix): 최대 출력을 위해 작업 유형을 모델 강점과 일치시킵니다.*

지금 가장 큰 가치를 얻고 있는 팀(teams)들은 하나의 모델(model)이나 하나의 런타임(runtime)에 충성하지 않습니다. 그들은 에이전트 런타임(agent runtime)이 특정 작업을 가장 잘 처리하는 모델(model)로 라우팅(routes)하는 모델 불가지론적(model-agnostic) 워크플로(workflows)를 구축하고 있습니다. MCP와 SKILL.md는 프로토콜(protocol) 수준에서 이를 가능하게 합니다. 런타임(runtime) 레이어(layer)는 이를 실용적으로 만듭니다.

이것이 진정한 변화입니다. “GPT-5.4가 Claude Code를 죽였다”거나 “Claude Code가 여전히 왕이다”가 아닙니다. 변화는 이렇습니다: 모든 것을 지배하는 하나의 모델(model) 시대가 끝나고 있습니다. 지능형 작업 라우팅(intelligent task routing)의 시대가 시작되고 있습니다. 그리고 승리하는 런타임(runtimes)은 그 라우팅(routing)을 원활하게 만드는 것들이 될 것입니다 — 그들의 모델(model)이 유일하게 필요한 것이라고 주장하는 것들이 아닙니다.

## 제 워크플로(Workflow)에서 실제로 바꿀 것들

그렇다면 모든 벤치마크(benchmark), 아키텍처(architecture) 다이어그램(diagrams), 그리고 상품화(commoditization) 이론을 거쳐 — 월요일부터 제가 실제로 다르게 할 일은 무엇일까요?

짧게 답하자면: 저는 모든 작업에 하나의 도구만 사용하는 것을 멈출 것입니다.

다음은 제 현재 설정의 솔직한 버전과 제가 변경할 부분입니다 — 모델(model) 결정과 런타임(runtime) 결정을 분리하여 설명합니다.

*이미지를 보려면 Enter를 누르거나 클릭하세요.*
*글쓴이 제공 이미지 — 이전 대 이후: 단일 도구 워크플로(single-tool workflow)에서 작업 라우팅 하이브리드(task-routed hybrid)로.*

**Claude Code에 남겨둘 것들 (런타임(runtime) 이점):**

*   **다중 파일(multi-file) 기능.** 프런트엔드(frontend) + 백엔드(backend) + 테스트(tests) + 문서(docs)를 한 번에 처리하는 모든 것. 이것은 Claude Code의 서브 에이전트(sub-agent) 아키텍처(architecture)가 제 역할을 하는 부분입니다 — 단순히 Opus 4.6 모델(model) 때문이 아니라, 적절한 CLAUDE.md 구성으로 단일 세션(session)에서 12개 파일(files)에 걸쳐 변경 사항을 조율하는 런타임(runtime) 덕분입니다. Codex CLI의 런타임은 아직 이를 오케스트레이션(orchestrate)할 수 없습니다.
*   **MCP(Model Context Protocol) 의존 세션(sessions).** Jira 컨텍스트(context), Slack 스레드(threads), Google Docs가 모두 코딩(coding) 세션(session)으로 유입되어야 할 때, Claude Code의 9,000개 이상의 플러그인(plugin) 생태계(ecosystem)는 있으면 좋은 것이 아니라 — 핵심 인프라(infrastructure)입니다. 이는 런타임(runtime) 이점이지, 모델(model) 이점이 아닙니다.
*   **아키텍처(architecture) 결정.** 마이그레이션(migration) 계획, 의존성 분석(dependency analysis), “이 서비스를 분리해야 하는가”와 같은 논의. 여기서는 둘 다 해당됩니다 — 100만 출력 토큰당 25달러의 Opus 4.6 모델(model)의 철저함과 Claude Code의 심층 컨텍스트 관리(context management)가 결합됩니다.

**GPT-5.4 + Codex CLI로 옮길 것들 (모델(model) 이점 + 런타임(runtime) 적합성):**

*   **빠른 스크립트(scripts) 및 일회성 수정.** GPT-5.4 모델(model)은 3~5배 빠릅니다. “이 로그 파일(log files)을 정리할 bash 스크립트(script)를 작성해 줘” 또는 “이 off-by-one 오류를 수정해 줘”와 같은 경우, 모델의 속도 이점은 분명하며 Codex CLI의 낮은 마찰 런타임(runtime)과 잘 맞습니다.
*   **중소 규모 PR(Pull Request)에 대한 PR 리뷰(reviews).** 더 낮은 토큰(token) 비용(모델(model) 이점)과 Codex CLI의 GitHub 네이티브(GitHub-native) 통합(integration)(런타임(runtime) 이점)이 결합되어 대량 작업에 더 나은 조합을 이룹니다.
*   **컴퓨터 자동화(automation) 작업.** GUI 테스트(testing), 양식 작성(form filling), 데스크톱(desktop) 앱 스크립팅(scripting) — GPT-5.4 모델(model)은 OSWorld에서 75%를 기록합니다. 이는 Claude Code의 터미널 네이티브(terminal-native) 런타임(runtime)이 제공하지 않는 진정한 모델(model) 수준의 기능입니다.

**최첨단 모델(frontier model)이 전혀 필요 없는 것들:**

*   **분류(classify), 추출(extract), 또는 요약(summarize)하는 프로덕션(production) API.** 문서에서 구조화된 필드(structured fields)를 추출하는 데이터 파이프라인(data pipeline)을 실행 중이라면, 미세 조정(fine-tuned)된 더 작은 모델(model)이 어떤 최첨단 모델(frontier model)보다 빠르고 저렴하며 배포하기 쉬울 것입니다. 100만 토큰당 15달러짜리 모델을 100만 토큰당 0.10달러짜리 작업에 사용하지 마세요.
*   **특정 도메인(scoped domains)을 가진 고객 대면 채팅(customer-facing chat).** 초점을 맞춘 지식 기반(knowledge base)을 가진 잘 조정된 오픈 가중치(open-weight) 모델(model)은 모든 것을 알지만 응답하는 데 3초가 걸리는 최첨단 모델(frontier model)보다 더 나은 지연 시간(latency)과 일관성(consistency)을 제공하는 경우가 많습니다.
*   **콘텐츠 검토(content moderation), 번역(translation), 구조화된 출력 생성(structured output generation).** 이들은 더 작은 모델(model) 티어(tier)에서 이미 해결된 문제입니다. 최첨단 모델(frontier models)은 품질 추가 없이 비용만 증가시킵니다.

이것이 GPT-5.4 대 Claude Code라는 틀이 완전히 놓치는 부분입니다. 네, 코딩 에이전트(coding agents)는 최첨단 추론(frontier reasoning)이 필요합니다. 하지만 이 에이전트(agents) 위에 구축하는 대부분의 소프트웨어(software)는 더 작고 전문화된 모델(models)을 통해 사용자에게 서비스를 제공할 것입니다. 아키텍처(architecture)는 이를 반영해야 합니다 — 개발을 위한 최첨단 모델, 프로덕션(production)을 위한 적정 규모 모델(right-sized models).

**커밋(commit)하기 전에 지켜볼 것들:**

*   **독립적인 벤치마크(benchmarks).** 이 글의 모든 수치는 벤더(vendors) 자체 또는 초기 제3자 보고서에서 나온 것입니다. GPT-5.4에 대한 Aider, BigCodeBench, LiveCodeBench 결과를 보기 전에는 벤치마크(benchmark)를 완전히 신뢰하지 않을 것입니다.
*   **대규모 토큰(token) 비용.** 4배 효율성은 훌륭해 보입니다 — 하지만 복잡한 작업에서도 유지될까요, 아니면 단순한 작업에서만 유지될까요? 핵심은 분포(distribution)에 있습니다.
*   **Codex CLI의 MCP(Model Context Protocol) 성숙도.** 아직 새롭습니다. Claude Code의 MCP는 몇 달 동안 실전에서 검증되었습니다. 한 분기(quarter) 정도 지켜보세요.

*이미지를 보려면 Enter를 누르거나 클릭하세요.*
*글쓴이 제공 이미지 — 하이브리드 스택(hybrid stack): 복잡성과 컨텍스트(context) 필요에 따라 도구 간에 작업을 라우팅(routing)합니다.*

불편한 진실: 단일 도구(single-tool) 충성도의 시대는 끝났습니다. “최고이기 때문에” 모든 것에 하나의 모델(model)과 하나의 런타임(runtime)을 고집하는 것은 성능 손실을 초래합니다. 2026년 초에 가장 빠르게 출시하는 팀(teams)들은 이미 하이브리드(hybrid) 설정(setups)을 운영하고 있습니다 — 작업에 적합한 모델(model)을 선택하고(속도에는 GPT-5.4, 깊이에는 Opus 4.6, 연구에는 Gemini 3.1 Pro), 워크플로(workflow)에 적합한 런타임(runtime)을 선택합니다(GitHub 네이티브(GitHub-native) 속도에는 Codex CLI, 오케스트레이션(orchestrated) 깊이에는 Claude Code).

제가 하지 않을 한 가지: 출시일 블로그(blog) 게시물만을 기반으로 기존 Claude Code 워크플로(workflow)를 GPT-5.4로 마이그레이션(migrate)하는 것입니다. 벤치마크(benchmark)는 독립적인 검증이 필요합니다. MCP(Model Context Protocol) 지원은 프로덕션(production) 테스트(testing)가 필요합니다. 토큰(token) 효율성 주장은 선별된 데모(demos)가 아닌 복잡한 작업에 대한 실제 검증이 필요합니다.

하이브리드(hybrid)를 구축하세요. 작업 유형별로 라우팅(route)하세요. 어떤 것에든 올인(all-in)하기 전에 독립적인 데이터를 기다리세요. 그것이 솔직한 조언입니다.
