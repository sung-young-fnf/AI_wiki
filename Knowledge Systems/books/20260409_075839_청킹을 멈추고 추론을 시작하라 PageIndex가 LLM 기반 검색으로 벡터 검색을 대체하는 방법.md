청킹을 멈추고 추론을 시작하라: PageIndex가 LLM 기반 검색으로 벡터 검색을 대체하는 방법

표준 RAG(Retrieval Augmented Generation) 파이프라인은 문서를 청킹(chunking)하고 임베딩(embedding)한 다음 유사성으로 검색합니다. 그러나 유사성이 곧 관련성은 아니며, 밀도 높은 전문 문서의 경우 이러한 간극이 정확성을 떨어뜨리는 지점입니다.

기본 RAG 스택은 지난 2년간 거의 변하지 않았습니다. 문서를 가져와 청크(chunk)로 분할하고, 각 청크를 임베딩하며, 벡터를 저장한 다음, 쿼리(query) 시점에 상위 k개의 가장 유사한 것을 검색하는 방식입니다. 이는 블로그 게시물이나 FAQ 페이지에는 충분히 잘 작동하지만, 200페이지 분량의 SEC 공시 서류, 규제 제출 문서, 기술 매뉴얼, 법률 계약서와 같이 기업에서 가장 중요한 문서들에서는 한계를 드러냅니다.

실제 표준 파이프라인은 다음과 같으며, 정확히 이러한 지점에서 문제가 발생합니다. 이는 쿼리(query)와 임베딩(embedding)이 가장 유사한 다섯 개의 청크(chunk)를 검색합니다. 그러나 '가장 가까운 것'이 '정확한 것'은 아닙니다. 질문이 총 가치를 묻는 경우, 가장 유사한 청크는 이연 자산이 어떻게 증가했는지 설명할 수 있습니다. 이는 의미론적으로는 유사하지만 사실과는 다릅니다. 실제 총계는 77페이지에서 "자세한 통계표는 부록 G를 참조하십시오"라는 문구로 참조된 부록 G에 있습니다. 어떤 임베딩 모델도 이러한 상호 참조(cross-reference)를 쿼리와 연결할 수 없습니다. 이러한 연결은 유사성이 아닌 추론을 요구하기 때문입니다.

10-K 보고서를 읽는 재무 분석가는 자신의 질문과 가장 유사해 보이는 단락을 검색하지 않습니다. 그들은 목차를 훑어보고, 어떤 섹션에 답이 있을지 추론하며, 해당 섹션으로 이동하여 내용을 읽고, 상호 참조를 따라갑니다. PageIndex는 바로 이러한 과정을 재현합니다.

### PageIndex의 실제 작동 방식

PageIndex는 전체 청크-임베딩-검색 파이프라인을 두 가지 단계로 대체합니다. 문서에서 계층적 트리(tree)를 구축한 다음, LLM이 해당 트리를 기반으로 추론하여 답을 찾는 방식입니다.

**1단계: 트리 구축(Tree construction)**
PageIndex는 PDF를 파싱(parsing)하여 JSON 트리(tree)를 생성합니다. 이는 요약, 페이지 범위, 중첩된 하위 섹션을 포함하는 기계가 읽을 수 있는 목차입니다.

각 `node_id`는 해당 페이지의 원본 콘텐츠(텍스트, 이미지 또는 표)에 직접 매핑됩니다. 이 트리는 문서를 512 토큰 조각으로 찢는 대신 문서의 자연스러운 계층 구조를 보존합니다. 임베딩도, 벡터 데이터베이스도 사용하지 않으며, 인덱스는 LLM의 컨텍스트(context) 창 내부에 존재하는 순수한 JSON입니다. PageIndex 팀은 이를 모델이 추론(inference) 중에 직접 참조하고 탐색하며 추론할 수 있기 때문에 '인-컨텍스트 인덱스(in-context index)'라고 부릅니다.

**2단계: 에이전트 기반 검색(Agentic retrieval)**
사용자가 질문을 하면 LLM은 트리를 읽고 다음과 같은 반복적인 루프(loop)를 따릅니다.
1.  답을 포함할 가능성이 있는 섹션을 식별합니다.
2.  가장 유망한 섹션의 원본 콘텐츠를 검색합니다.
3.  관련 정보를 추출합니다.
4.  질문에 답하거나 트리로 돌아가 다른 섹션을 시도합니다.

### 시작하기: 자체 호스팅

PageIndex를 로컬에서 실행하는 가장 빠른 방법은 자체 호스팅(self-hosted) 방식입니다. 이는 오픈소스 저장소를 사용하여 어떤 PDF든 트리 구조를 생성합니다. 기본 모델은 `gpt-4o-2024-11-20`이며, 선택적 매개변수(parameter)로 동작을 사용자 정의할 수 있습니다.

PageIndex는 또한 마크다운(Markdown) 입력을 직접 지원하여, PDF를 이미 변환했거나 구조화된 문서를 인덱싱하려는 경우 유용합니다. 마크다운 경로와 관련하여 한 가지 참고할 점은 `#` 헤딩(heading) 레벨을 사용하여 노드(node) 계층 구조를 결정한다는 것입니다. 만약 마크다운이 PDF나 HTML에서 기계적으로 변환된 경우, 헤딩이 종종 신뢰할 수 없습니다. 이러한 경우 PageIndex 팀은 구조를 보존하는 마크다운을 생성하기 위해 먼저 OCR(광학 문자 인식) 도구를 사용한 다음, 해당 출력에 대해 트리 생성기를 실행할 것을 권장합니다.

자체 호스팅 방식은 트리 구성에 대한 완전한 제어를 제공하며, 결과는 검사, 수정, 그리고 모든 다운스트림(downstream) 파이프라인에서 사용할 수 있는 JSON 파일입니다.

### 시작하기: Python SDK (클라우드 API)

프로덕션(production) 워크로드(workload)의 경우, PageIndex API를 통해 문서를 제출하고, 트리를 생성하며, 추론 기반 QA(질의응답)를 실행할 수 있습니다. 문서가 처리되면 트리 구조를 직접 검색할 수 있습니다. 또는 트리를 건너뛰고 바로 질의응답으로 넘어갈 수도 있습니다. 챗(Chat) API는 내부적으로 전체 에이전트 기반 검색 루프를 실행합니다.

다중 문서 챗(multi-document chat) 기능은 전년 대비 보고서 비교, 여러 회사에 걸친 공시 상호 참조, 기간별 회계 정책 변경 사항 식별 등 재무 분석에 특히 강력합니다.

### 시작하기: MCP 통합

PageIndex는 MCP(Multi-protocol Chat Protocol) 서버를 제공하여, 모든 MCP 호환 클라이언트(Claude Desktop, Cursor 또는 모든 에이전트 프레임워크)에 추론 기반 검색을 노출합니다. 이를 통해 컨텍스트 길이 제한에 부딪히지 않고 기존 도구에서 긴 PDF와 직접 대화할 수 있습니다.

로컬 서버는 파일 업로드와 인증을 자동으로 처리합니다. 직접 HTTP 연결은 URL을 통한 PDF만 지원합니다(로컬 파일 업로드 불가). 두 옵션 모두 에이전트(agent)에 동일한 추론 기반 검색을 제공합니다. 즉, LLM이 벡터 유사성에 의존하는 대신 문서의 트리 구조를 탐색합니다.

### 왜 벡터 검색은 전문 문서에서 한계를 드러내는가

문제는 나쁜 임베딩 모델이나 잘못된 청크(chunk) 크기가 아닙니다. 이는 청크-임베딩-검색 아키텍처(architecture) 자체에 내재된 다섯 가지 구조적 한계 때문입니다.

**쿼리-지식 공간 불일치:** 질문 "회사의 부채 추세는 어떻습니까?"는 내용이 아닌 의도를 표현합니다. 답변은 '장기 의무 및 신용 시설(Long-Term Obligations and Credit Facilities)'이라는 제목의 섹션에 있는데, 이는 쿼리와 표면적인 언어를 거의 공유하지 않습니다. 임베딩은 이러한 간극을 메울 수 없습니다. 해당 연결은 코사인 유사성이 아닌 도메인(domain) 추론을 요구하기 때문입니다.

**유사성이 곧 관련성은 아님:** 도메인(domain)별 문서에서는 많은 구절이 거의 동일한 의미론을 공유하지만 관련성에서 결정적으로 다릅니다. 예를 들어, 10-K 공시 서류에 수익 인식에 관한 수십 개의 단락이 있을 수 있지만, FY2023년 정책 변경에 대한 질문에 답하는 단락은 하나뿐입니다. 임베딩은 이들을 모두 거의 동일하게 평가합니다.

**하드 청킹(Hard chunking)은 컨텍스트(context)를 파괴함:** 문서를 512 토큰 청크(chunk)로 분할하면 문장, 표, 논리적 섹션을 가로지르게 됩니다. 두 개의 청크에 걸쳐 있는 표는 두 개의 무의미한 조각이 되며, 각주를 참조하는 단락은 각주가 다른 청크에 있을 때 그 참조를 잃게 됩니다.

**대화 메모리 없음:** 각 벡터 쿼리(query)는 독립적입니다. 사용자가 이전 턴에서 "금융 자산"에 대해 물었고 이제 "부채는 어떻습니까?"라고 묻는다면, 검색기는 동일한 보고서의 동일한 섹션을 찾아야 한다는 것을 알 방법이 없습니다.

**상호 참조(Cross-references) 소실:** 문서에 "표 5.3은 수입과 지출을 요약합니다. 자세한 통계표는 부록 G를 참조하십시오"라고 되어 있을 때, 벡터 검색기는 그 참조를 따를 메커니즘이 없습니다. 답은 부록 G에 있지만, 그곳에는 원본 질문과 의미론적으로 유사한 것이 없습니다. PageIndex 팀의 자체 테스트에서 이와 똑같은 시나리오가 나타났습니다. 총 이연 자산에 대한 쿼리는 증가분만 보고하는 섹션(75-82페이지)에 도달했습니다. 77페이지는 총계를 위해 부록 G를 참조했습니다. 추론 기반 검색기는 그 단서를 따랐지만, 벡터 검색기는 높은 확신으로 잘못된 섹션을 반환했습니다.

### PageIndex는 RAPTOR와 어떻게 다른가

RAPTOR(Recursive Abstractive Processing for Tree-Organized Retrieval, 스탠퍼드 대학교, ICLR 2024) 또한 트리를 구축하지만, 두 시스템은 근본적으로 다른 방식으로 작동합니다. RAPTOR는 전통적인 청킹으로 시작하여 청크를 임베딩하고, 가우시안 혼합 모델(Gaussian Mixture Models)을 사용하여 의미론적 유사성으로 클러스터링하며, 각 클러스터를 요약한 후 이 과정을 재귀적으로 반복하여 아래에서 위로 트리를 구축합니다. 검색 시에는 여전히 트리의 모든 레벨에서 벡터 유사성 검색을 사용합니다. RAPTOR는 인덱싱되는 방식을 개선하지만, 검색은 여전히 임베딩 기반으로 남아 있습니다.

PageIndex는 청킹과 임베딩을 완전히 건너뜁니다. 문서의 고유한 구조를 파싱(parsing)하여 트리를 구축하고, 검색 시 LLM이 추론을 통해 트리를 탐색합니다. 어떤 단계에서도 벡터 검색이 없습니다.

실질적인 차이점은 RAPTOR는 여전히 벡터 데이터베이스와 임베딩 모델을 필요로 하는 반면, PageIndex는 LLM과 문서만 필요로 한다는 점입니다. RAPTOR는 유사한 노드(node)를 찾지만, PageIndex는 찾고 있는 것에 따라 다음에 어떤 노드를 방문할지 결정합니다.

### 더 넓은 추세: 추론을 통한 검색

PageIndex는 임베딩 기반 검색에서 벗어나려는 더 큰 흐름의 일부입니다.

크로마(Chroma)의 '컨텍스트 로트(Context Rot)' 연구(2025년 7월)는 GPT-4.1, Claude 4, Gemini 2.5, Qwen3를 포함한 18개의 최첨단 LLM을 테스트한 결과, 입력 길이가 길어질수록 간단한 작업에서도 모델 성능이 크게 저하된다는 것을 발견했습니다. 이 결과는 '모든 것을 긴 컨텍스트(context) 창에 쏟아붓는 것'이 좋은 검색의 신뢰할 수 있는 대안이 아니며, 컨텍스트에 무엇이 들어가는지가 얼마나 많이 들어가는지보다 더 중요하다는 것을 강조합니다.

클로드 코드(Claude Code) 팀도 다른 방향에서 유사한 결론에 도달했습니다. 클로드 코드의 개발자인 보리스 체르니(Boris Cherny)는 2025년 5월 레이턴트 스페이스(Latent Space) 팟캐스트 인터뷰에서 초기 버전은 로컬 벡터 데이터베이스를 사용하는 RAG를 사용했지만, `grep` 및 파일 읽기 같은 도구를 사용하여 모델이 탐색하도록 하는 에이전트 기반 검색(agentic search)으로 전환했는데, 이는 이 방식이 모든 것을 능가했기 때문입니다. 그 이유는 정확성을 넘어섰는데, 단순성, 데이터 불일치 없음(no staleness), 외부 인덱스로 인한 보안 위험 없음, 동기화(sync) 드리프트(drift)로 인한 신뢰성 문제 없음 등이었습니다.

PageIndex는 동일한 원칙을 문서에 적용합니다. 즉, 외부 유사성 인덱스를 미리 계산하는 대신, 모델 자체의 추론이 검색을 처리하도록 합니다. 트리 구조는 LLM에게 추론할 대상을 제공합니다. 이는 임베딩을 필요로 하지 않으면서 효율적인 탐색을 가능하게 하는 문서의 구조화된 지도입니다.

### 상충 관계 (Tradeoffs)

추론 기반 검색은 무료가 아니며, 공정한 평가를 위해서는 그에 따른 비용을 명시해야 합니다.

**토큰 소비:** 트리 구성은 문서를 파싱(parsing)하고, 요약하며, 구조화하기 위해 여러 번의 LLM 호출을 필요로 합니다. 에이전트 기반 검색은 쿼리(query)당 여러 추론 단계(트리 읽기, 노드 선택, 콘텐츠 추출, 잠재적 반복)를 거쳐야 합니다. 반면 벡터 검색은 단일 임베딩 호출로 밀리초(millisecond) 단위로 결과를 반환합니다. PageIndex는 정확성을 높이는 대신 지연 시간(latency)과 비용이 증가하는 상충 관계(tradeoff)를 가집니다.

**규모:** PageIndex는 개별 문서나 소규모 컬렉션, 즉 재무 분석, 법률 검토 또는 규제 준수와 같은 시나리오에 탁월하게 작동합니다. 하지만 수천 개의 문서를 검색하는 경우, 벡터 데이터베이스가 여전히 명확한 이점을 가집니다. 벡터 데이터베이스는 순차적인 LLM 추론으로는 따라올 수 없는, 백만 개 문서 코퍼스(corpus)를 1초 미만에 검색할 수 있습니다.

**LLM 의존성:** 검색 품질은 전적으로 기본 모델의 추론 능력에 달려 있습니다. 트리를 탐색하는 모델의 성능이 약하면 어떤 섹션을 방문할지에 대해 더 나쁜 결정을 내릴 것입니다. 이는 검색 품질이 임베딩 품질에 달려 있는 벡터 검색과는 근본적으로 다른 실패 모드입니다.

**트리 구축 비용:** 트리 구축은 문서당 한 번 발생하는 비용이지만, 임베딩보다 더 비쌉니다. 자주 변경되는 코퍼스(corpus)의 경우, 트리 재구축 비용이 중요하게 작용합니다.

솔직히 말해, 벡터 RAG는 간단한 문서에 충분히 잘 작동하는 빠르고 저렴한 근사치입니다. PageIndex는 '충분히 잘 작동하는 것'으로는 부족한 문서에 대한 더 느리고, 더 비싸며, 더 정확한 접근 방식입니다. 이 두 가지 접근 방식은 서로 보완할 수도 있습니다. 벡터 검색이 대규모 코퍼스(corpus)를 관련 문서로 좁히면, PageIndex가 해당 특정 문서를 기반으로 추론하여 정확한 답변을 제공하는 식입니다.

### 어디서 시작해야 할까

중요한 문서 하나를 선택하십시오. 예를 들어 10-K 공시 서류, 규제 제출 문서, 100페이지 분량의 기술 매뉴얼 등 현재 RAG 파이프라인이 부정확한 답변을 제공하는 문서가 좋습니다.

완전한 제어를 원한다면 오픈소스 저장소를 복제(clone)하고, 트리를 생성하며, JSON 출력을 검사하십시오. 어떤 파이프라인에 연결하기 전에 계층 구조를 이해하는 것이 중요합니다.

빠른 결과를 원한다면 `pip install pageindex`를 실행하고, SDK를 통해 문서를 제출한 다음 챗(Chat) API를 사용하십시오. 에이전트 기반 검색 루프는 서버 측에서 실행되므로, 인프라를 직접 구축하지 않고도 추론 기반 답변을 얻을 수 있습니다.

기존 도구와 통합하고 싶다면 Claude Desktop 또는 Cursor에 MCP 서버를 추가하십시오. 컨텍스트 제한을 초과하는 PDF를 업로드하고 섹션 간 상호 참조를 요구하는 질문을 하십시오. 바로 그곳에서 PageIndex의 차이점을 확인할 수 있을 것입니다.

표준 RAG 파이프라인이 사라진 것은 아닙니다. 하지만 표, 부록, 상호 참조, 그리고 규제적 정확성 요구 사항을 포함하는 가장 중요한 문서의 경우, '그냥 청킹(chunking)하고 임베딩(embedding)하라'는 시대가 더 정확한 방식으로 대체되고 있습니다. LLM은 인간 전문가가 문서를 읽는 방식과 동일하게 읽을 수 있으며, 이는 어떤 임베딩 모델이 근사할 수 있는 것보다 더 나은 것으로 판명되었습니다.
