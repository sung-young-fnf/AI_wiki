# Kuzu 코드베이스 16만 3천 라인 분석: Apple이 이를 원했던 이유
10인 스타트업이 Apple의 온디바이스 AI 전략에 부족했던 그래프 엔진을 구축한 방법

저는 지난주에 Apple이 2025년 10월 조용히 인수한 임베디드 그래프 데이터베이스(embedded graph database)인 Kuzu를 구성하는 16만 3천 줄의 C++20 코드를 읽으며 시간을 보냈습니다.

이 인수는 EU 디지털 시장법(Digital Markets Act, DMA) 신고를 통해 드러났습니다. 10명의 직원, 온타리오주 워털루(Waterloo Ontario)에 기반을 둔 회사였고, 인수가 발표된 날 GitHub 저장소는 비공개 처리되었습니다.

모든 기술 언론은 Apple이 iWork에 데이터베이스를 추가할 것인가라는 동일한 관점으로 보도했습니다. 하지만 그러한 해석은 이 코드베이스에 실제로 담긴 내용을 놓치고 있습니다.

이 글을 마치면, 여러분은 이 인수가 우연이 아닌 전략적인 이유가 되는 세 가지 아키텍처 결정 — **팩터화된 실행(factorized execution)**, **최악의 경우 최적 조인(worst-case optimal joins)**, 그리고 **엄브라 스타일 저장 엔진(Umbra-style storage engine)** — 을 이해하게 될 것입니다.

이것들은 Neo4j나 다른 그래프 데이터베이스에 대한 점진적인 개선이 아닙니다.

이들은 Apple Intelligence가 직면한 정확한 문제, 즉 서버 없이 온디바이스(on-device)에서 여러분의 전체 디지털 라이프에 걸친 관계를 추론하는 문제를 해결하는 근본적인 설계 선택입니다.

## 코드베이스의 실제 내용

Kuzu는 임베디드 그래프 데이터베이스(embedded graph database)입니다.

서버 프로세스가 없습니다. SQLite처럼 인프로세스(in-process)로 실행됩니다.

구조화된 속성 그래프 모델(structured property graph model) 위에 Cypher 쿼리 언어(Cypher query language)를 사용하며, 28가지 논리적 타입(logical types)을 지원합니다. 여기에는 그래프 프리미티브(graph primitives) (NODE, REL, RECURSIVE_REL), INT128까지의 숫자 타입, 다섯 가지 타임스탬프 변형, 그리고 LIST, ARRAY, STRUCT, MAP, UNION을 포함한 중첩 타입(nested types)이 있습니다.

쿼리 파이프라인(query pipeline)은 교과서적인 데이터베이스 아키텍처를 훌륭하게 구현했습니다: ANTLR4 Cypher 파서(parser) → 의미 분석(semantic analysis)을 위한 바인더(binder) → 카디널리티 추정(cardinality estimation)을 포함한 DP 기반 조인 순서 플래너(join order planner) → 규칙 기반 옵티마이저(rule-based optimizer) → 물리적 플랜 매퍼(physical plan mapper) → 76개의 물리적 연산자(physical operators)를 갖춘 벡터화된(vectorized), 모셀 병렬(morsel-parallel) 프로세서.

하지만 세 가지 설계 결정이 Kuzu를 제가 검토했던 다른 모든 그래프 데이터베이스와 차별화합니다.

### 팩터화된 실행(Factorized Execution): 조합 폭발 문제 해결

그래프 쿼리(graph queries)에는 관계형 데이터베이스(relational databases)가 거의 직면하지 않는 구조적 문제가 있습니다. 두 번의 홉 탐색(two-hop traversal)을 예로 들어봅시다:

```cypher
MATCH (a)-[:KNOWS]->(b)-[:LIKES]->(c)
RETURN a.name, b.name, c.name
```

만약 `a1`이 100명을 알고 있고, 각 100명이 50가지 것을 좋아한다면, 단순한 실행기(naive executor)는 `a1.name`이 5,000번 중복된 5,000개의 행(rows)을 생성합니다. 실제 지식 그래프(knowledge graph)에서 세 번의 홉을 탐색하면 어떨까요? 중간 결과가 조합적으로 폭발하게 됩니다.

Kuzu는 **팩터화된 중간 표현(factorized intermediate representation)**을 통해 이를 완전히 방지합니다. 전체 데카르트 곱(Cartesian product)을 구체화(materializing)하는 대신, `a1.name`을 "평면(flat)" 값으로 한 번만 저장하고, `b`와 `c` 벡터는 "비평면(unflat)" 상태로 중복 없이 중첩(nested)하여 유지합니다:

```
팩터화된 표현(Factorized representation):
  평면(Flat):   a1.name           → 한 번 저장됨 (stored ONCE)
  비평면(Unflat): [b1, b2, ..., b100]  → 그룹당 한 번 저장됨 (stored once per group)
  비평면(Unflat): [c1..c50, c1..c50]   → 중첩 벡터, 중복 없음 (nested vectors, no duplication)
vs. 단순 구체화(Naive materialization):
  5,000행 × 3열 = 15,000 값 (대부분 중복)
```

**FactorizedTable**은 평면(flat) 및 비평면(unflat) 컬럼(columns)을 위해 별도의 저장 공간을 유지하며, 연산자(operators)들은 이 압축된 표현(compressed representation)을 전체 실행 파이프라인(execution pipeline)을 통해 전파합니다. 추천 탐색(recommendation traversals), 지식 그래프 추론(knowledge graph reasoning), 소셜 네트워크 쿼리(social network queries) — 그래프 작업 부하를 지배하는 다대다(many-to-many) 패턴 — 에 있어 이것은 밀리초와 분 사이의 차이를 만들어냅니다.

### 왜 삼각형 패턴이 대부분의 그래프 데이터베이스를 망가뜨리는가

Kuzu가 진정으로 독특해지는 지점은 바로 여기입니다.

Neo4j를 포함한 대부분의 프로덕션 그래프 데이터베이스는 다중 조인(multi-way joins)을 이진 조인(binary joins)의 시퀀스로 분해합니다. 이는 선형 경로 패턴(linear path patterns)에는 효과적이지만, 삼각형과 같은 순환 패턴(cyclic patterns)에서는 성능이 저하됩니다:

```cypher
MATCH (a)-[:E]->(b)<-[:F]-(c)-[:G]->(a)
```

이진 분해(binary decomposition)는 최종 결과보다 훨씬 더 큰 중간 결과(intermediate results)를 계산할 수 있습니다.

Kuzu는 **Intersect** 연산자를 통해 **최악의 경우 최적 조인(Worst-Case Optimal Joins, WCOJ)**을 구현합니다. 이는 N개의 해시 테이블(hash tables)(공유 변수에 대한 각 엣지 방향(edge direction)당 하나)을 구축하고, 모든 테이블을 동시에 탐색하며, 정렬된 배열(sorted arrays)에 대한 병합 조인(merge-join)을 통해 일치하는 ID 목록을 교차시킵니다.

모든 엣지(edges)가 일치하는 튜플(tuples)만이 구체화됩니다.

WCOJ는 AGM (Atserias-Grohe-Marx) 한계(bound)에 의해 제한되는 복잡도(complexity)를 달성합니다. 이는 어떤 이진 조인(binary join) 계획도 보장할 수 없는, 증명 가능한 최적의 성능입니다.

개인 지식 그래프(personal knowledge graph)에서 연락처, 위치, 이벤트, 메시지가 조밀하게 상호 연결되어 있다면, 순환 패턴(cyclic patterns)은 예외적인 경우가 아닙니다. 오히려 그것이 기본 토폴로지(default topology)입니다.

## Apple이 필요로 했던 저장 엔진

저장 계층은 Apple의 배포 제약 조건에 대한 사양서처럼 보입니다.

### 버퍼 관리(Buffer Management): 읽기 위주(Read-Heavy) 작업 부하를 위한 락-프리(Lock-Free) 읽기

Kuzu의 버퍼 매니저(buffer manager)는 엄브라 스타일의 vmcache를 구현합니다. 이는 페이지 상태 머신(page state machine)이 64비트 아토믹(atomic) (1개의 더티 비트(dirty bit) + 7개의 상태 비트(state bits) + 56비트 버전 카운터(version counter))으로 인코딩된 mmap(memory-mapped) 가상 메모리(virtual memory) 영역입니다.

핵심 최적화는 다음과 같습니다: UNLOCKED 상태의 페이지(pages)에 대해, 리더(readers)는 어떠한 락(lock)도 획득하지 않고 실행되며, 나중에 버전 카운터(version counter)를 통해 유효성을 검사합니다. 만약 버전이 변경되지 않았다면, 읽기 작업은 경쟁(contention) 없이 성공한 것입니다.

페이지 교체(eviction)는 MADV_DONTNEED를 사용하는 세컨드 찬스 클록 정책(second-chance clock policy)을 사용합니다. 이는 가상 주소(virtual addresses)를 언맵(unmapping)하지 않고 물리적 RAM을 해제합니다. 이 방식은 읽기 위주(read-heavy) 작업 부하에서 거의 락-프리(lock-free)에 가까운 성능을 제공합니다. 점진적으로 쓰여지고 지속적으로 쿼리되는 온디바이스 지식 그래프(on-device knowledge graph)는 정확히 이러한 접근 패턴을 가집니다.

### 컬럼형 압축(Columnar Compression): 모바일에서는 모든 바이트가 중요합니다

노드(Node) 스토리지는 컬럼형(columnar)으로, 262,144행(2¹⁸)의 **NodeGroup**으로 구성됩니다. 각 NodeGroup은 데이터 타입(data type)별로 독립적으로 압축되는 **ColumnChunk**를 포함합니다.

**ALP**는 특히 영리합니다. 이는 실제 부동소수점(floats)이 제한된 정밀도(precision)를 가진다는 사실을 활용합니다. 건강 센서 데이터(심박수는 소수점 한 자리, 혈중 산소는 두 자리)는 거의 최적으로 압축됩니다.

128–256GB 저장 공간을 가진 기기에서는 이는 '있으면 좋은 기능'이 아니라 필수입니다.

### 관계 저장(Relationship Storage): O(1) 이웃 접근

관계는 별도의 정방향(forward) 및 역방향(backward)을 가진 **압축 희소 행(Compressed Sparse Row, CSR)** 형식을 사용합니다. 이는 NodeGroup당 오프셋 배열(offset arrays), 이웃 ID 배열(neighbor ID arrays), 관계 ID, 그리고 속성 컬럼(property columns)으로 구성됩니다. 밀도 기반 패킹(Density-based packing)은 0.8의 점유율을 목표로 하여, 전체적인 재구조화(restructuring) 없이 삽입(inserts)을 위한 공간을 남겨둡니다.

결과적으로: 소스 노드(source node)당 O(1) 이웃 접근(neighbor access)이 가능합니다. "지난주에 Alice가 메시지를 보낸 모든 사람을 찾아라"와 같은 쿼리(query)의 경우, 비용은 전체 그래프 크기(graph size)가 아니라 Alice의 메시지 수에 비례합니다.

## 당신의 그래프: 아키텍처가 제품을 만나는 곳

Apple 생태계를 연결하는 개인 지식 그래프(personal knowledge graph)는 연락처, 앱, 위치, 캘린더 이벤트, 이메일, 메시지, 파일, 사진, 노래와 같은 개체(entities)를 노드(nodes)로 모델링하고, 관계(relationships)를 타입이 지정된 엣지(typed edges)로 모델링할 것입니다.

Kuzu의 속성 그래프 모델(property graph model)은 이를 스키마 우선(schema-first) 및 강력한 타입(strongly typed)으로 만듭니다.

### Cypher 엔진(Cypher engine)으로서의 Apple Intelligence

"지난달 하이킹 후 Bob이 그룹 채팅에서 언급했던 레스토랑을 찾아라"는 메시지(Messages) → 연락처(Contacts) → 개체(Entities) → 위치(Locations)를 가로지르는 다중 홉 조인(multi-hop join)이 됩니다.

벡터 인덱스 확장(vector index extension)은 의미론적 검색("휴가 계획에 대한 메시지")을 가능하게 하는 반면, 그래프 구조(graph structure)는 관계형 추론("...나의 하이킹 그룹 사람들과 관련된")을 가능하게 합니다.

전체 텍스트 검색(full-text search), 벡터 유사도(vector similarity), 그리고 그래프 탐색(graph traversal)이 하나의 임베디드 엔진(embedded engine)에서 세 개의 개별 시스템을 대체합니다.

### 장면 그래프(scene graph)로서의 사진

```
(Person:Alice)-[:APPEARS_IN]->(Photo)-[:TAKEN_AT]->(Location:Paris)-[:DURING]->(Event:Summer Trip)
```

알고리즘 확장(algo extension)은 PageRank, 연결 요소(connected components), Louvain 커뮤니티 감지(community detection), 최단 경로(shortest paths)와 함께 제공됩니다. 이는 사진, 사람, 장소의 밀접하게 연결된 클러스터(clusters)를 찾아 "추억(Memories)" 기능을 강화할 준비가 되어 있습니다.

### 시간적 지식 그래프(temporal knowledge graph)로서의 건강

수면 → 운동 → 심박수 → 약물은 관계 체인(relationship chains)으로 모델링됩니다. CSR 저장소는 순차적 탐색(sequential traversals)에 효율적이며, ALP 압축은 제한된 정밀도(limited-precision) 센서 부동소수점(sensor floats)을 거의 최적으로 처리합니다.

### 그래프 탐색(graph traversal)을 통한 콘텐츠 추천

PyTorch Geometric 통합(integration) — Kuzu는 FeatureStore 및 GraphStore 인터페이스(interfaces)를 모두 구현합니다 — 은 GNN(Graph Neural Network) 기반 추천 모델(recommendation models)이 그래프를 내보내지(export) 않고 직접 학습(train)할 수 있음을 의미합니다. 팩터화된 실행(factorized execution)은 단순한 추천 쿼리(recommendation queries)를 무력화시키는 팬아웃 패턴(fan-out patterns)을 처리합니다.

그리고 확장 시스템(extension system)은 파서(parser)부터 저장소(storage)에 이르기까지 모든 파이프라인 단계(pipeline stage)에 연결되므로, Apple은 포크(forking) 없이 독점적인 인덱스(indexes)와 함수(functions)를 추가할 수 있습니다. C API는 Swift로의 깔끔한 FFI(Foreign Function Interface) 경로를 제공합니다.

## 프라이버시 해자(Privacy Moat)

여기에 주목할 만한 이차적 함의(second-order implication)가 있습니다.

Google의 지식 그래프(Knowledge Graph)는 클라우드(cloud)에 존재합니다.

Meta의 소셜 그래프(social graph)는 클라우드에 존재합니다.

모든 주요 AI 비서의 맥락적 추론(contextual reasoning)은 여러분의 데이터를 서버로 왕복시키는 것에 의존합니다.

Apple의 전체적인 경쟁적 위치(competitive position)는 정반대의 전제 위에 구축되어 있습니다.

온디바이스 그래프 데이터베이스는 단순히 프라이버시 문제(privacy problem)를 피하는 것을 넘어, 구조적 이점(structural advantage)을 창출합니다.

기기를 더 많이 사용할수록 지식 그래프는 더욱 조밀해집니다. 이 그래프는 여러분의 것이며, 여러분의 하드웨어(hardware)에 있으며, 다른 누구의 모델(model)도 학습시키지 않고 여러분의 경험을 향상시킵니다. 경쟁자들이 몇 달 안에 따라잡을 수 있는 모델 가중치(model weights)와 달리, 수년간의 기기 사용을 통해 구축된 개인화된 지식 그래프(personalized knowledge graph)는 복제 불가능(non-replicable)합니다.

## 회의론자의 주장

합당한 반론: 이것은 10인 규모의 인재 인수(acqui-hire)일 뿐입니다. Apple은 소규모 기업을 꾸준히 인수하며, 대부분은 눈에 띄는 영향 없이 사라집니다.

이러한 해석에 반대하는 두 가지 요소가 있습니다.

DarwinAI (2024), Pointable (2025년 1월), WhyLabs (2025년 1월) 등 유사한 인수 사례에서 Apple의 패턴을 보면, 이러한 기술들이 12~18개월 이내에 OS 기능으로 등장하는 것을 알 수 있습니다.

그리고 EU DMA 신고 자체도 사소하지 않은 전략적 의도(strategic intent)를 보여줍니다. 이 신고는 핵심 플랫폼 서비스(core platform services)와 관련된 특정 중요성 임계값(significance threshold)을 초과하는 인수에 대해서만 발생합니다.

또 다른 반론 — 그래프 데이터베이스는 틈새시장(niche)이다 — 은 오늘날에는 사실입니다. 하지만 SQLite가 모든 모바일 앱(mobile app)에서 보이지 않는 인프라(infrastructure)가 되기 전에는 임베디드 분석 데이터베이스(embedded analytics databases)도 마찬가지였습니다.

배포 모델(deployment model) (임베디드(embedded), 제로-컨피그(zero-config), 서버 없음)이야말로 특수 기술을 주류로 끌어들이는 요소입니다.

Kuzu는 그래프를 위한 그러한 배포 모델입니다.

## 주목해야 할 점

온디바이스 머신러닝(ML) 애플리케이션(applications), 그래프 기반 분석(graph-based analytics), 또는 지식 그래프 아키텍처(knowledge graph architectures)를 구축하거나 평가하고 있다면, 세 가지를 추적할 가치가 있습니다.

첫째, Apple이 SwiftData 또는 새로운 지속성 프레임워크(persistence framework)를 통해 그래프 쿼리(graph query) 기능을 노출할지 여부입니다. C API는 이미 Swift 상호 운용성(interop)을 위한 적절한 추상화 계층(abstraction layer)입니다.

둘째, 2026년 또는 2027년 WWDC에서 Siri에 앱 간 관계 추론(cross-app relationship reasoning)이 필요한 "맥락적 인텔리전스(contextual intelligence)" 기능이 도입될지 여부입니다.

셋째, (GitHub에 여전히 MIT 라이선스로 공개된) 보관된 Kuzu 코드베이스가 다른 임베디드 그래프 프로젝트(embedded graph projects)에 영향을 미칠지 여부입니다.

가장 중요한 인수(acquisitions)는 20억 달러짜리 헤드라인을 장식하는 거래가 아닙니다.

그것은 데이터베이스 연구 외부에서는 아무도 이야기하지 않는 문제를 해결하고 있는, 16만 3천 줄의 프로덕션 품질 C++ 코드를 가진 10인 팀의 인수입니다.

Apple은 온디바이스 AI를 맥락적으로 만드는 데이터 계층(data layer)을 인수한 것입니다. 이는 단순히 빠르기만 한 것이 아니라, 여러분의 디지털 라이프가 어떻게 연결되는지 '인지'하는 것입니다.

문제는 그들이 이를 사용할지 여부가 아닙니다.

문제는 이것이 너무 깊이 임베디드되어 SQLite를 더 이상 의식하지 않게 된 것처럼, 여러분이 이를 의식하지 않게 되기까지 얼마나 걸릴지입니다.
