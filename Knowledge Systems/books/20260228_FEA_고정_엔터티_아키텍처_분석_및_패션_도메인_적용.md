# FEA 고정 엔터티 아키텍처 분석 및 패션 도메인 적용

> 원문: [Why I Stopped Letting LLMs Build My Knowledge Graphs — and What I Did Instead](https://medium.com/@balaaditya_25928/why-i-stopped-letting-llms-build-my-knowledge-graphs-and-what-i-did-instead-263e7b8e7ab6)

---

## 1. HyDE (Hypothetical Document Embeddings, 가설적 문서 임베딩)

2022년 Gao et al. 논문에서 제안된 기법.

### 문제

자연어로 된 도메인 개념 설명(예: "결제 처리를 포함한 결제 오케스트레이션")과 실제 코드(예: `func PlaceOrder(ctx context.Context, ...)`)는 임베딩 공간에서 완전히 다른 영역에 위치한다. 직접 유사도를 비교하면 거의 매칭이 안 됨 (평균 유사도 0.09).

### HyDE 해결법

1. 각 도메인 개념에 대해 LLM으로 **가상의 코드 스니펫**을 생성
2. 자연어 설명 대신 이 **생성된 코드를 임베딩**
3. 실제 코드 청크와 이 임베딩 간의 유사도를 비교

자연어 vs 코드가 아니라 **코드 vs 코드** 비교가 되므로 유사도가 0.09 → 0.30 이상으로 3배 넘게 향상.

**장점**: 개념당 한 번만 LLM을 호출하면 되는 일회성 비용이고, 이후 모든 매칭은 순수 수학(코사인 유사도)으로 처리.

## 2. FEA (Fixed Entity Architecture, 고정 엔터티 아키텍처)

이리나 아담칙 박사가 제안한 지식 그래프 구축 방식. 핵심 아이디어: **LLM이 엔터티를 알아서 발견하게 두지 말고, 사람이 직접 정의하라.**

### 3계층 구조

| 계층 | 내용 | 예시 |
|------|------|------|
| **Layer 1** | 고정 엔터티 온톨로지 (사람이 직접 정의) | "결제 처리", "인증", "주문 관리" 등 15개 도메인 개념 |
| **Layer 2** | 문서/코드 청크 | 실제 소스 코드, 문서 등 |
| **Layer 3** | NLP로 추출한 엔터티 | 코드에서 자동 추출된 함수명, 클래스명 등 |

계층 간 연결은 **LLM 호출 없이 코사인 유사도(수학)**로 수행.

### 기존 방식(Microsoft GraphRAG)과의 차이

- **기존**: LLM에게 "이 문서에서 엔터티를 찾아줘" → 수천 번 LLM 호출, 비용 높고, 중복 발생, 환각 위험
- **FEA**: 사람이 도메인 개념을 미리 정의 → 연결은 수학으로 → LLM 호출 거의 없음, 중복 없음, 결정론적

**"AI한테 뼈대를 만들게 하지 말고, 뼈대는 전문가가 잡고, AI는 최종 답변 생성에만 써라"**

## 3. "수학은 과소평가되어 있습니다" — 해설

지식 그래프에서 노드 간의 연결을 만들 때 두 가지 방법:

- **LLM에게 물어보기**: "이 코드와 이 개념이 관련 있어?" → 느리고, 비용 들고, 매번 답이 달라질 수 있음
- **수학으로 계산하기**: 두 임베딩 벡터의 코사인 유사도 계산 → 빠르고, 항상 같은 결과, 무료

**LLM 없이 단순 수학만 쓰는 부분이 오히려 시스템에서 가장 안정적.**

### "임베딩 공간의 격차" 설명

임베딩은 텍스트를 숫자 벡터(좌표)로 변환하는 것. 그런데:

- **"결제 처리 서비스"** 같은 자연어 → 임베딩 공간의 **A 영역**
- **`func PlaceOrder(ctx context.Context)`** 같은 코드 → 임베딩 공간의 **B 영역**

둘이 같은 의미인데도 완전히 다른 영역에 위치하여 유사도가 거의 0.

**HyDE 해결 방식:**

```
기존 (실패):
  "결제 처리 서비스" (자연어, A영역)  ←비교→  func PlaceOrder(...) (코드, B영역)
  → 유사도: 0.09 (거의 무관)

HyDE 적용 (성공):
  "결제 처리 서비스" → LLM에게 "이 개념의 가상 코드를 만들어줘"
  → func ProcessPayment(order Order) (가상 코드, B영역)  ←비교→  func PlaceOrder(...) (실제 코드, B영역)
  → 유사도: 0.35+ (의미 있는 연결)
```

## 4. Layer 3 NLP 추출 엔터티 — 상세 예시

### NLP 추출이란?

LLM을 쓰지 않고, **규칙 기반 도구(spaCy, 정규식)**로 텍스트에서 고유명사(엔터티)를 자동으로 추출.

### 추출 예시

코드 주석/문서:
```
# CartService handles checkout flow using Stripe for payments.
# Events are published to Kafka and cached in Redis.
# Deployed on AWS EKS with PostgreSQL as the primary database.
```

추출 결과:

| 추출된 엔터티 | 분류 |
|--------------|------|
| CartService | API/서비스명 |
| Stripe | 외부 조직/서비스 |
| Kafka | 기술/인프라 |
| Redis | 기술/인프라 |
| AWS EKS | 플랫폼 |
| PostgreSQL | 기술/데이터베이스 |

### 3계층 연결 구체적 예시

```
Layer 1 (사람이 정의한 고정 개념)
  ├── "결제 및 지급 흐름"
  ├── "장바구니 관리"
  ├── "사용자 인증"
  └── "주문 처리"

Layer 2 (실제 코드 청크)
  ├── func PlaceOrder(ctx, req) { ... }
  ├── func AddToCart(userId, item) { ... }
  ├── func ProcessPayment(order, card) { ... }
  └── func ValidateToken(jwt string) { ... }

Layer 3 (NLP로 자동 추출된 엔터티)
  ├── Stripe (결제 API)
  ├── Redis (캐시)
  ├── Kafka (메시지 큐)
  ├── CartService (서비스명)
  └── PostgreSQL (DB)
```

### 코사인 유사도로 연결

```
"결제 및 지급 흐름" (Layer 1)
    ↕ 코사인 유사도 0.42 → ProcessPayment() (Layer 2) ✅ 연결!
    ↕ 코사인 유사도 0.38 → PlaceOrder()     (Layer 2) ✅ 연결!
    ↕ 코사인 유사도 0.08 → ValidateToken()  (Layer 2) ❌ 무시
    ↕ 코사인 유사도 0.41 → Stripe           (Layer 3) ✅ 연결!
    ↕ 코사인 유사도 0.05 → Redis            (Layer 3) ❌ 무시
```

## 5. 임베딩 모델 vs LLM

| | **임베딩 모델** | **LLM** |
|---|---|---|
| **하는 일** | 텍스트 → 숫자 벡터 변환 | 텍스트 → 텍스트 생성 |
| **예시** | OpenAI `text-embedding-3-small` | GPT-4, Claude |
| **비용** | 매우 저렴 (100만 토큰에 ~$0.02) | 비쌈 (100만 토큰에 $3~$15+) |
| **속도** | 수천 개를 초 단위로 처리 | 한 번에 수 초 |
| **결과** | 항상 동일 (결정론적) | 매번 다를 수 있음 |

### 비용 차이 예시

코드 청크 1,000개 × 개념 15개 = 15,000쌍 연결 시:

- **LLM 방식**: 15,000번 LLM 호출 → 수십~수백 달러, 수 시간, 매번 결과 다름
- **FEA 방식**: 1,015번 임베딩 호출 → ~$0.01 미만, 수 초, 연결은 코사인 유사도(무료)

글에서 "LLM 없이"란 **텍스트를 생성하는 대규모 언어 모델을 안 쓴다**는 뜻. 임베딩 모델은 쓰지만 비용과 복잡도 면에서 차원이 다름.

## 6. 청크 크기와 임베딩

### 청크 크기에 따른 트레이드오프

```
너무 작음 (단어 1개)
  "Kafka" → 메시지 큐? 소설가? 맥락 없음 → 의미가 모호

적절함 (함수/문단 단위)
  "func PublishEvent(topic string) { kafka.Produce(...) }"
  → Kafka가 메시지 큐라는 맥락이 명확

너무 큼 (파일 전체)
  1000줄짜리 파일 통째로 임베딩
  → 여러 개념이 섞여서 벡터가 "평균화" → 유사도가 애매
```

**핵심: 하나의 청크 = 하나의 명확한 의미 단위**

### 다국어 임베딩과 동음이의어

- 다국어 모델(OpenAI 등): "Kafka" ↔ "카프카" → 유사도 높음 (~0.85+)
- 영어 전용 모델: "카프카"를 모름 → 유사도 낮음
- 동음이의어 문제: "Kafka"(메시지 큐) vs "Kafka"(소설가) → 문장 단위 임베딩으로 맥락 반영 필요

## 7. 패션 도메인에서의 속성 관리

### 구조화 속성 vs 비정형 정보

| 속성 유형 | 관리 방식 | 이유 |
|----------|----------|------|
| 소재, 핏, 기장, 색상, 사이즈 | DB/태그 | 선택지가 명확, 필터링이 핵심 |
| 스타일 무드, 트렌드 설명, 코디 설명 | 임베딩 | 유사한 "느낌"을 찾아야 함 |
| 아트웍, 그래픽 패턴 설명 | **둘 다** | 태그로 필터 + 설명으로 유사도 검색 |

### 실무 추천: 하이브리드 방식

```
┌─────────────────────────────────────────────┐
│              상품 데이터                      │
├──────────────────┬──────────────────────────┤
│  구조화 속성 (DB)  │  비정형 정보 (임베딩)      │
├──────────────────┼──────────────────────────┤
│  소재: 면 100%    │  스타일 설명:              │
│  핏: 오버핏       │  "미니멀한 모던 캐주얼,     │
│  기장: 크롭       │   깔끔한 라인에 드롭숄더    │
│  색상: 블랙       │   디테일이 포인트"         │
│  카테고리: 티셔츠  │                          │
│  시즌: 25SS      │  무드 키워드:              │
│  가격: 39,000    │  "미니멀, 모던, 클린"       │
├──────────────────┼──────────────────────────┤
│  → 필터 검색      │  → 유사도 검색             │
│  "오버핏 + 면"    │  "이런 분위기 옷 더 보여줘"  │
└──────────────────┴──────────────────────────┘
```

## 8. FEA를 패션 도메인에 적용

### Layer 1: 고정 엔터티 온톨로지 (패션 전문가가 직접 정의)

```
패션 고정 온톨로지 (예시 15개)
├── 미니멀 캐주얼
├── 스트릿 / Y2K
├── 오피스룩 / 비즈니스 캐주얼
├── 아메카지
├── 스포츠 / 애슬레저
├── 로맨틱 페미닌
├── 빈티지 / 레트로
├── 아웃도어 / 고프코어
├── 데님 컬처
├── 레이어드 스타일링
├── 시즌 트랜지션 (간절기)
├── 프린트 & 패턴
├── 오버사이즈 실루엣
├── 소재 중심 (니트, 린넨, 가죽 등)
└── 컬러 트렌드 (25SS)
```

### Layer 2: 상품 데이터

```
상품 A: "워싱 가공된 와이드 데님 팬츠. 90년대 무드의 루즈한
        실루엣, 백포켓 체인 디테일. 면 100%, 블루 워싱."

상품 B: "소프트 터치 오버핏 블레이저. 싱글 버튼 구성,
        드롭숄더 라인. 폴리 혼방, 차콜 그레이."

상품 C: "크롭 기장 래시가드 세트. 자외선 차단 기능성 원단,
        네온 컬러 배색. 스판덱스 혼방."
```

### Layer 3: NLP 추출 엔터티

```
상품 A에서 추출:
├── 워싱        (가공 기법)
├── 와이드       (실루엣)
├── 데님        (소재)
├── 90년대      (시대 레퍼런스)
├── 체인 디테일   (장식)
└── 면 100%     (소재 구성)

상품 B에서 추출:
├── 오버핏       (핏)
├── 블레이저     (아이템)
├── 싱글 버튼    (디테일)
├── 드롭숄더     (패턴)
├── 폴리 혼방    (소재)
└── 차콜 그레이   (컬러)
```

### 3계층 코사인 유사도 연결

```
[미니멀 캐주얼] (Layer 1)
    ↕ 0.41 → 상품 B "소프트 터치 오버핏 블레이저..."  ✅
    ↕ 0.12 → 상품 A "워싱 와이드 데님..."            ❌
    ↕ 0.06 → 상품 C "크롭 래시가드..."              ❌

[스트릿 / Y2K] (Layer 1)
    ↕ 0.44 → 상품 A "워싱 와이드 데님..."            ✅
    ↕ 0.09 → 상품 B "소프트 터치 블레이저..."         ❌

[스포츠 / 애슬레저] (Layer 1)
    ↕ 0.47 → 상품 C "크롭 래시가드..."              ✅

[오버사이즈 실루엣] (Layer 1)
    ↕ 0.43 → 상품 A "루즈한 실루엣..."              ✅
    ↕ 0.39 → 상품 B "오버핏 블레이저..."             ✅
```

### 최종 그래프

```
Layer 1 (고정 개념)        Layer 2 (상품)           Layer 3 (추출 엔터티)

[스트릿/Y2K] ──────────── 상품 A ──────── 데님
                          (워싱 와이드       워싱
[오버사이즈 실루엣] ──┤     데님 팬츠)       90년대
                    │                     체인 디테일
                    │
                    ├── 상품 B ──────── 블레이저
[미니멀 캐주얼] ──────    (오버핏            드롭숄더
                          블레이저)         차콜 그레이

[스포츠/애슬레저] ──────── 상품 C ──────── 래시가드
                          (크롭              네온
                           래시가드)         스판덱스
```

### 활용 시나리오

**1. 고객 질문: "요즘 Y2K 느낌 나는 하의 뭐 있어요?"**

```
"Y2K 느낌 하의" → 임베딩 → [스트릿/Y2K] 개념과 유사도 높음
→ 그래프 탐색 → 연결된 상품들 → 상품 A (워싱 와이드 데님)
→ Layer 3에서 "90년대", "워싱" 엔터티와 연결된 다른 상품도 탐색
```

**2. MD 업무: "오버사이즈 트렌드 상품 현황 파악"**

```
[오버사이즈 실루엣] 개념 노드에서 출발
→ 연결된 상품: A, B (2개)
→ 연결된 엔터티: 와이드, 오버핏, 드롭숄더, 루즈
→ "오버사이즈 카테고리에 하의는 있지만 아우터류가 부족하네"
```

**3. 유사 상품 추천: "이 상품과 비슷한 무드의 상품"**

```
상품 A 선택
→ 연결된 개념: [스트릿/Y2K], [오버사이즈 실루엣]
→ 같은 개념에 연결된 다른 상품들 탐색
→ 직접 임베딩 유사도만 쓸 때보다 "스타일 무드" 기준 추천이 가능
```

### 패션에서 FEA의 장점
