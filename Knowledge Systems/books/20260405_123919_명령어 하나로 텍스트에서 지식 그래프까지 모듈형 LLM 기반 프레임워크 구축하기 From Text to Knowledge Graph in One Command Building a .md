# 명령어 하나로 텍스트에서 지식 그래프까지: 모듈형 LLM 기반 프레임워크 구축하기 (From Text to Knowledge Graph in One Command: Building a Modular LLM-Backed Framework)

LLM은 법률 계약서를 요약하고, 의료 기록에 대한 질문에 답하며, 연구 논문에서 엔티티(Entity)를 추출할 수 있습니다. 하지만 이러한 문서로부터 구조화되고 신뢰할 수 있는 지식 그래프(Knowledge Graph)를 생성해 달라고 요청하면, 텍스트를 직접 읽는 것보다 환각(Hallucination)된 관계와 연결되지 않은 노드를 정리하는 데 더 많은 시간을 소비하게 될 것입니다.

이를 해결하기 위해 도구를 만들다 보면, 특정 제공업체에 종속된 스크립트와 하드코딩된 프롬프트가 얽혀 본인조차 이해하기 힘든 복잡한 파이프라인이 되어버리곤 합니다. 본 게시물은 비정형 텍스트에서 지식 그래프를 구축하고 증강하는 시리즈의 두 번째 파트로, 이러한 문제를 해결하기 위해 설계된 '지식 그래프 빌더(Knowledge Graph Builder)' 프레임워크의 아키텍처 설계와 실용적인 활용법에 초점을 맞춥니다.

### 핵심 아이디어: 추상화와 CLI 기반의 모듈화

지식 그래프 빌더는 비정형 텍스트에서 지식 그래프를 추출하는 모듈형 Python 프레임워크입니다. Google의 `langextract` 라이브러리를 사용하여 소스 텍스트의 특정 위치에 고정된 근거 기반 트리플(Grounded triples)을 추출하며, 이후 추론된 관계를 통해 그래프를 증강하고 표준 형식으로 변환 및 시각화합니다.

프레임워크 아키텍처의 핵심 원칙은 **"변경되기 쉬운 모든 구성 요소는 교체가 쉬워야 하며, 전체 프레임워크의 사용성을 유지해야 한다"**는 것입니다.

*   **클라이언트(Clients):** LLM 제공업체(Gemini, Ollama, LM Studio 등)를 위한 추상화된 래퍼.
*   **빌더(Builders):** 지식 그래프를 생성하고 개선하기 위해 LLM이 수행하는 다양한 작업의 핵심 로직.
*   **도메인(Domains):** 프롬프트 템플릿, 퓨샷 예시(Few-shot examples), 엔티티/관계 스키마의 패키지.
*   **I/O (Readers & Writers):** 입력 형식 감지 및 그래프 형식 변환.
*   **시각화(Visualisation):** 추출 방법에 따른 출처(Provenance)가 표시된 대화형 그래프 렌더링.

---

### 1. 빌더(Builder): 실제 작업이 수행되는 곳

빌더는 프레임워크가 지식 그래프를 얻기 위해 취할 수 있는 행동의 청사진을 정의합니다.

*   **추출 빌더(Extraction Builder):** 프롬프트 생성, 퓨샷 예시 주입, 트리플 정규화를 처리합니다. `langextract`를 통해 모든 트리플은 소스 텍스트의 정확한 위치와 연결되므로, 정보의 근거를 명확히 추적할 수 있습니다.
*   **증강 빌더(Augmentation Builder):** 추출된 그래프에 누락된 연결을 보완합니다. 예를 들어, 끊어진 컴포넌트 간의 가교 역할을 하는 트리플을 추론하여 그래프의 연결성을 높입니다.

추출(Extraction)과 증강(Augmentation)을 분리하는 이유는 신뢰성 때문입니다. 텍스트에 명시된 내용(단언된 지식)과 모델이 스스로 추론한 내용(증강된 지식)을 구분하는 것은 검증 가능성이 중요한 작업에서 매우 핵심적인 요소입니다.

### 2. 클라이언트(Client): 모든 제공업체를 위한 단일 인터페이스

클라이언트 계층은 클라우드 API나 로컬 호스팅 모델에 관계없이 모든 제공업체를 `BaseLLMClient` 추상 클래스로 통합합니다. 이를 통해 CLI에서 API 키, 모델 ID 등 개별 설정을 복잡하게 입력할 필요 없이 단일 구성 객체로 제어할 수 있습니다.

### 3. 도메인(Domain): 도메인 전환만으로 모든 설정 변경

지식 그래프 추출 시 좋은 결과를 얻으려면 도메인에 적합한 어휘와 스키마가 필요합니다. 도메인 모듈은 프롬프트, 예시, 스키마를 폴더 구조로 패키징하여 관리합니다. 덕분에 파이프라인 코드를 수정하지 않고도 `--domain legal`에서 `--domain medical`로 플래그 하나만 바꿔 전체 추출 어휘 체계를 전환할 수 있습니다.

### 4. I/O 및 시각화: 데이터의 원활한 흐름과 가독성

*   **I/O:** JSON, JSONL, CSV 등 다양한 형식을 자동으로 감지하여 로드합니다. 출력 시에는 `GraphML` 형식을 기본으로 사용하여 NetworkX, Gephi 등 기존 그래프 분석 도구와 호환됩니다. 각 트리플에는 명시적 추출인지 문맥적 추론인지에 대한 메타데이터가 포함됩니다.
*   **시각화:** 브라우저 기반의 대화형 네트워크 시각화를 제공합니다. 노드의 연결 중심성(Degree centrality)에 따라 크기를 조절하고, 색상과 선 스타일로 출처를 구분하며, 검색 및 경로 찾기 기능을 통해 복잡한 그래프를 쉽게 탐색할 수 있습니다.

---

### 파이프라인(Pipeline): 전체 워크플로우의 통합

각 모듈은 개별적으로도 유용하지만, 이를 하나로 묶어 선언적인 워크플로우로 실행할 때 진정한 가치가 발휘됩니다. YAML 설정 파일을 통해 클라이언트, 도메인, 입력 경로, 실행 단계(추출, 증강, 변환, 시각화)를 정의할 수 있습니다. 이 설정은 재현 가능하고 공유 가능하며, 버전 관리가 용이합니다.

```yaml
client:
  type: gemini
  model: gemini-2.0-flash
domain:
  name: legal
input:
  path: data/legal/background.jsonl
steps:
  - extract
  - augment
  - convert
  - visualize-network
```

### CLI: 명령어 하나로 실행

프레임워크의 모든 기능은 `kgb` 콘솔 스크립트를 통해 접근 가능합니다. 개별 단계를 세밀하게 실행하거나, 미리 정의된 YAML 설정을 호출하여 전체 파이프라인을 한 번에 실행할 수 있습니다.

```bash
# YAML 설정을 기반으로 전체 파이프라인 실행
kgb run-pipeline --config legal_gemini.yaml
```

### 결론: 모듈형 아키텍처의 이점

1.  **제공업체 독립성(Provider Independence):** Gemini, Ollama 등 모델을 변경해도 파이프라인 수정이 필요 없습니다.
2.  **도메인 이식성(Domain Portability):** 플래그 하나로 법률, 의료, 과학 등 다양한 분야에 적용 가능합니다.
3.  **재현성(Reproducibility):** YAML 설정을 통해 누구나 동일한 그래프를 정확히 재현할 수 있습니다.
4.  **점진적 채택(Incremental Adoption):** 기존에 추출된 데이터가 있다면 특정 단계(예: 시각화)부터 바로 시작할 수 있습니다.

이 프레임워크는 복잡하게 얽힌 스크립트 뭉치를 체계적인 모듈형 시스템으로 대체하여, 누구나 전문적인 지식 그래프 구축 파이프라인을 운영할 수 있게 돕습니다.
