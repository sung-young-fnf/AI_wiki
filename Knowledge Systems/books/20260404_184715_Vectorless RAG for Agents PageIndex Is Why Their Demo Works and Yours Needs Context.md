# Vectorless RAG for Agents: PageIndex Is Why Their Demo Works and Yours Needs “Context”

RAG(검색 증강 생성) 애플리케이션을 구축할 때 품질과 예산을 남모르게 갉아먹는 주범은 문서를 이른바 '의미론적 파편(semantic confetti)'으로 만드는 것입니다. 즉, 모든 내용을 청크(chunk)로 나누고 임베딩(embedding)한 뒤, 근사 최근접 이웃(ANN) 검색이 적절한 조각을 찾아내길 바라는 방식입니다.

문서 기반이 상당히 방대할 경우, 정보 검색은 수많은 청크 사이에서 '의미론적 숨바꼭질'로 변질되기 쉽습니다. 이 과정에서 수천 개의 토큰을 소모하며 정답 근처에서 확신에 찬 할루시네이션(hallucination, 환각)을 생성하게 될 수도 있습니다.

이러한 경우 **PageIndex**와 같은 접근 방식이 더 나은 해결책이 될 수 있습니다. 청크를 벡터화하고 검색하는 대신, 문서를 계층 구조의 트리(hierarchical tree)로 표현하는 방식입니다. 이는 기본적으로 LLM에 최적화된 목차와 같습니다.

모델은 트리를 따라 내려가며 추론하고(예: "위험 요인 섹션에서 유동성, 그다음 계약 위반 항목 확인..."), 정확히 필요한 가지에서 컨텍스트(context)를 추출합니다. 이를 통해 임베딩 편차나 "비슷해 보이는" 청크로 인한 오류, 그리고 단지 몇 개의 핵심 키워드가 겹친다는 이유로 관련 없는 섹션으로 잘못 빠지는 일을 방지할 수 있습니다.

PageIndex를 기반으로 구축된 금융 문서 분석 시스템인 **Mafin 2.5**는 FinanceBench 벤치마크에서 98.7%라는 최고 수준의 정확도를 기록하며 기존 벡터 기반 RAG 시스템을 크게 앞질렀습니다.

### PageIndex 작동 원리

1.  **트리 생성 (색인화):** PDF를 섹션 및 하위 섹션의 계층적 노드 구조로 변환합니다. 각 노드에는 제목, 노드 ID, 페이지 인덱스 및 관련 텍스트와 같은 메타데이터가 포함됩니다.
2.  **추론 기반 검색:** LLM이 다음에 열어볼 노드를 선택(트리 탐색)하며, 선택 근거(rationale)와 참조할 노드 ID 목록을 생성합니다.

### 주요 장점

*   **무차별 대입 없는 정밀도:** 모든 단락을 반복해서 임베딩하거나 방대한 검색 결과를 하류 작업으로 보낼 필요가 없어 비용이 절감됩니다.
*   **사용자 경험 및 설명 가능성(Explainability):** 단순히 "임베딩 결과입니다"라고 하는 대신, "경영진 검토 및 분석 섹션의 영업 실적 항목을 확인했습니다"라고 사용자에게 명확한 근거를 제시할 수 있습니다.
*   **지연 시간 및 토큰 소모 감소:** 트리 탐색을 통해 컨텍스트를 압축적으로 유지할 수 있어 응답이 빨라지고 불필요한 토큰 낭비가 줄어듭니다.
*   **구조 인지적 진실성(Structure-aware truthfulness):** SEC 공시나 실적 발표 자료 같은 복잡한 금융 문서에서는 내용이 '어디에 위치하는지'가 의미의 절반을 차지합니다. PageIndex는 문서 구조를 부차적인 요소가 아닌 검색의 핵심 원칙으로 취급합니다.

### 시작하기 및 설정

PageIndex는 설정이 간편하며 마크다운(Markdown)도 지원합니다.

**1. 환경 설정**
```bash
python -m venv .venv
source .venv/bin/activate
pip install --upgrade -r requirements.txt
```

**2. 실행**
`run_pageindex.py` 스크립트를 통해 PDF 색인화를 실행할 수 있습니다. 주요 옵션은 다음과 같습니다:
*   `--model`: 사용할 OpenAI 모델 (기본값: gpt-4o)
*   `--toc-check-pages`: 목차 확인 범위 (기본값: 앞 20페이지)
*   `--max-tokens-per-node`: 노드당 최대 토큰 수 (기본값: 20,000)

마크다운 파일의 경우 `--md_path` 플래그를 사용하여 트리 구조를 생성할 수 있습니다.

### 인덱싱 및 검색 워크플로우

1.  **인덱싱 (트리 생성):** PDF 입력을 받아 목차 형태의 계층적 노드 트리를 출력합니다.
2.  **검색 (트리 탐색):** 쿼리와 트리 구조를 제공하면, LLM이 '추론(thinking)'과 '노드 리스트'를 포함한 JSON을 출력합니다.
3.  **답변 합성:** 선택된 노드의 텍스트를 모아 최종 답변 프롬프트에 입력합니다.

### 프로덕션 통합 (SDK 및 API)

실제 서비스 환경에서는 호스팅 서비스의 SDK나 API를 활용할 수 있습니다.

*   **Python SDK:** `PageIndexClient`를 통해 문서를 제출하고 트리 구조나 OCR 결과를 가져올 수 있습니다.
*   **REST 엔드포인트:** `/doc/` 포스트를 통한 PDF 업로드, `/chat/completions`를 통한 대화형 검색 등을 지원합니다.

### 다중 문서 검색 패턴

여러 문서를 다룰 때는 다음과 같은 단계별 접근을 권장합니다.

1.  **메타데이터/의미론/설명 기반 검색:** 먼저 후보가 되는 문서를 선정합니다.
2.  **트리 탐색:** 선정된 문서 내부에서 트리 검색을 실행합니다.
3.  **답변 합성:** 추출된 정보를 바탕으로 최종 답변을 생성합니다.
