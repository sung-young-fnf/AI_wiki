# 바이너리 임베딩과 int8 리스코어링(rescoring)을 활용해 CPU만으로 4,000만 개의 문서를 200ms 이내에 검색하는 방법

### 4,000만 개의 문서. 단 하나의 CPU. GPU는 필요 없습니다.

**바이너리 검색과 int8 리스코어링이 대규모 시맨틱 검색(semantic search)의 규칙을 어떻게 조용히 바꿔놓고 있는지에 대하여.**

임베딩(embeddings) 작업을 오래 해본 사람이라면 누구나 겪게 되는 특유의 좌절감이 있습니다.

마침내 검색(retrieval) 시스템을 구축했습니다. 정확도는 높고, 시스템은 똑똑하게 작동합니다. 스스로가 자랑스러워질 때쯤, 여러분은 청구서를 보게 됩니다.

아니면 더 심각하게는 RAM 사용량을 보게 되죠.
그보다 더 최악인 상황은, 고작 "단순한" 시맨틱 검색 기능을 유지하기 위해 180GB의 메모리가 필요하다는 사실을 깨닫는 순간입니다.

대개 이 지점에서 우리는 조용히 이렇게 가정하곤 합니다.
"음... 제대로 된 벡터 검색을 하려면 이 정도 비용은 당연히 감수해야겠지."

하지만 어쩌면 그렇지 않을 수도 있습니다.

제가 스크롤을 멈추게 된 놀라운 사실이 여기 있습니다.

**4,000만 개의 텍스트를 약 200ms 만에 검색할 수 있습니다.**
* **CPU만 사용해서**
* **8GB의 RAM으로**
* **45GB의 디스크 용량으로**

GPU도, 특이한 하드웨어도, 마법도 필요 없습니다.
그저 매우 정교한 추론 전략(inference strategy)이 있을 뿐입니다.

그리고 이 방식을 한 번 이해하고 나면, 이전으로 돌아갈 수 없을 것입니다.

---

### float32 임베딩의 불편한 진실

솔직하게 말해봅시다.

우리 대부분은 모델이 제공하는 기본값이기 때문에 float32(fp32) 임베딩을 그대로 사용합니다. 이를 신성하고 불가침한 영역으로 여기며, 고정밀도가 아니면 안 된다고 생각하죠.

하지만 검색 과정에서 우리가 실제로 무엇을 하고 있는지 생각해보세요.

우리는 물리 방정식을 푸는 것이 아닙니다.
**문서의 순위를 매기고(ranking) 있을 뿐입니다.**

그리고 랭킹, 특히 퍼널(funnel) 상단에서 후보군을 추려내는 작업은 놀라울 정도로 정밀도에 관대합니다.

바로 이 틈새가 이 접근 방식이 파고드는 지점입니다.

---

### 핵심 비결 (The Trick)

신비감을 걷어내고 나면 이 설정은 지루할 정도로 단순합니다.

**재현율(recall)을 위해 바이너리 검색을 하고, 정밀도(precision)를 위해 int8 리스코어링을 수행한다.**

이것이 전부입니다.

품질을 버리는 것이 아니라, 단계를 나누는 것입니다.
저렴한 표현 방식(representation)이 먼저 힘든 일을 처리하게 하고, 정밀도는 정말 필요한 곳에만 다시 가져오는 방식입니다.

전체 흐름은 다음과 같습니다.

---

### 단계별 추론 전략

함께 디버깅하는 것처럼 천천히 살펴보겠습니다.

#### 1단계: 쿼리 임베딩
우리가 예상하는 지점에서 시작합니다. 밀집 임베딩(dense embedding) 모델을 사용하여 표준 fp32 벡터를 생성합니다.

```python
# 1. 쿼리를 float32로 임베딩
start_time = time.time()
query_embedding = model.encode_query(query)
embed_time = time.time() - start_time
```

아직 특별한 건 없습니다. 이것은 나중에 신뢰할 수 있는 참조 신호가 됩니다.

#### 2단계: 쿼리를 바이너리로 양자화(Quantization)
이제 사고방식을 전환해야 합니다. fp32 쿼리 임베딩을 가져와서 바이너리(binary)로 양자화합니다. 의미적 방향은 유지하면서 크기는 32배 줄어듭니다.

```python
# 2. 쿼리를 ubinary로 양자화
start_time = time.time()
query_embedding_ubinary = quantize_embeddings(
    query_embedding.reshape(1, -1), "ubinary"
)
quantize_time = time.time() - start_time
```

이 단계는 매우 빠릅니다. 의심스러울 정도로 빠르지만, 이후의 모든 과정을 가능하게 합니다.

#### 3단계: 바이너리 인덱스 검색
이 단계에서 규모(scale)는 더 이상 두려운 존재가 아닙니다. 바이너리 인덱스는 매우 작고 빠르며, 메모리에 가볍게 상주합니다.

```python
# 3. 바이너리 인덱스 검색 (정확 또는 근사 검색)
index = binary_ivf_index if use_approx else binary_index
start_time = time.time()
_scores, binary_ids = index.search(
    query_embedding_ubinary, top_k * rescore_multiplier
)
binary_ids = binary_ids[0]
search_time = time.time() - start_time
```

"4,000만 개 중 어떤 문서가 가장 좋은가?"라고 묻는 대신, **"유망해 보이는 약 40개의 후보는 무엇인가?"**라고 묻는 것입니다. 이 프레임의 전환이 중요합니다.

#### 4단계: 상위 후보에 대한 int8 임베딩 로드
이제 후보군이 좁혀졌으니, 더 많은 신호를 다시 도입합니다. 상위 바이너리 검색 결과에 대해서만 int8 임베딩을 로드합니다.

```python
# 4. 해당되는 int8 임베딩 로드
start_time = time.time()
int8_embeddings = np.array(
    title_text_int8_dataset[binary_ids]["embedding"], dtype=np.int8
)
load_int8_time = time.time() - start_time
```

여전히 작고 저렴하지만, 바이너리보다는 훨씬 더 표현력이 풍부합니다.

#### 5단계: 원본 fp32 쿼리를 이용한 리스코어링
정밀도가 돌아오는 순간입니다. 원본 fp32 쿼리 임베딩과 int8 문서 임베딩을 비교하여 점수를 다시 계산합니다.

```python
# 5. float32 쿼리 임베딩과 int8 문서 임베딩을 사용하여 리스코어링
start_time = time.time()
scores = query_embedding @ int8_embeddings.T
rescore_time = time.time() - start_time
```

이 단계는 비용이 거의 들지 않으면서, 이전에 "잃어버렸던" 품질의 대부분을 회복시켜 줍니다.

#### 6~7단계: 정렬 및 최종 텍스트 로드
이제 랭킹이 완료되었습니다. 마지막 단계에서만 실제 제목, URL, 본문 내용을 가져옵니다.

### 이것이 작동하는 이유 (처음에는 왜 틀린 것처럼 느껴질까?)

우리는 정밀도를 최우선으로 보호해야 한다고 배워왔기 때문에 이 방식이 틀린 것처럼 느껴질 수 있습니다. 하지만 검색은 분류(classification)도, 회귀(regression)도, 생성(generation)도 아닙니다.

**검색은 선별(triage)입니다.**

*   **바이너리 임베딩**은 관련 없는 항목을 걸러내는 데 경이적인 성능을 보입니다.
*   **int8 임베딩**은 유력한 후보들의 순위를 매기는 데 탁월합니다.
*   **fp32**는 정말 미묘한 차이가 필요할 때 최적이지만, 후보가 40개 정도로 줄어든 시점에서는 그런 경우가 드뭅니다.

따라서 모든 곳에 fp32 비용을 지불하는 대신, 정밀도를 스포트라이트처럼 단계적으로 투사하는 것입니다. 처음에는 넓은 빔으로, 나중에는 좁고 강한 빔으로 말이죠.

### 저장 용량 및 메모리 비교

표준 fp32 설정과 비교해 보겠습니다.

| 구분 | float32 검색 (기준점) | 바이너리 + int8 검색 |
| :--- | :--- | :--- |
| **RAM** | ~180GB | ~6GB |
| **디스크** | ~180GB | ~45GB |
| **속도** | 느림 | 매우 빠름 |
| **비용** | 고통스러움 | 저렴함 |

가장 핵심적인 것은 **바이너리 인덱스만 메모리에 유지하면 된다**는 점입니다. 여기서 실제 비용 절감 효과가 극대화됩니다.

### 성능 트레이드오프

품질 손실이 전혀 없는 척하지는 않겠습니다. 실험 결과는 다음과 같습니다.

| 표현 방식 | 인덱스 크기 | 속도 | 성능(정확도) |
| :--- | :--- | :--- | :--- |
| float32 | 1× | 1× | 100% |
| int8 / uint8 | 4× 작음 | 최대 4× 빠름 | ~99.3% |
| binary / ubinary | 32× 작음 | 최대 45× 빠름 | ~96% |

하지만 중요한 점은 이것입니다: **바이너리 검색을 통해 더 많은 후보를 뽑고 int8로 리스코어링함으로써, fp32 품질의 약 99%를 회복할 수 있다는 것**입니다. 마지막 1%를 위해 지불해야 했던 엄청난 비용을 거의 0으로 줄일 수 있습니다.

### 실무적인 변화

솔직히 말해서, 많은 것이 변합니다.

*   대규모 시맨틱 검색에 더 이상 GPU가 필수적이지 않습니다.
*   일반적인 하드웨어에서도 수준 높은 검색 기능을 배포할 수 있습니다.
*   모든 곳에 fp32를 쓰면서 내던 "사치세"를 멈출 수 있습니다.

무엇보다 가장 큰 변화는 심리적인 것입니다. 우리는 임베딩을 깨지기 쉬운 유리처럼 다뤄왔지만, 사실 그것들은 도구일 뿐입니다. 모든 못에 망치 하나만 사용하던 방식에서 벗어날 때 도구는 최고의 성능을 발휘합니다.
