이 글은 로컬 LLM(Local LLM)으로 지식 그래프를 구축하는 시리즈의 3부입니다. 1부에서는 개방형 가중치 모델(open-weight models) 네 가지의 엔티티-관계 추출(entity-relation extraction) 성능을 벤치마킹했습니다. 2부에서는 REBEL 데이터셋(dataset)으로 Qwen3–8B를 미세 조정(fine-tuning)하여 100% 구조적 신뢰성을 달성했지만, 도메인(domain)별 텍스트에서는 트리플 F1(Triple F1) 점수가 급락했습니다. 진단 결과는 다음과 같습니다. 하나의 미세 조정된 모델은 하나의 술어 어휘(predicate vocabulary)만 학습하며, 이를 여러 도메인에 걸쳐 전이(transfer)할 수 없다는 것입니다. 이 글은 이러한 문제에 대한 아키텍처(architectural) 솔루션에 관한 것입니다.

## 문제점: 하나의 어댑터로는 모든 도메인을 처리할 수 없다

2부에서 저희는 3,000개의 REBEL 샘플(sample)로 Qwen3–8B를 미세 조정하여 항상 완벽한 JSON을 생성하는 모델을 얻었습니다. 100% 스키마 적합성(schema conformance), 파싱(parsing) 실패 없음, 재시도(retry) 없음. 구조적으로는 문제가 해결되었습니다.

하지만 법률 문서에 이 모델을 적용했을 때, 예상했던 `(TechCorp Industries, filed_lawsuit_against, DataFlow Systems)` 대신 `(TechCorp Industries, filed a patent infringement lawsuit against, DataFlow Systems)`와 같은 트리플(triple)을 생성했습니다. 보류된 REBEL 데이터로 평가했을 때는 더 이상한 일이 일어났습니다. 모델이 동일한 텍스트에서 완전히 다르지만 동등하게 유효한 트리플을 추출한 것입니다. Deutsche Eislauf-Union과 Germany를 언급하는 구절에서 모델은 `(Deutsche Eislauf-Union, organizes, competition)`을 추출했지만, REBEL의 골드 라벨(gold label)은 `(Deutsche Eislauf-Union, country, Germany)`였습니다. 둘 다 올바르지만, 서로 일치하지 않았습니다.

핵심 문제는 프로덕션 파이프라인(production pipeline)이 한 가지 유형의 문서만 처리하지 않는다는 것입니다. 저희 파이프라인은 법률 서류, 의료 기록, 금융 보고서, 기술 문서를 처리합니다. 각 도메인에는 고유한 엔티티 유형(entity type), 술어 어휘, 그리고 추출 규칙(extraction conventions)이 있습니다. 법률 어댑터(adapter)는 `filed_lawsuit_against` 및 `Party` 엔티티를 출력해야 합니다. 의료 어댑터는 `treats_condition` 및 `Drug` 엔티티를 필요로 합니다. 금융 어댑터는 `acquired_by` 및 `Instrument` 엔티티를 필요로 합니다.

단순한 해결책은 효과가 없습니다. 모든 도메인에 걸쳐 하나의 모델을 훈련하면 술어 어휘 충돌이 발생합니다. 모델은 여러 도메인에 걸쳐 평균을 내어 어느 도메인도 제대로 처리하지 못하게 됩니다. 도메인별로 개별 미세 조정 모델을 실행하는 것은 GPU 메모리에 16GB × N개의 도메인을 로드해야 함을 의미하며, 이는 확장성(scales)이 매우 떨어집니다. 도메인별로 퓨샷 프롬프팅(few-shot prompting)을 사용하면 최고의 추출 품질을 얻을 수 있지만(1부에서 Llama 3.1이 퓨샷으로 0.732 트리플 F1을 달성했음을 보여줌), 스키마 적합성은 20%에 불과하여, 문서의 80%가 파싱 불가능한 출력물을 생성한다는 의미입니다.

저희는 하나의 모델처럼 확장되면서도 여러 모델처럼 전문화되는 솔루션이 필요했습니다.

## 아이디어: 모듈식 도메인 어댑터

LoRA(Low-Rank Adaptation) 미세 조정(fine-tuning)은 기본 모델(base model)의 가중치(weight)를 고정하고, 어텐션(attention) 및 피드포워드(feed-forward) 레이어(layer)를 수정하는 작은 어댑터 행렬(adapter matrix), 즉 저랭크 분해(low-rank decomposition) 세트를 훈련하는 방식으로 작동합니다. 랭크(rank) 64를 가진 Qwen3–8B 모델의 경우, 각 어댑터는 대략 250MB입니다. 기본 모델은 16GB입니다. 어댑터는 기본 모델의 1.5%에 불과합니다.

이러한 비대칭성이 바로 핵심 아이디어입니다. N개의 개별 미세 조정 모델 대신, 하나의 기본 모델을 로드하고 문서 도메인에 따라 경량 어댑터를 교체합니다. 메모리 계산은 다음과 같습니다:

네 개의 개별 미세 조정 모델: 4 × 16GB = 64GB. 이는 추출에만 3~4개의 GPU가 필요합니다.
하나의 기본 모델과 네 개의 LoRA 어댑터: 16GB + 4 × 250MB = 17GB. 이는 단일 RTX 3090에 KV 캐시(KV cache)를 위한 공간까지 포함하여 들어갑니다.
동일한 도메인 커버리지(domain coverage)에 99.7% 적은 메모리를 사용합니다.

하지만 이론이 실제로도 통할까요? 도메인별 어댑터에 대한 주요 관심사는 치명적인 망각(catastrophic forgetting)입니다. 즉, 법률 텍스트에 특화될 때 모델이 유효한 JSON을 생성하거나 일반적으로 엔티티(entity)를 추출하는 능력을 잃지 않을까요?

여기서 Biderman et al.의 2024년 ICML 논문 "LoRA Learns Less and Forgets Less"가 직접적으로 관련됩니다. 그들은 LoRA와 전체 미세 조정이 동일한 파레토 곡선(Pareto curve)에 있음을 보여주었습니다. LoRA는 대상 도메인을 덜 학습하지만, 기본 모델이 이미 알고 있는 것을 덜 망각합니다. 다중 도메인 추출의 경우, 이러한 절충(tradeoff)은 우리가 정확히 원하는 것입니다. 각 어댑터는 기본 모델의 구조적 JSON 생성 능력이나 일반 엔티티 인식(entity recognition) 능력을 파괴하지 않으면서 해당 도메인의 술어 어휘에 특화됩니다. 망각 저항성은 한계가 아니라 오히려 핵심적인 특징입니다.

이를 지원하는 인프라(infrastructure) 또한 성숙해졌습니다. S-LoRA(Sheng et al., 2023)는 통합 메모리 페이징(unified memory paging)을 사용하여 단일 GPU에서 수천 개의 동시 LoRA 어댑터를 서비스하는 것을 시연했습니다. Punica(Chen et al., 2023)는 다양한 어댑터에 걸쳐 동시에 추론(inference)을 일괄 처리(batching)하는 SGMV 커널(kernel)을 도입하여, 순차적 서비스보다 12배 향상된 처리량(throughput)을 달성했습니다. vLLM은 이 두 가지를 네이티브(natively)로 채택했습니다. 이는 실험적인 것이 아니라, 실제 환경에서 검증된 구현을 가진 프로덕션 인프라입니다.

## 저희의 파이프라인: 분류, 라우팅, 추출

아키텍처(architecture)는 세 가지 단계로 구성됩니다: 문서의 도메인을 결정하는 경량 분류기(lightweight classifier), 적절한 LoRA 어댑터를 선택하는 라우팅 레이어(routing layer), 그리고 추출 모델(extraction model) 자체입니다. 각 단계는 개별적으로 간단합니다. 이들이 함께 다중 도메인 문제를 해결합니다.

### 1단계: 문서 분류

라우팅 결정(어떤 어댑터가 이 문서를 처리해야 하는가?)은 빨라야 하며 완벽할 필요는 없습니다. 저희는 세 가지 접근 방식을 고려했으며, 각각 다른 장단점을 가집니다.

가장 간단한 방법은 메타데이터 기반 라우팅(metadata-based routing)입니다. 대부분의 프로덕션 파이프라인에서 문서들은 맥락(context)과 함께 도착합니다: 어떤 시스템에서 왔는지, 어떤 부서가 업로드했는지, 파일 유형은 무엇인지. 법률 문서 관리 시스템은 법률 어댑터에, 임상 기록 데이터베이스(database)는 의료 어댑터에 정보를 제공합니다. 이는 지연 시간(latency)을 전혀 추가하지 않으며 모델도 필요 없습니다. 깔끔한 메타데이터가 있는 파이프라인에 이것이 올바른 답입니다.

메타데이터를 사용할 수 없을 때는 경량 분류기가 잘 작동합니다. 미세 조정된 DistilBERT 또는 RoBERTa-small 모델은 CPU에서 5ms 이내에 문서를 도메인 카테고리(domain category)로 분류합니다. 이 모델들은 250MB로 단일 LoRA 어댑터와 유사하며, 수백 개의 라벨링된 문서 헤더(header)로 훈련할 수 있습니다. 특허 출원서와 임상 시험 보고서를 구별하기 위해 80억 개의 파라미터(parameter)가 필요하지 않습니다.

세 번째 옵션은 임베딩 기반 라우팅(embedding-based routing)입니다: 문서 임베딩(embedding)을 계산하고, 도메인 중심 벡터(domain centroid vectors)와 비교하여 가장 가까운 도메인으로 라우팅합니다. 도메인별 몇 가지 예시 문서에서 중심 벡터를 계산하는 것 외에는 훈련이 필요 없습니다. 훈련된 분류기보다 정확도는 약간 떨어지지만 즉시 배포 가능합니다.

저희는 모든 접근 방식에 신뢰도 임계값(confidence threshold)을 추가합니다. 분류기의 신뢰도가 70% 미만으로 떨어지면, 해당 문서는 도메인 전문화 없이 REBEL 베이스(base)로 훈련된 범용 어댑터(general-purpose adapter)로 라우팅됩니다. 잘못된 도메인의 어휘를 적용하는 것보다 일반 술어(generic predicates)로 추출하는 것이 더 낫습니다.

### 2단계: vLLM을 통한 어댑터 라우팅

서비스 레이어(serving layer)는 멀티 LoRA의 마법이 일어나는 곳입니다. vLLM은 `--enable-lora` 플래그(flag)를 통해 멀티 LoRA 어댑터를 네이티브로 지원합니다. 서버는 기본 모델을 한 번 로드하고 어댑터 가중치(weight)를 GPU 메모리(핫, hot) 또는 CPU 메모리(웜, warm)에 유지하며 필요에 따라 스왑(swap)합니다.

서버는 다음 명령으로 시작합니다:

```bash
vllm serve Qwen/Qwen3-8B \
  --enable-lora \
  --max-loras 4 \
  --max-lora-rank 64 \
  --max-cpu-loras 8 \
  --lora-modules \
    legal=./adapters/legal \
    medical=./adapters/medical \
    financial=./adapters/financial \
    technical=./adapters/technical
```

`--max-loras 4`는 네 개의 어댑터를 GPU 메모리에 동시에 유지합니다. `--max-cpu-loras 8`은 추가로 네 개의 어댑터를 CPU 메모리에 보관하여 빠른 스왑인을 가능하게 합니다. `--max-lora-rank`는 훈련 시 사용된 랭크(rank)와 일치해야 합니다. 불일치 시 조용히 잘못된 출력물을 생성합니다.

API는 OpenAI와 호환됩니다. 라우팅 레이어는 단순히 모델 파라미터(parameter)를 적절한 어댑터 이름으로 설정합니다:

```python
response = client.chat.completions.create(
    model="legal",  # 법률 LoRA 어댑터로 라우팅
    messages=[
        {"role": "system", "content": EXTRACTION_PROMPT},
        {"role": "user", "content": document_text}
    ]
)
```

미리 로드된 어댑터 간 전환은 사실상 무료입니다. 모델 재로드가 아니라 어텐션(attention) 레이어의 포인터 스왑(pointer swap)입니다. 어댑터가 CPU 메모리에서 로드되어야 하는 경우, 오버헤드(overhead)는 50~100ms입니다. 저희는 각 어댑터에 더미 요청(dummy request)을 보내 초기화하여 LRU 캐시(cache)를 미리 로드합니다(prime).

vLLM에 통합된 Punica의 SGMV 커널(kernel)은 서로 다른 어댑터를 대상으로 하는 요청들이 동일한 포워드 패스(forward pass)에서 일괄 처리될 수 있음을 의미합니다. 세 개의 법률 문서와 두 개의 의료 문서를 포함하는 배치(batch)는 동시에 처리됩니다. 도메인별로 먼저 정렬할 필요가 없습니다.

### 3단계: 도메인별 추출

모델 관점에서는 어댑터 간에 변경되는 것이 없습니다. 시스템 프롬프트(system prompt)는 동일합니다. JSON 스키마(schema)는 동일합니다. 기본 모델은 동일합니다. 변경되는 것은 어댑터 가중치에 학습된 동작입니다: 모델이 어떤 술어를 생성하고, 어떤 엔티티 유형을 인식하며, 복잡한 구절을 원자적(atomic) 트리플로 분해하는 방법입니다.

다음은 구체적인 예시입니다. 제약 회사의 인수합병에 대한 구절이 주어졌을 때, 동일한 시스템 프롬프트에 다른 어댑터를 적용하면 다음과 같은 결과가 나옵니다:

법률 어댑터:

```json
{
  "entities": [
    {"name": "Pfizer", "type": "Party"},
    {"name": "Seagen", "type": "Party"},
    {"name": "SEC", "type": "Regulatory_Body"}
  ],
  "triples": [
    {"subject": "Pfizer", "predicate": "acquired", "object": "Seagen"},
    {"subject": "SEC", "predicate": "approved_merger", "object": "Pfizer"}
  ]
}
```

금융 어댑터:

```json
{
  "entities": [
    {"name": "Pfizer", "type": "Company"},
    {"name": "Seagen", "type": "Company"},
    {"name": "$43 billion", "type": "Amount"}
  ],
  "triples": [
    {"subject": "Pfizer", "predicate": "acquired_for", "object": "$43 billion"},
    {"subject": "Seagen", "predicate": "acquired_by", "object": "Pfizer"}
  ]
}
```

동일한 구절, 동일한 모델 가중치, 동일한 프롬프트입니다. 다른 어댑터는 다른 추출 초점을 가집니다. 법률 어댑터는 규제 관계를 나타내고 `Party` 엔티티를 사용합니다. 금융 어댑터는 거래 금액을 나타내고 `Company` 엔티티를 사용합니다. 둘 다 올바르고 유용하며, 각각 다른 다운스트림(downstream) 쿼리 패턴(query pattern)에 사용됩니다.

추출 형식은 모든 어댑터에서 동일합니다. 즉, 동일한 엔티티 및 트리플 구조를 가진 동일한 JSON 스키마입니다. 이는 다운스트림 그래프 수집(graph ingestion)이 도메인별로 변경될 필요가 없음을 의미합니다. 단일 파이프라인이 어떤 어댑터가 트리플을 생성했는지에 관계없이 Neo4j에 데이터를 기록합니다.

## 여러 어댑터 훈련: 실용 가이드

### 데이터 준비

각 도메인 어댑터는 자체 훈련 세트(training set)가 필요하지만, 모두 공통 기반을 공유합니다. 저희가 채택한 방식은 두 개의 레이어(layer)를 가집니다.

기본 레이어는 모든 어댑터에서 동일합니다: 2,500~3,000개의 REBEL 샘플에서 품질(100~1500자 구절, 각 2~15개 트리플, 자기 참조 트리플 또는 날짜 엔티티 없음)에 따라 필터링된 데이터입니다. 이는 모델에 구조적 추출 패턴(structural extraction pattern)을 가르칩니다. 즉, 구절을 읽고, 엔티티를 식별하고, 트리플을 형성하고, 유효한 JSON을 출력하는 방법입니다. REBEL 변환 파이프라인은 2부에서 자세히 설명했습니다.

도메인 레이어는 각 어댑터를 차별화합니다: 해당 도메인이 요구하는 정확한 술어 어휘와 엔티티 유형을 사용하여 200~500개의 수동으로 라벨링된 예시입니다. 이 샘플들은 길거나 복잡할 필요가 없습니다. 중요한 것은 일관성입니다. 모든 예시는 동일한 술어 이름, 동일한 엔티티 유형 인벤토리(inventory), 그리고 동일한 세분화된 트리플 분해 규칙을 사용합니다.

저희는 이들을 대략 REBEL 85%, 도메인별 15%의 비율로 혼합합니다. REBEL 베이스는 작은 도메인 세트에 과적합(overfitting)되는 것을 방지하고, 도메인 오버레이(overlay)는 모델에 어떤 술어를 사용하고 어떤 엔티티 유형을 인식해야 하는지를 정확히 가르칩니다.

데이터 형식은 도메인에 걸쳐 동일합니다: 시스템 프롬프트, 사용자 메시지(구절), 어시스턴트 메시지(추출 JSON)를 포함하는 채팅 형식 JSONL(chat-format JSONL)입니다. 어시스턴트 메시지의 내용만 도메인 간에 다릅니다. 즉, 동일한 훈련 스크립트(script)가 모든 어댑터에 적용됩니다.

저희에게 상당한 재작업을 절약해 준 한 가지 실용적인 팁: 단 하나의 예시를 라벨링(labeling)하기 전에 술어 어휘를 정의하십시오. 도메인당 30~50개의 대상 술어 목록을 만들고 명확한 정의와 예시를 포함한 스프레드시트(spreadsheet)를 작성하십시오. `filed_lawsuit_against`와 `filed_suit_against`가 동일한 훈련 세트에 있으면 모델을 혼동시킬 것입니다. 술어의 일관성이 훈련 데이터의 양보다 훨씬 중요합니다.

### 훈련 구성

저희는 각 어댑터를 독립적으로 훈련하며, 절대로 함께 훈련하지 않습니다. 이는 Mixture-of-LoRAs 문헌(2024)에 의해 뒷받침되는 의도적인 선택이었습니다: 다른 도메인은 다른 데이터 규모와 훈련 난이도를 가지며, 공동 훈련은 작업 간 간섭을 유발합니다. 별도의 QLoRA 실행은 이를 완전히 피합니다.

결정적인 제약 조건: 모든 어댑터는 동일한 기본 모델 체크포인트(checkpoint)에서 훈련되어야 합니다. 이것이 어댑터를 교체 가능하게 만드는 이유입니다. 어댑터 A가 Qwen3–8B 커밋(commit) `abc123`에서 훈련되고 어댑터 B가 커밋 `def456`에서 훈련되었다면, 동일한 기본 모델에서 핫스왑(hot-swapped)될 때 올바르게 작동하지 않을 것입니다.

나머지 구성은 2부와 동일합니다:

*   랭크(Rank) 64, 알파(alpha) 128. 일반적인 기본값 r=16보다 높습니다. 도메인별 술어를 사용하는 구조적 추출에는 더 많은 어댑터 용량(capacity)이 필요합니다. 알파-랭크 비율 2는 학습률(learning rate) 스케일링(scaling)을 합리적으로 유지합니다.
*   코사인 스케줄(cosine schedule)을 사용한 학습률 3e-5. 전체 데이터셋에 대해 세 번의 에포크(epoch), 처음 3% 단계에 걸쳐 웜업(warmup).
*   4비트 양자화(quantization)를 사용한 QLoRA. 2개의 RTX 3090 GPU에 편안하게 들어갑니다.
*   사용자 턴 마스킹(User-turn masking). 손실(loss)은 입력 구절이 아닌 어시스턴트의 출력에 대해서만 계산됩니다.
*   훈련 시간: 어댑터당 30~45분. 네 개의 어댑터 = 총 2~3시간의 실제 시간(wall time). 며칠이 걸리지 않습니다.

### 가장 중요한 훈련 기법

저희 훈련 파이프라인에서 가장 큰 영향을 미치는 결정은 학습률이나 랭크 또는 에포크 수가 아닙니다. 그것은 일반 데이터 대 도메인별 데이터의 비율입니다.

순수하게 200개의 도메인별 예시로 훈련하면 술어 어휘에 과적합하지만 일반 추출 능력을 잃는 어댑터가 생성됩니다. `filed_lawsuit_against`를 출력하는 방법은 알지만 복잡한 다중 엔티티 구절을 원자적 트리플로 분해하는 방법을 잊어버립니다. 순수하게 REBEL로 훈련하면 그 반대의 결과가 나옵니다: 우수한 추출 구조를 가지지만 잘못된 술어를 사용합니다(2부의 문제).

혼합(blend)이 효과적입니다. REBEL은 "추출하는 방법"을 가르칩니다. 즉, 텍스트를 읽고, 엔티티를 식별하고, 주어-술어-목적어 트리플을 형성하고, 유효한 JSON을 출력하는 구조적 패턴입니다. 도메인 오버레이(overlay)는 "무엇을 추출할지"를 가르칩니다. 즉, 도메인이 요구하는 특정 술어, 엔티티 유형 및 분해 규칙입니다.

저희는 85/15(REBEL/도메인) 비율로 시작하여 검증 지표(validation metrics)를 기반으로 조정합니다. 어댑터가 올바른 술어를 생성하지만 트리플 분해가 불량하면 REBEL 비율을 늘립니다. 분해는 잘하지만 잘못된 술어를 사용하면 도메인 비율을 늘립니다. 실제로는 최적점(sweet spot)이 보통 80/20에서 90/10 사이입니다.

항상 도메인별 테스트 샘플과 일반 추출 홀드아웃(holdout) 데이터 모두에서 검증하십시오. 일반 추출 성능이 REBEL 전용 어댑터 대비 10% 이상 떨어지면 도메인 비율이 너무 높은 것입니다.

### 일반적인 함정 피하기

저희는 많은 어댑터를 훈련하면서 잘못되는 경우가 많다는 것을 알게 되었습니다. 다음은 간략한 요약입니다.

*   **어댑터 간 엔티티 유형 정렬(alignment).** 법률 어댑터가 `Organization`을 사용하고 금융 어댑터가 동일한 개념에 대해 `Company`를 사용하면, 다운스트림 지식 그래프에 병합되어야 할 중복 엔티티 유형이 생길 것입니다. 훈련 전에 모든 도메인에 걸쳐 공유 엔티티 유형 온톨로지(ontology)를 정의하십시오. `Statute` 또는 `Drug`과 같은 도메인별 유형은 공유 베이스를 확장하는 것이지 대체하는 것이 아닙니다.
*   **과도하게 훈련하지 마십시오(Overtrain).** 세 번의 에포크면 보통 충분합니다. LoRA는 하이퍼파라미터(hyperparameter)에 특히 민감합니다. Biderman et al. (2024)은 이를 명확하게 보여주었습니다. 검증 손실(validation loss)을 주시하고, 손실이 증가하면 조기에 중단하십시오. 저희는 훈련 손실이 계속 감소하더라도 4~5 에포크 이후 어댑터 성능이 저하되는 것을 목격했습니다.
*   **기본 모델도 테스트하십시오.** 법률 어댑터를 훈련한 후, 2부의 일반 REBEL 평가를 통해 실행하십시오. 엔티티 F1이 0.6 미만으로 떨어지거나 스키마 적합성이 95% 미만으로 떨어지면 뭔가 잘못된 것입니다. 아마도 도메인별 데이터가 너무 많거나 학습률이 너무 높았을 것입니다.
*   **어댑터 버전을 관리하십시오.** 프로덕션 환경에서는 도메인 어휘가 진화하고, 새로운 술어 유형이 출현하며, 추출 오류를 수정함에 따라 어댑터를 업데이트하게 될 것입니다. 어댑터를 모델 아티팩트(artifact)처럼 다루십시오: 버전을 관리하고, 훈련 데이터 출처(provenance)를 추적하고, 평가 지표(evaluation metrics)를 기록하며, 롤백(rollback) 사본을 보관하십시오. 잘못된 어댑터 업데이트가 프로덕션에 적용되면 지식 그래프에 손상된 트리플이 발생할 것입니다.

## 시스템 설정: 훈련부터 서비스까지

### 하드웨어 요구 사항

멀티 LoRA 접근 방식의 장점은 이미 가지고 있을 가능성이 있는 하드웨어에 잘 맞는다는 것입니다.

개발 및 개념 증명(proof of concept)을 위해서는 단일 RTX 3090(24GB)으로 충분합니다. 한 번에 하나의 어댑터를 훈련하고 2~3개의 동시 어댑터를 서비스할 수 있습니다. 이것이 2부의 실험에서 저희가 사용한 설정입니다.

소규모 프로덕션의 경우, 두 개의 RTX 3090 또는 단일 A100(40GB)이 최적점(sweet spot)입니다. 두 개의 GPU에 걸쳐 accelerate를 사용하여 훈련하면 훈련 시간이 절반으로 단축됩니다(어댑터당 33분에서 20분 미만으로). 4~8개의 어댑터 서비스는 편안합니다.

확장하려면 (16개 이상의 도메인 어댑터를 서비스하거나 수백 개의 동시 추출 요청을 처리하는 경우) A100 80GB 또는 H100이 어댑터 가중치와 높은 동시성(concurrency)에서 메모리를 지배하는 KV 캐시(KV cache) 모두를 위한 공간을 제공합니다.

저장은 결코 병목 현상(bottleneck)이 아닙니다. 각 어댑터는 랭크 64에서 약 250MB입니다. 100개의 어댑터는 디스크(disk)에서 25GB로, 기본 모델보다 작습니다.

### 훈련 파이프라인

새로운 도메인 어댑터를 추가하기 위한 엔드투엔드(end-to-end) 흐름은 다음과 같습니다:

**1단계: 훈련 데이터 준비.** REBEL 변환 파이프라인(`prepare_rebel.py`)을 실행하여 기본 레이어를 준비합니다. 술어 어휘를 따라 200~500개의 도메인별 예시를 수동으로 라벨링합니다. 이를 단일 JSONL 파일로 혼합합니다.

**2단계: 훈련.** `accelerate`를 사용하여 QLoRA 훈련을 실행합니다:

```bash
accelerate launch --num_processes 2 scripts/train_server.py \
  --data data/legal_train.jsonl \
  --output outputs/legal-adapter
```

**3단계: 평가.** 어댑터를 도메인별 벤치마크(benchmark) 및 일반 REBEL 홀드아웃(holdout) 데이터에 대해 테스트합니다:

```bash
python scripts/evaluate_local.py \
  --model-path outputs/legal-adapter-merged \
  --compare-with results/baseline_aggregate.csv
```

**4단계: 배포.** 어댑터 가중치를 서비스 디렉토리(directory)에 복사하고 vLLM에 등록합니다. 서버가 이미 실행 중인 경우, vLLM은 새 어댑터를 동적으로 인식하므로 재시작할 필요가 없습니다.

"새 도메인 어댑터가 필요하다"는 시점부터 "프로덕션에 서비스 중이다"는 시점까지 전체 주기는 하루가 걸립니다: 데이터 라벨링에 반나절, 훈련에 한 시간, 평가에 한 시간, 나머지는 검토 및 배포에 소요됩니다.

### 서비스 아키텍처

```
┌──────────────┐     ┌─────────────────┐     ┌──────────────────────┐
│   문서 수집  │────>│     분류기      │────>│    vLLM + 멀티 LoRA  │
│(Document Ingestion)│     (DistilBERT /  │     │                      │
│              │     │    메타데이터)    │     │  기본 모델: Qwen3-8B   │
└──────────────┘     └─────────────────┘     │  ├─ 법률 어댑터      │
                            │                │  ├─ 의료 어댑터      │
                       도메인 ID             │  ├─ 금융 어댑터      │
                            │                │  └─ 기술 어댑터      │
                            └───────────────>│                      │
                                             └──────────┬───────────┘
                                                        │
                                                   JSON 트리플
                                                        │
                                             ┌──────────▼───────────┐
                                             │    지식 그래프     │
                                             │   (Neo4j / Neptune)  │
                                             └──────────────────────┘
```

주요 vLLM 구성 플래그(flag)는 다음과 같습니다:

`--enable-lora`는 멀티 LoRA 지원을 활성화합니다. `--max-loras 4`는 몇 개의 어댑터가 GPU 메모리에 동시에 남아 있을지를 제어하며, 동시에 서비스할 것으로 예상되는 도메인 수에 맞춰 설정합니다. `--max-lora-rank 64`는 훈련 시 사용된 랭크와 일치해야 합니다. 불일치 시 조용히 잘못된 출력물을 생성합니다. `--max-cpu-loras 8`은 추가 어댑터를 CPU 메모리에 보관하여 디스크에서 재로딩할 필요 없이 빠르게 스왑인할 수 있도록 합니다. `--lora-modules`는 사람이 읽을 수 있는 어댑터 이름을 가중치 디렉토리(weight directory)에 매핑합니다.

요청 시에는 분류기의 출력이 vLLM 모델 이름에 직접 매핑됩니다. 라우팅 로직(routing logic)은 API 호출에서 모델 파라미터를 설정하는 단일 `if` 문입니다. 오케스트레이션 프레임워크(orchestration framework), 메시지 큐(message queue) 등 필요한 것 이상의 복잡성은 없습니다.

## 다음 단계: 자동 라우팅 및 어댑터 퓨전

저희의 현재 파이프라인은 문서 수준 분류를 사용합니다: 하나의 문서, 하나의 어댑터, 하나의 추출. 이는 저희 작업량의 대부분을 처리합니다. 하지만 일부 문서는 진정으로 멀티 도메인(multi-domain)입니다. 예를 들어, 금융 상품을 논의하는 법률 서류나 규제 준수 섹션이 포함된 임상 시험 보고서와 같은 경우입니다. 이러한 경우, 하나의 어댑터를 선택하면 다른 도메인의 추출 품질을 잃게 됩니다.

연구 커뮤니티는 이 문제를 적극적으로 연구하고 있으며, 몇 가지 접근 방식은 주목할 가치가 있습니다.

LoRA-Switch(Kong et al.)는 융합된 CUDA 커널(fused CUDA kernels)을 사용하는 토큰 수준 동적 라우팅(token-level dynamic routing)을 도입합니다. 문서당 하나의 어댑터를 선택하는 대신, 모델은 포워드 패스(forward pass) 중에 개별 토큰을 다른 어댑터로 라우팅합니다. 시스템-알고리즘 공동 설계(system-algorithm co-design)는 동적 라우팅의 지연 시간 오버헤드(latency overhead)를 정적 서비스 대비 2.5~3배에서 약 24%로 줄입니다. 동일한 단락에서 법률 및 금융 콘텐츠를 아우르는 구절의 경우, 이는 추출 품질을 의미 있게 향상시킬 수 있습니다.

LORAUTER(Ostapenko et al.)는 다른 접근 방식을 취합니다: 문서 분류가 아닌 태스크 표현(task representation)을 기반으로 라우팅합니다. 어댑터는 입력과 학습된 태스크 벡터(task vector) 간의 임베딩 유사성(embedding similarity)을 기반으로 선택됩니다. 별도의 분류기를 유지하는 것보다 더 우아하며, 어댑터 수보다는 태스크 수에 따라 확장됩니다.

MoLoRA(Li et al.)는 가장 진보된 방식으로, 학습된 게이팅(gating)을 통해 토큰별 LoRA 전문가 혼합(per-token mixture of LoRA experts)을 사용합니다. 주목할 만한 결과는 MoLoRA 라우팅을 사용하는 1.7B 모델이 추론 벤치마크에서 바닐라(vanilla) 8B 모델을 능가한다는 것입니다. 다중 도메인 추출의 경우, 이것이 가장 유망한 방향입니다. 하지만 아직 생산 배포하기에는 너무 새롭고, 프레임워크 통합은 초기 단계에 있습니다.

어댑터 퓨전(Adapter fusion) 및 LoRA 수프(LoRA Soups, Zhao et al.)는 반대 접근 방식을 취합니다: 어댑터 간 전환 대신, 추론 시 학습된 혼합 계수(mixing coefficients)로 가중치를 병합합니다. 이는 도메인이 겹칠 때 잘 작동합니다. 법률-금융 계약은 둘 중 하나를 명확히 선택하는 것보다 두 어댑터의 혼합에서 이점을 얻습니다. LoRAHub는 이것이 교차 작업 구성(cross-task composition)에 실용적임을 보여주었습니다.

저희 파이프라인의 경우, 문서 수준 라우팅은 95%의 경우를 깔끔하게 처리합니다. 나머지 5%인 모호하거나 진정으로 멀티 도메인 문서에서 이러한 기술들이 중요해질 것입니다. 저희는 동적 접근 방식 중 가장 성숙한 LoRA-Switch의 vLLM 통합을 가장 면밀히 주시하고 있습니다.

## 핵심 요약

*   **하나의 기본 모델 + N개의 어댑터가 N개의 개별 모델보다 우수합니다.** 메모리 계산은 압도적입니다: 네 개의 도메인에 대해 17GB 대 64GB를 단일 GPU에서 처리합니다. 어댑터는 모델 재로드 없이 핫스왑(hot-swappable) 가능하며, vLLM의 SGMV 커널은 동일한 포워드 패스에서 서로 다른 어댑터에 걸쳐 일괄 처리합니다.
*   **어댑터를 공동으로 훈련하지 않고 독립적으로 훈련하십시오.** 각 도메인은 자체 QLoRA 실행입니다: 동일한 기본 모델, 동일한 하이퍼파라미터(hyperparameter), 다른 훈련 데이터. 200~500개의 도메인별 예시를 REBEL 베이스와 혼합합니다. 일반 소비자 GPU에서 어댑터당 30~45분 소요됩니다.
*   **문서 분류는 라우팅 레이어이지 병목 현상(bottleneck)이 아닙니다.** 작은 분류기, 메타데이터 규칙 또는 임베딩 유사성(embedding similarity)이 라우팅을 처리합니다. 어댑터 선택은 완벽할 필요는 없지만, 대부분 정확해야 하며 불확실한 경우에는 범용 폴백(fallback)이 있어야 합니다.
*   **vLLM은 이를 오늘날 바로 프로덕션에 적용 가능하게 만듭니다.** 네이티브 멀티 LoRA 지원, OpenAI 호환 API, LRU 캐싱(caching), 동시 어댑터 일괄 처리(batching). 인프라는 존재하며, 엔지니어링 노력은 서비스 레이어가 아닌 훈련 데이터에 있습니다.
*   **LoRA의 망각 저항성은 핵심적인 특징입니다.** 각 어댑터는 기본 모델의 JSON 생성 또는 일반 추출 기능을 손상시키지 않으면서 전문화됩니다. Biderman et al. (2024)은 이를 경험적으로 보여주었습니다: LoRA는 덜 학습하면서 덜 망각하는 파레토 곡선에 있습니다. 모듈식 도메인 어댑터의 경우, 이러한 절충이 바로 원하는 것입니다.
*   **라벨링하기 전에 술어 어휘를 정의하십시오.** 이것이 추출 품질에 가장 큰 영향을 미칠 수 있는 단일 사항입니다. 미리 정의된 도메인당 30~50개의 깨끗하고 일관된 술어 세트는 일관성 없는 라벨링으로 훈련 데이터를 두 배로 늘리는 것보다 더 나은 결과를 낳습니다.

이것은 저희 그래프 RAG 추출 시리즈의 3부입니다. 1부에서는 추출을 위한 로컬 LLM 벤치마킹을 다룹니다. 2부에서는 미세 조정 및 도메인 전이 문제를 다룹니다. 모든 실험은 NVIDIA RTX 3090 GPU에서 vLLM을 추론(inference)에, TRL/PEFT를 미세 조정에 사용하여 실행되었습니다.
