# 16GB VRAM GPU 환경에서 Ollama용 최적의 LLM

거대 언어 모델(LLM)을 로컬에서 실행하면 데이터 프라이버시를 보호할 수 있고, 오프라인 사용이 가능하며, API 비용이 전혀 들지 않는다는 장점이 있습니다. 본 벤치마크는 NVIDIA RTX 4080 환경에서 Ollama를 통해 실행한 9가지 인기 모델의 성능 측정 결과를 담고 있습니다.

16GB VRAM GPU를 사용할 때는 성능과 속도 사이에서 끊임없는 선택(Trade-off)을 해야 합니다. 더 높은 품질을 위해 큰 모델을 선택할 것인가, 아니면 빠른 추론(Inference)을 위해 작은 모델을 선택할 것인가의 문제입니다.

### 핵심 요약 (TL;DR)

Ollama 0.15.2 버전과 RTX 4080 16GB 환경에서의 성능 비교 결과, **모델이 VRAM에 완전히 적재될 때 속도가 비약적으로 빨라집니다.** 예를 들어 VRAM에 모두 들어가는 GPT-OSS 20B 모델은 초당 139.93개의 토큰(tokens/sec)을 생성하는 반면, 대규모 CPU 오프로딩(CPU offloading)이 발생하는 GPT-OSS 120B 모델은 초당 12.64개에 그쳐 약 11배의 속도 차이를 보였습니다.

### 테스트 하드웨어 설정

*   **GPU:** NVIDIA RTX 4080 (16GB VRAM)
*   **CPU:** Intel Core i7-14700 (8 P-코어 + 12 E-코어)
*   **RAM:** 64GB DDR5-6000

16GB VRAM은 로컬 LLM 추론에서 가장 중요한 제약 요소입니다. 이 용량에 따라 모델을 GPU에서 전체 실행할지, 아니면 CPU 오프로딩이 필요할지가 결정됩니다. VRAM 용량을 초과하여 CPU 코어를 사용하게 되면 추론 속도에 직접적인 영향을 미치게 됩니다.

### 벤치마크 목적 및 파라미터

본 테스트의 목적은 실제 사용 환경에서의 추론 속도를 측정하는 것입니다. 각 모델이 텍스트를 얼마나 빨리 생성하는지, 그리고 VRAM 한계를 넘었을 때 발생하는 속도 저하가 어느 정도인지 파악하고자 했습니다.

*   **컨텍스트 크기(Context size):** 19,000 토큰
*   **프롬프트:** "호주의 주요 도시 간 날씨와 기후 비교"
*   **측정 지표:** 초당 토큰 생성 수(eval rate, tokens/s)

---

### CPU 오프로딩(CPU Offloading)의 이해

모델의 메모리 요구량이 가용 VRAM을 초과하면, Ollama는 모델 레이어를 GPU와 시스템 RAM(CPU)에 자동으로 분산 배치합니다. 이 경우 CPU와 GPU 간의 데이터 전송 병목 현상이 발생하여 성능이 크게 저하됩니다.

*   **100% GPU 모델:** 61~140 tokens/sec
*   **70~82% GPU 활용 모델:** 19~51 tokens/sec
*   **22% GPU 활용 모델 (대부분 CPU):** 12.6 tokens/sec

### 상세 벤치마크 결과

#### 1. 100% GPU로 실행되는 모델

*   **GPT-OSS 20B — 속도 챔피언**
    *   **속도:** 139.93 tokens/s
    *   **설명:** 속도가 중요한 애플리케이션에서 압도적인 승자입니다. 14GB의 VRAM만 사용하여 더 큰 컨텍스트 윈도우를 확보할 여유가 있습니다.
*   **Qwen3 14B — 최적의 균형**
    *   **속도:** 61.85 tokens/s
    *   **설명:** 지시 이행(Instruction-following) 능력이 가장 뛰어나며, 12GB 메모리 점유율로 대화형 서비스에 적합한 반응 속도를 보여줍니다.
*   **Ministral 3 14B — 빠르고 콤팩트함**
    *   **속도:** 70.13 tokens/s
    *   **설명:** Mistral 계열의 품질을 유지하면서 빠른 속도를 원하는 경우 훌륭한 선택지입니다.

#### 2. CPU 오프로딩이 필요한 모델

*   **Qwen3-VL 30B — 부분 오프로딩 성능 우수**
    *   **속도:** 50.99 tokens/s
    *   **설명:** 레이어의 30%를 CPU에서 처리함에도 불구하고 상당히 빠른 속도를 유지합니다. 시각 기능(Vision)을 지원하여 다중 모드(Multimodal) 작업이 가능합니다.
*   **Mistral Small 3.2 24B — 품질과 속도의 타협**
    *   **속도:** 18.51 tokens/s
    *   **설명:** 언어 품질은 우수하지만 속도 저하가 뚜렷합니다. 실시간 채팅보다는 지연 시간이 덜 중요한 작업에 적합합니다.
*   **GLM 4.7 Flash — MoE 사고 모델**
    *   **속도:** 33.86 tokens/s
    *   **설명:** 전문가 혼합(MoE, Mixture of Experts) 구조를 가진 모델로, 답변 전 내부 추론 과정을 거칩니다. CPU 오프로딩이 있음에도 구조적 이점 덕분에 합리적인 속도를 보여줍니다.
*   **GPT-OSS 120B — 대형 모델**
    *   **속도:** 12.64 tokens/s
    *   **설명:** 16GB VRAM에서 실행은 가능하지만 매우 느립니다. 실시간 대화보다는 속도가 중요하지 않은 배치 처리(Batch processing)에 적합합니다.

### 실무 권장 사항

**1. 대화형 채팅용**
VRAM에 100% 적재되는 모델을 권장합니다.
*   GPT-OSS 20B (최대 속도)
*   Ministral 3 14B (Mistral 품질)
*   Qwen3 14B (최고의 지시 이행 능력)

**2. 배치 처리용**
속도보다 품질이 중요한 경우에 적합합니다.
*   Mistral Small 3.2 24B (우수한 언어 품질)
*   Qwen3-VL 30B (시각 및 텍스트 기능)

### 결론

16GB VRAM 환경에서도 모델을 현명하게 선택한다면 매우 인상적인 속도로 LLM을 구동할 수 있습니다.

1.  **VRAM 한계 준수:** 실시간 대화가 목적이라면 VRAM 내에 모델을 모두 넣으십시오. 초당 140토큰의 20B 모델이 초당 12토큰의 120B 모델보다 훨씬 실용적입니다.
2.  **균형 잡힌 선택:** 순수 속도는 GPT-OSS 20B가 빠르지만, 지시 이행 능력까지 고려하면 Qwen3 14B가 가장 균형 잡힌 선택입니다.
3.  **CPU 오프로딩의 한계:** 오프로딩 시 3~10배의 속도 저하를 감수해야 합니다.
4.  **컨텍스트 크기 조절:** 19K와 같은 큰 컨텍스트는 VRAM 사용량을 크게 늘립니다. GPU 활용도를 높이려면 컨텍스트 크기를 적절히 제한하는 것이 좋습니다.
