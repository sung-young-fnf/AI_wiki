# Mac Mini의 은닉 상태: 실제 ML 연구를 위한 MLX 활용

대부분의 머신러닝(Machine Learning) 연구는 PyTorch를 실행하는 NVIDIA 하드웨어에서 이루어집니다. 툴체인(toolchain)은 성숙하고 생태계는 방대하며, 프로빙 실험(probing experiments)을 위해 트랜스포머(transformer)에서 내부 표현(internal representations)을 추출해야 한다면 HuggingFace와 CUDA GPU를 사용합니다. 이것이 기본 경로이며 잘 작동합니다.

저는 더 저렴한 경로를 택했습니다. 지난 몇 달 동안 저는 현재 준비 중인 논문을 위해 MLX를 주요 연구 플랫폼으로 사용하여 Apple Silicon에서 전적으로 대규모 언어 모델(LLM) 행동 연구를 위한 실험 파이프라인(experimental pipeline)을 운영해왔습니다.

이 게시물은 실제 경험이 어떠했는지에 대한 내용입니다. MLX가 연구에 충분히 잘 작동하는 부분, 추가적인 엔지니어링이 필요한 부분, 그리고 소비자용 하드웨어(consumer hardware)를 진지한 실험 플랫폼으로 다루는 것에 대해 제가 배운 점들을 다룹니다.

## 연구 문제 요약

저는 모델이 기본 지식과의 관계에 따라 질문 범주별로 LLM이 어떻게 다르게 동작하는지 연구하고 있습니다. 자세한 내용은 논문에 담을 예정이지만, 여기서는 엔지니어링 요구사항이 중요합니다. 즉, 신중하게 선별된 질문으로 구성된 벤치마크(benchmark), 여러 프롬프트 조건(prompting conditions)에서 모델 응답을 테스트하는 평가 시스템, 그리고 모델이 각 레이어에서 생성하는 것을 기반으로 선형 프로브(linear probes)를 훈련하기 위한 레이어별 활성화 추출(per-layer activation extraction)입니다. 이 모든 것은 32GB 통합 메모리(unified memory)를 갖춘 Mac Mini M4에서 mlx-community 저장소(repos) 중 하나에서 가져온 4비트 양자화(4-bit quantized)된 Mistral 7B를 사용하여 실행됩니다.

## MLX를 선택한 이유

저는 CUDA 클러스터(cluster)에 접근할 수 없는 독립 연구자이며, Mac Mini가 제가 가진 전부입니다.

하지만 덜 명확한 대답은 이러한 제약이 유용한 강제 함수(forcing function) 역할을 했다는 것입니다. 추론 예산(inference budget)이 단일 소비자용 기계인 경우, 실험 설계에 대해 더 깊이 고민하게 됩니다. 제대로 설계되지 않은 스크리닝 프로세스(screening process)를 더 많은 GPU를 투입하여 무작정 밀어붙일 수는 없습니다. 각 후보 질문은 결정론적 시딩(deterministic seeding)을 통해 10번의 추론 패스(inference passes)를 거치며, 낭비되는 토큰(tokens)은 병렬화할 수 없는 실제 시간(wall-clock time)을 소모합니다.

그 결과, 고정된 시드(pinned seeds)와 재현 가능한 스크리닝 실행(replayable screening runs)을 통해 기본적으로 재현 가능한 설정(reproducible setup)을 얻게 됩니다. 하드웨어 제약은 실험 설계에 더 규율을 요구합니다.

4비트 양자화(4-bit quantization)에는 논의할 가치가 있는 절충점(trade-off)이 있습니다. 양자화는 모델의 가중치(weights)를 압축하며, 이 압축은 정보를 손실시킵니다. 4비트 모델의 표현(representations)은 전체 정밀도(full-precision) 또는 8비트 버전보다 더 노이즈(noisier)가 많습니다. 즉, 활성화(activations)의 미묘한 효과가 사라지거나 모델의 행동 패턴이 약간 저하될 수 있습니다. 미세한 표현 차이에 의존하는 프로빙 실험(probing experiments)의 경우, 8비트 또는 전체 정밀도 가중치가 더 바람직할 것입니다. 32GB 통합 메모리에서는 4비트(약 4GB)의 7B 모델이 추출 파이프라인과 함께 편안하게 들어맞으며, 8비트(약 8GB)에서는 여전히 맞지만 여유 공간이 줄어들고, 전체 정밀도(약 14GB)에서는 빡빡할 것입니다.

실제로는 4비트에서 시작하는 주장은, 양자화 노이즈(quantization noise)를 뚫고 나타날 만큼 효과가 강하다면 그것은 아마 인공물(artifact)이 아닐 것이라는 점입니다. 4비트 활성화(activations)에서 명확한 분리(separation)를 달성하는 프로브(probe)는 견고한 무언가를 측정하고 있는 것입니다. 만약 4비트에서 효과가 사라진다면, 그것이 부재했는지 아니면 단순히 노이즈 플로어(noise floor) 아래에 있었는지 알 수 없는데, 이는 진정한 한계입니다. 그러나 초기 검증(initial validation)을 위해서는 4비트 양자화가 보수적인 필터(conservative filter) 역할을 합니다. 즉, 이를 통과하는 효과는 더 높은 정밀도(higher precision)에서 추가 조사를 할 가치가 있습니다.

## 은닉 상태(Hidden States) 추출: 훅(Hook)

핵심 엔지니어링 과제는 MLX 모델의 각 레이어에서 표현(representations)을 추출하는 것이었습니다. HuggingFace 트랜스포머(transformers)는 `output_hidden_states=True`를 일급 매개변수(first-class parameter)로 제공하지만, MLX-LM은 그렇지 않습니다.

제 해결책은 내부 모델의 `__call__` 메서드를 임시로 교체하는 것입니다. MLX-LM 모델은 언어 모델 헤드(language model head, `lm_head`)를 소유하는 외부 래퍼(outer wrapper)와 임베딩 레이어(embedding layer), 트랜스포머 레이어(transformer layers), 그리고 최종 정규화(final norm)를 소유하는 내부 트랜스포머(`model.model`)의 두 단계 구조를 가집니다. 이 훅(hook)은 레이어를 수동으로 반복하며 각 레이어의 출력을 수집합니다.

다음은 올바르게 구현하기까지 시간이 걸렸던 호환성 해결책(compatibility workarounds)을 포함한 전체 구현 코드입니다.

```python
import mlx.core as mx
import mlx.nn as nn
from mlx_lm import load

model, tokenizer = load("mlx-community/Mistral-7B-Instruct-v0.3-4bit")

def extract_hidden_states(text: str) -> mx.array:
    """Extract per-layer hidden states. Returns (n_layers+1, seq_len, hidden_dim)."""
    tokens = tokenizer.encode(text)
    input_ids = mx.array([tokens])  # (1, seq_len)
    inner_model = model.model
    collected_states: list[mx.array] = []
    # Save the original __call__ at the class level.
    original_call = inner_model.__class__.__call__
    def patched_call(self_inner, x, cache=None, mask=None, **kwargs):
        # Embedding - models name this layer differently.
        if hasattr(self_inner, 'embed_tokens'):
            h = self_inner.embed_tokens(x)
        elif hasattr(self_inner, 'embedding'):
            h = self_inner.embedding(x)
        else:
            raise AttributeError("Cannot find embedding layer")
        # Causal mask.
        if mask is None and h.shape[1] > 1:
            mask = nn.MultiHeadAttention.create_additive_causal_mask(h.shape[1])
            mask = mask.astype(h.dtype)
        # Iterate through transformer layers.
        for i, layer in enumerate(self_inner.layers):
            try:
                if cache is not None:
                    try:
                        h = layer(h, mask=mask, cache=cache[i])
                    except TypeError as te:
                        # Cache signature varies across architectures.
                        if "cache" in str(te) or "unexpected keyword" in str(te):
                            h = layer(h, mask=mask)
                        else:
                            raise
                else:
                    h = layer(h, mask=mask)
            except Exception as e:
                raise RuntimeError(f"Layer {i} forward pass failed: {e}") from e
            # Some layers return (hidden_state, cache_update) tuples.
            if isinstance(h, tuple):
                h = h[0]
            collected_states.append(h)
        # Final norm.
        h = self_inner.norm(h)
        collected_states.append(h)
        return h  # Outer model applies lm_head
    try:
        inner_model.__class__.__call__ = patched_call
        _logits = model(input_ids)
        mx.eval(_logits)  # Force computation before collecting.
        result = mx.stack(collected_states, axis=0)
        result = result[:, 0, :, :]  # Remove batch dim.
        mx.eval(result)  # Force evaluation before return.
    finally:
        inner_model.__class__.__call__ = original_call
    return result
```

코드가 왜 이렇게 이상하게 보이는지에 대한 몇 가지 참고 사항입니다.

`__call__`을 인스턴스(instance) 수준이 아니라 클래스(class) 수준에서 교체합니다. 이것이 중요한 이유는 MLX-LM의 외부 모델이 `self.model(x)`를 호출하며, 이는 클래스의 `__call__`을 통해 디스패치(dispatch)되기 때문입니다. 인스턴스를 패치(patching)하는 것은 묵묵히 무시될 것입니다.

두 `mx.eval()` 호출은 겉보기보다 더 중요합니다. MLX는 지연 평가(lazy evaluation)를 사용합니다. 즉, 결과가 필요할 때까지 계산이 실행되지 않습니다. 첫 번째 `mx.eval(_logits)` 없이는 수집된 상태(collected states)를 스택(stack)하려고 할 때 계산 그래프(computation graph)가 구체화되지 않습니다. 두 번째 `mx.eval(result)` 없이는 스택된 텐서(stacked tensor)가 다음 추출 호출(extraction call)에서 덮어쓰여질 그래프 노드(graph nodes)를 참조할 수 있습니다. 이 버그는 간헐적으로, 특정 메모리 압력(memory pressure) 하에서만 나타나 오류 대신 조용히 잘못된 값을 생성했기 때문에 찾기 가장 어려웠습니다.

`try/finally`는 이러한 종류의 패치(patching)가 항상 위험한 것과 같은 이유로 중요합니다. 포워드 패스(forward pass)가 예외(throws)를 발생시키면 원래 메서드를 복원해야 합니다. 그렇지 않으면 단 한 번의 추출 실패로 모델이 사용할 수 없는 상태가 됩니다.

모든 방어적 코드(defensive code) — 임베딩 이름 대체(`embed_tokens` 대 `embedding`), 캐시 시그니처 처리(cache signature handling), 튜플 반환 검사(tuple return check) — 는 커뮤니티 모델들이 내부 규칙(internal conventions)이 다양하기 때문에 존재합니다. 이것들은 MLX 자체의 버그가 아니라, 모델 인터페이스(model interfaces)가 아직 완전히 표준화되지 않은 생태계의 결과입니다. 각 해결책(workaround)을 진단하는 데 한나절, 수정하는 데 몇 줄의 코드가 소요되었습니다.

## 활성화(Activations)에서 프로브(Probes)로: 전체 파이프라인

단일 입력에서 레이어 출력(layer outputs)을 얻는 것이 대부분의 엔지니어링이 집중되는 부분입니다. 다른 모든 것은 더 일반적이지만, MLX와 NumPy 간의 메모리 관리(memory management)는 문서화할 만큼 중요합니다.

프로빙 실험(probing experiments)의 경우, 데이터셋의 모든 질문에 대해 각 레이어의 마지막 토큰 표현(last-token representation)을 원합니다. 질문당 레이어별로 하나의 벡터를 얻고, 간단한 분류기(classifier, 로지스틱 회귀(logistic regression))가 각 레이어에서 범주를 얼마나 잘 구별할 수 있는지 독립적으로 비교합니다.

질문은 한 번에 하나씩 처리됩니다. MLX는 HuggingFace가 제공하는 패딩(padding)/어텐션 마스크(attention-mask) 인프라(infrastructure)가 없기 때문에 가변 길이 입력(variable-length inputs)에 대해 배칭(batching)이 잘 되지 않으므로, 각 질문은 자체 포워드 패스(forward pass)를 거칩니다.

```python
import numpy as np

def extract_batch(model, questions, prompt_prefix=""):
    """Extract last-token hidden states for a list of questions."""
    all_states = []
    for i, q in enumerate(questions):
        text = prompt_prefix + q.text
        # Returns mx.array of shape (n_layers + 1, hidden_dim).
        states_mx = extract_last_token(model, text)
        # Convert to numpy immediately to free MLX memory.
        states_np = np.array(states_mx)
        all_states.append(states_np)
    # Stack in numpy, not MLX.
    return np.stack(all_states, axis=0)  # (n_questions, n_layers+1, hidden_dim)


def extract_last_token(model, text):
    """Convenience: extract full hidden states, take last token."""
    all_states = extract_hidden_states(text)
    return all_states[:, -1, :]  # (n_layers + 1, hidden_dim)
```

각 패스(pass) 직후 `np.array()`로 변환하는 것이 메모리 경계를 유지하는 방법입니다. 이것이 없으면 MLX는 메모리 풀(memory pool)에 수백 개의 텐서(tensors)를 축적하고, 몇백 개의 질문을 처리하는 32GB 기계에서는 그 풀이 가득 차게 됩니다. NumPy로 변환하면 다음 포워드 패스에서 재사용할 수 있도록 MLX 메모리가 해제됩니다.

(질문 수, 레이어 수+1, 은닉 차원) 형태의 NumPy 배열(array)을 얻으면, 프로빙(probing)은 `scikit-learn`을 사용하여 간단하게 수행할 수 있습니다.

```python
from sklearn.linear_model import LogisticRegression
from sklearn.model_selection import StratifiedKFold
from sklearn.preprocessing import StandardScaler
from sklearn.metrics import roc_auc_score

def probe_layer(X, y, n_folds=5):
    """Train a binary probe at a single layer. Returns mean AUROC."""
    skf = StratifiedKFold(n_splits=n_folds, shuffle=True, random_state=42)
    aurocs = []
    for train_idx, test_idx in skf.split(X, y):
        scaler = StandardScaler()
        X_train = scaler.fit_transform(X[train_idx])
        X_test = scaler.transform(X[test_idx])
        clf = LogisticRegression(max_iter=1000, class_weight="balanced")
        clf.fit(X_train, y[train_idx])
        proba = clf.predict_proba(X_test)[:, 1]
        aurocs.append(roc_auc_score(y[test_idx], proba))
    return np.mean(aurocs)


# Run independently at every layer.
n_layers = hidden_states.shape[1]
```

각 레이어에서 독립적으로 프로브를 실행하면 모델에서 범주 간의 구별이 가장 강력한 지점을 보여주는 곡선이 생성됩니다. 제 실험에서는 신호가 초기-중간 레이어에서 최고조에 달하고 최종 레이어로 갈수록 약해지는데, 이는 모델이 정보를 처리하는 방식에 대해 알려줍니다(해석은 여기보다는 논문에 있습니다).

모든 것은 압축된 `.npz` 파일로 저장되고 로드되므로, 한 번 추출을 실행하고 추론을 반복할 필요 없이 원하는 만큼 여러 번 프로빙할 수 있습니다. Mac Mini에서는 몇백 개의 질문에 대한 레이어 출력을 가져오는 데 약 20분이 걸립니다. 모든 프로브를 훈련하는 데는 1초 미만이 소요됩니다.

## 대규모 결정론적 스크리닝(Deterministic Screening at Scale)

벤치마크 구성은 결정론적 시딩(deterministic seeding)을 사용하여 온도(temperature) 0.7에서 각 후보 질문을 10번의 추론 패스(inference passes)로 실행합니다. 각 시드(seed)는 `base_seed + (question_index * n_runs) + run_index`로 계산되며, 범주별로 다른 기본 시드(base seed)를 사용합니다. 이는 동일한 시드로 재실행함으로써 개별 스크리닝 결과가 정확히 재현될 수 있음을 의미합니다.

MLX의 난수 생성(random number generation)은 이를 잘 지원합니다. 각 확률적 생성 호출(stochastic generation call) 전에 코드는 `mx.random.seed(seed)`를 고정하며, 이는 동일한 입력에 대해 동일한 토큰 샘플링(token sampling)을 생성합니다. 온도 0 조건(평가 단계에서 사용됨)의 경우, 샘플러(sampler)는 난수 생성기(RNG)를 완전히 우회하고 로짓(logits)에 대해 `mx.argmax`를 사용합니다.

스크리닝 기준(screening criteria)은 범주에 따라 다릅니다. 어떤 것은 높은 정확도(accuracy)를 요구하고, 어떤 것은 특정 정확도 대역(accuracy bands)을 목표로 하며, 어떤 것은 일관된 실패(consistent failure)를 요구합니다. 각 스크리닝 함수(screening function)는 자체 필터(filter)를 적용하지만, 모두 동일한 결정론적 시딩 인프라(deterministic seeding infrastructure)를 공유합니다.

초기 파일럿 결과는 프로브(probes)가 모델의 내부 표현(internal representations)에서 분리(separation)를 감지하며, 신호가 최종 레이어보다는 초기 레이어에 집중되어 있음을 보여줍니다. 프로빙 실험(probing experiments)의 숫자를 신뢰하는 것은 그 밑에 깔린 실험 인프라(experimental infrastructure)에 달려 있습니다.

## Mistral 7B 출력 구문 분석(Parsing): 겉보기보다 어려움

`mlx-lm`을 통해 Mistral 7B로 평가 작업(evaluation work)을 수행한다면, 예상보다 출력 구문 분석(output parsing)에 더 많은 시간을 할애하게 될 것입니다. 이 모델의 응답 형식은 너무 다양하여 단일 정규 표현식(regex)으로는 모두 처리할 수 없으며, 실패 모드는 조용합니다. 점수 대신 `None`을 받거나, 더 나쁘게는 응답에서 잘못된 숫자를 추출하게 됩니다.

신뢰도 구문 분석(confidence parsing)이 이를 잘 보여줍니다. 제 프롬프트(prompt)는 "Confidence (0-100):"으로 끝나며, 모델은 일반적으로 점수를 첫 번째 토큰(token)으로 응답하고 그 뒤에 줄 바꿈과 설명이 따릅니다. 주로 그렇습니다. 때로는 점수가 설명 뒤에 나타나기도 합니다. 때로는 분수(85/100)나 백분율(85%) 형식으로 표시되기도 합니다. 때로는 모델이 레이블(label)을 다시 언급하기도 합니다("Confidence: 85"). 때로는 문장 속에 숫자를 숨기기도 합니다.

저는 이를 순서대로 시도하는 5단계 우선순위 전략(five-priority strategy)으로 처리합니다.

```python
import re

def parse_confidence(text: str) -> float | None:
    """Extract a confidence score (0-100) from Mistral 7B output.
    Priority order:
      1. Leading bare number  ("85\n\nThe answer is...")
      2. Explicit label        ("Confidence: 85")
      3. Fraction              ("85/100")
      4. Percentage            ("85%")
      5. Trailing bare number  (last 80 chars of response)
    """
    stripped = text.strip()
    # Priority 1: Score as first token.
    lead_match = re.match(r"^(\d{1,3})\b", stripped)
    if lead_match:
        val = int(lead_match.group(1))
        if 0 <= val <= 100:
            return val / 100.0
    # Priority 2-4: Explicit patterns anywhere in the response.
    patterns = [
        r"[Cc]onfidence[:\s]*(\d{1,3})",
        r"(\d{1,3})\s*/\s*100",
        r"(\d{1,3})%",
    ]
    for pat in patterns:
        match = re.search(pat, text)
        if match:
            val = int(match.group(1))
            if 0 <= val <= 100:
                return val / 100.0
    # Priority 5: Any number in the tail of the response.
    tail = text[-80:]
    numbers = re.findall(r"\b(\d{1,3})\b", tail)
    for n_str in reversed(numbers):
        val = int(n_str)
        if 0 <= val <= 100:
            return val / 100.0
    return None
```

여기서 우선순위(priority ordering)가 중요합니다. 우선순위 1(선행 숫자)은 응답의 약 80%를 포착합니다. 모델이 점수를 먼저 제시할 때에는 모호성이 없으며 설명 텍스트에서 연도나 페이지 번호를 잘못 가져올 위험이 없습니다. 우선순위 5(후행 숫자)는 대체(fallback)로 사용되며, 모델이 때때로 추론(reasoning)에서 연도, 수량 또는 다른 숫자를 언급하기 때문에 잘못된 값을 추출할 가능성이 가장 높습니다. 검색 범위를 마지막 80자로 제한하면 오탐(false positives)이 줄어들지만 완전히 제거하지는 못합니다.

답변 확인(answer checking)에도 자체적인 문제가 있습니다. Mistral 7B의 응답은 장황함이 다양하므로, 저는 모델 출력과 참조 답변(reference answers)을 모두 정규화(normalise, 소문자화, 관사 및 구두점 제거, 공백 축소)한 다음 정규화된 참조가 정규화된 출력 어디에든 나타나는지 확인합니다.

```python
def normalise_answer(text: str) -> str:
    text = text.lower().strip()
    text = re.sub(r"\b(a|an|the)\b", " ", text)
    text = re.sub(r"[^\w\s]", "", text)
    text = re.sub(r"\s+", " ", text).strip()
    return text


def check_answer(model_output: str, reference_answers: list[str]) -> bool:
    model_norm = normalise_answer(model_output)
    for ref in reference_answers:
        ref_norm = normalise_answer(ref)
        if ref_norm and ref_norm in model_norm:
            return True
    return False
```

정확한 일치(exact matching) 대신 부분 문자열 일치(substring matching)를 사용하는 것은 의도적인 선택입니다. Mistral 7B는 거의 순수한 답변을 제공하지 않고, 설명으로 감쌉니다. "프랑스의 수도는 파리이며, 이곳은..."이라는 응답은 참조 답변 "파리"와 일치해야 합니다. 짧은 참조 답변은 관련 없는 단어의 부분 문자열로 나타날 경우 오탐을 생성할 수 있지만, 실제로는 참조 답변이 충분히 구체적이므로 문제가 되지 않았습니다.

기권 감지(abstention detection)는 더 간단하지만 여전히 모델에 특화되어 있습니다. Mistral 7B는 거절에 대해 "I don’t know", "I’m not sure", "I cannot determine" 등 약 12가지 변형을 포함하는 제한된 어휘를 사용합니다. 소문자화된 출력에 대한 구문 목록(phrase list)은 이를 안정적으로 포착합니다. 게이트 검사(gate check)는 파서(parser)가 신뢰도 조건 응답의 최소 90%에서 점수를 반환하도록 요구합니다. 그 미만에서는 보정 숫자(calibration numbers)를 신뢰할 수 없습니다. 제 구현은 약 96%를 달성합니다.

이 중 어느 것도 개별적으로는 복잡하지 않습니다. 이를 문서화할 가치가 있는 이유는 이러한 파서(parsers)가 평가의 모든 지표(metric)의 상류에 있기 때문입니다. 신뢰도 파서가 점수의 5%를 잘못 읽으면 예상 보정 오차(Expected Calibration Error)가 변동합니다. 답변 확인기가 체계적인 오탐을 보인다면 정확도 수치(accuracy numbers)는 부풀려집니다. 이러한 파서에는 자체 테스트 스위트(test suite)가 필요하며, 제 경험상 나머지 평가 코드보다 먼저 필요합니다.

## 생태계(Ecosystem)에 바라는 점

MLX는 충분히 활용할 수 있을 만큼 훌륭하며, 이는 대부분의 프레임워크가 주 커뮤니티(primary community) 외부에서 연구 용도로 관리하는 것보다 나은 수준입니다. 하지만 마찰 지점(friction points)이 있습니다.

레이어 표현(layer representations)을 추출하는 API가 있다면 모든 연구자가 자체 훅(hook)을 작성하는 수고를 덜 수 있을 것입니다. HuggingFace의 `output_hidden_states` 매개변수와 같이 생성 경로(generate path)에 내장된 기능은 연구 사용자에게 가장 유용한 추가 사항이 될 것입니다.

커뮤니티 컬렉션(community collection) 전반에 걸친 표준화된 모델 인터페이스(standardised model interfaces)는 임베딩 이름(embedding names), 캐시 시그니처(cache signatures), 그리고 반환 유형(return types)에 필요한 방어적 코딩(defensive coding)을 줄일 것입니다. 이는 부분적으로 모델이 어떻게 변환되는지에 대한 문제이자 API 설계 문제입니다.

지연 평가 모델(lazy evaluation model), 특히 활성화 수집(activation collection)과 같은 상태 저장 작업(stateful operations)과의 상호 작용에 대한 더 나은 문서화는 위에서 설명한 것처럼 조용히 실패하는 버그 클래스를 방지할 것입니다. 현재 문서는 중간 계산을 가로챌 필요가 없는 추론에 중점을 둔 사용자를 가정합니다.

## 그래서 작동하는가?

제가 시작한 질문은 소비자용 Apple Silicon이 출판 가능한 머신러닝 연구를 지원할 수 있는가였습니다. 대체로 그렇습니다. 툴링(tooling)은 PyTorch보다 덜 성숙하고, CUDA 사용자는 필요 없는 호환성 코드(compatibility code)를 작성해야 하며, 다른 주된 사용 사례를 위해 내려진 합리적인 설계 결정에서 비롯된 동작들을 디버깅(debugging)하는 데 몇 시간을 보낼 것입니다.

스크리닝 프로세스(screening processes)는 수백 개의 후보를 완벽한 재현성으로 처리하고, 활성화(activations)는 깨끗하게 나오며, 프로브(probes)는 훈련됩니다. 구성 중 GPU 활용률(utilisation)은 100%에 도달하며, 이는 올바른 병목 현상(bottleneck)입니다. 즉, 엔지니어링이 아닌 계산이 병목입니다.

Apple Silicon을 가지고 있고 CUDA에 접근할 수 없는 연구자들에게 MLX는 실행 가능한 연구 플랫폼입니다. 그렇게 사용하려면 엔지니어링 투자가 필요하지만, 그 엔지니어링은 다루기 쉽고 결과는 유효합니다.
