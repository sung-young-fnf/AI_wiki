# 로컬 LLM 하드웨어 완벽 가이드: 소비자 하드웨어에서 AI 모델을 실행하기 위한 사양

## 서론: AI 자체 호스팅(Self-Hosting)이 미래인 이유

AI를 위해 전적으로 클라우드 API에 의존하던 시대는 끝나가고 있습니다. 한때 ChatGPT나 Claude에 월 100달러 이상의 구독료를 지불해야 했던 일들이 이제는 개인 하드웨어에서 더 나은 프라이버시, 반복적인 비용 없이, 그리고 어떤 모델을 배포할지에 대한 완전한 통제권과 함께 가능해졌습니다.

오늘날의 오픈소스 대규모 언어 모델(LLM)들, 특히 Llama 3, DeepSeek R1, 그리고 Mistral은 실제 작업을 처리할 만큼 충분히 정교합니다. 하지만 문제는? 효율적인 실행을 위해 어떤 하드웨어가 필요한지 정확히 알아야 한다는 점입니다. 이 가이드는 예산에 맞춰 홈 랩(home lab)을 구축하든 고성능 Proxmox 서버를 만들든, 최적의 비용 대비 성능을 위한 정확한 사양을 공개합니다.

자체 호스팅(self-hosting)으로의 전환은 단순히 돈 문제만이 아닙니다. 이는 독립성에 관한 것입니다. 당신의 데이터는 네트워크를 벗어나지 않으며, 모델은 클라우드 지연 시간(latency) 없이 즉시 응답합니다. 당신은 어떤 모델의 어떤 버전을 실행할지 선택합니다. 개발자, 데브옵스(DevOps) 엔지니어, 그리고 AI 애호가들에게 이러한 자유는 하드웨어 투자 가치를 충분히 증명합니다.

## 핵심 요구 사항 이해하기: AI 하드웨어 스택(Stack)

### LLM 하드웨어 스택: 가장 중요한 구성 요소는 무엇인가?

LLM을 로컬에서 실행하는 것은 특정 구성 요소의 체인에 의존하지만, 모든 결정에 있어 가장 중요한 것은 바로 GPU의 VRAM(Video Random Access Memory, 비디오 메모리)입니다. 이를 모델의 작업 메모리라고 생각하십시오.

모든 LLM은 방대한 매개변수 파일(parameter file)로 존재합니다. 예를 들어, Llama 3 7B 모델은 완전 정밀도(full precision)에서 대략 3.5GB에 달합니다. 모델은 합리적으로 빠른 추론(inference)을 위해 VRAM에 완전히 로드되어야 합니다. 만약 VRAM에 맞지 않으면, 시스템은 CPU 처리와 시스템 RAM으로 폴백(fallback)하게 되는데, 이는 수백 배 더 느립니다. 이것이 바로 이 가이드의 모든 빌드 권장 사항을 형성하는 근본적인 제약입니다.

### GPU (그래픽 카드): 결정적인 요소

GPU는 LLM 추론의 핵심 두뇌입니다. 모든 것은 VRAM 용량에 달려 있습니다. 냉혹한 현실을 말씀드리자면, LLM 추론 성능 전투의 약 90%는 VRAM이 차지합니다. 이는 과장이 아닙니다.

**VRAM이 중요한 이유:** LLM을 로드할 때, 전체 매개변수 파일이 GPU 메모리에 들어가야 합니다. 70억 개 매개변수 모델(7B 모델)이 16비트 완전 정밀도(full 16-bit precision)일 경우 14GB가 필요합니다. 동일한 모델이 8비트 양자화(quantisation)되면 7GB가 필요하고, 4비트에서는 대략 3.5GB가 필요합니다. VRAM 용량이 부족하면 모델이 GPU 메모리와 시스템 RAM으로 분할됩니다.

그러면 모든 매개변수 조회(parameter lookup)에 느린 시스템 메모리로의 왕복(round-trip)이 필요하게 되어, 초당 토큰 처리 속도(tokens-per-second, TPS)가 10~15에서 1~3으로 급격히 떨어집니다. 이는 단순히 느려지는 것을 넘어, 사실상 사용 불가능한 수준이 됩니다.

**양자화(Quantisation)가 모든 것을 바꾼다:** 최신 LLM은 거의 항상 양자화된 형태(정밀도 축소 표현)로 배포됩니다. 4비트 모델은 놀라운 품질을 유지하면서 메모리 사용량을 4분의 1로 줄입니다. 8비트 모델은 품질 손실을 더 적게 하면서 메모리를 절반으로 줄입니다. 소비자 하드웨어의 경우, 70억 개 매개변수(7B)보다 큰 모델에서는 4비트 양자화가 표준입니다.

**모델 크기 및 양자화별 VRAM 요구 사항:**

*   **70억 개 매개변수 모델(7B 모델)**은 4비트 양자화에서 대략 3.5~4GB의 VRAM이 필요합니다. 8비트 양자화에서는 동일 모델에 약 7GB가 필요합니다. 16비트 완전 정밀도에서는 14GB가 필요한데, 이는 대부분의 소비자용 GPU에는 비실용적입니다. 실용적인 대화, 추론, 코딩 작업을 위해서는 7B 모델이 예산 빌드에 적합합니다.

*   **130억 개 매개변수 모델(13B 모델)**로 올라가면, 4비트 양자화 시 7GB의 VRAM이 필요합니다. 8비트는 13GB로 점프하고, 완전 정밀도는 26GB에 달합니다. 이때 RTX 4060 Ti 16GB가 적합해집니다. 이 카드는 13B 모델을 컨텍스트(context) 및 프롬프트 캐싱(prompt caching)을 위한 여유 공간과 함께 편안하게 처리할 수 있습니다.
    실제 사용 시 13B 모델은 문서 요약, 더 미묘한 추론, 컨텍스트 인지 코딩 생성과 같은 중급 작업에서 탁월한 성능을 발휘합니다.

*   **300억 개 매개변수 모델(30B 모델)**은 더 많은 부담을 줍니다. 4비트 양자화에서 16GB, 8비트에서 30GB, 완전 정밀도에서 60GB가 필요합니다. RTX 4090과 같은 하이엔드 소비자용 GPU만이 이 티어(tier)를 처리할 수 있으며, 이마저도 VRAM 용량에 제약을 받습니다. **700억 개 매개변수 모델(70B 모델)**은 소비자 하드웨어가 정말로 고전하는 지점입니다. 4비트 양자화에서 70B 모델은 35~40GB의 VRAM을 소비합니다. RTX 4090은 최대 24GB이므로, 극단적인 조치(CPU 오프로딩(CPU offloading), 다중 GPU 모델 샤딩(model sharding), 또는 4비트 미만의 양자화(품질이 현저히 저하됨)) 없이는 70B 모델을 GPU 메모리에 완전히 담을 수 없습니다.

*   **DeepSeek R1 6,710억 개 매개변수 모델(671B 모델)**은 완전히 다른 세상의 이야기입니다. 완전 정밀도에서는 1.3TB의 VRAM이 필요합니다. 4비트 양자화에서도 335GB가 필요합니다. 1.58비트까지 극단적으로 공격적인 양자화를 통해 약 131GB로 압축할 수 있지만, 이는 눈에 띄는 품질 저하(정확도 15~25% 손실)를 대가로 합니다.
    이것은 데이터 센터 규모에서만 가능하며, 극단적인 타협 없이는 소비자 하드웨어에서 실행하기 어렵습니다.

**2025년의 실용적인 GPU 선택:**

*   **RTX 3060 12GB**는 중고 가격 200~250달러로 가장 저렴한 진입점입니다. 완전한 7B 모델과 고도로 양자화된 13B 변형 모델을 처리합니다. 한계는 명확합니다. 12GB는 작은 모델이나 극단적인 양자화로 제한합니다. 7B 이상을 탐색할 계획이라면 이 카드는 빠르게 좌절감을 안겨줄 것입니다.
    RTX 3060은 기술적으로 2020년 출시 제품으로, GPU 기준으로 구형이지만 작은 모델 추론에는 여전히 유능합니다. Llama 3 7B 모델에서 초당 7~10 토큰(TPS)을 볼 수 있으며, 이는 취미 작업 및 학습에 완벽하게 사용할 수 있습니다.

*   **RTX 4060 Ti 16GB**(신제품 499달러부터, 2025년 7월 이후 널리 사용 가능)는 열정적인 사용자(enthusiast)와 진지한 취미 사용자(hobbyist)를 위한 최적의 선택입니다. 16GB는 컨텍스트 및 프롬프트 캐싱을 위한 여유 공간과 함께 4비트 13B 모델을 편안하게 실행합니다. 3060보다 GPU 코어당 1.7배 빠르며 165W만 소비하여 24시간 7일 배포에 전력 효율적입니다. 가능하다면 중고 제품을 250~300달러에 구입하십시오. Llama 3 13B 모델에서 초당 약 12~15 토큰(TPS)의 성능을 보이며, 예산 등급보다 눈에 띄게 빠르면서도 실제 작업을 위한 상호 작용성이 충분합니다.

*   **RTX 4090 24GB**는 신제품 1,200~1,500달러의 열정적인 사용자 표준입니다. 4비트 양자화에서 Llama 3 70B 모델을 허용 가능한 속도(약 7~9 TPS, 24GB VRAM 한계로 인한 모델 관리 필요)로 실행합니다. 여러 모델을 동시에 실행하거나 더 큰 컨텍스트 창(context window)을 처리하려면 이 카드가 최소 요구 사항입니다. 진지한 작업을 위해서는 24GB VRAM은 필수입니다. 실제 테스트에서는 13B 모델에서 초당 20~30 토큰(TPS)을 보여주며, 동시 작업 또는 더 큰 배치 크기(batch sizes)를 위한 충분한 여유 공간을 남깁니다.

*   **AMD 대안**인 RX 7800 XT 16GB 및 RX 6800 XT 16GB는 더 저렴한 비용으로 비슷한 VRAM을 제공하지만, AMD의 CUDA(Compute Unified Device Architecture) 등가 기술인 ROCm 지원은 NVIDIA의 생태계보다 성숙도가 떨어집니다.
    드라이버 생태계는 개선되고 있으며, KoboldCpp 및 MLC-LLM과 같은 도구는 이제 AMD를 지원합니다. 그러나 CUDA의 지배력은 NVIDIA 카드가 더 광범위한 소프트웨어 지원과 더 빠른 드라이버 업데이트를 누린다는 것을 의미합니다. 첫 빌드의 경우 NVIDIA가 더 안전한 선택이지만, ROCm의 특이점을 해결하고 일부 최첨단 최적화를 희생할 의향이 있는 사용자에게는 AMD도 훌륭한 선택이 될 수 있습니다.

### RAM (시스템 메모리): 안전망

시스템 RAM은 GPU VRAM이 가득 찼을 때 비상 오버플로우(overflow) 역할을 합니다. 16GB는 최소 요구 사항이며, OS, 시스템 오버헤드, 기본 컨텍스트를 처리합니다. 하지만 GPU VRAM이 가득 차고 RAM이 오버플로우 버퍼(spillover buffer)가 되는 순간, 추론 속도는 급격히 떨어집니다. 시스템 RAM으로 오프로딩되는 모델은 약 10분의 1 속도로 실행됩니다. 성능 저하는 미미한 수준이 아니라 치명적입니다. 초당 10~15 토큰 대신 1~3 토큰을 보게 될 것입니다. 진지한 LLM 작업을 위해서는 RAM을 모델 버퍼로 의존해서는 안 됩니다.

**실용적인 권장 사항:** 진지한 설정에는 **최소 32GB**가 필요합니다. LLM 추론 엔진과 함께 가상 머신(VM)을 실행하는 Proxmox 환경에서는 32GB가 훨씬 더 중요해집니다. 여러 VM은 메모리를 요구하고, LLM 추론 서비스도 메모리를 요구하며, OS도 메모리를 요구합니다. 여기서 메모리를 아끼려고 하면 모든 기가바이트의 부족함을 느끼게 될 것입니다.

**64GB**는 당신의 설정을 신뢰할 수 있는 다중 모델 플랫폼으로 변화시킵니다. 각각 32GB를 사용하는 두 개의 VM과 호스트 시스템을 위한 여유 공간, 그리고 LLM을 위한 RAM 기반 캐싱(caching)을 통해 고통스러운 리소스 경합(resource contention) 없이 여러 서비스를 실행할 수 있습니다.

CPU 오프로딩을 포함한 70B 모델 추론의 경우, 64~128GB가 표준입니다. 추가 RAM 투자는 프로덕션 워크로드(production workloads)를 실행하거나 여러 모델을 병렬로 테스트하기 시작하면 빠르게 성과를 거둘 것입니다.

### CPU (프로세서) 및 스토리지 (SSD)

**CPU**는 LLM 추론에 놀랍도록 중요하지 않습니다. GPU가 거의 모든 무거운 작업을 처리합니다. Intel Core i7과 N100과 같은 저전력 프로세서는 VRAM 용량이 일치하면 본질적으로 동일한 추론 속도를 제공합니다. CPU가 중요한 부분은 토큰화 속도(tokenisation speed, 텍스트를 모델이 읽을 수 있는 토큰으로 변환하는 속도)와 컨텍스트 길이입니다. 더 빠른 코어는 미미하게 도움이 되지만, CPU에 과도하게 투자할 만한 병목 현상은 아닙니다.

**현실 점검:** 오래된 Ryzen 5 5600X나 예산형 i5-12400도 충분합니다. GPU 가속이 CPU 성능을 완전히 압도합니다. 다른 워크로드(workloads)에 서버를 사용하지 않는 한, LLM 설정에 하이엔드 CPU를 페어링하려는 유혹을 피하십시오. 3.5GHz 이상의 쿼드 코어 프로세서는 토큰화를 완벽하게 처리합니다. 당신의 돈은 CPU가 아닌 GPU와 RAM에 투자해야 합니다.

반면 **스토리지(Storage)**는 CPU보다 더 중요합니다. LLM 모델 파일은 큽니다. 4비트 형태의 70B 모델은 35~40GB입니다. 여러 모델은 쉽게 200GB를 초과합니다. NVMe SSD는 SATA SSD보다 이러한 파일을 5~6배 빠르게 로드합니다. 그 차이는 확연합니다. NVMe는 10~20마이크로초의 지연 시간과 대략 3,500MB/s의 읽기 속도를 달성합니다.

SATA는 약 100마이크로초의 지연 시간과 약 600MB/s로 최고 속도를 냅니다. 모델 로딩에 있어서 NVMe는 초기화 시간을 30~50% 단축시킵니다. 여러 모델 사이를 전환하거나 서비스를 자주 재시작하는 경우, NVMe 드라이브는 번거로운 경험을 원활한 경험으로 바꿉니다.

**스토리지 권장 사항:** 예산 빌드의 경우 최소 512GB NVMe SSD를 사용하십시오. Proxmox를 사용하는 현실적인 다중 모델 시나리오에서는 1TB NVMe가 실용적인 목표입니다. 특정 모델보다는 용량이 중요합니다. WD Black SN850X 및 Samsung 990 Pro는 신뢰할 수 있는 선택입니다. 가능하면 QLC NAND(셀당 4비트)는 피하십시오. TLC(셀당 3비트)는 대규모 모델 파일을 반복적으로 읽는 LLM 워크로드에서 더 나은 지속 성능을 제공합니다.

## 권장 빌드: 예산형부터 고성능까지

### 빌드 1: "Ollama 스타터 키트" (예산: 300~400달러)

이 빌드는 적은 예산으로 LLM을 실험할 수 있음을 증명합니다. 목표 VRAM: 12GB.

*   **구성 요소:** NVIDIA RTX 3060 12GB GPU (중고 200~250달러). 예산형 B450/B550 AM4 칩셋 메인보드. AMD Ryzen 5 5600X CPU (중고 100~120달러) 또는 동급. 32GB DDR4 RAM (80~100달러). 512GB NVMe SSD (50~70달러). 650W 80+ 브론즈 PSU (60~80달러).
*   **총 비용:** 약 550~700달러 (신품 부품 기준; 중고 부품을 구매하면 더 저렴).
*   **최적의 용도:** 취미용, Ollama 학습, 7B 모델 테스트, 큰 투자 없이 로컬 AI 실험.
*   **성능 예상:** Llama 3 7B 모델에서 초당 7~10 토큰(TPS). 양자화된 13B 모델도 가능하지만 느립니다. 단일 모델 로딩 및 추론에 적합하며, 다중 모델 동시성에는 적합하지 않습니다. 이것은 학습용 장치이며, 프로덕션 시스템이 아닙니다.
*   **성공 요인:** RTX 3060 12GB는 GPU 기준으로 구형(2020년 출시)이지만 작은 모델에는 여전히 탁월합니다. 시스템 RAM은 단일 모델 워크로드에 충분합니다. 불필요한 기능 없이 Proxmox나 VM 없이, Ollama가 실행되는 Ubuntu Server를 직접 설치합니다. 이를 통해 양자화를 이해하고, 모델이 다양한 크기에서 어떻게 작동하는지 경험하며, 한두 달 후에는 더 큰 빌드에 투자할지 여부를 결정할 수 있습니다.

### 빌드 2: "데브옵스(DevOps) 홈 랩(Home Lab)" (중급: 800~1,200달러)

이것은 Proxmox에 가장 적합한 빌드입니다. 목표 VRAM: 최소 16GB GPU + 32GB 시스템 RAM.

*   **구성 요소:** NVIDIA RTX 4060 Ti 16GB (신품 499달러) 또는 중고 RTX 3090 (중고 시장 700~800달러). GPU 패스스루(passthrough)를 위한 VT-d/IOMMU 지원 Supermicro X12 시리즈 메인보드 또는 ASUS ProArt B550. AMD Ryzen 7 5800X3D 또는 Intel i7-12700K CPU (중고 200~250달러). 64GB DDR5 RAM (200~250달러). 1TB NVMe SSD (80~120달러). 850W 80+ 골드 등급 PSU (100~150달러).
*   **총 비용:** 신품으로 약 1,300~1,800달러; 중고 GPU 구매 시 900~1,200달러.
*   **최적의 용도:** 여러 VM과 함께 Proxmox 실행, 다른 워크로드와 함께 LLM 서비스 호스팅, 인프라 테스트, CI/CD 파이프라인(pipelines)과의 통합.
*   **성능 예상:** 4비트 Llama 3 13B 모델에서 초당 12~15 토큰(TPS). VM을 통한 리소스 격리(resource isolation)로 여러 7B 모델을 동시에 실행할 수 있습니다. 양자화를 통해 30B 모델도 원활하게 처리합니다. 실제 테스트에서는 이 등급의 시스템이 개발 환경, 모니터링 스택(stacks), 추론 서비스를 고통스러운 경합 없이 동시에 호스팅할 수 있음을 보여줍니다.
*   **이 빌드가 뛰어난 이유:** 64GB RAM은 이를 다중 워크로드 플랫폼으로 변모시킵니다. Proxmox GPU 패스스루(이 시리즈의 2일차 튜토리얼에서 다룰 내용)가 실제로 실용적이 됩니다. 하나의 VM에서 13B 모델을 실행하고, 다른 VM에서 개발 환경을 실행하면서도 시스템 안정성을 유지할 수 있습니다. RTX 4060 Ti의 전력 효율성(165W)은 24시간 7일 배포에도 합리적인 전기 요금을 의미합니다. 여기서는 더 이상 단순한 실험이 아닌, 실제 인프라를 구축하기 시작합니다. Proxmox를 진지하게 배우고, 리소스 할당을 이해하며, 프로덕션 수준의 VM 격리가 어떻게 작동하는지 경험하기 시작합니다.

### 빌드 3: "AI 훈련 파워하우스" (전문가용: 2,500달러 이상)

이 빌드는 진지한 작업, 즉 미세 조정(fine-tuning), 대규모 모델, 프로덕션 추론을 위한 것입니다. 목표 VRAM: 24GB+ GPU, 128GB+ 시스템 RAM, 다중 GPU 고려.

*   **구성 요소:** NVIDIA RTX 4090 24GB (1,200~1,500달러) 또는 듀얼 RTX 4090 또는 RTX 6000과 같은 전문가용 카드를 사용한 다중 GPU 설정. 다중 GPU 시나리오를 위한 다중 PCIe 슬롯 지원 ASUS Pro WS Z790-SAGE UEFI 메인보드. AMD Threadripper 5990WX 또는 Intel Xeon W5-3435X (500~1,000달러). 128GB ~ 256GB DDR5 RAM (600~1,200달러). 4TB NVMe SSD (여러 드라이브, 처리량 향상을 위한 RAID 0) (300~500달러). 1600W 80+ 플래티넘 등급 PSU (300~500달러).
*   **총 비용:** 하이엔드 단일 GPU의 경우 4,000~7,000달러; 다중 GPU 설정의 경우 8,000~15,000달러 이상.
*   **최적의 용도:** 프로덕션 LLM 서비스, 독점 모델 미세 조정, 연구, 고처리량 추론, 다중 모델 동시 서비스.
*   **성능 예상:** 4비트 Llama 3 70B 모델에서 초당 20~30 토큰(TPS). DeepSeek R1 671B 추론은 극단적인 양자화(1.58~2비트)로 가능하지만, 품질 저하가 눈에 띄게 됩니다. 사설 데이터셋에 대한 13B-70B 모델 미세 조정. 다중 GPU 병렬화 및 모델 샤딩(sharding) 전략을 통한 여러 70B급 모델의 동시 추론.
*   **이 빌드가 확장되는 이유:** RTX 4090의 24GB VRAM은 4비트 양자화된 70B 모델을 편안하게 담을 수 있습니다. 128GB 시스템 RAM은 모델 병렬화 전략(여러 GPU에 걸친 샤딩, 빠른 모델 전환을 위한 중복 로딩)을 가능하게 합니다. 높은 코어 수 CPU(Threadripper의 32개 이상 코어)는 배치 추론(batched inference)을 위한 전처리 및 토큰화를 가속화합니다. 이 설정은 소비자 하드웨어에서 데이터 센터 수준의 성능에 근접합니다.
    이제 더 이상 실험하는 것이 아니라, 작은 팀에 서비스를 제공하거나 API 엔드포인트(API endpoints)를 지원할 수 있는 합법적인 추론 서비스를 운영하는 것입니다.

## 소프트웨어 필수 조건: 프로덕션으로의 연결

이러한 빌드를 운영하려면 특정 소프트웨어 기반이 필요합니다.

*   **Proxmox VE**는 특히 중급 및 전문가용 빌드에서 가상화 하이퍼바이저(virtualisation hypervisor)가 됩니다. VM 리소스 할당을 관리하고, GPU 패스스루(게스트 VM에서 네이티브 NVIDIA 드라이버 지원)를 가능하게 하며, 워크로드를 격리합니다. 베어 메탈 하이퍼바이저(bare metal hypervisors)는 GPU 패스스루를 위해 상당한 문제 해결이 필요하지만, Proxmox는 이를 우아하게 처리합니다. 무료 오픈소스이며 프로덕션 환경에서 널리 채택되고 있습니다.

*   **Ubuntu Server**는 Proxmox VM 내의 게스트 OS입니다. 최고의 NVIDIA 드라이버 호환성, 안정적인 패키지 저장소, 그리고 광범위한 Ollama 지원을 제공합니다. CentOS/RHEL도 작동하지만, 취미 사용자에게 불필요한 마찰을 초래합니다. Windows VM도 작동하지만 오버헤드와 라이선스 비용이 추가되므로 특별한 이유가 없다면 건너뛰십시오. Ubuntu Server 22.04 LTS는 2025년 기준 표준이며, 5년간의 보안 업데이트를 제공합니다.

*   **Ollama**는 모든 것을 단순화합니다. 이는 단일 명령어로 LLM을 다운로드, 양자화하고 REST API를 통해 서비스하는 도구입니다. llama.cpp, GPTQ 툴링 또는 양자화 프레임워크와 씨름하는 대신, `ollama pull llama3:13b-q4_K_M`을 실행하면 몇 초 내에 모델이 `localhost:11434`에서 사용 가능해집니다. Ollama는 인프라 복잡성을 추상화합니다. 이면에서는 GPU 오프로딩을 최적화하고, 컨텍스트 창을 관리하며, 모델 로딩을 처리하지만, 당신은 그 복잡성을 결코 보지 못합니다. 이는 원시 하드웨어 가속과 사용자 친화적인 추론 사이의 다리 역할을 합니다.

*   **Open WebUI**는 Ollama를 ChatGPT와 유사한 인터페이스로 감쌉니다. REST API를 프로그래밍 방식으로 사용하지 않는 경우, Open WebUI는 네트워크의 비기술 사용자 또는 API 호출 없이 다른 모델을 테스트할 때 채팅, 이미지 이해 및 모델 전환을 위한 웹 기반 프런트엔드(front-end)를 제공합니다. 선택 사항이지만, 유용성을 극적으로 향상시킵니다.

*   **CUDA Toolkit 및 cuDNN**은 호스트 Proxmox 시스템에 설치되어야 합니다(버전 조정이 중요합니다). Proxmox는 GPU 드라이버 설치를 처리하지만, CUDA 라이브러리는 실제 GPU 컴퓨팅을 가능하게 합니다. NVIDIA RTX 카드용으로는 2025년 기준 CUDA 12.1+가 표준입니다. 이전 GPU(RTX 2000/3000 시리즈)는 CUDA 11.8이 필요할 수 있습니다. 버전 불일치는 "GPU가 감지되지 않음" 오류의 일반적인 원인이므로, 설치 전에 호환성을 확인하십시오. 특정 버전이 중요합니다. 잘못된 조합은 문제 해결에 몇 시간을 낭비하게 만듭니다.

## 결론: VRAM 결정이 전부다

VRAM은 로컬 LLM 배포의 결정적인 제약 요소입니다. CPU, RAM, 스토리지, 소프트웨어 등 다른 모든 것은 VRAM 최적화를 지원합니다. 12GB GPU는 7B 모델과 고도로 양자화된 13B 변형 모델로 제한됩니다. 16GB GPU는 13~30B 모델 범위를 편안하게 열어줍니다. 24GB GPU는 70B 모델의 진입점입니다. 이것은 임의적인 것이 아니라, 신경망이 작동하는 물리적 방식입니다. 메모리 액세스 패턴은 원시 컴퓨팅 성능보다 추론 지연 시간(inference latency)을 훨씬 더 지배합니다.

다음 단계는 즉각적입니다. 어떤 모델이 당신에게 중요한지에 따라 VRAM 목표를 선택하십시오. Llama 3 70B를 로컬에서 실행하고 싶다면 24GB는 필수입니다. 7B 추론 모델로 충분하다면, 예산형 RTX 3060으로 총 700달러 미만으로 달성할 수 있습니다. 예산을 초과하지 않으면서 유연성과 미래 대비(future-proofing)를 원한다면, RTX 4060 Ti 16GB가 2025년 최고의 가격 대비 VRAM 비율을 제공합니다.

하드웨어가 확정되면 Proxmox 설정이 시작됩니다. GPU 패스스루는 원시 하드웨어와 프로덕션화된 인프라 사이의 다리입니다. 이것이 바로 이 시리즈의 2일차에서 다룰 내용이며, GPU 패스스루 구성 튜토리얼은 진정한 로컬 AI 독립을 가능하게 하는 마지막 조각입니다.

당신은 "GPU가 있는 하드웨어가 있다"에서 "격리된 VM에서 여러 모델을 실행하는 프로덕션급 LLM 서버가 있다"로 나아갈 것입니다.
