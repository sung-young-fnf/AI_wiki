# Mac Mini에서 숨겨진 상태 활용하기: 실제 ML 연구를 위한 MLX 사용

대부분의 머신러닝(ML) 연구는 PyTorch(파이토치)를 실행하는 NVIDIA(엔비디아) 하드웨어에서 이루어집니다. 툴체인(toolchain)은 성숙하고 생태계는 방대하며, 프로빙(probing) 실험을 위해 트랜스포머(transformer)에서 내부 표현(internal representations)을 추출해야 할 경우, HuggingFace(허깅페이스)와 CUDA(쿠다) GPU를 사용합니다. 이는 기본 경로이며 잘 작동합니다.

저는 더 저렴한 경로를 택했습니다. 지난 몇 달 동안 저는 현재 준비 중인 논문을 위해 Apple Silicon(애플 실리콘) 상에서 MLX(엠엘엑스)를 주요 연구 플랫폼으로 사용하여 LLM(대규모 언어 모델) 행동 연구를 위한 실험 파이프라인(pipeline)을 전적으로 운영해 왔습니다.

이 게시물은 실제 환경에서 어떤 모습인지에 대한 내용입니다. MLX가 연구에 충분히 잘 작동하는 부분, 그 한계 주변에서 엔지니어링(engineering)이 필요한 부분, 그리고 소비자 하드웨어(consumer hardware)를 진지한 실험 플랫폼으로 다루는 것에 대해 제가 배운 점들을 다룹니다.

## 연구 문제(The research problem), 간략히

저는 모델이 기반 지식(underlying knowledge)과의 관계가 다양한 질문 범주에 따라 LLM이 어떻게 다르게 행동하는지를 연구하고 있습니다. 자세한 내용은 논문에 담을 예정이지만, 여기서는 엔지니어링 요구 사항이 중요합니다. 이는 신중하게 선별된 질문으로 구성된 벤치마크(benchmark), 여러 프롬프트(prompting) 조건에서 모델 응답을 테스트하는 평가 시스템, 그리고 각 레이어(layer)에서 모델이 생성하는 것을 기반으로 선형 프로브(linear probes)를 훈련하기 위한 레이어별 활성화(activation) 추출을 포함합니다. 이 모든 과정은 32GB의 통합 메모리(unified memory)를 갖춘 Mac Mini M4(맥 미니 M4)에서 mlx-community(엠엘엑스 커뮤니티) 저장소 중 하나에서 가져온 4비트 양자화(quantized) Mistral(미스트랄) 7B 모델을 사용하여 실행됩니다.

## MLX를 선택한 이유(Why MLX)

저는 CUDA 클러스터(cluster)에 접근할 수 없는 독립 연구자이며, 제가 가진 것은 Mac Mini입니다.

하지만 덜 명확한 대답은 이러한 제약이 유용한 강제 기능(forcing function)으로 작용했다는 것입니다. 추론(inference) 예산이 소비자용 장비 한 대뿐일 때, 실험 설계에 대해 더 깊이 고민하게 됩니다. 잘못 설계된 선별 과정을 더 많은 GPU를 투입하여 무작정 밀어붙일 수는 없습니다. 각 후보 질문은 결정론적 시딩(deterministic seeding)을 통해 10번의 추론 패스(inference passes)를 거치며, 낭비되는 토큰(token)은 병렬화(parallelise)할 수 없는 실제 시간(wall-clock time)을 소모합니다.

그 결과, 기본적으로 재현 가능(reproducible)하며 고정된 시드(seed)와 다시 실행할 수 있는 선별 실행(screening runs)을 갖춘 설정이 됩니다. 하드웨어 제약은 실험 설계에서 더 규율을 요구합니다.

4비트 양자화(quantization)에는 논의할 가치가 있는 트레이드오프(trade-off)가 있습니다. 양자화는 모델의 가중치(weights)를 압축하며, 이 압축 과정에서 정보가 손실됩니다. 4비트 모델의 표현(representations)은 전체 정밀도(full-precision) 또는 8비트 버전보다 노이즈(noise)가 더 많습니다. 즉, 활성화(activations)의 미묘한 효과가 희석될 수 있으며, 모델의 행동 패턴이 약간 저하될 수 있습니다. 미세한 표현적 차이에 의존하는 프로빙 실험의 경우, 8비트 또는 전체 정밀도 가중치가 더 바람직할 것입니다. 32GB의 통합 메모리에서, 4비트(약 4GB) 7B 모델은 추출 파이프라인과 함께 편안하게 들어맞습니다. 8비트(약 8GB)에서는 여전히 들어맞지만 여유 공간(headroom)이 줄어들며, 전체 정밀도(약 14GB)에서는 공간이 매우 협소할 것입니다.

실제로 4비트에서 시작하는 이유는, 양자화 노이즈를 뚫고 나타날 만큼 강력한 효과라면 그것은 아마 인공물(artifact)이 아닐 것이라는 점입니다. 4비트 활성화에서 명확한 분리(separation)를 달성하는 프로브는 견고한 무언가를 측정하고 있는 것입니다. 만약 효과가 4비트에서 사라진다면, 그것이 아예 없었는지 아니면 단지 노이즈 플로어(noise floor) 아래에 있었는지 알 수 없으며, 이는 진정한 한계입니다. 그러나 초기 검증(validation)을 위해 4비트 양자화는 보수적인 필터(filter) 역할을 합니다. 즉, 이를 통과한 효과는 더 높은 정밀도로 추가 조사를 할 가치가 있습니다.

## 숨겨진 상태 추출: 훅(hook)

핵심 엔지니어링 과제는 MLX 모델의 각 레이어에서 표현(representations)을 추출하는 것이었습니다. HuggingFace 트랜스포머는 `output_hidden_states=True`를 일급 매개변수(first-class parameter)로 제공하지만, MLX-LM은 그렇지 않습니다.

저의 해결책은 내부 모델의 `__call__` 메서드(method)를 임시로 교체하는 것입니다. MLX-LM 모델은 두 단계 구조를 가집니다. 즉, 언어 모델 헤드(LM Head)를 소유하는 외부 래퍼(outer wrapper)와 임베딩 레이어(embedding layer), 트랜스포머 레이어(transformer layers), 그리고 최종 정규화(final norm)를 소유하는 내부 트랜스포머(`model.model`)입니다. 이 훅(hook)은 레이어를 수동으로 반복하며 각 레이어의 출력을 수집합니다.

다음은 올바르게 구현하는 데 시간이 걸렸던 호환성 해결책(compatibility workarounds)을 포함한 전체 구현 코드입니다.

```python
import mlx.core as mx
import mlx.nn as nn
from mlx_lm import load

model, tokenizer = load("mlx-community/Mistral-7B-Instruct-v0.3-4bit")

def extract_hidden_states(text: str) -> mx.array:
    """Extract per-layer hidden states. Returns (n_layers+1, seq_len, hidden_dim)."""
    tokens = tokenizer.encode(text)
    input_ids = mx.array([tokens])  # (1, seq_len)
    inner_model = model.model
    collected_states: list[mx.array] = []
    # Save the original __call__ at the class level.
    original_call = inner_model.__class__.__call__
    def patched_call(self_inner, x, cache=None, mask=None, **kwargs):
        # Embedding - models name this layer differently.
        if hasattr(self_inner, 'embed_tokens'):
            h = self_inner.embed_tokens(x)
        elif hasattr(self_inner, 'embedding'):
            h = self_inner.embedding(x)
        else:
            raise AttributeError("Cannot find embedding layer")
        # Causal mask.
        if mask is None and h.shape[1] > 1:
            mask = nn.MultiHeadAttention.create_additive_causal_mask(h.shape[1])
            mask = mask.astype(h.dtype)
        # Iterate through transformer layers.
        for i, layer in enumerate(self_inner.layers):
            try:
                if cache is not None:
                    try:
                        h = layer(h, mask=mask, cache=cache[i])
                    except TypeError as te:
                        # Cache signature varies across architectures.
                        if "cache" in str(te) or "unexpected keyword" in str(te):
                            h = layer(h, mask=mask)
                        else:
                            raise
                else:
                    h = layer(h, mask=mask)
            except Exception as e:
                raise RuntimeError(f"Layer {i} forward pass failed: {e}") from e
            # Some layers return (hidden_state, cache_update) tuples.
            if isinstance(h, tuple):
                h = h[0]
            collected_states.append(h)
        # Final norm.
        h = self_inner.norm(h)
        collected_states.append(h)
        return h  # Outer model applies lm_head
    try:
        inner_model.__class__.__call__ = patched_call
        _logits = model(input_ids)
        mx.eval(_logits)  # Force computation before collecting.
        result = mx.stack(collected_states, axis=0)
        result = result[:, 0, :, :]  # Remove batch dim.
        mx.eval(result)  # Force evaluation before return.
    finally:
        inner_model.__class__.__call__ = original_call
    return result
```

코드가 이렇게 특이하게 보이는 몇 가지 이유에 대해 주목할 점이 있습니다.

이 코드는 인스턴스(instance) 수준이 아니라 클래스(class) 수준에서 `__call__`을 교체합니다. 이는 MLX-LM의 외부 모델이 `self.model(x)`를 호출할 때 클래스의 `__call__`을 통해 디스패치(dispatch)되기 때문에 중요합니다. 인스턴스에 패치(patch)하는 것은 조용히 무시될 것입니다.

저 두 `mx.eval()` 호출은 보이는 것보다 훨씬 중요합니다. MLX는 지연 평가(lazy evaluation)를 사용합니다. 즉, 결과가 필요할 때까지 계산이 실행되지 않습니다. 첫 번째 `mx.eval(_logits)` 없이는, 수집된 상태를 스택(stack)하려고 할 때 계산 그래프(computation graph)가 구체화되지 않습니다. 두 번째 `mx.eval(result)` 없이는 스택된 텐서(tensor)가 다음 추출 호출 시 덮어쓰여질 그래프 노드(graph nodes)를 참조할 수 있습니다. 이 버그(bug)는 특정 메모리 압력(memory pressure) 하에서만 간헐적으로 나타나고 오류 대신 조용히 잘못된 값을 생성했기 때문에 찾기 가장 어려웠습니다.

`try/finally`는 이러한 종류의 패치가 항상 위험한 것과 같은 이유로 중요합니다. 포워드 패스(forward pass)에서 예외(exception)가 발생하면 원본 메서드를 복원해야 합니다. 그렇지 않으면 단 한 번의 추출 실패로도 모델이 사용할 수 없는 상태가 됩니다.

모든 방어적인 코드(embedding 이름 폴백(fallback, `embed_tokens` 대 `embedding`), 캐시 시그니처(cache signature) 처리, 튜플 반환 확인 등)는 커뮤니티 모델들이 내부 컨벤션(convention)에서 다양하기 때문에 존재합니다. 이 중 어느 것도 MLX 자체의 버그가 아닙니다. 이는 모델 인터페이스(interface)를 아직 완전히 표준화하지 못한 생태계의 결과입니다. 각 해결책은 진단에 반나절, 수정에 몇 줄의 코드(code)가 들었습니다.

## 활성화에서 프로브까지: 전체 파이프라인(the full pipeline)

단일 입력에서 레이어 출력을 얻는 것이 대부분의 엔지니어링 작업이 들어가는 부분입니다. 다른 모든 것은 더 일반적이지만, MLX와 numpy(넘파이) 사이의 메모리 관리(memory management)는 기록할 만큼 중요합니다.

프로빙 실험을 위해 데이터셋(dataset)의 각 질문에 대해 모든 레이어에서 마지막 토큰 표현(last-token representation)을 원합니다. 질문당 레이어별 하나의 벡터(vector)를 사용하여, 단순 분류기(simple classifier, 로지스틱 회귀(logistic regression))가 각 레이어에서 범주를 얼마나 잘 독립적으로 구별할 수 있는지 비교합니다.

질문은 한 번에 하나씩 처리됩니다. HuggingFace가 제공하는 것과 같은 패딩(padding)/어텐션 마스크(attention-mask) 인프라(infrastructure)가 없기 때문에 MLX는 가변 길이 입력에 대해 배치(batch) 처리를 잘 하지 못합니다. 따라서 각 질문은 자체 포워드 패스(forward pass)를 가집니다.

```python
import numpy as np

def extract_batch(model, questions, prompt_prefix=""):
    """Extract last-token hidden states for a list of questions."""
    all_states = []
    for i, q in enumerate(questions):
        text = prompt_prefix + q.text
        # Returns mx.array of shape (n_layers + 1, hidden_dim).
        states_mx = extract_last_token(model, text)
        # Convert to numpy immediately to free MLX memory.
        states_np = np.array(states_mx)
        all_states.append(states_np)
    # Stack in numpy, not MLX.
    return np.stack(all_states, axis=0)  # (n_questions, n_layers+1, hidden_dim)


def extract_last_token(model, text):
    """Convenience: extract full hidden states, take last token."""
    all_states = extract_hidden_states(text)
    return all_states[:, -1, :]  # (n_layers + 1, hidden_dim)
```

각 패스(pass) 후에 즉시 `np.array()`로 변환하는 것이 메모리를 제한적으로 유지하는 방법입니다. 그렇지 않으면 MLX는 메모리 풀(memory pool)에 수백 개의 텐서(tensor)를 축적하고, 몇백 개의 질문을 처리하는 32GB 머신(machine)에서는 이 풀이 가득 차게 됩니다. numpy로 변환하면 다음 포워드 패스에서 재사용할 수 있도록 MLX 메모리가 해제됩니다.

(질문 수, 레이어 수 + 1, 히든 차원) 형태의 numpy 배열을 얻으면, 프로빙은 scikit-learn(사이킷런)을 사용하여 간단하게 수행됩니다.

```python
from sklearn.linear_model import LogisticRegression
from sklearn.model_selection import StratifiedKFold
from sklearn.preprocessing import StandardScaler
from sklearn.metrics import roc_auc_score

def probe_layer(X, y, n_folds=5):
    """Train a binary probe at a single layer. Returns mean AUROC."""
    skf = StratifiedKFold(n_splits=n_folds, shuffle=True, random_state=42)
    aurocs = []
    for train_idx, test_idx in skf.split(X, y):
        scaler = StandardScaler()
        X_train = scaler.fit_transform(X[train_idx])
        X_test = scaler.transform(X[test_idx])
        clf = LogisticRegression(max_iter=1000, class_weight="balanced")
        clf.fit(X_train, y[train_idx])
        proba = clf.predict_proba(X_test)[:, 1]
        aurocs.append(roc_auc_score(y[test_idx], proba))
    return np.mean(aurocs)


# Run independently at every layer.
n_layers = hidden_states.shape[1]
for layer in range(n_layers):
    X = hidden_states[:, layer, :]  # (n_questions, hidden_dim)

```

각 레이어에서 프로브를 독립적으로 실행하면 모델 내에서 범주 간의 구분이 가장 강력한 지점을 보여주는 곡선이 생성됩니다. 제 실험에서는 신호가 초기-중간 레이어에서 정점을 찍고 최종 레이어로 갈수록 약해지는데, 이는 모델이 정보를 처리하는 방식에 대해 알려주는 바가 있습니다(하지만 해석은 논문에 담을 것이므로 여기서는 다루지 않습니다).

모든 것은 압축된 `.npz` 파일로 저장되고 로드되므로, 추출을 한 번 실행하고 추론을 반복하지 않고 원하는 만큼 여러 번 프로빙할 수 있습니다. Mac Mini에서 수백 개의 질문에 대한 레이어 출력을 가져오는 데 약 20분이 걸립니다. 모든 프로브를 훈련하는 데는 1초 미만이 걸립니다.

## 대규모 결정론적 선별(Deterministic screening at scale)

벤치마크 구성은 각 후보 질문을 온도(temperature) 0.7에서 결정론적 시딩(deterministic seeding)을 통해 10번의 추론 패스를 거치게 합니다. 각 시드는 `base_seed + (question_index * n_runs) + run_index`로 계산되며, 범주별로 다른 기본 시드(base seed)를 사용합니다. 이는 동일한 시드로 다시 실행함으로써 개별 선별 결과가 정확히 재현될 수 있음을 의미합니다.

MLX의 난수 생성(random number generation)은 이를 잘 지원합니다. 각 확률적 생성(stochastic generation) 호출 전에 코드는 `mx.random.seed(seed)`를 고정하여 동일한 입력에 대해 동일한 토큰 샘플링(token sampling)을 생성합니다. 온도 0 조건(평가 단계에서 사용됨)의 경우, 샘플러(sampler)는 RNG(난수 생성기)를 완전히 우회하고 로짓(logits)에 대해 `mx.argmax`를 취합니다.

선별 기준은 범주에 따라 다릅니다. 일부는 높은 정확도를 요구하고, 다른 일부는 특정 정확도 범위(accuracy bands)를 목표로 하며, 또 다른 일부는 일관된 실패를 요구합니다. 각 선별 함수(function)는 자체 필터를 적용하지만, 모두 동일한 결정론적 시딩 인프라를 공유합니다.

초기 파일럿(pilot) 결과에 따르면, 프로브는 모델의 내부 표현에서 분리(separation)를 감지하며, 신호는 최종 레이어보다는 초기 레이어에 집중되어 있습니다. 프로빙 실험에서 얻은 숫자를 신뢰하는 것은 그 밑단의 실험 인프라에 달려 있습니다.

## Mistral 7B 출력 파싱: 생각보다 어렵다

mlx-lm을 통해 Mistral 7B로 평가 작업을 한다면, 예상보다 더 많은 시간을 출력 파싱(output parsing)에 할애하게 될 것입니다. 이 모델의 응답 형식은 단일 정규 표현식(regex)으로는 처리하기 어려울 정도로 다양하며, 오류 모드(failure modes)는 조용합니다. 즉, 점수 대신 `None`을 얻거나, 더 나쁘게는 응답에서 잘못된 숫자를 추출하게 됩니다.

신뢰도 파싱(confidence parsing)이 이를 잘 보여줍니다. 제 프롬프트(prompt)는 "Confidence (0-100):"으로 끝나며, 모델은 대개 점수를 첫 번째 토큰으로 응답하고 그 뒤에 줄 바꿈과 설명이 따릅니다. 주로 그렇습니다. 때로는 점수가 설명 뒤에 나타나기도 합니다. 때로는 분수(85/100)나 백분율(85%)로 형식화되기도 합니다. 때로는 모델이 레이블(label)을 다시 언급하기도 합니다(예: Confidence: 85). 때로는 숫자를 문장 속에 숨기기도 합니다.

저는 이를 5가지 우선순위 전략으로 순서대로 시도하여 처리합니다.

```python
import re

def parse_confidence(text: str) -> float | None:
    """Extract a confidence score (0-100) from Mistral 7B output.
    Priority order:
      1. Leading bare number  ("85\n\nThe answer is...")
      2. Explicit label        ("Confidence: 85")
      3. Fraction              ("85/100")
      4. Percentage            ("85%")
      5. Trailing bare number  (last 80 chars of response)
    """
    stripped = text.strip()
    # Priority 1: Score as first token.
    lead_match = re.match(r"^(\d{1,3})\b", stripped)
    if lead_match:
        val = int(lead_match.group(1))
        if 0 <= val <= 100:
            return val / 100.0
    # Priority 2-4: Explicit patterns anywhere in the response.
    patterns = [
        r"[Cc]onfidence[:\s]*(\d{1,3})",
        r"(\d{1,3})\s*/\s*100",
        r"(\d{1,3})%",
    ]
    for pat in patterns:
        match = re.search(pat, text)
        if match:
            val = int(match.group(1))
            if 0 <= val <= 100:
                return val / 100.0
    # Priority 5: Any number in the tail of the response.
    tail = text[-80:]
    numbers = re.findall(r"\b(\d{1,3})\b", tail)
    for n_str in reversed(numbers):
        val = int(n_str)
        if 0 <= val <= 100:
            return val / 100.0
    return None
```

여기서 우선순위 순서가 중요합니다. 우선순위 1(선행하는 숫자)은 응답의 약 80%를 처리합니다. 모델이 점수를 먼저 제시할 경우, 모호함이 없고 설명 텍스트에서 연도나 페이지 번호를 가져올 위험이 없습니다. 우선순위 5(후행하는 숫자)는 대체(fallback) 수단이며, 모델이 추론(reasoning)에서 연도, 수량 또는 기타 숫자를 언급하는 경우가 있기 때문에 잘못된 값을 추출할 가능성이 가장 높습니다. 검색을 마지막 80자(characters)로 제한하면 오탐(false positives)을 줄일 수 있지만 완전히 제거하지는 못합니다.

답변 확인(answer checking)에도 자체적인 문제가 있습니다. Mistral 7B의 응답은 장황함(verbosity)이 다양하므로, 저는 모델 출력과 참조 답변(reference answers)을 모두 정규화(normalise, 소문자화, 관사 및 구두점 제거, 공백 축소)한 다음, 정규화된 참조가 정규화된 출력 어디에든 나타나는지 확인합니다.

```python
def normalise_answer(text: str) -> str:
    text = text.lower().strip()
    text = re.sub(r"\b(a|an|the)\b", " ", text)
    text = re.sub(r"[^\w\s]", "", text)
    text = re.sub(r"\s+", " ", text).strip()
    return text


def check_answer(model_output: str, reference_answers: list[str]) -> bool:
    model_norm = normalise_answer(model_output)
    for ref in reference_answers:
        ref_norm = normalise_answer(ref)
        if ref_norm and ref_norm in model_norm:
            return True
    return False
```

정확한 일치(exact matching) 대신 부분 문자열 일치(substring matching)를 사용하는 것은 의도적인 선택입니다. Mistral 7B는 거의 단독 답변을 제공하지 않고, 설명으로 감싸서 제공합니다. "프랑스의 수도는 파리이며, 이곳은..."과 같은 문장은 참조 답변 "파리"와 일치해야 합니다. 짧은 참조 답변은 관련 없는 단어의 부분 문자열로 나타날 경우 오탐을 생성할 수 있지만, 실제로는 참조 답변이 충분히 구체적이어서 문제가 되지 않았습니다.

기권 감지(abstention detection)는 더 간단하지만 여전히 모델에 따라 다릅니다. Mistral 7B는 거절을 위해 제한된 어휘를 사용합니다. "모르겠습니다", "확신할 수 없습니다", "판단할 수 없습니다" 등 약 10여 가지 변형이 있습니다. 소문자화된 출력에 대한 구문 목록(phrase list)은 이러한 응답을 안정적으로 포착합니다. 게이트 검사(gate check)는 파서(parser)가 신뢰도 조건 응답의 최소 90%에서 점수를 반환하도록 요구합니다. 그 미만에서는 보정(calibration) 수치를 신뢰할 수 없습니다. 제 구현은 약 96%를 달성합니다.

이 중 어느 것도 개별적으로는 복잡하지 않습니다. 문서화할 가치가 있는 이유는 이 파서들이 평가의 모든 메트릭(metric) 상위에 있기 때문입니다. 만약 신뢰도 파서가 점수의 5%를 잘못 읽으면, Expected Calibration Error(예상 보정 오차)가 달라집니다. 답변 확인기가 체계적인 오탐을 보인다면, 정확도(accuracy) 수치는 부풀려집니다. 이 파서들은 자체 테스트 스위트(test suite)가 필요하며, 제 경험상 다른 평가 코드보다 먼저 필요합니다. 나중이 아닙니다.

## 생태계에 바라는 점(What I’d want from the ecosystem)

MLX는 기반을 다지기에 충분하며, 이는 대부분의 프레임워크(framework)가 주 커뮤니티(community) 외부에서 연구용으로 관리하는 것보다 나은 수준입니다. 하지만 마찰 지점(friction points)이 존재합니다.

레이어 표현을 추출하기 위한 API(응용 프로그래밍 인터페이스)는 모든 연구자가 자체 훅을 작성하는 수고를 덜어줄 것입니다. HuggingFace의 `output_hidden_states` 매개변수와 같이 `generate` 경로에 내장된 기능은 연구 사용자에게 가장 유용한 단일 추가 기능이 될 것입니다.

커뮤니티 컬렉션(collection) 전반에 걸쳐 표준화된 모델 인터페이스는 임베딩 이름, 캐시 시그니처, 반환 타입(return types)에 필요한 방어적 코딩(defensive coding)을 줄여줄 것입니다. 이는 부분적으로 모델이 어떻게 변환되는지에 대한 것이고, 부분적으로는 API 설계 문제입니다.

지연 평가 모델에 대한 더 나은 문서화, 특히 활성화 수집과 같은 상태 저장 작업(stateful operations)과의 상호 작용에 대한 설명은 위에서 설명한 것과 같이 조용히 실패하는 종류의 버그를 방지할 것입니다. 현재 문서는 중간 계산을 가로챌 필요가 없는 추론 중심의 사용자를 가정하고 있습니다.

## 그렇다면 작동하는가?(So does it work?)

제가 시작했던 질문은 소비자용 Apple Silicon이 출판 가능한 ML 연구를 지원할 수 있는가였습니다. 대체로, 그렇습니다. 툴링(tooling)은 PyTorch보다 덜 성숙하고, CUDA 사용자는 필요 없는 호환성 코드(compatibility code)를 작성하게 될 것이며, 다른 주요 사용 사례를 위해 내려진 합리적인 설계 결정에서 비롯된 동작들을 디버깅(debugging)하는 데 몇몇 오후를 보낼 것입니다.

선별 과정은 수백 개의 후보를 완전한 재현 가능성(reproducibility)으로 처리하고, 활성화는 깨끗하게 추출되며, 프로브는 훈련됩니다. 구성 시 GPU 활용률은 100%에 달하며, 이는 올바른 병목 현상(bottleneck)입니다. 즉, 엔지니어링이 아닌 계산이 병목입니다.

Apple Silicon을 가지고 있지만 CUDA에 접근할 수 없는 연구자들에게 MLX는 실행 가능한 연구 플랫폼입니다. 그렇게 사용하려면 엔지니어링 투자가 필요하지만, 그 엔지니어링은 다루기 쉽고 결과도 견고합니다.
