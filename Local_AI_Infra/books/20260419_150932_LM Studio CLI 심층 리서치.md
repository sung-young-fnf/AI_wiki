# LM Studio CLI (`lms`) 심층 리서치 리포트

> 작성일: 2026-04-19
> 출처: Brave Search + Perplexity (sonar-pro) 교차 분석

## 1. 개요

**`lms`**는 LM Studio의 공식 CLI 유틸리티로, 데스크톱 앱과 동일한 모델 라이프사이클·서버·런타임 제어 기능을 터미널에서 스크립트화할 수 있게 해주는 도구입니다. **LM Studio 0.2.22+** 버전부터 기본 번들로 포함되며, 내부적으로는 `lmstudio.js` SDK로 구축되어 있습니다.

- **공식 저장소**: [github.com/lmstudio-ai/lms](https://github.com/lmstudio-ai/lms)
- **공식 문서**: [lmstudio.ai/docs/cli](https://lmstudio.ai/docs/cli)
- **백엔드**: `llama.cpp` 또는 `MLX`(Apple Silicon)
- **API 호환**: OpenAI 호환 (기본 포트 `1234`), Anthropic 호환 일부

---

## 2. 설치 & 부트스트랩

LM Studio 데스크톱 앱이 이미 설치되어 있다면 별도 설치 불필요. PATH 등록만 필요할 수 있음.

```bash
# 공식 권장 방법 (npm 기반 부트스트랩)
npx lmstudio install-cli

# 직접 부트스트랩 (Mac/Linux)
~/.lmstudio/bin/lms bootstrap

# Windows
cmd /c %USERPROFILE%/.lmstudio/bin/lms.exe bootstrap

# 확인
lms --help
```

**리눅스 헤드리스(Unsloth 가이드 기준)**:

```bash
~/.lmstudio/bin/lms bootstrap
lms import <model.gguf> --symbolic-link --user-repo "<repo/name>" -y
```

## 3. 전체 명령어 목록 (v0.4.0 기준)

| 명령어 | 용도 |
|--------|------|
| `lms status` | LM Studio/llmster 데몬 상태 출력 |
| `lms server start` | 로컬 API 서버 시작 (기본 :1234) — `--port`, `--cors`, `--api-key` 등 옵션 |
| `lms server stop` | 서버 graceful shutdown (SIGTERM → 모델 언로드 → 포트 해제) |
| `lms ls` | 다운로드된 모델 목록 (`--json` 머신 리더블) |
| `lms ps` | 메모리에 로드된 모델 목록 (`--json`) |
| `lms get <model>` | 모델 다운로드 (HuggingFace 통합) — 0.3.5에서 추가 |
| `lms load <model>` | 모델 로드 (옵션: `--gpu`, `--context-length`, `--ttl`, `--id`) |
| `lms unload [<id>]` | 단건/전체(`--all`) 모델 언로드 |
| `lms chat` | 터미널 인터랙티브 채팅 세션 (0.4.0에서 stateful `/v1/chat` 엔드포인트와 연동) |
| `lms create` | LM Studio SDK 기반 신규 프로젝트 스캐폴딩 |
| `lms log stream` | 실시간 로그 스트리밍 (디버그·tool call 포맷 확인용) |
| `lms runtime` | 추론 런타임 관리 (llama.cpp / MLX 선택) |
| `lms runtime survey` | **0.4.0 신규** — 사용 가능한 GPU 정보 출력 |
| `lms daemon up/down` | **llmster** 헤드리스 데몬 제어 |
| `lms link` | LM Link(원격 공유) 관리 |
| `lms import <gguf>` | 외부 GGUF 모델 가져오기 (`--symbolic-link` 지원) |
| `lms version` | CLI 버전 |
| `lms bootstrap` | PATH 셋업 |

### 핵심 옵션 예시

```bash
# GPU 오프로드 비율 제어
lms load google/gemma-3-1b --gpu max         # 전체
lms load google/gemma-3-1b --gpu 0.5         # 50%
lms load google/gemma-3-1b --gpu off         # CPU only

# 컨텍스트 길이 + TTL(idle 후 자동 언로드, 초)
lms load google/gemma-3-1b --context-length 4096 --ttl 3600

# 커스텀 인스턴스 ID 부여 (멀티 로드 시 식별자)
lms load openai/gpt-oss-20b --id my-gpt
```

## 4. 헤드리스 운영 — `llmster`

LM Studio 0.3.5에서 등장한 **llmster**는 GUI 의존성 없는 데몬으로, 서버/클라우드/CI 환경에서 LM Studio 코어를 돌릴 수 있게 합니다.

```bash
lms daemon up           # 데몬 시작
lms get <model>         # 모델 다운로드
lms server start        # API 서버 기동
lms chat                # 인터랙티브 세션
```

**옵션 비교**:

1. **llmster (권장)** — 진정한 헤드리스, Linux systemd user service 등록 가능
2. **데스크톱 앱 헤드리스 모드** — 설정에서 "Run LLM server on login" 활성화, 트레이로 최소화 (초기 GUI 셋업 필요)

**JIT (Just-In-Time) 모델 로딩**: REST 엔드포인트 호출 시 모델이 자동으로 로드되며, idle TTL로 자동 언로드되어 VRAM 효율적 사용.

**API Key 생성**(이슈 #1674에서 요청, 일부 버전 지원):

```bash
openssl rand -hex 32
lms server start --api-key <generated-key>
```

## 5. 개발자 통합 (SDK 연계)

`lms`는 단독 사용보다 SDK와 결합 시 진가를 발휘합니다.

```typescript
// lmstudio.js / @lmstudio/sdk
import { LMStudioClient } from "@lmstudio/sdk";
const client = new LMStudioClient();
const model = await client.llm.model("openai/gpt-oss-20b");
const result = await model.respond("Who are you?");
```

```python
# lmstudio-python SDK — `lms get`로 모델 미리 다운로드
import lmstudio as lms
model = lms.llm("qwen3-4b-2507")
```

**0.4.0 주요 변경**:

- `/v1/chat` **stateful** 엔드포인트 (response_id로 멀티턴 체인)
- `api/v1/models/load`, `api/v1/models/unload` 추가
- `api/v1/models/load` 응답의 `model_instance_id` → `instance_id`로 **breaking change**
- Function/Tool calling 디버깅: `lms log stream` + chat completions 호출로 raw tool call format 관찰

## 6. Ollama vs LM Studio CLI 비교

| 항목 | LM Studio (`lms`) | Ollama |
|------|-------------------|--------|
| **철학** | GUI + CLI 하이브리드 (앱이 backend) | CLI/REST 우선, daemon 단독 |
| **설치 풋프린트** | 앱 ~500MB / llmster는 경량 | ~100MB idle |
| **모델 디스커버리** | 앱 내 HuggingFace UI 검색 강력, `lms get`도 지원 | `ollama pull <name>` (라이브러리 의존) |
| **동시 모델** | 단일 활성 모델 우선(인스턴스 ID로 멀티 가능) | 멀티모델 동시 서빙 견고 |
| **GPU 제어** | `--gpu max/0.5/off` 정밀 비율 + UI 슬라이더 | 자동 (Modelfile에서 일부 제어) |
| **커스터마이징** | context-length, TTL, instance ID | Modelfile (시스템 프롬프트, 파라미터) |
| **API** | OpenAI 호환 + stateful `/v1/chat` (0.4.0) | OpenAI 호환 + 자체 REST |
| **MLX(Apple) 가속** | ✅ 1급 지원 | 부분 지원 |
| **헤드리스** | llmster(0.3.5+)로 따라잡기 | 원래부터 daemon 중심 |
| **부팅** | 앱 콜드 스타트 2~3s | 거의 즉시 |
| **추론 속도** | 동일 (둘 다 llama.cpp) | 동일 |

**선택 기준**:

- **`lms`**: GUI에서 실험 후 CLI로 자동화하고 싶을 때, Apple MLX 활용, 비주얼 디버깅 필요 시
- **Ollama**: 순수 인프라/CI, 멀티모델 서빙, Modelfile 기반 형상관리

## 7. 실전 자동화 패턴

```bash
# 1) JSON 출력으로 스크립트 파이프
lms ps --json | jq '.[] | .identifier'

# 2) idle 시 자동 언로드 + 메모리 절약
lms load qwen3-4b --ttl 600 --gpu max

# 3) 서버 라이프사이클 관리
lms server start --port 1234 &
trap "lms server stop" EXIT

# 4) 로그 스트리밍으로 tool calling 포맷 확인
lms log stream &
curl http://localhost:1234/v1/chat/completions -d '{...tools:[...] }'

# 5) GGUF 직접 임포트 (Unsloth 등 외부 모델)
hf download unsloth/Qwen3-Coder-Next-GGUF --include "*UD-Q4_K_XL*"
lms import unsloth/Qwen3-Coder-Next-GGUF/<file>.gguf \
  --symbolic-link --user-repo "unsloth/Qwen3-Coder-Next-GGUF" -y
```

## 8. 한계 & 주의사항

1. **단일 활성 모델 우선** — 인스턴스 ID로 멀티 로드 가능하나 Ollama 대비 덜 자연스러움
2. **데스크톱 앱 의존성(전통적)** — llmster 등장 전엔 GUI 필요 → 서버 환경에서 불편했음. 0.3.5+에서 해소
3. **API Key 헤드리스 발급** — 이슈 #1674에서 추가 요청된 기능 (버전 별 지원 확인 필요)
4. **Breaking Change(0.4.0)** — `model_instance_id` → `instance_id` 마이그레이션 필요
5. **macOS AppleScript 환경**과는 무관 (LM Studio는 OS-네이티브 GUI + RPC 기반)

## 9. 본 프로젝트(roybongclaudebot)에 시사점

현재 프로젝트는 **Ollama(`gemma4:26b-a4b-it-q4_K_M`)**를 번역 백엔드로 사용 중입니다. LM Studio CLI로의 전환을 고려한다면:

- **장점**: MLX 가속(Apple Silicon에서 처리량↑), 동일 OpenAI 호환 엔드포인트라 `src/utils/llm.py`의 httpx 호출부만 base URL 변경(`http://localhost:11434` → `http://localhost:1234/v1`)으로 마이그레이션 가능, JIT 로딩으로 idle 메모리 절감
- **단점**: 현재 Ollama 기반 멀티 모듈(`src/utils/ollama.py`, `src/service/ollama_function_call_test/`)을 모두 손봐야 함. Ollama Function Calling 데모는 LM Studio Tool Use(`/v1/chat/completions`)로 재작성 필요
- **권장**: **하이브리드** — 번역 파이프라인은 Ollama 유지, MLX 가속이 필요한 멀티모달/이미지 분석에만 LM Studio 추가 도입
