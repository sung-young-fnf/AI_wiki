바로 이 질문에 대한 해답이 Exo입니다.

## Exo는 무엇인가요?

Exo는 Exo Labs가 관리하는 오픈소스 프로젝트입니다. 한마디로 말해, 모든 기기를 개인 AI 클러스터(personal AI cluster)로 연결하여 어떤 단일 머신에도 맞지 않는 최첨단 모델(frontier models)을 실행할 수 있게 해줍니다.

주요 기능 요약:
*   **자동 기기 검색(Automatic Device Discovery)** — Exo를 실행하는 기기들이 네트워크를 통해 자동으로 서로를 찾아냅니다. 수동 구성이 필요 없습니다.
*   **토폴로지 인식 자동 병렬화(Topology-Aware Auto Parallel)** — Exo는 사용 가능한 RAM, CPU/GPU 자원, 그리고 각 노드(node) 간의 네트워크 지연 시간(network latency)을 기반으로 모델을 기기별로 분할하는 최적의 방법을 찾아냅니다.
*   **텐서 병렬화(Tensor Parallelism)** — 모델 샤딩(model sharding)을 통해 두 대의 기기에서 최대 1.8배, 네 대의 기기에서 최대 3.2배의 속도 향상을 제공합니다.
*   **썬더볼트 5(Thunderbolt 5)를 통한 RDMA(Remote Direct Memory Access)** — 지원되는 하드웨어(M4 Pro/Max)에서 기기 간 지연 시간(inter-device latency)을 최대 99%까지 줄여줍니다.
*   **MLX 백엔드(MLX Backend)** — Apple Silicon에서 GPU 가속 추론(GPU-accelerated inference)을 위해 Apple의 MLX 프레임워크를 사용합니다.
*   **OpenAI 호환 API(OpenAI-compatible API)** — `http://localhost:52415/v1`를 노출하여 OpenAI와 통신하는 모든 도구가 대신 클러스터와 통신할 수 있습니다.
*   **54개 이상의 모델 지원** — 작은 Llama 모델부터 671B DeepSeek 변형 모델까지 지원합니다.
*   **Mac, Linux, 심지어 Raspberry Pi에서도 작동합니다.**

## 나의 설정: Mac Mini M4 + MacBook Pro M4 Max

이 실험을 위해 두 대의 머신을 연결했습니다.

*   **Mac Mini M4** – 16GB 통합 메모리(unified memory); 피크 시 55.1GB/64GB 사용 (86%)
*   **MacBook Pro M4 Max** – 64GB 통합 메모리; 보조 파티션(secondary partition)에서 9.8GB/16GB 사용 (61%)

이들을 합하면 클러스터는 Qwen3-Next-80B-A3B-Thinking-4bit (44GB 양자화 모델(quantized model)로, 어떤 단일 머신도 단독으로는 편안하게 처리할 수 없는 모델)를 로드할 수 있는 충분한 여유 공간을 확보합니다. 이 모델은 쿼리(query) 복잡성에 따라 TTFT(Time To First Token, 첫 토큰 생성 시간)가 약 4-11초인 상태로 초당 70-80 토큰(TPS, tokens per second)의 일관된 속도로 실행되었습니다. 온도: 부하 시 Mac Mini는 41-86°C까지 상승했으며, MacBook Pro는 48-53°C로 더 낮은 온도를 유지했습니다.

## 시작하기: Mac에서는 정말 간단합니다.

macOS용 Exo는 네이티브 앱(native app)으로 제공됩니다 (DMG 버전은 macOS Tahoe 26.2 이상 필요).

1.  릴리스 페이지(releases page)에서 `EXO-latest.dmg`를 다운로드합니다.
2.  응용 프로그램(Applications) 폴더에 복사하고 실행합니다.
3.  동일 네트워크에 있는 다른 모든 머신에서도 반복합니다.
4.  완료 — 노드(node)들이 자동으로 서로를 찾아 토폴로지 뷰(Topology view)에 나타납니다.

그게 전부입니다. 그냥 작동합니다.

## Linux 및 Windows 설정

Linux 사용자들은 소스(source)에서 실행해야 합니다. 먼저, 사전 요구 사항(prerequisites)을 설치합니다.

```bash
uv (Python dependency manager):
curl -LsSf https://astral.sh/uv/install.sh | sh
Node.js 18+ and npm
Rust (nightly):
curl — proto ‘=https’ — tlsv1.2 -sSf https://sh.rustup.rs | sh && rustup toolchain install nightly
```

그 다음 클론(clone)하고 실행합니다.

```bash
git clone https://github.com/exo-explore/exo
cd exo/dashboard && npm install && npm run build && cd ..
uv run exo
```

한 가지 중요한 주의사항: Linux에서 Exo는 현재 CPU(중앙 처리 장치)에서만 실행됩니다. Linux용 GPU(그래픽 처리 장치) 지원은 활발히 개발 중입니다 — NVIDIA 또는 AMD GPU를 활용하고 싶다면 추적해볼 가치가 있습니다.

## API: 드롭인(Drop-In) OpenAI 대체

Exo는 `http://localhost:52415/v1`에서 완전히 OpenAI 호환 REST API(OpenAI-compatible REST API)를 노출합니다. 이는 OpenAI SDK를 지원하는 모든 도구, 에이전트 프레임워크(agent framework) 또는 애플리케이션(application)이 대신 로컬 클러스터(local cluster)를 가리킬 수 있음을 의미합니다 — 코드 변경(code changes)이 필요 없습니다.

curl을 사용한 간단한 예시:

```bash
curl -N -X POST http://localhost:52415/v1/chat/completions \

    "messages": [

    ],
    "stream": false
  }'
```

## 썬더볼트 5를 통한 RDMA: 다음 단계

썬더볼트 5(Thunderbolt 5)를 지원하는 M4 Pro 또는 M4 Max 하드웨어(hardware)를 가지고 있다면, Exo는 macOS 26.2에 새로 추가된 기능인 RDMA(Remote Direct Memory Access)를 지원합니다. 이 기능은 노드 간 지연 시간(inter-node latency)을 최대 99%까지 줄여주며, 일반적으로 데이터센터 인터커넥트(data center interconnects)에서 기대할 수 있는 종류의 성능을 제공하는 것으로 알려져 있습니다.

현재 제 설정(스크린샷에서 RDMA Not Enabled 경고가 보입니다 — 제 머신들은 Wi-Fi에 연결되어 있고 썬더볼트 5가 아닙니다)에서는 이를 테스트할 수 없었지만, Jeff Geerling의 4× M3 Ultra Mac Studio 클러스터(cluster) 벤치마크(benchmark)는 Qwen3–235B가 프로덕션(production) 수준의 속도로 실행됨을 보여줍니다. 이는 이 도구가 도달할 수 있는 성능의 정점입니다.

## 실제 성능 수치

다음은 제가 테스트 쿼리(test queries)에서 관찰한 내용입니다.

*   "Why is Sky Blue?" — TTFT: 10,739ms, TPS: 75.2 tok/s (13.3 ms/tok)
*   "Write a snake game in Python" — TTFT: 4,049ms, TPS: 69.1 tok/s
*   일반 추론(inference): 세션(sessions) 전반에 걸쳐 68–75 TPS 지속

80B 매개변수(parameter) 사고 모델(thinking model)이 클라우드 비용 없이 전적으로 로컬 하드웨어(local hardware)에서 실행된다는 점을 고려하면, 이 수치들은 정말 인상적입니다. THINK 모드(사고의 사슬 추론)는 예상대로 TTFT를 증가시키지만, 이 모드를 활성화하면 모델 품질이 눈에 띄게 향상됩니다.

## Exo vs. LM Studio Link: 어떤 상황에 무엇을 사용해야 할까요?

이 두 도구는 인접하지만 서로 다른 문제를 해결합니다.

*   **LM Studio Link** — 강력한 머신 한 대를 가지고 있고 네트워크의 저사양 기기들에서 접근하고 싶을 때 사용합니다. 하나의 호스트(host), 여러 클라이언트(clients).
*   **Exo** — 여러 머신을 단일 가상 GPU 클러스터(virtual GPU cluster)로 결합하고 싶을 때 사용합니다. 여러 호스트(hosts), 하나의 모델.

만약 단일 머신이 지원하는 것보다 더 큰 모델을 실행하는 것이 목표라면 — Exo가 올바른 도구입니다. 편의성과 원격 접근(remote access)이 목표라면 — LM Studio Link가 여전히 훌륭합니다.

## 최종 의견

Exo는 제가 접해본 오픈소스 AI 도구 중 가장 실용적인 것 중 하나입니다. 진입 장벽은 (특히 Mac에서는) 놀라울 정도로 낮으며, 잠재력은 매우 높습니다. 불과 2년 전만 해도 동일한 Wi-Fi 네트워크에 연결된 두 대의 노트북에 분산하여 가정에서 사고 기능을 갖춘 80B 모델을 실행한다는 것은 공상처럼 들렸을 것입니다.

에이전트 기반 AI 시스템(agentic AI systems)을 구축하거나, 로컬 실험(local experiments)을 수행하거나, 단순히 하드웨어가 함께 작동할 때 무엇을 할 수 있는지 궁금하다면 Exo를 사용해보세요. Mac에서는 설정하는 데 2분밖에 걸리지 않을 것입니다. 하지만 그 함의는 당신을 훨씬 더 오랫동안 몰두하게 할 것입니다.
