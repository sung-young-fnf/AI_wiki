지금 이 글을 읽어볼 가치가 있는 두 가지 이유가 있습니다.

첫째, 오픈소스 모델(open-source model)이 이제 일상 업무의 상당 부분을 처리할 수 있습니다. 많은 코딩, 글쓰기, 자동화 및 에이전트(agentic) 작업에 대해 더 이상 모든 단계에서 GPT-5.4 또는 코덱스 5.3(Codex 5.3)을 기본으로 사용할 필요가 없습니다.

둘째, 클라우드 API(cloud API)로 전송되는 모든 프롬프트(prompt)는 사용자 기기를 떠나 서드파티 인프라(third-party infrastructure)를 통과합니다. 독점적인 코드베이스(codebase), 민감한 프로토타입(prototype) 또는 규제 산업에서는 이는 시작부터 불가능한 일입니다. 로컬에서 추론(inference)을 실행하면 모든 데이터가 장치 내(on-device)에 유지되며, 많은 팀에게 데이터 주권(data sovereignty)만으로도 설정 비용을 정당화할 수 있습니다.

클로드 코드(Claude Code)의 실제 사용량은 고빈도 사용자(heavy user)의 경우 월 100~200달러에 달합니다. 클라인(Cline) 또는 킬로 코드(Kilo Code)를 통해 클로드 소네트 4.6(Claude Sonnet 4.6)을 실행하는 데는 현재 API 요율로 시간당 약 3~8달러의 비용이 듭니다. 에이전트 스웜(agent swarms)을 실행하거나, 빈번하게 반복 작업하거나, 많은 하위 작업을 위임하는 경우, 로컬 모델은 프론티어 역량(frontier capability)이 필요 없는 작업에 프론티어 크레딧(frontier credit)을 소모하는 것을 멈출 수 있는 가장 쉬운 방법 중 하나입니다.

리더보드 헤드라인(leaderboard headline) 외에도, 클로드 코드(Claude Code) 및 코덱스(Codex)를 사용한 로컬 설정, 엔비디아(NVIDIA) 및 맥(Mac) 설정용 하드웨어 등급 가이드, 그리고 운영상 주의사항(operational gotchas), 실패 모드(failure modes), 트레이드오프(trade-offs) — 지연 시간(latency), 메모리 점유율(memory footprint), 도구 사용(tool use), 코딩 신뢰성(coding reliability), 지시 이행 품질(instruction-following quality) — 에 대해 설명하겠습니다.

## 모델 계열 선택 전 하드웨어 등급 가이드(Hardware Tier Guide)

실행할 수 있는 모델은 보유한 하드웨어에 의해 제약받습니다. 맥북 에어(MacBook Air)를 사용하면서 70B 모델을 평가하는 것은 무의미합니다.

전체 크기로 이미지를 보려면 Enter 키를 누르거나 클릭하세요.

Q4_K_M 양자화(quantization)는 대부분의 모델에서 MMLU(Massive Multitask Language Understanding) 벤치마크(benchmark) 점수를 전체 정밀도(full-precision) 대비 1~3점 이내로 유지합니다. 그러나 성능 저하는 작업에 따라 다르며, 다단계 수학적 추론과 같은 전문화된 워크로드(workload)에서는 5%를 초과할 수도 있습니다. Q4_K_M으로 시작하세요. 목표 작업에서 정확도 저하가 관찰되면 Q5_K_M으로 전환하세요.

### 1. Qwen3 (알리바바) — 코딩 및 다국어 에이전트에 최적

크기: 1.7B, 7B, 14B, 32B, 72B, 235B MoE(Mixture of Experts)
최적 하드웨어: RTX 4060 → Mac M4 → 워크스테이션(Workstation)

Qwen3 7B는 이 비교에서 8B 미만 파라미터(parameter) 모델 중 가장 높은 HumanEval(휴먼이발) 점수(76.0)를 기록했습니다. 이는 라마 3.3(Llama 3.3)의 72.6점보다 3.4점 높은 수치입니다. 이 모델의 다국어 지원은 4가지 주요 계열 중 가장 강력하며, 강력한 영어 성능과 더불어 CJK(중국어, 일본어, 한국어) 언어에서 특히 뛰어납니다.

32B 밀집(dense) 변형 모델은 단일 24GB GPU에서 진지한 에이전트(agentic) 코딩 작업에 가장 적합합니다. 2026년 3월 현재, Qwen3.5와 GLM-4.7-Flash는 24GB RAM/통합 메모리(unified memory) 장치에서 자율적으로 복잡한 작업을 잘 실행하는 가장 강력한 35B MoE 에이전트 및 코딩 모델입니다.

**클로드 코드(Claude Code)로 설정하기:**

```bash
# llama.cpp를 통해 Qwen3 32B Q4_K_M 다운로드
./llama-cli \
  -m qwen3-32b-instruct-q4_k_m.gguf \
  --server \
  --host 127.0.0.1 \
  --port 8080 \
  --ctx-size 32768

# 클로드 코드(Claude Code)를 로컬 서버(local server)로 지정
export ANTHROPIC_BASE_URL=http://127.0.0.1:8080
export ANTHROPIC_API_KEY=sk-dummy-local

# 이제 클로드 코드(Claude Code)가 로컬 Qwen3 인스턴스(instance)로 라우팅됩니다.
claude
```

**운영상 주의사항(Operational gotchas):** 서구 개발자들에게 주요 약점은 툴링(tooling)입니다. Qwen3의 도구 사용 스키마(tool-use schema)는 OpenAI 표준 함수 호출(function calling)과 약간 다른 규칙을 따르므로, 엄격한 OpenAI 형식 준수를 가정하는 일부 에이전트 프레임워크(agent framework)에서 눈에 띄지 않는(silent) 오류를 유발합니다. 에이전트 스웜(agent swarms)을 실행하기 전에 도구 사용을 명시적으로 테스트하세요.

**실패 모드(Failure modes):** 7B 변형 모델에서 24K 토큰(token)을 초과하는 긴 컨텍스트(context)에서의 성능 저하. 새로운 아키텍처(architectural) 문제에 대한 다단계 추론 신뢰성은 32B 이상 모델에 비해 현저히 떨어집니다.

### 2. DeepSeek-Coder V3 / DeepSeek V3 (딥시크) — 순수 코드 작업에 최적

크기: 7B, 14B, 33B, 236B MoE (V3 전체)
최적 하드웨어: RTX 4070 (14B) → 워크스테이션(Workstation) (전체 236B MoE)

DeepSeek V3는 235B 파라미터(parameter)(22B 활성)를 가지며 코딩 및 에이전트(agentic) 작업에 명시적으로 튜닝되었습니다. MoE 아키텍처(architecture)는 추론(inference) 중 활성 파라미터(parameter) 수가 전체 파라미터 수보다 훨씬 적다는 것을 의미합니다. 즉, 22B 규모의 컴퓨팅 요구사항(compute requirement)으로 235B 규모의 지식을 얻을 수 있습니다.

14B 밀집(dense) 변형 모델은 단일 16GB GPU에 여유롭게 실행되며, HumanEval(휴먼이발) 및 SWE-bench(스위-벤치) 서브셋(subset)에서 이전 세대의 많은 34B 모델을 능가합니다. 워크로드(workload)가 주로 코드 — 생성, 검토, 리팩토링(refactoring), 테스트 작성 — 인 팀에게 이 계열은 이 목록의 다른 어떤 모델보다 일관되게 체급 이상으로 성능을 발휘합니다.

**올라마(Ollama)로 설정하기:**

```bash
# DeepSeek-Coder V3 14B 풀(pull)
ollama pull deepseek-coder-v3:14b

# OpenAI 호환 엔드포인트(endpoint)로 서비스
ollama serve
# 엔드포인트: http://localhost:11434/v1

# 클로드 코드(Claude Code) 연결
export ANTHROPIC_BASE_URL=http://localhost:11434/v1
export ANTHROPIC_API_KEY=ollama
claude
```

**운영상 주의사항(Operational gotchas):** 코드 관련 작업이 아닌 다른 작업에 대한 지시 이행 능력은 코드 작업에 비해 현저히 약합니다. 코딩, 글쓰기, 도구 사용 등 혼합된 워크로드(workload)를 라우팅(routing)하는 경우, DeepSeek-Coder V3는 코딩은 잘 처리하지만 다른 모든 작업에서는 실망스러울 것입니다. 이 모델을 일반 에이전트(agent)가 아닌 전용 코드 작업자로 사용하세요.

**실패 모드(Failure modes):** 7B 및 14B 변형 모델에서는 마크다운(Markdown) 형식이 때때로 일관적이지 않습니다. 파일 시스템(filesystem)에 어떤 것도 커밋(commit)하기 전에 에이전트 출력에 대한 검토 단계를 계획하십시오.

### 3. Llama 3.3 (메타) — 최강의 생태계 및 도구 지원

크기: 8B, 70B
최적 하드웨어: RTX 4060 8GB (8B Q4) → Mac M3/M4 36GB 이상 (70B Q4)

Llama 3.3은 광범위한 커뮤니티 미세 조정(fine-tune)과 폭넓은 툴링(tooling) 지원을 갖춘 가장 큰 오픈 웨이트(open-weight) 생태계입니다. 대부분의 개발자에게 Llama 3.3 8B는 전반적으로 최고의 시작점입니다.

Llama가 순수 벤치마크(benchmark) 성능보다 생태계에서 앞서는 이유는 메타(Meta)의 모델 출시 주기(model release cadence)와 OpenAI 호환 API 형식 덕분에 모든 미세 조정(fine-tuning) 노력, 모든 최적화 라이브러리(optimization library), 그리고 로컬 모델 지원을 추가하는 모든 에이전트 프레임워크(agent framework)의 기본 대상이 되었기 때문입니다. 새로운 도구가 로컬 LLM(Local LLM) 지원을 추가할 때 Llama는 가장 먼저 테스트하는 모델 계열입니다.

이는 통합 시 예측치 못한 문제(surprises)가 적다는 것을 의미합니다. 눈에 띄지 않는 형식 오류가 적습니다. 도구 사용 시 스키마 불일치(schema mismatch)가 적습니다.

**도커 모델 러너(Docker Model Runner)로 설정하기:**

```bash
# 도커 모델 러너(Docker Model Runner)를 통해 Llama 3.3 8B 풀(pull) 및 실행
docker model pull ai/llama3.3:8B-Q4_K_M
docker model run ai/llama3.3:8B-Q4_K_M

# OpenAI 호환 엔드포인트(endpoint)
# http://localhost:12434/engines/llama.cpp/v1

# 클로드 코드(Claude Code) 통합
export ANTHROPIC_BASE_URL=http://localhost:12434/engines/llama.cpp/v1
export ANTHROPIC_API_KEY=sk-dummy-local
claude
```

**운영상 주의사항(Operational gotchas):** Llama 3.3 70B는 진지한 작업에 실행할 가치가 있는 변형 모델이지만, Q4에서 40GB 이상의 VRAM/통합 메모리(unified memory)가 필요합니다. 이는 맥 M3 Max(48GB) 또는 M4 Max(128GB) 또는 리눅스(Linux)의 다중 GPU 설정(multi-GPU setup)을 의미합니다. 8B 변형 모델은 일상적인 작업을 수행하는 데 실제로 유능하고 거의 모든 곳에 적합하지만, 다단계 추론 작업에서 8B와 70B 사이의 격차는 파라미터(parameter) 수 차이가 시사하는 것보다 큽니다.

**실패 모드(Failure modes):** 8B 변형 모델의 컨텍스트(context) 창 처리는 16K 토큰(token)을 넘어서면 성능이 저하됩니다. 컨텍스트를 엄격하게 범위 지정하거나 에이전트 프레임워크(agent framework)에서 슬라이딩 윈도우(sliding window) 전략을 사용하세요.

### 4. Phi-4 (마이크로소프트) — 자원 제약 환경에 최적

크기: Phi-4-mini (3.8B), Phi-4 (14B), Phi-4-reasoning (14B)
최적 하드웨어: 4GB RAM이 있는 모든 하드웨어 (mini) → RTX 4060 (14B)

Phi-4-mini (3.8B)는 저자원 하드웨어(low-resource hardware)에서 유일한 실제 선택지입니다. Phi-4-mini는 Phi-4 14B와는 별개의 모델이며, 자체적인 벤치마크(benchmark) 프로파일(profile)을 가지고 있습니다. 파라미터(parameter) 수 대비 고성능을 위해 설계되었으며, 자원 제약이 있는 장치를 대상으로 합니다.

Phi-4-reasoning 및 Phi-4-reasoning-plus는 박사 학위 수준의 질문에 대한 수학 및 과학 추론 대부분의 벤치마크(benchmark)에서 OpenAI o1-mini(오픈AI o1-미니) 및 DeepSeek-Distill-Llama-70B(딥시크-디스틸-라마-70B)를 능가했습니다. 이는 일반적인 능력에 대한 주장이 아니라, 구조화된 추론 작업에서 구체적이고 진정한 이점입니다.

CI/CD 파이프라인(CI/CD pipeline), 엣지 배포(edge deployment), IoT 기기 또는 엄격한 메모리 상한선(hard memory ceiling)이 있는 LLM 추론(inference)이 필요한 모든 시나리오에서, 이 목록의 다른 어떤 계열도 Phi-4-mini의 성능 대비 점유율 비율(performance-to-footprint ratio)과 경쟁할 수 없습니다.

**CI/CD 통합을 위한 설정:**

```bash
# CI 파이프라인(CI pipeline)용 llama.cpp의 Phi-4-mini
./llama-server \
  -m phi-4-mini-instruct-q4_k_m.gguf \
  --port 8080 \
  --ctx-size 8192 \
  --n-gpu-layers 0  # CI 러너(runner)용 CPU 전용

# GitHub 액션(GitHub Actions)에서 사용
- name: AI Code Review
  run: |
    curl http://localhost:8080/completion \
      -d '{"prompt": "Review this diff for security issues: '"$(git diff HEAD~1)"'",
           "max_tokens": 512}'
```

**운영상 주의사항(Operational gotchas):** 복잡한 다단계 추론 작업에서 한계가 드러나는데, 3.8B 변형 모델의 정확도는 7/8B 경쟁 모델에 비해 현저히 떨어집니다. 작은 모델은 양자화 손실(quantization loss)에 더 민감할 수 있으므로, 목표 작업에서 정확도 저하가 관찰되면 Q5_K_M을 평가해 보십시오.

**실패 모드(Failure modes):** 미니 변형 모델에서는 길고 여러 부분으로 구성된 시스템 프롬프트(system prompt)에 대한 지시 이행이 일관적이지 않습니다. 시스템 프롬프트를 200 토큰(token) 미만으로 유지하고 작업 지시를 가능한 한 단일 목표로 만드십시오.

### 5. Mistral Small 3 / Mixtral 8x7B (미스트랄 AI) — 처리량에 최적

크기: Mistral Small 3 (7B 밀집(dense)), Mixtral 8x7B MoE
최적 하드웨어: RTX 4060 8GB (Small 3) → RTX 4090 24GB (Mixtral 8x7B)

Mistral Small 3 7B는 처리량(throughput)이 가장 중요할 때 빛을 발합니다. 추론 속도(inference speed)와 강력한 지시 이행 능력으로 잘 알려져 있습니다.

병목 현상(bottleneck)이 추론 깊이(reasoning depth)보다는 초당 토큰(token) 수(대량 요약, 여러 문서를 동시에 처리하는 병렬 에이전트 워커(parallel agent worker), 실시간 응답 생성 등)에 있다면, Mistral Small 3는 이 등급의 다른 어떤 모델보다 GPU 비용당 더 많은 처리량(throughput)을 제공합니다. RTX 4060에서 Mistral Small 3 Q4는 초당 50~80 토큰으로 실행되는 반면, Llama 3.3 8B는 동일한 하드웨어에서 40~60 토큰으로 실행됩니다.

**운영상 주의사항(Operational gotchas):** Mixtral 8x7B는 전체 MoE(Mixture of Experts) 가중치 행렬(weight matrix)을 로드해야 하며, 추론(inference) 중 일부만 활성화되더라도 24~48GB의 시스템 메모리(system memory)가 필요합니다. 24GB VRAM과 64GB 시스템 RAM을 갖춘 머신에서는 오프로딩(offloading)이 작동하지만, 오프로드된 레이어(offloaded layer)에서 지연 시간(latency)이 발생합니다. 모델 사용을 결정하기 전에 실제 하드웨어에서 벤치마크(benchmark) 테스트를 실시하십시오.

**실패 모드(Failure modes):** 미스트랄(Mistral) 모델은 라마(Llama)와 약간 다른 채팅 템플릿(chat template) 형식을 사용합니다. 대부분의 프레임워크(framework)는 이를 자동으로 처리하지만, 프롬프트(prompt)를 수동으로 구성하는 API 통합(API integration)의 경우 잘못된 템플릿을 사용하면 지시 이행 성능이 저하될 것입니다. `tokenizer_config.json`을 확인하고 채팅 템플릿을 명시적으로 일치시키십시오.

### 6. GLM-4.7 (Z.AI / 지푸 AI) — 멀티모달 및 긴 컨텍스트 에이전트 작업에 최적

크기: GLM-4.7-Flash (9B), GLM-4.7 (35B MoE)
최적 하드웨어: RTX 4070 16GB (Flash) → RTX 4090 / Mac M4 Max (35B MoE)

GLM-4.7-Flash는 2026년 3월 현재 멀티모달(multimodal) 입력을 요구하는 에이전트(agentic) 워크플로우(workflow)에 사용될 수 있는 가장 강력한 35B MoE 에이전트 및 코딩 모델입니다.

GLM-4.7은 이 목록에서 서구 개발자에게 가장 덜 친숙한 계열이지만, 바로 그 이유 때문에 여기에 포함되었습니다. 이 모델은 이미지와 텍스트 입력을 네이티브(native)로 교차 처리(interleaved)하여, 코드와 함께 스크린샷, 다이어그램, UI 목업(mockup)을 처리하는 에이전트(agentic) 워크플로우(workflow)가 별도의 비전 모델(vision model) 없이 해당 작업을 로컬에서 라우팅(routing)할 수 있음을 의미합니다.

Z.AI의 GLM-5 및 GLM-4.7은 여전히 유효하지만, 개별 코드 프롬프트(code prompt)보다는 실제 에이전트 루프(agent loop) 내에서 테스트할 때 그 강점이 덜 명확해집니다. Flash 변형(9B 활성 파라미터(parameter))이 실용적인 진입점(entry point)입니다. 전체 35B MoE(Mixture of Experts) 모델에서 멀티모달(multimodal) 기능이 진정으로 차별화됩니다.

**운영상 주의사항(Operational gotchas):** GLM-4.7의 신뢰 점수(trust scoring)는 서구 모델과 다르게 보정되어 있습니다. 의도가 합법적일지라도 보안 관련 코드를 포함하는 요청을 거부할 가능성이 더 높습니다. 침투 테스트(penetration testing), 취약점 스캔(vulnerability scanning) 또는 보안 검토 작업(security review task)을 위한 시스템 프롬프트(system prompt)에 명시적인 컨텍스트(context)를 추가하십시오.

**실패 모드(Failure modes):** 도구 사용 스키마(tool-use schema) 호환성은 프레임워크(framework)마다 다릅니다. 배포하기 전에 사용하는 에이전트 프레임워크(agent framework)로 명시적으로 테스트하십시오. Flash 변형 모델의 함수 호출(function calling)은 동일한 파라미터(parameter) 규모에서 DeepSeek-Coder V3보다 신뢰성이 떨어집니다.

### 7. Qwen3.5 (알리바바) — 2026년 진지한 에이전트 워크로드에 전반적으로 최적

크기: 7B, 14B, 32B, 72B, 235B MoE
최적 하드웨어: RTX 4070 (14B) → Mac M4 Max (72B) → 워크스테이션(Workstation) (235B MoE)

Qwen3.5는 2026년 3월 현재 코딩 신뢰성(coding reliability)과 24GB 장치에서의 긴 컨텍스트(context) 처리를 모두 요구하는 워크로드(workload)에 가장 강력한 에이전트(agentic) 및 코딩 모델입니다.

중간 규모의 Qwen3.5 증류(distilled) 9B 모델은 일상적인 개발 작업에서 소형화(compactness)와 성능 사이의 최상의 균형을 이룹니다. 32B 밀집(dense) 변형 모델은 진지한 에이전트(agentic) 작업 — 다중 파일 리팩토링(refactoring), 복잡한 디버깅 세션(debugging session), 대규모 코드베이스(codebase)의 전체 컨텍스트(context)를 유지해야 하는 아키텍처(architecture) 수준의 추론 — 이 가능한 지점입니다.

**전체 로컬 에이전트 설정:**

```bash

git clone https://github.com/ggml-org/llama.cpp

  -DBUILD_SHARED_LIBS=OFF \
  -DGGML_CUDA=ON
cmake --build llama.cpp/build --config Release -j \
  --target llama-server

# Qwen3.5 32B MoE 서비스
./llama.cpp/build/bin/llama-server \
  -m qwen3.5-32b-instruct-q4_k_m.gguf \
  --host 127.0.0.1 \
  --port 8080 \
  --ctx-size 65536 \
  -ngl 40  # GPU의 레이어(layer)

# 클로드 코드(Claude Code) → 로컬 Qwen3.5

# ~/.claude.json에 추가하여 지속:
# "primaryApiKey": "sk-local-dummy"

```

**운영상 주의사항(Operational gotchas):** 긴 컨텍스트(context) 처리는 Qwen3.5의 차별점(differentiator)입니다. 특정 양자화된 변형 모델에 대한 정확한 지원 컨텍스트 창(context window) 길이는 허깅 페이스(Hugging Face)의 모델 카드(model card)를 확인하십시오. Q4_K_M은 전체 정밀도(full-precision)에 비해 일부 컨텍스트 창 구성을 잘라냅니다.

**실패 모드(Failure modes):** 235B MoE(Mixture of Experts)는 실용적인 추론 속도(inference speed)를 위해 다중 노드 설정(multi-node setup) 또는 NVLink로 연결된 듀얼 GPU(dual GPU) 장비가 필요합니다. 단일 GPU, 심지어 4090에서도 전체 MoE는 대화형(interactive) 사용에는 너무 느립니다. 이는 소비자용 하드웨어(consumer hardware)에서 배치 처리(batch processing) 모델입니다.

## 라우팅 가이드(Routing Guide)

더 이상 모든 단계에서 프론티어 클라우드 모델(frontier cloud model)을 기본으로 사용할 필요가 없습니다. 많은 코딩, 글쓰기, 자동화 및 에이전트(agentic) 작업에 대해 이제 로컬 모델(local model)이 작업을 안정적으로 처리합니다.

라우팅(routing)에 대해 생각하는 방법은 다음과 같습니다:

전체 크기로 이미지를 보려면 Enter 키를 누르거나 클릭하세요.

가장 생산적인 개발자들은 조합을 사용합니다. 자동화를 위한 터미널(Terminal) 에이전트(agent), 일상 업무를 위한 IDE(Integrated Development Environment) 에이전트, 그리고 프론티어 역량(frontier capability)이 필요 없는 작업을 위한 로컬 모델(local model)을 사용합니다.

목표는 클라우드 API(cloud API) 사용을 완전히 없애는 것이 아닙니다. 프론티어 역량(frontier capability)이 실제 병목 현상(bottleneck)인 작업에만 사용하도록 클라우드 API 사용을 의도적으로 만들고, 모든 것의 기본값이 아니게 하는 것입니다.
