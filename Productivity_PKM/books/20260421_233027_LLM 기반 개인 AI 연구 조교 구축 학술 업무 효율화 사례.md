# LLM 기반 개인 AI 연구 조교 구축: 학술 업무 효율화 사례

정보 과부하에서 효율적인 연구로의 여정.

대학원생으로서 저는 매일 정보 과부하와 씨름합니다. 논문 읽기, 제안서 작성, 인용 관리, 최신 사전 공개 논문 확인 등, 브라우저 탭은 마치 토끼처럼 불어납니다. 몇 년 동안 저는 실제 연구를 하는 시간보다 연구를 정리하는 데 더 많은 시간을 보내는 것 같았습니다. 저는 해결책이 필요했습니다. 제 작업의 맥락을 이해하고, 불필요한 정보를 걸러내며, 진정한 인지적 파트너 역할을 할 수 있는 솔루션 말입니다.

이 글은 제가 대규모 언어 모델(LLM)을 활용하여 개인 AI 연구 조교를 구축한 이야기입니다. 이는 제 사고를 대체하는 것이 아니라 보완하는 데 목적이 있었습니다. 제가 '헤르메스'(그리스 신화의 전령 신)라고 명명한 이 시스템은 이제 학술 처리의 번거로운 작업을 처리하여, 제가 창의성과 분석에 집중할 수 있도록 돕습니다.

이 글에서는 나만의 AI 연구 조교를 구축하는 데 필요한 아키텍처, 도구, 코드, 그리고 사고 과정에 대해 설명해 드리고자 합니다.

## 1. 문제점: 학술 정보의 홍수

제가 이 시스템이 왜 필요했는지 이해하려면, 일반적인 화요일의 제 상황을 알아야 합니다. 저는 '트랜스포머(Transformer) 효율성에 대한 논문 읽기'라는 구체적인 목표로 하루를 시작했습니다. 한 시간도 안 되어 다음과 같은 상황에 처했습니다.

*   arXiv에서 5개의 PDF 파일 열람
*   블로그 게시물 및 관련 논문이 담긴 10개의 브라우저 탭
*   마크다운(Markdown) 파일에 흩어진 노트들
*   20개의 새로 읽지 않은 항목이 있는 Zotero 라이브러리

지난주에 읽었던 특정 인용문을 찾는 것은 악몽과 같았습니다. 여러 논문 간의 아이디어를 연결하는 것은 오로지 제 불완전한 기억력에 의존해야만 했습니다. 저는 제 연구의 흐름을 놓치고 있었습니다.

## 2. 비전: 헤르메스가 수행해야 할 역할

저는 제 작업 흐름과 함께하는 조교를 구상했습니다. 논문을 직접 작성할 필요는 없었지만, 다음과 같은 기능을 수행해야 했습니다.

*   **모든 정보 수집 및 처리 (Ingest Everything):** 제가 입력하는 모든 PDF, 웹사이트, 노트를 자동으로 읽고 이해해야 합니다.
*   **질의응답 (Answer Questions):** "Attention is All You Need 논문의 주요 한계점은 무엇이었나요?" 또는 "다중 모달(multimodal) 학습에 대한 최신 논문들을 요약해 주세요."와 같은 자연어 질문을 할 수 있어야 합니다.
*   **아이디어 연결 (Connect Ideas):** 제가 읽었던 논문과 새로 발견하는 논문들 사이의 연결고리를 적극적으로 제안해야 합니다.
*   **초안 작성 지원 (Draft Content):** 기존 지식 기반을 바탕으로 문헌 검토 단락, 초록, 지도 교수에게 보내는 이메일 초안 작성을 도와야 합니다.
*   **개인 정보 보호 (Be Private):** 제 연구는 저의 지적 재산입니다. 핵심 처리는 공개 서버가 아닌 로컬 또는 제 자체 인프라에서 이루어져야 합니다.

## 3. 기술 스택: 적합한 도구 선택

AI 도구의 환경은 방대하지만, 저는 유연성, 개인 정보 보호, 비용 효율성을 우선시하는 스택을 선택했습니다.

*   **LLM 공급자 (LLM Provider):**
    *   **주력:** Ollama (로컬 모델 실행용). 주로 빠른 작업에는 `llama3:8b`를, 복잡한 추론에는 `mixtral:8x7b`를 사용합니다. 이는 제 노트와 작성 중인 초안의 개인 정보 보호를 보장합니다.
    *   **보조:** OpenAI API (GPT-4). 극도의 미묘함이 필요하거나 두 번째 의견이 필요할 때 GPT-4를 사용하지만, 개인 노트가 아닌 익명화된 논문 초록만 전송합니다.
*   **프레임워크 (Framework):** LangChain. 이는 전체 시스템을 연결하는 접착제 역할을 합니다. 체인 구축, 프롬프트 관리, 벡터 데이터베이스 연결을 위한 도구를 제공합니다.
*   **벡터 데이터베이스 (Vector Database):** ChromaDB. 가볍고 로컬에서 실행되며, 개인 프로젝트에 적합합니다.
*   **임베딩 모델 (Embeddings Model):** `BAAI/bge-small-en`. 이 모델은 Ollama/transformers를 통해 로컬에서 실행되며, 제 문서를 의미를 포착하는 숫자 표현(벡터)으로 변환합니다.
*   **프런트엔드 (Frontend):** Streamlit. Python 기반으로, 간단하고 상호작용적인 웹 UI를 몇 분 만에 구축할 수 있습니다.
*   **PDF 처리 (PDF Processing):** PyPDF2 및 pdfplumber (텍스트 추출용).

## 4. 1단계: 지식 기반 구축 — 나의 디지털 두뇌

첫 번째 단계는 기억력을 구축하는 것이었습니다. 모든 연구 자료를 위한 중앙 저장소가 필요했습니다. 저는 로컬 머신에 다음과 같은 폴더 구조를 만들었습니다.

```
~/research_knowledge_base/
├── papers/          # 모든 PDF 파일 (예: Author_Year_Title.pdf)
├── notes/           # 개인 .md 노트 (주제별 태그)
├── websites/        # 저장된 웹페이지 (.html 또는 .txt)
└── metadata/        # 처리된 내용을 추적하는 간단한 CSV
```

이 폴더가 모든 정보의 원천입니다. AI 조교의 역할은 이 폴더를 색인화하고 검색 가능하게 만드는 것입니다.

## 5. 2단계: 정보 수집 파이프라인 — 두뇌에 정보 공급

이는 제 폴더를 모니터링하고 문서를 AI가 이해할 수 있는 형식으로 변환하는 자동화된 프로세스입니다.

다음은 LangChain을 사용하여 이 파이프라인을 구동하는 Python 코드입니다 (간소화됨).

```python
# ingest.py
import os
from langchain.document_loaders import PyPDFLoader, TextLoader, UnstructuredMarkdownLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.embeddings import HuggingFaceEmbeddings
from langchain.vectorstores import Chroma
from langchain.schema import Document
import hashlib
# 설정
PERSIST_DIRECTORY = "chroma_db"
SOURCE_DIRECTORY = "~/research_knowledge_base"
EMBEDDING_MODEL = "BAAI/bge-small-en"
def hash_file_content(content):
    """문서 내용을 해싱하여 재처리 방지."""
    return hashlib.md5(content.encode()).hexdigest()
def load_documents():
    """소스 디렉토리에서 문서 로드."""
    documents = []
    for root, _, files in os.walk(SOURCE_DIRECTORY):
        for file in files:
            file_path = os.path.join(root, file)
            try:
                if file.endswith(".pdf"):
                    loader = PyPDFLoader(file_path)
                elif file.endswith(".md"):
                    loader = UnstructuredMarkdownLoader(file_path)
                elif file.endswith(".txt"):
                    loader = TextLoader(file_path, encoding="utf-8")
                else:
                    continue
                docs = loader.load()
                # 원본 파일 경로와 같은 메타데이터 추가
                for doc in docs:
                    doc.metadata["source"] = file_path
                    doc.metadata["file_hash"] = hash_file_content(doc.page_content)
                documents.extend(docs)
                print(f"Loaded: {file_path}")
            except Exception as e:
                print(f"Error loading {file_path}: {e}")
    return documents
def split_documents(documents):
    """문서를 더 작은 청크로 분할."""
    text_splitter = RecursiveCharacterTextSplitter(
        chunk_size=1000,
        chunk_overlap=200,
        length_function=len,
    )
    return text_splitter.split_documents(documents)
def create_or_update_vectorstore():
    """정보 수집을 실행하는 메인 함수."""
    print("Loading documents...")
    raw_docs = load_documents()
    print("Splitting documents...")
    chunks = split_documents(raw_docs)
    print(f"Created {len(chunks)} chunks.")
    print("Initializing embeddings model...")
    embeddings = HuggingFaceEmbeddings(model_name=EMBEDDING_MODEL)
    print("Creating/updating vector store...")
    # 간단화를 위해 다시 생성합니다. 실제 앱에서는 해시를 확인합니다.
    if os.path.exists(PERSIST_DIRECTORY):
        import shutil
        shutil.rmtree(PERSIST_DIRECTORY)
    vectorstore = Chroma.from_documents(
        documents=chunks,
        embedding=embeddings,
        persist_directory=PERSIST_DIRECTORY
    )
    vectorstore.persist()
    print("Ingestion complete!")
if __name__ == "__main__":
    create_or_update_vectorstore()
```

저는 이 스크립트를 크론잡(cron job)을 통해 매일 밤 실행합니다. 이 스크립트는 제 지식 기반을 스캔하고, 모든 문서를 1000자 청크(중복 포함)로 분할한 다음, 각 청크에 대한 임베딩을 생성하여 ChromaDB에 저장합니다.

## 6. 3단계: RAG 시스템 — 헤르메스의 ‘사고’ 방식

이제 마법 같은 부분입니다: 검색 증강 생성(Retrieval-Augmented Generation, RAG). 이것이 헤르메스의 핵심입니다. 제가 질문을 하면, 시스템은 단순히 답변을 추측하는 것이 아닙니다. 다음과 같은 정밀한 프로세스를 따릅니다.

*   **검색 (Retrieve):** 제 질문을 임베딩으로 변환하고, 벡터 데이터베이스에서 의미론적으로 가장 유사한 5~10개의 텍스트 청크를 검색합니다.
*   **증강 (Augment):** 검색된 청크를 신중하게 작성된 프롬프트에 삽입하여, LLM이 제공된 맥락만을 기반으로 질문에 답하도록 지시합니다.
*   **생성 (Generate):** 이 증강된 프롬프트를 LLM(로컬 Llama 3 또는 API를 통한 GPT-4)으로 보내고, 응답을 저에게 스트리밍합니다.

다음은 질의응답(Q&A) 체인을 위한 코드입니다.

```python
# qa_engine.py
from langchain.vectorstores import Chroma
from langchain.embeddings import HuggingFaceEmbeddings
from langchain.llms import Ollama
from langchain.chains import RetrievalQA
from langchain.prompts import PromptTemplate
PERSIST_DIRECTORY = "chroma_db"
EMBEDDING_MODEL = "BAAI/bge-small-en"
# 사실에 기반한 답변을 유도하기 위한 맞춤 프롬프트
PROMPT_TEMPLATE = """You are an AI research assistant. Use the following pieces of context to answer the question at the end. If you don't know the answer based *only* on the context, just say that you don't know. Don't try to make up an answer.
Context:
{context}
Question: {question}
Helpful Answer:"""
def get_qa_chain(model_name="llama3:8b"):
    # 벡터 저장소 로드
    embeddings = HuggingFaceEmbeddings(model_name=EMBEDDING_MODEL)
    vectorstore = Chroma(persist_directory=PERSIST_DIRECTORY, embedding_function=embeddings)
    # LLM 설정 (Ollama를 통한 로컬)
    llm = Ollama(model=model_name, temperature=0.3) # 사실적 답변을 위한 낮은 온도 설정
    # 프롬프트 생성
    prompt = PromptTemplate(template=PROMPT_TEMPLATE, input_variables=["context", "question"])
    # 검색 체인 생성
    qa_chain = RetrievalQA.from_chain_type(
        llm=llm,
        chain_type="stuff", # "stuff"는 검색된 모든 문서를 하나의 프롬프트에 넣습니다.
        retriever=vectorstore.as_retriever(search_kwargs={"k": 5}),
        return_source_documents=True, # 사용된 문서 표시
        chain_type_kwargs={"prompt": prompt}
    )
    return qa_chain
def ask_question(question):
    chain = get_qa_chain()
    result = chain({"query": question})
    return result
```

## 7. 4단계: 인터페이스 — 헤르메스와 대화

명령줄도 괜찮지만, 시각적인 인터페이스는 조교를 실제 도구처럼 느끼게 합니다. 저는 Streamlit을 사용하여 간단한 채팅 인터페이스를 구축했습니다.

```python
# app.py
import streamlit as st
from qa_engine import ask_question
import time
st.set_page_config(page_title="Hermes - AI Research Assistant", layout="wide")
st.title("🧑‍🔬 Hermes: 나의 개인 연구 조교")
# 제어판 사이드바
with st.sidebar:
    st.header("설정")
    st.info("로컬 지식 기반에 연결됨.")
    if st.button("🔄 지식 기반 재색인"):
        with st.spinner("재색인 중... 몇 분 소요될 수 있습니다."):
            # 실제 앱에서는 여기에 수집 함수를 호출합니다.
            # import ingest
            # ingest.create_or_update_vectorstore()
            time.sleep(2) # 작업 시뮬레이션
            st.success("지식 기반 재색인 완료!")
    st.header("세션 정보")
    st.write(f"**적용 문서:** `~/research_knowledge_base` 폴더 내의 모든 파일")
    st.write(f"**현재 LLM:** Llama 3 (로컬)")
# 메인 채팅 영역
if "messages" not in st.session_state:
    st.session_state.messages = [
        {"role": "assistant", "content": "안녕하세요! 저는 헤르메스입니다. 연구 자료에 대해 무엇이든 물어보세요."}
    ]
# 채팅 메시지 표시
for message in st.session_state.messages:
    with st.chat_message(message["role"]):
        st.markdown(message["content"])
# 채팅 입력
if prompt := st.chat_input("연구에 대해 질문하세요..."):
    # 사용자 메시지 추가
    st.session_state.messages.append({"role": "user", "content": prompt})
    with st.chat_message("user"):
        st.markdown(prompt)
    # 조교 응답 생성
    with st.chat_message("assistant"):
        message_placeholder = st.empty()
        full_response = ""
        try:
            # QA 엔진 호출
            result = ask_question(prompt)
            answer = result['result']
            source_docs = result['source_documents']
            # 스트리밍 효과 시뮬레이션
            for chunk in answer.split():
                full_response += chunk + " "
                time.sleep(0.05)
                message_placeholder.markdown(full_response + "▌")
            message_placeholder.markdown(full_response)
            # 확장 패널에 출처 표시
            with st.expander("출처 보기"):
                for i, doc in enumerate(source_docs):
                    st.caption(f"**출처 {i+1}:** `{doc.metadata['source']}`")
                    st.text(doc.page_content[:300] + "...")
                    st.divider()
        except Exception as e:
            st.error(f"오류가 발생했습니다: {e}")
            full_response = "죄송합니다. 오류가 발생했습니다. 로그를 확인해주세요."
        st.session_state.messages.append({"role": "assistant", "content": full_response})
```

## 8. 도전 과제 극복 및 향후 작업

이 시스템을 구축하는 과정이 순탄하지만은 않았습니다. 제가 겪었던 몇 가지 난관은 다음과 같습니다.

*   **도전 과제 1: PDF 파싱(Parsing) 지옥.** 학술 PDF는 복잡한 레이아웃(두 단 컬럼, 수학 공식, 표)을 가지는 경우가 많습니다. PyPDF2는 종종 텍스트를 잘못된 순서로 추출했습니다.
    *   **해결책:** 더 나은 레이아웃 감지를 위해 pdfplumber로 전환했으며, 현재는 PDF를 마크다운(Markdown)으로 변환하여 구조를 보존하는 Meta의 Nougat (Neural Optical Understanding for Academic Documents)을 실험하고 있습니다.

*   **도전 과제 2: ‘중간에 길을 잃는’ 문제 (Lost in the Middle Problem).** LLM은 긴 맥락의 중간에 있는 정보를 무시하는 경향이 있습니다.
    *   **해결책:** 저는 재순위화(Reranking) 단계를 구현하기 위해 노력하고 있습니다. 상위 10개 청크를 검색한 후, 교차 인코더(cross-encoder) 모델을 사용하여 관련성을 재평가하고, 상위 3개만 LLM에 전달합니다.

*   **도전 과제 3: 다단계 추론 (Multi-hop Reasoning).** 예를 들어, "모델 X의 아키텍처가 데이터셋 Y에서 발견된 학습 안정성 문제에 어떻게 영향을 미치는가?"와 같은 질문은 여러 논문의 정보를 연결해야 합니다.
    *   **해결책:** 이는 쿼리 변환(query transformation) 또는 반복 검색(iterative retrieval)과 같은 더 진보된 RAG 기술을 필요로 하며, 이것이 저의 다음 주요 과제입니다.

**향후 로드맵에 포함된 기능:**

*   **자동화된 문헌 격차 분석 (Automated Literature Gap Analysis):** 헤르메스가 특정 주제에 대한 여러 논문을 분석하고 부족한 점을 제안하도록 합니다.
*   **연구 아이디어 생성 (Research Idea Generation):** 헤르메스에게 최근 노트와 논문 목록을 제공하고, 세 가지 참신한 연구 가설을 제안하도록 요청합니다.
*   **Zotero와의 통합 (Integration with Zotero):** 제 참고 문헌 관리자에 새로 추가된 논문을 자동으로 수집합니다.

## 9. 결론: 나의 새로운 작업 방식

헤르메스 구축은 제 학술 작업 흐름을 근본적으로 변화시켰습니다. 더 이상 '월요일 아침 받은 편지함'을 두려워하지 않습니다. 제가 읽은 모든 것을 알고 즉시 기억해낼 수 있는 도구를 갖게 되었습니다. 이전에는 특정 인용문을 찾는 데 썼던 시간을 이제는 그 의미를 생각하는 데 사용합니다.

이 프로젝트는 AI의 이점을 누리기 위해 거대한 기업 인프라가 필요하지 않다는 것을 증명합니다. 오픈 소스 모델, 약간의 Python 코드, 그리고 명확한 비전만 있다면, 작업을 정리할 뿐만 아니라 적극적으로 참여하는 도구를 구축할 수 있습니다.

제 연구는 더 이상 혼란스러운 논문 더미가 아닙니다. 연결되고, 검색 가능하며, 상호작용하는 아이디어의 정원이며, 헤르메스는 저의 길잡이입니다.
