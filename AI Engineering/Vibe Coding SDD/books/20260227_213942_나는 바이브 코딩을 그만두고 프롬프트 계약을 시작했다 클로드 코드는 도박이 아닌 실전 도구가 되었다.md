# 나는 '바이브 코딩'을 그만두고 '프롬프트 계약'을 시작했다 — 클로드 코드는 도박이 아닌 실전 도구가 되었다

지난 화요일 새벽 2시, 나는 클로드 코드(Claude Code)가 방금 생성해 준 2,400줄의 코드를 삭제했다.

코드가 나빴기 때문은 아니다. 컴파일도 잘 되었고, 실행도 됐으며, 심지어 코드도 깔끔해 보였다.

하지만 엉뚱한 문제를 해결하고 있었다. 나는 로우 레벨 보안(Row-level security)이 적용된 슈퍼베이스(Supabase) 인증 플로우를 요청했는데, 결과물은 파이어베이스(Firebase)를 사용한 아주 아름답고 상용 수준인 인증 시스템이었다.

클로드 코드는 내 전체 기술 스택(Stack)에 대해 전형적인 '역전 카드'를 꺼내 든 셈이다. 피자를 주문했는데 완벽하게 조리된 리조또를 받은 격이랄까. 기술적으로는 인상적이지만, 근본적으로는 틀렸다.

그 순간 나는 불편한 진실을 깨달았다. 나는 클로드 코드로 코딩을 하고 있었던 게 아니라, 도박을 하고 있었던 것이다.

만약 당신도 클로드 코드에 자연어 프롬프트를 입력하며 운 좋게 결과가 나오길 바라고 있다면, 당신 역시 주사위를 굴리고 있는 것이다. 다만 그 게임의 마스터가 맥락은 전혀 모르면서 자신감만 넘치는 AI라는 점이 다를 뿐이다.

### 바이브 코딩(Vibe Coding)의 함정

현장에서 벌어지는 '바이브 코딩'의 모습은 대략 이렇다:

> "내 SaaS를 위한 대시보드를 만들어줘."

클로드 코드는 14개 파일에 걸쳐 3,000줄의 코드를 작성한다. 당신은 마치 '매직 아이' 포스터를 보듯 출력물을 가늘게 뜨고 살펴본다. 몇몇 부분은 맞아 보인다.

실행해 본다.

무언가 고장 난다.

당신은 "고쳐줘"라고 말한다.

AI는 하나를 고치고 다른 세 개를 망가뜨린다. 이것은 디버깅(Debugging)이 아니다. 월 100달러짜리 구독료를 내며 벌이는 '두더지 잡기' 게임일 뿐이다.

45분 후, 당신은 작동은 하지만 완전히 이해하지는 못하는 무언가를 출시(Shipping)하게 된다. 축하한다. 이제 당신은 자신이 코드를 썼다는 사실조차 기억하지 못하는 외계 지능이 작성한 코드베이스(Codebase)를 유지보수해야 한다. 무엇이 잘못될 수 있을까?

나는 수개월 동안 이 루프 속에서 살았다. 거의 전적으로 클로드 코드를 사용해 두 개의 SaaS 제품을 출시했는데, 과장하지 않고 내 시간의 30%는 클로드 코드가 엉뚱한 방향으로 자신 있게 구축해 놓은 것들을 되돌리는(Un-doing) 데 사용되었다. 이는 마치 기억상실증에 걸렸으면서 당신의 저장소(Repo)에 대한 개념이 전혀 없는 천재와 페어 프로그래밍(Pair programming)을 하는 것과 같았다.

문제는 클로드 코드의 지능이 아니었다. Opus 4.6 모델은 놀라울 정도로 유능하다. 기본적으로 지금까지 게시된 모든 스택 오버플로우(Stack Overflow) 답변을 읽고, 그중 틀린 답변까지 포함해 전부 기억하고 있는 시니어 엔지니어와 같다.

문제는 내 프롬프트가 '명세서(Specifications)'가 아닌 '바이브(Vibes, 느낌)'였다는 점이다.

"반응형으로 만들어줘", "에러 핸들링을 추가해줘", "베스트 프랙티스를 사용해줘."

이것들은 지시 사항이 아니다. 점성술에 가깝다. 클로드 코드는 점술가가 아니라 계약업체(Contractor)다. 계약업체에게 "집 지어줘, 예쁘게"라고 적힌 냅킨을 건네면서 꿈의 집이 완성되기를 기대하지는 않을 것이다. 집은 지어지겠지만, 벽도 있을지 모르지만, 화장실이 주방에 있을 수도 있다. 당신이 그러면 안 된다고 말한 적이 없기 때문이다.

### 해결책: 프롬프트 계약(Prompt Contracts)

몇 주 전, 전직 OpenAI 엔지니어가 제안한 '프롬프트 계약(Prompt Contracts)'이라는 개념을 우연히 발견했다. 아이디어는 간단하다. 프롬프트를 크리에이티브 브리프(Creative brief)처럼 쓰는 대신, 네 가지 강제 조항이 포함된 법적 계약서처럼 작성하는 것이다.

1. **목표(Goal)** — 정확한 성공 지표.
2. **제약 사항(Constraints)** — 넘지 말아야 할 엄격한 경계.
3. **출력 형식(Output Format)** — 기대하는 구체적인 구조.
4. **실패 조건(Failure Conditions)** — 결과물을 수용할 수 없게 만드는 요인.

나는 생각했다. "이거 챗GPT 블로그 포스팅용으로는 귀엽네. 하지만 카페인에 찌든 새벽 3시에 클로드 코드로 실제 백엔드를 구축할 때도 통할까?"

그래서 테스트해 보았다. 3주 동안 두 개의 SaaS 프로젝트에서 모든 '바이브 프롬프트'를 '프롬프트 계약'으로 대체했다.

스포일러를 하자면, 나는 이제 잠을 잘 잔다.

#### 구성 요소 1: 목표(GOAL) — "X를 만들어줘"라고 말하지 마라

가장 큰 업그레이드는 클로드 코드가 단 한 줄의 코드를 쓰기 전에 '완료'의 정의가 무엇인지 스스로 정의하게 만든 것이었다.

*   **변경 전 (바이브):** "앱에 구독 시스템을 추가해줘."
*   **변경 후 (계약):**
    > **GOAL:** 사용자가 3개 등급(무료/프로/팀)을 구독하고, 즉시 업그레이드/다운그레이드하며, `/settings/billing`에서 결제 상태를 볼 수 있는 스트라이프(Stripe) 구독 관리 기능을 구현하라.
    > **성공 기준:** 무료 사용자가 프로로 구독하고, 스트라이프 대시보드에서 결제를 확인하며, 5초 이내에 유료 기능에 접근할 수 있어야 함.

차이는 단순한 디테일이 아니라 '테스트 가능성'이다. 클로드 코드가 작업을 마치면 1분 이내에 목표 달성 여부를 확인할 수 있다. 모호함이 없다. "음, 대충 되는 것 같네"라는 식의 불확실성이 사라진다.

이것만으로도 클로드 코드와의 주고받는 횟수가 절반으로 줄었다. AI가 결승점이 어디인지 알면 트랙을 돌아다니며 셀카를 찍는 짓을 멈춘다.

#### 구성 요소 2: 제약 사항(CONSTRAINTS) — 당신을 지켜주는 벽

이 부분은 특히 클로드 코드에 중요하다. 제약 사항이 없으면 클로드 코드는 당신의 스택을 완전히 재발명하려 들 것이기 때문이다. 주방 리모델링 업자를 불렀더니 "풍수가 안 좋아서 거실 벽도 허물었습니다"라고 말하는 꼴이다.

나는 이제 모든 프로젝트 루트에 영구적인 제약 계층 역할을 하는 `CLAUDE.md` 파일을 둔다.

```markdown
# CLAUDE.md — 프로젝트 제약 사항 (항상 활성)
## 스택 (협상 불가, 어길 시 전체 롤백 예정)
- 프론트엔드: Next.js 14+ App Router, TypeScript strict 모드
- 백엔드: 실시간 데이터는 Convex, 인증 및 스토리지는 Supabase
- 인증: Clerk (직접 인증 시스템을 만들지 말 것)
- 스타일링: Tailwind 전용 — CSS 모듈이나 Styled-components 금지

## 엄격한 규칙
- 허락 없이 새로운 종속성(Dependency)을 설치하지 말 것
- 마이그레이션 계획을 보여주지 않고 데이터베이스 스키마를 수정하지 말 것
- 모든 API 호출은 Convex 함수를 통할 것 (컴포넌트에서 직접 Supabase 호출 금지)
- 환경 변수는 .env.local에 저장할 것 (하드코딩 금지)

## 패턴
- 기본적으로 서버 컴포넌트(Server components)를 사용하고, 인터랙션이 필요한 경우에만 클라이언트 컴포넌트 사용
- 모든 라우트 세그먼트에 에러 바운더리(Error boundaries) 적용
- 모든 사용자 입력에 Zod 유효성 검사 적용
```

`CLAUDE.md`가 없었을 때는 클로드 코드가 마음대로 Convex 대신 Prisma를 쓰기로 결정하거나, "더 대중적이라서" Clerk 대신 NextAuth로 바꾸곤 했다. 이제는 제 자리를 지킨다. 매번 말이다.

**프로 팁:** 새로운 클로드 코드 세션을 시작할 때 첫 메시지는 다음과 같아야 한다:
> "작업을 시작하기 전에 `CLAUDE.md`를 읽고 프로젝트 제약 사항을 이해했는지 확인해 줘."

#### 구성 요소 3: 형식(FORMAT) — 무엇을 건네줄지 정확히 지시하라

바이브 코딩은 클로드 코드가 구조를 결정하게 내버려 둔다. 이는 셰프에게 "나를 놀래켜줘"라고 말하고 신발 안에 담긴 해체주의 시저 샐러드를 받고 충격받는 것과 같다.

*   **변경 전:** "사용자 온보딩을 위한 API 엔드포인트를 만들어줘."
*   **변경 후 (계약):**
    > **FORMAT:**
    > 1. `convex/users.ts`에 Convex 함수 작성 (action이 아닌 mutation)
    > 2. `convex/schemas/onboarding.ts`에 입력 검증을 위한 Zod 스키마 작성
    > 3. `convex/types/user.ts`에서 TypeScript 타입 내보내기
    > 4. 공개 함수에 JSDoc 포함
    > 5. `{ success: boolean, userId: string, error?: string }` 형태로 반환

이제 매번 모듈화되고, 타입이 지정되고, 문서화된 코드를 얻는다. 클로드 코드가 스스로 이렇게 못 해서가 아니라, 명시적인 지시가 없으면 유지보수성이 아닌 '속도'에 최적화하기 때문이다.

#### 구성 요소 4: 실패 조건(FAILURE CONDITIONS) — 비밀 병기

이것이 모든 것을 바꾼 핵심 요소다. 목표가 '당근'이라면, 실패 조건은 '채찍'이다.

실패 조건을 정의하면 클로드 코드에게 '피해야 할 타겟'을 주게 된다. "앉아"라고 훈련시키는 것도 좋지만, "소파 위는 안 돼, 식탁 위도 안 돼"라고 가르치는 것이 가구를 보호하는 길이다.

내가 지난주에 사용한 실제 프롬프트 계약 예시다:

> `/dashboard` 페이지를 구축하라.
>
> **GOAL:** 실시간 업데이트로 사용자의 활성 프로젝트를 표시할 것. 첫 의미 있는 페인트(FMP)는 1초 미만이어야 함. 사용자는 인라인으로 프로젝트를 생성, 보관, 이름 변경할 수 있어야 함.
>
> **CONSTRAINTS:** 데이터에는 Convex `useQuery` 사용, 폴링(Polling)이나 SWR 금지. 인증 확인에는 Clerk `useUser()` 사용. 인증되지 않은 경우 `/sign-in`으로 리다이렉트. 컴포넌트 파일당 최대 150줄.
>
> **FORMAT:** `app/dashboard/page.tsx`(서버 컴포넌트 래퍼), `components/dashboard/ProjectList.tsx`(클라이언트 컴포넌트), `convex/projects.ts`(Convex 쿼리). Tailwind만 사용.
>
> **FAILURE CONDITIONS:**
> - Convex에 있어야 할 데이터에 `useState`를 사용하는 경우
> - 컴포넌트가 150줄을 초과하는 경우
> - 서버사이드에서 할 수 있는 데이터 페칭(Fetching)을 클라이언트사이드에서 하는 경우
> - Tailwind 유틸리티 클래스 외의 UI 라이브러리를 사용하는 경우
> - 로딩(Loading) 및 에러 상태가 누락된 경우
> - 함수 파라미터에 TypeScript 타입이 누락된 경우

결과는 완벽했다. 단 한 번의 시도에 깔끔하고 실시간으로 작동하는 대시보드가 만들어졌다. 실패 조건은 가드레일 역할을 한다. "좋은 코드"가 무엇인지 고민할 필요가 없다. 이미 무엇이 "나쁜 코드"인지 알려줬기 때문이다.

### 결과 (3주간의 프롬프트 계약 적용 후)

두 개의 프로젝트를 진행하며 작업 흐름을 추적해 본 결과, 패턴은 명확했다.

1.  **되돌리기(Undo/Revert) 비율 감소:** 생성물 3개 중 1개를 되돌리던 것에서 10개 중 1개 꼴로 줄었다.
2.  **커뮤니케이션 횟수 감소:** 프롬프트 입력 후 사용 가능한 코드가 나오기까지 평균 3회 정도 주고받던 것이 1.2회로 줄었다. 대부분 첫 번째나 두 번째 시도에 바로 사용 가능했다.
3.  **규칙 위반 제로:** `CLAUDE.md` 위반 사항(잘못된 라이브러리 사용 등)이 세션 시작 시 제약 사항 확인 절차를 거친 이후 사실상 제로가 되었다.

가장 놀라운 점은 시간이 지날수록 프롬프트가 길어지는 게 아니라 오히려 짧아졌다는 것이다. `CLAUDE.md`가 영구적인 제약 사항을 담고 있고 4개 구성 요소 구조가 익숙해지면, 복잡한 기능에 대한 프롬프트 계약을 작성하는 데 60초면 충분하다. 이 60초가 45분간의 디버깅과 커리어에 대한 회의감을 아껴준다.

### 오늘 바로 시작하는 방법

작업 방식을 완전히 갈아엎을 필요는 없다.

1.  **첫째,** 프로젝트 루트에 `CLAUDE.md`를 만들어 당신의 스택, 엄격한 규칙, 패턴을 적어라. 모든 클로드 코드 세션은 이 파일을 읽는 것으로 시작해야 한다.
2.  **둘째,** 다음에 "사용자 설정 페이지 추가해줘" 같은 바이브 프롬프트를 입력하려 할 때 잠시 멈춰라. 30초만 더 투자해서 **목표(GOAL)** 하나, **제약 사항(CONSTRAINT)** 하나, **실패 조건(FAILURE CONDITION)** 하나를 추가해라. 각각 하나씩만이라도 좋다.

프롬프트 계약 프레임워크는 더 많이 쓰기 위한 것이 아니다. 클로드 코드가 60분 동안 추측하게 두는 대신, 당신이 60초 동안 생각하기 위한 것이다.

나는 도박을 멈추고 제품을 출시하기 시작했다. 이제 당신 차례다.
