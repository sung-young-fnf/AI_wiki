미세 조정 없이 스스로 학습하는 에이전트 구축하기

### 1. 서론

LLM(Large Language Model)은 우리가 콘텐츠를 소비하고 생성하는 방식을 변화시켰습니다. 더 이상 구글에서 몇 시간 동안 검색하고 기사를 읽으며 정보를 수집하는 소위 '리서치'를 할 필요가 없습니다. 요즘은 질문이 있다면 LLM에 묻기만 하면 됩니다. LLM 분야의 치열한 경쟁 덕분에 이 모델들의 품질과 신뢰성은 크게 향상되었습니다. 소프트웨어 엔지니어에게 LLM은 진정한 판도를 바꾸는 요소였습니다. 질문 답변 외에도 LLM은 코드를 생성할 수 있으며, 경우에 따라 전체 애플리케이션을 생성하기도 합니다.

하지만 가장 간과하기 쉬운 장점은 다음과 같습니다. LLM은 데이터 과학 배경이 없어도 머신러닝(ML) 애플리케이션을 쉽게 구축할 수 있게 해줍니다. 애플리케이션의 복잡성에 따라 다양한 추론 모델을 사용하여 몇 줄의 코드로 예측, 분류 및 추천을 수행하는 에이전트(agentic) 애플리케이션을 만들 수 있습니다. 저처럼 ML 엔지니어가 아닌 사람에게 이러한 기능은 전통적인 경계를 넘어 고객에게 진정으로 유용하고 영향력 있는 애플리케이션을 구축할 수 있도록 해줍니다.

대부분의 경우 기본 제공(out-of-box) 모델이 대부분의 문제를 해결하지만, 실제 애플리케이션에서는 종종 도메인별 정확도가 필요합니다. 많은 개발자는 이를 위해 전문 지식이 필요한 비용이 많이 드는 프로세스인 미세 조정(fine-tuning)이 필요하다고 생각합니다. 하지만 그렇게 어려울 필요는 없습니다. 대부분의 경우, 업데이트된 시스템 프롬프트(system prompt)를 사용하거나 모델에 새로운 도구 세트(tool set)를 노출시켜 미세 조정 없이도 응답 품질을 향상시킬 수 있습니다.

미세 조정 없이 스스로 학습하는 에이전트를 구축하는 더 현명한 방법을 찾고 있다면 이 블로그가 적합합니다. 이 블로그에서는 모델을 미세 조정할 필요 없이 스스로 학습하는 에이전트를 구축하는 데 도움이 되는 두 가지 패턴을 공유할 것입니다.

### 2. Strands, AWS 에이전트 프레임워크

이 블로그에서는 AWS의 에이전트 프레임워크인 Strands를 사용하여 스스로 학습하는 에이전트를 구축할 것입니다. Strands는 몇 줄의 코드로 에이전트 애플리케이션을 만들 수 있는 경량 프레임워크입니다. AWS SDK를 기반으로 구축되어 에이전트를 빠르고 효율적으로 개발하기 위한 다양한 도구와 API를 제공합니다.

가장 흔한 질문 중 하나는 "왜 Strands인가? LangChain이나 다른 프레임워크는 왜 아닌가?"입니다. 답은 간단합니다. Strands를 사용하면 말 그대로 네 줄 미만의 코드로 에이전트를 구축할 수 있습니다. AWS가 제가 선호하는 클라우드 제공업체(cloud provider)이기 때문에 Strands는 Agent Core, Bedrock 등과 같은 AWS 서비스와의 원활한 통합도 제공합니다. 이러한 긴밀한 통합은 시작하는 데 필요한 상용구(boilerplate)의 양을 크게 줄여줍니다.

Strands는 re:Invent 2025에서 여러 흥미로운 발표와 강연을 진행했으며, 이 블로그는 이러한 업데이트와 논의에서 영감을 받았습니다.

### 3. [패턴-1]: 동적 툴링(Dynamic Tooling)을 통한 자체 개선 에이전트

#### 3.1. 문제점

검색, 채팅, 요약 등 다양한 작업을 처리할 수 있는 범용 에이전트를 구축한다고 상상해 보십시오. 일반적으로 이러한 에이전트는 고정되고 제한적인 도구 세트와 함께 제공됩니다. 시간이 지남에 따라 사용자가 에이전트와 상호 작용하는 방식을 관찰하고, 좋아요/싫어요, 댓글 또는 평점과 같은 피드백 신호를 기반으로 도구 세트를 개선합니다.

이 접근 방식의 문제는 피드백 루프(feedback loop)가 느리고 사람의 개입에 크게 의존한다는 것입니다. 팀은 피드백을 수집하고 분석한 다음 수동으로 새롭거나 개선된 도구를 설계하고 배포해야 합니다. 그 동안 사용자는 최적이 아닌 응답을 계속 받고, 개선 사항을 확인하려면 다음 반복(iteration)을 기다려야 합니다. 이는 종종 좌절감과 나쁜 사용자 경험으로 이어집니다.

#### 3.2. 해결책

이것이 동적 툴링이 강력해지는 지점입니다. 에이전트를 미리 정의된 도구 세트에 제한하는 대신, LLM이 주어진 작업에 기존 도구를 사용할 수 있는지 여부를 결정하도록 허용합니다. 만약 그러한 도구가 없다면, 에이전트는 즉석에서 새로운 도구를 생성하여 도구 세트에 동적으로 추가할 수 있으며, 이 모든 과정은 사람의 개입 없이 이루어집니다.

이러한 접근 방식을 통해 에이전트는 사용자 쿼리(query)를 기반으로 기능을 지속적으로 개선합니다. 이는 타당한 보안 문제를 야기하지만, 도구 실행에 대한 권한을 강화하거나 새로 생성된 도구를 샌드박스(sandboxed) 환경에서 실행함으로써 완화될 수 있습니다.

### 4. [패턴-2]: 동적 시스템 프롬프트(Dynamic System Prompt)를 통한 자체 개선 에이전트

#### 4.1. 문제점

시스템 프롬프트는 일반적으로 정적이며, 에이전트가 생성될 때 정의되고 일반적으로 변경되지 않습니다. 여기에는 타당한 이유가 있습니다. 예를 들어, 시스템 프롬프트를 수정하려면 에이전트가 예상치 못하거나 안전하지 않은 방식으로 동작하지 않도록 광범위한 테스트와 검증이 필요합니다.

하지만 응답 품질을 향상시키기 위해 시스템 프롬프트를 발전시켜야 하는 시나리오도 있습니다. 풀 리퀘스트(PR)를 검토하고 PR 워크플로우(workflow)의 일부로 피드백을 제공하는 코드 리뷰 에이전트를 생각해 봅시다. 대부분의 PR에서 에이전트는 잘 작동합니다. 하지만 특히 복잡한 코드 변경의 경우, 더 깊이 있고 미묘한 피드백을 제공하기 위해 여전히 사람이 검토해야 합니다.

시니어 엔지니어의 검토는 종종 코드 자체를 넘어 역사적 맥락, 팀별 코딩 가이드라인, 아키텍처 의도, 확립된 모범 사례 등을 통합합니다. 이상적으로는 에이전트가 이러한 피드백으로부터 학습하여 반복적인 사람의 검토 필요성을 줄이기를 원합니다. 실제로는 팀이 바쁘기 때문에 새로운 통찰력을 반영하도록 시스템 프롬프트를 수동으로 업데이트하는 작업이 종종 우선순위에서 밀려나 프롬프트가 현재 기대치와 동기화되지 않는 경우가 발생합니다.

#### 4.2. 해결책

이것이 동적 시스템 프롬프트가 더욱 강력해지는 지점입니다. 시스템 프롬프트를 수동으로 편집하는 대신, 자동으로 발전시킬 수 있습니다. PR이 병합될 때(merged), LLM은 리뷰 댓글을 분석하고 시스템 프롬프트의 관련 섹션을 업데이트하여 에이전트가 미래에 더 높은 품질의 피드백을 생성하도록 할 수 있습니다.

이러한 접근 방식은 에이전트가 사람의 개입을 최소화하면서 지속적으로 개선되도록 하며, 엔지니어는 프롬프트 유지 관리보다는 더 영향력 있는 작업에 집중할 수 있도록 해줍니다.

참고: 이 패턴은 코드 리뷰 사용 사례에만 국한되지 않습니다. 시스템 프롬프트는 파일이나 환경 변수(environment variables)에 저장될 수 있으며, 의미 있는 변경 사항이 발생하면 이와 동일한 동적 업데이트 메커니즘을 트리거할 수 있습니다. 아래에 표시된 것처럼 시스템 프롬프트 내에서 에이전트에게 자체 지침을 언제 어떻게 발전시켜야 하는지 명시적으로 지시할 수도 있습니다.

### 5. 결론

LLM과 에이전트 프레임워크는 최소한의 코드로 깊은 ML 전문 지식 없이도 강력한 애플리케이션을 구축할 수 있도록 합니다. 느린 사람의 피드백 루프(feedback loop)나 비용이 많이 드는 미세 조정(fine-tuning)에 의존하는 대신, 동적 툴링(dynamic tooling) 및 동적 시스템 프롬프트(dynamic system prompt)와 같은 패턴은 에이전트가 실시간으로 지속적으로 개선되도록 합니다.

Strands와 같은 프레임워크는 상용구(boilerplate)를 줄이고 AWS 서비스와 원활하게 통합함으로써 이러한 아이디어를 실용화합니다. 모델을 재훈련(retraining)하는 대신 프롬프트와 도구를 발전시키는 데 집중함으로써, 우리는 사용량으로부터 학습하고, 더 높은 품질의 응답을 제공하며, 지속적인 사람의 개입 없이도 효과적으로 확장할 수 있는 에이전트를 구축할 수 있습니다.
