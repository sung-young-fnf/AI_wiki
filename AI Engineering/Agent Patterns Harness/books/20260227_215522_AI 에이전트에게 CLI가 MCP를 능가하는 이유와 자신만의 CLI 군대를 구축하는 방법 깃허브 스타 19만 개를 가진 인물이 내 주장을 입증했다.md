# AI 에이전트에게 CLI가 MCP를 능가하는 이유와 자신만의 CLI 군대를 구축하는 방법: 깃허브 스타 19만 개를 가진 인물이 내 주장을 입증했다.

"mcp는 실수였다. bash가 낫다."
필 | 렌티어 디지털 자동화 (Phil | Rentier Digital Automation)
12분 분량의 읽을거리 · 2026년 2월 17일

---

여섯 단어. 지난달, 오픈클로(OpenClaw)의 개발자이자 깃허브(GitHub) 스타 19만 개를 보유하며 샘 알트만(Sam Altman)에게 갓 스카우트된 피터 스타인버거(Peter Steinberger)가 X에 올린 글입니다. 그리고 저의 즉각적인 반응은 그 글을 스크린샷 찍어 세 명의 개발자 친구에게 '내가 말했잖아!'라고 대문자로 써서 보내는 것이었습니다.

저는 수년간 우분투(Ubuntu)에서 개발해왔습니다.

제가 매일 사용하는 모든 도구는 CLI(명령줄 인터페이스)입니다.

Supabase CLI, Vercel CLI, Docker, git, n8n 등 제 전체 스택(stack)은 터미널(terminal)에서 실행됩니다. 지난해 MCP(모델 컨텍스트 프로토콜) 서버가 유행하기 시작했을 때, 몇 가지를 시도해봤습니다. 작동은 했습니다. 하지만 제 컨텍스트 창(context window)의 40%를 차지하고, 무작위로 충돌했으며, 한 줄 명령과 파이프(pipe)로 이미 처리할 수 있는 일에 불필요한 의존성(dependency)을 추가했습니다.

그래서 2026년 가장 뛰어난 오픈소스(open-source) 개발자가 CLI가 AI 에이전트와 현실 세계 간의 진정한 인터페이스(interface)라고 말하고, 오픈AI(OpenAI)가 동의하여 그를 고용할 정도라면, 이제 주목할 때가 된 것 같습니다.

요약: MCP 서버는 컨텍스트 창을 불필요하게 늘리고, 취약한 의존성을 추가하며, 도구가 CLI라면 애초에 존재하지 않을 문제를 해결하려 합니다 😂 피터 스타인버거는 오픈클로(OpenClaw)를 위해 약 10개의 커스텀(custom) CLI를 구축했고, 이 공로로 오픈AI(OpenAI)에 스카우트되었습니다.

오늘날 클로드 코드(Claude Code)를 사용해 동일한 패턴(pattern)을 적용하거나(CLAUDE.md 파일에 CLI를 문서화), 오픈클로(OpenClaw)에 스킬(skill)로 연결하거나, 앤트로픽(Anthropic)의 tool_use API를 활용하여 자신만의 자율 에이전트(autonomous agent)를 구축할 수 있습니다. CLI는 AI 에이전트와 현실 세계 간의 기본(native) 인터페이스입니다. GUI(그래픽 사용자 인터페이스)는 인간을 위한 것이고, API(애플리케이션 프로그래밍 인터페이스)는 서비스를 위한 것이며, CLI는 에이전트를 위한 것입니다.

## MCP에 반대하는 이유 (그리고 오픈클로 개발자가 내 터미널 방식에 동의하는 이유)

AI 에이전트가 외부 도구와 상호작용할 수 있는 방법을 최악에서 최고 순으로 나열해 봅시다.

GUI는 분명히 제외됩니다. CI/CD 파이프라인(pipeline)에 브라우저(browser)에서 버튼을 클릭하도록 요청하지 않는 것처럼, 동일한 논리가 에이전트에도 적용됩니다. 다음으로 넘어갑니다.

REST API와 SDK(소프트웨어 개발 키트)도 작동합니다. 하지만 모든 서비스는 고유한 인증 흐름(auth flow), 응답 형식(response format), 오류 처리(error handling) 방식을 가지고 있습니다. 결국 각 통합(integration)마다 래퍼 코드(wrapper code)를 작성하게 됩니다. 이는 SaaS(서비스형 소프트웨어) 백엔드(backend)에는 괜찮지만, 단순히 새 이메일이 있는지 확인해야 하는 에이전트에게는 과도한 작업입니다.

MCP(모델 컨텍스트 프로토콜)는 이러한 문제를 해결하기 위해 고안되었습니다. 에이전트를 도구에 연결하는 하나의 표준 프로토콜(protocol)이죠. 이론적으로는 훌륭하게 들립니다. 하지만 실제로는 어떨까요? 추가하는 모든 MCP 서버는 전체 스키마(schema)를 에이전트의 컨텍스트 창으로 쏟아냅니다. 도구 설명, 매개변수 목록, 기능 선언 등 이 모든 것이 말이죠. 에이전트가 실제 요청에 대해 생각하기 시작하기도 전에, 컨텍스트의 30~40%는 이미 MCP의 상투적인 내용(boilerplate)으로 소모됩니다.

피터 스타인버거는 이를 시도했고, 지원 기능도 구축했습니다. 그러고는 MCP 서버를 다시 CLI로 변환하는 도구인 MCPorter를 만들었습니다. 그만큼 그는 이 형식(format)이 잘못되었다고 생각합니다.

MCP가 생태계에 실제로 기여한 바에 대한 그의 정확한 언급은 이렇습니다: "MCP의 유일한 장점은 기업들이 일부 API를 공개하도록 만들었다는 점입니다." 가혹하지만, 정확한 평가입니다.

프로토콜 자체는 우회로였을 뿐이며, 그것이 강제로 존재하게 만든 API들이 진정한 선물입니다.

CLI가 승리하는 이유는 그 모든 불필요한 비대함(bloat)의 정반대이기 때문입니다. CLI는 다음과 같습니다:

*   **컨텍스트 오버헤드(overhead)가 없습니다.** 에이전트는 스키마를 로드할 필요가 없습니다. 한 페이지 분량의 문서(또는 `--help` 실행)를 읽고 모든 사용 가능한 명령을 파악합니다.
*   **구성 가능(Composable)합니다.** 한 CLI의 출력을 다른 CLI로 파이프할 수 있습니다. 예를 들어 `goplaces search "coffee" --json | jq '.[0].address'`와 같이 말이죠. MCP 서버로 이런 작업을 시도해보세요.
*   **2초 만에 테스트 가능합니다.** 터미널을 열고, 명령을 실행하고, 결과를 확인하면 됩니다. 서버를 띄울 필요도, 프로토콜 핸드셰이크(handshake)도, 웹소켓(WebSocket) 연결도 필요 없습니다.
*   **무료로 구조화된 출력(structured output)을 제공합니다.** `--json` 플래그(flag)를 추가하면 에이전트는 직렬화 계층(serialization layer) 없이 파싱 가능한(parseable) 데이터를 얻을 수 있습니다.

단 한 번의 `exec` 호출. 에이전트가 CLI를 사용하는 데 필요한 전부입니다. 미들웨어(middleware)도, 프로토콜도, 백그라운드(background)에서 RAM을 잡아먹으며 대기하는 서버 프로세스(server process)도 필요 없습니다.

---

그리고 이것은 단순히 이론적인 선호가 아닙니다.

스타인버거는 그의 전체 오픈클로(OpenClaw) 생태계(ecosystem)를 CLI를 중심으로 구축했습니다. 약 12개의 CLI를 만들었는데, 구글 맵스(Google Maps)용 `goplaces`, 아이메시지(iMessage)용 `imsg`, X/트위터(X/Twitter)용 `bird`, 왓츠앱(WhatsApp)용 `wacli`, 지메일(Gmail) 및 캘린더(Calendar)용 `gog`, 보안 카메라용 `camsnap`, AI 비전(vision)을 활용한 macOS 스크린샷(screenshot)용 `peekaboo`, 그리고 비디오(video)와 팟캐스트(podcast) 요약용 `summarize` 등이 있습니다. 각각은 '하나의 일을 잘 수행하고, `--json`을 지원하며, 명확한 `--help`를 제공한다'는 동일한 패턴을 따릅니다.

그리고 오픈AI(OpenAI)가 그를 고용했습니다. 그는 거의 1년 가까이 이 CLI 군대를 구축하는 데 시간을 보냈습니다. 샘 알트만(Sam Altman)은 예쁜 대시보드(dashboard)를 만든 사람을 스카우트한 것이 아니라, Bash가 최고의 에이전트 인터페이스임을 증명한 사람을 스카우트한 것입니다. 이것이 의미하는 바는 각자 판단에 맡기겠습니다.

## 지금 바로 CLI를 활용하여 더 빠르게 구축하기 (오픈클로 불필요)

이러한 이점을 누리기 위해 오픈클로(OpenClaw)가 필요하지 않습니다. 클로드 코드(Claude Code), 코덱스(Codex) 또는 셸(shell) 접근 권한이 있는 어떤 에이전트를 사용하든, 이미 인프라(infrastructure)를 갖추고 있습니다.

대부분의 사람이 놓치는 비법은 이것입니다. 에이전트는 이미 CLI를 호출할 수 있습니다. 하지만 여러분이 알려주지 않으면 여러분의 특정 CLI에 대해서는 알지 못합니다.

```markdown
# CLAUDE.md (root of your project)
# 프로젝트 루트(root)의 CLAUDE.md

## Available CLIs
## 사용 가능한 CLI

### Supabase
### 수파베이스
- `supabase db push` - 원격에 마이그레이션(migration) 적용
- `supabase functions deploy <name>` - 엣지 함수(edge function) 배포
- `supabase db dump --data-only` - 프로덕션(production) 데이터(data) 내보내기
- `supabase migration new <name>` - 새 마이그레이션 파일(file) 생성

### Vercel
### 버셀
- `vercel deploy --prod` - 프로덕션에 배포
- `vercel env pull .env.local` - 환경 변수(env variables) 동기화
- `vercel logs <url> --follow` - 프로덕션 로그(log) 실시간 확인

### 프로젝트(project) 특정
- `./scripts/check-mrr.sh` - 현재 MRR(월간 반복 수익), 신규 가입자, 이탈률을 JSON으로 출력
- `./scripts/seed-demo.sh` - 샘플 데이터로 데모 환경(environment) 초기화
```

그게 전부입니다.

클로드 코드(Claude Code)는 매 세션(session) 시작 시 CLAUDE.md 파일을 읽습니다. 다음번에 '프로덕션에 배포하고 MRR이 변경되었는지 확인해 줘'라고 말하면, 어떤 명령어를 실행해야 할지 정확히 알게 됩니다. 플러그인(plugin)도, MCP 서버도, 47개의 중첩된 키(key)를 가진 설정 파일(config file)도 필요 없습니다.

코덱스(Codex)의 경우에도 동일한 개념이며, 파일 이름은 AGENTS.md입니다.

커서(Cursor)의 경우 .cursorrules입니다.

파일 이름만 다를 뿐, 패턴은 동일합니다.

하지만 진정한 파워 플레이(power move)는 자신만의 CLI를 구축하는 것입니다. 'CLI 도구를 만들 시간이 없어'라고 생각하며 이 탭을 닫기 전에 말씀드리자면, 우리는 고작 20~30줄의 코드에 대해 이야기하는 것입니다. 정말입니다.

### 에이전트 친화적인 CLI를 위한 세 가지 규칙:

#### 1. `--json`을 이용한 구조화된 출력

에이전트는 박스 그리기 문자(box-drawing characters)로 된 예쁜 테이블을 파싱(parsing)할 수 없습니다. JSON이 필요합니다.

```javascript
#!/usr/bin/env node
// scripts/check-mrr.js
// scripts/check-mrr.js
import { createClient } from '@supabase/supabase-js'

const supabase = createClient(process.env.SUPABASE_URL, process.env.SUPABASE_KEY)
const args = process.argv.slice(2)
const jsonMode = args.includes('--json')
const { data } = await supabase
  .from('subscriptions')
  .select('plan, status, created_at')

const active = data.filter(s => s.status === 'active')
const mrr = active.reduce((sum, s) => sum + (s.plan === 'pro' ? 29 : 9), 0)
const today = data.filter(s =>
  new Date(s.created_at).toDateString() === new Date().toDateString()
)
const stats = {
  mrr,
  active_subscriptions: active.length,
  signups_today: today.length,
  timestamp: new Date().toISOString()
}
if (jsonMode) {
  console.log(JSON.stringify(stats))
} else {
  console.log(`MRR: $${mrr}`)
  console.log(`Active: ${active.length}`)
  console.log(`Signups today: ${today.length}`)
}
```

#### 2. 실제 내용을 설명하는 `--help`

에이전트는 사람이 README 파일을 읽듯이 `--help`를 읽습니다. 도움말 텍스트(help text)가 엉망이라면, 에이전트는 플래그(flag)를 환각(hallucinate)할 것입니다.

```bash
$ ./check-mrr.js --help
$ ./check-mrr.js --help
Usage: check-mrr [options]
사용법: check-mrr [options]

Check current SaaS metrics from Supabase.
Supabase에서 현재 SaaS 지표를 확인합니다.
Options:
  --json     Output as JSON (default: human-readable)
  --json     JSON으로 출력 (기본값: 사람이 읽을 수 있는 형식)
  --period   Filter: today | week | month (default: today)
  --period   필터: 오늘 | 주 | 월 (기본값: 오늘)
  --help     Show this message
  --help     이 메시지 표시
```

#### 3. 깔끔한 종료 코드(exit codes)

0 = 성공. 1 = 오류.

에이전트는 이를 사용하여 다음에 무엇을 할지 결정합니다. CLI가 실패했는데도 0으로 종료되면, 에이전트는 모든 것이 잘 진행되었다고 생각하고 다음 단계로 넘어갑니다.

저는 이 사실을 새벽 2시에 배포 스크립트(deploy script)가 조용히 실패하고 있는데도 클로드(Claude)가 계속 '배포 성공'이라고 말해줘서 힘들게 배웠습니다. 스크립트가 오류를 삼키고(swallowing errors) 어쨌든 0으로 종료된다는 것을 깨닫는 데 20분이나 걸렸죠. 이야기가 옆으로 샜네요.

이러한 CLI 몇 개를 갖추게 되면, 무언가 달라집니다.

더 이상 클로드 코드(Claude Code)에게 Supabase 쿼리(query)를 작성해달라고 요청하지 않게 됩니다. 대신 '내 지표를 확인하고, 어제보다 신규 가입자 수가 20% 이상 감소했다면, 팀에게 슬랙(Slack) 메시지를 초안해 줘'라고 말하게 됩니다. 클로드(Claude)는 CLI들을 연결하고, 로직(logic)을 결정하며, 결과에 따라 행동합니다. 이는 자동 완성(autocomplete)이 아닙니다. 이것이야말로 에이전트입니다.

## 확장성을 부여하는 패턴: CLI + 스킬 문서

그래서 스타인버거가 일찍 깨달았지만 대부분의 사람들이 아직 내재화하지 못한 것이 있습니다. 바로 문서화되지 않은 CLI는 에이전트에게 쓸모가 없다는 것입니다.

에이전트는 인간처럼 시행착오(trial and error)를 통해 CLI를 탐색할 수 없습니다. 어떤 명령이 존재하고, 어떤 플래그를 사용할 수 있으며, 출력(output)이 어떻게 생겼는지 미리 알아야 합니다. 그렇기 때문에 오픈클로(OpenClaw) 생태계의 모든 CLI는 에이전트용 설명서 역할을 하는 구조화된 문서인 SKILL.md 파일과 함께 제공됩니다.

이 패턴은 다음과 같습니다: CLI 바이너리(binary) + 스킬 문서 = 자율 기능(autonomous capability).

CLI는 작업을 수행하고, 스킬 문서는 에이전트에게 사용 방법을 알려줍니다. 이 둘은 어떤 에이전트든 쉽게 가져다 사용할 수 있는 자율적인 단위(self-contained unit)입니다. 스타인버거는 이것들을 '스킬'이라고 부릅니다. 이를 스킬, 도구, 또는 '지난 화요일 데이브가 작성한 Bash 스크립트'라고 부르든 개념은 동일합니다.

그리고 이 패턴을 사용하기 위해 오픈클로(OpenClaw)가 필요하지 않습니다. 사실 여러분은 이미 그렇게 하고 있습니다. CLI를 문서화하는 CLAUDE.md를 작성할 때, 그것이 바로 스킬 문서입니다. 차이점은 스타인버거가 형식을 표준화하고 그 위에 배포 계층(distribution layer)을 구축했다는 것입니다. 바로 3,000개 이상의 스킬을 찾아보고 설치할 수 있는 클로허브(ClawHub)입니다.

흥미로운 점은 무엇일까요? 이러한 스킬 중 어떤 것이든 여러분의 설정(setup)에 가져다 사용할 수 있다는 것입니다. 클로허브(ClawHub)의 모든 스킬은 독립적으로 설치 가능한 CLI(예: `brew install steipete/tap/goplaces`, `npm install -g @steipete/oracle` 등)와 읽을 수 있는 SKILL.md 파일로 이루어져 있습니다. 오픈클로 런타임(runtime)은 필요 없습니다. 바이너리(binary)를 설치하고 관련 명령어를 CLAUDE.md에 붙여넣기만 하면 클로드 코드(Claude Code)가 즉시 사용할 수 있습니다.

```markdown
# In your CLAUDE.md — stolen straight from ClawHub
# CLAUDE.md 파일 내 — 클로허브(ClawHub)에서 직접 가져온 내용

## goplaces (Google Maps CLI)
## goplaces (구글 맵스 CLI)
- `goplaces search "coffee near me" --open-now --json` - 장소 찾기
- `goplaces search "pizza" --lat 40.8 --lng -73.9 --radius-m 3000 --json` - 위치 기반 검색
- `goplaces details <place_id> --json` - 리뷰(review)가 포함된 전체 장소 상세 정보
- `goplaces resolve "Soho, London" --json` - 장소 이름 지오코딩(geocode)
Requires: GOOGLE_PLACES_API_KEY 환경 변수

## summarize (Video/Podcast/Web summarizer CLI)
## summarize (비디오/팟캐스트/웹 요약 CLI)
- `summarize --url "https://youtube.com/watch?v=xxx" --json` - 비디오 요약
- `summarize --url "https://some-blog.com/post" --json` - 웹페이지(web page) 요약
- `summarize --url "https://podcast.fm/ep42" --cli claude --json` - 사용할 모델 선택
```

이것이 바로 `goplaces`와 `summarize` — 스타인버거 자신의 도구 두 가지 — 가 오픈클로(OpenClaw) 의존성(dependency) 없이 클로드 코드(Claude Code) 내에서 실행되는 모습입니다. 단지 바이너리와 문서만 있으면 됩니다.

이것이 CLI 접근 방식이 MCP가 결코 할 수 없는 방식으로 확장되는 이유입니다. MCP 서버는 구성(configuration), 프로토콜 핸드셰이크, 그리고 컨텍스트 창 공간을 필요로 하는 실행 중인 프로세스(process)입니다. CLI 스킬은 정적 바이너리(static binary)와 텍스트 파일(text file)입니다. 하나는 인프라를 필요로 하고, 다른 하나는 `brew install`과 10줄의 마크다운(Markdown)만 있으면 됩니다.

## 오픈클로(OpenClaw)에 CLI 연결하기

이미 오픈클로(OpenClaw)를 실행 중이라면, CLI를 에이전트 스킬로 전환하는 데 약 5분이 소요됩니다.

시스템은 다음과 같이 작동합니다. 모든 스킬에는 CLI가 무엇을 하는지, 어떻게 설치하는지, 그리고 어떤 명령을 사용할 수 있는지 설명하는 SKILL.md 파일이 있습니다. 에이전트는 이 파일을 읽고 도구 사용 방법을 파악합니다.

```yaml
---
name: check-mrr
description: Check SaaS metrics (MRR, signups, churn) from Supabase.
metadata:
  openclaw:
    requires:
      env:
        - SUPABASE_URL
        - SUPABASE_KEY
      bins:
        - node
    primaryEnv: SUPABASE_URL
---

# check-mrr
# check-mrr
프로덕션 Supabase에서 현재 SaaS 지표를 가져옵니다.
## Install
## 설치
npm install -g @yourhandle/check-mrr
## Commands
## 명령어
- `check-mrr --json` - 전체 지표를 JSON으로 표시
- `check-mrr --period week` - 이번 주 지표
- `check-mrr --period month` - 월별 개요
## Output format (--json)
## 출력 형식 (--json)
{
  "mrr": 1247,
  "active_subscriptions": 89,
  "signups_today": 3,
  "timestamp": "2026-02-17T10:30:00Z"
}
```

`clawhub publish` 명령으로 클로허브(ClawHub)에 게시하면 오픈클로(OpenClaw)를 실행하는 누구나 여러분의 스킬을 설치할 수 있습니다. 하지만 진정한 가치는 로컬(local)에 있습니다. 이를 크론 작업(cron job)과 결합하면, 에이전트가 매일 아침 지표를 확인하고 뭔가 이상하면 왓츠앱(WhatsApp)으로 알려줍니다.

```json
// In openclaw.json
// openclaw.json 파일 내
{
  "cron": [
    {
      "schedule": "0 8 * * *",
      "message": "Run check-mrr --json. If signups_today is 0 or mrr dropped more than 5% from yesterday, alert me on WhatsApp with a summary. Otherwise just log it.",
      "channel": "whatsapp"
    }
  ]
}
```

이것이 전체 루프(loop)입니다. 크론(cron)이 에이전트를 트리거(trigger)하고, 에이전트는 스킬을 읽고, CLI를 호출하며, 결과를 해석하고, 무엇을 할지 결정합니다. 확인할 대시보드도 없고, 불필요한 알림(alert)으로 인한 알림 피로도 없습니다. 에이전트가 판단을 내립니다. 이는 스타인버거가 자신의 전체 설정(setup)에서 사용하는 것과 동일한 패턴입니다.

클로허브(ClawHub) 디렉토리(directory)에는 이미 3,000개 이상의 타사 스킬이 있으며, 대부분이 이 정확한 구조를 따릅니다. 위치 검색을 위한 `goplaces`, IMAP(Internet Message Access Protocol)을 통한 이메일(email)용 `himalaya`, X/트위터(X/Twitter)용 `bird` (😭), 스피커 제어용 `sonoscli` 등, 전체 군대와 같습니다. 여러분은 그것들을 설치하고, 에이전트는 그것들을 학습합니다. 끝입니다.

## 자신만의 에이전트 구축하기 (오픈클로 패턴, 오픈클로 없이)

그렇다면 오픈클로(OpenClaw)를 실행하고 싶지 않다면 어떨까요? 아마도 더 가볍고, 더 커스텀(custom)한 것을 원하거나, 아니면 처음부터 직접 만드는 것을 즐길 수도 있습니다. (이해합니다. 저는 모든 것을 직접 호스팅(self-host)합니다. 일종의 병이죠.)

핵심 패턴은 매우 간단합니다. `tool_use`와 함께 앤트로픽(Anthropic) API를 호출하고, 도구 호출을 CLI 실행에 매핑(map)하며, 에이전트가 완료될 때까지 루프(loop)를 도는 스크립트(script)입니다.

```javascript
import Anthropic from '@anthropic-ai/sdk' // 앤트로픽 SDK
import { execSync } from 'child_process' // 자식 프로세스 실행 모듈

const client = new Anthropic()
// 도구로 선언된 여러분의 CLI
const tools = [
  {
    name: "check_mrr",
    description: "Get current SaaS metrics (MRR, active subs, signups today)",
    input_schema: {
      type: "object",
      properties: {
        period: { type: "string", enum: ["today", "week", "month"], default: "today" }
      }
    }
  },
  {
    name: "deploy_production",
    description: "Deploy latest commit to Vercel production. Returns deploy URL.",
    input_schema: {
      type: "object",
      properties: {}
    }
  },
  {
    name: "send_slack",
    description: "Send a message to a Slack channel",
    input_schema: {
      type: "object",
      properties: {
        channel: { type: "string" },
        message: { type: "string" }
      },
      required: ["channel", "message"]
    }
  }
]
// 도구 이름을 CLI 명령에 매핑
function executeTool(name, input) {
  const commands = {
    check_mrr: `node ./scripts/check-mrr.js --json --period ${input.period || 'today'}`,
    deploy_production: `vercel deploy --prod --yes 2>&1`,
    send_slack: `curl -X POST -H 'Authorization: Bearer ${process.env.SLACK_TOKEN}' \
      -H 'Content-Type: application/json' \
      -d '{"channel":"${input.channel}","text":"${input.message}"}' \
      https://slack.com/api/chat.postMessage`
  }
  
  try {
    const result = execSync(commands[name], { encoding: 'utf-8', timeout: 30000 })
    return result
  } catch (err) {
    return JSON.stringify({ error: err.message, exitCode: err.status })
  }
}
// 에이전트 루프
async function runAgent(task) {
  let messages = [{ role: "user", content: task }]
  
  while (true) {
    const response = await client.messages.create({
      model: "claude-sonnet-4-5-20250514",
      max_tokens: 4096,
      system: "You are an autonomous agent. Use the available tools to complete tasks. Be concise in your reasoning.",
      tools,
      messages
    })
    // 클로드(Claude)가 대화를 마쳤다면, 종료
    if (response.stop_reason === "end_turn") {
      const text = response.content.find(b => b.type === 'text')
      return text?.text || 'Done.'
    }
    // 클로드(Claude)가 도구를 사용하려 한다면, 실행
    const toolBlocks = response.content.filter(b => b.type === 'tool_use')
    if (toolBlocks.length === 0) break
    messages.push({ role: "assistant", content: response.content })
    const toolResults = toolBlocks.map(block => ({
      type: "tool_result",
      tool_use_id: block.id,
      content: executeTool(block.name, block.input)
    }))
    messages.push({ role: "user", content: toolResults })
  }
}
// 실행
const result = await runAgent(
  "Check our MRR. If it's above $1000, deploy to production and notify #team on Slack with the metrics. If it's below, just send a warning to Slack."
)
console.log(result)
```

약 80줄. 이것이 여러분만의 미니 오픈클로(OpenClaw)입니다. 에이전트는 여러분이 부여한 작업에 따라 어떤 도구를 어떤 순서로 호출할지 결정합니다. 새로운 CLI를 추가하는 데는 30초밖에 걸리지 않습니다. 도구 정의를 추가하고, 명령 맵(commands map)에 한 줄을 추가하면 끝입니다.

자율적인 부분은 크론(cron)으로 감싸면 됩니다:

```bash
# crontab -e
# crontab -e
0 8 * * * cd /home/deploy/my-agent && node agent.js "아침 확인: 지표, 안정적이면 배포, 팀에 알림"
0 20 * * * cd /home/deploy/my-agent && node agent.js "하루 마감: 신규 가입자 요약, 이상 징후는 슬랙(Slack)에 알림"
```

또한 타이머(timer)가 있는 systemd 서비스(service)로 실행하거나, 서버(server)의 도커(Docker) 컨테이너(container)에 넣을 수도 있습니다. 결과는 동일하지만, 데브옵스(devops)의 방식만 다를 뿐입니다.

---

인프라(infrastructure)가 전혀 필요 없는 또 다른 옵션은 깃허브 액션즈(GitHub Actions)입니다. 러너(runner)에 CLI를 설치하고 앤트로픽(Anthropic) API를 호출하는 예약된 워크플로우(workflow)입니다:

```yaml
name: Daily Agent Run
이름: 일일 에이전트 실행
on:
  schedule:
    - cron: '0 8 * * *'

jobs:
  agent:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
      - run: npm install @anthropic-ai/sdk
      - run: node agent.js "Morning routine"
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
          SUPABASE_URL: ${{ secrets.SUPABASE_URL }}
          SUPABASE_KEY: ${{ secrets.SUPABASE_KEY }}
```

공개 저장소(public repos)에는 무료이며, 비공개 저장소(private ones)에는 월 2,000분까지 무료입니다. 30초 만에 실행되는 일일 에이전트(daily agent)에게는 나쁘지 않은 조건입니다.

그리고 n8n은요?

기술적으로는 `Execute Command` 노드(node)를 사용하거나 FastAPI 컨테이너(container)로 래핑(wrap)하여 n8n을 통해 CLI를 오케스트레이션(orchestrate)할 수 있습니다.

하지만 솔직히 말해서, 이 사용 사례에서는 스크립트(script) 접근 방식보다 더 많은 마찰(friction)이 발생합니다. n8n은 15단계의 시각적 워크플로우(visual workflows)와 복잡한 분기(branching)가 필요할 때 빛을 발하며, 'CLI를 호출하고 LLM이 결정하도록 하는' 경우에는 적합하지 않습니다.

n8n에서 커스텀 코드(custom code) 실행에 대해 더 깊이 알고 싶다면, 도커(Docker) + FastAPI 설정(setup)을 다루는 'n8n에서 파이썬(Python) 스크립트 호출 가이드'를 제가 작성했습니다.

## 이것이 여러분에게 의미하는 것

트렌드(trend)는 명확합니다. AI 에이전트 분야에서 가장 생산적인 개발자들은 MCP 서버를 쌓거나 프로토콜 어댑터(protocol adapter)를 구성하지 않습니다. 그들은 작고 날카로운 CLI를 작성하고 에이전트가 이를 호출하도록 합니다.

피터 스타인버거는 오픈클로(OpenClaw)를 통해 대규모로 이를 증명했습니다. 오픈AI(OpenAI)는 채용 제안으로 이를 입증했습니다. 그리고 여러분은 오늘 CLAUDE.md 파일과 20줄짜리 노드(Node) 스크립트로 시작할 수 있습니다.

스택(stack)은 중요하지 않습니다. 오픈클로(OpenClaw), 클로드 코드(Claude Code), 코덱스(Codex), 커스텀 에이전트 루프(custom agent loop) 등, 패턴은 동일합니다. 도구를 CLI로 감싸고, 에이전트를 위해 문서화하며, LLM(거대 언어 모델)이 오케스트레이션(orchestration)을 처리하도록 하세요.

여러분의 터미널은 내내 AI 인터페이스였습니다. 대부분의 사람들이 아직 깨닫지 못했을 뿐이죠.

이 내용이 이해되셨다면, 더 많은 실전 검증된(battle-tested) AI 자동화 콘텐츠를 위해 저를 팔로우(follow)해주세요. 다음으로는 제 SaaS(서비스형 소프트웨어)를 관리하는 완전한 자율 에이전트를 구축할 예정입니다. 배포, 모니터링, 지원 티켓(support tickets) 처리 등 모든 것을 CLI를 통해서만 할 것입니다. 대시보드도, GUI도 없습니다. 그저 랍스터(lobster)와 크론 작업(cron job)만 있을 뿐입니다. 🦞
