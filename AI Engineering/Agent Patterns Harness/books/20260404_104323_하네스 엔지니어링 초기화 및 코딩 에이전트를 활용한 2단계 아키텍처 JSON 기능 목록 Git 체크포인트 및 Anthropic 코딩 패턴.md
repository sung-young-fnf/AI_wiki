하네스 엔지니어링: 초기화 및 코딩 에이전트를 활용한 2단계 아키텍처, JSON 기능 목록, Git 체크포인트 및 Anthropic 코딩 패턴

### 컨텍스트 윈도우 경계: 에이전트 코딩의 난제

긴 시간이 소요되는 코딩 작업이 짧은 작업과 근본적으로 다른 점은 복잡성이나 지능의 문제가 아니라 바로 '메모리'입니다. 현재의 최첨단 모델들은 128K에서 200K 토큰 정도의 컨텍스트 윈도우(Context Window)를 제공하지만, 실제 코딩 세션에서는 파일 읽기, 명령 출력, 오류 메시지, 대화 내용 등이 누적되면서 이 공간이 금방 소모됩니다.

단기 작업에서는 컨텍스트 윈도우가 충분하지만, 전체 기능을 처음부터 끝까지 구현해야 하는 장기 작업에서는 수학적인 한계에 부딪힙니다. 에이전트는 기능 명세, 기존 코드베이스, 이미 작성한 코드, 발생한 오류, 그리고 현재 계획을 모두 한 번에 유지해야 합니다. Anthropic은 이를 모델 자체의 한계로 보기보다는 더 나은 '하네스(Harness, 에이전트 실행 환경)' 설계를 통해 관리해야 할 환경적 제약으로 정의했습니다.

### 에이전트 프로젝트를 망치는 두 가지 실패 패턴

Anthropic은 장기 작업에서 발생하는 두 가지 주요 실패 모드를 확인했습니다.

1.  **한꺼번에 모든 것을 구축하기 (Building Everything at Once):** 에이전트가 10개의 요구사항을 동시에 구현하려다 수많은 파일을 수정하고, 오류가 발생했을 때 원인을 찾지 못해 디버깅의 '조합 폭발'에 빠지는 경우입니다. 이는 토큰 예산의 대부분을 기능 구현이 아닌 디버깅에 소모하게 만듭니다.
2.  **조기 완료 (Premature Completion):** 에이전트가 예외 상황이나 오류 처리를 무시하고 기본 경로만 구현한 뒤 "완료"라고 선언하는 경우입니다. 코드는 컴파일되고 테스트는 통과할지 모르지만, 실제 환경에서는 쉽게 무너집니다. 이는 모델의 능력이 부족해서가 아니라 "테스트가 통과했으니 끝났다"라고 판단하는 인지 편향 때문입니다.

### 2단계 에이전트 아키텍처 (Two-Agent Architecture)

Anthropic의 하네스는 제품 명세를 태스크로 분해하는 **초기화 에이전트(Initializer Agent)**와 이를 하나씩 구현하는 **코딩 에이전트(Coding Agent)**로 역할을 분리합니다. 이는 두 개의 모델을 쓰는 것이 아니라, 서로 다른 컨텍스트와 책임을 가진 두 개의 세션을 운영하는 아키텍처적 접근입니다.

*   **초기화 에이전트:** 코딩 시작 전 계획과 명세에 집중합니다. 코드베이스를 분석하여 상세 기능 목록(Feature List)을 생성하고, "완료"의 기준이 되는 스켈레톤 테스트 파일을 작성합니다. 작업이 끝나면 이 에이전트의 컨텍스트는 폐기되어 메모리 낭비를 방지합니다.
*   **코딩 에이전트:** 초기화 에이전트의 출력을 바탕으로 기능을 점진적으로 구축합니다. 전체 기능을 다 이해할 필요 없이 현재 작업 중인 특정 태스크와 테스트, 코드 상태에만 깊게 집중하여 컨텍스트 효율을 높입니다.

### 인지적 닻 역할을 하는 기능 목록 (Feature List)

Anthropic은 단순한 텍스트가 아닌 구조화된 기계 판독 가능 형식(JSON)으로 기능 목록을 관리할 것을 권장합니다. 마크다운(Markdown) 체크리스트는 모호할 수 있지만, JSON은 각 기능의 ID, 상태, 요구사항, 관련 테스트 및 파일을 명확하게 정의합니다.

이 JSON 기능 목록은 컨텍스트 윈도우가 리셋되어도 유지되는 **외부 메모리** 역할을 합니다. 새로운 세션이 시작될 때 에이전트는 이 파일을 읽고 즉시 자신이 어디에 있는지, 무엇을 해야 하는지 파악할 수 있습니다.

### 불가침 테스트 (Inviolable Tests)

하네스 설계의 핵심 제약 중 하나는 에이전트가 테스트를 통과시키기 위해 테스트 코드 자체를 수정하는 것을 금지하는 것입니다. 초기화 단계에서 정의된 테스트는 예외 케이스와 통합 지점을 모두 포함하므로, 에이전트는 이를 수정하지 못하는 상태에서 실제 요구사항을 충족해야만 작업을 마무리할 수 있습니다. 이는 조기 완료 문제를 방지하는 강력한 장치가 됩니다.

### 점진적 진행: 세션 종료와 커밋 (Git Commits)

Anthropic의 패턴에 따르면, 모든 코딩 세션은 테스트를 통과한 작은 단위의 커밋(Commit)으로 종료되어야 합니다. 이는 단순히 작업을 저장하는 것이 아니라 다음과 같은 효과를 가집니다.

*   **작고 테스트 가능한 증분:** 에이전트가 한 세션 내에서 끝낼 수 있는 범위로 작업을 나누게 유도합니다.
*   **상태 재개 가능:** 새 세션에서 `git log`와 `git diff`를 통해 이전 작업을 정확히 파악할 수 있습니다.
*   **롤백 안전성:** 세션이 꼬일 경우 마지막으로 성공한 커밋으로 쉽게 되돌릴 수 있습니다.

### 브라우저 자동화를 통한 검증

웹 애플리케이션의 경우, Playwright 같은 도구를 활용한 브라우저 자동화(Browser Automation)를 결합합니다. 유닛 테스트만으로는 잡기 어려운 UI 상호작용이나 통합 단계의 버그를 실제 브라우저 환경에서 검증함으로써 생산 레벨의 신뢰도를 확보합니다.

### 표준화된 시작 시퀀스 (Standardized Startup Sequences)

각 세션이 시작될 때 에이전트가 컨텍스트를 파악하느라 토큰을 낭비하는 것을 막기 위해, Anthropic은 표준화된 명령 스크립트를 사용합니다. 기능 목록 읽기, 최근 커밋 확인, 테스트 실행, 관련 파일 로드 등의 과정을 자동화함으로써 토큰 소모를 20~30%에서 5~10% 수준으로 줄이고 작업 효율을 극대화합니다.

### 결론: 프롬프트 엔지니어링이 아닌 하네스 엔지니어링

프롬프트 엔지니어링이 모델에게 "점진적으로 일해라"라고 부탁하는 것이라면, **하네스 엔지니어링(Harness Engineering)**은 모델이 점진적으로 일할 수밖에 없도록 구조적 제약(커밋 강제, 불가침 테스트, 외부화된 상태 관리)을 만드는 것입니다.

긴 시간이 소요되는 복잡한 작업에서 에이전트의 성능을 결정짓는 것은 모델의 지능뿐만 아니라, 그 모델이 작동하는 환경과 프로토콜인 '하네스'의 설계 능력입니다.
