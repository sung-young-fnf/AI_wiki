# Anthropic의 매니지드 에이전트 출시와 에이전트 하네스(Harness)의 5계층 모델

에이전트 개발의 공식은 명확합니다. **에이전트(Agent) = 모델(Model) + 하네스(Harness)**.

최근 Anthropic이 매니지드 에이전트(Managed Agent)를 위한 하네스 계층 중 3가지를 출시하면서, 에이전트 하네스 엔지니어링(Harness Engineering)은 이제 피할 수 없는 과제가 되었습니다. 하지만 현재 '하네스'라는 용어는 팀마다 서로 다른 의미로 사용되고 있습니다. 어떤 팀은 단순한 프로젝트 규칙 파일을 가진 것을 하네스 엔지니어링이라 부르고, 어떤 팀은 정교한 아키텍처 제약, 컨텍스트 파이프라인, 서브 에이전트 오케스트레이션(Orchestration)을 갖춘 시스템을 의미합니다.

이러리한 혼란을 해결하기 위해, 본 글에서는 하네스를 5개의 계층으로 분해하는 모델을 제안합니다. 이를 통해 무엇이 부족한지 진단하고, 접근 방식을 비교하며, 어디에 집중적으로 투자해야 할지 명확한 기준을 제시하고자 합니다.

---

## 하네스(Harness)의 5계층 모델

하네스를 5개의 독립적인 계층으로 분해하면 각 계층은 고유한 책임, 도구, 변화 속도 및 담당 조직을 갖게 됩니다.

### 1계층: 제약 계층 (Constraint Layer) — "뼈대"
제약 계층은 코드가 가질 수 있는 구조적 규칙을 강제합니다. 이 계층은 완전히 결정론적(Deterministic)이며, LLM이 개입하지 않습니다. 에이전트가 넘지 말아야 할 경계가 어디인지를 정의합니다.
* **주요 요소:** 커스텀 린터(Linter) 규칙, 의존성 방향 및 모듈 경계를 강제하는 구조적 테스트 프레임워크(예: ArchUnit), API 계약 검증(OpenAPI 스키마) 등.
* **특징:** 변화 속도가 느리며, 아키텍처 팀이나 시니어 엔지니어가 담당합니다. 토큰 소모 없이 오류를 방지할 수 있어 가장 높은 레버리지를 가집니다.

### 2계층: 컨텍스트 계층 (Context Layer) — "기억"
컨텍스트 계층은 실행의 각 단계에서 모델이 무엇을 볼 수 있는지 제어합니다. 에이전트가 현재 무엇을 알고 있는지, 그 지식이 어떻게 선택되었는지를 결정합니다.
* **주요 요소:** `CLAUDE.md`와 같은 시스템 프롬프트 주입 파일, 점진적 공개(Progressive Disclosure) 기술, 진행 상황 기록(Progress file) 등.
* **특징:** 변화 속도가 중간 정도이며, 실제 코드를 작성하는 개발 팀이 담당합니다. 핵심은 '방대함'이 아니라 '정확하고 최신화된 정보'를 제공하는 것입니다.

### 3계층: 실행 계층 (Execution Layer) — "손"
실행 계층은 에이전트가 무엇을 할 수 있고 어떻게 수행하는지를 관리합니다.
* **주요 요소:** 도구 오케스트레이션(Tool Orchestration), MCP(Model Context Protocol) 서버 설정, 서브 에이전트 배정, 샌드박싱(Sandboxing), 권한 모델 등.
* **특징:** 도구가 너무 많으면 오히려 모델이 도구를 선택하는 데 너무 많은 토큰을 써서 성능이 저해되는 '멍청한 영역(Dumb Zone)'에 빠질 수 있습니다. 따라서 동적 도구 스코핑(Dynamic Tool Scoping)이 중요합니다.

### 4계층: 검증 계층 (Verification Layer) — "면역 체계"
검증 계층은 에이전트의 출력이 실제 환경에 적용되기 전, 정확성과 안전성을 확인합니다.
* **주요 요소:** 구조화된 검증 로직, 테스트 스위트, 에러 발생 시 재시도 또는 에스컬레이션(Escalation) 로직 등.
* **특징:** 신뢰성을 높이는 데 가장 영향력이 큰 계층입니다. '성공은 조용하게, 실패는 명확하게' 알리는 것이 핵심이며, 개발자와 QA가 공동으로 담당합니다.

### 5계층: 라이프사이클 계층 (Lifecycle Layer) — "신경계"
라이프사이클 계층은 실행 중인 프로세스로서의 에이전트를 관리합니다.
* **주요 요소:** 시작, 상태 모니터링, 안정적인 종료, 크래시 복구(Crash Recovery), 비용 추적, 인간 개입(Human-in-the-loop) 등.
* **특징:** 이 계층이 없으면 에이전트는 비용만 많이 들고 감시 불가능한 프로세스가 됩니다. 인프라 성격이 강하며, SRE(Site Reliability Engineering) 및 DevOps 팀이 담당합니다.

## 책임의 분리: 플랫폼 vs 개발자

Anthropic의 매니지드 에이전트 사례를 통해 보면, 플랫폼 제공자와 개발자의 역할 분담이 명확해집니다.

* **플랫폼 제공자 (Infrastructure):** **2, 3, 5계층**을 담당합니다. 세션 지속성, 샌드박스 관리, 비용 추적 등 인프라적 투자가 필요한 영역입니다.
* **개발자 (Application):** **1, 4계층**을 담당합니다. 아키텍처 경계, 수락 기준(Acceptance Criteria), 도메인 특화된 검증 로직 등 도메인 지식이 필요한 영역입니다.

가장 흔한 실수는 4계층(검증)에는 투자하면서 1계층(제약)을 간과하는 것입니다. 1계층의 결정론적인 제약을 강화하면 4계층의 검증 부담을 획기적으로 줄일 수 있습니다.

## 프레임워크 활용 가이드

### 진단 방법
현재의 하네스를 5개 계층에 따라 감사(Audit)하십시오.
* **구조적 오류**가 빈번하다면 $\rightarrow$ **1계층(제약)**에 투자.
* **도구 선택 오류**나 토큰 낭비가 심하다면 $\rightarrow$ **3계층(실행)**에 투자.
* **에이전트가 길을 잃거나** 비용이 폭증한다면 $\rightarrow$ **5계층(라이프사이클)**에 투자.

### 투자를 지양해야 할 때
하네스 엔지니어링은 투자 대비 회수 기간이 수주에서 수개월이 걸리는 작업입니다. 다음과 같은 경우에는 과도한 투자를 피해야 합니다.
* **프로토타이핑 단계:** 실험적인 프로젝트에 5계층을 모두 구축하는 것은 과잉 최적화입니다.
* **단발성 작업(Single-shot tasks):** 단계가 하나뿐인 작업에는 하네스 오버헤드가 오히려 방해가 됩니다.
* **레거시 코드베이스:** 구조가 엉망인 코드에 하네스를 억지로 입히는 것은 매우 어렵습니다.
* **런타임 미결정 상태:** 사용할 에이전트 런타임(Claude Code, Codex 등)이 정해지지 않았다면, 이식 가능한 구성 요소(제약, 지식 자산)에만 집중하십시오.
