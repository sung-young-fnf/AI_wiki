# LiteParse: LLM의 테이블 이해 능력을 신뢰하며 변환 작업을 중단하다

Pankaj 팔로우
6분 독서
· 2026년 3월 22일

135

LlamaIndex는 `LlamaParse`의 핵심 파싱 엔진을 오픈 소스화했으며, 문서 구조를 감지하고 마크다운(markdown)으로 변환하는 대신, 공간적 레이아웃(spatial layout)을 보존하고 모델이 나머지를 파악하도록 맡깁니다.

전체 크기로 이미지를 보려면 Enter 키를 누르거나 클릭하세요.
동일한 문서에서 나온 두 가지 경로. 하나는 왜곡되고, 다른 하나는 보존합니다.

이러한 반복적인 상황을 겪어보셨을 겁니다. 에이전트가 PDF 테이블에서 데이터가 필요하여 `pypdf`로 추출 코드(extraction code)를 작성하지만, 결과는 열 정렬이 맞지 않는(misaligned columns) 데이터입니다.

그래서 `pdfplumber`로 전환하면 정렬 문제는 해결되지만, 두 행이 한 행으로 조용히 병합됩니다.

다음으로 마크다운 변환기를 시도하면, 이번에는 헤더(header)가 한 열 오른쪽으로 이동하여 재무 요약(financial summary)의 모든 숫자가 잘못된 레이블에 첨부됩니다.

유료 구독 장벽에 부딪히셨나요? 걱정 마세요, 여기에서 전체 게시물을 무료로 읽으세요.

세 번의 반복과 수천 개의 토큰(token)이 소모된 후, 에이전트는 문서 내용에 대해 추론하는 것보다 문서를 파싱(parsing)하는 데 더 많은 연산 자원(compute)을 사용하게 됩니다.

문제는 에이전트의 추론 능력이 아닙니다. 해당 파이프라인의 모든 파서(parser)는 동일한 작업을 시도하고 있습니다: 문서의 구조를 감지하고, 마크다운(markdown)으로 변환한 다음, 변환이 정확하기를 바라는 것입니다. 이는 문서가 간단할 때는 작동하지만, 병합된 셀(merged cells), 다단계 헤더(multi-level headers) 또는 깔끔한 그리드(grid)에 맞춰지지 않는 열이 있을 때는 변환 과정에서 오류가 발생하며, 결과 마크다운은 문법적으로 유효(syntactically valid)하기 때문에 에이전트가 이를 감지할 수 없습니다.

LlamaIndex는 수년 동안 `LlamaParse`를 보험, 금융, 제조 분야에서 사용되는 실제 서비스용 문서 파서(production document parser)로 구축했으며, 그 과정에서 빠르고 경량화된 파싱(fast, lightweight parsing)에 실제로 무엇이 필요한지 알게 되었습니다.

이번 달, 그 핵심 엔진을 `LiteParse`라는 이름으로 오픈 소스화했습니다. 이는 TypeScript 기반의 CLI 및 라이브러리(TypeScript-native CLI and library)로, 클라우드 호출(cloud calls)이나 Python 종속성 없이, 그리고 파싱 루프(parsing loop)에 LLM 없이 완전히 사용자 머신에서 실행됩니다.

이는 3일 만에 GitHub에서 1,500개 이상의 별을 받았으며, 주목할 만한 설계 결정은 의도적으로 피하는 부분입니다: 문서의 구조를 전혀 이해하려고 시도하지 않는다는 점입니다.

### 레이아웃 보존, 변환 건너뛰기

LiteParse의 핵심 철학은 구조 감지(structure detection)가 LLM 파이프라인에 부적합한 추상화(abstraction)라는 것입니다.

테이블을 찾아서 마크다운으로 변환하는 대신, 모든 텍스트 요소를 페이지에 나타나는 좌표에 따라 공간 그리드(spatial grid)에 투영하여, 공백 자체가 구조가 되는 일반 텍스트(plain text)를 생성합니다:

```
Name        Age    City
John        25     NYC
Jane        30     LA
```

이는 파이프 문자나 마크다운 서식 없이, 원본 PDF에 나타나는 그대로 들여쓰기(indentation)와 공백(spacing)을 통해 열이 보존된 세 열 테이블의 실제 LiteParse 출력입니다.

이러한 접근 방식은 LLM이 이미 수백만 개의 ASCII 테이블, 들여쓰기된 코드 블록, README 파일의 열 형식 데이터(columnar data), 그리고 공백 정렬된 구성 출력(whitespace-aligned configuration outputs)으로 학습되었기 때문에 이 형식을 이해할 것이라는 가정에 기반합니다.

테이블의 공간 텍스트 렌더링(spatial text rendering)은 정확하게 변환되었을 수도 있고 아닐 수도 있는 마크다운 테이블(markdown table)보다 모델의 학습 분포(training distribution)에 더 가깝습니다. 반면 동일한 테이블의 PyPDF 출력은 일반적으로 세 열을 단일 텍스트 스트림(text stream)으로 연결하여 공간적 관계를 완전히 상실하게 됩니다.

그리드 투영 알고리즘(grid projection algorithm)은 추출된 각 텍스트 요소를 해당 페이지 좌표에 배치하고 전체 페이지를 일반 텍스트 그리드로 렌더링(render)하므로, 테이블이 무엇인지, 열이 어디서 시작되는지, 헤더가 행에 어떻게 매핑되는지를 식별할 필요가 없습니다.

두 텍스트 요소가 그리드에서 겹칠 경우, 하나는 밀려나는데(displaced), 이는 에이전트가 감지하고 해결할 수 있는 가시적인 오류(visible error)입니다. 이는 아무런 표시 없이 열을 조용히 다시 매핑하는 마크다운 변환과 다릅니다.

### 에이전트 워크플로(Agent workflow): 먼저 파싱(Parse), 다음 스크린샷(Screenshot)

LiteParse는 LlamaIndex 팀이 실제 서비스(production) 에이전트 워크플로에서 관찰한 특정 패턴을 중심으로 설계되었습니다.

Anthropic의 공식 PDF 스킬(skill)은 에이전트가 매 실행 시 `pypdf` 또는 `pdfplumber`로 Python 코드를 작성하도록 권장하는데, 이는 문서나 세션(session) 전반에 걸쳐 재사용할 수 없는 일회성 파싱 로직(disposable parsing logic)을 생성합니다.

두 번째 일반적인 패턴은 에이전트가 초기 이해를 위해 텍스트를 추출한 다음, 더 심층적인 시각적 분석이 필요한 페이지에 대해 스크린샷(screenshot)을 생성하는 것입니다.

LiteParse는 이 두 가지 기능을 단일의 안정적인 도구(stable tool)에 두 단계 워크플로(two-step workflow)로 통합했습니다.

첫 번째 단계는 빠른 텍스트 추출(fast text extraction)입니다:

```bash
npm i -g @llamaindex/liteparse

# PDF를 파싱하고 특정 콘텐츠 검색
lit parse quarterly-report.pdf | grep "revenue"
# 특정 페이지를 바운딩 박스(bounding boxes)와 함께 JSON 형식으로 파싱
lit parse quarterly-report.pdf --target-pages "1-5" --format json -o parsed.json
```

두 번째 단계는 에이전트가 특정 페이지에 대한 시각적 추론(visual reasoning)이 필요할 때 시작됩니다:

```bash
# 멀티모달 분석을 위한 고품질 스크린샷 생성
lit screenshot quarterly-report.pdf --target-pages "8,12" --dpi 300 -o ./charts
```

스크린샷은 멀티모달 모델(multimodal model)이 직접 분석할 수 있는 고해상도 페이지 렌더링(high-resolution page renders)입니다. 따라서 8페이지의 공간 텍스트 출력(spatial text output)이 에이전트가 텍스트만으로는 해석할 수 없는 차트 영역을 드러낼 경우, 해당 페이지를 스크린샷으로 찍어 비전 모델(vision model)에 넘겨 더 심층적인 분석을 수행합니다.

이 두 단계 패턴은 단일 명령으로 설치할 수 있는 재사용 가능한 에이전트 스킬(reusable agent skill)로 패키징(packaging)되어 있습니다:

```bash
npx skills add run-llama/llamaparse-agent-skills --skill liteparse
```

일단 설치되면, 에이전트는 대화(conversation)마다 일회성 추출 코드(throwaway extraction code)를 작성하는 대신 안정적이고 휴대 가능한 파싱 도구(stable, portable parsing tool)를 갖게 됩니다.

### LiteParse를 라이브러리(library)로 사용하기

CLI 외에도 LiteParse는 TypeScript(네이티브 구현)와 Python(Node.js 18+가 필요한 CLI 래퍼) 모두에서 임베디드 라이브러리(embedded library)로 작동합니다:

```typescript
import { LiteParse } from '@llamaindex/liteparse';

const parser = new LiteParse({ ocrEnabled: true });
// 파일 경로에서 파싱
const result = await parser.parse('document.pdf');
console.log(result.text);
// 디스크 I/O 없이 버퍼에서 파싱
const response = await fetch('https://example.com/report.pdf');
const buffer = Buffer.from(await response.arrayBuffer());
const fromBuffer = await parser.parse(buffer);
```

```python
# pip install liteparse (Node.js 18+ 필요)
from liteparse import LiteParse

parser = LiteParse()
result = parser.parse("document.pdf")
print(result.text)
```

버퍼(buffer) 입력 경로는 특히 샌드박스 환경(sandboxed environments)의 에이전트에게 유용합니다. PDF 버퍼는 임시 파일(temp files)을 디스크에 쓰지 않고 완전히 메모리에서 파싱(parse)되기 때문입니다. Office 문서(LibreOffice 경유) 및 이미지(ImageMagick 경유)와 같은 비 PDF 형식은 여전히 임시 파일을 통한 변환이 필요하지만, 핵심 PDF 경로는 완전히 메모리에서 유지됩니다.

### OCR: 내장(built-in) 및 플러그인(pluggable) 가능

Tesseract.js는 내장되어 있으며 CPU 코어 전반에서 자동으로 병렬화(parallelizes)되므로, 스캔된 문서나 임베디드 이미지(embedded images)가 포함된 페이지의 경우, 별도의 구성 없이 OCR이 활성화됩니다:

```bash
lit parse scanned-contract.pdf --ocr-server-url http://localhost:8828/ocr
```

더 높은 정확도가 필요할 경우, LiteParse를 외부 OCR 서버(external OCR server)에 연결할 수 있습니다.

저장소(repo)에는 EasyOCR 및 PaddleOCR용 즉시 사용 가능한 서버 래퍼(server wrappers)가 포함되어 있으며, HTTP 엔드포인트(HTTP endpoint)를 통해 텍스트(text), bbox(좌표), 신뢰도(confidence) 세 가지 필드를 반환하는 한 모든 OCR 엔진이 작동합니다.

이 플러그인 가능한(pluggable) 설계 덕분에 팀은 내장 Tesseract로 시작하여 나중에 애플리케이션 코드 변경 없이 더 정확한 엔진으로 교체할 수 있습니다.

### 다중 형식 지원(Multi-format support)

LiteParse는 모든 것을 통합된 파이프라인(unified pipeline)으로 변환하여 PDF 이상을 처리합니다.

Word 문서, PowerPoint 프레젠테이션, Excel 스프레드시트는 파싱 전 LibreOffice를 통해 PDF로 변환되며, 이미지(PNG, JPG, TIFF, SVG)는 ImageMagick을 통해 처리됩니다.

이러한 종속성(dependencies)은 별도로 설치해야 하지만, 일단 설치되면 전체 파이프라인(pipeline)이 동일한 공간 엔진(spatial engine) 아래 통합됩니다:

```bash
# 이 모든 형식은 동일한 파이프라인을 통해 작동합니다
lit parse presentation.pptx
lit parse spreadsheet.xlsx
lit parse scanned-receipt.jpg
```

### LiteParse의 적합한 용도와 그렇지 않은 용도

LlamaIndex는 두 파서(parser) 간의 경계에 대해 명확히 설명합니다.

LiteParse는 에이전트 및 실시간 파이프라인(real-time pipelines)을 위한 오픈 소스 고속 계층(open-source fast layer)으로, 문서를 빠르게 읽고 다음 단계로 진행해야 할 때 사용됩니다. 반면 LlamaParse는 밀집된 테이블, 다중 열 레이아웃, 차트, 손글씨 주석(handwritten annotations)이 포함된 복잡한 재무 보고서에서 모든 셀을 추출해야 하는 문서 인텔리전스(document intelligence) 제품을 위한 클라우드 기반 정확성 계층(cloud-based accuracy layer)입니다.

이 둘은 경쟁하기보다 서로 보완하도록 설계되었습니다.

기존 OCR 벤치마크(OCR benchmarks)인 OlmOCR이 공간 텍스트 출력(spatial text output)을 고려하지 않기 때문에 팀은 맞춤형 벤치마크(custom benchmark)를 구축했습니다. 페이지 스크린샷에서 질문-답변 쌍을 생성하고, 데이터셋을 수동으로 검수했으며, LLM을 판정자(LLM-as-judge)로 사용하여 PyPDF, PyMuPDF, Markitdown과 비교 평가했습니다.

LiteParse는 대규모 문서에서 최상위 수준의 파싱 지연 시간(parsing latency)과 함께 페이지 기반 QA(Question Answering)의 정확도 향상을 보여주었으며, 평가 코드와 데이터셋은 재현성(reproducibility)을 위해 GitHub와 HuggingFace에 공개되어 있습니다.

솔직히 말해, 이러한 벤치마크는 VLM(Vision-Language Model) 기반 도구가 아닌 기본적인 텍스트 추출기(text extractors)와 비교된다는 점입니다. 이는 Docling 또는 Reducto와 같은 상용 등급 대안(production-grade alternatives)과의 직접적인 정확도 비교는 없다는 의미입니다.

문서 에이전트에 대한 이전 게시물에서 우리는 Reducto가 LLM이 내용을 보기 전에 메타데이터에서 콘텐츠를 분리하는 접근 방식에 대해 다루었습니다. 이는 원시 파싱 출력(raw parse output)이 구조적 노이즈(structural noise)와 실제 내용을 혼합하기 때문입니다.

LiteParse는 그 반대 전략(opposite bet)을 취합니다. 즉, 구조를 전처리하여 제거하는 대신, 원본 레이아웃(original layout)을 보존하고 모델이 이를 직접 읽도록 신뢰합니다. 두 접근 방식 모두 반대편 끝에서 동일한 문제를 해결하며, 어떤 방식이 더 나은지는 문서에 따라 다릅니다.

간단한 테이블이 포함된 텍스트 위주의 PDF의 경우, LiteParse의 공간 보존(spatial preservation) 방식은 빠르고 신뢰할 수 있습니다. 반면 중첩된 구조와 복잡한 시각적 요소가 있는 문서는 Reducto 또는 LlamaParse가 더 적합할 가능성이 높습니다.

에이전트 스킬(agent skill) 패턴을 사용하면 LiteParse로 시작하고 공간 출력이 충분하지 않을 때 상위 솔루션으로 전환(escalate)하기 쉽습니다.

`LiteParse`는 Apache 2.0 라이선스가 적용됩니다. 벤치마크 데이터셋은 HuggingFace에 있습니다. 만약 실제 환경(production)에서 공간 텍스트 출력(spatial text output)을 마크다운 기반 파싱(markdown-based parsing)과 비교 테스트해 본 경험이 있으시다면, 어떤 접근 방식이 문서에 더 잘 맞았는지 듣고 싶습니다.
