# 프로덕션 등급 MCP 서버 및 에이전트 시스템의 11가지 계층 구축
멀티테넌트(multi-tenant) 격리, 인증(auth), 속도 제한(rate limit), 감사 추적(audit trail), 승인 게이트(approval gate) 등

MCP 서버는 프로덕션 에이전트(agent) 시스템의 핵심으로 부상하고 있습니다. 멀티테넌트(multi-tenant) 격리, 인증(auth), 속도 제한(rate limit), 감사 추적(audit trail), 파괴적 작업(destructive operation)에 대한 승인 게이트(approval gate)와 같이 모든 난제들이 바로 이곳에 존재하기 때문입니다. 이러한 기능이 없으면 에이전트는 테넌트 간 데이터 유출, 무한 루프로 인한 예산 소진, 인간의 승인 없는 환불 처리 등의 문제를 일으킬 수 있습니다. MCP 서버는 에이전트와 데이터 계층 사이에 안전한 단일 도구 표면(tool surface)으로 존재하며, 모든 에이전트 호출이 데이터 행에 접근하기 전에 인증, 정책 검사, 속도 제한 및 감사된 작업으로 전환함으로써 이러한 문제를 해결합니다.

이 블로그에서는 팀이 간과했을 때 새벽 3시 호출을 유발하는 12가지 구성 요소를 중심으로 설계된 프로덕션 등급 MCP 서버인 Atlas-MCP를 구축할 것입니다. 이 서버 위에, 서버의 도구들을 활용하여 실제 고객 지원 티켓을 처음부터 끝까지 처리하는 4개 에이전트 기반 지원 코파일럿(Copilot) (플래너, 리트리버, 신시사이저, 크리틱)도 함께 구축할 예정입니다.

모든 프로덕션 MCP 서버가 처리해야 할 두 가지 핵심 측면이 있습니다.

첫째는 에이전트의 행동, 도구 선택 정확도, 인자(argument) 유효성 검사, 인용(citation) 정확성, 그리고 환각(hallucination) 없이 구조화된 오류에서 복구하는 능력입니다.
둘째는 인프라 신뢰성, 멀티테넌트 격리, 인증, 속도 제한, 캐싱(caching), 서킷 브레이킹(circuit breaking), 관측 가능성(observability), 그리고 파괴적 작업에 대한 인간 승인 게이트입니다.

에이전트가 프로덕션 데이터베이스에 접근하기 전에 이 두 가지는 필수적으로 갖춰야 할 요소입니다.

## 모듈식 코드베이스 구축

일반적으로 파이썬 MCP 서버는 단일 `server.py` 파일과 몇 개의 `@tool` 데코레이터로 시작합니다. 하지만 인증, 속도 제한, 여러 백엔드, 테넌트 격리 기능이 추가되면 이 단일 파일은 유지보수하기 어려워집니다.

프로덕션 등급 MCP 서버를 위해서는 모든 관심사(concern)가 자체 패키지에 존재하는 모듈러 아키텍처(Modular Architecture) 패턴을 따릅니다.

인증 모듈은 캐싱에 대해 알지 못하고, 캐싱 모듈은 속도 제한에 대해 알지 못하며, 도구 실행 엔진은 각 모듈과 좁은 인터페이스(narrow interface)를 통해 통신합니다.

구축할 구조화된 레이아웃을 살펴보겠습니다…

```
src/atlas_mcp/
├── __init__.py
├── config.py                    # 구성 요소: 중앙 설정, 싱글톤
├── server.py                    # 구성 요소 1: 전송 및 세션
├── agents/                      # 멀티 에이전트 코파일럿 계층
│   ├── base.py
│   ├── planner.py
│   ├── retriever.py
│   ├── synthesizer.py
│   ├── critic.py
│   ├── orchestrator.py
│   └── mcp_client.py
├── auth/                        # 구성 요소 2 및 3: 인증(AuthN) + 인가(AuthZ)
│   ├── oauth.py
│   ├── middleware.py
│   └── policy.py
├── cache/                       # 구성 요소 9: 2계층 캐시
├── errors/                      # 구성 요소 10: SERF 프레임워크
├── governance/                  # 구성 요소 12: 테넌트 + 승인
├── observability/               # 구성 요소 11: 추적, 메트릭, 감사
├── ratelimit/                   # 구성 요소 8: Redis 토큰 버킷
├── reliability/                 # 구성 요소 7: 서킷 브레이커, 재시도, ATBA
├── tools/                       # 구성 요소 4 및 6: 레지스트리 + 실행
│   ├── atomic/
│   ├── composed/
│   └── workflow/
└── validation/                  # 구성 요소 5: 도구 엔벨로프 스키마
```

이 디렉터리 구조는 커 보이지만, 각 폴더는 12가지 프로덕션 구성 요소 중 하나와 명확하게 연결됩니다. 각 모듈은 하나의 계층을 담당하며 독립적으로 테스트, 교체 또는 이해될 수 있습니다.

`agents/`는 MCP 서버 위에 위치하는 4개 에이전트 기반 코파일럿을 포함합니다.
`auth/`는 인증(당신은 누구인가)과 인가(당신은 무엇을 할 수 있는가)를 분리합니다.
`tools/`는 원자적(atomic), 복합적(composed), 워크플로(workflow) 도구로 계층화되어 있습니다.
`reliability/`, `ratelimit/`, `cache/`, `observability/`는 모든 도구 호출을 감싸는 인프라 보호 계층입니다.

각각의 요소를 다음 섹션들에서 단계별로 구축할 것입니다.

### 의존성 관리

프로덕션 프로젝트에서 가장 먼저 필요한 것은 적절한 의존성 관리 전략입니다. 간단한 스크립트에는 `requirements.txt`가 적합하지만, 여러 기여자, 다양한 환경 및 빌드 파이프라인을 포함하는 모든 프로젝트에는 PEP 621을 따르는 `pyproject.toml`을 사용하여 의존성 그룹, 빌드 구성, 도구 설정을 하나의 파일에서 관리합니다.

`pyproject.toml`을 생성하고 프로젝트 메타데이터부터 시작하겠습니다.

```toml
[project]
name = "atlas-mcp"
version = "0.1.0"
description = "A production-grade MCP server for enterprise agents."
readme = "README.md"
requires-python = ">=3.11"
license = { text = "MIT" }
authors = [{ name = "Atlas-MCP Contributors" }]
```

이 블록은 패키지 이름, 버전, 최소 파이썬 버전 및 라이선스를 선언합니다. `requires-python = ">=3.11"`은 `StrEnum` 및 개선된 `asyncio` API와 같은 파이썬 3.11 기능을 사용하기 때문에 의도적으로 설정되었습니다.

다음은 핵심 런타임 의존성입니다. 전송, 인증, 속도 제한, 데이터 계층, 관측 가능성 및 신뢰성을 갖춘 스테이트풀(stateful) MCP 서버를 구축하므로 신중하게 라이브러리 목록을 작성해야 합니다.

```toml
dependencies = [
  # MCP 핵심
  "mcp>=1.2.0",

  # 전송 및 서버
  "starlette>=0.37",
  "uvicorn[standard]>=0.30",
  "httpx>=0.27",
  
  # 유효성 검사 및 설정
  "pydantic>=2.7",
  "pydantic-settings>=2.3",
  "pyyaml>=6.0",
  
  # 인증
  "authlib>=1.3",
  "pyjwt[crypto]>=2.9",

  # 인프라 클라이언트
  "asyncpg>=0.29",
  "redis[hiredis]>=5.0",
  "elasticsearch[async]>=8.14",
  "boto3>=1.34",
  "aioboto3>=13.0",

  # 신뢰성
  "tenacity>=8.5",

  # 관측 가능성
  "opentelemetry-api>=1.25",
  "opentelemetry-sdk>=1.25",
  "opentelemetry-exporter-otlp>=1.25",
  "opentelemetry-instrumentation-starlette>=0.46b0",
  "prometheus-client>=0.20",
  "structlog>=24.2",
]
```

주석을 사용하여 관심사(concern)별로 의존성을 그룹화한 것에 주목하십시오.

새로운 엔지니어가 팀에 합류했을 때 어떤 라이브러리가 어떤 계층에 속하는지 즉시 파악할 수 있으므로 중요합니다.

MCP SDK는 프로토콜을 제공하고, Starlette과 Uvicorn은 HTTP 전송을 제공하며, `pyjwt`와 `authlib`는 토큰 유효성 검사를 처리합니다. `asyncpg`와 `redis`는 데이터 계층의 비동기 드라이버이며, OpenTelemetry 스택은 분산 추적(tracing)을 제공합니다.

또한 선택적 그룹을 사용하여 개발 의존성을 런타임 의존성과 분리해야 합니다.

```toml
[project.optional-dependencies]
dev = [
  "pytest>=8.2",
  "pytest-asyncio>=0.23",
  "pytest-cov>=5.0",
  "ruff>=0.5",
  "mypy>=1.10",
  "httpx>=0.27",
]
```

이렇게 하면 `pip install -e .`을 실행할 때 런타임 의존성만 설치되며, `pip install -e .[dev]`를 실행하면 테스트 및 린트(lint) 도구가 추가로 설치됩니다. 프로덕션 이미지에는 `pytest`와 `mypy`가 포함되지 않아야 합니다.

다음으로 CLI 도구의 진입점(entry point)을 선언합니다.

```toml
[project.scripts]
atlas-mcp = "atlas_mcp.server:main"
atlas-copilot = "atlas_mcp.agents.cli:main"
```

`pip install` 후에는 사용자가 PATH에서 두 가지 명령어를 사용할 수 있습니다. `atlas-mcp`는 서버를 시작하고, `atlas-copilot`은 실행 중인 서버에 대해 4개 에이전트 기반 지원 코파일럿을 실행합니다. 여기에 콘솔 스크립트(console script)를 선언하면 별도의 `bin/` 디렉터리나 래퍼 셸 스크립트가 필요 없습니다.

마지막으로 빌드 시스템과 도구를 구성합니다.

```toml
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[tool.hatch.build.targets.wheel]
packages = ["src/atlas_mcp"]

[tool.ruff]
line-length = 100
target-version = "py311"

[tool.pytest.ini_options]
asyncio_mode = "auto"
testpaths = ["tests"]
```

`hatchling`은 `src/` 레이아웃을 깔끔하게 처리하는 최신 빌드 백엔드입니다. `ruff`는 `flake8`, `isort`, `pyupgrade`를 대체하는 빠르고 단일한 린터(linter)이며, 파이썬 3.11에 고정되어 있습니다. `Pytest`는 모든 비동기 테스트에 데코레이터를 붙일 필요가 없도록 자동 비동기 모드로 설정되어 있습니다.

### 환경 구성 설정

이제 설정 전략이 필요합니다. 표준적인 초보자 접근 방식은 모든 곳에서 `os.environ`을 직접 읽는 것인데, 이는 오타, 기본값 누락, 환경 간 구성 불일치로 이어집니다.

프로덕션 접근 방식은 설정 관리(Settings Management)입니다. 환경 변수에서 읽어와 유효성을 검사하고 나머지 코드에 노출하는 단일 유형 객체를 사용하는 것입니다. 이를 위해 Pydantic Settings를 사용합니다.

하지만 먼저 새로운 기여자가 어떤 변수가 존재하고 무엇을 위한 것인지 알 수 있도록 `.env.example` 파일이 필요합니다. 전송 및 인증 섹션부터 시작하겠습니다.

```dotenv
# ── 전송 (구성 요소 1) ───────────────────────────────────────
ATLAS_TRANSPORT=http
ATLAS_HTTP_HOST=0.0.0.0
ATLAS_HTTP_PORT=8080
ATLAS_STATELESS_MODE=true

# ── 인증 (구성 요소 2) ────────────────────────────────────────────
# 이 값들을 권한 부여 서버 (WorkOS AuthKit, Auth0,
# Descope, Keycloak, 또는 자체 OAuth 2.1 제공자)를 가리키게 하십시오.
ATLAS_AUTH_ISSUER=https://auth.atlas.local
ATLAS_AUTH_AUDIENCE=atlas-mcp
ATLAS_AUTH_JWKS_URL=https://auth.atlas.local/.well-known/jwks.json
ATLAS_AUTH_REQUIRE_PKCE=true
ATLAS_AUTH_ACCESS_TOKEN_TTL_SECONDS=900
```

`ATLAS_STATELESS_MODE=true`는 핵심적인 플래그입니다. 이는 Streamable HTTP 세션 관리자에게 프로세스 메모리에 세션 상태가 존재하지 않음을 알리며, 로드 밸런서가 어떤 요청이든 어떤 복제본(replica)으로든 라우팅할 수 있음을 의미합니다. 이는 2026년 MCP 로드맵의 최우선 프로덕션 수정 사항입니다.

인증을 위해 토큰이나 비밀(secret)을 여기에 절대 저장하지 않습니다. 우리는 외부 권한 부여 서버(authorization server)의 URL만 저장합니다. Atlas-MCP는 리소스 서버(resource server)이지 권한 부여 서버가 아니며, 이러한 분리는 매우 중요합니다. 이는 섹션 4(인증, 정책 및 거버넌스)에서 더 자세히 다룰 것입니다.

다음은 인가(authorization) 및 신뢰성(reliability)입니다.

```dotenv
# ── 인가 (구성 요소 3) ───────────────────────────────────
ATLAS_POLICY_DEFAULT_DENY=true
ATLAS_POLICY_FILE=config/policy.yaml

# ── 신뢰성 (구성 요소 7) ─────────────────────────────────────
ATLAS_CIRCUIT_BREAKER_FAILURE_THRESHOLD=5
ATLAS_CIRCUIT_BREAKER_RECOVERY_SECONDS=30
ATLAS_RETRY_MAX_ATTEMPTS=3
ATLAS_ATBA_TOTAL_BUDGET_MS=30000
```

`ATLAS_POLICY_DEFAULT_DENY=true`는 절대 끄지 않는 보안 기본값입니다. `policy.yaml`에 명시적으로 도구 호출을 허용하는 규칙이 없으면 호출은 거부됩니다. 이는 에이전트가 주도하는 시스템에 있어 유일하게 안전한 기본값입니다.

신뢰성 설정은 서킷 브레이커(Circuit Breaker) (오픈 전 윈도우 내 5회 실패, 복구 시도 전 30초 대기), 재시도 정책(최대 3회), 그리고 ATBA (적응형 타임아웃 예산 할당)의 에이전트 턴당 총 예산 30초를 조정합니다.

이러한 수치는 중요합니다. 답변에 90초가 걸리는 에이전트는 아무도 사용하지 않는 코파일럿입니다.

그 다음은 속도 제한, 캐싱, 관측 가능성, 거버넌스, 그리고 데이터 계층입니다.

```dotenv
# ── 속도 제한 (구성 요소 8) ───────────────────────────────────
ATLAS_RATE_LIMIT_DEFAULT_RPM=60
ATLAS_RATE_LIMIT_BURST=20
ATLAS_REDIS_URL=redis://redis:6379/0

# ── 캐시 (구성 요소 9) ───────────────────────────────────────────
ATLAS_CACHE_L1_MAX_ITEMS=10000
ATLAS_CACHE_L1_TTL_SECONDS=60
ATLAS_CACHE_L2_TTL_SECONDS=600

# ── 관측 가능성 (구성 요소 11) ──────────────────────────────────
ATLAS_OTEL_ENDPOINT=http://otel-collector:4317
ATLAS_SERVICE_NAME=atlas-mcp
ATLAS_LOG_LEVEL=INFO
ATLAS_METRICS_ENABLED=true
ATLAS_AUDIT_LOG_PATH=/var/log/atlas/audit.jsonl

# ── 거버넌스 (구성 요소 12) ─────────────────────────────────────
ATLAS_TENANT_HEADER=X-Tenant-Id
ATLAS_REQUIRE_TENANT=true
ATLAS_DESTRUCTIVE_TOOL_REQUIRES_APPROVAL=true

# ── 데이터 계층 ────────────────────────────────────────────────────
ATLAS_POSTGRES_DSN=postgresql://atlas:atlas@postgres:5432/atlas
ATLAS_ELASTICSEARCH_URL=http://elasticsearch:9200
ATLAS_S3_ENDPOINT=http://minio:9000
ATLAS_S3_BUCKET=atlas-mcp-data
ATLAS_VECTOR_DB_URL=http://qdrant:6333
```

속도 제한기는 테넌트-도구 쌍(tenant-tool pair)당 분당 60회 요청을 기본으로 하며, 20회의 버스트(burst)를 허용합니다. 캐시는 두 개의 계층으로 구성됩니다…

*   L1은 인-프로세스(in-process) 캐시로 1만 개의 항목과 60초 TTL을 가집니다.
*   L2는 Redis에 저장되며 600초로 더 긴 TTL을 가집니다.

관측 가능성(Observability)은 OTel 수집기(collector)를 가리킵니다. 거버넌스(Governance)는 모든 호출에 테넌트 헤더를 요구하며, 파괴적 도구는 승인을 받아야만 실행되도록 제한합니다.

이제 이 변수들을 단일 타입 설정 클래스(typed settings class)로 사용해 봅시다. `app/core/config.py` 파일을 생성하겠습니다.

```python
"""Atlas-MCP의 중앙 설정.
모든 서브시스템은 이곳에서 설정을 가져와 12가지 구성 요소가
하나의 공유된 진실 공급원을 갖도록 하며, 어떠한 것도 os.environ을
직접 읽지 않도록 합니다.
"""

from __future__ import annotations
from functools import lru_cache
from typing import Literal
from pydantic import Field
from pydantic_settings import BaseSettings, SettingsConfigDict

class ServerSettings(BaseSettings):
    """최상위 서버 구성."""
    model_config = SettingsConfigDict(env_prefix="ATLAS_", env_file=".env", extra="ignore")
```

`env_prefix="ATLAS_"`는 Pydantic Settings가 `ATLAS_`로 시작하는 변수만 로드하도록 지시하므로, 우리 설정이 호스트의 관련 없는 환경 변수를 실수로 읽는 일이 없습니다. `env_file=".env"`는 수동으로 환경 변수를 설정하지 않고도 로컬 개발이 가능하게 합니다.

다음으로 각 설정을 구성 요소별로 그룹화하여 타입이 지정된 속성으로 선언합니다.

```python
# ── 구성 요소 1: 전송 ──────────────────────────────────────
    transport: Literal["stdio", "http"] = "http"
    http_host: str = "0.0.0.0"
    http_port: int = 8080
    stateless_mode: bool = True


# ── 구성 요소 2: 인증 ─────────────────────────────────
    auth_issuer: str = "https://auth.atlas.local"
    auth_audience: str = "atlas-mcp"
    auth_jwks_url: str = "https://auth.atlas.local/.well-known/jwks.json"
    auth_signing_key_path: str | None = None
    auth_require_pkce: bool = True
    auth_access_token_ttl_seconds: int = 900  # 15분

    # ── 구성 요소 3: 인가 ──────────────────────────────────
    policy_default_deny: bool = True
    policy_file: str = "config/policy.yaml"
```

모든 속성에는 기본값이 있어 `.env` 파일 없이도 로컬 개발 환경에서 서버를 부팅할 수 있습니다. `Literal["stdio", "http"]` 타입은 중요합니다. Pydantic은 런타임에 잘못된 전송 방식을 발견하는 대신, 시작 시점에 이를 거부합니다.

신뢰성 및 속도 제한 섹션도 동일한 패턴을 따릅니다.

```python
# ── 구성 요소 7: 신뢰성 ────────────────────────────────────
    circuit_breaker_failure_threshold: int = 5
    circuit_breaker_recovery_seconds: int = 30
    retry_max_attempts: int = 3
    retry_base_delay_ms: int = 100
    atba_total_budget_ms: int = 30_000


# ── 구성 요소 8: 속도 제한 ──────────────────────────────────
    rate_limit_default_rpm: int = 60
    rate_limit_burst: int = 20
    redis_url: str = "redis://localhost:6379/0"

    # ── 구성 요소 9: 캐싱 ────────────────────────────────────────
    cache_l1_max_items: int = 10_000
    cache_l1_ttl_seconds: int = 60
    cache_l2_ttl_seconds: int = 600
```

마지막으로 캐시된 싱글톤(singleton)을 반환하는 `get_settings()` 함수를 노출합니다.

```python
@lru_cache(maxsize=1)
def get_settings() -> ServerSettings:
    """모듈 수준 싱글톤, 한 번 읽고 모든 곳에서 사용."""
    return ServerSettings()
```

`@lru_cache(maxsize=1)`은 이를 싱글톤으로 만듭니다. `get_settings()`를 가져오는 모든 모듈은 동일한 `ServerSettings` 인스턴스를 얻으므로 환경 변수 읽기는 시작 시점에 한 번만 발생합니다.

### 컨테이너화 전략

이제 이 시스템을 실제로 실행해야 합니다. 프로덕션 MCP 서버는 단독으로 실행되지 않습니다. 데이터용 Postgres, 전체 텍스트 검색용 Elasticsearch, 속도 제한 및 캐시용 Redis, 벡터 DB(Qdrant), S3 스토리지(로컬 MinIO), 그리고 OTel 수집기, Jaeger, Prometheus, Grafana가 필요합니다.

이 모든 것을 Docker Compose로 오케스트레이션(orchestrate)합니다. `docker-compose.yml`을 서비스별로 구축해 봅시다.

먼저 Atlas-MCP 애플리케이션 자체입니다.

```yaml
services:
  atlas-mcp:
    build: .
    image: atlas-mcp:dev
    restart: unless-stopped
    env_file: .env
    ports: ["8080:8080"]
    depends_on:
      postgres:      { condition: service_healthy }
      redis:         { condition: service_healthy }
      elasticsearch: { condition: service_healthy }
      qdrant:        { condition: service_started }
      minio:         { condition: service_started }
      otel-collector: { condition: service_started }
    volumes:
      - atlas-logs:/var/log/atlas
      - ./config:/app/config:ro
```

`depends_on`에 `condition: service_healthy`가 있는 것에 주목하십시오…

이는 중요합니다. Atlas-MCP는 Postgres, Redis, Elasticsearch가 단순히 실행 중일 뿐만 아니라 연결을 수락할 준비가 될 때까지 부팅할 수 없습니다. 각 서비스에는 아래에서 정의할 헬스체크(healthcheck)가 있으며, 이 헬스체크가 통과할 때까지 Docker Compose는 `atlas-mcp`를 시작하지 않을 것입니다.

`./config:/app/config:ro` 마운트는 읽기 전용이므로 정책 YAML 파일은 컨테이너 내부에서 변경될 수 없습니다. 정책은 보안 경계이며, 애플리케이션 버그로 인해 정책이 재작성되는 것을 원치 않습니다.

다음으로 헬스체크가 포함된 Postgres입니다.

```yaml
postgres:
    image: postgres:16-alpine
    restart: unless-stopped
    environment:
      POSTGRES_DB: atlas
      POSTGRES_USER: atlas
      POSTGRES_PASSWORD: atlas
    ports: ["5432:5432"]
    volumes:
      - pgdata:/var/lib/postgresql/data
      - ./deploy/sql/init.sql:/docker-entrypoint-initdb.d/init.sql:ro
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U atlas"]
      interval: 5s
      timeout: 3s
      retries: 10
```

`init.sql`을 `/docker-entrypoint-initdb.d/`로 마운트하는 것은 Postgres의 관례입니다. 여기에 놓인 `.sql` 파일은 첫 부팅 시 실행되므로, 우리의 스키마(schema)와 RLS(Row-Level Security) 정책이 자동으로 생성됩니다. 다음 섹션에서 해당 SQL을 작성할 것입니다.

다음으로 Redis, Elasticsearch, Qdrant, MinIO입니다.

```yaml
redis:
    image: redis:7-alpine
    restart: unless-stopped
    ports: ["6379:6379"]
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 10

  elasticsearch:
    image: docker.elastic.co/elasticsearch/elasticsearch:8.14.0
    restart: unless-stopped
    environment:
      discovery.type: single-node
      xpack.security.enabled: "false"
      ES_JAVA_OPTS: "-Xms512m -Xmx512m"
    ports: ["9200:9200"]
    volumes: [esdata:/usr/share/elasticsearch/data]
    healthcheck:
      test: ["CMD-SHELL", "curl -fsS http://localhost:9200/_cluster/health | grep -E 'green|yellow'"]
      interval: 10s
      timeout: 5s
      retries: 10

  qdrant:
    image: qdrant/qdrant:v1.11.0
    restart: unless-stopped
    ports: ["6333:6333"]
    volumes: [qdata:/qdrant/storage]

  minio:
    image: minio/minio:RELEASE.2024-08-17T01-24-54Z
    restart: unless-stopped
    command: server /data --console-address ":9001"
    environment:
      MINIO_ROOT_USER: minioadmin
      MINIO_ROOT_PASSWORD: minioadmin
    ports: ["9000:9000", "9001:9001"]
    volumes: [miniodata:/data]
```

로컬 개발을 위해 Elasticsearch 보안(`xpack.security.enabled: "false"`)을 비활성화합니다. 이는 현실적인 프로덕션 시뮬레이션보다는 빠른 피드백 루프가 목적이기 때문입니다. 프로덕션에서는 완전한 인증 기능을 갖춘 관리형 Elasticsearch를 사용할 것입니다. MinIO는 로컬 개발용 S3 호환 스토리지이며, AWS에서는 실제 S3로 교체됩니다.

이제 관측 가능성(observability) 스택입니다. 관측 가능성이 없는 MCP 서버는 맹목적으로 비행하는 것과 같아서, 추적(trace) 대신 기억에 의존하여 에이전트의 오작동을 디버깅하게 될 것입니다.

```yaml
otel-collector:
    image: otel/opentelemetry-collector-contrib:0.103.0
    restart: unless-stopped
    command: ["--config=/etc/otel/config.yaml"]
    volumes:
      - ./deploy/otel/config.yaml:/etc/otel/config.yaml:ro
    ports:
      - "4317:4317"   # OTLP gRPC
      - "4318:4318"   # OTLP HTTP
    depends_on: [jaeger]

  jaeger:
    image: jaegertracing/all-in-one:1.58
    restart: unless-stopped
    environment:
      COLLECTOR_OTLP_ENABLED: "true"
    ports:
      - "16686:16686"  # UI

  prometheus:
    image: prom/prometheus:v2.53.0
    restart: unless-stopped
    volumes:
      - ./deploy/prometheus/prometheus.yml:/etc/prometheus/prometheus.yml:ro
    ports: ["9090:9090"]

  grafana:
    image: grafana/grafana:11.1.0
    restart: unless-stopped
    environment:
      GF_SECURITY_ADMIN_USER: admin
      GF_SECURITY_ADMIN_PASSWORD: admin
    ports: ["3000:3000"]
    depends_on: [prometheus]
```

흐름은 다음과 같습니다. Atlas-MCP는 OTLP 스팬(span)을 OTel 수집기로 내보내고, 수집기는 이를 Jaeger로 전달하며, `localhost:16686`에서 추적을 볼 수 있습니다. 메트릭(metric)은 Atlas-MCP에서 Prometheus로 직접 전송되며(스크래핑 기반), `localhost:3000`의 Grafana에서 시각화할 수 있습니다.

마지막으로 명명된 볼륨(named volume)입니다.

```yaml
volumes:
  pgdata: {}
  esdata: {}
  qdata: {}
  miniodata: {}
  atlas-logs: {}
```

각 서비스는 영구적인 명명된 볼륨을 얻으므로 컨테이너 재시작 후에도 데이터가 유지됩니다.

이제 Dockerfile입니다. 두 단계 빌드(two-stage build)를 사용합니다. 휠(wheel)을 컴파일하는 두꺼운 빌더 스테이지(builder stage)와 서비스를 제공하는 데 필요한 것만 포함하는 슬림한 런타임 스테이지(runtime stage)입니다.

```dockerfile
# syntax=docker/dockerfile:1.6

# ── 1단계: 빌더 ──────────────────────────────────────────────
FROM python:3.11-slim AS builder
ENV PIP_NO_CACHE_DIR=1 PIP_DISABLE_PIP_VERSION_CHECK=1
RUN apt-get update && apt-get install -y --no-install-recommends \
      build-essential libssl-dev libffi-dev \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /build
COPY pyproject.toml ./
COPY src ./src

RUN pip install --upgrade pip build && \
    pip wheel --wheel-dir /wheels .
```

빌더는 `psycopg2` 및 `cryptography`와 같은 C 확장(C extension)을 컴파일하는 데 필요한 `build-essential`, `libssl-dev`, `libffi-dev`를 설치합니다. 그런 다음 모든 의존성에 대한 휠(portable binary artifacts)을 구축합니다.

다음은 런타임 스테이지입니다.

```dockerfile
# ── 2단계: 런타임 ──────────────────────────────────────────────
FROM python:3.11-slim


ENV PYTHONUNBUFFERED=1 PYTHONDONTWRITEBYTECODE=1 \
    ATLAS_TRANSPORT=http ATLAS_HTTP_HOST=0.0.0.0 ATLAS_HTTP_PORT=8080
RUN apt-get update && apt-get install -y --no-install-recommends \
      curl ca-certificates tini \
    && rm -rf /var/lib/apt/lists/*

# 비루트 사용자, 프로덕션 서버는 절대 루트로 실행되지 않습니다.
RUN useradd --system --uid 10001 --home /app atlas
WORKDIR /app
COPY --from=builder /wheels /wheels
RUN pip install --no-index --find-links=/wheels atlas-mcp && rm -rf /wheels

# 설정 및 로그 디렉터리.
RUN mkdir -p /app/config /var/log/atlas && chown -R atlas:atlas /app /var/log/atlas
COPY --chown=atlas:atlas config/ /app/config/
USER atlas
EXPOSE 8080

HEALTHCHECK --interval=10s --timeout=3s --start-period=15s --retries=3 \
    CMD curl -fsS http://localhost:8080/healthz || exit 1

ENTRYPOINT ["/usr/bin/tini", "--"]
CMD ["atlas-mcp"]
```

여기서 세 가지 주목할 점이 있습니다…

첫째, `useradd --system --uid 10001`은 비루트 사용자(non-root user)를 생성합니다. 프로덕션 서버는 절대 루트(root)로 실행되지 않습니다. 루트 프로세스에서 컨테이너 탈출은 호스트 침해로 이어지지만, 비루트 프로세스에서의 탈출은 격리됩니다.
둘째, `tini`가 진입점(entrypoint)으로 PID 1의 책임을 적절하게 처리하며, 좀비 프로세스(zombie process) 제거 및 시그널 전달을 수행합니다. 파이썬 프로세스는 `SIGTERM`을 자체적으로 우아하게 처리하지 못하는 것으로 악명이 높습니다.
셋째, `HEALTHCHECK`는 우리의 `/healthz` 엔드포인트에 핑(ping)을 보냅니다. Docker와 Kubernetes는 이를 사용하여 컨테이너가 정상 상태인지 판단하며, 비정상적인 컨테이너는 자동으로 교체됩니다.

지금까지 우리는 모듈식 코드베이스, 타입이 지정된 설정 계층, 그리고 헬스체크 및 관측 가능성을 갖춘 완전한 Docker Compose 스택을 구축했습니다.
