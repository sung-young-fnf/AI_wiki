# 5가지 에이전트 프레임워크. 하나의 패턴이 승리하다.
AutoGen vs. LangGraph vs. CrewAI vs. ByteDance의 DeerFlow vs. Anthropic — 그리고 이를 증명할 금융 에이전트

안드레이 카르파티(Andrej Karpathy)는 최근 8개의 AI 에이전트를 연구팀으로 구성하고, 각 에이전트에 전용 GPU(Graphics Processing Unit)를 할당하여 머신러닝(ML) 실험을 자율적으로 수행하게 했습니다. 그의 평가는 "작동하지 않고 엉망진창이다"였습니다.

이는 에이전트가 코드를 짤 수 없었기 때문이 아닙니다. 그들은 카르파티가 설명한 모든 것을 구현했습니다. 문제는 다른 모든 것이었습니다. 즉, 기준선(baselines)이 없고, 통제된 실험(controlled experiments)이 없었으며, 한 에이전트는 네트워크를 더 크게 만들면 손실(loss)이 개선된다는 것을 "발견"했는데, 이는 1학년 박사과정 학생도 알아챌 만한 결과였습니다.

하지만 문제에 대한 그의 이러한 관점이 제 머릿속에 각인되었습니다.

> 이제 당신은 하나의 조직을 프로그래밍하고 있으며, 소스 코드는 그 조직을 구성하는 프롬프트(prompts), 스킬(skills), 도구(tools), 그리고 프로세스(processes)의 집합이다.

이는 질문을 바꿉니다. 문제는 더 똑똑한 에이전트가 아닙니다. 문제는 에이전트 주변의 더 나은 인프라(infrastructure)입니다. 즉, 에이전트가 기능을 로드하고, 컨텍스트(context)를 관리하며, 상태(state)를 유지하고, 서로의 토큰(tokens)에 파묻히지 않고 조율하는 방식입니다. 인프라가 잘못되면, AutoGen 에이전트가 루프(loop) 안에서 똑같은 쉼표를 다시 추가하는 데 3시간을 소비하거나, 킬 스위치(kill switch) 없이 스스로를 재귀적으로 호출하는 에이전트 때문에 하루에 2,000달러짜리 청구서를 받게 됩니다.

직장에서 규제 준수에 민감한(compliance-sensitive) 연구 파이프라인을 위해 5가지 프레임워크를 분석한 후, 저는 "구성 가능한 인프라(composable infrastructure)가 모놀리식 프레임워크(monolithic frameworks)를 능가한다"는 결론에 도달했습니다. 점진적 스킬 로딩(progressive skill loading), 모듈식 미들웨어(modular middleware), 그리고 파일 시스템 우선 상태 관리(filesystem-first state management)는 단순한 기능이 아닙니다. 이것이야말로 확장 가능한(scalable) 아키텍처(architecture)입니다.

이 글에서는 그 이유와 함께, 이를 기반으로 금융 연구 에이전트를 구축할 때 어떤 모습인지 보여드리겠습니다.

## 5가지 철학을 5분 안에

5가지 프레임워크, 그리고 에이전트가 어떻게 조율되어야 하는지에 대한 5가지 접근 방식. 차이점은 기능에 있는 것이 아닙니다. 각 프레임워크가 어디에서 한계에 부딪히는지에 있습니다.

### AutoGen은 대화에 집중했습니다.

에이전트들은 메시지를 브로드캐스트하고 다음 발언자를 선택하는 `GroupChatManager`를 통해 서로 대화합니다. 데모에는 직관적이지만, 대규모로 확장하기에는 가혹합니다.

수치가 그 이유를 말해줍니다. 4개의 에이전트가 20개의 메시지를 교환한다는 것은 각 에이전트가 매 턴마다 20개의 메시지 모두를 처리한다는 뜻입니다. 이는 라운드당 80회의 메시지 읽기입니다. 메시지당 약 500토큰(tokens)이라고 가정하면, 단일 라운드에 에이전트가 실제 작업을 수행하기 전에 40,000개의 입력 토큰(input tokens)이 소모됩니다.

5라운드를 이어나가면, 조율 오버헤드(coordination overhead)만으로 200,000토큰을 사용하게 됩니다. 소넷(Sonnet) 요율 기준으로 작업당 약 0.60달러이며, 이는 단지 에이전트들이 서로의 메시지를 읽는 데 드는 비용입니다.

이 프로젝트는 또한 분열되었습니다. 원래 개발자들은 AG2로 포크(fork)했고, 마이크로소프트(Microsoft)는 핵심 부분을 v0.4로 재작성한 다음, 이를 Semantic Kernel과 병합하여 Microsoft Agent Framework를 만들었습니다. 만약 2024년에 AutoGen 기반으로 개발을 시작했다면, 이제 서로 다른 API(Application Programming Interface)를 가지며 명확한 마이그레이션 경로(migration path)가 없는 세 가지 코드베이스에 직면하게 됩니다. 이는 기술적인 위험뿐만 아니라 거버넌스(governance) 위험이기도 합니다.

### LangGraph는 그래프에 집중했습니다.

노드(nodes), 엣지(edges), 그리고 조건부 라우팅(conditional routing)을 갖춘 유향 상태 머신(directed state machines)입니다. 어떤 에이전트가 언제 실행되고 어떤 상태(state)가 전달될지 정확하게 정의합니다. 모든 노드 전환(node transition)에서의 체크포인트(checkpointing)는 일시 정지(pause), 재개(resume), 다시 재생(replay), 그리고 시간 여행 디버깅(time-travel debugging) 기능을 제공합니다.

절충점은 상용구 코드(boilerplate code)입니다. 조건부 라우팅(conditional routing), 상태 스키마(state schemas), 그리고 체크포인트(checkpointing)를 포함하는 기본적인 두 에이전트 파이프라인은 첫 번째 에이전트가 유용한 작업을 하기 전에 200줄 이상의 설정 코드를 필요로 합니다. 그리고 그래프를 리팩토링(refactor)할 때, 상태 스키마도 함께 변경됩니다. 기존 체크포인트는 무효화됩니다. 제어는 확실하지만, 그만큼 경직성(rigidity)도 따릅니다.

### CrewAI는 역할에 집중했습니다.

각 에이전트에 역할(role), 목표(goal), 그리고 배경(backstory)을 부여한 다음, 에이전트가 스스로 해결하도록 합니다. 아이디어에서 작동하는 데모까지 가장 빠른 경로이며, 코드 30줄 정도면 가능할 수도 있습니다.

문서에서 강조하지 않는 부분이 있습니다. 역할 기반 위임(role-based delegation)은 비결정적(non-deterministic)입니다. 동일한 입력이 LLM(Large Language Model)이 "이것을 누가 처리해야 하는가"를 해석하는 방식에 따라 다른 실행에서 다른 에이전트로 라우팅(route)될 수 있습니다. 이것이 바로 프로덕션 팀이 예측 불가능한 동작에 계속 부딪히자 CrewAI가 흐름(Flows, 결정적 이벤트 기반 파이프라인)을 추가해야 했던 이유입니다.

20억 개의 워크플로우(workflows)에 대한 그들의 블로그 게시물은 명확하게 말합니다. "100% 인간 검토로 시작하여 50%까지 점진적으로 줄여나가라." 이것은 모범 사례(best practice)가 아닙니다. 이는 자율적인 역할 위임이 안전장치(guardrails) 없이는 프로덕션 환경에서 안전하지 않다는 고백입니다.

### DeerFlow는 구성 가능한 인프라에 집중했습니다.

에이전트가 어떻게 대화하는지를 정의하는 대신, 에이전트가 기능을 로드하고, 컨텍스트를 처리하며, 상태를 유지하는 방식을 정의합니다. 9개의 미들웨어(middleware) 파이프라인이 모든 LLM 호출을 감쌉니다. 스킬(skills)은 점진적으로 로드됩니다(메타데이터(metadata) 우선, 트리거(trigger)될 때만 전체 지침 로드). 상태는 토큰(tokens)이 아닌 디스크(disk)에 저장됩니다.

에이전트 조율 계층은 LangGraph 위에 있지만, 진정한 아키텍처는 미들웨어입니다. 또한 여기에서 가장 최신 프레임워크(2026년 2월)이며, 이는 미들웨어 추상화(abstractions)는 깔끔하지만 실제 환경에서의 검증(battle-testing)은 아직 부족하다는 것을 의미합니다.

### Anthropic은 프레임워크 자체를 사용하지 않는 것에 집중했습니다.

프롬프트 체이닝(prompt chaining), 라우팅(routing), 병렬화(parallelization), 오케스트레이터-워커(orchestrator-workers)와 같은 6가지 구성 가능한 패턴(composable patterns)을 직접 연결합니다. 그들의 공식 입장은 "가장 성공적인 구현은 복잡한 프레임워크를 사용하지 않았다"입니다.

장점은 최대의 제어권과 프레임워크 종속성(framework lock-in)이 없다는 것입니다. 단점은 프레임워크가 대신 처리해주는 모든 것(상태 영속성(state persistence), 오류 복구(error recovery), 에이전트 라이프사이클(agent lifecycle), 컨텍스트 관리(context management) 등)을 스스로 책임져야 한다는 것입니다. 강력한 인프라 엔지니어를 가진 팀에게는 자유로울 수 있지만, 다른 모든 사람에게는 첫 번째 에이전트가 실제 작업을 실행하기 전에 기반 시스템(plumbing)을 구축하는 데 수개월이 걸리는 우회로가 될 수 있습니다.

그래프 기반 조율(LangGraph)은 상태를 선택적으로 전달하지만 여전히 누적됩니다. 구성 가능한 인프라(DeerFlow)는 컨텍스트를 적극적으로 관리합니다. 즉, 완료된 작업을 요약하고, 디스크로 오프로드(offload)하며, 필요할 때만 스킬을 로드합니다. 스펙트럼에서 오른쪽으로 갈수록 의사결정(decision)당 비용은 줄어듭니다.

## 구성 가능이 모놀리식을 능가하는 이유

'구성 가능(composable)'이라는 단어가 느슨하게 사용되므로, 구체적으로 설명하겠습니다. 구성 가능한 에이전트 아키텍처(agent architecture)는 세 가지 작업을 수행합니다.

### 1. 필요한 것만 로드합니다.

대부분의 프레임워크는 모든 기능(capability)을 처음부터 에이전트의 컨텍스트에 밀어 넣습니다. 모든 도구 설명, 모든 지침 세트, 모든 시스템 프롬프트(system prompt)가 에이전트가 사용하든 안 하든 토큰을 소모합니다.

점진적 스킬 로딩(progressive skill loading)은 이 방식을 뒤집습니다. 수십 개의 스킬을 가벼운 메타데이터(metadata, 이름과 한 줄 설명, 각각 약 100토큰)로 저장합니다. 전체 지침은 스킬이 트리거될 때만 로드합니다. 무거운 리소스(scripts, templates, reference docs)는 명시적으로 참조될 때만 가져옵니다.

이는 도서관 카드(library card)를 들고 다니는 것과 도서관 자체를 들고 다니는 것의 차이입니다. 이를 '도서관 카드 원칙(The Library Card Principle)'이라고 부릅니다. 모든 것을 인덱싱하고(index everything), 필요한 것만 로드합니다.

### 2. 상태를 토큰이 아닌 디스크에 저장합니다.

SEC(미국 증권거래위원회)의 단일 10-K 보고서는 60,000에서 100,000토큰에 이릅니다. 세 개의 보고서를 공유 에이전트 컨텍스트에 넣으면, 에이전트가 추론을 시작하기도 전에 180,000에서 300,000개의 입력 토큰을 소모하게 됩니다.

Claude Sonnet의 백만 입력 토큰당 3달러 요율로, 이는 호출당 1달러 미만입니다. 한 번은 재앙적이지 않지만, 하루에 100개의 연구 쿼리(query)를 실행하는 팀은 실제 분석 토큰 사용 전에 순수 보고서 컨텍스트만으로 매일 54달러에서 90달러를 지출하게 됩니다.

저는 이것을 '토큰 세금(The Token Tax)'이라고 부릅니다. 즉, 디스크에 존재해야 할 데이터를 컨텍스트에 보관하는 비용입니다.

파일 시스템 우선 상태 관리(Filesystem-first state management)가 이 문제를 해결합니다. 에이전트는 보고서를 읽고, 필요한 것을 추출하며, 요약을 디스크에 기록하고, 원본 토큰을 해제합니다. 다음 에이전트는 원본이 아닌 요약을 읽습니다. 컨텍스트 윈도우는 창고에 보관된 데이터가 아니라 작업 메모리(working memory)를 보유합니다.

### 3. 미들웨어를 모듈식 파이프라인으로 사용합니다.

컨텍스트 관리, 메모리, 샌드박싱(sandboxing), 그리고 요약 기능을 프레임워크의 핵심에 집어넣는 대신, 각 LLM 호출을 감싸는 독립적인 모듈(module)로 취급합니다. 긴 문서 요약이 필요한가요? 그것을 연결(plug in)하세요. 읽기 전용 연구 에이전트에 샌드박싱이 필요 없나요? 그것을 제외하세요. 각 미들웨어 컴포넌트(component)는 단일 책임(single responsibility)을 가지며, 파이프라인은 에이전트별, 작업별로 구성 가능합니다.

왜 이것이 1년 전보다 지금 더 중요할까요? 모델은 저렴해지고 있지만, 컨텍스트 윈도우를 잘 사용하는 것은 더 비싸지고 있기 때문입니다. GPT-4o의 입력 가격은 GPT-4 대비 10배 하락했습니다. 그러나 128K 컨텍스트 윈도우를 잘못 사용하는 비용은 전혀 떨어지지 않았습니다. 관련 없는 컨텍스트를 채워 넣는 것은 여전히 출력 품질을 저하시키고, 지연 시간(latency)을 증가시키며, 돈을 소모합니다. 더 저렴한 토큰은 낭비를 덜 눈에 띄게 할 뿐, 낭비가 덜 실제적인 것은 아닙니다.

모놀리식 프레임워크는 이 문제를 숨깁니다. 그들은 당신에게 큰 컨텍스트 윈도우를 주고 그것을 채우게 합니다. 구성 가능한 인프라는 당신에게 묻도록 강제합니다. "이 토큰은 제 역할을 하고 있는가?"

단일 작업인 "AAPL을 연구하고 투자 메모를 작성하라"에 대해 이 원칙들이 어떻게 적용되는지 살펴보겠습니다.

```
모놀리식 (모든 것을 컨텍스트에):
  시스템 프롬프트 + 지침:          2,000 토큰
  40개 스킬 정의 (전체):          40,000 토큰
  10-K 보고서 (원문):              80,000 토큰
  이전 대화 기록:                15,000 토큰
  ─────────────────────────────────────────────────
  LLM 호출당 입력:                 137,000 토큰
  × 파이프라인당 12회 호출:         1,644,000 토큰
  백만 입력 토큰당 $3 비용:                    $4.93

구성 가능 (점진적 + 파일 시스템):
  시스템 프롬프트 + 지침:          2,000 토큰
  40개 스킬 메타데이터 (이름만):        4,000 토큰
  1개 활성 스킬 본문:                   1,500 토큰
  디스크에서 구조화된 요약:          3,000 토큰
  ─────────────────────────────────────────────────
  LLM 호출당 입력:                  10,500 토큰
  × 파이프라인당 12회 호출:           126,000 토큰
  백만 입력 토큰당 $3 비용:                    $0.38
```

이는 실행당 13배 더 저렴합니다. 하루에 50개의 연구 파이프라인을 실행하는 팀은 구성 가능한 인프라를 사용하면 월 약 570달러를 지불하는 반면, 모놀리식 방식으로는 월 약 7,400달러를 지불합니다. 스킬을 추가할수록 격차는 더욱 벌어집니다. 새로운 기능 하나를 추가하는 데 1,000~2,000토큰(전체 정의) 대신 약 100토큰(메타데이터)이 소모됩니다.

그러나 비용만이 가장 강력한 주장은 아닙니다. 품질이 더 중요합니다. Anthropic과 Google의 연구 모두 LLM 정확도가 관련 없는 정보로 컨텍스트가 채워질수록 저하됨을 보여줍니다. 사용되지 않는 스킬 설명과 원본 보고서 텍스트가 120,000토큰을 차지하는 137,000토큰 입력은 단순히 비용만 더 드는 것이 아닙니다. 더 나쁘게 추론합니다.

![모놀리식 vs 구성 가능](https://miro.medium.com/v2/resize:fit:700/1*p3E5rW-5K_w1V3E2P7m0_Q.png)
*이미지 출처: 작성자 — 모놀리식 vs 구성 가능*

## DeerFlow 아키텍처: 구성 가능이 실제로 작동하는 방식

DeerFlow는 ByteDance의 오픈 소스 에이전트 프레임워크로, 2026년 2월 MIT 라이선스(MIT license)로 출시되었습니다. 이 아키텍처는 DeerFlow가 "승자"이기 때문이 아니라(너무 새롭고 검증되지 않아 그렇게 부르기 어렵습니다), 구성 가능한 인프라 원칙을 단일 코드베이스(codebase)에서 가장 명확하게 구현했기 때문에 연구할 가치가 있습니다. 이를 추천이 아닌 참조 아키텍처(reference architecture)로 생각하십시오.

내부적으로는 9개의 특수 노드(nodes)를 가진 LangGraph `StateGraph`입니다. 리드 에이전트(lead agent)는 사용자 요청을 받아 서브 태스크(sub-tasks)로 분해하고, `StepType` 분류에 따라 각 서브 태스크를 올바른 서브 에이전트 유형(연구원, 코더, 보고자, 분석가)으로 라우팅합니다.

서브 에이전트(sub-agents)는 격리된 환경에서 실행됩니다. 각 서브 에이전트는 `task_tool()`을 통해 별도의 인스턴스(instance)로 생성되며, 자체 컨텍스트 윈도우와 작업 디렉토리(working directory)를 가집니다. 이들은 결과를 리드 에이전트에 보고하고, 리드 에이전트는 이를 종합하여 작업을 계속합니다.

그것이 조율 계층입니다. 여기에서 구성 가능한 핵심 요소(pillars)가 나타납니다.

### 핵심 요소 1: 점진적 스킬 로딩

DeerFlow는 기능을 세 가지 계층(tiers)의 스킬로 구성합니다.

![필요할 때만 스킬 로드](https://miro.medium.com/v2/resize:fit:700/1*VqF2iYmU51R_5264U05-1Q.png)
*이미지 출처: 작성자 — 필요할 때만 스킬 로드*

수치가 중요합니다. 40개의 스킬이 설치되어 있다고 가정해 봅시다. 1계층에서는 항상 약 4,000토큰의 메타데이터가 존재합니다. 모든 것을 로드하는 모놀리식 프레임워크에서는 에이전트가 어떤 작업도 하기 전에 40,000에서 80,000토큰을 소모하게 됩니다. Claude Sonnet의 백만 입력 토큰당 3달러 요율로, 점진적 로딩 방식은 호출당 0.012달러인 반면, 모놀리식 방식은 호출당 0.24달러입니다. 하루 1,000회 이상의 에이전트 호출 시, 이는 12달러 대 240달러의 차이를 만듭니다. '도서관 카드 원칙'이 작동하는 방식입니다.

### 핵심 요소 2: 파일 시스템 우선 상태 관리

DeerFlow는 모든 세션(session)에 대해 디렉토리 구조를 강제합니다.

```
/mnt/user-data/
├── uploads/      ← 사용자 제공 파일
├── workspace/    ← 에이전트 작업 디렉토리 (중간 결과)
└── outputs/      ← 최종 결과물
```

이는 단순한 정리가 아닙니다. 이는 컨텍스트 관리 전략입니다. 서브 에이전트가 10-K 보고서 분석을 마치면, 구조화된 결과물을 `/workspace/`에 작성하고, `SummarizationMiddleware`는 대화 기록을 압축합니다. 다음 에이전트는 원본 대화가 아닌 요약 파일을 읽습니다. 완료된 작업은 컨텍스트에서 벗어나 디스크로 이동합니다.

그 결과, 컨텍스트 윈도우는 아무리 많은 데이터가 흘러도 여러 단계의 파이프라인(multi-step pipelines) 전반에 걸쳐 간결하게 유지됩니다.

![파일 시스템 상태 관리](https://miro.medium.com/v2/resize:fit:700/1*T4eT9W3-l8QyKxL7p8T-2w.png)
*이미지 출처: 작성자 — 파일 시스템 상태 관리*

### 핵심 요소 3: 미들웨어 파이프라인

![미들웨어 파이프라인](https://miro.medium.com/v2/resize:fit:700/1*g_e5eI5i7H0c7JpL35b80w.png)
*이미지 출처: 작성자 — 미들웨어 파이프라인*

이제 구체적인 부분입니다. DeerFlow의 모든 LLM 호출은 구성 가능한 미들웨어 스택(middleware stack)을 통과합니다. 기본적으로 9개의 모듈이 제공됩니다.

![미들웨어 참조표](https://miro.medium.com/v2/resize:fit:700/1*T49L5sK7Hh-5x71_mJb0Yg.png)
*이미지 출처: 작성자 — 미들웨어 참조표*

| 미들웨어             | 설명                                           |
| :------------------- | :--------------------------------------------- |
| RouterMiddleware     | 어떤 스킬을 활성화할지 결정합니다.             |
| ToolResolverMiddleware | 스킬 도구를 해결하고 로드합니다.               |
| ParameterBindingMiddleware | 스킬 호출에 매개변수를 바인딩합니다.           |
| ValidationMiddleware | 스키마에 대해 입/출력을 검증합니다.            |
| SummarizationMiddleware | 컨텍스트 윈도우 내용을 압축합니다.             |
| MemoryMiddleware     | 에이전트의 장기 메모리를 관리합니다.           |
| SandboxMiddleware    | 격리된 환경에서 코드를 실행합니다.             |
| ThreadDataMiddleware | 세션 특정 데이터를 관리합니다.                 |
| ErrorHandlingMiddleware | 예외를 포착하고 처리합니다.                    |

각 미들웨어는 `before()` 후크(hook, 프롬프트를 사전 처리)와 `after()` 후크(응답을 사후 처리)를 실행합니다. 이를 재정렬하거나, 비활성화하거나, 사용자 정의 미들웨어를 작성할 수 있습니다.

연구 에이전트는 `SummarizationMiddleware`와 `MemoryMiddleware`를 우선시할 수 있습니다. 코드 실행 에이전트는 `SandboxMiddleware`와 `ThreadDataMiddleware`를 우선시할 수 있습니다.

이 파이프라인은 장식적인 것이 아닙니다. 이곳에 구성 가능한 아키텍처가 실제로 존재합니다. 에이전트는 요약이 어떻게 작동하는지 알 필요가 없습니다. 스스로 메모리를 관리하지도 않습니다. 미들웨어가 이를 처리하며, 다른 작업에는 다른 미들웨어 스택이 적용됩니다.

## DeerFlow로 금융 연구 에이전트 구축하기

이제 실질적인 작업을 해봅시다. 주식 티커(stock ticker)를 입력받아 투자 연구 메모(investment research memo)를 생성하는 에이전트를 원한다고 가정해 봅시다. 기본 정보(fundamentals), 최근 보고서(recent filings), 뉴스 심리(news sentiment), 위험 평가(risk assessment), 동종 기업 비교(peer comparison) 등을 포함하는, 장난감 데모가 아니라 애널리스트에게 시작점으로 건넬 수 있는 결과물 말입니다.

DeerFlow에서 이것을 어떻게 설계할 수 있는지 보여드리겠습니다.

### 에이전트 파이프라인

네 개의 에이전트, 하나의 파이프라인, 각각 명확한 범위(scope)를 가집니다.

![에이전트 파이프라인](https://miro.medium.com/v2/resize:fit:700/1*cM6M4yG2d6Ff4FqQ14bFfQ.png)
*이미지 출처: 작성자 — 에이전트 파이프라인*

-   **리드 에이전트(Lead Agent):** 요청 분해, 태스크 라우팅(routing).
-   **연구원 에이전트(Researcher Agent):** 원시 데이터 수집 (보고서, 뉴스).
-   **분석가 에이전트(Analyst Agent):** 정량적 분석 실행.
-   **보고서 에이전트(Reporter Agent):** 최종 메모 작성.

리드 에이전트는 직접 연구를 수행하지 않습니다. "AAPL 연구" 요청을 세 가지 작업 패키지로 분해합니다. 원시 데이터 수집(연구원), 정량적 분석 실행(분석가), 최종 보고서 작성(보고자). 각 서브 에이전트는 `task_tool()`을 통해 자체 컨텍스트 윈도우, `/workspace/` 아래의 자체 작업 디렉토리, 그리고 필요한 미들웨어만 가지고 생성됩니다.

### 금융 연구 스킬 정의하기

여기에서 점진적 로딩이 그 가치를 발휘합니다. 금융 연구 스킬은 세 가지 계층을 가진 `SKILL.md` 파일에 정의됩니다.

```
# SKILL.md — financial-research
## Metadata (1계층, 항상 로드됨)
name: financial-research
description: SEC 보고서 분석, 재무 비율 계산,
             동종 기업 비교, 위험 점수화, 투자 메모 생성
## Body (2계층, 트리거될 때 로드됨)
### 연구 절차
1. SEC EDGAR에서 최신 10-K 및 10-Q 보고서 가져오기
2. 주요 재무 정보 추출: 매출, 마진, 현금 흐름, 부채 비율
3. 3-5개의 섹터 경쟁사와 동종 기업 비교 실행
4. 위험 요소 점수화: 규제, 집중, 거시적 노출
5. 이전 보고 기간 대비 중요 변경 사항 플래그 지정
### 출력 형식
- 분석가 에이전트 소비를 위한 구조화된 JSON
- 모든 달러 금액은 백만 단위, 비율은 소수점 둘째 자리까지
- 추출된 모든 데이터 포인트에 대한 출처 인용
## Resources (3계층, 참조될 때 로드됨)
- templates/investment-memo-template.md
- scripts/ratio-calculator.py
- reference/sector-classifications.json
```

리드 에이전트가 "AAPL 연구"를 보게 되면, 1계층 메타데이터와 일치시킵니다. 그때서야 연구원 서브 에이전트가 2계층 지침을 로드합니다. 비율 계산기 스크립트는 분석가가 실제로 계산을 실행해야 할 때만 로드됩니다.

다른 40개의 설치된 스킬(코드 검토, 데이터 시각화, 보고서 형식 지정)은 각각 약 100토큰으로, 건드리지 않은 채로 남아 있습니다.

### 미들웨어 구성: 이 사용 사례에 중요한 것

모든 미들웨어가 모든 작업에 대해 제 역할을 하는 것은 아닙니다. 다음은 금융 연구에 중요한 것과 그렇지 않은 것입니다.

![미들웨어 우선순위](https://miro.medium.com/v2/resize:fit:700/1*XkC8tW_M_B_F_fHqG_1hJg.png)
*이미지 출처: 작성자 — 미들웨어 우선순위*

| 미들웨어              | 중요도  | 이유                                           |
| :-------------------- | :------ | :--------------------------------------------- |
| RouterMiddleware      | 필수    | 서브 태스크(sub-tasks) 라우팅에 필요           |
| ToolResolverMiddleware | 필수    | SEC EDGAR, 시장 데이터 로드                    |
| ParameterBindingMiddleware | 필수    | 티커(ticker)를 도구에 전달                       |
| ValidationMiddleware  | 필수    | 구조화된 출력 보장                               |
| SummarizationMiddleware | 필수    | 대화 기록 압축                                   |
| MemoryMiddleware      | 선택 사항 | 단일 메모에는 장기 컨텍스트(context) 덜 중요     |
| SandboxMiddleware     | 필수    | `ratio-calculator.py` 스크립트용                 |
| ThreadDataMiddleware  | 선택 사항 | 세션별 데이터는 파일 시스템에서 처리             |
| ErrorHandlingMiddleware | 필수    | API 제한, 데이터 오류 포착                       |

### MCP 서버 설정: 금융 데이터에 연결

DeerFlow는 MCP(Model Context Protocol)를 사용하여 에이전트를 외부 도구에 연결합니다. 금융 연구를 위해 `conf.yaml`에 데이터 소스를 구성합니다.

```yaml
mcp:
  servers:
    sec-edgar:
      transport: stdio
      command: "python"
      args: ["-m", "mcp_sec_edgar"]
      add_to_agents: ["researcher"]
    market-data:
      transport: streamable-http
      url: "http://localhost:8080/market"
      add_to_agents: ["researcher", "analyst"]
    news-sentiment:
      transport: stdio
      command: "python"
      args: ["-m", "mcp_news"]
      add_to_agents: ["researcher"]
```

`add_to_agents` 필드가 구성 가능한 부분입니다. 연구원(Researcher)은 SEC EDGAR와 뉴스 도구를 얻습니다. 분석가(Analyst)는 시장 데이터를 얻습니다. 보고자(Reporter)는 둘 다 얻지 못합니다. 각 에이전트의 도구 세트는 그 에이전트의 작업에만 맞춰지며, 그 이상도 이하도 아닙니다.

### 파이프라인이 실제로 생성하는 것

연구원은 원시 결과물을 `/workspace/aapl-research/`에 작성합니다.

```
/mnt/user-data/workspace/aapl-research/
├── 10k-summary.json          ← 추출된 재무 정보
├── recent-news.json           ← 감성 점수가 매겨진 기사
├── peer-tickers.json          ← 식별된 비교 대상
└── raw-filings/               ← 전체 텍스트 (아카이브됨, 컨텍스트에 없음)
```

분석가는 요약(원본 보고서 아님)을 읽고, 샌드박스(sandboxed) 환경에서 비율 계산을 실행한 다음, 동일한 작업 공간에 분석 결과를 작성합니다. 보고자는 둘 다 읽고, 3계층 리소스의 메모 템플릿(memo template)을 적용하여 최종 결과물을 `/outputs/`에 작성합니다.

어떤 시점에도 어떤 에이전트도 전체 10-K 보고서를 컨텍스트에 보관하지 않습니다. `SummarizationMiddleware`는 각 서브 태스크가 완료된 후 압축을 수행합니다. 파일 시스템이 데이터를 전달하고, 컨텍스트 윈도우는 사고 과정을 전달합니다.

## 실제로 발생하는 문제점

깔끔한 아키텍처가 깔끔한 배포(deployments)를 의미하지는 않습니다. 실제로 이것을 구축할 때 발생할 수 있는 문제점은 다음과 같습니다.

### 요약이 중요한 정보를 누락합니다.

`SummarizationMiddleware`는 60,000토큰의 10-K 보고서를 3,000토큰 요약으로 압축합니다. 이는 95%의 축소입니다. 무엇인가 잘려나갑니다.

금융 문서에서 "잘려나가는 것"은 종종 숨겨진 위험 요소, 특수 관계자 거래, 또는 주석으로 달린 우발 부채(contingent liabilities)입니다. 이는 투자 분석에 가장 중요한 세부 정보들입니다.

해결책은 더 나은 요약이 아닙니다. 요약 전에 구조화된 추출을 수행하는 것입니다. 먼저 특정 필드(수익 부문, 부채 만기, 위험 요소, 법적 절차)를 구조화된 JSON으로 추출하는 1차 과정을 실행합니다. 그런 다음 서술형 섹션을 요약합니다. 구조화된 데이터는 요약 과정을 완전히 우회하여 바로 디스크에 저장됩니다.

### 외부 데이터 소스에서 속도 제한(rate-limit)이 걸립니다.

SEC EDGAR는 초당 10개의 요청을 제한합니다. 야후 파이낸스(Yahoo Finance), 블룸버그 API(Bloomberg APIs), 뉴스 서비스(news services) 모두 자체 제한을 가지고 있습니다. EDGAR를 동시에 호출하는 세 개의 서브 에이전트를 생성하는 순진한 에이전트 파이프라인은 몇 초 안에 스로틀링(throttled)될 것입니다.

완화책은 에이전트 수준이 아닌 MCP 서버 수준에 있습니다. MCP 서버 자체에 속도 제한(rate limiting) 및 응답 캐싱(response caching) 기능을 구축하십시오. 에이전트는 속도 제한에 대해 알 필요가 없습니다. 그들은 단지 더 느린 응답을 볼 뿐입니다. 보고서(10-K는 게시되면 변경되지 않음)에 TTL(Time-To-Live) 캐시를 추가하면, 동일한 회사에 대한 두 번째 연구 실행 비용은 거의 0의 외부 API 호출이 됩니다.

### 에이전트가 생성한 계산에는 근거(ground truth)가 없습니다.

분석가 에이전트가 부채-자본 비율(debt-to-equity ratio)을 계산할 때, 그것이 정확한지 어떻게 알 수 있을까요? 요약에서 숫자를 가져왔을 뿐, 원본 보고서에서 가져온 것이 아닙니다. 총부채를 총자본으로 나누었을 수도 있고, 장기부채를 주주 자본으로 나누었을 수도 있습니다. 둘 다 어떤 정의에 따라서는 "정확"하지만, 다른 숫자를 생성합니다.

파이프라인에 검증(assertion checks)을 구축하십시오. 디스크의 구조화된 JSON을 사용하여 `ratio-calculator.py`에서 동일한 계산을 실행한 다음, 에이전트의 출력과 비교합니다. 1% 이상 차이가 나면 인간 검토를 위해 플래그를 지정합니다. 이것은 기본적인 모델 검증(model validation)이며, SR 11-7(OCC의 모델 위험 관리 지침)의 기본 원칙을 에이전트 출력에 적용한 것입니다.

### 복잡한 엔터티(entities)에서 컨텍스트 오버플로우(overflow)가 발생합니다.

치폴레(Chipotle)와 같은 단일 제품 회사를 연구하면 파이프라인이 깔끔하게 실행됩니다. 60개 이상의 운영 회사를 가진 버크셔 해서웨이(Berkshire Hathaway)와 같은 복합 기업을 연구하면, 연구원 에이전트의 컨텍스트 윈도우가 세그먼트 데이터 추출을 마치기도 전에 가득 차게 됩니다.

아키텍처적 해결책은 분해(decomposition)입니다. 복합 기업 연구를 세그먼트별 서브 태스크로 나누고, 각 서브 태스크는 자체 컨텍스트 윈도우를 가집니다. DeerFlow의 `task_tool()`은 이를 지원하지만, 스킬 정의는 다중 세그먼트 회사를 감지하고 그에 따라 라우팅해야 합니다. 이 감지 로직은 프레임워크의 문제가 아니라 당신의 문제입니다.

## 규제 준수(Compliance) 관점

대부분의 프레임워크 비교는 개발자 경험(developer experience)에서 멈춥니다. 하지만 규제 산업(regulated industry)에서 일한다면, 정보 보안(infosec) 팀의 첫 번째 질문은 "얼마나 빨리 배포할 수 있는가?"가 아닙니다. "코드가 어디에서 실행되고, 무엇이 로깅되며, 누가 출력을 검토했는가?"입니다.

이러한 질문들은 어떤 기술적 한계보다 빠르게 에이전트 배포를 좌절시킵니다.

### 샌드박스 격리: "정보 보안 팀이 이것을 승인할 수 있습니까?"

CrewAI는 도구를 애플리케이션과 동일한 파이썬 프로세스(Python process)에서 실행합니다. 격리가 없습니다. 코드 실행 도구를 가진 에이전트는 애플리케이션과 동일한 권한을 가집니다.

AutoGen은 선택적 도커 컨테이너(Docker containers)를 제공하지만 강제하지는 않습니다. DeerFlow는 세 가지 샌드박스 모드(로컬 프로세스, 도커 컨테이너, 쿠버네티스 파드(Kubernetes pods))를 제공하며, 기본적으로 `SandboxMiddleware`를 통해 코드 실행을 라우팅합니다. 은행에서는 이것이 개념 증명(proof-of-concept)과 보안 검토를 통과한 프로덕션 배포(production deployment)의 차이입니다.

### 감사 추적(Audit Trail)의 공백: "무슨 일이 일어났고 왜 일어났습니까?"

대화 기반 프레임워크는 의사결정 과정을 채팅 로그(chat logs)에 묻어버립니다. 규제 기관이 에이전트가 특정 권고를 내린 이유를 물을 때, 수천 개의 뒤섞인 메시지를 파헤쳐 추론 사슬(reasoning chain)을 재구성해야 합니다.

이는 가설적인 우려가 아닙니다. OCC Bulletin 2011–12 및 연방준비제도 SR 11–7은 금융 기관에 대한 모델 위험 관리(model risk management) 표준을 정의합니다. 의사결정을 내리거나 정보 제공을 하는 에이전트 시스템은 이러한 프레임워크의 적용을 받습니다.

기준은 "무언가를 로깅했다"가 아닙니다. 재현성(reproducibility)입니다. 동일한 입력이 주어졌을 때, 시스템이 동일한 라우팅 결정과 동일한 출력을 생성한다는 것을 입증할 수 있는가?

대화 기반 조율은 에이전트 선택이 LLM 해석에 따라 달라지기 때문에 재현성을 거의 불가능하게 만듭니다. LLM 해석은 비결정적이기 때문입니다. 미들웨어 파이프라인은 설계상 결정적입니다. 동일한 입력, 동일한 미들웨어 스택, 동일한 라우팅입니다.

DeerFlow의 미들웨어 파이프라인은 자연스러운 로깅(logging) 지점을 생성합니다. 모든 LLM 호출은 동일한 스택을 통과합니다. 모든 미들웨어는 구조화된 로그를 내보낼 수 있습니다. 파일 시스템 우선 패턴은 중간 출력물이 일시적인(ephemeral) 컨텍스트가 아닌 파일로 지속됨을 의미합니다.

최종 메모에서 애널리스트 계산, 연구원 결과, 원본 데이터에 이르기까지 권고 사항을 추적할 수 있습니다. 이것은 채팅 기록이 아닌 감사 추적(audit trail)입니다.

### 휴먼 인 더 루프(Human-in-the-loop): 100%로 시작하여 점진적으로 줄입니다.

CrewAI의 20억 개 워크플로우 경험에 대한 블로그 게시물은 명확하게 말합니다. "100% 인간 검토로 시작하여, 신뢰가 쌓이면 점진적으로 50%까지 줄여나가라." 이것은 규제 준수 팀이 생각하는 방식과 정확히 일치합니다.

DeerFlow의 `ClarificationMiddleware`는 에이전트가 모호한 요청에 대해 행동하기 전에 이를 가로챕니다. 리드 에이전트 오케스트레이션(orchestration) 패턴(하나의 에이전트가 조율하고, 서브 에이전트가 실행하기 전에 사람이 계획을 검토)과 결합하여, 모든 단계를 수동으로 계측(instrument)하지 않고도 검토 가능한 체크포인트(checkpoints)를 얻을 수 있습니다.

FINRA(금융산업규제기구) 규제 환경에 있는 팀에게 이것은 Rule 3110에 따른 감독 의무와 직접적으로 연결됩니다. 투자 연구를 생성하는 에이전트는 인간 애널리스트와 동일한 감독 검토를 요구합니다. 질문은 인간 검토를 해야 하는지 여부가 아닙니다. 당신의 프레임워크가 검토를 아키텍처적으로 자연스럽게 만들 것인지, 아니면 나중에 덧붙이는 사후 고려 사항(bolt-on afterthought)으로 만들 것인지입니다.

### 비용 관리: 예산 항목으로서의 토큰

파일 시스템 우선 상태 관리(Filesystem-first state management)는 단순히 성능 패턴이 아닙니다. 비용 통제 패턴입니다. 중간 결과가 컨텍스트 대신 디스크에 저장될 때, 토큰 지출에 대한 구조적인 상한선(structural caps)이 생깁니다.

`SummarizationMiddleware`는 반응적으로가 아니라 사전적으로 압축합니다. 하루에 50개의 연구 파이프라인을 실행하는 팀에게는 이것이 예측 가능한 월별 청구서와 LLM 제공업체로부터의 예상치 못한 청구서 사이의 차이입니다.

## 한계점

DeerFlow는 제가 살펴본 다른 어떤 것보다 구성 가능한 아키텍처를 더 잘 구현합니다. 그렇다고 해서 주의사항 없이 프로덕션(production)에 바로 투입될 준비가 되었다는 의미는 아닙니다.

### 공개된 벤치마크(benchmarks)가 없습니다.

DeerFlow는 GAIA 점수(scores)도, SWE-bench 결과도 없습니다. 25,000개의 GitHub 별표는 인기가 있다는 것을 말해주지만, 더 낫다는 것을 말해주지는 않습니다.

### 서브 에이전트 격리는 양날의 검입니다.

각 서브 에이전트는 자체 컨텍스트 윈도우를 가지며, 이는 오염을 방지하지만 실시간 협업을 어렵게 합니다. 연구원(Researcher)은 작업 도중에 분석가(Analyst)에게 "이 데이터 포인트가 의심스럽나요?"라고 물을 수 없습니다. 모든 조율은 리드 에이전트를 통해 병목 현상(bottlenecks)을 겪습니다.

### ByteDance 종속성.

MIT 라이선스로 완전 오픈 소스이지만, ByteDance 엔지니어들이 로드맵(roadmap)을 주도합니다. LangChain의 자금 지원을 받는 회사의 지원을 받는 LangGraph나 벤처 캐피탈(venture-backed) 투자를 받고 기업 고객을 확보한 CrewAI와 비교해 보십시오. DeerFlow의 지속 가능성(sustainability) 이야기는 아직 쓰여지지 않았습니다.

### 토큰 오버플로우는 여전히 발생합니다.

`SummarizationMiddleware`는 해결책이 아니라 여유 공간(headroom)을 제공합니다. 수백 개의 보고서, 수천 페이지에 달하는 진정으로 방대한 문서 처리의 경우, 여전히 외부 청킹(chunking) 및 검색(retrieval) 기능이 추가로 필요합니다.

## 최종 생각

이 글은 "DeerFlow를 사용하라"는 것이 아닙니다.

하지만 에이전트 프레임워크 사용을 고려 중이라면, 다음 세 가지 핵심 원칙을 채택하는 것을 고려해 볼 수 있습니다.

### 1. 컨텍스트 윈도우를 감사하십시오.

구축하기 전에 에이전트의 컨텍스트에 실제로 무엇이 있는지 측정하십시오. 30% 이상이 사용되지 않는 도구 설명이나 원시 소스 문서라면, 당신은 '토큰 세금'을 내고 있는 것입니다.

### 2. 상태를 일찍 디스크로 옮기십시오.

컨텍스트 오버플로우(context overflow)가 강제할 때까지 기다리지 마십시오. 첫날부터 파일 시스템 우선 설계를 하십시오. 중간 결과는 대화 기록이 아닌 파일에 속합니다.

### 3. 미들웨어를 명시적으로 만드십시오.

DeerFlow의 파이프라인을 사용하든 직접 구축하든, 컨텍스트 관리, 요약, 샌드박싱을 모듈식으로 만들고 에이전트별로 구성할 수 있도록 하십시오. 10-K 보고서를 연구하는 에이전트는 보고서를 작성하는 에이전트와 동일한 미들웨어 스택을 필요로 하지 않습니다.

카르파티의 말이 맞습니다. 이제 당신은 하나의 조직을 프로그래밍하고 있습니다. 질문은 당신의 조직의 소스 코드가 채팅 메시지 더미인가 아니면 확장 가능한 구성형 아키텍처인가 하는 것입니다.
