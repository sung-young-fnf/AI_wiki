# Anthropic’s New Harness Engineering Playbook (Shows Why Your AI Agents Keep Failing)

오랫동안 실행되는 자율적인 작업(Long-running autonomous tasks)은 아무리 정교한 AI 에이전트라도 가장 빈번하게 실패하는 지점입니다. 앤스로픽(Anthropic)의 새로운 '하네스 엔지니어링(Harness Engineering)' 연구는 이러한 실패가 발생하는 이유에 대해 실무적인 통찰을 제공합니다.

앤스로픽 엔지니어링 팀은 2025년 11월과 2026년 3월에 발표된 두 편의 논문을 통해, 클로드(Claude)가 몇 시간에 걸친 자율 코딩 세션을 성공적으로 수행할 수 있도록 구축한 환경을 문서화했습니다. 놀랍게도 핵심은 더 똑똑한 모델이 아니라, 모델을 둘러싼 **더 영리한 환경**이었습니다. 그들은 이를 '하네스 엔지니어링'이라 부릅니다.

### AI 에이전트가 실패하는 이유: 장기 실행 에이전트의 문제

AI 에이전트는 짧은 작업(코드 리팩터링, 단일 기능 구현 등)에서는 잘 작동하지만, 여러 세션에 걸친 복잡한 작업을 맡기면 품질이 급격히 저하됩니다. 앤스로픽은 네 가지 구체적인 실패 패턴을 정리했습니다.

1.  **모든 것을 한 번에 시도함:** 에이전트는 계획 없이 한 번에 전체 앱을 구축하려다 컨텍스트 윈도우(Context window)를 소진하고, 다음 세션에 문서도 없는 미완성 코드를 넘겨줍니다.
2.  **컨텍스트 불안(Context anxiety):** 컨텍스트가 가득 차면 모델은 한계에 도달했음을 직감하고 작업을 서둘러 마무리하거나 요약만 하는 등 불완전한 상태에서 종료 신호를 보냅니다.
3.  **성급한 승리 선언:** 에이전트는 본질적으로 낙관적입니다. 일부 작업이 진행된 것을 보고 실제 구현되지 않은 기능이 있음에도 프로젝트가 완료되었다고 결론을 내립니다.
4.  **자기 평가 편향(Self-evaluation bias):** 자신이 만든 결과물을 스스로 검토하게 하면 대부분 "좋다"고 평가합니다. 인간 리뷰어가 보기엔 엉망인 코드라도 에이전트는 자신의 논리에 갇혀 오류를 잡아내지 못합니다.

더 나은 모델로 업그레이드한다고 해서 이 문제가 해결되지는 않습니다. 잘못 설계된 환경에서는 고성능 모델도 동일한 방식으로 실패합니다.

### 1단계 해결책: 초기화 에이전트와 코딩 에이전트

앤스로픽은 모델을 개조하는 대신 모델 주위에 **초기화 에이전트(Initializer Agent)**와 **코딩 에이전트(Coding Agent)**라는 두 가지 구조를 세웠습니다.

*   **초기화 에이전트:** 환경을 설정하고 향후 세션의 토대가 되는 세 가지 결과물을 만듭니다.
    *   `feature_list.json`: 사용자 요청을 개별 기능 단위로 분할한 구조화된 파일.
    *   `claude-progress.txt`: 각 세션이 끝날 때 기록하는 로그로, 다음 에이전트가 진행 상황을 파악하게 함.
    *   `init.sh` 및 Git 커밋: 개발 서버를 시작하고 기준 테스트를 실행하는 스크립트.
*   **코딩 에이전트:** 코드를 쓰기 전 항상 정해진 순서(작업 디렉토리 확인, 로그 및 Git 이력 확인, 우선순위 기능 선택 등)를 따릅니다. 가장 중요한 규칙은 **한 번에 하나의 기능만 작업하는 것**입니다.

### 2단계 해결책: 생성기-평가기 루프와 3중 에이전트 구조

복잡한 프로젝트에서는 에이전트가 기능을 흉내만 내거나 UI만 멀쩡하게 만드는 경우가 생깁니다. 이를 해결하기 위해 앤스로픽은 업무를 수행하는 에이전트와 이를 심사하는 에이전트를 분리한 3중 구조를 도입했습니다.

1.  **기획자(Planner):** 프롬프트를 상세한 제품 사양으로 확장합니다. 코드 구현보다는 제품의 범위와 방향성에 집중합니다.
2.  **생성기(Generator):** 기획자의 사양에 따라 '스프린트' 단위로 애플리케이션을 구축합니다. 작업을 시작하기 전 평가기와 무엇이 '완료'인지에 대한 계약을 맺습니다.
3.  **평가기(Evaluator):** 플레이라이트(Playwright) 등을 사용하여 실제 사용자처럼 앱을 조작하며 테스트합니다. 디자인 품질, 독창성, 기능성 등의 기준을 엄격하게 적용하여 합격/불합격(Pass/Fail)을 판정하고 구체적인 피드백을 생성기에 돌려보냅니다.

### 요약: 하네스(Harness)의 5가지 핵심 요소

'하네스'는 단순한 프롬프트가 아니라 모델을 둘러싼 제약 조건, 도구, 피드백 루프, 그리고 구조화된 환경 전체를 의미합니다. 성공적인 AI 에이전트 구축을 위한 하네스의 핵심은 다음과 같습니다.

*   **작업 분해(Task decomposition):** 작업을 검증 가능한 개별 단위로 나눕니다.
*   **세션 연속성(Session continuity):** 진행 상황 파일과 Git 이력을 통해 세션 간 공백을 메웁니다.
*   **외부 평가(External evaluation):** 제작자와 분리된 별도의 에이전트나 프로세스가 결과물을 채점합니다.
*   **기계적 제약(Mechanical constraints):** 모델이 임의로 수정할 수 없는 엄격한 규칙을 적용합니다.
*   **점진적 검증(Incremental verification):** 다음 단계로 넘어가기 전, 실제 사용자 관점에서 각 작업 단위를 테스트합니다.
