지난주, 저는 제 "AI 비서"(AI assistant)가 제 웹사이트의 스크린샷을 찍는 것을 보았습니다. 그리고 또 하나, 이어서 세 장 더 찍었죠. 항공편을 예약하려고 시도했지만, 약 10,000개의 토큰(tokens) (상호작용당 약 0.15달러에 해당)을 소모하고도 어떤 버튼이 "계속" 버튼인지 여전히 알아내지 못했습니다.

*비회원도 여기서 무료로 읽을 수 있습니다.*

이것은 단지 당황스러운 문제가 아니라, 근본적으로 잘못된 방식입니다.

지난 1년간 AI 에이전트(AI agents)들은 마치 낯선 도시에서 비틀거리는 취객처럼 웹사이트를 탐색해왔습니다. 스크린샷을 찍고, 픽셀을 응시하고, 버튼 위치를 추측하며, 디자이너가 폼(form)을 왼쪽으로 5픽셀만 옮겨도 30~40% 확률로 실패합니다. 전체 상호작용 모델은 웹을 실제 모습인 '구조화된 데이터(structured data)'가 아닌 일련의 이미지처럼 취급하는 '편법(hack)'에 기반을 두고 있었습니다.

그러다 2026년 2월 12일, 구글은 모든 것을 바꾸어 놓을 무언가를 크롬 146 카나리(Chrome 146 Canary)에 조용히 공개했습니다. 그것은 바로 WebMCP (웹 모델 컨텍스트 프로토콜: Web Model Context Protocol)이며, 만약 당신이 이에 주목하지 않는다면 뒤처지게 될 것입니다.

저는 과장하는 것이 아닙니다. 이것은 CSS가 테이블 레이아웃을 대체했던 순간과 같습니다. 단, 이번에는 웹사이트를 단순히 더 예쁘게 만드는 것을 넘어, 지능적으로 만들고 있습니다.

## 웹은 AI를 위해 설계되지 않았다 (그리고 우리는 그저 받아들였다)

현재 상황이 얼마나 터무니없는 일인지 이야기해 봅시다.

오늘날 AI 에이전트가 웹사이트와 상호작용할 때, 기본적으로 몇 초마다 스크린샷을 찍어 GPT-4V 또는 Gemini Flash와 같은 비전 모델(vision models)에 이미지로 제공합니다. 각 스크린샷은 대략 2,000개의 토큰을 소모합니다. 여러 페이지에 걸친 단일 제품 검색은 어떨까요? 이러한 상호작용이 20~30번 필요할 수 있습니다. 계산해 보면, 신발 하나를 검색하는 데만 40,000~60,000개의 토큰이 필요하다는 의미입니다.

느리고, 비용이 많이 들며, CSS를 업데이트하는 순간 작동이 멈춥니다.

다른 접근 방식은 DOM 스크래핑(DOM scraping)입니다. 에이전트는 전체 HTML 트리(HTML tree)를 다운로드하고, 수천 개의 div(div)를 파싱(parsing)하며, React 컴포넌트(React component) 구조를 이해하려고 노력하고, 클릭할 올바른 요소를 찾기를 바랍니다. 이 방법은 컨텍스트 윈도우(context window) 공간을 엄청나게 많이 차지하며, 최신 웹 앱(web apps)은 너무나 동적이기 때문에 여전히 계속 실패합니다.

두 접근 방식 모두 이를 위해 만들어진 것이 아닙니다. 웹은 AI 에이전트가 구조화된 작업(structured tasks)을 실행하는 것이 아니라, 사람이 버튼을 클릭하도록 설계되었습니다. 우리는 줄곧 '사각형 못을 둥근 구멍에 억지로 끼워 넣는' 격이었고, 모두가 그저… 받아들였습니다.

지금까지는 그랬습니다.

## WebMCP의 등장: 웹이 마침내 에이전트와 소통하다

WebMCP는 이러한 재앙에 대한 구글의 답변입니다. 이는 마이크로소프트(Microsoft)와 공동 개발한 W3C 표준(W3C standard)으로 (생각해 보면 꽤 놀라운 일이죠), 웹사이트가 `navigator.modelContext`라는 새로운 브라우저 API(browser API)를 통해 구조화된 도구(structured tools)를 AI 에이전트에게 직접 노출할 수 있도록 합니다.

저의 생각을 바꾼 것은 바로 이 점입니다: 백엔드 서버(backend server)가 필요 없습니다. Node.js 프로세스를 구동하거나 도커 컨테이너(Docker containers)를 구성할 필요도 없습니다. 웹페이지 자체가 서버가 됩니다. 브라우저가 모든 것을 처리합니다.

기술적 전환은 엄청납니다. 스크린샷 기반 상호작용과 비교했을 때 토큰 효율(token efficiency)이 89% 향상되었습니다. 구조화된 JSON 응답(structured JSON responses)의 경우, 액션당 수천 개였던 토큰 비용이 20~100개로 감소합니다. 정확도는 어떻고요? 초기 벤치마크(benchmarks)에 따르면, 이전에는 30~40%의 확률로 실패했던 작업에서 약 98%의 성공률을 보이고 있습니다.

이것은 점진적인 개선이 아닙니다. 이것은 패러다임의 전환(paradigm shift)입니다.

## WebMCP의 실제 작동 방식 (기술 심층 분석)

WebMCP는 두 가지 통합 경로를 제공하며, 두 가지 모두 우리가 지금까지 해왔던 방식에 비하면 말도 안 되게 간단합니다.

### 선언형 API (The Declarative API): 그저 작동하는 HTML 속성

적절한 ARIA 레이블(ARIA labels)을 추가하는 것만으로 폼(form)을 접근 가능하게 만들 수 있었던 때를 기억하십니까? WebMCP의 선언형 API(Declarative API)도 같은 방식으로 작동합니다.

기존 HTML 폼에 다음 세 가지 속성을 추가합니다:

```html
<form  
  toolname="login"  
  tooldescription="Log in to the application with email and password"
  toolautosubmit="true">
  
  <label for="email">Email</label>
  <input id="email" name="email" type="email" required>
  
  <label for="password">Password</label>
  <input id="password" name="password" type="password" required>
  
  <button type="submit">Login</button>
</form>
```

이것으로 끝입니다. 말 그대로 전체 통합 과정입니다.

이제 WebMCP를 지원하는 모든 AI 에이전트는 이 도구를 발견하고 직접 호출할 수 있습니다:

*   `toolname`: 함수 이름 (예: "login", "searchProducts", "bookFlight")
*   `tooldescription`: 도구가 무엇을 하는지 자연어로 설명
*   `toolautosubmit`: 에이전트가 자동으로 제출할지, 아니면 폼을 채우기만 할지 여부

만약 당신의 폼이 이미 적절한 `name` 속성(attributes)과 유효성 검사 패턴(validation patterns)으로 잘 구조화되어 있다면, 아마도 80%는 완료된 것입니다. 이 세 가지 속성만 추가하면 전체 폼을 AI가 접근 가능하도록 만들 수 있습니다.

### 명령형 API (The Imperative API): 복잡한 워크플로우를 위한 자바스크립트

단순한 폼을 넘어서는 동적 상호작용(dynamic interactions)의 경우, WebMCP는 `navigator.modelContext.registerTool()`을 통해 명령형 API(Imperative API)를 제공합니다.

다음은 전자상거래 사이트의 실제 예시입니다:

```javascript
navigator.modelContext.registerTool({
  name: "addToCart",
  description: "Add a product to the shopping cart by SKU and quantity",
  inputSchema: {
    type: "object",
    properties: {
      sku: {  
        type: "string",  
        pattern: "^[A-Z]{3}-\\d{6}$",
        description: "Product SKU code"
      },
      quantity: {  
        type: "number",  
        minimum: 1,
        maximum: 99,
        description: "Quantity to add"
      }
    },
    required: ["sku", "quantity"]
  },
  async execute(args) {
    // Your existing JavaScript logic
    const cart = getShoppingCart();
    cart.addItem(args.sku, args.quantity);
  
    return {
      content: [{
        type: "text",
        text: `Added ${args.quantity}x ${args.sku} to cart. Total items: ${cart.itemCount}`
      }]
    };
  }
});
```

여기서 강력해집니다. 기존 프론트엔드 코드(frontend code)를 재사용하는 것입니다. UI 버튼을 구동하는 동일한 함수들이 이제 AI 에이전트 상호작용을 구동합니다. 중복 로직(duplicate logic)도 없고, 별도로 유지보수해야 할 API 표면도 없습니다.

`inputSchema`는 JSON 스키마 유효성 검사(JSON Schema validation)를 사용하므로, 타입 안정성(type safety)과 적절한 오류 처리(error handling)가 자동으로 제공됩니다. 에이전트는 유효하지 않은 데이터를 전달할 수 없으며, 브라우저가 코드가 실행되기 전에 스키마를 강제합니다.

## 실제 통합: 실제로 필요한 것들

실제 애플리케이션에 WebMCP를 통합하는 과정을 안내해 드리겠습니다. 저는 지난주에 항공권 예약 사이트에 이 작업을 수행했으며, 한 시간도 채 걸리지 않았습니다.

### 1단계: 크롬에서 WebMCP 활성화

현재 WebMCP는 초기 프리뷰(early preview) 상태입니다. 실험적 플래그(experimental flag)가 활성화된 Chrome 146 Canary가 필요합니다:

*   크롬 카나리(Chrome Canary) 다운로드
*   `chrome://flags`로 이동
*   "Experimental Web Platform Features" 검색
*   활성화하고 재시작

### 2단계: 폴리필 (Polyfill) 추가

네이티브 브라우저 지원이 광범위하게 배포되기 전까지는 폴리필(polyfill)을 설치하세요:

```bash
npm install @mcp-b/global
```

그런 다음 메인 JavaScript 파일 상단에 임포트(import)합니다:

```javascript
import '@mcp-b/global';
```

이는 아직 `navigator.modelContext` API가 없는 브라우저에 해당 API를 추가합니다. 번들러(bundler)를 사용하지 않는 경우 스크립트 태그(script tag) 옵션도 있습니다.

### 3단계: 도구 등록

가장 중요한 사용자 플로우(user flows)부터 시작하세요. 항공권 예약 사이트의 경우, 세 가지 도구를 등록했습니다:

```javascript
// 도구 1: 항공편 검색
navigator.modelContext.registerTool({
  name: "searchFlights",
  description: "Search for available flights between cities on specific dates",
  inputSchema: {
    type: "object",
    properties: {
      origin: { type: "string", pattern: "^[A-Z]{3}$" },
      destination: { type: "string", pattern: "^[A-Z]{3}$" },
      departDate: { type: "string", format: "date" },
      returnDate: { type: "string", format: "date" }
    },
    required: ["origin", "destination", "departDate"]
  },
  async execute(args) {
    const results = await flightAPI.search(args);
    return {
      content: [{
        type: "text",
        text: `Found ${results.length} flights from ${args.origin} to ${args.destination}`
      }]
    };
  }
});

// 도구 2 & 3: 항공편 상세 정보 가져오기, 항공편 예약...
```

### 4단계: AI 에이전트 연결

WebMCP 브라우저 확장 프로그램(현재 구글의 프리뷰 프로그램(preview program)을 통해 얼리 액세스(early access) 중)을 설치한 다음, Claude Desktop 또는 다른 MCP 호환 AI 클라이언트(AI client)를 연결합니다:

```json
{
  "mcpServers": {
    "chrome-devtools": {
      "command": "npx",
      "args": ["@mcp-b/chrome-devtools-mcp@latest"]
    }
  }
}
```

이제 클로드(Claude)에게 "다음 주 화요일 JFK에서 LAX로 가는 가장 저렴한 항공편을 찾아줘"라고 말하면, 클로드는 UI를 클릭하며 탐색하려고 시도하는 대신, 당신의 `searchFlights` 도구를 직접 찾아 호출할 수 있습니다.

## 이것이 당신 생각보다 더 중요한 이유

WebMCP는 단순한 기술적 업그레이드가 아닙니다. 웹에서 가능한 모든 것을 근본적으로 변화시킵니다.

**개발자에게:** 더 이상 별도의 API를 유지보수할 필요가 없습니다. 프론트엔드 코드(frontend code) 자체가 API입니다. UI를 변경하더라도 도구들은 자동으로 업데이트됩니다. 유효성 검사를 추가하면, 사람과 AI 에이전트 모두에게 적용됩니다. 이는 DRY 원칙(DRY principle)이 논리적 결론에 도달한 모습입니다.

**사용자에게:** AI 에이전트가 마침내 복잡한 다단계 작업(multi-step tasks)을 안정적으로 완료할 수 있게 됩니다. 한 번의 대화로 항공편, 호텔, 렌터카를 예약할 수 있습니다. 15개의 탭을 열지 않고도 여러 사이트에서 제품을 비교할 수 있습니다. 노트북을 창밖으로 던지고 싶은 충동 없이 정부 양식을 작성할 수 있습니다.

**기업에게:** 이는 웹사이트를 에이전트 레디(agent-ready) 상태로 만드는 최초의 표준화된 방법입니다. 구글과 마이크로소프트 모두 이를 출시하고 있습니다. W3C가 이를 표준화하고 있습니다. 만약 에이전트 기반 웹(agent-driven web)에 대비하지 않는다면, 2012년에 모바일 반응형(mobile-responsive)이 아니었던 웹사이트들처럼 구식이 될 것입니다.

이것은 과장이 아닙니다. 사용자들이 모바일 기기로 이동하면서 모바일 우선 디자인(Mobile-first design)이 필수가 되었습니다. 사용자들은 AI 비서로 이동하고 있기 때문에 에이전트 우선 디자인(Agent-first design)이 필수가 될 것입니다.

## WebMCP vs. 앤트로픽의 MCP: 차이점 이해하기

이런 질문을 많이 받았습니다: "이게 앤트로픽(Anthropic)의 모델 컨텍스트 프로토콜(Model Context Protocol)과 같은 것 아닌가요?"

아닙니다. 둘은 완전히 다른 문제를 해결합니다.

앤트로픽의 MCP는 백엔드 프로토콜(backend protocol)입니다. JSON-RPC를 통해 AI 플랫폼을 서버 측 서비스(server-side services)에 연결합니다. 데이터베이스, 내부 도구, 엔터프라이즈 시스템(enterprise systems)에 AI 에이전트가 접근하기 위한 API 게이트웨이(API gateway)로 생각하면 됩니다. 이는 서버 인프라, 인증 계층(authentication layers), 백엔드 엔지니어링(backend engineering)을 필요로 합니다.

WebMCP는 전적으로 브라우저 내에서 작동합니다. 클라이언트 측(client-side)이며, 기존 프론트엔드 코드와 함께 작동하고, 브라우저의 보안 모델(same-origin policy, user permissions)을 활용합니다. 백엔드 인프라가 필요 없으며, 웹페이지가 서버입니다.

둘은 경쟁 관계가 아닌 상호 보완적입니다. 여행사는 ChatGPT Enterprise와의 직접적인 API 통합을 위해 MCP 서버를 운영할 수 있으며, 동시에 브라우저 기반 에이전트(browser-based agents)를 위해 소비자 웹사이트에 WebMCP를 구현할 수도 있습니다. 서로 다른 상호작용 패턴으로 충돌 없이 사용할 수 있습니다.

## 모두가 물어야 할 보안 문제

"잠깐, 그럼 어떤 AI 에이전트라도 제 웹사이트에서 함수를… 그냥 실행할 수 있다는 건가요?"

아닙니다. WebMCP는 '권한 우선(permission-first)' 원칙으로 설계되었습니다.

브라우저가 보안 중재자 역할을 합니다. 에이전트가 도구를 실행하기 전에 크롬은 사용자에게 "AI가 이 항공편을 예약하도록 허용하시겠습니까?"라고 프롬프트(prompt)를 띄울 것입니다. 인간이 통제권을 유지합니다. 이는 명백히 '인간 개입 프로토콜(human-in-the-loop protocol)'이며, 자율 자동화(autonomous automation)가 아닙니다.

도구는 해당 페이지의 오리진 보안 경계(origin security boundary)를 상속받습니다. 동일 출처 정책(same-origin policy)에 의해 교차 출처 공격(cross-origin attacks)이 방지됩니다. 도구에 인증(authentication)이 필요한 경우, 사용자 세션(user's existing session)을 사용합니다. 이는 인간 상호작용을 승인하는 것과 동일한 쿠키(cookies)와 토큰(tokens)을 활용합니다.

WebMCP에는 사용자가 에이전트와의 연결을 끊고 싶을 때 공유 세션 데이터(shared session data)를 지우는 `clearContext()`와 같은 메서드도 포함되어 있습니다.

여기서의 보안 모델은 실제로는 현재 에이전트 접근 방식보다 낫습니다. 스크린샷 기반 에이전트(screenshot-based agents)는 다른 탭의 민감한 데이터나 알림을 포함하여 화면의 모든 것을 볼 수 있습니다. WebMCP 도구는 당신이 명시적으로 정의한 것만 노출합니다.

## 현재 사용 가능한 것 (그리고 앞으로 나올 것)

WebMCP는 현재 초기 프리뷰(early preview) 상태입니다. 오늘 바로 할 수 있는 일은 다음과 같습니다:

*   플래그(flags)가 활성화된 Chrome 146 Canary에서 테스트
*   구글의 얼리 프리뷰 프로그램(Early Preview Program)에 참여하여 문서(documentation)에 접근
*   개발을 위해 `@mcp-b/global` 폴리필 사용
*   선언형(declarative) 및 명령형(imperative) API를 모두 사용하여 도구 구축
*   Claude Desktop 및 다른 MCP 클라이언트 연결

향후 예정 사항:

*   크롬 스테이블(Chrome stable) 버전의 네이티브 지원 (2026년 중순~하반기 예상)
*   엣지(Edge) 지원 (마이크로소프트가 사양(spec) 공동 저작 중)
*   공식 W3C 사양 초안(specification draft)
*   더 많은 AI 플랫폼에서 WebMCP 지원 추가
*   개발자 도구 및 디버깅 유틸리티(debugging utilities)

이 사양은 커뮤니티 인큐베이션(community incubation)에서 공식 W3C 표준화(standardization)로 전환되고 있습니다. 이 과정은 보통 수개월에서 1년이 걸리지만, 구글과 마이크로소프트 양쪽 모두 작동하는 구현체(implementations)를 출시하고 있다는 점은 매우 중요합니다. 이것은 '베이퍼웨어(vaporware)'가 아닙니다. 지금 바로 프로덕션 브라우저(production browsers)에서 실행되는 코드입니다.

## 지금 WebMCP를 구현해야 할까?

솔직한 제 생각은 이렇습니다: 사용자 대면 서비스(user-facing)를 구축하고 있다면, '예'입니다.

구현 비용은 최소한입니다. 기존 폼에 `toolname`과 `tooldescription`을 추가하는 데는 몇 분밖에 걸리지 않습니다. 기존 JavaScript 함수를 `registerTool()` 안에 래핑하는 것도 간단합니다.

장점은 엄청납니다. 당신이 준비되었든 아니든 다가오고 있는 에이전트 기반 웹에 대비하여 웹사이트의 미래를 보장하는 것입니다.

저는 프리뷰 단계에서 WebMCP를 구현한 사이트들의 초기 분석 데이터를 보았습니다. 에이전트 기반 전환율(Agent-driven conversions)은 기존 플로우(flows)보다 3~4배 더 높습니다. 사용자들은 이전에 중간에 포기했던 복잡한 작업들을 완료합니다. AI 에이전트가 간단한 거래 관련 업무를 처리할 수 있기 때문에 지원 티켓(Support tickets)이 감소합니다.

그리고 솔직히 말해서, 이것은 더 나은 엔지니어링(engineering)입니다. 깔끔하고 문서화가 잘 된 도구를 노출하도록 스스로를 강제하면 코드베이스(codebase)가 더 좋아집니다. 테스트를 작성하는 것이 코드 구조를 개선하는 것과 마찬가지로, WebMCP 통합은 좋은 아키텍처(architecture)를 위한 강제적인 기능입니다.

## 웹은 마침내 필요한 API 계층을 얻었다

모든 웹사이트는 이제 API가 될 것입니다. REST 엔드포인트(REST endpoints)나 GraphQL 서버(GraphQL servers)를 통해서가 아니라, 기존 프론트엔드 코드와 작동하는 브라우저 네이티브 도구 노출(browser-native tool exposure)을 통해서 말이죠.

에이전트들은 이미 여기에 있습니다. 지금 당장 당신의 사이트에서 비싼 스크린샷을 찍고 어떤 버튼을 클릭해야 할지 어림짐작하며 어설프게 돌아다니고 있습니다. WebMCP는 그들, 그리고 당신에게 더 나은 방법을 제공합니다.

문제는 당신이 WebMCP를 구현해야 하는지가 아닙니다. 문제는 AJAX가 동적 페이지를 가능하게 한 이후 웹 아키텍처(web architecture)의 가장 큰 변화에 일찍 참여할 것인가 아니면 늦을 것인가 하는 것입니다.

저는 어느 편에 설지 알고 있습니다.
