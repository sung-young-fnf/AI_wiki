# AI 코딩 에이전트의 문제 행동 교정: 안드레이 카르파티의 단일 마크다운 파일 솔루션

AI 코딩 에이전트에게는 성격 문제가 있습니다.

이들은 자신감 있고 빠르며 생산적입니다. 하지만 아첨꾼 같고, 가정을 너무 많이 하며, 혼란을 인정하는 것을 병적으로 어려워합니다. "사용자 데이터를 내보내세요"라고 요청하면, 에이전트는 사용자가 커피를 다 마시기도 전에 파일 형식을 조용히 선택하고, 데이터베이스 필드를 결정하며, 전달 메커니즘을 정하고, 400줄의 코드를 내놓습니다. 이 중 절반의 결정은 틀릴 것입니다.

이것은 사소한 불편함이 아닙니다. AI 에이전트가 한 줄 단위(자동 완성 제안, 함수 예측)로 작동했을 때는 잘못된 가정이 있어도 코드 한 줄만 잘못되었습니다. 사용자가 이를 찾아내고, 수정하고, 다음으로 넘어갈 수 있었습니다. 하지만 Claude Code와 같은 에이전트는 이제 프로젝트 단위로 작동합니다. 전체 코드베이스를 읽고, 여러 파일에 걸친 리팩터링을 수행하며, 테스트 스위트를 실행하고, 코드를 자율적으로 커밋합니다. 이러한 규모에서 잘못된 가정은 전체 저장소에 걸쳐 폭포수처럼 문제를 일으킵니다.

안드레이 카르파티(Andrej Karpathy)는 이를 직접 목격했습니다. 수동 코딩에서 약 80% 에이전트 주도 개발로 전환한 후, 그는 실패 패턴에 대한 솔직한 목록을 문서화했습니다. 이 관찰 내용은 빠르게 퍼져나갔고, 개발자 포레스트 창(Forrest Chang)은 이 관찰을 실용적인 것으로 바꿔냈습니다. 바로 Claude Code의 동작 방식을 재구성하는 `andrej-karpathy-skills`라는 단일 `CLAUDE.md` 파일입니다.

이 저장소는 프레임워크나 라이브러리 때문이 아니라, 네 가지 행동 원칙을 담은 하나의 설정 파일 때문에 3만 개 이상의 GitHub 스타를 얻었습니다. 이는 개발자들이 얼마나 이 해결책을 절실히 필요로 했는지를 보여줍니다.

## 카르파티가 지적한 문제

카르파티의 비판은 Claude Code의 코딩 능력에 관한 것이 아니었습니다. 문법은 괜찮고, 아키텍처 지식은 종종 인상적입니다. 문제는 행동에 있었습니다.

세 가지 특정 실패 모드가 계속 나타났습니다.

**1. 묵시적인 가정 (Silent assumptions).** 대규모 언어 모델(LLM)은 하나의 해석을 선택하고 그대로 진행합니다. 멈추거나, 명확한 질문을 하거나, 모호함을 표시하지 않습니다. 카르파티는 모델이 사용자 대신 잘못된 가정을 하고 확인 없이 그대로 진행한다고 단호하게 말했습니다. 모델은 혼란을 관리하지 않고, 명확화를 찾지 않으며, 불일치를 표면화하지 않고, 필요할 때 반대 의견을 제시하지 않습니다.

**2. 과도한 설계 (Overengineering).** 데이터베이스에 저장되는 간단한 부울 토글(boolean toggle)을 요청하면, Redis 캐싱 계층, 입력 유효성 검사 미들웨어, 이벤트 기반 알림 시스템을 받을 수도 있습니다. LLM은 장황한 추상화가 "전문적"이라고 여겨지는 방대한 기업 코드베이스로 훈련됩니다. 그래서 100줄이면 될 것을 1,000줄로 만드는 것을 기본값으로 삼습니다. 추상화를 부풀리고 일회성 로직에 엔터프라이즈 디자인 패턴을 도입합니다.

**3. 부수적인 피해 (Collateral damage).** 에이전트가 요청하지 않은 코드를 편집합니다. 인접한 줄을 재정렬하고, 따옴표 스타일을 바꾸고, 관련 없는 함수에 타입 힌트(type hint)를 추가하며, 완전히 이해하지 못하는 주석을 제거합니다. 깨끗한 Git diff는 실제 수정 사항이 40개의 미용적 변경 사항 아래에 묻히는 악몽으로 변합니다.

세 가지 모두의 근본 원인은 LLM이 제약보다는 생성에 최적화되어 있다는 점에 있습니다. 인간 피드백 기반 강화 학습(RLHF) 훈련은 거절이나 "도움이 되지 않는" 응답에 불이익을 주므로, 모델은 잘못된 요청에 거의 반대하지 않습니다. "무엇을 원하는지 혼란스럽습니다"라고 말하기보다는 잘못된 것을 열정적으로 만드는 것을 선호합니다.

## 하나의 파일에 담긴 네 가지 원칙

`forrestchang/andrej-karpathy-skills` 저장소의 `CLAUDE.md` 파일은 각 실패 모드를 특정 행동 제약으로 해결합니다. 이것들은 모호한 제안이 아닙니다. Claude Code가 저장소의 모든 작업에 접근하는 방식을 바꾸는 직접적인 지침입니다.

파일의 실제 내용은 다음과 같습니다.

### 1. 코딩 전에 생각하기 (Think Before Coding)

이 원칙은 에이전트가 생성을 멈추고 질문을 하도록 강제합니다.

규칙은 구체적입니다. 코드를 작성하기 전에 가정을 명시적으로 진술하세요. 여러 유효한 해석이 존재하는 경우, 하나를 조용히 선택하는 대신 모두 제시하세요. 더 간단한 접근 방식이 있다면 말하세요. 불분명한 점이 있다면 멈추고, 혼란을 명시하고, 질문하세요.

실제로 이것은 "검색을 더 빠르게 만드세요"라고 말할 때, 에이전트가 다음과 같은 옵션을 제시해야 함을 의미합니다. "캐싱 계층을 추가할 수 있습니다(응답은 빠르지만, 데이터가 오래될 위험 있음), 연결 풀링(connection pooling)을 구현할 수 있습니다(처리량은 좋지만, 오래될 위험 없음), 또는 데이터베이스 인덱스(database index)를 추가할 수 있습니다(구조적 변경, 장기적 이점). 어떤 방향을 원하십니까?" 훈련 데이터에서 가장 높은 통계적 가중치를 가진 솔루션을 그냥 만드는 대신 말입니다.

### 2. 단순성 우선 (Simplicity First)

이것은 과도한 설계 방지 규칙입니다. 문제를 해결하는 최소한의 코드. 추측성 내용은 배제합니다.

요청하지 않은 기능은 추가하지 마세요. 한 번만 사용되는 코드에 추상화를 만들지 마세요. 요청하지 않은 "유연성"이나 "설정 가능성"은 배제하세요. 불가능한 시나리오에 대한 오류 처리도 하지 마세요.

파일에는 구체적인 휴리스틱(heuristic)이 포함되어 있습니다. 200줄을 작성했는데 50줄로 줄일 수 있다면 다시 작성하세요. 그리고 에이전트가 스스로에게 물어야 할 직감적인 질문: "시니어 엔지니어라면 이것이 너무 복잡하다고 말할까?" 만약 그렇다면, 단순화하세요.

이는 할인 계산을 위한 전략 패턴(Strategy-pattern)과 같은 안티 패턴을 직접적으로 겨냥합니다. 10% 할인을 적용하는 함수를 요청했는데, 추상 기본 클래스, 세 개의 하위 클래스, 팩토리 메서드(factory method)는 필요하지 않습니다. 필요한 것은 다섯 줄의 코드입니다.

### 3. 정밀한 변경 (Surgical Changes)

경계 규칙입니다. 꼭 필요한 것만 건드리세요. 자신의 실수만 정리하세요.

기존 코드를 편집할 때는 인접한 코드, 주석, 포매팅을 "개선"하지 마세요. 고장 나지 않은 것을 리팩터링하지 마세요. 다른 방식으로 하고 싶어도 기존 스타일을 따르세요. 관련 없는 데드 코드(dead code)를 발견하더라도 언급만 하고 삭제하지 마세요.

여기에는 데드 코드 책임에 대한 현명한 구별이 있습니다. 변경 사항으로 인해 고아가 되는(사용되지 않는 임포트, 변수 또는 사용자의 수정으로 인해 쓸모없게 된 함수) 코드는 제거하세요. 그러나 인간이 명시적으로 요청하지 않는 한 기존의 데드 코드를 건드리지 마세요. 오래된 주석 처리된 함수가 쓸모없어 보일지라도, 거기에 어떤 이유가 있을 수 있습니다.

테스트는 간단합니다. Git diff의 모든 변경된 줄은 사용자의 요청과 직접적으로 연결되어야 합니다. 그 외에는 아무것도 건드리지 마세요.

### 4. 목표 기반 실행 (Goal-Driven Execution)

이 원칙은 모호한 명령을 검증 가능한 이정표로 바꿉니다.

"버그 수정" 대신, 에이전트는 버그를 재현하는 테스트를 작성하고, 실행하여 실패를 확인하며, 수정 사항을 작성하고, 다시 테스트를 실행하여 회귀 없이 통과하는지 확인해야 합니다. "유효성 검사 추가" 대신, 유효하지 않은 입력에 대한 테스트를 먼저 작성하고, 그런 다음 통과하도록 만들어야 합니다.

다단계 작업을 위해 파일은 간략한 계획을 요구합니다.

1.  [단계] → 확인: [체크]
2.  [단계] → 확인: [체크]
3.  [단계] → 확인: [체크]

이것이 작동하는 이유는 LLM이 잘 정의된 목표를 향해 반복하는 데 탁월하기 때문입니다. 카르파티 자신도 이렇게 언급했습니다. 모델에게 무엇을 할지 지시하지 말고, 성공 기준을 제공하고 작동하는지 지켜보세요. 강력한 기준은 에이전트가 독립적으로 작업하도록 합니다. "작동하게 만들어라"와 같은 약한 기준은 지속적인 인간 개입을 강제합니다.

## 설정 방법 (약 30초 소요)

워크플로우에 따라 세 가지 설치 경로가 있습니다.

**옵션 A: 플러그인 설치 (전역적으로 모든 프로젝트에 적용)**

Claude Code 내에서 다음을 실행합니다.

```
/plugin marketplace add forrestchang/andrej-karpathy-skills
```

그런 다음:

```
/plugin install andrej-karpathy-skills@karpathy-skills
```

이렇게 하면 작업하는 모든 프로젝트에서 행동 지침을 사용할 수 있습니다. 저장소별 설정이 필요 없습니다.

**옵션 B: 새 프로젝트 (파일 직접 다운로드)**

```
curl -o CLAUDE.md https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md
```

이를 프로젝트 루트에 배치합니다. Claude Code는 모든 세션 시작 시 `CLAUDE.md`를 자동으로 읽습니다.

**옵션 C: 기존 프로젝트 (현재 `CLAUDE.md`에 추가)**

```
echo "" >> CLAUDE.md
curl https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md >> CLAUDE.md
```

이는 행동 지침을 기존의 프로젝트별 지침과 병합합니다.

마지막 옵션이 중요합니다. 카르파티 지침은 기존 설정 위에 계층화되도록 설계되었습니다. 이 지침을 추가한 후에는 아래에 자신만의 프로젝트별 섹션을 추가할 수 있습니다.

```
## 프로젝트별 지침
- TypeScript strict 모드 사용
- 모든 API 엔드포인트는 테스트를 포함해야 함
- `src/utils/errors.ts`에 있는 기존 오류 처리 패턴 따르기
```

상위에는 전역 행동 원칙이, 하위에는 로컬 프로젝트 규칙이 옵니다. 서로 충돌하지 않습니다.

## 설치 후의 변화

차이점은 Git diff에서 나타납니다.

**이전:** Claude Code에게 날짜 구문 분석 버그를 수정해 달라고 요청합니다. 에이전트는 버그를 수정하고, 전체 날짜 유틸리티 파일을 리팩터링하며, 인접한 세 함수에 타입 주석을 추가하고, 모듈 전체에서 작은따옴표를 큰따옴표로 바꾸며, "겸사겸사" 새로운 헬퍼 함수를 도입합니다. 실제 수정 사항을 미용적인 변경 사항에서 분리해야 하므로 Pull Request (PR) 검토에 45분이 걸립니다.

**이후:** 에이전트는 날짜 구문 분석 버그를 수정합니다. diff에는 변경해야 할 줄만 정확히 포함됩니다. "47번째 줄의 `formatTimestamp` 함수가 사용되지 않는 것으로 보이지만, 이 수정의 범위 밖에 있으므로 수정하지 않았습니다."와 같은 언급을 할 수는 있습니다. 그것이 전부입니다. PR 검토에 3분이 걸립니다.

하지만 장단점은 분명합니다. 저장소의 문서 자체도 명확히 말합니다. 이 지침은 속도보다는 신중함을 선호합니다. 에이전트는 더 자주 멈출 것입니다. 앞만 보고 나아가는 대신 질문을 할 것입니다. 에이전트가 사용자가 당연하다고 생각하는 가정을 확인하느라 간단한 작업이 약간 느리게 느껴질 수 있습니다.

하지만 사소한 작업을 넘어선 모든 것에 대해서는 시간 절약 효과가 빠르게 복합적으로 나타납니다. 에이전트의 잘못된 추측으로 인한 재작업 감소. 부수적인 편집으로 인한 테스트 실패 감소. 코드 검토 중 "이 파일은 왜 바뀌었지?" 하는 순간 감소. 이러한 마찰은 의도적으로 초기에 발생하도록 설계된 것입니다.

## 심화 학습: 커뮤니티의 6단계 확장

이 저장소에 대한 가장 흥미로운 개발 중 하나는 Issue #11인데, 여기서 otonashi-labs라는 개발자가 네 가지 원칙을 다중 에이전트 워크플로우로 확장하는 6단계 협업 프레임워크를 공유했습니다.

핵심 통찰력: 코드를 구축하는 단일 에이전트는 자신의 가정에 대한 맹점을 가집니다. 그래서 관심사를 분리합니다. 탐색 에이전트가 먼저 시스템 설계를 감사합니다. 구현 코드를 한 줄도 작성하기 전에 모든 상태 전이를 매핑합니다. 그런 다음 인간 검토를 위해 실행을 일시 중지합니다. 인간이 아키텍처 계획을 승인한 후에야 구현 에이전트가 구축을 시작합니다. 그리고 구축 과정에서 격리되어 구축자의 편향을 공유하지 않는 완전히 별개의 검증 에이전트가 코드의 모든 가정을 교차 검증합니다.

이 전체 파이프라인을 채택할 필요는 없습니다. 하지만 그 밑바탕에 깔린 원칙은 내면화할 가치가 있습니다. 코드를 작성하는 에이전트는 해당 코드가 올바른지 판단하는 최악의 심판자입니다. 복잡한 작업을 수행하는 경우, Claude Code가 새로운 컨텍스트 창(동일 세션 내에서도)에서 자신의 출력을 검토하도록 하면 초기 생성에서 놓쳤던 문제를 잡아낼 수 있습니다.

## 마크다운 파일이 생각보다 중요한 이유

`CLAUDE.md`는 단순한 설정 꼼수가 아닙니다. 이는 개발 환경이 작동하는 방식의 더 큰 변화와 교차점에 있습니다.

Claude Code는 모든 세션 시작 시 이 파일을 컨텍스트로 로드합니다. 이는 파일 내용이 전체 상호 작용에서 에이전트의 기본 동작을 형성한다는 의미입니다. 이는 AI 팀원을 위한 온보딩(onboarding) 문서입니다. 인간 개발자를 온보딩하는 것과 마찬가지로, 문서의 품질이 작업의 품질을 직접적으로 결정합니다.

카르파티는 "LLM 위키" 패턴이라고 부르는 것을 옹호해 왔습니다. 프로젝트 지식을 저장소 자체 내의 평면 마크다운 파일로 유지하는 것입니다. 에이전트는 파일 시스템을 직접 읽을 수 있습니다. Notion, Confluence 또는 데이터베이스가 필요하지 않습니다. 코드 옆에 있는 일반 마크다운은 즉시 쿼리 가능한 지식 기반이 됩니다.

`CLAUDE.md`는 그 시스템의 첫 번째 계층입니다. 에이전트를 수정할 때마다 그 수정 사항이 파일로 다시 들어갈 수 있습니다. 에이전트가 프로젝트에서 `date-fns`를 사용하는 데 `moment.js`를 사용하려고 했나요? `CLAUDE.md`에 추가하세요. 훅(hooks)을 사용하는데 계속 React 클래스 컴포넌트를 생성하나요? 추가하세요. 이 파일은 프로젝트별 지능이 축적되는 기록이 됩니다.

시간이 지남에 따라 `CLAUDE.md`는 단순한 행동 지침을 넘어 조직의 기억이 됩니다.

## 솔직한 한계

이 파일이 모든 것을 해결하지는 못합니다. 몇 가지 유의할 사항이 있습니다.

이 지침은 에이전트의 출력을 평가할 수 있는 유능한 개발자에게 가장 잘 작동합니다. "단순성 우선" 원칙은 무엇이 과도하게 복잡한지 인식할 것을 요구합니다. "정밀한 변경"은 에이전트가 범위 밖의 코드를 건드렸을 때 이를 알아차릴 것을 요구합니다. Git diff를 비판적으로 읽을 수 없다면, 이 안전장치(guardrails)는 대부분의 가치를 잃습니다.

토큰 비용 고려 사항도 있습니다. `CLAUDE.md`는 모든 세션에서 컨텍스트로 로드됩니다. 지침으로 너무 비대해지면 모델이 일부를 무시하기 시작합니다. Anthropic의 문서 자체도 이에 대해 경고합니다. 파일을 실행 가능한 코드처럼 다루세요. 가차 없이 가지치기하세요. 모든 줄은 특정 실수를 방지해야 합니다. 그렇지 않다면 제거하세요.

그리고 행동 제약은 가끔 모델의 본능과 충돌합니다. 에이전트가 모순적인 충동을 만족시키려 하기 때문에 약간 부자연스러운 응답을 생성하는 것을 볼 수 있습니다. 훈련은 "도움이 되고 생성하라"고 말하고, `CLAUDE.md`는 "멈추고 질문하라"고 말합니다. 그 긴장감이 바로 핵심이지만, 눈에 띄게 느껴집니다.

## 다음 단계

`CLAUDE.md` 파일 없이 Claude Code를 사용하고 있다면, 에이전트를 제약 없이 내버려두는 것입니다. 일회성 스크립트나 빠른 실험에는 괜찮습니다. 하지만 유지 관리에 신경 쓰는 모든 프로젝트에서는 이 네 가지 원칙(먼저 생각하기, 단순하게 유지하기, 필요한 것만 변경하기, 성공의 기준 정의하기)이 수많은 정리 시간을 절약해 줄 것입니다.

플러그인 설치 또는 curl 명령어로 시작하세요. 일주일 동안 사용해 보세요. Git diff가 어떻게 변하는지 지켜보세요. 그런 다음 카르파티 지침 아래에 자신만의 프로젝트별 규칙을 추가하기 시작하세요.
