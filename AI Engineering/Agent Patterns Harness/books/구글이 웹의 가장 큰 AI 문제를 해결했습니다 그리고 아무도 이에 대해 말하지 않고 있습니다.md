# 구글이 웹의 가장 큰 AI 문제를 해결했습니다 (그리고 아무도 이에 대해 말하지 않고 있습니다)

**WebMCP 통합 전체 가이드, API 예시, 그리고 구글의 브라우저 네이티브 프로토콜이 어떻게 AI 에이전트의 효율적인 웹사이트 상호작용을 가능하게 하는가에 대하여**

---

지난주, 저는 저의 "AI 비서"가 제 웹사이트를 스크린샷으로 찍는 것을 지켜보았습니다. 한 번, 그리고 또 한 번, 그렇게 세 번을 더 찍더군요. 비행기 표를 예약하려던 그 비서는 약 10,000개의 토큰(상호작용당 약 0.15달러)을 소모하고도 어떤 것이 '계속하기(Continue)' 버튼인지 끝내 찾아내지 못했습니다.

이것은 단순히 창피한 일이 아니라, 근본적으로 시스템이 고장 났음을 의미합니다.

지난 1년 동안 AI 에이전트(AI agents)는 마치 외국 도시를 헤매는 술 취한 관광객처럼 웹사이트를 돌아다녔습니다. 스크린샷을 찍고, 픽셀을 찡그려 가며 살피고, 버튼 위치를 짐작합니다. 그리고 디자이너가 양식(form)을 왼쪽으로 5픽셀만 옮겨도 30~40%의 확률로 실패합니다. 상호작용 모델 전체가 일종의 '임시방편(hack)' 위에 구축되었기 때문입니다. 즉, 웹을 실제 구조화된 데이터(structured data)가 아닌 일련의 이미지 세트로 취급해 온 것이죠.

그러던 2026년 2월 12일, 구글은 크롬 146 카나리(Chrome 146 Canary) 버전에 모든 것을 바꿀 만한 기능을 조용히 출시했습니다. 바로 **WebMCP (웹 모델 컨텍스트 프로토콜, Web Model Context Protocol)**입니다. 지금 주의를 기울이지 않는다면 여러분은 뒤처지게 될 것입니다.

과장이 아닙니다. 이것은 CSS가 테이블 레이아웃(table layouts)을 대체했던 순간과 같습니다. 다만 이번에는 웹사이트를 더 예쁘게 만드는 것에 그치지 않고, 웹사이트를 '지능적'으로 만들고 있습니다.

## 웹은 AI를 위해 설계된 적이 없습니다 (그리고 우리는 그것을 받아들였을 뿐입니다)

현재 상황이 얼마나 터무니없는지 이야기해 봅시다.

오늘날 AI 에이전트가 웹사이트와 상호작용할 때, 기본적으로 몇 초마다 스크린샷을 찍어 GPT-4V나 제미나이 플래시(Gemini Flash) 같은 비전 모델(vision models)에 입력합니다. 스크린샷 한 장당 약 2,000개의 토큰이 소모됩니다. 여러 페이지에 걸친 단일 제품 검색을 하려면 이런 상호작용이 20~30번 필요할 수 있습니다. 계산해 보세요. 신발 한 켤레를 찾는 데 40,000~60,000개의 토큰이 드는 셈입니다.

느리고, 비싸며, CSS를 업데이트하는 순간 깨져버립니다.

대안적인 접근 방식은 **DOM 스크래핑(DOM scraping)**입니다. 에이전트가 전체 HTML 트리를 다운로드하고, 수천 개의 div 태그를 파싱하며, 리액트(React) 컴포넌트 구조를 이해하려 애쓰면서 클릭할 요소를 찾기를 기도하는 방식입니다. 이는 컨텍스트 윈도우(context window) 공간을 엄청나게 잡아먹으며, 현대적인 웹 앱이 너무나 동적이기 때문에 여전히 자주 실패합니다.

두 방식 모두 이를 위해 만들어진 것이 아닙니다. 웹은 인간이 버튼을 클릭하도록 설계되었지, AI 에이전트가 구조화된 작업을 실행하도록 설계되지 않았습니다. 우리는 사각형 구멍에 둥근 못을 억지로 끼워 맞추고 있었고, 모두가 그것을 당연하게 받아들였습니다.

지금까지는 말이죠.

## WebMCP의 등장: 마침내 웹이 에이전트와 대화하다

WebMCP는 이 재앙에 대한 구글의 해답입니다. 이는 마이크로소프트와 공동 개발한 W3C 표준으로, 웹사이트가 새로운 브라우저 API인 `navigator.modelContext`를 통해 구조화된 도구(tools)를 AI 에이전트에게 직접 노출할 수 있게 해줍니다.

제 생각을 바꾸게 된 결정적인 계기는 이것입니다. 백엔드 서버가 필요 없습니다. Node.js 프로세스를 띄우거나 도커(Docker) 컨테이너를 설정할 필요도 없습니다. 웹페이지 자체가 서버가 됩니다. 브라우저가 모든 것을 처리합니다.

기술적 변화는 엄청납니다. 스크린샷 기반 상호작용에 비해 토큰 효율성이 **89% 향상**됩니다. 토큰 비용은 작업당 수천 개에서 구조화된 JSON 응답당 20~100개 수준으로 떨어집니다. 정확도는 어떨까요? 초기 벤치마크에 따르면 기존에 30~40% 실패하던 작업에서 약 **98%의 성공률**을 보여줍니다.

이것은 점진적인 개선이 아니라 패러다임의 전환입니다.

## WebMCP 작동 방식 (기술적 심층 분석)

WebMCP는 두 가지 통합 경로를 제공하며, 두 가지 모두 기존 방식에 비해 놀라울 정도로 간단합니다.

### 1. 선언적 API (The Declarative API): 작동하는 HTML 속성

적절한 ARIA 레이블을 추가하는 것만으로 웹 접근성을 높일 수 있었던 때를 기억하시나요? WebMCP의 선언적 API도 같은 방식으로 작동합니다. 기존 HTML 폼(form)에 세 가지 속성을 추가하기만 하면 됩니다.

```html
<form  
  toolname="login"  
  tooldescription="이메일과 비밀번호로 애플리케이션에 로그인합니다"
  toolautosubmit="true">
  
  <label for="email">이메일</label>
  <input id="email" name="email" type="email" required>
  
  <label for="password">비밀번호</label>
  <input id="password" name="password" type="password" required>
  
  <button type="submit">로그인</button>
</form>
```

이게 전부입니다. 말 그대로 통합의 끝입니다. 이제 WebMCP를 지원하는 모든 AI 에이전트는 이 도구를 발견하고 직접 호출할 수 있습니다.

*   **toolname**: 함수 이름 (예: "login", "searchProducts", "bookFlight")
*   **tooldescription**: 도구가 하는 일에 대한 자연어 설명
*   **toolautosubmit**: 에이전트가 자동으로 제출할지, 아니면 양식만 채울지 여부

이미 폼 구조가 잘 잡혀 있고 적절한 name 속성과 유효성 검사 패턴이 있다면, 작업의 80%는 끝난 셈입니다. 이 세 가지 속성만 추가하면 웹사이트 전체를 AI가 접근 가능하게 만들 수 있습니다.

### 2. 명령형 API (The Imperative API): 복잡한 워크플로우를 위한 자바스크립트

단순한 폼을 넘어선 동적인 상호작용을 위해, WebMCP는 `navigator.modelContext.registerTool()`을 통한 명령형 API를 제공합니다. 이커머스 사이트의 실제 예시를 보겠습니다.

```javascript
navigator.modelContext.registerTool({
  name: "addToCart",
  description: "SKU와 수량을 입력하여 장바구니에 제품을 추가합니다",
  inputSchema: {
    type: "object",
    properties: {
      sku: {  
        type: "string",  
        pattern: "^[A-Z]{3}-\\d{6}$",
        description: "제품 SKU 코드"
      },
      quantity: {  
        type: "number",  
        minimum: 1,
        maximum: 99,
        description: "추가할 수량"
      }
    },
    required: ["sku", "quantity"]
  },
  async execute(args) {
    // 기존의 자바스크립트 로직 재사용
    const cart = getShoppingCart();
    cart.addItem(args.sku, args.quantity);
  
    return {
      content: [{
        type: "text",
        text: `장바구니에 ${args.sku} 제품 ${args.quantity}개를 추가했습니다. 총 아이템 수: ${cart.itemCount}`
      }]
    };
  }
});
```

이 지점에서 강력한 위력을 발휘합니다. 기존의 프론트엔드 코드를 그대로 재사용하는 것입니다. UI 버튼을 작동시키는 동일한 함수가 이제 AI 에이전트의 상호작용도 지원하게 됩니다. 로직을 중복해서 짤 필요도 없고, 별도의 API 인터페이스를 유지 관리할 필요도 없습니다.

또한 `inputSchema`는 JSON 스키마(JSON Schema) 유효성 검사를 사용하므로, 타입 안정성(type safety)과 적절한 오류 처리를 자동으로 얻을 수 있습니다. 에이전트는 잘못된 데이터를 전달할 수 없습니다. 브라우저가 여러분의 코드가 실행되기도 전에 스키마를 강제하기 때문입니다.

## 실제 세계에서의 통합: 무엇이 필요한가

실제 애플리케이션에 WebMCP를 통합하는 과정을 살펴보겠습니다. 지난주 비행기 예약 사이트에 적용해 보았는데, 1시간도 채 걸리지 않았습니다.

### 1단계: 크롬에서 WebMCP 활성화
현재 WebMCP는 초기 프리뷰 상태입니다. 실험적 플래그가 활성화된 **Chrome 146 Canary**가 필요합니다.
*   Chrome Canary 다운로드
*   `chrome://flags`로 이동
*   “Experimental Web Platform Features” 검색
*   활성화(Enable) 후 재시작

### 2단계: 폴리필(Polyfill) 추가
네이티브 브라우저 지원이 널리 퍼지기 전까지는 폴리필을 설치하세요.
`npm install @mcp-b/global`

그 후 메인 자바스크립트 파일 상단에 임포트합니다.
`import '@mcp-b/global';`

### 3단계: 도구 등록
가장 중요한 사용자 흐름부터 시작하세요. 비행기 예약 사이트의 경우 세 가지 도구를 등록했습니다 (예: 비행기 검색, 세부 정보 확인, 예약 등).

### 4단계: AI 에이전트 연결
WebMCP 브라우저 확장 프로그램을 설치한 후, 클로드 데스크톱(Claude Desktop)이나 다른 MCP 호환 AI 클라이언트를 연결합니다. 이제 클로드에게 "다음 주 화요일 JFK에서 LAX로 가는 가장 저렴한 비행기를 찾아줘"라고 말하면, 클로드는 UI를 클릭하려 애쓰는 대신 여러분이 등록한 `searchFlights` 도구를 직접 찾아 호출합니다.

## 이것이 생각보다 더 중요한 이유

WebMCP는 단순한 기술 업그레이드가 아닙니다. 웹에서 가능한 일의 근본을 바꿉니다.

*   **개발자에게**: 더 이상 별도의 API를 유지 관리할 필요가 없습니다. 프론트엔드 코드가 곧 API가 됩니다. UI를 바꾸면 도구도 자동으로 업데이트됩니다. 유효성 검사를 추가하면 인간과 AI 에이전트 모두에게 적용됩니다. 이것이 논리적 결론에 도달한 **DRY(Don't Repeat Yourself)** 원칙입니다.
*   **사용자에게**: AI 에이전트가 마침내 복잡한 다단계 작업을 안정적으로 수행할 수 있게 됩니다. 한 번의 대화로 비행기, 호텔, 렌터카를 예약할 수 있습니다. 15개의 탭을 열지 않고도 여러 사이트의 제품을 비교할 수 있습니다.
*   **비즈니스에**: 사이트를 에이전트 친화적으로 만드는 첫 번째 표준화된 방법입니다. 구글과 마이크로소프트가 이를 추진하고 있으며 W3C가 표준화하고 있습니다. 에이전트 중심의 웹을 준비하지 않는다면, 2012년에 모바일 대응을 하지 않았던 사이트들처럼 뒤처지게 될 것입니다.

## WebMCP vs. Anthropic의 MCP: 차이점 이해하기

"이것이 앤스로픽(Anthropic)의 모델 컨텍스트 프로토콜(Model Context Protocol)과 같은 것인가요?"라는 질문을 많이 받습니다.

아닙니다. 둘은 완전히 다른 문제를 해결합니다.

*   **Anthropic의 MCP**: 백엔드 프로토콜입니다. JSON-RPC를 통해 AI 플랫폼을 서버 측 서비스에 연결합니다. AI 에이전트가 데이터베이스, 내부 도구, 기업 시스템에 액세스하기 위한 API 게이트웨이라고 생각하면 됩니다. 서버 인프라와 인증 레이어가 필요합니다.
*   **WebMCP**: 브라우저 내에서 완전히 작동합니다. 클라이언트 측(client-side)이며 기존 프론트엔드 코드와 작동하고 브라우저의 보안 모델(동일 출처 정책, 사용자 권한)을 활용합니다. 백엔드 인프라가 필요 없으며, 웹페이지 자체가 서버 역할을 합니다.

둘은 경쟁 관계가 아니라 보완 관계입니다.

## 모두가 물어야 할 보안 문제

"잠깐, 그럼 어떤 AI 에이전트든 내 웹사이트에서 함수를 실행할 수 있다는 건가요?"

아닙니다. WebMCP는 설계 단계부터 **권한 우선(permission-first)** 방식을 따릅니다.

브라우저가 보안 중재자 역할을 합니다. 에이전트가 도구를 실행하기 전에 크롬은 사용자에게 묻습니다. "AI가 이 비행기를 예약하도록 허용하시겠습니까?" 인간이 통제권을 갖습니다. 이는 자율적인 자동화가 아니라 명백한 **휴먼 인 더 루프(human-in-the-loop, 인간 참여형)** 프로토콜입니다.

도구들은 해당 페이지의 출처(origin) 보안 경계를 상속받습니다. 동일 출처 정책(same-origin policy)에 의해 교차 출처 공격이 방지됩니다. 도구에 인증이 필요한 경우 사용자의 기존 세션(쿠키, 토큰 등)을 그대로 사용합니다.

## 결론: 오늘 WebMCP를 구현해야 할까요?

제 솔직한 의견은 '예'입니다. 사용자 대면 서비스를 만들고 있다면 더욱 그렇습니다.

구현 비용은 최소화됩니다. 기존 폼에 `toolname`과 `tooldescription`을 추가하는 데는 몇 분이면 충분합니다. 기존 자바스크립트 함수를 `registerTool()`로 감싸는 것도 간단합니다.

반면 얻을 수 있는 이점은 거대합니다. 준비 여부와 상관없이 다가올 에이전트 중심의 웹을 대비해 사이트를 미래 지향적으로 만들 수 있습니다.

웹은 방금 그토록 필요했던 API 레이어를 갖게 되었습니다. 모든 웹사이트는 이제 API가 될 것입니다. REST 엔드포인트나 GraphQL 서버를 통해서가 아니라, 기존 프론트엔드 코드와 함께 작동하는 브라우저 네이티브 도구 노출을 통해서 말이죠.
