로컬 AI 에이전트 구축: 모델, 메모리, 오케스트레이션 실전 가이드

로컬 AI 에이전트(Local AI agent)는 모델이 사용자 하드웨어에서 실행되고, 사용자를 대신하여 작업을 수행하며, 외부 API로 데이터를 전송하지 않고도 세션 간 컨텍스트를 유지하는 시스템입니다. 단순히 응답하고 잊어버리는 챗봇과 달리, 에이전트는 여러 단계에 걸쳐 추론하고, 도구를 사용하며, 시간이 지남에 따라 지식을 축적할 수 있습니다. 로컬에서 실행하면 클라우드 기반 대안과 관련된 개인 정보 보호 문제나 API 비용 없이 지능형 비서의 이점을 얻을 수 있습니다.

로컬 AI 에이전트를 구축하려면 다섯 가지 계층이 필요합니다. 사용자 하드웨어에서 추론(inference)을 실행하는 LLM 계층, 작업을 라우팅하고 실행하는 에이전트 프레임워크, 시간이 지남에 따라 에이전트를 더 똑똑하게 만드는 메모리 계층, 학습한 내용을 지속시키는 저장 계층, 그리고 실제 작업 방식과 연결하는 인터페이스 계층입니다.

이 글에서는 각 계층을 살펴보고, 옵션을 비교하며, 트레이드오프(trade-off)를 설명하여 실제 사용 사례에 맞는 스택을 구축할 수 있도록 돕습니다.

### 계층 1: LLM 계층

이 계층은 엔진에 해당합니다. 모델은 의도 감지(intent detection), 코드 생성(code generation), 그리고 스택의 나머지 부분이 의존하는 모든 구조화된 데이터 추출(structured data extraction)을 처리합니다. 다른 모든 구성 요소는 모델이 안정적으로 수행할 수 있는 기능을 중심으로 구축된 연결 파이프라인입니다.

로컬 추론(inference) 분야를 지배하는 세 가지 주요 도구는 다음과 같습니다.

*   **Ollama**: 로컬 추론을 위한 가장 쉬운 진입점입니다. 하나의 명령으로 설치하고, 하나의 명령으로 모델을 가져오면 즉시 로컬호스트(localhost)에서 OpenAI 호환 API를 실행할 수 있습니다. 또한 별도 구성 없이 모델 관리, 양자화(quantization) 선택, 컨텍스트 윈도우(context window)를 처리합니다. 하지만 이는 프로덕션 트래픽(production traffic)보다는 개인 비서나 개발자 도구에 더 적합합니다.

    ```bash
    ollama pull qwen3:8b
    ollama pull nomic-embed-text
    ```

*   **Llama.cpp**: Ollama보다 한 단계 낮은 수준의 라이브러리입니다. 데몬(daemon)이나 HTTP 계층 없이 양자화된 GGUF 모델을 직접 실행하는 C++ 추론 엔진입니다. Llama.cpp는 메모리 사용량(memory footprint)이 최소화되어 라즈베리 파이(Raspberry Pi), 임베디드 리눅스(embedded Linux)와 같이 Ollama가 너무 무거운 엣지 배포(edge deployment)에 적합합니다. 단점은 내장된 모델 관리가 없고 더 많은 수동 구성이 필요하다는 것입니다.

*   **vLLM**: 이 옵션은 프로덕션(production) 환경에 적합합니다. 클라우드 추론 API에서 대규모로 사용되는 PagedAttention 및 연속 배치(continuous batching) 기술을 구현합니다. 여러 사용자에게 서비스를 제공하거나 실제 처리량(throughput) 요구 사항이 있는 시스템을 구축하는 경우 vLLM이 적합한 계층입니다. 적절한 GPU 인프라가 필요하지만, Ollama 또는 Llama.cpp보다 더 잘 확장됩니다.

실제로는 단일 개발자를 위한 로컬 비서의 경우, 단순성이 처리량보다 중요한 상황에서는 Ollama가 적절한 기본 설정입니다. 특히 `qwen3:8b` 모델은 의도 감지를 위한 신뢰할 수 있는 JSON을 생성하고, 코드를 깔끔하게 생성하며, 8GB VRAM에 적합하다는 점에서 주목할 만합니다. 한 가지 주의할 점은 실제 콘텐츠를 반환하기 전에 출력을 `<think>…</think>` 블록으로 래핑하여 원시 응답을 처리하는 다운스트림(downstream) JSON 파서에 문제를 일으킬 수 있다는 것입니다. 이를 해결하기 위한 작은 인터셉트(intercept)로 `think: False`를 전역적으로 주입할 수 있습니다.

```python
_orig_ollama_chat = ollama.chat
def _no_think_chat(*args, **kwargs):
    opts = kwargs.get("options") or {}
    if isinstance(opts, dict):
        opts.setdefault("think", False)
        kwargs["options"] = opts
    return _orig_ollama_chat(*args, **kwargs)
ollama.chat = _no_think_chat
```

`setdefault`를 사용하면 개별 호출자는 필요할 때 명시적으로 사고 모드(thinking mode)를 선택할 수 있습니다. 도구 사용 파이프라인에서 JSON 추출 및 코드 생성의 경우, 사고 모드는 출력 품질을 개선하지 않으면서 지연 시간만 추가합니다.

### 계층 2: 에이전트 프레임워크 계층

LLM 계층은 텍스트 생성을 담당합니다. 반면 에이전트 프레임워크는 생성된 텍스트로 무엇을 할지, 그리고 다음에 무엇을 할지 결정합니다. 이 계층은 의도 분류(intent classification), 액션 라우팅(action routing), 실행을 처리합니다.

*   **OpenClaw**: 도구 우선(tool-first) 접근 방식이며 파일 시스템 네이티브(filesystem-native)입니다. 모든 에이전트 액션은 파일 생성, 코드 작성, 폴더 열기, 쉘 명령어 실행과 같이 선언된 스킬(skill)로 디스패치되는 명령입니다. 스킬의 이름과 디스패치 메서드를 선언하는 `SKILL.md` 파일을 작성하면 OpenClaw가 라우팅을 처리합니다.

    ```
    ---
    name: local-ai-assistant
    description: Local AI coding assistant with persistent memory
    command-dispatch: tool
    command-tool: exec
    command-arg-mode: raw
    ---
    ```

    대부분의 작업이 파일 I/O인 코딩 비서의 경우 이 모델이 자연스럽게 맞습니다. 한계는 복잡한 다단계 추론 체인(multi-step reasoning chains)이나 여러 에이전트 간의 조정을 위해 설계되지 않았다는 점입니다.

*   **LangGraph**: 정반대의 접근 방식을 취합니다. 노드(node)가 LLM 호출 또는 액션이고, 엣지(edge)가 전환인 명시적인 상태 그래프(state graph)를 정의하며, 프레임워크는 단계 간 상태를 관리합니다. 단점은 비즈니스 로직을 작성하기 전에 상태 스키마(state schema)와 그래프 토폴로지(graph topology)를 정의해야 하는 선행 설계 작업입니다. 간단한 도구 사용 작업에는 과도한 설계(overengineering)이지만, 복잡한 다단계 파이프라인에는 각 단계에서 발생한 일을 추적하고 검사할 수 있는 기록을 제공합니다.

*   **CrewAI**: 역할(role)을 가진 에이전트와 설명이 있는 작업(task)을 정의함으로써 한 단계 더 높은 수준으로 추상화하며, 프레임워크가 실행 순서와 에이전트 간 통신을 결정합니다. 이는 작동하는 다중 에이전트 프로토타입(multi-agent prototype)을 구축하는 가장 빠른 방법입니다. 단점은 추상화로 인해 실패 시 디버깅이 더 어려워진다는 것입니다.

실제로는 OpenClaw의 `command-dispatch: tool`과 `command-tool: exec`는 스킬이 호출될 때 파이썬 스크립트로 쉘을 벗어나게 하여 파일 작업이 프레임워크를 통해 흐르도록 합니다. 어떤 프레임워크를 선택하든 의도 분류(intent classification)는 모든 수신 메시지에서 가장 먼저 실행되는 작업입니다. LLM 분류와 정규식(regex) 폴백(fallback)으로 구성된 2단계 시스템은 단독 접근 방식보다 실패 모드를 더 잘 처리합니다.

```python
def detect_intent(message: str) -> dict:
    try:
        resp = ollama.chat(
            model=OLLAMA_CHAT_MODEL,
            messages=[{"role": "user", "content": INTENT_PROMPT.format(message=message)}],
            options={"temperature": 0, "num_predict": 1024},
        )
        return extract_json(resp["message"]["content"])
    except Exception:
        return keyword_intent_fallback(message)
```

정규식 폴백은 LLM이 잘못된 형식의 출력(malformed output)을 반환할 때 일반적인 패턴을 포착합니다. 사고 모드를 비활성화하고 온도를 0으로 설정하면 LLM 경로는 대부분의 요청을 처리하지만, 폴백이 에이전트 기능을 유지하는 역할을 합니다.

### 계층 3: 메모리 계층

메모리는 당신에게 적응하는 에이전트와 모든 대화를 처음처럼 대하는 에이전트를 구분하는 요소입니다. 메모리는 사용자의 관례, 선호도, 프로젝트 컨텍스트를 학습하고 요청 없이 적용하는 데 도움을 줍니다.

모든 메모리에서 핵심 문제는 인컨텍스트 메모리(in-context memory)에 있습니다. 일반적인 해결책으로는 시작 시 로드되는 `PREFERENCES.md`, 긴 시스템 프롬프트(system prompt), `CONTEXT.md` 파일 등이 있습니다. 이 모든 것에는 공통적인 결함이 있는데, 바로 컨텍스트 윈도우(context window) 내에 존재하여 컨텍스트 압축(context compaction), 토큰 제한(token limits), 세션 재시작(session restarts)에 의해 억제될 수 있다는 점입니다.

하지만 에이전트에게 실제로 필요한 것은 컨텍스트 윈도우 외부에 존재하며, 영구적으로 저장되고(durably stored), 관련성 있을 때 의미적으로 검색되며(semantically retrieved), 사실이 변경될 때 업데이트되는 메모리입니다.

다음은 다양한 사용 사례에 유용한 일반적인 메모리 접근 방식입니다.

*   **기본 RAG (검색 증강 생성 - Retrieval-Augmented Generation)**: 가장 일반적이고 완전히 초보자 친화적인 접근 방식입니다. 기본 RAG 흐름은 문서나 대화 기록을 청킹(chunking)하고, 임베딩(embedding)하며, 쿼리 시점에 최소한의 설정으로 상위 k개의 가장 유사한 청크를 검색하는 것을 포함합니다. 한계점은 메모리를 문서 저장소로 취급하여 장기적으로 보관할 가치가 있는 것과 일회성 쿼리를 구분하지 않는다는 것입니다.

*   **세션 범위 메모리 (Session-scoped memory) (LangChain의 `ConversationBufferMemory`, `ConversationSummaryMemory` 등)**: 세션 내 일관성을 해결하여 에이전트가 다섯 메시지 전에 말한 것을 잊지 않게 하며, 요약(summaries)을 통해 컨텍스트 윈도우가 가득 차는 것을 방지합니다. 하지만 세션 재시작 후에는 유지되지 않습니다. 코딩 비서의 경우 이것이 핵심적인 문제입니다.

*   **구조화된 영구 메모리 (Structured persistent memory) (Mem0)**: 메모리를 원시 텍스트(raw text)가 아닌 일급 데이터(first-class data)로 취급합니다. 메시지를 있는 그대로 저장하는 대신, 시스템은 LLM을 사용하여 개별 사실(discrete facts)을 추출한 다음, 해당 사실들을 벡터 데이터베이스(vector database)에 임베딩하고 저장합니다. 검색 시에는 쿼리가 임베딩되고, 현재 메시지가 사실이 저장될 때와 같은 단어를 사용하지 않더라도 의미적으로 유사한 사실이 반환됩니다. 사실은 세션 전반에 걸쳐 지속되며, 시간이 지남에 따라 변경될 때 중복 제거되고 업데이트될 수 있습니다. Mem0와 같은 라이브러리는 로컬 모델 지원을 통해 이 패턴을 구현하여 Ollama 기반 스택에 쉽게 통합할 수 있습니다.

실제로, 기본 RAG 시스템에 `pytest`를 선호한다고 말하면 원시 메시지를 저장합니다. Mem0와 같은 구조화된 메모리 시스템은 개별 선호도를 추출하여 저장합니다. 다음에 테스트를 포함한 함수를 요청할 때, 현재 메시지에 테스트 프레임워크에 대한 언급이 없더라도 메모리는 "사용자가 pytest를 선호함"을 반환합니다.

Mem0를 로컬 Ollama 설정에 통합하는 데는 작은 설정 스니펫만 필요합니다. 로컬 LLM, 로컬 임베딩 모델(embedding model), 선택한 벡터 저장소(vector store)를 가리키기만 하면 됩니다.

```python
from mem0 import Memory
config = {
    "llm": {
        "provider": "ollama",
        "config": {"model": "qwen3:8b", "ollama_base_url": "http://localhost:11434"},
    },
    "embedder": {
        "provider": "ollama",
        "config": {"model": "nomic-embed-text", "ollama_base_url": "http://localhost:11434"},
    },
    "vector_store": {
        "provider": "qdrant",
        "config": {"host": "localhost", "port": 6333, "embedding_model_dims": 768},
    },
}
memory = Memory.from_config(config)
memory.add("I always use type hints and pytest", user_id="dev")
results = memory.search("write a utility function", user_id="dev")
```

위 `llm` 블록은 Mem0가 `qwen3:8b`를 사용하여 원시 메시지를 있는 그대로의 텍스트가 아닌 개별 저장된 선호도로 변환하는 사실 추출을 수행하도록 지정합니다. `embedder` 블록은 `nomic-embed-text`를 사용하여 해당 사실들을 쓰기(write) 시점에 벡터로 변환하고 검색(retrieval) 시점에 쿼리를 임베딩합니다. 마지막으로 `memory.search()`는 키워드 일치가 아닌 의미에 따라 콘텐츠를 검색합니다.

메모리 접근 방식과 관계없이 과소평가되는 한 가지 개선 사항은 데이터를 저장하기 전에 필터링하는 것입니다. 모든 쓰기 작업 전에 실행되는 경량 분류기(lightweight classifier)를 선택하여 메모리 계층을 깨끗하게 유지하고 검색 품질을 높일 수 있습니다. 다음은 이를 수행하는 코드 스니펫입니다.

```python
def _is_worth_storing(self, user_message: str) -> bool:
    response = ollama.chat(
        model=OLLAMA_CHAT_MODEL,
        messages=[{"role": "user", "content": SMART_MEMORY_PROMPT.format(
            user_message=user_message
        )}],
        options={"temperature": 0, "num_predict": 512},
    )
    data = self._extract_json_robust(response["message"]["content"])
    return bool(data.get("worth_storing", False))
```

*   "재시도 데코레이터(retry decorator)를 작성해 주세요."
*   "감사합니다."

이 필터가 없으면 벡터 데이터베이스는 시간이 지남에 따라 검색 품질을 희석시키는 일회성 요청으로 가득 차게 됩니다.

### 계층 4: 저장 계층

메모리 계층은 무엇을 기억할지 결정합니다. 저장 계층은 이러한 메모리가 실제로 디스크에 저장되는 곳입니다.

*   **FAISS**: 빠르고 잘 테스트되었으며 인프라 종속성(infrastructure dependencies)이 없는 인-프로세스 라이브러리(in-process library)입니다. 따라서 파이썬 프로세스에 직접 임베드하여 메모리에서 실행하거나 디스크에 직렬화(serialize)할 수 있습니다. 유일한 한계는 운영적인 측면인데, 영속성(persistence)을 위해 명시적인 직렬화 호출이 필요하며, HTTP API나 내장된 복제(replication) 기능이 없습니다. 연구 파이프라인(research pipelines)과 배치 워크로드(batch workloads)에 유용하지만, 자주 재시작되는 장기 실행 비서에게는 더 취약합니다.

*   **Chroma**: 개발자를 위한 가장 간단한 벡터 데이터베이스입니다. 선택적인 임베디드 모드(embedded mode)와 함께 로컬 파이썬 프로세스로 실행됩니다. 데이터는 자동으로 로컬 디렉토리에 지속됩니다. 설치하려면 다음 명령을 실행하십시오.

    ```bash
    pip install chromadb
    ```

    수천 개의 저장된 사실(facts)을 가진 개인 비서의 경우 Chroma는 충분하고도 남습니다. Qdrant에 비해 필터링 기능이 더 제한적이라는 단점이 있지만, 인프라 종속성을 완전히 제거합니다.

*   **Qdrant**: 정식 서비스로 실행되며, 완전한 REST 및 gRPC API를 노출하고 페이로드 필터링(payload filtering) 및 명명된 벡터(named vectors)를 지원합니다. Qdrant는 수백만 개의 벡터와 동시 쿼리(concurrent queries)를 처리하는 배포 환경을 위해 설계되었습니다. 별도의 서비스로 실행되며, 디스크에 기본적으로 지속되고, 특별한 처리 없이 도커(Docker) 재시작을 견뎌냅니다.

    ```bash
    # Qdrant via Docker
    docker run -d - name qdrant-local -p 6333:6333 \
    -v $(pwd)/qdrant_storage:/qdrant/storage qdrant/qdrant
    ```

한 가지 사소하지만 문제를 일으킬 수 있는 설정 세부 사항은 벡터 저장소(vector store) 구성의 임베딩 차원(embedding dimension)이 임베딩 모델의 출력 차원과 정확히 일치해야 한다는 것입니다. 예를 들어, `nomic-embed-text` 모델은 768차원 벡터를 생성합니다. 이 값을 업데이트하지 않고 다른 임베딩 모델로 변경하면 모든 삽입 작업이 조용히 실패합니다.

```json
"vector_store": {
    "provider": "qdrant",
    "config": {
        "collection_name": "coding_assistant",
        "host": "localhost",
        "port": 6333,
        "embedding_model_dims": 768,  # must match your embedding model exactly
    },
}
```

도커(Docker) 종속성을 제거하고 싶다면 Chroma로 직접 교체할 수 있습니다. 구성에서 `vector_store` 제공자(provider)만 변경하면 나머지 스택은 동일하게 유지됩니다.

### 계층 5: 인터페이스 계층

이 계층은 스택의 마지막이자 가장 바깥쪽 계층으로, 사용자가 에이전트와 상호작용하는 방식을 결정합니다. 적절한 인터페이스는 에이전트가 기존 워크플로우에 어떻게 통합되는지에 전적으로 달려 있습니다. 가장 일반적으로 사용되는 인터페이스는 다음과 같습니다.

*   **웹 UI (Web UI)**: 로컬 에이전트에게 가장 유연한 인터페이스입니다. 브라우저에서 실행되며, 마크다운(markdown) 형식, 구문 강조된 코드(syntax-highlighted code), 축소 가능한 섹션(collapsible sections)을 지원하고 설치 없이 다양한 운영 체제에서 작동합니다. 코딩 비서의 경우, 웹 UI는 현재 파일, 생성된 코드, 에이전트의 메모리 상태를 한 화면에서 나란히 표시할 수 있습니다.

*   **CLI (명령줄 인터페이스)**: 쉘(shell) 워크플로우에 자연스럽게 통합되고, 출력을 다른 도구로 파이프(pipe)할 수 있으며, 시각적 오버헤드(visual overhead)가 없는 터미널(terminal) 비서입니다. 단점은 풍부한 출력(rich output)을 위해서는 추가적인 주의가 필요하며, 다중 턴 대화(multi-turn conversations)가 채팅 UI보다 어색하다는 점입니다. CLI는 빠른 조회, 파일 작업, 그리고 기존 터미널 세션 내에서 비서를 사용하고자 하는 스크립트 기반 워크플로우에 가장 적합합니다.

*   **IDE 플러그인 (IDE plugins)**: VS Code 또는 JetBrains에 내장된 비서 역할을 하며, 편집 중인 파일 옆에 존재하며, 어떤 파일이 열려 있는지 확인하고 인라인(inline)으로 편집을 제안합니다. 단점은 IDE 확장(extension)이 특정 확장 API에 대한 지식과 IDE 업데이트에 따른 지속적인 유지보수를 요구한다는 점입니다. 매일 사용되는 팀 도구로는 잠재적으로 가치가 있습니다.

*   **채팅 앱 (Chat apps) (Slack 봇, Discord 봇, Telegram 통합 등)**: 비서가 여러 장치에서 접근 가능하거나 팀 전체에서 공유되어야 할 때 채팅 앱이 유용합니다. 여러 장치에서 또는 이동 중에 에이전트에 접근하고 싶을 때 잘 작동합니다. 특히 코딩 비서의 경우 채팅 인터페이스는 코드 검토나 파일 편집에 적합하지 않다는 한계가 있지만, Q&A, 상태 확인, 경량 요청에 가장 적합합니다.

### 마무리하며

각 계층은 주변 계층을 제약하는 결정을 내립니다. LLM 선택은 에이전트 프레임워크가 호출하는 API 형태를 결정합니다. 메모리 접근 방식은 저장 계층이 무엇을 지원해야 하는지 결정합니다. 인터페이스 선택은 출력 형식(output formatting)이 얼마나 중요한지 결정합니다.

단일 개발자를 위한 로컬 코딩 비서의 경우, 각 계층을 교체 가능하게 유지하면서 마찰을 최소화하는 스택은 다음과 같습니다.

*   인터페이스: 로컬호스트(localhost) 기반 웹 UI
*   프레임워크: OpenClaw - 도구 디스패치(tool-dispatch), SKILL.md 라우팅
*   LLM: `qwen3:8b`용 Ollama (JSON 신뢰성을 위해 사고 모드(think mode) 비활성화)
*   메모리: 값 필터링을 통한 구조화된 영구 메모리(structured persistent memory), `nomic-embed-text`를 통한 임베딩
*   저장소: Chroma (인프라 불필요) 또는 Qdrant (더 견고함)

각 계층은 독립적으로 교체 가능하므로, 각 수준에서 가장 간단한 옵션으로 시작하여 요구 사항이 증가함에 따라 개별 계층을 업그레이드할 수 있습니다. Chroma를 Qdrant로, 또는 Ollama를 vLLM으로 전환해도 그 위나 아래의 어떤 것도 다시 구축할 필요가 없습니다.

다른 무엇보다 먼저 투자할 가치가 있는 계층은 메모리입니다. 잘 설계된 영구 메모리 계층을 가진 적당한 로컬 모델은 매 세션을 처음부터 시작하는 더 큰 모델보다 꾸준히 더 나은 성능을 보입니다.

이 스택을 엔드투엔드(end-to-end)로 실행하고 싶다면, 로컬 AI 에이전트 파이프라인을 실행하는 전체 작동 소스 코드는 다음에서 확인할 수 있습니다: https://github.com/AashiDutt/OpenClaw_Mem0_Oll
