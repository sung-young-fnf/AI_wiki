# Why Your AI Agent Keeps Failing: A Deep Dive into Harness Engineering

OpenAI는 5개월 동안 Codex를 활용해 사람이 단 한 줄의 코드도 작성하지 않고 100만 줄의 프로덕션 코드를 배포했습니다. 이 성과 뒤에 숨겨진 엔지니어링 원리를 살펴봅니다.

2026년 2월, OpenAI는 엔지니어링 팀이 5개월간 1,500개의 풀 리퀘스트(Pull Request)를 처리하며 100% Codex로 코드를 작성한 사례를 공개했습니다. 이 글에서 가장 인상적인 문장은 다음과 같았습니다.

> "엔지니어링 팀의 주요 업무는 에이전트가 유용한 작업을 수행할 수 있도록 지원하는 것으로 변모했다."

비슷한 시기, LangChain 엔지니어들이 수행한 실험에 따르면 동일한 모델(gpt-5.2-codex)을 사용하더라도 주변 시스템을 어떻게 구성하느냐에 따라 벤치마크 점수가 52.8%에서 66.5%로 상승했습니다. 이러한 13.7%포인트의 성능 향상은 모델이 아닌 **하네스(Harness)**의 변경만으로 이뤄졌습니다.

**에이전트 = 모델 + 하네스(Agent = Model + Harness).** 모델이 성능의 천장을 결정한다면, 하네스는 그 천장에 얼마나 가까이 도달할지를 결정합니다. 이 글은 에이전트 시스템이 실패하는 구체적인 이유와 이를 해결하기 위한 하네스 구성 요소에 대해 다룹니다.

## 4가지 구조적 실패 모드

이것들은 단순한 버그가 아니라 대규모 언어 모델(LLM)의 구조적 특성입니다.

1.  **세션 간 상태 보존 실패:** LLM은 영구적인 메모리가 없습니다. 복잡한 프로젝트에서 다음 세션은 이전 세션이 무엇을 했는지 알지 못하며, 작업을 처음부터 다시 시작하거나 이미 완료된 일을 반복합니다.
2.  **한 번에 해결하려는 욕심(One-shot Greed):** 에이전트는 전체 목표를 한 번에 끝내려 합니다. 이로 인해 작업 중간에 컨텍스트 창(Context window)이 가득 차버리고, 코드베이스는 불안정한 미완성 상태로 남게 됩니다.
3.  **성급한 완료 판단(Premature Completion):** 에이전트는 작업이 끝나지 않았음에도 완료되었다고 주장합니다. 이는 기만이 아니라 신뢰할 수 없는 자기 평가 메커니즘 때문입니다. 코드가 작성된 것과 기능이 실제로 작동하는 것을 구분하지 못하는 것입니다.
4.  **파멸의 루프(The Doom Loop):** 어려운 문제에 직면하면 에이전트는 약간의 변형만 준 채 같은 방식을 반복합니다. LLM의 추론은 어려움에 처했을 때 뒤로 물러나 재평가하기보다 유사한 솔루션 공간 내에 머무는 경향이 있습니다.

## 하네스의 5가지 핵심 구성 요소

### 1. 가독성 있는 환경 (Readable Environment)
LLM은 상태가 없으므로(Stateless), 모든 상태는 외부 환경에 저장되어야 합니다. OpenAI 팀은 모든 내용을 하나의 거대한 `AGENTS.md` 파일에 담으려 했으나, 정보 오염과 유지보수의 어려움에 직면했습니다. 해결책은 정보 축적이 아닌 **정보 설계(Information Architecture)**였습니다.

`AGENTS.md`를 목차로 삼고, 제품 사양, 설계 문서, 실행 계획, DB 스키마 등을 별도 문서로 분리하여 필요할 때만 로드하는 방식입니다. 이를 통해 에이전트는 현재 작업에 필요한 맥락만 정확히 파악할 수 있습니다.

### 2. 작업 상태 머신 (Task State Machine)
상태 손실과 성급한 완료 문제를 해결하기 위해 작업 관리를 JSON 상태 머신으로 외부화합니다. 여기서 핵심은 상태의 기본값을 `pending`(대기)이 아닌 **`fail`(실패)**로 설정하는 것입니다. 에이전트의 임무는 단순히 작업을 완료하는 것이 아니라, 작업이 통과되었음을 증명하는 것이 됩니다.

프로젝트를 수백 개의 작은 작업으로 쪼개고, 각 작업에 기계가 읽을 수 있는 **수락 기준(Acceptance Criteria)**을 부여합니다. 이를 통해 새로운 에이전트 세션이 시작되어도 30초 이내에 프로젝트 상태를 재구성하고 작업을 이어갈 수 있습니다.

### 3. 검증 루프 (Verification Loop)
에이전트는 자신이 작성한 코드가 있는 동일한 컨텍스트 내에서 스스로를 평가할 때 편향되기 쉽습니다. LangChain은 `PreCompletionChecklistMiddleware`를 통해 에이전트가 완료를 선언하기 전 반드시 테스트 통과 여부와 회귀 테스트(Regression test)를 확인하도록 강제합니다.

OpenAI는 에이전트 런타임에 Chrome DevTools 프로토콜을 통합하여, 수정 전후의 비디오 증거를 제출하게 함으로써 텍스트 설명을 넘어선 확실한 검증 체계를 구축했습니다.

### 4. 아키텍처 강제 (Architecture Enforcement)
에이전트는 코드베이스의 패턴을 그대로 복제합니다. 기술 부채가 있는 코드라면 부채까지 복제합니다. 이를 방지하려면 아키텍처 제약 사항을 문서가 아닌 **도구(Tooling)**에 인코딩해야 합니다.

OpenAI는 커스텀 린터(Linter)와 구조적 테스트를 통해 의존성 방향이나 명명 규칙 위반 시 커밋(Commit) 자체를 차단합니다. Go나 TypeScript와 같은 강한 타입 시스템(Strong typing)을 활용하면 상당수의 오류를 컴파일 단계에서 걸러낼 수 있습니다.

### 5. 루프 감지 (Loop Detection)
에이전트가 동일한 파일에 대해 반복적인 수정을 시도하며 정체될 때 이를 감지하고 개입해야 합니다. `LoopDetectionMiddleware`는 파일 편집 횟수가 임계치를 넘으면 에이전트에게 다른 접근 방식을 고려하거나 도움을 요청하도록 개입하여 토큰 낭비와 지연을 방지합니다.

## 요약

모델 자체가 병목 현상은 아닙니다. 에이전트의 실패는 LLM의 구조적 한계에서 기인하며, 이는 더 나은 프롬프트가 아닌 **엔지니어링된 하네스**로 해결해야 합니다.

*   상태를 정보 설계 기반으로 외부화할 것.
*   작업 상태를 기계가 읽을 수 있는 형태로 관리하고 기본값을 '실패'로 둘 것.
*   실제 증거를 바탕으로 한 검증 루프를 구축할 것.
*   아키텍처 제약을 도구화하여 강제할 것.
*   추론의 관성을 끊어주는 실시간 루프 감지를 도입할 것.

이 다섯 가지 구성 요소는 유기적으로 결합되어 상태가 없고 탐욕적이며 자기 평가가 부족한 LLM이 복잡한 장기 프로젝트에서 안정적으로 작동할 수 있게 만듭니다.
