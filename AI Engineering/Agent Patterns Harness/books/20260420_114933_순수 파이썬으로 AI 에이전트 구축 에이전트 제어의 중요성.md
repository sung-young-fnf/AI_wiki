# 순수 파이썬으로 AI 에이전트 구축: 에이전트 제어의 중요성

에이전트 프레임워크(agent framework)를 사용할 때와 마찬가지로, 저 역시 에이전트가 잘못된 결정을 내린 이유를 파악해야 했을 때 한계에 부딪혔습니다.

프레임워크는 대화 내용과 도구 호출(tool call)을 보여주고, 단계별 멋진 그래프까지 보여줄 수 있었습니다. 하지만 정확한 루프 로직(loop logic)과 실패 지점을 명확하게 보여주지는 못했습니다. 저는 모델이 4단계에서 무엇을 보았는지 정확히 알아야 5단계에서 왜 문제가 발생했는지 이해할 수 있었습니다.

그래서 저는 내부 엔지니어링 도우미를 순수 파이썬(pure Python)으로 다시 구축했습니다. 이 도우미는 내부 문서를 검색하고, 서비스 소유권 세부 정보를 얻으며, 최근 배포(deployment)를 검사해야 합니다. 이는 개발자들이 5개의 다른 대시보드를 일일이 뒤지지 않고도 운영 질문에 답할 수 있도록 돕는 실용적인 도구입니다.

직접 구축하면서 불필요한 복잡성이 제거되었습니다. 유용한 에이전트는 기본적으로 엄격한 도구 계약(tool contract), 루프 제어(loop control) 및 검증 단계(verification step)를 포함하는 상태 머신(state machine)이라는 것이 드러났습니다. 오늘은 에이전트 프레임워크를 사용하는 방법을 배우는 것이 아닙니다. 추상화(abstraction)를 제거했을 때, 하나의 좋은 에이전트가 실제로 어떻게 작동하는지 살펴볼 것입니다.

## 제한된 루프(Bounded Loop) 사고 모델

이 아키텍처(architecture)는 간단합니다. 시스템은 질문을 읽고, 다음 행동을 결정하여 도구를 호출합니다. 그 결과를 저장하고 충분한 증거를 확보했는지 확인합니다. 그리고 사용자에게 답변하거나 루프를 계속 진행합니다.

이 아키텍처에서 실제로 중요한 다섯 가지가 있습니다. 어떤 일이 진행 중인지 추적하기 위한 상태(state)가 필요합니다. 외부와 상호 작용하기 위한 도구(tools)가 필요합니다. 다음 단계를 선택하기 위한 결정자(decider)가 필요합니다. 도구를 실행하기 위한 실행자(executor)가 필요하며, 마지막으로 무한 루프(infinite loop)를 방지하기 위한 검증자(verifier)와 중단 정책(stop policy)이 필요합니다.

프레임워크(framework)는 일반적으로 도구 스키마(tool schema)가 모델에 어떻게 전달되고 상태가 단계마다 어떻게 업데이트되는지 숨깁니다. 또한 관찰 내용이 프롬프트(prompt)로 다시 유입되는 방식도 모호하게 만듭니다. 직접 루프를 구축하여 이해하고 나면, 디지털 두뇌가 아닌 토큰 예산(token budget)을 가진 표준 소프트웨어 루프로 보이기 시작할 것입니다.

## 1단계: 상태를 명시적으로 정의하기

숨겨진 상태(hidden state)는 디버깅(debugging)을 불가능하게 만드는 지점입니다. 모델이 특정 순간에 무엇을 알고 있는지 정확히 모르면 시스템을 수정할 수 없습니다.

루프가 중요하게 여기는 모든 것을 담을 깔끔한 객체가 필요합니다. 이 객체는 원래 질문, 도구 호출(tool call) 기록, 수집된 증거, 최종 답변, 그리고 루프가 중단된 이유를 추적해야 합니다. 또한 예산을 준수하기 위해 수행된 단계 수와 소비된 입력 토큰(input tokens)을 추적해야 합니다.

```python
from dataclasses import dataclass, field
from typing import Any

@dataclass
class ToolCall:
    tool_use_id: str
    tool_name: str
    arguments: dict[str, Any]
    result: Any = None

@dataclass
class AgentState:
    question: str
    tool_history: list[ToolCall] = field(default_factory=list)
    evidence: list[str] = field(default_factory=list)
    answer: str | None = None
    stop_reason: str | None = None
    steps: int = 0
    input_tokens_used: int = 0

    @property
    def is_done(self) -> bool:
        return self.answer is not None or self.stop_reason is not None

    def evidence_summary(self -> str:
        if not self.evidence:
            return "No evidence gathered yet."
        return "\n".join(
            f"[{i+1}] {e}" for i, e in enumerate(self.evidence)
        )

    def last_tool(self) -> str | None:
        return self.tool_history[-1].tool_name if self.tool_history else None
```

이 코드는 의도적으로 단순하게 작성되었습니다. 어떤 라이브러리 클래스에서 마법 같은 상속(inheritance)도 없습니다. 이는 어떤 단계에서든 출력하여 정확히 무슨 일이 일어나고 있는지 확인할 수 있는 데이터클래스(dataclass)입니다.

`tool_history`와 `evidence`가 별도의 리스트(list)인 점에 주목하세요. `tool_history`는 실행된 원본 API 로그(log)입니다. `evidence`는 모델이 읽기를 원하는 사실들의 정제된 문자열 표현(string representation)입니다. 이 둘을 분리하면 나중에 프롬프트(prompt)를 훨씬 쉽게 포맷할 수 있고, 원본 JSON 덤프(JSON dumps)로 인해 컨텍스트 창(context window)이 가득 차는 것을 방지할 수 있습니다. 또한 `tool_use_id`를 `ToolCall` 객체에 명시적으로 저장합니다. 언어 모델(language model)이 이 ID를 생성하며, API 계약(contract)을 충족시키기 위해 이 ID를 전달해야 합니다.

## 2단계: 도구를 계약처럼 정의하기

대부분의 에이전트 불안정성(instability)은 언어 모델(language model)의 외피를 쓴 잘못된 도구 설계에서 비롯됩니다. 도구 설명이 모호하거나 타입(typing)이 느슨하면, 모델은 인자(argument)를 환각(hallucinate)하여 백엔드(backend)를 충돌시킬 수 있습니다.

도구는 엄격한 계약(contract)입니다. 명확한 이름, 정확한 설명, 엄격한 입력 스키마(input schema), 그리고 안정적인 반환 형태(return shape)가 필요합니다. 우리는 도구 이름을 파이썬 함수(Python function)와 JSON 스키마(JSON schema)에 매핑하는 도구 레지스트리(tool registry)를 구축할 것입니다.

```python
import json


def search_docs(query: str, max_results: int = 3) -> list[dict]:
    """Search internal engineering docs. Returns list of {title, snippet, url}."""
    return [
        {"title": "Deploy rollback runbook", "snippet": "To roll back a deploy...", "url": "https://docs.internal/rollback"}
    ]

def get_service_owner(service_name: str) -> dict:
    """Look up the on-call owner for a service. Returns {service, owner, slack_channel}."""
    return {"service": service_name, "owner": "platform-team", "slack_channel": "#platform-oncall"}

def get_recent_deploys(service_name: str, limit: int = 5) -> list[dict]:
    """Fetch recent deploys for a service. Returns list of {sha, timestamp, author, status}."""
    return [
        {"sha": "a1b2c3d", "timestamp": "2026-04-06T14:32:00Z", "author": "alice", "status": "success"},
        {"sha": "e4f5g6h", "timestamp": "2026-04-06T09:11:00Z", "author": "bob", "status": "rolled_back"},
    ]

# -- Tool registry: maps names to (callable, tool schema) --

TOOL_REGISTRY: dict[str, tuple[callable, dict]] = {
    "search_docs": (
        search_docs,
        {
            "name": "search_docs",
            "description": "Search internal engineering documentation by keyword or question.",
            "input_schema": {
                "type": "object",
                "properties": {
                    "query": {"type": "string", "description": "Search query"},
                    "max_results": {"type": "integer", "description": "Max results", "default": 3},
                },
                "required": ["query"],
            },
        },
    ),
    "get_service_owner": (
        get_service_owner,
        {
            "name": "get_service_owner",
            "description": "Get the current on-call owner and Slack channel for a given service.",
            "input_schema": {
                "type": "object",
                "properties": {
                    "service_name": {"type": "string", "description": "Exact service name, e.g. 'payments-api'"},
                },
                "required": ["service_name"],
            },
        },
    ),
    "get_recent_deploys": (
        get_recent_deploys,
        {
            "name": "get_recent_deploys",
            "description": "List recent deployments for a service, newest first.",
            "input_schema": {
                "type": "object",
                "properties": {
                    "service_name": {"type": "string", "description": "Exact service name"},
                    "limit": {"type": "integer", "description": "Number of deploys", "default": 5},
                },
                "required": ["service_name"],
            },
        },
    ),
}

def get_tool_schemas() -> list[dict]:
    return [schema for _, schema in TOOL_REGISTRY.values()]

def call_tool(name: str, arguments: dict) -> str:
    if name not in TOOL_REGISTRY:
        return json.dumps({"error": f"Tool {name} does not exist."})
        
    func, _ = TOOL_REGISTRY[name]
    try:
        result = func(**arguments)
        return json.dumps(result, default=str)
    except Exception as e:
        return json.dumps({"error": str(e)})
```

여기 스텁(stub) 구현은 의도적인 교육용 예시입니다. 실제 프로덕션 환경에서는 `search_docs`가 벡터 데이터베이스(vector database)에 접속하고 `get_recent_deploys`는 CI/CD 파이프라인(CI/CD pipeline)을 쿼리(query)할 것입니다. 이 글은 특정 검색 백엔드(backend)보다는 에이전트 루프 자체에 초점을 맞춥니다.

레지스트리(registry) 패턴은 도구를 쉽게 찾을 수 있도록 하고 스키마(schema)를 코드 바로 옆에 유지합니다. `call_tool` 함수가 JSON 문자열(string)을 반환하는 이유는 이것이 도구 결과 블록(tool result block)으로 대화에 다시 들어가는 내용과 정확히 일치하기 때문입니다. 언어 모델(language model)은 읽을 텍스트(text)가 필요하며, JSON 문자열은 구조화된 데이터(structured data)를 직렬화(serialize)하는 가장 안전한 방법입니다. 또한 여기에는 기본적인 try-catch 블록(block)을 추가했습니다. 만약 모델이 도구 이름을 환각하거나 잘못된 인자(argument)를 전달하면, 모델이 다음 차례에 이를 읽고 실수를 수정할 수 있도록 오류를 문자열로 반환합니다.

## 3단계: 다음 행동 결정자(Next-Action Decider)

이 부분이 모델이 루프(loop)에 진입하는 곳입니다. 우리는 현재 상태(state)를 확인하고 바로 다음 행동을 선택하는 간단한 결정자(decider)를 사용합니다.

만약 에이전트에게 너무 멀리까지 계획하도록 강요하면, 두 번째 단계에서 예상치 못한 증거가 반환되는 순간 계획은 쓸모없어집니다. 토큰 낭비가 증가하고 신뢰성(reliability)이 떨어집니다. 우리는 한 단계를 수행하고 결과를 관찰한 다음 다시 생각하는 겸손한 결정자를 원합니다.

```python
import anthropic

client = anthropic.Anthropic()
MODEL = "claude-4-6-opus"

SYSTEM_PROMPT = """You are an internal engineering assistant. You help engineers 
answer operational questions by searching docs, looking up service owners, 
and checking recent deploys.

Rules:
- Use the provided tools to gather evidence before answering.
- Do not guess. If you lack evidence, make a tool call.
- When you have sufficient evidence, respond with a direct answer.
- Avoid redundant tool calls. Check what evidence you already have."""

def run_decider(state: AgentState) -> anthropic.types.Message:
    """Send current state to the model and get the next action."""
    messages = _build_messages(state)
    response = client.messages.create(
        model=MODEL,
        max_tokens=1024,
        system=SYSTEM_PROMPT,
        tools=get_tool_schemas(),
        messages=messages,
    )
    
    state.input_tokens_used += response.usage.input_tokens
    return response

def _build_messages(state: AgentState) -> list[dict]:
    """Reconstruct the conversation history from agent state."""
    
    # We inject the current evidence summary into the first message
    # so the model knows exactly what it has found and what the verifier said.
    context = f"Question: {state.question}\n\nCurrent Evidence:\n{state.evidence_summary()}"
    messages = [{"role": "user", "content": context}]

    for tc in state.tool_history:
        messages.append({
            "role": "assistant",
            "content": [
                {
                    "type": "tool_use",
                    "id": tc.tool_use_id,
                    "name": tc.tool_name,
                    "input": tc.arguments,
                }
            ],
        })
        
        messages.append({
            "role": "user",
            "content": [
                {
                    "type": "tool_result",
                    "tool_use_id": tc.tool_use_id,
                    "content": json.dumps(tc.result, default=str),
                }
            ],
        })

    return messages
```

여기서 중요한 가르침은 `_build_messages`입니다. 모델은 자신의 행동 기록을 볼 필요가 있습니다. 도구 호출을 결정하면 API는 `tool_use` 블록을 반환합니다. 도구를 실행할 때, 우리는 `tool_result` 블록에 결과를 모델로 다시 전달해야 합니다. 이 블록들은 모델이 생성한 정확한 고유 ID(unique ID)로 연결되어야 합니다.

또한 `state.evidence_summary()`를 초기 사용자 프롬프트에 직접 주입합니다. 만약 우리의 검증 단계(verification step)가 답변을 '미지원(unsupported)'으로 플래그하면, 그 경고는 증거 목록에 추가됩니다. 증거 목록을 다시 프롬프트로 전달함으로써, 모델은 다음 차례에 실제로 그 경고를 보고 다른 접근 방식을 시도해야 한다는 것을 알게 됩니다.

`run_decider`가 실제로 아무것도 실행하지 않는다는 점에 주목하세요. 이 함수는 상태(state)를 모델에 전달합니다. 모델은 답변할 준비가 되었다는 신호로 텍스트(text)를 응답하거나, 계속 진행하기를 원한다는 신호로 도구 사용 블록(tool use block)을 응답합니다. 이것이 우리가 의존하는 계약(contract)입니다.

## 4단계: 루프 실행하기

이것이 아키텍처의 핵심 부분입니다. 모든 것을 하나의 읽기 쉬운 `while` 루프(loop)로 묶습니다.

먼저 중단 조건(stop conditions)을 확인합니다. 그런 다음 결정자(decider)를 호출합니다. 응답 유형에 따라 실행을 라우팅(route)합니다. 만약 도구 호출(tool call)이라면 이를 실행하고 결과를 추가합니다. 텍스트(text)라면 검증 단계(verification step)를 실행하고 답변을 확정하거나 모델이 계속 찾도록 유도합니다.

```python
MAX_STEPS = 10
MAX_INPUT_TOKENS = 50_000
MAX_CONSECUTIVE_SAME_TOOL = 2

def run_agent(question: str) -> AgentState:
    state = AgentState(question=question)

    while not state.is_done:
        # --- Stop policy checks ---
        if state.steps >= MAX_STEPS:
            state.stop_reason = f"Hit max steps ({MAX_STEPS})"
            break
        if state.input_tokens_used >= MAX_INPUT_TOKENS:
            state.stop_reason = f"Token budget exceeded ({MAX_INPUT_TOKENS})"
            break
        if _stuck_in_loop(state):
            state.stop_reason = "Repeated same tool call - forcing answer"
            break

        # --- Decide the next action ---
        response = run_decider(state)
        state.steps += 1

        # --- Route on response type ---
        tool_use_block = _extract_tool_use(response)

        if tool_use_block is None:
            raw_answer = _extract_text(response)
            
            # --- Verify before committing the answer ---
            if _answer_is_supported(state, raw_answer):
                state.answer = raw_answer
            else:
                state.evidence.append("[verifier] Answer lacked sufficient evidence. Gather more.")
        else:
            name = tool_use_block["name"]
            args = tool_use_block["input"]
            use_id = tool_use_block["id"]
            
            result = call_tool(name, args)
            
            tc = ToolCall(
                tool_use_id=use_id, 
                tool_name=name, 
                arguments=args, 
                result=json.loads(result)
            )
            state.tool_history.append(tc)
            state.evidence.append(f"Tool '{name}' returned: {result}")

    # If we broke out without an answer, force a best-effort response
    if state.answer is None and state.stop_reason:
        state.answer = _force_answer(state)

    return state


def _extract_tool_use(response) -> dict | None:
    for block in response.content:
        if block.type == "tool_use":
            return {"id": block.id, "name": block.name, "input": block.input}
    return None

def _extract_text(response) -> str:
    return "".join(b.text for b in response.content if b.type == "text")

def _stuck_in_loop(state: AgentState) -> bool:
    recent = state.tool_history[-MAX_CONSECUTIVE_SAME_TOOL:]
    if len(recent) < MAX_CONSECUTIVE_SAME_TOOL:
        return False
    
    # Check if the last two calls have identical names and arguments
    return len({(tc.tool_name, json.dumps(tc.arguments, sort_keys=True)) for tc in recent}) == 1

def _force_answer(state: AgentState) -> str:
    """When the loop exits without an answer, ask the model to do its best."""
    messages = _build_messages(state)
    messages.append({
        "role": "user",
        "content": (
            f"The agent loop ended because: {state.stop_reason}. "
            "Based on the evidence gathered so far, give your best answer. "
            "If evidence is insufficient, say so clearly."
        ),
    })
    response = client.messages.create(
        model=MODEL, max_tokens=1024, system=SYSTEM_PROMPT, messages=messages
    )
    state.input_tokens_used += response.usage.input_tokens
    return _extract_text(response)
```

`_stuck_in_loop` 함수는 실제 운영에서 매우 중요한 작은 디테일입니다. 모델은 혼란스러울 때 종종 호출(call)을 반복합니다. 에이전트가 동일한 쿼리(query)로 검색 도구를 세 번 연속으로 호출하는 것은 실패 모드(failure mode)입니다. 모델은 원하는 답변을 얻지 못했기 때문에 다시 시도하면 마법처럼 작동할 것이라고 가정하여 이러한 행동을 합니다. 우리는 최근 도구 호출의 문자열화된(stringified) 인자(argument)를 비교하여 이러한 정확한 동작을 잡아냅니다.

`_force_answer` 함수는 우리의 폴백(fallback) 역할을 합니다. 에이전트가 토큰 예산(token budget)에 도달하거나 루프에 갇히면, 우리는 루프를 벗어납니다. 사용자에게 빈 화면을 반환하는 것은 좋지 않은 사용자 경험(UX)입니다. 수집된 증거를 모델에 마지막으로 한 번 더 전달하고, 찾은 모든 것을 기반으로 신중한 답변(hedged answer)을 작성하도록 요청하는 것이 훨씬 좋습니다. 예산 추적의 정확성을 유지하기 위해 여기서 `state.input_tokens_used`를 증가시키는 점도 주목하세요.

## 5단계: 검증 계층(Verification Layer) 추가하기

에이전트는 너무 빨리 답변합니다. 도구를 오용하고, 빈약한 증거를 수집하면서도 엄청나게 자신감 있는 태도를 보입니다.

우리는 제안된 답변을 살펴보고 증거가 실제로 이를 뒷받침하는지 묻는 비판 계층(critique layer)이 필요합니다. 이를 위해 복잡한 자체 프롬프트(prompt)를 가진 또 다른 에이전트 전체가 필요하지 않습니다. 루프 내에 제한된 검사 단계(bounded check stage)만 있으면 됩니다. 저렴한 언어 모델(language model) 호출을 수행하는 간단한 함수 하나로 자신감 넘치는 헛소리로부터 시스템을 구할 수 있습니다.

```python
def _answer_is_supported(state: AgentState, proposed_answer: str) -> bool:
    
    evidence_block = state.evidence_summary()

    check_prompt = f"""A model proposed this answer to the user's question.

Question: {state.question}

Proposed answer: {proposed_answer}

Evidence gathered:
{evidence_block}

Is the proposed answer well-supported by the evidence above?
Respond with exactly one word: SUPPORTED or UNSUPPORTED.
If the evidence is thin, ambiguous, or the answer makes claims beyond what the tools returned, say UNSUPPORTED."""

    response = client.messages.create(
        model=MODEL,
        max_tokens=16,
        messages=[{"role": "user", "content": check_prompt}],
    )
    
    state.input_tokens_used += response.usage.input_tokens
    verdict = _extract_text(response).strip().upper()
    
    return verdict == "SUPPORTED"
```

우리는 한 단어만 원하므로 `max_tokens=16`으로 설정했습니다. 추론 단락(paragraph of reasoning)은 원하지 않습니다. 이진 판정(binary verdict)을 원합니다. 이렇게 하면 검증 호출이 매우 빠르고 저렴해집니다. 또한 이 검증으로 소비된 토큰을 총 상태(state)에 추가하여 추적합니다.

만약 검증자가 답변이 '미지원(unsupported)'이라고 판단해도 시스템은 충돌하지 않습니다. 우리는 단순히 증거 목록에 경고를 추가하고 루프를 다시 실행합니다. `_build_messages`에서 증거 목록을 사용자 프롬프트에 주입하기 때문에, 결정자(decider)는 다음 차례에 이 경고를 보게 됩니다. 결정자는 이전 답변이 거부되었음을 깨닫고 보통 더 나은 데이터를 얻기 위해 다른 도구를 호출하기로 결정합니다.

이제 전체 시스템을 처음부터 끝까지 실행하고 디버깅 가능성(debuggability)의 효과를 확인할 수 있습니다.

```python
if __name__ == "__main__":
    state = run_agent("Who owns payments-api and was there a rollback in the last day?")

    print(f"Answer:  {state.answer}\n")
    print(f"Steps:   {state.steps}")
```

이 코드를 실행하면 깔끔하고 검사 가능한 출력(output)을 얻을 수 있습니다.

진정한 보상은 아래의 `print` 문(statement)에 있습니다. 모든 루프 단계, 모든 사용된 토큰, 그리고 모든 호출된 도구가 완전히 보입니다. 만약 답변이 틀렸다면 도구 배열(tool array)을 확인합니다. 도구 배열의 순서가 틀렸다면 결정자 프롬프트(decider prompt)를 확인합니다. 실패 지점을 즉시 격리할 수 있습니다.

## 이렇게 구축하면서 배운 점

이러한 아키텍처를 순수 파이썬(pure Python)으로 구축하는 것은 에이전트 엔지니어링(agent engineering)의 현실과 마주하게 합니다.

**도구(Tools)는 프롬프트(prompts)보다 훨씬 더 중요합니다.** 약한 도구는 약한 에이전트를 만듭니다. 만약 검색 도구가 완벽하게 포맷된 의미론적 쿼리(semantic query)를 요구하여 결과를 반환한다면, 에이전트는 계속해서 실패할 것입니다. 도구가 정돈되지 않은 자연어 문자열(natural language string)을 받아 백엔드(backend)에서 퍼지 매칭(fuzzy matching)을 처리한다면 에이전트는 성공합니다. 시스템의 지능은 주로 도구의 허용 범위에 의해 결정됩니다.

**상태(State)는 에이전트의 성격보다 중요합니다.** 개발자들은 에이전트가 선임 스태프 엔지니어(senior staff engineer)처럼 행동하도록 시스템 프롬프트(system prompt)를 작성하는 데 시간을 보냅니다. 하지만 상태 객체가 손상되면(corrupted) 그것은 중요하지 않습니다. 깔끔한 상태 전환(clean state transition)은 신뢰할 수 있는 동작(reliable behavior)의 기반입니다.

**검증(Verification)은 에이전트의 자신감보다 중요합니다.** 언어 모델(language model)은 유용하게 만들어졌습니다. 즉, 추측하더라도 자신감 있게 들리도록 만들어졌다는 뜻입니다. 유용한 에이전트는 자신의 증거가 빈약할 때를 압니다. 검증 단계는 여러분이 작성할 수 있는 가장 저렴한 보험 정책입니다.

## 마법이 아닌 소프트웨어 엔지니어링

에이전트에 대한 과대광고(hype)는 에이전트를 새로운 종류의 소프트웨어처럼 들리게 합니다. 하지만 그렇지 않습니다. 에이전트는 루프(loop) 내부의 함수(function)에 연결된 언어 모델(language model)입니다.

브랜딩(branding)을 벗겨내면, 실제 작업은 그저 표준 소프트웨어 엔지니어링(software engineering)일 뿐입니다. 엄격한 계약을 정의하고 상태(state)를 깔끔하게 관리하는 것입니다. 무한 루프를 잡기 위한 방어적 코드(defensive code)를 작성하는 것입니다. 토큰 예산(token budget)을 측정하고 타임아웃(timeout)을 강제하는 것입니다.

프로덕션(production) 환경에서는 프레임워크(framework)를 반드시 사용해야 합니다. 프레임워크는 시간을 절약하고 팀 전체의 패턴을 표준화합니다. API 연결과 동시성(concurrency) 관리에 필요한 상용구 코드(boilerplate)를 줄여줍니다. 하지만 프레임워크가 추상화(abstract)하는 루프를 이해한 후에 사용해야 합니다.

한 번쯤은 좋은 에이전트를 직접 손으로 만들어보세요. 상태 객체(state object)를 작성하고, 도구 결과를 프롬프트(prompt)로 다시 연결하세요. 검증 단계(verification step)를 작성하고, 터미널(terminal)에서 루프가 단계별로 실행되는 것을 지켜보세요. 그렇게 하고 나면, 다시는 프레임워크 데모를 예전과 같은 시선으로 보지 않게 될 것입니다.
