# I Deleted the Admin Panel. And I’m Never Going Back.

모든 스타트업에는 백로그의 한 구석을 차지하며 무시당하고 혐오받는 프로젝트가 하나씩 있습니다. 바로 관리자 페이지(Admin Panel)입니다.

이는 지원 팀을 위한 내부 대시보드입니다. 20개의 탭, 50개의 검색 필터, 수시로 깨지는 페이지네이션(Pagination), 그리고 제대로 작동하지 않는 CSV 내보내기 버튼들로 가득합니다. 엔지니어들은 이를 개발하는 것을 싫어하고, 프로덕트 매니저는 유지보수를 꺼리며, 지원 팀은 속도가 느려 사용을 기피합니다.

그래서 저는 모든 CTO가 꿈꾸는 일을 실행에 옮겼습니다. 바로 관리자 페이지를 삭제한 것입니다.

저는 전체 React 애플리케이션을 Text-to-SQL 에이전트와 연결된 단일 채팅 인터페이스로 대체했습니다. 아무도 보지 않는 데이터의 UI를 구축하는 일을 중단했고, 그 결과 개발 속도는 하룻밤 사이에 3배나 향상되었습니다.

### 1. 핵심 지식: 데이터에 부과되는 'UI 세금(UI Tax)'

엔지니어링 시간은 유한합니다. 관리자 페이지를 구축할 때, 노출하려는 모든 데이터 조각마다 'UI 세금'을 지불하게 됩니다.

전통적인 내부 도구의 비용 방정식은 다음과 같습니다:
`비용 = 프론트엔드 코드 + 백엔드 API + QA 테스트 + 배포`

예를 들어 '우편번호로 필터링'하는 버튼을 추가하려 한다면:
*   **프론트엔드:** 드롭다운 추가 (2시간)
*   **백엔드:** SQL WHERE 절 추가 (1시간)
*   **QA:** 페이지네이션 오류 여부 테스트 (2시간)

결국 데이터 한 열을 보여주기 위해 총 5시간이 소요됩니다.

반면 Text-to-SQL 방식을 사용하면 UI는 단순히 텍스트 상자 하나로 충분합니다. 복잡성은 모델로 전이됩니다. 데이터베이스 구조를 한 번만 정의하면 됩니다. 모델 실행 비용(Inference cost)은 아주 저렴하며, 코드를 작성하는 대신 스키마(Schema)를 작성하게 됩니다.

### 2. 흐름: 드롭다운에서 대화로

'투박한 관리자 페이지'와 'AI 데이터 분석가'의 차이는 명확합니다. 구식 방식은 복잡한 UI 미로를 통과해야 하지만, 새로운 방식은 직접 쿼리(Query)를 던집니다. 지연 시간(Latency)은 버튼을 클릭하는 시간이 아니라 생각하는 시간에서 발생합니다. AI는 SQL 쿼리를 직접 생성하므로 무거운 React 단일 페이지 애플리케이션(SPA)을 로드하는 것보다 실행 속도가 훨씬 빠릅니다.

### 3. 코드 비교: React vs. 프롬프팅(Prompting)

"생애 가치(LTV) 기준 상위 5명의 사용자"를 보여주는 기능을 구현한다고 가정해 봅시다.

**구식 방식 (고통스러운 과정)**
백엔드 로직과 프론트엔드 컴포넌트를 모두 작성해야 합니다. Express로 API를 만들고, React에서 데이터를 호출하여 테이블을 렌더링합니다. 이 과정에 약 30분이 소요되며, 2개의 파일을 수정하고 배포가 필요합니다.

**새로운 방식 (스키마 정의)**
프롬프트나 메타데이터에 테이블 구조를 한 번만 정의합니다.

```python
# SQL을 작성하지 않고 구조만 설명합니다.
schema_definition = """
Table 'users' (id, name, email, created_at)
Table 'orders' (id, user_id, total, created_at)
Relationship: users.id -> orders.user_id
"""

# AI가 SQL 전문가로서 스키마를 사용해 답변하도록 설정
# 기능 구현 완료: 30초 소요, 파일 수정 없음, 배포 불필요
```

### 4. 수학적 계산: AI가 승리하는 시점

이 방식은 고객용 앱에는 적용되지 않습니다. 사용자에게는 시각적 피드백과 탐색 가능성이 중요하기 때문입니다. 하지만 내부 도구(관리자 페이지)의 사용 패턴은 다릅니다. 목적이 뚜렷하고(High Intent), 빈도가 높으며(High Frequency), 인내심이 낮습니다(Low Patience).

고유한 쿼리의 수를 N이라고 할 때:
*   N이 작다면(예: 단순 사용자 목록) 정적 UI를 만드세요.
*   N이 크다면(예: 비기술 직군의 다양한 비즈니스 질문) AI가 승리합니다.

UI 구축은 고정 비용이 크지만, AI 사용은 가변 비용이 거의 제로에 가깝습니다. 수주가 걸리는 엔지니어링 시간을 고려하면 내부 도구에서 AI의 효율성은 즉각적으로 나타납니다.

### 5. 보안 문제: 해결책

GPT-4를 운영 데이터베이스에 그대로 연결할 수는 없습니다. 프롬프트 인젝션(Prompt Injection)을 통한 테이블 삭제나 개인정보(PII) 유출 위험이 있기 때문입니다.

**해결책: 읽기 전용 권한(Read-Only Roles) + 행 수준 보안(Row-Level Security)**
에이전트에게 루트 권한을 주는 대신 특정 권한만 부여합니다.
*   **읽기 전용 액세스:** `GRANT SELECT` 권한만 부여.
*   **제한된 뷰(Views):** 비밀번호나 주민번호 같은 민감한 열을 제외한 정제된 뷰를 생성하여 에이전트가 해당 뷰만 조회하게 합니다.

모델은 물리적으로 민감한 열을 볼 수 없으므로 유출 가능성도 차단됩니다.

### 6. 결론: 복잡함을 삭제하라

우리는 흔히 드롭다운 메뉴를 만드는 것을 '엔지니어링'이라고 착각하지만, 이는 디지털 조립 라인 작업에 가깝습니다.

관리자 페이지 구축을 멈추십시오. 그것들은 배포되는 순간부터 구식이 되는 기술 부채(Technical Debt)입니다. 지원 팀에게는 채팅창을, 데이터 팀에게는 쿼리 인터페이스를 제공하십시오. 개발자들이 화면의 픽셀을 옮기는 일에 시간을 쓰게 하지 마십시오.

관리자 페이지를 삭제하고 엔지니어들을 자유롭게 하십시오.

#### 핵심 요약 (Key Takeaways)
*   **CRUD는 범용재(Commodity):** 생성, 읽기, 수정, 삭제 로직을 작성하는 것은 가치가 낮은 작업입니다.
*   **Text-to-SQL:** LLM을 사용하여 SQL 쿼리를 생성하면 커스텀 UI 필터가 필요 없습니다.
*   **스키마의 중요성:** 데이터베이스 스키마가 잘 정의되어 있다면, 새로운 코드 없이도 LLM이 쿼리를 수행할 수 있습니다.
*   **보안 우선:** 원시 DB 자격 증명을 노출하지 마십시오. 읽기 전용 권한과 정제된 뷰를 사용하십시오.
*   **내부용 vs 고객용:** 이 방식은 내부 도구에 적합합니다. 고객용 앱은 여전히 탐색을 위한 시각적 UI가 필요합니다.
