다른 언어들 간 MCP 서버 성능을 실제로 측정하면 어떤 결과가 나올까요?

대부분의 MCP 서버 개발자는 편의성에 따라 언어를 선택합니다. 머신러닝(ML) 분야에 있다면 파이썬(Python)을, 웹 개발 출신이라면 Node.js를, 백엔드(backend) 엔지니어라면 자바(Java)나 고(Go)를 선택하죠. 프로토콜(protocol)이 런타임(runtime)보다 더 중요하다고 가정하는 경향이 있습니다.

TM Dev Lab의 티아고 멘데스(Thiago Mendes)가 발표한 벤치마크는 이러한 가정을 강력하게 뒤집었습니다. 390만 건의 요청과 세 번의 독립적인 테스트 라운드에 걸쳐, 언어 간 성능 격차는 단순히 2배나 3배가 아니라, 경우에 따라서는 자릿수(orders of magnitude)만큼 엄청난 것으로 드러났습니다.

## 설정

네 개의 MCP 서버(자바(Java) (스프링 부트(Spring Boot) + 스프링 AI(Spring AI)), 고(Go) (공식 SDK(Software Development Kit)), Node.js, 파이썬(Python) (FastMCP))가 동일한 제약 조건(1 CPU 코어, 1GB RAM)으로 컨테이너화(containerized)되었습니다. 부하는 k6를 사용하여 생성되었으며, 5분 동안 지속적으로 50개의 동시 가상 사용자(virtual users, VU)로 증가시켰고, 이 과정을 세 번 반복했습니다.

각 서버는 동일한 네 가지 도구를 노출했습니다:

*   `calculate_fibonacci`: CPU 집약적인 재귀 연산
*   `fetch_external_data`: I/O 집약적인 HTTP 호출
*   `process_json_data`: JSON 변환
*   `simulate_db_query`: 제어된 지연 시간(latency) 시뮬레이션

이는 특정 성능 차원(dimensions)에 부하를 가하고, 단순히 특정 워크로드(workload)에 유리한 결과만을 선택하는 것이 아니라, 전반적인 성능을 파악하기 위함이었습니다.

### 아키텍처(Architecture) 개요

```
k6 부하 생성기 (50개 가상 사용자(VU) · 5분 · 3회 라운드)
             |
     도커 브리지 네트워크(Docker Bridge Network)
     |          |          |          |
  Java:8080  Go:8081  Python:8082  Node:8083
     |          |          |          |
  [calculate_fibonacci · fetch_external_data]
  [process_json_data  · simulate_db_query  ]
             |
  측정 지표(Metrics): 지연 시간(latency) · 초당 요청 수(RPS) · CPU 사용률 · 메모리
```

## 결과

주요 결과 표는 다음과 같습니다:

| 서버      | 평균 지연 시간 (Avg Latency) | 처리량 (초당 요청 수, RPS) | CPU 사용률 | 메모리 |
| :-------- | :-------------------------- | :----------------------- | :--------- | :----- |
| Java      | 0.84 ms                     | 1,624                    | 28.8%      | 226 MB |
| Go        | 0.86 ms                     | 1,624                    | 31.8%      | 18 MB  |
| Node.js   | 10.66 ms                    | 559                      | 98.7%      | 110 MB |
| Python    | 26.45 ms                    | 292                      | 93.9%      | 98 MB  |

자바(Java)와 고(Go)는 완전히 다른 수준의 성능을 보였습니다. 평균 지연 시간은 밀리초 미만이었고, 초당 1,600개 이상의 요청을 처리했으며, CPU 사용률은 35% 미만을 유지했습니다. 반면 파이썬(Python)은 평균 26.45ms의 지연 시간과 최대 292 RPS를 기록했으며, CPU 사용률은 거의 94%에 육박했습니다.

가장 눈에 띄는 개별 결과는 사용 가능한 도구들을 나열하는 기본적인 메타데이터(metadata) 호출인 `tools_list` 작업에서 나타났습니다. 자바(Java)는 이 작업을 0.31ms만에 처리했지만, 파이썬(Python)은 28.88ms가 걸렸습니다. 이는 가장 가벼운 작업 중 하나여야 할 작업에서 93배의 차이를 보인 것입니다.

## 파이썬(Python)이 어려움을 겪는 이유

간단히 말해, 파이썬(Python)의 전역 인터프리터 락(Global Interpreter Lock, GIL) 때문입니다. GIL은 CPU 바운드(CPU-bound) 작업의 진정한 병렬 실행을 방해합니다. 동시 부하(concurrent load) 상황에서 async/await는 I/O 동시성(concurrency)을 제공하지만, 실제 연산이 필요한 작업들은 대기열에 쌓여 순차적으로 실행됩니다. 그 결과 높은 지연 시간(latency)과 최고치에 달하는 CPU 사용률을 초래합니다.

```python
# Python — FastMCP
@mcp.tool()
async def calculate_fibonacci(n: int) -> dict:
    result = fibonacci(n)   # GIL이 여기에서 동시 CPU 작업을 차단합니다.
    return {"result": result, "server": "python"}
```

고(Go)의 고루틴(goroutine) 모델은 동일한 워크로드(workload)를 매우 다르게 처리합니다:

```go
// Go — 공식 SDK
server.AddTool("calculate_fibonacci", mcp.Tool{...},
  func(args map[string]interface{}) (*mcp.ToolResponse, error) {
    n := int(args["n"].(float64))
    // 고루틴(goroutines)은 동시성을 기본적으로 처리합니다 — GIL도 병목 현상도 없습니다.
    return &mcp.ToolResponse{Content: [...]}, nil
  })
```

두 언어의 동시성 모델은 근본적으로 다르며, 지속적인 부하 상황에서는 그 격차가 더욱 커집니다.

## 도구별 상세 분석

벤치마크는 개별 도구별 지연 시간(latency)도 분석했으며, 몇 가지 흥미로운 패턴을 드러냈습니다.

| 도구                  | Java      | Go        | Node.js                | Python                 |
| :-------------------- | :-------- | :-------- | :--------------------- | :--------------------- |
| calculate_fibonacci   | 0.37 ms   | 0.39 ms   | 7.11 ms (19×)          | 30.83 ms (84×)         |
| fetch_external_data   | 1.32 ms   | 1.29 ms   | 19.18 ms (15×)         | 80.92 ms (63×)         |
| process_json_data     | 0.35 ms   | 0.44 ms   | 7.48 ms (21×)          | 34.24 ms (97×)         |
| simulate_db_query     | 10.37 ms  | 10.71 ms  | 26.71 ms (2.6×)        | 42.57 ms (4.1×)        |
| tools_list            | 0.31 ms   | 0.72 ms   | 7.56 ms (24×)          | 28.88 ms (93×)         |

지연 시간(latency) 시뮬레이션 도구(`simulate_db_query`)는 실제 대기 시간을 포함하고, 네 가지 런타임(runtime) 모두 거의 같은 속도로 대기하기 때문에 격차가 줄어듭니다. 그러나 런타임의 동시성(concurrency) 모델이 실제로 중요한 컴퓨팅 및 I/O 작업에서는 격차가 크게 벌어집니다.

## 알아두어야 할 주의 사항

Node.js 서버는 CVE-2026-25536(동시 요청 간 세션(session) 데이터 유출)에 대한 보안 완화(security mitigation)를 위해 요청당 서버 인스턴스(instance) 생성을 사용했습니다. 이러한 아키텍처(architecture) 선택은 오버헤드(overhead)를 추가했으며, 공유 인스턴스 패턴(shared-instance pattern)은 훨씬 더 나은 성능을 보였을 것입니다. 이러한 결과는 보안이 강화된 구성(configuration)을 반영한 것이지, Node.js의 최고 성능을 의미하는 것은 아닙니다.

파이썬(Python) 또한 기본으로 제공되는 단일 워커(single-worker) uvicorn 설정으로 테스트되었습니다. uvloop를 사용한 다중 워커(multi-worker) 배포(deployments)는 처리량을 상당히 향상시킬 수 있습니다.

이는 벤치마크가 최적화된 프로덕션(production) 설정이 아닌, 현실적인 기본 구성(configurations)을 측정했기 때문에 중요합니다. 규모에 맞게 파이썬(Python) 또는 Node.js 배포(deployments)를 튜닝(tuned)한 팀은 다른 수치를 볼 수 있습니다.

네 가지 언어 모두에서 유지된 한 가지는 바로 제로(zero) 에러(error)율이었습니다. 390만 건의 모든 요청이 언어에 관계없이 성공적으로 완료되었습니다. 프로토콜(protocol) 자체는 견고합니다.

## 이것이 당신의 스택(Stack)에 실제로 의미하는 것

내부 도구, 개발자용 MCP 서버 또는 프로토타입(prototype)을 구축한다면 파이썬(Python)과 Node.js는 완전히 합리적인 선택입니다. 개발자 경험(developer experience)이 더 빠르고, 생태계(ecosystem)가 풍부하며, 트래픽(traffic) 양이 적을 때는 사용자에게 지연 시간(latency) 차이가 보이지 않습니다.

하지만 MCP 서버가 실제 프로덕션(production) 부하를 처리해야 한다면, 경제성은 빠르게 바뀝니다. 파이썬(Python)으로 동일한 트래픽(traffic)을 처리하려면 고(Go)보다 약 5배 더 많은 CPU가 필요합니다. 규모가 커질수록 이는 실제 인프라(infrastructure) 비용으로 이어집니다.

고(Go)는 프로덕션(production) 환경에서 실용적인 기본 선택지로 두드러집니다. 자바(Java)와 동등한 성능, 12배 더 작은 메모리 점유 공간(memory footprint), JVM(Java Virtual Machine) 워밍업(warmup) 불필요, 그리고 테스트 라운드 전반에 걸친 뛰어난 일관성(consistency) (처리량 변동성(throughput variability) 단 0.5%)을 제공합니다. 자바(Java)는 가장 낮은 지연 시간(latency)이 필요하고 기존 JVM 생태계(ecosystem)를 활용할 수 있는 경우에 유리합니다. 파이썬(Python)과 Node.js는 트래픽(traffic)이 적은 시나리오나 개발 환경에 가장 적합합니다.

이 벤치마크는 단 하나의 놀라운 결과를 도출한 것은 아닙니다. 언어 런타임(runtime)은 항상 성능 차이가 있었습니다. 이 벤치마크가 한 일은 개발자들이 흔히 스타일 선호도(style preference)로 여기는 결정에 명확한 수치를 제시했다는 점입니다.
