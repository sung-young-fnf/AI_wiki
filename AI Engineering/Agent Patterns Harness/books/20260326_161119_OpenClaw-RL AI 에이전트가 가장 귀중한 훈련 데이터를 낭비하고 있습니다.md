# OpenClaw-RL: AI 에이전트가 가장 귀중한 훈련 데이터를 낭비하고 있습니다

OpenClaw-RL은 모든 에이전트 시스템이 낭비하는 다음 상태 신호(next-state signal)를 회수합니다. 그리고 사용자가 대화하는 중에도 이를 수행합니다.

만다르 카르하데 (Mandar Karhade, MD. PhD.)
7분 독서
·
2026년 3월 14일

---

## 요약 (TLDR)

*   모든 에이전트 상호작용은 다음 상태 신호(next-state signal)를 생성합니다. 모든 기존 시스템은 이를 낭비합니다. OpenClaw-RL은 이를 실시간 훈련 데이터로 회수합니다.
*   완전 비동기(Fully async) 아키텍처: 에이전트는 백그라운드에서 학습하는 동안에도 계속 서비스를 제공합니다. 중단이 전혀 없습니다.
*   두 가지 방법(이진 강화 학습(Binary RL) + 온-폴리시 증류(On-Policy Distillation))을 결합하여 개인화 점수를 16단계 만에 0.17에서 0.81로 높입니다.
*   개인 에이전트(personal agent) 및 범용 에이전트(general agent)(터미널, GUI, SWE, 도구 호출) 모두에 적용됩니다. 오픈 소스(Open source), 프린스턴 대학교.

---

핵심은 다음과 같습니다. 여러분이 배포한 모든 AI 에이전트는 이미 훈련 데이터의 금광을 보유하고 있다는 것입니다. 모든 상호작용, 모든 사용자 응답, 모든 도구 실행 결과, 모든 터미널 출력, 모든 GUI 상태 변화 — 이 모든 것이 풍부하고, 구조화되어 있으며, 무료로 존재합니다.

그리고 모든 시스템은 이 데이터를 버립니다.

사용자가 "아니요, 제가 말한 건 다른 파일이었어요"라고 말합니다. 이것이 바로 훈련 신호(training signal)입니다. 에이전트의 코드 편집 후 테스트 스위트(test suite)가 스택 트레이스(stack trace)를 반환합니다. 훈련 신호입니다. 터미널이 종료 코드(exit code)를 내놓습니다. 훈련 신호입니다. 이 모든 것이 단순히 "다음 행동을 위한 맥락(context)"으로 버려집니다. 그레디언트(gradient)로 변환되지 않고, 가중치(weight)를 업데이트하는 데 사용되지 않습니다. 그저... 소비되고 잊힐 뿐입니다.

프린스턴 대학교의 한 팀이 이러한 낭비를 더 이상 무시할 수 없게 만드는 논문을 발표했습니다. 바로 OpenClaw-RL입니다. 기존 아이디어를 새로운 약어(acronym)로 재포장하는 대부분의 논문과 달리, 이 논문은 전체 에이전트형 강화 학습(Agentic RL) 커뮤니티가 간과해 왔던 근본적인 문제를 실제로 밝혀냈습니다.

## ‘다음 상태 신호(Next-State Signal)’란 정확히 무엇인가요?

간단합니다. 에이전트가 행동 a_t를 취한 후 어떤 일이 발생합니다. 사용자가 응답하거나, 도구가 출력을 반환하거나, GUI가 전환됩니다. 이러한 "어떤 일"이 바로 다음 상태 s_{t+1}입니다. 대규모 언어 모델(LLM)을 위한 표준 강화 학습(RL)에서 이 신호는 완전히 무시되거나, 긴 궤적(trajectory)의 맨 마지막에 터미널 결과 보상(terminal outcome reward)으로 축소됩니다.

OpenClaw-RL은 이 신호가 두 가지 명확한 형태의 복구 가능한 정보(recoverable information)를 포함한다고 주장합니다. 첫째, **평가 신호(evaluative signal)**입니다. 행동이 제대로 작동했는가? 사용자의 재질문은 불만을 나타냅니다. 테스트 통과는 성공을 알립니다. PRM(Process Reward Model)은 이를 턴(turn)별 스칼라 보상(scalar reward)으로 변환할 수 있습니다. 둘째, **지시 신호(directive signal)**입니다. 행동이 어떻게 달라졌어야 했는가? 사용자가 "파일을 먼저 확인했어야지"라고 작성하는 것은 단순한 부정적인 피드백이 아닙니다. 이는 어떤 토큰(token)을 어떻게 변경해야 하는지 알려주는 것입니다. 스칼라 보상은 이를 포착할 수 없습니다. 더 풍부한 정보가 필요합니다.

솔직히 말해서, 평가 부분 자체가 완전히 새로운 것은 아닙니다. PRM은 수학적 추론 분야에서 한동안 연구되어 왔습니다. 하지만 이를 이질적인 상호작용 스트림(대화, 터미널, GUI, SWE 작업, 도구 호출) 전반에 걸쳐 실시간 온라인 프로세스 보상(live, online process reward)으로 동시에 적용하는 것은 다릅니다. 이는 일반적인 "고정된 데이터셋으로 훈련했다"는 이야기가 아닙니다. 이번에는 실시간입니다.

## 작동을 가능하게 하는 아키텍처

OpenClaw-RL은 슬라임 비동기(async) 프레임워크를 기반으로 구축되었으며, 모든 것을 분리합니다. 차단 종속성(blocking dependency) 없이 실행되는 네 가지 독립적인 루프(loop)가 있습니다.

모델은 다음 요청을 처리하는 동안, PRM은 이전 응답을 평가하고, 트레이너(trainer)는 두 번의 상호작용 전의 그레디언트(gradient) 업데이트를 적용합니다. 어떤 구성 요소도 다른 구성 요소를 기다리지 않습니다. 개인 에이전트의 경우, 여러분의 장치는 기밀 API를 통해 연결됩니다. 대규모 범용 에이전트의 경우, 수백 개의 병렬 환경이 클라우드 서비스에서 실행됩니다.

아! 그리고 모든 API 요청을 "주요(main-line) 턴"(훈련 가능: 실제 응답 및 도구 실행)과 "보조(side) 턴"(훈련 불가능: 메모리 구성, 보조 쿼리)으로 분류합니다. 따라서 무엇을 학습해야 할지 정확히 알고 있습니다.

## 이진 강화 학습(Binary RL): 거칠지만 신뢰할 수 있는 신호

PRM은 다음 상태 피드백을 기반으로 각 에이전트 응답을 평가합니다. 사용자가 만족스러워했는가? 도구 호출이 성공했는가? 판독기(judge)는 이에 점수를 매깁니다: +1 (좋음), -1 (나쁨), 또는 0 (중립). 견고성(robustness)을 위해 다수결 투표를 통해 여러 독립적인 평가를 수행합니다.

이 스칼라 보상들은 비대칭 경계(asymmetric bounds)를 가진 PPO 방식의 클립된 대리 손실(clipped surrogate loss)로 입력됩니다. 이는 표준적이고 잘 이해된 최적화 기법입니다. 이진 강화 학습(Binary RL)은 점수가 매겨진 모든 턴(turn)을 받아들이고, 간결한 반응을 포함한 모든 다음 상태 신호와 작동하며, 광범위한 그레디언트 적용 범위(gradient coverage)를 제공합니다.

하지만 문제는 이것이 거친 정보라는 것입니다. 시퀀스(sequence)당 하나의 스칼라 값만 제공됩니다. 사용자가 "파일을 편집하기 전에 확인했어야지"라고 말할 때, 이진 강화 학습(Binary RL)은 -1을 부여할 뿐입니다. 그것이 전부입니다. 무엇을 어떻게 변경해야 하는지에 대한 모든 지시 정보는? 손실됩니다.

## 회고 기반 온-폴리시 증류(Hindsight-Guided On-Policy Distillation): 비밀 병기

이것이 바로 OpenClaw-RL이 정말로 영리해지는 부분입니다. OPD(On-Policy Distillation)는 이진 강화 학습(Binary RL)이 버리는 지시 정보를 회수하며, 별도의 더 강력한 교사 모델(teacher model)을 필요로 하지 않고 이를 수행합니다.

네 단계로 이루어집니다.

1.  **1단계**: 판독기(judge)는 다음 상태 신호에서 간결한 "힌트"(1~3문장 분량의 실행 가능한 수정 사항)를 추출합니다.
2.  **2단계**: 품질 필터링; 명확하고 추출 가능한 수정 방향이 있는 턴(turn)만이 선택됩니다.
3.  **3단계**: 힌트는 원래 프롬프트(prompt)에 추가되어 "강화된 교사 맥락(enhanced teacher context)"을 생성합니다. 이는 사용자가 미리 수정 사항을 알려줬다면 모델이 보았을 내용입니다.
4.  **4단계**: 이 강화된 맥락(context) 하에서 정책 모델(policy model)이 쿼리되고, "교사(teacher)"(힌트가 보강된) 분포(distribution)와 "학생(student)"(원래) 분포(distribution) 간의 토큰(token)별 로그 확률(log-probability) 차이가 어드밴티지 신호(advantage signal)가 됩니다.

$$A_t = \log \pi_{teacher}(a_t | s_{enhanced}) - \log \pi_{\theta}(a_t | s_t)$$

이는 자기 증류(self-distillation) 방식이기 때문에 매우 훌륭합니다. 외부 교사(teacher)가 필요 없고, 미리 수집된 피드백 쌍도 필요 없습니다. 모델은 대화에 이미 존재했던 회고(hindsight)를 사용하여 스스로 학습합니다. 일부 토큰(token)은 강화되고, 다른 토큰은 억제됩니다. 토큰(token)별 방향성 지침은 단일 스칼라 값보다 훨씬 더 풍부한 정보를 제공합니다.

하지만 잠깐. OPD는 까다롭습니다. 명확한 수정 방향이 존재하는 턴(turn)에서만 훈련합니다. 샘플이 희소합니다(Sparse samples). 이것이 두 가지 방법이 함께 작동해야 하는 이유입니다.

## 결합된 방법이 수치를 변화시킵니다

수치는 거짓말을 하지 않습니다. 기본 개인화 점수(personalization score) 0.17에서 시작합니다.

정말인가요? 이진 강화 학습(Binary RL)만으로는 16단계 후에 실제로 성능이 저하됩니다. OPD만으로는 지연된 이득을 보이지만(희소한 샘플임을 기억하세요) 결국 크게 상승합니다. 하지만 두 가지를 결합하면? 단 8번의 업데이트 만에 0.17에서 0.76으로 상승합니다.

구체적으로 말하면, 숙제에 OpenClaw를 사용한 학생은 에이전트가 AI처럼 들리지 않게 학습하기까지(더 이상 "굵은" 글씨체나 로봇 같은 단계별 설명 없음) 단 36번의 문제 해결 상호작용만 필요했습니다. 숙제를 채점하는 교사는 에이전트가 더 친근하고 구체적인 피드백을 작성하기 시작하기까지 단 24번의 상호작용만 필요했습니다. 학생 시나리오는 0.17에서 0.76으로 상승했고, 교사 시나리오는 0.22에서 0.90으로 상승했습니다.

## 개인 에이전트(Personal Agent)에만 국한되지 않습니다

동일한 인프라(infrastructure)는 터미널(terminal), GUI, SWE(Software Engineering), 도구 호출(tool-call) 설정 전반에 걸쳐 대규모 병렬화(massive parallelization)(128개 환경은 터미널용, 64개는 GUI/SWE용, 32개는 도구 호출용)를 통해 범용 에이전트형 강화 학습(general agentic RL)을 처리합니다. 다양한 설정에 따라 다른 모델을 사용합니다: 터미널용 Qwen3–8B, GUI용 Qwen3VL-8B-Thinking, SWE용 Qwen3–32B.

범용 에이전트의 경우, 프로세스 보상(process reward)과 결과 보상(outcome reward)을 통합합니다. 도구 호출(tool-call) 결과는 통합 보상 사용 시 0.30 대 결과만 사용 시 0.17이었습니다. 이는 단계별 PRM 신호를 추가하는 것만으로 76% 향상된 수치입니다. 절대적인 수치가 엄청난 것은 아니지만, 방향성 증거는 명확합니다. 즉, 프로세스 보상은 장기적인 에이전트형 작업에 중요하며, OpenClaw-RL은 이를 대규모로 실용화합니다.

## 이것이 실제로 중요한 이유

현재 모든 에이전트형 프레임워크는 메모리 파일, 시스템 프롬프트(system prompt), 스킬 라이브러리(skill library)를 통해 적응합니다. 기본 모델의 가중치(weight)는 결코 변하지 않습니다. OpenClaw-RL은 가중치를 변경합니다. 사용 중에도. 서비스 중단 없이 말이죠.

전체 스택(stack)(정책 모델(policy model), 판독기(judge), 트레이너(trainer))은 자체 인프라(infrastructure)에서 실행됩니다. 제3자 API 호출이 없습니다. 대화 데이터는 로컬(local)에 유지됩니다. 그리고 완전한 관측 가능성(observability)을 위해 모든 것이 실시간으로 JSONL에 기록됩니다.

이것은 동일한 루프(loop) 내에서, 동일한 다음 상태 신호를 사용하여, 이질적인 상호작용 유형 전반에 걸쳐 개인 에이전트 개인화와 범용 에이전트 훈련을 통합하는 최초의 시스템입니다. 대화, 터미널, GUI, 코드 리포지토리(code repository), 도구 호출 — 이 모든 것은 단지 다른 전이 함수(transition function)를 가진 MDP(Markov Decision Process)일 뿐입니다. 훈련 신호는 보편적입니다.

사용되면서 개선되는 에이전트는 더 이상 연구 프로토타입(research prototype)이 아닙니다. 이것은 오픈 소스(open source)이며, 비동기(async) 방식이고, 여러분의 하드웨어에서 실행되며, 여러분 스스로 생성하고 버리고 있던 데이터로부터 학습합니다.

이것은 제 관점입니다. 여러분은 편안한 대로 하시면 됩니다. 하지만 에이전트형 시스템을 구축하고 있으면서 훈련을 위한 다음 상태 신호(next-state signal)를 회수하지 않는다면, 가장 자연스럽고, 가장 풍부하며, 가장 정보가 풍부한 감독 소스(supervision source)를 간과하고 있는 것입니다. OpenClaw-RL은 바로 그 방법을 여러분에게 정확히 보여주었습니다.
