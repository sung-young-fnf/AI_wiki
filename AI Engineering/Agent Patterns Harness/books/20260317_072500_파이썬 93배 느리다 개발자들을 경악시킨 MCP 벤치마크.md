실제로 여러 언어별 MCP 서버 성능을 측정하면 어떤 결과가 나올까요?

대부분의 개발자는 MCP(Multi-Agent Communication Protocol) 서버를 구축할 때 익숙한 언어를 선택합니다. 머신러닝(ML) 분야에 있다면 파이썬(Python)을, 웹 개발 출신이라면 Node.js를, 백엔드 엔지니어라면 자바(Java)나 고(Go)를 선택하죠. 이들은 프로토콜(protocol)이 런타임(runtime)보다 더 중요하다고 가정합니다.

_엔터 키를 누르거나 클릭하여 전체 크기 이미지 보기_
AI 생성 이미지

TM Dev Lab의 티아고 멘데스(Thiago Mendes)가 발표한 벤치마크는 이러한 가설에 강력하게 도전했습니다. 390만 건의 요청과 세 번의 독립적인 테스트 라운드를 거친 결과, 언어 간 성능 격차는 2배나 3배 수준이 아니라 일부 경우에는 자릿수가 다른 엄청난 수준인 것으로 나타났습니다.

## 설정 (The Setup)

자바(Java)(Spring Boot + Spring AI), 고(Go)(공식 SDK), Node.js, 파이썬(Python)(FastMCP)으로 네 개의 MCP 서버를 구축했으며, 각각 동일한 제약 조건(1 CPU 코어, 1GB RAM)으로 컨테이너화(containerized)되었습니다. 부하(load)는 k6를 사용하여 생성되었으며, 50개의 동시 가상 사용자(virtual user, VU)가 5분 동안 지속적으로 증가하는 형태로 세 번 반복 수행되었습니다.

각 서버는 다음 네 가지 도구(tool)를 노출했습니다.

*   `calculate_fibonacci` (CPU 집약적인 재귀 계산)
*   `fetch_external_data` (I/O 집약적인 HTTP 호출)
*   `process_json_data` (JSON 변환)
*   `simulate_db_query` (제어된 지연 시간 시뮬레이션)

이는 특정 작업 부하(workload)만 선택하는 것이 아니라, 다양한 성능 차원을 스트레스 테스트(stress test)하여 포괄적인 그림을 얻기 위함이었습니다.

## 결과 (The Results)

다음은 주요 결과표입니다.

| 서버       | 평균 지연 시간 (Avg Latency) | 처리량 (RPS) | CPU 사용률 (CPU Usage) | 메모리 (Memory) |
| :--------- | :--------------------------- | :----------- | :--------------------- | :-------------- |
| **Java**   | **0.84 ms**                  | **1,624**    | **28.8%**              | **226 MB**      |
| **Go**     | **0.86 ms**                  | **1,624**    | **31.8%**              | **18 MB**       |
| **Node.js**| 10.66 ms                     | 559          | 98.7%                  | 110 MB          |
| **Python** | 26.45 ms                     | 292          | 93.9%                  | 98 MB           |

자바(Java)와 고(Go)는 완전히 다른 수준이었습니다. 평균 지연 시간(latency)은 밀리초(ms) 미만이었고, 초당 1,600건 이상의 요청을 처리했으며, CPU 사용률은 35% 미만으로 안정적이었습니다. 반면 파이썬(Python)은 평균 26.45ms의 지연 시간을 기록했고, 최대 292 RPS를 처리했으며, CPU 사용률은 거의 94%에 달했습니다.

가장 눈에 띄는 개별 결과는 사용 가능한 도구 목록을 나열하는 기본적인 메타데이터(metadata) 호출인 'tools_list' 작업에서 나왔습니다. 자바(Java)는 이 작업을 0.31ms 만에 처리했지만, 파이썬(Python)은 28.88ms가 걸렸습니다. 이는 가장 저렴해야 할 작업에서 무려 93배의 차이를 보인 것입니다.

## 파이썬이 고전하는 이유 (Why Python Struggles)

간단히 말해 파이썬(Python)의 전역 인터프리터 잠금(Global Interpreter Lock, GIL) 때문입니다. GIL은 CPU 바운드(CPU-bound) 작업의 진정한 병렬 실행을 방해합니다. 동시 부하(concurrent load) 상황에서 비동기(async/await)는 I/O 동시성을 제공하지만, 실제 연산이 필요한 작업들은 순차적으로 대기하며 실행됩니다. 그 결과 높은 지연 시간(latency)과 최고치에 달하는 CPU 사용률을 초래합니다.

```python
# 파이썬(Python) — FastMCP
@mcp.tool()
async def calculate_fibonacci(n: int) -> dict:
    result = fibonacci(n)   # 여기서 GIL이 동시 CPU 작업을 차단함
    return {"result": result, "server": "python"}
```

고(Go)의 고루틴(goroutine) 모델은 동일한 작업 부하를 매우 다르게 처리합니다.

```go
// 고(Go) — 공식 SDK
server.AddTool("calculate_fibonacci", mcp.Tool{...},
  func(args map[string]interface{}) (*mcp.ToolResponse, error) {
    n := int(args["n"].(float64))
    // 고루틴(goroutine)이 동시성을 자체적으로 처리 — GIL 없음, 병목 현상(bottleneck) 없음
    return &mcp.ToolResponse{Content: [...]}, nil
  })
```

이러한 동시성 모델은 근본적으로 다르며, 지속적인 부하(sustained load) 상황에서는 그 격차가 더욱 커집니다.

## 도구별 분석 (Per-Tool Breakdown)

벤치마크는 또한 개별 도구(tool)별 지연 시간(latency)을 분석하여 몇 가지 흥미로운 패턴을 밝혀냈습니다.

| 도구                  | Java      | Go        | Node.js                | Python                 |
| :-------------------- | :-------- | :-------- | :--------------------- | :--------------------- |
| `calculate_fibonacci` | 0.37 ms   | 0.39 ms   | 7.11 ms (19×)          | 30.83 ms (84×)         |
| `fetch_external_data` | 1.32 ms   | 1.29 ms   | 19.18 ms (15×)         | 80.92 ms (63×)         |
| `process_json_data`   | 0.35 ms   | 0.44 ms   | 7.48 ms (21×)          | 34.24 ms (97×)         |
| `simulate_db_query`   | 10.37 ms  | 10.71 ms  | 26.71 ms (2.6×)        | 42.57 ms (4.1×)        |
| `tools_list`          | 0.31 ms   | 0.72 ms   | 7.56 ms (24×)          | 28.88 ms (93×)         |

지연 시간 시뮬레이션 도구(`simulate_db_query`)는 실제 대기(waiting)가 포함되어 모든 네 가지 런타임(runtime)이 거의 동일한 속도로 대기하기 때문에 격차가 좁혀집니다. 반면 런타임의 동시성 모델이 실제로 중요한 연산(compute) 및 I/O 작업에서는 격차가 폭발적으로 커집니다.

## 알아둘 가치가 있는 주의사항 (The Caveats Worth Knowing)

Node.js 서버는 CVE-2026–25536(동시 요청 간 세션 데이터 유출)에 대한 보안 완화 조치로 요청당 서버 인스턴스(instance)를 생성하는 방식을 사용했습니다. 이러한 아키텍처(architecture) 선택은 오버헤드(overhead)를 추가했으며, 공유 인스턴스 패턴을 사용했다면 성능이 훨씬 더 좋았을 것입니다. 이 결과는 Node.js의 최고 성능이 아니라 보안 강화 구성(security-hardened configuration)을 반영합니다.

파이썬(Python) 또한 기본 설정(out of the box)인 단일 워커(single-worker) Uvicorn 설정으로 테스트되었습니다. uvloop를 사용한 다중 워커(multi-worker) 배포(deployment)는 처리량(throughput)을 상당히 향상시킬 수 있습니다.

이것이 중요한 이유는 벤치마크가 최적화된 프로덕션(production) 설정이 아닌, 현실적인 기본 구성(default configurations)을 측정하기 때문입니다. 파이썬(Python)이나 Node.js 배포를 확장성(scale)에 맞춰 튜닝한 팀이라면 다른 수치를 볼 수 있을 것입니다.

네 가지 언어 모두에서 공통적으로 나타난 한 가지는 바로 '오류율 제로'였습니다. 모든 390만 건의 요청이 언어와 관계없이 성공적으로 완료되었습니다. 프로토콜(protocol) 자체는 견고합니다.

## 이것이 여러분의 스택에 실제로 의미하는 바 (What This Actually Means for Your Stack)

내부 도구, 개발자용 MCP 서버 또는 프로토타입(prototype)을 구축한다면 파이썬(Python)과 Node.js는 완전히 합리적인 선택입니다. 개발자 경험(developer experience)이 더 빠르고, 생태계(ecosystem)가 풍부하며, 트래픽이 적은 환경에서는 사용자에게 지연 시간(latency) 차이가 느껴지지 않습니다.

하지만 MCP 서버가 실제 프로덕션(production) 부하를 처리해야 한다면 경제성은 빠르게 달라집니다. 파이썬(Python)으로 동일한 트래픽을 처리하려면 고(Go)보다 약 5배 더 많은 CPU가 필요합니다. 대규모 환경에서는 이는 실제 인프라(infrastructure) 비용으로 이어집니다.

고(Go)는 프로덕션(production) 환경에서 실용적인 기본 선택지로 돋보입니다. 자바(Java)와 동등한 성능을 제공하며, 메모리 사용량(memory footprint)은 12배 더 적고, JVM 워밍업(warmup)이 필요 없으며, 테스트 라운드 전반에 걸쳐 뛰어난 일관성(처리량 변동률 0.5%에 불과)을 보여줍니다. 절대적으로 가장 낮은 지연 시간(latency)이 필요하고 기존 JVM 생태계(ecosystem)를 활용할 수 있다면 자바(Java)가 우세합니다. 파이썬(Python)과 Node.js는 트래픽이 적은 시나리오나 개발 환경에 가장 적합합니다.

이번 벤치마크는 단 하나의 놀라운 결과를 내놓은 것이 아닙니다. 언어 런타임(runtime)은 항상 성능 차이가 있어왔습니다. 하지만 이 벤치마크는 개발자들이 흔히 '스타일 선호'로 여기는 결정에 명확한 수치를 제시했다는 데 의미가 있습니다.
