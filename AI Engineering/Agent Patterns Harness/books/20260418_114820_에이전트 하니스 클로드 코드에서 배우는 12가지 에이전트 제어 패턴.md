# 에이전트 하니스: 클로드 코드에서 배우는 12가지 에이전트 제어 패턴

대규모 언어 모델(LLM)을 위한 제어 계층 설계에 대한 실무자 가이드입니다.

AI 에이전트를 구축했습니다.

처음 다섯 단계 정도는 아름답게 작동합니다.

그러다 뭔가 고장 납니다.

존재하지 않는 함수를 환각합니다.
자신이 무엇을 하고 있었는지 잊어버립니다.
건드리지 말아야 할 파일을 덮어씁니다.
쓸모없는 루프에 빠져듭니다.

프롬프트를 조정해봅니다.
모델을 바꿔봅니다.
더 많은 컨텍스트를 추가합니다.

여전히 불안정합니다.

그래서 모델을 탓합니다.

하지만 불편한 진실은 이렇습니다.

문제는 모델이 아닙니다.
문제는 바로 여러분의 하니스(Harness)입니다.

멋진 데모와 프로덕션 수준의 AI 시스템을 가르는 것은 지능이 아니라 제어입니다.

고급 에이전트의 진정한 마법은 LLM 내부에 숨겨져 있지 않습니다.
그것은 LLM을 둘러싼 계층, 즉 다음을 결정하는 시스템에 있습니다.

*   모델이 무엇을 보는지
*   무엇을 할 수 있는지
*   실패로부터 어떻게 복구하는지
*   무엇이 중요한지 어떻게 기억하는지

이 계층에는 이제 이름이 있습니다.

**하니스 엔지니어링 (Harness Engineering)**

그리고 일단 이것을 이해하면, 다시는 에이전트를 똑같은 방식으로 구축하지 않을 것입니다.

## 하니스 엔지니어링은 실제로 무엇을 의미할까요?

대부분의 사람들은 AI 에이전트 구축이 프롬프트(prompt)에 관한 것이라고 생각합니다.

그렇지 않습니다.

프롬프트는 그저 표면적인 계층일 뿐입니다.
진정한 시스템은 그 아래에 존재합니다.

하니스 엔지니어링은 LLM을 둘러싼 제어 계층, 즉 모델이 시간이 지남에 따라 어떻게 작동할지를 형성하는 코드, 구조 및 규칙을 설계하는 관행입니다.

이는 다음 사이에 존재하는 모든 것입니다.

원시 모델 출력 → 실제 세계의 행동

### 더 정확한 정의

하니스 엔지니어링은 다음과 같습니다.

LLM이 컨텍스트(context)를 인식하고, 행동을 실행하고, 메모리를 관리하며, 실패를 처리하는 방식을 조절하는 주변 시스템의 의도적인 설계.

여기에는 다음이 포함됩니다.

*   **컨텍스트 구성 (Context construction)** (프롬프트에 무엇이 들어가는지)
*   **메모리 시스템 (Memory systems)** (단계별로 무엇이 유지되는지)
*   **도구 오케스트레이션 (Tool orchestration)** (에이전트가 무엇을 할 수 있는지)
*   **권한 경계 (Permission boundaries)** (무엇을 할 수 있도록 허용되는지)
*   **복구 로직 (Recovery logic)** (문제가 발생했을 때 어떻게 되는지)

그리고 핵심 통찰은 다음과 같습니다.

모델이 아니라 하니스(harness)가 에이전트의 신뢰성을 결정합니다.

## 이것이 지금 중요한 이유

최근 연구와 실제 구현 사례들이 한 가지를 명확히 보여주었습니다.

아무리 좋은 모델이라도 구조 없이는 실패합니다.

실제로 오늘날 에이전트 구축은 여전히 매우 수동적입니다.

*   실패를 관찰하고
*   휴리스틱(heuristics)으로 해결하며
*   끝없이 반복합니다.

이는 전통적인 소프트웨어 엔지니어링보다는…
유도된 제약 조건을 통해 시스템을 훈련시키는 것과 더 비슷합니다.

그렇기 때문에 하니스 엔지니어링은 세 가지 분야의 교차점에 있습니다.

### 1. 시스템 설계 (Systems Design)

여러분은 분산 시스템을 구축하는 것입니다.

*   상태(state)는 여러 단계를 거쳐 흐르고
*   결정은 미래의 행동에 영향을 미치며
*   포함되지 않으면 실패가 연쇄적으로 발생합니다.

나쁜 하니스는 취약한 에이전트를 만듭니다.
좋은 하니스는 탄력적인 시스템을 만듭니다.

### 2. UX: 하지만 모델을 위한 UX

이렇게 생각해보세요.

여러분의 LLM이 여러분의 사용자(user)입니다.

*   프롬프트는 UI
*   컨텍스트는 내비게이션
*   도구는 행위 유발성 (affordances)

경험이 혼란스럽거나, 과부하되거나, 일관성 없다면...

모델은 사람과 마찬가지로 제대로 작동하지 않습니다.

### 3. 프롬프트 아키텍처 (Prompt Architecture)

프롬프트는 여전히 중요하지만, 더 큰 시스템의 일부로서만 중요합니다.

다음과 같은 것들이 없으면:

*   구조화된 메모리
*   제어된 실행
*   단계별 워크플로

...아무리 좋은 프롬프트라도 실제 세계의 복잡성 앞에서 무너집니다.

## 누락된 정신 모델 (Missing Mental Model)

경험 많은 AI 엔지니어들이 직관적으로 해왔던 것이 이제 명시적으로 드러나고 있습니다.

여러분은 LLM을 단순히 "호출"하는 것이 아닙니다.
여러분은 LLM이 작동할 "환경"을 설계하는 것입니다.

그리고 그 환경, 즉 하니스가 다음을 결정합니다.

*   안정성 (Stability)
*   확장성 (Scalability)
*   안전성 (Safety)
*   성능 (Performance)

## 전환: 프롬프팅 → 엔지니어링

우리는 다음에서 다음으로 전환하고 있습니다.

**오래된 사고방식 → 새로운 사고방식**

"더 나은 프롬프트를 작성하라" → "더 나은 시스템을 설계하라"

상태 비저장 호출 → 상태 저장 실행

잘 되기를 바란다 → 행동을 보장한다

출력 디버깅 → 제약 조건 설계

## 대부분의 에이전트가 실패하는 이유

에이전트가 고장 나는 것은 모델이 "멍청해서"인 경우는 드뭅니다.

그것은 다음과 같은 이유 때문입니다.

*   컨텍스트가 과부하되거나 누락되었기 때문
*   메모리가 일관성 없기 때문
*   행동이 제약되지 않았기 때문
*   실패가 관리되지 않았기 때문

다시 말해:

하니스가 없거나, 제대로 설계되지 않았기 때문입니다.

## 획기적인 통찰 (Breakthrough Insight)

프로덕션 수준의 코딩 에이전트에 대한 심층 분석을 통해 강력한 사실이 밝혀졌습니다.

아마추어 에이전트와 프로덕션 준비 시스템의 차이는 지능이 아니라 **패턴**에 있습니다.

재사용 가능하고 조합 가능한 디자인 패턴 말입니다.

다음과 같은 패턴입니다.

*   동작을 안정화하고
*   워크플로를 구조화하며
*   복잡성을 관리하고
*   장기적인 자율성을 가능하게 합니다.

그리고 바로 이것이 우리가 다음에 탐구할 내용입니다.

취약한 에이전트를 신뢰할 수 있는 시스템으로 바꾸는 12가지 하니스 엔지니어링 패턴.

## 프로덕션 준비 에이전트를 위한 12가지 패턴

고급 AI 에이전트에서 "마법"처럼 보이는 것은 사실 구조(structure)입니다.

실제 시스템(Generative Programmer가 분석한 Claude Code 스타일 에이전트와 같은)의 상세한 분석은 중요한 사실을 밝혀냅니다.

프로덕션 에이전트는 더 똑똑한 것이 아니라, 더 잘 조직되어 있습니다.

그들은 소수의 재사용 가능한 패턴에 의존합니다. 일단 이 패턴들을 보면, 어디에서나 그것들을 발견하기 시작할 것입니다.

자, 이 패턴들을 분석해 봅시다.

### 메모리 및 컨텍스트 (5가지 패턴)

#### 1. 영구 지침 파일 (CLAUDE.md 패턴)

대부분의 에이전트는 세션 사이에 모든 것을 잊어버립니다.

프로덕션 에이전트는 그렇지 않습니다.

그들은 영구 지침 파일, 즉 다음과 같은 내용을 정의하는 살아있는 문서(종종 'living document' CLAUDE.md 또는 유사한 이름으로 불림)에 의존합니다.

*   코딩 표준
*   프로젝트 구조
*   명명 규칙
*   행동 규칙

이 파일은 항상 컨텍스트에 주입되어 장기적인 "성격 + 정책 계층" 역할을 합니다.

**예시:**
세션 간에 작업하는 코딩 에이전트는 다음을 절대 잊지 않습니다.

*   "JavaScript 대신 TypeScript를 사용하세요."
*   "모든 API 경로는 /api/v1/을 따라야 합니다."
*   "/core/에 있는 파일은 확인 없이 절대 수정하지 마세요."

매번 지침을 반복하는 대신, 하니스가 이를 강제합니다.

**사실:** 영구 지침 계층은 프롬프트 비대화(prompt bloat)를 줄이고 세션 간 출력의 안정성을 높입니다.

#### 2. 범위 지정 컨텍스트 구성 (Scoped Context Assembly)

전체 코드베이스를 컨텍스트에 덤핑하는 것은 실수입니다.

프로덕션 에이전트는 범위(scope)에 따라 관련 있는 것만 로드합니다.

다른 디렉터리 → 다른 규칙 → 다른 컨텍스트.

**예시:**
모노레포(monorepo)에서:

*   `/frontend/`는 UI 가이드라인을 로드합니다.
*   `/backend/`는 API 계약을 로드합니다.
*   `/infra/`는 배포 구성을 로드합니다.

하니스는 에이전트가 작동하는 위치에 따라 컨텍스트를 동적으로 구성합니다.

**사실:** 더 작고, 범위가 지정된 프롬프트는 정확성을 향상하고 환각 현상을 줄입니다.

#### 3. 계층화된 메모리 (3계층 설계)

모든 메모리가 동등하게 취급되어서는 안 됩니다.

프로덕션 시스템은 세 가지 계층을 사용합니다.

*   **컴팩트 인덱스 (Compact Index)** → 모든 것의 빠른 요약
*   **온디맨드 파일 (On-Demand Files)** → 상세한 주제별 메모리
*   **아카이브된 기록 (Archived Transcripts)** → 전체 기록, 거의 접근되지 않음

이는 컨텍스트 과부하를 방지하면서 장기적인 지식을 보존합니다.

**예시:**
5만 개의 토큰을 채워 넣는 대신:

*   에이전트는 요약 인덱스를 읽습니다.
*   필요할 때만 자세한 메모를 가져옵니다.
*   필요하지 않으면 오래된 기록은 무시합니다.

**사실:** 계층화된 메모리는 컨텍스트 한도에 도달하지 않고도 장기적인 작업을 가능하게 합니다.

#### 4. 드림 통합 (자동 드림 모드) (Dream Consolidation (AutoDream Mode))

여기서부터 흥미로워집니다.

프로덕션 에이전트는 메모리를 저장할 뿐만 아니라, 메모리를 정리합니다.

유휴 시간 동안 백그라운드 프로세스는 다음을 수행합니다.

*   중복을 제거하고
*   유사한 아이디어를 병합하며
*   불필요한 컨텍스트를 압축합니다.

이를 "AI를 위한 수면 모드"라고 생각할 수 있습니다.

**예시:**
한 기능에 대한 여러 번의 반복 후:

*   불필요한 메모가 병합되고
*   오래된 단계가 제거되며
*   최신 이해만 남습니다.

**사실:** 통합 없이는 메모리 시스템이 시간이 지남에 따라 저하됩니다.

#### 5. 점진적 컨텍스트 압축 (Progressive Context Compaction)

대화가 길어질수록 컨텍스트는 비싸집니다.

프로덕션 시스템은 다단계 요약을 사용하여 시간이 지남에 따라 컨텍스트를 압축합니다.

*   초기 단계 → 전체 상세 내용
*   중간 단계 → 요약된 청크
*   후기 단계 → 고수준 추상화

**예시:**
200단계 워크플로는 다음과 같이 됩니다.

*   마지막 5단계 (자세히)
*   마지막 20단계 (요약)
*   이전 단계 (압축된 통찰력만)

**사실:** 점진적 압축은 장기 실행 에이전트에게 필수적입니다.

### 워크플로 및 오케스트레이션 (3가지 패턴)

#### 6. 탐색 → 계획 → 실행 루프 (Explore → Plan → Act Loop)

아마추어 에이전트는 곧바로 행동에 뛰어듭니다.

프로덕션 에이전트는 그렇지 않습니다.

그들은 실행을 세 가지 명확한 단계로 분리합니다.

*   **탐색 (읽기 전용)** → 정보 수집
*   **계획** → 무엇을 할지 결정
*   **실행** → 변경 사항 적용

각 단계는 점차적으로 더 많은 권한을 가집니다.

**예시:**
코드를 편집하기 전에:

*   탐색: 파일을 읽습니다.
*   계획: 변경 사항을 개략적으로 작성합니다.
*   실행: 편집 사항을 적용합니다.

**사실:** 이는 파괴적인 실수를 극적으로 줄여줍니다.

#### 7. 병렬 워크스트림 (Parallel Workstreams)

하나의 에이전트를 사용할 수 있는데 왜 여러 에이전트를 사용하지 않을까요?

프로덕션 시스템은 독립적인 작업을 위해 여러 하위 에이전트를 생성합니다.

그리고 나서 결과를 병합합니다.

**예시:**
기능 요청의 경우:

*   에이전트 1 → 백엔드 로직
*   에이전트 2 → 프론트엔드 UI
*   에이전트 3 → 테스트

감독 에이전트가 모든 것을 결합합니다.

**사실:** 병렬화는 속도와 모듈성을 크게 향상시킵니다.

#### 8. 검증 게이트 (Verification Gate)

이것은 가장 중요한 패턴 중 하나입니다.

파괴적인 행동을 하기 전에 에이전트는 검증 단계를 통과해야 합니다.

**예시 확인 사항:**

*   "이것이 중요한 파일을 삭제합니까?"
*   "이것이 기존 기능을 손상시킵니까?"
*   "이것이 지침과 일치합니까?"

검증을 통과한 후에만 → 행동이 허용됩니다.

**사실:** 검증 게이트는 안전한 자율성에 필수적입니다.

### 도구 및 권한 (2가지 패턴)

#### 9. 최소 권한 범위 (Minimal Permission Scope)

에이전트에게 전체 접근 권한을 주지 마세요.

지금 당장 필요한 것만 주세요.

**예시:**

*   탐색 단계 → 읽기 전용 접근
*   계획 단계 → 도구 실행 금지
*   실행 단계 → 제한된 쓰기 접근

**사실:** 과도한 권한을 가진 에이전트는 파괴적인 실패의 가장 큰 원인입니다.

#### 10. 도구 어댑터 계층 (Tool Adapter Layer)

직접적인 도구 호출은 취약합니다.

프로덕션 시스템은 에이전트와 도구 사이에 어댑터 계층을 사용합니다.

이는 다음을 가능하게 합니다.

*   테스트
*   모의(mocking)
*   로깅
*   안전한 실행

**예시:**
API를 직접 호출하는 대신:

에이전트 → 어댑터 → API

어댑터는 테스트 중에 응답을 시뮬레이션할 수 있습니다.

**사실:** 이 추상화는 성숙한 소프트웨어 시스템에서 표준이며, 이제 AI 에이전트에도 적용됩니다.

### 자동화 (2가지 패턴)

#### 11. 헤드리스 배치 모드 (Headless Batch Mode)

모든 에이전트가 UI를 필요로 하는 것은 아닙니다.

프로덕션 시스템은 에이전트를 파이프라인의 일부로 백그라운드에서 실행합니다.

**예시:**

CI/CD 파이프라인은 에이전트를 실행하여 다음을 수행합니다.

*   코드 리팩터링
*   의존성 업데이트
*   테스트 생성

사람의 개입이 필요 없습니다.

**사실:** 헤드리스 에이전트는 진정한 기업 자동화를 가능하게 합니다.

#### 12. 자가 치유 루프 (Self-Healing Loop)

문제가 발생할 것입니다.

프로덕션 에이전트는 충돌하지 않고 복구합니다.

그들은 다음 실패 신호를 감지합니다.

*   오류 메시지
*   테스트 실패
*   유효하지 않은 출력

그리고 조정을 거쳐 재시도합니다.

**예시:**

*   API 호출 실패 → 에이전트가 수정된 매개변수로 재시도
*   테스트 실패 → 에이전트가 코드를 다시 작성하고 다시 실행

**사실:** 자가 치유는 장기 실행 자율 시스템에 필수적입니다.

## 큰 그림

개별적으로는 이러한 패턴들이 유용합니다.

하지만 함께하면 강력한 것이 됩니다.

바로 구조화되고, 탄력적이며, 프로덕션 수준의 에이전트 시스템입니다.

이것이 차이점입니다.

아마추어 에이전트 → 반응한다
프로덕션 에이전트 → 작동한다

그리고 이러한 패턴으로 설계하기 시작하면...

여러분은 "프롬프팅"을 멈추고 지능을 "엔지니어링"하기 시작할 것입니다.

## 지능형 하니스 런타임(IHR): 하니스를 실행 가능하게 만들기

이제 패턴들을 이해했을 것입니다.

메모리 시스템을 설계할 수 있고.
워크플로를 구조화할 수 있으며.
권한을 제어할 수 있습니다.

하지만 더 깊은 질문이 있습니다.

이 모든 것이 실제로 어떻게 실행될까요?

하니스는 단순히 디자인이 아니기 때문입니다.

그것은 살아있는 시스템입니다.

그리고 여기서 최근 연구에서 강력한 아이디어가 나옵니다.

**지능형 하니스 런타임 (Intelligent Harness Runtime, IHR)**

### 핵심 변화: 정적 하니스 → 실행 가능한 시스템

대부분의 개발자들은 하니스를 다음과 같은 구성처럼 취급합니다.

*   일련의 프롬프트
*   몇 가지 규칙
*   약간의 오케스트레이션 로직

하지만 고급 시스템에서 하니스는 수동적이지 않습니다.

그것은 런타임에 해석됩니다.

이것이 IHR의 핵심 아이디어입니다.

LLM은 단순히 출력을 생성하는 것이 아니라, 하니스 자체를 적극적으로 해석하고 실행합니다.

### IHR은 실제로 무엇을 할까요?

각 단계에서 시스템은 의사 결정 주기(decision cycle)를 반복합니다.

*   하니스(규칙 + 구조)를 읽습니다.
*   현재 상태(메모리 + 파일 + 진행 상황)를 읽습니다.
*   환경(도구 + 출력 + 제약 조건)을 읽습니다.
*   런타임 헌장(예산, 권한, 실패 규칙)을 따릅니다.
*   다음 최적의 행동을 결정합니다.

그리고 반복합니다.

이는 전통적인 에이전트와 근본적으로 다른 것을 만듭니다.

모델이 무엇을 출력해야 할지뿐만 아니라 어떻게 행동해야 할지에 대해 지속적으로 추론하는 폐쇄 루프 시스템(closed-loop system)입니다.

### 이것이 중요한 이유

IHR이 없으면 대부분의 에이전트는 다음과 같이 작동합니다.

입력 → 출력 → 완료

IHR을 사용하면 동작이 다음과 같이 됩니다.

입력 → 추론 → 행동 → 관찰 → 조정 → 반복

이 루프는 다음을 가능하게 합니다.

*   장기적인 작업
*   적응적 행동
*   자체 수정
*   제약 조건을 통한 다단계 추론

다시 말해:

IHR은 LLM을 함수(function)에서 런타임(runtime)으로 전환시킵니다.

### IHR의 세 가지 핵심 구성 요소

이것이 실제로 어떻게 작동하는지 이해하려면 세 부분으로 나누어 살펴보세요.

#### 1. 루프 내 LLM 인터프리터 (The In-Loop LLM Interpreter)

이것은 루프 안에 있는 두뇌입니다.

LLM을 한 번 호출하는 대신, 지속적으로 루프 안에 유지합니다.

각 단계에서 모델은 다음을 수행합니다.

*   하니스를 해석하고
*   현재 상태를 평가하며
*   다음 행동을 선택합니다.

더 이상 단순히 질문에 답하는 것이 아닙니다.

제약 조건 하에서 결정을 내리는 것입니다.

**예시**

버그를 수정하는 코딩 에이전트를 상상해 보세요.

*   1단계 → 오류 로그를 읽습니다.
*   2단계 → 관련 파일을 검사합니다.
*   3단계 → 수정을 제안합니다.
*   4단계 → 패치를 적용합니다.
*   5단계 → 테스트를 실행합니다.
*   6단계 → 실패 감지 → 재시도합니다.

각 단계는 LLM에 의해 동적으로 결정됩니다.

**사실:** 이 루프식 실행은 에이전트가 단순히 완료뿐만 아니라 반복이 필요한 작업을 처리할 수 있도록 합니다.

**핵심 통찰 (Key Insight)**

모델은 더 이상 상태 비저장(stateless)이 아닙니다.

자신의 환경을 인식하는 상태 인식 인터프리터(state-aware interpreter)가 됩니다.

#### 2. 백엔드 (도구 + 다중 에이전트 인터페이스) (The Backend (Tools + Multi-Agent Interface))

인터프리터만으로는 행동 없이는 쓸모가 없습니다.

여기서 백엔드가 등장합니다.

백엔드는 다음을 제공합니다.

*   터미널 접근 (파일 시스템, 코드 실행)
*   API 및 외부 도구
*   다중 에이전트 조정 인터페이스

이를 실행 계층이라고 생각할 수 있습니다.

**예시**

에이전트가 다음을 결정할 때:

*   "파일 읽기" → 백엔드가 가져옵니다.
*   "테스트 실행" → 백엔드가 명령을 실행합니다.
*   "하위 에이전트 생성" → 백엔드가 조정합니다.

LLM은 직접 작업을 수행하지 않습니다.

행동을 요청하고, 백엔드가 이를 이행합니다.

**이 계층이 중요한 이유**

구조화된 백엔드 없이는:

*   도구 호출이 일관성 없어지고
*   출력이 신뢰할 수 없게 되며
*   디버깅이 불가능해집니다.

적절한 백엔드를 사용하면:

모든 행동은 관찰 가능하고, 제어 가능하며, 재현 가능합니다.

**다중 에이전트 확장 (Multi-Agent Extension)**

병렬 워크스트림도 여기에 존재합니다.

런타임은 다음을 수행할 수 있습니다.

*   하위 에이전트를 생성하고
*   작업을 할당하며
*   출력을 병합합니다.

이 모든 것이 동일한 시스템 내에서 이루어집니다.

#### 3. 런타임 헌장 (시스템 규칙) (The Runtime Charter (Rules of the System))

이것은 가장 저평가되고, 가장 중요한 구성 요소입니다.

런타임 헌장은 다음을 정의합니다.

*   유효한 상태로 간주되는 것
*   허용되는 행동
*   실패의 모습
*   복구가 이루어져야 하는 방식
*   절대 위반되어서는 안 되는 제약 조건

본질적으로 에이전트의 헌법입니다.

**예시**

런타임 헌장에는 다음이 포함될 수 있습니다.

*   "검증 없이 프로덕션 파일을 절대 수정하지 마십시오."
*   "실패한 API 호출은 최대 3회까지 재시도하십시오."
*   "테스트 커버리지가 80% 미만으로 떨어지면 중단하십시오."
*   "모든 파일 쓰기 작업을 기록하십시오."

이것은 프롬프트가 아닙니다.

이것은 강제된 행동 계약입니다.

**이것이 모든 것을 바꾸는 이유**

대부분의 에이전트는 규칙이 암묵적이기 때문에 실패합니다.

런타임 헌장은 규칙을 명시적으로 만듭니다.

"모델이 잘 행동하기를 바란다"를 "시스템이 잘 행동하는 것을 보장한다"로 바꿉니다.

**실패 처리 (Failure Handling)**

가장 큰 장점 중 하나는:

실패가 더 이상 예측 불가능하지 않다는 것입니다.

실패는 다음과 같습니다.

*   분류되고
*   예상되며
*   체계적으로 처리됩니다.

### 모든 것을 종합하기

이 세 가지 구성 요소를 결합하면 다음을 얻게 됩니다.

*   사고 루프 (LLM 인터프리터)
*   실행 계층 (백엔드 도구)
*   지배 시스템 (런타임 헌장)

이것들이 함께 형성하는 것은 바로:

**실행 가능한 하니스(executable harness)** 입니다.

### 정신 모델 업그레이드 (The Mental Model Upgrade)

**IHR 이전:**

프롬프트를 작성합니다.
좋은 출력을 기대합니다.

**IHR 이후:**

시스템을 정의합니다.
에이전트는 그 시스템 내에서 작동합니다.

### 진정한 돌파구 (The Real Breakthrough)

이것이 최근 연구와 실제 시스템에서 나타나는 더 깊은 통찰력입니다.

AI 에이전트의 미래는 더 나은 프롬프트가 아닙니다.

더 나은 런타임입니다.

## 시작하는 방법: 5단계 하니스 설계 프로세스

이 시점에서 아이디어가 복잡하게 느껴질 수 있습니다. 하지만 시작하는 것은 사실 간단합니다.

다음은 오늘 바로 따를 수 있는 실용적인 5단계 프로세스입니다.

### 1. CLAUDE.md (또는 이에 상응하는) 파일을 먼저 작성하세요.

무엇을 구축하기 전에 에이전트의 규칙을 정의하세요.

이 파일에는 다음이 포함되어야 합니다.

*   코딩 표준
*   제약 조건
*   해야 할 것과 하지 말아야 할 것
*   예상되는 행동

이를 에이전트의 운영 매뉴얼로 생각하세요.
이것이 없으면 모든 세션이 처음부터 시작됩니다.

### 2. 작업을 단계별로 매핑하세요.

워크플로를 다음과 같은 명확한 단계로 나눕니다.

*   탐색 (읽기 전용)
*   계획 (추론 전용)
*   실행 (실행)

그런 다음 각 단계에 권한을 할당합니다.

예를 들어:

*   탐색 → 파일만 읽기
*   실행 → 제한된 쓰기 접근

이렇게 하면 대부분의 파괴적인 실수를 방지할 수 있습니다.

### 3. 중요한 상태를 외부화하세요.

어떤 정보가 다음 상황에서 생존해야 하는지 결정하세요.

*   긴 작업
*   컨텍스트 한도
*   재시작

그런 다음 (프롬프트 안에만 넣지 말고) 파일에 저장하세요.

예시:

*   요약
*   결정
*   중간 출력

중요한 것은 컨텍스트 창(context window) 밖에 존재해야 합니다.

### 4. 실패 분류 체계를 정의하세요.

에이전트가 실패할 수 있는 모든 방법을 나열합니다.

*   잘못된 출력
*   도구 오류
*   불완전한 작업
*   안전하지 않은 행동

그런 다음 각 경우에 무엇이 발생해야 하는지 정의합니다.

*   재시도
*   롤백
*   에스컬레이션
*   중지

이렇게 하면 실패가 놀라움이 아닌 관리 가능한 이벤트가 됩니다.

### 5. 작은 검증 하니스를 먼저 구축하세요.

전체 시스템으로 시작하지 마세요.

대신:

*   최소한의 설정을 만들고
*   하나의 워크플로를 테스트하며
*   행동을 검증하세요.

안정된 후에만 확장하세요.

이렇게 하면 복잡성이 줄어들고 학습 속도가 빨라집니다.

## 결론

이제 여러분은 하니스를 직접 설계할 수 있습니다.

패턴을 이해했고.
시스템을 구조화하는 방법을 알며.
