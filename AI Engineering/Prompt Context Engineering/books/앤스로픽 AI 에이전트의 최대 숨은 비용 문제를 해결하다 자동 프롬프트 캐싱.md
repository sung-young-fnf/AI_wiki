# 앤스로픽, AI 에이전트의 최대 숨은 비용 문제를 해결하다 (자동 프롬프트 캐싱)

단 한 번의 설정 변경만으로 클로드(Claude) API 비용을 10분의 1 수준으로 줄이고, 매 API 호출마다 발생하던 불필요한 비용 누수를 막을 수 있습니다.

그렇습니다.

에이전트가 클로드(Claude)를 호출할 때마다 매번 동일한 시스템 프롬프트, 도구 정의, 지침을 반복해서 전송하며 비용을 지불하고 있다는 사실을 알고 계셨나요?

이는 클로드가 해당 내용을 반복해서 읽어야 하기 때문이 아닙니다.

API가 **상태 비저장 방식(stateless)**이기 때문입니다. 즉, API는 이전 대화의 내용을 기억하지 못하므로, 여러분의 실행 환경은 매 요청마다 모든 내용을 처음부터 다시 보내야만 합니다.

예를 들어, 10,000 토큰의 시스템 프롬프트를 사용하는 코딩 에이전트를 50번의 턴(turn) 동안 실행한다고 가정해 봅시다. 여러분은 이미 첫 번째 턴에서 보냈던 것과 동일한 지침을 위해 500,000 토큰만큼의 비용을 조용히 지불하고 있는 셈입니다.

대부분의 개발자는 청구서를 보기 전까지 이 사실을 깨닫지 못합니다. 앤스로픽(Anthropic)은 최근 이 문제를 해결할 새로운 기능을 발표했습니다.

이제 클로드 API는 시스템 지침, 도구 정의, 컨텍스트와 같은 프롬프트의 정적인 부분을 자동으로 캐싱하고 이를 다음 턴에서 재사용합니다.

**캐시 히트(Cache hit)** 비용은 표준 입력 토큰 가격의 10%에 불과합니다. 즉, 위 예시에서의 500,000 토큰 오버헤드가 단 50,000 토큰 비용으로 줄어드는 것입니다.

저는 코드 실행(code execution)과 MCP를 통해 토큰 사용량을 150,000개에서 2,000개로 줄여 AI 에이전트의 비대화 문제를 해결한 사례부터, '클로드 코드(Claude Code)' 작동 방식의 핵심인 프롬프트 캐싱 메커니즘에 이르기까지 앤스로픽의 인프라 발표를 밀착 취재해 왔습니다.

**자동 캐싱(Auto-caching)**은 API를 사용하는 모든 개발자가 수동 작업 없이도 이러한 혜택을 누릴 수 있게 해주는 핵심 요소입니다.

이 글에서는 프롬프트 캐싱의 작동 원리와 클로드 코드 팀이 실제 운영 과정에서 얻은 교훈을 분석해 보겠습니다.

### 프롬프트 캐싱의 작동 원리

사용법을 알아보기 전에, 내부에서 어떤 일이 일어나는지 이해해야 합니다.

#### 클로드를 호출할 때마다 일어나는 두 가지 단계

클로드 API에 요청을 보내면 모델은 두 가지 구별된 단계를 거칩니다.

1.  **프리필 단계(Prefill phase):** 클로드가 전체 프롬프트를 처리하는 단계입니다. 응답의 첫 단어가 생성되기 전에 시스템 지침, 도구 정의, 대화 기록의 모든 토큰을 읽고 계산합니다.
2.  **디코드 단계(Decode phase):** 클로드가 실제 응답을 토큰 단위로 생성하는 단계입니다.

여기서 비용이 많이 발생하는 단계는 바로 **프리필 단계**입니다.

만약 현재 프롬프트의 접두사(prefix)가 이전 요청과 일치한다면, 이미 수행된 프리필 계산을 반복할 필요가 없습니다. 그 결과값을 저장하고 해싱하여 재사용할 수 있습니다. 이것이 프롬프트 캐싱의 핵심 원리입니다.

#### 접두사 매칭(Prefix Match)

클로드의 캐싱은 정확한 **접두사 매칭**을 통해 작동합니다.

특정 블록에 `cache_control` 마크를 표시하면, API는 요청의 시작부터 해당 지점까지의 모든 내용에 대해 암호화 해시(cryptographic hash)를 생성합니다.

다음 요청에서 접두사가 동일하면 해당 해시를 찾아 프리필 단계를 건너뛰고, 표준 요율의 10%만 청구합니다.

이 접두사에서 단 한 글자라도 틀리면(타임스탬프가 다르거나, 도구 순서가 바뀌거나, 공백이 추가되는 등), 완전히 다른 해시가 생성됩니다. 이것이 바로 순서가 중요한 이유이며, 무해해 보이는 변경이 예기치 못한 비용 발생으로 이어지는 이유입니다.

### 실제 사례로 보는 토큰 계산

이해를 돕기 위해 실제 시나리오를 살펴봅시다. 코드 리뷰 에이전트를 구축한다고 가정해 보겠습니다.

*   **지침 및 페르소나가 포함된 시스템 프롬프트:** 8,000 토큰
*   **도구 정의 (파일 읽기, 검색, 테스트 실행 등):** 4,000 토큰
*   **프로젝트 컨텍스트 (CLAUDE.md, 코딩 표준):** 3,000 토큰

사용자의 첫 메시지가 전달되기도 전에 이미 **15,000 토큰의 정적 컨텍스트**가 발생합니다. 이 에이전트가 파일을 읽고, 체크하고, 결과를 보고하며 40번의 대화를 주고받는다고 합시다.

*   **캐싱이 없을 때:** 15,000 토큰 × 40턴 = 600,000 입력 토큰 비용 청구
*   **캐싱을 사용할 때:**
    *   1턴: 15,000 토큰 (캐시 작성 비용)
    *   2~40턴: 15,000 토큰 × 39턴 × 10% = 58,500 토큰 비용 청구
    *   **총 정적 토큰 비용:** 15,000 + 58,500 = **73,500 토큰**

단일 세션에서 정적인 부분에 대해서만 비용이 600,000개에서 73,500개로 줄어들었습니다. 수백 명의 동시 사용자가 유사한 세션을 실행하는 규모에서는, 이 차이가 단순히 '있으면 좋은 기능'을 넘어 제품의 경제적 생존 여부를 결정짓는 요소가 됩니다.

### 캐시 중단점(Cache Breakpoint)의 작동 방식

자동 캐싱 기능이 나오기 전에는 개발자가 요청 내에서 캐싱할 위치를 `cache_control` 중단점을 사용하여 수동으로 지정해야 했습니다.

```json
{
  "messages": [
    { "role": "user", "content": "이 파일을 리뷰해줘" },
    { "role": "assistant", "content": "리뷰 결과입니다..." },
    {
      "role": "user",
      "content": "이제 테스트 코드를 확인해줘",
      "cache_control": { "type": "ephemeral" }
    }
  ]
}
```

이 중단점은 두 가지 역할을 합니다.
1. 해당 블록을 포함하여 그 이전의 모든 내용을 캐싱하도록 클로드에게 지시합니다.
2. 다음 요청 시, 일치하는 해시를 찾기 위해 이전 최대 20개 블록까지 역방향으로 검색하도록 지시합니다.

수동 방식의 문제는 대화가 길어짐에 따라 개발자가 계속해서 중단점을 최신 블록으로 옮겨줘야 한다는 점이었습니다. 한 번이라도 실수하면 캐시 히트를 놓치거나 순서가 어긋나, 이미 지불한 토큰 비용을 다시 전액 지불해야 했습니다.

### 자동 캐싱(Auto-Caching): 무엇이 바뀌었나?

자동 캐싱은 API 요청의 최상위 레벨에 단 하나의 `cache_control` 필드만 추가하면 됩니다.

```json
{
  "cache_control": { "type": "ephemeral" },
  "messages": [
    { "role": "user", "content": "이 파일을 리뷰해줘" },
    { "role": "assistant", "content": "리뷰 결과입니다..." },
    { "role": "user", "content": "이제 테스트 코드를 확인해줘" }
  ]
}
```

이제 API가 요청에서 가장 길게 일치하는 접두사를 자동으로 식별하고, 캐시 중단점을 마지막으로 캐싱 가능한 블록으로 이동시키며, 대화가 길어짐에 따라 이를 자동으로 재사용합니다.

이 기능 덕분에 긴 대화가 이어지는 코딩 에이전트나 문서 분석 파이프라인의 경제성이 확보됩니다. 전체 토큰의 60~70%를 차지하는 정적인 부분의 비용을 획기적으로 낮출 수 있기 때문입니다.

### 캐시 효율을 극대화하는 규칙

클로드 코드 팀은 프롬프트 캐싱을 중심으로 에이전트 실행 환경을 설계했습니다. 그들이 얻은 몇 가지 핵심 교훈은 다음과 같습니다.

#### 1. 프롬프트를 '정적인 내용'에서 '동적인 내용' 순으로 배치하세요
캐싱은 접두사 매칭 방식이므로 요청의 순서가 모든 것을 결정합니다.
*   **기본 시스템 지침** (모든 세션 공통)
*   **도구 정의** (모든 세션 공통)
*   **프로젝트 메모리(CLAUDE.md 등)** (프로젝트별 캐싱)
*   **세션 상태** (세션별 캐싱)
*   **대화 메시지** (턴마다 증가)

이 순서를 지키면 모든 세션이 캐시 히트로 시작됩니다. 시스템 프롬프트에 타임스탬프를 넣거나 도구 정의 순서가 매번 바뀌면 캐시가 깨져 비용이 전액 청구됩니다.

#### 2. 세션 중간에 도구를 변경하지 마세요
대화 도중에 도구를 추가하거나 제거하면 캐시된 접두사가 변경됩니다. 전체 캐시가 무효화되고 처음부터 다시 캐시를 구축해야 합니다.

#### 3. 프롬프트 수정 대신 시스템 메시지를 활용하세요
중간에 파일이 업데이트되거나 사용자 선호도가 바뀌었을 때 시스템 프롬프트 자체를 수정하고 싶은 유혹이 생기지만, 이는 해시를 깨뜨립니다. 대신 다음 사용자 메시지 내에 `<system-reminder>` 태그 등을 사용하여 업데이트된 정보를 전달하세요.

#### 4. 대화 도중에 모델을 교체하지 마세요
Opus로 긴 대화를 나누다가 간단한 질문을 위해 Haiku로 바꾸면, Haiku에는 해당 대화의 캐시가 없으므로 100,000 토큰의 컨텍스트를 처음부터 다시 구축하는 비용을 전액 지불해야 합니다. 저렴한 모델로 바꾸는 것이 오히려 더 비싼 결과를 초래할 수 있습니다.

#### 5. 컴팩션(Compaction): 세심한 컨텍스트 관리
컨텍스트 창이 가득 차서 대화를 요약하고 새 세션을 시작해야 할 때, 단순히 요약 API 호출을 따로 하면 캐시 효율이 떨어집니다. 부모 세션과 거의 동일한 시스템 프롬프트와 도구 구성을 유지하면서 요약 요청을 마지막 메시지로 추가하는 방식이 캐시 보존에 유리합니다.

### 결론

프롬프트 캐싱은 다음 두 가지 조건이 충족될 때 가장 큰 효과를 발휘합니다.
1. **정적 컨텍스트가 큼**
2. **세션이 길게 유지됨**

**코딩 에이전트, 문서 분석 파이프라인, 고객 지원 봇, 연구 어시스턴트**와 같은 유즈케이스에서 이 기능은 게임 체인저가 될 것입니다.

앤스로픽의 이번 **자동 캐싱** 기능은 그동안 개발자에게 전가되었던 복잡한 최적화 작업을 API 레벨에서 해결해 주었습니다. 이제 몇 번의 대화 이상을 주고받는 서비스를 구축하고 있다면, 이 기능을 무시할 이유가 전혀 없습니다. 지금 바로 적용하여 비용을 절감해 보세요!
