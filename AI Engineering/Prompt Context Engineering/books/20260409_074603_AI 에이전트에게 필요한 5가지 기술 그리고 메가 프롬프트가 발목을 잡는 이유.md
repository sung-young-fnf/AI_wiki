AI 에이전트에게 필요한 5가지 기술 (그리고 메가 프롬프트가 발목을 잡는 이유)

2026년, 단일 프롬프트에서 모듈형 스킬 아키텍처로의 전환은 실제 운영 AI 에이전트의 가장 큰 발전 요소입니다. 어떤 프레임워크에서도 이를 구현하는 방법을 소개합니다.

여러분의 에이전트 시스템 프롬프트는 5만 토큰에 달합니다. 조직이 문서화한 모든 정책, 모든 예외 상황, 모든 워크플로우(workflow)를 담고 있죠. 그리고 사용자가 환불 정책을 묻든 날씨를 묻든, 매 턴마다 그 모든 것에 대한 비용을 지불하고 있습니다.

이것이 바로 메가 프롬프트 시대이며, 지금 한계에 봉착했습니다.

서서히, 혹은 우아하게 무너지는 것이 아닙니다. 천정부지로 치솟는 비용, 지시 이행 능력 저하, 그리고 모델이 847번째 줄에서 지시한 내용을… 그냥 무시해 버리는 설명할 수 없는 오류와 같이 가장 큰 타격을 주는 방식으로 한계가 드러나고 있습니다.

더 나은 방법이 있습니다. 바로 스킬(Skills) 아키텍처이며, 이미 Claude, OpenAI Codex, GitHub Copilot, VS Code, 그리고 LangChain의 Deep Agents와 같은 오픈 소스 프레임워크 전반에서 수렴하고 있습니다. 가장 좋은 점은 특정 벤더에 종속되지 않는다는 것입니다.

### 이 글에서 다룰 내용:

*   **메가 프롬프트 문제:** 모든 것을 하나의 거대한 시스템 프롬프트에 담는 것이 왜 비싸고, 신뢰할 수 없으며, 에이전트 기능이 성장할수록 더 악화되는지.
*   **스킬(Skills) 아키텍처:** 점진적 공개(progressive disclosure)가 적용된 간단한 폴더 구조가 어떻게 단일 프롬프트를 모듈형의 조합 가능한 기능으로 대체하는지.
*   **크로스-프레임워크(Cross-Framework) 구현:** Pydantic AI 실습과 함께 모든 에이전트 프레임워크에서 스킬 런타임을 구축하기 위한 구체적인 패턴.
*   **프로덕션(Production) 준비:** 스킬을 지속 가능한 조직 자산으로 만드는 유효성 검사, 모니터링, 안전 완화 조치 및 거버넌스 모델.

### 메가 프롬프트 문제, 생각보다 심각합니다

모든 것을 하나의 시스템 프롬프트에 쌓아 올릴 때 실제로 어떤 일이 발생하는지 이야기해 봅시다.

첫째, 비용 문제입니다. 토큰 요금은 전송하는 텍스트에 비례하므로, 여러분이 추가하는 모든 지시사항, 모든 예외 상황, 모든 "만약을 대비한" 정책은 매 턴마다 비용이 청구됩니다. 5만 토큰짜리 시스템 프롬프트는 일회성 투자가 아닙니다. 모든 상호작용에 대한 반복적인 세금입니다.

하지만 비용이 전부가 아닙니다.

Liu 등 연구진의 연구에서 문서화된 "중간 실종(lost-in-the-middle)" 효과는 언어 모델이 긴 컨텍스트(context)에서 U자형 주의 곡선을 보인다는 것을 보여줍니다. 시작과 끝 부분의 지시사항은 잘 이행되지만, 중간에 묻힌 지시사항은 놓치거나, 중요도가 낮게 평가되거나, 아예 무시됩니다.

여기에 도구 스키마(tool schema)까지 추가해 봅시다. Anthropic 엔지니어링 팀의 실제 사례에서는 대화가 시작되기도 전에 58개의 도구가 약 5만 5천 토큰을 소모했습니다. 추가적인 MCP 서버를 사용하면 도구 정의 오버헤드(overhead)가 10만 토큰 이상에 달할 수 있습니다. 이는 단 하나의 사용자 메시지가 도착하기도 전에 전체 컨텍스트 창이 기능 설명으로 채워진다는 의미입니다.

이 도표가 모든 것을 설명합니다. 여러분의 에이전트는 항상 등에 도서관을 짊어지고 다니며, 생각할 여유 공간이 거의 없습니다.

엔지니어링 교훈은 명확합니다. 컨텍스트 내 불필요한 토큰은 중립적이지 않습니다. 예산을 소모하고, 지연 시간을 증가시키며, 지시 이행 능력을 저하시킵니다. 우리는 다른 아키텍처가 필요합니다.

### 스킬(Skills) 도입: AI 에이전트를 위한 점진적 공개(Progressive Disclosure)

해결책은 더 나은 메가 프롬프트를 작성하는 것이 아닙니다. 메가 프롬프트 작성을 완전히 중단하는 것입니다.

에이전트 스킬(Agent Skills) 아키텍처는 하나의 거대한 프롬프트를 모듈형 기능 라이브러리로 대체하며, 각 기능은 간단한 폴더로 패키징됩니다. 에이전트는 필요할 때 필요한 것만 로드합니다. 이는 도서관의 모든 책을 들고 다니는 것과, 카드 목록을 들고 다니다가 주제가 나오면 책꽂이에서 책을 꺼내는 것의 차이라고 생각할 수 있습니다.

디스크에서 스킬은 놀랍도록 간단합니다. YAML 프론트매터(frontmatter)와 마크다운(Markdown) 본문을 포함하는 필수 `SKILL.md` 파일이 들어있는 폴더입니다. 선택적으로 더 깊이 있는 리소스를 위해 `scripts/`, `references/`, `assets/` 디렉터리를 추가할 수 있습니다.

```
my-skill/
├── SKILL.md          # 필수: 지시사항 + 메타데이터
├── scripts/          # 선택 사항: 실행 가능한 헬퍼 스크립트
│   └── validate.py
├── references/       # 선택 사항: 상세 문서, 예시
│   └── api-guide.md
└── assets/           # 선택 사항: 템플릿, 설정 파일
    └── template.json
```

YAML 프론트매터에는 두 가지 필드만 필요합니다. 이름(디렉터리 이름과 일치하는 케밥 케이스(kebab-case))과 해당 스킬이 무엇을 하고 언제 사용해야 하는지 설명하는 설명(description)입니다. 이것이 최소 실행 가능 스킬입니다.

하지만 진정한 마법은 형식이 아닌, 로딩 전략에 있습니다.

### 3단계 점진적 공개(Progressive Disclosure) 모델

이것이 스킬을 작동시키는 핵심 아키텍처 개념입니다. agentskills.io의 공개 사양과 Anthropic의 개발자 문서 모두 동일한 3단계 모델을 설명하고 있습니다.

**1단계: 메타데이터(Metadata).** 시작 시 에이전트는 각 스킬의 이름과 설명 필드만 로드합니다. 이는 스킬당 약 100토큰에 해당합니다. 라이브러리에 50개의 스킬이 있더라도 검색에 총 5천 토큰이 사용됩니다. 이를 메가 프롬프트의 5만 토큰 이상과 비교해 보세요.

**2단계: 지시사항(Instructions).** 에이전트가 스킬이 현재 작업과 관련이 있다고 판단하면, `SKILL.md`의 전체 본문을 컨텍스트(context)에 로드합니다. 사양은 이를 5천 토큰 미만, 500줄 미만으로 유지할 것을 권장합니다. 더 깊이 있는 내용은 참조 파일로 푸시됩니다.

**3단계: 리소스(Resources).** 작업에 상세 참조 자료, 템플릿 또는 실행 가능한 헬퍼 스크립트가 필요한 경우, 에이전트는 `references/` 또는 `assets/`에서 파일을 로드하고 `scripts/`에서 코드를 실행할 수 있습니다. 이는 지시사항이 명시적으로 요구할 때만 발생합니다.

이 방식이 작동하는 핵심 제약 조건은 다음과 같습니다. 사양은 작성자에게 파일 참조를 `SKILL.md`로부터 "한 단계 깊이"로 유지하도록 지시합니다. 중첩된 검색 체인은 없습니다. 점진적 공개(progressive disclosure)는 모델이 필요할 때 다음 파일을 안정적으로 찾을 수 있을 때만 작동합니다.

### 실제 환경에서 스킬이 성공하는 이유

이 아키텍처는 우아하지만, 실제 운영 환경에서 어떤 이점을 가져다줄까요? 네 가지가 있습니다.

**관련성에 따라 확장되는 토큰 효율성.** 모든 요청에 대해 영구적인 "조직 지식세"를 지불하는 대신, 소량의 초기 검색 비용을 지불하고 필요할 때만 심층적인 지시를 로드합니다. 50개의 스킬 라이브러리는 검색에 약 5천 토큰이 들고, 하나의 활성화된 스킬을 로드하면 3~5천 토큰이 더 추가됩니다. 이는 동등한 메가 프롬프트의 5만 토큰 이상과 비교하면 매 턴마다 1만 토큰 수준입니다.

**조합 가능성(Composability).** 스킬은 함께 존재하도록 설계되었습니다. "브랜드 보이스" 스킬, "보안 검토" 스킬, "릴리스 노트" 스킬은 단일 프롬프트 블롭(prompt blob)으로 병합하지 않고도 다양한 워크플로우(workflow)에서 조합될 수 있습니다. Anthropic의 지침은 에이전트가 여러 스킬을 동시에 로드할 수 있으며, 작성자는 스킬이 서로 배타적이라고 가정하지 않고 함께 작동하도록 작성해야 한다고 명시합니다.

**프레임워크 및 팀 간의 이식성(Portability).** 이 부분이 사람들을 놀라게 합니다. 에이전트 스킬은 Claude 전용 기능이 아닙니다. `SKILL.md`와 YAML 프론트매터(frontmatter)를 사용하는 동일한 폴더 구조는 OpenAI의 Codex, GitHub Copilot, VS Code 에이전트 및 LangChain의 Deep Agents 프레임워크 전반에서 문서화되고 지원됩니다. 여러분이 중요하게 생각하는 아티팩트(artifact)(스킬 폴더)는 모델에 구애받지 않습니다. 런타임 정책(검색 방식, 샌드박스 존재 여부)만이 특정 하네스(harness)에 연결됩니다.

**버전 제어(Version Control) 및 거버넌스(Governance).** 파일 기반 스킬은 Git 워크플로우(workflow)와 자연스럽게 호환됩니다. 코드 검토, 태그 지정, 감사, 롤백이 가능합니다. 조직은 절차적 지식을 UI에 붙여넣기된 비공식적인 행동이 아닌, 검토된 아티팩트(artifact)로 취급할 수 있습니다.

### 모든 프레임워크에서 스킬 런타임 구축하기

이제 실용적인 부분입니다. 스킬을 구현하기 위해 특정 벤더의 SDK가 필요하지 않습니다. 일반적인 알고리즘은 5단계이며, 모든 에이전트 프레임워크에 적용될 수 있습니다.

이 글의 나머지 부분에서는 작동하는 스킬 런타임을 구축할 것입니다. 검색 레지스트리로 시작하여 활성화 로직을 추가하고, Pydantic AI를 위한 완전한 통합 패턴으로 마무리할 것입니다. 결국 여러분은 자신만의 에이전트 스택에 맞게 적용할 수 있는 코드를 갖게 될 것입니다.

### 1단계: 스킬 검색(Skill Discovery)

첫 번째 작업은 디렉터리에서 스킬을 스캔하고 해당 메타데이터(metadata)를 레지스트리에 파싱(parsing)하는 것입니다.

```python
import os
import yaml
from dataclasses import dataclass, field
from pathlib import Path

@dataclass
class SkillMetadata:
    name: str
    description: str
    path: Path
    metadata: dict = field(default_factory=dict)

class SkillRegistry:
    def __init__(self, skill_dirs: list[str]):
        self.skills: dict[str, SkillMetadata] = {}
        for directory in skill_dirs:
            self._scan_directory(Path(directory))
    
    def _scan_directory(self, base_path: Path):
        """SKILL.md 파일을 스캔하고 프론트매터를 파싱합니다."""
        for skill_dir in base_path.iterdir():
            skill_file = skill_dir / "SKILL.md"
            if skill_dir.is_dir() and skill_file.exists():
                meta = self._parse_frontmatter(skill_file)
                if meta:
                    meta.path = skill_dir
                    self.skills[meta.name] = meta
    
    def _parse_frontmatter(self, path: Path) -> SkillMetadata | None:
        """SKILL.md에서 YAML 프론트매터를 추출합니다."""
        content = path.read_text()
        if not content.startswith("---"):
            return None
        _, fm, _ = content.split("---", 2)
        data = yaml.safe_load(fm)
        return SkillMetadata(
            name=data["name"],
            description=data["description"],
            path=path.parent,
            metadata=data.get("metadata", {})
        )
    
    def get_catalog(self) -> str:
        """모델 컨텍스트를 위한 간결한 카탈로그를 반환합니다 (스킬당 ~100토큰)."""
        lines = []
        for skill in self.skills.values():
            lines.append(f"- {skill.name}: {skill.description}")
        return "\n".join(lines)
```

이를 통해 시작 시 스킬 디렉터리를 스캔하고 간결한 카탈로그를 구축하는 레지스트리를 얻게 됩니다. `get_catalog()`가 이름과 설명만 반환한다는 점에 주목하세요. 이것이 1단계입니다. 모델이 사용 가능한 것이 무엇인지 알기에 충분한 정보입니다.

### 2단계: 스킬 활성화(Skill Activation)

레지스트리가 구축되었으니 스킬을 검색할 수 있습니다. 하지만 에이전트가 실제로 스킬을 로드할 방법이 필요합니다. 활성화 로직을 추가해 봅시다. 이는 `SKILL.md`의 전체 본문을 컨텍스트(context)로 읽어들이는 것을 의미합니다.

```python
class SkillRegistry:
    # ... 이전 메서드들 ...
    
    def activate_skill(self, name: str) -> str:
        """SKILL.md 전체 본문을 로드합니다 (2단계 활성화)."""
        if name not in self.skills:
            raise ValueError(f"알 수 없는 스킬: {name}")
        
        skill = self.skills[name]
        content = (skill.path / "SKILL.md").read_text()
        
        # 프론트매터를 제거하고 본문만 반환합니다.
        parts = content.split("---", 2)
        body = parts[2].strip() if len(parts) >= 3 else content
        return body
    
    def load_reference(self, name: str, ref_path: str) -> str:
        """스킬에서 참조 파일을 로드합니다 (3단계 리소스)."""
        skill = self.skills[name]
        full_path = skill.path / ref_path
        
        # 보안: 경로가 스킬 디렉터리를 벗어나지 않도록 합니다.
        if not full_path.resolve().is_relative_to(skill.path.resolve()):
            raise ValueError("참조 경로가 스킬 디렉터리를 벗어납니다.")
        
        return full_path.read_text()
```

이제 우리의 레지스트리는 모든 세 가지 수준을 처리합니다. `get_catalog()`는 검색을, `activate_skill()`은 지시사항 로드를, `load_reference()`는 더 깊이 있는 리소스를 가져옵니다. `load_reference()`의 경로 탐색(path traversal) 검사에 주목하세요. 실행 가능한 코드를 번들링(bundling)하는 스킬은 공급망과 유사한 공격 표면(supply-chain attack surface)을 도입하므로, 처음부터 방어적으로 접근해야 합니다.

### 3단계: Pydantic AI에 통합하기

검색 및 활성화 로직이 준비되었으므로, 이를 실제 에이전트 프레임워크에 연결해 봅시다. Pydantic AI의 도구 등록 및 실행 그래프 인트로스펙션(run graph introspection)은 이러한 통합에 자연스럽게 적합합니다.

```python
from pydantic_ai import Agent

# 스킬 디렉터리로 레지스트리를 초기화합니다.
registry = SkillRegistry(skill_dirs=["./skills", "./org-skills"])

# 시스템 프롬프트에 스킬 카탈로그가 포함된 에이전트를 생성합니다.
agent = Agent(
    model="claude-sonnet-4-5-20250929",
    system_prompt=(
        "당신은 전문 스킬에 접근할 수 있는 유용한 비서입니다.\n\n"
        "사용 가능한 스킬:\n"
        f"{registry.get_catalog()}\n\n"
        "사용자 요청이 스킬과 일치하면, activate_skill 도구를 사용하여 지시를 로드한 후 진행하세요."
    ),
)

@agent.tool_plain
def activate_skill(name: str) -> str:
    """스킬의 전체 지시를 컨텍스트에 로드합니다.
    사용자 요청이 사용 가능한 스킬과 일치할 때 사용합니다."""
    return registry.activate_skill(name)

@agent.tool_plain
def load_skill_reference(skill_name: str, ref_path: str) -> str:
    """활성화된 스킬에서 참조 파일을 로드합니다.
    스킬 지시가 특정 파일을 지시할 때만 사용합니다."""
    return registry.load_reference(skill_name, ref_path)

@agent.tool_plain
def run_skill_script(skill_name: str, script_path: str, args: str = "") -> str:
    """스킬에 번들된 스크립트를 실행합니다.
    스킬 지시가 명시적으로 요구할 때만 사용합니다."""
    import subprocess
    skill = registry.skills[skill_name]
    full_path = skill.path / script_path
    
    # 보안 검사
    if not full_path.resolve().is_relative_to(skill.path.resolve()):
        return "오류: 스크립트 경로가 스킬 디렉터리를 벗어납니다."
    if not full_path.exists():
        return f"오류: 스크립트를 {script_path}에서 찾을 수 없습니다."
    
    result = subprocess.run(
        f"{full_path} {args}",
        shell=True, capture_output=True, text=True, timeout=30
    )
    return result.stdout or result.stderr
```

이것이 전체 스킬 런타임입니다. 에이전트는 시스템 프롬프트에 가벼운 카탈로그를 가지고 시작하며, 필요할 때 `activate_skill`을 사용하여 전체 지시를 로드하고, 더 깊이 있는 작업을 위해 참조를 가져오거나 스크립트를 실행할 수 있습니다. 모든 조각은 이전 조각 위에 구축됩니다.

### 크로스-프레임워크(Cross-Framework) 이야기

핵심 통찰은 이것입니다. 이 동일한 패턴은 어디에서나 작동합니다. 검색-활성화-실행 흐름은 Pydantic AI에만 국한되지 않습니다. LangChain의 Deep Agents는 동일한 메커니즘을 설명합니다. 시작 시 프론트매터(frontmatter)를 파싱하고, 일치하는 스킬이 발견되면 전체 스킬을 로드합니다. OpenAI의 Codex Skills 문서도 동일한 `SKILL.md` 구조와 동일한 점진적 공개(progressive disclosure) 동작을 사용합니다.

`@agent.tool_plain` 데코레이터를 LangChain의 `@tool`이나 OpenAI의 함수 호출 스키마(function calling schema), 또는 사용자 지정 프레임워크의 일반 함수로 바꾸세요. 스킬 폴더는 변하지 않습니다.

### 스킬을 프로덕션(Production) 준비 완료 상태로 만들기

스킬은 강력합니다. 또한 에이전트 동작에 영향을 미치는 코드이므로, 모든 프로덕션 종속성(production dependency)에 적용하는 것과 동일한 엄격함이 필요합니다.

### 유효성 검사(Validation): 측정할 내용

Anthropic의 공식 지침은 모든 프레임워크에 적용될 수 있는 세 가지 구체적인 성공 기준을 권장합니다.

**활성화 정확도.** 스킬이 관련 쿼리(query)의 90% 이상에서 트리거(trigger)되는가? 10~20개의 대표적인 프롬프트로 테스트하고 자동 활성화와 수동 호출을 비교 측정합니다.

**워크플로우(Workflow) 효율성.** 스킬이 합리적인 수의 도구 호출로 작업을 완료하는가? 스킬 사용 전후의 실행을 비교하고 품질과 토큰 소비량 모두를 추적합니다.

**신뢰성.** 워크플로우(workflow)당 실패한 도구 호출은 0개 (또는 거의 0개)여야 합니다. 오류, 재시도, 폴백(fallback) 비율을 추적합니다.

프레임워크별 관측 가능성(observability)을 위해 Pydantic AI의 평가 시스템은 OpenTelemetry 트레이스(trace)를 기반으로 구축된 스팬(span) 기반 평가를 지원합니다. 이는 최종 출력뿐만 아니라 내부 에이전트 동작(도구 호출, 활성화 패턴, 실행 흐름)을 평가할 수 있다는 의미입니다. 이것이 바로 스킬이 요구하는 종류의 평가입니다.

### 안전(Safety): 스킬은 종속성(Dependency)입니다. 그렇게 다루세요.

이 부분에서 많은 팀이 예상치 못한 상황에 처합니다. 스킬은 `scripts/`에 실행 가능한 코드를 번들링(bundling)할 수 있고 에이전트에게 추가 리소스를 로드하도록 지시할 수 있기 때문에, 공급망 공격 표면(supply-chain attack surface)을 도입합니다.

연구 결과가 이를 뒷받침합니다. 대규모 실증 연구에 따르면 분석된 스킬의 26% 이상이 프롬프트 인젝션(prompt injection), 데이터 유출(data exfiltration), 권한 상승(privilege escalation)을 포함한 최소 한 가지의 취약점 패턴을 포함하고 있었습니다. 실행 가능한 스크립트를 번들링(bundling)하는 스킬은 지시 전용 스킬보다 취약점을 포함할 가능성이 2.12배 높았습니다.

**구체적인 완화 조치:**

*   사양의 실험적인 `allowed-tools` 프론트매터(frontmatter)를 사용하여 스킬이 호출할 수 있는 도구를 제한하세요. 스킬이 파일 읽기만 필요한 경우 광범위한 실행 권한을 부여하지 마세요.
*   중요한 유효성 검사 단계에는 결정론적 스크립트를 선호하세요. Anthropic의 지침이 잘 설명하듯이, 코드는 결정론적이지만 언어 해석은 그렇지 않습니다. 단계가 반드시 신뢰할 수 있어야 한다면, 스크립트로 만드세요.
*   실행을 샌드박스(sandbox)에 가두세요. 스킬 실행 환경에서 네트워크 접근이나 런타임(runtime) 패키지 설치를 허용하지 마세요.
*   서드파티 스킬(third-party skills)을 서드파티 종속성(dependency)처럼 취급하세요. 이는 출처 추적, 코드 검토, 서명, 그리고 개발(dev)에서 스테이징(staging), 프로덕션(production)으로의 단계적 배포를 의미합니다.

### 미래 대비(Future-Proofing) 명제

벤더 종속성(vendor lock-in)에 대한 결론은 예상보다 더 낙관적입니다.

스킬 폴더 형식(디렉터리 + `SKILL.md` + 선택적 리소스)은 agentskills.io에서 명시적으로 표준화되어 있습니다. 독점적인 UI 뒤에 숨겨져 있지 않습니다. 여러 주요 플랫폼이 이미 이 정확한 아티팩트(artifact) 경계에서 수렴하고 있습니다. Claude, OpenAI Codex, GitHub Copilot, VS Code, 그리고 LangChain의 Deep Agents는 모두 동일한 구조를 문서화합니다.

따라서 다음과 같은 방어 가능한 명제를 제시합니다. 스킬 폴더를 조직 에이전트 지식의 지속 가능한 단위로 투자하세요. 필요에 따라 런타임(runtime)을 교체하세요. 모델과 에이전트 하네스(harness)는 변하겠지만, 스킬 라이브러리는 이식하고, 감사하고, 지속적으로 개선할 수 있는 버전 관리된 자산입니다.

메가 프롬프트 시대는 시작할 수 있는 방법을 제공했습니다. 스킬 시대는 확장할 수 있는 방법을 제공합니다.

### 주요 요점

*   **메가 프롬프트의 한계는 현실입니다.** 토큰 비용은 프롬프트 크기에 비례하여 증가하고, "중간 실종(lost-in-the-middle)" 효과는 신중하게 작성된 지시사항이 무시되고 있음을 의미합니다. 점진적 공개(progressive disclosure)는 이 두 가지 문제를 모두 해결합니다.
*   **스킬은 그저 폴더일 뿐입니다.** YAML 프론트매터(frontmatter)와 마크다운(Markdown) 본문이 있는 `SKILL.md` 파일. 선택적인 스크립트와 참조 파일. 그게 전부입니다. 단순함이 바로 기능입니다.
*   **이 패턴은 프레임워크에 구애받지 않습니다.** 검색, 활성화, 실행의 5단계. Pydantic AI, LangChain, OpenAI 또는 사용자 지정 스택에서 작동합니다. 스킬 폴더는 이식 가능한 아티팩트(artifact)입니다.
*   **스킬을 프로덕션 종속성(production dependency)으로 취급하세요.** 활성화 정확도를 검증하고, 토큰 사용량을 모니터링하며, 실행을 샌드박스(sandbox)에 가두고, 서드파티 스킬(third-party skills)을 서드파티 코드 검토 방식과 동일하게 검토하세요.
