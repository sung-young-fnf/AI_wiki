단 하나의 변경만으로 Claude API(Application Programming Interface) 비용을 10분의 1로 줄이고, 모든 API 호출에서 발생하는 불필요한 지출을 막을 수 있습니다.

그렇습니다.

에이전트가 Claude를 호출할 때마다 동일한 시스템 프롬프트, 도구 정의, 명령에 대해 계속해서 비용을 지불하고 있다는 사실을 알고 계셨습니까?

Claude가 반복을 요구해서가 아니라,

API가 무상태(stateless)이기 때문입니다. 즉, 턴(turn) 사이에 아무것도 기억하지 못하므로, 여러분의 하니스(harness)가 모든 요청에 대해 처음부터 모든 것을 다시 전송해야 합니다.

유료 Medium 회원이 아닌 경우 여기에서 기사를 무료로 읽을 수 있지만, 제 작업을 지원하기 위해 Medium 가입을 고려해 주세요. 감사합니다!

10,000 토큰(token)짜리 시스템 프롬프트로 코딩 에이전트를 50턴 실행하면, 첫 번째 턴에 이미 보냈던 동일한 명령 500,000 토큰에 대해 모르는 사이에 비용을 지불하게 됩니다.

대부분의 개발자는 청구서를 받기 전까지는 이 문제를 알아차리지 못합니다. Anthropic이 이 새로운 해결책을 막 내놓았습니다.

Claude API는 이제 프롬프트의 정적 부분인 시스템 명령, 도구 정의, 컨텍스트(context)를 자동으로 캐싱(caching)하고 여러 턴에 걸쳐 재사용합니다.

캐시 히트(cache hit) 비용은 표준 입력 토큰 가격의 10%에 불과하며, 이는 500,000 토큰의 오버헤드(overhead)가 청구되는 50,000 토큰으로 감소한다는 의미입니다.

저는 Anthropic의 인프라(infrastructure) 출시 소식을 면밀히 다뤄왔습니다. 코드 실행(code execution)과 MCP를 통해 토큰 사용량을 150,000개에서 2,000개로 줄여 AI 에이전트 비대화 문제를 해결한 방법부터, Claude Code 작동 방식의 핵심인 프롬프트 캐싱 메커니즘(mechanism)에 이르기까지 말입니다.

자동 캐싱(auto-caching)은 수동 오버헤드(overhead) 없이 API를 기반으로 구축하는 모든 개발자가 이 모든 것을 활용할 수 있도록 하는 핵심 요소입니다.

이 글에서는 프롬프트 캐싱이 어떻게 작동하는지, 그리고 Claude Code 팀이 실제 환경(production)에서 이를 운영하면서 배운 교훈들을 자세히 설명하겠습니다.

## 프롬프트 캐싱 작동 방식

사용 방법을 알아보기 전에, 어떤 일이 일어나는지 이해해야 합니다.

### Claude 호출 시 발생하는 두 가지 현상

Claude API에 요청을 보내면, 모델은 두 가지 개별 단계를 실행합니다.

첫 번째는 **프리필 단계(prefill phase)**입니다. 이 단계에서 Claude는 전체 프롬프트를 처리합니다. 시스템 명령, 도구 정의, 대화 기록의 모든 토큰은 응답의 단 한 단어가 생성되기 전에 읽고 연산됩니다.
두 번째는 **디코드 단계(decode phase)**입니다. 이 단계에서 Claude는 실제 응답을 토큰(token) 단위로 생성합니다.

프리필 단계가 비용이 많이 드는 단계입니다.

프롬프트가 이전 요청과 접두사(prefix)를 공유하는 경우, 프리필 연산은 이미 완료되었으므로 다시 수행될 필요가 없습니다.

그 결과는 저장되고, 해시(hash)되어 재사용될 수 있습니다. 이것이 프롬프트 캐싱의 전체 전제입니다.

### 접두사 일치

Claude의 캐싱은 정확한 접두사 일치(exact prefix matching)를 통해 작동합니다.

`cache_control`로 블록(block)을 표시하면, API는 요청 시작부터 해당 지점까지의 모든 것에 대한 암호화 해시(cryptographic hash)를 생성합니다.

다음 요청에서 해당 접두사가 동일하면, 해시를 찾아 해당 토큰에 대한 프리필을 건너뛰고 표준 요금의 10%만 청구합니다.

해당 접두사 내의 단 한 글자 차이(예: 다른 타임스탬프(timestamp), 재배열된 도구, 추가 공백)라도 완전히 다른 해시를 생성합니다.

이것이 순서가 매우 중요한 이유이며, 무해해 보이는 일부 변경 사항이 아무런 경고도 없이 비용을 발생시킬 수 있는 이유입니다. (이에 대해서는 나중에 더 자세히 설명하겠습니다.)

### 실제 토큰 계산

실제 시나리오를 통해 쉽게 이해해 봅시다.

코드 검토 에이전트를 구축한다고 가정해 봅시다.

설정은 다음과 같습니다.

*   지침과 페르소나(persona)가 포함된 시스템 프롬프트: 8,000 토큰
*   도구 정의 (파일 읽기, 검색, 테스트 실행): 4,000 토큰
*   프로젝트 컨텍스트 (CLAUDE.md, 코딩 표준): 3,000 토큰

이는 단 하나의 사용자 메시지가 오기 전에 15,000 토큰의 정적 컨텍스트입니다.

이제 에이전트가 파일을 읽고, 검사를 실행하고, 결과를 보고하는 40턴의 주고받는 과정을 통해 검토 세션을 실행합니다.

**캐싱(caching) 없이:**

15,000 토큰 × 40턴 = 600,000 입력 토큰이 정가로 청구됩니다.

**캐싱(caching) 사용 시:**

*   1턴: 15,000 토큰 정가 (캐시 쓰기)
*   2~40턴: 15,000 토큰 × 39턴, 10% 가격 적용 = 58,500 토큰 청구
*   총 정적 토큰 비용: 15,000 + 58,500 = 73,500 토큰

단일 세션의 정적 부분만으로도 600,000 토큰에서 73,500 토큰으로 감소한 것입니다.

대규모로, 수백 명의 동시 사용자가 유사한 세션을 실행할 때, 이러한 차이는 더 이상 '있으면 좋은' 기능이 아니라 여러분의 제품이 재정적으로 실행 가능하게 만드는 이유가 됩니다.

### 캐시 중단점(Cache Breakpoint) 작동 방식

자동 캐싱(auto-caching) 이전에는 요청에 `cache_control` 중단점(breakpoint)을 수동으로 배치하여 Claude에게 정확히 어디를 캐싱할지 알려줘야 했습니다.

```json
{
  "messages": [
    { "role": "user", "content": "Review this file" },
    { "role": "assistant", "content": "Here are my findings..." },
    {
      "role": "user",
      "content": "Now check the tests",
      "cache_control": { "type": "ephemeral" }
    }
  ]
}
```

중단점은 두 가지 역할을 합니다.

1.  해당 블록까지의 모든 것을 캐싱하도록 Claude에게 지시합니다.
2.  그리고 다음 요청에서 Claude에게 최대 20개 블록까지 뒤로 검색하여 일치하는 해시를 찾도록 지시합니다.

이 수동 방식의 문제는 대화가 길어질수록 중단점을 최신 블록으로 계속 이동해야 한다는 점입니다.

한 턴을 놓치거나 순서를 잘못 지정하면, 캐시 히트(cache hit)를 놓치거나 이미 비용을 지불한 토큰에 대해 정가를 지불하게 됩니다.

바로 이 지점에서 자동 캐싱이 해결책을 제공합니다.

## 자동 캐싱(Auto-Caching): 무엇이 달라졌나

수동 캐싱도 작동했지만, 모든 부담은 개발자에게 있었습니다.

대화를 직접 추적하고, 중단점 위치를 결정하며, 새로운 메시지가 쌓일 때마다 매 턴마다 중단점을 앞으로 이동시켜야 했습니다.

간단한 앱에서는 관리할 수 있지만, 수십 턴에 걸쳐 실행되는 장기 에이전트의 경우 심각한 문제를 야기합니다.

### 하나의 매개변수(Parameter)

자동 캐싱을 사용하면, API 요청의 최상위 수준에 단 하나의 `cache_control` 필드를 추가합니다.

```json
{
  "cache_control": { "type": "ephemeral" },
  "messages": [
    { "role": "user", "content": "Review this file" },
    { "role": "assistant", "content": "Here are my findings..." },
    { "role": "user", "content": "Now check the tests" }
  ]
}
```

API는 요청에서 가장 긴 일치하는 접두사(prefix)를 자동으로 식별하고, 캐시 중단점(breakpoint)을 마지막으로 캐싱 가능한 블록(block)으로 이동시키며, 대화가 진행됨에 따라 모든 후속 턴에서 재사용합니다.

### 세션 예시 분석

*   **1턴 요청이 들어오면** — API는 시스템 프롬프트, 도구, 컨텍스트에 대한 캐시를 기록합니다.
*   **2턴 요청이 도착하면** — API는 1턴에서 일치하는 접두사를 찾아 캐시를 적중시키고, 여러분은 새 메시지에 대해서만 정가를 지불합니다.
*   **10턴, 20턴, 40턴** — 동일합니다. 캐싱된 접두사는 대화와 함께 성장하며, 새로운 토큰에 대해서만 정가로 비용이 청구됩니다.

이것이 장기 실행 코딩 에이전트와 문서 분석 파이프라인(pipeline)을 대규모로 재정적으로 실행 가능하게 만듭니다. 이는 전체 토큰 수의 60~70%를 쉽게 차지할 수 있는 프롬프트의 정적 부분에 해당합니다.

### 수동 캐싱과 함께 작동

더 세분화된 제어가 필요한 경우, 자동 캐싱이 블록 수준의 `cache_control`을 대체하지는 않습니다.

예를 들어, 시스템 프롬프트는 모든 사용자에게 전역적으로 공유되지만 프로젝트 컨텍스트는 사용자마다 변경되기 때문에, 시스템 프롬프트를 프로젝트 컨텍스트와 독립적으로 캐싱하고 싶다고 가정해 봅시다.

이러한 특정 블록에 명시적인 중단점(breakpoint)을 여전히 배치할 수 있으며, 자동 캐싱은 그 외의 모든 것을 처리합니다.

캐싱 규칙은 변경되지 않았고, 구현 오버헤드(overhead)만 달라졌습니다. 자동 캐싱은 중단점 관리의 수동 작업을 제거합니다.

하지만,

개발자들이 세션 도중에 스스로 캐시를 손상시키는 방식으로부터 보호해주지는 않으며, 이러한 실수는 생각보다 흔합니다.

## 캐시를 유지하는 규칙

Claude Code 팀은 전체 에이전트 하니스(harness)를 프롬프트 캐싱을 중심으로 구축했으며, 대부분의 팀이 가동 시간(uptime)을 모니터링하는 방식과 동일하게 캐시 적중률(cache hit rate)을 모니터링하고, 적중률이 떨어지면 경고 및 사고(incident)를 발생시킵니다.

이러한 교훈은 X에 공유된 Thariq의 기사에서 발췌한 것입니다.

그들이 배운 내용은 다음과 같습니다.

### 1. 프롬프트 순서를 정적에서 동적으로 정렬하세요

캐싱은 접두사 일치(prefix matching)를 통해 작동하므로, 요청의 순서가 모든 것을 결정합니다. Claude Code 팀은 모든 요청을 다음과 같이 구성합니다.

*   **기본 시스템 명령** — 모든 세션에서 전역적으로 캐싱됨
*   **도구 정의** — 모든 세션에서 전역적으로 캐싱됨
*   **CLAUDE.md 및 프로젝트 메모리** — 프로젝트별로 캐싱됨
*   **세션 상태** — 세션별로 캐싱됨
*   **대화 메시지** — 매 턴마다 증가

정적 콘텐츠를 먼저 배치하면 모든 세션이 캐시 히트로 시작됩니다. 동적 콘텐츠를 마지막에 배치하면 새로운 토큰에 대해서만 정가가 부과됩니다.

**이 순서를 깨뜨리는 요소:**

*   시스템 프롬프트에 포함된 타임스탬프(timestamp)
*   비결정론적 순서로 로드되는 도구 정의
*   세션 간에 미세하게 변경되는 매개변수(parameter)

이 중 어떤 것이라도 접두사를 변경하고, 다른 해시를 생성하며, 매 턴마다 경고 없이 정가 청구서를 안겨줄 것입니다.

### 2. 세션 중간에 도구를 변경하지 마세요

대화 중간에 도구를 추가하거나 제거하는 순간, 캐싱된 접두사가 변경됩니다. 전체 캐시가 사라지고, 처음부터 다시 구축하기 위해 정가를 지불하게 됩니다.

Claude Code 팀이 사용하는 해결책: 모드가 변경될 때 도구를 바꾸는 대신, `EnterPlanMode`와 `ExitPlanMode`를 자체 도구로 추가했습니다.

도구 정의는 절대 변경되지 않습니다. 모델은 현재 모드를 설명하는 시스템 메시지를 받지만, 접두사는 동일하게 유지됩니다.

### 3. 프롬프트 편집 대신 시스템 메시지를 사용하세요

세션 중간에 무언가가 변경될 때 (파일 업데이트, 시간 변경, 사용자 선호도 등) 본능적으로 시스템 프롬프트를 편집하게 됩니다.

하지만,

모든 편집은 해시를 손상시키고 전체 캐시 재구축을 유발합니다. 대신, 업데이트된 정보를 다음 사용자 메시지 또는 도구 결과 내부에 `<system-reminder>` 태그로 전달하세요.

모델은 이를 읽고 업데이트를 이해하며, 캐싱된 접두사는 그대로 유지됩니다.

### 4. 대화 중간에 모델을 전환하지 마세요

이것은 사람들을 당황하게 만듭니다.

Opus로 100,000 토큰 깊이의 대화를 진행 중입니다. 간단한 질문이 나왔고, Haiku가 훨씬 저렴한 비용으로 쉽게 처리할 수 있는 종류의 질문입니다. 전환하는 것이 현명한 움직임처럼 느껴집니다.

하지만,

프롬프트 캐시는 각 모델마다 고유합니다. Haiku는 귀하의 대화에 대한 캐시가 없습니다. 처음부터 100,000 토큰의 컨텍스트를 재구축하기 위해 정가를 지불하게 되며, 이는 더 저렴한 모델로 전환하는 것이 더 많은 비용을 들게 한다는 의미입니다.

하위 작업에 다른 모델이 필요한 경우, 하위 에이전트(subagent)를 사용하세요.

Opus는 관련 컨텍스트만 포함된 집중된 핸드오프(handoff) 메시지를 준비하여 Haiku에 전달하고, 메인 세션 캐시는 그대로 유지됩니다.

### 5. 압축(Compaction): 아무도 이야기하지 않는 예외 사례

모든 장기 실행 에이전트는 결국 컨텍스트 창(context window) 제한에 도달합니다. 이는 대화를 요약하고 새로운 세션에서 계속 진행해야 합니다. 이를 압축(compaction)이라고 합니다.

순진한 구현 방식은 캐시를 손상시킵니다.

대부분의 개발자는 압축을 다른 시스템 프롬프트와 도구 없이 별도의 API 호출로 실행합니다. API 관점에서 보면, 이는 부모 대화와 전혀 다르게 보입니다.

**Claude Code 팀의 해결책은 캐시 안전 포킹(cache-safe forking)입니다.**

*   부모 대화와 정확히 동일한 시스템 프롬프트, 도구, 컨텍스트를 사용하세요.
*   전체 대화 기록을 앞에 추가하세요.
*   압축 프롬프트를 최종 사용자 메시지로 추가하세요.

API는 부모 세션의 마지막 턴과 거의 동일하게 보이는 요청을 인식합니다. 캐시가 적중되고, 정가로 청구되는 유일한 토큰은 압축 프롬프트 자체입니다.

## 마지막 생각

모든 사용 사례가 프롬프트 캐싱으로부터 동일하게 이점을 얻는 것은 아니며, 전체 프롬프트 아키텍처(architecture)를 재구성하기 전에 이 점에 대해 솔직해질 필요가 있습니다.

두 가지 조건이 충족될 때 유용합니다. 정적 컨텍스트(static context)가 크고, 세션이 길게 실행될 때입니다. 턴이 많을수록 캐싱된 접두사가 그 가치를 더합니다. 다음 시나리오에서 가장 큰 차이를 만듭니다.

*   **코딩 에이전트** — 대규모 시스템 프롬프트, 많은 도구 정의, 긴 다중 턴(multi-turn) 세션.
*   **문서 분석 파이프라인(pipeline)** — 많은 문서나 사용자에게 동일한 지침과 컨텍스트를 반복적으로 사용하는 에이전트.
*   **고객 지원 봇** — 회사 지식, 제품 세부 정보, 대화 간에 변경되지 않는 정책 지침을 포함하는 대규모 시스템 프롬프트.
*   **연구 보조 에이전트** — 에이전트가 여러 턴에 걸쳐 정보를 읽고, 통합하고, 반복하는 긴 세션.

정적 컨텍스트가 1,000 토큰 미만이고 세션이 5턴을 거의 초과하지 않는다면, 자동 캐싱은 큰 효과를 보지 못할 것입니다.

프롬프트 캐싱은 Claude API에서 항상 사용 가능했지만, 수동 관리는 개발자에게 복잡성을 가중시켰습니다. 자동 캐싱은 Claude API를 기반으로 구축하는 모든 사람이 이 최적화를 쉽게 이용할 수 있도록 합니다.

몇 턴 이상 실행되는 것을 구축하고 있다면, 이것을 무시할 여유가 없을 것입니다!
