Best LLMs for OpenCode — Tested Locally

최근 AI 개발 도구 생태계에서 가장 유망한 도구 중 하나인 OpenCode를 Ollama 기반의 로컬 LLM(Large Language Models) 및 OpenCode Zen의 무료 모델들과 비교 테스트했습니다.

### 요약: OpenCode를 위한 최적의 LLM
*   **로컬 실행 부문 우승자: llama.cpp 기반 Qwen 3.5 27b (IQ3_XXS 양자화)**
    *   IQ3_XXS 양자화(Quantization) 버전임에도 8개의 유닛 테스트를 모두 통과한 완벽한 Go 프로젝트를 생성했습니다. 16GB VRAM 환경(CPU+GPU 혼합)에서 초당 34개 토큰(tokens/sec)의 속도를 기록하며, 별다른 제약 없이 로컬 작업의 주력 모델로 사용하기 적합합니다.

*   **Qwen 3.5 35b (llama.cpp): 코딩은 빠르나 검증 필수**
    *   에이전트 기반 코딩에는 뛰어나지만, 구조화된 규칙 준수 테스트(마이그레이션 맵 생성)에서 신뢰도 문제가 발견되었습니다. 특정 양자화 버전에서는 페이지 슬러그(slug)를 누락하거나 잘못된 경로를 생성하는 오류를 보였습니다. 코딩 품질 자체는 훌륭하나, 출력물에 대한 별도의 검증 절차가 반드시 필요합니다.

*   **의외의 복병: Bigpicle (OpenCode Zen 제공)**
    *   테스트 중 가장 빠른 완료 시간(1분 17초)을 기록했습니다. 특히 코딩 시작 전 검색 도구(Exa Code Search)를 사용해 IndexNow 프로토콜 명세를 직접 확인하는 능력이 돋보였습니다.

*   **조건부 우수: GPT-OSS 20b**
    *   기본 모드에서는 실패했으나, 'High Thinking' 모드를 활성화하면 매우 유능한 코딩 어시스턴트로 변모합니다. 플래그 파싱(flag parsing)과 배치 로직을 정확히 구현하며 유닛 테스트를 빠르게 통과했습니다.

*   **추천하지 않는 모델 (에이전트 코딩 기준)**
    *   **Qwen 3 14b, devstral-small-2:24b, GPT-OSS 20b(기본 모드):** 단순 채팅이나 생성 작업에는 좋지만, 에이전트 모드에서는 환각(Hallucination) 증상을 보이거나 기본적인 파일 조작에서 실수를 범하는 등 한계를 보였습니다.

### 테스트 방법
각 모델을 OpenCode 환경에서 실행하며 다음 두 가지 과제를 부여했습니다.
1.  **IndexNow CLI 도구 제작:** Bing 등 검색 엔진에 웹사이트 변경 사항을 알리는 Go 언어 기반 CLI 도구 생성.
2.  **웹사이트 마이그레이션 맵 작성:** 특정 URL 형식을 새로운 구조로 변경하되, 슬러그(slug, 경로의 마지막 부분)는 그대로 유지하는 전략 수립.

### 모델별 상세 결과

#### Qwen 3.5 9b & 9b-q8_0
*   기본 버전은 도구를 전혀 호출하지 못하고 실패했습니다. q8_0 양자화 버전은 파일을 생성하긴 했으나, 의존성 문제로 실행되지 않는 결과물을 내놓았습니다.

#### Qwen 3 14b
*   문서를 찾지 못했음에도 존재하지 않는 API 엔드포인트와 인증 방식을 자신 있게 제시하는 심각한 환각 증상을 보였습니다.

#### GPT-OSS 20b
*   **기본 모드:** 정보 수집에 실패했지만, 환각을 일으키지 않고 실패 과정을 투명하게 기록했습니다.
*   **High Thinking 모드:** 완전히 다른 결과를 보여주었습니다. 정확한 IndexNow 사양에 맞춘 도구를 제작했으며, 유닛 테스트까지 완벽히 통과했습니다. 초기 구축 시간도 22.5초로 매우 빨랐습니다.

#### Qwen3-coder 30b
*   실제 엔드포인트에서 발생하는 에러를 보고 코드를 수정하는 좋은 직관을 보였습니다. 하지만 하드웨어 자원 활용이 비효율적이라 실행 속도가 매우 느렸습니다.

#### Qwen 3.5 35b (Ollama)
*   결과물은 훌륭했으나 구축 시간이 19분 이상 소요되었습니다. 27GB 모델을 CPU/GPU 혼합 방식으로 구동할 때 발생하는 지연 시간(Latency)이 실시간 작업 흐름을 방해했습니다.

#### Bigpicle (Big-pickle)
*   코딩 전 프로토콜을 직접 검색하여 정확한 엔드포인트 목록을 확보하는 등 가장 인상적인 성능을 보였습니다. 도구 완성까지 단 1분 17초가 걸렸습니다.

#### Qwen 3.5 27b (llama.cpp, IQ3_XXS)
*   로컬 실행 모델 중 가장 놀라운 성능을 보였습니다. 8개 테스트를 모두 통과한 도구와 설치 가이드가 포함된 README를 생성했습니다. 초당 34토큰의 속도와 높은 결과물 품질을 동시에 잡은 모델입니다.

### 마이그레이션 맵 결과 (규칙 준수 테스트)
"기존 슬러그를 반드시 유지하라"는 엄격한 조건 하에 테스트를 진행했습니다.

*   **우수:** Minimax-m2.5, Nemotron 3, Qwen 3.5 27b Q3_XXS 모델은 규칙을 거의 완벽하게 준수했습니다.
*   **미흡:** Qwen 3.5 35b 계열은 페이지 제목을 바탕으로 슬러그를 새로 쓰거나, 슬러그 자체를 누락하여 여러 페이지를 동일한 URL로 연결하는 치명적인 오류를 범했습니다. 특히 IQ4_XS 양자화 버전은 결과물을 아예 사용할 수 없는 수준이었습니다.

### 결론 및 제언
16GB VRAM GPU 환경에서 llama.cpp를 이용해 Qwen 3.5 모델들을 양자화하여 사용하는 워크플로우는 매우 실용적입니다.

1.  **주력 모델 추천:** **Qwen 3.5 27b (IQ3_XXS)**. 지시 사항을 정확히 따르며 빠르고 신뢰할 수 있습니다. 클라우드 모델과 대등한 성능을 로컬에서 보여줍니다.
2.  **주의 사항:** **35b 모델**은 코딩 능력 자체는 강력하지만, 구조화된 규칙이 필요한 작업에서는 예기치 못한 환각(Hallucination)을 보일 수 있습니다. 출력 결과에 대한 자동화된 검증 단계를 반드시 포함해야 합니다.
