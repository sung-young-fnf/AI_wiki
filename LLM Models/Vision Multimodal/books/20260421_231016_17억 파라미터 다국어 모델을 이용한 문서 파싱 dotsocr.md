# 17억 파라미터 다국어 모델을 이용한 문서 파싱: dots.ocr

dots.ocr은 텍스트 읽기, 테이블 추출, 수학 공식 파싱 등 모든 작업을 레이아웃을 유지하며 수행할 수 있는 혁신적인 오픈소스 문서 이해 모델입니다.

검출(detection) 및 인식(recognition)을 위해 별도의 파이프라인이나 여러 모델을 필요로 하는 기존의 OCR(광학 문자 인식) 솔루션과 달리, dots.ocr은 단일 17억 파라미터 시각-언어 모델(vision-language model)을 사용하여 모든 것을 엔드-투-엔드(end-to-end)로 처리합니다.

이는 레이아웃 분석기, OCR 엔진, 후처리기를 연결할 필요 없이 입력(이미지 또는 PDF)과 작업을 설명하는 프롬프트(prompt)만 제공하면 dots.ocr이 나머지를 처리한다는 의미입니다.

이 모델은 영어, 중국어는 물론 티베트어, 칸나다어와 같은 저자원 언어까지 특별한 구성 없이 100개 이상의 언어를 기본적으로 지원합니다.

상대적으로 컴팩트한 크기(17억 파라미터)에도 불구하고, dots.ocr은 벤치마크(예: OmniDocBench)에서 최첨단(state-of-the-art) 정확도를 달성하며, 종종 몇 배나 더 큰 OCR 모델보다 뛰어난 성능을 보입니다.

## 설정 및 설치

dots.ocr을 로컬에서 시작하는 것은 간단합니다. 이 프로젝트는 오픈소스(MIT 라이선스)이며 코드는 GitHub에서 확인할 수 있습니다. 모델을 컴퓨터에 설정하려면 다음 단계를 따르세요:

**(1) 환경 준비**: Python 3.10+ (프로젝트에서는 3.12 권장)와 PyTorch가 설치되어 있는지 확인하세요.

dots.ocr은 GPU(Graphics Processing Unit)와 함께 사용할 때 가장 잘 작동하지만(속도 향상), CPU(Central Processing Unit)만으로도 사용 가능합니다(추론(inference) 속도 저하). 파이썬 가상 환경(Python virtual environment) 또는 콘다 환경(Conda environment)을 사용하는 것이 좋습니다.

```bash
conda create -n dots_ocr python=3.12
conda activate dots_ocr
```

**(2) dots.ocr 설치**: 저장소(repository)를 복제(clone)하고 종속성(dependencies)을 설치하세요. 현재 이 저장소는 PyPI(Python Package Index)에 등록되어 있지 않으므로 소스에서 설치해야 합니다.

```bash
git clone https://github.com/rednote-hilab/dots.ocr.git 
cd dots.ocr 

# (선택 사항) 특정 CUDA 버전에 맞는 PyTorch 설치, 예:
pip install torch==2.7.0 torchvision==0.22.0 torchaudio==2.7.0 --index-url https://download.pytorch.org/whl/cu118 
pip install -e .
```

dots.ocr의 17억 파라미터 모델 가중치(model weights)는 별도로 다운로드해야 합니다(Git 저장소에 포함하기에는 너무 큽니다). 프로젝트는 저자의 모델 허브(model hub)에서 이를 가져오는 스크립트를 제공합니다.

```bash
python3 tools/download_model.py
```

격리된 설정(isolated setup)을 선호하거나 설치 문제에 직면한 경우, 개발자들은 도커허브(DockerHub)에서 즉시 사용 가능한 도커 이미지(Docker image)를 제공합니다. 구성된 환경을 위해 이미지를 풀(pull)하고 실행할 수 있습니다.

```bash
docker pull rednotehilab/dots.ocr   # 도커 이미지 다운로드
```

실제로, 모델의 크기와 대규모 컨텍스트(context) (매우 상세한 JSON 결과를 많은 토큰으로 출력할 수 있음)로 인해, 특히 대용량 문서의 경우 원활한 추론(inference)을 위해서는 상당한 메모리(예: 16GB 이상)를 갖춘 GPU가 권장됩니다. CPU에서 실행하는 것도 허깅 페이스(Hugging Face) 모드(아래에서 설명)를 사용하여 가능하며, 작은 이미지나 테스트에는 적합할 수 있지만, 더 긴 처리 시간을 예상해야 합니다.

## dots.ocr 실행: 로컬 vs 클라우드 배포

설치 후, dots.ocr로 추론을 실행하는 몇 가지 옵션이 있습니다.

**(1) 가장 빠른 성능을 위해**, 저자들은 최적화된 LLM(대규모 언어 모델) 추론 엔진인 vLLM 사용을 권장합니다. dots.ocr은 vLLM (버전 0.11.0 이상)에 공식적으로 통합되어 있어, 최적화된 GPU 활용률(utilization)로 모델을 제공하는 로컬 서버를 구동할 수 있습니다.

vLLM이 설치되어 있다면, 서버 시작은 한 줄 명령어로 간단합니다:

```bash
vllm serve rednote-hilab/dots.ocr --trust-remote-code --gpu-memory-utilization 0.95
```

vLLM의 장점은 LLM 제공에 매우 최적화되어 있어, 특히 배치 처리(batch processing)나 긴 출력 처리에 있어 dots.ocr의 추론을 훨씬 빠르고 메모리 효율적으로 만든다는 것입니다. dots.ocr을 백엔드 서비스에 통합하거나 높은 처리량(throughput)이 필요한 경우, vLLM이 최적의 선택입니다.

**(2) 별도의 서버를 실행하지 않으려면**, 허깅 페이스(Hugging Face)의 트랜스포머(Transformers) 라이브러리를 사용하여 코드에서 직접 모델을 로드할 수 있습니다. dots.ocr 모델 가중치는 트랜스포머와 호환됩니다 (사용자 정의 모델링 코드를 사용하므로 `trust_remote_code=True` 필요). 저장소는 이 모드를 위한 데모 스크립트도 제공합니다.

**(3) 적합한 로컬 환경이 없다면**, 클라우드에서 dots.ocr을 시도해 볼 수 있습니다. 저자들은 브라우저에서 사용해 볼 수 있는 라이브 데모 웹 앱(그라디오(Gradio) 인터페이스)을 호스팅하고 있으며, 이는 빠른 테스트나 비개발자에게 보여주기에 좋습니다. 또한 Replicate를 통해 커뮤니티에서 제공하는 REST API(종량제(pay-per-use) 가격 정책)도 있어, 모델을 직접 호스팅하지 않고도 빠른 통합에 유용합니다.

## dots.ocr 사용하기: 일반적인 사용 사례 및 예시

간단히 설명하기 위해, 아래 예시에서는 vLLM 서버가 아닌 트랜스포머(허깅 페이스) 방식을 코드에서 직접 사용합니다.

먼저, 필요한 구성 요소를 가져오고 파이썬 스크립트(Python script) 또는 대화형 환경(interactive environment)에서 모델을 로드합니다:

```python
import torch
from dots_ocr.parser import DotsOCRParser

# 파서 초기화; use_hf=True를 사용하여 Hugging Face 모드 사용 (vLLM 서버 필요 없음)
parser = DotsOCRParser(use_hf=True)
```

`use_hf=True`일 때, 파서(parser)는 모델과 프로세서(processor)를 메모리에 로드합니다(17억 파라미터 가중치를 처음 로드할 때 시간이 다소 걸릴 수 있습니다). 기본적으로 `./weights/DotsOCR` (다운로드 스크립트가 생성한 디렉토리)에서 로드하려고 시도합니다. 모든 설정이 완료되면, HF 모델이 로드되었음을 나타내는 메시지가 표시될 것입니다.

이제 파서를 사용하여 문서를 처리할 수 있습니다.

### 문서에서 텍스트 추출하기 (기본 OCR)

일반적인 사용 사례 중 하나는 문서에서 모든 텍스트 콘텐츠를 추출하는 것입니다(레이아웃 위치 무시). dots.ocr은 이 모드에서 일반 OCR로 작동할 수 있습니다. 이를 위해 `prompt_ocr` 모드를 사용할 수 있습니다. 이 모드는 모델에게 레이아웃 주석(layout annotations)을 출력하지 않고, 모든 텍스트(페이지 헤더/푸터 제외)를 올바른 순서로 읽도록 지시합니다.

```python
result = parser.parse_file("sample_page.png", prompt_mode="prompt_ocr")
```

이 코드에서 "sample_page.png"는 문서 페이지의 이미지 스캔(JPEG/PNG)일 수 있습니다. 마찬가지로 PDF 파일을 지정할 수도 있으며, 파서는 내부적으로 PDF를 페이지로 분할하고 각 페이지를 처리합니다(PDF는 다음 섹션에서 논의). `parse_file` 메서드는 결과 딕셔너리(dictionary) 목록(페이지당 하나)을 반환합니다. 단일 이미지의 경우, 하나의 요소로 구성된 목록을 얻게 됩니다.

출력 텍스트는 디스크에 마크다운(Markdown) 파일로 저장되며, 결과 딕셔너리에 해당 경로가 포함됩니다.

```python
text_md_path = result[0]['md_content_path']
with open(text_md_path, 'r', encoding='utf-8') as f:
    extracted_text = f.read()
print(extracted_text[:500])  # 추출된 텍스트의 처음 500자 출력
```

이제 후처리 작업(예: 색인(indexing), 검색 등)에 사용할 수 있는 전체 텍스트 콘텐츠를 얻었습니다.

텍스트 추출 시 몇 가지 유의할 점:

*   **읽기 순서(Reading Order)**: dots.ocr은 다단 레이아웃(multi-column layout)에서도 텍스트의 사람이 읽는 순서를 보존하도록 설계되었습니다. 마크다운에서 추출된 텍스트는 이미 논리적인 순서로 정렬되어 있습니다(뒤섞이지 않음).
*   **정확도(Accuracy)**: 이 모델은 벤치마크에서 매우 높은 텍스트 인식 정확도(95% 이상)를 달성하며, 까다로운 글꼴이나 저자원 언어에서도 전문 OCR 엔진과 유사하거나 우수한 성능을 보이는 경우가 많습니다.
*   **다국어 텍스트(Multilingual text)**: 이 모드는 이미지에 있는 텍스트를 원본 언어와 스크립트로 출력합니다. 번역은 수행되지 않습니다. 예를 들어, 문서가 프랑스어인 경우 추출된 텍스트는 프랑스어입니다. dots.ocr은 비라틴 문자(non-Latin scripts)(예: 중국어, 아랍어, 데바나가리 등)도 쉽게 처리할 수 있습니다.

### 테이블 및 양식 파싱

dots.ocr의 또 다른 강력한 기능은 테이블과 같은 구조화된 콘텐츠를 추출하는 것입니다. 기존 OCR은 일반 텍스트를 제공할 수 있지만, dots.ocr은 테이블을 실제 테이블(구조 포함)로 캡처하고 유용한 형식(예: HTML)으로 출력할 수 있습니다.

예를 들어, 테이블이 있는 문서 페이지(예: 재무 보고서(financial report) 또는 양식(form))가 있다고 가정해 봅시다. 기본 레이아웃 파싱 프롬프트(layout parsing prompt)를 사용하여 구조화된 결과를 얻을 수 있습니다.

```python
result = parser.parse_file("report_page.png", prompt_mode="prompt_layout_all_en")
```

여기서 `prompt_layout_all_en`은 모델에게 모든 레이아웃 요소(layout elements) (텍스트 블록, 테이블, 공식 등)를 감지하고 그 내용을 인식하도록 요청하는 기본 프롬프트입니다. 이 모드는 각 요소의 경계 상자(bounding box, bbox), 카테고리(category) 및 콘텐츠가 포함된 JSON을 반환합니다. 또한 인식된 모든 콘텐츠를 읽기 순서로 연결한 마크다운 파일을 저장합니다(테이블 및 기타 요소 포함).

결정적으로, 출력은 특정 서식 규칙을 따릅니다: 테이블은 HTML로, 공식은 LaTeX(라텍스)로, 일반 텍스트는 마크다운으로 프롬프트에 따라 서식이 지정됩니다.

예를 들어, JSON 출력에서 다음과 같은 항목을 볼 수 있습니다:

```json
{
  "category": "Table",
  "bbox": [100, 200, 500, 300],
  "text": "<table>\n  <tr><th>Year</th><th>Revenue</th></tr>\n  <tr><td>2023</td><td>$5M</td></tr>\n  <tr><td>2024</td><td>$6M</td></tr>\n</table>"
}
```

이 HTML을 렌더링하거나 추가로 파싱하여 셀 값을 얻을 수 있습니다. dots.ocr은 본질적으로 레이아웃 분석(layout analysis)과 OCR을 동시에 수행하여 별도의 테이블 추출 도구 없이 구조화된 데이터를 제공합니다.

테이블 외에도 동일한 출력에는 텍스트 단락("category": "Text"와 마크다운 텍스트), 제목, 섹션 헤더, 공식(LaTeX 문자열 포함) 등과 같은 다른 요소가 포함될 수 있습니다. 각 요소의 카테고리는 필요한 경우 후처리하는 데 도움이 됩니다(예: 모든 테이블이나 모든 그림을 쉽게 분리할 수 있음).

마크다운 출력 파일(이 예시에서는 `report_page.md`)에는 읽기 쉬운 형식으로 모든 콘텐츠가 포함됩니다. 예를 들어, 테이블은 마크다운에 HTML 블록으로 나타날 수 있고, 단락은 일반 텍스트 섹션으로 나타날 수 있습니다. 이 마크다운은 문서를 다른 형식으로 변환하려는 경우(예: PDF를 마크다운으로 변환 등) 구조를 선형화된(linearized) 방식으로 보존하므로 유용합니다.

### 다국어 문서 파싱

dots.ocr은 광범위한 언어로 훈련되어 다국어 문서에 매우 강력합니다. 비영어 텍스트를 파싱하기 위해 특별한 스위치가 필요하지 않으며, 동일한 메서드를 사용하면 모델이 언어를 인식하고 원본 스크립트로 텍스트를 출력합니다.

예를 들어, 일본어 신문 스캔 페이지나 혼합 언어 문서(예: 중국어 주석이 있는 영어 보고서)가 있다고 가정해 봅시다:

```python
result = parser.parse_file("invoice_cn.pdf", prompt_mode="prompt_layout_all_en")
```

프롬프트가 영어("prompt_layout_all_en" 템플릿이 모델에 영어로 지시)임에도 불구하고, 모델은 100개 이상의 언어로 콘텐츠를 올바르게 추출합니다. 출력 JSON/마크다운에는 해당 언어의 텍스트가 포함됩니다. 따라서 일본어 텍스트는 일본어 문자로, 중국어는 중국어 문자로 이미지에 있는 그대로 유지되며, 음역(transliteration)이나 번역(translation)은 수행되지 않습니다.

이는 글로벌 애플리케이션에 특히 유용합니다. 아랍어, 러시아어, 힌디어로 된 문서를 입력하여 추가 학습이나 언어 구성 없이 구조화된 결과를 얻을 수 있습니다. 내부 테스트에서 dots.ocr은 저자원 언어에서도 강력한 성능을 보였으며, 이전 OCR 모델에 비해 오류율을 크게 줄였습니다.

### PDF 및 다중 페이지 문서 작업

지금까지는 단일 페이지 예시를 살펴보았습니다. 하지만 여러 페이지로 구성된 PDF는 어떨까요? dots.ocr은 이미지와 마찬가지로 PDF도 쉽게 파싱할 수 있습니다. 내부적으로는 PyMuPDF 라이브러리(fitz)를 사용하여 PDF 페이지를 래스터화(rasterize)하여 처리합니다.

`parse_file`에 PDF 파일 경로를 전달하면 페이지 반복(page iteration)을 자동으로 처리합니다:

```python
results = parser.parse_file("multilingual_report.pdf", prompt_mode="prompt_layout_all_en")
```

PDF에 예를 들어 10페이지가 있다면, 결과는 10개의 딕셔너리(페이지당 하나) 목록이 됩니다. 스크립트는 또한 출력 디렉토리(output directory)에 .jsonl 파일을 출력하며, 각 줄은 페이지에 대한 JSON 결과입니다. 각 페이지의 콘텐츠는 별도의 하위 폴더(subfolder)에 저장됩니다(예: `output/multilingual_report/page_0/...`에 해당 페이지의 JSON, MD, 이미지 등). 콘솔 로그(console log)에는 페이지가 병렬로 처리되는 동안 진행률 표시줄(progress bar)이 표시됩니다.

dots.ocr의 파서는 PDF 처리 속도를 높이기 위해 멀티스레딩(multithreading)을 지원하며, 기본적으로 최대 16개의 스레드를 사용할 수 있지만, 특히 페이지가 많은 PDF의 경우 CPU 코어가 더 많으면 이를 늘릴 수 있습니다. 예를 들어, `parser.parse_file("bigdoc.pdf", prompt_mode="prompt_layout_all_en")`는 `parser.num_thread`에 지정된 스레드 수를 자동으로 사용합니다(이전 초기화에서 기본값은 64였지만, 재정의할 수 있습니다). 명령줄 인터페이스(CLI) 사용 시, 대용량 PDF의 경우 `--num_threads 64`를 전달할 수 있습니다. 각 페이지의 OCR이 독립적이므로 이는 페이지 처리를 병렬화하는 데 유용합니다.

#### PDF를 위한 출력

각 페이지의 출력 구조는 단일 이미지의 경우와 동일하며, JSON 레이아웃, 마크다운 텍스트, 그리고 레이아웃이 그려진 이미지로 구성됩니다. 각 페이지의 마크다운을 취합(aggregate)함으로써 사실상 전체 PDF의 콘텐츠를 얻을 수 있습니다.

매우 큰 PDF(수백 페이지)는 처리 시간이 더 오래 걸리고 더 많은 메모리를 소비한다는 점을 염두에 두세요. 저자들은 모델이 아직 매우 높은 처리량(throughput)의 PDF 처리에 최적화되어 있지 않으므로, 한 번에 수천 페이지를 OCR로 처리할 계획이라면 배치(batch)로 처리하거나 인내심을 가져야 할 수도 있다고 언급합니다. 그렇지만 일반적인 문서(수십 페이지)의 경우 잘 작동합니다.

#### DPI 및 이미지 스케일링

일부 PDF 텍스트가 인식되지 않는 경우, 렌더링된 이미지의 해상도 때문일 수 있습니다. 기본적으로 dots.ocr은 PDF 페이지 이미지에 200 DPI(인치당 도트 수)를 사용하며, 이는 일반적으로 충분합니다. 명령줄 인터페이스(CLI)에서 `--dpi` 매개변수를 전달하거나 `parser = DotsOCRParser(dpi=300, use_hf=True)`로 설정하여 더 높은 해상도를 조정할 수 있지만, 이는 더 많은 픽셀을 처리해야 하는 대가입니다. 또한, 극도로 큰 이미지(초고해상도 스캔)는 다운스케일링되거나 메모리 문제를 일으킬 수 있습니다. 모델은 약 11 메가픽셀(megapixels) 미만의 이미지에서 가장 잘 작동합니다. 필요한 경우, 지나치게 큰 이미지의 크기를 조정하거나 파서에 내장된 `min_pixels/max_pixels` 옵션을 사용하여 이미지 크기를 제한하세요.

### 출력 구조 이해하기

dots.ocr로 문서를 파싱한 후, 일반적으로 애플리케이션에서 사용하기 위해 출력을 검토합니다.

*   **JSON 레이아웃 데이터**: 레이아웃 파싱 프롬프트의 경우, 모델은 페이지에서 감지된 모든 요소가 포함된 구조화된 JSON(structured JSON)을 출력합니다(.json 파일로 저장됨). 각 요소에는 좌표(bbox), 카테고리 레이블(예: "Text", "Table", "Title" 등) 및 해당 요소의 인식된 텍스트 콘텐츠가 포함됩니다. 이 JSON은 가장 기계 친화적인 출력이며, 이를 로드하여 문서의 구조를 프로그래밍 방식으로 탐색할 수 있습니다.
*   **마크다운(Markdown) 텍스트 파일**: dots.ocr은 또한 페이지당(또는 이미지당) 마크다운(.md) 파일을 생성하며, 이는 레이아웃 요소의 모든 텍스트 콘텐츠를 읽기 순서로 연결합니다. 마크다운은 기본적인 구조를 보존합니다. 예를 들어, 페이지에 제목, 테이블 및 일부 단락이 있다면, MD 파일은 제목(아마도 제목으로), 그 다음 테이블(HTML 테이블 블록 또는 마크다운 테이블로), 그 다음 텍스트 단락을 나열합니다.
*   **주석이 달린 이미지**: 육안 검토(visual inspection)를 위해, 이 도구는 원본 페이지/이미지에 감지된 레이아웃 경계 상자(bounding box)가 그려진 이미지를 저장합니다. 이는 디버깅(debugging)에 매우 유용하며, 이미지를 열어 모델이 모든 텍스트 영역, 테이블 등을 올바르게 윤곽을 표시했는지 확인할 수 있습니다. 각 영역에는 일반적으로 해당 카테고리가 주석 처리(annotated)되어 있습니다.

핵심은 파싱의 핵심 작업이 모두 모델에 의해 수행되며, 필요에 따라 추가로 처리할 수 있는 풍부한 출력을 얻는다는 것입니다.

## 결론

효율성과 성능을 고려할 때, dots.ocr은 최고 수준의 OCR 결과를 얻기 위해 거대한 모델 앙상블(ensemble of models)이 필요하지 않음을 증명합니다. 정교하게 제작된 17억 파라미터 모델 하나만으로도 우아하게 작업을 수행할 수 있습니다.
