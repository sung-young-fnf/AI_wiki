# 4가지 Gemma 4 모델 테스트: 26B 모델의 '최고의 편법'

구글이 이번 주 아파치 2.0(Apache 2.0) 라이선스 하에 Gemma 4를 공개했습니다. 이는 무료 사용 및 상업적 활용이 가능하며, 아무런 제약이 없습니다. E2B, E4B, 26B MoE, 31B Dense의 네 가지 모델 크기가 동시에 출시되었습니다. 저는 출시 당일 이 네 모델을 로컬 머신에 설치하고 동일한 작업을 실행했으며, 예상치 못한 결과를 발견했습니다.

26B 모델이 4B 모델보다 더 빨랐습니다. 훨씬 더 빨랐죠. 그리고 제가 실행한 모든 벤치마크에서 31B 모델과 2% 이내의 성능 차이를 보였습니다.

제가 정확히 무엇을 발견했으며, 어떤 모델을 실제로 사용해야 하는지 알려드리겠습니다.

## 구글이 실제로 공개한 것

결과를 살펴보기 전에, 이름이 혼란스러울 수 있으므로 잠시 설명이 필요합니다.

*   **E2B** — '실질적 2B(Effective 2B)'. 실제 파라미터(parameter)는 23억 개입니다. 스마트폰이나 CPU 전용 노트북에서 실행되도록 설계되었습니다. 구글은 이 모델을 온디바이스(on-device) AI용으로 홍보하고 있습니다.
*   **E4B** — '실질적 4B(Effective 4B)'. 45억 개의 파라미터를 가집니다. 최소 8GB의 RAM이 필요합니다. 중간급 머신에서 매일 사용하기 좋은 강력한 모델입니다.
*   **26B A4B (MoE)** — 이것이 흥미로운 모델입니다. 총 252억 개의 파라미터를 가지고 있지만, 단일 추론(inference) 과정에서 38억 개만 활성화됩니다. 이는 전문가 혼합(Mixture of Experts, MoE) 아키텍처로, 모델이 전체 네트워크를 매번 실행하는 대신 각 토큰을 전문가 레이어(layer)의 하위 집합으로 라우팅합니다. 그 결과, 약 4B 모델급의 연산 비용(compute cost)으로 26B 모델급 출력을 제공합니다.
*   **31B Dense** — 플래그십 모델입니다. 모든 307억 개 파라미터가 모든 토큰에 대해 작동합니다. 최고 품질을 자랑하지만, 가장 높은 비용이 듭니다. 양자화(quantization)하지 않으면 80GB H100이 필요하지만, Q4 양자화 시 24GB GPU에서도 실행 가능합니다.

네 모델 모두 멀티모달(multimodal, 텍스트 + 이미지)이며, 아파치 2.0 라이선스를 따르고, 출시 당일부터 HuggingFace와 Ollama에서 사용할 수 있습니다.

## 제 테스트 환경

저는 M3 Max MacBook Pro(128GB 통합 메모리)와 대형 모델을 위한 별도의 RTX 4090 워크스테이션(24GB VRAM)에서 Ollama를 통해 네 모델을 모두 다운로드했습니다.

```bash

ollama pull gemma4:e2b
ollama pull gemma4:e4b
ollama pull gemma4:26b
ollama pull gemma4:31b

ollama list
# 이름              ID            크기    수정일시
# gemma4:e2b        a1b2c3d4e5f   7.2 GB  2분 전
# gemma4:e4b        b2c3d4e5f6a   9.6 GB  4분 전
# gemma4:26b        c3d4e5f6a7b   18 GB   8분 전
# gemma4:31b        d4e5f6a7b8c   20 GB   12분 전
```

저는 각 모델을 세 가지 작업에 대해 테스트했습니다: 다단계 추론 문제, 코드 생성 작업, 그리고 장문(long-context) 요약. 각 작업에서 첫 토큰 생성 시간(Time-to-first-token), 총 생성 시간, 그리고 출력 품질(1-10점 척도로 수동 평가)을 측정했습니다.

## 결과: E2B — 규모 대비 인상적이지만, 그 규모에 제약받다

E2B는 23억 개의 파라미터라는 점을 감안할 때 정말 인상적입니다. 간단한 Q&A, 기본적인 요약, 간단한 파이썬 코드를 잘 처리합니다. 200ms 이내에 생성을 시작하여 반응이 빠릿하게 느껴집니다.

```python
import ollama
response = ollama.chat(
    model='gemma4:e2b',
    messages=[{
        'role': 'user',
        'content': '에라토스테네스의 체를 사용하여 n까지의 모든 소수를 찾는 파이썬 함수를 작성하세요.'
    }]
)
print(response['message']['content'])
# 출력: 약 1.2초 만에 깔끔하고 올바른 구현
# 첫 토큰 생성 시간: 180ms
# 총 생성 시간: 약 120개 토큰에 1.2초
```

하지만 다단계 추론에서는 한계를 보였습니다. 동시에 3~4개 이상의 논리적 종속성을 유지해야 하는 문제를 주면 중간 단계에서 환각(hallucination) 현상을 보이기 시작했습니다. 제 추론 벤치마크에서 6.1/10점을 기록했는데, 이는 존중할 만하지만 신뢰할 수는 없는 수준입니다.

**E2B에 대한 평가**: 온디바이스(on-device) 애플리케이션, 엣지 추론(edge inference) 또는 200ms 미만의 첫 토큰 지연 시간이 필요할 때 사용하세요. 깊은 추론이 필요한 작업에는 신뢰하지 마십시오.

## 결과: E4B — 일상적인 작업용 모델

E4B는 대부분의 개발자에게 흥미로운 지점입니다. 45억 개의 파라미터로 8GB RAM 이상의 모든 머신에서 편안하게 실행되며, E2B보다 품질 향상이 의미 있습니다.

추론 점수: 7.4/10. 모든 다단계 문제를 올바르게 처리했습니다. 코드 품질도 훌륭했습니다. 프롬프트 없이도 적절한 오류 처리와 타입 힌트(type hint)가 포함된 FastAPI 엔드포인트(endpoint)를 생성했습니다.

```python
import ollama
# 스트리밍 예시 — 대화형 앱에 적합
stream = ollama.chat(
    model='gemma4:e4b',
    messages=[{
        'role': 'user',
        'content': 'URL 목록을 받아 asyncio로 동시에 가져와서 상태 코드를 반환하는 FastAPI 엔드포인트를 구축하세요.'
    }],
    stream=True
)
for chunk in stream:
    print(chunk['message']['content'], end='', flush=True)
# 약 280개 토큰을 3.1초 만에 생성
# aiohttp를 사용한 올바른 비동기 구현, 적절한 오류 처리
# 첫 토큰 생성 시간: 220ms
```

한 가지 놀라웠던 점은 장문(long-context) 요약(16K 토큰)에서 E4B가 문서 중간을 넘어서면서 일관성을 잃기 시작했다는 것입니다. 처음 절반은 정확히 요약했지만, 나머지 절반에 대해서는 그럴듯한 세부 정보를 지어냈습니다. 신뢰할 수 있는 출력을 위해서는 E4B를 8K 토큰 미만으로 유지하는 것이 좋습니다.

**E4B에 대한 평가**: 표준 노트북에서 로컬 개발 작업에 가장 좋은 옵션입니다. 견고한 추론 능력, 빠른 속도, 토큰당 0달러의 비용을 제공합니다.

## 결과: 26B MoE — 제 예상을 깬 모델

이 모델은 제가 테스트를 두 번 다시 실행하게 만들었는데, 제 타이밍이 틀렸다고 생각했기 때문입니다.

26B MoE는 제 RTX 4090에서 E4B보다 더 빠르게 토큰을 생성했습니다. Q4 양자화 상태에서 18GB VRAM에 들어가며 초당 42토큰(tokens/second)을 생성합니다. 동일 하드웨어에서 E4B는 초당 38토큰을 생성했습니다. 오타가 아닙니다.

이것이 바로 전문가 혼합(MoE) 아키텍처가 의도한 대로 작동하는 방식입니다. 토큰당 38억 개의 파라미터만 활성화되므로 연산 비용은 26B 모델보다는 4B 모델에 가깝습니다. 하지만 라우팅 메커니즘(routing mechanism) 덕분에 출력은 전체 26B 파라미터 공간을 활용하며, 컨텍스트에 따라 다른 슬라이스를 사용합니다.

E4B보다 품질 격차가 상당합니다. 추론 점수: 9.1/10. 모든 다단계 문제를 올바르게 처리했고, 더 많은 엣지 케이스(edge case)를 포괄하는 더 나은 코드를 작성했으며, 128K 컨텍스트 윈도우(context window) 전체에서 성능 저하 없이 일관성을 유지했습니다.

```python
import requests
import json
# Ollama REST API 직접 사용
def chat(model: str, prompt: str) -> dict:
    response = requests.post(
        'http://localhost:11434/api/chat',
        json={
            'model': model,
            'messages': [{'role': 'user', 'content': prompt}],
            'stream': False
        }
    )
    return response.json()
# 추론 작업에서 26B MoE 테스트
result = chat(
    model='gemma4:26b',
    prompt="""기차 한 대가 오전 9시에 시티 A를 출발하여 시티 B를 향해 시속 60마일로 달립니다.
    또 다른 기차 한 대가 오전 10시 30분에 시티 B를 출발하여 시티 A를 향해 시속 80마일로 달립니다.
    두 도시는 285마일 떨어져 있습니다. 몇 시에 두 기차가 만날까요?
    단계별 추론 과정을 보여주세요."""
)
print(result['message']['content'])
# 정답: 오후 12시 30분
# 추론: 명확하고 모든 단계를 보여주며, 1.5시간의 선행 시간을 고려함
# 첫 토큰 생성 시간: 195ms
# 총 생성 시간: 약 200개 토큰에 2.8초
```

저에게 결정적인 벤치마크는 AIME 2026(고급 수학 추론)에서 26B MoE가 88.3%를 기록한 반면 31B Dense는 89.2%를 기록했다는 점입니다. 실행 비용이 7배 더 저렴한 모델과의 차이가 불과 0.9%입니다.

**26B MoE에 대한 평가**: 이 모델은 대부분의 개발자가 사용해야 하는 모델입니다. 24GB GPU에 들어가고, 예상보다 빠르며, 플래그십 모델과 1% 이내의 벤치마크 성능을 보여줍니다. 바로 '스위트 스팟'입니다.

## 결과: 31B Dense — 최고이지만, 생각보다 격차가 작다

31B Dense는 이 모델군에서 의심할 여지 없이 최고의 모델입니다. 추론 점수: 9.4/10. LiveCodeBench v6에서는 26B MoE의 77.1% 대비 80.0%를 기록했습니다. 코드가 더 깔끔하고, 엣지 케이스가 더 잘 처리되며, 128K 토큰에 걸쳐 장문 일관성이 완벽합니다.

또한, 훨씬 더 큰 하드웨어를 요구하는 몇몇 모델을 제외하고 전 세계 모든 오픈 모델 중에서 Arena AI 리더보드 3위에 랭크되었습니다.

```python
import ollama
# 31B Dense — 품질이 타협 불가능한 작업용
response = ollama.chat(
    model='gemma4:31b',
    messages=[
        {
            'role': 'system',
            'content': '코드 검토를 하는 선임 엔지니어입니다. 문제점에 대해 구체적으로 설명하세요.'
        },
        {
            'role': 'user',
            'content': '''이 파이썬 코드에서 버그와 보안 취약점을 검토하세요:
def get_user(user_id):
    conn = sqlite3.connect("users.db")
    query = f"SELECT * FROM users WHERE id = {user_id}"
    result = conn.execute(query).fetchone()
    return result'''
        }
    ]
)
print(response['message']['content'])
# 올바르게 식별: SQL 인젝션(injection) 취약점, 연결 정리 없음,
# 오류 처리 없음, 반환 타입 어노테이션(annotation) 누락
# 또한 예시 수정과 함께 매개변수화된 쿼리(parameterized queries) 제안
# 첫 토큰 생성 시간: 310ms
# 총 생성 시간: 약 300개 토큰에 4.1초
```

문제는, 양자화하지 않으면 80GB H100이 필요하거나, Q4 양자화 시 24GB GPU가 필요하다는 점입니다(이 경우 품질이 약간 저하됩니다). 대규모 추론을 실행하는 팀의 경우, 26B MoE 대비 비용 차이가 빠르게 누적됩니다.

## 직접 비교

GPU에서 26B MoE가 E4B보다 초당 토큰(tokens/second) 속도가 더 빠르다는 점은 표에서 가장 특이한 숫자이며, 이는 실제입니다.

## 어떤 모델을 실제로 사용해야 할까?

*   **E2B**는 휴대폰, Raspberry Pi, 또는 적절한 GPU가 없는 환경에서 실행되어야 하는 것을 구축할 때 사용하세요. 이 모델군에서 CPU 전용으로 잘 실행되는 유일한 모델입니다.
*   **E4B**는 8~16GB RAM이 장착된 노트북을 사용하며 일상적인 코딩 및 글쓰기 지원을 위한 유능한 로컬 모델을 원할 때 사용하세요. 프롬프트는 8K 토큰 미만으로 유지하십시오.
*   **26B MoE**는 24GB GPU를 가지고 있다면 사용하세요. 이것은 80%의 개발자에게 정답입니다. 플래그십에 가까운 품질, 더 작은 모델보다 GPU에서 더 빠르고, 소비자용 하드웨어에 적합합니다. 저 역시 제 프로젝트에서 이 모델을 기본으로 사용할 것입니다.
*   **31B Dense**는 서비스형 추론(inference as a service)을 실행하고 최대 정확도가 필요하거나, H100에 접근할 수 있고 품질이 타협 불가능한 경우에 사용하세요.

## 아파치 2.0 라이선스가 모든 것을 바꾼다

솔직히 말해서, 네 가지 모델 모두 아파치 2.0 라이선스 하에 제공된다는 사실은 어떤 벤치마크 결과만큼이나 중요합니다. 이 모델들을 파인튜닝(fine-tune)하고, 상업용 제품에 배포하며, 그 위에 SaaS를 구축할 수 있으며, 토큰 기반 추론 비용을 전혀 지불할 필요가 없습니다.

현재 내부 도구를 구동하기 위해 최첨단(frontier) 모델에 1백만 입력 토큰당 2.50달러를 지불하는 팀의 경우, 26B MoE가 단일 4090에서 쿼리당 0달러로 실행된다는 점을 고려하면 비용 계산이 크게 달라집니다.
