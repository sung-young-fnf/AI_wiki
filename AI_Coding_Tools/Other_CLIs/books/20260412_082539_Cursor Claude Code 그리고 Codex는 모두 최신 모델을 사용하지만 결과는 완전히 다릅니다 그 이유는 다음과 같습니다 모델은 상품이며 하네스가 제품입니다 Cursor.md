Cursor, Claude Code, 그리고 Codex는 모두 최신 모델을 사용하지만 결과는 완전히 다릅니다. 그 이유는 다음과 같습니다: 모델은 상품이며, 하네스가 제품입니다: Cursor 클라우드 에이전트의 실제 아키텍처 심층 분석

여러분의 AI 코딩 어시스턴트가 여러분이 기차를 타고 있는 동안 PR을 제출했습니다. 이 어시스턴트는 기능이 작동함을 증명하기 위해 UI를 클릭하는 자체 비디오를 첨부했으며, 자체적으로 병합 충돌을 해결하고 단일 커밋으로 스쿼시(squash)했습니다. 이것은 단순한 데모가 아닙니다. Cursor 자체 병합된 풀 리퀘스트(pull requests, PR)의 35%가 현재 이런 방식으로 생성됩니다. 그러나 대부분의 엔지니어들이 이것을 볼 때 놓치는 부분은, 이 에이전트 내부의 모델(GPT-5, Claude, Gemini)이 스택에서 가장 중요하지 않은 부분이라는 것입니다. 이를 작동하게 만드는 것은 모델을 둘러싼 모든 것: VM 격리(VM isolation), 코드베이스 온보딩(codebase onboarding), 병렬 오케스트레이션(parallel orchestration), 비디오 아티팩트 캡처(video artifact capture), 다중 모델 라우팅(multi-model routing)입니다. Cursor는 이를 클라우드 에이전트(cloud agent)라고 부릅니다. 더 정확한 용어는 하네스(harness)입니다.

**요약 (TL;DR)**

*   Cursor 클라우드 에이전트(Cloud Agents)는 완전한 개발 환경을 갖춘 격리된 리눅스 VM에서 자율 코딩 작업을 실행합니다. 사용자당 최대 10~20개의 에이전트가 병렬로 실행됩니다.
*   핵심 기능은 에이전트가 생성한 소프트웨어를 직접 구축하고, 테스트하고, 사용한 다음 PR에 비디오 증명을 첨부한다는 것입니다. 이것은 자동 완성(autocomplete)이 아닙니다. 이것은 위임된 엔지니어링(delegated engineering)입니다.
*   Cursor 자체 병합된 PR의 35%는 에이전트로부터 생성됩니다. Cursor 3 "Glass"(2026년 4월 2일)는 에이전트 오케스트레이션(agent orchestration)을 중심으로 전체 IDE를 재구축했으며, 회사는 연간 반복 매출(ARR) 20억 달러를 달성했습니다.
*   모델은 상품화된 자원(commodity)입니다. Cursor는 GPT-5, Claude Opus, Gemini 및 자체 Composer 2를 라우팅(routing)합니다. 여러분이 지불하는 것은 모델을 둘러싼 하네스(VM 격리, 코드베이스 온보딩, 아티팩트 생성, 병렬 오케스트레이션)입니다.
*   Claude Code와 OpenAI Codex는 근본적으로 다른 하네스로 동일한 문제를 해결하며, 모델 계층은 교체 가능하지만 하네스 계층은 그렇지 않다는 것을 증명합니다.

**왜 중요한가**

"AI가 코드를 제안한다"에서 "AI가 테스트된 기능을 엔드 투 엔드(end to end)로 배포한다"로의 전환이 현재 빠르게 진행되고 있으며, 대부분의 엔지니어링 팀이 예상했던 것보다 빠르게 진행되고 있습니다. 2025년 10월, Cursor는 백그라운드 워커(background workers)로 클라우드 에이전트를 출시했습니다. 2026년 2월까지, 이 에이전트들은 브라우저를 열고, UI를 클릭하며, 작업에 대한 비디오 증명을 기록하는 등 자체 컴퓨터를 제어할 수 있게 되었습니다. 2026년 4월까지, Cursor 3는 채팅 패널을 완전히 제거하고, 에이전트 창(Agents Window)을 중심으로 IDE를 재구축하여, 개발자들이 프로젝트 관리자가 작업을 검토하듯이 자율 작업을 배포하고 모니터링할 수 있게 했습니다.

이 현상은 Cursor에만 국한되지 않습니다. 하네스 패턴(harness pattern)이 모든 곳에서 나타나고 있기 때문입니다. Claude Code는 작업 트리 격리(worktree isolation)를 통해 백그라운드 에이전트를 제공합니다. OpenAI Codex는 GitHub 이슈(issues)에서 샌드박스화된 작업(sandboxed tasks)을 실행합니다. GitHub Copilot은 자체 코딩 에이전트를 추가했습니다. 모든 도구는 "AI가 다른 작업을 하는 동안 작동한다"는 동일한 약속으로 귀결됩니다. 그러나 구현 방식은 근본적으로 다릅니다. Cursor의 하네스를 이해하면 이 모든 도구를 평가하는 방법을 알 수 있습니다. 왜냐하면 차별화 요소는 모델이 아니라 항상 모델을 둘러싼 인프라(infrastructure)이기 때문입니다.

**엔진 대 자동차**

여러분은 엔진만 보고 자동차를 사지 않습니다. 섀시(chassis), 서스펜션(suspension), 변속기(transmission), 안전 시스템(safety systems) 및 이들을 연결하는 대시보드(dashboard) 때문에 자동차를 삽니다. 엔진은 교체 가능합니다. V6 엔진을 전기 모터로 바꿔도 자동차는 여전히 주행합니다. 클라우드 에이전트도 마찬가지입니다. LLM은 스택의 가장 아래에 있는 구성 요소일 뿐입니다. 하네스는 원시 모델 기능을 코드를 배포하는 데 유용하게 만드는 그 위의 모든 것입니다.

Cursor의 하네스는 다섯 가지 계층으로 이루어져 있습니다. 가장 위에는 **인터페이스 계층(interface layer)**이 있습니다: 작업이 들어오는 곳(Slack, GitHub, 모바일, IDE). 그 아래에는 **오케스트레이션 계층(orchestration layer)**이 있습니다: 작업이 어떻게 계획되고, 어떤 모델이 선택되며, 얼마나 많은 에이전트가 병렬로 실행되는지. 그다음 **실행 계층(execution layer)**이 있습니다: 코드가 실제로 실행되는 곳(격리된 VM, 여러분의 노트북 아님). 이어서 **검증 계층(verification layer)**이 있습니다: 에이전트가 자신의 출력이 작동함을 증명하는 방법(컴퓨터 사용, 비디오, 스크린샷, 로그). 마지막으로 **출력 계층(output layer)**이 있습니다: 개발자가 받는 것(채팅 메시지가 아닌 아티팩트가 포함된 PR).

모델은 이 다섯 가지 계층 아래에 위치합니다. GPT-5를 Claude, Gemini, Composer 2로 바꿔도 하네스는 동일하게 작동합니다. 이것은 이론적인 주장이 아닙니다. Cursor는 실제로 동일한 에이전트 세션 내에서 다른 하위 작업에 여러 모델을 라우팅하며, 최상의 결과를 선택하기 위해 여러 모델에 대해 동일한 작업을 병렬로 실행합니다. 모델은 엔진입니다. 하네스는 자동차입니다. 그리고 Cursor의 293억 달러 가치는 자동차에 대한 가치 평가입니다.

**Cursor 클라우드 에이전트 심층 분석**

**코드베이스 온보딩(Codebase Onboarding)**

에이전트는 한 줄의 코드를 작성하기 전에 여러분의 레포지토리(repository)를 읽습니다. Cursor는 대규모 레포지토리(repositories) 전반에 걸친 코드베이스 리콜(codebase recall)을 위해 맞춤형 임베딩 모델(embedding model)을 구축했습니다. 클라우드 에이전트가 시작되면, 하위 에이전트(subagents)들이 병렬로 확장되어 코드베이스의 다른 부분을 탐색하며, 각 하위 에이전트는 해당 하위 작업에 가장 적합한 모델을 사용합니다. 한 하위 에이전트는 프런트엔드 컴포넌트 트리(frontend component tree)를 색인화하고, 다른 하나는 API 경로(API routes)를 매핑하며, 세 번째는 데이터베이스 스키마(database schema)를 읽을 수 있습니다. 그 결과는 에이전트가 어떤 파일도 건드리기 전에 프로젝트에 대한 실무 지식을 제공하는 컨텍스트 맵(context map)을 생성합니다.

새로운 레포지토리의 경우, cursor.com/onboard에서 온보딩(onboarding)을 시작할 수 있습니다. 에이전트는 자체 환경을 구성하고, 종속성(dependencies)을 설치하며, 작동하는 애플리케이션의 데모 비디오를 기록합니다. 여러분은 데모를 시청하여 에이전트가 여러분의 프로젝트를 올바르게 이해했는지 확인합니다. 이것은 README 파서(parser)가 아닙니다. 이는 검증된 기준선(baseline)을 생성하는 코드베이스에 대한 능동적인 탐색입니다.

**VM 샌드박스(Sandbox)**

각 클라우드 에이전트는 파일 시스템, 터미널(terminal), 브라우저, 실행 중인 애플리케이션 인스턴스(application instance)를 포함한 완전한 개발 환경을 갖춘 자체 격리된 리눅스 VM을 얻습니다. 여러분의 노트북은 관여하지 않습니다. 에이전트 간에 리소스 경쟁이 없으며, 에이전트와 여러분 사이에도 리소스 경쟁이 없습니다. 이러한 격리가 병렬 처리(parallelism)를 가능하게 합니다. 여러분은 10~20개의 에이전트를 동시에 실행할 수 있으며, 각 에이전트는 자체 샌드박스(sandbox)에서 완전히 다른 브랜치(branch)에서 완전히 다른 작업을 수행합니다.

클라우드 에이전트 이전에는 로컬 에이전트 모드(local agent mode)는 여러분의 기기가 편집기를 실행하고, 에이전트를 실행하고, 테스트 중인 애플리케이션을 실행하는 등 두 가지 작업을 동시에 수행해야 함을 의미했습니다. 한 노트북에서 세 개의 에이전트를 실행하면 메모리 압력, 포트 충돌 및 제트 엔진 소리 같은 팬 소음이 발생했습니다. 클라우드 VM은 이 모든 문제를 해결합니다.

2026년 3월, Cursor는 엔터프라이즈 팀(enterprise teams)을 위한 자체 호스팅 클라우드 에이전트(self-hosted cloud agents)를 출시했습니다. 동일한 기능: 격리된 VM, 완전한 개발 환경, 다중 모델 하네스(multi-model harnesses), 플러그인(plugins). 차이점은 여러분의 코드베이스(codebase), 빌드 출력(build outputs) 및 비밀(secrets)이 네트워크를 벗어나지 않는다는 것입니다. 에이전트는 여러분의 인프라(infrastructure)에서 도구 호출(tool calls)을 로컬로 처리합니다. 동일한 하네스, 다른 신뢰 경계(trust boundary)입니다.

**계획 모드(Plan Mode) 및 모델 라우팅(Model Routing)**

Cursor의 복잡한 기능 워크플로우(workflow)는 두 단계로 나뉩니다. 첫째, 여러분은 모델과 함께 로컬에서 반복하여 상세한 계획을 만듭니다: 기능이 무엇을 해야 하는지, 어떤 파일을 건드리는지, 승인 기준(acceptance criteria)이 어떻게 보이는지. 계획이 확정되면, 구현을 위해 클라우드 에이전트에게 보냅니다. 여러분은 다음 작업으로 이동합니다. 에이전트는 여러분이 동의한 계획에 따라 백그라운드에서 작동합니다.

모델 라우팅(model routing)은 개발자 수준이 아닌 하네스 수준에서 이루어집니다. Cursor는 작업 유형과 복잡성에 따라 각 하위 작업에 가장 적합한 모델을 선택합니다. GPT-5 Codex 에이전트 하네스는 장기적인 클라우드 환경에서 작동하도록 특별히 개편되었습니다. Claude는 추론 중심의 하위 작업을 처리합니다. Gemini는 대규모 컨텍스트 창(context windows)을 효율적으로 처리합니다. 사용자 상호 작용에서 강화 학습(reinforcement learning)으로 훈련된 Cursor 자체 모델인 Composer 2는 어려운 작업에서도 강력한 결과로 일상적인 코딩을 처리합니다.

가장 흥미로운 오케스트레이션 패턴(orchestration pattern)은 경쟁(race)입니다. Cursor는 동일한 문제를 여러 모델에 병렬로 보내고 최상의 결과를 선택합니다. 팀은 이것이 최종 출력 품질을 크게 향상시킨다고 보고하는데, 특히 몇 가지 정확한 변경이 필요한 어려운 버그(bugs)의 경우 더욱 그렇습니다. 이것이 하네스 수준에서 모델 종속성(model lock-in)이 중요하지 않은 이유입니다. 하네스가 경쟁을 주도하고, 모델은 경쟁자입니다.

**컴퓨터 사용(Computer Use)**

2026년 2월 24일, Cursor는 클라우드 에이전트를 이전의 모든 것과 차별화하는 기능을 출시했습니다. 이제 에이전트는 자신이 만든 소프트웨어를 사용할 수 있습니다. 각 VM에는 브라우저가 포함되어 있습니다. 에이전트는 애플리케이션을 구축하고, 실행하고, 로컬호스트(localhost)로 이동하여 사람이 하는 방식으로 UI와 상호 작용합니다: 버튼을 클릭하고, 양식을 채우고, 페이지를 탐색하고, 요소(elements)가 올바르게 렌더링(render)되는지 확인하는 것입니다.

에이전트가 이 검증 과정에서 문제를 발견하면, 작업을 멈추고 실패를 보고하지 않습니다. 코드로 돌아가 문제를 해결하고, 다시 빌드(rebuild)하고, 다시 테스트합니다. 이 루프(loop)는 에이전트가 변경 사항이 실제로 작동함을 검증할 때까지 계속됩니다. 검증이 통과되면, 에이전트는 전체 세션의 비디오를 녹화하고, 주요 상태의 스크린샷을 찍고, 로그(logs)를 수집합니다. 이 모든 것은 아티팩트(artifacts)로 풀 리퀘스트(pull request)에 첨부됩니다.

Cursor는 이 기능을 내부적으로 사용하고 있습니다 (dogfooding). 그들은 클라우드 에이전트를 사용하여 Cursor Marketplace의 소스 코드 링크를 구축했습니다: 에이전트는 기능을 구현하고, 가져온 Prisma 플러그인으로 이동했으며, 각 컴포넌트를 클릭하여 GitHub 링크가 작동하는지 확인한 다음, 메인(main) 브랜치로 리베이스(rebase)하고, 병합 충돌을 해결하며, 단일 커밋으로 스쿼시(squash)했습니다. 보안 작업을 위해, 그들은 Slack에서 클라우드 에이전트를 시작하여 클립보드 데이터 유출 취약점(clipboard exfiltration vulnerability)을 분류했습니다. 에이전트는 익스플로잇 페이지(exploit page)를 구축하고, 백엔드 서버(backend server)를 시작하고, 브라우저에 로드하며, 완전한 공격 흐름을 기록했습니다. 요약은 Slack 스레드(thread)에 나타났습니다.

**증거로서의 PR**

클라우드 에이전트로부터의 풀 리퀘스트(pull request)는 단순한 차이(diff)가 아닙니다. 이는 차이, 비디오 데모, 스크린샷, 로그, 그리고 깨끗한 커밋 히스토리(리베이스, 충돌 해결, 스쿼시)를 포함합니다. 이 PR을 검토할 때, 여러분은 코드를 읽고 그것이 작동하는지 정신적으로 시뮬레이션하는 것이 아닙니다. 여러분은 에이전트가 기능을 시연하는 30초짜리 비디오를 시청합니다. 이것은 검토 병목 현상(review bottleneck)을 근본적으로 변화시킵니다: 실행이 아닌 의도(intent)를 검증합니다.

이 워크플로우를 채택한 팀의 경우, 코드 검토 대화가 바뀌었습니다. 검토자들은 "이것이 작동하는가?"라고 묻는 대신 "이것이 우리가 실제로 원했던 것인가?"라고 묻기 시작했습니다. 기계적인 검증은 에이전트가 처리합니다. 인간의 판단은 제품 결정(product decisions)을 위해 남겨집니다. 이는 엔지니어링 집중의 의미 있는 재배분입니다.

**하네스에 매핑하기**

이제 실제로 어떤 일이 일어나는지 보셨으니, 이전 섹션의 하네스 스택(harness stack)과 어떻게 연결되는지 설명합니다. 하네스의 각 계층은 방금 살펴본 워크플로우의 한 부분에 해당합니다.

*   **인터페이스 계층(Interface layer).** 클라우드 에이전트는 Slack, GitHub, Linear, cursor.com/agents (웹), 모바일 또는 데스크톱 IDE에서 트리거(trigger)될 수 있습니다. Cursor 3의 에이전트 창(Agents Window)은 채팅 패널을 영구적인 오케스트레이션 패널(orchestration panel)로 대체했습니다: 파일 차이(file diffs) 및 진행률 표시기(progress indicators)와 함께 계획(Planning), 실행(Executing), 검토(Reviewing) 또는 완료(Done) 상태를 보여주는 작업 카드(task cards)가 있습니다. 여러분은 여러 작업을 보내고, 이를 카드로 모니터링하며, 결과가 나타나면 검토합니다. 정신 모델(mental model)은 페어 프로그래밍(pair programming)에서 프로젝트 관리(project management)로 전환되었습니다.
*   **오케스트레이션 계층(Orchestration layer).** 계획 모드(Plan mode), 모델 라우팅(model routing), 경쟁 패턴(race pattern) 및 하위 에이전트 배포(subagent dispatch)가 여기에 있습니다. 이 계층은 작업이 어떻게 수행되고 어떤 모델이 작업을 수행할지 결정합니다. 위 경쟁 패턴에서 보듯이, Cursor는 단일 모델에 의존하지 않습니다. 동일한 문제에 대해 여러 모델을 실행하고 최상의 출력을 선택함으로써 위험을 분산시킵니다.
*   **실행 계층(Execution layer).** VM 격리(VM isolation) 및 자체 호스팅 옵션(self-hosted option). 이 계층은 코드가 어디에서 실행되고 누가 환경을 제어할지 결정합니다. 신뢰 경계(trust boundary) 결정이 여기에 있습니다: Cursor 호스팅은 코드가 여러분의 기기를 벗어남을 의미하고; 자체 호스팅은 네트워크 내에 머무름을 의미합니다.
*   **검증 계층(Verification layer).** 컴퓨터 사용, 비디오 녹화, 스크린샷 캡처(screenshot capture) 및 로그 수집. 이 계층은 코드 차이(code diff)를 증명으로 전환합니다. 이 기능이 없으면 클라우드 에이전트는 원격 코드 생성기에 불과합니다. 이 기능이 있으면 작업을 시연하는 자율 엔지니어입니다.
*   **출력 계층(Output layer).** 아티팩트가 포함된 PR. 에이전트의 작업과 개발자의 검토 사이의 루프(loop)를 닫는 결과물(deliverable) 형식입니다.

하네스는 다섯 개의 계층으로 구성되며, 모델은 가장 아래에 한 개의 계층으로 존재합니다. 모델은 교체 가능합니다. 하네스는 그렇지 않습니다.

**하네스를 기반으로 구축된 플랫폼**

클라우드 에이전트 하네스는 기반입니다. Cursor는 그 위에 네 가지 플랫폼 기능을 구축했습니다.

*   **자동화(Automations).** 여러분이 없는 상태에서도 일정에 따라 또는 외부 도구의 이벤트에 따라 트리거(trigger)되는 이벤트 중심 에이전트(event-driven agents)입니다. 트리거가 발생하면 Cursor는 클라우드 샌드박스(cloud sandbox)를 실행하고, 여러분의 지침을 따르며, 구성한 MCP(Multi-Competency Protocols)를 사용하고, 선택적으로 이전 실행 결과를 기억하여 시간이 지남에 따라 개선될 수 있습니다. 주목할 만한 한계: 자동화는 아직 컴퓨터 사용을 지원하지 않으므로 자동화된 에이전트는 시각적 검증을 수행할 수 없습니다.
*   **버그봇 자동 수정(Bugbot Autofix).** 버그봇(Bugbot)은 코드 검토자로 시작했습니다. 이제 PR에서 문제를 발견하면, 자체 VM에서 클라우드 에이전트를 실행하고, 수정 사항을 테스트하며, 풀 리퀘스트에 직접 수정 사항을 제안합니다. 버그봇 자동 수정 제안의 35% 이상이 기본 PR에 병합되고 있습니다. 해결률(버그봇이 플래그한 버그 중 병합 전에 수정되는 비율)은 6개월 동안 52%에서 76%로 증가했으며, 실행당 식별되는 문제의 평균 수는 거의 두 배로 늘어났습니다. 이 도구는 단순히 더 많은 것을 찾아내는 것이 아니라 더 정확해지고 있습니다.
*   **플러그인 생태계(Plugin ecosystem).** Atlassian, Datadog, GitLab, Glean, Hugging Face, monday.com, PlanetScale을 포함한 파트너사의 30개 이상의 플러그인(plugins)이 있습니다. 대부분의 플러그인에는 클라우드 에이전트가 수동으로 또는 자동화를 통해 시작될 때 사용할 수 있는 MCP가 포함되어 있습니다. 여기에서 하네스는 통합 플랫폼(integration platform)이 됩니다: 단일 작업 내에서 Jira에서 읽고, Datadog에 쓰고, PlanetScale을 쿼리(query)할 수 있는 에이전트입니다.
*   **Cursor 3 "Glass".** 2026년 4월 2일에 출시되었으며, 에이전트 오케스트레이션(agent orchestration)을 중심으로 재구축된 IDE입니다. 에이전트 창(Agents Window)은 채팅 패널을 대체합니다. 다중 레포 레이아웃(Multi-repo layout)은 에이전트가 단일 작업 공간(workspace)에서 여러 레포지토리(repositories)에 걸쳐 읽고 쓸 수 있게 합니다. 디자인 모드(Design Mode)는 UI 컴포넌트(components)를 위한 시각적 편집기를 제공합니다: 라이브 미리보기에서 요소를 클릭하고, 자연어로 변경 사항을 설명하면, 에이전트가 실시간으로 미리보기가 업데이트되면서 소스 코드(source code)를 수정합니다. Cursor는 출시 시점에 ARR 20억 달러를 달성했으며 AI 코딩 도구 시장의 약 25%를 차지합니다.

**설계 선택 및 장단점**

**Cursor가 제대로 한 일**

*   **비디오 아티팩트는 검토를 10배 빠르게 만듭니다.** 여러분은 실행이 아닌 의도를 검증합니다. 30초짜리 비디오는 차이(diff)를 읽는 것만으로는 잡아낼 수 없는 시각적 회귀(visual regressions)를 포착합니다.
*   **병렬 에이전트(Parallel agents)는 잘 정의된 작업의 처리량(throughput)을 증대시킵니다.** 여러분이 아키텍처(architecture)에 집중하는 동안 10개의 버그 수정이 동시에 실행됩니다. 생산성 계산은 간단합니다: 각 에이전트가 잘 범위가 지정된 작업에서 30분을 절약한다면, 10개의 에이전트는 한 세션에서 5시간을 절약합니다.
*   **다중 표면 액세스(Multi-surface access)는 개발자들이 이미 작업하는 곳에서 이들을 지원합니다.** Slack에서 에이전트를 시작하고, 휴대폰에서 PR을 검토하며, GitHub 이슈에서 작업을 보냅니다. 인터페이스 계층은 하네스에서 가장 화려하지 않은 부분이지만, 일상적인 행동을 바꿀 만큼 충분한 마찰을 제거합니다.
*   **자체 호스팅 옵션(Self-hosted option)은 가장 어려운 엔터프라이즈 반대 의견을 해결합니다.** "우리 코드는 네트워크를 벗어날 수 없습니다"는 Cursor Cloud를 완전히 배제하는 요인이었습니다. 2026년 3월 이후, 이 반대 의견에는 답이 생겼습니다: 동일한 하네스, 여러분의 인프라.

**Cursor가 잘못한 일 (또는 아직 해결하지 못한 일)**

*   **게으른 삭제(Lazy delete).** 병렬 워커(Parallel workers)는 가끔 `// ... existing code ...`와 같은 플레이스홀더 주석을 사용하여 실제 코드를 소리 없이 삭제합니다. 차이(diffs)를 신중하게 검토해야 합니다. 이것은 드문 예외 사항(edge case)이 아닙니다; 여러 독립적인 검토자들이 이를 지적했습니다.
*   **크레딧 소진(Credit burn).** 고강도 에이전트 워크플로우(agentic workflows)는 크레딧을 빠르게 소진합니다. 병렬 에이전트에 크게 의존하는 개발자의 경우, 프로 플랜(Pro plan)의 컴퓨팅 상한선(compute ceiling)이 월 중순에 도달합니다. 에이전트 모드(Agent mode)는 작업당 약 0.04달러로 여러 모델 호출을 실행하며, Claude Sonnet은 Gemini보다 요청당 2.4배 더 비쌉니다. 하네스가 어떤 모델을 라우팅할지 결정하며, 이 라우팅 결정은 실제 비용에 영향을 미칩니다.
*   **작업 범위 지정(Task scoping)이 진정한 기술입니다.** 너무 광범위한 작업("인증 모듈 리팩토링")은 광범위한 검토를 요구하는 대규모 변경을 생성합니다. 너무 좁은 작업("이 변수 이름 바꾸기")은 과잉입니다. 최적의 지점은 중간 크기의 개별 작업입니다: "/api/users의 기존 미들웨어 패턴을 사용하여 /api/auth/login 경로에 속도 제한을 추가합니다." 이 범위 지정을 배우는 데는 며칠의 연습이 필요합니다.
*   **레거시 코드베이스(Legacy codebases)의 문제.** 클라우드 에이전트는 일관된 규칙과 좋은 테스트 커버리지(test coverage)를 가진 잘 구조화된 현대적인 코드베이스에서 가장 잘 작동합니다. 일관성이 없는 패턴을 가진 레거시 모놀리스(legacy monolith)에 에이전트를 사용하면 더 많은 수동 제어와 낮은 성공률을 예상해야 합니다.
*   **에이전트 복구(Agent recovery)는 투박합니다.** 에이전트가 잘못된 방향으로 가면, 수정 워크플로우(일시 중지, 잘못된 방향 설명, 재지정)는 이전의 대화형 채팅 모델보다 더 많은 마찰을 생성합니다. Cursor 3의 에이전트 창(Agents Window)은 포크(fork)하고 다시 시작할 수 있는 작업 카드(task cards)로 이를 개선했지만, Cursor 2의 반복적인 주고받는 방식보다는 여전히 거칠습니다.
*   **그림자 코드(Shadow code).** CTO와 엔지니어링 관리자에게 가장 큰 운영상의 두려움은 인간 개발자들이 이해하거나 제대로 검토하지 못하는 AI가 자율적으로 작성한 로직(logic)입니다. Cursor의 엔터프라이즈 플랜(Enterprise plan)은 AI 코드 추적 API(AI code tracking API)와 감사 로그(audit logs)를 포함하여 어떤 모델이 어떤 라인을 작성했는지 추적합니다. 그러나 검토 규율(review discipline)은 도구가 아닌 팀에서 나와야 합니다.

**Claude Code와 Codex가 다르게 해결하는 방법**

이것은 Claude Code나 OpenAI Codex에 대한 제품 리뷰가 아닙니다. 하네스 비교입니다. 세 가지 도구 모두 동일한 작업을 해결합니다: 개발자가 다른 작업을 하는 동안 코드를 자율적으로 배포하는 것입니다. 이들은 동일한 계열의 최신 모델(frontier models)을 사용합니다. 결과는 하네스가 완전히 다르기 때문에 완전히 다릅니다.

**Claude Code 백그라운드 에이전트(Background Agents)**

Claude Code는 거의 모든 면에서 Cursor와 반대되는 아키텍처적 접근 방식(architectural bet)을 취합니다. 터미널 우선(terminal-first)이며, 코드는 여러분의 로컬 머신에 남아 있습니다. 클라우드 VM은 없습니다. 여러 에이전트에 걸친 병렬 오케스트레이션(parallel orchestration) 대신, Claude Code는 단일 복잡한 작업에 깊이 파고듭니다: 작업 트리 격리(worktree isolation)를 통한 하위 에이전트 생성(subagent spawning)은 Claude가 프로젝트의 여러 브랜치(branches)에서 동시에 간섭 없이 작업할 수 있음을 의미하지만, 이 모든 것은 로컬에서 실행됩니다.

추론 깊이(reasoning depth)에서 Claude Code는 앞서 나갑니다. Opus와 최대 200K 토큰(tokens)의 컨텍스트(context)를 통해, Claude Code는 대규모 코드베이스(codebase) 전반에 걸쳐 지속적인 추론이 필요한 아키텍처 리팩토링(architectural refactors) 및 복잡한 다중 파일 변경을 처리합니다. MCP 통합은 기본적이며 깊이 있습니다: Claude Code는 프로토콜 서버(protocol servers)의 전체 생태계를 통해 로컬 도구, 데이터베이스 및 서비스에 연결됩니다.

Claude Code에는 비디오 아티팩트, 클라우드 VM, 또는 GUI 오케스트레이션 계층이 없습니다. 출력은 비디오 증명이 포함된 PR이 아닌 차이(diff) 및 대화 로그(conversation log)입니다. 신뢰 모델(trust model)은 로컬 우선(local-first)입니다: 여러분의 코드는 절대 여러분의 기기를 떠나지 않으므로, 엄격한 데이터 상주(data residency) 요구 사항이 있는 팀에게 기본 선택이 됩니다.

Dispatch 기능(2026년 3월 17일 출시)은 원격 제어 계층(remote control layer)을 추가합니다: QR 코드를 스캔하면 여러분의 휴대폰이 데스크톱에서 실행되는 Claude Code 세션(session)의 무전기가 됩니다. 이것은 "여러분이 없는 동안 AI가 작동한다"는 문제를 해결하지만, 병렬 클라우드 VM이 아닌 단일 기기의 단일 에이전트로 해결합니다.

**OpenAI Codex**

Codex는 웹 인터페이스(web interface) 또는 GitHub 이슈(GitHub issues)에서 트리거되는 클라우드 샌드박스(cloud sandboxes)에서 작업을 실행합니다. 하네스는 Cursor보다 간단합니다: 한 번에 하나의 작업, 긴밀한 GitHub 통합, 비디오 아티팩트 없음, 모델 경쟁 없음. 대부분의 유용한 에이전트 작업은 이슈 트래커(issue tracker)의 잘 범위가 지정된 이슈에서 시작하여 PR로 끝난다는 것이 그들의 핵심이라고 봅니다.

Codex가 강점인 부분은 단순성입니다. GitHub 이슈를 가리키면, 이슈를 읽고, 브랜치를 생성하고, 변경 사항을 구현하며, PR을 엽니다. 워크플로우는 선형적이고 예측 가능합니다. 약점은 검증의 부재(컴퓨터 사용 불가, 비디오 증명 없음)와 병렬 처리의 부족입니다.

**하네스 비교**

| 기능 (Feature) | Cursor | Claude Code | OpenAI Codex |
|---|---|---|---|
| 아키텍처 (Architecture) | 클라우드 VM, 다중 에이전트, 병렬 | 로컬 우선, 단일 에이전트, 하위 에이전트 격리 | 클라우드 샌드박스, 단일 에이전트, 선형 |
| 검증 (Verification) | 비디오 아티팩트, 컴퓨터 사용 | 차이, 대화 로그 | 차이만 |
| 신뢰 모델 (Trust Model) | 클라우드 호스팅 (자체 호스팅 옵션) | 로컬 우선 | 클라우드 호스팅 |
| 사용자 경험 (UX) | IDE (에이전트 창), 웹, 모바일, Slack, GitHub | 터미널, 원격 제어 (모바일) | 웹, GitHub |
| 모델 전략 (Model strategy) | 다중 모델 라우팅, 경쟁 패턴 | 심층 통합 (Opus) | GPT 패밀리 |

**각 하네스가 빛을 발하는 때**

*   **Cursor**는 병렬 처리할 10개 이상의 잘 정의된 작업이 있고, 검토자를 위한 시각적 증명을 원하며, 팀이 Slack과 GitHub를 통해 소통하고, 클라우드 VM의 코드에 익숙하거나 (또는 자체 호스팅할 수 있는 경우) 유리합니다.
*   **Claude Code**는 깊은 추론을 요구하는 하나의 복잡한 아키텍처 작업이 있고, 코드가 로컬에 있어야 하며, 로컬 도구와 MCP 통합이 필요하거나, 터미널 우선으로 작업하는 경우 유리합니다.
*   **Codex**는 워크플로우가 GitHub 이슈 중심이고, 작업 범위가 잘 지정되어 있으며, 이슈에서 PR까지 가장 간단한 경로를 원하는 경우 유리합니다.

최신 모델(Frontier models)들은 세 가지 모두에 사용할 수 있습니다. 완전히 다른 하네스. 완전히 다른 개발자 경험. 하네스가 제품입니다.

**가격 책정: 실제로 무엇에 돈을 지불하는가**

Cursor는 크레딧 기반 요금제(credit-based billing)를 사용합니다. 모든 유료 플랜에는 달러로 표시되는 크레딧 풀(credit pool)이 포함됩니다. AI 요청을 하면, 어떤 모델을 사용하고 작업이 얼마나 복잡한지에 따라 두 가지 요소를 기준으로 크레딧이 차감됩니다.

크레딧 계산은 중요합니다. 20달러 프로 풀(Pro pool)은 약 225개의 Claude Sonnet 요청, 550개의 Gemini 요청 또는 500개의 GPT-5 요청을 포함합니다. 에이전트 모드(Agent mode)는 작업당 약 0.04달러로 여러 모델 호출을 실행하며, Claude Sonnet은 Gemini보다 요청당 2.4배 더 비쌉니다. 하네스가 어떤 모델을 라우팅할지 결정하며, 이 라우팅 결정은 실제 비용에 영향을 미칩니다.

핵심 통찰: 여러분은 모델 액세스(model access) 비용을 지불하는 것이 아닙니다. Claude, GPT-5, Gemini는 각 API를 통해 더 저렴하게 얻을 수 있습니다. 여러분은 하네스에 비용을 지불하는 것입니다: VM 인프라, 코드베이스 색인화(codebase indexing), 아티팩트 생성 파이프라인(artifact generation pipeline), 플러그인 생태계 및 이 모든 것을 연결하는 오케스트레이션 계층(orchestration layer)입니다.

**요약 및 다음 단계**

*   Cursor 클라우드 에이전트(Cloud Agents)는 비디오 증명이 포함된 격리된 VM에서 자율 코딩 작업을 실행합니다. Cursor 자체 PR의 35%가 이런 방식으로 배포됩니다. 모델은 교체 가능합니다. 하네스가 제품입니다.
*   하네스는 다섯 가지 계층(인터페이스, 오케스트레이션, 실행, 검증, 출력)으로 구성되며, 이는 기본 모델보다 개발자 경험을 훨씬 더 많이 결정합니다. Cursor, Claude Code, Codex는 이를 증명합니다: 동일한 모델, 근본적으로 다른 하네스, 근본적으로 다른 결과.
*   결정은 "어떤 모델이 가장 좋은가"가 아닙니다. 결정은 "어떤 하네스가 우리 팀의 신뢰 모델, 병렬 처리 요구 사항 및 워크플로우에 가장 적합한가"입니다.
