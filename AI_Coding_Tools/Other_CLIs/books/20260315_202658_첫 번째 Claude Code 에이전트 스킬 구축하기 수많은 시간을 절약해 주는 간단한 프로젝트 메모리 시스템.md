## AI 기억 상실(AI Amnesia)의 진정한 비용

당신이 공감할 만한 상황을 그려보겠습니다:

**1개월 차: 발견**

*   **오류:** `localhost:3000`에서 CORS 정책(CORS policy)으로 요청 차단됨
*   [2시간 동안 디버깅(debugging), 프록시(proxy) 구성 시도, 헤더(header) 변경, Nginx 재작성]
*   **해결책:** `package.json`에 프록시 구성 추가

**6개월 차: 데자뷔**

*   **오류:** `localhost:3000`에서 CORS 정책으로 요청 차단됨
*   **개발자:** "이거 왠지 익숙한데... 어떻게 고쳤더라?"
*   [오래된 커밋(commit) 검색, 스택 오버플로우(Stack Overflow) 다시 확인, 기억 없는 Claude에게 문의]
*   [정확히 동일한 프록시 구성 해결책을 다시 찾아내는 데 1시간 소요]

익숙하게 들리시나요? 여기 불편한 진실이 있습니다. AI 코드 어시스턴트는 실제로는 이 문제를 더 악화시킬 뿐, 개선하지 않습니다. 기억이 없다면:

*   각 새 채팅 세션(chat session)은 지식 없이 처음부터 시작됩니다.
*   모든 버그는 처음 해결하는 것처럼 느껴집니다.
*   해결책은 반복적으로 "재발견"됩니다 (저는 한 번 6개월 동안 동일한 CORS 문제를 네 번 해결했습니다).
*   시간이 지나도 당신이나 AI 어시스턴트에게 학습이 축적되지 않습니다.

숨겨진 비용은 엄청납니다. 이미 해결했던 문제를 다시 해결하는 데 일주일에 30분만 소비해도 연간 26시간이 됩니다. 시간당 100달러로 계산하면 개발자 한 명당 연간 2,600달러의 시간이 낭비되는 셈입니다.

하지만 AI 어시스턴트가 기억할 수 있다면 어떨까요?

## Claude Code 에이전트 스킬(Agent Skill)이란 무엇인가요?

프로젝트 메모리(project-memory)에 뛰어들기 전에, 스킬이 실제로 무엇인지 이해해 봅시다. Claude에게 특정 워크플로(workflow)를 가르치거나 도메인 지식(domain knowledge)을 주고 싶었던 적이 있다면, 스킬이 그 답입니다.

에이전트 스킬은 단순히 `SKILL.md` 파일과 선택적 지원 리소스(resource)를 포함하는 폴더입니다. `SKILL.md` 파일은 두 부분으로 구성됩니다:

*   **YAML 프런트매터(YAML frontmatter):** Claude에게 스킬 활성화 시기를 알려주는 메타데이터(metadata)
*   **마크다운 본문(Markdown body):** 스킬이 활성화되었을 때 Claude가 따르는 지침

스킬을 설치할 수 있는 재사용 가능한 "전문가 모드"라고 생각하면 됩니다. "프로젝트 메모리를 설정해 줘"라고 말하면 Claude는 단지 추측하는 것이 아닙니다. 해당 작업을 위한 구체적이고 테스트된 지침을 로드(load)합니다.

에이전트 스킬 구조 해부학
전체 크기 이미지 보려면 Enter 또는 클릭: 이름과 설명을 포함하는 YAML 프런트매터와 개요, 사용 시기, 핵심 기능, 예시, 성공 기준 섹션이 있는 마크다운 본문을 보여주는 SKILL.md 파일 구조

이 이미지는 이름과 설명을 포함하는 YAML 프런트매터와 개요, 사용 시기, 핵심 기능, 예시, 성공 기준 섹션이 있는 `SKILL.md` 파일 구조를 보여줍니다.

스킬은 일반적으로 두 곳에 저장될 수 있습니다:

전체 크기 이미지 보려면 Enter 또는 클릭

스킬은 사용자 홈 디렉토리(`~/.claude/skills/`)에 저장됩니다. 이 스킬들은 모든 프로젝트에서 전역적으로 사용 가능하며 코드 검토, 문서화와 같은 일반적인 목적의 스킬들입니다. 그리고 `.claude/skills/`에 저장되는 프로젝트별 스킬들이 있습니다. 이 스킬들은 해당 프로젝트에만 로컬(local)하게 적용되는 프로젝트별 워크플로, 팀 컨벤션(convention)을 위한 것입니다. 일반적으로라고 말하는 이유는 엔터프라이즈(enterprise) 스킬, 모노레포(mono-repo) 수준에서 공유되는 스킬 등도 있을 수 있기 때문입니다. 또한 사용 중인 제공자(provider)에 따라 다릅니다 (`~/.codex/skills/`, `~/.config/opencode/skill/`, `~/.gemini/skills/`, `.claude/skills/`, `.codex/skills/`, `.opencode/skills/` 등 몇 가지만 언급하자면).

💡 **직접 해보기:** `ls ~/.claude/skills/`를 실행하여 이미 설치된 스킬을 확인하세요. 해당 디렉토리(directory)가 아직 존재하지 않는다면, 첫 번째 스킬을 설치할 때 생성됩니다.

### 스킬 활성화 작동 방식

요청을 하면 Claude Code는 에이전틱 스킬을 작성하기 위한 점진적 공개 패턴(progressive disclosure pattern)을 따릅니다. 필요한 것만 필요한 시기에 로드합니다. 이는 상호 작용을 빠르게 유지하면서도 강력한 기능을 온디맨드(on demand)로 사용할 수 있도록 합니다.

전체 크기 이미지 보려면 Enter 또는 클릭: 세 가지 단계(발견: 디렉토리 스캔, 메타데이터 로드; 매칭: 요청이 스킬과 일치하는지 확인; 실행: SKILL.md 로드, 지침 따르기, 작업 완료)를 보여주는 스킬 활성화 흐름도

스킬 활성화 흐름도는 세 가지 단계(발견(Discovery): 디렉토리 스캔, 메타데이터(metadata) 로드; 매칭(Matching): 요청이 스킬과 일치하는지 확인; 실행(Execution): `SKILL.md` 로드, 지침 따르기, 작업 완료)를 보여줍니다.

핵심 통찰력: Claude는 스킬 디렉토리를 스캔(scan)하고 일치하는 것을 찾을 때까지 메타데이터(이름 + 설명)만 읽습니다. 그때서야 전체 지침을 로드합니다. 이는 수십 개의 스킬이 설치되어 있어도 일반적인 상호 작용을 늦추지 않는다는 의미입니다.

이제 메커니즘을 이해했으니, 프로젝트 메모리가 어떻게 이를 실천하는지 살펴봅시다.

bugs.md, decisions.md, key_facts.md, issues.md를 포함하는 `docs/project_notes/` 폴더와 프로젝트 메모리를 구성하는 프로젝트 노트 파일에 연결되는 구성 파일 CLAUDE.md 및 AGENTS.md를 보여주는 프로젝트 메모리 파일 구조를 확인하세요.

각 메모리 파일은 프로젝트 지식을 캡처하는 데 특정한 역할을 합니다. 예를 들어, `bugs.md`는 검색 가능한 버그 데이터베이스(database) 역할을 하며, "BUG-018: Pulumi 상태 불일치(state drift) — pulumi refresh — yes 실행"과 같은 항목은 문제와 해결책을 모두 문서화하여 동일한 문제가 다시 나타날 때 미래의 시간 낭비를 방지합니다. 저는 이것을 프로젝트에서 반복되는 문제를 추적하는 데 사용합니다. `decisions.md` 파일은 "ADR-012: 모든 차트(chart)에 D3.js 사용 (팀 전문 지식)"과 같은 아키텍처 결정 기록(ADR, Architectural Decision Records)을 통해 아키텍처 선택을 추적하여 일관성을 보장하고 향후 상충되는 기술 결정을 방지합니다. 한편, `key_facts.md`는 "스테이징 API: https://api.staging.example.com:8443"과 같은 필수 프로젝트 세부 정보를 위한 빠른 참조 역할을 하여 포트(port)와 URL에 대한 추측을 없앱니다. 마지막으로, `issues.md`는 "TICKET-456: 사용자 인증 구현 — 2024-01-15"와 같은 항목으로 시간 순서대로 작업 로그(log)를 유지하여 완료된 작업을 특정 티켓(ticket) 및 날짜에 연결하는 감사 추적(audit trail)을 생성합니다.

이 스킬의 각 부분이 어떻게 작동하는지 자세히 설명하겠습니다.

### 파트 1: 에이전트 스킬의 프런트매터 (스킬의 "언제")

```yaml
---
name: project-memory
description: Set up and maintain a structured project memory system in docs/project_notes/
  that tracks bugs with solutions, architectural decisions, key project facts, and work
  history. Use this skill when asked to "set up project memory", "track our decisions",
  "log a bug fix", "update project memory", or "initialize memory system".
---
```

`description` 필드는 매우 중요합니다. 이것은 스킬의 가장 중요한 부분이라고 할 수 있습니다. Claude에게 언제 활성화될지 알려줍니다. 설명에 포함된 특정 트리거 구문("set up project memory", "track our decisions", "log a bug fix")에 주목하세요. 이러한 구문 중 어느 것이든 말하면 Claude는 일치하는 것을 인식하고 전체 지침을 로드합니다. (전체 스킬은 이 글의 끝에 있습니다.)

💡 **직접 해보기:** 자신만의 스킬 설명을 작성할 때는 사용자가 자연스럽게 말할 만한 3~5가지 특정 구문을 포함하세요. "결정 추적을 도와줘" 대 "결정 추적 설정해 줘" 대 "우리의 아키텍처 결정을 기록하고 싶어"와 같이 다양한 표현을 시도하여 활성화를 테스트해 보세요.

### 파트 2: 이 에이전트 스킬을 사용하는 시점 (상황별 트리거)

```markdown
## When to Use This Skill

Invoke this skill when:

- Starting a new project that will accumulate knowledge over time
- The project already has recurring bugs or decisions that should be documented
- The user asks to "set up project memory" or "track our decisions"
- Encountering a problem that feels familiar ("didn't we solve this before?")
- Before proposing an architectural change (check existing decisions first)
```

이 섹션은 트리거를 강화하고 Claude가 메모리가 유용한 더 넓은 맥락을 이해하는 데 도움이 됩니다. "전에 이걸 해결하지 않았던가?"라는 문구는 메모리가 도움이 될 수 있는 정확한 좌절의 순간을 포착하는 데 특히 강력합니다.

### 파트 3: 초기 설정 지침 (에이전트 스킬의 "무엇")

```markdown
## Core Capabilities

### 1. Initial Setup - Create Memory Infrastructure

When invoked for the first time in a project, create the following structure:

docs/
└── project_notes/
    ├── bugs.md         # Bug log with solutions
    ├── decisions.md    # Architectural Decision Records
    ├── key_facts.md    # Project configuration and constants
    └── issues.md       # Work log with ticket references

**Directory naming rationale:** Using `docs/project_notes/` instead of `memory/`
makes it look like standard engineering organization, not AI-specific tooling.
```

이 설계 결정은 생각보다 중요합니다. 메모리가 일반 문서처럼 보이게 만드세요. `ai-memory/` 또는 `claude-context/`에 파일을 만들면 개발자들은 이를 AI 도구로 간주하고 유지 관리하지 않을 것입니다. `docs/project_notes/`에 있으면 AI 지원을 사용하든 안 하든 누구든지 업데이트할 수 있는 표준 엔지니어링 문서처럼 보입니다.

### 파트 4: 메모리 항목 형식 (구조화된 지식)

이 스킬은 각 항목 유형에 대한 일관된 형식을 정의합니다. Claude는 구조화된 형식을 몇 달 후에도 안정적으로 구문 분석(parse)할 수 있기 때문에 일관성이 중요합니다.

전체 크기 이미지 보려면 Enter 또는 클릭: bugs.md 항목 형식(이슈, 근본 원인, 해결책, 예방)과 decisions.md ADR 형식(컨텍스트, 결정, 대안, 결과)의 나란히 비교

위는 `bugs.md` 항목 형식(이슈, 근본 원인, 해결책, 예방)과 `decisions.md` ADR 형식(컨텍스트, 결정, 대안, 결과)의 나란히 비교입니다.

버그 항목의 '예방(Prevention)' 필드는 특히 가치가 있습니다. 이것은 버그를 "우리가 고친 문제"에서 "우리가 배운 교훈"으로 바꿉니다. 6개월 후, Claude는 당신이 동일한 문제에 부딪히기 전에 경고할 수 있습니다.

### 파트 5: CLAUDE.md를 위한 메모리 프로토콜 (마법)

여기서 진정한 힘이 나타납니다. 이 스킬은 CLAUDE.md를 구성하여 Claude가 기본적으로 메모리를 인식하도록 만듭니다.

```markdown
### 2. Configure CLAUDE.md - Memory-Aware Behavior

Add or update the following section in the project's `CLAUDE.md` file:

## Project Memory System

### Memory-Aware Protocols

**Before proposing architectural changes:**
- Check `docs/project_notes/decisions.md` for existing decisions
- Verify the proposed approach doesn't conflict with past choices

**When encountering errors or bugs:**
- Search `docs/project_notes/bugs.md` for similar issues
- Apply known solutions if found
- Document new bugs and solutions when resolved

**When looking up project configuration:**
- Check `docs/project_notes/key_facts.md` for credentials, ports, URLs
- Prefer documented facts over assumptions
```

여기서 마법이 일어납니다. 이 프로토콜(protocol)들이 CLAUDE.md에 있으면, Claude는 결정을 내리거나 제안을 하기 전에 자동으로 메모리 파일을 확인합니다. "버그 로그를 확인해 줘"라고 말할 필요가 없습니다. 그것이 기본 동작이 됩니다.

메모리가 일상적인 개발 워크플로(workflow)를 어떻게 변화시키는지 구체적인 예를 공유하겠습니다.

메모리 인식 버그 해결

### 예시 1: 버그 해결 속도 (실제 수치)

**시나리오:**

10월 20일, 배포 중 Pulumi 상태 불일치(state drift) 오류가 발생했습니다. 오류 메시지는 모호했습니다: `error: update failed`. 45분 동안 디버깅하고, IAM 권한을 확인하고, 최근 변경 사항을 검토하고, 다른 배포 전략을 시도한 후, 해결책인 `pulumi refresh --yes`를 찾아 상태 파일을 동기화했습니다.

저는 이를 BUG-018로 문서화했습니다:

```markdown
### 2024-10-20 - BUG-018: Pulumi State Drift During Deploy

**Issue**: Deploy failed with cryptic "update failed" error after manual 
GCP console changes

**Root Cause**: Pulumi state file out of sync with actual infrastructure 
after teammate made console changes

**Solution**: Run `pulumi refresh --yes` before deploy to sync state

**Prevention**: Add `pulumi refresh --yes` to CI/CD pipeline before deploys; 
document that manual console changes require refresh
```

이틀 후인 10월 22일, 동일한 오류가 나타났습니다. 이번에는:

*   저는 Claude에게 오류에 대해 알렸습니다.
*   Claude는 `bugs.md`를 확인하고 BUG-018을 찾았습니다.
*   Claude는 `pulumi refresh --yes`를 제안했습니다.
*   문제는 5분 만에 해결되었습니다.

**결과:** 45분이 5분으로 단축되었습니다. 해결 시간이 89% 감소했습니다.

## 보안: 절대 저장하면 안 되는 것

이 섹션은 매우 중요합니다. 프로젝트 메모리를 사용하기 전에 주의 깊게 읽으십시오.

`key_facts.md` 파일은 프로젝트 구성(configuration)을 편리하게 찾아볼 수 있도록 설계되었습니다. 그러나 부주의하면 편리함이 위험을 초래할 수 있습니다. 메모리 파일에 절대로 보안 비밀(secrets)을 저장하지 마십시오. 이것들은 저장소(repo)에 접근할 수 있는 누구든지 읽을 수 있는 버전 관리되는 마크다운(Markdown) 파일입니다. 이 스킬은 이 주의 사항을 명시합니다.

### 위험 영역

`key_facts.md` (또는 어떤 메모리 파일에도) 다음을 절대 저장하지 마세요:

전체 크기 이미지 보려면 Enter 또는 클릭: `key_facts.md`에 절대 저장하면 안 되는 것 (요약)
*   **인증 자격 증명(Authentication Credentials):** 비밀번호, API 키, 액세스 토큰 또는 모든 형태의 인증 비밀
*   **서비스 계정 키(Service Account Keys):** GCP/AWS JSON 키 파일, 개인 키 또는 서비스 계정 자격 증명
*   **OAuth 비밀(OAuth Secrets):** 클라이언트 비밀, 새로 고침 토큰 또는 OAuth 구성 비밀
*   **데이터베이스 자격 증명(Database Credentials):** 비밀번호 또는 모든 데이터베이스 인증 세부 정보가 포함된 연결 문자열(connection string)
*   **인프라 비밀(Infrastructure Secrets):** SSH 개인 키, VPN 자격 증명 또는 네트워크 액세스 자격 증명

### 안전 영역

`key_facts.md`에 저장해도 안전한 것:

전체 크기 이미지 보려면 Enter 또는 클릭: 저장해도 안전한 것 (요약)
*   **호스트 이름 및 URL:** `api.staging.example.com` 또는 `https://api.example.com/v1`과 같은 공개 도메인 이름
*   **포트 번호:** `PostgreSQL: 5432` 또는 `Redis: 6379`와 같은 표준 포트
*   **프로젝트 식별자(Project Identifiers):** GCP 프로젝트 ID, AWS 계정 ID 또는 이와 유사한 비민감 식별자
*   **서비스 계정 이메일 주소:** `service-account@project.iam.gserviceaccount.com`과 같은 ID 정보
*   **환경 이름:** `staging`, `production`, `dev`와 같은 환경 레이블(label)
*   **공개 엔드포인트(Public Endpoints):** 공개적으로 접근 가능한 모든 URL 또는 API 엔드포인트
*   **구성 값(Configuration Values):** 타임아웃(timeout) 값, 재시도 횟수, 기능 플래그(feature flags)와 같은 비민감 설정

### 보안 비밀(Secrets)의 보관 장소

전체 크기 이미지 보려면 Enter 또는 클릭: 보안 비밀의 보관 장소 (요약)
*   **.env 파일 (gitignored):** `DATABASE_PASSWORD=secret123`와 같은 로컬 개발 비밀에 가장 적합
*   **클라우드 비밀 관리자(Cloud Secrets Managers):** 프로덕션 시스템은 GCP Secret Manager, AWS Secrets Manager 또는 Azure Key Vault를 사용해야 함
*   **CI/CD 변수(Variables):** 자동화된 파이프라인은 GitHub Actions secrets, GitLab CI/CD 변수 또는 이와 유사한 것을 사용해야 함
*   **비밀번호 관리자:** 1Password, LastPass, Bitwarden 또는 이와 유사한 도구를 통한 자격 증명 팀 공유
*   **환경 변수:** 배포 시점에 주입되는 런타임(runtime) 비밀, 버전 제어에 커밋(commit)되지 않음
*   **Kubernetes Secrets:** 쿠버네티스 시크릿 오브젝트(Secret objects)를 사용하는 컨테이너화된 애플리케이션용
*   **하드웨어 보안 모듈(HSM):** 엔터프라이즈 환경에서 고도로 민감한 암호화 키(cryptographic keys)용

💡 **직접 해보기:** 지금 당장, 기존의 `key_facts.md` 파일(있는 경우)에서 `password`, `secret`, `key`, `token`과 같은 패턴(pattern)을 검색하세요. 비밀 정보가 발견되면 즉시 이동시키고 손상된 자격 증명(credential)을 교체하세요.

## 흔한 함정 (그리고 피하는 방법)

수십 개의 프로젝트에 프로젝트 메모리를 사용하고 다른 사람들이 이를 채택하도록 도운 후, 저는 동일한 실수를 반복해서 보았습니다. 이러한 실수로부터 배워서 좌절감을 덜어내세요.

### 함정 1: 문서화 유령 도시

**문제:** 열정적으로 프로젝트 메모리를 설정하고 세 가지 항목을 추가한 다음 다시는 손대지 않습니다. 6개월 후, 파일은 오래되고 쓸모없게 됩니다.

**해결책:** 메모리 업데이트를 할 일 목록(todo list)이 아닌 워크플로에 추가하세요. 버그를 수정할 때마다 Claude에게 "이것을 `bugs.md`에 기록해 줘"라고 요청하세요. 모든 아키텍처 토론 후에는 "이 결정에 대한 ADR(아키텍처 결정 기록)을 추가해 줘"라고 말하세요. 의도적인 행동이 아닌 반사적인 행동으로 만드세요.

저는 또한 워크플로를 아키텍트 에이전트, 코드 에이전트, 그리고 그 사이에 있는 사람으로 나누는 다른 스킬을 작성했습니다. 이 스킬은 Claude용 훅(hook) 또는 OpenCode용 플러그인(plugin)을 설치할 수 있는 기능을 가지고 있으며, 주요 결정 및 리디렉션(redirection)의 로그 파일을 자동으로 추적합니다. 저는 나중에 백그라운드 에이전트(background agent)를 시작하여 놓쳤을 수도 있는 주요 결정, 버그, 핵심 사실을 스캔하도록 할 수 있습니다. 이런 방식으로 잊어버리더라도 백그라운드 에이전트가 로그를 스캔하고 중요한 사실을 찾을 수 있습니다.

### 함정 2: 소설 길이의 항목

**문제:** 버그 항목이 여러 페이지짜리 에세이가 됩니다. 결정 기록에는 전체 토론 기록이 포함됩니다. 너무 길어서 아무도 읽지 않습니다.

**해결책:** 각 항목은 30초 안에 스캔할 수 있어야 합니다. 구조화된 형식(이슈, 근본 원인, 해결책, 예방)을 사용하세요. 더 자세한 정보가 필요하면 별도의 문서에 링크하세요.

### 함정 3: 만능 파일

**문제:** 모든 정보가 `key_facts.md`에 들어가서 500줄로 늘어나고 조직화되어 있지 않습니다.

**해결책:** 네 개의 파일을 각각의 목적에 맞게 사용하세요:

*   버그는 `bugs.md`에
*   결정은 `decisions.md`에
*   구성은 `key_facts.md`에
*   작업 기록은 `issues.md`에

`key_facts.md`가 100줄을 초과하면 아마도 사실로 위장된 결정이 포함되어 있을 것입니다.

### 함정 4: 데자뷔(Déjà Vu) 함정

**문제:** 당신은 다음과 같은 답답한 순간을 경험합니다: "이 버그 전에 본 적 있어. 고쳤던 것도 알아. 해결책도 있어." 하지만 세부 사항을 기억할 수 없어서 30분을 더 투자하여 해결책을 다시 찾아냅니다.

**해결책:** 이것이 바로 프로젝트 메모리가 없애는 문제입니다. 데자뷔 느낌이 들 때, 첫 번째 반응은 "`bugs.md` 파일을 확인해 봐"여야 합니다. 반복되는 CORS 문제요? 문서화되어 있습니다. 화요일마다 실패하는 불안정한 테스트요? 해결책이 기록되어 있습니다. 인증 예외 상황요? 이미 해결되고 기록되어 있습니다.

아름다운 점은 프로젝트 메모리가 기관 지식(institutional knowledge)을 생성한다는 것입니다. 팀에서 흔히 "부족 지식(tribal knowledge)"이라고 부르는 것을 누군가의 머리에 가두는 대신 접근 가능하게 만듭니다. 팀이 모든 차트에 D3.js를 사용하기로 결정하면 그 결정은 기록됩니다. 6개월 후, 신규 팀원이 합류하여 "어떤 차트 라이브러리를 사용해야 하나요?"라고 물으면 Claude는 추측하지 않습니다. `decisions.md`를 확인하고 올바른 답변을 제공합니다.

**실제 영향:** 더 이상 같은 문제를 두 번 해결할 필요가 없습니다. 더 이상 일관되지 않은 기술 선택이 없습니다. 더 이상 "분명히 전에 이걸 고쳤는데"라는 좌절감이 없습니다. 메모리가 지속되기 때문에 반복되는 문제는 사라집니다.

### 함정 5: 비밀 저장소

**문제:** 개발자들이 "편의를 위해" API 키를 `key_facts.md`에 저장합니다. 키는 영원히 Git 기록에 남게 됩니다. `.env`를 사용하고 `.env`를 `.gitignore`에 추가하세요. 우리는 문명인이고 사회에는 규칙이 있습니다. :)

**해결책:** 위 보안 섹션을 참조하세요. 이미 비밀을 커밋(commit)했다면 즉시 교체하세요. Git 기록에서 삭제하는 것만으로는 충분하지 않습니다. 비밀이 이미 다른 곳에 복제되었을 수 있기 때문입니다.

### 함정 6: 고립된 메모리

**문제:** 한 개발자가 프로젝트 메모리를 충실히 유지 관리합니다. 다른 개발자들은 그것이 존재하는지조차 모릅니다. 지식 사일로(silo)가 형성됩니다.

**해결책:** README 또는 온보딩 문서에 프로젝트 메모리에 대한 섹션을 추가하세요. PR(Pull Request) 검토에서 메모리 파일을 언급하세요: "이 버그 수정은 `bugs.md`에 문서화되어야 합니다."

### 함정 7: 오래된 결정

**문제:** ADR-003은 "모든 API에 REST 사용"이라고 되어 있지만, 팀은 2년 전에 GraphQL로 마이그레이션(migrate)했습니다. 이제 메모리 파일에는 오해의 소지가 있는 정보가 포함되어 있습니다.

**해결책:** 분기별로 결정 기록을 검토하세요. 오래된 결정은 `**Status**: Superseded by ADR-027`과 같이 대체됨으로 표시하세요. 삭제하지 마세요. 미래의 개발자들이 왜 오래된 코드가 REST를 사용하는지 궁금해할 수 있습니다.

## 나만의 에이전트 스킬 구축하기

프로젝트 메모리 스킬은 당신이 만들 수 있는 모든 스킬에 적용할 수 있는 핵심 원칙을 보여줍니다. 이 가이드라인은 수십 개의 스킬을 만들고 어떤 스킬이 사용되고 어떤 스킬이 버려지는지 보면서 얻은 것입니다.

### 1. 실제로 겪는 실제 문제를 해결하세요.

다음과 같은 상황을 발견할 때 스킬을 만드세요:

*   Claude (또는 동료)에게 같은 설명을 반복하는 경우
*   같은 정보를 반복해서 찾아보는 경우
*   같은 다단계 프로세스를 수동으로 따르는 경우
*   Claude가 프로젝트에 대해 "그냥 알았으면" 하는 바람이 있는 경우

**안티패턴(Anti-pattern):** 가상의 문제를 위한 스킬을 만들지 마세요. 고통을 경험하지 않았다면 올바른 해결책을 설계할 수 없을 것입니다.

### 2. 초점을 맞추세요 (하나의 스킬 = 하나의 목적).

모든 것을 다 하는 "슈퍼 스킬"을 만들려는 충동을 억제하세요. 구성 가능하고 집중된 스킬이 더 재사용성이 높고 유지 관리가 쉽습니다.

*   **좋은 예:** 프로젝트 메모리(project-memory)는 메모리를 처리합니다. 코드 검토(code-review)는 검토를 처리합니다.
*   **나쁜 예:** 프로젝트-모든것(project-everything)은 메모리, 검토, 배포, 커피 주문까지 관리하려고 합니다.

### 3. 명확하고 자연스러운 트리거(Trigger) 구문을 사용하세요.

설명에는 사용자가 자연스럽게 말할 만한 특정 구문이 포함되어야 합니다:

*   "프로젝트 메모리 설정"
*   "이 버그 기록"
*   "핵심 사실 업데이트"
*   "우리의 결정 추적"

"문서화 서브시스템 초기화"와 같은 전문 용어가 많은 트리거는 피하고, "정말로 이렇게 말할까?"라고 자문하며 트리거를 테스트하세요.

### 4. 표준 문서처럼 보이게 만드세요.

`docs/`에 있는 파일은 유지 관리됩니다. `ai-stuff/`에 있는 파일은 무시됩니다.

팀이 이미 따르는 규칙을 사용하여 파일과 디렉토리 이름을 지정하세요. 결정이 일반적으로 `adr/` 폴더에 들어간다면 그곳에 두세요. 팀이 Notion을 사용한다면 Notion 형식으로 내보내는 스킬을 만드는 것을 고려하세요.

### 5. 템플릿과 예시를 포함하세요.

원하는 정확한 형식을 보여주세요. Claude는 추상적인 지침보다 예시를 더 안정적으로 따를 것입니다.

**덜 효과적:**

관련 정보로 버그 항목을 만드세요.

**더 효과적:**

```markdown
### YYYY-MM-DD - Bug Title

**Issue**: What happened
**Root Cause**: Why it happened
**Solution**: How you fixed it
**Prevention**: How to avoid it
```

## 직접 시도해 보세요

지금 바로 프로젝트 메모리를 워크플로에 추가할 준비가 되셨나요? 다음 5분 안에 시작하는 방법입니다.

### 빠른 시작

```bash
# skilz CLI 설치
pip install skilz

# project-memory 에이전트 스킬을 전역으로 설치 (모든 프로젝트에서 작동)
skilz install https://github.com/SpillwaveSolutions/project-memory
```

그런 다음 어떤 프로젝트에서든 Claude Code를 여세요:

"이 프로젝트의 프로젝트 메모리를 설정해 줘"

Claude는 메모리 인프라(infrastructure)를 생성하고 CLAUDE.md를 구성하며, 당신은 바로 시작할 수 있습니다.

만약 작동하지 않는다면 다음 중 하나를 시도해 볼 수 있습니다:

"프로젝트 메모리 스킬을 사용하여 이 프로젝트의 프로젝트 메모리를 설정해 줘"

또는...

"`/project-memory-skill` 이 프로젝트의 프로젝트 메모리를 설정해 줘"

이 스킬은 `AGENTS.md`와도 즉시 작동합니다. 제가 OpenCode와 Claude Code를 모두 자주 사용하기 때문에, 둘 다에서 작동하도록 작성했습니다. 다시 말하지만, 이것은 제가 작성한 첫 번째 스킬 중 하나이며 약간의 개선이 필요할 수 있습니다.

### 첫 주 도전 과제

모든 것을 한 번에 문서화하려고 하지 마세요. 대신:

*   **1일차:** 프로젝트 메모리 설정
*   **2일차:** `key_facts.md`에 항목 하나 추가 (항상 잊어버리는 URL이나 포트)
*   **3일차:** 버그를 수정할 때 `bugs.md`에 문서화
*   **4일차:** `decisions.md`에 하나의 아키텍처 결정 기록
*   **5일차:** Claude에게 이전에 문서화했던 것에 대해 질문

5일차쯤 되면 Claude가 며칠 전에 알려줬던 것을 기억할 때 첫 번째 "아하!" 순간을 경험하게 될 것입니다. 그때 가치가 현실이 됩니다.

### 성공의 모습

**1주일 후,** 당신은 다음을 달성해야 합니다:

*   메모리 파일 전반에 걸쳐 3~5개의 항목
*   Claude가 메모리를 사용하여 당신을 도운 사례 최소 1회
*   문서화 습관 형성

**1개월 후,** 당신은 다음을 달성해야 합니다:

*   메모리 파일 전반에 걸쳐 15~25개의 항목
*   메모리 조회를 통해 시간을 절약한 여러 사례
*   동료들이 "Claude가 어떻게 그걸 알아?"라고 묻는 상황

## 향후 개선 사항: 프로젝트 메모리를 더욱 향상시키기

이 스킬이 AI 기억 상실의 핵심 문제를 해결하지만, 개선의 여지는 많습니다. 프로젝트 메모리를 다음 단계로 끌어올릴 수 있는 몇 가지 개선 사항은 다음과 같습니다.

### 1. 자동 목차 생성

**필요성:** `bugs.md`, `decisions.md`, `key_facts.md`가 50-100개 항목을 넘어서면 특정 정보를 찾는 것이 더 어려워집니다. AI 코딩 어시스턴트는 에이전틱 검색(agentic search)을 사용하지만, 구조화된 탐색은 사람과 AI 모두에게 도움이 됩니다.

**해결책:** 각 파일이 특정 임계값(예: 20개 항목)을 초과할 때 파일 상단에 자동으로 목차를 생성합니다. 이는 의미론적 검색(semantic search) 기능을 향상시키고 수동 탐색을 훨씬 빠르게 만듭니다. 이는 부분적인 발견 아키텍처(partial discovery architecture)에 도움이 될 것입니다.

```markdown
## Table of Contents
- [Authentication Issues](#authentication-issues)
- [Database Performance](#database-performance)
- [CORS Problems](#cors-problems)
```

### 2. 자동 파일 보관

**필요성:** 2년 전의 버그 항목은 최근 항목만큼 관련성이 없습니다. 모든 것을 한 파일에 보관하면 노이즈(noise)가 발생하고 검색 품질이 저하됩니다. 이는 목차와 별개입니다.

**해결책:** `bugs.md` 및 `issues.md`에 대한 자동 보관(archiving)을 구현합니다:

*   6~12개월보다 오래된 버그를 `bugs-archive-2025.md`로 이동합니다.
*   메인 파일에 참조를 유지합니다: `bugs-archive-2025.md`에서 이전 항목 참조
*   결정 및 핵심 사실은 일반적으로 보관할 필요가 없습니다. 변경하기로 결정할 때까지 무기한으로 관련성을 유지합니다.

### 3. 다중 에이전트 지원 (Claude를 넘어서)

**필요성:** 이 스킬이 원래 생성되었을 때 Claude Code 스킬은 공개 표준이 아니었습니다. 이제 에이전트 스킬 표준이 존재하므로 프로젝트 메모리를 진정으로 보편적으로 만들어야 합니다.

**해결책:** 스킬이 여러 AI 코딩 어시스턴트를 감지하고 구성하도록 만듭니다:

*   Claude Code를 사용하는 경우 `CLAUDE.md`를 업데이트합니다.
*   Gemini를 사용하는 경우 `GEMINI.md`를 업데이트합니다.
*   Codex 또는 다른 도구를 사용하는 경우 `AGENT.md`를 업데이트하거나 구성 파일 위치를 묻습니다.

스킬은 설정 중에 "어떤 AI 코딩 어시스턴트를 사용하십니까?"라고 묻고 모든 관련 파일을 자동으로 구성할 수 있습니다.

**진화는 계속됩니다:** 이러한 향후 개선 사항은 AI 코딩 어시스턴트와 에이전틱 검색 기능 모두에 대한 더 깊은 이해를 반영합니다. 생태계가 성숙하고 에이전트가 프로젝트 메모리를 사용하는 방법에 대해 더 많이 알게 됨에 따라 스킬은 계속 진화할 것입니다.

스킬 시스템의 아름다움은 기존 기능을 손상시키지 않고 이러한 향상 기능을 점진적으로 추가할 수 있다는 것입니다. 기본부터 시작하여 필요에 따라 정교함을 더하십시오.

이 매우 간단한 스킬이 마음에 드셨고 에이전틱 스킬을 좀 더 잘 이해하는 데 도움이 되셨기를 바랍니다. 그렇다면 '좋아요'와 댓글을 남겨주세요.

## Overview

프로젝트를 위한 기관 지식을 유지하기 위해 `docs/project_notes/`에 구조화된 메모리 시스템을 구축합니다. 이 스킬은 네 가지 주요 메모리 파일(`bugs`, `decisions`, `key facts`, `issues`)을 설정하고 `CLAUDE.md` 및 `AGENTS.md`를 구성하여 이들을 자동으로 참조하고 유지 관리하도록 합니다. 결과적으로 코딩 세션과 다양한 AI 도구 전반에 걸쳐 과거 결정, 문제 해결책 및 중요한 구성 세부 정보를 기억하는 프로젝트를 얻을 수 있습니다.

## When to Use This Skill

다음과 같은 경우 이 스킬을 호출합니다:

- 시간이 지남에 따라 지식이 축적될 새로운 프로젝트를 시작할 때
- 프로젝트에 이미 문서화되어야 할 반복적인 버그나 결정이 있을 때
- 사용자가 "프로젝트 메모리를 설정해 줘" 또는 "우리의 결정을 추적해 줘"라고 요청할 때
- 사용자가 버그 수정, 아키텍처 결정 또는 완료된 작업을 기록하고 싶을 때
- 익숙하게 느껴지는 문제에 직면했을 때 ("전에 이거 해결하지 않았던가?")
- 아키텍처 변경을 제안하기 전에 (기존 결정을 먼저 확인)
- 여러 개발자나 AI 도구(Claude Code, Cursor 등)가 있는 프로젝트에서 작업할 때

## Core Capabilities

### 1. Initial Setup - Create Memory Infrastructure

프로젝트에서 처음 호출될 때 다음 구조를 생성합니다:

```
docs/
└── project_notes/
    ├── bugs.md         # 버그 로그 및 해결책
    ├── decisions.md    # 아키텍처 결정 기록
    ├── key_facts.md    # 프로젝트 구성 및 상수
    └── issues.md       # 티켓 참조가 있는 작업 로그
```

**디렉토리 이름 지정 근거:** `memory/` 대신 `docs/project_notes/`를 사용하면 AI 전용 도구가 아닌 표준 엔지니어링 조직처럼 보입니다. 이는 사람 개발자의 채택 및 유지 관리를 증가시킵니다.

**초기 파일 내용:** 이 스킬의 `references/` 디렉토리에서 템플릿을 복사합니다:
- 초기 `bugs.md`에는 `references/bugs_template.md`를 사용합니다.
- 초기 `decisions.md`에는 `references/decisions_template.md`를 사용합니다.
- 초기 `key_facts.md`에는 `references/key_facts_template.md`를 사용합니다.
- 초기 `issues.md`에는 `references/issues_template.md`를 사용합니다.

각 템플릿에는 형식 예시와 사용 팁이 포함되어 있습니다.

### 2. Configure CLAUDE.md - Memory-Aware Behavior

프로젝트의 `CLAUDE.md` 파일에 다음 섹션을 추가하거나 업데이트합니다:

```markdown
## Project Memory System

This project maintains institutional knowledge in `docs/project_notes/` for consistency across sessions.

### Memory Files

- **bugs.md** - Bug log with dates, solutions, and prevention notes
- **decisions.md** - Architectural Decision Records (ADRs) with context and trade-offs
- **key_facts.md** - Project configuration, credentials, ports, important URLs
- **issues.md** - Work log with ticket IDs, descriptions, and URLs

### Memory-Aware Protocols

**Before proposing architectural changes:**
- Check `docs/project_notes/decisions.md` for existing decisions
- Verify the proposed approach doesn't conflict with past choices
- If it does conflict, acknowledge the existing decision and explain why a change is warranted

**When encountering errors or bugs:**
- Search `docs/project_notes/bugs.md` for similar issues
- Apply known solutions if found
- Document new bugs and solutions when resolved

**When looking up project configuration:**
- Check `docs/project_notes/key_facts.md` for credentials, ports, URLs, service accounts
- Prefer documented facts over assumptions

**When completing work on tickets:**
- Log completed work in `docs/project_notes/issues.md`
- Include ticket ID, date, brief description, and URL

**When user requests memory updates:**
- Update the appropriate memory file (bugs, decisions, key_facts, or issues)
- Follow the established format and style (bullet lists, dates, concise entries)

### Style Guidelines for Memory Files

- **Prefer bullet lists over tables** for simplicity and ease of editing
- **Keep entries concise** (1-3 lines for descriptions)
- **Always include dates** for temporal context
- **Include URLs** for tickets, documentation, monitoring dashboards
- **Manual cleanup** of old entries is expected (not automated)
```

### 3. Configure AGENTS.md - Multi-Tool Support

프로젝트에 `AGENTS.md` 파일(에이전트 워크플로 또는 다중 도구 프로젝트에 사용됨)이 있는 경우 동일한 메모리 프로토콜을 추가합니다. 이는 Claude Code, Cursor, GitHub Copilot 또는 기타 AI 도구를 사용하든지 간에 일관성을 보장합니다.

**`AGENTS.md`가 존재하는 경우:** 위와 동일한 "Project Memory System" 섹션을 추가합니다.

**`AGENTS.md`가 존재하지 않는 경우:** 사용자에게 생성할 것인지 묻습니다. 많은 프로젝트는 여러 AI 도구를 사용하며 공유 메모리 프로토콜의 이점을 얻습니다.

### 4. Searching Memory Files

문제에 직면하거나 결정을 내릴 때, 메모리 파일을 적극적으로 검색합니다:

**bugs.md 검색:**
```bash
# 유사한 오류 찾기
grep -i "connection refused" docs/project_notes/bugs.md

# 날짜 범위로 버그 찾기
grep "2025-01" docs/project_notes/bugs.md
```

**decisions.md 검색:**
```bash
# 기술에 대한 결정 확인
grep -i "database" docs/project_notes/decisions.md

# 모든 ADR 찾기
grep "^### ADR-" docs/project_notes/decisions.md
```

**key_facts.md 검색:**
```bash
# 데이터베이스 연결 정보 찾기
grep -A 5 "Database" docs/project_notes/key_facts.md

# 서비스 계정 조회
grep -i "service account" docs/project_notes/key_facts.md
```

**더 복잡한 검색을 위해 Grep 도구 사용:**
- 모든 메모리 파일에서 검색: `Grep(pattern="oauth", path="docs/project_notes/")`
- 컨텍스트 인식 검색: `Grep(pattern="bug", path="docs/project_notes/bugs.md", -A=3, -B=3)`

### 5. Updating Memory Files

사용자가 업데이트를 요청하거나 해결된 문제를 문서화할 때, 적절한 메모리 파일을 업데이트합니다:

**버그 항목 추가:**
```markdown
### YYYY-MM-DD - 간략한 버그 설명
- **문제**: 무엇이 잘못되었는지
- **근본 원인**: 왜 발생했는지
- **해결책**: 어떻게 해결되었는지
- **예방**: 미래에 어떻게 피할 수 있는지
```

**결정 추가:**
```markdown
### ADR-XXX: 결정 제목 (YYYY-MM-DD)

**컨텍스트:**
- 왜 결정이 필요했는지
- 어떤 문제를 해결하는지

**결정:**
- 무엇이 선택되었는지

**고려된 대안:**
- 옵션 1 -> 왜 거부되었는지
- 옵션 2 -> 왜 거부되었는지

**결과:**
- 이점
- 장단점
```

**핵심 사실 추가:**
- 범주별로 구성 (GCP 프로젝트, 데이터베이스, API, 로컬 개발 등)
- 명확성을 위해 글머리 기호 목록 사용
- 프로덕션 및 개발 세부 정보 모두 포함
- 쉬운 탐색을 위해 URL 포함
- 저장하지 말아야 할 보안 가이드라인은 `references/key_facts_template.md`를 참조하세요.

**작업 로그 항목 추가:**
```markdown
### YYYY-MM-DD - TICKET-ID: 간략한 설명
- **상태**: 완료 / 진행 중 / 차단됨
- **설명**: 1-2줄 요약
- **URL**: https://jira.company.com/browse/TICKET-ID
- **참고**: 중요한 컨텍스트
```

### 6. Memory File Maintenance

**오래된 항목 주기적으로 정리:**
- 사용자는 수동 정리에 대한 책임이 있습니다 (자동화 없음).
- 더 이상 관련 없는 매우 오래된 버그 항목(6개월 이상)을 제거합니다.
- `issues.md`에서 완료된 작업(3개월 이상)을 보관합니다.
- 모든 결정을 유지합니다 (가볍고 역사적 컨텍스트를 제공합니다).
- 프로젝트 구성이 변경되면 `key_facts.md`를 업데이트합니다.

**충돌 해결:**
- `decisions.md`와 충돌하는 것을 제안하는 경우, 왜 결정을 재검토해야 하는지 설명합니다.
- 선택이 변경되면 결정 항목을 업데이트합니다.
- 개정 날짜를 추가하여 진화를 보여줍니다.

이 스킬에는 올바른 형식을 보여주는 `references/`의 템플릿 파일이 포함되어 있습니다:

- **`references/bugs_template.md`** - 예시가 포함된 버그 항목 형식

User: "그 CORS 수정 사항을 버그 로그에 추가해 줘."

```

```

## Tips for Effective Memory Management

1.  **적극적으로 행동하세요**: 해결책을 제안하기 전에 메모리 파일을 확인하세요.
2.  **간결하게 작성하세요**: 항목을 간결하게 유지하세요 (설명은 1-3줄).
3.  **날짜를 포함하세요**: 시간적 컨텍스트를 위해 항상 날짜를 포함하세요.
4.  **링크를 포함하세요**: 티켓, 문서, 모니터링 대시보드(dashboard)로 연결되는 URL을 포함하세요.
5.  **선택적으로 작성하세요**: 모든 버그가 아닌, 반복되거나 교육적인 문제에 초점을 맞추세요.
