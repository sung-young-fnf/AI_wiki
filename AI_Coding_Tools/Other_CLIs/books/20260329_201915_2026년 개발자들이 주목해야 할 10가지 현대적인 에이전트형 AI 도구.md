개발 워크플로우(workflow)가 더욱 복잡해지면서, 도구가 많다고 해서 항상 좋은 것은 아니라는 것을 깨달을 수 있습니다. 도구 간 전환, 동일한 명령 반복 실행, API 연결, 또는 단일 작업을 완료하기 위해 워크플로우의 여러 부분을 관리하는 데 많은 시간을 할애할 수 있습니다.

대부분의 도구는 단순히 명령을 실행하도록 만들어졌을 뿐, 사용자와 협력하도록 설계되지 않았습니다. 이 때문에 여전히 모든 것을 직접 조율하고, 도구 사이를 오가며 각 단계를 처리하고, 전체 프로세스를 체계적으로 유지해야 합니다.

**에이전트형 도구(Agentic tools)**는 단순히 지시에 응답하는 것을 넘어, 작업을 이해하고 코드베이스(codebase)와 상호작용하며 훨씬 적은 수동 노력으로 여러 단계의 작업을 자동화하는 데 도움을 줄 수 있습니다.

이러한 아이디어를 발전시키고 있는 몇 가지 현대적인 에이전트형 도구들을 살펴보겠습니다.

### 1. Goose 🦢
출처: https://github.com/block/goose

Goose는 사용자 컴퓨터에 상주하는 완전 자율 개발자 에이전트(autonomous developer agent)입니다. 툴킷(toolkits)을 통해 확장 가능하도록 설계되었습니다. 일반적인 챗봇과 달리 Goose는 셸 명령(shell commands)을 실행하고, 파일을 편집하며, Jira 및 GitHub와 같은 외부 API와 직접 상호작용할 수 있습니다.

**최적의 활용 사례 🎯**
-   코드 리팩토링(refactoring) 및 테스트 실행과 같은 작업을 스스로 처리합니다.
-   사용자 정의 워크플로우(custom workflows)를 학습하고 따릅니다.
-   단일 파일만이 아닌 전체 코드베이스(codebase)를 이해합니다.

**최적의 활용 사례 🎯**
-   AI가 전체 프로젝트를 볼 때 주요 리팩토링에 가장 적합합니다.
-   프로젝트 구조로 새로운 AI 세션(session)을 온보딩하는 데 좋습니다.
-   코드에서 README와 같은 전체 문서를 생성하는 데 유용합니다.

전체 크기 이미지 보려면 엔터 키를 누르거나 클릭하세요

**최적의 활용 사례 🎯**

### 5. Rivet 🔗
전체 크기 이미지 보려면 엔터 키를 누르거나 클릭하세요
출처: https://rivet.ironcladapp.com/

Rivet은 Ironclad에서 복잡한 LLM 작업(operations)을 위해 특별히 구축한 오픈 소스 시각적 프로그래밍 환경입니다. AI 로직(logic)을 노드 그래프(node graph)로 시각화하여, 복잡한 프롬프트(prompts) 체인, 로직 게이트(logic gates) 및 데이터 변환(data transformations)을 구축할 수 있습니다. 데이터를 와이어(wires)를 통해 실시간으로 흐르는 것을 관찰하고 에이전트가 어디에서 잘못되었는지 정확히 찾아낼 수 있습니다.

**최적의 활용 사례 🎯**
-   여러 결정 및 API 호출을 포함하는 복잡한 로직을 시각화합니다.
-   비개발자가 프롬프트를 안전하게 조정하는 데 좋습니다.
-   각 노드(node)의 입력 및 출력을 추적하고 디버깅하는 데 유용합니다.

### 6. Flowise 🦜
전체 크기 이미지 보려면 엔터 키를 누르거나 클릭하세요
출처: https://flowiseai.com/

LangChain 앱을 처음부터 작성해 본 적이 있다면, 반복적인 코드(boilerplate)가 빠르게 지저분해질 수 있다는 것을 알 것입니다. Flowise는 LangChain(그리고 이제 LlamaIndex)을 위한 오픈 소스 시각적 인터페이스입니다. PDF 로더(loaders), 벡터 스토어(Vector Stores), 임베딩(Embeddings), LLM 등 구성 요소를 드래그 앤 드롭(drag-and-drop)하고 와이어(wires)로 연결한 다음, 전체 체인(chain)을 깔끔한 API 엔드포인트(API endpoint)로 즉시 노출할 수 있습니다. 이는 몇 시간의 코딩을 몇 분의 상자 연결로 바꿉니다.

**최적의 활용 사례 🎯**
-   "PDF와 채팅" 봇을 빠르게 프로토타이핑(prototyping)하는 데 좋습니다.
-   비개발자가 코드를 건드리지 않고 프롬프트를 조정하는 데 이상적입니다.
-   시각적 로직(Visual Logic)에서 API를 즉시 배포하는 데 유용합니다.

전체 크기 이미지 보려면 엔터 키를 누르거나 클릭하세요
출처: https://flowiseai.com/

**피해야 할 경우 ⛔**
-   약간의 시각적 오버헤드(overhead)로 인해 초저지연(ultra-low latency) 요구 사항에는 적합하지 않습니다.
-   체인 내에서 고도로 사용자 정의된 로직(logic)에는 제한적일 수 있습니다.

### 7. Portkey 🔑
전체 크기 이미지 보려면 엔터 키를 누르거나 클릭하세요
출처: https://portkey.ai/

Portkey를 스마트 라우터(smart router)라고 생각해보세요. 사용자 코드와 LLM 공급자(providers) 사이에서 작동합니다. OpenAI에 장애가 발생하면 Portkey가 이를 감지하여 앱이 충돌하지 않고 즉시 트래픽을 다른 공급자로 재라우팅(reroutes)합니다. 재시도(retries), 캐싱(caching) 및 대체(fallbacks)를 자동으로 처리하여 에이전트를 즉시 프로덕션 수준(production-grade)으로 만듭니다.

**최적의 활용 사례 🎯**
-   주요 공급자가 다운(down)되더라도 앱을 온라인 상태로 유지합니다.
-   비용과 속도를 최적화하기 위해 프롬프트 라우팅에 좋습니다.
-   단일 대시보드(dashboard)에서 요청, 비용 및 지연 시간을 추적하는 데 유용합니다.

전체 크기 이미지 보려면 엔터 키를 누르거나 클릭하세요
출처: https://github.com/Portkey-AI/gateway

**피해야 할 경우 ⛔**
-   다운타임(downtime)이 중요하지 않은 소규모 앱에는 과도합니다.
-   프롬프트가 단일 공급자에서만 작동하는 경우에는 도움이 되지 않습니다.

### 8. Warp ⚡
전체 크기 이미지 보려면 엔터 키를 누르거나 클릭하세요
출처: https://www.warp.dev/

Warp는 Rust로 구축된 터미널 앱이지만, 기존 터미널보다 현대적인 텍스트 편집기를 사용하는 느낌을 줍니다. 대부분의 터미널에서는 모든 것이 하나의 긴 텍스트 스트림으로 나타나 관리하기 어려울 수 있습니다. Warp에서는 각 명령과 그 출력(output)이 **블록(Blocks)**이라는 명확한 섹션으로 정리됩니다. 이를 통해 작업을 더 쉽게 읽고, 복사하고, 편집하고, 공유할 수 있습니다.

Warp는 또한 명령줄 내부에 AI가 내장되어 있습니다. 따라서 "파일 압축 해제 방법(how to untar a file)"과 같은 것을 온라인에서 검색하기 위해 터미널을 떠날 필요가 없습니다. 단순히 입력하면 Warp가 사용자가 원하는 것을 이해하고 올바른 명령을 제안합니다.

**최적의 활용 사례 🎯**
