# Mistral Vibe: 터미널 기반 AI 코딩 어시스턴트

레거시 현대화 프로젝트를 시작한 지 3주째입니다. COBOL 코드베이스는 50,000줄에 달하고, 문서화는 미흡하며, 팀원들은 "AI가 실제로 도움이 될 수 있을까요?"라고 묻고 있습니다.

대부분의 대규모 언어 모델(LLM)은 함수 자동 완성이나 리팩터링을 제안하는 정도입니다.

하지만 실제로 필요한 것은 50개의 파일을 탐색하고, 의존성을 이해하며, 웹에서 최신 아키텍처 패턴을 가져오고, 동일한 Python 코드를 생성하고, 테스트를 작성하며, 개발 흐름을 유지하는 동안 전체 마이그레이션을 조율할 수 있는 도구입니다.

Mistral Vibe는 바로 이러한 도구입니다. 코드 스니펫만 작성하는 것이 아닌, 터미널 기반(terminal-native) AI입니다. 개발 환경 전체를 운영하고, MCP 서버를 통해 외부 도구와 연결되며, 병렬 에이전트를 통해 작업을 확장합니다.

이 가이드에서는 Mistral Vibe를 실제로 사용하는 방법을 보여줍니다.

## 왜 중요한가

AI 코딩 도구는 우리가 함수를 작성하고 파일을 편집하는 방식을 변화시켰습니다. Mistral Vibe는 전체 개발 워크플로를 조율하는 방식을 변화시킵니다.

그 차이는 범위와 자율성(autonomy)에 있습니다. IDE 플러그인은 현재 파일이나 주변 몇 개의 파일만 볼 수 있습니다. 반면 Vibe는 프로젝트 수준에서 작동합니다. 전체 코드베이스를 분석하고, 스키마 컨텍스트를 위해 데이터베이스를 쿼리하며, 실시간 문서를 가져와 의사결정에 활용하고, 테스트 스위트(test suites)를 실행하며, 의미 있는 메시지로 Git 커밋을 생성합니다.

그리고 가장 중요한 점은 Vibe가 필요한 곳이면 어디에서든 실행된다는 것입니다. 로컬 노트북, 에어갭(air-gapped) 데이터센터, 유럽 주권 클라우드 등 어디에서든 사용할 수 있습니다. 소유한 코드는 외부 인프라에 절대 노출되지 않습니다.

## 정신 모델: 터미널의 내장 지능 계층

터미널을 미션 컨트롤 센터라고 생각해 보세요. Git, pytest, Docker, kubectl, psql 등 각각 고유한 인터페이스와 특징을 가진 도구들이 있습니다. 여러분은 수십 개의 도구를 수동으로 조율하는 지휘자입니다.

Vibe는 여러분의 오케스트레이션 레이어(orchestration layer)가 됩니다. 여러분이 원하는 결과를 설명하면, Vibe가 실행을 조율합니다.

```
사용자: "모든 Flask 라우트를 FastAPI 비동기 핸들러로 리팩터링하고, 테스트를 실행하고, 실패를 분석하여 수정 사항을 제안해 줘."
Vibe: [47개 파일 읽기]
      [23개 함수 재작성]
      [pytest 실행]
      [14개 실패 분석]
      [파일:라인 참조와 함께 수정 제안]
```

컨텍스트 전환(context switching)이 없습니다. 도구 플래그(tool flags) 관리에 대한 정신적 부담도 없습니다. 오직 지속적이고 지원되는 흐름(flow)만 있습니다.

## 작동 방식: 설치부터 운영까지

자동화된 워크플로를 실행하기 위한 완전한 설정 과정을 처음부터 살펴보겠습니다.

### 1부: 설치 및 구성

**1단계: Vibe 설치**

```bash
pip install mistral-vibe
vibe --version
```

**2단계: MCP 런타임 확인**

MCP 서버는 Python 또는 Node.js 러너가 필요합니다:

```bash
# Python 서버
uvx --version
# 누락된 경우: pip install uv

# Node.js 서버
npx --version
# 누락된 경우: nodejs.org에서 설치
```

**3단계: 설정 파일 생성**

```bash
mkdir -p ~/.vibe
touch ~/.vibe/config.toml
```

`~/.vibe/config.toml` 파일 열기:

```toml
[models]
default = "devstral-2"
[agent]
enable_parallel = false  # 순차적으로 시작하여 나중에 활성화
max_concurrent_agents = 4
```

**4단계: 실행**

```bash
vibe
# ✓ Mistral Vibe 준비 완료
```

다음 명령을 시도해 보세요:

```
> /status      # 토큰 사용량 보기
> /help        # 모든 명령 보기
> /model       # 현재 모델
> /exit        # 종료 (또는 Ctrl+D)
```

끝났습니다. 이제 실행 중입니다.

### 2부: 핵심 기능

#### `@` 파일 탐색 자동 완성

**사용 이유:** 익숙하지 않은 코드베이스를 탐색하거나 전체 경로를 입력하지 않고 특정 파일을 참조해야 할 때 유용합니다. `ls` 또는 `find` 명령을 실행하기 위해 컨텍스트 전환을 할 필요가 없습니다.

`@` 기호는 코드베이스 탐색기입니다:

```bash
# 프로젝트 복제
git clone https://github.com/pallets/flask
cd flask/examples/tutorial
vibe
```

```
# Vibe 내부
> 이 프로젝트에 무엇이 있나요? @
[TAB 키를 눌러 모든 파일 보기]
> @flaskr/auth.py를 분석하고 인증 흐름을 설명해 줘
> @flaskr/db.py를 최신 SQLAlchemy 비동기 패턴과 비교해 줘
```

#### `!` Bash 실행 접두사

**사용 이유:** 명령을 실행하고 즉시 Vibe에게 결과를 해석해 달라고 요청해야 할 때 유용합니다. 터미널과 AI 어시스턴트 간의 복사-붙여넣기 과정을 없애줍니다.

Vibe를 벗어나지 않고 명령 실행:

```
> !pytest -v
# 테스트 실행, 결과 출력
> 테스트가 실패했어. 출력을 분석하고 무엇이 문제인지 알려줘.
> !git status
> 커밋되지 않은 변경 사항을 검토하고 적절한 커밋 메시지를 제안해 줘.
```

#### 코드베이스 분석

**사용 이유:** 레거시 프로젝트에 온보딩하거나 존재하지 않는 문서를 생성할 때 유용합니다. Vibe는 전체 구조를 읽고 코드 패턴으로부터 아키텍처를 추론합니다.

```
> 전체 프로젝트 구조를 분석하고 다음을 설명하는 README.md를 생성해 줘:
  - 아키텍처 개요
  - 설정 지침
  - API 엔드포인트
  - 테스트 접근 방식
```

Vibe는 모든 파일을 읽고, 관계를 이해하며, 문서를 생성합니다.

#### 코드 생성

**사용 이유:** 상용구(boilerplate)가 많은 기능을 구현하거나 정립된 패턴을 따를 때 유용합니다. Vibe는 오류 처리, 유효성 검사, 로깅이 이미 내장된 프로덕션 준비 코드(production-ready code)를 생성합니다.

```
> 사용자 등록을 위한 새로운 FastAPI 엔드포인트를 생성해 줘. 다음을 포함해야 해:
  - pydantic을 사용한 이메일 유효성 검사
  - bcrypt를 사용한 비밀번호 해싱
  - SQLAlchemy 비동기(async)를 사용한 데이터베이스 삽입
  - 적절한 오류 처리 및 로깅
  - 성공 시 JWT 토큰(JWT token) 반환
```

#### 대화 압축 (Conversation compaction)

**사용 이유:** 토큰 비용이 누적되는 여러 시간의 세션에서 작업할 때 유용합니다. 압축은 작업 컨텍스트를 유지하면서 API 비용을 70~80% 절감합니다.

긴 대화는 토큰을 소모합니다. 사용량을 추적하세요:

```
> /status
# 입력 토큰: 45,230
> /compact
# 압축 완료: 45,230 → 10,450 토큰 (-77%)
> /status
# 입력 토큰: 10,450
```

**작동 방식:** Vibe는 주요 의사결정을 추출하고, 불필요한 대화를 제거합니다. 컨텍스트는 유지하면서 비용을 줄일 수 있습니다.

### 3부: MCP 기능

#### MCP란 무엇인가요?

모델 컨텍스트 프로토콜(Model Context Protocol, MCP)은 Vibe를 외부 도구에 연결합니다. 플러그인과 비슷하지만 표준화되어 있습니다.

#### 웹 fetch 설정

**사용 이유:** 코드베이스에 없는 최신 문서, 모범 사례 또는 API 사양(specs)이 필요할 때 유용합니다. Vibe는 터미널을 벗어나지 않고 실시간 웹 콘텐츠를 가져와 해석합니다.

`~/.vibe/config.toml`에 추가:

```toml
[[mcp_servers]]
name = "fetch"
transport = "stdio"
command = "uvx"
args = ["mcp-server-fetch"]
[tools.fetch_fetch]
permission = "ask"  # 각 fetch 전에 프롬프트
```

재로드:

```
> /reload
# ✓ 1개 모델 · 1개 MCP 서버
```

**웹 fetch 사용**

```
> https://docs.python.org/3/library/asyncio.html에서 문서를 가져와 비동기 모범 사례를 요약해 줘.

> FastAPI 문서를 가져와 Flask의 의존성 주입(dependency injection) 방식과 비교해 줘.
> 최신 OWASP Top 10 취약점을 조사하고 @app/auth.py에 대해 감사(audit)를 실행해 줘.
```

#### 브라우저 자동화 설정

**사용 이유:** UI 렌더링을 검증하고, 시각적 문제를 디버그하거나, 종단 간(end-to-end) 테스트를 자동화해야 할 때 유용합니다. Playwright를 통해 Vibe는 브라우저를 제어하고 사람이 보는 것을 캡처할 수 있습니다.

시각적 테스트를 위한 Playwright 추가:

```toml
[[mcp_servers]]
name = "playwright"
transport = "stdio"
command = "npx"
args = ["-y", "@playwright/mcp"]

[tools.playwright_navigate]
permission = "ask"
[tools.playwright_screenshot]
permission = "always"  # 안전한 작업, 자동 승인
[tools.playwright_click]
permission = "ask"
```

**Playwright 사용**

```
> https://github.com/mistralai로 이동하여 스크린샷을 찍어줘.

> localhost:3000으로 이동하여 대시보드 스크린샷을 찍고 모든 UI 요소가 올바르게 렌더링되었는지 확인해 줘.
> 스테이징 앱으로 이동하여 테스트 자격 증명으로 로그인 폼을 채우고 리다이렉션을 확인해 줘.
```

#### Git 통합 설정

**사용 이유:** 실제로 변경 사항을 설명하는 AI 생성 커밋 메시지를 원하거나, 진화 과정을 이해하기 위해 저장소(repository) 히스토리를 분석해야 할 때 유용합니다. Vibe는 여러분의 Git 기록 관리자이자 커밋 작성자가 됩니다.

```toml
[[mcp_servers]]
name = "git"
transport = "stdio"
command = "uvx"
args = ["mcp-server-git", "--repository", "."]

[tools.git_status]
permission = "always"
[tools.git_commit]
permission = "ask"
[tools.git_push]
permission = "never"  # 푸시 차단, 명시적 작업 필요
```

**Git 사용**

```
> 내 Git 상태는? 커밋되지 않은 변경 사항이 있어?
> 변경 사항의 diff를 보여주고 무엇을 수정했는지 설명해 줘.
> 내 변경 사항에 대한 의미 있는 커밋 메시지를 생성해 줘.
> 마지막 10개 커밋을 분석하고 프로젝트의 진화 과정을 요약해 줘.
```

#### 데이터베이스 접근 설정

**사용 이유:** 스키마 관계를 이해하고, ORM 모델을 생성하거나, 수동으로 SQL을 작성하지 않고 쿼리를 최적화해야 할 때 유용합니다. Vibe는 데이터베이스를 쿼리하고 실제 테이블 구조를 기반으로 개선 사항을 제안합니다.

PostgreSQL의 경우:

```toml
[[mcp_servers]]
name = "postgres"
transport = "stdio"
command = "uvx"
args = ["mcp-server-postgres", "postgresql://user:pass@localhost/dbname"]

[tools.postgres_query]
permission = "ask"
[tools.postgres_list_tables]
permission = "always"
```

SQLite의 경우:

```toml
[[mcp_servers]]
name = "sqlite"
transport = "stdio"
command = "uvx"
args = ["mcp-server-sqlite", "--db-path", "./app.db"]

[tools.sqlite_read_query]
permission = "ask"
[tools.sqlite_write_query]
permission = "never"
```

**데이터베이스 통합 사용**

```
> 모든 테이블을 나열하고 스키마를 설명해 줘.

> 지난 7일 동안 생성된 계정에 대해 사용자 테이블을 쿼리해 줘.
> 데이터베이스 스키마를 분석하고 SQLAlchemy 모델을 생성해 줘.
> @app/queries.py를 검토하고 인덱스 최적화를 제안해 줘.
```

#### 권한 전략 (Permission strategy)

**사용 이유:** AI 자율성에 대한 정밀한 제어를 원할 때 유용합니다. 권한 설정을 통해 의도치 않은 파괴적인 작업을 방지하면서 안전한 읽기 및 분석 작업을 자유롭게 실행할 수 있습니다.

세 가지 수준으로 자율성을 제어합니다:

*   `always` = 자동 승인 (읽기 같은 안전한 작업)
*   `ask` = 실행 전 확인 (쓰기, 커밋, 쿼리)
*   `never` = 완전히 차단 (푸시, 파괴적인 작업)

도구 이름은 `{서버_이름}_{도구_이름}` 패턴을 따릅니다.

예시: `playwright_navigate` = Playwright 서버, `navigate` 도구

```
> https://github.com/topics/python으로 이동하여 스크린샷을 찍어줘.
> 스크린샷을 분석하고 상위 10개 인기 저장소(repo)를 나열해 줘.
> 데이터베이스에 연결하여 스키마를 분석하고 정규화 개선을 제안해 줘.
> Git 상태를 확인하고, 커밋되지 않은 변경 사항을 보여주고, 커밋 메시지를 생성해 줘.
```

### 4부: 고급 기능

#### 병렬 실행 활성화

**사용 이유:** 각 변경 사항이 독립적인 수십 개의 파일을 리팩터링할 때 유용합니다. 병렬 에이전트(parallel agents)가 동시에 작업하여 경과 시간을 2~4배 단축시키면서도 개발 흐름을 유지할 수 있습니다.

#### 자동화를 위한 헤드리스(Headless) 모드

**사용 이유:** Vibe를 CI/CD 파이프라인이나 크론 잡(cron jobs)에 통합할 때 유용합니다. 헤드리스 모드는 비대화형으로 실행되고, 구조화된 결과를 출력하며, 기존 자동화 워크플로에 적합합니다.

비대화형으로 Vibe 실행:

```bash
# 단일 명령
vibe --non-interactive "코드베이스를 분석하고 summary.txt에 요약을 출력해 줘."

# 특정 모델
vibe --model devstral-2-small --non-interactive "@src/auth.py에 대한 테스트를 생성해 줘."

# JSON 출력
vibe --output results.json --non-interactive "코드 리뷰를 실행해 줘."
```

#### CI/CD 통합

**사용 이유:** 사람 리뷰어를 추가하지 않고도 모든 풀 리퀘스트(pull request)에서 자동화된 코드 리뷰를 원할 때 유용합니다. Vibe는 변경 사항을 분석하고, 문제를 표시하며, 자동으로 피드백을 게시하여 병합 전에 문제를 발견합니다.

GitHub Actions 워크플로:

```yaml
name: AI Code Review
on: [pull_request]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Setup Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.11'
      
      - name: Install Vibe
        run: pip install mistral-vibe
      
      - name: Configure
        run: |
          mkdir -p ~/.vibe
          cat > ~/.vibe/config.toml << 'EOF'
          [models]
          default = "devstral-2"
          
          [tools.read_file]
          permission = "always"
          
          [tools.write_file]
          permission = "never"
          EOF
      
      - name: Run Review
        env:
          MISTRAL_API_KEY: ${{ secrets.MISTRAL_API_KEY }}
        run: |
          vibe --non-interactive --output review.json << 'PROMPT'
          PR 변경 사항을 검토해 줘:
          1. 코드 품질 (1-10점)
          2. 보안 취약점
          3. 성능 영향
          4. 테스트 커버리지 격차
          JSON 형식으로 출력해 줘.
          PROMPT
      
      - name: Comment on PR
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const review = JSON.parse(fs.readFileSync('review.json'));
            await github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `## 🤖 AI 코드 리뷰\n\n${review.summary}`
            });
```

**안전한 자동화 권한**

CI/CD를 위해 분석 전용 모드를 설정합니다:

```toml
# 자유롭게 읽고 분석
[tools.read_file]
permission = "always"

[tools.search_files]
permission = "always"
# CI에서 모든 수정 차단
[tools.write_file]
permission = "never"
[tools.run_command]
permission = "never"
[tools.git_commit]
permission = "never"
```

**철학:** CI는 분석하고 보고합니다. 사람은 변경 사항을 승인합니다.

#### 사용자 지정 MCP 서버

**사용 이유:** Vibe와 상호 작용해야 하는 내부 도구, 배포 시스템 또는 독점 API가 있을 때 유용합니다. 사용자 지정 서버는 Vibe의 기능을 고유한 인프라로 확장합니다.

자신만의 통합(integrations) 구축:

```python
# my_tools_server.py
from fastmcp import FastMCP
mcp = FastMCP("Internal Tools")
@mcp.tool()
def deploy_to_staging(service_name: str) -> str:
    """스테이징 환경에 서비스를 배포"""
    # 배포 로직
    return f"스테이징에 {service_name} 배포 완료"
@mcp.tool()
def check_service_health(service_name: str) -> dict:
    """서비스 상태를 확인"""
    # 상태 확인 로직
    return {"status": "healthy", "uptime": "99.9%"}
if __name__ == "__main__":
    mcp.run()
```

설정:

```toml
[[mcp_servers]]
name = "internal_tools"
transport = "stdio"
command = "python"
args = ["/path/to/my_tools_server.py"]

[tools.internal_tools_deploy_to_staging]
permission = "ask"
[tools.internal_tools_check_service_health]
permission = "always"
```

사용:

```
> user-service의 상태를 확인해 줘.
> auth-service를 스테이징에 배포해 줘.
```

## 설계 선택 및 장단점

### 선택 1: CLI 대 IDE 플러그인

| 접근 방식 | 최적의 용도 | 한계 |
|---|---|---|
| CLI (Vibe) | 다중 파일 리팩터링, 시스템 수준 변경, 자동화, 숙련된 엔지니어 | 에디터에서 컨텍스트 전환 필요 |
| IDE 플러그인 (VS Code 등) | 코드 라인별 코딩, 인라인 제안, 초급 개발자 | 에디터 컨텍스트로 제한, 스크립트 작성 어려움 |

여러 도구에 걸쳐 오케스트레이션이 필요할 때는 Vibe를 사용하세요. 단일 파일 편집에는 IDE 플러그인을 사용하세요.

### 선택 2: 모델 선택

| 모델 | 파라미터 | 지연 시간 | 사용 사례 |
|---|---|---|---|
| devstral-2 | 123B | 3–5초 | 복잡한 리팩터링, 아키텍처 |
| devstral-2-small | 24B | 0.5–1초 | 일상적인 작업, 빠른 반복 |
| codestral | 24B | <500ms | 코드 자동 완성(FIM) |

동적으로 전환:

```
> /model devstral-2-small
```

### 선택 3: 권한 전략

3단계 접근 방식:

**항상 자동 승인:**
*   파일 읽기
*   웹 fetch
*   Git 상태 확인
*   스크린샷

**실행 전 확인:**
*   파일 쓰기
*   데이터베이스 쿼리
*   Git 커밋
*   폼 제출

**절대 허용 안 함:**
*   Git 푸시
*   데이터베이스 쓰기
*   파괴적인 파일 작업
*   운영 환경 배포

### 선택 4: 병렬 대 순차

| 모드 | 속도 | 비용 | 사용 시기 |
|---|---|---|---|
| 순차 (Sequential) | 기준선 | $ | 작은 코드베이스 (10개 파일 미만), 간단한 작업 |
| 병렬 (4개 에이전트) | 2–4배 빠름 | $1.17 (17% 추가) | 대규모 리팩터링, 시간 민감 작업 |

병렬 실행은 중복된 컨텍스트 로딩으로 인해 비용이 추가되지만, 인간의 시간을 절약해 줍니다.
