# Python 개발자인 내가 실제로 사용하는 3가지 MCP 서버

지난달 저는 스스로가 당황스러운 상황에 놓여 있다는 것을 깨달았습니다. 네 개의 터미널과 세 개의 브라우저 탭을 열어둔 채, 슬랙(Slack)에서 업무 진척을 묻는 메시지에 시달리고 있었습니다. 제가 막혔던 이유는 파이썬(Python)이 어려워서가 아니라, 도구들이 서로 제각각 놀며 저를 방해하고 있었기 때문입니다.

그래서 저는 대부분의 숙련된 개발자들이 결국 그러하듯, 도구를 더하는 것을 멈추고 냉정하게 쳐내기 시작했습니다. 복잡하고 불안정한 실험적 도구들은 버리고, 매주 실질적으로 의존하고 있는 세 가지 MCP(Model Context Protocol) 서버만 남겼습니다. 이들은 단순히 유행하는 도구가 아니라, 인지 부하(cognitive load)를 줄여주고 불필요한 연결 코드(glue code)를 제거하며 제가 구조적인 문제에만 집중할 수 있게 도와주었습니다.

## 1. Filesystem MCP Server — 20개의 스크립트를 대체한 단 하나의 도구

Filesystem MCP 서버는 로컬 프로젝트 파일에 대해 구조화되고 안전하며 예측 가능한 접근 권한을 제공합니다. 

이 서버를 사용하기 전 저의 워크플로우는 이랬습니다.
- 디렉토리를 스캔하는 스크립트 하나
- 파일을 요약하는 스크립트 하나
- TODO나 설정값의 차이(config drift)를 검색하는 스크립트 하나
- 그리고 이들을 실행할 때마다 드는 정신적 비용

하지만 Filesystem MCP 서버를 사용하면서 이 모든 번거로움이 단일 인터페이스로 통합되었습니다. 대부분의 'AI 개발 도구'는 문맥 파악(context acquisition)에서 실패하곤 합니다. 그들은 일반적인 파이썬은 알지만, '당신의 파이썬 코드'는 모르기 때문입니다. 이 서버는 바로 그 문제를 해결해 줍니다.

**주요 활용 사례:**
- 수동으로 grep을 쓰지 않고 대규모 레거시 코드베이스 조사
- 여러 환경 간의 설정 파일 비교
- 기존 구조를 존중하는 방식의 리팩토링 생성
- 존재조차 잊고 있었던 불필요한 파일(dead files) 탐색

500개가 넘는 파일을 가진 대형 프로젝트를 다루고 있다면, 이 서버는 필수적입니다. 이는 단순한 파일 접근이 아니라 '구조화된 파일 접근'을 가능하게 합니다.

## 2. Git MCP Server — CLI가 아닌 지식 그래프로서의 Git

저는 Git을 좋아하지만, 기본적인 명령어를 넘어서는 복잡한 작업에는 질색합니다. `git log`는 강력하지만 불친절합니다. Git MCP 서버는 저장소의 이력을 단순히 훑어보는 대상이 아니라 논리적으로 추론할 수 있는 지식 그래프(Knowledge Graph)로 바꿔줍니다.

이 도구 덕분에 저는 "무엇이 깨졌는가?"라는 질문 대신 "**이 코드가 왜 존재하는가?**"라는 훨씬 더 본질적인 질문을 던지기 시작했습니다.

**실제 활용 방식:**
- 기술 부채가 언제, 누구에 의해 도입되었는지 추적
- 특정 워크어라운드(workaround) 코드가 제거되지 않은 이유 파악
- 단순한 느낌이 아닌, 과거의 맥락을 바탕으로 PR(Pull Request) 리뷰
- 추측에 의존하지 않고 팀원들에게 의사결정 근거 설명

Git은 단순한 버전 관리 도구가 아니라 '의도'를 기록한 타임머신입니다. MCP는 이를 우리가 실제로 활용할 수 있게 만들어 줍니다. 이 도구가 나쁜 커밋 메시지까지 고쳐주지는 않겠지만, 미처 발견하지 못했던 패턴을 표면 위로 드러내 줍니다.

## 3. HTTP / API MCP Server — 더 이상 직접 작성할 필요 없는 연결 코드

모든 파이썬 개발자는 결국 어느 시점에서 'API 관리자'가 됩니다. 직접 의도한 것은 아니지만, 수동으로 엔드포인트를 테스트하고, curl 명령어를 복사해서 붙여넣고, 유지보수하고 싶지도 않은 얇은 래퍼(wrapper) 코드를 작성하게 됩니다. HTTP MCP 서버는 이런 단계를 끝내주었습니다.

대부분의 버그는 경계면(boundaries)에서 발생하기 때문에 이 서버는 매우 중요합니다.

**이 서버가 제공하는 이점:**
- 모킹(mock) 데이터의 괴리 없이 실제 API 응답 확인
- 라이브 시스템에 대해 가설 검증
- 운영 환경에 배포하기 전 통합 단계 디버깅
- 스테이징과 운영 환경 간의 동작 차이 비교

무언가 잘못되었다고 느낄 때 코드를 파헤치는 대신, "API가 실제로 무엇을 반환하는가?", "지난주와 지금의 차이는 무엇인가?"를 즉시 물어볼 수 있습니다. 이러한 명확성은 수많은 시간을 절약해 줍니다. 이는 테스트를 건너뛰기 위한 수단이 아니라, 추측을 멈추기 위한 수단입니다.

## 결론: 개발자가 놓치기 쉬운 패턴

위의 서버들의 공통점은 단순히 코드를 생성해 주는 것이 아니라 **'불확실성'을 줄여준다는 것**입니다. 이것이 진정한 생산성 증폭기입니다. 대부분의 개발자는 더 빨리 타이핑하게 해주는 도구를 찾지만, 저는 더 빨리 결정하게 도와주는 도구를 원합니다.

**도구 도입 전 명심할 점:**
1. 사용하는 서버의 수를 줄이고, 대신 깊게 사용하세요.
2. 특정 스크립트를 대체하지 못하는 도구라면 의심해 보세요.
3. 설정만 복잡하게 만들고 명확성을 주지 못한다면 삭제하세요.
4. 도구는 정신적인 부담(mental stack)을 줄여야지, 확장해서는 안 됩니다.

파이썬이 어려운 것이 아니라면, 그것은 아마 여러분의 도구가 충분히 주관을 가지고(opinionated) 여러분을 돕지 못하고 있기 때문일 것입니다. 작게 시작해 보세요. 하나의 MCP 서버를 골라 일주일 동안 사용해 보며, 그동안 수동으로 해왔던 일들 중 무엇을 멈추게 되었는지 관찰해 보시기 바랍니다. 그것이 바로 생산성의 신호입니다.
