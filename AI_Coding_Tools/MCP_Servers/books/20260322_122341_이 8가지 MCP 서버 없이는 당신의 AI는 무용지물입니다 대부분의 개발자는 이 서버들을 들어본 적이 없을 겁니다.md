## #8 — Vercel

어떤 애플리케이션을 배포했는데 실패했다고 가정해 봅시다. 이제 에디터(editor)를 떠나 Vercel을 열고, 배포 목록을 확인하고, 로그를 스크롤하며 실패 로그를 검색합니다.

[Vercel의 MCP 서버 사용하기]
Vercel MCP는 팀, 프로젝트, 배포 관리는 물론 문서 검색 도구를 제공합니다.
vercel.com

Vercel MCP를 사용하면, 당신의 AI가 배포 환경에 직접 연결됩니다. 당신은 그저 프롬프트(prompt)만 주면 됩니다:

“최신 배포 로그를 가져와 줘. 빌드가 왜 실패했지?”

그러면 AI가 로그를 가져와 정확한 오류를 찾아내고 수정 사항을 제안합니다. 당신은 작업 흐름을 유지할 수 있습니다.

이 MCP의 취미(Hobby) 등급은 무료입니다. 프로(Pro) 등급은 사용자당 월 20달러부터 시작합니다.

## #7 — Docker

이것은 매우 흔한 문제를 해결합니다: 내 컴퓨터에서는 작동하지만 CI(Continuous Integration)에서는 작동하지 않는 문제죠. 그래서 이제 환경 변수를 비교하고, 기본 이미지를 확인하고, 추측해야 합니다.

[발표: Docker MCP | Docker]
자동 MCP 툴킷 실행을 위해 Docker Desktop 4.48 이상 버전 필요
www.docker.com

Docker MCP를 사용하면 AI가 컨테이너(container)를 직접 검사할 수 있습니다. 프롬프트만 주면 됩니다:

“이 컨테이너는 왜 CI에서는 실패하고 로컬(locally)에서는 실패하지 않지?”

그러면 Dockerfile 빌드 로그, 레이어 캐시, 환경 차이점을 확인할 것입니다. 이제 더 이상 추측할 필요가 없습니다. 컨테이너를 실행한다면, 이 기능은 매우 빠르게 막강해집니다.

대부분의 Docker MCP 설정은 자체 호스팅(self-hosted)이거나 오픈 소스(open source)이며, MCP 자체에 대한 직접적인 구독 비용은 없습니다. 인프라 비용만 지불하면 됩니다. 컨테이너 워크플로(workflow)의 경우, 이는 엄청난 생산성 향상을 가져옵니다.

## #6 — Apify

웹사이트에서 데이터가 필요하다고 말할 때마다, 당신은 기본적으로 웹 스크레이퍼(web scraper)를 구축하는 것이고, 스크레이퍼는 고장 나기 마련입니다. 셀렉터(selector)가 변경되었거나 레이아웃이 바뀌었을 수 있죠.

[Apify MCP 서버]
액터(Actor)와 도구를 사용하여 Apify MCP 서버를 구성하여 MCP 클라이언트(client)와 원활하게 통합하세요.
mcp.apify.com

Apify는 이 모든 번거로움을 제거합니다. 2,000개 이상의 액터에 대해 유지보수되는 스크레이퍼에 접근할 수 있습니다. 프롬프트만 주면 됩니다:

“Google 지도에서 (당신의 도시 이름)에 있는 모든 IT 기업을 찾아줘. 이름, 웹사이트, 전화번호를 CSV(Comma Separated Values)로 내보내 줘. CSV 파일로 저장해 줘.”

끝입니다.

매월 5달러의 무료 크레딧이 제공됩니다. 유료 요금제는 약 49달러부터 시작합니다. 만약 당신의 앱이 외부 데이터에 의존한다면, 이것은 활용해야 할 중요한 MCP입니다.

## #5 — Playwright

이것은 개인적으로 가장 좋아하는 것입니다. Apify는 데이터를 수집합니다. Playwright는 액션(action)을 수행합니다.

[GitHub - microsoft/playwright-mcp: Playwright MCP 서버]
Playwright MCP 서버. GitHub 계정을 만들어 microsoft/playwright-mcp 개발에 기여하세요.
github.com

이것은 당신의 AI에 제어 가능한 브라우저(browser)를 제공합니다. 클릭, 입력, 스크롤, 업로드, 다운로드. 레거시(legacy) 대시보드에 로그인하고, 2단계 인증을 처리하며, 일일 보고서를 다운로드해야 할 때도 가능합니다.

로컬(locally)에서 실행됩니다. CI에서도 실행됩니다. 무료입니다. 실제 예시와 전체 설정(setups)을 보려면 (GitHub를 참조하세요).

## #4 — Ref

여기에 문제가 있습니다. 당신이 AI에게 특정 라이브러리(library)에 대해 물어보면, AI는 전체 문서, 수천 줄의 코드, 히스토리, 튜토리얼(tutorial), 예제 등 이 모든 것에서 단 하나의 함수를 찾기 위해 검색합니다.

[Ref.]
당신의 에이전트(agent)를 위한 문서. Ref.는 6개의 리포지토리(repository)를 사용할 수 있습니다. GitHub에서 그들의 코드를 팔로우하세요.
github.com

Ref는 이 문제를 해결합니다. AI가 문서에 매우 구체적인 질문을 할 수 있게 합니다. 따라서 다음과 같이 하는 대신:

“Playwright 문서를 읽어 줘”

당신의 프롬프트는 다음과 같을 것입니다:

“page.waitForSelector의 매개변수는 무엇인가요?”

그러면 해당 함수만, 즉 시그니처(signature)와 설명만 반환합니다. 불필요한 정보는 없습니다.

공개 문서(public documentation)는 무료입니다. 비공개 문서(private docs)를 인덱싱(indexing)하고 싶다면 유료 요금제를 사용하세요. 대규모 라이브러리를 사용한다면, 이것은 당신의 AI가 집중하고 효율적으로 작동하도록 돕습니다.

## #3 — File System MCP

이것이 없다면, 당신의 AI는 당신이 붙여넣은 코드만 볼 수 있습니다. 그래서 하나의 파일을 붙여넣으면 변경을 제안하지만, 해당 함수가 어디에 사용되는지, 무엇이 그에 의존하는지, 어떤 파일이 임포트(import)되었는지 알지 못합니다. AI는 고립된 상태에서 작동하는 것입니다.

[servers/src/filesystem at main · modelcontextprotocol/servers]
모델 컨텍스트 프로토콜(Model Context Protocol) 서버. GitHub 계정을 만들어 modelcontextprotocol/servers 개발에 기여하세요.
github.com

파일 시스템 MCP(File System MCP)를 사용하면, AI가 당신의 전체 프로젝트를 볼 수 있습니다. 프롬프트만 주면 됩니다:

“사용되지 않는 컴포넌트(components)를 찾아 줘.”

그러면 전체 src 폴더를 스캔합니다. 이제 AI는 단편적인 코드 조각(snippets)이 아니라 관계를 이해하게 됩니다.

이것은 무료입니다. 로컬(locally)에서 실행됩니다. 그리고 프로젝트 수준의 변경을 원한다면, 이것이 바로 그것을 가능하게 합니다.

## #2 — Exa

Google은 인간, 광고, SEO(Search Engine Optimization), 클릭을 위해 만들어졌습니다. Exa는 AI를 위해 만들어졌습니다. 의미를 기반으로 검색하고 구조화된 결과(structured results)를 반환합니다.

[Exa MCP 서버 - Claude, Cursor & VS Code를 위한 AI 웹 검색]
당신의 AI 어시스턴트(assistant)에 웹 검색 기능을 부여하세요. Claude Desktop, Cursor, VS Code 및 기타 AI 도구를 Exa에 연결하여...
exa.ai

프롬프트만 주면 됩니다:

“2026년에 React에서 발생한 이 경쟁 조건(race condition)에 대한 논의를 찾아 줘.”

그러면 GitHub 이슈, 관련 스레드(thread), 그리고 실제 수정 사항을 가져옵니다.

1,000회 무료 요청. 유료 요금제는 월 약 50달러부터 시작합니다. 빠르게 변화하는 스택(stacks)으로 작업한다면, Exa가 당신의 작업을 더욱 빠르게 만들 것입니다.

## #1 — Sequential Thinking

모델은 빠른 답변을 좋아합니다. 하지만 때로는 너무 빠릅니다.

[servers/src/sequentialthinking at main · modelcontextprotocol/servers]
모델 컨텍스트 프로토콜(Model Context Protocol) 서버. GitHub 계정을 만들어 modelcontextprotocol/servers 개발에 기여하세요.
github.com

순차적 사고(Sequential Thinking)는 모델이 먼저 계획한 다음 답변하도록 돕습니다. 문제를 분해하고, 가정을 나열하며, 단계를 요약한 다음 코드를 생성합니다.

이것은 아키텍처(architecture), 복잡한 디버깅(debugging) 또는 시스템 설계(system design)에 사용하세요. 간단한 스크립트(scripts)를 위한 것이 아닙니다. 진지한 로직(logic)을 위한 것입니다.

이것은 무료입니다. 오픈 소스(open source)입니다.

순차적 사고는 추측과 엔지니어링(engineering)의 차이입니다.

---

### 컨텍스트(Context) 붙여넣기를 멈추세요. 시스템(System)을 연결하기 시작하세요.

이 도구들을 함께 연결하면, 컨텍스트를 붙여넣는 대신 시스템을 연결하기 시작합니다.

그것이 바로 변화입니다. 채팅에서 구축으로. 고립된 코드 조각에서 완전한 시스템 인식으로. 추측에서 엔지니어링으로.

MCP는 기능이 아닙니다. 그것은 말하는 AI와 실제로 작동하는 AI의 차이입니다.
