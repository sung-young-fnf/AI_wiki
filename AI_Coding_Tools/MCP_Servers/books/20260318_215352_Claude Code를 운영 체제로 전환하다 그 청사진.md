### 요약 (TL;DR)

Claude Code는 터미널이 있는 챗봇이 아닙니다. 이는 프로그래밍 가능한 플랫폼입니다. 이 청사진은 Claude Code를 스킬(skills), 훅(hooks), 에이전트(agents), 그리고 MCP 서버(MCP servers)를 포함하는 6계층 운영 체제로 전환하며, 제로 트러스트(zero-trust) 보안 모델 하에서 작동합니다. 시스템은 다음을 포함합니다:

- **78개의 권한 규칙(허용 40개 + 거부 38개)**: 모든 명령, 모든 쓰기, 모든 MCP 접근이 명시적으로 허용되거나 차단되는 제로 트러스트(zero-trust) 모델. (Anthropic Documentation 참조)
- **17개의 라이프사이클(lifecycle) 이벤트(17 hooks)**: 실행 전후의 모든 작업을 가로채는 Bash 스크립트(또는 HTTP 엔드포인트, LLM 프롬프트). 비밀 감지, 파괴적인 명령 차단, 자동 포맷팅, 자동 실패 로깅. 순수한 결정론(determinism).
- **10가지 범주로 구성된 32개의 스킬(skills)**: 환각 방지부터 생성형 AI까지, 각 스킬은 공통의 5단계 아키텍처 패턴(ASSESS, ANALYZE, PLAN, EXECUTE, VALIDATE)을 따릅니다. 6개의 전용 AI/ML 스킬(NLP, vision, MLOps, RL, generative, agents) 포함.
- **5개 도메인에 걸친 10개의 전문 에이전트(agents)**: 일반적인 비서가 아닌, 전용 도구, 모델 및 색상을 가진 자율적인 부서. 코드 에이전트는 더 모듈화된 플러그인(plugins)으로 대체되었습니다.
- **통합 계층으로서의 6개의 MCP 서버**: 실시간 문서화, Git(깃) 작업, 웹 읽기, Jupyter(주피터) 노트북. Claude를 외부 세계와 연결하는 신경계.
- **영구 메모리(persistent memory)**: `memory/` 디렉터리를 통해 Claude는 세션 간 학습 내용을 유지할 수 있습니다. 확인된 패턴, 아키텍처 결정, 워크플로 선호도 등.
- **모든 것은 오픈 소스(open-source)**: `claude-code-blueprint`는 이 글에 설명된 `CLAUDE.md`, 스킬, 에이전트, 훅, 규칙, 명령 및 설정을 포함합니다. 클론(clone)하고, 적용하고, 반복하세요.

핵심 질문: Claude Code는 터미널이 있는 챗봇이 아닙니다. 프로그래밍 가능한 플랫폼입니다. 대부분의 사용자는 터미널만 활용합니다.

3개월 전, Claude는 망설임 없이 내게 거짓말을 했습니다.

나는 FastAPI 애플리케이션 작업을 하고 있었습니다. 엔드포인트에 조건부 종속성(conditional dependency)을 주입해야 했습니다. 나는 Claude Code에 `Depends()`의 정확한 시그니처(signature)를 물었습니다. 답변은 2초도 안 되어 도착했고, 마치 외과의사가 시술을 설명하는 것처럼 확신에 차 있었습니다: `Depends(dependency, use_cache=True, allow_override=False)`. 세 개의 매개변수. 상세한 설명. 사용 예시. 모든 것이 명확하고, 구조화되어 있으며, 설득력이 있었습니다.

나는 그 세 개의 매개변수에 맞춰 코드를 리팩터링(refactor)했습니다. `allow_override`를 종속성 로직에 통합하는 데 40분을 보냈습니다. 그리고 테스트를 실행했습니다.

모든 것이 망가졌습니다.

`Depends()`는 하나의 인자(argument)만 받습니다. 종속성 그 자체입니다. 그게 전부입니다. 다른 두 매개변수는 존재하지 않습니다. 존재한 적도 없습니다. Claude는 피타고라스 정리를 인용하는 것과 같은 확신으로 아무것도 없는 상태에서 그것들을 날조했습니다.

이것이 LLM(대규모 언어 모델)의 근본적인 문제입니다: 그들은 자신이 무엇을 모르는지 모릅니다. 구두 시험에서 부정행위를 하는 학생은 질문을 받으면 약간의 불편함을 보일 것입니다. 환각을 일으키는 LLM은 절대적인 확신을 가지고 당신의 눈을 응시합니다. 경고 신호도, 어조의 변화도, "내 생각에는"이라는 말도 없습니다. 단지 검증된 사실처럼 전달되는 틀린 답변일 뿐입니다.

40분간의 작업. 지어낸 함수 시그니처 때문에 허비되었습니다.

그날, 나는 그런 일이 다시는 일어나지 않도록 결심했습니다. 내 기억에 의존하거나, 모델에 의존하는 것이 아니라, 환각 발생을 구조적으로 어렵게 만드는 시스템을 구축함으로써 말입니다.

그 시스템이 바로 여기서 설명할 내용입니다. 튜토리얼도 아니고, 팁 목록도 아닙니다. 모든 계층이 기능을 가지고 있고, 오류가 코드에 도달하기 전에 가로채지며, 모델이 결코 최후의 방어선이 아닌 Claude Code 환경의 완전한 아키텍처입니다.

### Claude Code 아키텍처: 6계층 시스템

자세히 들어가기 전에 전체 그림을 살펴보겠습니다. 시스템은 각각 고유한 책임을 가진 6개의 계층으로 나뉩니다:

```
+---------------------------------------------------+
|                  LAYER 6: MCP                     |
|        6개 서버 (context7, fetch, git,           |
|         huggingface, jupyter, ide)                |
+---------------------------------------------------+
|               LAYER 5: SECURITY                   |
|     40개 허용 + 38개 거부 + 17개 라이프사이클 훅    |
|     + 정규식 비밀 + 실패 시 차단(fail-closed) 트랩  |
+---------------------------------------------------+
|               LAYER 4: AGENTS                     |
|        10개의 전문 에이전트 (5개 도메인)          |
|     다중 에이전트 브레인스토밍 (4개 병렬)          |
+---------------------------------------------------+
|               LAYER 3: SKILLS                     |
|     32개 스킬 (10개 카테고리) 자동 로드           |
|     공통 패턴: 5단계 AAPEV                        |
+---------------------------------------------------+
|               LAYER 2: MEMORY                     |
|   영구 memory/ + 프로젝트 수준 rules/             |
|   세션 간 학습 내용                              |
+---------------------------------------------------+
|           LAYER 1: CLAUDE.md (KERNEL)             |
|   환각 방지 + 신뢰도 수준 +                        |
|   툴체인 + 코드 표준 + /compact 규칙              |
+---------------------------------------------------+
```

6계층 아키텍처는 `CLAUDE.md`를 커널(kernel, 1계층)로 시작하여, 영구 메모리(persistent memory, 2계층), 컨텍스트(context) 인식 스킬(3계층), 전문 에이전트(4계층), 훅을 포함한 제로 트러스트(zero-trust) 보안(5계층), 그리고 MCP 서버 통합(6계층)을 추가합니다. 각 계층은 이전 계층을 기반으로 구축됩니다. 어떤 계층을 제거하더라도 시스템의 성능은 저하되지만, 시스템이 완전히 망가지지는 않습니다.

운영 체제에 대한 비유는 단순한 수사가 아닙니다. `CLAUDE.md`는 커널입니다: 이는 근본적인 행동을 정의합니다. 영구 메모리는 파일 시스템입니다: 이는 세션 간 학습 내용을 유지합니다. 스킬은 공유 라이브러리입니다: 컨텍스트가 필요할 때 온디맨드(on-demand)로 로드됩니다. 에이전트는 프로세스입니다: 전용 리소스를 가진 자율적인 엔터티입니다. 보안은 권한 계층입니다: 누가, 어디서, 어떤 조건에서 무엇을 할 수 있는지를 정의합니다. 그리고 MCP 서버는 주변 장치 드라이버입니다: 외부 세계와의 인터페이스입니다.

이것은 강제된 비유가 아닙니다. 시스템이 실제로 작동하는 방식입니다.

### `CLAUDE.md`를 커널(Kernel)로

대부분의 가이드에서는 `CLAUDE.md`에 "선호도"를 넣으라고 합니다. 코드 스타일. 좋아하는 프레임워크. TypeScript(타입스크립트)를 사용하라는 리마인더(reminder) 정도.

그것만으로는 충분하지 않습니다.

`CLAUDE.md`는 선호도 파일이 아닙니다. 시스템의 커널(kernel)입니다. 모든 세션, 모든 도구, 모든 응답의 기본 동작을 정의합니다. `CLAUDE.md`에 없는 모든 것은 선택 사항입니다. 그 안에 있는 모든 것은 필수 사항입니다.

내 `CLAUDE.md`에는 다음이 포함됩니다:

#### 환각 방지 프로토콜

이것이 첫 번째 규칙입니다. FastAPI 사건이 발생한 날 작성되었습니다. Claude가 기술적인 질문에 답변하기 전에 따라야 할 의사 결정 트리입니다:

답변하기 전에 (BEFORE answering):
+-- API/라이브러리 질문 --> Context7 우선(FIRST)
+-- 최근 사실/뉴스   --> WebSearch 우선(FIRST)
+-- 파일 내용         --> Read 우선(FIRST)
+-- 불확실            --> "확인이 필요합니다" + 도구 사용

절대 (NEVER):
+-- 함수 시그니처를 날조하지 마십시오
+-- 라이브러리 버전을 추측하지 마십시오
+-- 검증 없이 API 동작을 가정하지 마십시오
+-- 인용이나 출처를 날조하지 마십시오

각 브랜치(branch)는 직접적인 명령입니다. 제안이 아닙니다. "가급적"이 아닙니다. Claude가 API에 대한 질문에 답변해야 한다면, 먼저 Context7(실시간 문서를 제공하는 MCP 서버)에 질의해야 합니다. 확인할 수 없다면, 그렇게 말해야 합니다.

이것은 단순한 장치가 아닙니다. 운영상의 제약 조건입니다. Claude가 "함수는 3개의 매개변수를 받는다"고 중간(MEDIUM) 신뢰도 수준으로 답변할 때, 나는 확인해야 한다는 것을 압니다. 높은(HIGH) 신뢰도와 Context7 링크와 함께 답변할 때는 신뢰할 수 있습니다. 이 둘의 차이는 40분의 리팩터링 시간을 허비하는 것과 같습니다.

#### `/compact` 시 보존

Claude Code에는 컨텍스트(context) 제한이 있습니다. 대화가 너무 길어지면 `/compact` 명령이 컨텍스트를 압축합니다. 문제는 명시적인 지시 없이는 압축이 중요한 정보를 잃어버린다는 것입니다. 내 `CLAUDE.md`는 무엇이 보존되어야 하는지를 명시합니다:

컨텍스트가 압축될 때, 항상 다음을 보존하십시오:
- 경로와 함께 수정된 파일 목록
- 현재 Git 브랜치와 커밋되지 않은 변경 사항
- 보류 중인 작업 및 TODO 항목
- 테스트 결과 및 실패
- 세션 중 내려진 주요 아키텍처 결정

이 규칙이 없으면 모든 압축은 부분적인 전두엽 절제술(lobotomy)과 같습니다. 이 규칙이 있으면 구조화된 요약이 됩니다.

### 영구 메모리 (Persistent Memory)

Claude Code에는 근본적인 문제가 있습니다: 모든 세션이 백지상태에서 시작됩니다. 아키텍처, 선호도, 규칙에 대해 한 시간 동안 설명해도 다음 날이면 모두 잊혀집니다. 컨텍스트 창은 휘발성입니다.

해결책은 `memory/` 디렉터리입니다. Claude Code는 `~/.claude/projects/<project>/memory/`에 파일을 쓰고 읽을 수 있습니다. 이 파일들은 세션 간에 지속됩니다. `MEMORY.md`는 모든 대화에 자동으로 로드됩니다. 이는 영구적인 작업 메모리입니다. (메모리 문서 참조)

```
memory/
├── MEMORY.md          <-- 자동 로드됨 (최대 200줄)
├── patterns.md        <-- 프로젝트에서 확인된 패턴
├── debugging.md       <-- 반복되는 문제에 대한 해결책
└── architecture.md    <-- 구조적 결정
```

`MEMORY.md`는 인덱스(index)입니다. 간결하게 유지되어야 합니다(200줄 이후는 잘립니다). 위성 파일(satellite files)에는 세부 정보가 포함됩니다. Claude는 여러 세션에 걸쳐 이 파일들을 업데이트합니다. 일시적인 메모가 아니라, 여러 상호 작용을 통해 검증된 확인된 학습 내용입니다.

내가 여기에 저장하는 것:

- 프로젝트에서 검증된 아키텍처 패턴(가설이 아님)
- 확인된 워크플로 선호도("항상 uv 사용", "자동 커밋(auto-commit) 금지")
- 중요 경로 및 프로젝트 구조
- 반복되는 문제에 대한 해결책

내가 저장하지 않는 것: 현재 세션 컨텍스트, 미검증 정보, 추측성 결론. 메모리는 임시 작업 공간이 아닌 사실의 원장(ledger)입니다.

`/compact`와의 차이점: `compact`는 세션 내에서 컨텍스트를 보존합니다. 메모리는 세션 간 학습 내용을 보존합니다. 하나는 RAM이고, 다른 하나는 영구 저장 장치(persistent storage)입니다.

### 경로 기반 규칙 (Path-Based Rules)

`CLAUDE.md`는 전역적입니다. 모든 것에 적용됩니다. 하지만 Python(파이썬)과 TypeScript(타입스크립트)를 사용하는 프로젝트는 동일한 규칙을 가질 수 없습니다. "ruff 사용"과 "prettier 사용"을 같은 파일에 넣으면 Claude는 무엇이 어디에 적용되는지 분류해야 합니다. 이는 결정론(determinism)이 가능한 영역에서 확률론적인 작업입니다.

`~/.claude/rules/` 디렉터리가 이를 해결합니다. 각 파일에는 로드 시점을 정의하는 `paths:` 프런트매터(frontmatter)가 있습니다:

```
~/.claude/rules/
├── python.md         <-- *.py, pyproject.toml (40줄)
└── typescript.md     <-- *.ts, *.tsx, package.json (38줄)
```

`python.md`에는 `uv`, `ruff`, `pytest`, 타입 힌트(type hints) 및 비동기 패턴(async patterns)이 포함됩니다. `typescript.md`에는 `prettier`, `vitest`, 엄격 모드(strict mode), `as` 대신 `satisfies` 사용, `enum` 사용 금지가 포함됩니다. `paths:` 프런트매터는 글로브(glob)처럼 작동합니다: 편집 중인 파일이 패턴과 일치하면 규칙이 로드됩니다.

솔직한 한마디: 이 글을 쓰는 시점에는 경로 범위 지정(path-scoping)에 알려진 버그가 있습니다. 모든 `rules/` 파일은 `paths:`와 관계없이 세션 시작 시 로드됩니다. 실제로는 78줄의 규칙이 항상 로드됩니다. 치명적인 것은 아닙니다: 78줄은 200K 컨텍스트 창에서 약 100토큰입니다. 그리고 버그가 수정되면, 내 파일을 변경하지 않아도 범위 지정이 자동으로 작동할 것입니다.

모든 것을 `CLAUDE.md`에 중앙 집중화하는 것보다 장점: 내 커널은 109줄을 유지합니다. 언어별 규칙은 커널과 함께 존재하며 주요 파일의 가독성을 해치지 않습니다.

#### 스킬의 구조

Claude Code 스킬이란 무엇일까요? Claude Code 스킬은 `~/.claude/skills/`에 YAML 프런트매터가 포함된 마크다운 파일이며, 대화 컨텍스트가 스킬의 설명과 일치할 때 Claude가 자동으로 로드합니다. API, SDK, 빌드 단계 없이 모델이 지시처럼 따르는 평범한 텍스트입니다.

모든 스킬은 동일한 구조를 따릅니다. 다음은 환각 방지 스킬의 프런트매터입니다:

```yaml
---
name: anti-hallucination
description: |-
  API 시그니처, 라이브러리 메서드, 사실적 주장을 Context7과 WebSearch를 사용하여
  답변하기 전에 검증합니다.
  사용자가 함수 매개변수, 메서드 시그니처, 라이브러리 버전, API 동작에 대해 질문할 때
  반드시 사용해야 합니다.
allowed-tools:
  - mcp__context7__resolve-library-id
  - mcp__context7__get-library-docs
  - WebSearch
  - Read
  - Grep
---
```

설명이 핵심입니다. Claude는 이를 사용하여 스킬을 로드할지 여부를 결정합니다. `allowed-tools` 필드는 화이트리스트(whitelist)입니다: 스킬은 나열된 도구만 사용할 수 있습니다. 프런트매터 이후의 내용은 행동을 정의하며, 완전한 의사 결정 트리를 포함합니다:

질문 유형?
+-- API/라이브러리 시그니처  --> Context7 우선(FIRST), 그 다음 답변
+-- 최근 이벤트/사실      --> WebSearch 우선(FIRST)
+-- 파일 내용           --> Read 도구 우선(FIRST)
+-- 코드 동작           --> Read + trace 우선(FIRST)
+-- 역사적 사실         --> 학습 데이터 사용 가능
+-- 확인할 수 없음        --> "모릅니다"라고 명시

#### 공통 아키텍처 패턴

모든 32개의 스킬은 5단계 패턴을 따릅니다:

1단계: ASSESS(평가) / CLARIFY(명확화)    --> 문제 이해
2단계: ANALYZE(분석) / RESEARCH(조사)  --> 행동 전 조사
3단계: PLAN(계획) / DESIGN(설계)       --> 계획 제안
4단계: EXECUTE(실행) / IMPLEMENT(구현) --> 변경 사항 적용
5단계: VALIDATE(검증) / VERIFY(확인)   --> 결과 테스트

이 패턴은 모델에 인지적 규율(cognitive discipline)을 부여합니다. 구현(4단계)으로 바로 넘어가는 대신, Claude는 먼저 이해하고(1단계), 연구하고(2단계), 계획해야 합니다(3단계). 5단계는 검증으로 루프(loop)를 닫습니다. 선임 엔지니어에게 기대할 수 있는 것과 동일한 추론 방식입니다.

#### 다중 에이전트 브레인스토밍 (Multi-Agent Brainstorm)

`brainstorm` 스킬은 아마도 시스템에서 가장 흥미로운 부분일 것입니다. 이는 여러 에이전트를 병렬로 조율하여 다양한 각도에서 문제를 공격합니다:

1단계: CLARIFY(명확화)    --> 주제 이해 (2-3가지 질문)
2단계: MAP(매핑)        --> 전문가 에이전트 선택
3단계: EXPLORE(탐색)    --> 에이전트 병렬 생성
4단계: SYNTHESIZE(종합) --> 결합: 합의점, 이견, 권고

브레인스토밍 프롬프트(prompt)를 받는 각 에이전트는 다음을 수행해야 합니다:

- 해당 전문 분야에서 2~3가지 접근 방식을 제안
- 각 접근 방식의 장단점과 위험성 제시
- 출처를 통해 각 권고 사항 정당화
- 신뢰도 수준 선언
- 확인할 수 없는 사항 플래그(flag) 지정

합성(synthesis)은 합의점(강한 신호), 이견(결정 지점)을 식별하고, 논거를 갖춘 권고 사항을 생성합니다. 이것은 투표가 아닙니다. 구조화된 숙고(deliberation)입니다.

### 제로 트러스트 권한 및 훅 (Zero-Trust Permissions and Hooks)

이것이 시스템이 진지해지는 부분입니다.

Claude Code는 셸(shell) 명령을 실행할 수 있습니다. 파일을 쓸 수 있습니다. 프로덕션(production) 코드를 수정할 수 있습니다. 기본 신뢰(Default trust)는 언젠가 문제를 일으키기 전까지는 유효한 전략입니다. 내 시스템은 반대 원칙에서 시작합니다: 명시적으로 허용되지 않는 한 모든 것이 금지됩니다.

#### 레벨 1: 정적 권한 (Static Permissions)

`settings.json` 파일은 78개의 권한 규칙을 정의합니다. 40개의 허용. 38개의 거부. (권한 문서 참조) 다음은 발췌 내용입니다:

허용됨 (Allowed):

```json
"Bash(git:*)", "Bash(uv:*)", "Bash(npm run:*)",
"Bash(python:*)", "Bash(cargo:*)", "Bash(docker:*)",
"Bash(ruff:*)", "Bash(gh:*)",
"mcp__context7__*", "mcp__fetch__*", "mcp__git__*"
```

거부됨 (Denied):

```json
"Bash(chmod 777:*)", "Bash(sudo:*)", "Bash(:|:*)",
"Bash(dd if:*)", "Bash(eval :*)", "Bash(nc :*)",
"Write(~/.ssh/*)", "Write(*.pem)", "Write(*secret*)",
"Edit(.env.production)", "Edit(*credentials*)"
```

논리는 간단합니다: Claude는 Git, Python, Cargo, Docker를 실행할 수 있습니다. `sudo`, `chmod 777`, `eval`을 실행할 수 없으며, `~/.ssh/` 또는 이름에 "secret"이 포함된 파일에 쓸 수 없습니다. 각 권한은 명시적인 결정입니다.

#### 레벨 2: PreToolUse 훅 (PreToolUse Hooks)

정적 권한만으로는 충분하지 않습니다. `Bash(rm:*)`가 허용된다면, 전체 명령이 `rm -rf /`일 경우 위험할 것입니다. PreToolUse 훅은 실행 전에 모든 명령의 내용을 검사합니다.

`bash-guard`는 파괴적인 패턴을 가로챕니다:

```bash
#!/usr/bin/env bash
set -euo pipefail

# 실패 시 차단(Fail-closed): 오류 발생 시 기본적으로 차단
trap 'echo "{\"hookSpecificOutput\":{...\"deny\"...}}"' ERR
input=$(cat)
command=$(echo "$input" | jq -r '.tool_input.command // empty')
# Sudo/Su: 경고뿐 아니라 차단
if echo "$command" | grep -qE \
  '(^|;|&&)\s*(sudo|su |doas |pkexec)'; then
    # --> 거부(DENY): 권한 상승 차단됨
    exit 0
fi
# 파괴적인 패턴
# 다음을 감지: rm -rf /, 포크 폭탄(fork bombs), chmod -R 777,
#          curl 파이프 Bash, mkfs, dd to device
if echo "$command" | grep -qE "$dangerous_patterns"; then
    # --> 거부(DENY): 위험한 명령 감지됨
    exit 0
fi
# 난독화(Obfuscation): eval $var, base64 decode pipe shell
if echo "$command" | grep -qE "$obfuscation_patterns"; then
    # --> 거부(DENY): 간접 실행 감지됨
    exit 0
fi
```
하나의 스크립트에 세 가지 보호 기능: 권한 상승 차단, 파괴적인 패턴 감지(포크 폭탄, `rm -rf`, Bash로 파이프 연결), 그리고 난독화 감지(`eval $var`, `base64 decode`를 셸로 연결).

`write-guard`는 민감한 파일을 보호하고 콘텐츠에서 비밀 정보를 스캔합니다:

```bash
# === 민감한 파일 보호 ===
# 다음 파일에 대한 쓰기를 차단: .env, credentials, id_rsa,
# .pem, .key, .ssh/, .npmrc, .pypirc, .aws/, kubeconfig...
protected='\.env($|\.)|credentials|id_rsa|\.pem$|\.key$|\.ssh/'
if echo "$file_path" | grep -qiE "$protected"; then
    # --> 거부(DENY): 보호된 파일
fi
# === 작성된 콘텐츠의 비밀 정보 감지 ===
# 15개 이상의 키 및 토큰 형식을 포함하는 정규식:
#   - OpenAI 키 (sk-...)
#   - AWS 키 (AKIA...)
#   - GitHub 토큰 (ghp_..., gho_..., github_pat_...)
#   - GitLab, Slack, Google, Stripe, SendGrid 토큰
#   - SSH/TLS 개인 키 (BEGIN ... KEY)
#   - DB 연결 URL (인라인 비밀번호 포함)
#   - JWT (eyJ...eyJ...sig 형식)
#
if echo "$content" | grep -qE "$secret_patterns"; then
    # --> 거부(DENY): 잠재적인 비밀 정보 감지됨
fi
```

이 정규식(regex)은 가장 흔한 우발적 유출을 커버합니다: 주요 클라우드 및 SaaS 제공업체의 API 키, 인증서, 인증 토큰, 그리고 일반 텍스트 비밀번호가 포함된 데이터베이스 연결. 완벽한 목록은 아니지만, 가장 빈번한 사례에는 충분합니다.

#### 레벨 3: 실패 시 차단 패턴 (The Fail-Closed Pattern)

모든 훅에서 가장 중요한 단 한 줄은 아마도 이것일 것입니다:

```bash
trap 'echo "{\"hookSpecificOutput\":{
  \"permissionDecision\":\"deny\",
  \"permissionDecisionReason\":\"Hook error - fail-closed\"}}"' ERR
```

훅 자체가 충돌하거나, `jq`가 입력을 파싱(parse)하지 못하거나, 변수가 비어 있거나, 무엇이든 잘못되면: 기본 응답은 거부입니다. 허용이 아닙니다. "지켜보자"가 아닙니다. 거부입니다. 이것이 실패 시 차단(fail-closed) 원칙입니다: 의심스러울 때에는 차단합니다.

#### Bash를 넘어: HTTP 및 프롬프트 훅 (HTTP and Prompt Hooks)

나의 모든 훅은 Bash 스크립트입니다. 하지만 더 이상 유일한 옵션은 아닙니다. Claude Code는 이제 세 가지 훅 유형을 지원합니다:

- `"type": "command"`: 셸 스크립트 (내가 사용하는 방식)
- `"type": "http"`: URL에 JSON을 POST하고 JSON을 반환받음
- `"type": "prompt"`: 이벤트를 해석하는 LLM 서브 에이전트

HTTP 훅은 흥미로운 가능성을 열어줍니다: 모든 도구 실패를 Slack(슬랙) 웹훅으로 보내거나, 보안 측정 항목을 대시보드에 푸시하거나, 허용/거부 결정을 중앙 집중식 서비스에 위임하는 것 등입니다. 프롬프트 훅은 정규식보다 미묘한 결정을 가능하게 하지만, 토큰 비용과 지연 시간(latency)이 발생합니다. 나는 결정론과 속도를 위해 Bash를 고수하지만, 아키텍처는 세 가지 모두를 지원합니다. (훅 문서 참조)

#### 전송 전 스캔 (The Pre-Send Scan)

마지막으로 하나의 훅은 사용자 프롬프트가 API로 전송되기 전에 가로챕니다:

```bash
# 사용자 프롬프트에 비밀 정보 언급을 감지합니다
# (하지만 비밀 정보에 대한 질문은 감지하지 않음)
if echo "$prompt" | grep -qiE \
  '(password|api.?key|token|credential)'; then
    if ! echo "$prompt" | grep -qiE \
      '(how|help|explain|what is)'; then
        echo '{"systemMessage":"Warning: your prompt
          mentions sensitive information."}'
    fi
fi
```

프롬프트에 직접 비밀번호를 입력하면(비밀번호에 대한 질문이 아닌 경우), 텍스트가 Anthropic(앤트로픽) 서버로 전송되기 전에 훅이 경고합니다.

#### 안전망: PostToolUseFailure

PostToolUseFailure는 도구 실패를 가로챕니다: 실패한 `Read`, 타임아웃(timeout)된 Bash, 응답하지 않는 MCP 등. 이러한 오류가 사라지게 내버려두는 대신, 훅은 이를 로깅(logging)합니다:

```bash
TOOL_NAME=$(echo "$INPUT" | jq -r '.tool_name // "unknown"')
ERROR=$(echo "$INPUT" | jq -r '.error // .stderr // "no details"' \
  | head -c 500)

LOG_FILE="$HOME/.claude/logs/tool-failures.log"
# 1MB에서 자동 로테이션
if [ -f "$LOG_FILE" ] && \
   [ "$(stat -f%z "$LOG_FILE")" -gt 1048576 ]; then
    mv "$LOG_FILE" "$LOG_FILE.old"
fi
echo "[$(date '+%Y-%m-%d %H:%M:%S')] FAIL tool=$TOOL_NAME \
  error=$(echo "$ERROR" | tr '\n' ' ' | head -c 200)" \
  >> "$LOG_FILE"
```

이것은 엄밀히 말하면 보안은 아닙니다. 관측 가능성(observability)입니다. 훅이 작업을 차단하면 즉시 알 수 있습니다. 하지만 도구가 조용히 실패하면(MCP 타임아웃, 파일 없음, 예상치 못한 형식), 로깅 없이는 이러한 오류는 보이지 않습니다. PostToolUseFailure는 이 격차를 해소합니다. 1MB에서 자동 로테이션은 디스크를 깨끗하게 유지합니다.

### 훅이 포맷팅 및 보안에서 LLM을 능가하는 이유

이 시스템의 모든 결정에 적용하는 간단한 규칙이 있습니다: 린터(linter)가 더 잘하는 일을 LLM에게 시키지 마십시오.

코드 포맷팅(formatting)이 완벽한 예시입니다. Claude는 Python 코드를 포맷할 수 있습니다. 사실 꽤 잘합니다. 하지만 "꽤 잘한다"는 결정론적이지 않습니다. LLM은 컨텍스트, 토큰 샘플링의 확률론적 분위기, 달의 위상에 따라 `dict[str, Any]`를 다르게 포맷할 것입니다.

린터는 `dict[str, Any]`를 동일한 방식으로 포맷합니다. 항상. 예외 없이.

**전체 크기로 이미지를 보려면 Enter 키를 누르거나 클릭하세요.**
![훅은 결정론적 함수입니다: 동일한 입력, 동일한 출력, 항상. LLM은 확률 분포입니다: 동일한 입력, 다른 출력, 항상. 포맷팅과 보안에서는 함수가 필요합니다.](image_placeholder_url_hooks_vs_llms)
훅은 결정론적 함수입니다: 동일한 입력, 동일한 출력, 항상. LLM은 확률 분포입니다: 동일한 입력, 다른 출력, 항상. 포맷팅과 보안에서는 함수가 필요합니다.

`Write/Edit`에 대한 PostToolUse 훅은 이 원칙을 적용합니다:

```bash
if [ -n "$file_path" ] && [ -f "$file_path" ]; then
    case "$file_path" in
        *.ts|*.tsx|*.js|*.jsx|*.mjs|*.cjs)
            npx prettier --write "$file_path" 2>/dev/null || true
            ;;
        *.py)
            ruff format --quiet "$file_path" 2>/dev/null || true
            ;;
        *.json)
            npx prettier --write "$file_path" 2>/dev/null || true
            ;;
    esac
fi
```

Claude가 작성하는 모든 파일은 해당 결정론적 도구(TypeScript/JavaScript/JSON용 Prettier, Python용 Ruff)에 의해 자동으로 재포맷됩니다. Claude는 최종 포맷팅을 보지도 못합니다. Claude는 작성하고, 훅이 재포맷하며, 결과는 표준 준수를 보장합니다.

동일한 원칙이 취약점 감지에도 적용됩니다. 모든 `npm install` 후, `posttooluse-bash-vuln.sh` 훅은 출력에서 CVE(Common Vulnerabilities and Exposures) 언급을 스캔합니다:

```bash
if echo "$command" | grep -qE \
  '^(npm|pnpm|yarn) (install|add|remove)'; then
    if echo "$stdout" | grep -qiE '(vulnerabilit|CVE-)'; then
        echo '{"systemMessage":"Vulnerabilities detected.
          Run npm audit for details."}'
    fi
fi
```

결정론적입니다. 재현 가능합니다. 소비되는 토큰은 없습니다.

### Claude Code가 챗봇이 아닌 플랫폼인 이유

Claude Code는 도구가 아닙니다. 플랫폼입니다.

차이점은 구조적입니다. 도구는 사용하지만, 플랫폼은 프로그래밍합니다. `settings.json`은 시스템 구성 파일입니다. 훅은 이벤트 리스너(event listener)입니다. 스킬은 온디맨드(on-demand) 모듈입니다. 에이전트는 격리된 리소스를 가진 프로세스입니다. MCP 서버는 주변 장치 드라이버입니다.

Claude Code에 대한 대부분의 글은 고립된 부품들을 보여줍니다: "훅을 작성하는 방법", "유용한 스킬", "설치해야 할 5가지 MCP 서버". 이것들은 구성 요소입니다. 이러한 구성 요소들이 어떻게 서로를 강화하는 시스템을 형성하는지는 보여주지 않습니다.

환각 방지 프로토콜은 Context7이 접근 가능한 MCP 서버이기 때문에 작동합니다. 보안 훅은 정적 권한이 경계를 정의하기 때문에 작동합니다. 다중 에이전트 브레인스토밍은 에이전트가 전용 도구와 모델을 가지고 있기 때문에 작동합니다. 한 계층을 제거하면 다른 계층의 효율성이 떨어집니다.

내 입장: Claude Code의 가치는 모델 자체에 있는 것이 아닙니다. 모델을 둘러싼 인프라(infrastructure)에 있습니다. 모델은 엔진입니다. `CLAUDE.md`, 스킬, 에이전트, 훅, 권한 및 MCP 서버는 차체, 변속기, 브레이크 및 대시보드입니다.

마지막으로 한 가지 더. 이 시스템은 쇼케이스가 아닙니다. 나의 일상적인 작업 환경입니다. 모든 스킬, 모든 훅, 모든 권한은 내가 직면했던 실제 문제를 해결하기 위해 작성되었습니다. 환각 방지 프로토콜은 날조된 함수 시그니처에서 탄생했습니다. `write-guard`는 실수로 API 키가 파일에 작성되는 것에서 탄생했습니다. `bash-guard`는 잘못된 `rm -rf`에서 탄생했습니다. 이것은 예방적 엔지니어링이 아닙니다. 결정론적 코드로 응축된 반응적 엔지니어링입니다.

### 이 분석의 한계

- **생존자 편향(Survivorship bias)**. 나는 나에게 효과적인 시스템을 보여주고 있습니다. 나의 워크플로(기술 문서, Python/TypeScript API, ML 연구)가 모든 사용 사례를 대표하지는 않습니다. 모바일 개발자나 데이터 엔지니어는 다른 설정을 가질 것입니다.
- **토큰 비용**. 32개의 스킬은 지연 로딩을 사용하더라도 컨텍스트를 소비합니다. 최소한의 설정과 비교하여 토큰 오버헤드(overhead)를 정확히 측정하지는 않았습니다. 직관적으로는 상당할 것입니다.
- **Bash 훅의 취약성**. 훅은 `jq`, `grep` 및 Claude Code의 안정적인 JSON 형식에 의존합니다. 입력 형식의 업데이트는 시스템을 깨뜨릴 수 있습니다. 훅 자체에 대한 자동화된 테스트는 없습니다.
- **정규식(Regex)은 보안이 아닙니다**. 정규식을 통한 비밀 스캔은 경험적 방법(heuristic)이며, 보장은 아닙니다. Base64 인코딩(encoded)되거나 여러 줄에 걸친 비밀 정보는 통과할 수 있습니다. 이것은 비밀 관리자(secrets manager)를 대체할 수 없습니다.
- **유지보수**. 32개의 스킬, 10개의 에이전트, 17개의 훅. 이는 유지보수해야 할 시스템입니다. API는 진화하고, 모델은 변경되며, 패턴은 오래됩니다. 초기 투자는 상당합니다. 좋은 소식은 플러그인을 통해 가장 일반적인 에이전트(코드 검토, 테스트, 프런트엔드)의 유지보수를 커뮤니티 패키지에 위임할 수 있다는 것입니다.

### 이것이 당신에게 미치는 변화

큐레이션되고 바로 사용할 수 있는 선택 항목이 포함된 동반 저장소인 `claude-code-blueprint`를 게시했습니다. 여기에는 `CLAUDE.md`, 8개의 스킬, 4개의 에이전트, 11개의 훅(2개는 정제됨), 규칙, 명령 및 권한 템플릿이 포함됩니다. 내 설정의 미러(mirror)가 아닙니다. 사용에 맞게 조정된 시작점입니다.

Claude Code를 처음 사용하는 경우: `blueprint`의 `CLAUDE.md`로 시작하십시오. 환각 방지 프로토콜과 신뢰도 수준을 추가하십시오. 이것이 노력 대비 효과 비율이 가장 좋은 변화입니다. 30분 정도의 시간을 할애하십시오.

이미 중간 수준의 설정을 가지고 있는 경우: 보안 훅을 추가하십시오. `blueprint`의 `write-guard`와 `bash-guard`에는 구조와 위협 범주가 포함되어 있습니다. 자신만의 정규식을 추가하십시오. 적응하고 테스트하는 데 오후 한나절 정도 걸릴 것입니다. 주요 사용 사례에 맞는 스킬 3-5개를 추가하십시오.

완전한 시스템을 구축할 준비가 되었다면: 도메인별로 에이전트를 설계하고, MCP 서버를 구성하고, 제로 트러스트 권한을 작성하십시오. `blueprint`는 골격을 제공하며, 적응하고 반복하는 데 일주일 정도의 시간을 할애하십시오. 시스템은 나중에가 아니라 미리 보상합니다.

### Claude Code 설정 FAQ

**SDK나 API 대신 마크다운 파일인 이유는 무엇입니까?**

Anthropic의 아키텍처적 선택입니다. 스킬과 에이전트는 모델이 지시로 해석하는 평범한 텍스트입니다. 장점: 빌드 단계 없음, 종속성 없음, 완전한 이식성. 단점: "런타임(runtime)"은 모델 자체이며, 비결정론적인 해석이 발생할 수 있습니다.

**Claude Code 훅은 실행 속도를 늦춥니까?**

각 훅에는 타임아웃(timeout, 유형에 따라 2~15초)이 있습니다. 실제로는 나의 훅은 100ms 이내에 실행됩니다. `write-format.sh`가 가장 느리지만(`prettier`/`ruff` 호출), 15초 타임아웃으로 인해 투명하게 작동합니다. (Anthropic 문서 참조)

**이것이 Claude 외의 다른 모델에서도 작동합니까?**

아니요. 스킬, 에이전트, 훅 및 `CLAUDE.md`는 Claude Code에만 해당됩니다. 아키텍처 개념(계층, 제로 트러스트, 결정론)은 전이 가능하지만, 구현은 Anthropic 생태계에 묶여 있습니다.

**32개의 스킬이 컨텍스트 창을 포화시키지 않습니까?**

로딩은 지연(lazy)됩니다. Claude는 대화 컨텍스트와 설명이 일치하는 경우에만 스킬을 로드합니다. 실제로는 2~4개의 스킬이 동시에 로드되며, 그 이상은 거의 없습니다. `MAX_MCP_OUTPUT_TOKENS`가 25,000이고 최대 출력 토큰이 64,000이므로 넉넉한 여유가 있습니다.

**작동하지 않는 훅은 어떻게 디버깅합니까?**

훅은 디버깅을 위해 `stderr`에 내용을 작성합니다(Claude에게는 보이지 않지만 로그에는 보입니다). `set -euo pipefail + trap ERR` 패턴은 깨진 훅이 조용히 실패하는 대신 차단되도록 보장합니다. 개발 중에는 `echo "DEBUG: ..." >&2`를 추가합니다. 그리고 `PostToolUseFailure` 훅을 추가한 이후, 모든 도구 실패는 `~/.claude/logs/tool-failures.log`에 자동으로 로깅됩니다.

**영구 메모리가 오래되지 않습니까?**

실제 위험입니다. `MEMORY.md`는 자동 덤프가 아닙니다. 큐레이션된 원장입니다. 여러 세션을 통해 확인된 패턴만 저장하며, 가설은 저장하지 않습니다. 그리고 정보가 오래되면 Claude가 업데이트하거나 제거합니다. 규칙: 프로젝트에 대해 검증되지 않은 것은 메모리에 절대 쓰지 마십시오.

### 결론: 자신감 넘치는 거짓말쟁이에서 검증하는 시스템으로

#### `Depends()`로 돌아가서

3개월 전, Claude는 눈 하나 깜빡하지 않고 `Depends()`에 대한 세 개의 매개변수를 날조했습니다. 40분을 허비하고, 무의미한 리팩터링을 했으며, LLM을 신뢰하는 것은 유효 기간이 있는 전략이라는 것을 깨달았습니다.

오늘날, 같은 질문은 검증 체인(verification chain)을 거칠 것입니다: 환각 방지 프로토콜은 Context7을 트리거하고, 이는 Starlette/FastAPI 문서를 실시간으로 질의하여 실제 시그니처(`Depends(dependency)`, 하나의 인자)를 높은(HIGH) 신뢰도와 출처 링크와 함께 반환할 것입니다. 거짓말은 내 코드에 도달하기 전에 가로채질 것입니다.

주목할 만한 모순이 있습니다. Claude가 거짓말하는 것을 막는 시스템은 Claude로 구축되었습니다. 위험한 명령을 차단하는 훅은 Claude가 위험한 명령을 실행할 수 있는 터미널에서 작성되었습니다. 제로 트러스트 시스템을 구축하는 데 필요한 신뢰는 시스템 자체가 제거하도록 설계된 바로 그 신뢰와 같습니다.

#### 나의 입장

Claude Code는 2026년에 거의 아무도 그렇게 사용하지 않는 프로그래밍 가능한 개발 플랫폼입니다. 모델 때문이 아닙니다. 인프라(infrastructure) 때문입니다. 훅, 스킬, 에이전트, 권한 및 MCP는 LLM이 프로세서이고 나머지는 결정론적 코드인 최소한의 운영 체제를 형성합니다.

Claude Code를 터미널이 있는 챗봇으로 취급하는 사용자는 그 기능의 일부만을 사용하는 것입니다. 프로그래밍 가능한 플랫폼으로 취급하는 사용자는 더 빠르게 출시하고 더 적게 고장냅니다.

하지만 시계 장치는 아직 완성되지 않았습니다. HTTP 및 프롬프트 훅(위에 문서화됨)을 통해 Bash 스크립트를 엔드포인트(endpoint)나 전문 서브 에이전트로 대체할 수 있습니다. 플러그인은 전체 시스템을 패키징하여 배포할 수 있습니다. MCP 프로토콜은 매주 새로운 서버를 추가하고 있습니다.

그리고 현재 가장 중요한 실험적 기능인 에이전트 팀(Agent Teams)이 있습니다. 단일 Claude Code 인스턴스가 조용히 서브 에이전트를 조율하는 대신, 여러 인스턴스가 직접 통신하며 병렬로 협업합니다. 리더가 작업을 할당하고, 팀원들은 각자의 컨텍스트에서 작업하며, 중개자 없이 조율합니다. 이는 내 시스템의 자연스러운 진화입니다: 단일 코어 OS에서 멀티코어(multi-core) OS로 이동하는 것입니다. 이 기능은 아직 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` 뒤에 있지만, 방향은 명확합니다.

질문은 더 이상 "Claude Code가 내 IDE를 대체할 수 있는가"가 아닙니다. "에이전트의 프로그래밍 가능성을 운영 체제 자체가 되기 전까지 얼마나 밀어붙일 수 있는가"입니다.

나는 답을 가지고 있지 않습니다. 하지만 그 질문은 40분의 리팩터링 가치가 있다는 것을 압니다.
