Hermes 에이전트 심층 분석: 자가 개선 AI 에이전트의 작동 원리

## 서론

Hermes 에이전트는 Nous Research가 개발한 오픈 소스 AI 에이전트입니다. 다중 에이전트 오케스트레이션(multi-agent orchestration)을 중심으로 구축된 OpenClaw와 달리, Hermes는 단일 에이전트(single agent)로, 구성 업데이트가 아닌 실제 사용을 통해 실행 시간이 길어질수록 능력이 향상됩니다.

대부분의 에이전트는 발생했던 일을 기억하지만, Hermes는 한 단계 더 나아갑니다. Hermes는 성공적이었던 부분을 추출하여 재사용 가능한 스킬(skill)로 작성하고, 유사한 문제가 발생할 때 다음번에 이를 불러옵니다. 학습 루프(learning loop)는 자체적으로 실행되며, 메모리 아키텍처가 캐시 인식(cache-aware) 방식이므로 에이전트가 더 많이 학습해도 토큰 비용이 계속 증가하지 않습니다.

이 글은 학습 루프, 4계층 메모리 시스템, 게이트웨이(gateway), 에이전트 루프 내부 작동 방식, 터미널 백엔드(terminal backend), 스킬, 도구(tool), 예약된 자동화(scheduled automations), 세션 지속성(session persistence), 그리고 Nebius Token Factory를 이용한 Hermes 에이전트의 대규모 운영에 대해 설명합니다.

## 학습 루프 (The Learning Loop)

OpenClaw와 같은 에이전트들은 세션 전반에 걸쳐 컨텍스트(context)를 유지하고 중앙 허브를 통해 라우팅(routing)하는데, 이는 단순한 사용 사례에는 잘 작동하지만, '무엇이 발생했는지'를 저장하는 것과 '무엇이 효과적이었는지'를 저장하는 것 사이에는 간극이 있습니다. Hermes는 이 간극을 중심으로 설계되었으며, 완료된 워크플로우(workflow)가 에이전트가 다음번에 동일한 단계를 반복하지 않고 따를 수 있는 재사용 가능한 절차로 변환됩니다.

이것이 학습 루프의 역할입니다. 모든 세션 아래에서 실행되는 폐쇄형 피드백 사이클로, 메모리, 스킬, 세션 검색이 모두 동일한 연속적인 프로세스의 결과물입니다.

이 루프는 네 가지 움직이는 부분으로 구성되어 있으며, 각 부분은 사이클의 다른 지점에서 작동합니다. 이들이 어떻게 연결되는지 살펴보겠습니다.

### 에이전트가 큐레이션(Curated)하는 메모리와 주기적인 안내 (Agent-Curated Memory with Periodic Nudges)

대부분의 에이전트는 모든 것을 기록하고 유용한 것을 검색하지 못하거나, 아예 아무것도 기록하지 않고 매 세션마다 새로 시작합니다. Hermes는 에이전트 스스로 무엇을 보관할 가치가 있는지 결정하는 책임을 부여함으로써 이 두 가지 문제를 모두 피합니다. 이를 '주기적인 안내(periodic nudge)'라는 메커니즘을 통해 수행합니다.

세션 중 정해진 간격으로, 에이전트는 내부 시스템 수준 프롬프트(internal system-level prompt)를 받게 되며, 이는 과거에 무슨 일이 있었는지 되돌아보고 메모리에 지속할 가치가 있는 것이 있는지 평가하도록 요청합니다. 이는 사용자 입력 없이 작동하며, 에이전트는 최근 활동을 스캔하여 향후 세션에서 유용할 임계값을 넘어서는 것이 있다면 메모리 파일에 기록합니다.

그 결과, 메모리는 모든 상호작용의 덤프(dump)가 되는 대신 큐레이션된 상태를 유지합니다.

### 자율적인 스킬 생성 (Autonomous Skill Creation)

에이전트가 작업을 완료하면, 에이전트가 거쳐온 경로가 문서화할 가치가 있는지 확인합니다. 트리거(trigger)는 구체적입니다. 5개 이상의 도구 호출(tool calls), 오류로부터의 복구, 사용자 수정, 또는 성공적이었던 명확하지 않은 워크플로우가 해당됩니다.

확인이 통과되면, 에이전트는 `~/.hermes/skills/`에 스킬 파일(skill file)을 생성합니다. 이는 로그 항목이 아니라 에이전트가 향후 세션에서 원래 단계를 반복하지 않고 따를 수 있는 재사용 가능한 명령어 세트(instruction set)입니다.

각 스킬 파일은 이름, 짧은 설명, 관련된 단계, 그리고 워크플로우의 일부인 도구 호출이나 파일 참조를 포함하며, 그 구조는 `agentskills.io` 오픈 표준을 따르므로 호환되는 에이전트 간에 스킬을 이식할 수 있습니다.

### 스킬 자가 개선 (Skill Self-Improvement)

스킬은 한 번 작성되면 고정되지 않으며, 에이전트는 계속해서 이를 사용하고 실행 중에 더 나은 경로를 찾으면 업데이트합니다.

`skill_manage` 도구를 통해 6가지 스킬 액션(skill actions)을 사용할 수 있습니다.
`create`, `patch`, `edit`, `delete`, `write_file`, `and remove_file`.

에이전트는 대부분의 업데이트에 `patch`를 기본으로 사용하며, 이전 문자열과 대체 문자열만 전달하여 특정 변경 사항을 적용합니다. 이는 전체 스킬 내용이 아닌 변경된 텍스트만 도구 호출에 나타난다는 의미입니다.

`edit`보다 `patch`를 선호하는 것은 정확성(correctness)과 효율성(efficiency)을 모두 고려한 결정입니다. 전체 재작성은 이미 작동하는 것을 손상시킬 위험이 있는 반면, `patch`는 변경된 부분만 수정하고 토큰 사용량 측면에서 더 효율적입니다.

### LLM 요약 기능을 갖춘 FTS5 세션 검색 (FTS5 Session Search with LLM Summarization)

모든 세션은 SQLite 아카이브에 기록되고 FTS5를 사용하여 색인됩니다. 따라서 에이전트는 전체 이전 세션을 메모리에 로드하는 대신 과거 컨텍스트를 검색합니다. 검색된 결과는 주입되기 전에 LLM 요약(LLM summarization)을 거치므로 현재 작업과 관련된 내용만 들어옵니다.

이 계층은 '언제 무엇이 일어났는지'에 대한 에피소드 기억(episodic memory)을 처리하며, '어떻게 해야 하는지'에 대한 절차 기억(procedural memory)을 처리하는 스킬 계층(skills layer)과 의도적으로 분리되어 있습니다. 두 계층은 함께 작동하지만 다른 질문에 답하며, 모든 것을 하나로 섞는 대신 별도의 저장소에 보관하는 것은 전체 메모리 시스템이 어떻게 구성되는지에까지 이어지는 설계 결정입니다. 다음 섹션에서 다룰 내용입니다.

## 다단계 메모리 시스템 (The Multi-Level Memory System)

모든 것을 하나의 메모리 저장소에 섞어두면 대부분의 에이전트 메모리 시스템이 시간이 지남에 따라 신뢰할 수 없게 됩니다. 따라서 Hermes는 이를 네 가지 별개의 계층으로 분리하며, 각 계층은 특정 작업, 디스크상의 특정 위치, 그리고 읽히는 특정 순간을 가집니다.

### 프롬프트 메모리 (Prompt Memory) `MEMORY.md` 및 `USER.md`

이것은 '항상 켜져 있는' 계층으로, 에이전트가 요청하지 않아도 모든 세션 시작 시 로드되는 컨텍스트입니다. 두 파일은 `~/.hermes/memories/`에 있으며, 첫 번째 메시지가 처리되기 전에 시스템 프롬프트에 직접 주입됩니다.

두 파일을 합한 총 문자 제한은 3,575자로, 축적보다는 큐레이션을 강제하기 위해 의도적으로 엄격하게 설정되었습니다. 에이전트는 `add`, `replace`, `remove`의 세 가지 작업을 통해 메모리 도구를 통해 이들을 관리합니다.

중요한 세부 사항은 `MEMORY.md` 또는 `USER.md`에 대한 세션 중 편집은 대화 도중에 적용되지 않고 다음 세션에서만 적용된다는 것입니다.

### 세션 검색 (Session Search) SQLite + FTS5

여기서 이해할 가치가 있는 차이점은 에이전트가 세션 검색을 사용하는 시점과 프롬프트 메모리를 사용하는 시점입니다.

*   프롬프트 메모리(Prompt memory)는 항상 켜져 있습니다. 즉, 에이전트가 로드하기로 결정하지 않아도 로드됩니다.
*   세션 검색(Session search)은 의도적인 검색 단계입니다. 에이전트는 과거 컨텍스트가 현재 작업과 관련이 있다고 판단할 때 SQLite 아카이브에 대해 쿼리(query)를 실행합니다.

실질적인 경계는 영구성(permanence)입니다. 어떤 정보가 모든 향후 대화에 존재할 만큼 중요하다면, `MEMORY.md` 또는 `USER.md`에 속합니다. 특정 주제가 다시 나올 때만 유용하다면, 세션 아카이브에 남아 온디맨드(on-demand)로 검색될 수 있습니다. 에이전트는 주기적인 안내(periodic nudge) 중에 이 판단을 내려, 모든 것을 한 곳에 기본값으로 두는 대신 정보 조각이 어떤 계층에 속하는지 결정합니다.

### 스킬 절차 기억 (Skills Procedural Memory)

스킬의 생성 측면은 섹션 1에서 살펴보았으므로, 여기서는 스킬이 어떻게 저장되고 에이전트가 토큰 예산을 소모하지 않고 스킬을 로드하는 방법에 중점을 둡니다.

모든 스킬은 `~/.hermes/skills/`에 개별 마크다운 파일(markdown files)로 저장됩니다. 새로 설치할 때, 리포지토리(repository)의 번들 스킬(bundled skills)이 이 디렉토리에 복사되며, 에이전트가 생성하거나 Skills Hub에서 설치한 스킬도 함께 추가됩니다. 로딩 전략은 점진적 공개 패턴(progressive disclosure pattern)을 따릅니다. 기본적으로 시스템 프롬프트(system prompts)에는 스킬 이름과 짧은 요약만 포함되며, 스킬의 전체 내용은 에이전트가 현재 작업과 관련이 있다고 판단할 때만 로드됩니다.

이는 존재하는 스킬의 수에 관계없이 토큰 사용량을 일정하게 유지합니다. 200개의 스킬을 가진 에이전트도 40개의 스킬을 가진 에이전트와 거의 동일한 컨텍스트 비용을 지불합니다. 이는 상세한 내용이 실제로 필요할 때만 컨텍스트에 입력되기 때문입니다.

### 혼초 계층 사용자 모델링 (Honcho Layer User Modeling)

지금까지의 세 가지 계층은 모두 에이전트가 무언가를 적극적으로 기록해야 합니다. 네 번째 계층은 다르게 작동합니다. 명시적인 기록을 기다리는 대신, 세션 전반에 걸쳐 수동적으로 사용자의 그림을 구축하여 시간이 지남에 따라 변화하는 선호도, 커뮤니케이션 스타일, 도메인 지식을 추적합니다. 이것이 바로 Honcho이며, 나머지 위에 위치하는 선택적 사용자 모델링 계층입니다.

이것은 변증법적 모델링(dialectic modeling) 접근 방식을 사용하여 12개의 식별 계층에 걸쳐 사용자와 에이전트를 서로 관련하여 모델링합니다.

이것은 선택 사항이며, 대부분의 작업별 또는 자동화 설정에는 다른 세 계층으로도 충분합니다. Hermes를 일상적인 개인 비서로 사용하고 실제로 작업하는 방식에 맞춰진 응답을 원할 때 그 오버헤드(overhead)의 가치를 발휘합니다.

다음으로 다룰 게이트웨이는 이 네 계층 모두에 플랫폼 간 액세스를 가능하게 하여 플랫폼을 전환할 때 컨텍스트를 잃지 않도록 합니다.

## 게이트웨이 (The Gateway)

학습 루프와 메모리 시스템은 에이전트가 필요할 때 도달할 수 있어야만 유용합니다. 이것이 바로 게이트웨이가 처리하는 역할입니다. 게이트웨이는 Hermes를 계속 실행하고 페어링(paired)된 모든 플랫폼에 연결 상태로 유지하는 지속적인 백그라운드 서비스(persistent background service)입니다. 따라서 에이전트는 필요할 때 시작하는 것이 아니라 항상 켜져 기다리고 있는 존재입니다.

### 플랫폼 어댑터 및 세션 라우팅 (Platform Adapters and Session Routing)

Hermes는 CLI, Telegram, Discord, Slack, WhatsApp, Signal, Email에 연결되며, 각기 자체 어댑터(adapter)를 가지고 있지만 모두 공유 세션 라우팅 계층으로 연결됩니다. Telegram에서 시작된 대화는 세션이 플랫폼이 아닌 ID에 연결되어 있기 때문에 터미널에서 계속될 수 있습니다.

Telegram은 Project Conversations를 통해 한 단계 더 나아가며, Private Chat Topics를 통해 단일 채팅 내에서 격리된 워크플로우를 실행할 수 있습니다. 각 주제는 자체 스킬 바인딩(skill bindings)과 세션 컨텍스트를 가집니다.

`gateway/` 디렉토리는 메시징, 세션 라우팅, 전달, 페어링, 크론 틱킹(cron ticking)의 다섯 가지를 처리합니다.

*   페어링(Pairing)은 새로운 플랫폼이 에이전트 인스턴스와 연결되는 방식입니다.
*   크론 틱킹(Cron ticking)은 예약된 자동화가 적절한 시간에 트리거되고 올바른 플랫폼으로 다시 라우팅되는 방식입니다.

전체 시스템은 `hermes gateway` 명령으로 시작되는 시스템 서비스로 실행됩니다. 이는 터미널을 닫은 후에도 백그라운드에서 계속 실행된다는 의미입니다.

### 다른 에이전트 아키텍처와 비교하여 이것이 제공하는 기능 (What This Enables vs. Other Agent Architectures)

OpenClaw에서 게이트웨이는 전달을 처리하며, 그곳에서 책임이 끝납니다. 스킬 생성, 메모리 기록, 예약된 자동화 출력은 모두 게이트웨이 자체와 연결되지 않은 별도의 메커니즘을 통해 이루어집니다.

Hermes에서 게이트웨이는 동일한 루프의 일부입니다. 들어오는 메시지가 스킬 생성을 트리거할 수 있으며, 예약된 자동화는 동일한 게이트웨이 계층을 통해 출력을 다시 기록하고, 세션 라우팅이 시스템에 내장되어 있고 별도로 추가되지 않았기 때문에 플랫폼 간 연속성(cross-platform continuity)이 작동합니다.

게이트웨이가 외부 통신을 처리하고 메모리 시스템이 컨텍스트 지속성을 처리하므로, 다음으로 이해해야 할 부분은 메시지가 도착하는 순간부터 응답이 다시 나가는 순간까지 단일 턴(turn) 내에서 발생하는 일입니다.

## 에이전트 루프 (The Agent Loop)

CLI, Telegram 또는 다른 연결된 플랫폼에서 Hermes에 도달하는 모든 메시지는 `run_agent.py`에 구현된 동일한 동기식 오케스트레이션 엔진(synchronous orchestration engine)을 통과합니다. 이곳에서 메모리, 스킬, 도구, 그리고 게이트웨이가 모두 만나 정의된 순서로 실행됩니다.

### 턴 라이프사이클 (Turn Lifecycle)

메시지가 도착하면 에이전트는 태스크 ID(task ID)를 생성하고, 캐시된 시스템 프롬프트(cached system prompt)를 로드하거나 메모리 계층, 스킬 인덱스, 관련 컨텍스트 파일에서 새로운 프롬프트를 구축합니다. API 호출이 나가기 전에 사전 압축 검사(preflight compression check)가 실행되어 대화 기록이 컨텍스트 제한에 가까워지지 않는지 확인합니다.

모델이 도구 호출(tool calls)을 반환하면 에이전트는 이를 실행하고 결과를 추가한 다음 다른 API 호출을 위해 루프(loop)로 돌아갑니다. 모델이 최종 텍스트를 반환하면 세션은 SQLite에 지속(persist)되고, 응답은 게이트웨이를 통해 나갑니다.

### 통합으로서의 압축 (Compression as Consolidation)

사전 검사가 긴 대화를 감지하면, 하드 컨텍스트 제한(hard context limits)에 도달하기 전에 센티넬(sentinel)이 트리거됩니다. 보조 모델(auxiliary model)이 전체 대화를 스캔하여 3,575자 제한 내에서 메모리에 보관할 가치가 있는 것을 추출하고, 중간 턴(middle turns)을 삭제하는 대신 요약합니다.

요약된 턴을 원래 대화에 연결하는 참조 체인(reference chain)인 Lineage는 SQLite에 보존되므로, 압축 이벤트 후에도 에이전트가 이전 컨텍스트를 추적할 수 있습니다.

### 프롬프트 캐싱 (Prompt Caching)

Hermes는 안정적인 소스에서 시스템 프롬프트를 구축하므로, 접두사(prefix)는 턴 전반에 걸쳐 일관성을 유지하며, 대부분의 API 제공업체는 이를 캐시하여 후속 턴에서 지연 시간과 비용을 줄입니다. 세 가지 요인이 이 캐시를 깨뜨립니다. 세션 도중 모델을 전환하거나, 메모리 파일을 변경하거나, 컨텍스트 파일을 변경하는 경우입니다. 이 중 어느 하나라도 전체 재처리를 강제하므로, 프로덕션(production)에서 사용하는 제공업체가 중요합니다.

그리고 해당 제공업체가 세션 도중 중단되더라도 에이전트 루프는 멈추지 않습니다. `config.yaml`에 추론 제공업체(inference providers)의 순서 목록을 구성할 수 있으며, Hermes는 턴 라이프사이클을 방해하지 않고 자동으로 다음 제공업체로 폴백(fall through)합니다. 세션은 계속되고 컨텍스트는 그대로 유지되며, 사용자에게는 실패가 드러나지 않습니다.

## 터미널 백엔드 (Terminal Backends)

게이트웨이가 통신을 처리하지만, 터미널 백엔드는 실제 작업이 발생하는 위치를 결정합니다. Hermes는 여섯 가지 옵션을 제공하므로 에이전트는 실제 작업이 있는 곳에서 실행됩니다.

### 6가지 백엔드 (The 6 Backends)

백엔드는 모든 배포 컨텍스트(deployment contexts)를 포괄합니다.

*   **Local(로컬)**은 사용자 머신에서 직접 실행됩니다.
*   **Docker(도커)**는 사용자 환경을 벗어나지 않으면서 격리(isolation)를 위한 컨테이너 계층을 추가합니다.
*   **SSH**는 에이전트가 원격 서버(remote server)에서 완전히 실행될 수 있도록 합니다. 이는 작업이 파일, 데이터베이스 또는 해당 서버에 있는 서비스를 포함할 때 유용합니다.
*   **Daytona**와 **Modal**은 모두 서버리스(serverless)입니다. 즉, 유휴 상태일 때는 실행 환경이 휴면 상태가 되고 필요할 때 다시 활성화됩니다. 이는 Hermes를 지속적으로 실행하지만 사용량이 고르지 않을 경우 비용 측면에서 매우 중요합니다.
*   **Singularity(싱귤래리티)**는 Docker를 사용할 수 없거나 허용되지 않는 HPC(고성능 컴퓨팅) 및 연구 환경을 위한 옵션입니다.

이들 사이의 실질적인 선택은 수행하는 작업에 달려 있습니다. 격리가 중요하지 않은 개인 용도의 경우 **Local**, 호스트 시스템에서 에이전트를 샌드박스(sandboxed) 처리하려면 **Docker**, 작업이 서버에 있다면 **SSH**, HPC 클러스터의 경우 **Singularity**, 유휴 기간 동안 거의 비용이 들지 않는 서버리스 실행을 원한다면 **Modal** 또는 **Daytona**를 사용합니다.

### 컨테이너 강화 및 제로 텔레메트리 (Container Hardening and Zero Telemetry)

Docker와 함께 실행할 때 Hermes는 읽기 전용 루트 파일 시스템(read-only root filesystem)을 적용하고, Linux 기능(capabilities)을 드롭하며, 기본적으로 네임스페이스 격리(namespace isolation)를 제공합니다. 이는 선택적 설정이 아닌 아키텍처 기본값이므로 에이전트는 지정된 디렉토리 외부에는 쓸 수 없거나 권한을 에스컬레이션(escalate)할 수 없습니다.

제로 텔레메트리(Zero telemetry)도 동일한 방식으로 작동합니다. 개인 정보 보호 토글(privacy toggle)이 아니라 에이전트가 작동하는 방식의 내장 속성(built-in property)으로서 설계상 어떤 것도 사용자 머신을 떠나지 않습니다.

## 스킬 및 도구 (Skills and Tools)

도구와 스킬은 모두 Hermes가 기능을 확장하는 방식의 일부이지만, 작동 수준이 다르며 둘 중 하나를 시작하기 전에 그 차이점을 명확히 하는 것이 중요합니다.

*   **도구(Tools)**는 에이전트가 호출할 수 있는 것입니다. 터미널 명령 실행, 웹 검색, 이미지 생성과 같은 개별적인 기능입니다.
*   **스킬(Skills)**은 에이전트가 이러한 도구를 사용하여 '무엇을 할 줄 아는지'입니다. 에이전트가 이미 한 번 파악하고 기록해 둔, 도구 호출을 워크플로우로 연결하는 재사용 가능한 절차입니다.

### 내장 도구 (Built-in Tools)

40개 이상의 내장 도구는 다섯 가지 광범위한 범주를 포괄합니다.

*   **실행 도구(Execution tools)**는 터미널 명령과 코드 실행을 처리합니다.
*   **웹 도구(Web tools)**는 검색 및 브라우저 자동화를 다룹니다.
*   **미디어 도구(Media tools)**는 비전, 이미지 생성, 텍스트 음성 변환을 포함합니다.
*   **조정 도구(Coordination tools)**는 서브 에이전트 위임(subagent delegation)과 다중 모델 추론(multi-model reasoning)을 처리합니다. 이를 통해 에이전트는 병렬 워크스트림(parallel workstreams)을 위해 격리된 서브 에이전트를 시작하거나 특정 작업을 더 적합한 모델로 라우팅할 수 있습니다.
*   마지막 범주는 **메모리 및 계획 도구(memory and planning tools)**로, 에이전트가 세션 중에 자체 메모리 계층과 프로그래밍 방식으로 상호 작용하는 방식입니다.

모델 측면에서 Hermes는 Nous Portal을 단일 제공자 엔드포인트(single provider endpoint)로 통해 400개 이상의 모델에 연결되며, Hugging Face는 완전한 HF 추론 API(Hugging Face Inference API) 통합, 내장 모델 선택기, 설정 마법사를 갖춘 일등 제공업체(first-class provider)로 지원됩니다.

또한 MCP(Multi-Context Prompting) 기능을 지원하며, 더 나아가고자 하는 개발자를 위해 Hermes는 `pre_llm_call`, `post_llm_call`, `on_session_start`, `on_session_end`의 네 가지 플러그인 훅(plugin hooks)을 노출합니다. 이들을 통해 Hermes 위에 사용자 정의 로직(custom logic)을 추가하여 코드베이스를 포크(fork)하거나 내부적인 것을 건드리지 않고도 에이전트 루프의 특정 지점에 기능을 추가할 수 있습니다.

### 스킬 시스템 (Skills System)

Hermes는 MLOps, GitHub 워크플로우, 연구, 생산성 작업과 같은 영역을 다루는 40개 이상의 번들 스킬과 함께 제공됩니다. 이들은 새로 설치할 때 `~/.hermes/skills/`에 복사되며, 에이전트가 자율적으로 생성하거나 Skills Hub에서 설치한 모든 스킬과 함께 저장됩니다.

```
~/.hermes/skills/                  # 단일 진실 소스 (Single source of truth)
├── mlops/                         # 카테고리 디렉토리
│   ├── axolotl/
│   │   ├── SKILL.md               # 주요 지침 (필수)
│   │   ├── references/            # 추가 문서
│   │   ├── templates/             # 출력 형식
│   │   ├── scripts/               # 스킬에서 호출 가능한 헬퍼 스크립트
│   │   └── assets/                # 보조 파일
│   └── vllm/
│       └── SKILL.md
├── devops/
│   └── deploy-k8s/                # 에이전트 생성 스킬
│       ├── SKILL.md
│       └── references/
├── .hub/                          # Skills Hub 상태
│   ├── lock.json
│   ├── quarantine/
│   └── audit.log
└── .bundled_manifest              # 시드된 번들 스킬 추적
```

저장 형식은 `agentskills.io` 오픈 표준을 따르므로 스킬은 호환되는 에이전트 간에 이식 가능하며 변환 단계 없이 공유할 수 있습니다. 기본적으로 이름과 요약만 로드되며, 전체 내용은 요청 시에 로드됩니다.

## 예약된 자동화 (Scheduled Automations)

Hermes는 반복 작업을 단순한 셸 스크립트(shell scripts)나 AI를 호출하는 크론 작업(cron jobs)이 아닌 일등 에이전트 작업(first-class agent tasks)으로 처리합니다. 메시지를 예약하면 Hermes는 지침을 구문 분석하고(parses the instruction), `cron/` 디렉토리에 작업을 저장하며, 게이트웨이의 크론 틱킹(cron ticking)이 나머지를 처리합니다.

예정된 시간이 되면 에이전트 루프는 메모리와 스킬에 대한 완전한 액세스 권한으로 작업을 실행한 다음, 게이트웨이를 통해 지정된 위치로 출력을 라우팅합니다. 이는 모든 대화형 세션(interactive session)과 동일한 파이프라인이지만, 메시지 대신 클럭 틱(clock tick)에 의해 트리거될 뿐입니다.

## 세션 지속성 (Session Persistence)

지금까지 다룬 모든 것, 즉 학습 루프, 메모리 계층, 게이트웨이 라우팅, 에이전트 루프, 예약된 자동화는 단일 세션을 넘어 상태가 지속되는 것에 의존합니다. 이러한 내구성(durability)은 외부 서버 종속성(external server dependency)이 없는 휴대용 파일 기반 저장소인 `hermes_state.py`가 관리하는 SQLite 데이터베이스에서 나옵니다.

각 턴이 끝난 후 에이전트는 대화, 도구 호출 및 결과를 데이터베이스에 기록하며, 온디맨드 검색(on-demand retrieval)을 위해 FTS5를 통해 색인됩니다.

원시 트랜스크립트(raw transcripts)는 JSONL 파일로 이동하고, 크론 정의는 디스크에 별도로 저장되며, WAL(Write-Ahead Logging) 모드는 단일 작성자(single writer)와 함께 동시 읽기(concurrent reads)를 허용하여 여러 세션이 병렬로 실행될 때 신뢰성을 유지합니다.

이것이 전체 아키텍처입니다. 위의 모든 것은 로컬에서 잘 실행되지만, 인프라를 직접 관리하지 않고 Hermes-4-405B를 실행하고 싶다면 Nebius Token Factory가 필요합니다.

어떻게 작동하는지 살펴보겠습니다.

## Nebius를 선택한 이유 (Why I Picked Nebius?)

Nebius Token Factory는 모델 추론(model inference)을 위해 특별히 구축된 추론 플랫폼으로, Qwen 3.5, Nemotron Super, GLM-5와 같은 강력한 오픈 소스 모델을 관리형 엔드포인트(managed endpoints)로 사용할 수 있습니다.

Hermes 에이전트의 경우, GPU 프로비저닝(GPU provisioning), 로드 밸런싱(load balancing), 콜드 스타트(cold starts)를 직접 관리할 필요 없이 Nous Research의 Hermes-4-405B 모델이나 Nebius의 다른 모델을 실행할 수 있습니다.

단순한 호스팅된 엔드포인트 이상으로 만드는 것은 추론 계층을 둘러싼 것입니다. Nebius는 채팅 완료 기록을 남기며, 이를 Data Lab으로 직접 파이프하여 실제 Hermes 세션에서 파인튜닝 데이터셋(fine-tuning dataset)을 구축할 수 있습니다. 거기서 Token Factory에서 파인튜닝하고 체크포인트(checkpoint)를 사용자 고유의 커스텀 엔드포인트(custom endpoint)로 배포합니다. Hermes 설정이 특정 도메인(예: 지원, 연구, 코딩)을 처리하는 경우, 일반적인 기본 모델이 아니라 사용자가 사용하는 방식에 정확하게 맞춰진 모델을 얻게 됩니다.

또한 관찰 가능성(observability)이 내장되어 있습니다. 세션 전반에 걸쳐 요청 볼륨, 지연 시간, 토큰 사용량을 모니터링할 수 있습니다. 이는 Hermes가 예약된 자동화를 실행하거나 여러 사용자를 위해 여러 작업을 처리할 때 무언가가 느리거나 중단되는 시점을 알아야 할 때 중요합니다.

## Hermes를 Hermes-4-405B에 사용하도록 구성하기 (Configuring Hermes to Use Hermes-4–405B)

구성(configuration)은 몇 줄에 불과합니다. API 키를 설정한 다음, `hermes model`을 실행하여 모델 구성(model configuration)을 열고, 사용자 정의 OpenAI 호환 엔드포인트(custom OpenAI-compatible endpoint)를 선택한 다음, 기본 URL과 모델 ID를 설정합니다.

```
export NEBIUS_API_KEY=your_key_here
hermes model
# Select: Custom OpenAI-compatible endpoint
# Base URL: https://api.tokenfactory.nebius.com/v1/
# Model: NousResearch/Hermes-4-405B
```

Hermes-4-405B는 128K 컨텍스트 윈도우(context window)를 가지고 있어, 센티넬이 트리거되기 전에 사전 검사(pre-flight check)가 더 많은 공간을 확보하여 조기에 압축될 걱정 없이 긴 세션을 위한 충분한 여유를 제공합니다.

## Nebius가 가치가 있을 때 (그리고 아닐 때) (When Nebius Is Worth It (And When It Is Not))

405B 모델을 직접 실행한다는 것은 GPU 프로비저닝, 추론 워커(inference workers) 간의 로드 밸런싱, 유휴 세션을 위한 콜드 스타트(cold-start) 동작을 관리해야 함을 의미합니다. Nebius의 관리형 엔드포인트는 이 모든 것을 사용자 측에서 제거합니다.

하지만 관리형 엔드포인트가 항상 올바른 선택인 것은 아닙니다. Hermes를 개인 비서로 로컬에서 실행하는 경우, 더 작은 로컬 모델을 사용하는 것이 잘 작동하고 비용도 적게 듭니다.

Nebius Token Factory는 Hermes 에이전트 설정의 여러 지점에 적합합니다. 어떤 모델이 워크플로우에 적합한지 아직 평가 중이라면, Nebius Playground를 통해 무엇이든 확정하기 전에 모델을 테스트하고 비교할 수 있습니다. 준비가 되면 프로덕션(production) 측면도 처리합니다. 동시 세션(concurrent sessions), 예약된 자동화를 위한 보장된 가동 시간(guaranteed uptime), 그리고 GPU를 직접 프로비저닝할 필요 없는 관리형 인프라를 제공합니다.

## 결론

이것이 전체 아키텍처입니다. 단 하나의 메시지로 시작하는 것에 비해 많은 움직이는 부분이 있지만, 각 부분에는 이유가 있습니다. 에이전트는 일반적인 작업뿐만 아니라 사용자의 특정 작업에서 더 나아져야 합니다. `patch`를 `edit`보다 선호하는 것, 네 가지 메모리 계층, 프롬프트 캐싱 전략, 이 모든 결정은 시간이 지남에 따라 정확하고 저렴하게 실행되도록 하는 것이지, 단순히 바로 사용 가능한(out of the box) 능력만을 위한 것이 아닙니다.

이것은 간단한 작업을 위해 빠르게 설정하는 가벼운(lightweight) 설정이 아닙니다. 이것은 실행하고 유지 관리해야 하는 인프라(infrastructure)이며, 사용 사례가 좁고 수명이 짧다면 이러한 오버헤드는 과도하게 느껴질 것입니다.

그러나 매일 플랫폼을 넘나들며 반복되고 진화하는 작업을 처리하는 무언가를 구축하고 있다면 Hermes는 그만한 가치가 있습니다. OpenClaw가 여러 에이전트에 걸쳐 모듈성과 오케스트레이션을 제공하는 반면, Hermes는 시간이 지남에 따라 컨텍스트를 얻는 단일 에이전트를 제공합니다. 이를 테스트하든 여러 사용자를 위한 서비스로 실행하든, Nebius Token Factory는 인프라를 직접 관리하지 않고도 목표에 도달하는 간단한 방법입니다. 어떤 것이 합리적인지는 실제로 무엇을 구축하려는지에 전적으로 달려 있습니다.
