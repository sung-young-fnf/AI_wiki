Paperclip + Claude Code: 로컬 설치를 위한 가장 빠른 방법

약 2주 전, 한가한 주말에 저는 진지하게 사업을 시작할지 몇 시간을 고민했습니다. 제 전략은 "제가 CEO가 되고 모든 부하 직원은 AI 에이전트로 두는 것"이었습니다.

어제 인터넷을 검색하다가 Paperclip이라는 새로운 AI 도구에 대한 정보를 접했습니다. 초보자도 AI를 사용하여 회사를 운영할 수 있게 해준다고 합니다.

하지만 필요한 것은 강력한 에이전트가 아니라 "에이전트를 관리하는 시스템"입니다. OpenClaw, Claude, Cursor, Codex 등 강력한 AI 에이전트는 더 이상 드문 것이 아닙니다.

그럼에도 불구하고 Paperclip이 갑자기 큰 주목을 받은 이유는 또 다른 새로운 에이전트를 만들었기 때문이 아니라, 여러 AI를 직원처럼 할당하고, 관리하고, 예산을 책정하고, 조직으로 운영한다는 아이디어를 구체적인 방식으로 보여주었기 때문입니다.

Paperclip은 스스로를 "인간이 없는 회사를 위한 오케스트레이션 레이어(orchestration layer)"라고 설명합니다. 더 간단히 말하면, 에이전트를 만드는 방법을 가르치는 것이 아니라, 에이전트 그룹을 사용하여 회사를 운영하는 방법을 가르쳐줍니다.

실제로 2026년 3월 26일 기준으로 Paperclip GitHub 저장소는 약 33,000개의 스타(stars)와 4,700개의 포크(forks)를 기록했습니다. 이렇게 짧은 시간 안에 이러한 반응을 얻은 것은 단순한 소문 때문만이 아니라, 많은 개발자와 기업이 직면하고 있는 "다음 과제"를 직접적으로 해결하고 있기 때문일 것입니다.

### Paperclip이 독특한 이유

Paperclip 자체는 단일 AI가 아니라, OpenClaw, Claude, Codex, Cursor와 같은 여러 에이전트를 "고용"하고, 역할을 할당하고, 명령 및 제어 구조(command and control structure)를 생성하며, 워크플로우(workflow)를 관리하기 위한 기반으로 설계되었습니다.

이것이 이 프로젝트의 핵심입니다. 지금까지 AI 활용은 한 번에 하나의 강력한 모델이나 편리한 에이전트를 숙달하는 방향으로 진행되었습니다. 그러나 실제 상황에서는 AI가 많아질수록 관리가 어려워집니다.

누가 무엇을 하는지 불분명해지고, 작업이 중복되며, 토큰(token) 소비가 증가하고, 언제 작업을 중단해야 할지 알 수 없게 됩니다. Paperclip은 이러한 "다중 AI 사용 시대의 운영 문제"를 직접적으로 해결하는 제품입니다.

### Paperclip은 어떻게 작동하는가?

문서와 코드를 살펴보니 프로젝트의 설계 개념이 상당히 흥미로웠습니다. 또 다른 AI 모델을 만드는 것도 아니고, "에이전트 프레임워크(Agent framework)"도 아닙니다. 에이전트를 어떻게 만드는지에는 전혀 관심이 없습니다. Paperclip이 관심을 두는 것은 에이전트가 일단 있으면, 어떻게 이들을 합법적인 회사처럼 운영할 수 있는가입니다.

몇 가지 언급할 만한 가치가 있다고 생각하는 점이 있습니다.

**첫째, 에이전트를 차별하지 않습니다.**

OpenClaw, Claude, Cursor 또는 Python을 사용하여 직접 만든 스크립트 등 "하트비트 신호(heartbeat signal)"를 수신하기만 한다면 Paperclip은 이를 통합할 수 있습니다. 약간의 백그라운드 구성을 통해 크로스 플랫폼 "가상 팀(virtual team)"을 구축할 수 있습니다.

**둘째, 각 작업에 "이유(why)" 라벨을 제공합니다.**

이 세부 사항은 매우 중요하다고 생각합니다. 일반적인 작업 관리 도구는 에이전트에게 "무엇을 할지"만 알려줍니다. Paperclip은 각 작업에 "가계도(family tree)"를 추가합니다. 이 작업이 어떤 회사의 목표에서 파생되었는지, 이전 단계에 누가 있었는지, 그 위의 단계에 누가 있었는지를 알려줍니다. 에이전트는 작업을 받을 때 "이 코드를 작성해야 한다"는 것뿐만 아니라 "월 100만 달러의 매출을 달성하기 위해 이 코드를 작성하고 있다"는 것도 알게 됩니다.

프로젝트 관리를 해본 사람이라면 이 "이유(why)"가 얼마나 중요한지 이해할 것입니다.

**셋째, 에이전트에게 급여를 지급하고 예산 제한을 둡니다.**

각 에이전트에 대해 "Claude는 이번 달에 최대 50달러를 지출할 것이다"와 같은 월별 예산을 설정할 수 있습니다. 예산이 초과될 위험이 있는 경우, 시스템은 자동으로 회로 차단기(circuit breaker)를 작동시켜 에이전트가 작업을 중단하게 합니다. 이 설계는 "AI는 매우 빠르게 실행되지만, 현금을 훨씬 더 빠르게 소진한다"는 말을 떠올리게 합니다. 이 메커니즘이 없다면 수백 달러를 순식간에 날려버리는 악순환이 발생할 수 있습니다.

**넷째, 승인 절차를 거쳐야 합니다.**

Paperclip은 인간을 배제할 의도가 없습니다. 대신, 인간을 "이사회(board)"에 배치했습니다. 에이전트가 새로운 작업을 원하면 승인해야 합니다. 에이전트가 실행 계획을 세우면 검토해야 합니다. 에이전트가 잘못된 방향으로 가면 즉시 중단시킬 수 있습니다.

또한, 모든 작업은 변조할 수 없는 방식으로 기록됩니다. 문제가 발생하면 정확히 어떤 단계에서 잘못되었는지 다시 확인해 볼 수 있습니다.

### Paperclip 사용 방법

먼저 Node.js 20+와 pnpm 9.15+를 설치해야 합니다. 아직 설치되지 않았다면 공식 웹사이트에서 다운로드하여 설치하거나 선호하는 패키지 관리자를 사용할 수 있습니다.

Paperclip은 원클릭 설치 방법을 제공합니다. 터미널(terminal)을 열고 다음 명령어를 실행합니다.

```
npx paperclipai onboard --yes
```

서버가 실행되면 가장 먼저 "회사"를 만들고 첫 번째 직원인 CEO 에이전트를 고용합니다. 그 다음 "다음"을 누릅니다.

첫 번째 에이전트의 기본 이름을 그대로 사용합니다. 이 에이전트와 가장 자주 대화하게 될 것입니다. 다음으로 어댑터 유형(adapter type)을 선택해야 합니다. 여기서는 Claude Code를 사용하지만, 환경에 OpenCLAW가 이미 설치되어 있다면 OpenCLAW를 선택할 수도 있습니다.

Paperclip에 의해 자동으로 감지되어야 합니다. 그렇지 않은 경우, Claude Code를 사용하며, 모델(model)로는 Sonnet 4.6을 사용할 것입니다.

다음으로 환경을 테스트합니다. 터미널에 API 키를 추가하고 Claude Code를 설치했는지 확인해야 합니다.

이제 다음 단계로 넘어갑니다.

방금 생성한 에이전트에게 실제 수행할 수 있는 작업을 할당하여 역할을 더 명확하게 정의해야 합니다.

"당신은 CEO입니다. 당신의 임무는 회사의 방향을 설정하는 것입니다"와 같은 설명을 작성합니다.

특정 작업이 필요할 때, 에이전트는 생성된 기술을 사용하여 다른 에이전트에게 작업을 위임할 수 있어야 합니다.

다음으로 작업 제목을 만듭니다. 예를 들어, "내 AI 에이전트 에이전시를 위한 스킬을 만들고 잠재 고객에게 DM을 보내세요."

그런 다음 "다음"을 클릭합니다.

이제 실행할 차례입니다. 회사 이름과 회사를 운영하는 데 도움을 줄 에이전트가 모두 준비되었습니다. 이 에이전트가 우리가 주로 상호 작용할 에이전트가 될 것입니다.

다음으로 이슈(issue)를 생성하고 엽니다.

실행되면 React 인터페이스가 있는 Node.js 서비스가 표시됩니다. "회사"를 만들고, 몇몇 에이전트를 구성하고, 예산과 목표를 설정하면 AI 팀이 나머지를 자동으로 처리합니다.

보시다시피, 여전히 문제가 있습니다. Paperclip이 API 키를 인식하지 못합니다.

먼저 Claude API 키를 가져와 이 명령을 실행합니다. 그런 다음 다음 명령을 사용하여 Claude Code를 설치합니다.

```
export ANTHROPIC_API_KEY="sk-ant-..."
```

그런 다음 이 명령을 사용하여 Claude Code를 설치합니다.

```
npm install -g @anthropic-ai/claude-code
```

설치 단계를 따르고 설정 질문에 답변합니다. 모든 것을 완료한 후 API 키가 터미널에 올바르게 추가되었는지 확인하세요. 그 후 모든 것이 제대로 작동하면 에이전트와 상호 작용을 시작할 수 있습니다.

다음으로 대시보드(dashboard)로 돌아가 Agent CEO를 찾아봅니다. "구성(Configuration)"을 클릭한 다음 "환경 테스트(test the environment)"를 클릭합니다.

테스트가 통과되는 것을 볼 수 있으며, Paperclip이 API 키를 성공적으로 감지했음을 확인할 수 있습니다.

이제 대시보드에 있습니다. 보시다시피, 사용 가능한 에이전트는 원래 에이전트 하나뿐입니다.

여기서 새로운 이슈를 만들고 CEO에게 처리할 작업을 할당할 수 있습니다. 예를 들어, "모든 클라이언트의 콘텐츠 전략을 담당할 콘텐츠 관리자를 고용하세요"라고 프롬프트할 수 있습니다.

작업 설명에서 이 콘텐츠 관리자가 영업 스크립트 작성 외의 모든 콘텐츠 관련 작업을 처리할 것임을 명시합니다.

그런 다음 작업을 CEO에게 할당합니다. 에이전트가 새로운 작업을 처리하도록 하려면 언제든지 새로운 이슈를 생성할 수 있습니다.

### Paperclip의 한계는 무엇인가?

에이전트의 품질은 여전히 사용자의 문제입니다. Paperclip은 에이전트를 오케스트레이션(orchestrates)하지만, 더 스마트하게 만들지는 않습니다. Claude Code 에이전트가 잘못된 코드를 작성해도 Paperclip은 이를 감지하지 않습니다. 추적하고 기록하며 감사할 수 있도록 하지만, 출력의 품질은 전적으로 기본 에이전트에 달려 있습니다.

"아무도 없는(nobody)" 프레임워크라는 아이디어는 오해의 소지가 있습니다. Paperclip의 창시자조차 실제 작업에 필요해서 만들었다고 말합니다. 여전히 전략을 세우고, 출력을 검토하고, 판단을 내려야 합니다.

오류 전파는 실제 문제입니다. 한 에이전트가 실수를 하고 그것을 다른 에이전트에게 전달하면 오류가 증폭됩니다. 예산 통제가 도움이 되지만, 의미론적 오류(semantic errors)는 포착하지 못합니다. 즉, 에이전트가 자신 있게 잘못된 메시지를 올바른 수신자에게 보내는 경우입니다.

### 결론

이전에는 AI 도구를 개별 비서로 사용했습니다. 이제 Paperclip은 AI 도구를 조직 구조, 정렬된 목표, 비용 통제 및 거버넌스 메커니즘(governance mechanisms)을 갖춘 실제 회사로 조직화할 수 있음을 보여줍니다.

이러한 사고방식의 전환은 도구 자체보다 더 중요할 수 있습니다. AI가 정말 직원처럼 일할 수 있다면, 우리는 그들을 어떻게 관리해야 할까? 어떻게 효과적인 메커니즘을 구축해야 할까? 하는 생각을 하게 합니다.

물론 Paperclip은 아직 초기 단계에 있으며, 많은 기능이 여전히 개발 중입니다. 하지만 현재 버전만으로도 충분히 흥미롭습니다.

여러 AI 도구를 사용하고 있거나 1인 사업의 가능성을 탐색하려는 경우, 이 프로젝트는 분명히 살펴볼 가치가 있습니다.
