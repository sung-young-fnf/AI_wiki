# OpenClaw 내부 구조: 지속성(Persistent) AI 에이전트의 실제 작동 원리

## 서론

최근 'OpenClaw(이전 명칭 ClawdBot)'가 큰 주목을 받고 있습니다. 텔레그램을 통해 웹사이트 전체를 재구축하거나, 인간은 오직 손님으로만 참여하는 AI 에이전트 플랫폼을 만드는 등 놀라운 사례들이 등장하고 있습니다. 반면, 시스템에 대한 전체 권한을 가진 AI가 프롬프트 주입(Prompt injection) 공격을 받아 수천 개의 이메일을 삭제하는 것과 같은 보안 위험도 현실로 다가오고 있습니다.

웹 인터페이스 뒤에 숨어 있는 ChatGPT나 Claude와 달리, OpenClaw는 사용자의 하드웨어에서 지속성 프로세스(Persistent process)로 실행됩니다. 사용자는 왓츠앱(WhatsApp), 텔레그램(Telegram), 슬랙(Slack) 등을 통해 메시지를 주고받으며, 에이전트는 사용자가 잠든 사이에도 업무를 확인합니다. 이 에이전트는 파일 시스템, 터미널, 그리고 연결된 모든 API에 접근할 수 있습니다.

OpenClaw의 기술적 아키텍처는 생각보다 단순하면서도 강력합니다. 그 내부 구조를 자세히 살펴보겠습니다.

## 게이트웨이 아키텍처: 중앙 신경계

OpenClaw는 기본적으로 로컬 머신의 `127.0.0.1:18789` 포트에서 대기하는 단일 Node.js 프로세스로 실행됩니다. 이 프로세스를 **게이트웨이(Gateway)**라고 부르며, 왓츠앱, 텔레그램, 디스코드, 슬랙, 시그널 등 모든 메시징 플랫폼과의 연결을 동시에 관리합니다.

게이트웨이는 시스템의 중앙 신경계와 같습니다. 모든 플랫폼에서 들어오는 메시지는 게이트웨이를 통과하며, 에이전트가 생성한 응답 또한 게이트웨이를 통해 나갑니다. 모든 통신은 웹소켓(WebSocket) 프로토콜을 통해 이루어지므로, 실시간 양방향 메시징이 가능합니다.

### 세션 상태, 라우팅 및 보안

게이트웨이는 세 가지 핵심 기능을 수행합니다:
1. 세션 상태 관리 (Session state management)
2. 메시지 라우팅 (Message routing)
3. 보안 집행 (Security enforcement)

메시지가 도착하면 게이트웨이는 사용자, 대화 문맥 또는 설정된 라우팅 규칙에 따라 어떤 에이전트 세션이 이를 처리할지 결정합니다. 적절한 세션 상태를 로드하고 메시지를 에이전트에게 전달한 뒤, LLM(거대언어모델)이 응답을 생성할 때까지 기다렸다가 다시 해당 플랫폼으로 응답을 보냅니다.

이러한 중앙 집중식 설계는 기술적 난제를 해결해 줍니다. 예를 들어 왓츠앱 웹은 한 번에 하나의 활성 세션만 허용하는데, 게이트웨이가 단일 세션 역할을 수행하면서 내부적으로 여러 에이전트 대화를 관리함으로써 충돌을 방지합니다.

### 연결 및 인증 흐름

새로운 플랫폼이 연결을 시도하면 게이트웨이는 페어링 저장소(Pairing store)를 확인합니다. 승인되지 않은 장치는 연결이 거부되며, 명시적인 승인을 거친 후에야 특정 권한이 부여된 **장치 토큰(Device token)**을 발급받습니다. 이후의 연결은 이 토큰을 통해 인증됩니다.

## 에이전트 루프: 메시지에서 실행까지

메시지가 게이트웨이에 도달하면 단순히 LLM으로 전달되는 것이 아니라, 일련의 처리 과정을 거칩니다.

1. **컨텍스트 로드:** 해당 세션은 파일 시스템에서 대화 기록을 불러옵니다. 여기에는 현재 대화뿐만 아니라 과거의 모든 상호작용이 포함되어 있어 에이전트가 이전 대화 내용을 기억할 수 있습니다.
2. **LLM 및 도구 활용:** 에이전트는 메시지를 사용 가능한 도구(Tools) 및 기술(Skills) 정보와 함께 LLM에 전달합니다. 모델은 요청을 분석하고 일정 확인이나 이메일 전송과 같은 도구 호출이 필요한지 결정하여 실행합니다.
3. **지속성 유지:** OpenClaw 세션은 초기화되지 않습니다. 모든 정보가 로컬 파일로 저장되므로, 채팅창을 닫아도 문맥(Context)은 사라지지 않습니다. 또한 사용자는 복잡한 추론에는 Claude를, 창의적인 작업에는 GPT-4를 사용하는 식으로 모델을 자유롭게 구성할 수 있습니다.

## 지속성 메모리: 모든 것은 파일이다

OpenClaw는 별도의 데이터베이스를 사용하지 않고, 모든 데이터를 마크다운(Markdown) 파일 형태로 `~/clawd/` 디렉토리에 저장합니다.

- `AGENTS.md`: 에이전트의 행동 정의
- `SOUL.md`: 페르소나 및 핵심 지침
- `TOOLS.md`: 사용 가능한 도구 목록
- `~/clawd/skills/`: 설치된 기술들
- `2026-02-10-conversation.md`: 타임스탬프가 찍힌 메모리 로그

모든 데이터가 평문(Plain text)이므로 텍스트 에디터로 쉽게 확인하고 수정할 수 있습니다. 또한 Git을 활용해 변경 사항을 추적하거나 백업하는 것도 매우 간편합니다.

메모리는 단기 문맥을 담는 **일간 로그**와 에이전트가 중요하다고 판단하여 영구적으로 기록하는 **장기 메모리** 계층으로 나뉩니다. 이를 통해 시스템 장애가 발생하더라도 에이전트는 재시작 시 파일에서 상태를 읽어와 작업 중단 지점부터 다시 시작할 수 있습니다.

## 하트비트(Heartbeat): 능동적인 에이전트

대부분의 AI 비서는 사용자의 질문을 기다리지만, OpenClaw는 스스로 움직입니다. 크론(Cron) 작업이 설정된 간격(기본 30분)마다 에이전트를 깨우는 **하트비트(Heartbeat)** 기능을 수행합니다.

에이전트는 `HEARTBEAT.md`의 지침을 확인하고 사고 루프를 실행하여 사용자에게 알릴 내용이 있는지 스스로 판단합니다. 예를 들어, 새벽에 서버가 다운되거나 모니터링 중인 주가가 급락했을 때 에이전트가 먼저 메시지를 보낼 수 있습니다.

### 저비용 확인(Cheap Checks) 우선 원칙

매번 LLM을 호출하면 API 비용이 과다하게 발생하므로, OpenClaw는 2단계 접근 방식을 사용합니다. 먼저 이메일 확인이나 시스템 경고와 같은 간단한 스크립트를 실행합니다. 이 단계에서는 비용이 들지 않습니다. 여기서 유의미한 변화가 감지되었을 때만 LLM을 호출하여 상황을 해석하고 의사결정을 내립니다.

## 기술 및 실행: 기능 확장

OpenClaw는 기능을 컴파일된 코드가 아닌 마크다운 파일로 정의하는 **기술 기반 아키텍처(Skill-based architecture)**를 사용합니다. 새로운 기술을 설치하면 서버 재시작 없이 즉시 실행 가능합니다.

보안을 위해 **샌드박스 모드(Sandboxed mode)**와 **전체 권한 모드(Full access mode)**를 선택할 수 있습니다. 샌드박스 모드에서는 파일 시스템 쓰기나 셸 접근이 차단되지만, 전체 권한 모드에서는 터미널 명령어를 실행하거나 브라우저를 제어할 수 있습니다.

## 보안 및 다중 플랫폼 처리

게이트웨이는 라우팅 계층에서 보안을 강화합니다. 허용 목록(Allow list)에 있는 사용자나 그룹의 메시지만 처리하며, 그 외의 접근은 에이전트에게 도달하기 전에 차단됩니다.

또한 **채널 레이어(Channel Layer)**를 통해 각 플랫폼(왓츠앱, 텔레그램 등)의 서로 다른 메시지 형식을 표준화된 내부 구조로 변환합니다. 덕분에 사용자는 "그룹 채팅에서의 호출(@mention)에만 응답하라"와 같은 규칙을 모든 플랫폼에 동일하게 적용할 수 있습니다.

OpenClaw의 보안 아키텍처는 LLM이 속을 수 있다는 가정하에 다중 방어 계층을 둡니다.
- 위험한 작업(파일 삭제, 결제 등)에 대한 사용자 승인 워크플로우
- 읽기/쓰기 권한 분리
- 장치 토큰별 권한 제한

## 결론

OpenClaw의 아키텍처는 게이트웨이, 에이전트 루프, 파일 기반 메모리, 기술 확장, 그리고 하트비트라는 단순한 요소들의 조합으로 이루어져 있습니다. 데이터베이스나 마이크로서비스 없이도 지속적이고 능동적인 AI 에이전트를 구현했습니다.

이 설계는 편의성보다는 투명성과 통제권을 우선시합니다. 사용자는 에이전트가 무엇을 알고 있는지, 어떤 권한을 가졌는지 명확히 파악할 수 있습니다. 시스템에 대한 깊은 접근 권한을 가진 AI를 운영하는 만큼, 그에 따르는 보안 위험을 이해하고 관리하는 것이 OpenClaw 활용의 핵심입니다.
