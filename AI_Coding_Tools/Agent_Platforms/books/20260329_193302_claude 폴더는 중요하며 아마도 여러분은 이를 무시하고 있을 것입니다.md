**TLDR (요약)**

*   `.claude` 폴더는 Claude Code(클로드 코드) 기능의 약 80%를 제어합니다.
*   구성에는 세 가지 계층이 있으며, 이들은 계단식(cascading)으로 적용됩니다: 정책(Policy, 시스템 전반, IT에서 설정하며 재정의 불가) → 사용자(User, `~/.claude/`, 모든 프로젝트에 걸친 개인 선호 사항) → 프로젝트(Project, 레포지토리(`.claude/` 내`, Git을 통해 팀 공유).
*   `.claude` 폴더 안의 모든 것은 엄격한 '제안'입니다.
*   모든 것을 `~/.claude/CLAUDE.md`에 몰아넣지 마세요. 사용자 규칙(Userules)/스킬(skills)/을 활용하세요.
*   스킬(Skills), 에이전트(agents), 규칙(rules), 명령어(commands)는 사용자 및 프로젝트 레벨 모두에 존재합니다. 기능은 동일하지만 범위가 다릅니다.
*   Claude 머신 로컬 메모리는 여러 머신에 동기화(sync)되지 않습니다.

**본문으로 돌아가서:**

현재 Claude Code를 사용하는 대부분의 사람들은 테슬라(Tesla)를 사고는 담배 라이터만 사용하는 것과 같습니다. 그들은 터미널(terminal)을 엽니다. `claude`를 입력합니다. 프롬프트(prompt)를 붙여넣습니다. 답변을 받습니다. 그리고 다음 작업으로 넘어갑니다.

하지만 중요한 사실은 이렇습니다: 진정한 힘은 프롬프트(prompt)에 있는 것이 아닙니다. 그것은 여러분과 모델(model) 사이에 놓인 구성 계층(configuration layer)에 있습니다. 그 계층이 바로 `.claude` 폴더입니다. 만약 여러분이 이 폴더를 한 번도 열어보거나, 구조화하거나, 그 안에 무엇이 들어가는지 생각해본 적이 없다면, Claude Code 기능의 80%를 활용하지 못하고 있는 것입니다.

이것은 일반적인 "Claude Code 설치 방법" 튜토리얼이 아닙니다. 이 글은 초보자가 처음 사용하는 `CLAUDE.md`부터 경로 범위 규칙(path-scoped rules), 사용자 지정 서브 에이전트(custom subagents), 세션 간 지속적인 메모리(persistent memory)를 갖춘 고급 팀 설정에 이르기까지, 구성 시스템(configuration system)의 전체 해부(anatomy)입니다.

### 정신 모델(Mental Model): 세 가지 계층, 하나의 계층 구조

단 하나의 파일도 만지기 전에, 아키텍처(architecture)를 이해해야 합니다. Claude Code의 구성은 단일한 설정 파일이 아닙니다. 세 가지 명확한 범위(scope)를 가진 계층화된 시스템이며, CSS처럼 계단식으로 적용됩니다. 가장 넓은 범위가 기준선(baseline)을 설정하고, 더 좁은 범위가 이를 구체화하거나 재정의(override)할 수 있습니다.

**정책(Policy)**은 모든 것을 감싸고 있습니다.

이는 조직이 마련한 안전장치(guardrail)입니다. 만약 정책이 "Bash(curl *)를 거부(deny)한다"고 명시한다면, 그것은 최종적입니다. 어떠한 사용자 설정도, 프로젝트 설정도, 그 어떤 것도 이를 재정의할 수 없습니다.

**사용자(User)**는 정책 안에 있습니다. 이는 여러분의 개인 기본 설정(personal defaults)입니다. 코딩 스타일 선호도, 전역 `CLAUDE.md`, 개인 스킬(skills) 및 에이전트(agents)가 여기에 해당합니다. 프로젝트 레벨 설정이 달리 명시하지 않는 한, 여러분의 머신(machine)에 있는 모든 프로젝트에 적용됩니다.

**프로젝트(Project)**는 사용자 안에 있습니다. 이는 가장 구체적인 범위입니다. 팀이 공유하는 규칙, 프로젝트별 권한, 이 코드베이스(codebase)를 위한 사용자 지정 명령어(custom commands)가 여기에 포함됩니다. 프로젝트 설정은 동일한 키(key)에 대해 사용자 설정을 재정의하지만 (정책을 재정의하는 경우는 없습니다).

가장 높은 우선순위부터 가장 낮은 우선순위까지의 전체 우선순위는 다음과 같습니다:

*   **정책 / 관리형(Managed)** (엔터프라이즈 IT(enterprise IT)) — 절대적인 권한, 재정의 불가 (지금은 무시하세요, 아마도 사용할 일이 없을 것입니다)
*   **명령줄 인수(Command line arguments)** (`--model`, `--agent` 등) — 세션별 재정의(session-specific overrides)
*   **로컬 프로젝트(Local project)** (`.claude/settings.local.json`) — 여러분의 머신 전용 프로젝트 재정의
*   **공유 프로젝트(Shared project)** (`.claude/settings.json`) — 팀 공유 구성 (지금은 무시하세요, 아마도 사용할 일이 없을 것입니다)
*   **사용자(User)** (`~/.claude/settings.json`) — 여러분의 전역 기본 설정(global defaults)

두 설정이 충돌하면, 우선순위가 높은 범위가 승리합니다. 예외는 없습니다. 병합(merging)도 없습니다. 그저 승리할 뿐입니다.

이제 가장 바깥쪽부터 가장 안쪽까지 각 계층을 살펴보겠습니다.

### 계층 1: 정책(Policy) (시스템 전반)

대부분의 개별 개발자(individual developers)는 이 계층을 건드릴 일이 없을 것입니다. 하지만 보안 팀(security team), 데브옵스(DevOps), 또는 IT 정책을 가진 조직에서 일한다면, 이곳은 그들이 통제하는 계층입니다.

정책 레벨 구성은 시스템 경로에 존재합니다:

*   macOS: `/Library/Application Support/ClaudeCode/`
*   Linux: `/etc/claude-code/`
*   Windows: `%PROGRAMDATA%\ClaudeCode\`

이 디렉토리(directories) 내부에서 IT는 다음을 배치할 수 있습니다:

*   **`CLAUDE.md`** — 해당 머신의 모든 사용자를 위한 모든 Claude Code 세션에 로드되는 조직 전반의 지침. "모든 코드는 배포 전에 보안 검토(security review)를 통과해야 합니다." "메인(main) 브랜치에 직접 커밋(commit)하지 마세요." 이 지침들은 어떠한 개별 사용자나 프로젝트에 의해서도 제외되거나 재정의될 수 없습니다.
*   **`settings.json`** — 조직 전반의 권한 정책(permission policies). 만약 관리형 정책이 `Bash(curl *)`를 거부한다면, 해당 도구는 모든 곳에서 차단됩니다. 어떠한 프로젝트 `settings.json`도 이를 다시 허용할 수 없습니다.
*   **`managed-settings.json`** — 동일한 목적의 대체 경로로, 종종 MDM(Mobile Device Management) 또는 구성 관리 도구(Puppet, Chef, Ansible)를 통해 배포됩니다.

정책의 핵심은 다음과 같습니다: **항상 로드되고 항상 승리합니다.** 관리형 `CLAUDE.md`는 `claudeMdExcludes`에 의해 제외될 수 없습니다. 관리형 거부(deny) 규칙은 프로젝트 허용(allow) 규칙에 의해 재정의될 수 없습니다. 이것은 의도된 설계입니다. 이는 조직의 안전망(safety net)입니다.

만약 여러분이 개인 개발자(solo developer)이거나 개인 프로젝트를 진행하고 있다면, 이 계층은 존재하지 않으므로 완전히 건너뛸 수 있습니다. 하지만 팀이나 엔터프라이즈(enterprise)를 위해 Claude Code를 설정하고 있다면, 이곳에서 협상 불가능한 표준(non-negotiable standards)을 적용하게 됩니다.

### 계층 2: 사용자(User) (개인/전역)

이것은 여러분의 계층입니다. `~/.claude/` 안의 모든 것은 여러분의 소유이며, 여러분의 머신에 있는 모든 프로젝트에 적용되며, 버전 관리(version control)에 절대 포함되지 않습니다.

여기에 포함되는 내용은 다음과 같습니다:

```
~/.claude/
├── CLAUDE.md              # 여러분의 전역 지침 (모든 프로젝트)
├── settings.json          # 여러분의 전역 설정 및 권한
├── settings.local.json    # 머신-로컬 설정 (동기화되지 않음)
├── rules/                 # 개인 규칙 (모든 프로젝트)
│   ├── preferences.md
│   └── workflows.md
├── commands/              # 개인 슬래시 명령어 (→ /user:command-name)
├── skills/                # 개인 스킬 (모든 프로젝트)
├── agents/                # 개인 에이전트 (모든 프로젝트)
├── agent-memory/          # 영구 에이전트 메모리 (사용자 범위)
└── projects/              # 세션 이력 + 프로젝트별 자동 메모리
    └── <project-hash>/
        └── memory/
            ├── MEMORY.md
            └── ...topic files
```

**`~/.claude/CLAUDE.md` - 여러분의 전역 지침**

이 파일은 여러분이 어떤 프로젝트에 있든 관계없이 모든 Claude Code 세션에 로드됩니다. 이것은 특정 코드베이스(codebase)에 관한 것이 아니라 여러분 자신에 관한 선호 사항을 위한 적절한 장소입니다.

여기에 넣기 좋은 내용:

*   "저는 `npm`보다 `pnpm`을 선호합니다."
*   "항상 클래스 기반(class-based) 패턴보다 함수형(functional) 패턴을 사용하세요."
*   "테스트를 작성할 때는 'arrange-act-assert' 구조를 사용하세요."
*   "저는 시니어 엔지니어(senior engineer)입니다. 기본적인 설명은 건너뛰세요."

여기에 넣기 좋지 않은 내용:

*   프로젝트별 빌드 명령어 (이는 프로젝트 `CLAUDE.md`에 속합니다)
*   팀원들과 공유되어야 하는 모든 것 (이는 프로젝트 레벨 구성에 속합니다)

**`~/.claude/settings.json` - 여러분의 전역 권한**

프로젝트 레벨 `settings.json`과 동일한 구조를 가지지만 (이것은 나중에 다룰 것입니다), 모든 곳에 적용됩니다. 만약 Claude가 항상 묻지 않고 `git status`를 실행할 수 있기를 원한다면, 모든 프로젝트에 넣는 대신 여기에 한 번만 넣으세요.

**`~/.claude/rules/` - 개인 규칙**

여기에 있는 Markdown 파일은 전역 `CLAUDE.md`와 함께 로드됩니다. 프로젝트 레벨 규칙(계층 3에서 다룰 예정)과 동일한 개념이지만, 모든 프로젝트에 걸쳐 여러분에게 범위가 지정됩니다. Claude가 모든 곳에서 따라야 할 개인 코딩 컨벤션(coding conventions)에 유용합니다.

**`~/.claude/commands/`, `skills/`, `agents/` - 개인 도구**

여기에 배치된 슬래시 명령어(slash commands)는 `/user:command-name`으로 나타납니다 (대신 `/project:command-name`이 아님). 여기에 배치된 스킬과 에이전트는 모든 프로젝트에서 사용 가능합니다. 일일 스탠드업 도우미, 여러분이 선호하는 컨벤션을 따르는 커밋 메시지 생성기, 개인 코드 검토 에이전트 등은 사용자 레벨에 속합니다.

**`~/.claude/projects/` - 메모리**

이곳은 Claude가 자신을 위해 메모(notes)를 하는 곳입니다. Claude가 여러분의 프로젝트에서 작업함에 따라, 발견한 빌드 명령어, 디버깅 패턴, 아키텍처(architecture) 노트, 코드 스타일 관찰 등 통찰력(insights)을 자동으로 저장합니다. 미래의 대화에서 정보가 유용할지 여부에 따라 무엇을 기억할 가치가 있는지 결정합니다.

```
~/.claude/projects/<project-hash>/memory/
├── MEMORY.md          # 간결한 색인, 모든 세션에서 로드됨 (처음 200줄)
├── debugging.md       # 디버깅 패턴에 대한 상세 노트
├── api-conventions.md # API 설계 결정
└── ...
```

`MEMORY.md`는 색인(index) 역할을 합니다. 처음 200줄은 모든 세션 시작 시 로드됩니다. 200줄을 넘어가는 내용은 세션 시작 시 로드되지 않습니다. 토픽 파일(Topic files)은 Claude가 필요할 때 요청 시(on demand) 읽습니다.

모든 자동 메모리(auto memory)는 편집하거나 삭제할 수 있는 일반 Markdown 파일입니다. 세션 내에서 `/memory`를 실행하여 탐색할 수 있습니다.

**중요:** 자동 메모리는 머신-로컬(machine-local)입니다. 여러 머신이나 클라우드 환경에 동기화되지 않습니다. 동일한 Git 레포지토리 내의 모든 워크트리(worktrees)와 서브 디렉토리(subdirectories)는 하나의 자동 메모리 디렉토리를 공유합니다. 새 노트북으로 이동하면 처음부터 다시 시작해야 합니다.

### 계층 3: 프로젝트(Project) (레포지토리 레벨)

이것은 대부분의 사람들이 상호작용하는 계층이며, 가장 넓은 표면적(surface area)을 가집니다. 프로젝트의 루트 디렉토리(root directory)에 존재하며, (대부분) Git에 커밋(commit)되어 팀 전체가 공유합니다.

다음은 전체 구조입니다:

```
your-project/
├── CLAUDE.md                      # 팀 지침 (커밋됨)
├── CLAUDE.local.md                # 개인 재정의 (gitignored)
│
└── .claude/
    ├── settings.json              # 권한 + 구성 (커밋됨)
    ├── settings.local.json        # 개인 권한 재정의 (gitignored)
    │
    ├── rules/                     # 모듈화된 지침 파일
    │   ├── code-style.md
    │   ├── testing.md
    │   ├── api-conventions.md
    │   └── frontend/
    │       └── react-patterns.md  # 경로 범위 규칙(Path-scoped rules) 지원
    │
    ├── skills/                    # 자동 호출 또는 슬래시 명령어 워크플로
    │   ├── security-review/
    │   │   ├── SKILL.md
    │   │   └── DETAILED_GUIDE.md
    │   └── deploy/
    │       ├── SKILL.md
    │       └── templates/
    │           └── release-notes.md
    │
    ├── commands/                   # 슬래시 명령어 (여전히 작동하며, 스킬과 병합됨)
    │   ├── review.md              # → /project:review
    │   └── fix-issue.md           # → /project:fix-issue
    │
    └── agents/                    # 전문 서브 에이전트 페르소나
        ├── code-reviewer.md
        └── security-auditor.md
```

패턴을 주목하세요: 일부 파일은 커밋되어 (팀과 공유) 있고 일부는 `.gitignore` 처리되어 (개인 전용) 있습니다. 이는 의도된 것이며 사용자(User) 대 프로젝트(Project)의 구분을 더 세밀하게 반영합니다. 파일이 커밋되었나요? >> 네, 전체 팀이 CLAUDE.local.md 변경 사항과 코더의 동작을 봅니다. 아니요 >> 그러면 그것은 여러분만을 위한 것입니다. 이 프로젝트에만 영향을 미치고 싶다면 `.claude/settings.json`을 사용하세요.

각 부분을 자세히 살펴보겠습니다.

**`CLAUDE.md`: AI를 위해 작성할 가장 중요한 파일**

Claude Code 세션을 시작하면, 가장 먼저 `CLAUDE.md`를 읽습니다. 그 내용을 컨텍스트 창(context window)에 직접 로드하고 대화 내내 이를 염두에 둡니다. `CLAUDE.md`에 무엇을 쓰든, Claude는 따를 것입니다. 만약 구현 전에 항상 테스트를 작성하라고 지시하면, 그렇게 할 것입니다. "오류 처리에는 `console.log`를 절대 사용하지 말고, 항상 사용자 지정 로거 모듈(custom logger module)을 사용하세요"라고 말하면, 매번 이를 존중할 것입니다.

하지만 잠깐. 이것은 강제적인 구성(enforced configuration)이 아닙니다. 이것은 컨텍스트(context)입니다. Claude는 이를 읽고 따르려 노력하지만, 특히 모호하거나 충돌하는 지침의 경우 엄격한 준수(strict compliance)를 보장하지는 않습니다. 지침이 더 구체적이고 간결할수록, Claude는 더욱 일관적으로 따릅니다.

`CLAUDE.md`는 여러 범위에 존재할 수 있습니다 (계층을 기억하세요):

| 범위       | 위치                                                         | 로드 시기                                         |
| :--------- | :----------------------------------------------------------- | :------------------------------------------------ |
| **정책**   | `/Library/Application Support/ClaudeCode/CLAUDE.md` (macOS), `/etc/claude-code/CLAUDE.md` (Linux) | 항상. 제외될 수 없음.                             |
| **사용자** | `~/.claude/CLAUDE.md`                                        | 모든 세션, 모든 프로젝트                           |
| **프로젝트** | `./CLAUDE.md` 또는 `./.claude/CLAUDE.md`                     | 이 프로젝트에서 작업할 때                          |
| **하위 디렉토리** | `./src/api/CLAUDE.md`                                        | Claude가 해당 디렉토리의 파일을 읽을 때 요청 시(on demand) |

Claude Code는 현재 작업 디렉토리에서 디렉토리 트리(directory tree)를 따라 위로 올라가면서 `CLAUDE.md`를 읽습니다. 만약 `foo/bar/`에서 Claude Code를 실행하면, `foo/bar/CLAUDE.md`와 `foo/CLAUDE.md`의 지침을 모두 로드합니다. 이 모든 것이 결합됩니다. 정책 + 사용자 + 프로젝트 + 하위 디렉토리가 모두 계층화되어 함께 적용됩니다.

**`CLAUDE.md`에 실제로 속하는 내용**

대부분의 사람들은 너무 많이 쓰거나 너무 적게 씁니다. 다음은 효과적인 방법입니다.

작성할 내용:

*   빌드(Build), 테스트(test), 린트(lint) 명령어 (`npm run test`, `make build` 등)
*   주요 아키텍처 결정 ("우리는 Turborepo를 사용하는 모노레포(monorepo)를 사용합니다")
*   명확하지 않은 함정 ("TypeScript 엄격 모드(strict mode)가 켜져 있어, 사용되지 않는 변수는 오류입니다")
*   임포트(Import) 컨벤션, 이름 지정 패턴, 오류 처리 스타일
*   주요 모듈의 파일 및 폴더 구조

작성하지 말아야 할 내용:

*   린터(linter) 또는 포맷터(formatter) 구성에 속하는 모든 것
*   이미 링크할 수 있는 전체 문서
*   이론을 설명하는 긴 단락

`CLAUDE.md` 파일당 200줄 미만을 목표로 하세요. 이것은 공식 문서의 권장 사항이며, 엄격한 제한은 아닙니다. Claude는 길이에 관계없이 전체 파일을 로드합니다. 200줄에서 잘리는 것은 없습니다. 하지만 파일이 길어질수록 더 많은 컨텍스트(context)를 소비하고 준수도(adherence)는 떨어집니다. 500줄짜리 `CLAUDE.md`는 완전히 로드되겠지만, 집중된 100줄짜리 파일만큼 신뢰성 있게 따르지는 않을 것입니다. 이 200줄 제한은 `MEMORY.md`에만 적용됩니다. `CLAUDE.md` 파일은 길이에 관계없이 완전히 로드되지만, 짧은 파일이 더 나은 준수도를 나타냅니다.

다음은 최소한이지만 효과적인 예시입니다:

```markdown
# Project: Acme API

## Commands
npm run dev     # 개발 서버 시작
npm run test    # 테스트 실행 (Jest)
npm run lint    # ESLint + Prettier 검사
npm run build   # 프로덕션 빌드

## Architecture
- Express REST API, Node 20
- Prisma ORM을 통한 PostgreSQL
- 모든 핸들러는 src/handlers/에 있습니다.
- 공유 타입은 src/types/에 있습니다.

## Conventions
- 모든 핸들러에서 요청 유효성 검증(request validation)에 zod를 사용하세요.
- 반환 형식은 항상 `{ data, error }`입니다.
- 클라이언트에게 스택 트레이스(stack traces)를 절대 노출하지 마세요.
- `console.log` 대신 로거 모듈(logger module)을 사용하세요.

## Watch out for
- 테스트는 실제 로컬 DB를 사용하며, 목(mock)이 아닙니다. 먼저 `npm run db:test:reset`을 실행하세요.
- 엄격한 TypeScript: 사용되지 않는 임포트(imports)는 절대 허용되지 않습니다.
```

이것은 약 20줄입니다. Claude가 끊임없는 설명 없이 생산적으로 작업하는 데 필요한 모든 것을 제공합니다.

**`CLAUDE.local.md`: 개인 재정의 (강력함)**

때로는 팀 전체가 아닌 여러분에게만 특별한 선호 사항이 있을 수 있습니다. 예를 들어, 다른 테스트 러너(test runner)를 선호하거나, Claude가 항상 특정 패턴을 사용하여 파일을 열도록 하고 싶을 수 있습니다.

프로젝트 루트에 `CLAUDE.local.md`를 생성하세요. Claude는 이를 메인 `CLAUDE.md`와 함께 읽으며, 자동으로 `.gitignore` 처리되므로 여러분의 개인적인 변경 사항이 레포지토리(repo)에 들어가지 않습니다.

**가져오기 시스템(The Import System): `@path` 구문**

`CLAUDE.md` 파일은 `@path/to/import` 구문을 사용하여 추가 파일을 가져올 수 있습니다. 가져온 파일은 시작 시 참조하는 `CLAUDE.md`와 함께 컨텍스트에 확장되어 로드됩니다. 상대 경로와 절대 경로 모두 작동합니다. 가져온 파일은 최대 5단계까지 재귀적으로 다른 파일을 가져올 수 있습니다.

```markdown
@README 프로젝트 개요 및 @package.json에서 사용 가능한 npm 명령어를 확인하세요.

# 추가 지침
- git 워크플로 @docs/git-instructions.md
```

체크인하고 싶지 않은 개인 선호 사항의 경우, 홈 디렉토리에서 파일을 가져오세요:

```markdown
# 개인 선호 사항
- @~/.claude/my-project-instructions.md
```

Claude Code가 외부 가져오기를 처음 만날 때, 승인 대화 상자(approval dialog)가 표시됩니다. 거부하면 가져오기는 비활성화된 상태로 유지됩니다.

**`rules/` 폴더: `CLAUDE.md`가 너무 커질 때**

`CLAUDE.md`는 단일 프로젝트에 잘 작동합니다. 하지만 팀이 성장하거나 프로젝트가 복잡해지면, 아무도 유지 보수하지 않고 모두가 무시하는 300줄짜리 `CLAUDE.md`가 됩니다.

`.claude/rules/` 폴더가 이를 해결합니다.

`.claude/rules/` 내부의 모든 Markdown 파일은 `CLAUDE.md`와 함께 자동으로 로드됩니다. 하나의 거대한 파일 대신, 관심사별로 지침을 분할합니다:

```
.claude/rules/
├── code-style.md
├── testing.md
├── api-conventions.md
└── security.md
```

각 파일은 집중되어 있고 업데이트하기 쉽습니다. API 컨벤션을 담당하는 팀원은 `api-conventions.md`를 편집합니다. 테스트 표준을 담당하는 사람은 `testing.md`를 편집합니다. 서로의 작업을 방해하지 않습니다.

**경로 범위 규칙(Path-Scoped Rules): 진정한 파워 무브**

여기서 흥미로운 부분이 시작됩니다. 규칙 파일에 YAML 프론트매터(YAML frontmatter)를 추가하면, Claude가 일치하는 파일과 작업할 때만 활성화됩니다:

```yaml
---
paths:
  - "src/api/**/*.ts"
  - "src/handlers/**/*.ts"
---

# API 설계 규칙
- 모든 핸들러는 `{ data, error }` 형식을 반환합니다.
- 요청 본문 유효성 검증에 zod를 사용하세요.
- 내부 오류 세부 정보를 클라이언트에게 절대 노출하지 마세요.
```

Claude는 React 컴포넌트(component)를 편집할 때는 이 파일을 로드하지 않습니다. `src/api/` 또는 `src/handlers/` 내부에서 작업할 때만 로드됩니다. `paths` 필드가 없는 규칙은 모든 세션에서 무조건 로드됩니다.

`CLAUDE.md`가 너무 복잡하게 느껴질 때 이것이 올바른 패턴입니다. 그리고 계층을 기억하세요: 사용자 레벨 규칙도 `~/.claude/rules/`에 존재할 수 있으며, 여러분의 머신에 있는 모든 프로젝트에 적용됩니다.

**`settings.json`: 권한 및 프로젝트 구성**

`CLAUDE.md`는 지침과 컨텍스트를 다루는 반면, `settings.json`은 Claude의 동작을 프로그래밍 방식으로 제어할 수 있게 해줍니다. 이는 권한, 환경 변수(environment variables), 도구 접근을 관리합니다.

```json
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git status)",
      "Bash(git diff *)",
      "Read",
      "Write",
      "Edit"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(curl *)",
      "Read(./.env)",
      "Read(./.env.*)"
    ]
  }
}
```

`$schema` 라인은 VS Code 또는 Cursor에서 자동 완성(autocomplete) 및 인라인 유효성 검사(inline validation)를 가능하게 합니다. 항상 포함해야 합니다.

`allow` 목록에는 Claude가 확인을 요청하지 않고 실행되는 명령어가 포함됩니다. `deny` 목록에는 완전히 차단되는 명령어가 포함되며, 어떤 경우에도 마찬가지입니다. 둘 중 어느 목록에도 없는 것이 있다면, Claude는 진행하기 전에 질문합니다. 이 중간 지점은 의도된 것입니다.

그리고 `CLAUDE.md`와 마찬가지로, 커밋되지 않는 개인 권한 재정의를 위한 `settings.local.json`이 있습니다. 자동으로 `.gitignore` 처리됩니다.

여기서 계층화가 가장 중요하게 작용합니다. 권한에 대한 우선순위는 다음과 같습니다:

**정책 거부 >> 프로젝트 거부 >> 사용자 거부 >> 프로젝트 허용 >> 사용자 허용**

더 높은 범위에서의 거부는 더 낮은 범위에서의 허용에 의해 재정의될 수 없습니다. 만약 정책이 `Bash(curl *)`를 거부한다면, 여러분 프로젝트의 `Bash(curl *)`에 대한 허용 목록은 조용히 무시됩니다. 이것은 의도대로 작동하는 안전 아키텍처(safety architecture)입니다.

**스킬(Skills): 승진한 명령어**

여기서 역사가 흥미로워집니다. Claude Code는 이전에 "명령어" (`.claude/commands/*.md`)와 "스킬" (`.claude/skills/*/SKILL.md`)이라는 별도의 개념을 가지고 있었습니다. 이들은 병합되었습니다. `.claude/commands/deploy.md`에 있는 파일과 `.claude/skills/deploy/SKILL.md`에 있는 스킬은 모두 `/deploy`를 생성하며 동일하게 작동합니다. 기존 `.claude/commands/` 파일도 계속 작동합니다.

하지만 스킬은 명령어가 가지고 있지 않던 선택적 기능(optional features)을 추가합니다:

*   지원 파일(templates, scripts, reference docs)을 위한 디렉토리(directory)
*   사용자 또는 Claude가 스킬을 호출하는지 제어하는 YAML 프론트매터(YAML front-matter)
*   Claude가 관련 있을 때 자동으로 스킬을 로드하는 기능

**결정적인 차이: 누가 실행을 트리거(Trigger)하는가**

이것이 중요한 차이입니다. 두 개의 프론트매터 필드가 호출(invocation)을 제어합니다:

(전체 크기로 이미지를 보려면 Enter를 누르거나 클릭하세요)
두 개의 프론트매터 필드가 사용자, Claude, 또는 둘 다 스킬을 호출할 수 있는지 제어합니다 (Claude HTML로 생성됨)

배포(deploy) 스킬은 `disable-model-invocation: true`여야 합니다. 코드가 준비되었다고 Claude가 배포를 결정하는 것을 원하지 않을 것입니다. 레거시 시스템 컨텍스트(legacy-system-context) 스킬은 `user-invocable: false`여야 합니다. Claude는 그 지식이 필요하지만, `/legacy-system-context`는 사용자에게 의미 있는 동작이 아닙니다.

**계층을 가로지르는 스킬 범위 지정(Skill Scoping)**

스킬은 동일한 세 계층 모델을 따릅니다:

*   `.claude/skills/`의 프로젝트 스킬 → `/project:skill-name`으로 표시
*   `~/.claude/skills/`의 사용자 스킬 → `/user:skill-name`으로 표시
*   마켓플레이스를 통해 설치된 플러그인 스킬(Plugin skills) → `/plugin-name:skill-name`으로 표시

동일한 계층 구조입니다. 프로젝트는 가장 구체적이며, 사용자는 개인 도구 키트이고, 플러그인은 외부 패키지(external packages)입니다.

**동적 컨텍스트 주입(Dynamic Context Injection)**

`!`<command>`` 구문은 스킬 내용이 Claude에게 전송되기 전에 셸 명령어(shell commands)를 실행합니다. 명령어 출력은 플레이스홀더(placeholder)를 대체합니다:

```yaml
---
name: pr-summary
description: 풀 리퀘스트(pull request)의 변경 사항 요약
context: fork
---

## 풀 리퀘스트 컨텍스트
- PR diff: !`gh pr diff`
- PR comments: !`gh pr view --comments`

이 풀 리퀘스트를 요약...
```

각 `!`<command>``는 즉시 실행되며 (Claude가 아무것도 보기 전에), 출력은 플레이스홀더를 대체하고, Claude는 실제 데이터가 완전히 렌더링(rendered)된 프롬프트(prompt)를 받습니다. 이것은 전처리(preprocessing)이며, Claude가 실행하는 것이 아닙니다.

### 에이전트(Agents): 전문 서브 에이전트 페르소나

작업이 전담 전문가(dedicated specialist)의 도움이 필요할 만큼 충분히 복잡할 때, `.claude/agents/`에 서브 에이전트(subagent)를 정의합니다. 각 에이전트는 자체 시스템 프롬프트(system prompt), 도구 접근, 모델 선호도(model preference)를 가진 Markdown 파일입니다:

```yaml
---
name: code-reviewer
description: 전문 코드 검토자. PR 검토 시 적극적으로(PROACTIVELY) 사용하세요.
model: sonnet
tools: Read, Grep, Glob, Bash
memory: project
---

당신은 정확성과 유지 보수성(maintainability)에 중점을 둔 시니어 코드 검토자입니다.

코드를 검토할 때:
- 스타일 문제뿐만 아니라 버그(bugs)를 표시하세요.
- 모호한 개선 사항 대신 구체적인 수정 사항을 제안하세요.
- 엣지 케이스(edge cases) 및 오류 처리(error handling)의 누락을 확인하세요.
```

Claude가 설명과 일치하는 작업을 만나면, 자체적으로 격리된 컨텍스트 창(isolated context window)에서 이 에이전트를 생성합니다. 에이전트는 독립적으로 작동하고 결과를 반환합니다. 여러분의 메인 세션은 수천 개의 토큰으로 구성된 중간 탐색으로 복잡해지지 않습니다.

주요 구성 옵션:

*   **`tools`**: 에이전트가 할 수 있는 것을 제한합니다. 보안 감사자(security auditor)는 `Read`, `Grep`, `Glob`만 필요합니다.
*   **`model`**: 집중된 작업에 더 저렴하고 빠른 모델을 사용합니다. `Haiku`는 읽기 전용 탐색에 잘 작동합니다.
*   **`memory`**: 세션 간 지속적인 학습을 위한 `user`, `project`, 또는 `local` 범위.
*   **`isolation: worktree`**: 병렬 작업을 위해 임시 Git 워크트리(worktree)에서 에이전트를 실행합니다.
*   **`context: fork` (스킬 내에서)**: 포크된 서브 에이전트 컨텍스트 내에서 스킬을 실행합니다.
*   **`hooks`**: 이 에이전트에 범위가 지정된 생명주기 훅(Lifecycle hooks) (`PreToolUse`, `PostToolUse`, `Stop`).

그리고 그렇습니다, 에이전트도 동일한 계층화를 따릅니다: `.claude/agents/`의 프로젝트 에이전트, `~/.claude/agents/`의 개인 에이전트. 프로젝트 에이전트는 팀과 공유됩니다. 개인 에이전트는 어디든 여러분과 함께합니다.

### 전체 그림: 세 가지 계층 모두 함께

이제 각 계층을 개별적으로 이해했으니, 모든 것이 어떻게 함께 작동하는지 살펴보겠습니다:

**계층 1: 정책(POLICY) (시스템 전반)**
`/Library/Application Support/ClaudeCode/` (macOS)
`/etc/claude-code/` (Linux)
```
├── CLAUDE.md              # 조직 전반 지침 (제외 불가)
├── settings.json          # 조직 전반 권한 (재정의 불가)
└── managed-settings.json  # MDM 배포를 위한 대체 경로
```
↓ 정책이 항상 승리합니다 ↓

**계층 2: 사용자(USER) (개인/전역)**
`~/.claude/`
```
├── CLAUDE.md              # 여러분의 전역 지침
├── settings.json          # 여러분의 전역 권한
├── settings.local.json    # 머신-로컬 (동기화되지 않음)
├── rules/                 # 개인 규칙 (모든 프로젝트)
├── commands/              # → /user:command-name
├── skills/                # 개인 스킬 (모든 프로젝트)
├── agents/                # 개인 에이전트 (모든 프로젝트)
├── agent-memory/          # 영구 에이전트 메모리
└── projects/              # 프로젝트별 자동 메모리
    └── <project>/
        └── memory/
            └── MEMORY.md
```
↓ 사용자 기본값이 프로젝트에 의해 재정의됩니다 ↓

**계층 3: 프로젝트(PROJECT) (레포지토리)**
`your-project/`
```
├── CLAUDE.md              # 팀 지침 (커밋됨)
├── CLAUDE.local.md        # 여러분의 개인 재정의 (gitignored)
└── .claude/
    ├── settings.json      # 팀 권한 (커밋됨)
    ├── settings.local.json# 여러분의 권한 재정의 (gitignored)
    ├── rules/             # 모듈화된 지침 파일
    ├── skills/            # → /project:skill-name
    ├── commands/          # → /project:command-name (레거시, 여전히 작동)
    └── agents/            # 팀 서브 에이전트 페르소나
```

모든 기능은 여러 계층에 존재합니다. `CLAUDE.md`, `settings.json`, 규칙, 스킬, 에이전트; 이들은 모두 사용자 레벨(user-level) 및 프로젝트 레벨(project-level) 변형을 가집니다. 그리고 정책(Policy)이 모든 것을 감싸고 있습니다. 이를 내면화하면 전체 시스템은 예측 가능해집니다.

### 실제 패턴: 사용자 계층을 통한 재사용 가능한 지식 분할

이제 세 계층을 이해했으니, 여러 프로젝트를 구축하기 시작하는 순간 생기는 질문은 "반복되는 내용은 어디에 두어야 하는가?"입니다.

만약 여러분이 아이디어부터 MVP(Minimum Viable Product)까지의 프로젝트를 정기적으로 출시하는 사람이라면, 모든 프로젝트에 적용되는 자신만의 의견을 가지고 있을 것입니다. 선호하는 스택(stack). 모노레포(monorepo) 대 멀티레포(multi-repo). `.env` 대 `.env.production`에 무엇을 넣을지. 배포 플랫폼. 항상 존재해야 하는 개발 명령어. 개발 철학. 프로젝트 구조.

본능적으로 이 모든 것을 `~/.claude/CLAUDE.md`에 몰아넣으려고 할 것입니다. 하나의 파일, 전역(global), 모든 곳에 로드. 끝.

그렇게 하지 마세요.

이유는 다음과 같습니다. `CLAUDE.md`에는 엄격한 줄 제한이 없습니다. 길이가 아무리 길어도 완전히 로드됩니다. 하지만 공식 문서에서는 준수도(adherence)가 길이가 길어질수록 떨어지기 때문에 200줄 미만을 목표로 할 것을 권장합니다.

여러분의 스택 선호도, 환경 변수 컨벤션, 배포 규칙, 그리고 전체 스캐폴딩(scaffolding) 플레이북은 쉽게 300줄 이상이 될 수 있습니다. Claude는 이 모든 것을 읽을 것입니다. 다만, 그 모든 것을 신뢰성 있게 따르지는 않을 것입니다. 그리고 그 비대한 컨텍스트(bloated context)는 여러분이 6개월 전에 출시한 프로젝트에서 변수 이름을 바꾸거나 오타를 수정하도록 Claude에게 요청할 때조차 모든 세션에 로드됩니다. 이것은 Claude가 실제 작업을 생각해야 할 창을 잠식하는 낭비되는 컨텍스트입니다.

올바른 방법은 이 지식을 얼마나 자주 필요한지에 따라 분할하는 것입니다:

(전체 크기로 이미지를 보려면 Enter를 누르거나 클릭하세요)
재사용 가능한 지식을 얼마나 자주 필요한지에 따라 분할하기 (Claude HTML로 생성됨)

**`~/.claude/CLAUDE.md` — 신분증(The Identity Card)**

짧고 명확합니다. 50줄 미만. 이것은 여러분이 무엇을 하든 변하지 않는 사실입니다:

```markdown
# 전역 선호 사항

## 타협 불가 사항
- 항상 TypeScript 엄격 모드(strict mode)
- 프로젝트가 진정으로 단일 목적이 아니라면 모노레포(Turborepo)
- npm이나 yarn 대신 pnpm
- 클래스 기반보다 함수형 패턴
- 저는 시니어 엔지니어(senior engineer)입니다; 기본적인 설명은 건너뛰세요.

## 스택 기본값
- 프론트엔드: Next.js (App Router) + Tailwind + shadcn/ui
- 백엔드: MVP(Minimum Viable Product)를 위한 Next.js API 라우트(API routes), 이를 초과하면 별도 서비스
- DB: Prisma를 통한 Postgres (MVP용 Supabase, 프로덕션용 전용 DB)
- 인증: 복잡성에 따라 Clerk 또는 NextAuth
- 배포: 앱용 Vercel, 엣지/워커(edge/workers)용 Cloudflare

## 철학
- 대충 만들고(Ship ugly), 빠르게 출시하고(ship fast), 성과가 있으면 리팩토링(refactor)
- 모든 환경 변수(env var)는 명확한 소유자(앱, 인프라, 또는 비밀)를 가집니다.
- 배포에 2개 이상의 명령어가 필요하면 개발자 경험(DX)이 망가진 것입니다.
```

약 25줄입니다. Claude는 이 내용을 모든 세션에서 읽습니다. Claude는 여러분이 누구인지 압니다. 여러분의 기본값을 압니다. 다시 말할 필요가 없습니다.

**`~/.claude/rules/` — 항상 적용되는 컨벤션**

이 파일들은 전역 `CLAUDE.md`와 함께 자동으로 로드됩니다. 각 관심사가 집중되고 유지 보수하기 쉽도록 분리되어 있으며, `CLAUDE.md`가 간결하게 유지됩니다.

`~/.claude/rules/env-conventions.md`:

```markdown
# 환경 변수 컨벤션

## 파일 구조
- .env.local        → 개발자 머신 비밀 (gitignored)
- .env.development  → 비-비밀 개발 기본값 (committed)
- .env.production   → 비-비밀 프로덕션 기본값 (committed)
- .env.example      → 모든 키를 포함한 템플릿, 값 없음 (committed)

## 이름 지정 규칙
- 스코프(scope) 접두사: 클라이언트용 NEXT_PUBLIC_, 서버 전용은 접두사 없음
- 데이터베이스: DATABASE_URL, DIRECT_URL (Prisma용)
- 인증: AUTH_SECRET, AUTH_[PROVIDER]_ID, AUTH_[PROVIDER]_SECRET
- 외부 API: [SERVICE]_API_KEY, [SERVICE]_BASE_URL
- 기능 플래그: FEATURE_[NAME]=true|false

## 커밋된 환경 변수 파일에 절대 포함되지 않는 것
- API 키, 토큰, 모든 종류의 비밀
- 자격 증명이 포함된 데이터베이스 연결 문자열
- 팀원 머신마다 다른 모든 것
```

`~/.claude/rules/deployment.md`:

```markdown
# 배포 컨벤션

## 기본: Vercel
- 엣지-헤비(edge-heavy)가 아니라면 모든 Next.js 앱에 Vercel 사용
- vercel.json은 필요할 때만 (rewrites, headers)
- 모든 PR에 대한 미리 보기 배포(Preview deployments)
- 메인 브랜치에서만 프로덕션 배포(Production deploys)

## Cloudflare를 사용할 때
- 다음의 경우 Cloudflare Workers/Pages 사용: 엣지 함수(edge functions), R2 스토리지(storage) 필요,
  콜드 스타트(cold start)가 중요한 프로젝트, 비용에 민감한 고트래픽 앱
- 로컬 개발에는 wrangler 사용: wrangler dev

## 모든 프로젝트에 존재해야 하는 명령어
- pnpm dev          → 로컬 개발
- pnpm build        → 프로덕션 빌드 (배포 전에 통과해야 함)
- pnpm lint         → eslint + prettier
- pnpm test         → vitest
- pnpm db:push      → 개발 DB에 스키마 푸시
- pnpm db:migrate   → 마이그레이션 실행 (프로덕션)
- pnpm db:studio    → Prisma Studio 열기
```

각각 약 25~30줄입니다. 모든 세션에 로드됩니다. `CLAUDE.md`와 결합하면, 항상 활성화되는 총 비용은 약 80줄입니다. 예산 범위 내에 편안하게 들어갑니다. 그리고 파일이 분리되어 있기 때문에, 환경 변수 이름 지정 규칙을 건드리지 않고 배포 컨벤션을 업데이트할 수 있습니다.

**`~/.claude/skills/mvp-scaffold/` - 무거운 작업, 요청 시(On Demand)**

이것이 핵심 통찰력입니다. 전체 스캐폴딩(scaffolding) 플레이북; 상세한 모노레포 레이아웃, 환경 변수 템플릿, 배포 체크리스트, 단계별 설정; 이것들은 프로덕션 버그(production bug)를 디버깅할 때 로드될 필요가 없습니다. 새로운 것을 시작할 때만 중요합니다.

**이 분할이 작동하는 이유**

전체 패턴은 세 가지 지식 유형에 깔끔하게 매핑됩니다:

*   **정체성 (여러분이 누구인지)** → `~/.claude/CLAUDE.md` → 항상 로드됨, 매우 저렴
*   **컨벤션 (일을 처리하는 방법)** → `~/.claude/rules/` → 항상 로드됨, 관심사별 분리, 여전히 저렴
*   **플레이북 (필요할 때 무엇을 해야 하는지)** → `~/.claude/skills/` → 요청 시 로드됨, 사용하지 않을 때는 비용 없음

항상 활성화되는 총 컨텍스트: 약 80줄. 전체 스캐폴딩 기능: 요청 시 200줄 이상 사용 가능. 모든 세션은 정체성과 컨벤션을 얻습니다. 플레이북이 필요하지 않는 한, 어떤 세션도 플레이북 비용을 지불하지 않습니다.

이것이 패턴입니다. 일단 이 패턴을 이해하면, `~/.claude/CLAUDE.md`를 잡동사니 서랍으로 취급하는 대신, 전체 `~/.claude/` 디렉토리를 적절한 지식 아키텍처(knowledge architecture)로 취급하기 시작할 것입니다.

### Claude Code 대 Cowork: 같은 엔진, 다른 조종석(Cockpit)

여기서 미묘한 차이가 발생합니다. Claude Code와 Cowork는 모두 동일한 Claude 에이전트 SDK(Agent SDK)를 기반으로 구축되었습니다. 동일한 모델. 동일한 도구 기능. 동일한 기본 아키텍처. 그러나 인터페이스(interfaces)가 근본적으로 다르며, 이것이 `.claude`가 실제로 작동하는 방식에 변화를 줍니다.

**Claude Code**는 터미널 우선(terminal-first) 경험입니다. 여러분은 셸(shell)에 있으며, `.claude/` 구성에 완전히 접근할 수 있고, `CLAUDE.md` 파일을 작성하고, 사용자 지정 슬래시 명령어(custom slash commands)를 생성하고, 서브 에이전트(subagents)를 정의합니다. 철저히 개발자 도구(developer tooling)입니다.

**Cowork**는 데스크톱 GUI(Graphical User Interface) 경험입니다. 터미널을 전혀 건드리지 않을 수도 있는 지식 작업자(knowledge workers)를 위해 설계되었습니다. 사용자 컴퓨터의 샌드박스(sandboxed environment) 환경에서 실행되며, 작업 공간 폴더(workspace folder)에 접근하여 파일을 생성하고, 코드를 실행하고, 작업을 자동화할 수 있습니다.

**`.claude`가 다르게 동작하는 방식**

Claude Code에서 세 계층 모델은 완전히 노출됩니다:

*   프로젝트 내의 `.claude/` 파일을 수동으로 생성하고 편집합니다.
*   `~/.claude/`는 직접 접근 가능한 개인 전역 구성입니다.
*   코드베이스에서 `CLAUDE.md`를 부트스트랩(bootstrap)하기 위해 `/init`을 실행합니다.
*   설정 우선순위가 명시적이며, 모든 계층을 제어합니다.
*   자동 메모리는 `~/.claude/projects/<project>/memory/`에 지속됩니다.

Cowork에서는 동일한 계층이 존재하지만 추상화되어 있습니다: Cowork가 Claude Code와 동일한 규칙 세트를 따르기를 바랍니다. Anthropic이 아직 이를 처리할 실질적인 방법을 제공하지 않은 것이 짜증 납니다.

*   스킬과 플러그인(plugins)은 Cowork 인터페이스를 통해 관리되며, 종종 직접 작성된 `SKILL.md` 파일 대신 플러그인 마켓플레이스에서 설치됩니다.
*   `.claude/skills/` 폴더 개념은 여전히 적용되지만, 스킬은 일반적으로 `SKILL.md` 파일, MCP 커넥터(connectors), 지원 리소스와 함께 플러그인으로 번들링(bundled)됩니다.
*   Cowork 세션은 샌드박스 VM(Virtual Machine)에서 실행됩니다; 해당 VM 내의 `.claude` 디렉토리 구조는 세션 범위(session-scoped)입니다.
*   사용자 레벨 구성은 세션 간에 지속되지만, 메커니즘은 추상화되어 있습니다.
*   Cowork는 예약된 작업(scheduled tasks), MCP 커넥터(Notion, Slack 등), 그리고 Claude Code가 명시적인 MCP 서버 구성을 통해 처리하는 브라우저 자동화(browser automation)를 지원합니다.

*   **지식 작업, 문서 처리, 연구 또는 비-코드 작업을 한다면:** Cowork는 구성 파일을 수동으로 관리할 필요 없이 동일한 기능을 제공합니다.
*   **둘 다 사용한다면 (많은 파워 유저들이 그렇습니다):** 여러분의 레포지토리(repos)에 있는 프로젝트 레벨 `.claude/` 구성이 Claude Code에 직접 피드되는 반면, Cowork는 동일한 개념(SKILL.md, 지원 파일, MCP 구성)을 배포 가능한 번들(distributable bundles)로 패키징하는 설치된 플러그인과 스킬에 더 많이 의존한다는 점을 이해해야 합니다.

### 한계점(The Limitations)

완벽한 것은 없습니다. `.claude` 시스템도 마찬가지입니다. 다음은 여러분이 알아야 할 사항입니다:

*   **`CLAUDE.md`는 컨텍스트이지 강제가 아닙니다.** Claude는 이를 읽고 따르려 노력합니다. 하지만 시스템 프롬프트(system prompt)의 일부가 아니라 사용자 메시지(user message)로 전달됩니다. 엄격한 준수(strict compliance)를 보장하지는 않습니다. 모호한 지침은 모호한 준수를 낳습니다.
*   **200줄 초과 메모리 비대화.** `MEMORY.md`의 처음 200줄은 모든 대화 시작 시 로드됩니다. 200줄을 넘어가는 내용은 세션 시작 시 로드되지 않습니다. Claude는 상세한 노트를 별도의 토픽 파일로 이동시켜 `MEMORY.md`를 간결하게 유지합니다. 이 200줄 제한은 `MEMORY.md`에만 적용됩니다. `CLAUDE.md` 파일은 길이에 관계없이 완전히 로드되지만, 짧은 파일이 더 나은 준수도를 나타냅니다.
*   **`CLAUDE.md`는 대화 메모리에 보존되지 않습니다.** `/compact` 후 `CLAUDE.md`는 디스크에서 다시 읽혀져 재주입됩니다. 하지만 `CLAUDE.md`가 아닌 대화에서만 Claude에게 지침을 주었다면, 압축(compaction) 후에는 사라집니다. 기록하거나 잃어버리세요.
*   **여러 `CLAUDE.md` 파일 간의 충돌하는 지침.** 두 개의 `CLAUDE.md` 파일이 동일한 동작에 대해 다른 지침을 제공하면, Claude는 임의로 하나를 선택할 수 있습니다. 모노레포(monorepos)에서는 이것이 실제 문제가 됩니다. 관련 없는 팀의 `CLAUDE.md` 파일을 건너뛰기 위해 `settings.local.json`에서 `claudeMdExcludes`를 사용하세요. 저를 믿으세요, 이런 일은 누구에게나 일어날 것입니다. 그렇습니다, 이런 일이 발생했을 때 고통스럽고 Claude는 이를 쉽게 발견할 수 있도록 만들지도 않았습니다.
*   **스킬에는 문자 예산이 있습니다.** 스킬 설명은 Claude가 무엇을 사용할 수 있는지 알도록 컨텍스트에 로드됩니다. 스킬이 많으면 문자 예산(컨텍스트 창의 2%, 16,000자 폴백(fallback) 포함)을 초과할 수 있습니다. 제외된 스킬에 대한 경고를 확인하려면 `/context`를 실행하세요.
*   **서브 에이전트(Subagents)는 다른 서브 에이전트를 생성할 수 없습니다.** 워크플로가 중첩된 위임(nested delegation)을 요구한다면, 메인 대화에서 서브 에이전트를 연결(chain)하거나 스킬을 대신 사용해야 합니다.
*   **자동 메모리(Auto-memory)는 머신-로컬(machine-local)입니다.** 여러 머신이나 클라우드 환경에 동기화되지 않습니다. 동일한 Git 레포지토리 내의 모든 워크트리(worktrees)와 서브 디렉토리(subdirectories)는 하나의 자동 메모리 디렉토리를 공유하지만, 새 노트북으로 이동하면 처음부터 다시 시작해야 합니다.
*   **플러그인 에이전트(Plugin agents)에는 보안 제한이 있습니다.** 플러그인에서 제공하는 서브 에이전트는 훅(hooks), `mcpServers`, 또는 `permissionMode` 프론트매터 필드를 지원하지 않습니다. 이들은 조용히 무시됩니다. 필요하다면 에이전트 파일을 `.claude/agents/`에 수동으로 복사하세요.

### 실용적인 설정: 5단계로 제로에서 프로덕션까지

처음부터 시작한다면, 세 계층을 자연스럽게 따르는 진행 과정은 다음과 같습니다:

**1단계: `CLAUDE.md` (프로젝트 계층)로 시작합니다.** Claude Code 내에서 `/init`을 실행합니다. 코드베이스를 분석하여 스타터 `CLAUDE.md`를 생성합니다. 필수적인 내용으로 편집합니다. 200줄 미만을 목표로 하세요. 엄격한 제한은 없지만, 준수도를 위해서는 짧을수록 좋습니다. https://code.claude.com/docs/en/memory

**2단계: 권한을 추가합니다 (프로젝트 계층).** 스택에 대한 허용/거부(allow/deny) 규칙을 사용하여 `.claude/settings.json`을 생성합니다. 최소한 실행 명령어를 허용하고 `.env` 읽기를 거부하세요.

**3단계: 첫 번째 스킬을 추가합니다 (프로젝트 계층).** 가장 많이 하는 워크플로를 위해 하나 또는 두 개의 스킬을 만드세요. 코드 검토 및 문제 해결이 좋은 시작점입니다. 이들을 `.claude/skills/`에 넣으세요.

**4단계: 성장함에 따라 모듈화합니다 (프로젝트 계층).** `CLAUDE.md`가 복잡해지면, 지침을 `.claude/rules/` 파일로 분할합니다. 적절한 경우 경로별로 범위를 지정(scope)합니다.

**5단계: 개인 계층을 설정합니다 (사용자 계층).** 개인 선호 사항("항상 `npm` 대신 `pnpm` 사용"; "클래스 기반보다 함수형 패턴 선호")을 포함하는 `~/.claude/CLAUDE.md`를 추가합니다. 여러 프로젝트에서 반복하는 작업을 위해 `~/.claude/agents/`에 하나 또는 두 개의 개인 에이전트를 생성합니다.

이것이 실제로 95%의 프로젝트에 필요한 전부입니다. 스킬, 에이전트, 훅, 경로 범위 규칙; 이들은 최적화(optimization)입니다. `CLAUDE.md`는 가장 높은 레버리지(leverage)를 가진 파일입니다. 먼저 이것을 제대로 만드세요. 다른 모든 것은 그 위에 쌓입니다.

### `.claude` 폴더는 인프라(Infrastructure) 제안/지침이며, 하드 구성(hard configuration)으로 취급하지 마세요.

대부분의 사람들이 놓치는 부분이 있습니다. `.claude` 폴더는 `.prettierrc`나 `.eslintrc`와 같지 않습니다. 이들은 구성 파일(configuration files)입니다. `.claude` 폴더는 인프라(infrastructure)입니다. 여러분과 모델 사이에 있는 운영 체제(operating system) 계층입니다.

세 계층. 정책(Policy)은 안전장치(guardrails)를 설정합니다. 사용자(User)는 여러분의 정체성을 설정합니다. 프로젝트(Project)는 컨텍스트를 설정합니다. 지침, 권한, 스킬, 에이전트, 메모리 등 모든 기능은 각 계층에 존재하며, 우선순위는 항상 동일합니다.

여러분이 누구인지, 프로젝트가 무엇을 하는지, 그리고 Claude가 따라야 할 규칙을 더 명확하게 정의할수록, Claude를 수정하는 데 시간을 덜 쓰고 유용한 작업을 수행하는 데 더 많은 시간을 할애할 수 있습니다. `.claude/`에 투자한 모든 시간은 모든 세션, 팀의 모든 개발자, 모든 미래 대화에서 배당금(dividends)을 지불합니다.
