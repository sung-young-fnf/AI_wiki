# Claude Code 모범 사례

Claude Code를 최대한 활용하기 위한 팁과 패턴을 환경 설정부터 병렬 세션 전반으로 확장하는 방법까지 다룹니다.

Claude Code는 에이전트 기반 코딩 환경입니다. 질문에 답하고 기다리는 챗봇과 달리, Claude Code는 파일을 읽고, 명령을 실행하며, 변경 사항을 적용하고, 자율적으로 문제를 해결할 수 있습니다. 사용자는 이 과정을 지켜보거나, 방향을 지시하거나, 또는 완전히 맡겨둘 수도 있습니다.

이는 작업 방식을 변화시킵니다. 코드를 직접 작성하고 Claude에게 검토를 요청하는 대신, 원하는 바를 설명하면 Claude가 이를 구축하는 방법을 찾아냅니다. Claude는 탐색하고, 계획하며, 구현합니다.

하지만 이러한 자율성에도 학습 곡선이 존재합니다. Claude는 이해해야 할 특정 제약 조건 내에서 작동합니다.

이 가이드는 Anthropic 내부 팀과 다양한 코드베이스, 언어 및 환경에서 Claude Code를 사용하는 엔지니어들에게 효과적인 것으로 입증된 패턴을 다룹니다. 에이전트 루프가 내부적으로 어떻게 작동하는지에 대해서는 "Claude Code 작동 방식 (How Claude Code works)"을 참조하십시오.

대부분의 모범 사례는 한 가지 제약 조건에 기반합니다: Claude의 컨텍스트 창(context window)은 빠르게 채워지며, 채워질수록 성능이 저하됩니다.

Claude의 컨텍스트 창은 모든 메시지, Claude가 읽는 모든 파일, 모든 명령 출력(command output)을 포함하여 전체 대화를 담고 있습니다. 하지만 이 공간은 빠르게 채워질 수 있습니다. 단일 디버깅 세션이나 코드베이스 탐색만으로도 수만 개의 토큰을 생성하고 소비할 수 있습니다.

컨텍스트가 채워질수록 LLM(Large Language Model) 성능이 저하되기 때문에 이는 중요합니다. 컨텍스트 창이 가득 차면 Claude는 이전 지침을 "잊어버리거나" 더 많은 실수를 할 수 있습니다. 컨텍스트 창은 관리해야 할 가장 중요한 리소스입니다. 세션이 실제로 어떻게 채워지는지 보려면, 시작 시 무엇이 로드되고 각 파일 읽기가 얼마나 비용이 드는지에 대한 대화형 가이드(interactive walkthrough)를 시청하십시오. 맞춤형 상태 표시줄로 컨텍스트 사용량을 지속적으로 추적하고, 토큰 사용량을 줄이는 전략은 "토큰 사용량 줄이기 (Reduce token usage)"를 참조하십시오.

## Claude가 자신의 작업을 검증할 방법을 제공하세요

Claude가 스스로 확인할 수 있도록 테스트, 스크린샷 또는 예상 결과(expected output)를 포함하세요. 이는 가장 높은 레버리지를 얻을 수 있는 단일한 조치입니다.

Claude는 테스트를 실행하고, 스크린샷을 비교하며, 출력을 검증하는 등 자신의 작업을 스스로 검증할 수 있을 때 훨씬 더 나은 성능을 보입니다.

명확한 성공 기준이 없으면, 보기에는 그럴듯하지만 실제로는 작동하지 않는 결과를 만들어낼 수 있습니다. 이 경우 사용자가 유일한 피드백 루프가 되며, 모든 실수가 사용자의 주의를 필요로 하게 됩니다.

| 전략 | 개선 전 (Before) | 개선 후 (After) |
|---|---|---|
| 검증 기준 제공 | "이메일 주소를 검증하는 함수를 구현해 줘" | "validateEmail 함수를 작성해 줘. 예시 테스트 케이스: user@example.com은 참, invalid는 거짓, user@.com은 거짓. 구현 후 테스트를 실행해 줘" |
| UI 변경 사항 시각적으로 검증 | "대시보드를 더 멋지게 만들어 줘" | "[스크린샷 붙여넣기] 이 디자인을 구현해 줘. 결과 스크린샷을 찍어 원본과 비교하고, 차이점을 나열한 뒤 수정해 줘" |
| 증상이 아닌 근본 원인 해결 | "빌드가 실패해" | "빌드가 이 오류로 실패해: [오류 붙여넣기]. 이를 수정하고 빌드가 성공하는지 확인해 줘. 근본 원인을 해결하고 오류를 억제하지 마" |

UI 변경 사항은 Chrome용 Claude 확장 프로그램 (Claude in Chrome extension)을 사용하여 검증할 수 있습니다. 이 확장 프로그램은 브라우저에서 새 탭을 열고 UI를 테스트하며, 코드가 작동할 때까지 반복합니다.

사용자의 검증은 테스트 스위트(test suite), 린터(linter) 또는 출력을 확인하는 Bash 명령일 수도 있습니다. 검증을 확고하게 만드는 데 투자하십시오.

## 먼저 탐색하고, 다음으로 계획한 뒤, 코딩하세요

잘못된 문제를 해결하는 것을 피하기 위해 연구와 계획을 구현과 분리하세요.

Claude가 바로 코딩으로 넘어갈 경우 잘못된 문제를 해결하는 코드를 생성할 수 있습니다. 계획 모드(Plan Mode)를 사용하여 탐색과 실행을 분리하세요.

권장되는 워크플로우는 네 가지 단계로 구성됩니다:

1.  **탐색 (Explore)**
    계획 모드(Plan Mode)로 진입하세요. Claude는 파일을 읽고 변경 사항을 적용하지 않고 질문에 답합니다.
    ```
    claude (Plan Mode)
    /src/auth를 읽고 세션 및 로그인 처리 방식을 이해해 줘.
    또한 비밀(secrets)을 위한 환경 변수(environment variables) 관리 방식도 살펴봐 줘.
    ```

2.  **계획 (Plan)**
    Claude에게 상세한 구현 계획을 세우도록 요청하세요.
    ```
    claude (Plan Mode)
    Google OAuth를 추가하고 싶어. 어떤 파일을 변경해야 할까?
    세션 흐름은 어떻게 되지? 계획을 세워 줘.
    ```
    Ctrl+G를 눌러 Claude가 진행하기 전에 텍스트 편집기에서 계획을 열어 직접 편집할 수 있습니다.

3.  **구현 (Implement)**
    일반 모드(Normal Mode)로 돌아가 Claude가 계획에 따라 코드를 작성하도록 하고, 이를 검증하세요.
    ```
    claude (Normal Mode)
    계획에 따라 OAuth 흐름을 구현해 줘.
    콜백 핸들러(callback handler)에 대한 테스트를 작성하고,
    테스트 스위트를 실행한 뒤 모든 실패를 수정해 줘.
    ```

4.  **커밋 (Commit)**
    Claude에게 설명적인 메시지로 커밋하고 PR(Pull Request)을 생성하도록 요청하세요.
    ```
    claude (Normal Mode)
    설명적인 메시지로 커밋하고 PR을 열어 줘.
    ```

계획 모드(Plan Mode)는 유용하지만, 오버헤드(overhead)도 발생시킵니다.

범위가 명확하고 수정 사항이 작은 작업(예: 오타 수정, 로그 라인 추가, 변수 이름 변경)의 경우 Claude에게 직접 처리하도록 요청하세요.

계획은 접근 방식이 불확실하거나, 여러 파일을 수정해야 하거나, 수정하려는 코드에 익숙하지 않을 때 가장 유용합니다. 변경 사항을 한 문장으로 설명할 수 있다면 계획 단계를 건너뛰십시오.

## 프롬프트에 구체적인 컨텍스트를 제공하세요

지침이 더 정확할수록 필요한 수정 횟수가 줄어듭니다.

Claude는 의도를 추론할 수 있지만, 사용자의 마음을 읽을 수는 없습니다. 특정 파일을 참조하고, 제약 조건을 언급하며, 예시 패턴을 지적하십시오.

| 전략 | 개선 전 (Before) | 개선 후 (After) |
|---|---|---|
| 작업 범위 지정. 파일, 시나리오, 테스트 선호도 명시. | "foo.py에 테스트를 추가해 줘" | "foo.py에 대해 사용자가 로그아웃된 엣지 케이스(edge case)를 다루는 테스트를 작성해 줘. 목(mock)은 사용하지 마." |
| 소스 지정. 질문에 답할 수 있는 소스를 Claude에게 지시. | "ExecutionFactory는 왜 이렇게 이상한 API를 가지고 있지?" | "ExecutionFactory의 Git 히스토리를 살펴보고 API가 어떻게 만들어졌는지 요약해 줘" |
| 기존 패턴 참조. 코드베이스의 패턴을 Claude에게 지적. | "달력 위젯을 추가해 줘" | "홈페이지에 기존 위젯이 어떻게 구현되어 있는지 살펴보고 패턴을 이해해 줘. HotDogWidget.php가 좋은 예시야. 이 패턴을 따라 사용자가 월을 선택하고 전/후로 페이지를 넘겨 연도를 선택할 수 있는 새 달력 위젯을 구현해 줘. 코드베이스에 이미 사용된 라이브러리 외에는 다른 라이브러리 없이 처음부터 구축해 줘." |
| 증상 설명. 증상, 예상 위치, "수정된" 모습 제공. | "로그인 버그를 수정해 줘" | "사용자들이 세션 시간 초과 후 로그인에 실패한다고 보고했어. src/auth/의 인증 흐름, 특히 토큰 갱신을 확인해 줘. 문제를 재현하는 실패하는 테스트를 작성한 다음 수정해 줘." |

모호한 프롬프트는 탐색 중이고 경로 수정(course-correct)을 감당할 수 있을 때 유용할 수 있습니다. "이 파일에서 무엇을 개선할 수 있을까?"와 같은 프롬프트는 사용자가 생각하지 못했던 것을 드러낼 수 있습니다.

## 풍부한 콘텐츠를 제공하세요

파일을 참조하거나, 스크린샷/이미지를 붙여넣거나, 데이터를 직접 파이프(pipe)할 때 @를 사용하세요.

Claude에게 풍부한 데이터를 제공하는 방법은 여러 가지가 있습니다:

*   코드가 어디에 있는지 설명하는 대신 @를 사용하여 파일을 참조하세요. Claude는 응답하기 전에 해당 파일을 읽습니다.
*   이미지를 직접 붙여넣으세요. 이미지를 복사/붙여넣기 하거나 프롬프트로 드래그 앤 드롭(drag and drop)하세요.
*   문서 및 API 참조를 위한 URL을 제공하세요. 자주 사용하는 도메인을 허용 목록(allowlist)에 추가하려면 `/permissions`를 사용하세요.
*   `cat error.log | claude`와 같이 실행하여 데이터를 파이프하여 파일 내용을 직접 보낼 수 있습니다.
*   Claude가 필요한 것을 직접 가져오도록 하세요. Bash 명령, MCP 도구 또는 파일을 읽는 방식으로 Claude에게 직접 컨텍스트를 가져오도록 지시하세요.

## 환경을 설정하세요

몇 가지 설정 단계를 통해 모든 세션에서 Claude Code를 훨씬 더 효과적으로 사용할 수 있습니다. 확장 기능의 전체 개요와 각 기능을 언제 사용해야 하는지에 대해서는 "Claude Code 확장 (Extend Claude Code)"을 참조하십시오.

### 효과적인 CLAUDE.md를 작성하세요

`/init`을 실행하여 현재 프로젝트 구조를 기반으로 초기 `CLAUDE.md` 파일을 생성한 다음, 시간이 지남에 따라 개선하십시오.

`CLAUDE.md`는 Claude가 모든 대화를 시작할 때 읽는 특별한 파일입니다. Bash 명령, 코드 스타일 및 워크플로우 규칙을 포함하십시오. 이는 Claude가 코드만으로는 추론할 수 없는 영구적인 컨텍스트를 제공합니다.

`/init` 명령은 코드베이스를 분석하여 빌드 시스템, 테스트 프레임워크 및 코드 패턴을 감지하여 개선할 수 있는 견고한 기반을 제공합니다.

`CLAUDE.md` 파일에는 정해진 형식이 없지만, 짧고 사람이 읽을 수 있도록 유지하십시오. 예를 들어:

```markdown
CLAUDE.md
# 코드 스타일
- CommonJS (require) 대신 ES 모듈 (import/export) 구문을 사용하세요.
- 가능한 경우 임포트(import)를 구조 분해(destructure)하세요 (예: import { foo } from 'bar').

# 워크플로우
- 일련의 코드 변경을 마친 후에는 반드시 타입 체크(typecheck)를 수행하세요.
- 성능을 위해 전체 테스트 스위트(test suite) 대신 단일 테스트를 실행하는 것을 선호합니다.
```

`CLAUDE.md`는 모든 세션에서 로드되므로, 광범위하게 적용되는 내용만 포함하십시오. 특정 도메인 지식이나 가끔만 관련되는 워크플로우의 경우 스킬(skills)을 대신 사용하십시오. Claude는 모든 대화를 복잡하게 만들지 않고 필요할 때 스킬을 로드합니다.

간결하게 유지하십시오. 각 줄에 대해 다음과 같이 질문하십시오: "이것을 제거하면 Claude가 실수를 할까?" 그렇지 않다면 제거하십시오. 과도하게 길어진 `CLAUDE.md` 파일은 Claude가 실제 지침을 무시하게 만듭니다!

| ✅ 포함할 내용 | ❌ 제외할 내용 |
|---|---|
| Claude가 추측할 수 없는 Bash 명령 | Claude가 코드를 읽어서 파악할 수 있는 모든 것 |
| 기본값과 다른 코드 스타일 규칙 | Claude가 이미 알고 있는 표준 언어 규칙 |
| 테스트 지침 및 선호하는 테스트 러너 | 상세 API 문서 (대신 문서 링크) |
| 저장소 에티켓 (브랜치 명명, PR 규칙) | 자주 변경되는 정보 |
| 프로젝트에 특정한 아키텍처 결정 | 긴 설명 또는 튜토리얼 |
| 개발 환경 특이 사항 (필수 환경 변수) | 코드베이스의 파일별 설명 |
| 일반적인 문제(gotchas) 또는 명확하지 않은 동작 | "깔끔한 코드 작성"과 같은 자명한 관행 |

원하지 않는 작업을 계속 수행하는데도 금지 규칙이 있다면, 파일이 너무 길어 규칙이 무시될 가능성이 높습니다. `CLAUDE.md`에 답변이 있는 질문을 Claude가 계속 묻는다면, 문구가 모호할 수 있습니다. `CLAUDE.md`를 코드처럼 다루십시오: 문제가 발생하면 검토하고, 정기적으로 정리하며, 변경 사항이 Claude의 동작에 실제로 영향을 미치는지 관찰하여 테스트하십시오.

"IMPORTANT" 또는 "YOU MUST"와 같이 강조 표시를 추가하여 지침 준수도를 높일 수 있습니다. 팀이 기여할 수 있도록 `CLAUDE.md`를 Git에 체크인(check in)하십시오. 이 파일은 시간이 지남에 따라 가치가 높아집니다.

`CLAUDE.md` 파일은 `@path/to/import` 구문을 사용하여 추가 파일을 가져올 수 있습니다:

```markdown
CLAUDE.md
프로젝트 개요는 @README.md를, 사용 가능한 npm 명령은 @package.json을 참조하세요.

# 추가 지침
- Git 워크플로우: @docs/git-instructions.md
- 개인 오버라이드(overrides): @~/.claude/my-project-instructions.md
```

`CLAUDE.md` 파일은 여러 위치에 배치할 수 있습니다:

*   **홈 폴더 (`~/.claude/CLAUDE.md`):** 모든 Claude 세션에 적용됩니다.
*   **프로젝트 루트 (`./CLAUDE.md`):** 팀과 공유하기 위해 Git에 체크인하십시오.
*   **프로젝트 루트 (`./CLAUDE.local.md`):** 개인 프로젝트별 메모; 이 파일을 `.gitignore`에 추가하여 팀과 공유되지 않도록 하십시오.
*   **부모 디렉토리:** `root/CLAUDE.md`와 `root/foo/CLAUDE.md`가 모두 자동으로 가져와지는 모노레포(monorepo)에 유용합니다.
*   **하위 디렉토리:** Claude는 해당 디렉토리의 파일로 작업할 때 하위 `CLAUDE.md` 파일을 필요에 따라 가져옵니다.

### 권한을 설정하세요

분류기(classifier)가 승인을 처리하도록 자동 모드(auto mode)를 사용하거나, 특정 명령을 허용 목록(allowlist)에 추가하기 위해 `/permissions`를, 또는 OS 수준 격리(OS-level isolation)를 위해 `/sandbox`를 사용하세요. 각 방법은 사용자가 통제권을 유지하면서도 방해를 줄여줍니다.

기본적으로 Claude Code는 파일 쓰기, Bash 명령, MCP 도구 등 시스템을 수정할 수 있는 작업에 대해 권한을 요청합니다. 이는 안전하지만 지루합니다. 열 번째 승인 후에는 더 이상 실제로 검토하는 것이 아니라 단순히 클릭만 하게 됩니다. 이러한 방해를 줄이는 세 가지 방법이 있습니다:

*   **자동 모드 (Auto mode):** 별도의 분류기 모델(classifier model)이 명령을 검토하고 위험해 보이는 것만 차단합니다: 범위 확대(scope escalation), 알 수 없는 인프라(unknown infrastructure) 또는 악성 콘텐츠 기반 작업. 작업의 일반적인 방향은 신뢰하지만 모든 단계를 클릭하고 싶지 않을 때 가장 좋습니다.
*   **권한 허용 목록 (Permission allowlists):** `npm run lint` 또는 `git commit`과 같이 안전하다고 알려진 특정 도구를 허용합니다.
*   **샌드박싱 (Sandboxing):** 파일 시스템(filesystem) 및 네트워크 접근을 제한하는 OS 수준 격리를 활성화하여 Claude가 정의된 경계 내에서 더 자유롭게 작업할 수 있도록 합니다.

권한 모드(permission modes), 권한 규칙(permission rules) 및 샌드박싱(sandboxing)에 대해 더 자세히 알아보십시오.

### CLI 도구를 사용하세요

외부 서비스와 상호 작용할 때 Claude Code에게 `gh`, `aws`, `gcloud`, `sentry-cli`와 같은 CLI 도구를 사용하도록 지시하세요.

CLI 도구는 외부 서비스와 상호 작용하는 가장 컨텍스트 효율적인 방법입니다. GitHub를 사용한다면 `gh` CLI를 설치하십시오. Claude는 이를 사용하여 이슈를 생성하고, 풀 리퀘스트(pull request)를 열고, 댓글을 읽는 방법을 알고 있습니다. `gh`가 없어도 Claude는 GitHub API를 사용할 수 있지만, 인증되지 않은 요청은 종종 속도 제한(rate limit)에 도달합니다.

Claude는 또한 이미 알지 못하는 CLI 도구를 학습하는 데 효과적입니다. "foo-cli-tool --help"를 사용하여 foo 도구에 대해 학습한 다음, A, B, C를 해결하는 데 사용해 줘"와 같은 프롬프트를 시도하십시오.

### MCP 서버를 연결하세요

`claude mcp add`를 실행하여 Notion, Figma 또는 데이터베이스와 같은 외부 도구를 연결하세요.

MCP 서버를 통해 Claude에게 이슈 트래커(issue tracker)의 기능을 구현하고, 데이터베이스를 쿼리하며, 모니터링 데이터를 분석하고, Figma의 디자인을 통합하며, 워크플로우를 자동화하도록 요청할 수 있습니다.

### 훅(Hook)을 설정하세요

예외 없이 매번 발생해야 하는 작업에는 훅(hooks)을 사용하세요.

훅은 Claude의 워크플로우 내 특정 지점에서 스크립트를 자동으로 실행합니다. 조언적인 `CLAUDE.md` 지침과 달리, 훅은 결정론적이며 해당 작업이 발생함을 보장합니다.

Claude가 사용자 대신 훅을 작성할 수 있습니다. "모든 파일 편집 후 eslint를 실행하는 훅을 작성해 줘" 또는 "마이그레이션(migrations) 폴더에 대한 쓰기를 차단하는 훅을 작성해 줘"와 같은 프롬프트를 시도하십시오. `.claude/settings.json`을 직접 편집하여 훅을 수동으로 구성하고, `/hooks`를 실행하여 구성된 내용을 탐색할 수 있습니다.

### 스킬(Skill)을 생성하세요

Claude에게 도메인 지식과 재사용 가능한 워크플로우를 제공하려면 `.claude/skills/`에 `SKILL.md` 파일을 생성하십시오.

스킬은 프로젝트, 팀 또는 도메인에 특정한 정보로 Claude의 지식을 확장합니다. Claude는 관련될 때 자동으로 스킬을 적용하거나, 사용자가 `/skill-name`으로 직접 호출할 수 있습니다.

`.claude/skills/`에 `SKILL.md`가 포함된 디렉토리를 추가하여 스킬을 생성하십시오:

```markdown
.claude/skills/api-conventions/SKILL.md
---
name: api-conventions
description: 우리 서비스의 REST API 디자인 규칙
---
# API 규칙
- URL 경로에는 케밥 케이스(kebab-case)를 사용하세요.
- JSON 속성에는 카멜 케이스(camelCase)를 사용하세요.
- 목록 엔드포인트(list endpoints)에는 항상 페이지네이션(pagination)을 포함하세요.
- URL 경로(/v1/, /v2/)에 API 버전을 명시하세요.
```

스킬은 직접 호출할 수 있는 반복 가능한 워크플로우를 정의할 수도 있습니다:

```markdown
.claude/skills/fix-issue/SKILL.md
---
name: fix-issue
description: GitHub 이슈 수정
disable-model-invocation: true
---
GitHub 이슈를 분석하고 수정: $ARGUMENTS.

1. `gh issue view`를 사용하여 이슈 상세 정보 가져오기
2. 이슈에 설명된 문제 이해하기
3. 코드베이스에서 관련 파일 검색하기
4. 이슈 수정을 위한 필요한 변경 사항 구현하기
5. 수정을 검증하기 위한 테스트 작성 및 실행하기
6. 코드가 린팅(linting) 및 타입 체크(type checking)를 통과하는지 확인하기
7. 설명적인 커밋 메시지 생성하기
8. 푸시(Push) 및 PR(Pull Request) 생성하기
```
`/fix-issue 1234`를 실행하여 호출하십시오. 수동으로 트리거하려는 부작용(side effects)이 있는 워크플로우에는 `disable-model-invocation: true`를 사용하십시오.

### 맞춤형 서브 에이전트(subagent)를 생성하세요

Claude가 격리된 작업에 위임할 수 있는 특수화된 도우미를 `.claude/agents/`에 정의하십시오.

서브 에이전트(subagents)는 자체 컨텍스트(context)와 허용된 도구 세트를 가지고 실행됩니다. 이들은 많은 파일을 읽거나, 주 대화를 복잡하게 만들지 않고도 특수한 집중이 필요한 작업에 유용합니다.

```markdown
.claude/agents/security-reviewer.md
---
name: security-reviewer
description: 보안 취약점 코드를 검토합니다.
tools: Read, Grep, Glob, Bash
model: opus
---
당신은 선임 보안 엔지니어입니다. 다음 사항에 대해 코드를 검토하세요:
- 주입 취약점 (Injection vulnerabilities, SQL, XSS, 명령 주입)
- 인증 및 인가(authorization) 결함
- 코드 내 비밀 정보 또는 자격 증명
- 안전하지 않은 데이터 처리

특정 라인 참조와 제안된 수정 사항을 제공하세요.
```
Claude에게 서브 에이전트를 명시적으로 사용하도록 지시하세요: "이 코드의 보안 문제를 검토하기 위해 서브 에이전트를 사용해 줘."

### 플러그인을 설치하세요

`/plugin`을 실행하여 마켓플레이스(marketplace)를 탐색하십시오. 플러그인은 구성 없이 스킬, 도구 및 통합 기능을 추가합니다.

플러그인은 커뮤니티와 Anthropic에서 제공하는 스킬, 훅(hooks), 서브 에이전트(subagents) 및 MCP 서버를 단일 설치 가능한 단위로 묶습니다. 타입 언어(typed language)로 작업하는 경우, Claude에게 정밀한 심볼 내비게이션(symbol navigation)과 편집 후 자동 오류 감지 기능을 제공하는 코드 인텔리전스 플러그인(code intelligence plugin)을 설치하십시오.

스킬, 서브 에이전트, 훅 및 MCP 중에서 선택하는 방법에 대한 지침은 "Claude Code 확장 (Extend Claude Code)"을 참조하십시오.

## 효과적으로 소통하세요

Claude Code와 소통하는 방식은 결과의 품질에 상당한 영향을 미칩니다.

### 코드베이스 관련 질문을 하세요

선임 엔지니어에게 물을 법한 질문을 Claude에게 하십시오.

새로운 코드베이스에 온보딩할 때(onboarding) Claude Code를 학습 및 탐색 용도로 사용하십시오. 다른 엔지니어에게 물을 법한 질문을 Claude에게 할 수 있습니다:

*   로깅(logging)은 어떻게 작동하나요?
*   새로운 API 엔드포인트(endpoint)를 어떻게 만드나요?
*   foo.rs 파일 134행의 `async move { ... }`는 무슨 역할을 하나요?
*   `CustomerOnboardingFlowImpl`은 어떤 엣지 케이스(edge case)를 처리하나요?
*   이 코드는 333행에서 `bar()` 대신 `foo()`를 왜 호출하나요?

이러한 방식으로 Claude Code를 사용하는 것은 효과적인 온보딩 워크플로우이며, 적응 시간을 단축하고 다른 엔지니어의 부담을 줄여줍니다. 특별한 프롬프트는 필요 없습니다: 직접 질문하십시오.

### Claude가 당신을 인터뷰하게 하세요

더 큰 기능의 경우, Claude가 먼저 당신을 인터뷰하게 하십시오. 최소한의 프롬프트로 시작하고, `AskUserQuestion` 도구를 사용하여 Claude에게 당신을 인터뷰하도록 요청하세요.

Claude는 기술 구현, UI/UX, 엣지 케이스(edge cases) 및 트레이드오프(tradeoffs)를 포함하여 당신이 아직 고려하지 못했을 수 있는 사항에 대해 질문합니다.

```markdown
[간략한 설명]을 구축하고 싶어. AskUserQuestion 도구를 사용해서 나를 자세히 인터뷰해 줘.

기술 구현, UI/UX, 엣지 케이스, 우려 사항 및 트레이드오프에 대해 질문해 줘. 명백한 질문은 하지 말고, 내가 고려하지 못했을 수 있는 어려운 부분들을 파고들어 줘.

모든 것을 다룰 때까지 인터뷰를 계속하고, 그 다음 완전한 사양(spec)을 SPEC.md에 작성해 줘.
```
사양(spec) 작성이 완료되면, 새로운 세션을 시작하여 이를 실행하십시오. 새 세션은 구현에만 집중하는 깔끔한 컨텍스트를 가지며, 참조할 수 있는 문서화된 사양을 갖게 됩니다.

## 세션을 관리하세요

대화는 지속적이며 되돌릴 수 있습니다. 이를 활용하세요!

### 빠르고 자주 경로를 수정하세요

Claude가 잘못된 길로 가는 것을 발견하는 즉시 수정하십시오.

최고의 결과는 긴밀한 피드백 루프(feedback loop)에서 나옵니다. Claude가 가끔 첫 시도에 문제를 완벽하게 해결하기도 하지만, 빠르게 수정하는 것이 일반적으로 더 나은 해결책을 더 빨리 만들어냅니다.

*   **Esc:** Esc 키로 Claude의 작업을 중간에 중단하세요. 컨텍스트는 유지되므로 방향을 다시 지시할 수 있습니다.
*   **Esc + Esc 또는 `/rewind`:** Esc를 두 번 누르거나 `/rewind`를 실행하여 되감기 메뉴(rewind menu)를 열고 이전 대화 및 코드 상태를 복원하거나, 선택한 메시지부터 요약할 수 있습니다.
*   **"Undo that" (이것을 취소해 줘):** Claude에게 변경 사항을 되돌리도록 요청하십시오.
*   **`/clear`:** 관련 없는 작업 간에 컨텍스트를 초기화하세요. 관련 없는 컨텍스트가 많은 긴 세션은 성능을 저하시킬 수 있습니다.

한 세션에서 동일한 문제에 대해 Claude를 두 번 이상 수정했다면, 컨텍스트가 실패한 접근 방식으로 복잡해진 것입니다. `/clear`를 실행하고, 학습한 내용을 통합한 더 구체적인 프롬프트로 새로 시작하십시오. 더 나은 프롬프트로 시작하는 깔끔한 세션은 누적된 수정 사항이 있는 긴 세션보다 거의 항상 더 나은 성능을 보입니다.

### 컨텍스트를 적극적으로 관리하세요

관련 없는 작업 간에 `/clear`를 실행하여 컨텍스트를 초기화하세요.

Claude Code는 컨텍스트 제한에 가까워지면 대화 기록을 자동으로 압축하여 중요한 코드와 결정을 보존하고 공간을 확보합니다.

긴 세션 동안 Claude의 컨텍스트 창(context window)은 관련 없는 대화, 파일 내용 및 명령으로 채워질 수 있습니다. 이는 성능을 저하시키고 때로는 Claude의 집중을 방해할 수 있습니다.

*   컨텍스트 창을 완전히 초기화하려면 작업 간에 `/clear`를 자주 사용하십시오.
*   자동 압축(auto compaction)이 트리거되면 Claude는 코드 패턴, 파일 상태 및 주요 결정을 포함하여 가장 중요한 내용을 요약합니다.
*   더 많은 제어를 위해서는 `/compact <지침>`을 실행하십시오. 예를 들어 `/compact API 변경 사항에 집중해 줘`와 같이 사용합니다.
*   대화의 일부만 압축하려면 Esc + Esc 또는 `/rewind`를 사용하고 메시지 체크포인트(checkpoint)를 선택한 다음 "여기부터 요약 (Summarize from here)"을 선택하십시오. 이는 이전 컨텍스트를 그대로 유지하면서 해당 시점부터의 메시지를 압축합니다.
*   `CLAUDE.md`에서 "압축 시, 항상 수정된 파일의 전체 목록과 모든 테스트 명령을 보존해 줘"와 같은 지침으로 압축 동작을 사용자 지정하여 중요한 컨텍스트가 요약 후에도 살아남도록 보장하십시오.
*   컨텍스트에 남아 있을 필요가 없는 간단한 질문에는 `/btw`를 사용하십시오. 답변은 닫을 수 있는 오버레이(overlay)로 나타나며 대화 기록에 들어가지 않으므로, 컨텍스트를 늘리지 않고도 세부 사항을 확인할 수 있습니다.

### 조사를 위해 서브 에이전트(subagent)를 사용하세요

"서브 에이전트를 사용하여 X를 조사해 줘"와 같이 연구 작업을 위임하십시오. 서브 에이전트는 별도의 컨텍스트에서 탐색하여 주 대화를 구현에 집중할 수 있도록 깔끔하게 유지합니다.

컨텍스트가 기본적인 제약 조건이므로, 서브 에이전트는 사용 가능한 가장 강력한 도구 중 하나입니다. Claude가 코드베이스를 연구할 때 많은 파일을 읽는데, 이 모든 것이 컨텍스트를 소비합니다. 서브 에이전트는 별도의 컨텍스트 창에서 실행되며 요약된 내용을 보고합니다:

```
서브 에이전트를 사용하여 우리 인증 시스템이 토큰 갱신을 어떻게 처리하는지,
그리고 재사용할 수 있는 기존 OAuth 유틸리티가 있는지 조사해 줘.
```

서브 에이전트는 코드베이스를 탐색하고, 관련 파일을 읽으며, 발견한 내용을 보고하는데, 이 모든 과정이 주 대화를 복잡하게 만들지 않습니다.

Claude가 무언가를 구현한 후 검증을 위해 서브 에이전트를 사용할 수도 있습니다:

```
이 코드의 엣지 케이스를 검토하기 위해 서브 에이전트를 사용해 줘
```

### 체크포인트(checkpoint)를 사용하여 되감으세요

Claude의 모든 작업은 체크포인트(checkpoint)를 생성합니다. 대화, 코드 또는 둘 다를 이전 체크포인트로 복원할 수 있습니다.

Claude는 변경 전 자동으로 체크포인트를 만듭니다. Escape 키를 두 번 누르거나 `/rewind`를 실행하여 되감기 메뉴(rewind menu)를 엽니다. 대화만 복원하거나, 코드만 복원하거나, 둘 다 복원하거나, 선택한 메시지부터 요약할 수 있습니다. 자세한 내용은 "체크포인트 (Checkpointing)"를 참조하십시오.

모든 움직임을 신중하게 계획하는 대신, Claude에게 위험한 시도를 해보라고 지시할 수 있습니다. 만약 작동하지 않으면, 되감기(rewind)하고 다른 접근 방식을 시도하십시오. 체크포인트는 세션 간에도 유지되므로 터미널을 닫았다가 나중에 다시 되감을 수 있습니다.

체크포인트는 Claude에 의해 이루어진 변경 사항만 추적하며, 외부 프로세스는 추적하지 않습니다. 이는 Git을 대체하는 것이 아닙니다.

### 대화를 다시 시작하세요

`claude --continue`를 실행하여 중단했던 부분부터 다시 시작하거나, `--resume`을 사용하여 최근 세션 중에서 선택하십시오.

Claude Code는 대화를 로컬에 저장합니다. 작업이 여러 세션에 걸쳐 진행될 때 컨텍스트를 다시 설명할 필요가 없습니다:

```
claude --continue    # 가장 최근 대화 재개
claude --resume      # 최근 대화 목록에서 선택
```

세션에 "oauth-migration" 또는 "debugging-memory-leak"과 같이 설명적인 이름을 부여하여 나중에 쉽게 찾을 수 있도록 `/rename`을 사용하십시오. 세션을 브랜치(branches)처럼 다루십시오: 서로 다른 작업 흐름은 별도의 영구적인 컨텍스트를 가질 수 있습니다.

## 자동화 및 확장

하나의 Claude를 효과적으로 사용할 수 있게 되면, 병렬 세션, 비대화형 모드(non-interactive mode) 및 분산 처리(fan-out) 패턴을 사용하여 생산성을 배가시키십시오.

지금까지의 내용은 한 명의 사람, 하나의 Claude, 하나의 대화를 전제로 했습니다. 그러나 Claude Code는 수평적으로 확장됩니다. 이 섹션의 기술은 더 많은 작업을 수행하는 방법을 보여줍니다.

### 비대화형 모드(non-interactive mode)를 실행하세요

CI, 사전 커밋 훅(pre-commit hooks) 또는 스크립트에서 `claude -p "prompt"`를 사용하십시오. 스트리밍 JSON 출력(streaming JSON output)을 위해서는 `--output-format stream-json`을 추가하십시오.

`claude -p "your prompt"`를 사용하면 세션 없이 비대화형(non-interactive)으로 Claude를 실행할 수 있습니다. 비대화형 모드는 Claude를 CI 파이프라인, 사전 커밋 훅 또는 모든 자동화된 워크플로우에 통합하는 방법입니다. 출력 형식(output formats)을 통해 결과를 프로그래밍 방식으로 파싱(parse)할 수 있습니다: 일반 텍스트, JSON 또는 스트리밍 JSON.

```
# 일회성 쿼리
claude -p "이 프로젝트가 무엇을 하는지 설명해 줘"

# 스크립트용 구조화된 출력
claude -p "모든 API 엔드포인트를 나열해 줘" --output-format json

# 실시간 처리를 위한 스트리밍
claude -p "이 로그 파일을 분석해 줘" --output-format stream-json
```

### 여러 Claude 세션을 실행하세요

개발 속도를 높이고, 격리된 실험을 실행하거나, 복잡한 워크플로우를 시작하기 위해 여러 Claude 세션을 병렬로 실행하십시오.

병렬 세션을 실행하는 세 가지 주요 방법이 있습니다:

*   **Claude Code 데스크톱 앱:** 여러 로컬 세션을 시각적으로 관리합니다. 각 세션은 자체적으로 격리된 작업 트리(worktree)를 가집니다.
*   **웹 기반 Claude Code:** Anthropic의 보안 클라우드 인프라(secure cloud infrastructure)에서 격리된 VM(Virtual Machine)으로 실행됩니다.
*   **에이전트 팀 (Agent teams):** 공유 작업, 메시징 및 팀 리더와 함께 여러 세션을 자동화하여 조정합니다.

작업 병렬화 외에도, 여러 세션은 품질 중심 워크플로우를 가능하게 합니다. Claude는 방금 작성한 코드에 편향되지 않으므로 새로운 컨텍스트는 코드 검토를 개선합니다.

예를 들어, 작성자/검토자 패턴(Writer/Reviewer pattern)을 사용하십시오:

| 세션 A (작성자) | 세션 B (검토자) |
|---|---|
| API 엔드포인트에 대한 속도 제한기(rate limiter)를 구현해 줘 | `@src/middleware/rateLimiter.ts`에 구현된 속도 제한기를 검토해 줘. 엣지 케이스(edge cases), 경쟁 조건(race conditions) 및 기존 미들웨어(middleware) 패턴과의 일관성을 찾아 줘. |
| 다음은 검토 피드백이야: [세션 B 출력]. 이 문제들을 해결해 줘. | |

테스트에서도 유사한 작업을 수행할 수 있습니다: 한 Claude에게 테스트를 작성하게 하고, 다른 Claude에게는 그 테스트를 통과하는 코드를 작성하게 하십시오.

### 파일 분산 처리(Fan out)를 수행하세요

각 작업에 대해 `claude -p`를 호출하는 태스크를 반복하십시오. 배치 작업(batch operations)의 권한 범위를 지정하려면 `--allowedTools`를 사용하십시오.

대규모 마이그레이션(migrations) 또는 분석의 경우, 많은 병렬 Claude 호출을 통해 작업을 분산할 수 있습니다:

1.  **작업 목록 생성**
    Claude에게 마이그레이션해야 할 모든 파일(예: 마이그레이션해야 할 2,000개의 Python 파일 목록)을 나열하도록 하십시오.
2.  **목록을 반복하는 스크립트 작성**
    ```bash
    for file in $(cat files.txt); do
      claude -p "$file을 React에서 Vue로 마이그레이션해 줘. OK 또는 FAIL을 반환해 줘." \
        --allowedTools "Edit,Bash(git commit *)"
    done
    ```
3.  **몇 개의 파일로 테스트한 다음 대규모로 실행**
    처음 2~3개 파일에서 발생하는 문제점을 기반으로 프롬프트를 다듬은 다음, 전체 세트에서 실행하십시오. `--allowedTools` 플래그는 Claude가 수행할 수 있는 작업을 제한하며, 이는 무인으로 실행할 때 중요합니다.

Claude를 기존 데이터/처리 파이프라인에 통합할 수도 있습니다:
`claude -p "<프롬프트>" --output-format json | your_command`

개발 중 디버깅을 위해서는 `--verbose`를 사용하고, 프로덕션에서는 이를 끄십시오.

### 자동 모드(auto mode)로 자율적으로 실행하세요

백그라운드 안전 검사(background safety checks)를 통해 중단 없는 실행을 위해서는 자동 모드(auto mode)를 사용하십시오. 분류기 모델(classifier model)은 명령이 실행되기 전에 이를 검토하여, 범위 확대(scope escalation), 알 수 없는 인프라(unknown infrastructure) 및 악성 콘텐츠 기반 작업은 차단하면서 일상적인 작업은 프롬프트 없이 진행되도록 합니다.

`claude --permission-mode auto -p "모든 린트 오류를 수정해 줘"`

`-p` 플래그를 사용하는 비대화형 실행(non-interactive runs)의 경우, 분류기가 작업을 반복적으로 차단하면 자동 모드는 중단됩니다. 이는 사용자에게 돌아갈 수 있는 대안이 없기 때문입니다. 자동 모드가 언제 실패하는지에 대한 임계값은 해당 문서를 참조하십시오.

## 흔한 실패 패턴을 피하세요

다음은 흔한 실수들입니다. 이를 조기에 인지하면 시간을 절약할 수 있습니다:

*   **과도한 정보 혼재 세션 (The kitchen sink session).** 하나의 작업으로 시작했다가 관련 없는 질문을 하고, 다시 첫 번째 작업으로 돌아갑니다. 컨텍스트가 관련 없는 정보로 가득 차게 됩니다.
    **해결책:** 관련 없는 작업 사이에는 `/clear`를 사용하십시오.
*   **반복적인 수정 (Correcting over and over).** Claude가 잘못된 작업을 수행하고, 사용자가 수정해도 여전히 잘못되어 또 다시 수정합니다. 컨텍스트가 실패한 접근 방식으로 오염됩니다.
    **해결책:** 두 번 이상 수정에 실패한 후에는 `/clear`를 사용하고, 학습한 내용을 포함하여 더 나은 초기 프롬프트를 작성하십시오.
*   **과도하게 상세한 CLAUDE.md (The over-specified CLAUDE.md).** `CLAUDE.md` 파일이 너무 길면, 중요한 규칙이 잡음에 묻혀 Claude가 절반을 무시하게 됩니다.
    **해결책:** 과감하게 정리하십시오. Claude가 지침 없이도 올바르게 수행하는 내용이라면 삭제하거나 훅(hook)으로 전환하십시오.
*   **신뢰 후 검증 누락 (The trust-then-verify gap).** Claude가 그럴듯해 보이는 구현을 만들지만 엣지 케이스(edge cases)를 처리하지 못합니다.
    **해결책:** 항상 검증(테스트, 스크립트, 스크린샷)을 제공하십시오. 검증할 수 없다면 배포하지 마십시오.
*   **무한 탐색 (The infinite exploration).** 범위를 지정하지 않고 Claude에게 무언가를 "조사해 달라"고 요청합니다. Claude는 수백 개의 파일을 읽어 컨텍스트를 채웁니다.
    **해결책:** 조사의 범위를 좁히거나 서브 에이전트(subagents)를 사용하여 탐색이 주 컨텍스트를 소비하지 않도록 하십시오.

## 직관을 개발하세요

이 가이드의 패턴은 불변의 것이 아닙니다. 일반적으로 잘 작동하는 시작점이지만, 모든 상황에 최적이지 않을 수 있습니다.

때로는 하나의 복잡한 문제에 깊이 몰입해 있어 과거 기록이 중요할 때는 컨텍스트를 축적하도록 두어야 합니다. 때로는 작업이 탐색적이기 때문에 계획을 건너뛰고 Claude가 스스로 파악하도록 해야 할 수도 있습니다. 때로는 문제를 제한하기 전에 Claude가 어떻게 해석하는지 보고 싶기 때문에 모호한 프롬프트가 정확히 올바를 수 있습니다.

무엇이 효과적인지 주의를 기울이세요. Claude가 훌륭한 결과물을 생성할 때, 당신이 무엇을 했는지 주목하세요: 프롬프트 구조, 제공한 컨텍스트, 사용했던 모드. Claude가 어려움을 겪을 때는 그 이유를 질문하세요. 컨텍스트가 너무 복잡했나? 프롬프트가 너무 모호했나? 작업이 한 번에 처리하기에 너무 컸나?

시간이 지나면서, 어떤 가이드로도 포착할 수 없는 직관을 개발하게 될 것입니다. 언제 구체적으로 지시하고 언제 개방적으로 접근해야 할지, 언제 계획하고 언제 탐색해야 할지, 언제 컨텍스트를 지우고 언제 축적하도록 두어야 할지 알게 될 것입니다.
