## AI가 실제로 잘 못하는 것

AI 코딩 도구가 잘 못하는 것에 대해 솔직해져 봅시다.

**프로덕션 코드(production code) 작성:**
*   엣지 케이스(edge case)에서의 환각(hallucination) 현상
*   보안 취약점 (SQL 인젝션(SQL injection), XSS)
*   부하 상황에서의 경쟁 조건(race condition)
*   장기 실행 프로세스에서의 메모리 누수(memory leak)
*   비즈니스 로직(business logic)에 대한 이해 부족

**컨텍스트(context) 이해:**
*   팀의 코딩 표준(coding standards)을 읽지 못함
*   아키텍처(architecture) 결정 사항을 모름
*   회사의 기술 스택(tech stack) 제약 조건을 무시함
*   성능 요구사항(performance requirements)에 대한 이해 부족

**트레이드오프(tradeoff) 결정:**
*   속도 vs 유지보수성(maintainability)? AI는 무작위로 선택합니다.
*   단순성 vs 유연성(flexibility)? AI는 복잡성을 선택합니다.
*   DRY(Don't Repeat Yourself) vs KISS(Keep It Simple, Stupid)? AI는 Stack Overflow를 그대로 복사합니다.

저는 AI에게 엔지니어링에서 가장 어려운 부분들을 요청하고 있었습니다.

실패하는 것은 당연했습니다.

## AI가 실제로 놀라운 능력을 발휘하는 것

그러다 저는 접근 방식을 바꿨습니다.

AI에게 코드를 작성하게 하는 대신, 다음 5가지 작업에 활용했습니다.

### 1. 레거시 코드(Legacy Code) 설명

**기존 방식:**
*   문서화되지 않은 코드를 읽는 데 2시간 소요
*   회의 중인 시니어 개발자에게 문의
*   여전히 절반은 이해 못 함

**새로운 방식:**
*   나: "이 함수는 어떤 기능을 하나요?"
*   AI: "이 함수는 지수 백오프(exponential backoff)를 사용하는 재시도 메커니즘을 구현합니다. 네트워크 오류 발생 시 재시도 사이에 1초, 2초, 4초를 기다리며, 최대 3번 시도합니다."
*   나: "뮤텍스 록(mutex lock)은 왜 사용했나요?"
*   AI: "여러 고루틴(goroutine)이 재시도 카운터에 동시에 접근할 때 발생하는 경쟁 조건(race condition)을 방지합니다."

**레거시 함수당 절약 시간:** 30분 → 2분
**영향:** 신입 개발자 온보딩(onboarding) 3배 가속

### 2. 테스트 케이스(Test Case) 작성

**기존 방식:**
*   함수 작성
*   나중에 (어쩌면) 테스트 작성
*   엣지 케이스 놓침
*   운영 환경(production)에서 버그 발생

**새로운 방식:**
*   나: "이 결제 함수에 대한 테스트 케이스를 생성해 줘"

*   그다음 AI의 실수를 검토, 수정하고 비즈니스 관련 케이스를 추가합니다.

**결과:** 6개월 만에 테스트 커버리지(test coverage) 40% → 92%

### 3. 문서(Documentation) 생성

**기존 방식:**
*   코드 작성
*   나중에 문서화하겠다고 약속
*   결국 문서화하지 않음
*   팀원들이 영원히 같은 질문을 함

**새로운 방식:**
*   나: "이 API 엔드포인트(API endpoint)를 문서화해 줘"
*   AI: [다음 내용 생성:]
    *   엔드포인트 설명
    *   요청/응답 예시
    *   오류 코드(error codes)
    *   속도 제한(rate limits)
    *   인증 요구사항(authentication requirements)
*   그다음 부정확한 부분을 수정하고 비즈니스 컨텍스트를 추가합니다.

**영향:** "X 사용법"에 대한 지원 문의(support tickets) 70% 감소

### 4. 코드 리뷰(Code Review) 지원

**기존 방식:**
*   PR(Pull Request) 검토
*   미묘한 버그 놓침
*   승인
*   운영 환경에서 버그 발생

**새로운 방식:**
*   나: "이 PR에서 문제점을 검토해 줘"
*   AI: [다음 문제점 지적:]
    *   47번째 줄의 잠재적 널 포인터(null pointer)
    *   23번째 줄의 누락된 오류 처리(error handling)
    *   89번째 줄의 SQL 인젝션(SQL injection) 위험
    *   156번째 줄의 메모리 누수(memory leak)
*   나는 AI의 발견 사항을 검토하고, 오탐(false positive)을 무시하며, 실제 문제를 조사합니다.

**결과:** 지난달 병합 전 12개의 운영 환경 버그 발견

### 5. 리팩터링(Refactoring) 제안

**기존 방식:**
*   코드가 지저분해짐
*   "나중에 리팩터링하자"
*   나중은 오지 않음
*   코드베이스(codebase)가 유지보수 불가능해짐

**새로운 방식:**
*   나: "이 함수에 대한 리팩터링을 제안해 줘"
*   AI:
    1.  데이터베이스 로직(database logic)을 리포지토리 패턴(repository pattern)으로 추출
    2.  200줄짜리 함수를 4개의 집중된 함수로 분할
    3.  중첩된 `if` 문을 조기 반환(early return)으로 대체
    4.  매직 넘버(magic numbers) 대신 상수를 사용
*   나는 제안을 평가하고, 합리적인 것을 구현하며, 나쁜 조언은 무시합니다.

**영향:** 기술 부채(tech debt) 통제, 코드베이스 실제 개선

AI는 운영 환경 문제를 해결할 수는 없지만, 당신이 잠든 동안 보고서를 작성할 수 있습니다.

## 제가 따르는 새로운 규칙

**규칙 1: AI는 제안하고, 제가 결정합니다.**
AI 코드를 맹목적으로 받아들이지 마세요. 항상 검토하고, 항상 테스트하세요.

**규칙 2: AI는 설명하고, 제가 이해합니다.**
AI를 사용하여 더 빠르게 학습하고, 학습을 건너뛰지 마세요.

**규칙 3: AI는 생성하고, 제가 검증합니다.**
테스트, 문서, 리팩터링 – AI는 초안을 작성할 수 있습니다. 저는 정확성을 검증합니다.

**규칙 4: AI는 검토하고, 제가 판단합니다.**
AI는 문제를 지적합니다. 저는 그것이 실제 문제인지 판단합니다.

**규칙 5: AI가 핵심 로직을 작성하게 두지 마세요.**
인증(authentication), 결제(payments), 데이터 유효성 검사(data validation) – 오직 사람이 직접 작성해야 합니다.

## 아무도 말해주지 않는 가혹한 진실

AI 코딩 도구는 과대 포장되어 있습니다.

그들이 약속하는 것:
*   개발자 생산성 10배 향상
*   지루한 작업 제거
*   창의적인 문제에 집중

그들이 제공하는 것:
*   더 빠른 코드 작성
*   더 많은 디버깅 시간
*   기술 부채(technical debt) 폭발

하지만 올바르게 사용하면:
*   AI를 조수(보조 역할)로 활용 (대체가 아님)
*   인간의 판단이 최상위
*   특정 작업에 집중

그것들은 실제로 유용합니다.

## 여러분께 드리는 도전

AI가 코드를 작성하게 하지 마세요.
대신 AI를 다음 용도로 사용하세요.
*   하루에 1개의 레거시 함수 설명
*   자신의 코드에 대한 테스트 케이스 생성
*   문서 초안 작성
*   PR(Pull Request)에서 문제점 검토
*   리팩터링 제안

결과를 추적하세요:
*   병합 전 발견된 버그 수
*   테스트 커버리지(test coverage) 증가량
*   작성된 문서량
*   절약된 시간 대 디버깅 시간

장담컨대, 여러분은 더 생산적이 될 것입니다.

기술 부채(technical debt) 없이.

## 결론

AI는 개발자를 대체하지 않을 것입니다.

하지만 AI를 활용할 줄 아는 개발자가 그렇지 않은 개발자를 대체할 것입니다.

핵심은 AI에게 코드를 작성하게 하는 것이 아닙니다.

AI를 언제 사용하고, 언제 직접 해야 하는지를 아는 것입니다.

**AI를 다음 용도로 사용하세요.**
*   기존 코드 설명
*   테스트 생성
*   문서 작성
*   코드 리뷰 지원
*   리팩터링 제안

**AI를 다음 용도로 사용하지 마세요.**
*   핵심 비즈니스 로직(business logic)
*   보안에 민감한 코드
*   운영 환경에 중요한 기능
*   당신이 이해하지 못하는 모든 것

이것이 AI가 당신을 더 빠르게 만들지, 아니면 더 느리게 만들지의 차이입니다.
