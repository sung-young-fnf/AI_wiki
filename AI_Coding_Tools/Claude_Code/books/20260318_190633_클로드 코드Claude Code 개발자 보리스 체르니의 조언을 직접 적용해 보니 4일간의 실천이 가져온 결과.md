# 클로드 코드(Claude Code) 개발자 보리스 체르니의 조언을 직접 적용해 보니 — 4일간의 실천이 가져온 결과

실제 사이드 프로젝트에 보리스 체르니(Boris Cherny)의 클로드 코드(Claude Code) 워크플로우 팁을 적용한 후기 — 무엇이 효과 있었고, 무엇이 그렇지 않았으며, 가장 큰 변화는 무엇이었나.

클로드 코드(Claude Code)가 유틸리티 함수(utility function)를 리팩터링(refactoring)하는 과정에서 도입한, 세 개 파일 깊숙이 파묻힌 타입스크립트(TypeScript) 타입 불일치(type mismatch).

12분 동안 부풀어 오른 컨텍스트 윈도우(context window)를 스크롤하며 그것을 쳐다보다가, 문제가 코드에 있는 것이 아니라 나 자신에게 있다는 것을 깨달았습니다.

저는 계획, 코딩, 디버깅, 테스트 등 모든 작업을 하나의 클로드 코드(Claude Code) 세션으로 실행하고 있었고, 그 결과 컨텍스트(context)는 진흙탕이 되어버렸습니다.

**투명성 고지:** 이 글의 일부는 인공지능(AI) 도구의 도움을 받아 조사하고 다듬었습니다. 하지만 직접적인 테스트, 시간 측정, 실제 사례는 지난 4일간 저의 실제 워크플로우(workflow)에서 나온 것입니다.

그것은 지난 수요일 저녁의 일입니다. 보리스 체르니(Boris Cherny)가 2026년 1월부터 공유해 온 워크플로우 팁(workflow tips)을 4일 동안 체계적으로 적용한 결과, 그 사이드 프로젝트(side project)는 일요일에는 완전히 다른 상태가 되었습니다. 도구가 변한 것이 아니라, 저의 접근 방식이 변했기 때문입니다.

저는 클로드 코드(Claude Code)가 처음 공개 출시된 이후로 계속 사용해 왔습니다. 12개월 회고록에서 썼듯이, 이는 저의 주된 코딩 에이전트(coding agent)이며, 대부분의 경우 환상적인 작업을 수행합니다. 그러나 도구를 사용하는 것과 도구를 잘 사용하는 것에는 차이가 있습니다.

앤트로픽(Anthropic)에서 클로드 코드(Claude Code)를 만든 보리스 체르니는 2026년 1월부터 다섯 개의 스레드(thread)를 통해 총 42가지 팁을 공유했는데, 각 팁은 그와 클로드 코드 팀이 매일 실제로 일하는 방식에서 비롯된 것입니다.

그의 설정은 그 자신의 말에 따르면 "놀라울 정도로 평범(vanilla)"합니다. 이 세부 사항이 저에게 깊이 남았습니다. 이 도구를 만든 사람이 자신의 구성을 지나치게 복잡하게 만들지 않는다면, 저는 왜 제 설정을 조정하는 데 몇 시간을 보내고 있었을까요?

그래서 저는 실험을 진행했습니다. 4일 동안.

하나의 사이드 프로젝트 — 여러 저장소(repository)에서 AI 에이전트(AI agent)의 작업 완료를 추적하기 위한 Next.js 대시보드. 합리적으로 적용할 수 있는 모든 보리스의 팁을 실제 작업에 시험해 보았습니다. 결과는 다음과 같습니다.

## 사이드 프로젝트: 제가 만들던 것

이 프로젝트는 간단하지만 사소하지는 않습니다. PostgreSQL 백엔드(backend)를 사용하는 Next.js 15 애플리케이션(application)으로, 여러 클로드 코드(Claude Code) 세션에서 다양한 저장소의 작업 완료 데이터를 집계하도록 설계되었습니다.

병렬 AI 코딩 에이전트를 실행하는 모든 사람을 위한 경량형 명령 센터(command center)라고 생각하면 됩니다.

스택(stack)은 다음과 같습니다. 타입스크립트 엄격 모드(TypeScript strict mode), Next.js 앱 라우터(App Router), 프리즈마 ORM(Prisma ORM), 테일윈드 CSS(Tailwind CSS), 버셀(Vercel)에 배포. 제가 이 실험을 시작할 당시 12개 디렉터리(directory)에 걸쳐 약 45개 파일이 있었습니다. 장난감 프로젝트는 아니지만 모놀리스(monolith)도 아니었습니다. 워크플로우 변경에 완벽한 테스트 기반이었습니다.

## 1일차: 컨텍스트 관리 및 병렬 세션

보리스의 근본적인 통찰은 얼핏 단순해 보입니다. 컨텍스트(context)는 가장 소중한 자원이며, 빠르게 저하됩니다.

그는 터미널(terminal)에서 1번부터 5번까지 번호가 매겨진 5개의 클로드 코드(Claude Code) 세션을 병렬로 실행하며, 세션이 입력을 필요로 할 때 시스템 알림을 받습니다.

또한 claude.ai에서 5~10개의 세션을 더 실행합니다. 총 몇 개냐고요? 때로는 15개의 동시 세션입니다.

저는 15개까지 가지는 않았습니다. 3개의 병렬 터미널 세션과 2개의 브라우저 세션으로 시작했습니다.

**제가 바꾼 것:** 계획, 구현, 테스트, 디버깅을 순차적으로 수행하는 하나의 긴 세션 대신, 각 세션에 단일 책임을 할당했습니다.

이는 제가 클로드 코드(Claude Code)와 싸우기를 멈추는 데 7개월이 걸렸다는 글에서 설명한 가드레일 시스템(guardrail system)의 직접적인 발전이었습니다. 다만 이제 가드레일은 행동적인 것이 아니라 구조적인 것이 되었습니다.

*   세션 1은 API 라우트(API route) 구현을 담당했습니다.
*   세션 2는 프런트엔드 대시보드 컴포넌트(frontend dashboard components) 작업을 수행했습니다.
*   세션 3은 전용 디버깅 및 테스트 세션이었습니다.

**어떤 일이 일어났나:** 첫 한 시간은 더 느리게 느껴졌습니다. 터미널 사이를 컨텍스트 스위칭(context-switching)하며 어떤 세션이 주의를 필요로 하는지 확인해야 했습니다. 두 번째 시간에는 무언가가 연결되었습니다.

세션 1이 API 라우트 작업을 마치는 동안 세션 2는 여전히 차트 컴포넌트(chart components)를 반복하고 있었습니다.

기다릴 필요가 없었습니다. 병목 현상(bottleneck)은 "클로드가 생각 중입니다"에서 "이 결과물을 검토해야 합니다"로 바뀌었습니다.

**수치:** 이 실험 전에는 평범한 날에 집중적인 코딩 세션에서 대략 3~4개의 의미 있는 기능 또는 수정 사항을 완료했습니다.

1일차에는 병렬 세션을 사용하여 API 계층, 두 개의 대시보드 뷰(dashboard view), 데이터 집계 서비스(data aggregation service), 데이터베이스 스키마 마이그레이션(database schema migration), 그리고 웹소켓 알림 시스템(WebSocket notification system)을 포함한 6개의 완성된 컴포넌트를 제공했습니다.

**소요 시간:** 4.5시간의 실제 작업. 이는 처리량(throughput)이 50%에서 75% 증가한 수치이지만, 1일차는 프로젝트의 신선한 추진력이라는 이점도 있었기 때문에 이 수치에 대해서는 조심해야 합니다.

**솔직한 부분:** 저는 5개 세션 중 하나를 완전히 포기했습니다. 제대로 범위가 지정되지 않은 캐싱 계층(caching layer)을 구현하려고 시도하다가 엉뚱한 방향으로 빠졌기 때문입니다. 보리스도 자신의 세션 중 10%에서 20%를 버린다고 인정합니다. 그것이 옳게 느껴졌습니다. 나쁜 세션을 조기에 중단하는 것이 그것을 계속 끌고 가는 것보다 비용이 적게 듭니다.

## 2일차: 플랜 모드(Plan Mode)가 모든 것을 바꾸었다

이 날은 저의 전체적인 사고방식을 바꾼 날이었습니다.

보리스는 이에 대해 단호합니다. 모든 복잡한 작업은 플랜 모드(plan mode)에서 시작해야 합니다. 예외는 없습니다. 그의 워크플로우(workflow)는 구체적입니다. Shift+Tab으로 플랜 모드에 진입하고, 계획이 확고해질 때까지 클로드와 왔다 갔다 한 다음, 자동 승인 모드(auto-accept mode)로 전환합니다.

그의 주장:
> 좋은 계획 후에는 클로드가 일반적으로 한 번에 구현을 완료할 수 있다.

저는 이전에 플랜 모드를 사용해 본 적이 있었습니다. 가끔씩. 기억이 날 때마다요. 하지만 그것을 필수로 취급한 적은 없었습니다.

**테스트:** 저는 대시보드(dashboard)와 에이전트 작업 데이터(agent task data) 간의 실시간 동기화(real-time synchronization)를 구축해야 했습니다.

여기에는 웹소켓 연결(WebSocket connections), 데이터베이스 폴링(database polling), 프런트엔드(frontend)의 상태 관리(state management), 오류 복구(error recovery)가 포함되었습니다. 일반적으로는 2~3시간 동안 왔다 갔다 하는 수정 작업을 거쳐야 하는 다중 파일, 다중 관심사 기능입니다.

저는 스스로를 먼저 플랜 모드로 강제했습니다. 아키텍처(architecture)에 대해 클로드와 18분 동안 논의했습니다. 우리는 서버 센트 이벤트(Server-Sent Events, SSE)를 사용할지 웹소켓(WebSockets)을 사용할지 논쟁했습니다.

클로드는 제가 고려하지 못했던 폴백(fallback) 패턴을 제안했습니다. 즉, 기본적으로 SSE를 사용하고 브라우저가 지원하는 경우 웹소켓 업그레이드(WebSocket upgrade)를 수행하는 것입니다. 우리는 건드릴 모든 파일, 모든 함수 시그니처(function signature), 모든 오류 상태를 계획했습니다.

그런 다음 자동 승인 모드(auto-accept mode)로 전환했습니다.

클로드는 전체 기능을 한 번에 구현했습니다. 11개 파일이 수정되었습니다. 첫 빌드(build)에서 오류는 0개였습니다. 계획에서 정의했던 테스트(test)는 첫 실행에서 모두 통과했습니다.

**시간 비교:**
계획 18분 + 구현 12분 + 검토 5분 = 총 35분.
동일한 기능 범위가 규율 있는 계획 없이 일반적으로 수정 작업을 포함하여 2~3시간이 걸렸습니다.
이는 대략 75%에서 80%의 시간 절감 효과입니다.

**효과가 없었던 경우:** 작은 작업들. 단일 툴팁 컴포넌트(tooltip component)를 추가하기 위해 플랜 모드를 사용하려고 했을 때, 오버헤드(overhead)가 가치가 없었습니다. 보리스도 이 점을 언급합니다. 변경 사항을 한 문장으로 설명할 수 있다면 계획을 건너뛰세요.

2일차에 30초짜리 변경 사항을 계획하는 데 8분을 낭비하며 그 임계값을 어렵게 배웠습니다.

## 3일차: 살아있는 기억으로서의 CLAUDE.md

보리스의 팀은 CLAUDE.md를 살아있는 문서로 취급합니다. 전체 팀이 매주 여러 번 기여합니다. 클로드가 실수를 하면 문서화합니다. 코드 리뷰(code review) 중에는 클로드에게 가이드라인(guideline)을 자동으로 업데이트하도록 태그(tag)를 지정합니다.

그의 황금률: 모든 수정 후에는 프롬프트(prompt)를 다음으로 끝내라.
>"다음에 같은 실수를 하지 않도록 CLAUDE.md를 업데이트해."

이 실험 전 저의 CLAUDE.md는 847줄이었습니다. 이것은 문제입니다. 저는 CLAUDE.md 항목의 강력함에 대해 광범위하게 글을 썼지만, '많을수록 좋다'는 함정에 빠져 있었습니다.

보리스의 팀은 그들의 CLAUDE.md를 약 2,500 토큰(tokens)으로 유지합니다.

앤트로픽(Anthropic)의 모범 사례 문서(best practices documentation)는 다음과 같이 명시적으로 경고합니다.
> ...만약 클로드가 계속해서 규칙을 무시한다면, 파일이 너무 길어서 그 규칙이 잊히고 있을 가능성이 높다.

**정리:** 3일차 아침에 40분을 들여 CLAUDE.md를 대폭 정리했습니다. 클로드가 기본적으로 이미 올바르게 따르는 모든 지시사항—표준 타입스크립트(TypeScript) 규칙, 기본 형식 지정 규칙, 클로드가 훈련을 통해 알고 있는 것들—을 제거했습니다.

길었던 설명을 간결한 규칙으로 대체했습니다. 코드베이스(codebase)의 파일별 설명을 삭제했습니다.

**결과:** 847줄이 127줄이 되었습니다. 500 토큰(tokens) 미만.

**복리 효과:** 정리 후, 저는 "실수를 규칙으로" 만드는 연습을 시작했습니다. 클로드가 제가 수정한 어떤 작업을 할 때마다—명명된 내보내기(named export) 대신 기본 내보내기(default export)를 사용하거나, 테스트 설정이 요구하는 비동기 래퍼(async wrapper) 없이 테스트를 작성하거나, 잘못된 경로 별칭(path alias)에서 임포트(import)하는 등—클로드에게 CLAUDE.md를 업데이트하라고 지시했습니다.

3일차와 4일차에 걸쳐 CLAUDE.md는 127줄에서 156줄로 늘어났습니다. 하지만 모든 새로운 줄은 이론적인 가이드라인이 아니라 실제 수정 사항을 나타냈습니다. 4일차 오후가 되자 클로드는 반복적으로 저지르던 모든 실수를 멈추었습니다.

**중요한 지표:** 1일차에는 평균 3~4번의 상호작용마다 한 번씩 클로드를 수정했습니다. 4일차에는 그 횟수가 8~10번의 상호작용마다 한 번으로 줄어들었습니다. CLAUDE.md는 복리 효과를 내고 있었습니다.

## 4일차: 서브 에이전트, 슬래시 명령어, 그리고 검증

마지막 날은 인프라(infrastructure) 구축에 관한 것이었습니다. 즉, 모든 미래 세션을 더 좋게 만드는 시스템을 구축하는 것이었습니다.

### 컨텍스트 보호를 위한 서브 에이전트 (Subagents for Context Protection)

보리스는 전문화된 서브 에이전트(subagents)를 사용합니다. 정리(cleanup)를 위한 코드 심플리파이어(Code Simplifier), 종단간 테스트(end-to-end testing)를 위한 베리파이 앱(Verify App), 빌드 통과(build pass)를 보장하는 빌드 밸리데이터(Build Validator). 핵심 통찰은 서브 에이전트가 강력하다는 것이 아닙니다. 저는 이전에 서브 에이전트 아키텍처(subagent architectures)에 대해 썼고, 심지어 CTO 팀 전체를 서브 에이전트로 구축하기도 했습니다.

저는 `.claude/agents/`에 세 개의 커스텀 에이전트(custom agents)를 만들었습니다. 이는 저의 `claude-code-tresor` 저장소의 패턴을 기반으로 하지만, 보리스 수준의 단순함으로 축소했습니다.

```markdown
# .claude/agents/code-reviewer.md
---
name: code-reviewer
description: 타입 안전성, 오류 처리 및 아키텍처 일관성을 위해 코드를 검토합니다.
model: opus
tools: ["read", "grep", "glob"]
---

다음 변경 사항에 대해 검토해 주세요:
1. TypeScript 엄격 모드 위반 사항
2. 누락된 오류 경계 (error boundaries)
3. 기존 패턴과 일치하지 않는 이름 지정
4. 테스트되지 않은 엣지 케이스 (edge cases)
구체적으로 언급해 주세요. 줄 번호를 참조하세요. 수정 사항을 제안해 주세요.
```

클로드에게 복잡한 PR(Pull Request) 수준의 변경 사항을 검토하도록 요청해야 할 때, 주 구현 컨텍스트(main implementation context)를 오염시키지 않고 코드 리뷰어 에이전트(code-reviewer agent)를 불러낼 수 있었습니다.

주요 세션은 계속 집중했습니다. 리뷰어 에이전트(reviewer agent)는 작업을 수행하고 결과를 반환했습니다.

**저를 놀라게 한 것:** 서브 에이전트 패턴(subagent pattern)은 리뷰 자체보다는 주 컨텍스트를 깨끗하게 유지하는 데 가장 가치 있었습니다.

서브 에이전트 없이 3시간 작업한 후에는 보통 주 컨텍스트가 60~70% 용량이었습니다. 서브 에이전트가 조사 및 검증을 처리하면서, 4일차에는 주 세션의 컨텍스트 사용량이 약 40%로 끝났습니다.

### 반복 작업을 위한 슬래시 명령어 (Slash Commands for Repetitive Workflows)

보리스는 매일 수십 번 `/commit-push-pr`을 사용합니다. 원칙은 이렇습니다. 하루에 두 번 이상 하는 작업이라면 슬래시 명령어(slash command)로 만드세요.

이는 제가 10개의 커스텀 명령어 라이브러리(custom command library)를 구축했을 때 발견한 것과 완벽하게 일치하지만, 보리스의 접근 방식은 더 간결하고 집중적입니다.

저는 이 프로젝트를 위해 세 가지 명령어를 만들었습니다.

```markdown
# .claude/commands/sync-and-test.md
다음 순서로 실행합니다:
1. `npm run typecheck`을 실행하고 오류를 보고합니다.
2. `npm run test`를 실행하고 실패를 보고합니다.
3. 모든 것이 통과하면 `npm run build`를 실행하여 프로덕션 빌드를 확인합니다.
4. 결과를 한 단락으로 요약합니다.
```

이전에는 30~45분마다 테스트 및 빌드 지침을 수동으로 입력했습니다. 슬래시 명령어는 이를 세 번의 키 입력으로 줄여주었습니다.

### 협상 불가능한 검증 (Verification as Non-Negotiable)

보리스는 검증을 "훌륭한 결과를 얻기 위한 아마도 가장 중요한 것"이라고 부릅니다. 그의 팀은 UI 테스트를 위해 크롬용 클로드 확장 프로그램(Claude in Chrome extension)을 사용합니다. 그들은 테스트 스위트(test suites)를 자동으로 실행합니다. 검증 없이는 아무것도 배포되지 않습니다.

저는 모든 파일 편집 후에 타입스크립트 컴파일러(TypeScript compiler)를 실행하는 `PostToolUse` 훅(hook)을 추가했습니다.

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "npx tsc --noEmit 2>&1 | head -20 || true"
          }
        ]
      }
    ]
  }
}
```

이것은 타입 오류를 즉시 잡아내어, 나중에 더 큰 버그로 이어지는 것을 방지합니다. 4일차에 이 훅은 나중에 더 큰 버그로 이어질 수 있었던 4개의 타입 불일치(type mismatch)를 잡아냈습니다.

## 아직 알아내는 중인 것

이 실험은 4일 동안 진행되었습니다. 패턴을 파악하기에는 충분하지만, 영구적인 결론을 내리기에는 충분하지 않습니다. 제가 아직 확실하지 않은 부분은 다음과 같습니다.

*   **워크트리(Worktrees) vs. 개별 체크아웃(separate checkouts).** 보리스와 그의 팀은 병렬 세션(parallel sessions)을 위해 깃 워크트리(git worktrees)를 선호합니다. 저는 같은 체크아웃(checkout)에서 별도의 터미널 탭(terminal tabs)을 사용했는데, 두 세션이 동시에 같은 유틸리티 파일(utility file)을 편집할 때 한 번 충돌이 발생했습니다. 이것은 제가 클로드 코드(Claude Code)를 위한 깃허브 워크플로우(GitHub workflow)로 부분적으로 해결한 문제입니다.

    다음 스프린트(sprint)를 위해 워크트리를 제대로 설정해야 하지만, 중간 규모 프로젝트의 솔로 개발자(solo developer)에게 여러 작업 디렉터리(working directories)를 관리하는 오버헤드(overhead)가 그만한 가치가 있는지는 테스트하지 않았습니다.

*   **10~20%의 세션 포기율.** 보리스는 세션의 10~20%를 포기한다고 말합니다. 4일 동안 저는 약 15%를 포기했습니다. 하지만 제가 충분히 빨리 포기하고 있는지는 확실하지 않습니다.

    포기한 세션 중 두 개는 제가 종료하기 전에 20분 이상을 소모했습니다. 이상적인 것은 실패하는 세션을 5분 이내에 인지하는 것이라고 생각하며, 저는 아직 그 수준에 도달하지 못했습니다.

*   **컨텍스트 윈도우 모니터링.** 보리스가 권장하는 항상 보이는 컨텍스트 사용량을 위한 `/status` 줄 사용자 정의를 설정하지 않았습니다. 그랬어야 했습니다. 클로드가 낮은 품질의 답변을 주기 시작했을 때에야 컨텍스트가 저하되었음을 여러 번 깨달았습니다. 그 때는 이미 토큰(tokens)을 낭비한 후였습니다.

*   **플러그인(Plugins) 및 마켓플레이스(Marketplace).** 보리스의 2026년 2월 팁에서는 플러그인 시스템—LSP(Language Server Protocol), MCP(Modular Contextual Prompt), 스킬(skills), 그리고 마켓플레이스에서 설치할 수 있는 에이전트—이 소개되었습니다. 저는 4일 동안 이 부분을 전혀 탐색하지 않았습니다. 이것이 다음 과제입니다.

## 보리스가 놓치지 않는, 대부분의 가이드가 놓치는 점

현재 미디엄(Medium)과 인터넷 전반에는 수십 개의 클로드 코드(Claude Code) "모범 사례" 기사가 떠돌고 있습니다. 대부분은 문서 요약처럼 읽힙니다. 보리스의 접근 방식이 다른 점은 팁 아래에 깔린 철학입니다.

그는 클로드 코드(Claude Code)를 대화하는 챗봇(chatbot)으로 보지 않습니다. 그는 그것을 스케줄링 가능한 역량(capacity you schedule)으로 봅니다. 각 세션은 고유한 컨텍스트(context)를 가진 별도의 작업자이며, 모든 것을 담당해야 하는 단일 비서가 아닙니다. 이러한 재정의(reframe)는 작업 방식을 조직하는 모든 것을 바꿉니다.

"놀라울 정도로 평범한(surprisingly vanilla)" 설정은 대부분의 사람들이 놓치는 또 다른 통찰입니다. 보리스는 클로드 코드(Claude Code)를 크게 사용자 정의하지 않습니다. 그는 수십 개의 MCP 서버(MCP servers)로 구성된 정교한 시스템을 가지고 있지 않습니다.

그는 2,000줄짜리 CLAUDE.md를 가지고 있지 않습니다.

힘은 기본 원칙(fundamentals)을 규율 있게 적용하는 것에서 나옵니다.
> ...코딩하기 전에 계획하고, 작업을 검증하고, 컨텍스트를 깨끗하게 유지하고, 학습을 복리화(compound)하라.

4일간 이를 테스트한 결과, 저는 가장 높은 레버리지(high-leverage) 변화가 바로 **플랜 모드(plan mode) 규율**이라고 믿습니다. 플랜 모드가 비밀스러운 기능이기 때문이 아니라—모두가 그것이 존재한다는 것을 압니다.

하지만 모든 사소하지 않은 작업을 계획하도록 강제하는 것이 가장 비싼 실패 모드를 제거하기 때문입니다.
> ...잘못된 것을 만들고 나서 중간에 방향을 수정하는 것.

## 오늘 당장 적용할 수 있는 것

이 글에서 한 가지를 얻어가야 한다면, 이것을 명심하세요. CLAUDE.md를 500 토큰(tokens) 미만으로 정리하고 오늘부터 "실수를 규칙으로" 만드는 연습을 시작하세요.

모든 수정은 영구적인 개선이 됩니다. 일주일이 지나면 클로드가 반복적인 실수를 덜 하는 것을 볼 수 있을 것입니다. 한 달이 지나면 복리 효과는 혁신적일 것입니다.

세 가지를 얻어가야 한다면:

1.  **CLAUDE.md를 무자비하게 정리하세요.** 클로드가 이미 알고 있는 모든 것을 제거하세요. 프로젝트별 규칙과 학습된 수정 사항만 유지하세요.
2.  **2개 이상의 파일에 영향을 미치는 모든 작업에 플랜 모드(plan mode)를 사용하세요.** 15~20분의 계획 투자는 1~2시간의 수정 작업을 절약합니다.
3.  **최소 2개의 병렬 세션(parallel sessions)을 실행하세요.** 각 세션에 단일 책임을 할당하세요. 나쁜 세션은 조기에 종료하세요.

나머지—서브 에이전트(subagents), 훅(hooks), 슬래시 명령어(slash commands), 워크트리(worktrees)—는 그 기반 위에 구축됩니다. 기본을 먼저 제대로 잡으세요. 보리스가 그랬습니다. 그의 설정은 평범합니다. 하지만 결과는 그렇지 않습니다.

저는 여전히 반복 작업을 하고 있습니다. 다음 주에는 적절한 깃 워크트리(git worktrees)를 설정하고 보리스가 2월 스레드에서 소개한 플러그인 마켓플레이스(plugin marketplace)를 탐색할 예정입니다.

대시보드 프로젝트는 이달 말까지 프로덕션(production)에 배포될 예정입니다. CLAUDE.md 연습의 복리 효과가 지속된다면, 그때쯤 클로드는 일주일 온보딩(onboarding) 후 대부분의 인간 개발자가 알게 되는 것보다 이 코드베이스(codebase)를 더 잘 알게 될 것입니다.

그것이 바로 보리스가 계속해서 지적하는 진정한 약속입니다. AI가 코드를 작성하는 것이 아니라, 개발 환경이 매 세션마다 더 똑똑해진다는 것입니다. 4일이 지난 지금, 저는 그것을 보기 시작했습니다.
