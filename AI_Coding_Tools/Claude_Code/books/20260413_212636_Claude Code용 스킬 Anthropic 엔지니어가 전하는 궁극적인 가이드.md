# Claude Code용 스킬: Anthropic 엔지니어가 전하는 궁극적인 가이드

스킬은 클로드 코드(Claude Code)에서 가장 인기 있는 확장 지점(extension point) 중 하나로 빠르게 자리 잡았습니다. 유연하고 생성하기 쉬우며 공유도 간편합니다.

하지만 이러한 유연성 때문에 스킬을 이해하기 어렵기도 합니다. 어떤 방식이 가장 효과적일까요? 어떤 스킬을 구축해야 할까요? 좋은 스킬은 어떻게 작성하며, 언제 다른 사람들과 공유하는 것이 합리적일까요?

Anthropic에서는 클로드 코드 내에서 스킬을 광범위하게 사용하고 있으며, 현재 수백 개의 스킬이 프로덕션 환경에서 운영되고 있습니다. 아래는 그 경험을 통해 얻은 교훈들입니다.

## 스킬이란 무엇인가?

흔한 오해는 스킬이 "단지 마크다운(Markdown) 파일"이라는 것입니다. 실제로는 가장 강력한 부분은 스킬이 폴더라는 점입니다. 스크립트, 자산(asset), 데이터 등을 포함할 수 있으며, 에이전트(agent)는 이 모든 것을 발견하고 탐색하며 사용할 수 있습니다.

클로드 코드에서 스킬은 동적 훅(dynamic hook)을 포함한 풍부한 구성 옵션을 지원합니다. 가장 흥미로운 스킬들은 이러한 옵션과 폴더 구조를 심도 있게 활용합니다.

## 스킬의 유형

내부 스킬들을 분석한 결과, 여러 반복적인 범주로 나뉜다는 것을 알게 되었습니다. 최고의 스킬들은 이 중 한 가지 범주에 명확하게 들어맞습니다. 여러 범주를 모호하게 넘나드는 스킬들은 에이전트와 사용자 모두를 혼란스럽게 하는 경향이 있습니다.

이 목록이 모든 것을 다루지는 않지만, 자신의 스킬 세트에서 놓치고 있을 수 있는 부분을 파악하는 데 도움이 될 것입니다.

1.  **라이브러리 및 API 참조 (Library & API References)**
    라이브러리, CLI(Command Line Interface), 또는 SDK(Software Development Kit)를 올바르게 사용하는 방법을 설명하는 스킬입니다. 내부 라이브러리나 클로드 코드가 다루기 어려워하는 공개 라이브러리를 대상으로 할 수 있습니다. 일반적으로 `examples/` 폴더와 알려진 함정(gotchas) 목록을 포함합니다.
    *   예시:
        *   `billing-lib`: 내부 결제 라이브러리의 예외 케이스 및 문제점
        *   `internal-platform-cli`: CLI의 모든 하위 명령어와 예시
        *   `frontend-design`: 디자인 시스템으로 클로드의 작업 개선

2.  **제품 검증 (Product Verification)**
    코드 동작을 테스트하고 검증하는 방법을 설명하는 스킬입니다. 종종 플레이라이트(Playwright), tmux 등과 같은 외부 도구와 함께 사용됩니다. 이러한 스킬은 정확성을 위해 매우 중요합니다. 한 명의 엔지니어가 일주일 동안 스킬을 다듬는 데 전념할 가치가 있을 때가 많습니다.
    *   기법: 시각적 검증을 위해 클로드의 출력 영상을 기록하고, 모든 단계에 프로그래밍 방식의 단언(assertion)을 추가합니다. 스킬 내부에 스크립트를 포함하세요.
    *   예시:
        *   `signup-flow-driver`: 헤드리스 브라우저에서 회원가입 → 이메일 확인 → 온보딩 흐름을 단언과 함께 실행
        *   `checkout-verifier`: 스트라이프(Stripe) 테스트 카드를 구동하고, 인보이스 상태를 확인
        *   `tmux-cli-driver`: TTY(Teletypewriter)가 필요한 대화형 CLI 테스트

3.  **데이터 검색 및 분석 (Data Retrieval & Analysis)**
    데이터 및 모니터링 스택에 연결되는 스킬입니다. 자격 증명, 대시보드 ID, 일반적인 워크플로우에 대한 지침을 사용하여 데이터를 가져오는 라이브러리를 포함할 수 있습니다.
    *   예시:
        *   `funnel-query`: 회원가입 → 활성화 → 결제를 위한 이벤트 조인 방법 및 정식 `user_id` 테이블
        *   `cohort-compare`: 통계적 유의미성으로 두 코호트(cohort) 간 유지율/전환율 비교
        *   `grafana`: 데이터 소스 UID, 클러스터 이름, 문제 → 대시보드 매핑

4.  **비즈니스 프로세스 및 팀 자동화 (Business Processes & Team Automation)**
    반복적인 워크플로우를 자동화하는 스킬입니다. 일반적으로 간단한 지침이지만, 다른 스킬이나 MCP(Master Control Program)에 의존할 수도 있습니다. 과거 실행 결과를 로그 파일에 저장하는 것이 유용합니다. 이는 모델이 일관성을 유지하고 이전 컨텍스트(context)를 기억하는 데 도움이 됩니다.
    *   예시:
        *   `standup-post`: 작업 추적기 + GitHub 활동 + Slack을 취합 → 형식화된 스탠드업 보고서 (변경 사항만)
        *   `create-ticket`: 스키마(열거형, 필수 필드) 및 사후 작업(검토자 핑, Slack 링크) 강제 적용
        *   `weekly-recap`: PR(Pull Request) + 종료된 티켓 + 배포를 결합 → 형식화된 주간 요약

5.  **코드 템플릿 및 스캐폴딩 (Code Templates & Scaffolding)**
    코드베이스 내 특정 기능에 대한 프레임워크 상용구(boilerplate)를 생성하기 위한 스킬입니다. 조합 가능한 스크립트와 결합할 수 있습니다. 스캐폴딩(scaffolding)에 코드만으로는 다룰 수 없는 자연어 요구사항이 포함될 때 특히 유용합니다.
    *   예시:
        *   `new-workflow`: 어노테이션(annotation)이 있는 새 서비스/워크플로우/핸들러 스캐폴딩
        *   `new-migration`: 마이그레이션 파일 템플릿 + 일반적인 문제점
        *   `create-app`: 사전 구성된 인증(auth), 로깅(logging), 배포 설정이 포함된 새 내부 애플리케이션 생성

6.  **코드 품질 및 코드 리뷰 (Code Quality & Code Review)**
    품질 표준을 강제하고 코드 리뷰를 지원하는 스킬입니다. 최대의 신뢰성을 위해 결정론적 스크립트나 도구를 포함할 수 있습니다. 훅(hook)이나 GitHub Actions를 통해 자동으로 실행될 수 있습니다.
    *   예시:
        *   `adversarial-review`: "신선한 시각"으로 비판할 하위 에이전트(sub-agent)를 생성하고, 수정 사항을 적용하며, 피드백이 사소해질 때까지 반복
        *   `code-style`: 클로드가 기본적으로 제대로 처리하지 못하는 스타일 강제
        *   `testing-practices`: 테스트 작성 방법 및 무엇을 테스트할지에 대한 지침

7.  **CI/CD 및 배포 (CI/CD & Deployment)**
    코드를 가져오고, 출시하고, 배포하기 위한 스킬입니다. 데이터 수집을 위해 다른 스킬을 참조할 수 있습니다.
    *   예시:
        *   `babysit-pr`: PR 모니터링 → 불안정한 CI 재시도 → 병합 충돌 해결 → 자동 병합 활성화
        *   `deploy-service`: 빌드 → 스모크 테스트(smoke test) → 오류율 비교를 통한 점진적 트래픽 전환 → 회귀(regression) 시 자동 롤백(rollback)
        *   `cherry-pick-prod`: 격리된 작업 트리(worktree) → 체리픽(cherry-pick) → 충돌 해결 → 템플릿에서 PR 생성

8.  **런북 (Runbooks)**
    증상(Slack 스레드, 경고, 오류 서명)을 받아 여러 도구를 통해 조사를 수행하고 구조화된 보고서를 생성하는 스킬입니다.
    *   예시:
        *   `service-debugging`: 가장 바쁜 서비스에 대한 증상을 도구 및 쿼리 패턴에 매핑
        *   `oncall-runner`: 경고 수신 → 일반적인 문제 확인 → 보고서 출력
        *   `log-correlator`: 요청 ID(request ID)가 주어지면 해당 요청을 처리한 모든 시스템에서 일치하는 로그 가져오기

9.  **인프라 운영 (Infrastructure Operations)**
    일상적인 유지 관리 및 운영 절차를 위한 스킬입니다. 일부는 파괴적인 작업을 포함할 수 있으므로, 안전 장치를 추가하는 것이 유용합니다. 이는 중요한 운영 중 모범 사례를 따르기 쉽게 만듭니다.
    *   예시:
        *   `resource-orphans`: 고아 파드(pods)/볼륨(volumes) 찾기 → Slack에 게시 → 쿨다운(cooldown) → 사용자 확인 → 연쇄 정리
        *   `dependency-management`: 내부 종속성 승인 프로세스
        *   `cost-investigation`: 스토리지/이그레스(egress) 비용이 증가한 이유 (특정 버킷 및 요청 패턴)

## 훌륭한 스킬을 구축하기 위한 팁

필요한 스킬 유형을 파악했다면, 다음 논리적인 질문은 "실제로 어떻게 작성해야 하는가?"입니다. 다음은 우리가 개발한 모범 사례들입니다.

### 뻔한 내용은 작성하지 마세요

클로드 코드는 이미 코드베이스에 대해 많은 것을 알고 있으며, 클로드는 일반적으로 코딩, 특히 많은 표준 접근 방식에 능숙합니다. 스킬은 클로드의 기본 사고방식을 넘어서는 정보에 집중해야 합니다.

*   좋은 예시: `frontend-design` 스킬. Anthropic 엔지니어는 클로드의 디자인 감각을 향상시키고, Inter 폰트나 보라색 그라데이션과 같은 진부한 표현을 피하기 위해 사용자들과 반복적으로 이 스킬을 구축했습니다.

### "함정(Gotchas)" 섹션을 포함하세요

어떤 스킬에서든 가장 가치 있는 내용은 `Gotchas` 섹션, 즉 흔한 실수들입니다. 이는 클로드가 스킬을 사용할 때 실제로 겪는 문제를 기반으로 해야 합니다. 이상적으로는 새로운 문제가 나타날 때마다 스킬을 업데이트해야 합니다.

### 파일 시스템 및 점진적 공개를 활용하세요

스킬은 단순히 마크다운 파일이 아니라 폴더입니다. 전체 파일 시스템을 컨텍스트 엔지니어링 도구로 취급하세요. 클로드에게 스킬 내에 어떤 파일이 존재하는지 알려주면, 클로드는 필요할 때 해당 파일을 읽을 것입니다.

가장 간단한 형태는 다른 마크다운 파일에 대한 링크입니다(예: 자세한 함수 시그니처를 위한 `references/api.md`). 최종 출력을 위해서는 `assets/`에 복사할 템플릿을 포함할 수 있습니다.

참조 문서, 스크립트, 예제가 있는 폴더는 클로드가 훨씬 효율적으로 작업하는 데 도움이 됩니다.

### 클로드를 엄격한 규칙에 묶어두지 마세요

클로드는 지침을 따르기 위해 노력하며, 스킬은 자주 재사용되기 때문에 지나치게 구체적인 지침은 역효과를 낼 수 있습니다. 클로드에게 필요한 정보를 제공하되, 각 상황에 적응할 수 있는 유연성을 남겨두세요.

### 초기 설정 계획하기

일부 스킬은 시작 시 사용자 제공 컨텍스트(context)가 필요합니다. 예를 들어, 스탠드업(standup)을 게시할 Slack 채널이 있습니다.

*   좋은 패턴: 스킬 디렉토리 내 `config.json`에 설정을 저장합니다. 구성 파일이 없으면 에이전트가 사용자에게 질문합니다. 선택지가 있는 구조화된 질문이 필요한 경우 `AskUserQuestion` 도구를 사용하세요.

### `description` 필드는 모델을 위한 것입니다

세션이 시작되면 클로드 코드는 사용 가능한 모든 스킬 목록과 그 설명을 수집합니다. 클로드는 이 목록을 사용하여 "이 요청에 적합한 스킬이 있는가?"를 결정합니다. `description`은 요약이 아니라, 스킬이 트리거되어야 하는 조건 집합입니다.

### 메모리 및 데이터 영속성 (Memory & Data Persistence)

일부 스킬은 자체 내부에 데이터를 저장하여 메모리 저장소 역할을 합니다. 이는 추가 전용 텍스트 로그(append-only text log)나 JSON 파일부터 SQLite 데이터베이스까지 다양합니다.

예를 들어, `standup-post` 스킬은 `standups.log` 파일을 유지할 수 있습니다. 다음 실행 시 클로드는 이 기록을 읽어 어제 이후 변경된 내용을 파악합니다.

*   ⚠️ **주의**: 스킬 디렉토리 내의 데이터는 업데이트 시 삭제될 수 있습니다. 플러그인당 안정적인 디렉토리로 `${CLAUDE_PLUGIN_DATA}`를 사용하세요.

### 스크립트를 유지하고 코드를 생성하세요

클로드의 가장 강력한 도구 중 하나는 코드 그 자체입니다. 스크립트와 라이브러리는 클로드가 상용구(boilerplate)를 다시 구현하는 대신, 다음에 무엇을 할지 결정하는 '구성'에 더 많은 단계를 할애하게 합니다.

데이터 과학 스킬의 경우, 이벤트 소스에서 데이터를 가져오는 함수 라이브러리를 포함하세요. 복잡한 분석을 위해서는 클로드에게 도우미 함수(helper functions)를 제공하세요. 그러면 클로드는 "화요일에 무슨 일이 일어났나?"와 같은 프롬프트에 맞춰 해당 함수들을 구성하여 스크립트를 즉석에서 생성할 것입니다.

### 온디맨드 훅 (Hooks on Demand)

스킬은 스킬이 호출될 때만 활성화되고 세션 동안 활성 상태를 유지하는 훅(hook)을 포함할 수 있습니다. 이는 가끔은 중요하지만 항상 켜져 있으면 번거로울 수 있는 선택적 훅에 사용됩니다.

*   예시:
    *   `/careful`: 프로덕션 작업 시에만 필요한 Bash의 `PreToolUse` 매처(matcher)를 통해 `rm -rf`, `DROP TABLE`, `force-push`, `kubectl delete`를 차단합니다.
    *   `/freeze`: 지정된 디렉토리 외부에서의 모든 편집/쓰기(Edit/Write)를 차단합니다 (로그를 추가하고 싶지만 우발적인 변경을 피하고 싶을 때 디버깅에 매우 유용).

## 스킬 공유

스킬의 주요 이점 중 하나는 팀과 공유할 수 있다는 것입니다.

두 가지 주요 방법:
*   스킬을 저장소(`/.claude/skills`)에 커밋(commit)합니다.
*   플러그인을 생성하고 클로드 코드 플러그인 마켓플레이스(plugin marketplace)를 사용합니다.

소규모 팀이 적은 저장소에서 작업할 때는 스킬을 저장소에 커밋하는 것이 잘 작동합니다. 하지만 커밋된 스킬은 각각 모델의 컨텍스트(context)를 약간 증가시킵니다. 대규모 환경에서는 내부 플러그인 마켓플레이스를 통해 스킬을 배포하고 팀에게 어떤 스킬을 설치할지 선택권을 줄 수 있습니다.

### 마켓플레이스 관리

어떤 스킬이 마켓플레이스에 포함될지 어떻게 결정하나요? 그리고 사람들은 어떻게 스킬을 제안하나요?

이러한 결정을 위한 중앙 집중식 팀은 없습니다. 우리는 유용한 스킬을 유기적으로 발견합니다. 동료들에게 제공하고 싶은 스킬이 있다면, GitHub의 샌드박스(sandbox) 폴더에 업로드하고 Slack이나 포럼에서 링크를 공유하세요.

스킬이 인기를 얻으면 (작성자가 결정), PR(Pull Request)을 열어 공식 마켓플레이스로 옮길 수 있습니다.

*   **경고**: 품질이 낮거나 중복되는 스킬을 생성하기 쉽기 때문에 출시 전에 일부 큐레이션(curation) 메커니즘이 중요합니다.

### 스킬의 구성 (Composition of Skills)

때로는 스킬들이 서로 의존하기도 합니다. 예를 들어, 파일을 업로드하는 스킬과 CSV 파일을 생성하여 업로드하는 CSV 생성 스킬이 있을 수 있습니다. 종속성 관리(dependency management)는 아직 마켓플레이스나 스킬에 내장되어 있지 않지만, 다른 스킬을 이름으로 참조할 수 있습니다. 그러면 모델은 해당 스킬이 설치되어 있을 경우 호출할 것입니다.

### 스킬 사용량 측정

스킬 채택을 평가하기 위해, 우리는 회사 내에서 스킬 사용량을 기록하는 `PreToolUse` 훅(hook)을 사용합니다 (문서에서 샘플 코드 사용 가능). 이 접근 방식은 인기 있는 스킬이나 예상보다 적게 사용되는 스킬을 파악하는 데 도움이 됩니다.

## 결론

스킬은 강력하고 유연한 에이전트 프리미티브(agentic primitive)입니다. 하지만 우리는 아직 초기 단계이며, 전체 커뮤니티가 함께 모범 사례를 찾아가고 있습니다.

이 글은 최종 규칙집이 아니라, 우리에게 효과적인 것들을 모아놓은 것으로 생각해 주세요. 스킬을 배우는 가장 좋은 방법은 작게 시작하고, 실험하며, 자신의 워크플로우에 무엇이 맞는지 확인하는 것입니다. 대부분의 내부 스킬은 몇 줄의 코드와 하나의 "함정"으로 시작하여, 클로드가 새로운 예외 상황을 만날 때마다 사람들이 추가하면서 개선되었습니다.
