# Claude Code로 프로덕션 앱 구축: 실용 가이드

스마트한 자동 완성 기능을 기대하며 Claude Code를 사용하기 시작했습니다. 하지만 제가 얻은 것은 새로운 것을 가르칠 때마다 생산성이 향상되는 주니어 개발자에 더 가까웠습니다.

Claude Code에 스택(stack)과 워크플로우를 더 많이 가르칠수록, 생산성과 정확성이 더욱 높아집니다.

이 글은 제가 Claude Code를 사용하는 방식에 가장 큰 변화를 가져온 핵심 구성 요소, 주의할 점, 그리고 습관들을 담고 있습니다.

## 1. 서론

Claude Code는 AI 기반의 에이전트형(agentic) 코딩 도구입니다. 우리가 원하는 것을 일반 언어로 설명하면, Claude Code는 접근 방식을 계획하고, 여러 파일에 걸쳐 코드를 작성하며, 작동 여부를 검증합니다.

Claude는 전체 개발 워크플로우를 자동화하는 데 사용될 수 있습니다. Claude Code를 활용하여 테스트되지 않은 코드에 대한 테스트 작성, 프로젝트 전반의 린트(lint) 오류 수정, 병합 충돌(merge conflicts) 해결, 의존성(dependencies) 업데이트, 릴리스 노트 작성, 기능 구축, 작업 스케줄링, 버그 수정과 같은 반복적인 작업을 처리할 수 있습니다. 또한 주제를 연구하고 Slack, GitHub, 데이터베이스와 같은 외부 서비스에 연결할 수 있습니다.

아침 PR 검토, 야간 CI 실패 분석, 주간 의존성 감사, 병합 후 문서 동기화와 같은 반복적인 작업을 스케줄링할 수도 있습니다.

Claude Code는 지침, 워크플로우, 규칙, 스킬(skills), 플러그인(plugins), 명령어, 훅(hooks)을 추가하여 확장할 수 있습니다. 이 글에서는 프로젝트에 Claude Code를 확장하는 방법을 설명할 것입니다.

## 2. 글의 목적

이 글은 Claude Core 확장(extension)의 핵심 구성 요소를 소개하는 것으로 시작합니다.

그다음으로, 영구적인 지침을 위한 CLAUDE.md, 재사용 가능한 워크플로우를 위한 스킬(skills), 병렬 작업을 위한 서브에이전트(subagents), 자동화를 위한 훅(hooks), 그리고 팀 간 설정 공유를 위한 플러그인(plugins)에 대해 설명합니다.

마지막으로, Claude Code를 최대한 활용하기 위한 실용적인 팁을 제시하는 것을 목표로 합니다.

## 3. Claude Code로 무엇을 할 수 있는가

Claude Code를 무엇에 활용할 수 있는지 간략하게 알아보겠습니다. Claude Code는 다음 용도로 사용될 수 있습니다:

*   우리의 워크플로우에 맞춰진 AI 에이전트 구축
*   문서화, 배포, 코드 리뷰, 실시간 웹 애플리케이션 디버깅 자동화, Git 작업 수행(스테이징, 커밋 메시지, PR), Slack, Discord, Telegram으로부터 이벤트 수신, MCP(Multi-Agent Communication Protocol)를 통해 외부 도구 연결(Google Drive 문서 읽기, Jira 티켓 업데이트, 데이터베이스, API 및 내부 도구 접근).
*   반복 작업을 자동화하기 위해 로컬 또는 Anthropic 관리 인프라에서 스케줄링된 작업 실행.

Claude Code는 또한 기존 코드베이스를 분석하고, 프로젝트의 패턴과 규칙에 맞는 테스트를 생성하며, 완료되면 풀 리퀘스트(pull request)를 열 수 있습니다:

*   프로젝트 루트 디렉토리로 이동
*   `claude`를 입력하여 Claude Code 시작
*   높은 수준의 개요 요청, 데이터 모델 및 인증과 같은 특정 구성 요소 심층 분석, 관련 파일 찾기, 구성 요소 상호 작용 방식 파악, 실행 흐름 이해, 코드베이스 리팩토링(refactoring).

저는 VS Code 확장(extension)을 사용해왔습니다. 이 확장은 인라인 차이점(inline diffs), @-멘션(@-mentions), 계획 검토, 대화 기록을 편집기에서 직접 제공합니다.

터미널, 데스크톱 앱, 웹, 그리고 IntelliJ IDEA, PyCharm, WebStorm 및 기타 JetBrains IDE용 플러그인을 사용할 수도 있습니다.

## 4. Claude Code 확장하기

별도 설정 없이 Claude Code는 일반적인 방식으로 작동합니다. 확장 기능(extensions)을 통해 Claude에게 우리 팀의 작업 방식을 가르치고, 실제 도구 및 데이터베이스에 연결하며, 다단계 워크플로우를 팀원 누구나 실행할 수 있는 단일 명령어로 전환할 수 있습니다.

그 결과, 매 세션마다 스택, 규칙, 프로세스를 이미 알고 있는 코딩 어시스턴트(coding assistant)를 얻을 수 있습니다.

이 섹션에서는 핵심 Claude Code 확장 기능을 소개합니다:

*   **CLAUDE.md**: Claude가 모든 세션에서 읽는 영구적인 컨텍스트(context)를 유지하는 곳입니다. 규칙(Rules)은 CLAUDE.md 지침의 범위가 지정된 버전입니다. 모든 것을 하나의 큰 CLAUDE.md 파일에 넣는 대신, `.claude/rules/` 내부에 별도의 파일로 지침을 구성하고 특정 파일 유형 또는 하위 디렉토리를 대상으로 지정할 수 있습니다.
*   **스킬(Skills)**: 재사용 가능한 온디맨드(on-demand) 지식 및 호출 가능한 워크플로우를 추가합니다.
    *   팀이 공유할 수 있는 반복 가능한 워크플로우를 패키징하기 위해 `/review-pr` 또는 `/deploy-staging`과 같은 사용자 지정 명령어를 생성할 수 있습니다.
*   **MCP**: Claude를 외부 서비스 및 도구에 연결합니다.
*   **서브에이전트(Subagents)**: 작업을 서브에이전트에 위임합니다. 이 에이전트들은 격리된 컨텍스트에서 자체 루프를 실행하고 요약을 반환합니다.
*   **에이전트 팀(Agent teams)**: 공유된 작업과 P2P 메시징(peer-to-peer messaging)을 통해 여러 독립적인 세션을 조정합니다.
*   **훅(Hooks)**: 루프 외부에서 완전히 결정론적 스크립트(deterministic scripts)로 실행됩니다. 훅은 Claude의 워크플로우 특정 지점에서 스크립트를 자동으로 실행합니다. 주요 이벤트: SessionStart, PreToolUse, PostToolUse, Notification, Stop, PermissionRequest.
*   **플러그인(Plugins)**: 이러한 기능을 패키징하고 배포합니다.

이러한 확장 기능이 프로젝트 폴더에 위치하는 폴더 구조는 다음과 같습니다:

```
your-project/
├── CLAUDE.md                          # Claude가 모든 세션에서 읽는 영구적인 컨텍스트
│
└── .claude/
    ├── settings.json                  # MCP 서버, 훅, 권한, 허용 목록
    ├── rules/                         # 파일 유형 또는 하위 디렉토리별 범위 지정 지침
    │   ├── code-style.md
    │   ├── testing.md
    │   └── security.md
    ├── commands/                      # 사용자 지정 슬래시 명령어 (예: /review-pr, /deploy-staging)
    │   ├── review-pr.md
    │   └── deploy-staging.md
    ├── skills/                        # 재사용 가능한 온디맨드 지식 및 워크플로우
    │   └── api-conventions/
    │       └── SKILL.md
    └── agents/                        # 프로젝트 서브에이전트 및 에이전트 팀
    |  └── security-reviewer.md
    └── mcp-servers.json         ← MCP 통합(integrations)
```

```
# 사용자 레벨 — 모든 프로젝트에서 사용 가능
~/.claude/
├── settings.json                      # 전역 MCP, 훅, 권한
├── skills/
│   └── <skill-name>/
│       └── SKILL.md
└── agents/
    └── <agent-name>.md
```

```
# 플러그인 패키징
.claude-plugin/
└── plugin.json
```

`settings.json`은 MCP 연결, 훅, 권한 허용 목록을 하나의 파일에서 모두 처리합니다.
`.claude/` 아래의 모든 항목은 프로젝트 범위이며, `~/.claude/` 아래의 모든 항목은 개인용으로 모든 프로젝트에서 작동합니다.

CLAUDE.md는 프로젝트 규칙을 위해, 스킬은 배포 워크플로우를 위해, MCP는 데이터베이스 연결을 위해, 훅은 모든 편집 후 린팅(linting)을 실행하기 위해 사용할 수 있습니다.

## 5. Claude Code 작동 방식

이 섹션에서는 에이전트 루프(agentic loop), 컨텍스트 윈도우(context window) 및 권한에 대한 개요를 제공합니다.

### 에이전트 루프 (Agentic Loop)

Claude에게 작업을 주면, 이는 에이전트 루프(agentic loop)라고 알려진 세 단계를 거쳐 작동합니다:

*   컨텍스트 수집
*   조치 실행
*   결과 검증

사용자는 언제든지 중단할 수 있습니다. 이는 일반적으로 Claude를 다른 방향으로 유도하거나, 추가 컨텍스트를 제공하거나, 다른 접근 방식을 시도하도록 요청할 때 필요합니다.

에이전트 루프는 두 가지 구성 요소에 의해 구동됩니다:

*   **추론하는 모델(models)**: 세션 중에는 `/model`로 전환하거나 `claude --model <name>`으로 시작합니다.
*   **행동하는 도구(tools)**: Claude는 파일을 검색하여 코드를 이해하고, 변경 사항을 편집하고, 테스트를 실행하여 결과를 확인하는 데 도구를 사용합니다. 각 도구 사용은 루프로 다시 피드백되어 Claude의 다음 결정을 알리는 정보를 반환합니다.

### 컨텍스트 윈도우 (Context Window)

컨텍스트 윈도우(context window)는 관리해야 할 가장 중요한 리소스입니다. 여기에는 대화 기록, 파일 내용, 명령어 출력, CLAUDE.md, 자동 메모리, 로드된 스킬, 시스템 지침이 포함됩니다.

컨텍스트가 채워질수록 LLM(Large Language Model) 성능이 저하됩니다. 이는 노이즈를 추가하고, Claude의 효율성을 떨어뜨리며, 스킬이 제대로 트리거되지 않게 합니다. 컨텍스트 윈도우가 가득 차면 Claude는 이전 지침을 '잊거나' 더 많은 실수를 할 수 있습니다.

Claude Code는 한계에 접근할 때 오래된 도구 출력을 지우고 필요한 경우 대화를 요약하여 컨텍스트를 자동으로 관리합니다.

### 체크포인트 (Checkpoints)

Claude가 파일을 편집하기 전에 현재 내용의 스냅샷(snapshot)을 생성합니다. 체크포인트(checkpoints)는 세션에 로컬입니다. 이는 Git 변경 사항과는 별개로 파일 변경 사항만 다룹니다. 원격 시스템(데이터베이스, API, 배포)에 영향을 미치는 작업은 체크포인트될 수 없으며, 이 때문에 Claude는 외부 부작용(side effects)이 있는 명령어를 실행하기 전에 확인을 요청합니다.

### 권한 모드 (Permission Modes)

3가지 권한 모드가 있습니다:

*   **기본(Default)**: Claude는 파일 편집 및 셸(shell) 명령어 실행 전에 확인을 요청합니다.
*   **자동 편집 수락(Auto-accept edits)**: Claude는 확인 없이 파일을 편집하지만, 명령어에 대해서는 여전히 확인을 요청합니다.
*   **계획 모드(Plan mode)**: Claude는 읽기 전용 도구만 사용하여, 실행 전에 사용자가 승인할 수 있는 계획을 생성합니다.

## 6. CLAUDE.md 및 MEMORY.md

CLAUDE.md는 Claude가 모든 세션 시작 시 읽는 마크다운(markdown) 파일입니다. 이는 코딩 표준, 아키텍처 결정, 보안 정책, 선호 라이브러리, 팀 워크플로우를 저장하는 데 사용됩니다. 프로젝트 루트, 회사 레벨 또는 하위 디렉토리에 존재할 수 있습니다. `/init`을 실행하여 코드베이스로부터 자동으로 생성할 수 있습니다.

MEMORY.md는 Claude가 작업하면서 스스로 작성하며, 빌드 명령어, 프로젝트 패턴, 세션 전반의 디버깅 통찰력과 같은 정보를 자동으로 저장합니다. 처음 200줄은 각 세션 시작 시 로드됩니다. `/memory`를 통해 자동 메모리를 켜거나 끌 수 있으며, 프로젝트 설정에서 `autoMemoryEnabled`를 설정할 수도 있습니다.

Claude Code는 또한 모든 대화를 로컬에 저장하고, 영향을 받은 파일을 스냅샷하여 변경 사항을 되돌릴 수 있게 하며, 언제든지 세션을 재개, 되감기, 또는 포크(fork)할 수 있도록 합니다. `claude --continue`를 사용하여 가장 최근 대화를 이어서 진행하고, `claude --resume`을 사용하여 목록에서 선택하거나, `claude --from-pr 123`을 사용하여 특정 풀 리퀘스트(pull request)와 연결된 세션을 재개할 수 있습니다. 원본 세션에 영향을 주지 않고 다른 접근 방식을 탐색하려면 `--fork-session` 플래그를 사용하십시오.

## 7. 스킬 (Skills)

스킬(Skills)은 필요에 따라 지식, 워크플로우, 지침을 로드하여 Claude에게 새로운 기능을 제공합니다. Claude는 모든 세션 시작 시 스킬 설명을 확인하지만, 실제로 스킬이 필요할 때만 전체 내용을 로드하여 컨텍스트를 가볍게 유지합니다.

스킬을 사용하여 규칙, 스타일 가이드, 도메인 지식, 또는 배포, 커밋, 코드 생성과 같은 특정 작업에 대한 단계별 지침을 추가할 수 있습니다.

스킬에는 두 가지 유형이 있습니다: 세션 내내 Claude가 참고하는 참조 스킬(예: API 스타일 가이드), 그리고 `/deploy`와 같은 슬래시 명령어로 직접 트리거하는 액션 스킬입니다. Claude Code는 `/simplify`, `/batch`, `/debug`와 같은 내장 스킬을 기본으로 제공합니다.

### 스킬 생성하기

1.  스킬 디렉토리와 그 안에 `SKILL.md` 파일을 생성합니다.
    *   모든 프로젝트의 경우: `~/.claude/skills/<skill-name>/SKILL.md`
    *   단일 프로젝트의 경우: `.claude/skills/<skill-name>/SKILL.md`

2.  모든 `SKILL.md` 파일은 두 부분으로 구성되어야 합니다: Claude에게 언제 스킬을 사용할지 알려주는 YAML 프론트매터(frontmatter) 블록(`---` 마커 사이), 그리고 실제 지침이 포함된 마크다운 내용입니다. 스킬 이름은 슬래시 명령어(slash command)가 되며, 설명은 Claude가 자동으로 로드할 시점을 결정하는 데 도움을 줍니다.

    ```yaml
    ---
    description: Review code for bugs, security, and performance
    disable-model-invocation: true
    ---
    ```

    *   다음 사항에 대해 코드를 검토합니다:
        *   잠재적인 버그 또는 예외 상황(edge cases)
        *   보안 문제
        *   성능 문제
        *   가독성 개선

    간결하고 실행 가능하도록 작성하십시오.

스킬 폴더에는 템플릿(templates), 참조 문서, 예시 출력, Claude가 실행할 수 있는 스크립트도 포함될 수 있습니다:

```
my-skill/
├── SKILL.md           # 주요 지침 (필수)
├── template.md        # Claude가 채울 템플릿
├── reference.md       # 필요할 때 로드되는 상세 문서
├── examples/
│   └── sample.md      # 예상 형식을 보여주는 예시 출력
└── scripts/
    └── validate.sh    # Claude가 실행할 수 있는 스크립트
```

### 스킬 사용하기

스킬은 `/skill-name`으로 직접 호출하거나, 요청이 스킬 설명과 일치할 때 Claude가 자동으로 선택하도록 할 수 있습니다. 스킬은 메인 대화 또는 서브에이전트 내에서 실행될 수 있습니다.

## 8. 플러그인 (Plugins)

플러그인(Plugins)은 여러 리포지토리(repositories)에서 동일한 설정을 재사용하거나 마켓플레이스(marketplace)를 통해 다른 사람들에게 배포하기 위한 패키징 계층입니다.

플러그인은 스킬, 훅, 서브에이전트, MCP 서버를 단일 설치 가능한 단위로 묶습니다. 플러그인 스킬은 네임스페이스(namespaced) 처리되어(`/my-plugin:review`처럼) 여러 플러그인이 공존할 수 있습니다.

플러그인 생성에는 기본적으로 6단계가 있습니다:

1.  폴더 구조 생성
    ```bash
    mkdir -p my-marketplace/.claude-plugin
    mkdir -p my-marketplace/plugins/quality-review-plugin/.claude-plugin
    mkdir -p my-marketplace/plugins/quality-review-plugin/skills/quality-review
    ```
2.  `/quality-review` 스킬이 수행하는 작업을 정의하는 `SKILL.md` 파일 생성.
3.  플러그인을 설명하는 `plugin.json` 파일 생성:
    ```json
    my-marketplace/plugins/quality-review-plugin/.claude-plugin/plugin.json
    {
      "name": "quality-review-plugin",
      "description": "Adds a /quality-review skill for quick code reviews",
      "version": "1.0.0"
    }
    ```
4.  플러그인을 나열하는 마켓플레이스 카탈로그 생성:
    ```json
    my-marketplace/.claude-plugin/marketplace.json
    {
      "name": "my-plugins",
      "owner": {
        "name": "Your Name"
      },
      "plugins": [
        {
          "name": "quality-review-plugin",
          "source": "./plugins/quality-review-plugin",
          "description": "Adds a /quality-review skill for quick code reviews"
        }
      ]
    }
    ```
5.  마켓플레이스 추가 및 플러그인 설치
    ```bash
    /plugin marketplace add ./my-marketplace
    /plugin install quality-review-plugin@my-plugins
    ```
6.  플러그인 테스트
    ```bash
    /quality-review
    ```

`/plugin`을 실행하여 마켓플레이스를 탐색하십시오. 플러그인은 구성 없이 스킬, 도구, 통합 기능을 추가합니다.

## 9. 서브에이전트 (Subagents)

서브에이전트(Subagents)는 자체 시스템 프롬프트(system prompt), 도구 접근 권한, 권한을 가지고 자체 컨텍스트 윈도우에서 실행되는 전문화된 AI 어시스턴트입니다. Claude는 서브에이전트의 설명과 일치하는 작업을 만나면 해당 서브에이전트에 작업을 위임하며, 서브에이전트는 독립적으로 작동하고 메인 대화에 요약만 반환합니다.

Claude Code는 탐색, 계획, 터미널 명령어 실행을 위한 내장 서브에이전트를 제공하며, 이는 적절할 때 자동으로 사용됩니다.

### 서브에이전트를 사용하는 이유

*   **컨텍스트를 깨끗하게 유지합니다.** 테스트 실행, 문서 가져오기, 로그 파일 처리와 같은 작업은 많은 출력을 생성합니다. 이를 서브에이전트에 위임하면 모든 노이즈가 서브에이전트의 컨텍스트에 유지되고 관련 요약만 사용자에게 돌아옵니다.
*   **도구 제한을 적용합니다.** 서브에이전트가 접근할 수 있는 도구를 제한하여, 서브에이전트가 수행할 수 있는 작업에 대해 더 엄격한 통제를 할 수 있습니다.
*   **비용을 제어합니다.** 모든 작업을 더 비싼 모델을 통해 실행하는 대신, Haiku와 같이 더 빠르고 저렴한 모델로 간단한 작업을 라우팅(route)합니다.

### 서브에이전트 생성하기

*   Claude Code에서 `/agents`를 실행하고 '새 에이전트 생성'을 선택합니다.
*   개인용(Personal)을 선택하여 `~/.claude/agents/`에 저장하고 모든 프로젝트에서 사용 가능하게 하거나, 프로젝트용(Project)을 선택하여 현재 코드베이스의 `.claude/agents/`에 유지합니다.
*   Claude가 언제 자동으로 위임할지 알 수 있도록 명확한 설명을 제공하십시오.

### 서브에이전트 사용하기

Claude는 작업이 서브에이전트의 설명과 일치할 때 자동으로 서브에이전트에 위임합니다. @멘션(@-mention)으로 직접 트리거할 수도 있습니다. 서브에이전트는 메인 에이전트에게만 결과를 보고하며, 서로 통신하지 않습니다.

**서브에이전트 예시: .claude/agents/security-reviewer.md**

```yaml
---
name: security-reviewer
description: 코드의 보안 취약점을 검토합니다.
tools: Read, Grep, Glob, Bash
model: opus
---
```

당신은 선임 보안 엔지니어입니다. 다음 사항에 대해 코드를 검토하십시오:

*   인젝션 취약점(Injection vulnerabilities) (SQL, XSS, 명령 주입)
*   인증 및 권한 부여 결함
*   코드 내 시크릿(secrets) 또는 자격 증명(credentials)
*   안전하지 않은 데이터 처리

특정 줄 참조와 제안된 수정 사항을 제공하십시오.

*   코드베이스별 에이전트에는 프로젝트 서브에이전트(`.claude/agents/`)를 사용하십시오. 팀도 혜택을 받을 수 있도록 버전 제어 시스템에 체크인하십시오.
*   모든 프로젝트에서 사용 가능한 개인 에이전트에는 사용자 서브에이전트(`~/.claude/agents/`)를 사용하십시오.

서브에이전트는 마크다운 파일로 수동 생성하거나, CLI 플래그를 통해 정의하거나, `/agents` 명령어를 사용하여 플러그인을 통해 배포할 수도 있습니다.

## 10. 에이전트 팀 (Agent Teams)

에이전트 팀(Agent teams)을 통해 여러 Claude Code 인스턴스를 동시에 함께 실행할 수 있습니다. 한 세션은 팀 리더 역할을 하여 작업을 조정하고 결과를 통합하며, 팀원들은 자체 컨텍스트 윈도우에서 독립적으로 작업하고 서로 직접 통신합니다.

이는 한 번에 여러 각도에서 문제를 탐색하고 싶을 때 유용합니다. 예를 들어, 하나는 UX에 집중하고, 다른 하나는 기술 아키텍처에 집중하며, 또 다른 하나는 반대 의견을 제시하는 역할(devil’s advocate)을 할 수 있습니다.

### 에이전트 팀과 서브에이전트의 차이점

서브에이전트는 단일 세션 내에서 작동합니다. 에이전트 팀은 별도의 세션에 걸쳐 조정되며, 각 팀원은 자체적인 전체 Claude Code 인스턴스를 실행합니다.

### 에이전트 팀을 사용하기 전에

*   이는 실험적인 기능이며 기본적으로 비활성화되어 있습니다. `settings.json` 또는 환경에 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS`를 추가하여 활성화하십시오.
*   각 팀원이 자체 컨텍스트 윈도우를 실행하므로 단일 세션보다 훨씬 많은 토큰을 사용합니다.
*   팀원들이 독립적으로 작동할 수 있을 때 가장 잘 작동합니다. 순차적인 작업, 동일한 파일을 건드리는 작업, 또는 많은 의존성(dependencies)을 가진 작업의 경우 단일 세션 또는 서브에이전트가 더 효율적입니다.
*   세션 재개, 작업 조정, 종료 동작과 관련하여 알려진 제한 사항이 있습니다.

## 11. 팁 및 모범 사례

다음은 Claude Code를 사용하면서 제가 얻은 팁과 모범 사례(best practices)입니다:

*   **대화를 통해 계획 모드(plan mode)에서 계획을 검토하고 다듬은 다음, Claude가 계획을 구현하도록 하십시오.**
    *   Shift+Tab을 두 번 눌러 계획 모드에 진입합니다. 계획 모드는 읽기 전용 분석입니다.
    *   이것은 변경 사항이 여러 파일에 걸쳐 있거나, 접근 방식이 확실하지 않거나, 코드베이스에 익숙하지 않을 때 가장 유용합니다.
    *   차이점(diff)을 한 문장으로 설명할 수 있다면 계획을 건너뛰십시오.
*   **먼저 광범위한 질문을 하고, 점차 구체화하십시오.**
    *   프로젝트에서 사용되는 코딩 규칙과 패턴에 대해 질문하는 것으로 시작하십시오.
    *   프로젝트별 용어집(glossary)을 요청하십시오.
*   **@를 사용하여 Claude가 읽을 때까지 기다릴 필요 없이 파일이나 디렉토리를 빠르게 포함시키십시오.**
*   **데이터를 직접 파이프(pipe)합니다: `cat error.log | claude`**
*   **결정론적 동작(deterministic behaviour)을 위해 훅(hooks)을 사용하십시오.**
    *   훅은 무조건적으로 실행됩니다. Claude의 결정에 의존하는 대신, 항상 어떤 일이 일어나도록 하고 싶을 때 사용할 수 있습니다. 훅은 결정론적이며, `Claude.md` 지침과는 달리 작업이 반드시 발생하도록 보장합니다. 이는 Claude Code 작업 전후에 셸(shell) 명령어를 실행할 수 있게 합니다.
    *   훅은 프로젝트 규칙을 강제하고, 반복적인 작업을 자동화하며, 데이터를 전처리(preprocessing)하는 데 유용합니다 (예: Claude가 보기 전에 10,000줄 로그에서 `ERROR` 라인을 grep하는 등).
    *   조건부 로직(conditional logic)의 경우 프롬프트(prompt) 기반 또는 에이전트 기반 훅을 사용하십시오.
    *   Claude를 사용하여 훅을 작성할 수 있습니다. "모든 파일 편집 후 eslint를 실행하는 훅을 작성해 줘" 또는 "마이그레이션 폴더(migrations folder)에 대한 쓰기를 차단하는 훅을 작성해 줘"와 같은 프롬프트를 시도해 보십시오. `.claude/settings.json`을 직접 편집하여 훅을 수동으로 구성하고, `/hooks`를 실행하여 구성된 내용을 탐색하십시오.
    *   `.claude/settings.json`에 추가:
        ```json
          {
            "hooks": {
              "PostToolUse": [
                {
                  "matcher": "Edit|Write", // matcher: 파일 편집을 위한 `Edit` 또는 `Write`
                  "hooks": [{ "type": "command", "command": "npx prettier --write $FILE" }]
                }
              ],
              "Notification": [
                {
                  "hooks": [{ "type": "command", "command": "notify-send 'Claude needs input'" }] // 예시: Claude가 입력을 필요로 할 때 데스크톱 알림 보내기
                }
              ]
            }
          }
        ```
*   **대규모 기능(large features)의 경우 Claude가 당신을 인터뷰하도록 하십시오.**
    *   최소한의 프롬프트로 시작하여 Claude에게 `AskUserQuestion` 도구를 사용하여 당신을 인터뷰하도록 요청하십시오.
    *   사양(spec)이 완료되면, 구현을 위해 새로운 세션을 열어 실행에 초점을 맞춘 깨끗한 컨텍스트를 사용하도록 하십시오.
    *   결국 빌드(build) 전반에 걸쳐 참조할 수 있는 문서화된 사양을 얻게 됩니다.
*   **노력 및 사고 수준을 조정하십시오.**
    *   세 가지 영구적인 수준이 있습니다: 낮음(low), 중간(medium), 높음(high), 그리고 Opus 4.6에서는 최대(max)가 있습니다.
    *   대부분의 코딩 작업에는 중간 수준이 권장됩니다. 더 높은 수준은 Claude가 일상적인 작업에 대해 과도하게 생각하게 만들 수 있습니다.
    *   어려운 디버깅(debugging)이나 복잡한 아키텍처 결정에는 높음 또는 최대 수준을 예약하십시오.
    *   `/effort low`로 변경하거나 `/effort auto`로 재설정하십시오.
    *   `/config`에서 사고를 비활성화하거나 간단한 작업의 경우 `MAX_THINKING_TOKENS=8000`으로 설정하십시오.
*   **에이전트 안전을 위해 샌드박싱(sandboxing)을 활성화하십시오.**
    *   `/sandbox`를 실행하여 Claude가 위험을 줄이고 자유롭게 작업할 수 있는 정의된 경계(boundaries)를 생성하십시오.
    *   프롬프트 인젝션(prompt injection)이 Claude의 동작을 조작하더라도 샌드박스는 시스템을 안전하게 유지합니다.
    *   Claude Code는 시작된 폴더와 그 하위 폴더에만 쓰기 작업을 할 수 있으며, 작업 디렉토리 외부 읽기는 가능합니다.
*   **속도가 중요한 작업에는 `/fast` 모드를 사용하십시오.**
    *   `/fast`로 켜서 빠른 반복(iteration) 또는 실시간 디버깅을 수행하십시오. 이는 Opus 4.6을 약 2.5배 더 빠르게 만듭니다.
    *   대기 시간(latency)보다 비용이 더 중요할 때는 끄십시오.
    *   Claude Opus 4.6에서만 사용 가능합니다.
*   **컨텍스트를 적극적으로 관리하십시오.**
    *   `/context`를 실행하여 무엇이 공간을 차지하고 있는지 확인하십시오.
    *   MCP 서버는 모든 요청에 도구 정의를 추가합니다. 결과적으로 몇몇 서버는 시작하기 전에 상당한 컨텍스트를 소모할 수 있습니다. `/mcp`를 실행하여 서버별 비용을 확인하고 사용하지 않는 서버를 비활성화하십시오.
*   **압축(compaction)을 제어하십시오.**
    *   Claude는 자동으로 압축하지만, 초기 지침이 손실될 수 있습니다.
    *   무엇을 보존할지 제어하기 위해 `CLAUDE.md`에 '압축 지침(Compact Instructions)' 섹션을 추가하십시오.
    *   중요한 부분을 대상으로 API 변경 사항에 집중하도록 `/compact` 명령어를 실행하십시오.
*   **`/clear`를 사용하고 더 나은 초기 프롬프트를 작성하십시오.**
    *   Claude를 반복적으로 수정하는 것은 실패한 접근 방식으로 컨텍스트를 오염시킵니다.
    *   `/clear`를 실행하여 재설정한 다음, 먼저 `/rename`을 사용하여 나중에 `/resume`으로 세션을 찾을 수 있도록 하십시오.
    *   사용자 지정 압축 지침을 추가하십시오: `/compact Focus on code samples and API usage`
*   **전문 지침을 스킬로 이동하십시오.**
    *   스킬은 호출될 때만 온디맨드로 로드되며, 기본 컨텍스트를 부풀리지 않습니다.
    *   수동으로 호출하는 스킬에 `disable-model-invocation: true`를 설정하여 필요할 때까지 설명을 컨텍스트에서 제외하십시오.
*   **코드 리뷰(code review)에는 새로운 세션을 사용하십시오.**
    *   새로운 컨텍스트는 코드 리뷰를 향상시킵니다. Claude는 자신이 방금 작성한 코드에 편향되지 않을 것입니다.
    *   동일한 시작점에서 병렬 작업을 수행할 때 `--fork-session`을 사용하여 각 터미널(terminal)이 자체적인 깨끗한 세션을 얻도록 하십시오.
    *   여러 터미널에서 동일한 세션을 재개하는 것을 피하십시오. 메시지가 뒤섞여 대화가 혼란스러워질 수 있습니다.
*   **구현 후 검토에는 `/simplify`를 사용하십시오.**
    *   `/simplify`를 실행하여 최근 변경된 파일의 코드 재사용성, 품질, 효율성을 검토하십시오.
    *   세 개의 검토 에이전트를 병렬로 생성하고, 발견 사항을 집계하며, 수정 사항을 적용합니다.
    *   초점을 전달하십시오: `/simplify focus on memory efficiency`
*   **체크포인트와 되감기(rewind)를 사용하십시오.**
    *   Claude가 잘못된 방향으로 가고 있다면 즉시 Escape를 눌러 중지하십시오.
    *   Escape를 두 번 누르거나(`/rewind` 실행) 이전 체크포인트로 대화와 코드를 복원하십시오.
    *   체크포인트는 파일 변경 사항을 되돌릴 수 있게 하며, 권한은 Claude가 요청 없이 무엇을 할 수 있는지 제어합니다.
*   **완료 시 서브에이전트가 메모리를 업데이트하도록 지시하십시오.**
    *   여러 서브에이전트를 생성하여 코드베이스의 다른 영역을 동시에 조사할 수 있습니다.
    *   서브에이전트에게 "시작 전에 메모리를 확인하고" "완료 시 메모리를 업데이트하라"고 지시하여 시간이 지남에 따라 축적되는 지식을 구축하도록 하십시오.
    *   특정 작업에 대해 서브에이전트가 실행되도록 보장하려면 @-멘션(@-mention)을 사용하십시오.
    *   서브에이전트는 메인 에이전트에게만 보고한다는 점에 유의해야 합니다. 그들은 서로 통신하지 않습니다. 서브에이전트는 메인 컨텍스트를 깨끗하게 유지하는 가장 효과적인 도구 중 하나입니다.
*   **효과적인 CLAUDE.md 파일을 작성하십시오.**
    *   `Claude.md` 파일당 200줄 미만을 목표로 하십시오. 파일이 길어지면 더 많은 컨텍스트를 소비하고 지침 준수율을 낮춥니다.
    *   구체적이고 실행 가능한 규칙을 사용하십시오: "코드를 제대로 포맷해" 대신 "2칸 들여쓰기 사용".
    *   관련 지침을 그룹화하기 위해 마크다운 헤더와 글머리 기호를 사용하십시오.
    *   블록 레벨 HTML 주석(`<!-- notes -->`)은 컨텍스트에 주입되기 전에 제거됩니다. 토큰을 소비하지 않고 유지 관리자 메모용으로 사용하십시오.
    *   대규모 프로젝트의 경우 `CLAUDE.md`를 여러 파일로 분할하십시오.
    *   Claude는 작업 디렉토리부터 루트까지 `CLAUDE.md` 파일을 읽고, 하위 디렉토리 내의 중첩된 파일들을 발견합니다.
    *   파일 유형 또는 하위 디렉토리에 범위가 지정된 주제별 파일에는 `.claude/rules/`를 사용하십시오.
    *   `.claude/rules/`는 심볼릭 링크(symlinks)를 지원합니다 — 여러 프로젝트에서 공통 규칙 세트를 공유하십시오.
    *   모노레포(monorepos)에서는 `claudeMdExcludes`를 사용하여 다른 팀의 관련 없는 `CLAUDE.md` 파일을 건너뛰십시오.
    *   `@path/to/import` 구문을 사용하여 추가 파일을 가져오십시오(import).
*   **지침을 주기적으로 감사(audit)하십시오.**
    *   두 규칙이 서로 모순될 경우, Claude는 임의로 하나를 선택할 수 있습니다.
    *   오래되거나 충돌하는 규칙을 제거하기 위해 `CLAUDE.md`, 중첩 파일, `.claude/rules/`를 정기적으로 검토하십시오.
    *   Claude가 지침 없이도 이미 올바르게 작업을 수행한다면, 해당 지침을 삭제하거나 훅으로 전환하십시오.
    *   과도하게 상세한 `CLAUDE.md`는 함정입니다 — 가차 없이 가지치기(prune)하십시오.
*   **Claude에게 수정 로그(Correction Log)를 유지하도록 요청하십시오.**
    *   `docs/claude-corrections.md`에 수정 사항 로그를 유지하도록 `CLAUDE.md`에 지침을 추가할 수 있습니다. Claude는 파일을 읽을 수 있기 때문에 이러한 파일은 Claude의 메모리 역할을 하여 도움이 됩니다.
    *   `Claude.md`에 지침을 추가하여 Claude가 각 세션 시작 시 수정 파일을 읽도록 하십시오.
    ```markdown
    ## 세션 시작 시

    다른 작업을 하기 전에 `docs/mistakes.md`를 읽으십시오.
    ## 수정 로그

    실수를 저지르고 수정될 때, `docs/claude-corrections.md`에 다음 형식으로 항목을 추가하십시오:

    ### [실수에 대한 간략한 설명]
    - **실수:** 무엇을 잘못했는지
    - **교정:** 올바른 접근 방식은 무엇인지
    - **규칙:** 앞으로 기억해야 할 일반적인 규칙

    모든 작업을 시작하기 전에 매 세션 시작 시 `docs/claude-corrections.md`를 항상 읽으십시오.
    ```
*   **Claude가 자신의 작업을 검증할 방법을 항상 제공하십시오.**
    *   테스트 케이스를 포함하거나, 예상 UI의 스크린샷을 붙여 넣거나, 정확한 출력을 정의하십시오.
    *   시각적 작업의 경우: 스크린샷을 붙여 넣고 Claude에게 자신의 구현을 스크린샷과 비교하도록 요청하십시오.
    *   UI 검증에는 Chrome용 Claude 확장 프로그램을 사용하십시오 — 이는 탭을 열고, UI를 테스트하며, 반복합니다.
    *   검증은 테스트 스위트(test suite), 린터(linter), 또는 출력을 확인하는 Bash 명령어일 수 있습니다.
*   **안전한 명령어를 허용 목록(allowlist)에 추가하십시오.**
    *   Claude가 매번 묻지 않도록 신뢰할 수 있는 명령어를 `.claude/settings.json`에 추가하십시오 (예: `npm test`, `git status`, `npm run lint`).
    *   부작용(side effects)이 있는 명령어에 대한 프롬프트를 유지하면서 여러 편집을 일괄적으로 수락하려면 '편집 수락(Accept Edits)' 모드를 사용하십시오.
    *   `/permissions`를 실행하여 권한 설정을 정기적으로 감사하십시오.
*   **토큰 비용을 추적하고 줄이십시오.**
    *   `/cost` (API 사용자) 또는 `/stats` (구독자)를 사용하여 사용량을 모니터링하십시오.
    *   MCP 서버보다 CLI 도구(`gh`, `aws`, `gcloud`)를 선호하십시오. CLI 도구는 컨텍스트 효율성이 높고 도구별 목록을 추가하지 않습니다.
    *   자세한 작업(테스트, 로그, 문서 가져오기)은 서브에이전트에 위임하십시오. 이들은 메인 컨텍스트에 요약만 반환합니다.
    *   조직 전체 비용 추적을 위해 텔레메트리(telemetry)를 내보내십시오.
    *   `CLAUDE_CODE_ENABLE_TELEMETRY=1`로 활성화하십시오.
    *   기본 내보내기 간격: 메트릭(metrics)은 60초, 로그는 5초.
    *   사용자 프롬프트 내용은 기본적으로 수집되지 않습니다 — 포함하려면 `OTEL_LOG_USER_PROMPTS=1`로 설정하십시오.
*   **컨텍스트 내에서 빠른 질문을 위해 `/btw`를 사용하십시오.**
    *   `/btw`는 전체 컨텍스트를 보지만 도구 접근 권한이 없습니다. 답변은 폐기되며 기록에 추가되지 않습니다.
    *   서브에이전트를 실행하지 않고 빠른 질문에 사용하십시오.
*   **일정 간격으로 프롬프트를 실행하려면 `/loop`를 사용하거나, 설정된 일정에 따라 자동으로 실행되는 반복 작업을 생성하려면 `/schedule`을 사용하십시오.**
    *   구문: `/loop [interval] <prompt>` — 예: `/loop 5m 배포가 완료되었는지 확인`
    *   스킬 호출 가능: `/loop 20m /review-pr 1234`
    *   작업은 Claude Code가 실행 중이고 유휴 상태일 때만 발생합니다 — 터미널을 닫으면 모든 것이 취소됩니다. 놓친 작업에 대한 따라잡기 기능은 없습니다.
    *   CLI, 데스크톱 앱, 또는 웹에서 `/schedule`을 사용하여 작업을 생성하십시오.
    *   빠른 폴링(polling)을 위해 CLI 세션 내에서 프롬프트를 반복하려면 `/loop`를 사용하십시오.
*   **문제 해결을 위해 `/debug`를 사용하십시오.**
    *   현재 세션에 대한 디버그 로깅(debug logging)을 활성화합니다.
    *   분석에 집중하기 위해 문제를 선택적으로 설명하십시오: `/debug auth 모듈의 메모리 누수`
    *   디버그 로깅은 `claude --debug`로 시작하지 않는 한 기본적으로 꺼져 있습니다.
*   **프로젝트 간에 스킬을 공유하려면 플러그인을 사용하십시오.**
    *   플러그인은 스킬, 에이전트, 훅, MCP 서버로 Claude Code를 확장합니다.
    *   빠른 반복을 위해 `.claude/`에 독립 실행형 구성으로 시작한 다음, 팀 전체에 공유할 플러그인으로 변환하십시오.
    *   `/plugin` → 'Discover' 탭을 실행하여 사용 가능한 플러그인을 탐색하거나, `claude.com/plugins`를 방문하십시오.
    *   공식 마켓플레이스에서 설치: `/plugin install <name>@claude-plugins-official`
    *   Claude Code 버전 1.0.33 이상이 필요합니다.
    *   신뢰할 수 있는 소스에서만 플러그인을 설치하십시오 — 플러그인은 사용자 권한으로 임의의 코드(arbitrary code)를 실행할 수 있습니다.
*   **오래 실행되는 작업에 대한 데스크톱 알림을 받으십시오.**
    *   플랫폼의 기본 알림 명령어를 호출하는 알림(Notification) 훅을 `~/.claude/settings.json`에 추가하십시오.
    *   Claude가 권한을 기다리거나, 유휴 상태로 준비되었거나, 인증을 완료할 때 발생합니다.
*   **설치 확인을 위해 `claude doctor`를 실행하십시오.**
    *   설치 및 구성에 대한 자세한 확인을 제공합니다.
*   **세션 분석을 위해 `/insights` 및 `/stats`를 사용하십시오.**
    *   `/insights` 프로젝트 영역, 상호 작용 패턴, 마찰 지점(friction points)을 포함하여 Claude Code 세션을 분석하는 보고서를 생성합니다.
    *   `/stats` 일일 사용량, 세션 기록, 연속 사용(streaks), 모델 선호도를 시각화합니다.
*   **기기 간 작업.**
    *   휴대폰, 브라우저, 데스크톱 또는 Claude iOS 앱에서 동일한 세션을 계속 진행하십시오. 모든 인터페이스는 동일한 `CLAUDE.md` 파일, 설정, MCP 서버를 공유합니다.
    *   웹 또는 iOS에서 장기 실행 작업을 시작한 다음, `/teleport`를 사용하여 터미널로 가져오십시오.
    *   시각적 차이점(diff) 검토를 위해 `/desktop`으로 터미널 세션을 데스크톱 앱으로 넘기십시오.
*   **에이전트 팀 토큰 비용 효율적으로 관리.**
    *   토큰 사용량은 활성 팀원의 수와 각 팀원이 실행되는 시간에 비례합니다.
    *   팀원에게는 Sonnet을 사용하십시오. 이는 조정 작업에서 기능과 비용의 균형을 이룹니다.
    *   팀을 작게 유지하십시오. 각 팀원은 자체 컨텍스트 윈도우를 실행하므로 토큰 사용량은 팀 규모에 대략적으로 비례합니다.
    *   생성 프롬프트(spawn prompts)를 집중적으로 유지하십시오. 팀원들은 `CLAUDE.md`, MCP 서버, 스킬을 자동으로 로드하지만, 생성 프롬프트의 모든 내용은 시작부터 컨텍스트에 추가됩니다.
