몇 주 전, GitHub에 흥미로운 벤치마크(mame/ai-coding-lang-bench)가 등장했습니다. 스스로를 Ruby 커미터(committer)라고 밝힌 저자는 간소화된 Git 시스템을 구현하기 위해 13개 프로그래밍 언어를 대상으로 클로드 코드(Claude Code)를 실행했습니다. 총 600회의 실행, 명확한 수치, 재현 가능한 방법론이 적용되었습니다.

그 결과는 매우 놀랍습니다. 특히 Go 엔지니어라면 주목할 가치가 있습니다.

## 실험

이 방법론은 단순함 속에서 우아함을 지니고 있습니다.

*   **과제**: 간소화된 Git과 유사한 버전 관리 시스템(version control system) 구현
*   **두 단계**: v1 (기본 기능) + v2 (기능 확장)
*   **언어**: 총 13개 언어, 각 언어별 20회 테스트
*   **측정 지표**: 시간, 비용(API 토큰), 생성된 코드 라인 수
*   **총 600회 실행**, 결과 평균 산출

한 가지 영리한 설계 선택은, 저자가 SHA-256 대신 사용자 지정 해시 알고리즘(custom hash algorithm)을 사용하여 언어별 라이브러리 종속성(library dependencies)의 차이를 피했다는 점입니다. 이는 순수한 언어 수준의 생성 특성을 분리하여, 동일한 문제를 다르게 표현했을 때의 특징을 파악합니다.

이 벤치마크는 우리가 거의 정량화하지 않는 것, 즉 언어가 AI 코딩 에이전트(AI coding agent)에게 얼마나 효율적으로 사고하게 하는지를 측정합니다.

## 순위: 동적 언어의 압도적 승리

수치는 거짓말을 하지 않습니다.

_전체 크기 이미지 보려면 엔터 키를 누르거나 클릭하세요_

Ruby, Python, JavaScript가 압도적으로 우세합니다. 이들은 가장 빠르고, 가장 저렴하며, 가장 안정적입니다.

다음으로 TypeScript를 주목하십시오. 이는 JavaScript의 정적 타입(statically typed) 슈퍼셋(superset)입니다. 일반 JavaScript보다 거의 2배나 시간이 더 걸리고 비용도 60% 더 듭니다. 동일한 기본 로직(underlying logic)임에도 불구하고 AI 코딩 성능에는 극적인 차이가 발생합니다.

흥미롭지 않나요?

## 타입 체크(Type-Checking)의 세금

가장 놀라운 발견은 타입 체커(type checkers)를 추가하는 것이 AI 코딩 성능에 상당한 불이익을 준다는 점입니다.

*   Python + mypy: 일반 Python보다 1.6~1.7배 느림
*   Ruby + Steep: 일반 Ruby보다 2.0~3.2배 느림

생각해보면 이는 직관적으로 이해됩니다. 정적 타입 시스템(static type systems)은 AI가 여러 제약 조건을 동시에 충족하도록 요구합니다: 타입 정확성 + 비즈니스 로직(business logic) + API 호환성 + 컴파일러(compiler) 요구사항. 더 많은 제약 조건은 더 많은 추론 단계, 더 많은 토큰, 더 많은 시간, 더 많은 비용으로 이어집니다.

이는 타입 시스템의 결함이 아닙니다. 코드 생성 과정에서 제약 조건을 만족시켜야 하는 기본적인 속성입니다. 모델은 더 어려운 문제를 해결해야 하는 것입니다.

그 함의는 명확합니다: 타입 체커를 추가하는 것은 단순히 안전성을 더하는 것이 아니라, AI 생성 복잡도에 비례하여 증가하는 '세금'을 추가하는 것입니다.

## 간결성의 역설 (The Compactness Paradox)

_전체 크기 이미지 보려면 엔터 키를 누르거나 클릭하세요_

이 부분은 직관과 상충됩니다.

OCaml과 Haskell은 가장 간결한 코드(약 216~224라인)를 생성합니다. 기능적으로는 인상적입니다. 그러나 생성 속도에서는 중간에서 낮은 순위를 기록했습니다.

왜 그럴까요? 표현력이 풍부한 언어의 간결한 코드는 종종 더 정교한 추론을 요구합니다. 관용적인 Haskell 코드를 생성한다는 것은 모델이 타입 클래스(type classes), 모나드 합성(monadic composition), 지연 평가(lazy evaluation)를 동시에 추론해야 함을 의미합니다. 모델은 더 적은 코드를 생성하기 위해 "더 열심히 생각"합니다.

인지적 밀도(Cognitive density)는 토큰 비용을 발생시킵니다. 장황한 코드를 더 많이 생성하는 것이 밀도 높고 관용적인 코드를 더 적게 생성하는 것보다 비용이 저렴할 수 있습니다.

이는 일반적인 직관과 반대됩니다. 우리는 종종 표현력이 풍부한 언어가 사용하기에 "더 간단"하다고 가정합니다. 인간에게는 그럴 수 있지만, AI 생성 모델에게는 반드시 그렇지는 않습니다.

## 신뢰성은 어떨까?

총 600회의 실행 중 단 3회의 실패만 발생했습니다.

*   Rust: 2회 실패
*   Haskell: 1회 실패

일반적인 통념은 "정적 타입이 버그를 방지한다"는 것입니다. 그러나 이 벤치마크에서는 모든 실패가 정적 타입 언어에서 발생했습니다. 이것이 정적 타이핑이 나쁘다는 의미는 아니지만, 타입 시스템이 AI 생성 코드를 자동으로 더 신뢰성 있게 만든다는 가정에 도전합니다.

모델의 실패 모드는 인간의 실패 모드와 다릅니다. 타입은 인간의 추론을 안내하지만, AI 생성에는 때때로 연쇄적인 실패(cascading failures)를 야기하는 방식으로 제약을 가할 수 있습니다.

## 그렇다면 Go는 어디에 있을까?

Go는 요약된 데이터에서 명시적으로 강조되지는 않았지만, 그 위치는 기본 원칙에서 쉽게 추론할 수 있습니다.

Go는 TypeScript 및 Rust와 유사하게 명시적 타입 선언(explicit type declarations)을 사용하는 정적 타입 언어입니다. 관찰된 패턴에 따르면:

*   Go는 중간 계층에 위치할 가능성이 높습니다. Ruby/Python보다는 느리지만, Go의 의도적인 언어 단순성 때문에 TypeScript보다는 빠를 것입니다.
*   Go의 제네릭(generics) 복잡성 부족은 도움이 됩니다. Go 1.18 이전에도, 그리고 이후에도 Go는 AI 추론을 어렵게 하는 고급 타입 시스템 기능을 피합니다.
*   Go의 장황한 에러 핸들링(error handling)은 불리하게 작용합니다. `if err != nil` 체인은 반복적이지만 필수적이며, 상용구(boilerplate) 코드로 토큰 수를 증가시킵니다.

핵심 통찰력은 다음과 같습니다. Go의 단순성은 더 복잡한 타입 시스템(Rust, Haskell, 고급 제네릭을 사용하는 TypeScript)을 가진 언어와 비교했을 때, AI 코딩 시대에 상대적인 장점이 됩니다.

## Go 엔지니어를 위한 전략적 실행 계획

_전체 크기 이미지 보려면 엔터 키를 누르거나 클릭하세요_

저자의 결론은 깊은 공감을 불러일으킵니다.

"동적 언어로 프로토타입을 만들고 정적 언어로 마이그레이션(migrate)하는 전통적인 전략은, 특히 AI 시스템이 언어 간 번역(cross-language translation) 능력을 향상시킬수록 전략적으로 건전하게 유지됩니다."

Go 엔지니어에게 이는 구체적인 워크플로우로 이어집니다.

*   **1단계: Python 또는 Ruby로 프로토타이핑**. AI를 사용하여 문제 공간(problem space) — 데이터 구조, API 계약, 엣지 케이스(edge cases) —를 빠르게 탐색합니다. 동적 언어는 모델이 빠르고 저렴하게 반복 작업을 수행할 수 있도록 합니다.
*   **2단계: API 계약 강화**. 무엇을 구축할지 이해하면 타입이 명세(specification)가 됩니다. 바로 이 지점에서 정적 타이핑의 이점이 발휘됩니다.
*   **3단계: AI 지원을 받아 Go로 마이그레이션**. AI 모델이 언어 간 번역 능력을 향상시킬수록 이 단계는 점점 더 저렴해집니다. Go의 강점 — 고루틴(goroutines), 단일 바이너리(single binary), 메모리 효율성, 강력한 표준 라이브러리(robust standard library) —를 잘 이해된 문제 영역에 가져옵니다.

이는 Go에 대한 비판이 아닙니다. 벤치마크는 프로토타이핑(prototyping) 규모 작업에 대한 AI 생성 속도를 측정합니다. Go의 장점은 가치 사슬(value stack)의 다른 계층에 존재합니다: 프로덕션 신뢰성(production reliability), 운영 단순성(operational simplicity), 팀 확장성(team scalability) 등입니다.

## 더 큰 그림

이 벤치마크가 드러내는 것은 AI 코딩 성능이 이제 런타임 성능(runtime performance)이나 개발자 인체 공학(developer ergonomics)만큼이나 중요한 일급 언어 속성(first-class language property)이 되었다는 점입니다.

HTTP 서버(HTTP servers)와 데이터베이스 클라이언트(database clients)를 벤치마크하듯이, 언어 및 툴체인(toolchain) 선택 시 AI 생성 효율성을 고려해야 합니다. 측정 지표는 다릅니다. 초당 요청 수(RPS: Requests Per Second)가 아니라 기능당 비용(dollars per feature)입니다.

클로드 코드(Claude Code), 커서(Cursor), 코파일럿(Copilot)이 주요 개발 표면(development surfaces)이 됨에 따라, 언어 설계는 새로운 축을 따라 점점 더 평가될 것입니다: AI가 이 언어에서 얼마나 효율적으로 추론할 수 있는가?

AI 시대에 성공하는 언어가 런타임 시대에 성공하는 언어와 같지 않을 수도 있습니다. 두 가지 차원을 모두 이해하는 것이 새로운 엔지니어링 지식(engineering literacy)입니다.

## 요약

*   동적 언어(Ruby, Python, JS)는 AI 기반 프로토타이핑에서 압도적입니다. 가장 빠르고, 저렴하며, 안정적입니다.
*   타입 체크(type-checking) 오버헤드(overhead)는 실재합니다. mypy는 1.6~1.7배, Steep는 2~3.2배 더 많은 시간이 소요됩니다.
*   코드 간결성 ≠ 생성 속도. 표현력이 풍부한 언어(OCaml, Haskell)는 적은 코드를 생성하더라도 더 느릴 수 있습니다.
*   정적 타입이 AI 실패를 방지하지 못합니다. 600회 실행 중 발생한 3회의 실패는 모두 정적 타입 언어에서 발생했습니다.
*   Go 엔지니어를 위해: 프로토타입은 동적 언어로, 프로덕션은 Go로 마이그레이션하세요. AI 번역 기술이 빠르게 향상되고 있습니다.
*   AI 코딩 성능은 이제 런타임 성능 및 생태계 성숙도와 함께 언어 선택의 중요한 기준이 됩니다.

판도가 바뀌고 있습니다. 가장 현명한 움직임은 이를 명확히 이해하고 저항하지 않는 것입니다.
