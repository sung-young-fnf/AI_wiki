# .claude/ 폴더 구조화로 Claude Code 프로젝트 효율 극대화하기: 지침, 규칙, 훅, 스킬 및 권한 실전 가이드

대부분의 Claude Code 사용자는 `.claude/` 폴더가 존재한다는 것을 알지만, 이를 어떻게 구성해야 할지 신중하게 고민하는 사람은 훨씬 적습니다.

처음에는 문제가 없어 보일 수 있습니다. 작은 프로젝트는 기본적인 `CLAUDE.md` 파일, 몇 가지 설정, 그리고 한두 개의 추가 파일만으로도 운영될 수 있습니다. 그러나 프로젝트가 커지면서 이러한 임시방편적인 설정은 한계를 드러내기 시작합니다. 지침은 유지보수하기 어려워지고, 워크플로(workflow)는 엉뚱한 곳에 흩어지며, 폴더는 유용한 구성과 설명하기 어려운 잡동사니로 뒤섞이게 됩니다.

이것이 바로 구조가 중요한 이유입니다.

잘 정리된 `.claude/` 폴더는 Claude를 안내하고 신뢰하며 실제 프로젝트 전반에 걸쳐 확장하기 쉽게 만듭니다. 이는 광범위한 지침과 세부 규칙을, 재사용 가능한 워크플로와 자동화된 작업을, 팀 전체 표준과 개인적인 선호를 분리하는 데 도움이 됩니다. `.claude/`를 무작위 구성 파일의 쓰레기장으로 취급하는 대신, 프로젝트의 운영 계층(operating layer)의 일부로 여기기 시작하게 됩니다.

이 가이드에서는 효율을 극대화하기 위해 `.claude/` 폴더를 어떻게 구조화해야 하는지, 각 부분이 무엇을 포함해야 하는지, 그리고 워크플로가 더욱 정교해짐에 따라 설정을 깔끔하게 유지하는 방법을 살펴보겠습니다.

## 목차:

*   1. `.claude/` 구조가 중요한 이유
*   2. 핵심부터 시작하기: 최상위 레벨에 두어야 할 것
*   3. 간결한 지침 유지하기: `CLAUDE.md` vs `rules/`
*   4. 실행을 위한 구성: `hooks/`, `commands/`, 그리고 재사용 가능한 워크플로
*   5. 전문화된 기능 구조화: `skills/`와 `agents/`
*   6. 팀 구조와 개인 구조 분리하기
*   7. 효율적인 `.claude/` 폴더를 위한 실용적인 청사진
*   8. 피해야 할 일반적인 구조적 실수

## 1. `.claude/` 구조가 중요한 이유

대부분의 사람들은 우연히 `.claude/` 폴더를 발견합니다!

프로젝트에 나타난 것을 보고 Claude Code와 관련이 있다고 인식한 후, 특정 설정을 변경할 필요가 없는 한 그냥 둡니다. 시간이 지나면서 이 폴더는 명확한 구조 없이 지침, 스크립트, 규칙, 실험 파일들이 뒤섞이게 됩니다.

이는 초반에는 작동할 수 있습니다. 하지만 프로젝트가 커지면 더 이상 작동하지 않습니다.

제대로 정리되지 않은 `.claude/` 폴더는 제대로 정리되지 않은 코드베이스(codebase)와 동일한 종류의 마찰을 만듭니다. 지침은 유지보수하기 어려워지고, 중요한 규칙은 묻혀버립니다. 자동화 스크립트는 명명 규칙(naming conventions) 없이 쌓여갑니다. 팀원들은 새로운 지침을 어디에 추가해야 할지, 어떤 파일이 여전히 활성화되어 있는지 확신하지 못합니다. Claude와 함께 작업하는 것을 더 쉽게 만드는 대신, 폴더는 오히려 혼란을 가중시킵니다.

잘 구조화된 `.claude/` 폴더는 그 반대입니다. 이는 Claude Code 설정의 모든 부분에 명확한 위치를 제공합니다. 주요 지침은 찾기 쉽고, 재사용 가능한 규칙은 일회성 선호 사항과 분리됩니다. 훅(hooks)과 워크플로 파일은 예측 가능한 위치에 존재합니다. 스킬(skills)과 에이전트(agents)는 실제로 필요할 때 흩어진 실험이 아닌 의도적인 도구로 조직됩니다.

이러한 구조가 중요한 이유는 Claude Code가 Claude뿐만 아니라 당신과 당신의 팀에게도 이해하기 쉬운 설정일 때 가장 잘 작동하기 때문입니다. 폴더가 깔끔하면 자신감 있게 업데이트할 수 있습니다. 지저분하면 작은 개선조차 위험하게 느껴지기 시작합니다.

목표는 가능한 가장 큰 `.claude/` 폴더를 만드는 것이 아닙니다. 목표는 워크플로가 더욱 정교해짐에 따라 탐색하기 쉽고, 유지보수하기 쉬우며, 확장하기 쉬운 폴더를 만드는 것입니다. 실제로, 최대 효율성은 명확성에서 비롯됩니다. 모든 파일은 목적이 있어야 하고, 모든 폴더는 특정 문제를 해결해야 하며, 전체 구조는 한눈에 이해되어야 합니다.

그것이 바로 `.claude/`를 제대로 정리하는 진정한 가치입니다. 단순히 구성 파일을 저장하는 것이 아니라, Claude가 프로젝트 내에서 어떻게 작동하는지를 형성하는 운영 계층을 구축하는 것입니다.

각 구성 요소를 자세히 살펴보기 전에, 효율적인 `.claude/` 폴더의 전체 구조를 살펴보는 것이 도움이 됩니다. 이것을 프로젝트가 더욱 정교해짐에 따라 구축하고자 하는 목표 레이아웃(layout)이라고 생각하십시오.

```
your-project/
├── CLAUDE.md                  # 주요 프로젝트 지침
├── CLAUDE.local.md            # 개인적인 재정의 (커밋되지 않음)
└── .claude/
    ├── settings.json          # 제어 계층
    ├── rules/                 # 모듈식 지침
    ├── hooks/                 # 자동화 스크립트
    ├── commands/              # 재사용 가능한 프롬프트 워크플로
    ├── skills/                # 패키지화된 기능
    └── agents/                # 전문화된 서브 에이전트
```

## 2. 핵심부터 시작하기: 최상위 레벨에 두어야 할 것

하위 폴더들을 넘어서면, 설정에서 가장 중요한 부분은 여전히 최상위 레벨입니다. 이곳은 Claude가 중첩된 디렉터리(directories)를 뒤지지 않고도 프로젝트를 최상위 레벨에서 정의하는 파일을 찾아야 하는 곳입니다.

대부분의 경우, 두 가지가 먼저 중요합니다. 바로 주요 지침 파일과 주요 구성 파일입니다.

프로젝트 루트에 있는 `CLAUDE.md`는 Claude가 코드베이스를 이해하는 진입점 역할을 해야 합니다. 이곳에는 Claude가 거의 모든 세션에서 필요로 하는 필수적인 컨텍스트(context)를 배치합니다. 즉, 스택(stack), 주요 아키텍처(architecture) 결정, 중요한 컨벤션, 그리고 일반적인 개발 워크플로를 정의하는 명령(commands) 등이 포함됩니다. 팀원이 이 저장소에서 Claude가 어떻게 작동해야 하는지 이해하고 싶다면, 이 파일이 가장 먼저 확인해야 할 파일이 되어야 합니다.

`.claude/` 폴더 내에서 `settings.json` 파일도 같은 이유로 최상위 레벨에 있어야 합니다. 이것은 단순한 지원 파일이 아닙니다. `settings.json`은 Claude Code의 운영 측면(operational side)을 제어합니다. 즉, 찾기 쉽고 업데이트하기 쉬워야 하는 권한(permissions), 훅(hooks), 그리고 프로젝트 레벨의 동작(behavior)을 제어합니다. 이 파일을 폴더 구조 깊숙이 묻어두면 실질적인 이득 없이 일상적인 유지보수만 어려워집니다.

최상위 레벨은 의도적으로 가볍게 유지해야 합니다. 많은 사람들이 하는 실수는 `.claude/`의 루트를 자신들이 만드는 모든 스크립트, 워크플로, 실험적 메모를 위한 저장 공간으로 취급하는 것입니다. 이는 일반적으로 빠르게 혼란을 초래합니다. 대신, 최상위 레벨은 프로젝트를 전역적으로 정의하는 파일만 포함해야 합니다. 더 구체적인 내용은 전용 폴더로 이동해야 합니다.

간단하게 생각하자면 다음과 같습니다:

*   `CLAUDE.md`: 프로젝트 작동 방식 설명
*   `settings.json`: 프로젝트 내 Claude 작동 방식 제어

나머지 모든 것은 하위 폴더에 정리됩니다.

이러한 분리는 각 파일이 다른 목적을 수행하기 때문에 유용합니다. `CLAUDE.md`는 지침에 관한 것이고, `settings.json`은 제어에 관한 것입니다. 하나는 Claude에게 코드베이스에서 무엇이 중요한지 알려주고, 다른 하나는 Claude가 무엇을 할 수 있는지, 그리고 작업 주변에서 어떤 자동 동작이 발생해야 하는지를 형성합니다.

예를 들어, FastAPI 프로젝트에서 `CLAUDE.md`는 모든 API 스키마(schemas)는 `schemas/`에, 모든 서비스 로직(logic)은 `services/`에 있으며, 모든 엔드포인트(endpoint)는 Pydantic 모델로 입력을 검증해야 한다고 설명할 수 있습니다.

한편, `.claude/settings.json`은 Claude가 테스트 명령을 실행하고, `.env` 파일에 대한 접근을 거부하며, 파일 편집 후 포맷팅 훅(formatting hook)을 트리거(trigger)하도록 허용할 수 있습니다. 이 두 파일은 함께 작동하지만, 서로 섞여서는 안 됩니다.

최상위 레벨은 공유 파일과 개인 재정의(personal overrides) 사이를 명확히 구분해야 합니다. 공유 `CLAUDE.md`는 팀 전체가 이점을 얻기 때문에 루트(root)에 속합니다. `CLAUDE.local.md`와 같은 개인 재정의는 다릅니다. 이는 로컬(local) 환경 설정(preferences)을 위한 것이며 팀의 공유 구조에서 벗어나 있어야 합니다. 동일한 논리가 `.claude/` 내의 `settings.local.json`에도 적용됩니다.

다시 말해, 최상위 레벨은 가장 큰 질문에 먼저 답해야 합니다. Claude는 이 프로젝트에 대해 무엇을 알아야 하는가? Claude는 여기서 무엇을 할 수 있는가? 나머지 모든 것은 그 기반 아래에서 정리될 수 있습니다.

그것이 바로 구조를 효율적으로 만듭니다. 가장 중요한 파일은 명확하게 유지되고, 더 전문화된 부분은 핵심 설정을 복잡하게 하지 않고 더 쉽게 관리할 수 있는 폴더로 이동됩니다.

## 3. 간결한 지침 유지하기: `CLAUDE.md` vs `rules/`

`.claude/` 폴더를 비효율적으로 만드는 가장 쉬운 방법 중 하나는 `CLAUDE.md`에 너무 많은 내용을 담는 것입니다.

처음에는 프로젝트가 작기 때문에 이 파일이 보통 잘 작동합니다. 몇 가지 명령, 일부 코딩 컨벤션, 그리고 아키텍처에 대한 짧은 메모를 추가합니다. 그러나 코드베이스가 커지면 `CLAUDE.md`는 종종 팀이 Claude에게 기억시키고 싶은 모든 것을 담는 만능 문서가 됩니다. 이때부터 명확성을 잃기 시작합니다.

더 나은 구조는 `CLAUDE.md`를 프로젝트의 전체 지식 기반(knowledge base)이 아닌 운영 가이드로 취급하는 것입니다.

`CLAUDE.md`는 Claude가 대부분의 세션에서 필요로 하는 지침을 담아야 합니다.

*   프로젝트란 무엇인가
*   어떻게 구성되어 있는가
*   어떤 명령이 가장 중요한가
*   어떤 컨벤션이 광범위하게 적용되는가
*   Claude가 놓치지 말아야 할 제약사항은 무엇인가

`rules/` 폴더는 더 구체적인 지침이 속해야 하는 곳입니다. 여기에는 코드베이스의 특정 부분, 특정 워크플로, 또는 테스트나 보안과 같은 특정 엔지니어링(engineering) 문제와 관련된 지침이 포함됩니다.

간단하게 생각하자면 다음과 같습니다:

*   `CLAUDE.md`는 전역 지침을 담는다.
*   `rules/`는 전문화된 지침을 담는다.

이러한 분리는 구조를 유지보수하기 쉽게 만들고 확장하기 쉽게 만듭니다.

### 실용적인 예시

Next.js 프런트엔드(frontend), FastAPI 백엔드(backend), 그리고 보고 작업(reporting jobs)을 위한 데이터 파이프라인(data pipeline)이라는 세 가지 주요 영역을 가진 제품을 작업한다고 상상해 보십시오. `CLAUDE.md`에 모든 것을 설명하려고 하면 파일이 매우 빠르게 복잡해집니다. 프런트엔드 컨벤션이 백엔드 유효성 검사 규칙과 데이터 파이프라인 메모 옆에 놓입니다. Claude는 더 많은 컨텍스트를 얻지만, 항상 더 나은 컨텍스트를 얻는 것은 아닙니다.

더 깔끔한 설정은 다음과 같을 것입니다:

```
your-project/
├── CLAUDE.md
└── .claude/
    └── rules/
        ├── frontend.md
        ├── backend-api.md
        ├── testing.md
        └── data-pipelines.md
```

이 설정에서:

*   `CLAUDE.md`: 제품을 상위 레벨에서 설명합니다.
*   `frontend.md`: UI 컨벤션에 중점을 둡니다.
*   `backend-api.md`: API 동작 및 유효성 검사 규칙에 중점을 둡니다.
*   `testing.md`: 테스트를 작성하고 실행하는 방법을 설명합니다.
*   `data-pipelines.md`: 배치 작업(batch jobs) 및 스케줄링된 작업(scheduled tasks)을 위한 컨벤션을 담습니다.

이는 하나의 큰 지침 파일보다 훨씬 유지보수하기 쉽습니다.

### `CLAUDE.md`에 무엇을 유지해야 하는가?

좋은 `CLAUDE.md`는 일반적으로 다음을 포함합니다:

*   주요 스택
*   상위 레벨 아키텍처
*   가장 중요한 개발 명령
*   광범위한 코드 컨벤션
*   중요한 프로젝트 전반의 경고 또는 제약사항

FastAPI 프로젝트의 짧은 예시는 다음과 같습니다:

```markdown
# 프로젝트: 고객 인사이트 API

## 스택
- FastAPI
- PostgreSQL
- SQLAlchemy
- Pytest

## 구조
- `app/api/`는 라우트 정의를 포함합니다.
- `app/services/`는 비즈니스 로직을 포함합니다.
- `app/models/`는 ORM 모델을 포함합니다.
- `app/schemas/`는 요청 및 응답 스키마를 포함합니다.

## 명령
- `pytest`는 테스트 스위트를 실행합니다.
- `alembic upgrade head`는 마이그레이션을 적용합니다.
- `ruff check .`는 린팅을 실행합니다.
- `ruff format .`은 코드를 포맷합니다.

## 컨벤션
- 모든 요청 본문을 Pydantic 스키마로 검증하십시오.
- 라우트 핸들러를 얇게 유지하고, 비즈니스 로직은 서비스에 속해야 합니다.
- API 응답에 내부 예외 세부 정보를 노출하지 마십시오.
```

이것은 Claude에게 세부 사항에 압도되지 않고 신뢰할 수 있는 시작점을 제공하기 때문에 유용합니다.

### `rules/`로 무엇을 옮겨야 하는가?

지침이 좁아지거나 특정 영역에 국한될 경우, `CLAUDE.md`에서 옮겨야 합니다.

예를 들어, `backend-api.md`에는 다음과 같은 규칙이 포함될 수 있습니다:

```markdown
# 백엔드 API 규칙

- 모든 새로운 엔드포인트는 요청 및 응답 스키마를 포함해야 합니다.
- 데이터베이스 세션을 위해 의존성 주입(dependency injection)을 사용하십시오.
- 컬렉션 엔드포인트에 대해 페이지 매김된(paginated) 결과를 반환하십시오.
- 외부 API 실패를 공유 로거(logger)로 기록하십시오.
- 라우트 파일 내 로직보다 서비스 계층 함수를 선호하십시오.
```

그리고 `frontend.md`에는 다음이 포함될 수 있습니다:

```markdown
# 프런트엔드 규칙

- 클라이언트 상호작용이 필요하지 않는 한 서버 컴포넌트(server components)를 선호하십시오.
- UI 상태(state)가 페이지 전체에서 공유되지 않는 한 로컬에 유지하십시오.
- 새로운 디자인 시스템 컴포넌트(design system components)를 생성하기 전에 기존 컴포넌트를 재사용하십시오.
- 페이지별 컴포넌트는 가능하면 해당 라우트 옆에 두십시오.
```

이제 각 파일은 명확한 목적을 가집니다.

### `rules/`가 더 나은 선택이 되는 시점

일반적으로 다음 경우에 `rules/`로 분리해야 합니다:

*   `CLAUDE.md`가 너무 복잡하게 느껴지기 시작할 때
*   저장소의 다른 부분에 다른 지침이 필요할 때
*   다른 사람들이 다른 표준을 가질 때
*   팀이 컨벤션을 자주 업데이트할 때
*   특정 경로(paths)나 관심사에 지침 범위를 지정하고 싶을 때

이때부터 모듈성(modularity)의 진가가 발휘됩니다. 매번 하나의 큰 문서를 편집하는 대신, 문제에 해당하는 파일만 업데이트합니다.

### 경로 기반 규칙으로 더 강력한 구조

저장소가 커짐에 따라 일부 규칙을 특정 파일에만 적용하도록 하는 것이 유용할 수 있습니다. 예를 들어:

```
.claude/
└── rules/
    ├── frontend.md
    ├── backend-api.md
    ├── tests.md
    └── migrations.md
```

그렇게 하면 백엔드(backend) 관련 지침이 프런트엔드(frontend) 작업을 복잡하게 하지 않으며, 마이그레이션(migration) 관련 지침은 Claude가 데이터베이스(database) 변경 작업을 할 때만 나타납니다.

구문에 깊이 들어가지 않더라도, 중요한 조직적 아이디어는 간단합니다. 전역 지침은 광범위하게 유지하고, 특정 대상에 대한 지침은 모듈식 파일로 옮깁니다.

### 이 구조가 더 효율적인 이유는 무엇인가?

이러한 분리는 세 가지 방식으로 효율성을 향상시킵니다.

첫째, `CLAUDE.md`의 가독성을 유지합니다. 이는 사람과 Claude 모두 주요 프로젝트 컨텍스트를 빠르게 추출하는 것을 더 쉽게 만듭니다.
둘째, 유지보수 오버헤드(overhead)를 줄입니다. 단 하나의 컨벤션이 변경될 때마다 하나의 큰 파일을 계속 다시 열 필요가 없습니다.
셋째, 팀 소유권(team ownership)을 위한 더 깔끔한 경로를 만듭니다. 테스트, 프런트엔드 작업 또는 API 설계 담당자는 주요 지침 파일을 공유된 잡동사니로 만들지 않고 자신의 규칙 파일을 업데이트할 수 있습니다.

실제로, 최고의 설정은 가장 많은 지침을 가진 것이 아닙니다. 지침이 적절한 레벨에 배치된 것입니다.

## 4. 실행을 위한 구성: `hooks/`, `commands/`, 그리고 재사용 가능한 워크플로

`.claude/` 폴더에 깔끔한 지침 계층이 갖춰지면, 다음 단계는 지침을 행동으로 전환하는 파일을 정리하는 것입니다.

여기서 많은 설정들이 복잡해지기 시작합니다. 팀은 포맷팅(formatting)을 위한 셸 스크립트(shell script), 풀 리퀘스트(pull request) 검토를 위한 프롬프트(prompt) 파일, 어쩌면 테스트 확인을 위한 또 다른 스크립트를 추가하고, 얼마 지나지 않아 이 파일들은 `.claude/`의 루트에 생성자에게만 이해되는 이름으로 흩어지게 됩니다.

더 나은 접근 방식은 지침 파일과 실행 파일(action files)을 분리하는 것입니다.

지침 파일은 Claude가 어떻게 행동해야 하는지 설명합니다. 실행 파일은 Claude가 실제로 반복적으로 수행하는 작업을 지원합니다. 이것이 `hooks/`와 `commands/` 같은 폴더가 구조 내에서 자체적인 위치를 가질 자격이 있는 이유입니다.

### `hooks/`의 역할

`hooks/` 폴더는 Claude의 워크플로 특정 지점에서 자동으로 실행되는 스크립트를 배치하는 곳입니다.

이 파일들은 일반적인 문서가 아닙니다. 운영 스크립트입니다. 이들의 역할은 보통 다음 세 가지 중 하나입니다:

*   위험한 작업 방지
*   Claude의 출력물 정리 또는 유효성 검사
*   워크플로 요구사항 강제

깔끔한 구조는 다음과 같을 수 있습니다:

```
.claude/
├── settings.json
├── hooks/
│   ├── block-dangerous-commands.sh
│   ├── format-edits.sh
│   └── run-tests-before-stop.sh
└── commands/
    ├── review-pr.md
    ├── write-tests.md
    └── summarize-changes.md
```

이 설정에서:

*   `block-dangerous-commands.sh`는 안전 훅입니다.
*   `format-edits.sh`는 파일 변경 사항을 깔끔하게 유지합니다.
*   `run-tests-before-stop.sh`는 Claude가 작업을 마치기 전에 품질을 확인합니다.

폴더 없이 이 세 스크립트를 모두 `.claude/`에 넣는 것보다 훨씬 이해하기 쉽습니다.

### 실용적인 훅(hook) 예시

모든 Claude 편집이 자동으로 포맷팅되어야 하는 파이썬(Python) 서비스에서 작업한다고 상상해 보십시오. 워크플로는 다음을 사용할 수 있습니다: `ruff format .` (포맷팅용), `ruff check .` (린팅(linting)용).

좋은 구조는 포맷팅 로직(logic)을 전용 스크립트에 배치할 것입니다:

```bash
#!/usr/bin/env bash
jq -r '.tool_input.file_path' | xargs ruff format
```

그리고 그 스크립트는 여기에 위치할 것입니다:

```
.claude/
└── hooks/
    └── format-edits.sh
```

중요한 점은 구조적이라는 것입니다. 훅이 작더라도 `hooks/`에 있어야 합니다. 왜냐하면 그 폴더가 팀에게 어떤 종류의 파일인지 정확히 알려주기 때문입니다.

### `commands/`의 역할

`commands/` 폴더는 재사용 가능한 프롬프트(prompt) 워크플로에 유용합니다.

이들은 훅처럼 자동적이지 않습니다. 대신, 당신이나 당신의 팀이 반복적으로 실행하는 작업을 패키지화합니다. 예를 들어:

*   풀 리퀘스트 검토
*   누락된 테스트 작성
*   릴리즈 노트(release notes)를 위한 변경 사항 요약
*   보고된 문제 디버깅(debugging)
*   마이그레이션(migration) 단계 초안 작성

`commands/`의 가치는 반복적인 프롬프트 작성을 줄인다는 것입니다. 매번 같은 지침을 다시 작성하는 대신, 해당 워크플로를 명확한 한 곳에 한 번 저장합니다.

예를 들어:

```
.claude/
└── commands/
    ├── review-pr.md
    ├── fix-bug.md
    ├── write-tests.md
    └── prepare-release-notes.md
```

각 파일은 하나의 명확한 목적을 가져야 합니다. 이것이 효율성의 핵심입니다.

### 실용적인 명령(command) 예시

팀이 병합(merging) 전에 Claude에게 백엔드 변경 사항을 검토하도록 자주 요청한다고 가정해 봅시다. 매번 새로운 프롬프트를 작성하는 대신, 다음을 만듭니다.

```markdown
# 리뷰-PR

현재 변경 사항을 다음 사항에 중점을 두고 검토하십시오:
- 정확성
- 누락된 엣지 케이스(edge cases)
- API 계약 변경
- 테스트 커버리지(test coverage) 격차
요약:
1. 심각한 문제
2. 중간 위험 문제
3. 제안된 개선 사항
```

다음과 같이 저장됩니다: `.claude/commands/review-pr.md`

이는 워크플로를 즉시 더 쉽게 재사용하고 시간이 지남에 따라 개선하기 쉽게 만듭니다.

### 왜 이들을 분리해야 하는가?

흔한 실수는 이 파일들의 목적을 모호하게 하는 것입니다.

예를 들어:

*   재사용 가능한 검토 프롬프트를 `CLAUDE.md` 안에 넣기
*   훅 스크립트를 설정 파일 바로 옆에 배치하기
*   하나의 큰 스크립트로 포맷팅, 테스트, 알림, 로깅(logging)을 한꺼번에 처리하기

이는 일반적으로 유지보수 문제를 야기합니다.

더 깔끔한 패턴은 다음과 같습니다:

*   `CLAUDE.md`는 프로젝트를 설명합니다.
*   `rules/`는 지침을 구체화합니다.
*   `hooks/`는 자동 운영 스크립트를 포함합니다.
*   `commands/`는 재사용 가능한 프롬프트 워크플로를 포함합니다.

이러한 분할은 폴더를 더 쉽게 훑어볼 수 있게 하고, 확장하기 쉽게 하며, 팀원들이 이해하기 쉽게 만듭니다.

```
.claude/
├── settings.json
├── rules/
│   ├── frontend.md
│   ├── backend-api.md
│   └── testing.md
├── hooks/
│   ├── block-dangerous-commands.sh
│   ├── format-edits.sh
│   └── run-tests-before-stop.sh
└── commands/
    ├── review-pr.md
    ├── write-tests.md
    └── summarize-changes.md
```

이는 `hooks/`와 `commands/`가 지침 계층 안에 있는 것이 아니라 그 옆에 있음을 보여주기 때문에 잘 작동합니다.

### 좋은 명명 전략

이 섹션은 명명(naming)에 대해 언급하기에도 좋은 곳입니다. 왜냐하면 명명이 구조의 일부이기 때문입니다.

좋은 이름은 목적을 명확하게 합니다:

*   `format-edits.sh`
*   `block-dangerous-commands.sh`
*   `review-pr.md`
*   `write-tests.md`

모호한 이름은 폴더를 사용하기 어렵게 만듭니다:

*   `script1.sh`
*   `helper.sh`
*   `misc.md`
*   `temp-review.md`

누군가 `.claude/hooks/` 또는 `.claude/commands/`를 열면, 각 파일의 목적을 즉시 이해해야 합니다.

### 웹 앱(web app) 팀을 위한 현실적인 예시

서비스형 소프트웨어(SaaS) 웹 앱의 경우, 구조는 다음과 같을 수 있습니다:

```
.claude/
├── hooks/
│   ├── lint-edits.sh
│   ├── block-prod-commands.sh
│   └── notify-on-completion.sh
└── commands/
    ├── review-ui-changes.md
    ├── generate-test-cases.md
    ├── investigate-bug-report.md
    └── draft-release-summary.md
```

이것은 각 파일이 반복되는 필요를 반영하기 때문에 강력한 설정입니다:

*   훅(hooks)은 보호하고 검증합니다.
*   명령(commands)은 반복 작업을 가속화합니다.

### 이 구조가 더 효율적인 이유는 무엇인가?

이 구조는 운영 로직(operational logic)을 지침 파일에서 분리하여 효율성을 향상시킵니다.

`CLAUDE.md`를 프로세스 세부 정보로 부풀리거나 유틸리티(utility) 파일을 여기저기 흩뿌리는 대신, 각 종류의 아티팩트(artifact)에 고유한 위치를 부여합니다. 이는 워크플로를 유지보수하기 쉽게 만들고, 팀원을 온보딩(onboarding)하기 쉽게 하며, 새 파일이 어디에 속해야 하는지 파악하기 쉽게 만듭니다.

실제로, `.claude/` 폴더는 사람들이 어떤 것이 다음 중 무엇인지 더 이상 추측할 필요가 없을 때 훨씬 더 유용해집니다:

*   프로젝트 규칙
*   자동화 스크립트
*   재사용 가능한 작업 정의

그 명확성이 설정을 확장 가능하게 만듭니다.

## 5. 전문화된 기능 구조화: `skills/`와 `agents/`

`.claude/` 폴더의 핵심이 정리되면, 다음 질문은 더 전문화된 기능(capabilities)이 필요한지 여부입니다.

여기서 `skills/`와 `agents/`가 등장합니다.

이 두 폴더는 유용하지만, 대부분의 프로젝트에서 시작점이 되어서는 안 됩니다. 팀에 자체적인 구조를 가질 만큼 복잡한 반복적인 워크플로가 있을 때 의미가 있습니다. `CLAUDE.md`, `rules/`, `hooks/`, 그리고 `commands/`가 이미 대부분의 필요를 충족한다면, 이는 좋은 신호입니다. 이는 당신의 설정이 여전히 간단하고 효율적임을 의미합니다.

Claude가 더 전문화된 작업을 더 깔끔하고 재사용 가능한 방식으로 처리하기를 원할 때 `skills/`와 `agents/`를 추가합니다.

### `skills/` 폴더에 속하는 것

스킬(skill)은 패키지화된 기능(packaged capability)으로 생각하는 것이 가장 좋습니다. 워크플로가 다음을 가질 때 유용합니다:

*   명확한 목적
*   반복 가능한 프로세스
*   지원 파일 또는 상세 지침
*   짧은 명령 파일로는 더 이상 충분하지 않을 정도의 복잡성

이것이 `skills/`가 느슨한 파일들보다는 자체 포함된 디렉터리(directories)들의 폴더로 가장 잘 작동하는 이유입니다. 깔끔한 구조는 다음과 같습니다:

```
.claude/
└── skills/
    ├── api-review/
    │   ├── SKILL.md
    │   └── checklist.md
    ├── release-prep/
    │   ├── SKILL.md
    │   └── release-template.md
    └── docs-audit/
        ├── SKILL.md
        └── style-guide.md
```

각 스킬은 자체적인 공간을 가지고 있기 때문에 효율적입니다. 로직(logic), 설명, 지원 파일들이 함께 유지됩니다.

### 실용적인 스킬(skill) 예시

팀이 정기적으로 릴리즈(releases)를 준비한다고 상상해 보십시오. 이 워크플로는 다음을 포함합니다:

*   최근 변경 사항 확인
*   버전 노트(version notes) 검토
*   릴리즈 요약 초안 작성
*   마이그레이션(migrations) 또는 주요 변경 사항(breaking changes) 존재 여부 확인

이것은 간단한 재사용 가능한 프롬프트 그 이상입니다. 컨텍스트(context)를 가진 반복 가능한 프로세스입니다. 따라서 모든 것을 `commands/`에 저장하는 대신, 다음과 같이 패키지화합니다:

```
.claude/
└── skills/
    └── release-prep/
        ├── SKILL.md
        └── release-template.md
```

`SKILL.md`는 스킬의 목적을 정의하고, `release-template.md`는 Claude에게 재사용 가능한 출력 구조를 제공합니다. 이는 릴리즈(release) 지침을 여러 명령 파일에 흩뿌리는 것보다 훨씬 깔끔합니다.

### 워크플로가 명령(command)으로 유지되어야 하는 시점

이러한 구분은 구조상 중요합니다. 작업이 다음과 같을 때 일반적으로 명령(command)으로 충분합니다:

*   짧고
*   직접적이며
*   주로 프롬프트 기반이고
*   하나의 파일로 쉽게 표현될 수 있을 때

작업이 다음을 가질 때 스킬(skill)이 더 합리적입니다:

*   여러 단계가 있고
*   보조 문서가 필요하며
*   더 신중하게 표준화할 만큼 중요하고
*   시간이 지남에 따라 재사용될 때

간단하게 생각하자면:

*   `commands/` = 가벼운 재사용 가능한 작업
*   `skills/` = 더 깊이 있는 패키지화된 워크플로

### `agents/` 폴더에 속하는 것

`agents/` 폴더는 전문화된 서브 에이전트(subagents)를 위한 것입니다. 이들은 특정 종류의 작업에 대해 더 좁고 역할 기반의 페르소나(persona)를 원할 때 유용합니다. 모든 작업에 동일한 일반 Claude 설정을 사용하는 대신, 다음과 같은 집중된 역할을 정의합니다:

*   코드 검토자
*   보안 감사자
*   문서 편집자
*   마이그레이션 확인자
*   성능 검토자

깔끔한 구조는 다음과 같습니다:

```
.claude/
└── agents/
    ├── code-reviewer.md
    ├── security-auditor.md
    ├── docs-writer.md
    └── performance-checker.md
```

각 에이전트(agent) 파일이 단일하고 명확한 목적을 가지고 있기 때문에 잘 작동합니다.

### 실용적인 에이전트(agent) 예시

팀이 병합(merging) 전에 Claude에게 API 변경 사항을 정확성과 엣지 케이스(edge cases)에 대해 검토하도록 자주 요청한다고 가정해 봅시다. 일반적인 프롬프트도 작동할 수 있지만, 시간이 지남에 따라 더 엄격한 경계를 가진 전용 검토자 페르소나(persona)를 원할 수 있습니다.

그것이 다음과 같은 에이전트(agent)의 강력한 사례입니다: `.claude/agents/code-reviewer.md`

해당 에이전트(agent)는 다음 사항에 중점을 두도록 설계될 수 있습니다:

*   정확성
*   위험한 가정
*   누락된 엣지 케이스
*   테스트 커버리지(test coverage) 격차

구조적 이점은 전문화된 검토 로직(logic)이 더 이상 무작위 프롬프트 안에 숨겨져 있거나 여러 파일에 흩어져 있지 않다는 것입니다. 전용 보금자리가 있습니다.

### 두 폴더를 깔끔하고 목적 지향적으로 유지하기

`skills/`와 `agents/`에서 가장 큰 실수 중 하나는 이들을 실험적인 저장 공간으로 취급하는 것입니다. 이는 일반적으로 다음과 같은 것들로 가득 찬 폴더로 이어집니다:

*   미완성 스킬
*   중복된 에이전트
*   중첩되는 검토자 페르소나
*   `helper.md` 또는 `analysis.md`와 같은 모호한 이름

이는 설정을 더 효율적으로 만드는 것이 아니라 사용하기 어렵게 만듭니다. 더 나은 규칙은 다음과 같습니다:

*   각 스킬은 하나의 반복적인 워크플로를 해결해야 합니다.
*   각 에이전트는 하나의 전문화된 역할을 소유해야 합니다.
*   두 파일이 심하게 중복된다면, 아마도 병합하거나 단순화해야 할 것입니다.

```
.claude/
├── commands/
│   ├── review-pr.md
│   └── write-tests.md
├── skills/
│   ├── release-prep/
│   │   ├── SKILL.md
│   │   └── release-template.md
│   └── docs-audit/
│       ├── SKILL.md
│       └── style-guide.md
└── agents/
    ├── code-reviewer.md
    ├── security-auditor.md
    └── docs-writer.md
```

이것은 다음 진행을 보여주기 때문에 유용합니다:

*   `commands/`는 가벼운 재사용 가능한 작업을 위한 것
*   `skills/`는 더 풍부한 패키지화된 워크플로를 위한 것
*   `agents/`는 전문화된 페르소나를 위한 것

### 제품 팀을 위한 현실적인 예시

서비스형 소프트웨어(SaaS) 팀의 경우, 구조는 다음과 같을 수 있습니다:

```
.claude/
├── skills/
│   ├── release-prep/
│   │   ├── SKILL.md
│   │   └── release-template.md
│   ├── api-audit/
│   │   ├── SKILL.md
│   │   └── api-checklist.md
│   └── onboarding-docs/
│       ├── SKILL.md
│       └── docs-outline.md
└── agents/
    ├── code-reviewer.md
    ├── security-auditor.md
    └── documentation-editor.md
```

이러한 종류의 구조는 팀에 한 줄짜리 명령 이상을 필요로 하는 몇 가지 반복적인 워크플로가 있을 때 잘 작동합니다.

### 아직 이 폴더들을 추가하지 않아야 할 때

이것은 이 글을 실용적으로 유지하는 데 중요합니다.

`skills/`와 `agents/`가 존재한다는 이유만으로 추가하지 마십시오. 명확한 구조적 이유가 있을 때만 추가하십시오.

다음과 같은 경우에는 아직 필요하지 않을 가능성이 높습니다:

*   대부분의 작업이 여전히 간단할 때
*   팀이 동일한 고급 워크플로를 거의 반복하지 않을 때
*   명령 파일이 이미 문제를 해결할 때
*   아직 기본적인 지침과 권한을 다듬고 있을 때

효율성은 더 많은 폴더를 가지는 것에서 오는 것이 아닙니다. 워크플로가 정당화할 때만 구조를 추가하는 것에서 옵니다.

### 이 구조가 더 효율적인 이유는 무엇인가?

`skills/`와 `agents/`는 혼란을 가중시키지 않으면서 반복을 줄일 때 효율성을 향상시킵니다.

좋은 스킬(skill)은 복잡한 워크플로를 한 곳에 패키지화하여 유지합니다. 좋은 에이전트(agent)는 전문화된 역할을 좁고 재사용 가능하게 유지합니다. 둘 다 의도적으로 사용될 때 `.claude/` 폴더를 확장하기 쉽게 만듭니다.

그러나 핵심은 규율입니다. 이 폴더들은 작고, 목적 지향적이며, 훑어보기 쉬워야 합니다. 실험의 쓰레기장이 되면 더 이상 도움이 되지 않습니다.

최고의 `.claude/` 구조는 가장 진보된 기능을 가진 것이 아닙니다. 고급 기능이 워크플로를 명확하게 개선할 때만 도입된 것입니다.

## 6. 팀 구조와 개인 구조 분리하기

`.claude/` 폴더는 하나의 공유 공간으로 생각하는 것을 멈추고 팀 구성(team configuration)과 개인 구성(personal configuration)이라는 두 개의 분리된 계층으로 취급하기 시작할 때 훨씬 관리하기 쉬워집니다.

이러한 구분은 대부분의 사람들이 예상하는 것보다 더 중요합니다.

프로젝트 초기 단계에는 모든 것이 한곳에 모이는 것이 일반적입니다. 개발자는 `CLAUDE.md`에 개인적인 선호 사항을 추가합니다. 누군가는 자신의 기기에서만 의미 있는 워크플로를 위해 `settings.json`을 조정합니다. 또 다른 사람은 유용한 에이전트를 추가하지만, 이는 팀의 필요보다는 자신의 작업 스타일을 반영합니다.

처음에는 심각해 보이지 않습니다. 그러나 시간이 지남에 따라 공유된 설정은 프로젝트를 명확하게 나타내지 않기 때문에 이해하기 어려워집니다. 이는 프로젝트 표준과 개별 습관의 혼합을 나타내기 시작합니다.

마찰은 거기서 시작됩니다.

효율적인 설정은 프로젝트가 필요로 하는 것과 개별 개발자가 선호하는 것 사이에 깔끔한 분리를 만듭니다.

프로젝트 레벨 구조는 팀 전체가 일관성 있게 작업하는 데 도움이 되는 규칙, 워크플로, 보호 장치를 포함해야 합니다. 이 파일들은 모든 사람이 저장소를 복제하고 해당 코드베이스에서 Claude를 사용하기 시작할 때 상속받기를 원하는 파일입니다. 반대로, 개인적인 선호 사항은 로컬(local)에 유지되어야 합니다. 프로젝트 내의 재정의(override) 파일에 있거나 사용자의 전역 `~/.claude/` 디렉터리에 있어야 합니다.

건강한 구조는 종종 다음과 같습니다:

```
your-project/
├── CLAUDE.md
├── CLAUDE.local.md
└── .claude/
    ├── settings.json
    ├── settings.local.json
    ├── rules/
    ├── hooks/
    ├── commands/
    ├── skills/
    └── agents/
~/.claude/
├── CLAUDE.md
├── settings.json
├── skills/
├── agents/
└── projects/
```

이 레이아웃(layout)은 경계가 명확하기 때문에 잘 작동합니다.

프로젝트 레벨에서는 파일들이 이 저장소 내에서 Claude가 어떻게 행동해야 하는지를 정의합니다. 여기에는 공유 지침, 모듈식 규칙, 자동화 훅(hooks), 재사용 가능한 팀 워크플로, 그리고 반복되는 프로젝트 문제를 해결하는 모든 전문화된 스킬(skills) 또는 에이전트(agents)가 포함됩니다.

개인 레벨에서는 파일들이 한 개발자가 여러 저장소에서 어떻게 작업하기를 선호하는지를 정의합니다. 여기에는 개인적인 작성 선호 사항, 비공개 스킬, 사용자 정의 에이전트, 또는 팀 저장소에 커밋(commit)되어서는 안 되는 기기별 상태(machine-specific states)가 포함될 수 있습니다.

좋은 규칙은 간단합니다. 구성이 팀 전체가 더 일관성 있게 작업하는 데 도움이 된다면, 프로젝트에 속합니다. 주로 한 사람의 워크플로를 반영한다면, 그 사람의 로컬(local) 또는 전역 설정(global setup)에 속합니다.

예를 들어, 결제 서비스에서 작업하는 백엔드(backend) 팀을 상상해 보십시오. 그들 모두 Claude가 동일한 API 컨벤션을 따르고, `.env`와 같은 민감한 파일을 피하며, 주요 변경 사항을 마치기 전에 검사를 실행하고, 풀 리퀘스트(pull request)에 동일한 검토 워크플로를 사용하기를 원합니다. 이것들은 모든 사람의 일관성을 향상시키기 때문에 공유 프로젝트 구조에 속합니다.

그 설정은 다음과 같을 수 있습니다:

```
your-project/
├── CLAUDE.md
└── .claude/
    ├── settings.json
    ├── rules/
    │   ├── api.md
    │   ├── testing.md
    │   └── security.md
    ├── hooks/
    │   ├── block-secrets-access.sh
    │   └── run-tests-before-stop.sh
    └── commands/
        ├── review-pr.md
        └── investigate-bug.md
```

그 구조의 모든 것은 팀 가치를 가집니다. 저장소에 합류하는 모든 개발자는 즉시 이점을 얻어야 합니다.

이제 이를 개인 설정(personal setup)과 비교해 보십시오. 어떤 개발자는 여러 프로젝트에 걸쳐 Claude를 사용하여 기술 문서를 작성할 수 있습니다. 또 다른 개발자는 분석 작업을 위해 비공개 SQL 검토 스킬(skill)에 의존할 수 있습니다. 이들은 유용할 수 있지만, 반드시 프로젝트 표준은 아닙니다. 팀이 명시적으로 채택하기로 결정하지 않는 한 개인적인 상태로 유지되어야 합니다.

개인 구조는 다음과 같을 수 있습니다:

```
~/.claude/
├── CLAUDE.md
├── skills/
│   ├── article-outline/
│   │   ├── SKILL.md
│   │   └── outline-template.md
│   └── sql-review/
│       ├── SKILL.md
│       └── checklist.md
└── agents/
    ├── documentation-editor.md
    └── sql-reviewer.md
```

이러한 종류의 설정은 공유 저장소에 혼란을 주지 않으면서 한 사람의 워크플로를 개선합니다.

유용한 중간 계층도 있습니다. 프로젝트 내의 로컬 재정의(override) 파일입니다. `CLAUDE.local.md` 및 `.claude/settings.local.json`과 같은 파일은 누군가가 버전 관리되는 팀 설정을 변경하지 않고 특정 저장소에서 동작을 조정하고 싶을 때 유용합니다. 이는 모든 로컬 선호 사항을 팀 결정으로 만들지 않고도 사람들이 워크플로를 실험하거나 개인화할 여지를 줍니다.

예를 들어, 어떤 개발자는 훅(hook)을 테스트하는 동안 약간 다른 로컬 권한(permissions)을 원할 수 있습니다. 또 다른 개발자는 Claude가 자신의 검토 프로세스를 위해 더 짧은 요약을 생성하도록 원할 수 있습니다. 이것들은 로컬 재정의 파일의 좋은 사용 사례입니다. 프로젝트에 가깝게 유지되지만, 다른 모든 사람을 위한 공유 계약의 일부가 되지는 않습니다.

이것이 구조적으로 중요한 이유는 `.claude/` 설정이 파일 유형별로만 정리되어서는 안 되기 때문입니다. 소유권별로도 정리되어야 합니다.

이러한 소유권 분할은 팀이 성장함에 따라 구성을 유지보수 가능하게 만듭니다. 공유 파일은 모든 사람이 실제 프로젝트 표준을 나타낸다는 것을 알기 때문에 신뢰할 수 있습니다. 개인 파일은 개인들이 저장소에 혼란을 주지 않고 자신만의 작업 스타일에 맞춰 Claude를 조정할 수 있기 때문에 유연하게 유지됩니다.

이것이 진정한 요점입니다. 프로젝트 구조는 일관성에 관한 것이고, 개인 구조는 개별 효율성에 관한 것입니다.

이 두 가지가 혼합되면 `.claude/` 폴더는 신뢰하기 어렵고 유지보수하기 어려워집니다. 분리되어 유지될 때, 설정은 더 깔끔하고, 확장하기 쉬우며, 시간이 지남에 따라 팀이 함께 작업하기 훨씬 쉬워집니다.

## 7. 효율적인 `.claude/` 폴더를 위한 실용적인 청사진

주요 부분들이 명확해지면, 다음 단계는 실제 프로젝트에서 유지보수하기 쉬운 구조로 통합하는 것입니다.

여기서 많은 팀들이 일을 너무 복잡하게 만듭니다. 그들은 `CLAUDE.md`가 무엇을 하는지 이해하고, 규칙과 훅(hooks)을 실험한 다음, 관리할 수 있는 속도보다 더 빠르게 폴더를 추가하기 시작합니다. 그 결과는 고급스러워 보이지만 기본 설정보다 사용하기 어렵게 느껴지는 `.claude/` 설정입니다.

더 나은 접근 방식은 실용적인 청사진(blueprint)의 관점에서 생각하는 것입니다. 즉, 확장할 수 있을 만큼 잘 정리되어 있으면서도 팀의 누구라도 빠르게 이해할 수 있을 만큼 간단한 구조입니다.

강력한 기준선은 다음과 같습니다:

```
your-project/
├── CLAUDE.md
├── CLAUDE.local.md
└── .claude/
    ├── settings.json
    ├── settings.local.json
    ├── rules/
    │   ├── backend.md
    │   ├── frontend.md
    │   ├── testing.md
    │   └── security.md
    ├── hooks/
    │   ├── block-dangerous-commands.sh
    │   ├── format-edits.sh
    │   └── run-checks-before-stop.sh
    ├── commands/
    │   ├── review-pr.md
    │   ├── write-tests.md
    │   └── summarize-changes.md
    ├── skills/
    │   └── release-prep/
    │       ├── SKILL.md
    │       └── release-template.md
    └── agents/
        ├── code-reviewer.md
        └── security-auditor.md
```

이 구조는 각 계층이 명확한 역할을 가지고 있기 때문에 작동합니다.

*   `CLAUDE.md`는 Claude에게 거의 모든 세션에서 필요한 광범위한 컨텍스트(context)를 제공합니다. 프로젝트, 주요 명령(commands), 상위 레벨 아키텍처(architecture), 그리고 저장소 전반에 적용되는 컨벤션(conventions)을 설명합니다.
*   `settings.json`은 운영 측면(operational side)을 제어합니다. 권한(permissions), 훅(hooks), 그리고 프로젝트 전반의 동작(behavior)이 속하는 곳입니다. Claude가 코드베이스 내에서 어떻게 작동하도록 허용되는지를 정의하므로 찾기 쉬운 곳에 있어야 합니다.
*   `rules/` 폴더는 모듈식 지침을 담습니다. 모든 표준을 `CLAUDE.md`에 넣는 대신, 프로젝트는 더 집중된 지침을 `backend.md`, `frontend.md`, `testing.md`, `security.md`와 같은 별도의 파일로 분리합니다. 이는 업데이트를 더 쉽게 하고 주요 지침 파일이 비대해지는 것을 방지합니다.
*   `hooks/` 폴더는 자동화 스크립트(scripts)를 예측 가능한 한 곳에 보관합니다. 이들은 위험한 명령을 차단하거나, 편집된 파일을 포맷팅(formatting)하거나, Claude가 중지하기 전에 검사를 실행하는 등 안전 또는 일관성을 강제하기 위해 자동으로 실행되는 파일들입니다.
*   `commands/` 폴더는 가벼운 재사용 가능한 워크플로(workflows)를 저장합니다. 이는 풀 리퀘스트(pull request) 검토, 테스트 작성, 최근 변경 사항 요약과 같이 자주 반복되지만 여전히 단일 프롬프트(prompt) 파일에 깔끔하게 맞는 작업에 가장 적합합니다.
*   `skills/` 폴더는 더 복잡한 워크플로가 추가적인 구조를 정당화하기 시작하는 곳입니다. 이 예시에서 `release-prep/`는 간단한 지침 이상을 포함할 가능성이 높기 때문에 자체 폴더로 패키지화됩니다. 체크리스트, 템플릿, 그리고 `SKILL.md`에 명확한 워크플로 설명이 필요할 수 있습니다.
*   `agents/` 폴더는 전문화된 역할(roles)을 담습니다. 성숙한 설정에서는 이들이 좁고 목적 지향적이어야 합니다. 팀이 Claude에게 정확성과 엣지 케이스(edge cases)를 정기적으로 검토하도록 요청한다면 `code-reviewer.md` 에이전트(agent)는 의미가 있습니다. 보안 검토가 자체적인 관점을 가진 반복적인 워크플로라면 `security-auditor.md` 에이전트는 의미가 있습니다.

이 청사진(blueprint)을 효율적으로 만드는 것은 가능한 모든 폴더를 포함하기 때문이 아닙니다. 모든 폴더가 존재해야 하는 이유가 있기 때문입니다.

그 구분이 중요합니다. 작은 프로젝트는 아직 `skills/`나 `agents/`가 필요하지 않을 수 있습니다. 매우 간단한 저장소는 다음만 필요할 수 있습니다:

```
your-project/
├── CLAUDE.md
└── .claude/
    ├── settings.json
    ├── rules/
    │   ├── testing.md
    │   └── security.md
    └── hooks/
        └── format-edits.sh
```

그것도 여전히 강력한 구조입니다. 사실, 초기에는 종종 더 나은 구조입니다. 효율성은 모든 폴더를 미리 채우는 것에서 오는 것이 아닙니다. 워크플로가 실제로 필요로 할 때만 구조를 추가하는 것에서 옵니다.

청사진(blueprint)을 생각하는 유용한 방법은 다음과 같은 진행으로 보는 것입니다:

*   `CLAUDE.md`와 `settings.json`으로 시작하기
*   하나의 지침 파일이 더 이상 확장되지 않을 때 `rules/` 추가하기
*   자동화 또는 강제가 필요할 때 `hooks/` 추가하기
*   프롬프트(prompts)가 반복될 때 `commands/` 추가하기
*   워크플로(workflows)가 더 깊어질 때 `skills/` 추가하기
*   전문화가 결과를 명확하게 개선할 때 `agents/` 추가하기

그러한 진행은 폴더가 무작위로 성장하는 것을 막습니다.

프런트엔드(frontend) 앱과 백엔드(backend) API를 가진 제품 팀의 또 다른 예시는 다음과 같습니다:

```
.claude/
├── settings.json
├── rules/
│   ├── ui-components.md
│   ├── api-contracts.md
│   └── testing.md
├── hooks/
│   ├── lint-edits.sh
│   └── block-prod-commands.sh
├── commands/
│   ├── review-ui-changes.md
│   └── investigate-bug.md
└── agents/
    ├── ui-reviewer.md
    └── api-reviewer.md
```

이 버전은 더 좁지만, 동일한 구조적 로직(logic)을 따릅니다. 그것이 청사진(blueprint)의 주요 요점입니다. 정확한 파일은 프로젝트마다 달라지겠지만, 조직 원칙은 안정적으로 유지됩니다.

좋은 `.claude/` 구조는 다음 질문들에 즉시 답해야 합니다:

*   프로젝트 전반의 지침은 어디에 있는가?
*   모듈식 규칙은 어디에 가는가?
*   자동화 스크립트는 어디에 속하는가?
*   재사용 가능한 워크플로는 어디에 가는가?
*   어떤 파일이 공유되고, 어떤 파일이 개인적인가?
*   어떤 부분이 활성화되어 있고, 어떤 부분이 단지 실험인가?

폴더가 이러한 질문들에 명확하게 답한다면, 이미 제 역할을 하고 있는 것입니다.

이 섹션의 주요 요점은 간단합니다. 최고의 `.claude/` 폴더는 가장 기능이 풍부한 폴더가 아닙니다. 구조가 프로젝트와 함께 성장하고, 모든 추가가 설정을 더 어렵게 만드는 대신 더 쉽게 이해할 수 있도록 하는 폴더입니다.

## 8. 피해야 할 일반적인 구조적 실수

`.claude/` 폴더는 너무 단순해서 비효율적이 되는 것이 아닙니다. 파일들이 그 목적을 잃을 때 비효율적이 됩니다.

이는 보통 점진적으로 발생합니다. 팀은 명확한 설정으로 시작하여 몇 가지 빠른 수정 사항, 두어 개의 실험 파일, 일부 개인적인 선호 사항, 그리고 결코 정리되지 않는 소수의 워크플로를 추가합니다. 심각하게 고장 난 것은 없어 보이지만, 구조는 신뢰하기 어려워집니다. 사람들은 새로운 파일을 어디에 넣어야 할지, 어떤 규칙이 여전히 활성화되어 있는지, 또는 스크립트가 실제 팀 표준을 반영하는지 아니면 단순히 누군가의 임시 해결책인지를 알 수 없게 됩니다.

이것이 바로 `.claude/`의 가장 큰 문제가 보통 기술적인 것이 아니라 구조적인 이유입니다.

첫 번째 실수는 `CLAUDE.md`에 너무 많은 내용을 담는 것입니다. 이것은 새로운 지침을 추가하기 가장 쉬운 곳이기 때문에 가장 흔한 문제 중 하나입니다. 시간이 지나면서 `CLAUDE.md`는 프로젝트 개요, 코딩 규칙, 검토 체크리스트, 테스트 노트, 예외 사항, 일회성 알림 등이 뒤섞인 긴 문서로 변모합니다. 그 시점에서는 상위 레벨 가이드로서의 기능을 멈추고 복잡한 저장 파일처럼 행동하기 시작합니다. 더 나은 접근 방식은 `CLAUDE.md`를 광범위한 프로젝트 컨텍스트에 집중시키고 전문화된 지침을 `rules/`로 옮기는 것입니다.

두 번째 실수는 필요하기도 전에 폴더를 추가하는 것입니다. 일부 팀은 `skills/`, `agents/`, 그리고 대규모 명령 컬렉션을 일찍 만드는데, 그 구조가 강력해 보이기 때문입니다. 그러나 비어 있거나 거의 사용되지 않는 폴더는 설정을 더 성숙하게 만들지 않습니다. 오히려 더 혼란스럽게 만드는 경우가 많습니다. 워크플로가 여전히 간단하다면, 간단하게 유지하십시오. 명령 파일만으로도 충분한 경우가 많습니다. 프로젝트에 아직 전문화된 에이전트가 필요하지 않다면, 구조를 채우기 위해 굳이 만들지 마십시오.

또 다른 흔한 문제는 팀 표준과 개인적인 선호 사항을 섞는 것입니다. 이는 일반적으로 로컬 작업 습관이 커밋(commit)된 파일로 새어 들어갈 때 발생합니다. 개발자는 공유 지침에 자신이 선호하는 문구를 추가하고, 자신의 기기에서만 의미 있는 워크플로를 커밋하거나, 명확한 이유 없이 개인 에이전트를 팀 파일로 만듭니다. 일단 그렇게 되면, 저장소는 팀의 공유 워크플로를 반영하는 것을 멈추고 개별 스타일을 반영하기 시작합니다. 이것이 바로 프로젝트 구조와 개인 구조가 분리되어야 하는 이유입니다.

명명(naming)은 구조가 종종 무너지는 또 다른 지점입니다. `misc.md`, `helper.sh`, `test2.md`, 또는 `temp-review.md`와 같은 파일로 가득 찬 폴더는 매우 빠르게 탐색하기 어려워집니다. 이름은 목적을 명확하게 해야 합니다. `review-pr.md`라는 파일은 명확합니다. `block-dangerous-commands.sh`라는 스크립트는 명확합니다. 좋은 명명은 구조 자체가 설명되기 때문에 마찰을 줄입니다.

또한 오래된 파일(stale files) 문제가 있습니다. 이는 `commands/`, `skills/`, `agents/`에서 특히 흔합니다. 팀은 워크플로를 실험하다가 사용을 중단하지만 파일을 그대로 둡니다. 규칙 파일이 더 새로운 파일로 대체되었지만, 오래된 파일은 결코 제거되지 않습니다. 훅(hook)이 실제로 비활성화되었음에도 불구하고 스크립트는 여전히 폴더에 남아 있습니다. 시간이 지남에 따라 `.claude/` 디렉터리는 불필요한 파일들을 축적하기 시작합니다. 이는 설정의 어떤 부분이 실제이고 어떤 부분이 잔여물인지 구분하기 어렵게 만듭니다. 효율적인 `.claude/` 폴더는 코드베이스 자체처럼 주기적인 정리가 필요합니다.

또 다른 실수는 모든 지침을 동등하게 취급하는 것입니다. 그렇지 않습니다. 일부는 `CLAUDE.md`에 속하고, 일부는 `rules/`에 속하며, 일부는 `commands/`에 속하고, 일부는 `.claude/`에 전혀 없어야 합니다. 예를 들어, 어떤 것이 실제로 포맷터(formatter) 규칙, 린터(linter) 규칙, 또는 빌드(build) 규칙이라면, Claude 지침으로 반복되기보다는 실제 도구 구성에 속해야 합니다. 좋은 `.claude/` 구조는 저장소의 나머지 부분을 대체하려고 하기보다 보완할 때 가장 잘 작동합니다.

폴더가 복잡하게 느껴질 때마다 다음과 같은 질문을 하는 것이 유용한 체크리스트입니다:

*   이 파일은 하나의 명확한 목적을 가지고 있는가?
*   이것은 공유 프로젝트 구조에 속하는가, 아니면 개인 구성에 속하는가?
*   이것은 광범위한 지침인가, 모듈식 규칙인가, 자동화 스크립트인가, 아니면 재사용 가능한 워크플로인가?
*   현재 이름이 충분히 구체적인가?
*   이 파일이 여전히 활발하게 사용되는가?

이 질문들에 대한 답이 명확하지 않다면, 구조는 아마도 단순화될 필요가 있습니다.

실수(mistakes)에 대한 간략한 시각적 요약은 다음과 같습니다:

```
.claude/ 구조의 흔한 실수
│
├── 너무 많은 내용이 담긴 CLAUDE.md
├── 너무 이른 시기에 너무 많은 폴더 추가
├── 개인 선호 사항과 혼합된 공유 파일
├── 모호하거나 불분명한 파일 이름
├── 방치된 오래된 워크플로
└── 잘못된 위치에 저장된 지침
```

목표는 가능한 가장 정교한 `.claude/` 폴더를 구축하는 것이 아닙니다. 목표는 모든 파일이 제자리를 갖는 폴더를 구축하는 것입니다.

그것이 바로 이 글의 핵심 요점입니다. 좋은 `.claude/` 구조는 복잡성에 관한 것이 아닙니다. 명확성에 관한 것입니다. 폴더가 잘 정리되어 있으면 Claude를 안내하고 신뢰하며 성장하는 프로젝트 전반에 걸쳐 사용하기가 더 쉬워집니다. 구조가 지저분하면 좋은 지침조차도 그 주변의 시스템이 유지보수하기 어려워지기 때문에 가치를 잃습니다.

최고의 설정은 보통 작게 시작하고, 의도적으로 유지되며, 워크플로가 명확히 요구할 때만 성장합니다.

최고의 `.claude/` 폴더는 가장 많은 파일, 가장 많은 폴더, 또는 가장 진보된 설정을 가진 폴더가 아닙니다. 모든 부분이 명확한 목적을 가진 폴더입니다.

이는 보통 핵심을 단순하게 유지하고, 확장이 필요할 때만 지침을 분할하며, 자동화를 지침과 분리하여 정리하고, 팀 구성과 개인 선호 사항을 명확히 구분하는 것을 의미합니다. 또한 워크플로가 실제로 필요하기 전에 추가 구조를 추가하려는 충동을 억제하는 것을 의미합니다.

`.claude/`가 잘 정리되면 Claude와 함께 작업하는 것이 훨씬 쉬워집니다. 설정은 예측 가능하고, 유지보수 가능하며, 팀이 공유하기 쉬워집니다. 그리고 프로젝트가 성장함에 따라 폴더도 복잡해지지 않고 함께 성장할 수 있습니다.

결국, 최대 효율성은 명확성에서 비롯됩니다. 구조가 깔끔할수록 당신과 Claude 모두 이를 더 잘 사용하기가 쉬워집니다.
