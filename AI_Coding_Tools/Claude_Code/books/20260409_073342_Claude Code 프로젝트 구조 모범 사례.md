Claude Code 프로젝트 구조 모범 사례

Claude Code를 사용하여 작업할 때, 프로젝트를 잘 정리하는 것은 큰 차이를 만듭니다. Claude는 명확한 컨텍스트(context), 일관된 구조, 그리고 명시적인 지침이 있을 때 가장 잘 작동합니다.

다음은 훌륭한 프로젝트 구조를 만들기 위해 따라야 할 5가지 필수 원칙입니다.

1.  CLAUDE.md를 루트(root)에 배치하기

CLAUDE.md는 Claude Code에 프로젝트별 컨텍스트를 제공하는 데 매우 중요한 파일입니다. 이 파일은 AI가 프로젝트 요구 사항과 코드베이스(codebase)를 이해하기 위한 온보딩 가이드 역할을 합니다.

Claude는 시작할 때 CLAUDE.md 파일을 자동으로 읽습니다.

이 파일에 포함해야 할 내용의 예시는 다음과 같습니다:

```markdown
# Project Overview
프로젝트에 대한 간략한 설명.

# Architecture
주요 구성 요소 설명.

# Tech Stack
Next.js
TypeScript
ShadCN UI
Tailwind

# Coding Conventions
- TypeScript 엄격 모드(strict mode) 사용
- 함수형 컴포넌트(functional components) 선호
- 기본 내보내기(default exports) 피하기

# Folder Structure
디렉토리 설명.

# Commands
npm run dev
npm run build

# Important Rules
성능 요구 사항
접근성 요구 사항
테스트 전략
```

이 파일은 프로젝트의 루트 폴더에 배치해야 합니다.

```
my-project/
 ├── CLAUDE.md
 ├── package.json
 ├── src/
 ├── docs/
 └── scripts/
```

간단한 참고: 프로젝트에 기존 코드베이스가 있는 경우, CLAUDE.md를 처음부터 작성할 필요가 없습니다. 대신 Claude Code에서 `/init` 명령을 입력하면, 이 파일의 초안을 생성해 줍니다.

`/init`

2.  대규모 CLAUDE.md 지침을 임포트(import) 파일로 분할하기

CLAUDE.md 파일은 빠르게 커질 수 있습니다. Claude Code가 이 파일을 항상 읽고 컨텍스트에 추가하기 때문에, 파일 크기가 너무 큰 것은 권장되지 않습니다. 일반적인 규칙으로, 파일이 200줄 이상으로 커진다면 몇 개의 파일로 분할해야 합니다.

Claude Code는 CLAUDE.md 내에서 임포트(import)를 지원합니다.

`@path/to/import.md`

Claude가 언제 임포트를 수행해야 하는지에 대한 컨텍스트만 제공하면 됩니다. 임포트가 추가된 업데이트된 CLAUDE.md 파일은 다음과 같습니다:

```markdown
# CLAUDE.md

# Architecture
프로젝트 아키텍처(architecture)는 @claude/architecture.md를 확인하세요.

# Coding Conventions
코딩 컨벤션(coding conventions)은 @claude/coding_conventions.md를 확인하세요.

# UI Guidelines
UI 가이드라인은 @claude/ui_guidelines.md를 확인하세요.
```

프로젝트 디렉토리의 파일 구조는 다음과 같습니다:

```
CLAUDE.md
claude/
  architecture.md
  coding_conventions.md
  ui_guidelines.md
```

이 구조는 다음과 같은 이유로 여러분과 AI 모두에게 유용합니다:

*   유지 보수 용이 (수정 사항이 있을 경우)
*   더 빠른 Claude 컨텍스트 로딩
*   프로젝트 전반에 걸쳐 재사용 가능 (모든 유형의 웹 개발 프로젝트에 적합)

3.  컨텍스트를 위한 `/docs` 폴더 추가하기

Claude가 프로젝트와 미묘한 차이점을 더 잘 이해하도록 돕고 싶다면, `docs/` 폴더를 추가하는 것이 좋습니다. Claude는 마크다운(markdown) 형식의 문서를 매우 잘 읽습니다.

예를 들어, 이번 분기에 제품에 추가할 예정인 향후 기능에 대한 정보나 제품이 제공하는 현재 API 호출에 대한 정보를 추가할 수 있습니다.

```
docs/
  project_roadmap.md
  api.md
```

이제 특정 기능을 구현할 때 Claude에게 문서를 확인하도록 프롬프트(prompt)할 수 있습니다 (예: 웹 서비스의 상태 확인).

`docs/api.md를 읽고 API 호출을 통해 이용 가능한 서비스 상태 모니터를 구현하세요.`

4.  워크플로우(workflows)를 위한 `/workflows` 폴더 추가하기

Claude가 작업을 수행할 때 특정 워크플로우를 따르기를 원한다면, 해당 워크플로우를 전용 디렉토리에 배치해야 합니다. 예를 들어, 웹 개발 프로젝트의 경우 다음과 같을 수 있습니다.

```
claude/
  workflows/
     build-new-component.md
     code-refactoring.md
     write-auto-tests.md
     migrate-db.md
```

다음은 `build-new-component.md` 워크플로우가 어떻게 보이는지에 대한 예시입니다 (참고: 이것은 단지 예시이며, 실제 워크플로우는 더 복잡할 수 있습니다).

```markdown
# build-component.md

웹 서비스용 새 컴포넌트 생성을 요청받으면, 다음 요구 사항에 따라 생성하세요:
- TypeScript
- ShadCN UI
- 접근성(accessible) 고려
- 모바일 우선(mobile-first)
- Tailwind 스타일링
```

다음과 같이 호출할 수 있습니다.

`claude/prompts/build-component.md를 따라 대시보드 카드(dashboard card)를 만드세요.`

이 구조의 큰 장점은 하나의 워크플로우에서 다른 워크플로우를 트리거(trigger)할 수 있다는 것입니다. 예를 들어, Claude가 새 컴포넌트(`build-new-component.md`에 따라)를 코딩하는 경우, `build-new-component.md` 파일에서 바로 이 새 코드를 검증하는 자동 테스트(auto tests) 작성을 트리거할 수 있습니다.

```markdown
# build-component.md

웹 서비스용 새 컴포넌트 생성을 요청받으면, 다음 요구 사항에 따라 생성하세요:
- TypeScript
- ShadCN UI
- 접근성(accessible) 고려
- 모바일 우선(mobile-first)
- Tailwind 스타일링

@workflows/write-auto-tests.md에 따라 이 컴포넌트에 대한 자동 테스트를 반드시 작성하세요.
```

5.  Claude 서비스 작업을 위한 `/tools` 폴더 사용하기

Claude Code의 큰 장점은 워크플로우를 간소화하기 위한 특정 서비스 스크립트를 작성한다는 것입니다. 예를 들어, 데이터베이스 마이그레이션(database migration) 작업(`migrate-db.md`에 명시된)을 정리해 달라고 요청하면, Claude는 일부 작업을 처리할 파이썬 스크립트(Python script)를 작성할 가능성이 높습니다. 모든 서비스 스크립트를 `/tools`라는 하나의 폴더에 보관할 것을 제안합니다.

```
tools/
  migrate-db.py
  seed-data.py
  export-data.py
```

간단한 참고: 왜 폴더 이름이 `/scripts`가 아니라 `/tools`인지 궁금할 수 있습니다. 웹 개발 프로젝트에서 '스크립트'라는 이름은 일반적으로 서비스가 사용하는 프런트엔드 또는 백엔드 스크립트를 연상시키지, 사용하는 서비스 도구(service tools)를 연상시키지 않습니다. 따라서 프로젝트 팀이 폴더를 확인할 때 혼란을 최소화하기 위해 이 명칭을 사용합니다.
