# Claude Code 설정: 프로덕션 개발을 위한 실용 가이드

---

## 서론

모든 AI 코딩 세션은 같은 방식으로 시작됩니다. 프로젝트, 제약 조건, 선호 사항을 설명합니다. 그러면 Claude는 잊어버립니다. 다시 설명합니다. 앱을 충돌시킬 만한 것을 제안합니다. 당신은 그것을 수정합니다. 이 과정이 계속 반복됩니다.

저는 몇 주 동안 Claude Code를 사용하여 Rust + iOS 애플리케이션을 구축했는데, 이 반복되는 문제에 계속 부딪혔습니다. AI 코딩 어시스턴트는 유능했지만 기억력이 없었습니다. 모든 세션이 마치 영화 '사랑의 블랙홀(Groundhog Day)'처럼 매일 반복되었습니다.

그러다 Claude Code에 영구적인 컨텍스트(persistent context)를 위한 기본 기능이 있다는 것을 발견했습니다. 대부분의 개발자들은 이러한 기능의 존재를 알지 못합니다. 저는 이 기능들을 기반으로 시스템을 구축했고, 모든 것이 바뀌었습니다.

이 가이드는 그 시스템을 공유합니다. 이것은 이론적인 것이 아닙니다. 제가 프로덕션 개발을 위해 매일 사용하는 정확한 설정입니다.

## 이 가이드 활용법

이 가이드는 세 부분으로 구성됩니다.

**1부: 개념(The Concept)** — 우리가 무엇을 구축하고 왜 작동하는지 이해합니다. 먼저 이 부분을 읽으세요.
**2부: 빠른 시작(Quick Start)** — 10분 안에 작동하는 설정을 만듭니다. 오늘 바로 사용하고 싶다면 이 부분을 따르세요.
**3부: 심층 분석(Deep Dive)** — 각 구성 요소를 자세히 배웁니다. 설정을 확장할 때 이 부분을 참조하세요.

의심이 든다면 1부를 읽으세요. 확신이 있다면 2부로 건너뛰세요. 이미 Claude Code를 사용하고 있으며 수준을 높이고 싶다면 3부로 바로 이동하세요.

## 1부: 개념

### AI 코딩 어시스턴트의 문제점

AI 코딩 어시스턴트는 상태 비저장(stateless)입니다. iOS 앱이 백그라운드 스레드에서 Metal GPU를 사용하면 충돌한다는 사실을 기억하지 못합니다. 팀이 상속(inheritance)보다 조합(composition)을 선호한다는 것을 알지 못합니다. 지난주에 데이터베이스 스키마(database schema)를 설명하는 데 세 시간을 보냈다는 것을 기억하지 못합니다.

모든 세션에서 당신은 처음부터 시작합니다.

일부 개발자들은 각 대화 시작 시 컨텍스트를 붙여넣는 방식으로 문제를 해결합니다. 다른 개발자들은 "Claude에게 알려줄 것들"이라는 텍스트 파일을 유지합니다. 두 가지 접근 방식 모두 수동적이고 오류 발생 가능성이 높으며 확장성이 떨어집니다.

### 해결책: 구조화된 컨텍스트(Structured Context)

Claude Code는 간단하지만 강력한 기능으로 이 문제를 해결합니다. 모든 세션 시작 시 `CLAUDE.md`라는 파일을 자동으로 읽습니다.

이는 프로젝트 컨텍스트를 한 번 작성하면 Claude가 이를 영원히 알게 된다는 의미입니다.

하지만 제가 몇 달 동안 발견한 핵심 통찰력은 다음과 같습니다. `CLAUDE.md`는 당신의 컨텍스트를 *포함해서는 안 됩니다*. 그것은 당신의 컨텍스트를 *가리켜야 합니다*.

책의 목차처럼 생각해보세요. 목차는 챕터를 포함하지 않습니다. 챕터를 어디서 찾을 수 있는지 알려줍니다. 마찬가지로, `CLAUDE.md`는 작업 중인 내용에 따라 상세한 컨텍스트 파일로 Claude를 안내하는 경량 라우터(router)여야 합니다.

이 접근 방식은 세 가지 이점이 있습니다.

1.  **관련 컨텍스트만, 전부가 아님** — iOS 코드를 작업할 때 데이터베이스 스키마가 컨텍스트 창을 어지럽힐 필요가 없습니다. 라우터 패턴은 필요한 것만 로드합니다.
2.  **시간이 지남에 따라 유지 관리 용이** — 작고 집중된 파일은 하나의 거대한 문서보다 업데이트하기 쉽습니다.
3.  **프로젝트 간 재사용 가능** — iOS 제약 조건 파일은 어떤 iOS 프로젝트에도 복사할 수 있습니다.

### 아키텍처(The Architecture)

제가 사용하는 시스템은 다섯 가지 구성 요소로 이루어져 있습니다.

![다섯 가지 구성 요소 아키텍처 다이어그램](image_placeholder_for_architecture)
_엔터 키를 누르거나 클릭하여 전체 크기로 이미지를 확인하세요._

`CLAUDE.md`는 당신의 진입점(entry point)입니다. 자동으로 로드되어 Claude에게 다른 모든 것을 어디서 찾을 수 있는지 알려줍니다.

**메모리(Memory)**는 시간이 지남에 따라 변경되는 컨텍스트를 담습니다. 무엇을 구축하는지, 왜 구축하는지, 프로젝트의 어느 단계에 있는지, 어떤 결정을 내렸는지 등입니다.

**규칙(Rules)**은 위반 시 충돌, 보안 취약점 또는 운영 중단을 초래하는 것과 같은 엄격한 제약 조건을 인코딩합니다. 이것들은 협상 불가능합니다.

**페르소나(Personas)**는 전문가 역할을 정의합니다. Claude가 일반론자가 아닌, 메모리 효율성을 우선시하는 iOS 전문가나 스키마(schemas) 및 인덱스(indexes)를 생각하는 데이터베이스 설계자(database architect)가 될 수 있도록 합니다.

**명령(Commands)**은 다단계 워크플로(multi-step workflows)를 자동화합니다. 다섯 개의 Git 명령을 입력하는 대신 `/commit`을 입력하면 Claude가 나머지를 처리합니다.

### 왜 이 시스템이 작동하는가

이 시스템의 힘은 개별 구성 요소에 있는 것이 아니라, 구성 요소들이 함께 작동하는 방식에 있습니다.

Swift 파일을 편집할 때, Claude는 자동으로 iOS 제약 조건을 로드할 수 있습니다. 데이터베이스 변경에 대해 물어보면, 데이터베이스 설계자 페르소나를 채택할 수 있습니다. 커밋(commit)할 준비가 되면, 하나의 명령으로 스테이징(staging), 커밋, 푸시(push), 풀 리퀘스트(pull request) 생성을 처리합니다.

그 결과: Claude는 모든 문서를 읽고, 모든 결정을 기억하며, 모든 규칙을 따르는 팀원처럼 행동합니다. 실제로 기억하기 때문이 아니라, 관련 컨텍스트가 자동으로 로드되기 때문입니다.

## 2부: 빠른 시작

10분 안에 작동하는 설정을 만들 수 있습니다. 방법은 다음과 같습니다.

### 1단계: 폴더 구조 생성

프로젝트 루트에서 터미널(terminal)을 열고 다음을 실행하세요.

```bash
mkdir -p .claude/memory .claude/rules .claude/personas .claude/commands
```

이렇게 하면 필요한 폴더가 생성됩니다. 이것이 전체 구조입니다.

### 2단계: 진입점 생성

`.claude/CLAUDE.md` 파일을 생성합니다. 이 파일은 Claude가 자동으로 로드하는 유일한 파일이므로, 이것이 시작점입니다.

다음은 최소 버전입니다.

```markdown
# 프로젝트: [당신의 프로젝트 이름]
## 이것은 무엇인가
[당신의 프로젝트를 설명하는 한 단락]
## 빠른 명령
- 빌드: `[당신의 빌드 명령]`
- 테스트: `[당신의 테스트 명령]`
- 실행: `[당신의 실행 명령]`
## 컨텍스트 파일
이 프로젝트를 작업할 때, 자세한 컨텍스트는 다음 파일을 참조하세요:
- 제품 및 비즈니스 컨텍스트: [productContext.md](memory/productContext.md)
- 기술 아키텍처: [systemPatterns.md](memory/systemPatterns.md)
- 현재 상태: [activeContext.md](memory/activeContext.md)
```

대괄호로 묶인 섹션을 프로젝트 세부 정보로 대체하세요. 짧게 유지하세요. 이 파일은 30초 안에 스캔할 수 있어야 합니다.

### 3단계: 비즈니스 컨텍스트 추가

`.claude/memory/productContext.md` 파일을 생성합니다. 이곳에 무엇을 만들고 왜 만드는지 설명합니다.

새 팀원을 온보딩(onboarding)하듯이 작성하세요. 다음 내용을 포함합니다.

-   **해결하는 문제** — 어떤 문제점을 해결합니까?
-   **누구를 위한 것인지** — 사용자(user)는 누구입니까?
-   **작동 방식** — 제품에 대한 고수준 설명
-   **성공의 모습** — 작동하는지 어떻게 측정합니까?

길 필요는 없습니다. 몇 단락이면 충분합니다. 목표는 Claude가 합리적인 제안을 하는 데 필요한 비즈니스 컨텍스트를 제공하는 것입니다.

예를 들어, 바쁜 직장인을 위한 모바일 앱을 구축하고 있다면 Claude는 그것을 알아야 합니다. 연구원을 위한 데이터 분석 도구를 구축하는 경우와는 다른 권장 사항을 제시할 것입니다.

### 4단계: 현재 상태 추가

`.claude/memory/activeContext.md` 파일을 생성합니다. 이것은 "현재 우리가 어디에 있는지"를 나타내는 파일입니다.

다음 내용을 포함합니다.

-   **현재 집중하는 업무** — 이번 주에 무엇을 작업하고 있습니까?
-   **최근 결정** — 무엇을 결정했고 왜 결정했습니까?
-   **진행 방해 요인(Blockers)** — 무엇이 진행을 방해하고 있습니까?
-   **다음 단계** — 현재 작업 후에 무엇이 옵니까?

다른 파일과 달리 이 파일은 자주 변경됩니다. 주요 작업을 완료하거나, 중요한 결정을 내리거나, 새로운 작업 흐름을 시작할 때 업데이트하세요.

**전문가 팁(Pro tip):** 이 파일을 Git에 커밋(commit)하세요. 내일 새로운 Claude 세션을 시작할 때, Claude는 당신이 어디에서 멈췄는지 정확히 알 것입니다.

### 끝입니다!

이제 작동하는 Claude Code 설정이 완료되었습니다. 프로젝트에서 새 Claude Code 세션을 시작하면 `CLAUDE.md`가 자동으로 로드되고, 이 파일은 당신의 컨텍스트 파일을 가리킵니다.

다음번에 Claude에게 프로젝트에 대해 물어볼 때, Claude는 컨텍스트를 갖게 될 것입니다. 지난 대화를 기억해서가 아니라, 당신의 컨텍스트가 영구적인 파일에 인코딩되어 있기 때문입니다.

3부의 모든 내용은 이 기초 위에 구축됩니다. 필요하다고 느낄 때 구성 요소를 추가하세요. 모든 것을 한 번에 구현할 필요는 없습니다.

## 3부: 심층 분석

### 3.1 진입점: CLAUDE.md

`CLAUDE.md`에서 가장 흔한 실수는 너무 많은 것을 넣는 것입니다.

저는 개발자들이 전체 데이터베이스 스키마, 모든 코딩 표준, 모든 API 엔드포인트, 전체 제품 로드맵을 포함하는 500줄짜리 파일을 만드는 것을 보았습니다. 이는 본래의 목적에 어긋납니다. Claude의 컨텍스트 창은 즉시 가득 차고, 대부분의 컨텍스트는 현재 작업과 관련이 없습니다.

`CLAUDE.md`는 저장소(repository)가 아니라 라우터(router)여야 합니다.

좋은 `CLAUDE.md`는 세 가지 섹션으로 구성됩니다.

**개요(Overview)** — 이 프로젝트는 무엇이며, 한두 문장으로 설명합니다. 이는 Claude의 방향을 즉시 잡게 합니다.

**빠른 참조(Quick reference)** — 빌드 명령, 테스트 명령, 일반적인 작업. 포스트잇에 적어둘 만한 것들입니다.

**포인터(Pointers)** — 필요할 때를 기준으로 정리된 상세 컨텍스트 파일 링크. 프론트엔드(frontend)를 작업 중이라면, UI 패턴을 찾을 수 있는 곳입니다. 데이터베이스를 건드린다면, 스키마 참조가 여기 있습니다.

전체 파일을 50줄 미만으로 유지하세요. 만약 더 길다면, 별도의 파일에 속해야 할 내용을 넣고 있는 것입니다.

### 3.2. 메모리 파일: 진화하는 컨텍스트(Context That Evolves)

`memory/` 폴더는 시간이 지남에 따라 변경되는 프로젝트에 대한 컨텍스트를 담고 있습니다.

저는 세 가지 파일을 사용합니다.

`productContext.md`는 "무엇을 만들고 있으며 왜 만드는가?"에 답합니다. 이것은 비즈니스 컨텍스트입니다. 문제, 사용자, 목표입니다. 제품 방향이 변경될 때 변경되며, 이는 드물어야 합니다.

```markdown
# 제품 컨텍스트

## 비전
이것이 어떤 문제를 해결하는가? 누구를 위해?

## 현재 목표
이번 분기에 무엇을 구축하고 있는가?

## 사용자 페르소나
누가 이것을 사용하며, 어떻게 사용하는가?
```

`systemPatterns.md`는 "여기서 어떻게 구축하는가?"에 답합니다. 이것은 기술 컨텍스트입니다. 아키텍처, 패턴, 규칙입니다. 새로운 패턴을 채택하거나 아키텍처 결정을 내릴 때 변경됩니다.

```markdown
# 시스템 패턴

## 아키텍처
- 프론트엔드: React + TypeScript
- 백엔드: Rust + Tauri
- 데이터베이스: 벡터 검색(vector search)을 사용하는 SQLite

## 핵심 패턴
- 데이터 액세스에 리포지토리 패턴(repository pattern) 사용
- 모든 비동기(async) 작업은 Result 타입을 사용
- UI 컴포넌트(component)는 아토믹 디자인(atomic design)을 따름
```

`activeContext.md`는 "지금 우리는 어디에 있는가?"에 답합니다. 이것은 세션 컨텍스트입니다. 현재 작업, 최근 결정, 방해 요인, 다음 단계입니다. 자주 변경되며, 때로는 하루에도 여러 번 변경됩니다.

```markdown
# 활성 컨텍스트

**현재 작업**: 사용자 인증(user authentication) 구현 중
**진행 방해 요인**: OAuth 자격 증명(credentials) 대기 중
**최종 업데이트**: 2025-01-19

## 최근 결정
- 모바일 지원을 위해 세션 토큰(session tokens) 대신 JWT(JSON Web Token) 선택
- 비밀번호 해싱(password hashing)에 bcrypt 사용

## 다음 단계
1. 로그인 엔드포인트(endpoint) 구현
2. 토큰 갱신(refresh) 로직 추가
3. 인증 미들웨어(middleware) 생성
```

핵심 통찰력은 안정적인 컨텍스트와 변동성이 큰 컨텍스트를 분리하는 것입니다. 제품 비전은 자주 변하지 않습니다. 현재 작업은 끊임없이 변합니다. 다른 파일에 두면, 다른 파일에 영향을 주지 않고 하나를 업데이트할 수 있습니다.

### 3.3. 규칙: 협상 불가능한 제약 조건

규칙은 컨텍스트와 다릅니다. 컨텍스트는 Claude가 좋은 제안을 하는 데 도움이 됩니다. 규칙은 Claude가 위험한 제안을 하는 것을 방지합니다.

규칙은 세 가지 구성 요소로 이루어져야 합니다.

**제약 조건(The constraint)** — 명확하고 모호하지 않게 진술합니다. "iOS 공유 확장(Share Extensions)에서는 절대 Metal GPU를 사용하지 마세요."

**결과(The consequence)** — 이 규칙이 위반되면 어떤 일이 발생합니까? "iOS가 복구 불가능한 상태로 앱을 즉시 충돌시킬 것입니다."

**올바른 접근 방식(The correct approach)** — 대신 무엇을 해야 합니까? "신경 엔진(Neural Engine)에서 실행되고 백그라운드 컨텍스트에서 안전한 Vision 프레임워크를 사용하세요."

결과가 가장 중요한 부분입니다. Claude는 규칙이 존재하는 이유를 이해할 때 규칙을 더 잘 따를 가능성이 높습니다. "X를 하면 충돌할 것이므로 X를 하지 마세요"가 "X를 하지 마세요"보다 더 설득력이 있습니다.

규칙을 도메인별로 정리하세요. 저는 iOS 플랫폼 제약 조건, Rust 언어 패턴, 데이터베이스 규칙에 대한 별도의 파일을 가지고 있습니다. iOS 코드를 작업할 때는 Rust 규칙을 로드할 필요가 없으며, 그 반대도 마찬가지입니다.

```markdown
# [플랫폼/도메인] 제약 조건

## 규칙 1: [짧은 이름]

### 규칙
한 문장으로 명확하게 진술합니다.

### 이유
무엇이 망가지나요? 결과는 무엇인가요?

### 잘못된 방법 ❌
```code
// 문제를 일으키는 패턴
```

### 올바른 방법 ✅
```code
// 작동하는 패턴
```

### 3.4. 페르소나: 전문가 역할

페르소나는 Claude를 일반론자(generalist)에서 도메인 전문가(domain expert)로 변모시킵니다.

일반론자 AI는 정확하지만 일반적인 조언을 줄 것입니다. iOS 전문가 페르소나는 메모리 제약 조건, 앱 라이프사이클(app lifecycle), 플랫폼 규칙을 고려한 조언을 줄 것입니다. 실제 iOS 개발자가 물어볼 만한 질문을 할 것입니다.

페르소나 파일은 다음을 정의해야 합니다.

**전문 지식(Expertise)** — 이 전문가는 무엇을 깊이 알고 있습니까? 어떤 프레임워크, 패턴, 도구에 능숙합니까?

**우선순위(Priorities)** — 이 전문가는 무엇을 최적화합니까? iOS 전문가는 메모리 효율성과 사용자 경험을 우선시할 수 있습니다. 데이터베이스 설계자는 쿼리 성능과 데이터 무결성을 우선시할 수 있습니다.

**의사 결정 프레임워크(Decision framework)** — 이 전문가는 문제에 어떻게 접근합니까? 어떤 질문을 합니까? 어떤 절충안(tradeoffs)을 고려합니까?

```markdown
# [역할] 페르소나

## 전문 지식
이 전문가는 무엇을 깊이 알고 있는가?

## 우선순위
무엇을 최적화하는가?

## 의사 결정 프레임워크
문제에 어떻게 접근하는가?

## 일반적인 작업
주로 어떤 도움을 주는가?
```

`CLAUDE.md`에서 참조하거나("iOS 코드를 작업할 때는 iOS 전문가처럼 행동하라"), 대화에서 명시적으로 ("데이터베이스 설계자처럼 행동하고 이 스키마를 검토하라") 페르소나를 호출합니다.

페르소나는 선택 사항이지만 강력합니다. 페르소나 없이 시작하고, 동일한 도메인별 고려 사항을 반복적으로 설명하고 있다는 것을 알게 되면 추가하세요.

### 3.5. 명령: 자동화된 워크플로

명령(Commands)을 사용하면 다단계 작업을 자동화할 수 있습니다.

`.claude/commands/`에 워크플로를 설명하는 Markdown 파일을 생성합니다. 명령 이름(예: `/commit`)을 입력하면 Claude가 단계를 실행합니다.

예를 들어, 커밋 워크플로는 다음을 수행할 수 있습니다.

1.  `git status`를 실행하여 변경 사항 확인
2.  `git diff`를 실행하여 변경 사항 검토
3.  변경 사항 분석 및 커밋 메시지 초안 작성
4.  관련 파일 스테이징(stage)
5.  커밋 생성
6.  원격 저장소(remote)로 푸시(push)
7.  요청 시 풀 리퀘스트(pull request) 생성

이러한 명령을 수동으로 입력하는 대신 `/commit`을 입력하면 Claude가 처리합니다.

명령에 적합한 후보는 다음과 같은 작업입니다.

-   여러 단계가 있는 경우
-   자주 수행하는 경우
-   일관된 패턴을 가진 경우
-   수동으로 수행할 때 오류가 발생하기 쉬운 경우

저는 Git 워크플로, 테스트 실행, 빌드 프로세스에 명령을 사용합니다. 각 명령은 무언가를 수행하는 "올바른 방법"을 인코딩하므로 세부 사항을 기억할 필요가 없습니다.

### 3.6. 고급: 자동 컨텍스트 주입(Automatic Context Injection)

지금까지는 Claude에게 파일을 읽으라고 지시해야 했습니다. 하지만 Claude Code는 더 강력한 것을 지원합니다. 자동으로 실행되는 훅(hooks)입니다.

`Pre-tool` 훅(pre-tool hooks)은 Claude가 도구를 사용하기 전에 실행됩니다. 이를 사용하여 Claude가 하려는 작업에 따라 컨텍스트를 주입할 수 있습니다.

예를 들어: Claude가 Swift 파일을 편집하려고 할 때 자동으로 iOS 제약 조건을 로드합니다. 데이터베이스 마이그레이션(migration)을 수정하려고 할 때 스키마 참조를 로드합니다.

이것은 어떤 파일이 편집되고 있는지 확인하고 관련 컨텍스트를 출력하는 작은 셸 스크립트(shell script)가 필요합니다. 설정이 더 복잡하므로 먼저 기본 시스템에 익숙해지는 것을 권장합니다. 동일한 유형의 파일에 대해 동일한 컨텍스트를 반복적으로 로드하고 있다는 것을 알게 되면 훅이 가치가 있습니다.

`.claude/settings.json`을 생성합니다.

```json
{
  "hooks": {
    "preToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": ".claude/hooks/pre-tool-use.sh \"$TOOL_NAME\" \"$TOOL_INPUT\""
          }
        ]
      }
    ]
  }
}
```

## 마지막 생각

이 설정 뒤에 있는 통찰력은 간단합니다. 구조가 기억력보다 낫다는 것입니다.

AI 어시스턴트는 영구적인 메모리를 가지고 있지 않지만, 파일을 읽을 수 있습니다. 컨텍스트를 자동으로 로드되는 파일에 인코딩함으로써, AI가 실제로 기억하는 것에 의존하지 않고도 메모리의 이점을 얻을 수 있습니다.

이것은 Claude를 더 똑똑하게 만드는 것이 아닙니다. 특정 상황에서 Claude가 똑똑해지는 데 필요한 컨텍스트를 제공하는 것입니다.

Anthropic 팀은 자동 로딩 파일, 사용자 지정 명령, 도구 훅(tool hooks)과 같은 올바른 기본 기능(primitives)으로 Claude Code를 구축했습니다. 이 가이드는 그것들을 사용하는 한 가지 방법을 보여줍니다. 다른 방법들도 분명 있을 것입니다.

이 접근 방식으로 흥미로운 것을 구축한다면, 소식을 듣고 싶습니다.
