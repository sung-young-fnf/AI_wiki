이미지를 전체 크기로 보려면 Enter를 누르거나 클릭하세요.
아이디어부터 구현까지 — AI 에이전트에게 뛰어난 엔지니어링 팀이 따르는 것과 동일한 구조화된 프로세스를 제공하는 5단계 스펙 주도 워크플로우(Made with Google Gemini).
엔지니어링 규율을 잃지 않으면서 AI로 더 빠르게 출시하는 실용적인 방법

예상 독서 시간: 8분

AI는 코드를 몇 초 만에 작성할 수 있습니다.

저는 이 점이 좋습니다. 여러분도 그럴 것입니다.

하지만 실제 프로젝트에서 AI 코딩 에이전트(AI coding agents)를 사용해 보셨다면 불편한 점을 발견했을 것입니다. 바로 결과물의 품질이 엄청나게 일관되지 않다는 것입니다. 어떤 세션(session)에서는 깔끔하고 잘 설계된 코드를 생성합니다. 하지만 다음 세션에서는 — 동일한 도구, 동일한 모델임에도 불구하고 — 처음부터 다시 작성하는 것보다 고치는 데 더 많은 시간이 걸리는 뒤죽박죽 코드를 생성합니다.

문제는 AI가 아닙니다. 문제입니다 — 아니, 오히려 프로세스의 부재가 문제입니다.

저는 이 문제를 해결하는 패턴을 발견했습니다. 바로 Claude Skills를 활용한 스펙 주도 개발(spec-driven development)입니다.

하나의 거대한 프롬프트(prompt)와 기원에 의존하는 대신, 아이디어부터 코드 출시까지 구조화된 워크플로우(workflow)를 실행합니다. 각 단계는 다음 단계에 공급될 스펙(spec)을 생성합니다.

가장 좋은 점은 이미 Claude Code 플러그인(plugin)으로 패키징되어 오늘 바로 설치할 수 있다는 것입니다.

## 진정한 문제: 에이전트도 프로세스가 필요하다

여기 AI 과장 광고(hype)가 간과하는 진실이 있습니다. 구조화된 프로세스는 인간만의 것이 아니라 AI 에이전트에게도 마찬가지로 필요합니다.

시니어 엔지니어(senior engineer)가 어떻게 일하는지 생각해 보세요. 그들은 모호한 아이디어에서 곧바로 코드를 작성하지 않습니다. 문제 공간(problem space)에 대해 생각합니다. 아키텍처(architecture)를 스케치합니다. 작업을 관리 가능한 조각으로 나눕니다. 의존성에 따라 작업을 순서화합니다. 그리고 나서 구축합니다.

우리는 이러한 규율을 인간에게 기대합니다. 왜냐하면 그것이 없을 때 무슨 일이 일어나는지 알기 때문입니다. 범위 확장(scope creep), 재작업(rework), 아키텍처 드리프트(architectural drift), 그리고 해결해야 할 문제를 제대로 해결하지 못하는 기능들 말입니다.

그렇다면 AI와 함께 작업할 때는 왜 이 모든 것을 내던질까요?

AI 에이전트에게 '알림 시스템을 구축해 줘'와 같은 단일 프롬프트(prompt)를 제공할 때, 당신은 에이전트에게 제품 결정, 기술 결정, 구현 결정을 한꺼번에 내리도록 요구하는 것입니다.

이는 신입 사원에게 '여기 코드베이스가 있습니다. 이 기능을 구축하고, 모든 것을 스스로 알아내며, 질문하지 마세요'라고 말하는 것과 같습니다.

어떤 누구도 그런 식으로 팀을 관리하지 않을 것입니다. 하지만 대부분의 사람들이 AI 코딩 에이전트를 그렇게 사용합니다.

결과는 예측 가능합니다.

*   **범위가 모호합니다(Scope is fuzzy)** — 에이전트는 당신이 원하는 것을 추측하며, 종종 틀립니다.
*   **아키텍처는 우발적입니다(Architecture is accidental)** — 결정은 설계 과정이 아닌 구현 중에 암묵적으로 이루어집니다.
*   **작업이 뒤얽힙니다(Tasks are entangled)** — 의존성 순서가 없으므로 변경 사항이 서로 충돌합니다.
*   **재작업이 가중됩니다(Rework compounds)** — AI가 생성한 하나의 결정을 수정하면 다섯 가지 다른 결정을 수정해야 하는 연쇄 반응이 발생합니다.

해결책은 '더 나은' AI를 사용하는 것이 아닙니다. 해결책은 인간 팀에게 제공할 것과 동일한 구조화된 프로세스를 에이전트에게 제공하는 것입니다.

그것이 바로 스펙 주도 개발(spec-driven development)이 하는 일입니다.

## 작동 방식: 5단계, 각 단계는 이전 단계 위에 구축

이 플러그인(plugin)은 순차적인 워크플로우(workflow)를 구성하는 다섯 가지 Claude Skills를 제공합니다.

`/ideation` → `/tech-spec` → `/sprint-planning` → `/task-breakdown` → `/implement`

각 단계는 이전 단계에서 생성된 스펙(spec)을 읽으므로, 컨텍스트(context)가 손실되지 않고 누적됩니다. 각 단계의 역할과 중요성은 다음과 같습니다.

### 1. 아이디어 도출(Ideation) — 무엇을 만들지 결정

`/ideation` 스킬(skill)은 당신의 브레인스토밍(brainstorming) 파트너입니다. 문제나 기능 아이디어를 제공하면, 3~5가지의 완전한 솔루션 접근 방식을 생성합니다 — 개별 기능 목록이 아니라, 나란히 비교할 수 있는 완전한 제품 개념입니다.

각 접근 방식에 대해 다음을 얻을 수 있습니다.

*   명확한 개념 설명
*   주요 기능(Key capabilities)
*   다른 접근 방식과의 차별점
*   상대적인 노력 추정치(Relative effort estimate)

논의하고, 범위를 좁히고, 선택합니다. 그런 다음 스킬은 결과를 저장합니다.

**출력:** `specs/PROJECT.md` 및 `specs/ROADMAP.md`

**중요한 이유:** 첫 번째 '충분히 좋은' 아이디어에 갇히지 않습니다. 커밋(commit)하기 전에 문제 공간(problem space)을 신중하게 탐색합니다.

### 2. 기술 스펙(Tech Spec) — 어떻게 만들지 결정

`/tech-spec` 스킬은 승인된 프로젝트 스펙과 로드맵(roadmap)을 받아 제품 의도(product intent)를 엔지니어링 결정으로 전환합니다.

*   스택(Stack) 및 근거 (언어, 프레임워크(framework), 데이터베이스(database) — 그 이유와 함께)
*   아키텍처 개요 및 데이터 흐름(data flow)
*   핵심 데이터 모델(data models) 및 관계
*   API 표면(surface)
*   1회차 주기(cycle 1)를 위한 프로젝트 설정 체크리스트(checklist)

**출력:** `specs/TECH_SPEC.md`

**중요한 이유:** 아키텍처가 우발적이 아닌 의도적으로 됩니다. 필요한 것이 Postgres인데 에이전트가 SQLite를 조용히 선택하거나, 관계형(relational)이어야 할 데이터 모델(data model)을 평탄화(flatten)하지 않습니다.

### 3. 스프린트 계획(Sprint Planning) — 한 주기 범위 지정

`/sprint-planning` 스킬은 로드맵 단계를 다음을 포함하는 사용자 스토리(user stories)로 변환합니다.

*   명확한 승인 기준(acceptance criteria) (모호하지 않고 테스트 가능함)
*   MoSCoW 우선순위 (필수 / 권장 / 있으면 좋음)
*   스토리(stories) 간 의존성

**출력:** `specs/cycles/CYCLE_01.md`

**중요한 이유:** 팀과 AI는 지금 무엇이 범위에 포함되고 무엇이 연기되었는지 정확히 압니다. 스프린트(sprint) 중간에 기능 범위가 확장되지 않습니다.

### 4. 작업 세분화(Task Breakdown) — 작업을 실행 가능하게 만들기

`/task-breakdown` 스킬은 사용자 스토리(user stories)를 실용적인 개발자 작업으로 변환합니다.

*   고유 ID 및 한 줄 제목
*   구체적인 설명 (무엇을 어디에 구현할지)
*   작업별 승인 확인
*   의존성을 고려한 구현 순서

**출력:** `specs/cycles/CYCLE_01_TASKS.md`

**중요한 이유:** 모호성이 줄어들고, 블로커(blocker)가 적으며, 실행이 더 원활해집니다. 모든 작업은 한 세션(session)에 완료할 수 있을 만큼 충분히 작고, 개발자 — 또는 AI —가 무엇을 해야 할지 정확히 알 만큼 구체적입니다.

### 5. 구현(Implement) — 제어하에 실행

`/implement` 스킬은 의존성 순서에 따라 작업 목록을 처리하며 두 가지 모드를 지원합니다.

*   **검토 모드(Review mode)** — 각 작업 후에 일시 중지하여 검사하고 조정할 수 있습니다.
*   **연속 모드(Continuous mode)** — 차단될 때만 일시 중지하고 주기를 진행합니다.

한 줄의 코드를 작성하기 전에 모든 스펙 파일(spec file)을 읽습니다. 완료된 작업을 체크하고, 편차를 문서화합니다.

**중요한 이유:** 진행 상황을 시각적으로 확인할 수 있으며, AI는 당신이 설정한 규칙을 벗어나지 않습니다.

## 스펙이 누락된 계층인 이유

대부분의 사람들은 AI 생산성을 생성 속도로 생각합니다. 얼마나 빨리 코드를 작성할 수 있는가? 분당 몇 줄인가?

그것은 잘못된 측정 기준입니다.

올바른 측정 기준은 재작업률(rework rate)입니다. AI가 생성한 것 중 실제로 얼마나 많이 유지합니까? 수정, 방향 전환 또는 폐기하는 데 얼마나 많은 시간을 보냅니까?

스펙(spec)은 에이전트가 코드를 만지기 전에 문제를 제한하기 때문에 재작업을 줄입니다. 각 스펙 파일은 당신과 AI가 무엇을, 왜, 어떻게 구축할지에 대해 정렬하는 체크포인트(checkpoint)입니다.

이것을 다음과 같이 생각해 보세요.

*   **스펙이 없을 때:** AI가 당신의 의도를 추측합니다. 아키텍처는 구현에서 나옵니다. 범위는 AI가 결정하는 모든 것입니다. 코드가 작성된 후에 재작업이 발생합니다.
*   **스펙이 있을 때:** AI는 당신이 문서화한 의도를 읽습니다. 아키텍처는 구현 전에 정의됩니다. 범위는 스프린트 계획에서 합의한 것입니다. 불일치는 코드가 존재하기 전에 포착됩니다.

스펙 파일은 또한 AI 어시스턴트(assistant)의 가장 고질적인 문제 중 하나인 컨텍스트 손실(context loss)을 해결합니다. 긴 코딩 세션(coding session)에서 에이전트의 초기 결정은 컨텍스트에서 사라집니다. 스펙 파일을 사용하면 프로젝트의 결정은 항상 사용 가능합니다. 채팅 기록처럼 사라지는 것이 아니라 디스크(disk)에 있는 문서입니다.

## 간단한 실제 예시

팀 알림을 구축하고 싶다고 가정해 봅시다.

이 워크플로우가 없다면, 종종 다음을 얻게 됩니다.

*   부분적인 기능 커버리지(feature coverage)
*   불분명한 재시도 동작(retry behavior)
*   엉망인 환경 설정 처리(preference handling)
*   일관성 없는 아키텍처

스펙 주도 플로우(flow)를 사용하면:

*   **아이디어 도출(Ideation)** — 앱 내 알림, 다채널 전달 또는 다이제스트(digest) 모델 중 선택합니다. 장단점을 비교합니다. 하나를 선택합니다.
*   **기술 스펙(Tech Spec)** — 이벤트 유형, 큐잉 전략(queueing strategy), 재시도 로직(retry logic), 사용자 설정 모델을 정의합니다. 신중하게 스택(stack)을 선택합니다.
*   **스프린트 계획(Sprint Planning)** — 첫 번째 주기(cycle)의 범위를 정합니다: 핵심 이벤트 파이프라인(pipeline) + 앱 내 전달. 이메일과 다이제스트는 2회차 주기로 연기합니다.
*   **작업 세분화(Task Breakdown)** — 구체적인 작업을 생성합니다: 이벤트 스키마(schema) 설정, 큐 소비자(queue consumer) 구축, 알림 API(API) 생성, 프론트엔드(frontend) 연결.
*   **구현(Implement)** — 모든 단계에서 승인 확인을 통해 의존성 순서대로 출시합니다.

동일한 기능. 훨씬 더 나은 프로세스. 극적으로 적은 재작업.

## 더 깊은 통찰: 프로세스는 프롬프팅의 한 형태이다

여기 대부분의 사람들이 AI 에이전트와 작업할 때 놓치는 점이 있습니다.

구조화된 프로세스는 워크플로우(workflow) 수준에서의 프롬프트 엔지니어링(prompt engineering)입니다.

프로젝트를 아이디어 도출 → 스펙 → 계획 → 작업 → 구현으로 세분화할 때, 당신은 단지 자신의 사고를 조직하는 것만이 아닙니다. 각 단계에서 AI가 받는 컨텍스트(context)를 형성하고 있는 것입니다. AI가 더 나은 선택을 하도록 의사결정 공간을 제한하고 있는 것입니다.

단일의 거대한 프롬프트는 AI에게 모든 것을 동시에 처리하도록 요구합니다. 구조화된 워크플로우는 AI가 한 번에 한 가지 유형의 결정에 집중할 수 있도록 합니다. 아이디어 도출 중에는 제품 결정, 기술 스펙 중에는 아키텍처 결정, 계획 중에는 범위 지정 결정, 코딩 중에는 구현 결정에 집중하게 합니다.

이것은 사슬 사고 추론(chain-of-thought reasoning)의 기본 원리와 동일합니다. 단지 프롬프트 수준이 아닌 프로젝트 수준에 적용된 것일 뿐입니다.

그리고 이것은 확장 가능합니다(scales). 주말 프로젝트를 구축하든, 여러 스프린트(sprint)에 걸친 이니셔티브(initiative)를 통해 팀을 이끌든, 동일한 다섯 단계가 적용됩니다. 깊이만 달라질 뿐입니다. 주말 프로젝트는 아이디어 도출에 10분을 할애하고 스프린트 계획을 건너뛸 수도 있지만, 순서는 유지됩니다.

## 시간이 지남에 따라 얻는 것

이 워크플로우를 일관되게 실행하면 누적되는 이점을 얻을 수 있습니다.

*   **더 명확한 범위 결정(scope decisions)** — 아무도 요청하지 않은 것을 구축하는 것을 멈춥니다.
*   **더 강력한 기술 일관성(technical consistency)** — 아키텍처 결정이 문서화되고 재사용됩니다.
*   **재작업 감소** — 불일치는 PR 검토(PR review) 단계가 아닌 스펙 단계에서 포착됩니다.
*   **더 쉬운 온보딩(onboarding)** — 새로운 팀원(또는 새로운 AI 세션)은 스펙을 읽고 전체 컨텍스트를 얻을 수 있습니다.
*   **영구적인 프로젝트 기억(Persistent project memory)** — 당신의 결정은 사라지는 채팅 기록이 아닌 문서에 살아있습니다.

마지막 요점은 과소평가(underrated)되어 있습니다. 6개월 후 누군가 '왜 알림에 이벤트 소싱(event sourcing)을 선택했나요?'라고 물으면 — 답은 `TECH_SPEC.md`에 있으며, 찾을 수 없는 슬랙(Slack) 스레드(thread)에 있지 않습니다.

## 이 워크플로우가 방지하는 일반적인 실수

*   아이디어에서 코드로 바로 넘어가는 것 — 나중에 비싼 비용을 치르게 될 변경(pivot)을 방지하는 사고 과정을 건너뛰는 것.
*   테스트 가능한 승인 기준(acceptance criteria) 없이 스토리(stories)를 작성하는 것 — '이것이 완료되었는가?' 논쟁으로 이어지는 것.
*   의존성 순서 없이 작업을 태스크(task)로 나누는 것 — 블로커(blocker)와 낭비되는 노력을 생성하는 것.
*   구현 중간에 아키텍처 결정을 내리는 것 — 잘못된 선택을 했다는 것을 발견하는 가장 비싼 단계.
*   AI 세션(session) 간 컨텍스트(context)를 잃는 것 — 문서화된 결정에 기반하여 구축하는 대신 매번 처음부터 시작하는 것.

## 마지막 생각

저는 AI가 프로세스(process)를 대체해야 한다고 생각하지 않습니다.

저는 AI가 좋은 프로세스를 더 저렴하고 빠르게 만들어야 한다고 생각합니다.

AI 에이전트로부터 적절한 결과를 원한다면, 우리는 모든 훌륭한 엔지니어(engineer)에게 기대할 것과 동일한 구조화된 사고 과정을 통해 그들을 안내해야 합니다. AI가 멍청해서가 아니라 — 복잡한 작업은 분해(decomposition)를 요구하기 때문입니다. 그 작업을 수행하는 것이 인간이든 인공지능이든 상관없이 말입니다.

이 플러그인(plugin)은 정확히 그것을 수행합니다. 팀을 효과적으로 만드는 엔지니어링 규율을 반복 가능한 AI 네이티브(AI-native) 워크플로우(workflow)로 인코딩(encode)합니다.
