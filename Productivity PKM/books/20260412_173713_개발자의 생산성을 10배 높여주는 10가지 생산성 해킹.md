# 개발자의 생산성을 10배 높여주는 10가지 생산성 해킹

어느 평범한 월요일이었습니다. 특별한 일은 없었습니다. 서버 장애도, 화난 클라이언트도 없었습니다. 그저 화면을 응시하던 저는, 단 몇 분이면 끝날 문제에 반나절을 허비했다는 사실을 깨달았습니다.

에디터는 열려 있었고, 코드도 멀쩡했습니다. 하지만 제 정신은 산만했습니다. 너무 많은 탭, 채 끝내지 못한 생각들, 그리고 하루 종일 '바쁘게' 움직였음에도 실제로 끝낸 일은 하나도 없는, 그 특유의 공허한 피로감이 몰려왔습니다. 커피가 두 번째로 식어버린 것을 발견했을 때, 깨달음이 찾아왔습니다.

**저는 코딩을 못 하는 게 아니라, 일하는 법을 못 하는 것이었습니다.**

대부분의 개발자가 남몰래 겪는 문제입니다. 우리는 언어, 프레워크, 시스템 디자인을 배우는 데 수년을 투자하고 클린 코드와 성능에 집착합니다. 하지만 끊임없는 방해와 깊은 사고(Deep thinking)가 공존해야 하는 이 직업에서, 어떻게 주의력과 에너지, 시간을 관리해야 하는지는 아무도 가르쳐주지 않습니다.

그래서 우리는 더 열심히 몰아붙입니다. 더 긴 시간, 더 많은 탭, 더 많은 도구, 그리고 속도를 약속하지만 결국 우리를 더 지치게 만드는 '생산성 해킹'에 매달립니다. 결국 우리가 지치는 이유는 열심히 일하지 않아서가 아니라, 우리의 노력이 사방으로 흩어졌기 때문입니다.

이 글은 단순히 '더 많이' 하는 법에 관한 것이 아닙니다. 집중력이 쉽게 깨지고, 문맥 전환(Context switching)이 빈번하며, 시간이 순식간에 사라지는 환경에서 **'더 잘'** 일하는 법에 관한 이야기입니다.

### 1. 터미널의 지루한 작업 자동화하기 (뇌의 집중력을 위해)

매일 입력하는 명령어를 생각해보세요. 서버 시작, 테스트 실행, Git 상태 확인, 포트 종료, 의존성 재설정 등. 손가락은 익숙하게 움직이지만, 그 과정에서 뇌는 아주 작은 결정들을 내리고 있습니다. "플래그가 뭐였지?", "오늘은 몇 번 포트였지?" 이런 작은 결정들이 하루에 수백 번 반복되면 집중력은 조용히 고갈됩니다.

저만의 규칙은 이렇습니다. **"두 번 이상 반복하는 일은 즉시 자동화한다."**

*   **Level 1: 간단한 별칭(Alias) 활용**
    가장 빠르게 효과를 볼 수 있는 방법입니다. 복잡한 명령어를 짧은 단축키로 만드세요.
    ```bash
    # Git
    alias gs='git status'
    alias gco='git checkout'
    alias gc='git commit -m'
    # Navigation
    alias ..='cd ..'
    # Dev & test
    alias dev='npm run dev'
    alias test='npm test'
    ```
*   **Level 2: 워크플로우를 위한 함수(Function) 작성**
    단순한 별칭으로 부족한 복잡한 과정은 함수로 만듭니다.
    ```bash
    # 새 브랜치 생성 및 푸시
    gnb() {
      git checkout -뮬 "$1"
      git push -u origin "$1"
    }
    # 특정 포트의 프로세스 종료
    killport() {
      lsof -ti:$1 | xargs kill -9
    }
    ```
*   **Level 3: 반복적인 설정을 위한 프로젝트 스크립트**
    서비스 시작, 마이그레이션, 데이터 시딩(Seeding) 등 프로젝트마다 반복되는 의식을 스크립트로 만드세요.
*   **Level 4: 프로젝트 간 표준 명령어 통일**
    프로젝트마다 명령어가 다르면 뇌는 매번 적응해야 합니다. `make dev`, `make test`처럼 프로젝트에 상관없이 동일한 명령어를 사용할 수 있도록 표준화하세요.

**핵심:** 자동화는 타이핑 속도를 높이는 것이 아니라, **정신적 노이즈(Mental noise)를 제거**하여 진짜 중요한 문제에 에너지를 보존하는 것입니다.

### 2. IDE 마스터하기: 코딩 초보에서 디지털 거장으로

IDE(통합 개발 환경)는 개발자의 가장 중요한 도구입니다. 마우스를 사용하는 대신 키보드 중심의 워크플로우를 구축하면 생산성을 2~3배 높일 수 있습니다.

*   **Phase 1: 마우스 의존도 낮추기**
    탐색을 위해 마우스를 쓰는 것은 비효나적입니다. 파일 검색(`Cmd/Ctrl + P`), 심볼 찾기(`Cmd/CLtrl + Shift + O`), 정의로 이동(`F12`), 다중 커서 편집(`Cmd/Ctrl + D`) 등을 익히세요.
*   **Phase 2: 스니펫(Snippet)으로 반복 작업 자동화**
    보일러플레이트(Boilerplate) 코드를 일일이 타이핑하지 마세요. 특정 키워드만 입력하면 구조가 완성되는 스니펫을 활용해 인지 에너지를 아끼세요.
*   **Phase 3: IDE의 지능적 기능 활용**
    심볼 이름 변경(Rename Symbol), 고급 다중 커서, 내장 Git 도구 등을 적극적으로 사용하세요.
*   **Phase 4: 디지털 작업 공간 최적화**
    화면 분할을 전략적으로 사용하고, 자주 사용하는 파일은 탭으로 고정(Pin)하세요.

### 3. 두 개의 터미널 규칙 (뇌를 진정시키는 방법)

문맥 전환(Context switching)의 비용을 줄이기 위해 터미널을 두 개로 분리합니다.

*   **터미널 A (작업용):** 프로젝트 루트에서 실행되는 안정적인 공간입니다. 개발 서버, 테스트, 빌드 등 예측 가능한 작업만 수행합니다.
*   <'span'>>**터미널 B (탐색용):** 로그 확인, 설정 변경, Docker 컨테이너 확인 등 실험적이고 혼란스러운 작업을 위한 공간입니다.

이렇게 분리하면 메인 작업 공간을 깨뜨리지 않고도 자유롭게 탐색할 수 있어 정신적 피로가 급격히 줄어듭니다.

---

### 4. 실효성 있는 타임박싱(Timeboxing)

전통적인 포모도로(Pomodoro) 기법은 개발자의 깊은 몰입(Deep work)을 방해할 수 있습니다. 저는 타이머를 '방해 도구'가 아닌 **'의사결정 도표'**로 사용합니다.

1.  **수행 가능한 작업 정의:** "인증 기능 작업"이 아니라 "JWT 검증 로직 구현"처럼 구체적으로 정합니다.
2.  **정직한 시간 예측:** 예상 시간보다 20~30% 정도 여유를 둡니다.
3.  **타이머 종료 후 평가:** 타이머가 울렸을 때 무조건 멈추는 것이 아니라, **"지금 몰입 상태인가?"**를 스스로 묻습니다. 몰입 중이라면 계속하고, 흐름이 끊겼다면 그때 휴식합니다.

### 5. 15분 규칙 (미루기와 과몰입 방지)

*   **일을 미루고 있을 때:** 딱 15분만 그 일을 하겠다고 자신과 약속하세요. 시작이 가장 어렵습니다. 일단 시작하면 관성이 생겨 계속하게 될 확률이 높습니다.
*   **지나친 탐구(Rabbit hole)에 빠졌을 때:** 15분 타이머를 맞추고 탐구를 시작하세요. 타이머가 울리면 멈추고 자문하세요. "이것이 지금 문제를 해결하는 데 정말 필요한가?"

### 6. 문서 우선 작성법 (Documentation-First)

코드를 다 짠 후 문서를 쓰는 것은 '기억을 재구성'하는 고된 작업입니다. 반대로, 구현 전에 짧은 설명을 먼저 작성해 보세요.

구현 전, 다음 질문에 답을 적어보는 것만으로도 설계 결함을 발견할 수 있습니다.
*   이 기능은 무엇을 하는가?
*   사용자는 어떻게 사용하는가?
*   입력값과 예외 상황(Edge case)은 무엇인가?

문서가 설계도(Blueprint) 역할을 하게 되면, 코딩은 단순한 '구현(Implementation)'의 과정이 됩니다.

---

### 7. 2분 코드 리뷰 규칙 (병목 현상 제거)

코드 리뷰가 늦어지는 이유는 리뷰가 '무겁게' 느껴지기 때문입니다.

**"2분 내로 끝낼 수 있는 리뷰라면 즉시 수행한다."**

작은 풀 리퀘스트(PR)를 즉시 처리하면 팀 전체의 흐름이 빨라지고, 리뷰어의 심리적 부채도 줄어듭니다. 큰 리뷰는 별도의 집중 시간을 할당하여 처리하세요.

---

### 8. 나를 위한 데일리 스탠업(Daily Standup)

매일 아침 에디터를 켜기 전, 5분만 투자해 세 가지만 기록하세요.
1.  **어제 완료한 일**
2.  **오늘 완료할 목표**
3.  **나를 가로막는 장애물(Blockers)**

이 기록은 단순한 보고용이 아니라, 하루의 방향을 잡아주는 내비게이션 도구입니다.

---

### 9. 유사 작업 배치 처리(Batching)

슬랙 메시지 하나 때문에 몰입이 깨지면, 다시 원래 상태로 돌아오는 데 생각보다 많은 시간이 걸립니다. 비슷한 성격의 작업을 묶어서 처리하세요.

*   **딥 워크(Deep work) 시간:** 기능 개발, 복잡한 디버깅
*   **협업 시간:** 코드 리뷰, 회의, 논의
*   **유지보수 시간:** 문서 작성, 리팩터링, 작은 버그 수정

슬랙이나 이메일 확인 시간도 정해두고, 그 외의 시간에는 알림을 꺼두는 것이 좋습니다.

### 10. AI를 '두뇌 대체재'가 아닌 '증폭기'로 활용하기

AI는 강력하지만, 맹신은 위험합니다.

*   **AI가 잘하는 것:** 보일러플레이트 생성, 코드 설명, 단순 변환, 디버깅 보조.
*   **AI가 위험한 것:** 아키텍처 결정, 보안이 중요한 로직, 비즈니스 로직 설계.

**핵점:** AI에게 초안을 작성하게 하되, 최종 판단과 검증은 반드시 인간인 개발자가 해야 합니다. AI를 통해 실행 속도를 높이되, 설계와 판단력은 유지하세요.

---

### 결론: 생산성은 마찰을 제거하는 과정입니다

생산성은 더 많은 일을 하는 것이 아니라, **마찰(Friction)을 제거하는 것**입니다.

위의 10가지 방법 중 지금 당장 당신의 가장 큰 고통을 해결해 줄 수 있는 **두 가지만 선택**하여 2주 동안 실천해 보세요. 생산성은 의지력이 아니라, 잘 설계된 시스템에서 나옵니다.
