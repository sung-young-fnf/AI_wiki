# LLM Wiki 스킬: Claude Code와 Obsidian으로 제2의 뇌 구축하기

카르파티(Karpathy)의 위키 패턴은 지식을 검색하는 대신 '컴파일(compile)'합니다. 여기 프로덕션 환경에서 작동하도록 제가 구축한 완벽한 Claude Code 스킬이 있습니다.

3개월간의 아키텍처 결정 사항들이 14개의 팀 스레드, 두 개의 Notion 데이터베이스, 1월 이후 아무도 업데이트하지 않은 공유 Google 문서, 그리고 제 기억 속에 흩어져 있었습니다. 지난 화요일, 한 신입 엔지니어가 알림 서비스에 REST 대신 이벤트 기반 메시징(event-driven messaging)을 선택한 이유를 물었습니다. 저는 답을 알고 있었습니다. 회의에서 논쟁하고, 화이트보드에 스케치하고, 풀 리퀘스트(pull request) 댓글에서 옹호했던 내용이었죠. 하지만 실제 근거를 찾는 데 25분간 스크롤해야 했습니다.

같은 주에 안드레이 카르파티(Andrej Karpathy)는 “LLM Wiki”라는 제목의 gist를 공개했고, 이는 4일 만에 5,000개의 별과 약 3,000개의 포크(fork)를 얻었습니다. 이 gist의 한 문장이 전체 문제의 프레임을 재구성했습니다. 바로 '검색(retrieval)'에서 '컴파일(compilation)'로의 전환이라는 것이죠. 더 이상 '이 질문에 답하는 문서를 찾아라'가 아닙니다. 대신, '이미 종합된 답변을 포함하고 있는 영구적이고 상호 참조되는 지식 기반을 구축하고 유지하라'는 것입니다.

저는 3월부터 카르파티의 작업에 대해 글을 써왔습니다. 그가 '오토리서치(autoresearch)'를 발표했을 때, 저는 최적화 루프(optimization loop)를 ML 훈련(training)을 넘어 재사용 가능한 Claude Code 스킬로 만들었습니다. 그가 '에이전트허브(AgentHub)'를 출시했을 때, 저는 작동하는 에이전트 템플릿(agent template)과 함께 실용적인 설정 가이드를 작성했습니다. LLM Wiki는 이 삼부작을 완성합니다. 오토리서치는 에이전트에게 최적화하는 방법을, 에이전트허브는 에이전트에게 협업하는 방법을, 그리고 LLM Wiki는 에이전트에게 기억하는 방법을 가르칩니다.

이 글은 세 가지 내용을 다룹니다.

첫째, 이 패턴의 핵심 변화, 즉 왜 '컴파일'이 '검색'보다 우월한지를 설명합니다.

둘째, 위키가 실제로 작업 방식을 어떻게 변화시키는지 두 가지 사용 사례를 통해 보여줍니다.

셋째(이 부분은 다른 어떤 가이드에서도 다루지 않습니다), 오늘 바로 설치하여 사용할 수 있는 완전하고 프로덕션 환경에 적합한 Claude Code 스킬을 제공합니다.

## 핵심 변화: 검색보다 컴파일

모두가 아는 표준 패턴은 RAG(Retrieval Augmented Generation)입니다. 문서를 업로드하면 LLM이 쿼리 시점에 관련 청크(chunk)를 찾아 답변을 생성합니다. 다른 질문을 하면 LLM은 모든 것을 처음부터 다시 발견합니다. 카르파티의 비유는 정확합니다. 이는 마치 도서관의 모든 책을 읽지만, 대화가 멈추는 순간 모든 것을 잊어버리는 연구 조교를 두는 것과 같습니다. 질문을 하면 그는 책장으로 달려가고, 답변을 가지고 돌아옵니다. 후속 질문을 하면 다시 달려갑니다. 그는 어떤 것도 연결하지 않습니다.

위키 패턴은 이를 역전시킵니다. 새로운 소스(source)를 추가할 때, LLM은 단순히 나중에 검색하기 위해 색인을 생성하지 않습니다. LLM은 소스를 읽고 중요한 정보를 추출하며, 기존 위키 페이지를 업데이트하고, 새로운 페이지를 생성하며, 이전 소스와의 모순을 표시하고, 상호 참조를 유지합니다. 지식은 한 번 컴파일되면, 그 상태로 유지됩니다.

세 가지 계층이 이 작업을 가능하게 합니다:

*   **원시 소스(Raw sources)** — 변경 불가능한 문서 모음입니다. 기사, 회의록, 아키텍처 결정, 연구 논문 등이 해당됩니다. LLM은 이들로부터 읽기만 하며, 절대 수정하지 않습니다. 이는 '진실의 원천(source of truth)'입니다.
*   **위키(The wiki)** — LLM이 생성한 마크다운(Markdown) 파일 디렉터리입니다. 요약, 엔티티 페이지, 개념 페이지, 비교 페이지 등이 포함됩니다. LLM은 이 계층을 전적으로 소유합니다. 사용자는 읽고, LLM은 작성하고 유지 관리합니다.
*   **스키마(The schema)** — 위키가 어떻게 구성되어 있고 어떤 워크플로우(workflow)를 따라야 하는지 LLM에 알려주는 구성 파일(CLAUDE.md 또는 스킬 정의)입니다. 이는 LLM을 일반적인 챗봇이 아닌, 규율 있는 위키 관리자로 만듭니다.

대부분의 글에서 간과되는 핵심 통찰력은 지식 기반을 유지 관리하는 데 지루한 부분은 읽거나 생각하는 것이 아니라는 점입니다. 그것은 바로 '장부 정리(bookkeeping)'입니다. 상호 참조 업데이트, 요약 최신화, 새로운 증거가 이전 주장과 모순될 때 기록하는 것 등이죠. 인간은 유지 관리 부담이 가치보다 빠르게 증가하기 때문에 위키를 포기합니다. LLM은 지루해하지 않습니다. 상호 참조 업데이트를 잊지 않으며, 한 번의 작업으로 15개의 파일을 처리할 수 있습니다.

## 실제로 작동하는 두 가지 사용 사례

이 gist는 의도적으로 추상적입니다. 여기 제가 이 패턴이 명확한 가치를 제공하는 것을 본 두 가지 사용 사례가 있습니다.

### 사용 사례 1: CTO 의사결정 위키

이 문제는 엔지니어링을 이끄는 누구에게나 익숙합니다. 아키텍처 결정은 회의록, Slack 스레드, 풀 리퀘스트(pull request) 댓글, 비공식 대화 등 수개월에 걸쳐 축적됩니다. 6개월 후 누군가 “왜 Y 대신 X를 선택했나요?”라고 물으면, 그 근거는 흩어져 있거나 완전히 사라집니다.

위키는 이를 변화시킵니다. `raw/` 디렉터리를 만들고 결정 사항을 포함하는 아티팩트(artifact)를 공급하기 시작합니다:

```
raw/
├── meetings/
│   ├── 2026-01-15-queue-architecture.md
│   ├── 2026-02-03-auth-provider-eval.md
│   └── 2026-03-20-scaling-retrospective.md
├── adrs/
│   ├── adr-007-event-messaging.md
│   └── adr-008-database-migration.md
└── postmortems/
    └── 2026-02-28-notification-outage.md
```

이 소스들을 수집(ingesting)한 후, 위키는 각 서비스 및 벤더에 대한 엔티티 페이지, 논의했던 아키텍처 패턴에 대한 개념 페이지, 그리고 상호 참조가 포함된 의사결정 로그를 포함하게 됩니다. 신입 엔지니어가 알림 서비스에 대해 물었을 때, 그 답변은 이미 컴파일되어 있으며, 트레이드오프(trade-off)가 논의되었던 특정 회의를 인용하고 있습니다.

3회 수집 후 엔티티 페이지의 예시:

```markdown
---
title: "Notification Service"
type: entity
confidence: 0.85
sources:
  - raw/meetings/2026-01-15-queue-architecture.md
  - raw/adrs/adr-007-event-messaging.md
  - raw/postmortems/2026-02-28-notification-outage.md
last_updated: 2026-04-08
stale: false
---
# Notification Service
## Architecture
Event-driven via RabbitMQ (chosen over REST polling in January 2026).
Decision rationale: 3x throughput improvement under load testing,
decoupled failure domains. See [[ADR-007-Event-Messaging]].
## Known Issues
- February 2026 outage caused by queue overflow during batch processing.
  Root cause: missing backpressure configuration. See [[Postmortem-2026-02-28]].
- Current retry policy is exponential with 5 attempts. Team discussed
  switching to dead-letter queue pattern but deferred to Q2.
## Open Questions
- Should we migrate to NATS for multi-region? Discussed but not decided
  in [[Meeting-2026-03-20]].
```

모든 숫자에는 출처가 있고, 모든 결정에는 맥락이 있습니다. 미해결 질문은 명확하게 명시되고 추적됩니다.

### 사용 사례 2: 기술 콘텐츠 연구 위키

저는 한 달에 8~10개의 미디엄(Medium) 아티클을 작성합니다. 각 아티클은 기존 자료를 읽고, 공백을 파악하며, 증거를 수집하는 연구를 필요로 합니다. 위키가 없으면 그 연구는 아티클이 발행된 후 사라져 버립니다. 위키 패턴을 사용하면 각 아티클을 위해 읽은 모든 소스가 동일한 지식 기반으로 유입됩니다. 개념 페이지는 아티클 전반에 걸쳐 축적됩니다. 이 글을 위해 LLM Wiki 주제를 연구하기 시작했을 때, 위키는 이미 저의 오토리서치(autoresearch) 및 에이전트허브(AgentHub) 연구에서 컴파일된 요약본을 가지고 있었습니다. 여기에는 카르파티의 설계 철학, 그의 “아이디어 파일(idea file)” 배포 모델, 그리고 두 프로젝트에 대한 커뮤니티 반응이 포함되어 있었습니다.

수집(Ingest) 워크플로우:

```
# Clip an article to raw/
raw/articles/2026-04-06-mindstudio-llm-wiki-guide.md
# Ingest it
> /wiki-ingest raw/articles/2026-04-06-mindstudio-llm-wiki-guide.md
# The agent reads the source, creates a summary page,
# updates the existing "LLM Wiki" concept page,
# adds a new "MindStudio" entity page,
# updates the index, appends to the log.
```

이 글을 위해 다섯 개의 경쟁 아티클을 연구한 후, “LLM Wiki” 개념 페이지에는 이미 모든 접근 방식의 종합적인 내용(어떤 부분이 일치하고, 어떤 부분이 모순되며, 어떤 공백이 남아있는지)이 포함되어 있었습니다. 저는 어떤 것도 다시 읽을 필요가 없었습니다. 컴파일은 이미 완료된 상태였습니다.

## LLM Wiki 스킬 구축하기: 완전한 단계별 안내

이 부분은 다른 어떤 곳에서도 찾을 수 없는 내용입니다. 카르파티의 gist는 의도적으로 추상적인 '아이디어 파일(idea file)'입니다. 등장한 GitHub 리포지토리(repository, `claude-obsidian`, `MehmetGoekce/llm-wiki`, `obsidian-wiki` 등)들은 구현체이지만, 그 어느 것도 이 패턴을 설치 가능하고 이식 가능하며 확장 가능한 구조를 가진 재사용 가능한 Claude Code 스킬로 패키징하지는 못했습니다.

다음은 전체 스킬 아키텍처(architecture)입니다.

### 스킬 디렉터리 구조

```
llm-wiki/
├── SKILL.md                    # 메인 진입점 — 트리거, 작업, 규칙
├── references/
│   ├── schema.md               # 위키 구조, 페이지 유형, 프런트매터 사양
│   ├── page-templates.md       # 모든 페이지 유형에 대한 템플릿
│   └── lint-rules.md           # 상태 점검 및 모순 탐지
├── commands/
│   ├── wiki-init.md            # 새 위키 볼트 부트스트랩
│   ├── wiki-ingest.md          # 위키에 소스 처리
│   ├── wiki-query.md           # 종합 및 인용을 통한 쿼리
│   └── wiki-lint.md            # 유지 관리 및 상태 점검
└── hooks/
    └── session-start.md        # 세션 시작 시 컨텍스트 자동 로드
```

### SKILL.md — 메인 진입점

`SKILL.md` 파일은 Claude Code가 스킬을 언제 어떻게 사용할지 이해하기 위해 읽는 파일입니다. 다음은 전체 파일입니다.

```markdown
---
name: llm-wiki
description: >
  카르파티의 LLM Wiki 패턴을 사용하여 영구적으로 축적되는 지식 기반을 구축하고 유지 관리합니다.
  세 가지 작업(operation)을 관리합니다. 수집(ingest): 새 소스를 위키로 처리,
  쿼리(query): 컴파일된 지식에서 답변 종합, 린트(lint): 위키의 모순, 노후화,
  고아 페이지(orphan page) 상태 점검. Obsidian을 뷰잉 레이어(viewing layer)로,
  Git을 버전 관리(version history)에 활용합니다.
triggers:
  - /wiki-init
  - /wiki-ingest
  - /wiki-query
  - /wiki-lint
  - 사용자가 "위키", "지식 기반", "제2의 뇌"를 언급할 때
  - 사용자가 과거 연구, 결정 또는 축적된 지식에 대해 물을 때
context: fork
---
# LLM Wiki 스킬
## 아키텍처
세 가지 계층, 엄격한 분리:
1.  **raw/** — 변경 불가능한 원시 문서. 사용자(사람)가 파일을 추가합니다.
    에이전트는 raw/에서 읽기만 하며, 절대 쓰지 않습니다.
2.  **wiki/** — 에이전트가 유지 관리하는 페이지. 에이전트가 이 디렉터리를
    전적으로 소유합니다. 페이지를 생성, 업데이트 및 상호 참조합니다.
    사용자(사람)는 wiki/를 읽지만 직접 편집하는 경우는 드뭅니다.
3.  **이 스킬 (스키마)** — 에이전트를 일반적인 챗봇이 아닌,
    규율 있는 유지 관리자로 만드는 규칙 및 워크플로우(workflow).
## 작업(Operations)
### /wiki-init
새 위키 볼트(vault)를 부트스트랩(bootstrap)합니다. `commands/wiki-init.md`를 읽으세요.
### /wiki-ingest <경로-또는-URL>
새로운 소스를 위키로 처리합니다. `commands/wiki-ingest.md`를 읽으세요.
### /wiki-query <질문>
컴파일된 위키에 질의합니다. `commands/wiki-query.md`를 읽으세요.
### /wiki-lint
위키의 상태를 점검합니다. `commands/wiki-lint.md`를 읽으세요.
## 규칙 (항상 준수)
1.  **raw/를 절대 수정하지 마세요.** 원시 소스는 변경 불가능한 진실입니다.
2.  **항상 출처를 인용하세요.** 위키 페이지의 모든 주장은
    특정 원시 소스 파일을 참조해야 합니다.
3.  **[[위키링크(wikilinks)]]를 사용하세요.** 모든 상호 참조는 Obsidian의
    이중 대괄호 문법을 사용합니다. 그래프 연결성을 위해.
4.  **모든 페이지에 YAML 프런트매터(frontmatter)를 사용하세요.** 제목, 유형, 신뢰도,
    소스, 최종 업데이트 날짜, 노후화(stale) 플래그가 포함됩니다.
5.  **모든 작업 후에 index.md를 업데이트하세요.** 색인(index)은
    사용자(및 에이전트)가 위키를 탐색하는 방법입니다.
6.  **모든 작업 후에 log.md에 추가하세요.** 형식:
    `## [YYYY-MM-DD] operation | description`
7.  **모든 작업 후에 Git 커밋(commit)을 하세요.**
    수집/질의 제출/린트(lint) 작업당 하나의 커밋과 설명적인 메시지를 사용합니다.
```

### 스키마: 페이지 유형 및 프런트매터

이 내용은 `references/schema.md`에 들어갑니다.

```markdown
# 위키 스키마
## 페이지 유형
### 소스 요약
수집(ingest) 중에 생성됩니다. 원시 소스 파일당 하나입니다.
-   위치: wiki/sources/<슬러그>.md
-   내용: 메타데이터(metadata), 주요 주장, 관련성 평가
-   연결: 업데이트하는 개념 및 엔티티 페이지
### 개념 페이지
아이디어, 패턴 또는 도메인 주제를 나타냅니다.
-   위치: wiki/concepts/<이름>.md
-   내용: 정의, 여러 소스의 증거, 미해결 질문, 관련 개념
-   새로운 소스에서 개념을 언급할 때마다 시간이 지남에 따라 성장
### 엔티티 페이지
특정 대상을 나타냅니다: 사람, 도구, 서비스, 회사.
-   위치: wiki/entities/<이름>.md
-   내용: 속성(attributes), 언급 타임라인(timeline), 관계
-   새로운 소스가 엔티티를 참조할 때마다 업데이트됨
### 비교 페이지
에이전트 또는 사용자가 경쟁 접근 방식을 식별할 때 생성됩니다.
-   위치: wiki/comparisons/<A>-vs-<B>.md
-   내용: 기준, 각 측의 증거, 명확한 경우 판결
### 종합 페이지
쿼리 결과 중 보존할 가치가 있는 내용에서 생성됩니다.
-   위치: wiki/syntheses/<주제>.md
-   내용: 종합된 답변, 참조한 소스, 신뢰도
## YAML 프런트매터 사양 (모든 페이지에 필수)
    ---
    title: "페이지 제목"
    type: source | concept | entity | comparison | synthesis
    confidence: 0.0-1.0
    sources:
      - raw/path/to/source1.md
      - raw/path/to/source2.md
    last_updated: YYYY-MM-DD
    stale: false
    tags: [tag1, tag2]
    ---
## 특수 파일
### index.md
콘텐츠 지향적 카탈로그. 각 페이지는 다음을 포함하여 나열됩니다:
-   링크, 한 줄 요약, 유형, 신뢰도, 소스 수.
-   유형별로(소스, 개념, 엔티티 등) 정리됩니다.
-   모든 수집(ingest) 작업 시 업데이트됩니다.
### log.md
연대순, 추가 전용. 형식:
    ## [2026-04-08] ingest | Article: "LLM Wiki Patterns"
    - Created: wiki/sources/llm-wiki-patterns.md
    - Updated: wiki/concepts/RAG.md, wiki/entities/Karpathy.md
    - New page: wiki/concepts/Knowledge-Compilation.md
### overview.md
전체 위키의 상위 수준 종합. 5번째 수집(ingest) 후 또는
명시적 요청 시 업데이트됩니다. 현재 지식 상태, 주요 주제,
미해결 질문을 기록합니다.
```

### 수집(Ingest) 명령: 단계별 안내

이것이 가장 중요한 작업입니다. `commands/wiki-ingest.md` 파일에는 에이전트가 따를 정확한 지침이 포함되어 있습니다.

```markdown
# /wiki-ingest <소스-경로-또는-URL>
## 워크플로우(Workflow)
### 1단계: 소스 검증
-   `raw/`에 소스 파일이 존재하는지 확인 (또는 URL을 `raw/`로 다운로드)
-   이 소스가 이전에 수집(ingested)되었는지 확인 (`log.md` 검색)
-   이미 수집되었다면 사용자에게 질문: "이 소스는 [날짜]에 이전에 수집되었습니다. 업데이트를 반영하기 위해 다시 수집할까요?"
### 2단계: 읽고 분석
-   전체 소스 문서 읽기
-   식별: 주요 주장, 언급된 엔티티, 논의된 개념,
    데이터 포인트, 결론, 그리고 기존 위키 내용과 모순되는 주장
### 3단계: 사용자와 논의
-   소스에서 3~5가지 주요 핵심 요약 제시
-   질문: "강조하거나 건너뛰고 싶은 부분이 있나요?"
-   진행하기 전에 사용자 응답 대기
### 4단계: 페이지 생성/업데이트
-   `wiki/sources/`에 소스 요약 페이지 생성
-   언급된 각 엔티티에 대해:
    -   엔티티 페이지가 존재하면: 새 정보로 업데이트,
        프런트매터(frontmatter)에 소스 추가, 신뢰도 업데이트
    -   새로운 경우: 템플릿에서 엔티티 페이지 생성
-   논의된 각 개념에 대해:
    -   개념 페이지가 존재하면: 새 증거 섹션 추가,
        모순 여부 확인, 신뢰도 업데이트
    -   새로운 경우: 템플릿에서 개념 페이지 생성
-   소스가 두 가지 접근 방식을 비교하는 경우:
    비교 페이지 생성 고려
### 5단계: 탐색 업데이트
-   새/수정된 페이지로 `index.md` 업데이트
-   모든 세부 정보와 함께 작업(operation)을 `log.md`에 추가
-   5번째 수집(ingest) 작업인 경우 `overview.md` 업데이트
### 6단계: 커밋(Commit)
-   변경된 모든 파일 Git에 추가
-   커밋 메시지: "ingest: [소스 제목] — [N]개의 페이지 변경"
## 규칙
-   `raw/`의 파일을 절대 수정하지 마세요.
-   위키 페이지에는 항상 소스 인용을 포함하세요.
-   주장의 신뢰도가 0.5 미만이면 명시적으로 표시하세요.
-   새로운 주장이 기존 위키 페이지와 모순되면 '모순(Contradictions)' 섹션을 추가하세요.
    조용히 덮어쓰지 마세요.
```

### 쿼리 명령

`commands/wiki-query.md` 파일:

```markdown
# /wiki-query <질문>
## 워크플로우(Workflow)
### 1단계: 색인(Index) 읽기
-   관련 페이지를 식별하기 위해 `wiki/index.md` 읽기
-   답변을 포함할 가능성이 가장 높은 페이지 선택
### 2단계: 관련 페이지 읽기
-   선택된 위키 페이지 읽기 (원시 소스 아님)
-   위키 페이지가 중요한 주장에 대해 원시 소스를 참조하는 경우,
    원시 소스와 대조하여 확인
### 3단계: 답변 종합
-   여러 위키 페이지에서 답변을 컴파일
-   인용 포함: "[[페이지-이름]]에 따르면, ..."
-   신뢰도 수준 및 모순 사항 기록
### 4단계: 파일 저장 제안
-   질문: "이 답변을 종합 페이지로 저장할까요?"
-   그렇다면: `wiki/syntheses/<주제>.md` 생성
-   `index.md` 및 `log.md` 업데이트
## 규칙
-   항상 참조한 위키 페이지를 인용하세요.
-   위키에 답변할 충분한 정보가 없으면 그렇게 말하세요.
    일반 지식으로 회귀하지 마세요.
-   훈련 데이터보다 위키 콘텐츠를 선호하세요.
```

### 린트(Lint) 명령

`commands/wiki-lint.md` 파일:

```markdown
# /wiki-lint
## 실행할 검사
1.  **고아 페이지(Orphan Page) 탐지**
    -   인바운드 [[위키링크]]가 없는 페이지 찾기
    -   연결 제안 또는 검토 플래그 지정
2.  **노후화(Staleness) 검사**
    -   `last_updated`가 30일 이상인 페이지 플래그 지정
    -   `stale: true`인 페이지 플래그 지정
3.  **모순 스캔(Contradiction Scan)**
    -   동일한 주제에 대한 페이지 간 주장 비교
    -   출처가 승인 없이 불일치하는 부분 플래그 지정
4.  **신뢰도 감사(Confidence Audit)**
    -   신뢰도 0.5 미만 페이지 목록화
    -   신뢰도를 높일 수 있는 소스 제안
5.  **누락된 페이지**
    -   존재하지 않는 페이지를 가리키는 [[위키링크]] 찾기
    -   생성 제안
6.  **색인(Index) 동기화**
    -   모든 위키 페이지가 `index.md`에 나타나는지 확인
    -   색인 항목이 삭제된 페이지를 가리키지 않는지 확인
## 출력
-   각 검사별 개수가 포함된 상태 보고서 출력
-   주의가 필요한 특정 페이지 목록
-   공백을 기반으로 찾아야 할 새로운 소스 제안
-   린트(lint) 결과를 `log.md`에 추가
```

### 세션 시작 훅(Hook)

`hooks/session-start.md` 파일은 에이전트가 모든 세션에서 컨텍스트를 갖도록 보장합니다.

```markdown
# 세션 시작 훅(Hook)
위키를 포함하는 디렉터리에서 세션이 시작될 때:
1.  `wiki/index.md` 읽기 (전체 파일)
2.  `wiki/log.md`의 마지막 10개 항목 읽기
3.  총 페이지 수와 마지막 작업 날짜 기록
4.  사용자가 경로를 지정할 필요 없이 위키 내용에 대한 질문에 답변할 준비가 되어 있어야 함
```

### 설치

스킬을 Claude Code 스킬 디렉터리에 복사하고 Obsidian 볼트(vault)를 가리키도록 합니다.

```bash
# 스킬 복제 또는 복사
cp -r llm-wiki/ ~/.claude/skills/llm-wiki/
# Obsidian 볼트에서 Claude Code 열기
cd ~/Documents/ObsidianVault
claude
# 위키 부트스트랩
> /wiki-init
# 소스 수집 시작
> /wiki-ingest raw/articles/first-article.md
```

`/wiki-init` 명령은 전체 디렉터리 구조를 생성하고, Git 추적을 초기화하며, 비어 있는 `index.md`, `log.md`, `overview.md` 파일을 만듭니다. 이후 모든 수집(ingest) 작업은 위키를 점진적으로 구축합니다.

## 문제점

이 패턴은 강력하지만, 마찰이 없는 것은 아닙니다.

**규모 한계.** 에이전트가 `index.md`를 읽어 관련 페이지를 찾는 색인 기반 접근 방식은 약 200페이지 및 100개 소스까지 잘 작동합니다. 그 이상이 되면, 색인 자체가 효율적인 스캔(scanning)에 너무 커집니다. 완화책은 시맨틱 검색(semantic search)을 추가하는 것입니다. 토비 루트케(Tobi Lutke)의 `qmd` 도구는 MCP 서버 인터페이스를 통해 마크다운 파일에 대한 하이브리드 BM25/벡터 검색(vector search)을 제공합니다. 하지만 대부분의 개인 위키에서는 몇 달 동안 이 한계에 도달하지 않을 것입니다.

**컴파일 중 환각(hallucination).** LLM은 수집(ingest) 중에 소스를 잘못 해석할 수 있습니다. 주장을 잘못된 회의에 귀속시키거나, 유사하지만 별개의 두 개념을 하나의 페이지로 병합하는 것을 본 적이 있습니다. 완화책은 엄격한 인용 강제(모든 주장은 특정 원시 소스 파일을 참조해야 함)와 모순을 찾아내는 정기적인 린트(lint) 작업입니다. 원시 소스는 변경 불가능하게 유지되므로 항상 확인할 수 있습니다.

**컨텍스트 윈도우(context window) 압력.** 긴 문서(10,000단어짜리 연구 논문, 전체 회의록)의 단일 수집(ingest)은 토큰(token) 제한에 부딪힐 수 있으며, 특히 에이전트가 업데이트를 확인하기 위해 기존 위키 페이지를 읽어야 할 때 더욱 그렇습니다. 완화책은 스킬의 설계에 있습니다. 에이전트는 먼저 색인(index)을 읽고(간결함), 업데이트할 페이지를 식별하며(선별적), 해당 특정 페이지만 로드합니다.

**단일 에이전트 쓰기.** 두 에이전트(또는 두 개의 Claude Code 세션)가 위키에 동시에 쓰기 작업을 하면 병합 충돌(merge conflicts)이 발생합니다. 카르파티의 gist는 이 문제를 해결하지 않습니다. 단독 사용의 경우 문제가 되지 않지만, 팀 사용의 경우 조정이 필요합니다. 한 번에 하나의 에이전트 세션만 사용하거나, 에이전트가 변경 사항을 제안하고 사람이 승인하는 풀 리퀘스트(pull request) 검토 패턴을 사용해야 합니다.

**유지 관리 규율.** 위키는 데이터를 공급할 때만 성장합니다. 방치하면 노트가 단순히 오래되는 정적인 노트 앱과 달리, 방치된 위키는 다음 수집(ingest) 작업에서 조정해야 할 노후화(staleness)를 발생시킵니다. 소스를 발견하는 즉시 수집하는 습관을 들이고, 일괄 처리하지 마세요.

## 아직 해결 중인 문제

개인 위키가 언제 팀 위키가 되는가? 스키마 변경은 분명합니다. 기여자 추적, 검토 게이트(review gates), 접근 제어 등을 추가하는 것이죠. 하지만 더 어려운 질문은 '신뢰'입니다. 개인 위키는 사용자가 맥락을 알기 때문에 개념 페이지에 0.7의 신뢰도를 허용할 수 있습니다. 팀 위키는 다른 사람이 사용자 배경 지식 없이 읽을 것이기 때문에 더 높은 기준이 필요합니다.

린트(lint) 작업이 자율적으로 실행될 수 있을까? 인간 개입 없이 위키를 매일 밤 상태 점검하고, 모순을 찾아내고, 공백을 메울 소스를 제안하는 에이전트 말이죠. 패턴은 이를 지원하지만, 아직 신뢰 모델은 그렇지 않습니다.

그리고 제가 계속해서 되묻는 질문은: 우리는 사고를 더 잘하게 만드는 도구를 구축하고 있는가, 아니면 우리 대신 생각하는 도구를 구축하고 있는가? 이 위키 패턴은 흥미로운 경계에 놓여 있습니다. 에이전트는 장부 정리를 담당합니다. 하지만 인간은 여전히 무엇을 수집할지, 어떤 질문을 할지, 그리고 컴파일된 지식이 무엇을 의미하는지 결정합니다. 이러한 업무 분할은 적절하다고 느껴지며, 저는 이것이 계속 유지될지 지켜보고 있습니다.

## LLM Wiki 및 Claude Code에 대한 자주 묻는 질문

**카르파티의 LLM Wiki는 무엇이며 RAG와 어떻게 다른가요?**

LLM Wiki는 LLM이 원시 문서를 쿼리 시점에 검색하는 대신, 구조화된 마크다운 파일을 컴파일하고 유지 관리하는 영구적인 지식 기반입니다. RAG 시스템에서는 모델이 모든 질문에 대해 처음부터 관련 정보를 다시 발견합니다. 위키 패턴에서는 지식이 수집(ingest) 중에 한 번 컴파일됩니다. 요약이 작성되고, 상호 참조가 유지되며, 모순이 표시된 다음 컴파일된 상태로 유지됩니다. 위키는 소스가 추가될 때마다 더욱 풍부해집니다. RAG는 상태를 유지하지 않습니다.

**Claude Code 및 Obsidian으로 LLM Wiki를 어떻게 설정하나요?**

`llm-wiki` Claude Code 스킬을 설치하고, Obsidian 볼트(vault) 디렉터리에서 Claude Code를 연 다음, `/wiki-init`를 실행하여 구조를 부트스트랩(bootstrap)합니다. 이 스킬은 세 가지 계층 아키텍처(원시 소스, 위키 페이지, 스키마)를 생성하고, Git 추적을 초기화하며, 색인 및 로그 파일을 설정합니다. 그 다음 `raw/`에 소스 파일을 넣고 `/wiki-ingest`를 실행하여 각 파일을 처리합니다. Obsidian은 그래프 뷰, 백링크, 실시간 편집 기능을 통해 뷰잉 레이어(viewing layer)를 제공하고, Claude Code는 컴파일 및 유지 관리를 수행합니다.

**팀 지식 관리용으로 LLM Wiki 패턴을 사용할 수 있나요?**

네, 하지만 기본 패턴에는 포함되지 않은 조정 메커니즘이 필요합니다. 단독 사용의 경우, 한 번에 하나의 Claude Code 세션만 사용하면 쓰기 충돌을 피할 수 있습니다. 팀의 경우, 검토 큐를 추가해야 합니다. 에이전트가 Git 브랜치(branch) 형태로 위키 업데이트를 제안하고, 사람이 병합(merge) 전에 검토하는 방식입니다. 신뢰도 점수와 소스 인용이 포함된 YAML 프런트매터(frontmatter)는 검토자에게 변경 사항을 승인하거나 거부하는 데 필요한 컨텍스트를 제공합니다. 몇몇 커뮤니티 구현(AI-Context-OS, Continuity)은 다중 사용자 조정 계층을 추가했습니다.

**LLM Wiki는 성능 저하 없이 얼마나 커질 수 있나요?**

색인 기반 접근 방식은 약 200페이지 및 100개 소스까지 잘 작동합니다. 이 규모에서는 에이전트가 전체 색인을 읽어 관련 페이지를 찾습니다. 그 이상이 되면 검색 계층이 필요합니다. `qmd`는 MCP 서버를 통해 하이브리드 BM25/벡터 검색(vector search)을 제공하거나, 더 간단한 `grep` 기반 검색 스크립트를 구축할 수 있습니다. 위키 패턴 자체는 무한정 확장 가능하며, 제한 요소는 에이전트가 읽을 관련 페이지를 검색하는 방법입니다.
