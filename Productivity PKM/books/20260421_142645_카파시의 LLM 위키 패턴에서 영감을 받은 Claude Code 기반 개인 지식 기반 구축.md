# 카파시의 LLM 위키 패턴에서 영감을 받은 Claude Code 기반 개인 지식 기반 구축

3계층 아키텍처가 어떻게 LLM을 실제로 서가를 정리하는 사서로 바꾸는가

지난 주말, 저는 Andrej Karpathy(안드레이 카파시)의 LLM 위키(LLM Wiki)라는 아이디어, 즉 언어 모델(LLM)이 단순히 질문에 답하는 것을 넘어 여러분을 대신해 살아있는 지식 기반을 유지하는 패턴에 대한 글을 우연히 발견했습니다. 글을 읽으면서 조용히 '이거다!' 싶었고, 제가 매주 작업하는 두 가지 영역에 이 시스템을 적용하고 싶다고 생각했습니다.

한 시간 뒤, 저는 이를 구축했습니다. 미디엄(Medium) 글쓰기를 위한 위키 하나와 SQL, 파이썬(Python), 파워 BI(Power BI)를 사용한 BI 보고를 위한 위키 하나, 이렇게 두 개의 구조화된 위키를 클로드 코드(Claude Code)와의 대화만으로 완전히 만들었습니다.

여기서 이 패턴이 무엇인지, 왜 중요한지, 그리고 여러분이 직접 어떻게 구축할 수 있는지 설명해 드립니다.

## 카파시의 제안

LLM 위키 패턴은 간단한 아키텍처적 통찰력에 기반합니다. 우리 대부분은 AI를 동일한 방식으로 사용합니다. 질문하고, 답을 얻고, 탭을 닫습니다. 다음 날, 같은 질문의 약간 다른 버전을 다시 묻습니다. 지식이 결코 축적되지 않습니다.

카파시의 대안은 세 가지 계층으로 구성됩니다.

*   **원천 자료(Raw sources)** — 여러분이 관리하는 변경 불가능한 문서들입니다. 읽었던 기사, 작동했던 코드 조각, 분석 보고서, 회의록 등이 해당하며, 한 번 들어가면 절대 변하지 않습니다.
*   **위키(The wiki)** — LLM이 생성하고 유지 관리하는 마크다운(Markdown) 페이지입니다. 요약, 패턴, 상호 참조, 작동 예시 등을 포함하며, 이 계층이 성장합니다.
*   **스키마(The schema)** — LLM의 작동 방식을 알려주는 구성 문서입니다. 명명 규칙, 페이지 구조, 수행할 작업, 각 작업의 책임자 등을 정의합니다.

두 개의 보조 파일이 모든 것을 연결합니다. 분류별 검색을 위한 `index.md`와 모든 변경 사항을 시간순으로 추적하는 `log.md`입니다.

핵심적인 변화는 철학적인 것입니다. 매번 원천 문서에서 정보를 검색하는 표준 RAG(Retrieal Augmented Generation) 접근 방식 대신, LLM은 구조화된 지식을 점진적으로 구축합니다. 새로운 원천 자료는 기존 페이지에 통합되고, 상호 참조는 업데이트되며, 모순은 플래그가 지정됩니다. 위키가 기억하므로 여러분은 기억할 필요가 없습니다.

## 이 아이디어가 저에게 공감을 준 이유

저는 두 가지 주제에 대해 글을 씁니다. 블로그 글쓰기 기술(특히 미디엄(Medium)에서)과 BI(Business Intelligence) 보고서 작성 기술입니다. 두 영역 모두 답답한 패턴을 공유합니다. 바로 이미 배운 것을 계속해서 재발견한다는 점입니다.

미디엄의 알고리즘은 조회 비율과 박수(claps)를 어떻게 가중할까요? 저는 이 내용을 네 번이나 찾아봤습니다. 연간 성장률(year-over-year growth)을 위한 가장 깔끔한 DAX(Data Analysis Expressions) 패턴은 무엇일까요? 저는 그것을 작성했다가 `SAMEPERIODLASTYEAR` 구문을 잊어버리고 처음부터 다시 작성하기를 셀 수 없이 반복했습니다.

전통적인 메모 작성으로는 이 문제를 해결할 수 없습니다. 제 메모는 길고, 체계적이지 못하며, 결국에는 버려집니다. 유지 관리 부담이 가치보다 빠르게 증가합니다. 이는 카파시가 지적한 실패 모드와 정확히 일치합니다.

LLM 위키는 유지 관리 방정식을 뒤집습니다. 요약, 상호 참조, 일관성 유지와 같은 지루한 작업은 언어 모델이 잘하는 일입니다. 인간은 흥미로운 부분, 즉 무엇을 배울지 선택하고, 올바른 질문을 하고, 창의적인 결정을 내리는 일에 계속 책임감을 가집니다.

## 한 세션 만에 구축한 것

클로드 코드(Claude Code)를 열고 제가 원하는 것을 설명했습니다. 단 한 번의 대화로, 두 도메인에 대한 완전한 구조를 생성했습니다.

### 블로그 글쓰기 위키 (6 페이지)

스키마(schema)는 위키의 범위를 콘텐츠 전략, 글쓰기 기술, 미디엄(Medium) 특정 메커니즘, 편집 워크플로우, 분석으로 정의합니다. 각 페이지는 YAML 프론트매터(YAML front-matter), 위키 스타일의 상호 링크(cross-links), 출처 인용과 함께 일관된 형식을 따릅니다.

초기 페이지는 다음을 포함합니다.

*   **미디엄 플랫폼 가이드** — 배포, 파트너 프로그램(Partner Program), 순위 신호가 실제로 작동하는 방식
*   **글쓰기 구조 및 후크** — 서두 패턴, 본문 구조(리스트형, 서사형, 문제-해결형), 글의 속도 조절 규칙
*   **미디엄 SEO 전략** — 미디엄의 도메인 권한이 중요한 이유, 온페이지 최적화(on-page optimization), 검색을 위한 콘텐츠 전략
*   **콘텐츠 아이디어 발상** — 아이디어가 어디서 오는지, 검증 체크리스트, 기둥-위성 모델(pillar-and-satellite model)
*   **편집 워크플로우** — 아이디어 발상부터 게시 후 분석까지 5단계 파이프라인
*   **목소리와 톤** — 디너 파티 테스트(dinner party test), 목소리 속성, 콘텐츠 유형별 톤 조절

### BI 보고 위키 (7 페이지)

이 위키는 제가 매일 사용하는 전체 스택(stack)을 다룹니다. 쿼리(query)를 위한 SQL, 데이터 정리 및 자동화를 위한 파이썬(Python), 시각화 및 제공을 위한 파워 BI(Power BI)입니다.

초기 페이지는 다음을 포함합니다.

*   **SQL 쿼리 패턴** — CTE(Common Table Expression), 윈도우 함수(window functions), 조건부 집계(conditional aggregation), 날짜 스파인(date spines), 피해야 할 안티패턴(anti-patterns)
*   **SQL 성능 최적화** — EXPLAIN 플랜(EXPLAIN plans), 인덱싱 전략(indexing strategy), 사그이블 프레디케이트(sargable predicates), 구체화된 뷰(materialized views)
*   **데이터 모델링: 스타 스키마(Star Schema)** — 팩트(facts), 디멘션(dimensions), 날짜 디멘션(date dimension) 설계, 느리게 변하는 디멘션(slowly changing dimensions)
*   **BI를 위한 파이썬** — 파이썬이 SQL 및 파워 BI와 함께 어떻게 사용되는지, 판다스(pandas) 핵심 사항, 자동화 패턴
*   **파이썬-SQL 통합** — SQLAlchemy(SQLAlchemy) 연결, 매개변수화된 쿼리(parameterized queries), 진자 템플릿(Jinja templating), 데이터 품질 테스트
*   **DAX 핵심 사항** — 측정값(measures) 대 계산 열(calculated columns), CALCULATE(CALCULATE), 시간 인텔리전스(time intelligence), 반복자 함수(iterator functions)
*   **파워 BI 데이터 모델링** — 버티팩 최적화(Vertipaq optimization), 파워 쿼리 폴딩(Power Query folding), 행 수준 보안(row-level security), 배포 파이프라인(deployment pipelines)

모든 페이지에는 작동하는 코드 예제가 포함되어 있습니다. SQL 페이지에는 실제 쿼리가 있고, DAX 페이지에는 실제 측정값이 있으며, 파이썬 페이지에는 실제 판다스(pandas) 변환이 있습니다. 이것은 추상적인 문서가 아니라, 제가 복사하여 사용할 수 있는 참조 자료입니다.

## 위키가 성장하는 방식

진정한 가치는 초기 페이지에 있는 것이 아닙니다. 다음에 일어날 일에 있습니다.

### 수집(Ingest)

새로운 미디엄(Medium) 알고리즘 변경에 대한 기사, 동료의 기발한 DAX(Data Analysis Expressions) 패턴, 컨퍼런스 강연에서 얻은 성능 팁 등 유용한 것을 읽을 때마다 `raw/` 폴더에 넣고 클로드 코드(Claude Code)에 수집(ingest)하라고 요청합니다. LLM은 원본을 읽고, 핵심 내용을 추출하며, 관련 위키 페이지를 업데이트합니다. 새로운 개념은 새로운 페이지를 만들고, 기존 페이지는 더욱 풍부해집니다.

### 조회(Query)

뭔가를 기억해야 할 때, 인터넷 검색 대신 위키를 조회합니다. LLM은 기존 페이지에서 답변을 종합하고, 출처를 인용합니다. 만약 답변이 재사용하기에 충분히 중요하면, 자동으로 새로운 위키 페이지가 됩니다.

### 린트(Lint)

주기적으로 LLM에게 위키의 상태를 확인해 달라고 요청합니다. LLM은 페이지 간의 모순, 오래된 조언(미디엄은 기능이 끊임없이 변함), 인바운드 링크가 없는 고립된 페이지, 주제는 참조되지만 페이지가 아직 존재하지 않는 간극 등을 찾습니다.

## 업무 분담

이 시스템이 지속 가능한 이유는 명확한 책임 분담에 있습니다.

| 여러분(You)           | LLM(The LLM)                          |
| :-------------------- | :------------------------------------ |
| 무엇을 배울지 선택     | 요약 및 구조화                       |
| 창의적인 방향 설정     | 상호 참조 및 연결                   |
| 판단 결정             | 일관성 유지                          |
| 무엇을 게시할지 결정   | 간극 및 모순 사항 보고             |

여러분은 주도권을 잡습니다. LLM은 여러분이 직접 하지 않을 장부 정리 업무를 처리합니다.

## 시작하기

만약 여러분도 이 시스템을 시도해보고 싶다면, 최소한의 구조는 다음과 같습니다.

```
my-wiki/
├── schema.md     ← 규칙 및 관례
├── index.md      ← 페이지 카탈로그
├── log.md        ← 변경 내역
├── raw/          ← 여러분의 원천 자료
└── wiki/         ← LLM이 관리하는 페이지
```

도메인 범위, 파일 명명 규칙, 그리고 세 가지 작업(수집(ingest), 조회(query), 린트(lint))을 정의하는 스키마(schema)를 작성하세요. 그런 다음 선호하는 코딩 보조 도구와 대화하여 여러분이 이미 알고 있는 내용을 기반으로 초기 위키 페이지를 생성하도록 요청하세요.

첫 번째 세션은 기반을 제공합니다. 두 번째 세션에서 첫 번째 외부 원천 자료를 수집하고 위키가 스스로 업데이트되는 것을 볼 때, 이 패턴의 진정한 가치를 깨닫게 될 것입니다.

## 제가 배우고 있는 것

며칠 동안 이 시스템을 사용하면서 세 가지가 두드러지게 나타났습니다.

첫째, 위키는 제가 읽는 방식을 변화시킵니다. 새로운 지식이 착륙할 곳이 있다는 것을 알면 더 주의 깊게 읽게 됩니다. 저는 이미 이 내용이 어떤 페이지를 업데이트할지, 무엇과 모순될지, 어떤 간극을 채울지에 대해 생각하고 있습니다.

둘째, 스키마(schema)가 가장 중요한 파일입니다. 위키 페이지 자체가 아니라 — 페이지는 재생성될 수 있습니다. 스키마는 무엇이 중요한지, 어떻게 구성되어야 하는지, LLM이 무엇에 신경 써야 하는지에 대한 여러분의 판단을 인코딩합니다.

셋째, 이것은 개인 인프라입니다. 제품이나 프레임워크가 아닙니다. 한 번에 하나의 원천 자료씩, 축적되는 지식을 구축하는 조용한 실천입니다.

카파시는 LLM 위키를 Vannevar Bush(배니바 부시)의 Memex(메멕스)의 현대판이라고 설명했습니다. 메멕스는 1945년에 한 사람이 읽었던 모든 것을 저장하고 검색하기 위해 상상되었던 장치입니다. 우리는 그 아이디어를 따라잡을 기술을 80년 동안 기다려왔습니다.

저는 마침내 그 기술이 도래했다고 생각합니다.
